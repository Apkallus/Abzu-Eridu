# 动态分配器误用

glibc 堆由许多不同的组件组成，这些组件在性能和安全之间取得平衡。
在本堆介绍中，将针对线程缓存层 `tcache` 进行利用。
`tcache` 是一个快速的线程特定缓存层，通常是处理动态内存分配的程序首先交互的层。

## 动态分配器误用资源

错误使用堆的方式
- 未释放内存
    - 内存占用
    - 敏感数据长期留在内存里
- 释放已经释放的内存
    - 重复释放/或 freelist 被破坏，会让 allocator 的“空闲块目录”出现重复/伪造条目，从而让两次 malloc 返回同一地址或返回异常地址。
- 使用已经释放的内存
    - 若该 chunk 已进入 freelist，其 payload 被当作链表节点字段，继续写可能破坏 freelist；继续读可能读到“分配器写入的指针/结构信息”或新对象数据。
- 堆缓冲区越界
    - 改坏相邻对象数据或 chunk 元数据；或泄露相邻对象/元数据内容。

每个 tcache 桶（bin）是一个单链表，最多只能容纳 7 个空闲的 chunk

在 64 位系统上，tcache 管理 24 字节到 1032 字节（0x408） 范围内的 chunk
- 范围，格式 `用户请求大小范围 (对应的chunk实际大小)`：
    - 索引1为24(32)
    - 中间索引均为15(16)
    - 索引63为32(32)

分配释放的顺序
- 新释放的在链表头
- 新分配的使用链表头

块结构：
- 元数据
    1. `prev_size` (前一块大小)
        - 在 tcache 和 fastbin 中，它不被使用。
    2. `size` (当前块大小 & 标志位)
        - 存储当前 chunk 的总大小（包括元数据），由于大小是 8 字节对齐的，所以最后 3 个比特位可以借用来作为标志位
        - 标志位 (Flags)：
            1. PREV_INUSE (第 0 位)：最关键的一位。如果为 1，表示前一个相邻的 chunk 正在被使用；如果为 0，则表示前一个 chunk 是空闲的。分配器使用此位判断能否合并前一个空闲块
            2. IS_MMAPPED (第 1 位)：标记该 chunk 是否由 mmap 系统调用分配。如果是，则不参与常规的 bins 管理。
            3. NON_MAIN_ARENA (第 2 位)：标记该 chunk 是否属于非主分配区（thread arena）。
- 用户数据区
    - tcache
        1. `*next` 指针到下一块的用户数据区开头
            - 从 glibc 2.32 开始，next 不是明文指针，而是和 heap 地址 XOR 混淆过的值；
            - safe-linking:
                - *next = 下一块的指针^（当前块的 next 槽位地址>>12） 混淆后的值
                    - 右移12位导致
                        - 有效的高3个半字节不变（异或0）
                        - 1页内的堆块使用相同混淆操作数（地址低3个半字节的差异消失）
                        - 由不变的高位即可从同一页相同混淆操作数混淆后的值中恢复真实地址（低位使用高位的值进行异或，回推得到更多解码的高位值，最终得到所有半字节）
                            - 每轮恢复3个半字节：
                                - 混淆值^(混淆值>>12)^(混淆值>>24)^(混淆值>>36)
                                    - 或 mid = 混淆值^(混淆值>>12)
                                        res = mid^(mid>>24)
                                - 或精确的恢复半字节后使用到之后的混淆值
                    - 若下一块指针为 null，则立即得到了混淆操作数
                - 最低半字节必为 `0x0`
        2. `key` 安全字段，用于检测 `double free` 错误或攻击
            - `free` 时取 `key` 与进程固定的 `tcache_key` 进行比较，若匹配则阻止释放。攻击时只需修改任意篡改此值即可通过双重释放的检测。
                - 双重释放后， `*next` 指针指向当前地址
                - malloc 将消除 key 值
    - 普通空闲块
        1. `*fd` 前向指针
        2. `*bk` 后向指针
        3. `*fd_nextsize` 前向指针所指块大小的指针
        4. `*bk_nextsize` 后向指针所指块大小的指针

技巧：
- `house of spirit`：
    1. 构造块结构的元数据头部
        - 伪造的 chunk 头为用户数据大小+头部16字节后的16字节对齐，设置前一块是否存在的 flag 为 1 可避免对前一块进行一致性检查；并且避免合并空闲块
        - 猜测：当前一块是否使用的flag置位时，头部按照 +0x8计算后进行16字节对齐
    2. 释放结构中对应用户数据区位置的指针，此时构造块进入空闲列表
    3. 下一次分配将使用此栈中构造块


- tcache poisoning：
    1. 可控块进入空闲列表或已存在于空闲列表的块
    2. 修改已存在于空闲列表中的空闲块的元数据，通常为 *next 指针
        - 新版 glibc 中 next 不是明文指针，而是和 heap 地址 XOR 混淆过的值；
    3. 下一次分配将使用设置的地址

- 重叠分配：
    - 分配块1/2，并分配目标块在块2之后
    - 在块1溢出修改块2 元数据 size，并释放
        - 若在释放后修改，由于此时块已按照原大小进入小桶，于是分配时对应原大小
        - 而在使用中修改，则可进入修改大小后对应的桶，于是分配时对应修改后的大小
    - 修改大小后的分配得到的块将包含目标块，拥有块2相同的位置，以及扩展的大小

触发分配：
- 直接使用 malloc
- 间接使用 printf/scanf 等调用 malloc 的函数
    - setbuf 控制是否使用缓冲区
    - scanf 写入前调用 `malloc_usable_size`，而此函数预期传入的是 malloc 返回的堆块用户指针，若 scanf 的地址不包含块构造则可能崩溃
        - 对于 bss. 段其中包含大量0，于是大小检测时移动指针到元数据位置读取的当前块大小、之前块是否被使用的flag值很可能为0。
            - “可能” 就此返回而无进一步验证，从而可控
        - 对于已拥有值的字段可能对应一个很大的块大小并且 flag 位为已使用
            - “可能” 继续进行之后的块验证或处理导致异常崩溃

可能的错误：
- 篡改后由于 count 计数不足而导致新分配块而不是得到目标块
- 篡改地址无法通过 tcache 16字节对齐检测
- 篡改地址的 safe-linking 异或校验
- tcache 每个 bin 的上限通常为 7
- 分配释放顺序：空闲链表的新块加入头部，分配时从头部移除
- tcache 向不可写段写入

tips：
- free 将块放入空闲链表时，对数据区的 *next 与 key 槽的更新，可用来覆盖
    - 从表头取出实际地址进行 safe-linking 的加密，保存到本块的 *next 槽中
- malloc 将空闲块移出空闲链表时，对数据区的 key 槽进行清0，可用来覆盖
    - 对 *next 槽位进行 safe-linking 的解密，保存到空闲链表头

### 寻找无边秘密 (简单)

思路：

    ```sh
    objdump -R /challenge/seeking-spanless-secrets-easy

    ```

    ```sh
    53 @ 00402238  sprintf(&buf, "%%%us", malloc_usable_size(ptr_array[index]));   
    55 @ 00402295  __isoc99_scanf(&buf, ptr_array[index]); 
    ```
    - 虽然仍有 `malloc_usable_size`，但 bss. 段空间大多为 0，于是读取当前块大小为 0 后无报错继续
        - scanf 宽度为 0 被当作未设置宽度来处理，于是可写入任意长度字符串

### 栈欺骗 (简单)

利用TCACHE漏洞使malloc()返回一个栈指针。

- 新增 `stack_free/stack_scanf/stack_malloc_win` 3个功能
    - `stack_malloc_win` 在内存分配到栈的匹配位置时（ret-0x58）调用 win，如题目分配时返回栈指针

        ```c
        32 @ 00402101  printf("[*] if (malloc(%d) == %p) win()\n", 0x6a, &stack_cmp_addr)   
        33 @ 0040211c  printf("[*] malloc_usable_size(malloc(%d)) = %d\n", 0x6a, 0x78)   
        34 @ 00402126  void* malloc_ptr = malloc(0x6a)   
        35 @ 0040214d  printf("[*] malloc(%d) = %p\n", 0x6a, malloc_ptr)   
        36 @ 00402164  if (malloc_ptr == &stack_cmp_addr)
        ```
    - `stack_free` 将释放栈中 ret-0x58 处的指针
    - `stack_scanf` 将输入写到栈中的 ret-0x98

- `house of spirit`：
    1. 构造块结构的元数据头部
        - 伪造的 chunk 头需通过 glibc 的各种一致性/范围/对齐检查
    2. 释放结构中对应用户数据区的指针
    3. 下一次分配将使用栈中的构造块

思路：
- 从 stack_buf 到目标栈地址块的用户段长度为 0x98-0x58，-0x8 到块元数据的 size
    - 用户段 0x6a 添加元数据并对齐后为 0x80
    - 设置块大小为 0x80+0x1，flag 位设置为前一块正在使用的 1避免可能的一致性检查错误

### 栈召唤 (简单)

利用在栈指针上调用free()来读取秘密数据。

```c
void stack_buf  {Frame offset -198} // stack_scanf
void stack_block  {Frame offset -158}  // stack_free
char goal[0x15]  {Frame offset -ad} 
char input_buf[0x88]  {Frame offset -98} 

// send_flag
52 @ 00402341  if (memcmp(&input_buf, &goal, 0x10) != 0)
54 @ 0040234a  puts("Authorized!")   
55 @ 00402354  win()
```
