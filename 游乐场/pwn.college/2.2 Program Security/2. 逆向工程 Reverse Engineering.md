## 补丁

在处理二进制文件时，有时你可能想改变二进制文件本身的行为。这可以通过对二进制文件打补丁来实现。打补丁是修改二进制文件中的汇编指令以改变其行为的过程。以下一系列挑战将允许你在运行时对二进制文件中的有限字节进行补丁，以改变其行为。

### Patched Up（简单）

逆向此挑战以找到正确的许可证密钥，但你的输入在与正确密钥比较之前会以某种方式被修改。此挑战允许你在二进制文件中修补 5 个字节。

分析：
- 取 `__elf_header` 地址 `0x400000`，并设置内存权限可写
- 使用十六进制scanf，读取2字节偏移，读取1字节进行覆盖。循环5次
- 读取输入密钥
- 使用不可逆的 md5 处理密钥
- 与目标密钥 `EXPECTED_RESULT` 进行比较 `0x404010`，长度 0x1a

思路：
- 输入固定字符串得到md5值后，使用此值覆盖二进制文件的密钥。然而5次修改无法覆盖目标密钥
- 修改指令的控制流，篡改其判断条件

    ```c
    00401bc5  e886f6ffff         call    memcmp 
    00401bca  85c0               test    eax, eax 
    00401bcc  7514               jne     0x401be2
    ```
    - `7514` 是短跳转
        - `0x75` = `JNE rel8`
        - `0x14` = 位移
    - `74 JE`，`75 JNE`

```c
int32_t interpret_sys(struct test* base, char sys_flag, char reg_code)

//////

// 返回 reg_select 对应的寄存器保存的值（寄存器映射）

// 在 buf 的 buf_offset 处写入 value
```

### The Yanalyzer（简单）

此挑战使用基于虚拟机的混淆：逆向自定义模拟器和架构以理解如何获取 flag！这是一个完整的端到端混淆挑战，就像你在现实世界的混淆代码中可能看到的那样！


```c
// main 中将指令包复制到buf中
18 @ 00402388  memcpy(&base, &vm_code, vm_code_length);

struct test
{
    char ins_op_pk_buf [0x300];
    char buf [0x100];
    char reg_a;
    char reg_b;
    char reg_c;
    char reg_d;
    char reg_e;
    char reg_f;
    char reg_g;
};

// 循环构建并调用 interpret_instruction 执行指令
// 取 reg_f 的值，+1 后放回
// 使用 reg_f 的值对 buf 进行3字节组的索引偏移
// 对应的3字节组的每一字节进行“位移”后“或” -> 三字节小端序 
interpreter_loop(struct test* base)
interpret_instruction(base, three_bytes[2] << 0x10 | three_bytes[1] << 8 | three_bytes[0]);

// 3字节组中包含指令代码与操作数
char interpret_instruction(struct test* base, struct ins_op_code ins_op_pk)
interpret_imm(base, rsi_2);
// 构建结构
struct ins_op_code
{
    char op1;
    char ins;
    char op2;
};

// 若 op2 存在，寄存器 reg_e++， read_register 读取 op2 对应寄存器的值后， write_memory 写入 reg_e 偏移的内存
// 若 op1 存在， read_memory 读取 reg_e 对应内存值后， write_register 写入 op1 寄存器，寄存器 reg_e--
struct test* interpret_stk(struct test* base, struct ins_op_code ins_op_pk)

// if (ins_op_pk.op1 && !(base->reg_g & ins_op_pk.op1)) 则不跳转
// 若 op1 为 0 或 条件寄存器与op1的&不为0则跳转：read_register 取出 op2 寄存器的值更新指令寄存器 reg_f
interpret_jmp(struct test* base, struct ins_op_code ins_op_pk)
```
- 获取指令包长度以及指令包，并解码指令
    - `dump binary memory out.bin (char*)&vm_code (char*)&vm_code + 0x273`
    - 三字节的小端序无需额外排序
    - 注意，指令流中也通过 `imm() stk()` 设置指令寄存器 REG_F 进行跳转
- 打印:
    一个不包含实际值的静态总览，一个包含栈与内存值的动态解析
    - 指令行号
    - 函数功能
    - 参数的寄存器
    - sys 打印对应的调用信息
    - jmp 打印对应的跳转条件
    - stack 打印push/pop操作的栈计数与值

    示例：
    ```c
    0x7 sys_write (reg_a, reg_b, min(reg_c, 0x100-reg_b))
    write(1, 0xb6, 0x5)
    bytearray(b'KEY: ')

    0x7b imm (reg_a, 0x30)
    0x89 add (reg_b, reg_c)
    0x9c stack push reg_c pop reg_d
        push value = 0xe, sp = 0x2
        pop value = 0xe, sp = 0x2

    0x65 imm (rip_reg, 0x7b)
    >>      jmp 0x7b

    0x83 cmp (reg_d, reg_c)
    0x84 imm (reg_d, 0x66)
    0x85 >>      jmp (==) -> reg = reg_d, value = 0x66
    ```
- 构建指令映射
- 寄存器功能：
    - reg_g 条件寄存器
    - reg_e rsp
    - reg_f rip

```c

add(op1&结果寄存器, op2寄存器)        操作数相加后放回op1
stk(pop寄存器, push寄存器)              模拟栈使用 REG_E 作为 rsp，若同时设置则为push后pop更新寄存器

    跳转条件匹配 REG_G
```

    - 后缀接 sys_exit
    - 若返回码为 None 则定位

        - exit1 系统调用使用寄存器编码写入返回值的时候由于 0xff 不对应寄存器的异常退出

- `exit()` 仅在两种状态触发

        ins_stk
        ins_jmp
        ins_sys

        - 此时可确认指令的字节位置，以及jmp的指令编码，2个操作符顺序未知

2. 若条件寄存器未被设置，初始为0，那么在 jmp指令对寄存器顺序进行探测

    - 在跳转条件为 0 时，进入跳转分支而使用指令编码读取寄存器触发 exit1
    - 得到位置： 0 为跳转条件op1，0xff 为寄存器op2
