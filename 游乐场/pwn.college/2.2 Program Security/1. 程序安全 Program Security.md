### 循环狂乱（简单）

溢出缓冲区并粉碎栈以获取 flag，但这次是在一个带有栈金丝雀的 PIE 二进制文件中。请注意，这需要精心设计载荷！

虽然启用了金丝雀（Canary）保护，但该程序一次只读取 1 字节的输入，并通过栈上的一个局部变量 n 来跟踪已读取的字节数和偏移量。
相关代码大致如下：

```c
while (n < size) {
  n += read(0, input + n, 1);
}
```

你可以利用这个局部变量 n 来跳过金丝雀。

当你覆盖 n 时，你将改变程序对已读字节数的理解。当它再次执行 read(0, input + n, 1) 时，它将读取到你控制的一个偏移量位置。这允许你将写入位置重新定位到金丝雀之后，从而写入返回地址！

关于有效负载（payload）的大小，有一个巧妙的逻辑：你不需要考虑最终会跳过多少字节。在上述 while 循环中，有效负载的大小决定了将被读取到的最右侧字节。就本挑战而言，通过篡改 n 来“跳过”的字节与正常读入的字节没有区别：只有 n 和 size 的值决定了何时停止读取，而不是实际读入的字节数。

虽然完整地址未知，但我们知道地址的最后三位半字节（nibbles）永远不会改变，因为内存页面对齐到 0x1000。因此，我们可以覆盖保存的返回地址的最低有效字节（通过调试可知），将其重定向到任何具有相同高 7 字节地址的指令。由于地址对齐，这个最低字节在不同次执行间是恒定的，所以这种方法总是有效。

本挑战还有一个额外要求：win_authed() 函数必须首先通过认证。只有当你向其提供参数 0x1337 时，它才会判定你获胜。

```c
void win_authed(int token) {
  if (token != 0x1337) return;
  puts("You win! Here is your flag: ");
  sendfile(1, open("/flag", 0), 0, 256);
  puts("");
}
```

那么如何通过这个检查？有一种方法可以绕过它：你不需要直接调用函数的开头。你可以通过 objdump 分析程序，在 win_authed 函数中找到检查指令之后的那条指令的地址，然后将返回地址覆盖为那个地址，从而跳过 token 检查。

- 分析：

## 复杂损坏

### 复杂损坏

漏洞利用始于两个核心思想的交汇点：损坏程序内存和获取控制流。<br>
内存损坏提供了立足点，通过覆盖关键数据（如返回地址或其他有价值的指针）来控制程序的执行。<br>
Shellcode 提供了有效载荷，这是一系列精心设计的指令，一旦执行被重定向，便会执行特定操作。<br>
当两者结合时，这些概念阐明了软件漏洞利用的基础：不仅仅是使程序崩溃，而是刻意塑造其行为以执行攻击者控制的代码。
