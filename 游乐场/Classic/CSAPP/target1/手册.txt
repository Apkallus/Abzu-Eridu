运行可执行程序 ./ctarget 和 ./rtarget 时出现的错误
加入 -q 标志，防止程序与不存在的分级服务器联系
gdb向程序传递参数:
(gdb) run -q
积累更多使用调试工具（如 GDB 和 OBJDUMP）的经验。

README.txt：描述该目录内容的文件。
ctarget：一个易受代码注入攻击的可执行程序。
rtarget：一个易受面向返回编程攻击的可执行程序。
cookie.txt：一个 8 位十六进制代码，在攻击中作为你的唯一标识符。
farm.c：目标程序 “gadget farm” 的源代码，用于生成面向返回编程攻击。
hex2raw：一个生成攻击字符串的工具。

你的解决方案不能通过攻击绕过程序中的验证代码。具体来说，在攻击字符串中用于 ret 指令的任何地址，只能是以下目标之一：
	touch1、touch2 或 touch3 函数的地址。
	你注入代码的地址。
	“gadget farm” 中某个 gadget 的地址。
你只能使用 rtarget 文件中地址在 start_farm 和 end_farm 函数地址之间的代码构建 gadget。

CTARGET 和 RTARGET 都会从标准输入读取字符串，它们使用如下定义的 getbuf 函数来实现：
1 unsigned getbuf()
2 { 
3     char buf[BUFFER_SIZE];
4     Gets(buf); 
5     return 1;
6 }

Gets 函数与标准库函数 gets 类似，它从标准输入读取一个字符串（以‘\n’或文件结束符结尾），并将其（连同一个空终止符）存储到指定的目标位置。
在这段代码中，目标位置是一个声明为具有 BUFFER_SIZE 字节的数组 buf。在生成你的目标程序时，BUFFER_SIZE 是特定于你程序版本的编译时常量。

Gets () 和 gets () 函数无法判断目标缓冲区是否足够大，以存储读取的字符串。它们只是简单地复制字节序列，这可能会超出目标分配的存储边界。

如果用户输入并由 getbuf 读取的字符串足够短，getbuf 显然会返回 1，如下列执行示例所示：
unix> ./ctarget
Cookie: 0x1a7dd803
Type string: Keep it short! 
No exploit. Getbuf returned 0x1
Normal return

通常，如果你输入一个长字符串，就会出错：
unix> ./ctarget 
Cookie: 0x1a7dd803
Type string: This is not a very interesting string, but it has the property ...
Ouch!: You caused a segmentation fault! Better luck next time

（注意，显示的 cookie 值会与你的不同。）程序 RTARGET 也会有相同的行为。
正如错误消息所示，缓冲区溢出通常会导致程序状态损坏，从而引发内存访问错误。
你的任务是更巧妙地输入字符串，让 CTARGET 和 RTARGET 执行更有趣的操作，这些字符串称为利用字符串（exploit string）。

CTARGET 和 RTARGET 都接受几个不同的命令行参数：
	-h：打印可能的命令行参数列表。
	-q：不将结果发送到评分服务器。
	-i FILE：从文件而非标准输入提供输入。
你的利用字符串通常会包含与可打印字符的 ASCII 值不对应的字节值。
程序 HEX2RAW 可用于生成这些原始字符串。有关如何使用 HEX2RAW 的更多信息，请参阅附录 A。

重要事项：
你的利用字符串在任何中间位置都不能包含字节值 0x0a，因为这是换行符（‘\n’）的 ASCII 代码。
当 Gets 遇到这个字节时，它会认为你想要终止字符串。
HEX2RAW 期望以一个或多个空格分隔的两位十六进制值。因此，如果你想创建一个十六进制值为 0 的字节，需要将其写为 00。
要创建 0xdeadbeef 这个字，你应该将 “ef be ad de” 传递给 HEX2RAW（注意，由于小端字节序，需要反转顺序）。
