Host请求头字段用于指定所请求资源的互联网主机和端口号，
该信息源自用户或引用资源提供的原始URI（通常为HTTP URL，详见第3.2.2节）。
Host字段的值必须代表原始URL所指向的源服务器或网关的命名权威。
这种机制使得源服务器或网关能够区分内部存在歧义的URL，
例如单一IP地址上托管多个主机名的服务器根路径"/"URL。

User-Agent请求头字段包含发起请求的用户代理信息。
该字段主要用于统计用途、追踪协议违规行为，以及通过自动识别用户代理来定制响应，从而规避特定用户代理的限制。


Accept请求头字段可用于指定响应可接受的特定媒体类型。通过Accept头部，
可以明确表示请求仅限于一小部分期望的类型，例如请求内联图像时的情况。

Accept-Encoding请求头字段与Accept类似，
但限制了响应中可接受的内容编码（参见第3.5节）。  

Accept-Language请求头字段与Accept类似，
但用于限制请求方期望响应返回的自然语言集合。语言标签的定义详见第3.10章节。


Connection通用头字段允许发送方指定针对该特定连接所需的选项，且代理不得在后续连接中转发这些选项。

Upgrade-Insecure-Requests 是一个 HTTP 请求头/响应头，
用于强制将不安全的 HTTP 请求自动升级为 HTTPS 请求，提升网站安全性。

Referer[sic]请求头部字段允许客户端向服务器指明当前请求URI的来源资源地址
（即“referrer”，尽管该头部字段名称存在拼写错误）。
通过Referer请求头部，服务器可生成资源的反向链接列表，用于兴趣追踪、日志记录、优化缓存等用途。
此外，该字段还能帮助追踪失效链接或错误键入的链接以便维护。
若请求URI来自不具备自身URI的来源（例如用户键盘输入），则不得发送Referer字段。



Fetch Metadata 请求头是 W3C 提出的 Web API 扩展，用于为 HTTP 请求添加元数据描述，
帮助服务器更精准识别请求来源、目的和上下文。这些头部由浏览器自动添加，
禁止人为篡改（Forbidden Header），可有效防御 CSRF、XSSI 等攻击

1. Sec-Fetch-Dest：请求资源的用途
指示客户端获取资源后的使用场景，服务器据此优化资源类型（如返回图片或脚本的不同格式）

2. Sec-Fetch-Mode：请求的模式
描述请求的跨域策略和资源类型，与 Fetch API 的 mode 属性对应

3. Sec-Fetch-Site：请求来源与目标的关系
标识请求发起方与目标资源的站点关系

Priority 请求头是 HTTP/2 和 HTTP/3 协议中用于优化资源加载优先级的请求头字段，
通过设置资源请求的“紧迫性”，指导服务器或中间设备调整传输顺序和带宽分配
