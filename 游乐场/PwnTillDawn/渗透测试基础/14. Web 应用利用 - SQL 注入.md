# Web 应用利用 - SQL 注入

## 什么是 SQL 注入？

SQL 注入（SQLi）是你在 Web 应用渗透测试中可能遇到的最著名、最危险的漏洞之一。本质上，攻击者可以利用 Web 服务器构建的包含用户输入的 SQL 查询未得到妥善保护这一事实。例如，如果 SQL 查询使用了客户端输入字段且未进行任何净化（如果查询没有参数化），就可以插入一些会被后端数据库解释的 SQL 代码，从而返回与预期不同的结果。

我们可以区分三种主要类型的 SQL 注入：

- **In-Band SQLi**：对攻击者来说最容易利用，因为它利用应用程序在错误消息中输出信息这一事实。
- **盲 SQL 注入（Blind SQLi）**：在这种情况下，应用程序返回非常基本的错误消息或根本不返回错误消息。
- **带外 SQL 注入（Out-of-Band SQLi）**：这种类型的 SQL 注入不太常见。攻击者需要另一个通道来获取对其查询的响应（例如来自服务器的 DNS 请求）。

## 如何利用 SQL 注入？

在本节中，我们将看到如何手动以及使用命令行工具利用 SQL 注入。首先要做的是检查 Web 应用程序是否容易受到 SQL 注入。我们可以在应用程序的每个动态字段中输入单引号（`'`），观察应用程序如何处理。也可以在某些字段中输入计算表达式，例如输入 `1+1`，看输出是否为 `2`。

一个经典的可能存在 SQL 注入的地方是登录表单。例如，我们在用户名字段中输入：

```sql
admin' or 1=1 #
```

在密码字段中输入任意字符串。如果表单存在漏洞，单引号后面我们输入的文本将被解释。由于 `1=1` 总是为真，应用程序将授予用户 `admin` 的登录权限。井号（`#`）将注释掉查询的其余部分。

由于语法可能因数据库引擎而异，需要区分数据库类型。你可以使用以下链接查看常见的数据库类型及其 SQL 注入语法：

- MSSQL：<https://github.com/swisskyrepo/PayloadsAllTheThings/blob/master/SQL%20Injection/MSSQL%20Injection.md>
- MySQL：<https://github.com/swisskyrepo/PayloadsAllTheThings/blob/master/SQL%20Injection/MySQL%20Injection.md>
- OracleSQL：<https://github.com/swisskyrepo/PayloadsAllTheThings/blob/master/SQL%20Injection/OracleSQL%20Injection.md>
- PostgreSQL：<https://github.com/swisskyrepo/PayloadsAllTheThings/blob/master/SQL%20Injection/PostgreSQL%20Injection.md>

如前所述，可以使用工具自动化 SQL 注入利用。然而，即使工具非常有用，也不能在所有情况下工作，这就是为什么有时需要手动执行 SQL 注入。

最著名的工具是 sqlmap，它包含在 Kali Linux 中。最基本的使用方法是：

```bash
sqlmap -r req
```

其中 `req` 是一个包含来自 Burp 工具请求的文本文件（你可以直接复制粘贴）。

## 示例

测试 SQL 注入的一个非常典型且可靠的方法是尝试中断由用户输入生成的 SQL 查询。这可以通过在用户输入（最常见的是登录表单）中附加或前置一个单引号或双引号来实现。

考虑 `http://10.150.150.11/` 上的登录表单。当我们在用户名字段中输入任意内容，在密码字段中只输入一个单引号时，会得到一个通用错误。注意当我们尝试双引号（`"`）时会发生什么：

![SQL 注入错误](/Content/img/lessons/sqli_errorYYYY10DD2110SS.jpg)

我们可以看到 SQL 查询出现问题。可以通过使用 Burp 代理进一步确认：

![Burp 中的 SQL 注入错误](/Content/img/lessons/sqli_error_burpYYYY10DD2110SS.jpg)

我们已经成功中断了原始的 SQL 查询——这是存在 SQL 注入漏洞的一个非常强烈的指示。进一步测试通过提交一个永远为真的语句（使用布尔逻辑）实现了 SQL 注入并成功绕过登录：

![SQL 注入绕过登录](/Content/img/lessons/sqliYYYY10DD2110SS.jpg)
