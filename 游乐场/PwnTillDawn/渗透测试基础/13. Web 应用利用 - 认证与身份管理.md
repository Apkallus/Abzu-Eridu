# Web 应用利用 - 认证与身份管理

认证是验证给定用户或客户端身份的过程。此过程的主要目标是确保已认证的用户确实是他们所声称的身份，但一旦用户登录到 Web 应用程序，他们就不再通过初始凭证被识别。

由于 HTTP 是一种无状态协议，应用程序必须维护一种方法，以识别随时间推移属于同一会话的请求。

大多数框架都内置了会话管理，将会话标识符存储在 cookie 中，这些是维护会话状态的推荐方式。然而，我们经常看到开发人员别出心裁地创建新的会话管理机制。在这种情况下，攻击者就有机会攻入应用程序。

## 主要的弱点和漏洞有哪些？

当面对受认证机制保护的应用程序时，应检查以下几个弱点：

### 1. 默认凭证 / 密码

你经常会遇到使用流行的开源或商业软件的 Web 应用程序，这些软件可以在服务器上以最少的配置或定制安装。

通常，这些应用程序安装后，没有进行正确配置，初始认证和配置提供的默认凭证从未更改。这些默认凭证为渗透测试人员所熟知，不幸的是，恶意攻击者也知道，他们可以利用它们获得对各种类型应用程序的访问。

最常见的默认凭证是用户名和密码都为“admin”。

以下网站可用于检查默认凭证：<https://cirt.net/passwords>

### 2. 用户名枚举

用户名枚举是一种常见的弱点，通常在认证或重置密码时可以发现。通过观察应用程序在提供有效/无效用户名时返回的错误消息或响应的差异，我们可以推断出该用户名是否存在。

下图展示了一个用户名枚举的示例。左侧，使用用户“admin”进行认证失败，而使用用户“ThisUserDoesNotExist”进行认证时返回了不同的错误。

![用户名枚举示例](/Content/img/lessons/2020-11-16_14-39-2320201117083449.png)

不要只看错误消息，有时在 HTTP 响应中可以看到细微的差异。

仅凭用户名枚举无法让你入侵应用程序，但它可以增加你进行成功密码猜测攻击的机会。

### 3. 可猜测的凭证

事实是：大多数应用程序强制使用弱密码策略，允许最终用户设置非常简单且易于猜测的密码，例如：

- qwerty
- password
- 111111
- 12345678
- abc123
- 1234567
- password1
- 12345

如果应用程序未使用帐户锁定机制、双因素认证，并且如果你能够识别出有效的用户名，那么执行密码暴力破解攻击可能成为你的入侵途径。

在 wizlynx，我们在执行此类攻击时通常会查看前 50 个最常用的密码。

![常用密码示例](/Content/img/lessons/120201117084851.png)

### 4. 使用 SQL 注入绕过认证

正如你将在后续课程中学到的，SQL 注入攻击包括通过客户端向应用程序输入数据来注入 SQL 查询。成功的 SQL 注入利用可以让攻击者从数据库读取敏感数据、修改数据库数据（插入/更新/删除）、执行管理操作，在极少数情况下，还可以绕过认证。

要测试此类漏洞，你可以首先输入可能揭示 SQL 注入漏洞存在的特殊字符（例如单引号字符）。

![SQL 注入测试](/Content/img/lessons/2020-11-16_15-17-2920201117084140.png)

然后，你还可以使用真值语句来绕过认证。

![使用真值绕过认证](/Content/img/lessons/2020-11-16_15-19-4320201117084042.png)

### 5. 关键功能缺少认证

开发人员通常不会在整个应用程序中正确设置认证和授权。因此，你很可能会发现应用程序的某些部分或功能可以被任何未认证的用户访问。

发现此类弱点主要有三种方法：

- 首先，当目标应用程序可通过互联网访问时，你应该在搜索引擎中搜索带有目标 URL 的已索引资源。
- 另一个重要方法是使用 DirBuster 等工具进行适当的侦察，并匿名爬取应用程序链接。
- 在执行灰盒 Web 应用程序渗透测试时，重复认证请求，但删除请求中的会话标识符。

下面是一个针对 `/admin` 的已认证 GET 请求示例——如您所见，该请求包含会话标识符。

![包含会话标识符的认证请求](/Content/img/lessons/2020-11-16_15-32-0520201117084742.png)

重放请求并删除 Cookie 会得到相同的结果，因为在此示例中，`/admin` URI 配置错误，不需要认证证明：
