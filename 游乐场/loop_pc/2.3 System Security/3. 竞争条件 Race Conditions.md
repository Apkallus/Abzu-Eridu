# 竞争条件

想象一下，你正处于一场与时间赛跑的肾上腺素飙升的竞赛中，两个或多个程序正在激烈地竞争访问或更改共享数据。
这就是竞争条件利用的惊险世界！就像大胆的黑客从正在关闭的安全门下滑过一样，这些利用在恰当的毫秒间潜入以篡改数据，导致系统以意想不到且往往是灾难性的方式运行。
在这场数字竞赛中，瞬间便可决定安全与入侵的区别，上演一场在眨眼间展开的高风险戏剧。
通过掌握竞争条件的机制，你不仅是在学习编程，更是在踏入一个时机决定一切的领域，而奖赏则是加固或利用系统漏洞。
你的代码成为系统资源赛道上的高速赛车，理解竞争条件是你通往胜利者圈子的门票！

### level5.0

这个竞争条件涉及多个步骤，这使得利用它的可靠性降低！

此挑战将验证文件所在的目录属于root用户，并且其他用户无法在该目录中创建文件。

- 不跟随链接，检查文件本身：路径不含 flag，文件不为链接
- 跟随链接，检查文件路径字符串对应目录的权限：root 为所有者以及组，其他用户不可写

思路：
- 由于此处检查路径字符串对应目录，普通文件与最终的文件链接必然在一层目录中
- root 目录设置同名目录链接或 root 普通文件链接（仅使用字符串而并不检查是否实际为目录）
- 同名目录 + 普通文件
- 同名目录 + flag 文件链接
- 似乎按对应顺序更容易

- send_redacted_flag 将 flag 字符串添加到前缀字符串的空字节后，而 receive_message 使用空字节判断字符串长度，于是无法直接读取

思路：
- 尝试消除前缀字符串末尾的空字节：
    在程序1读取所有字符串后，程序2覆盖字符串，若程序1读取信息时程序2覆盖了空字节且未加入其空字节，那么程序1的字符串长度包含 flag 字符串，此时可得到全部字符串
    - 程序1调用 send_redacted_flag 读取全部内容到全局指针的buf
    - 程序2调用 send_message 覆盖前缀字符串末尾的空字节后停止
        - 由于循环判断字符串是否走完，于是覆盖的字符串应当长于到空字节的数量
    - 程序1调用 receive_message 获取空字节被覆盖而 strlen 得到包含 flag 字符串长度的目标字符串
- 未使用的另一个思路：检查字符串长度与实际使用字符串内容的时间窗口
    - 程序a send_message 发送足够长的载荷字符串
    - 程序a receive_message 的 strlen 得到此长度后内核控制流转移
    - 程序b send_redacted_flag 读取 flag 字符串到全局变量指针 buf
    - 内核控制流返回 程序a 后输出 flag 字符串

- 使用之前的思路2代码

```c
// receive_message 之前为
21 @ 00401755  uint64_t nbytes = strlen(&global_message)   
22 @ 0040177a  write(fileno(*(fsbase - 8)), &global_message, nbytes)

// receive_message 此处为

```
- 之前的窗口： 
    - 程序1 send_message 发送载荷字符串
    - 程序1 receive_message 的 strlen **显式**得到载荷字符串长度，内核控制流转移
    - 程序2 send_redacted_flag 此时全局变量指针处为 flag 字符串
    - 程序1 恢复内核控制流后读取 flag 字符串
- 此处的窗口：
    - 程序1 send_message 发送载荷字符串
    - 程序1 receive_message  的 fprintf **隐式**得到载荷字符串长度，内核控制流转移
        - 猜测1：似乎 fprintf 在隐式得到长度后调用底层类似 write 的系统函数输出，否则检查字符串结尾的空字节则不应当输出空字节（但实际由此输出了在空字节后的 flag 字符串）
        - 猜测2：或者 fprintf 共用一个缓存，导致缓存被覆盖
    - 程序2 send_redacted_flag 此时全局变量指针处为 flag 字符串
    - 程序1 恢复内核控制流后读取 flag 字符串
