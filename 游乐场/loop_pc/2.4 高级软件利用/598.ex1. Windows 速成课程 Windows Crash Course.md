# Windows 速成课程

学习在新操作系统中工作，就如同重新蹒跚学步一般。请以全新的面貌重温早期的概念，以此开启你的旅程。

注意：本道场的开发进度较为*缓慢*，包括基础设施支持的迭代。相关功能可能会随时调整！

## 资源

### Dojo 平台上的 Windows 环境 - 入门指南

开始步骤：
  - 启动挑战，进入常规的 Linux 环境。
  - 在 Linux 环境中使用 `windows` 命令来 `start`（启动）、`stop`（停止）或 `connect`（连接）Windows 虚拟机。虚拟机启动可能需要片刻时间。
  - Windows 桌面图形界面可通过网页访问： <https://pwn.college/workspace/desktop-windows>。

获取 flag：
  - 挑战程序位于 `C:\challenge\` 目录下。
  - Flag 文件位于 `C:\flag`，但 hacker 用户无法直接访问它，即便通过挑战程序也不行！
  - Flag 可以通过运行在 Windows 机器上的一个 "flag 服务" 来获取。
  - 该 "flag 服务" 将通过位于 `\\.\pipe\LOCAL\FlagPipe` 的命名管道传输 flag。
  - "flag 服务" 仅会向位于 `C:\challenge\` 的挑战程序发送 flag。

其他细节：
  - 从练习环境启动虚拟机时，你将获得 Windows 机器的管理员权限。你可能需要先“注销”再“登录”，Windows 环境才会识别此权限。
  - 你的 Linux 主目录可通过 Windows 的 Z 盘访问。建议利用此方式在两个环境之间传输文件，而不是直接依赖 Z 盘进行操作。
  - 挑战程序在 Windows 中位于 `C:\challenge`，在 Linux 中位于 `/challenge`。
  - **请勿运行位于 `Y:` 盘的挑战文件**。该程序无法获取 flag，其存在仅为实现细节。

请在练习模式下使用 Windows 虚拟机，利用已安装的工具开发你的漏洞利用程序。

要获取 flag，请**在 Windows 环境中**运行你的漏洞利用程序，目标为位于 `C:\challenge\` 的挑战程序。

```asm
'win() is located at: 0x140007c60\r\n'

00000001`40014b7a 8b542420           mov     edx, dword ptr [toRead (rsp+20h)]
00000001`40014b7e 488d4c2430         lea     rcx, [buffer{[0]} (rsp+30h)]
bu 00000001`40014b83 e8c8dffeff         call    babywinsimpleoverflownoaslr!@ILT+6976(fgets) (140002b50)
...
00000001`40014b91 4881c428030000     add     rsp, 328h
00000001`40014b98 c3                 ret 
```

tips:
- windbg attach 运行时进程或打开 exe 文件
    - attach 或需切换线程
- `uf challenge` 打印函数反汇编
- 30h 对应 0x30
- fgets 为字符串型读取
- 仍使用 pwntool

### level3

启用 ASLR 的缓冲区溢出

```asm
lea     rcx, [rsp+0x30 {buffer}] 
call    j_fgets

add     rsp, 0x1e8
```
- 仍然提供 win 的地址

### level4

若 `win` 函数位于 DLL 中该如何处理？

```asm
lea     rcx, [rsp+0x30 {stack_buf}] 
call    j_fgets

add     rsp, 0x248
```

tips:
- 不同于每次程序启动时随机的 linux，Windows 仅在系统启动时随机一次
    - 使用 windbg 查看运行时 win DLL 地址
- lm 列出模块，其中包括 win.dll
- 获取 win() 函数地址
    - 使用 win.dll 运行时基址 + win()函数的 DLL 文件相对偏移
    - 或查看模块函数中 win!win 的运行时地址

### Cookie Monster 1

Cookie 怪兽饿了，想吃 flag。你能喂饱它吗？

```c
challenge() 
// void rsp_addr  {Frame offset -238} 
// enum PAGE_PROTECTION_FLAGS lpflOldProtect  {Frame offset -218} 
// int32_t n  {Frame offset -214} 
// char stack_buf[0x1f0]  {Frame offset -208} 
// int64_t canary  {Frame offset -18}

    void rsp_addr 
    int64_t canary = __security_cookie ^ &rsp_addr 
    int32_t n = 0x600 
    enum PAGE_PROTECTION_FLAGS lpflOldProtect = 0 
    char stack_buf[0x1f0] 
    VirtualProtect(&stack_buf, 0x1e8, PAGE_EXECUTE_READWRITE, &lpflOldProtect) 
    j_cookie_monster()
    j_printf("Send your payload!\n") 
    j_fgets(&stack_buf, n, get_fp(0)) 
    // lpflOldProtect 作为 buf 索引写入空字节
    uint64_t result = lpflOldProtect 
    stack_buf[result] = 0 
    j___security_check_cookie(canary ^ &rsp_addr) 
    return

uint64_t cookie_monster()
// int64_t rsp_addr  {Frame offset -188} 
// int64_t* canary_ptr  {Frame offset -160} 
// int64_t canary  {Frame offset -18} 
// uint64_t result  {Register rax}

    int64_t rsp_addr 
    int64_t canary = __security_cookie ^ &rsp_addr 
    int64_t* canary_ptr = &canary 
    j_printf("rsp is located at: %lp\n", &rsp_addr) 
    j_printf("The stack cookie is: %lp\n", *canary_ptr) 
```

tips:
- cookie 值随机一次后复用
    - 异或编码得到金丝雀，且异或还原 cookie
- call 压栈，rsp 计算有 0x8 偏移
- ret 到 shellcode 时 rsp 在高地址，shellcode 本身可能被后续系统函数覆盖
