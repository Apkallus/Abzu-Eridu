仅限内核的资源：
- `hlt` 停止 CPU
- `in`与`out` 硬件端口
- `cr3` 页表寄存器（控制虚拟内存）
- `MSR_LSTAR` 作为 `syscall` 跳转目标地址寄存器，使用 `rdmsr/wrmsr` 访问

权限模型：
- 环0 内核态
- 环3 用户态
- `syscall` 切换权限状态，并在内核与程序代码之间进行控制流跳转

内核模块交互：
- 内核注册设备/模块
    - dev `register_chrdev()`
    - proc `proc_create()`
- 内核模块设置接口的处理函数映射结构
    ```c
    static struct file_operations fops = {
        .read = device_read,
        .write = device_write,
        .open = device_open,
        .release = device_release
    };
    ```
- 用户态 `open()` 得到模块 fd 后，读/写/`ioctl` 此 fd 以使用模块接口进行交互

内核模块命令：
- `insmod` 安装模块到内核
- `lsmod` 列出当前安装的模块
- `rmmod` 移除模块

权限凭证：
- 内核使用凭证结构保存进程的凭证信息（uid/suid/euid）
- 命令
    - `prepare_kernel_cred()` 创建凭证结构
    - `commit_creds()` 应用凭证结构
 

seccomp 过滤器结构：
- 位于 `current_task_task` 结构中，使用字段 `.thread_info.flags` 的 `TIF_SECCOMP` 位控制
    - 寄存器 `gs` 的某个偏移保存的指针指向 `current_task_task` 结构
    - 清除 `TIF_SECCOMP` 则当前进程逃出 seccomp 的过滤，但子进程仍继承 seccomp 过滤

页表：
- 页表项使得连续的虚拟内存可分散到不连续的物理内存中
- 多级页表增加在物理内存的开销的同时，增加理论内存上限
- `cr3` 寄存器保存页表基址
- 虚拟地址（48位）使用页表转为物理地址

    ```log
    | 9位 (PGD) | 9位 (PUD) | 9位 (PMD) | 9位 (PTE) | 12位 (页内偏移) |
    47        39|38       30|29       21|20       12|11            0
    ```
    
- 从虚拟地址取各级页表索引

    ```c
    uint64_t virt_addr = 0x404060;

    uint64_t pgd_idx = (virt_addr >> 39) & 0x1ff;
    uint64_t pud_idx = (virt_addr >> 30) & 0x1ff;
    uint64_t pmd_idx = (virt_addr >> 21) & 0x1ff;
    uint64_t pte_idx = (virt_addr >> 12) & 0x1ff;
    uint64_t offset  = virt_addr & 0xfff;
    ```

- 使用 pgd 基址与页表索引得到物理地址
    - 各级页表的低12位为状态，高位为基址
    - 内核物理地址直接映射到虚拟地址，示例：
        - 虚拟 0xffff88807c732000
        - 物理 0x7c732000
        - 起始 0xffff888000000000

    ```c
    // 示意的 _v 后缀为虚拟地址， _p 后缀为物理地址
    pgd_base_v = ...
    kernel_origin_v = pgd_base_v  >> 32 << 32;

    // pgd 虚拟基址到 pud 虚拟基址
    pgd_entry_v = pgd_base_v + pgd_idx*8;
    pgd_entry_p = attack_getdata(pgd_entry_v);
    pud_base_p = pgd_entry_p & ~0xfff;
    pud_base_v = pud_base_p | kernel_origin_v;

    // 页表索引复用示意 top_base_v 到 down_base_p
    // 之后  | kernel_origin_v 得到 down_base_v
    uint64_t col(uint64_t top_base_v, uint64_t top_idx){
        uint64_t top_entry_v = top_base_v + top_idx*8;
        uint64_t top_entry_p = attack_getdata((void*)top_entry_v);
        uint64_t down_base_p = top_entry_p & ~0xfff;
        return down_base_p; 
    }

    // 逐层得到基址
    uint64_t pud_base_v = col(pgd_base_v, pgd_idx) | kernel_origin_v; 
    uint64_t pmd_base_v = col(pud_base_v, pud_idx) | kernel_origin_v;
    uint64_t pte_base_v = col(pmd_base_v, pmd_idx) | kernel_origin_v;

    // 得到最终物理页
    uint64_t page_p = col(pte_base_v, pte_idx);
    // 定位目标地址
    uint64_t final_p = page_p + offset;
    uint64_t final_v = final_p | kernel_origin_v;

    attack_getdata((void*)final_v)
    ```
    - 此处无大页处理 

缓解：
- `SMEP` 内核态不能执行用户态页
- `SMAP` 内核态默认不能随意读写用户态页

攻击面：
- 旧版 linux 内核提权 `commit_creds(prepare_kernel_cred(0))`
- seccomp 逃逸：`current->thread_info.flags &= ~(1 << TIF_SECCOMP)`
- root 凭证执行命令 `run_cmd("/path/to/my/command")`
    - 使用绝对路径


`current_task_struct` 简写为 `current`
- 寄存器 `gs` 指向 `__per_cpu_start`，而其中某个偏移保存的指针指向 `current` 结构

    ```sh
    nm -n /opt/linux/vmlinux | grep -E '__per_cpu_start|current_task'
    0000000000000000 D __per_cpu_start
    0000000000015d00 D current_task
    ```
- 进程内存访问
    - 父进程 `current->real_parent`
    - 子进程链表头部 `current->children`
        - 子进程链表指针位置 `current->sibling`
        - 双向链表指针指向 sibling 处，而不是结构起始位置
            - 获得链表指针指向的子进程结构 sibling 地址后需 -sibling 偏移到结构基址
    - `access_process_vm(子进程任务结构指针，子进程虚拟内存地址，内核缓冲区，标志)`
        - 标志控制读写，读时无需置位
- 残留数据的空闲内存页面分配
    - 分配内存页面，得到内存地址 `__get_free_pages(gfp_t gfp_mask, unsigned int order)`
        - `gfp_mask` 使用常用标志 `GFP_KERNEL` 0xcc0
    - `strnstr` c 库函数查找子串
    - 实用：
        - `__get_free_pages(0xcc0, 0)` 分配 1<<0 = 1页
        - `strnstr(addr, "pwn.college{", 4096)` c 库函数查找子串
        - 循环，每次分配基础的1页，并在这1页中查找子串

tips：
- `/proc/kallsyms` 内核函数地址
    - 在未使用地址随机时，shellcode 使用得到的固定地址以 call 调用内核函数
        - 若 shellcode 作为函数被调用，结尾使用 ret 干净退出，若使用栈需还原
    - `F` 为全局符号， `f` 为本地符号
    - 找到 `noop_ret` 内核函数地址，并在 shellcode 中 call，gdb 设置此 noop 符号为断点
        - 但实际上 `noop_ret` 被其他地方多次触发，应当断在目标设备函数处而不是全局符号

- pwn.college 模块的 vm 在 root 权限下，GDB 连接后即可调试内核函数
    - 模拟器在 1234 端口打开服务器端，`vm debug` 将 gdb 连接到此端口进行调试
    - vm 终端 `cat /proc/kallsyms | tail` 得到目标设备函数地址
    - 外部终端 
        - `vm debug` 
        - `(gdb) b * 设备函数地址` 
        - `(gdb) c`
    - vm 终端执行命令或程序以触发设备交互的对应函数
    - 外部终端 gdb 命中断点，从而进行内核函数调试
    - 命中断点后第一次或需 si 之后才可 ni
- `dmesg -W`
	等待并打印新信息
- `pahole -C task_struct /opt/linux/vmlinux` 查看数据结构信息
    - 偏移为10进制

    ```c
    struct task_struct *       parent;               /*  1192     8 */
    struct list_head           children;             /*  1200    16 */
    struct list_head           sibling;              /*  1216    16 */
    ```
- 调试时查看结构信息 `p *(struct task_struct*)$rdi`
- shellcode 内核空间：
    - 内核页可使用 asm 标签
    - 内核栈可分配 rsp 并在 ret 前还原
