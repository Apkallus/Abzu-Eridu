seccomp 过滤器结构：
- 位于 `current_task_task` 结构中，使用字段 `.thread_info.flags` 的 `TIF_SECCOMP` 位控制
    - 寄存器 `gs` 的某个偏移保存的指针指向 `current_task_task` 结构
    - 清除 `TIF_SECCOMP` 则当前进程逃出 seccomp 的过滤，但子进程仍继承 seccomp 过滤

页表：
- 页表项使得连续的虚拟内存可分散到不连续的物理内存中
- 多级页表增加在物理内存的开销的同时，增加理论内存上限
- `cr3` 寄存器保存页表基址
- 虚拟地址（48位）使用页表转为物理地址

    ```log
    | 9位 (PGD) | 9位 (PUD) | 9位 (PMD) | 9位 (PTE) | 12位 (页内偏移) |
    47        39|38       30|29       21|20       12|11            0
    ```
    
- 从虚拟地址取各级页表索引

    ```c
    uint64_t virt_addr = 0x404060;

    uint64_t pgd_idx = (virt_addr >> 39) & 0x1ff;
    uint64_t pud_idx = (virt_addr >> 30) & 0x1ff;
    uint64_t pmd_idx = (virt_addr >> 21) & 0x1ff;
    uint64_t pte_idx = (virt_addr >> 12) & 0x1ff;
    uint64_t offset  = virt_addr & 0xfff;
    ```

- 使用 pgd 基址与页表索引得到物理地址
    - 各级页表的低12位为状态，高位为基址
    - 内核物理地址直接映射到虚拟地址，示例：
        - 虚拟 0xffff88807c732000
        - 物理 0x7c732000
        - 起始 0xffff888000000000

    ```c
    // 示意的 _v 后缀为虚拟地址， _p 后缀为物理地址
    pgd_base_v = ...
    kernel_origin_v = pgd_base_v  >> 32 << 32;

    // pgd 虚拟基址到 pud 虚拟基址
    pgd_entry_v = pgd_base_v + pgd_idx*8;
    pgd_entry_p = attack_getdata(pgd_entry_v);
    pud_base_p = pgd_entry_p & ~0xfff;
    pud_base_v = pud_base_p | kernel_origin_v;

    // 页表索引复用示意 top_base_v 到 down_base_p
    // 之后  | kernel_origin_v 得到 down_base_v
    uint64_t col(uint64_t top_base_v, uint64_t top_idx){
        uint64_t top_entry_v = top_base_v + top_idx*8;
        uint64_t top_entry_p = attack_getdata((void*)top_entry_v);
        uint64_t down_base_p = top_entry_p & ~0xfff;
        return down_base_p; 
    }

    // 逐层得到基址
    uint64_t pud_base_v = col(pgd_base_v, pgd_idx) | kernel_origin_v; 
    uint64_t pmd_base_v = col(pud_base_v, pud_idx) | kernel_origin_v;
    uint64_t pte_base_v = col(pmd_base_v, pmd_idx) | kernel_origin_v;

    // 得到最终物理页
    uint64_t page_p = col(pte_base_v, pte_idx);
    // 定位目标地址
    uint64_t final_p = page_p + offset;
    uint64_t final_v = final_p | kernel_origin_v;

    attack_getdata((void*)final_v)
    ```
    - 此处无大页处理

内核 slab 对象堆：
- 概念
    - 对象大小相同
    - 总大小至少为连续的1页
    - 可能拥有多个 slab 板
    - 每个 cpu 拥有专用 slab 板
    - 对象释放时进入当前 slab 板的空闲链表，新释放对象进入链表头
    - 当分配用完当前 slab 板的空闲链表时，分配到半满的其他 slab 板，若仍无空闲块则分配一个新板
- 特性
    - kmalloc 使用通用分配到 kmalloc-x
    - 使用安全 flag 将分配到 kmalloc-cg-x 特殊缓存中
- 安全
    - next 指针位于对象逻辑大小中段: 当 size 为 0x100 时，位于 0x80
        - 8的倍数向下对齐
    - 空闲列表随机化一次
        - 应对：
            - 定位：分配控制对象0，测试对象1，对象0进行越界覆盖，当对象1检测到被覆盖时即为相邻
            - 喷射：分配 n 个对象后，使用 n-1 个设置对象批量越界覆盖，使用第 n 个触发对象尝试触发
                - 所有设置对象进行相邻溢出数据覆盖
                - 此时板中唯1的触发对象很可能被篡改
                - 使用此对象触发
            - pin：UAF 读取 next 地址后，篡改地址围绕 next 对象进行泄露与覆盖
                - 控制对象0使用释放命令后，得到悬空指针的 UAF（注，此处并不 close 而无双重释放）
                - 控制对象0读取得到 slab 板 next 对象基址
                - 分配控制对象0/篡改对象1，释放命令控制对象0回到空闲链表
                - 控制对象0设置本对象的 next 指针，污染空闲链表到对篡改对象1基址+8处重叠的泄露伪对象2
                    - 分配控制对象0/泄露伪对象2，应用空闲链表劫持
                    - 篡改对象1读取泄露伪对象2的函数指针，释放命令控制对象0回到空闲链表
                    - 得到内核模块函数地址
                - 使用 dmesg 得到 oops 的内核泄露地址，计算相对偏移后得到运行时内核基址
                - 控制对象0设置本对象的 next 指针，污染空闲链表到对篡改对象1基址-8处重叠的覆盖伪对象3
                    - 分配控制对象0/覆盖伪对象3，应用空闲链表劫持
                    - 覆盖伪对象3覆盖篡改对象1的函数指针与参数结构

    - 空闲链表硬化：
        - 概念：启用硬化时，该指针会通过将指针存放的地址与每个缓存的随机数进行异或运算来混淆
            - `freelist_ptr_encode`
                ```c
                encoded = (unsigned long)ptr ^ s->random ^ swab(ptr_addr);
                ```
            - `freelist_ptr_decode`
                ```c
                decoded = (void *)(ptr.v ^ s->random ^ swab(ptr_addr));
                ```
            - `swab()` 逆序
        - 应对：
            同一位置的 UAF 异或解码
            - 获取编码地址
                - 控制对象0使用释放命令后，得到悬空指针的 UAF（注，此处并不 close 而无双重释放）
                - 控制对象0第一次 UAF 读取得到 slab 板编码后的 next 值 `obj_0_mg_ptr`
                - 分配0~7，释放控制对象0 (控制对象0位于空闲链表尾部指向 NULL)
                - 控制对象0第二次 UAF 读取得到 slab 板对 NULL 编码后的 next 值 `obj_0_mg_null`
            - 计算解码
                - 异或两次编码值，消除两项相同操作数，得到探测对象1的 ptr 值 `obj_1_ptr`
            - 获取计算 random
                - 分配0，顺序释放探测对象1/控制对象0（对象1到链表尾，对象0到链表头）
                - 探测对象1 UAF 读取得到 slab 板对 NULL 编码后的 next 值 `obj_1_mg_null`
                - 使用对象1的 NULL 编码值以及之前得到的对象1指针及槽位偏移，计算随机值 
                    `random = my_swab(obj_1_ptr + next_offset) ^ obj_1_mg_null`
                - 使用 ptr 与 random 得到控制对象0的 `swab(&ptr)` obj_0_swab 之后用来编码地址（或者以对象1为核心，直接使用已知值）
            ```c
            ptr = (swab(&ptr) ^ ptr ^ random) ^ (swab(&ptr) ^ NULL ^ rand)
            ```

            越界读写
            - 修改相邻位置对象的 next 值

- pipe_buffer
    - 概念
        ```c
        /**
        * struct pipe_buffer - Linux 内核管道缓冲区
        * @page: 包含管道缓冲区数据的物理页
        * @offset: 数据在 @page 中的偏移量
        * @len: 数据在 @page 中的长度
        * @ops: 与此缓冲区关联的操作集。参见 @pipe_buf_operations。
        * @flags: 管道缓冲区标志。见上文。
        * @private: 由 ops 拥有的私有数据。
        **/
        struct pipe_buffer {
            struct page *page;
            unsigned int offset, len;
            const struct pipe_buf_operations *ops; // anon_pipe_buf_ops
            unsigned int flags;
            unsigned long private;
        };
        ```
        - `ops` 字段为 `pipe_buf_operations` 内核结构的常量指针
            - 泄露内核基址
            - 控制流劫持
        - 需写入以更新结构
        
        ```c
        struct pipe_buf_operations {
            int (*confirm)(struct pipe_inode_info *, struct pipe_buffer *);
            void (*release)(struct pipe_inode_info *, struct pipe_buffer *);
            bool (*try_steal)(struct pipe_inode_info *, struct pipe_buffer *);
            bool (*get)(struct pipe_inode_info *, struct pipe_buffer *);
        };
        ```

    - `pipe()` 创建管道
        - `pipefd[0]` 管道读取端，`pipefd[1]` 管道写入端

    - `fcntl()` 对打开的文件描述符执行操作

    - 使用错误测试得到一批地址，从中寻找与无害测试得到的栈地址相同段的栈地址
- 用户态到内核态的切换函数相对稳固 `_copy_from_user` 无 oops
- 用户态内的错误无 oops
- 内核态内的错误将触发 oops `page_fault_oops`
    - slab 空闲链表劫持
- 内核地址中触发错误：
    - 在高位跨页写入无效页面
    - 释放无效指针
    - 清零之后用到的指针
    - 多次运行时可能第一次与之后的错误信息不同，随机化也可能导致每次重启后的错误遵循多个不同但固定的模式（即，一次尝试可能不足够，应当多次尝试）
- 强化空闲链表似乎添加了任务错误时的清理，而清理时若存在无效指针释放则致命崩溃
    - release 仅在文件引用计数为0时触发，当 fork 子进程增加引用计数后不会触发
    - 错误探测后的空闲链表头已被污染，之后的代码需 free 有效对象以重获控制

tips：
- `/proc/kallsyms` 内核函数地址
    - 在未使用地址随机时，shellcode 使用得到的固定地址以 call 调用内核函数
        - 若 shellcode 作为函数被调用，结尾使用 ret 干净退出，若使用栈需还原
    - `F` 为全局符号， `f` 为本地符号
    - 找到 `noop_ret` 内核函数地址，并在 shellcode 中 call，gdb 设置此 noop 符号为断点
        - 但实际上 `noop_ret` 被其他地方多次触发，应当断在目标设备函数处而不是全局符号
    - `_text` 或 `_stext` 即为内核基址

- pwn.college 模块的 vm 在 root 权限下，GDB 连接后即可调试内核函数
    - 模拟器在 1234 端口打开服务器端，`vm debug` 将 gdb 连接到此端口进行调试
    - vm 终端 `sudo cat /proc/kallsyms | tail -n20` 得到目标设备函数地址
    - 外部终端 
        - `vm debug` 
        - `(gdb) b * 设备函数地址` 
        - `(gdb) c`
    - vm 终端执行命令或程序以触发设备交互的对应函数
    - 外部终端 gdb 命中断点，从而进行内核函数调试
    - 命中断点后第一次或需 si 之后才可 ni
- `dmesg -W`
	等待并打印内核日志新信息
- `pahole -C task_struct /opt/linux/vmlinux` 查看数据结构信息
    - 偏移为10进制

    ```c
    struct task_struct *       parent;               /*  1192     8 */
    struct list_head           children;             /*  1200    16 */
    struct list_head           sibling;              /*  1216    16 */
    ```
- 调试时查看结构信息 `p *(struct task_struct*)$rdi`
- shellcode 内核空间：
    - 内核页可使用 asm 标签
    - 内核栈可分配 rsp 并在 ret 前还原
- 双重释放将创造自我指向的循环
- 设备模块基址与内核基址为不同段
