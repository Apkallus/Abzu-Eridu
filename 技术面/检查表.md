竞争条件：
- 对变量/地址的检查与执行时间/内容不同，导致两个操作实际对应不同数据。如：
    - 访问内存检查后，再次读取内存使用。两次内存读取的时间窗口内，内存值发生变化
    - 多进程/线程共享变量


shellcode/rop:
- 泄露或爆破金丝雀/返回地址，以及栈中遗留的数据段/libc/栈地址
    - 泄露 puts/write
    - 爆破 hit/miss 预言机
    - plt 函数调用入口


堆：
- 地址对齐限制的地址泄露或覆盖
    - 构造块读取堆地址
        - 同页块的 safe-linking 解码 -> 得到地址>>12的混淆值 -> 编码任意地址到 next 指针 -> 劫持空闲链表到到目标地址
        - 分配到空闲链表尾部块时，其 next 值为混淆值 & NULL，直接得到混淆值


地址链：
- 数据段地址
    - `plt 表`指向数据段本身的 `got 表` 并调用函数
    - `got 表`在函数已使用时保存指向 `libc` 的指针

- libc
    - `__environ` 环境变量符号保存`主栈地址`
    - gdb 测量固定偏移 
        - `libc 基址`与 gdb 得到的`线程栈地址`之间，偏移固定

- 线程堆
    - 未对齐 +0x8
        - 堆起始位置之前 `fs_base` 线程结构指针
            -  是否一定拥有本线程的 `fs_base` 指针❓
    - 对齐 +0x0
        - 堆起始位置 `libc` 函数地址
            - 是否一定为 `libc` 函数 `_IO_wfile_jumps` 位置 `0x000f20`❓
    - 待测试❗
        - `线程堆`空间拥有指向位于 `libc` 的主空间的指针

- 主栈
    - 未对齐 +0x8
        - 函数返回地址 `PIE` 程序段
        - main 返回到 `__libc_start_main+243`
    - 对齐 +0x0
        - `rsp - 0x20` 为 `PIE` 的 main 函数基址


- ld
    - 一篇关于[在内存中 pivoting](https://nickgregory.me/post/2019/04/06/pivoting-around-memory/) 所用方法的简短文章。


- 当前段拥有直接指向其它段的指针，或指向当前段中的结构的字段中拥有指向其它段的指针

侦察
- `checksec` 查看 elf 文件安全信息
- `/proc/cpuinfo` 查看 cpu 支持 
    - `shstk|ibt`
- `/proc/进程id/fd` 查看 fd
- `strace` 查看调用
