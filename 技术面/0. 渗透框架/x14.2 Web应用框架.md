# 第14章 评估Web应用框架

应用框架解释并执行用Java、PHP、Python和Ruby等语言编写的代码。框架可以捆绑在更大的Web服务器软件包中（例如Microsoft IIS中的ASP.NET），也可以使用不同的应用和Web服务器组件运行（例如运行在Apache Tomcat之上的JBoss Application Server）。

在许多情况下，开发人员负责应用程序安全，而IT运维人员负责基础设施安全。如果双方忽视加固这些层之间的框架组件，可能存在可利用的缺口。

## 框架与数据存储分析

应用框架指纹识别：

*   审查HTTP头（例如`Server`和`X-Powered-By`）
*   分析连接时设置的cookie（主要是会话变量）
*   分析文件名、扩展名和目录结构
*   审查内容中的元数据（例如HTML和JavaScript）

可用于进一步调查和分析应用框架组件的自动化工具包括
- BlindElephant
- WAFP
- clusterd

前两个工具对用PHP编写的内容管理系统（CMS）平台（例如WordPress、Joomla和Drupal）进行指纹识别，而clusterd可以对应用服务器（包括Apache Tomcat、JBoss和Adobe ColdFusion）进行指纹识别。

根据CMS的不同，应随后使用特定工具，来识别可利用的漏洞。如：
- CMS Explorer
- WPScan
- OWASP Joomla Vulnerability Scanner

## 理解常见缺陷

在未经加固的应用框架组件中存在可远程利用的漏洞。严重的可远程利用缺陷源于软件缺陷和缺乏安全控制，可能导致以下后果：

*   未授权访问管理接口
*   信息泄露和敏感数据暴露
*   任意文件上传和代码执行

通过减少攻击面和维护软件（即及时打补丁），可以缓解并解决大多数问题。

## PHP

PHP由解释器核心和可选子系统（包括FTP、Mail、OpenSSL、Phar和ZIP）组成。除了漏洞库列出的内容外，利用已知漏洞需要攻击者提供随后被解析的恶意内容（例如PHP脚本或存档文件）。已解决问题的有用列表可以在PHP ChangeLog[^10]中找到。

[^10]: [PHP ChangeLog](https://www.php.net/ChangeLog-8.php)。

### PHP管理控制台

流行的服务器管理控制台包括Parallels Plesk和phpMyAdmin（用于通过HTTP管理MySQL）。Plesk通常通过TCP端口8443使用专用服务器实例暴露，而phpMyAdmin通常可通过`/phpmyadmin`路径访问。phpMyAdmin常见的弱用户名/密码组合有`root/（空）`、`root/password`和`root/root`。

这些控制台中的已知漏洞大多数导致任意代码执行，但需要经过认证的访问。这些软件包中还存在大量XSS和CSRF缺陷。

### PHP CMS软件包

Drupal、Joomla、SilverStripe和WordPress是用PHP编写的流行CMS软件包。这些软件包易受攻击版本中的各个组件容易受到攻击，导致SQL注入、XSS、文件上传和代码执行。

Metasploit包含很多针对这四个CMS平台的利用模块，NVD列出了大量高严重性漏洞。许多缺陷存在于可选模块和插件中。在识别出特定的CMS（使用BlindElephant或类似工具）后，手动调查其配置以发现可利用的条件。注意：其中许多需要特权访问CMS才能利用。

在测试期间，使用Nmap的`http-drupal-enum`脚本枚举服务器上已安装的Drupal模块和主题，包括Views和OpenID。

## Apache Tomcat

Tomcat由Coyote（代理入站连接的HTTP连接器）、Catalina（Java servlet容器）和Jasper（JSP引擎，为Catalina编译代码以供处理）组成。

包括JBoss和Apache Struts在内的应用服务器也使用Coyote作为Web服务器。在测试期间，会发现此类服务器返回的HTTP头为`Server: Apache-Coyote/1.1`。为避免将应用服务器误标为Apache Tomcat（即Catalina），应通过审查返回的内容（包括HTTP头和cookie）并扫描特定文件的存在来识别底层容器。

### Manager应用程序

manager应用程序通过`/manager/html`暴露管理界面。需要凭证，常见的默认用户名/密码组合包括`admin/（空）`、`cxuser/cxuser`、`j2deployer/j2deployer`、`ovwebusr/OvW*busr1`、`tomcat/tomcat`和`vcx/vcx`。

认证后，可以使用Metasploit的`tomcat_mgr_deploy`模块部署JSP命令shell。

### 已知的Tomcat缺陷

过时的Tomcat版本中存在可利用的漏洞。XSS和CSRF缺陷适用于包括`/manager/html`和`/jsp/cal/cal2.jsp`在内的组件

许多商业产品捆绑了有缺陷的Apache Tomcat组件。

### 攻击Apache JServ协议

Apache JServ协议（AJP，也称为JK协议）通常通过TCP端口8009暴露。该服务由Apache Coyote运行，充当前端表示服务器（例如Apache HTTP Server）和后端servlet容器（即Catalina）之间的二进制连接器。如果该服务可用，可以通过配置本地Apache HTTP Server使用`mod_jk`转发流量来与其交互，方式类似于与暴露的Tomcat实例交互。[^16]

[^16]: DiabloHorn, [“8009, the Forgotten Tomcat Port”](http://bit.ly/2aDhQmy), DiabloHorn.com blog, 2011年10月19日。

Nmap包含五个与AJP端点交互的NSE脚本。

| 名称 | 用途 |
|------|------|
| ajp-auth | 检索认证方案和域 |
| ajp-brute | 通过AJP进行暴力破解密码 |
| ajp-headers | 请求内容时检索服务器头 |
| ajp-methods | 通过AJP枚举支持的HTTP方法（通过OPTIONS） |
| ajp-request | 通过AJP请求URI并返回输出 |

## JBoss测试

JBoss应用服务器（也称为WildFly）使用Apache Coyote HTTP连接器作为其Web服务器。HTTP和RMI（远程方法调用）服务通过JMX应用服务器核心（MBeanServer）提供对单个MBean[^17]的访问。

[^17]: MBean是遵循JMX规范的托管Java对象。

包括Oracle Application Server、WebLogic、IBM WebSphere和JBoss在内的服务器都支持RMI。

### 通过HTTP进行服务器分析

`/web-console/ServerInfo.jsp`和`/status?full=true`网页通常会暴露服务器详细信息。然后可以与可用的应用程序交互，并将JBoss版本与NVD交叉引用以识别弱点。

通过GET请求传递的参数会暴露在状态页面和Web访问日志文件中。在某些情况下，可能获得敏感数据（例如会话令牌、用户名、文件名和路径），这些数据可用于访问特权系统组件。

### Web控制台和Invoker Servlet

可以通过JBoss中的以下路径暴露管理servlet（取决于版本）：`/admin-console`、`/jmx-console`、`/management`和`/web-console`。默认凭据是`admin/admin`。获得访问权限后，可以使用可用的invoker servlet与暴露的MBean交互：

*   `/web-console/Invoker`（JBoss 6和7版本）
*   `/invoker/JMXInvokerServlet`和`/invoker/EJBInvokerServlet`（JBoss 5及更早版本）

使用Google查找暴露的JBoss invoker servlet

### 识别MBean

每个MBean暴露一个管理接口，由可读/可写属性和一组可调用操作组成。查阅文档说明的Java类及其各自的接口。该类包含应用程序代码，MBean定义属性和可调用操作（如果有）。

#### 通过HTTP枚举MBean

JMX管理控制台 `/jmx-console` 列出已注册的MBean。

#### 通过RMI注册表服务枚举MBean

RMI注册表服务通过JBoss中的TCP端口1098和1099（类似于RPC端口映射器），使用Java命名和目录接口（JNDI）协议详细列出已注册的MBean。

Metasploit的`java_rmi_registry`模块支持通过该服务进行枚举。Juan Vazquez在他的在线演讲“Abusing Java Remote Interfaces”[^19]中详细介绍了该技术。

[^19]: Juan Vazquez, [“Abusing Java Remote Interfaces”](http://bit.ly/2axE4X4), SlideShare.net, 2015年4月13日。

### 利用MBean

可以通过HTTP invoker servlet和暴露的RMI接口访问MBean。有用的策略包括通过`jboss.system:type=ServerInfo`枚举操作系统详细信息，以及通过向量执行命令shell。

| 名称 | 属性/操作 | 描述 |
|------|------------|------|
| `jboss.admin:service=DeploymentFileRepository` (JBoss 4和5) | `store` | 上传任意JSP文件^[a] |
| `jboss.system:service=MainDeployer` (JBoss 4) | `deploy` | 从远程位置部署WAR文件^[b] |
| `jboss.deployer:service=BSHDeployer` | `createScriptDeployment` | 执行任意Java代码^[c] |
| `jboss.deployment:flavor=URL,type=DeploymentScanner` | `URLs` | 从远程位置部署WAR文件 |

^[a]: Metasploit [`jboss_deploymentfilerepository`](http://bit.ly/2axDJnq) 模块。
^[b]: Metasploit [`jboss_maindeployer`](http://bit.ly/2axEpsP) 模块。
^[c]: Metasploit [`jboss_bshdeployer`](http://bit.ly/2axEdKj) 模块。

其他有用的MBean存在于Oracle Java实现中（被包括WebLogic Server在内的产品使用）[^20]，包括`javax.management.loading.MLet`，它可能被滥用来加载文件。

[^20]: 参见GitHub上Mogwai的 [Java Management Extensions (JMX) Exploitation Toolkit](http://bit.ly/2aDi5ya)。

#### 通过HTTP

可以通过JMX控制台手动与MBean交互。也可以使用Metasploit `jboss_maindeployer` 模块生成 shell。

#### 通过RMI

可以使用RMI端点与有用的MBean交互，包括：

*   JBoss JMX RMI连接器（默认在TCP端口1090上监听）[^21]
*   在TCP端口（包括4444、4445、4446和4447）上运行的RMI服务

[^21]: Renaud Dubourguais, [“Hacking (and Securing) JBoss AS”](http://bit.ly/2aunDAi), Hervé Schauer Consultants白皮书, 2012年。

可以使用`twiddle.sh`实用程序通过RMI与可用的MBean交互。

安装`twiddle.sh`且无错误执行后，可以使用它通过RMI读取和写入MBean属性，以及调用操作。默认情况下，该工具使用RMI注册表服务查找正确的端点。

### 利用RMI分布式垃圾收集器

RMI服务（包括注册表服务和各个RMI端点）暴露了一个垃圾收集器机制，攻击者可以利用它加载和执行任意代码，如Metasploit `java_rmi_server` 模块。该利用使用远程类加载，因此对JMX RMI连接器（例如JBoss中的端口1090）无效。

JBoss应用服务器中导致代码执行、文件泄露和任意文件创建的可利用软件缺陷

## Apache Struts

Struts是一个Java servlet容器，默认使用Apache Coyote HTTP连接器作为其Web服务器。通常可以通过目录名和HTTP头（例如`struts-time`）中的“struts”字符串以及`action`和`do`文件扩展名来识别使用该框架构建的应用程序。容器本身拥有各种漏洞缺陷

### 利用DefaultActionMapper

一个特别严重的漏洞涉及Struts中的DefaultActionMapper。[^26]该机制旨在执行操作，如以下JSP片段所示：

```jsp
<s:form action="foo">
  ...
  <s:submit value="Register"/>
  <s:submit name="redirect:http://www.google.com/" value="Cancel"/>
</s:form>
```

如果用户点击取消按钮，将请求以下URI：

*   `/foo.action?redirect:http://www.google.com/`

Struts 2.3.15及更早版本中的DefaultActionMapper未正确清理输入，这意味着在请求操作文件并提供有效的redirect或action参数时，可以注入OGNL表达式（导致服务器端代码执行）（例如`/foo.action?action:%{3*4}`，如果服务器评估该表达式，将返回“12”）。

可以使用以下两种结构与DefaultActionMapper交互：

*   `/struts2-blank/example/X.action`
*   `/struts2-showcase/employee/save.action`

在环境中识别到DefaultActionMapper暴露后，可以手动执行任意命令或使用Metasploit `struts_default_action_mapper` 生成命令shell

## JDWP

Java调试线协议（JDWP）通过TCP端口（包括5005、8000、8080、8787和9009）提供Java应用程序的远程调试。

工具：
- Metasploit `java_jdwp_debugger` 模块
- Nmap 的NSE脚本

    | 脚本 | 描述 |
    |------|------|
    | `jdwp-exec` | 注入执行shell命令的Java类 |
    | `jdwp-info` | 注入返回操作系统详细信息的Java类 |
    | `jdwp-inject` | 注入任意Java类 |

每个都需要弱的服务器权限来注入Java类

有关JDWP缺陷的进一步讨论，请参阅Christophe Alladoum, [“Hacking the Java Debug Wire Protocol”](http://bit.ly/2azB3p6), IOActive Blog, 2014年4月23日。Alladoum的博文详细介绍了通过[jdwp-shellifier](http://bit.ly/2aDiiBq)执行命令。

## Adobe ColdFusion

ColdFusion应用服务器解释ColdFusion标记语言（CFML）内容。包括Railo在内的其他引擎也解析CFML，因此使用该语言并不一定表示底层服务器是Adobe ColdFusion。

### ColdFusion分析

在ColdFusion中触发错误处理可以提供服务器目录结构、变量名和数据库模式的详细信息。

工具：
- MSF `coldfusion_version`
- clusterd

### 暴露的管理接口

ColdFusion管理接口通常通过以下路径暴露：

*   `/CFIDE/administrator/index.cfm`
*   `/CFIDE/componentutils/login.cfm`
*   `/CFIDE/main/ide.cfm`
*   `/CFIDE/wizards/`

这些接口在认证时通常配置为使用静态用户名`admin`。此配置容易受到暴力破解密码的攻击。

ColdFusion远程开发服务（RDS）使用`/CFIDE/main/ide.cfm`路径提供数据源浏览、远程文件访问和通过HTTP的调试功能。

ColdFusion MX Developer Edition和其他客户端包提供对RDS功能的完全访问。[^31]

### Apache Solr漏洞

Adobe ColdFusion 9及更高版本捆绑了Apache Solr：一个全文搜索引擎，可通过TCP端口8983访问。Solr用Java编写，也可以运行在Apache Tomcat和Jetty之上（可通过包括80、8080、8083、8084、8984和8985在内的TCP端口访问）。Google搜索可以揭示可互联网访问的Solr管理界面。

此类接口位于每个Solr集合之下。ColdFusion中默认集合是`core0`，因此可以查询`http://server:8983/solr/core0/admin/`以显示索引内容（例如使用`*`查询）。
本地服务器环境变量也通过`get-properties.jsp`页面显示。

在测试期间，首先评估`/solr`路径以显示集合名称，然后继续。

## Django

Python应用程序通常运行在Django框架之上（通过诸如Apache或Nginx之类的Web服务器）。管理界面通常可通过`/admin`和`/django-admin`路径访问（默认使用`admin/admin`凭据），可以通过HTTP服务器头（包括`X-Django-Request-Time`、`X-Django-Request-Timestamp`和`X-Django-User`）识别该框架。

Django核心中存在一些重大缺陷。可选框架和软件包中也存在漏洞。在测试期间，识别Django系统组件并与NVD交叉引用以列出已知的暴露。

## Rails

Rails用Ruby编写，是一个用于提供Ruby Web应用程序的Web应用服务器。与其他框架（包括Django和PHP）一样，Rails通常通过Apache HTTP Server或Nginx呈现。可以通过以下方式识别其存在：

*   HTTP服务器头字段（例如`X-Rails-Env`、`X-Rails-Time`和`X-Powered-By`）
*   应用程序中使用RESTful URI路径（例如`/project/new`和`/user/81/`）
*   存在以下静态文件：
    *   `/images/rails.png`
    *   `/javascripts/application.js`
    *   `/javascripts/prototype.js`
    *   `/stylesheets/application.css`

识别到暴露的Rails实例后，查询漏洞库以了解服务器核心中的已知漏洞。可选软件包（称为gems）也存在可利用缺陷

### 使用应用程序的Secret Token

Rails应用程序的`secret_token`值可用于创建包含序列化数据的恶意cookie，该cookie随后在服务器端执行。这些令牌通常通过版本控制暴露，或通过目录遍历从应用程序的`config/initializers/secret_token.rb`文件中读取（例如CVE-2016-0752）。

获得`secret_token`值后，可以使用Metasploit的`rails_secret_deserialization`模块生成命令shell。

Rob Heaton, [“How to Hack a Rails App Using Its secret_token”](http://bit.ly/2aunbSs), RobertHeaton.com, 2013年7月22日。

## 应用框架安全检查清单

在加固Web应用服务器时，应考虑以下对抗措施：

*   确保应用框架组件（包括依赖项和间接暴露的元素）已打上最新补丁，以缓解已知缺陷。
*   仔细处理和考虑仓库和版本控制系统中的秘密。例如，可以使用Rails应用程序的`secret_token`值在服务器端执行代码。
*   不要将管理界面或特权功能（例如phpMyAdmin和Django管理控制台）暴露给不可信网络。
*   如果可能，将公开暴露的Web应用组件与管理功能解耦，以防止通过CSRF、持久性XSS或社会工程进行权限提升。

*   移除向攻击者泄露有用Web和应用服务器详细信息的HTTP头（例如`X-Powered-By`和`X-Runtime`）。
