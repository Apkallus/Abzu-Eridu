# 评估Web服务器

Web服务器需要高度的安全保障，因为它们通常暴露在不可信网络中。

用于指纹识别、调查和确认可用HTTP服务中漏洞的简洁方法论，包括以下步骤：

1.  识别代理机制
2.  枚举虚拟主机和可访问的网站
3.  对每个识别出的站点：
    1.  分析服务器软件和可用子系统
    2.  主动扫描和爬取以识别有用的内容和功能
    3.  攻击暴露的认证机制
    4.  确认服务器软件中的漏洞

Web应用通常通过负载均衡器呈现，因此前两步很重要。例如，客户端通过TLS连接到负载均衡器，然后负载均衡器根据提供的Host值将请求定向到内部的应用服务器（通过HTTP）。

在测试期间，会遇到以下三种情况之一：

*   直接访问托管单个站点的单台服务器
*   直接访问托管多个站点（虚拟主机）的单台服务器
*   通过代理间接访问多个应用服务器

通过对从每个服务器端点接收到的材料进行主动测试和被动分析，映射并测试可用的Web应用组件。

## 识别代理机制

负载均衡器和反向代理在大型环境中很常见；它们用于在多个后端应用服务器之间分发和定向请求。系统通常支持HTTP 1.1方法（特别是GET、POST和HEAD）。

识别服务器将连接转发到其他地方的一种简单方法是提供一个不带Host字段的HEAD请求，以及另一个带有效Host字段的请求

通过修改请求中的Host字段，有可能滥用配置错误的代理并连接到任意主机。通过提供内部IP地址或有效的内部主机名，可能能够通过可访问的HTTP服务器将连接定向到非公共资源

## 枚举有效主机

大多数Web服务器和反向代理解析HTTP 1.1的Host值并相应地定向请求。攻击者获取可莪能有效的主机名有三种基本方法：

*   在客户端环境查找
*   通过Netcraft、Google、DNS等渠道进行开源查询
*   对暴露的Web服务器和应用进行主动测试

主动测试技术包括：

*   网站爬取和HTML解析以识别主机名
*   X.509证书分析，通过TLS获取服务器主机名值
*   分析服务器响应以获取主机名和IP地址详细信息
*   暴力破解有效主机名

工具
- 使用Wikto通过主动爬取识别关联主机名
- 使用Metasploit的`vhost_scanner`模块进行主动暴力破解主机名

在测试期间，使用更大的字典可能会产生结果。

## Web服务器分析

掌握了环境中的有效网站列表（即IP地址、协议和主机组合）后，采用手动和自动化的策略来测试每个可用服务器的配置。

包括Apache HTTP Server和Microsoft IIS在内的实现支持许多模块和子系统（例如认证机制、WebDAV和TLS）。有些是服务器软件自带的（例如Apache中的mod_cgi和IIS中的Microsoft ASP.NET支持），尽管许多需要安装和配置。

通过以下方式推断Web服务器的配置：

*   分析对HTTP请求的响应
*   审查请求内容后返回的服务器HTTP头
*   爬取每个站点并分析目录结构、文件名和内容

### 分析服务器响应

使用HEAD/GET方法为给定的HTTP资源从Web服务器检索状态码和头（本质上执行ping操作）。在返回的 `Server` 头中，可能找到服务器软件和底层配置的详细信息(若未被混淆)。

通过浏览这些子系统的项目页面并搜索NVD，调查已知的安全漏洞。

OPTIONS请求应返回给定资源允许的HTTP请求方法。

某些方法支持服务器端文件上传和内容修改。例如，对某个资源发出OPTIONS请求时，可能会发现支持有用的方法，包括PUT和PROPPATCH（分别用于上传内容和修改文件属性）。

### HTTP头审查

对资源的请求通常会返回包含有用HTTP头的响应

| 头                   | 备注                                                     |
|----------------------|----------------------------------------------------------|
| ETag                 | 可用于指纹识别设备固件                                   |
| Content-Location     | 可能泄露主机名或内部IP地址详细信息                       |
| Location             | 重定向时使用，可能指向内部IP或主机名                     |

使用`tree`工具显示目录结构。

要强制wget为提供的主机名使用特定的IP地址（例如，使用特定的代理地址或无法解析的名称），编辑Kali Linux中的`/etc/hosts`文件，将名称指向给定的地址。

## 主动扫描

在进行手动调查后，应该已编译了有效的HTTP和HTTPS端点、相关的虚拟主机以及感兴趣的应用程序和URL路径的列表。然后，应进行主动扫描以执行以下操作：

*   识别Web应用防火墙（WAF）机制
*   指纹识别Web服务器和应用框架软件
*   暴露潜在有用的内容和功能

### WAF检测

WAF系统用于解析HTTP流量，并阻止与已知签名（例如SQL注入和XSS字符串）匹配的入站查询和出站响应。将WAF构建到Web服务器中（例如Apache mod_security模块），将其作为专用设备运行，或将其作为云服务操作。

指纹识别工具：
- [wafw00f](https://github.com/EnableSecurity/wafw00f)
- Nmap `http-waf-fingerprint` 脚本

WAF的存在反过来要求对攻击流量（例如命令注入）进行混淆以逃避阻止。

一种有效的WAF规避策略是绕过安全机制路由HTTP请求。迭代修改本地的`/etc/hosts`文件并评估每个HTTP/S端点，以识别绕过WAF的路由。

### 服务器与应用框架指纹识别

工具：
- WhatWeb

## 确认Web服务器漏洞

在对操作系统、Web服务器进行指纹识别并识别有用内容后，调查潜在的漏洞

### 审查暴露的内容

主动扫描通常会揭示可用的数据和URL路径（例如`robots.txt`、`phpinfo.php`和`/server-status/`）。暴露的内容可能包含以下内容：

*   用户名、会话令牌和凭证
*   软件包的详细信息（例如版本信息和设置）
*   本地文件和目录结构的详细信息（例如绝对路径）

例如，`.DS_Store`和`/.svn/entries`结构会泄露文件名和目录结构。日志文件也可能包含有用的数据（例如base64编码的凭证）。

在识别到可访问的`/.svn/entries`（Subversion 1.6及更早版本）或`/.svn/wc.db`文件（Subversion 1.7及更高版本）后，使用Metasploit[^10]和`pillage-svn`[^11]下载源代码。

如果识别到暴露的Git仓库（通过`/.git/index`），请使用`gitpillage.sh`（可以在DVCS Pillage Toolkit[^12]中找到）。仓库通常包含有用的秘密，包括令牌和API密钥。


[^10]: Metasploit [svn_wcdb_scanner](http://bit.ly/2aDcgAI) 模块。
[^11]: 参见GitHub上的 [pillage-svn](https://github.com/lanjelot/pillage-svn)。
[^12]: 参见GitHub上的 [DVCS-pillage](https://github.com/evilpacket/DVCS-Pillage)。

### 暴力破解密码

在测试期间发现（Nikto）的需身份验证的URL路径后
- 尝试手动登录查看详细验证信息
- 对目标验证方式（无账户锁定）执行爆破工具（Hydra）

### 调查支持的HTTP方法

常规的HTTP 1.1方法很少容易受到攻击（例如，代理连接到任意主机，或访问敏感内容）。在测试期间可能遇到的有用方法包括：

*   TRACE
*   PUT和DELETE
*   WebDAV方法

#### TRACE

如果支持TRACE方法，并且Web服务器运行的应用程序容易受到XSS攻击，则可以发起跨站追踪（XST）[^14]攻击以获取用户会话信息。此向量特别有用，因为它可以揭示受`HttpOnly`标志保护的cookie。

相比之下，如果服务器仅运行不处理用户输入的静态网站，则TRACE支持的实际安全影响可以忽略不计。

#### PUT和DELETE

使用PUT和DELETE方法上传和删除服务器端内容（结合有效凭证或在识别到可写目录时）。

工具：
- 手动：telnet
- 自动
    - MSF [^15][^16]
    - `davtest`[^17]

[^15]: Metasploit [http_put](http://bit.ly/2aDdcoz) 模块。

#### WebDAV方法

如果支持PROPFIND方法，使用Metasploit[^18][^19]揭示系统信息。

有了有效凭证，使用`cadaver`实用程序[^20]上传、下载、搜索和操作服务器端内容。没有足够的权限，需要依赖可写目录来上传数据。

使用`davtest`识别全局可写目录。

### 服务器漏洞

指纹识别基础设施后查找漏洞库的已知缺陷

## Web服务器加固

在加固Web服务器时，应考虑以下对抗措施：

*   确保服务器软件、库和依赖项都已打上最新补丁。

*   通过移除不必要的模块和禁用冗余子系统来减少攻击面。同时考虑禁用认证子系统以消除暴力破解的风险。

*   在的环境中禁用对不必要HTTP方法的支持（例如PUT、DELETE、TRACE和WebDAV方法）。
*   设置服务器以移除向攻击者泄露有用Web和应用服务器详细信息的HTTP头

*   如果不存在索引文件（例如`default.asp`、`index.htm`和`index.html`），防止目录索引，以防止Web爬虫和攻击者识别敏感信息。
*   如果环境中发生应用程序异常（崩溃），不要向公众暴露调试信息。返回不包含任何敏感材料的通用404或500错误页面。
