# Web应用架构

当今的应用程序通过客户端处理和服务器API提供丰富的用户体验，支持移动应用、桌面浏览器和第三方集成。

系统组件日益解耦以促进可扩展性（例如负载均衡器、应用服务器、消息队列服务和键值存储），这在使用第三方服务时引入了风险。

## Web应用类型

应用类别包括零售、银行、博彩、社交网络和信息站点（例如博客和新闻媒体）。

考虑一个独立的Web服务器，通过内容管理系统（CMS）提供营销内容：浏览器通过明文HTTP与站点交互，应用托管在单台服务器上。

大型Web应用（例如Facebook、eBay和银行站点）很复杂，利用内容分发网络（CDN）并支持原生移动应用。组件跨多个层级运行，使用各种协议和数据格式。


## Web应用层级

大多数应用使用表示层、应用层和数据层中的组件。

这些技术中的许多都存在漏洞，确保小缺陷不能被组合起来利用系统非常重要。从设计角度来看，控制层之间的数据流至关重要。

### 表示层

移动客户端和Web浏览器使用JavaScript和客户端技术支持丰富的功能，这些技术与服务器API和端点交互。处理越来越多地在客户端系统上进行，HTTP用于通过标准化格式（例如HTML、XML和JSON）传输数据。

表示层中使用的两种协议：

*   TLS，通过HTTPS提供传输层安全
*   HTTP，包括支持流式传输和状态跟踪的功能

#### TLS

TLS提供了以下好处：

*   通过非对称密码学和证书使用进行认证
*   通过对称密码学提供机密性
*   通过HMAC或使用认证密码提供完整性

安全性依赖于客户端和服务器的配置（即底层数学是可靠的，但实现可能有缺陷）。

#### HTTP

服务器通过HTTP向客户端（包括Web浏览器、移动应用和第三方）发送数据。该协议越来越多地通过安全连接（HTTPS）呈现，以减轻网络嗅探风险。

客户端首先提供HTTP方法、目标资源和协议版本。随后的行包括HTTP头和客户端提供的数据。方法使用不同的头和数据格式——例如，GET请求的格式与POST不同。收到请求后，服务器返回一个状态码，以及HTTP头和供客户端解析的数据

HTTP扩展和特性构成了Web应用的基础：

*   客户端请求方法
    *   HTTP方法
    *   WebDAV扩展
    *   专有的Microsoft扩展
    *   常见请求方法头
*   服务器状态码
*   其他服务器特性
    *   支持持久连接和缓存
    *   HTTP认证机制
    *   设置Cookie

**客户端请求方法**

| 方法      | 备注                                   |
|-----------|----------------------------------------|
| GET       | 用于检索服务器端内容                   |
| POST      | 用于在消息体中向服务器发送数据         |
| HEAD      | 用于检查服务器端内容而不检索它         |
| OPTIONS   | 枚举特定URL支持的HTTP方法              |
| PUT       | 如果权限允许，允许上传文件             |
| DELETE    | 如果权限允许，执行服务器端文件删除     |
| TRACE     | 回显请求内容以用于调试目的             |
| CONNECT   | 提供指向任意主机和端口的代理能力       |

**服务器状态码**

当收到HTTP请求时，服务器应响应一个状态码和包含待客户端解释数据的消息体。

| 状态码 | 备注                                                               |
|--------|--------------------------------------------------------------------|
| 100 Continue | 服务器已收到请求头，客户端应继续发送请求体，通常响应HTTP PUT或POST请求 |
| 200 OK | 成功HTTP请求的标准响应                                             |
| 201 Created | 请求已成功处理并创建了新资源                                       |
| 301 Moved Permanently | 此请求及所有未来请求应重定向到给定的URI                            |
| 302 Found | 临时重定向到给定的URI                                              |
| 304 Not Modified | 指示自客户端在请求头中指定的版本以来，资源未被修改                 |
| 400 Bad Request | 由于语法错误，无法满足请求                                         |
| 401 Unauthorized | 需要认证或认证失败                                                 |
| 403 Forbidden | 请求有效，但服务器拒绝执行                                         |
| 404 Not Found | 页面或资源不存在时的常见错误                                       |
| 405 Method Not Allowed | 使用的HTTP方法不被该资源允许                                       |
| 500 Internal Server Error | 通用错误消息                                                       |
| 501 Not Implemented | 服务器不识别请求方法                                               |
| 502 Bad Gateway | 服务器作为代理，从上游服务器收到无效响应                           |
| 503 Service Unavailable | 由于高负载或维护，服务器当前不可用                                 |
| 504 Gateway Timeout | 服务器作为代理，未及时收到上游服务器的响应                         |

**支持持久连接和缓存**

流式传输内容的应用使用持久HTTP连接和特定的数据编码。大多数Web服务器和浏览器支持以下特性：

*   Keep-alive
*   分块编码
*   缓存

Keep-alive功能允许客户端在单个会话中发出多个请求。`Content-Length`头定义了每个请求发送的数据量。分块编码支持流式传输和其他动态呈现材料（无论是向客户端还是从客户端）的用例。这是通过`Transfer-Encoding: chunked`头与keep-alive会话结合实现的。

浏览器和代理根据`Cache-Control`头[^6]设置的指令缓存内容。材料使用标志标记，包括`public`、`private`、`no-cache`和`no-store`。`max-age`限定符用于定义应保留数据的旧副本的时间长度。

**HTTP认证机制**

跟踪状态对许多应用至关重要（例如，区分未认证用户和已登录用户，或已支付商品的客户和未支付的客户），但HTTP是无状态协议。因此，应用通过以下方式跟踪状态：

*   设置Cookie
*   将令牌放置在HTML中，在执行操作时呈现
*   处理HTTP referrer头（显示用户最后访问的页面）

Web应用的行为——已认证用户获得一个会话令牌（设置为cookie），在每个HTTP请求中呈现(类似Kerberos认证:用户在成功认证后获得一个票据，该票据有给定的有效期，随后在每个请求中呈现)。

包括Microsoft IIS在内的Web服务器通常支持HTTP认证，无论运行在其上的应用如何。攻击者可以使用`Authorization`请求头通过支持的方法（例如WebDAV或HTTP PUT功能）上传恶意内容。

大多数Web服务器支持的认证机制是Basic和Digest。[^7] Basic机制很弱：用户凭证经过base64编码并以明文发送，容易通过网络嗅探被窃取。Digest机制被提出来克服这个问题，利用MD5和共享密钥来避免发送明文凭证；然而，它容易受到重放攻击。

Microsoft Web服务器支持额外的认证类型：

*   NTLM [^8]
*   Negotiate（简单受保护协商 [SPNEGO]）[^9]

NTLM机制使用base64编码的挑战-响应来认证用户。Negotiate可以在客户端和安全支持提供程序（SSP）之间代理NTLM或Kerberos凭证。

**设置Cookie**

Cookie用于跟踪用户并在客户端存储材料，可以由基础设施硬件（如负载均衡器）、Web应用框架（例如Microsoft ASP.NET）和Web应用设置。Cookie通过`Set-Cookie`服务器头发送给客户端。

Cookie由名称-值对和属性组成。每个属性定义了浏览器应如何处理cookie。缺乏安全属性的Cookie可以通过XSS或嗅探明文HTTP流量等方式获取。

| 名称        | 用途                                                       |
|-------------|------------------------------------------------------------|
| Domain      | 定义cookie的域范围                                         |
| Path        | 定义域内URL路径范围                                        |
| Expires     | 指示浏览器在指定时间删除cookie                             |
| Max-Age     | 指示浏览器在指定时间删除cookie                             |
| Secure      | 此标志指示浏览器仅通过HTTPS连接传输cookie                  |
| HttpOnly    | 此标志指示浏览器仅通过HTTP(S)传输cookie，而不是其他方式（例如JavaScript） |


客户端随后使用`Cookie`头在每个请求中呈现名称-值对。

### CDN

CDN用于通过从“更接近”客户端的系统提供静态资产（例如图像、可下载文件和流式内容）来减少Web应用中的延迟。

运营商在全球维护接入点（POP）。当用户向CDN主机名发出请求时，基于位置、可用性、成本和其他指标，使用DNS和BGP将请求路由到服务器IP。

然而，当CDN用于提供敏感或私有内容（例如Facebook和Instagram用户的照片）时，会出现问题。如果攻击者知道图像的有效URL，他可以在没有认证的情况下向CDN呈现该URL并获取材料。唯一缓解内容被窥探的是一个不可预测的标识符：

### 负载均衡器

负载均衡系统用于在物理、虚拟和云环境中的许多应用服务器之间分发入站会话。

包括F5 Networks在内的产品供应商生产裸机和虚拟系统，而包括Amazon、Microsoft和Google在内的云提供商在其基础设施即服务（IaaS）平台内提供负载均衡。TLS通常在负载均衡器处终止，而在环境内部使用明文HTTP。

### 表示层数据格式

HTTP `Content-Type`头用于描述正在传输的数据格式。特别是，定义了类型、子类型和可选参数（例如语言或字符集）。常见的媒体类型包括标记和对象表示语言（HTML、XML和JSON）、图像格式（JPEG、GIF和PNG）以及JavaScript。IANA维护着已注册媒体类型的列表，其中包括以下类型：[^10]

*   application/javascript
*   application/json
*   application/xml
*   image/gif
*   image/jpeg
*   image/png
*   text/html

`Content-Encoding`头通常用于描述数据的压缩。客户端使用编码和媒体类型头来处理数据（例如，执行JavaScript，或解压缩并渲染网页及其图像）。类型混淆漏洞可以被利用来执行持久性XSS。

## 应用层

应用服务器支持用Microsoft ASP.NET、Java、Python和Ruby等语言编写的代码执行。连接器和适配器用于在客户端和应用之间进行通信代理。

Java应用服务器组件使用的协议包括JMX、RMI和AJP。Microsoft应用倾向于使用RPC、HTTP和COM机制进行通信。外部依赖可能还包括LDAP，以支持外部认证提供程序（例如Microsoft Active Directory）。

### 应用层数据格式

应用层中使用的媒体类型与表示层中使用的类似，包括JSON和XML。SAML和其他格式支持单点登录和其他功能。

应用组件在传输之前通常会对材料进行序列化。序列化（称为编组）是将数据结构或对象状态转换为可存储、随后在同一或另一环境中重建（称为解组）的格式的过程。

包括Rails[^12]和Django[^13]在内的Web应用框架具有已知的序列化弱点，发送到应用服务器的恶意内容在解组和处理时可能导致漏洞利用（导致代码执行、信息泄露和其他问题）。Gabriel Lawrence和Chris Frohoff在AppSecCali上的演讲详细介绍了这些缺陷的实际利用。[^14]

## 数据层

Web应用中使用的数据存储包括数据库、键值存储和分布式文件系统。连接器用于与数据层组件交互，方式与表示层和应用层之间的交互相同，包括：

*   用于MySQL、PostgreSQL、Microsoft SQL Server等的ODBC和JDBC驱动程序
*   MongoDB、Memcached、Redis等使用的专有协议
*   通过HTTP的REST API（如Amazon S3、WebHDFS等使用的）

服务也可能通过UDP运行以减少开销并提高吞吐量（例如Memcached和NFS）。认证机制各不相同（例如，Redis默认不提供认证，Apache Hadoop使用Kerberos），数据格式范围从人类可读的文档到机器可读的XML、JSON和二进制材料。
