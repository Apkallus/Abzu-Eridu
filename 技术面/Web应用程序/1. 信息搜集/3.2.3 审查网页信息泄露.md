# 审查网页内容中的信息泄露

## 概要

程序员在源代码中包含详细的注释和元数据是非常常见的，甚至是推荐的。然而，HTML 代码中包含的注释和元数据可能会泄露内部信息。应进行注释和元数据审查，以确定是否有任何敏感信息被泄露。此外，某些应用程序可能会在重定向响应的主体中泄露信息。

对于现代 Web 应用程序，前端使用客户端 JavaScript 变得越来越流行。流行的前端构建技术使用客户端 JavaScript，如 React、Angular 或 Vue。与 HTML 代码中的注释和元数据类似，许多程序员也在前端的 JavaScript 变量中硬编码敏感信息。敏感信息可以包括（但不限于）：
- 私有 API 密钥（例如，不受限制的 Google Map API 密钥）
- 内部 IP 地址
- 敏感路由（例如，指向隐藏管理页面或功能的路径）
- 甚至凭据。

对于大型 Web 应用程序，性能问题是程序员非常关心的问题。程序员使用不同的方法来优化前端性能，包括 Syntactically Awesome Style Sheets（Sass）、Sassy CSS（SCSS）、webpack 等。使用这些技术时，前端代码有时会变得更难理解和调试，因此程序员通常会部署源映射文件以方便调试。“源映射”是一种特殊文件，它将资产（CSS 或 JavaScript）的压缩/混淆版本与原始创作版本连接起来。源映射文件或用于调试的文件如果发布到生产环境，将使源代码更具可读性。这会使攻击者更容易从前端发现漏洞或收集敏感信息。应进行 JavaScript 代码审查，以确定是否有任何调试文件从前端暴露。

## 测试目标

- 审查网页注释、元数据和重定向主体，以发现任何信息泄露。
- 收集 JavaScript 文件并审查 JS 代码，以更好地理解应用程序并发现任何信息泄露。
- 确定是否存在源映射文件或其他前端调试文件。

## 如何测试

### 审查网页注释和元数据

HTML 注释通常被开发人员用来包含有关应用程序的调试信息。有时，他们会忘记注释并将其留在生产环境中。测试人员应查找以 `<!--` 开头的 HTML 注释。

检查 HTML 源代码中是否包含可能帮助攻击者更深入了解应用程序的敏感信息的注释。可能是 SQL 代码、用户名和密码、内部 IP 地址或调试信息。

```html
...
<!-- Query: SELECT id, name FROM app.users WHERE active='1' -->
...
```

检查 HTML 版本信息以获取有效的版本号和数据类型定义（DTD）URL

```html
<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01//EN" "https://www.w3.org/TR/html4/strict.dtd">
```

- `strict.dtd` -- 默认严格 DTD
- `loose.dtd` -- 宽松 DTD
- `frameset.dtd` -- 框架集文档的 DTD

某些 `META` 标签不提供主动攻击向量，但允许攻击者对应用程序进行画像：

```html
<META name="Author" content="Andrew Muller">
```

一个常见（但不符合 [WCAG](https://www.w3.org/WAI/standards-guidelines/wcag/)）的 `META` 标签是 [Refresh](https://en.wikipedia.org/wiki/Meta_refresh)。

```html
<META http-equiv="Refresh" content="15;URL=https://www.owasp.org/index.html">
```

`META` 标签的一个常见用途是指定搜索引擎可能用来提高搜索结果质量的关键字。

```html
<META name="keywords" lang="en-us" content="OWASP, security, sunshine, lollipops">
```

尽管大多数 Web 服务器通过 `robots.txt` 文件管理搜索引擎索引，但也可以通过 `META` 标签进行管理。下面的标签将建议机器人不要索引包含该标签的 HTML 页面，也不要跟踪其链接。

```html
<META name="robots" content="none">
```

[互联网内容选择平台（PICS）](https://www.w3.org/PICS/) 和 [Web 描述资源协议（POWDER）](https://www.w3.org/2007/powder/) 提供了将元数据与互联网内容关联的基础设施。

### 识别 JavaScript 代码并收集 JavaScript 文件

程序员经常在前端用 JavaScript 变量硬编码敏感信息。测试人员应检查 HTML 源代码，查找 `<script>` 和 `</script>` 标签之间的 JavaScript 代码。测试人员还应识别外部 JavaScript 文件以审查代码（JavaScript 文件的扩展名为 `.js`，其名称通常放在 `<script>` 标签的 `src`（源）属性中）。

检查 JavaScript 代码中是否有任何敏感信息泄露，这些信息可能被攻击者用来进一步滥用或操纵系统。查找诸如 API 密钥、内部 IP 地址、敏感路由或凭据等值。例如：

```javascript
const myS3Credentials = {
  accessKeyId: config('AWSS3AccessKeyID'),
  secretAccessKey: config('AWSS3SecretAccessKey'),
};
// 或
var conString = "tcp://postgres:1234@localhost/postgres";
```

API 密钥：测试人员可以检查 API 密钥限制是否按服务或 IP、HTTP 引荐来源、应用程序、SDK 等设置。

例如，如果测试人员发现一个 Google Map API 密钥，他们可以检查此 API 密钥是否受 IP 限制，或仅受限于 Google Map API。如果 Google API 密钥仅受限于 Google Map API，攻击者仍然可以使用该 API 密钥查询不受限制的 Google Map API，应用程序所有者必须为此付费。

```html

<script type="application/json">
...
{"GOOGLE_MAP_API_KEY":"AIzaSyDUEBnKgwiqMNpDplT6ozE4Z0XxuAbqDi4", "RECAPTCHA_KEY":"6LcPscEUiAAAAHOwwM3fGvIx9rsPYUq62uRhGjJ0"}
...
</script>
```

敏感路由：例如指向内部或隐藏管理页面的链接。

```html
<script type="application/json">
...
"runtimeConfig":{"BASE_URL_VOUCHER_API":"https://staging-voucher.victim.net/api", "BASE_BACKOFFICE_API":"https://10.10.10.2/api", "ADMIN_PAGE":"/hidden_administrator"}
...
</script>
```

#### 提高 JavaScript 文件的代码可读性

JavaScript 文件通常被压缩以减小其大小，提高 Web 性能。此过程会去除不必要的字符，如空格和注释，使代码更难阅读。

一个工具是 JSbeautify （[https://github.com/beautify-web/js-beautify](https://github.com/beautify-web/js-beautify)）。它可以重新格式化格式不良的 JavaScript，将代码解压缩，并部分去混淆 JavaScript。

```bash
# 以 example.js 作为输入，并将格式化结果返回到 beautify-example.js
js-beautify example.js > beautify-example.js
```

#### 自动化 JS 文件信息搜集

搜集 JS 文件并去重

使用 Python 工具 [LinkFinder](https://github.com/GerbenJavado/LinkFinder) 自动化发现这些文件内的端点和参数。它利用 “jsbeautifier” 和一套全面的正则表达式来匹配端点。

将多个 JavaScript 文件输入到 Linkfinder 并提取相关细节。为此，我们将使用以下 bash 脚本：

**代码**
```bash
while read url; do ./linkfinder.py -i $url -o cli >> paypal-endpoints.txt; done < ../paypal-js.txt
```

该命令使用一个循环，读取 “paypal-js.txt” 文件中的每一行（包含在先前步骤中提取的 URL），然后为每个 URL 执行 “Linkfinder.py”，并将结果输出到 “paypal-endpoints.txt”。

使用 [Secret Finder](https://github.com/m4ll0k/SecretFinder) 工具提取各种敏感细节，包括 API 密钥、访问令牌、授权、JWT 令牌、用户名和密码。以下命令将 “1.js” 文件作为输入，并将返回任何相关细节：

**命令**
```bash
python3 SecretFinder.py -i https://example.com/1.js -o cli
```


### 识别源映射文件

源映射文件通常在开发者工具打开时加载。测试人员还可以通过在每个外部 JavaScript 文件的扩展名后添加“.map”扩展名来查找源映射文件。例如，如果测试人员看到一个 `/static/js/main.chunk.js` 文件，他们可以通过访问 `/static/js/main.chunk.js.map` 来检查其源映射文件。

检查源映射文件中是否有任何可能帮助攻击者更深入了解应用程序的敏感信息。例如：

```json
{
  "version": 3,
  "file": "static/js/main.chunk.js",
  "sources": [
    "/home/sysadmin/cashsystem/src/actions/index.js",
    "/home/sysadmin/cashsystem/src/actions/reportAction.js",
    "/home/sysadmin/cashsystem/src/actions/cashoutAction.js",
    "/home/sysadmin/cashsystem/src/actions/userAction.js",
    "..."
  ],
  "..."
}
```

当站点加载源映射文件时，前端源代码将变得可读且更易于调试。

### 识别泄露信息的重定向响应

虽然重定向响应通常不应包含任何重要的 Web 内容，但不能保证它们不包含内容。因此，虽然 300 系列（重定向）响应通常包含“正在重定向到 `https://example.com/`”类型的内容，但它们也可能泄露内容。

考虑一种情况，重定向响应是身份验证或授权检查的结果，如果检查失败，服务器可能会响应用户重定向回“安全”或“默认”页面，但重定向响应本身可能仍然包含未在浏览器中显示但确实传输到客户端的内容。这可以通过使用浏览器开发者工具或通过个人代理（如 ZAP、Burp、Fiddler 或 Charles）来观察。

### 审查生成文件的元数据泄露

Web 应用程序可能会生成可下载的文件，如 PDF、电子表格、发票或报告。这些文件可能包含描述用于创建它们的软件或库的嵌入元数据。

元数据字段如 *Producer*、*Creator*、*Application* 或 *Library Version* 可能揭示后端技术和特定版本号。此信息可帮助攻击者指纹识别应用程序并识别公开已知的漏洞（CVE）。

例如，生成的 PDF 可能暴露用于创建它的库：

`Producer: iText 2.1.7`

攻击者随后可以搜索影响该版本的已知漏洞。

#### 测试文件元数据

从应用程序下载生成的文件并检查其元数据。

对于 PDF 文件：

```bash
exiftool file.pdf
pdfinfo file.pdf
strings file.pdf | grep -i producer
```

示例输出：

`Producer: iText 2.1.7`

对于 Office 文档（DOCX、XLSX）：

```bash
exiftool file.docx
```

检查元数据字段如：

- Producer
- Creator
- Application
- Creation Tool
- Library Version
