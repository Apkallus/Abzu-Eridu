# 测试 OAuth 弱点

|ID          |
|------------|
|WSTG-ATHZ-05|

## 概述

[OAuth2.0](https://oauth.net/2/)（以下简称 OAuth）是一个授权框架，允许客户端代表其用户访问资源。

为实现此目的，OAuth 严重依赖令牌在不同实体之间进行通信，每个实体具有不同的[角色](https://datatracker.ietf.org/doc/html/rfc6749#section-1.1)：

- **资源所有者：** 授予资源访问权限的实体，即所有者，在大多数情况下就是用户本人。
- **客户端：** 代表资源所有者请求访问资源的应用程序。这些客户端分为两种[类型](https://oauth.net/2/client-types/)：
    - **公共客户端：** 无法保护机密的客户端（例如前端应用程序，如 SPA、移动应用程序等）。
    - **机密客户端：** 能够通过安全保存其注册机密来与授权服务器进行安全认证的客户端（例如后端服务）。
- **授权服务器：** 保存授权信息并授予访问权限的服务器。
- **资源服务器：** 提供客户端访问内容的应用程序。

由于 OAuth 的职责是将所有者的访问权限委托给客户端，因此这对攻击者来说是一个非常吸引人的目标，而糟糕的实现会导致对用户资源和信息的未授权访问。

为了向客户端应用程序提供访问权限，OAuth 依赖于几种[授权类型](https://oauth.net/2/grant-types/)来生成访问令牌：

- [授权码](https://oauth.net/2/grant-types/authorization-code/)：由公共客户端和机密客户端共同使用，用于将授权码交换为访问令牌，但建议仅用于机密客户端。
- [用于代码交换的证明密钥（PKCE）](https://oauth.net/2/pkce/)：PKCE 建立在授权码授权之上，为其提供更强的安全性，使其可用于公共客户端，并改善机密客户端的安全态势。
- [客户端凭据](https://oauth.net/2/grant-types/client-credentials/)：用于机器对机器通信，其中“用户”是请求访问资源服务器上自身资源的机器。
- [设备码](https://oauth.net/2/grant-types/device-code/)：用于输入能力受限的设备。
- [刷新令牌](https://oauth.net/2/grant-types/refresh-token/)：由授权服务器提供的令牌，允许客户端在访问令牌失效或过期时刷新用户访问令牌。此授权类型与其他授权类型结合使用。

在 [OAuth2.1](https://oauth.net/2.1/) 发布时，以下两种流程将被弃用，不建议使用：

- [隐式流*](https://oauth.net/2/grant-types/implicit/)：PKCE 的安全实现使此流程过时。在 PKCE 出现之前，隐式流被客户端应用程序（如[单页应用程序](https://en.wikipedia.org/wiki/Single-page_application)）使用，因为 [CORS](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS) 放宽了站点间通信的[同源策略](https://developer.mozilla.org/en-US/docs/Web/Security/Same-origin_policy)。有关不推荐隐式授权的更多信息，请参阅此[部分](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics#section-2.1.2)。
- [资源所有者密码凭据](https://oauth.net/2/grant-types/password/)：用于将用户的凭据直接交换给客户端，然后客户端将其发送给授权服务器以换取访问令牌。有关不推荐此流程的原因，请参阅此[部分](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics#section-2.4)。

*：隐式流在 OAuth 中已被弃用，但在 Open ID Connect（OIDC）中仍然是获取 `id_tokens` 的可行解决方案。请注意理解隐式流的使用方式，如果仅使用 `/authorization` 端点获取访问令牌，而不以任何方式依赖 `/token` 端点，则可识别为隐式流。相关示例可参考[此处](https://auth0.com/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)。

*请注意，OAuth 流程是一个复杂的话题，上述内容仅涵盖关键领域的摘要。内联引用包含有关特定流程的更多信息。*

## 测试目标

- 确定 OAuth2 实现是否存在漏洞，或是否使用了已弃用或自定义的实现。

## 如何测试

### 测试已弃用的授权类型

已弃用的授权类型因安全性和功能原因而被淘汰。识别它们是否被使用，可以让我们快速检查它们是否容易受到与其使用相关的任何威胁。有些可能不在攻击者的范围内，例如客户端可能使用用户凭据的方式。这应被记录并提交给内部工程团队。

对于公共客户端，通常可以在对 `/token` 端点的请求中识别授权类型。它通过参数 `grant_type` 在令牌交换中指示。

以下示例展示了带有 PKCE 的授权码授权。

```http
POST /oauth/token HTTP/1.1
Host: as.example.com
[...]

{
  "client_id":"example-client",
  "code_verifier":"example",
  "grant_type":"authorization_code",
  "code":"example",
  "redirect_uri":"https://client.example.com"
}
```

`grant_type` 参数的值及其指示的授权类型为：

- `password`：表示 ROPC 授权。
- `client_credentials`：表示客户端凭据授权。
- `authorization_code`：表示授权码授权。

隐式流类型不由 `grant_type` 参数指示，因为令牌是在对 `/authorization` 端点的响应中呈现的，而是通过 `response_type` 识别。示例如下。

```http
GET /authorize
  ?client_id=<some_client_id>
  &response_type=token 
  &redirect_uri=https%3A%2F%2Fclient.example.com%2F
  &scope=openid%20profile%20email
  &state=<random_state>
```

以下 URL 参数指示所使用的 OAuth 流程：

- `response_type=token`：表示隐式流，因为客户端直接请求授权服务器返回令牌。
- `response_type=code`：表示授权码流，因为客户端请求授权服务器返回代码，随后将用该代码交换令牌。
- `code_challenge=sha256(xyz)`：表示 PKCE 扩展，因为没有其他流程使用此参数。

以下是一个带有 PKCE 的授权码流的授权请求示例：

```http
GET /authorize
    ?redirect_uri=https%3A%2F%2Fclient.example.com%2F
    &client_id=<some_client_id>
    &scope=openid%20profile%20email
    &response_type=code
    &response_mode=query
    &state=<random_state>
    &nonce=<random_nonce>
    &code_challenge=<random_code_challenge>
    &code_challenge_method=S256 HTTP/1.1
Host: as.example.com
[...]
```

#### 公共客户端

建议公共客户端使用带有 PKCE 扩展的授权码授权。带有 PKCE 的授权码流的授权请求应包含 `response_type=code` 和 `code_challenge=sha256(xyz)`。

令牌交换应包含授权类型 `authorization_code` 和 `code_verifier`。

公共客户端不合适的授权类型有：

- 不带 PKCE 扩展的授权码授权
- 客户端凭据
- 隐式流
- ROPC

#### 机密客户端

建议机密客户端使用授权码授权。也可使用 PKCE 扩展。

机密客户端不合适的授权类型有：

- 客户端凭据（机器对机器除外——见下文）
- 隐式流
- ROPC

##### 机器对机器

在无用户交互且客户端仅为机密客户端的情况下，可使用客户端凭据授权。

如果知道 `client_id` 和 `client_secret`，可以通过传递 `client_credentials` 授权类型来获取令牌。

```bash
$ curl --request POST \
  --url https://as.example.com/oauth/token \
  --header 'content-type: application/json' \
  --data '{"client_id":"<some_client_id>","client_secret":"<some_client_secret>","grant_type":"client_credentials"}' --proxy https://localhost:8080/ -k
```

### 凭据泄露

根据流程不同，OAuth 会以 URL 参数的形式传输多种类型的凭据。

以下令牌可被视为泄露的凭据：

- 访问令牌
- 刷新令牌
- 授权码
- PKCE code_challenge / code_verifier

由于 OAuth 的工作方式，授权 `code` 以及 `code_challenge` 和 `code_verifier` 可能是 URL 的一部分。如果 `response_mode` 未设置为 [`form_post`](https://openid.net/specs/oauth-v2-form-post-response-mode-1_0.html)，隐式流会将授权令牌作为 URL 的一部分传输。这可能导致请求的令牌或代码在 referrer 头、日志文件及代理中泄露，因为这些参数通过查询字符串或片段传递。

隐式流泄露令牌所带来的风险远高于泄露 `code` 或其他 `code_*` 参数，因为后者绑定到特定客户端，在泄露时更难被滥用。

为了测试此场景，请使用 HTTP 拦截代理（如 ZAP）拦截 OAuth 流量。

- 逐步执行授权过程，识别 URL 中存在的任何凭据。
- 如果与 OAuth 流程相关的页面中包含了任何外部资源，请分析对这些资源发出的请求。凭据可能通过 referrer 头泄露。

逐步执行 OAuth 流程并使用应用程序后，HTTP 拦截代理的请求历史中会捕获到一些请求。在请求历史中搜索包含授权服务器和客户端 URL 的 HTTP referrer 头（例如 `Referer: https://idp.example.com/`）。

审查 HTML meta 标签（尽管此标签并非在所有浏览器上[受支持](https://caniuse.com/mdn-html_elements_meta_name_referrer)）或 [Referrer-Policy](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Referrer-Policy) 有助于评估是否通过 referrer 头发生了凭据泄露。

## 修复

- 在实施 OAuth 时，始终考虑所使用的技术，以及应用程序是能够避免泄露机密的服务器端应用程序，还是无法避免的客户端应用程序。
- 在几乎所有情况下，使用带有 PKCE 的授权码流。机器对机器流程可能是一个例外。
- 使用 POST 参数或头部值来传输机密。
- 当没有其他可能性时（例如无法迁移的旧应用程序），实施额外的安全头部，例如 `Referrer-Policy`。
