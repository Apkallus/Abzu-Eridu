# 第 19 章：第二部分总结

当今的 Web 应用存在着大量的漏洞。其中一些漏洞很容易分类，就像我们在第二部分中评估和测试的那些漏洞。另一些漏洞则更为小众——如果某个应用具有不常见的安全模型或拥有在别处找不到的独特架构功能，则这些漏洞可能是该应用所独有的。

最终，要全面测试一个 Web 应用，需要具备常见漏洞原型知识、批判性思维技能和领域知识，以便能够发现最常用原型之外的深层逻辑漏洞。第一部分和第二部分中呈现的基础技能应足以让您在未来参与的任何 Web 应用安全渗透测试项目中顺利起步。

从现在开始，请注意您测试的任何应用中的业务模型。所有应用都存在诸如 XSS、CSRF 或 XXE 等漏洞的风险，但只有通过深入了解应用中的底层业务模型和业务逻辑，您才能识别出更高级和更具体的漏洞。

如果您觉得第二部分中介绍的漏洞在现实场景中难以应用，请思考原因。可能是您正在测试的应用经过了全面加固，但更可能的情况是，虽然您已经掌握了开发和部署这些攻击的知识，但您可能需要进一步改进或应用您的侦察技能，以便在应用中找到可以成功部署这些攻击的弱点。

本书第二部分学到的技能直接建立在第一部分技能的基础之上。此外，当您进入本书关于 Web 应用安全的最后一部分（防御机制以抵御攻击）时，这些技能也将对您大有裨益。

在您进行本书最后部分的学习时，请牢记迄今为止已掌握的侦察技术和进攻性黑客技术。当您学习防御示例时，请不断思考黑客如何在有适当防御和无适当防御的情况下发现并利用应用。

您将了解到 Web 应用防御措施常常是不完美的，这就是为什么它们经常被称为“缓解措施”而非“修复方案”。凭借第一部分和第二部分的知识，您或许能够找到在第三部分中绕过或削弱特定防御措施的方法。第三部分中介绍的防御措施大多被认为是行业最佳实践，但许多并非无懈可击，通常应结合多种防御措施而非仅依赖单一防御。

最后需要指出，第二部分中介绍的技术确实具有危险性。这些是真实攻击者经常使用的真实攻击手段。欢迎您在您自己的 Web 应用上进行测试，但未经 Web 应用所有者的明确书面许可，请勿对他人拥有的 Web 应用进行测试。

之前章节中的技术既可被用于善意目的，也可被用于恶意目的。因此，必须慎重考虑这些技术的应用和使用，不可随意部署。

其中一些技术还可能导致服务器或客户端机器被入侵，即使获得了应用所有者的许可。请牢记每次攻击的影响，并确保在开始之前，应用所有者了解在线测试所涉及的风险。
