# 第 14 章：拒绝服务（DoS）

分布式拒绝服务（DDoS）攻击或许是最为人所知、也最被广泛报道的攻击类型之一。这种攻击是拒绝服务（DoS）的一种形式，由大量设备组成的网络向服务器发送海量请求，从而拖慢服务器速度或使其对合法用户不可用。

DoS 攻击有多种形式，从众所周知的、涉及数千台或更多协调设备的分布式版本，到影响单个用户的代码级 DoS（由错误的正则表达式实现导致验证文本字符串耗时过长）不等。DoS 攻击的严重程度也各不相同，从拖慢活动服务器、导致账单无效，到使用户的网页加载速度比平时稍慢，或使视频在缓冲期间暂停。

因此，测试 DoS 攻击（尤其是严重程度较低的攻击）非常困难。大多数漏洞赏金计划都完全禁止提交 DoS 漏洞，以防止赏金猎人干扰正常的应用使用。

到本章结束时，您将了解各种常见的 DoS 攻击、几种高级的 DoS 攻击形式，以及所有 DoS 攻击背后的共同概念。有了这些知识，您将能够通过开发自己的 DoS 攻击来攻击 Web 应用。

> 由于 DoS 漏洞会通过应用干扰正常用户的使用，因此在本地开发环境中测试 DoS 漏洞最有效，这样真实用户不会经历服务中断。

除了少数例外，DoS 攻击通常不会对应用造成永久性损害，但会干扰合法用户对应用的可用性。

根据具体的 DoS 攻击类型，有时可能很难找到导致用户体验降级的 DoS 接收器。

## 正则表达式 DoS

基于正则表达式 DoS 的漏洞是当今 Web 应用中最常见的 DoS 形式之一。一般来说，这些漏洞的风险范围从非常轻微到中等，通常取决于正则表达式解析器的位置。

退一步说，正则表达式通常用于 Web 应用中验证表单字段，并确保用户输入服务器期望的文本。通常这意味着只允许用户向密码字段输入应用选择接受的字符，或者限制评论的最大字符数，以便完整评论在 UI 中显示良好。

正则表达式最初由研究形式语言理论的数学家设计，用于以非常紧凑的方式定义字符串的集合和子集。当今 Web 上的几乎每种编程语言都包含自己的正则表达式解析器，浏览器中的 JavaScript 也不例外。

在 JavaScript 中，正则表达式通常通过两种方式定义：

```javascript
const myregex = /username/;      // 字面量定义
const myregex = new RegExp('username'); // 构造函数
```

完整的正则表达式课程超出了本书范围，但重要的是要注意，正则表达式通常是搜索或匹配文本的快速且强大的方式。至少掌握正则表达式的基础知识绝对值得。

就本章而言，我们只需要知道 JavaScript 中两个正斜杠之间的任何内容都是正则表达式字面量：`/test/`。

正则表达式也可用于匹配范围：

```javascript
const lowercase = /[a-z]/;
const uppercase = /[A-Z]/;
const numbers = /[0-9]/;
```

我们可以将这些与逻辑运算符（如 OR）组合：

```javascript
const youori = /you|i/;
```

等等。您可以在 JavaScript 中轻松测试一个字符串是否匹配正则表达式：

```javascript
const dog = /dog/;
dog.test('cat'); // false
dog.test('dog'); // true
```

如前所述，正则表达式通常解析得非常快。很少有正则表达式慢到足以拖慢 Web 应用。话虽如此，正则表达式可以被专门构造以运行缓慢。这些被称为恶意正则表达式（有时也称为邪恶正则表达式），当允许用户提供自己的正则表达式用于其他 Web 表单或服务器时，这是一个巨大的风险。恶意正则表达式也可能被意外引入应用，尽管开发者对正则表达式不够熟悉以致于避免一些常见错误的情况可能很少见。

一般来说，大多数恶意正则表达式都是使用加号“+”运算符形成的，该运算符将正则表达式变为“贪婪”操作。贪婪正则表达式会测试一个或多个匹配项，而不是在找到第一个匹配项时停止。

恶意正则表达式在发现失败案例时会触发回溯。考虑正则表达式：`/^((ab)*)+$/`。此正则表达式执行以下操作：

1. 在行首定义捕获组 `((ab)*)+`。
2. `(ab)*` 表示匹配 0 到无限个 `ab` 组合。
3. `+` 表示查找 #2 的每个可能匹配项。
4. `$` 表示匹配到字符串末尾。

使用输入 `abab` 测试此正则表达式实际上运行得相当快，不会引起太多问题。向外扩展模式，`ababababababab` 也会运行得相当快。如果我们将此模式修改为 `abababababababa` 并增加一个额外的 “a”，正则表达式将突然变慢，可能需要几毫秒才能完成。

这是因为正则表达式在末尾之前都是有效的，此时引擎将回溯并尝试找到组合匹配：

- `(abababababababa)` 无效。
- `(ababababababa)(ba)` 无效。
- `(abababababa)(baba)` 无效。
- 多次迭代后：`(ab)(ab)(ab)(ab)(ab)(ab)(ab)(a)` 无效。

本质上，由于正则表达式引擎试图穷尽所有可能的 `(ab)` 有效组合，在确定字符串本身无效之前（在检查完所有可能组合之后），它必须完成等于字符串长度的组合数。

使用正则表达式引擎快速尝试此技术的结果如表 14-1 所示。

**表 14-1. 正则表达式（恶意输入）匹配时间（/^((ab)*)+$/）**

| 输入 | 执行时间 |
| :--- | :--- |
| `abababababababababababa`（23 字符） | 8 ms |
| `ababababababababababababa`（25 字符） | 15 ms |
| `abababababababababababababa`（27 字符） | 31 ms |
| `ababababababababababababababa`（29 字符） | 61 ms |

如您所见，使用此恶意正则表达式构造用于破坏正则表达式解析器的输入，每增加两个字符就会使解析器完成匹配的时间加倍。这种情况会继续下去，最终很容易导致 Web 服务器性能显著下降（如果在服务端计算）或完全崩溃 Web 浏览器（如果在客户端计算）。

有趣的是，此恶意正则表达式并非对所有输入都易受攻击，如表 14-2 所示。

**表 14-2. 正则表达式（安全输入）匹配时间（/^((ab)*)+$/）**

| 输入 | 执行时间 |
| :--- | :--- |
| `ababababababababababab`（22 字符） | <1 ms |
| `abababababababababababab`（24 字符） | <1 ms |
| `ababababababababababababab`（26 字符） | <1 ms |
| `abababababababababababababab`（28 字符） | >1 ms |

这意味着在 Web 应用中使用的恶意正则表达式可能潜伏数年，直到黑客找到导致正则表达式解析器执行大量回溯的输入。因此，它似乎会凭空出现。

正则表达式 DoS 攻击比您想象的更常见，如果找到正确的载荷，它们可以轻易使服务器宕机或使客户端机器无法使用。请注意，开源软件通常更容易受到恶意正则表达式的影响，因为很少有开发者能够检测到恶意正则表达式。

## 逻辑 DoS 漏洞

在逻辑 DoS 漏洞中，服务器资源被非法用户耗尽。结果，合法用户经历性能下降或服务中断（如图 14-1 所示）。

> **图 14-1. 服务器资源被非法用户耗尽，导致合法用户性能下降或服务中断**

正则表达式是了解 DoS 漏洞和利用 DoS 的简单入门，因为它为研究和尝试攻击提供了一个集中的起点（任何存在正则表达式解析器的地方）。然而，重要的是要注意，由于 DoS 的广泛性，DoS 漏洞几乎可以在任何类型的软件中找到！

逻辑 DoS 漏洞是最难发现和利用的漏洞之一，但它们在现实中的出现频率比预期更高。这些漏洞需要一定的专业知识来确定和利用，但在掌握了几种查找技术之后，您可能能够找到许多。

首先，我们需要思考什么使 DoS 攻击起作用。DoS 攻击通常基于消耗服务器或客户端硬件资源，使其无法用于合法目的。这意味着我们首先要寻找 Web 应用中资源密集型的操作。一个非穷尽的列表可能包括：

- 您可以确认是同步运行的任何操作
- 数据库写入
- 磁盘写入
- SQL 连接（JOIN）
- 文件备份
- 循环逻辑操作

通常，Web 应用中复杂的 API 调用不仅包含其中一种，而是包含多种操作。例如，一个照片分享应用可能暴露一个允许用户上传照片的 API 路由。在上传过程中，此应用可能执行：

- 数据库写入（存储关于照片的元数据）
- 磁盘写入（记录照片成功上传）
- SQL 连接（积累足够的用户和相册数据以填充元数据写入）
- 文件备份（以防灾难性服务器故障）

我们无法轻松地测量我们无权访问的服务器上这些操作的持续时间，但我们可以结合计时和估算来确定哪些操作比其他操作更长。例如，我们可以首先从开始到结束对请求进行计时。这可以使用浏览器开发者工具完成。

我们也可以通过同时多次发起相同请求并观察响应是否交错，来测试操作是否在服务器上同步发生。每次这样做时，我们应该编写脚本并平均大约一百个 API 调用，以便我们的指标不受随机差异的影响。

也许当我们测试时，服务器恰好遇到流量高峰，或开始资源密集型的定时任务。平均请求时间将为我们提供哪些 API 调用耗时显著的更准确度量。

我们还可以通过仔细分析网络载荷和 UI 来近似后端代码的结构。如果我们知道应用支持这些类型的对象：

- 用户对象
- 相册对象（用户拥有相册）
- 照片对象（相册包含照片）
- 元数据对象（照片拥有元数据）

那么我们可以看到每个子对象都由 ID 引用：

```json
// 照片 #1234
{
  "image": "data",
  "metadata": "123abc"
}
```

我们可能会假设用户、相册、照片和元数据存储在不同的表或文档中，具体取决于使用的数据库是 SQL 还是 NoSQL。如果在我们的 UI 中，我们发起请求查找与用户关联的所有元数据，那么我们知道后端必须运行复杂的连接操作或迭代查询。假设此操作位于端点 `GET /metadata/:userid`。

此操作的规模根据用户使用应用的方式而有很大差异。高级用户可能需要大量硬件资源来执行此操作，而新用户则不需要。

我们可以测试此操作并观察其扩展方式，如表 14-3 所示。

**表 14-3. 按账户类型划分的 GET /metadata/:userid 响应时间**

| 账户类型 | 响应时间 |
| :--- | :--- |
| 新账户（1 个相册，1 张照片） | 120 ms |
| 普通账户（6 个相册，60 张照片） | 470 ms |
| 高级用户（28 个相册，490 张照片） | 1,870 ms |

鉴于操作根据用户账户类型的扩展方式，我们可以推断，我们可以创建一个配置文件，通过 `GET /metadata/:userid` 消耗服务器资源时间。如果我们编写客户端脚本将相同或相似的图像重新上传到大量相册中，我们可以拥有一个包含 600 个相册和 3,500 张照片的账户。

之后，只需对端点 `GET /metadata/:userid` 重复发起请求，就会导致其他用户的服务器性能显著下降，除非服务端代码极其健壮，并在每个请求的基础上限制资源。这些请求可能会超时，但即使服务器软件超时且未将结果发送回发起请求的客户端，数据库很可能仍会协调资源。

这只是一个关于如何发现和利用逻辑 DoS 攻击的示例。当然，这些攻击因案例而异——因此称为“逻辑 DoS”，由您所利用的应用中的特定应用逻辑定义。

## 分布式 DoS（DDoS）

在 DDoS 中，服务器资源被大量非法用户耗尽。由于他们集体发起请求，甚至可能执行标准请求。大规模情况下，这将淹没合法用户的服务器资源（见图 14-2）。

> **图 14-2. DDoS 服务器资源被大量非法用户集体耗尽**

DDoS 攻击全面涵盖超出了本书的范围，但您至少应在概念上熟悉它们的工作原理。与单个黑客针对另一个客户端或服务器以拖慢其速度的 DoS 攻击不同，分布式攻击涉及多个攻击者。攻击者可以是其他黑客或网络僵尸（僵尸网络）。

理论上，这些僵尸可以利用任何类型的 DoS 攻击，但规模更大。例如，如果服务器在其 API 端点之一中使用正则表达式，僵尸网络可以让多个客户端同时向同一 API 端点发送恶意载荷。然而，在实践中，大多数 DDoS 攻击不执行逻辑或基于正则表达式的 DoS，而是攻击较低层（通常在网络层而非应用层）。大多数基于僵尸网络的 DDoS 攻击将直接针对服务器的 IP 地址发起请求，而非针对任何特定的 API 端点。这些请求通常是用户数据报协议（UDP）流量，试图淹没服务器用于合法请求的可用带宽。

如您所想，这些僵尸网络通常不是由单个黑客拥有的设备，而是黑客或黑客团体通过互联网分发的恶意软件所控制的设备。它们是真实的人拥有的真实计算机，但安装了允许远程控制的软件。这是一个大问题，因为它使检测非法客户端变得更加困难。（他们是真实用户吗？）

如果您能够访问僵尸网络，或可以为安全测试目的模拟僵尸网络，那么尝试结合网络层和应用层攻击将是明智的。前述任何针对服务器的 DoS 攻击都容易受到 DDoS 攻击。一般来说，DDoS 攻击对单个客户端效果不佳，尽管也许在范围内，可以播种大量正则表达式易受攻击的载荷，稍后交付给客户端设备并执行。

## 高级 DoS

虽然本章已涵盖了最常见的 DoS 攻击类型，但由于 DoS 的性质，还存在许多晦涩和较少见的形式。任何降低或阻碍 Web 应用可用性的自动化努力，无论其来源如何，都可以被视为拒绝服务。

### YoYo 攻击

YoYo 攻击是一种较新的通过 DoS 攻击 Web 应用的方法，它利用像 Amazon Web Services（AWS）或 Google Cloud Platform（GCP）等大型云提供商内置的自动缩放功能。自动缩放器的理论基础是，Web 应用可以在流量较少时使用有限的计算资源（例如低 CPU、低 RAM）。这可以节省成本。然而，当负载增加时，云提供商会自动增加分配给给定 Web 应用的计算资源量——同时产生相应的硬件账单。

YoYo 攻击依赖于此自动缩放功能，攻击者在短时间内向 Web 应用发送大量请求——触发自动缩放器增加硬件资源。云提供商为目标 Web 应用扩展计算资源，然后流量突然停止。此时，云提供商很快将不得不开始减少计算资源。

一旦计算资源缩减回基线，攻击者开始另一轮请求以触发再次扩展。此过程反复进行。

YoYo 攻击不仅在硬件重新分配期间导致最终用户体验降级，还会导致云托管成本激增。如果攻击持续任何实质性时间，这些成本可能相当可观。

### 压缩攻击

允许文件上传的应用通常容易受到涉及畸形文件的应用层 DoS 攻击。考虑一个类似于 YouTube 的视频托管网站：这样的网站接受视频文件上传，但在向最终用户提供服务之前，会通过压缩器和优化器（例如 FFMPEG）运行视频文件。此过程减少带宽压力，并确保所有视频都是浏览器兼容的文件格式。

然而，在压缩和优化步骤中，最终用户提供的原始视频文件必须被解析。在这种情况下，精心制作的视频文件可能显著消耗服务端计算资源，从而降低所有其他使用该服务的用户的体验。

随着越来越多的网站依赖用户提交的数据，这些形式的 DoS 漏洞变得越来越常见。例如，CVE-2021-38094 描述了针对开源 FFMPEG（版本 4.1）视频压缩和转换库的此类攻击。

在 CVE-2021-38094 场景中，通过 FFMPEG 运行的视频文件包含超出有效输入文件预期范围的数据，将触发文件 libavfilter/vf_convolution.c 中函数 filter_sobel() 内的整数溢出错误。一旦发生此溢出，将导致一系列有问题的函数调用，导致极高的 CPU 和内存消耗，并可能导致 FFMPEG 进程崩溃。当然，这会降低或破坏依赖 FFMPEG 的服务对其他合法用户的可用性。

基于压缩的 DoS 攻击通常难以发现，因为许多 Web 应用依赖第三方软件来处理文件上传。像 FFMPEG 和 ImageMagick 这样的工具简化了开发者对复杂文件的上传和处理——但也为攻击者构造和利用畸形上传载荷暴露了显著的表面积。

### 基于代理的 DoS

DoS 攻击通常需要攻击者付出大量计算资源才能执行。当攻击拥有大量计算资源的 Web 应用时，通常需要花费成本才能以足够的请求命中 Web 应用以降低用户性能。

在这些情况下，一些攻击者寻找从合法 Web 应用窃取计算资源，并将 Web 应用的网络请求重定向到目标网站的方法。这降低了 DoS 攻击的成本，并使受害者的 Web 应用更难确定谁在发起攻击。

考虑像 Google、Bing 或 Yahoo! 这样的搜索引擎。这些搜索引擎拥有大量计算资源，专门用于抓取和缓存来自新网站的数据。

这些爬虫扫描新发现的网站，对其进行评级，并将数据片段存储在自己的数据库中。在爬虫运行时，每个请求的页面需要少量计算资源。有趣的是，大多数 Web 应用开发者为了让其应用被索引以便更多合法用户找到，特意在其 robots.txt 文件中将爬虫加入白名单。

由于这些搜索引擎爬虫一开始就拥有大量计算资源，有时可以“诱骗”爬虫向目标网站发送过多抓取请求以达到 DoS 目的。通过利用新的域名和子域名（将请求代理到目标 Web 应用），黑客可以诱骗爬虫反复索引相同的内容。扩展到数千或数百万个子域名，这种方法可以显著降低 Web 应用的性能，而无需自己支付所有攻击的计算资源费用。

我将这些形式的 DoS 攻击称为基于代理的 DoS，因为黑客使用搜索引擎等中介服务向目标 Web 应用发送流量，而不是自己将流量发送给目标。随着 Web 应用——尤其是用于测试和抓取其他 Web 应用的 Web 应用——的发展，这种攻击形式将越来越常见于黑客之中。

## 小结

经典的 DDoS 攻击是迄今为止最常见的 DoS 形式，但它们只是众多旨在消耗服务器资源以致合法用户无法使用的攻击之一。DoS 攻击可能发生在应用栈的许多层——从客户端到服务器，在某些情况下甚至发生在网络层。这些攻击可以一次影响一个用户或大量用户，损害范围从降低应用性能到完全锁定应用。

在寻找 DoS 攻击时，最好调查哪些服务器资源最有价值，然后开始尝试找到使用这些资源的 API。服务器资源的价值因应用而异，但可以是标准的，如 RAM/CPU 使用率，或者更复杂的，如在队列中执行的功能（用户 a → 用户 b → 用户 c 等）。

虽然通常只会造成烦恼或中断，但某些 DoS 攻击也可能泄露数据。请注意任何 DoS 尝试所导致的日志和错误。
