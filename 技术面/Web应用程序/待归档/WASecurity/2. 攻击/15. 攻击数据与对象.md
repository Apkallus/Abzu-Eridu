# 第 15 章：攻击数据与对象

大多数现代编程语言在实现逻辑程序设计时都利用两种不同的能力：数据（通常以对象的形式表示）和操作（通常以函数的形式表示）。即使在不支持面向对象编程（OOP）的编程语言中，对象通常也被定义为**一等公民**。

“一等公民”是一个编程语言设计概念，指的是编程语言中的一个实体，它可以被赋值、重新赋值、修改、作为参数传递给函数，并从函数中返回。几乎所有现代编程语言都将对象（数据）定义为一等公民，但并非所有现代语言都将函数（操作）定义为一等公民。因此，可以说大多数现代编程语言将存储数据和操作数据的功能划分为两个不同的语言特性。

本章内容是关于在数据以对象形式存储并通过函数操作时，如何利用这些数据进行攻击的方法。这些技术适用于多种现代编程语言，它们滥用了将数据存储为一等公民所带来的强大副作用。

## 批量赋值（Mass Assignment）

攻击一等公民对象的第一种也是最常见的方法是批量赋值攻击。虽然通常使用现代标准化术语“批量赋值攻击”或“批量赋值漏洞”来指代，但历史上也曾使用“自动绑定攻击”（针对 Spring MVC 和 ASP.NET 框架）和“对象注入攻击”（针对遗留 PHP 漏洞）等术语。所有这些术语都指同一套利用系统。

批量赋值漏洞允许攻击者通过向函数调用传递对象中额外的、未经键验证的字段，来更改本不应该被更改的字段。尽管听起来有些令人困惑，但在阅读了本节中的示例后，它就会变得非常容易识别。

批量赋值漏洞通常出现在那些大量传递状态相关数据（以对象形式）的系统。一个典型的容易受到批量赋值攻击的应用是视频游戏，它会经常传递用户状态信息。另一种可能易受攻击的应用是实现了“动态表单”的 Web 应用，即表单的标题、数据类型等可以由最终用户更改。如果没有考虑到安全性进行编程，这两种应用都很容易受到批量赋值攻击。

让我们花点时间评估以下代码片段，它们描述了一种常见的批量赋值漏洞形式：

```javascript
/*
 * 这是一个服务端 API 端点，用于更新基于 Web 的视频游戏 "MegaGame" 的玩家数据。
 */
app.post("updatePlayerData", function(req, res, next) {
  // 如果客户端发回了玩家状态数据，则在数据库中更新
  if (!!req.body.data) {
    db.update(session.currentUser, req.body.data);
    return res.sendStatus(200); // 成功
  } else {
    return res.sendStatus(400); // 错误
  }
});
```

在这个批量赋值漏洞的示例中，视频游戏 MegaGame 实现了一个可通过 HTTP POST 访问的 API 端点。该端点从客户端（Web 浏览器）接收一个参数，形式为状态对象。如果状态对象不为空，API 端点会调用数据库辅助库，使用客户端（代表正在玩游戏的用户）提供的状态来更新当前用户的游戏状态。

API 端点中引用的 `db.update()` 函数编写如下：

```javascript
const update = function(data) {
  for (const [key, value] of Object.entries(data)) {
    database.upsert({ `${key}`: `${value}` })
  }
}
```

请注意，这个 `update()` 函数中的问题（漏洞）并不在于它 upsert（如果不存在则插入，如果存在则更新）客户端提供的数据。相反，批量赋值漏洞之所以出现，是因为该函数没有验证客户端提供的任何信息，尤其是 `data` 对象中的键。

该应用开发者的假设是客户端会传回包含以下数据的载荷：

```javascript
const data = {
  playerId: 123,
  playerPosition: { "x": 125, "y": 346 },
  playerHP: 90
};
```

因为 `update()` 函数缺乏任何验证，并直接接受来自客户端的原始数据，黑客可以在传输过程中修改载荷，或者发送一个恶意载荷，包含以下数据：

```javascript
const data = {
  playerId: 123,
  playerPosition: { "x": 125, "y": 346 },
  playerHP: 90,
  isAdmin: true // 这是载荷的“攻击”部分
};
```

通过修改载荷使其包含 `isAdmin` 键，`update()` 函数将在用户数据库行中写入或修改 `isAdmin` 条目。现在，用户通过滥用 `update()` 函数中未进行清理和验证的事实，将其在游戏中的角色提升为管理员。如果 `update()` 函数验证了对象的键并对 `isAdmin` 键进行了清理，那么当这个恶意载荷命中 `updatePlayerData` 端点时，就不会发生角色升级。

这是一个完美的批量赋值漏洞示例，也是最常见的对象漏洞形式之一，也是最容易利用的漏洞之一。

## 不安全的直接对象引用（IDOR）

不安全的直接对象引用攻击（IDOR）是一种常见的 Web 应用漏洞，它在 2007 年被列入 OWASP 十大漏洞列表时受到关注。从那时起，IDOR 漏洞在 Web 上变得不那么常见；然而，由于其简单的特性，它们仍然不时出现，并且利用起来几乎不需要什么技术。

当服务器上的对象可以通过用户提供的参数（如 URL 查询参数、HTTP POST 请求体或 URL 中的 `:id` 字段）直接访问时，就会发生 IDOR 漏洞。由于信任最终用户提供正确的参数来引用服务器上的特定对象，最终用户极有可能访问到其他对象，从而导致权限提升。

考虑以下 HTTP POST 端点：

```javascript
app.get('/files/:id', function(req, res, next) => {
  return res.sendFile(`/filesystem/files/${req.params.id}`);
});
```

此端点根据由 `:id` 表示的文件名返回 `/filesystem/files` 下的任何文件。在客户端，开发者可能期望最终用户能够访问一个特定的文件。例如，开发者可能希望最终用户能够下载一个名为 `my-report-card.txt` 的文件。

为了请求文件 `my-report-card.txt`，最终用户的浏览器会发起以下请求：

```
HTTP GET https://mywebsite.com/files/my-report-card.txt
```

攻击此 IDOR 很简单：只需在浏览器中拦截 HTTP GET 请求并更改文件名。或者使用像 Burp Suite 或 ZAP 这样的 HTTP 代理工具，在请求到达服务器之前拦截并修改它。

如果文件名被更改为 `other-report-card.txt`，即使当前用户不应该能够访问它，服务器也会尝试返回文件 `other-report-card.txt`。因此，IDOR 不仅是一种通过对象引用攻击对象的方法，也是一种权限提升的形式。

## 序列化攻击

当数据在浏览器客户端、服务器和服务之间来回传递时，以原始格式发送通常不切实际、成本高昂且困难重重。为了解决这个问题，许多 Web 应用在通过网络发送数据之前会对原始数据进行序列化。

### Web 序列化解析

序列化的一个关键概念是，序列化的格式应该易于反序列化，即还原为其原始数据形式。此外，它的格式应该便于在需要时存储在磁盘上。

序列化的一个示例是从浏览器的 JavaScript 执行上下文中获取一个内存对象，并将其转换为 JavaScript 对象表示法（JSON）。表示一个序列化的 `farmer` 类的 JSON “数据块”如下所示：

```json
{
  "name": "Joe Carrot",
  "age": 62,
  "location": "Montana, USA",
  "crops": {
    "wheat": {
      "measurement": "acres",
      "amount": 100
    }
  }
}
```

在此示例中，与农夫“Joe Carrot”相关的数据可以轻松存储在磁盘上或通过网络传输。在需要时，它可以被反序列化并加载到内存中——创建新的内存地址和指针，并允许编程语言对其进行修改。

Web 应用中使用的其他流行的数据序列化格式包括 XML、YAML 和 base64（一种以二进制形式序列化文本的格式）。

### 攻击弱序列化

通过序列化攻击来攻击 Web 应用的第一步是找到应用中执行数据序列化的函数。下一步是使用测试数据调用该函数，以便比较一组序列化函数的输入和输出。

考虑流行的 npm 库 [serialize-javascript](https://oreil.ly/zCgW7)。该库每周下载量超过 3000 万次，并用于数万个基于 JavaScript 的应用。然而，3.0.9 及更低版本容易受到代码注入攻击，其 `serialization()` 函数被用作载荷的传递机制。

在这种情况下，将载荷 `{"foo": /1"/, "bar": "a\"@__R-<UID>-0__@"}` 输入 `serialization()` 函数将导致序列化 JSON 为 `{"foo": /1"/, "bar": "a\/1"/}`。生成的 JSON 对象的结构会转义引号（引号是正确 JSON 对象的预期部分），因此如果它曾经通过 `eval()` 函数运行，就会导致代码执行。

像 `eval('(' + serialize({"foo": /1" + console.log(1) /i, "bar": '"@__R-<UID>-0__@'}) + ')');` 这样的概念验证代码会导致代码执行，因为序列化器无法将此输出字符串格式化为正确转义的 JSON。

所有序列化攻击都遵循相同的步骤：

1. 找到一个执行序列化的函数。
2. 仔细阅读该函数，或使用常见载荷进行测试。
3. 在发现无法正确序列化数据后，创建一个能够执行脚本的载荷。
4. 使用能够执行脚本的载荷调用该函数。
5. 获得远程代码执行（服务端）或 XSS（客户端）。

由于大多数网站并不实现自己的序列化器，因此有可能找到存在漏洞的开源序列化器，并扫描易受攻击的应用和方法。这是一种高级攻击技术，但它是现实中寻找和利用序列化漏洞的最佳方法之一。

## 小结

尽管多年来 Web 已经发生了显著演变，但 Web 编程语言仍然使用存在已久的编程语言设计模式。当您开始理解任何复杂应用底层的系统时，如何从根本上利用这些系统就变得显而易见。本章中针对对象和数据的攻击旨在展示，即使在编程语言设计经历了数十年的改进之后，底层系统往往仍然是黑客的可行目标。
