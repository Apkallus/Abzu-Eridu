# 第 18 章：业务逻辑漏洞

在前面的章节中，我们讨论了许多影响大多数 Web 应用的常见漏洞。这些漏洞很容易用诸如注入或拒绝服务等术语进行分类。上述漏洞几乎总是呈现出一种一致的形态，这使得它们很容易按类别定义。这也意味着，针对常见漏洞的攻击或缓解策略在所有受影响的应用中相对一致。

到目前为止，我们研究的是典型的 Web 应用漏洞。但当我们遇到一个仅存在于单个应用中的漏洞时会发生什么？独特的漏洞最常出现的原因是应用实现了特定的业务逻辑规则。然后黑客找到利用这些编程规则并获得非预期结果的方法。换句话说，我们迄今为止研究的漏洞可能因应用逻辑而发生，但并非因业务规则而发生。

应用逻辑结合信息和指令以执行 Web 应用常见的任务，如渲染图像或执行网络调用。而业务规则则特定于实施该业务的具体企业。业务规则的一个示例是：仅当当前时间距离预订时间超过 24 小时时，才允许乘客取消预订。

业务规则实现中的漏洞通常被称为业务逻辑漏洞。这种形式的漏洞更难发现和利用，因为按照定义，它们需要理解特定于特定应用的一套业务规则。

这与使用相对一致的方式在许多应用中实现的标准应用逻辑的常见漏洞形成对比。由于这种增加的复杂性，业务逻辑漏洞往往是最难发现和利用的漏洞——但通常也具有最大的影响，如果被利用，会带来最大的回报。

最后需要注意的是，业务逻辑漏洞虽然难以手动发现，但几乎也不可能使用标准的自动化工具找到。这是因为几乎所有设计用于查找漏洞的工具都不是为特定应用和一组业务规则设计的，而是配置为攻击大量使用共享应用框架、语言和系统设计的应用。

闲话少说，让我们评估一些常见的业务逻辑漏洞——并讨论利用它们的方法。

## 自定义数学漏洞

与数学相关的漏洞是最常见的业务逻辑漏洞形式之一。几乎所有应用都代表用户使用数学进行计算，但数学运算背后的逻辑往往不同。

考虑示例应用 MegaBank。在前面的章节中，我们注意到 MegaBank 为其客户托管的 Web 应用允许已认证用户通过向另一用户发送货币来完成交易。

让我们考虑在像 MegaBank 这样的应用中转账是什么样的：

1. 当前已认证用户（用户 A）在用户界面中指定他们希望向用户 B 发送 500 美元。
2. 服务器验证用户 B 存在。
3. 服务器向用户 A 发送确认请求。
4. 用户 A 向服务器发送确认“接受”请求。
5. 服务器通过将数据库中与用户 A 关联的余额字段减少 500 来从用户 A 中扣除。
6. 服务器将 500 添加到数据库中与用户 B 关联的余额字段。
7. 服务器向用户 A 发送交易完成的确认。

现在，这种转账方法在冗余性、故障安全和并发性方面存在许多问题。出于本章的目的，让我们仅评估服务器上为完成此交易而执行的数学运算可能出现什么问题。

首先，虽然服务器检查以确保用户 B 存在，但并未检查用户 A 是否实际拥有转账给用户 B 所需的资金。一个简单的对用户 A 余额的缺失验证（检查是否有足够的金额进行转账）可能会带来多种后果。

在最好的情况下，数据库模式中的验证会取消交易并报错——尽管应用在其代码中缺少适当的业务逻辑检查。在最坏的情况下，用户 A 的资金在 $0 处停止，而用户 B 的资金继续每笔交易增加 $500。这允许用户 B 和用户 A 合作，在银行注意到之前提取数千美元。

上述示例实际上是一个非常简单的业务逻辑漏洞。

如果一个应用未能实现以下两个要求，它将成为黑客利用的绝佳目标：

- 满足其业务逻辑期望的适当数学运算
- 对数学函数输入进行适当的验证

由于这些漏洞依赖于银行关于转账的特定业务模型，因此不太可能被任何静态分析工具发现。

你可以将针对此漏洞的攻击描述为黑客找到一种方法，在遵守编程规则的同时绕过预期功能。

## 编程副作用

大多数应用在编程时都考虑到了主要用例，该用例用于开发应用将运行的业务规则。通常，随着 Web 应用复杂性的增加，应用将开始产生副作用，即由于以开发者未预见的方式使用编程功能而发生的非预期变化。

考虑另一个 Web 应用：MegaCrypto。在这个假设的 Web 应用中，MegaBank 用户能够使用新的 MegaCrypto 门户网站相互购买、出售和转换加密货币。MegaBank 预测这将提高其在更年轻、更懂技术的客户中的受欢迎程度。

与其他交易所一样，MegaBank 知道它需要流动性来支持用户的购买、出售和加密货币转换。它决定最初提供五种主要的加密货币币种。MegaCrypto 出售这些货币的价格将由市场需求决定。当 MegaCrypto 推出时，MegaBank 假设本地市场汇率将反映 MegaCrypto 决定在其平台上提供的加密货币币种的全球市场汇率。在 MegaCrypto 上出售和交换的币种如下：

- MegaCoin
- ByteCoin
- DoggoCoin
- GhostCoin
- HardDriveCoin

因为此应用的预期功能是市场需求决定加密货币的购买或交换汇率，当前汇率如下：

- MegaCoin——$1
- GhostCoin——$2,000
- HardDriveCoin——$10
- ByteCoin——$50,000
- DoggoCoin——$5

每晚午夜，MegaCrypto 通过首先从其用户处购买（如果已提交卖出订单），然后从另一家交易所购买来补充其流动性池。MegaCrypto 的业务逻辑以这种方式运作，首先从其出售的用户处回购加密货币，因为这减少了 MegaCrypto 原本需要支付给另一家交易所的流动性购买费用。

MegaCrypto 的客户 Henry Hacker 能够查看公共记录，并注意到 MegaCrypto 在流动性低时首先从其用户处回购币种。换句话说，他已经逆向工程了 MegaCrypto 用于维护流动性池的业务逻辑。他在侦察日志中记录了这一情况。

Henry Hacker 观察到，新交易所在任何给定时间持有的总币量相当低。换句话说，在满负荷情况下，该交易所的流动性仅占全球加密货币市场的一小部分——实际上，它仅占一组投资者能够汇集资金的一小部分。在任何给定时间，流动性储备中持有的总币量约为 100 万美元。

在执行了几次交易并观察新交易所按市场汇率从现有用户处购买以补充其流动性池后，Henry Hacker 决定请他的富豪朋友筹集一些资金供他投资。他获得了 100 万美元的初始资本。

Henry 看到，随着时间的推移，MegaCrypto 交易所对 ByteCoin 的需求平均为每天 20 笔购买（按每枚 $50,000 的价格计算，总额为 100 万美元）。他注意到每年平均有 20 枚 ByteCoin 从 MegaCrypto 的流动性池流出到其用户的 MegaCrypto 钱包。

同样，每天下午 MegaCrypto 以当地市场汇率在其本地用户池中创建 20 枚 ByteCoin 的买单。通过在夜间以当地汇率从本地用户回购币种，MegaCrypto 最终从交易费中获得可观收益，其用户也很满意。

Henry 决定编写一个算法，每晚自动以 $50,000 的汇率购买 20 枚 ByteCoin，这与全球市场汇率相匹配。之后，他的脚本将立即以每枚 $100,000 的价格将这些 ByteCoin 挂单出售。在将此算法部署到生产环境后，他在前一天仅投资 100 万美元的情况下，醒来时账户中有 200 万美元。

Henry 所做的实际上可能并不违法，但根据 MegaCrypto 的标准，这绝对会被视为一个漏洞。如果不迅速修复，可能每天给 MegaCrypto 造成数百万美元的损失。

这里的问题是 MegaCrypto 的应用逻辑没有考虑到有限的本地流动性池可能不够大，无法模拟全球市场价格。此外，如此小的总流动性池允许非常富有的个人（或在本例中是一个团体）操纵市场上下波动。

MegaCrypto 是一个数字货币交易所，这意味着它所要做的就是在应用中包含逻辑，以防止在没有本地币种以全球市场汇率提供时购买本地币种（并转向其他地方）。

表面上看是经济问题，实际上也是一个业务逻辑漏洞。MegaCrypto 的这些重大损失本可以通过适当的边缘情况检测来避免，该检测允许 MegaCrypto 仅在价格在平均全球市场价格的一定百分比阈值内时才回购本地币种。

这类漏洞需要深入理解复杂系统，但一旦获得这些知识，就可能以高利润且难以检测的方式攻击 Web 应用或一组数字系统。

## 准现金攻击

到目前为止，我们看了因数学运算不当而产生的业务逻辑漏洞，以及因预期功能的副作用未被适当纳入初始应用架构而产生的业务逻辑漏洞。

然而，有时主要功能本身也可能容易受到业务逻辑漏洞的影响，尤其是当主要功能涉及多个协同工作的系统时。在信用卡行业，最常见（且代价高昂）的业务逻辑漏洞形式之一是准现金交易漏洞。

考虑我们示例公司 MegaBank 提供的以下信用卡。这张卡称为 MegaCard，具有以下属性：

- 25% 年利率
- $0 年费
- 所有购买 5% 现金返还（前三年）

这是 MegaBank 向其客户提供的第一张卡。它是内部开发的，而不是 MegaBank 销售由 Visa 或 Mastercard 等主要成熟卡供应商生产的产品。

假设是 MegaBank 从其其他业务部门赚取大量资金，因此在这张卡推出后的前三年内，5% 的年度现金返还将得到适当资金支持，以吸引新客户。假设所有竞争对手都提供较低的现金返还率，因此该利率将推动 MegaCard 的大规模市场普及。

Henry Hacker 看到高现金返还率，决定申请一张 MegaCard。起初他对非常高的现金返还率感到满意，但在被 EvilCorp 解雇后，他急需现金。

在 EvilCorp 工作时，他花了很多时间处理 API，用于将替代数字支付方法集成到 EvilCorp 的业务部门中。其中一个工具称为 PayBuddy。它提供一个在线门户，供应商可以注册、创建发票、接收付款，并将收入直接存入企业银行账户。

PayBuddy 每笔交易仅收取 1% 的费用，这对 Henry 的前雇主 EvilCorp 很有吸引力。在学习如何使用 PayBuddy 的 API 时，Henry Hacker 设置了自己的商家账户，将其连接到他的银行账户，并在下班后用于测试。

在被 EvilCorp 解雇后不久，Henry 想出了一个主意：

1. 使用他在 PayBuddy 上的商家账户，Henry 会创建一张 $1,000 的发票，用于一分钟的咨询。
2. 然后他会用他的 MegaCard 支付 $1,000 的发票。
3. 现在他的 MegaCard 上会有 $1,000 的信用额度，但他的商家账户中会有 $990 的借方（$1,000，扣除 1% 的费用）。
4. 接下来，他将获得 5%（卡片奖励率）的 $1,000 购买金额的返现，即 $50，存入他的个人银行账户。
5. 在第 4 步之后，Henry 会将 $990 从 PayBuddy 转入他的个人银行账户。
6. 最终，Henry 的个人银行账户将包含 $1,040（来自 PayBuddy 的 $990 和来自 MegaCard 的 $50 奖励），比他开始时的 $1,000 多 $40。

通过反复重复这一系列交易，Henry 每个周期将不断为他的个人银行账户增加 $40 的净资产。手动操作这将是一个缓慢而乏味的过程。然而，Henry Hacker 创建了一个脚本，能够每分钟重复上述所有步骤一次。有了这种自动化，他现在每小时赚取 $2,400 的 MegaCard 奖励。在短短 24 小时内，Henry 可以通过此漏洞赚取 $57,600 的奖励——足以支付他一整年的租金。

最后一步，Henry 在 MegaBank 注意到之前兑现了他的余额。MegaBank 只能承担账单，因为从技术上讲，Henry 没有违反服务协议中的任何规则。

在这种情况下，MegaCard 按设计工作。该卡是“安全的”，对购买提供 5% 的现金返还，并且易于使用。MegaCard 的设计没有考虑到的是，MegaCard 客户同时也可以是供应商，从而能够对自己进行快速自动化的购买。

这被称为准现金交易漏洞，因为 PayBuddy 实际上并不是银行账户。相反，它是一个金融中介，允许程序化的商业交易，这些交易最终将以真实货币结算，但在此之前，它们实际上只是数据库中的字段。

如果 MegaCard 独立存在于更传统（非数字）的经济中，Henry Hacker 可能以易于预测的方式进行交易。购买杂货、看电影或在餐厅用餐是 MegaBank 假设 Henry Hacker 会为了使用这张新信用卡而进行的活动。

不幸的是，由于 MegaBank 未在运行 MegaCard 的服务器上的应用逻辑中包含制衡机制，Henry 能够利用此业务逻辑漏洞赚取数万美元。

## 易受攻击的标准与约定

根据您所针对的应用，业务逻辑漏洞可能从容易发现到需要大量领域特定知识才能识别不等。然而，您可以应用一些技巧来使发现这些漏洞更容易一些。这些技巧通常表现为由常见反模式产生的边缘情况，而程序员在不理解其影响的情况下使用了这些反模式。让我们讨论数字精度损失，这是其中一个技巧。

如前所述，业务逻辑漏洞通常涉及某种形式的数学运算，要么输入处理不当，要么结构不正确导致非预期输出。如果代码在您无法访问的服务器上执行，可能难以正确猜测数学函数的结构以及哪些边缘情况已被考虑在内。

幸运的是，由于数学是一个标准化的系统，一些常见错误可能会引导您找到逻辑漏洞。其中一个例子源于理解大多数编程语言如何处理数字。

几乎每种主要编程语言都使用 IEEE754 浮点数格式（JavaScript、Java、Ruby、Python 等）。使用这种数字格式是因为计算机中的内存是（或至少曾经是）稀缺的，导致人们希望使用更有效的方法来存储极长的数字，而不是简单地将每个位存储在内存中。

与其他格式相比，IEEE754 通过使用浮点（可变位置小数与固定位置小数）并依赖科学计数法来表示数字，提供了一种高效的数字存储机制。那么使用 IEEE754 的缺点是什么？

您看，为了实现更节省内存的数字存储，IEEE754 有一个权衡，即牺牲精度以换取空间。这意味着，在任何涉及小数点的计算中，结果值可能被近似，但与真实数学结果不完全相同。

尝试以下步骤：

1. 打开 Chrome Web 浏览器。
2. 导航到开发者工具（通常是 F12 快捷键）。
3. 点击 Console 标签。
4. 对 Chrome 的 v8 JavaScript 解释器执行 JavaScript 代码。
5. 输入以下代码：`0.1 + 0.2`。
6. 点击 Enter。

您会期望在这种情况下 JavaScript 解释器返回正确的数学结果 0.3，但实际返回的是 `0.30000000000000004`。发生这种情况是因为 IEEE754 数字格式无法表示 0.1 + 0.2 的和，因此它以精度损失来近似结果。

在本例中，精度损失为 0.0000000000000004，这是一个极小的数字，在大多数情况下不会导致任何不正确的计算。然而，在处理金融计算时——尤其是快速自动化的计算——这种计算误差可能在短时间内累积成一笔可观的金额。

想象一个金融应用在重新加载时重新计算用户账户余额。如果 0.0000000000000004 的误差直接加到余额上，需要数百万次重新计算才能使真实余额与记录余额显著不同。

另一方面，如果涉及更复杂的计算，其中和相乘或更糟的是求幂，这可能很快失控，导致记录余额与实际余额出现巨大差异。因此，许多业务逻辑漏洞源于不当的数学计算。理解数学运算的常见实现方式将使您在 Web 应用中寻找这些漏洞时具有优势。

总之，虽然业务逻辑漏洞特定于应用的业务逻辑，但大多数应用在实现上有相似之处，可以用于洞察业务逻辑漏洞可能存在于何处。

在本例中，通过了解编程语言默认如何处理浮点数，您获得了一些可以开始手动测试的表面区域。

## 利用业务逻辑漏洞

现在我们已经看到了许多业务逻辑漏洞，显然逻辑漏洞很难具体分类，因为它们与特定应用的特定预期用例紧密相关。

当尝试使用业务逻辑漏洞攻击 Web 应用时，您需要做的第一件事是深入了解该应用的预期用例。使用 MegaCard 示例——将您的思维框架设定为 MegaBank 的典型 MegaCard 客户。您想使用该卡购买杂货、看电影和外出就餐。

写下您能想到的该应用或技术的每一个预期用例。然后，对于每个预期用例，确保详细说明您想象它在后端（应用逻辑中）是如何工作的。

例如，MegaCard 的笔记可能如下所示：

1. 我使用 MegaCard 在 FoodCo 购买杂货。
2. 一旦我刷卡，MegaBank 和 FoodCo 之间会执行某种类型的网络握手。
3. MegaBank 的数据库将相当于我购买的食品价值的信用添加到我的卡上，这将在我还清余额时从我的银行账户中结算。
4. MegaBank 的服务器计算相当于食品购买金额 5% 的金额，并在单独的交易中，将其从我的个人银行账户中扣除。

更复杂的应用功能（如市场操纵示例）将需要更多时间和精力来理解和绘制地图。一旦您绘制了所有预期功能以及您对应用在服务器内部工作原理的最佳猜测，现在您就准备好开始计划攻击了。

对于您记录的每个功能，考虑可能未得到适当处理的边缘情况。这些可能如下所示：

- 如果我以加元购买 $1,000 的杂货，我会获得 5%（$50）的美元返现，还是会进行货币转换？
- 如果我在 PayBuddy 上设置一个在线商家商店，考虑到费用与奖励的差异（1% 对 5%），我能否在零成本交易上净赚 4%？
- 如果我设置一个在线商家账户，我能否执行一张 -$1,000 的负发票，将资金加到我的信用卡余额中而不是减去？
- 如果我购买然后立即退款，我是否仍保留 5% 的奖励？

这些边缘情况是您在考虑通过业务逻辑漏洞利用 MegaCard 时的初始攻击向量。
