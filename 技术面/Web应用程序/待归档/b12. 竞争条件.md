# 第12章：竞争条件（Race Conditions）

竞争条件是现代Web应用程序中最有趣的漏洞之一。它们源于开发人员常犯的简单编程错误，而这些错误已被证明代价高昂：攻击者曾利用竞争条件从网上银行、电商网站、股票经纪公司和加密货币交易所窃取资金。

让我们深入了解这些漏洞发生的原因和方式，以及如何发现和利用它们。

## 机制

当两段本应按顺序执行的代码被无序执行时，就会发生竞争条件。要理解其工作原理，你首先需要理解**并发**的概念。在计算机科学中，并发是指在不影响程序结果的情况下同时执行程序不同部分的能力。并发可以大幅提高程序性能，因为程序操作的不同部分可以同时运行。

并发有两种类型：多处理和多线程。**多处理**是指使用多个中央处理单元（CPU）——计算机中执行指令的硬件——来执行同时计算。而**多线程**是指单个CPU提供多个线程（即并发执行）的能力。这些线程实际上并不是同时执行的；相反，它们轮流使用CPU的计算能力。当一个线程空闲时，其他线程可以继续利用未使用的计算资源。例如，当一个线程在等待用户输入时被挂起，另一个线程可以接管CPU执行其计算。

安排多个线程执行顺序的过程称为**调度**。不同系统根据其性能优先级使用不同的调度算法。例如，某些系统可能通过先执行最高优先级的任务来调度任务，而另一些系统可能轮流分配计算时间，而不考虑优先级。

正是这种灵活的调度导致了竞争条件。当开发人员未遵循某些安全的并发原则时，就会发生竞争条件，我们将在本章后面讨论。由于调度算法可以在两个线程的执行之间随时切换，你无法预测线程执行每个操作的顺序。

为了理解执行顺序为何重要，让我们考虑一个示例（来源：维基百科：https://en.wikipedia.org/wiki/Race_condition）。假设两个并发执行线程各自试图将全局变量的值增加1。如果该变量初始值为0，它最终应变为2。理想情况下，线程应按表12-1所示的阶段执行。

**表12-1：两个线程操作同一变量的正常执行**

| 阶段 | 线程1 | 线程2 | 变量A的值 |
|------|-------|-------|-----------|
| 阶段1 | | | 0 |
| 阶段2 | 读取A的值 | | 0 |
| 阶段3 | A增加1 | | 0 |
| 阶段4 | 写入A的值 | | 1 |
| 阶段5 | | 读取A的值 | 1 |
| 阶段6 | | A增加1 | 1 |
| 阶段7 | | 写入A的值 | 2 |

但如果两个线程同时运行，而不考虑访问相同资源时可能发生的冲突，执行可能被调度为表12-2所示。

**表12-2：竞争条件导致的计算错误**

| 阶段 | 线程1 | 线程2 | 变量A的值 |
|------|-------|-------|-----------|
| 阶段1 | | | 0 |
| 阶段2 | 读取A的值 | | 0 |
| 阶段3 | | 读取A的值 | 0 |
| 阶段4 | A增加1 | | 0 |
| 阶段5 | 写入A的值 | | 1 |
| 阶段6 | | A增加1 | 1 |
| 阶段7 | | 写入A的值 | 1 |

在这种情况下，全局变量的最终值变为1，这是不正确的。结果值应为2。

总之，当一个线程的执行结果取决于另一个线程的结果，并且两个线程在操作相同资源时未考虑其他线程也在使用这些资源时，就会发生竞争条件。当这两个线程同时执行时，可能会产生意外结果。某些编程语言（如C/C++）由于其内存管理方式，更容易出现竞争条件。

### 竞争条件何时成为漏洞

当竞争条件影响安全控制机制时，它就成为一个漏洞。在这些情况下，攻击者可以诱导一种敏感操作在安全检查完成之前执行的情况。因此，竞争条件漏洞也被称为**检查时与使用时**（time-of-check vs time-of-use）漏洞。

想象一下，前面示例中的两个线程正在执行更敏感的操作：银行账户之间的转账。应用程序必须执行三个子任务才能正确转账。首先，它必须检查转出账户是否有足够的余额。然后，必须将钱添加到目标账户。最后，必须从转出账户扣除相同金额。

假设你拥有两个银行账户，账户A和账户B。账户A中有500美元，账户B中有0美元。你同时发起两笔从账户A到账户B的500美元转账。理想情况下，当发起两笔转账请求时，程序应按表12-3所示运行。

**表12-3：两个线程操作同一银行账户的正常执行**

| 阶段 | 线程1 | 线程2 | 账户A+B余额 |
|------|-------|-------|-------------|
| 阶段1 | 检查账户A余额（$500） | | $500 |
| 阶段2 | 向账户B添加$500 | | $1,000（A中$500，B中$500） |
| 阶段3 | 从账户A扣除$500 | | $500（A中$0，B中$500） |
| 阶段4 | | 检查账户A余额（$0） | $500（A中$0，B中$500） |
| 阶段5 | | 转账失败（余额不足） | $500（A中$0，B中$500） |

最终你的钱数正确：两个银行账户中共计500美元。但如果你能同时发送两个请求，你可能能够诱导线程执行呈现表12-4所示的情况。

**表12-4：竞争条件导致的错误转账结果**

| 阶段 | 线程1 | 线程2 | 账户A+B余额 |
|------|-------|-------|-------------|
| 阶段1 | 检查账户A余额（$500） | | $500 |
| 阶段2 | | 检查账户A余额（$500） | $500 |
| 阶段3 | 向账户B添加$500 | | $1,000（A中$500，B中$500） |
| 阶段4 | | 向账户B添加$500 | $1,500（A中$500，B中$1,000） |
| 阶段5 | 从账户A扣除$500 | | $1,000（A中$0，B中$1,000） |
| 阶段6 | | 从账户A扣除$500 | $1,000（A中$0，B中$1,000） |

请注意，在这种情况下，你最终的钱比开始时多。你的账户中不再是500美元，现在总计拥有1,000美元。你通过利用竞争条件漏洞凭空多出了500美元！

虽然竞争条件通常与金融网站相关，但攻击者也可以在其他情况下使用它们，例如操纵在线投票系统。假设一个在线投票系统执行三个子任务来处理在线投票。首先，它检查用户是否已经投票。然后，它将一票添加到所选候选人的票数中。最后，它记录该用户已投票，以防止他们再次投票。

假设你尝试同时为候选人A投票两次。理想情况下，应用程序应拒绝第二次投票，遵循表12-5中的过程。

**表12-5：两个线程操作同一用户投票的正常执行**

| 阶段 | 线程1 | 线程2 | 候选人A票数 |
|------|-------|-------|-------------|
| 阶段1 | | | 100 |
| 阶段2 | 检查用户是否已投票（未投） | | 100 |
| 阶段3 | 增加候选人A票数 | | 101 |
| 阶段4 | 标记用户已投票 | | 101 |
| 阶段5 | | 检查用户是否已投票（已投） | 101 |
| 阶段6 | | 拒绝用户投票 | 101 |

但如果投票应用程序存在竞争条件漏洞，执行可能变成表12-6所示的场景，这赋予用户可能无限投票的能力。

**表12-6：用户通过滥用竞争条件能够投两次票**

| 阶段 | 线程1 | 线程2 | 候选人A票数 |
|------|-------|-------|-------------|
| 阶段1 | | | 100 |
| 阶段2 | 检查用户是否已投票（未投） | | 100 |
| 阶段3 | | 检查用户是否已投票（未投） | 100 |
| 阶段4 | 增加候选人A票数 | | 101 |
| 阶段5 | | 增加候选人A票数 | 102 |
| 阶段6 | 标记用户已投票 | | 102 |
| 阶段7 | | 标记用户已投票 | 102 |

攻击者可以遵循此过程一次性发送两个、十个甚至数百个请求，然后观察在用户被标记为已投票之前哪些投票请求被处理。

大多数竞争条件漏洞被用于操纵金钱、礼品卡积分、投票、社交媒体点赞等。但竞争条件也可以用于绕过访问控制或触发其他漏洞。你可以在HackerOne Hacktivity feed上阅读一些真实的竞争条件漏洞：https://hackerone.com/hacktivity?querystring=race%20condition/。

## 防御措施

防止竞争条件的关键是在执行期间通过使用**同步**方法来保护资源，即确保使用相同资源的线程不会同时执行的机制。

**资源锁**是这些机制之一。它们通过锁定资源来阻止其他线程操作同一资源。在银行转账示例中，线程1可以在修改账户A和B的余额之前锁定它们，以便线程2必须等待其完成才能访问资源。

大多数具有并发能力的编程语言都内置了某种同步功能。你必须意识到应用程序中的并发问题并相应地应用同步措施。除同步外，遵循安全编码实践（如最小权限原则）可以防止竞争条件演变为更严重的安全问题。

**最小权限原则**是指应用程序和进程应仅被授予完成其任务所需的权限。例如，当应用程序仅需要读取文件时，不应授予任何写入或执行权限。你应精确授予应用程序所需的权限，而不是更多。这降低了攻击期间系统完全受损的风险。

## 寻找竞争条件

寻找竞争条件很简单，但通常涉及运气成分。通过遵循以下步骤，你可以确保最大化成功的机会。

### 第1步：寻找容易出现竞争条件的功能

攻击者使用竞争条件来颠覆访问控制。理论上，任何敏感操作依赖于访问控制机制的应用程序都可能存在漏洞。

大多数情况下，竞争条件出现在处理数字的功能中，如在线投票、在线游戏分数、银行转账、电子商务支付和礼品卡余额。在应用程序中寻找这些功能，并记录更新这些数字所涉及的请求。

例如，假设在你的代理中，你发现了用于从银行网站转账的请求。你应该复制此请求用于测试。在Burp Suite中，你可以通过右键点击请求并选择 `Copy as curl command` 来复制请求。

### 第2步：发送同时请求

然后，你可以通过同时向服务器发送多个请求来测试和利用目标中的竞争条件。

例如，如果你的银行账户中有3,000美元，想看看是否能转出超过你拥有的金额，你可以通过 `curl` 命令同时向服务器发送多个转账请求。如果你从Burp复制了命令，可以简单地将命令多次粘贴到终端中，并在每个之间插入 `&` 字符。在Linux终端中，`&` 字符用于在后台同时执行多个命令：

```bash
curl (transfer $3000) & curl (transfer $3000) & curl (transfer $3000)
& curl (transfer $3000) & curl (transfer $3000) & curl (transfer $3000)
```

请务必测试应被允许一次但不允许多次的操作！例如，如果你的银行账户余额为3,000美元，测试转账5,000美元是没有意义的，因为任何单个请求都不会被允许。但测试多次转账10美元也没有意义，因为即使没有竞争条件你也应该能做到这一点。关键在于通过执行不应重复的操作来测试应用程序的极限。

### 第3步：检查结果

检查你的攻击是否成功。在我们的示例中，如果同时请求后你的目标账户最终多出的金额超过3,000美元，则攻击成功，你可以确定转账余额端点存在竞争条件。

请注意，攻击是否成功取决于服务器的进程调度算法，这取决于运气。但你在短时间内发送的请求越多，攻击成功的可能性就越大。此外，许多竞争条件测试第一次不会成功，因此在放弃之前多尝试几次是个好主意。

### 第4步：创建概念验证

一旦发现竞争条件，你需要在报告中提供漏洞证明。最好的方法是列出利用漏洞所需的步骤。例如，你可以如下列出利用步骤：

1. **创建一个余额为3,000美元的账户**和另一个余额为零的账户。余额为3,000美元的账户将是转账的源账户，余额为零的账户将是目标账户。
2. **执行此命令**：
   ```bash
   curl (transfer $3000) & curl (transfer $3000) & curl (transfer $3000)
   & curl (transfer $3000) & curl (transfer $3000) & curl (transfer $3000)
   ```
   这将尝试多次同时向另一账户转账3,000美元。
3. **你应该在目标账户中看到超过3,000美元**。如果未看到超过3,000美元，请撤销转账并再尝试几次攻击。

由于竞争条件攻击的成功取决于运气，请确保包含如果第一次测试失败则重试的说明。如果漏洞存在，经过几次尝试后攻击最终应会成功。

## 提升竞争条件影响

竞争条件的严重性取决于受影响的功能。在确定特定竞争条件的影响时，注意攻击者在金钱奖励或社会影响力方面可能获得多少。

例如，如果在取款、资金转账或信用卡支付等关键功能上发现竞争条件，该漏洞可能导致攻击者无限获利。在报告中证明竞争条件的影响并阐明攻击者将能够实现什么。

## 找到你的第一个竞争条件！

现在你已准备好找到第一个竞争条件了。按照以下步骤使用这种巧妙的技术操纵Web应用程序：

1. **识别目标应用程序中容易出现竞争条件的功能**，并复制相应的请求。
2. **同时向服务器发送多个这些关键请求**。你应该构造应被允许一次但不允许多次的请求。
3. **检查结果**，看攻击是否成功。并尝试多次执行攻击以最大化成功机会。
4. **考虑你刚发现的竞争条件的影响**。
5. **起草你的第一份竞争条件报告！**
