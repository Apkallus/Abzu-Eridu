# 测试网络基础设施配置

## 概要

互联且异构的 Web 服务器基础设施（可能包含数百个 Web 应用程序）的内在复杂性，使得配置管理和审查成为测试和部署每个应用程序的基本步骤。只需一个漏洞就可能破坏整个基础设施的安全，甚至微小且看似无关紧要的问题也可能演变为同一服务器上另一个应用程序的严重风险。为了解决这些问题，在映射整个架构之后，对配置和已知安全问题执行深入审查至关重要。

Web 服务器基础设施的适当配置管理对于维护应用程序本身的安全性非常重要。如果诸如 Web 服务器软件、后端数据库服务器或身份验证服务器等元素未得到适当审查和保护，它们可能会引入不必要的风险或新的漏洞，从而危及应用程序本身。

例如，一个允许远程攻击者泄露应用程序源代码的 Web 服务器漏洞（在 Web 服务器和应用服务器中都曾多次出现）可能会危及应用程序，因为匿名用户可以利用源代码中披露的信息对应用程序或其用户发起攻击。

需要采取以下步骤来测试配置管理基础设施：

- 需要确定构成基础设施的不同元素，以便理解它们如何与 Web 应用程序交互以及如何影响其安全性。
- 需要审查基础设施的所有元素，以确保它们不包含任何已知漏洞。
- 需要审查用于维护所有不同元素的管理工具。
- 需要审查身份验证系统，以确保它们满足应用程序的需求，并且不能被外部用户操纵以获取访问权限。
- 应维护应用程序所需的已定义端口列表，并将其置于变更控制之下。

在映射了构成基础设施的不同元素（参见 映射网络和应用程序架构）之后，可以审查每个已发现元素的配置，并测试是否存在任何已知漏洞。

## 测试目标

- 审查部署在网络中的应用程序配置，并验证它们不易受攻击。
- 验证所使用的框架和系统是安全的，并且不会因未维护的软件或默认设置和凭据而容易受到已知漏洞的影响。

## 如何测试

### 已知的服务器漏洞

应用程序架构各个区域（无论是 Web 服务器还是后端数据库）中的漏洞都可能严重危及应用程序。例如，考虑一个允许远程未经身份验证的用户将文件上传到 Web 服务器甚至替换现有文件的服务器漏洞。该漏洞可能危及应用程序，因为恶意用户可能能够替换应用程序本身或引入影响后端服务器的代码，因为其应用程序代码将像任何其他应用程序一样运行。

如果测试需要通过盲渗透测试进行，则审查服务器漏洞可能很难做到。在这种情况下，通常需要使用自动化工具从远程站点测试漏洞。然而，测试某些漏洞可能对 Web 服务器产生不可预测的结果，而测试其他漏洞（如直接涉及拒绝服务攻击的漏洞）可能无法进行，因为如果测试成功，将导致服务停机。

一些自动化工具会根据它们获取的 Web 服务器版本来标记漏洞。这会导致误报和漏报。一方面，如果 Web 服务器版本已被本地站点管理员删除或混淆，扫描工具将不会将该服务器标记为易受攻击，即使它确实是。另一方面，如果提供软件的供应商在修复漏洞时不更新 Web 服务器版本，扫描工具将标记不存在的漏洞。后一种情况实际上非常常见，因为某些操作系统供应商将安全漏洞的补丁向后移植到它们在操作系统中提供的软件中，但不会完全更新到最新的软件版本。这种情况发生在大多数 GNU/Linux 发行版中，如 Debian、Red Hat 和 SuSE。在大多数情况下，应用程序架构的漏洞扫描只会发现与架构“暴露”元素（如 Web 服务器）相关的漏洞，并且通常无法发现与未直接暴露的元素（如身份验证后端、后端数据库或正在使用的反向代理 [1]）相关的漏洞。

最后，并非所有软件供应商都公开披露漏洞，这意味着这些弱点可能不会记录在已知漏洞数据库 [2] 中。这些信息仅向客户披露，或通过没有附带公告的修复程序发布。这降低了漏洞扫描工具的有效性。通常，这些工具对常见产品（如 Apache Web 服务器、Microsoft IIS 或 IBM 的 Lotus Domino）的漏洞覆盖率会非常好，但对知名度较低的产品则缺乏覆盖。

这就是为什么当测试人员获得有关软件的内部信息（包括版本、发行版和应用的补丁）时，审查漏洞是最好的做法。有了这些信息，测试人员可以从供应商处检索数据，并分析架构中的潜在漏洞及其对应用程序的潜在影响。在可能的情况下，可以测试这些漏洞以确定其实际影响，并检测是否存在任何可能降低或否定成功利用可能性的外部元素（如入侵检测或防御系统）。测试人员甚至可以通过配置审查确定该漏洞实际上不存在，因为它影响的软件组件未被使用。

供应商为软件提供生命周期支持，若在不受支持的旧软件版本中出现漏洞，系统人员可能不会直接意识到它。并且公告可能不会将该版本列为易受攻击，因为它已不再受支持。即使他们意识到漏洞和相关系统风险，也由于各种因素不升级到新的软件版本：会给应用程序架构带来显著的停机时间，或者由于与最新软件版本不兼容而需要重新编码应用程序。

### 管理工具

任何 Web 服务器基础设施都需要存在管理工具来维护和更新应用程序使用的信息。这些信息包括静态内容（网页、图形文件）、应用程序源代码、用户身份验证数据库等。例如，一些 Web 服务器将使用本身是 Web 服务器的管理界面（如 iPlanet Web 服务器）进行管理，或者通过纯文本配置文件（如 Apache 的情况 [3]）进行管理，或者使用操作系统 GUI 工具（如使用 Microsoft 的 IIS 服务器或 ASP.Net 时）。

在大多数情况下，服务器配置通过各种文件维护工具进行管理，通过 FTP 服务器、WebDAV、网络文件系统（NFS、CIFS）或其他机制进行管理。显然，构成应用程序架构的元素的操作系统也将使用其他工具进行管理。应用程序还可能包含用于管理应用程序数据（用户、内容等）的嵌入式管理界面。

在映射了用于管理架构不同部分的管理界面之后，对其进行审查，重要的是：

- 确定控制对这些界面访问的机制及其相关的脆弱性。这些信息可能在线获得。
- 确保默认用户名和密码已更改。

一些公司外包部分内容，如内容管理到其他方。这个外部公司可能只提供内容的某些部分（如新闻更新或促销），或者它可能完全管理 Web 服务器（包括内容和代码）。在这些情况下，通常会发现可从互联网访问的管理界面，因为使用互联网比提供一条专用线路（通过仅管理接口将外部公司连接到应用程序基础设施）更便宜。在这种情况下，测试管理界面是否易受攻击至关重要。
