# API 测试概述

## Web API 简介

Web 应用程序编程接口（API）促进不同软件系统之间通过网络或互联网进行通信和数据交换。Web API 使不同应用程序能够以标准化和高效的方式相互交互，使它们能够利用彼此的功能和数据。

云计算、微服务架构和单页应用程序等不同技术的采用，都促进了 API 作为一种架构趋势的普及。

随着任何新概念的引入，都可能存在需要测试的缺陷和漏洞。否则，安全性不佳的 API 可能为敏感数据提供不受限制的直接访问路径。

本章试图引导安全研究人员了解测试 API 所需的概念。本节特别研究了不同的 API 技术及其发展历史。

## 使用哪种 API 技术？

在我们就所测试 API 的类型做出假设之前，了解安全研究人员可能遇到的整个问题范围的广度是很有帮助的。这些包括：

1. 表述性状态转移（REST）API
2. 简单对象访问协议（SOAP）API
3. GraphQL API
4. gRPC 远程过程调用（gRPC）
5. WebSockets API

## REST（表述性状态转移）API

### 什么是 REST？

REST 是一套用于与 Web 资源交互的规则和约定。URI、HTTP 方法、头部和状态码等关键组件支持 REST 的原则。

### 历史

由于其简单性、可扩展性以及与现有 Web 基础设施的兼容性，基于 REST 的 API 在撰写本文时已成为互联网上最常见的 API 架构。基于 REST 的 API 并非立即出现，而是经历了一条从研究到采用的漫长道路。

1994 年，HTTP 规范的主要作者之一 Roy Fielding 在加州大学欧文分校开始了他关于 REST 的博士论文工作。到 2000 年，他发表了论文[《架构风格与基于网络的软件架构设计》](https://ics.uci.edu/~fielding/pubs/dissertation/top.htm)，在其中将 REST 作为一种架构风格引入并定义。REST 旨在利用 HTTP 的现有功能，强调可扩展性、无状态交互和统一接口。

在 2010 年代，REST 因其简单性和与 Web 底层架构的兼容性而成为 Web API 的事实标准。RESTful API 的广泛使用是由移动应用程序、云计算和微服务架构的增长推动的。Swagger/OpenAPI、RAML 和 API Blueprint 等工具和框架的开发促进了 REST API 的设计、文档化和测试。

到 2020 年代，现代发展通过 GraphQL 等技术演进 REST。此外，OpenAPI/Swagger 规范成为描述 REST API 的广泛采用的标准，实现了更好的集成和自动化。

### 统一资源标识符

REST API 使用统一资源标识符（URI）来访问资源。URI 是 REST 架构的关键要素。URI 是一个字符串，用于唯一标识特定资源。URI 在互联网上广泛用于定位和交互资源，如网页、文件和服务。

URI 由多个组件组成，每个组件都有特定的用途。[RFC3986](https://tools.ietf.org/html/rfc3986) 中定义的通用 URI 语法如下：

> `URI = scheme "://" authority "/" path [ "?" query ] [ "#" fragment ]`

对于 REST，**scheme（方案）** 通常为 `HTTP` 或 `HTTPS`，但一般表示用于访问资源的协议或方法。其他常见方案包括 `ftp`、`mailto` 和 `file`。

**authority（权限/主机）** 指定资源所在服务器的域名或 IP 地址，并且可能包括端口号。它还可能包括用户信息作为子组件。

**path（路径）** 指定资源在服务器上的具体位置。我们关注 URI 的路径，因为它是用户与资源之间的关系。例如，`https://api.example.com/admin/testing/report` 可能显示一个测试报告。管理员用户与其报告之间存在关系。

任何 URI 的路径都将定义一个 REST API 资源模型。资源由正斜杠分隔，并基于自上而下的设计。

例如：

- `https://api.example.com/admin/testing/report`
- `https://api.example.com/admin/testing/`
- `https://api.example.com/admin/`

**query（查询）** 为资源提供附加参数。它以 `?` 开头，由 `&` 分隔的键值对组成。

**fragment（片段）** 指示资源的特定部分，例如网页中的一个章节。它以 `#` 开头。值得注意的是，片段标识符仅在客户端处理，不会发送到服务器。

### HTTP 方法

REST API 使用标准 HTTP 方法对资源执行操作，遵循 [RFC7231](https://tools.ietf.org/html/rfc7231) 中定义的 [HTTP 请求方法](https://tools.ietf.org/html/rfc7231#section-4)。这些方法映射到 CRUD，即计算机科学中持久存储的四个基本函数。CRUD 代表创建（Create）、读取（Read）、更新（Update）和删除（Delete），这是可以对数据执行的四种操作。

HTTP 请求方法为：

| 方法   | 描述                                   |
|---------|-----------------------------------------------|
| GET     | 获取资源状态的表示形式    |
| POST    | 创建新资源                         |
| PUT     | 更新资源                             |
| DELETE  | 移除资源                             |
| HEAD    | 获取与资源状态相关的元数据 |
| OPTIONS | 列出可用方法                        |

#### 头部

REST 依赖头部来支持在请求或响应中传递附加信息。这些包括：

- `Content-Type`：指示资源的媒体类型（例如 `application/json`）。
- `Authorization`：包含认证凭据（例如令牌）。
- `Accept`：指定响应可接受的媒体类型。

#### 状态码

符合 REST 原则的应用程序 API 使用 HTTP 响应消息的响应状态码来通知客户端其请求的结果。

| 响应码 | 响应消息      | 描述   |
|---------------|-----------------------|--------------------------------------------------------------------------------------------------------|
| 200           | OK                    | 处理客户端请求成功                                                              |
| 201           | Created               | 新资源已创建                                                                                   |
| 301           | Moved Permanently     | 永久重定向                                                                                  |
| 304           | Not Modified          | 与缓存相关的响应，当客户端拥有与服务器相同的资源副本时返回 |
| 307           | Temporary Redirect    | 资源的临时重定向                                                                      |
| 400           | Bad Request           | 客户端请求格式错误                                                                        |
| 401           | Unauthorized          | 客户端不允许发出请求或访问特定资源                                 |
| 403           | Forbidden             | 禁止客户端访问该资源                                                             |
| 404           | Not Found             | 资源不存在或根据请求不正确                                               |
| 405           | Method Not Allowed    | 使用了无效方法或未知方法                                                                  |
| 500           | Internal Server Error | 服务器因内部错误无法处理请求                                              |
