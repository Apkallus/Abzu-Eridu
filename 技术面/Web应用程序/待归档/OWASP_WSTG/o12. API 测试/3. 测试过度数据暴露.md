# 测试过度数据暴露

|ID          |
|------------|
|WSTG-APIT-03|

## 概述

当 API 在其响应中返回的信息超出客户端显示或运行所需的信息时，就会发生过度数据暴露。这通常发生在开发人员将整个后端对象（如数据库模型或内部数据结构）直接序列化为 API 响应时，依赖于客户端代码在向用户呈现之前过滤掉敏感字段。

问题是 API 响应直接对任何能够发出请求的人可访问，无论用户界面选择显示什么。检查原始 API 流量的攻击者可以观察所有返回的字段，包括 UI 有意隐藏的字段。这可能暴露密码、认证令牌、内部标识符、个人身份信息（PII）、财务数据、基础设施细节或内部业务逻辑。

此漏洞映射到 [OWASP API Security Top 10 API3:2023 损坏的对象属性级授权](https://owasp.org/API-Security/editions/2023/en/0xa3-broken-object-property-level-authorization/)。

## 测试目标

- 识别包含超出客户端应用程序显示或所需数据的 API 响应。
- 检测 API 响应中返回的敏感字段，如密码哈希、认证令牌、内部对象 ID、PII 或基础设施细节。
- 确定 API 是否依赖客户端过滤而非服务器端字段选择来控制数据暴露。

## 如何测试

### 将 API 响应与显示数据进行比较

最直接的测试是比较用户界面显示的内容与 API 实际返回的内容。

1. 在浏览器中打开应用程序，导航到显示用户或对象数据的页面（例如个人资料页面、订单摘要或仪表板）。
2. 打开浏览器的开发者工具（网络选项卡）或配置拦截代理（如 Burp Suite 或 ZAP）。
3. 识别填充页面内容的 API 请求。
4. 将 UI 中显示的字段与 API 响应中的完整字段集进行比较。

例如，用户资料页面可能仅显示姓名和电子邮件，但底层 API 响应可能返回：

```json
{
  "id": 12045,
  "name": "Alice Johnson",
  "email": "alice@example.com",
  "password_hash": "$2b$12$LJ3m4ys5qGn...",
  "role": "admin",
  "internal_account_id": "ACC-2024-88421",
  "ssn": "123-45-6789",
  "api_key": "REDACTED_EXAMPLE_KEY_12345",
  "created_by": "system_migration_v2",
  "last_login_ip": "10.0.3.47"
}
```

在这种情况下，API 返回了密码哈希、社会安全号码、API 密钥、内部标识符和内部 IP 地址，而这些 UI 从未显示。

### 检查不同 API 端点的响应

不要将测试限制在单个端点上。检查多种端点类型的响应：

- **用户和账户端点：** `GET /api/users/{id}`、`GET /api/me`、`GET /api/account`
- **列表和搜索端点：** `GET /api/users`、`GET /api/orders?status=completed`（列表端点经常为列表中的每个项目返回相同的完整对象，从而放大了暴露）
- **相关对象端点：** `GET /api/orders/{id}` 可能将完整的用户对象、支付详情或送货地址嵌入到订单响应中
- **错误响应：** 失败的请求可能返回包含堆栈跟踪、内部文件路径、数据库查询详情或配置值的详细错误消息
- **管理或内部端点：** 检查本应用于内部使用的端点是否可从外部访问并返回提升的数据

### 使用不同的用户权限级别进行测试

同一端点可能根据请求用户的角色返回不同数量的数据。测试：

- 低权限用户在请求同一资源时是否收到与管理员相同的字段。
- 未认证请求是否返回本应需要认证的数据。
- 请求其他用户的数据（IDOR）是否返回与请求您自己的数据相同的丰富对象。

例如，`GET /api/users/124` 可能返回包括敏感字段的完整对象，无论请求者是用户 124、其他用户还是未认证用户。

### 分析嵌套和嵌入对象

现代 API 经常在响应中嵌入相关对象。单个 API 调用可能返回深度嵌套的结构，暴露来自多个后端系统的数据。

示例：一个订单端点返回嵌入的客户对象、支付对象和发货对象：

```json
{
  "order_id": "ORD-9923",
  "items": [...],
  "customer": {
    "id": 5001,
    "name": "Bob Smith",
    "email": "bob@example.com",
    "phone": "+1-555-0199",
    "date_of_birth": "1985-03-14",
    "loyalty_tier": "gold"
  },
  "payment": {
    "method": "card",
    "card_last_four": "4242",
    "card_brand": "Visa",
    "card_fingerprint": "fp_abc123xyz",
    "billing_address": {...}
  },
  "internal_notes": "Customer flagged for manual review - suspected fraud"
}
```

应审查每个嵌套对象是否存在超出消费客户端需求的字段。

### 检查 GraphQL 响应中的敏感数据

GraphQL API 呈现了此问题的特定形式。虽然 GraphQL 允许客户端选择他们想要的字段，但模式可能暴露默认 UI 查询未请求但攻击者可以添加到自定义查询中的敏感字段。

1. 如果启用了内省，检索模式并审查每个类型上的可用字段（请参阅[测试 GraphQL](99-Testing_GraphQL.md)）：

    ```graphql
    {
      __type(name: "User") {
        fields {
          name
          type { name }
        }
      }
    }
    ```

2. 查找诸如 `passwordHash`、`apiKey`、`ssn`、`internalId`、`role` 或 `isAdmin` 等可查询但前端未使用的字段。
3. 尝试直接查询这些字段：

    ```graphql
    {
      user(id: "123") {
        name
        email
        passwordHash
        apiKey
        role
      }
    }
    ```

### 检查 JavaScript 源映射和客户端捆绑包

前端 JavaScript 捆绑包和源映射可能无意中揭示预期的 API 响应结构，包括 UI 有意隐藏的字段。如果源映射可公开访问（例如在 `main.js.map` 处），它们可能暴露：

- 前端期望从 API 响应中获取的完整字段列表
- 内部 API 端点路径
- 认证和授权流程逻辑
- 硬编码的 API 密钥、令牌或配置值

测试源映射可用性：

```http
GET /static/js/main.js.map
GET /assets/app.js.map
```

如果源映射可访问且包含 `sourcesContent`，则可以重建原始源代码并分析预期的 API 响应结构和隐藏字段。

### 检查错误响应和调试输出

API 可能通过错误响应泄露敏感信息，特别是在开发或调试模式下运行时：

- **堆栈跟踪** 揭示内部文件路径、框架版本和代码结构
- **数据库错误消息** 暴露表名、列名或查询语法
- **详细的验证错误** 列出所有预期字段，包括公共 API 中未记录的字段
- **调试头部** 如 `X-Debug-Token`、`X-Powered-By` 或包含内部信息的自定义头部

通过发送格式错误的输入、无效认证或请求不存在的资源来触发错误响应，并检查完整的响应（包括头部）。

## 修复

- **实施服务器端响应过滤。** 绝不在 API 响应中返回原始数据库对象或完整的模型序列化。明确定义响应模式，为每个端点和上下文仅包含客户端所需的字段。
- **使用数据传输对象（DTO）或序列化白名单。** 将内部对象映射到特定于用途的响应对象，这些对象仅包含允许的字段。大多数 Web 框架为此提供了机制（例如 Django REST Framework 中的序列化器、Spring 中的 JsonView 或 Rails 中的 ActiveModel::Serializers）。
- **应用字段级访问控制。** 某些字段（如电子邮件地址或电话号码）可能对某个消费者合适，但对另一个不合适。根据请求用户的角色及其与所请求资源的关系，实施包含或排除字段的逻辑。
- **避免暴露内部标识符。** 使用面向外部的标识符（如 UUID 或不透明令牌），而不是顺序的内部数据库 ID，后者会揭示记录计数并允许枚举。
- **限制 GraphQL 模式。** 完全从 GraphQL 模式中移除敏感字段，而不是依赖解析器级别的检查。如果某个字段必须存在于模式中供内部使用，请应用授权指令以防止未经授权的访问。
- **在生产中禁用源映射。** 不要将 JavaScript 源映射部署到生产环境。配置构建管道以从生产工件中排除 `.map` 文件。
- **清理错误响应。** 在生产中返回通用错误消息。在服务器端记录详细的错误信息，但不要将堆栈跟踪、内部路径或查询详情包含在面向客户端的响应中。

## 工具

- **Burp Suite：** 使用 Comparer 工具将显示的 UI 内容与原始 API 响应进行差异比较。Repeater 和 Intruder 工具可用于系统地使用不同参数和认证上下文测试端点。
- **ZAP（Zed Attack Proxy）：** 拦截和检查 API 响应，使用主动扫描规则识别信息泄露，并使用模糊测试功能触发详细错误响应。
- **Postman：** 使用不同的认证令牌发送 API 请求，并比较响应正文以识别依赖于角色的数据暴露差异。
- **浏览器开发者工具：** 网络选项卡提供对 API 响应的即时可见性，无需设置代理。适用于快速比较 UI 渲染数据与原始响应内容。
- **jq：** 命令行 JSON 处理器，用于提取和比较 API 响应中的字段：`curl -s https://api.example.com/users/me | jq 'keys'` 列出所有返回的字段。
