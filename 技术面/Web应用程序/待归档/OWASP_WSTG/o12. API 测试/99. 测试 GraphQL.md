# 测试 GraphQL

|ID          |
|------------|
|WSTG-APIT-99|

## 概述

GraphQL 在现代 API 中已变得非常流行。它提供了简单性和嵌套对象，从而加快了开发速度。虽然每种技术都有其优势，但它也可能使应用程序暴露于新的攻击面。本场景的目的是提供针对使用 GraphQL 的应用程序的一些常见错误配置和攻击向量。某些向量是 GraphQL 特有的（例如[内省查询](#introspection-queries)），有些则是 API 通用的（例如[SQL 注入](#sql-injection)）。

本节中的示例将基于一个易受攻击的 GraphQL 应用程序 [poc-graphql](https://github.com/righettod/poc-graphql)，它在 Docker 容器中运行，映射 `localhost:8080/GraphQL` 作为易受攻击的 GraphQL 节点。

## 测试目标

- 评估是否部署了安全且可用于生产的配置。
- 针对通用攻击验证所有输入字段。
- 确保应用了适当的访问控制。

## 如何测试

测试 GraphQL 节点与测试其他 API 技术并无太大不同。请考虑以下步骤：

### 内省查询

内省查询是 GraphQL 让你了解支持哪些查询、哪些数据类型可用以及许多其他详细信息的方法，这些信息在你测试 GraphQL 部署时将会用到。

[GraphQL 网站描述了内省](https://graphql.org/learn/introspection/)：

> “向 GraphQL 模式询问它支持哪些查询的信息通常很有用。GraphQL 允许我们通过内省系统来实现这一点！”

有几种方法可以提取这些信息并可视化输出，如下所述。

#### 使用原生 GraphQL 内省

最直接的方法是使用个人代理发送带有以下载荷的 HTTP 请求，该载荷取自 [Medium](https://medium.com/@the.bilal.rizwan/graphql-common-vulnerabilities-how-to-exploit-them-464f9fdce696) 上的一篇文章：

```graphql
        ...InputValue
      }
    }
  }
}

  kind
  name
  description

    name
    description

    }

    }

  }
  inputFields {
    ...InputValue
  }
  interfaces {
    ...TypeRef
  }
  enumValues(includeDeprecated: true) {
    name
    description
    isDeprecated
    deprecationReason
  }
  possibleTypes {
    ...TypeRef
  }
}
fragment InputValue on __InputValue {
  name
  description
  type {
    ...TypeRef
  }
  defaultValue
}
fragment TypeRef on __Type {
  kind
  name
  ofType {
    kind
    name
    ofType {
      kind
      name
      ofType {
        kind
        name
        ofType {
          kind
          name
          ofType {
            kind
            name
            ofType {
              kind
              name
              ofType {
                kind
                name
              }
            }
          }
        }
      }
    }
  }
}
```

#### 使用 GraphiQL

[GraphiQL](https://github.com/graphql/graphiql) 是一个基于 Web 的 GraphQL IDE。它是 GraphQL 项目的一部分，主要用于调试或开发目的。最佳实践是不允许用户在生成部署中访问它。如果你正在测试临时环境，可能可以访问它，从而在使用内省查询时节省一些时间（尽管你当然可以在 GraphiQL 界面中使用内省功能）。

GraphiQL 有一个文档部分，它使用来自模式的数据来创建所使用的 GraphQL 实例的文档。此文档包含数据类型、变更以及基本上可以通过内省提取的每条信息。

#### 使用 GraphQL Playground

[GraphQL Playground](https://github.com/graphql/graphql-playground) 是一个 GraphQL 客户端。它可用于测试不同的查询，以及将 GraphQL IDE 划分为不同的 Playground，并按主题或通过为它们分配名称来分组。与 GraphiQL 非常相似，Playground 可以为你创建文档，而无需手动发送内省查询和处理响应。它还有另一个巨大的优势：它不需要 GraphiQL 界面可用。你可以通过 URL 将工具指向 GraphQL 节点，或使用数据文件在本地使用它。GraphQL Playground 可用于直接测试漏洞，因此你无需使用个人代理来发送 HTTP 请求。这意味着你可以使用此工具与 GraphQL 进行简单的交互和评估。对于其他更高级的载荷，请使用个人代理。

请注意，在某些情况下，你需要在底部设置 HTTP 头部，以包含会话 ID 或其他认证机制。这仍然允许创建具有不同权限的多个“IDE”，以验证是否确实存在授权问题。

#### 通过字段建议导致的模式泄露

许多 GraphQL 服务器实现（包括 Apollo Server 和 `graphql-js`）在客户端引用不存在的字段名时，默认返回“您的意思是...？”的建议。此行为有助于开发，但即使已明确禁用了内省，也会泄露模式信息。

发送一个稍微拼写错误的字段名来测试此问题。如果服务器在错误响应中返回建议，则该响应泄露了真实字段名，而无需发出内省查询。

例如，发送一个包含不存在字段 `userss` 的请求：

```graphql
query {
  userss {
    id
  }
}
```

易受攻击的服务器在其响应中揭示了真实字段名：

```json
{
  "errors": [
    {
      "message": "无法在类型 \"Query\" 上查询字段 \"userss\"。您是否指的是 \"users\"？",
      "locations": [{ "line": 2, "column": 3 }]
    }
  ]
}
```

遍历猜测名称的变体（例如 `userss`、`usesr`、`usres`）以逐个建议枚举有效的查询和变更名称。这样做可以在没有内省访问权限的情况下重建模式的很大一部分。

[Clairvoyance](https://github.com/nikitastupin/clairvoyance) 通过将单词列表与建议响应相结合，逐步构建模式，从而自动化此过程。

#### 内省结论

内省是一种有用的工具，允许用户获取有关 GraphQL 部署的更多信息。然而，这也将允许恶意用户访问相同的信息。最佳实践是限制对内省查询的访问，因为如果完全禁用此功能，某些工具或请求可能会失败。由于 GraphQL 通常连接到系统的后端 API，因此最好强制执行严格的访问控制。

仅禁用内省是不够的。在生产中也要禁用字段建议，因为它们独立地允许模式枚举。

### 授权

内省是寻找授权问题的第一个地方。如前所述，应限制对内的省访问，因为它允许数据提取和数据收集。一旦测试人员可以访问模式并了解要提取的敏感信息，他们就应该发送查询，这些查询不会因权限不足而被阻止。GraphQL 默认不强制执行权限，因此由应用程序来执行授权强制。

在之前的示例中，内省查询的输出显示有一个名为 `auth` 的查询。这似乎是提取敏感信息（如 API 令牌、密码等）的好地方。

### 注入

GraphQL 是应用程序 API 层的实现，因此它通常将请求转发到后端 API 或直接转发到数据库。这允许你利用任何底层漏洞，如 SQL 注入、命令注入、跨站脚本等。使用 GraphQL 只是改变了恶意载荷的入口点。

你可以参考 OWASP 测试指南中的其他场景来获取一些想法。

GraphQL 还有标量（scalars），通常用于没有原生数据类型的自定义数据类型，如 DateTime。这些类型的数据没有现成的验证，使它们成为测试的良好候选对象。

#### SQL 注入

示例应用程序在查询 `dogs(namePrefix: String, limit: Int = 500): [Dog!]` 中设计上存在漏洞，因为参数 `namePrefix` 被拼接到 SQL 查询中。拼接用户输入是应用程序的一种常见不良实践，可能使其暴露于 SQL 注入。

以下查询从数据库中的 `CONFIG` 表中提取信息：

```graphql
query sqli {
  dogs(namePrefix: "ab%' UNION ALL SELECT 50 AS ID, C.CFGVALUE AS NAME, NULL AS VETERINARY_ID FROM CONFIG C LIMIT ? -- ", limit: 1000) {
    id
    name
  }
}
```

对此查询的响应为：

```json
{
  "data": {
    "dogs": [
      {
        "id": 1,
        "name": "Abi"
      },
      {
        "id": 2,
        "name": "Abime"
      },
      {
        "id": 3,
        "name": "..."
      },
      {
        "id": 50,
        "name": "$Nf!S?(.}DtV2~:Txw6:?;D!M+Z34^"
      }
    ]
  }
}
```

该查询包含示例应用程序中签名 JWT 的秘密，这是非常敏感的信息。

为了知道在任何特定应用程序中要查找什么，收集有关应用程序如何构建以及数据库表如何组织的信息将很有帮助。你还可以使用像 `sqlmap` 这样的工具来查找注入路径，甚至自动化从数据库中提取数据。

#### 跨站脚本（XSS）

当攻击者注入可执行代码，随后由浏览器运行时，就会发生跨站脚本。在[输入验证](../07-Input_Validation_Testing/README.md)章节中了解 XSS 的测试。你可以使用[测试反射型跨站脚本](../07-Input_Validation_Testing/01-Testing_for_Reflected_Cross_Site_Scripting.md)中的载荷来测试反射型 XSS。

### 拒绝服务（DoS）查询

GraphQL 暴露了一个非常简单的接口，允许开发者使用嵌套查询和嵌套对象。这种能力也可能被恶意利用，通过调用类似递归函数的深度嵌套查询，并耗尽 CPU、内存或其他计算资源来导致拒绝服务。

回顾*图 12.1-1*，可以看到可以创建一个循环，其中 Dog 对象包含 Veterinary 对象。可能存在无限数量的嵌套对象。

这允许进行深度查询，有可能使应用程序过载：

```graphql
      id
      name
      dogs {
        id
        name

          id

            id

              id

```

有多种安全措施可以实施以防止这些类型的查询，详见[修复](#remediation)部分。滥用查询可能导致 GraphQL 部署的 DoS 问题，应纳入测试。

### 批处理攻击

GraphQL 支持将多个查询批处理到单个请求中。这允许用户高效地请求多个对象或对象的多个实例。然而，攻击者可以利用此功能执行批处理攻击。在一个请求中发送多个查询如下所示：

```graphql
[
  {
    query: < query 0 >,
    variables: < variables for query 0 >,
  },
  {
    query: < query 1 >,
    variables: < variables for query 1 >,
  },
  {
    query: < query n >
    variables: < variables for query n >,
  }
]
```

在示例应用程序中，可以使用可猜测的 ID（它是一个递增整数）在一个请求中发送以提取所有兽医名称。攻击者随后可以利用这些名称来获取访问令牌。与其通过多个请求进行（这可能被网络安全措施如 Web 应用防火墙或速率限制器如 Nginx 阻止），这些请求可以被批处理。这意味着只有少数几个请求，这可能允许有效的暴力破解而不被检测到。以下是一个示例查询：

```graphql
query {
  Veterinary(id: "1") {
    name
  }
  second:Veterinary(id: "2") {
    name
  }
  third:Veterinary(id: "3") {
    name
  }
}
```

这将为攻击者提供兽医的名称，并且如前所述，名称可用于批处理多个请求，以请求这些兽医的认证令牌。例如：

```graphql
query {
  auth(veterinaryName: "Julien")
  second: auth(veterinaryName:"Benoit")
}
```

批处理攻击可用于绕过站点上强制执行的许多安全措施。它也可用于枚举对象并尝试暴力破解多因素认证或其他敏感信息。

### 通过别名绕过查询黑名单

GraphQL 允许为查询中的任何字段指定别名——用于标记结果的替代名称。这是一个标准的语言特性，但可能被滥用，以规避那些查找特定字段名的实施不当的安全控制。

一些部署使用对原始 GraphQL 文档进行简单字符串或正则表达式匹配来实现查询黑名单（例如，仅当字段名 `adminUsers` 以 `adminUsers {` 或 `adminUsers(` 形式出现时拒绝任何包含该字段名的查询，而无需完全解析语法或构建 AST）。如果实现在请求中的任何位置将 `adminUsers` 作为子字符串进行搜索，或者正确解析查询结构，则仍会检测到别名调用，因为底层的字段名 `adminUsers` 仍然存在于操作中，并未被别名重命名或移除。然而，那些不处理 `aliasName: fieldName` 语法的简单检查可以被绕过，方法是在别名下发出相同的字段，而 `adminUsers` 解析器继续在服务器上执行。请注意，在 GraphQL 术语中，以下示例中的 `adminUsers` 是一个字段名；操作名会出现在 `query` 关键字之后（例如，`query GetAdmins { ... }`）。

首先，直接发送查询以确认它被阻止：

```graphql
query {
  adminUsers {
    id
    name
    email
  }
}
```

响应（被黑名单阻止）：

```json
{
  "errors": [
    {
      "message": "400 Bad Request：不允许查询"
    }
  ]
}
```

然后，使用别名重新发送相同的查询：

```graphql
query {
  s: adminUsers {
    id
    name
    email
  }
}
```

响应（绕过了黑名单，返回了数据）：

```json
{
  "data": {
    "s": [
      { "id": 1, "name": "Admin", "email": "admin@example.com" }
    ]
  }
}
```

服务器将 `s` 解析为 `adminUsers`，因为别名是 GraphQL 层的构造——底层解析器正常运行。仅在请求正文中匹配字面字段名的黑名单将不会捕获别名调用。

对于每个似乎被阻止或受限的查询，尝试为其添加任意别名前缀（例如 `x:`、`s:`、`bypass:`），并观察响应是否从错误变为成功的数据载荷。

### 详细错误消息

GraphQL 在运行时可能遇到意外错误。当发生此类错误时，服务器可能发送一个错误响应，可能揭示内部错误详细信息或应用程序配置或数据。这允许恶意用户获取更多关于应用程序的信息。作为测试的一部分，应通过发送意外数据（称为模糊测试的过程）来检查错误消息。应搜索响应中可能通过此技术泄露的潜在敏感信息。

### 底层 API 的暴露

GraphQL 是一项相对较新的技术，一些应用程序正在从旧 API 过渡到 GraphQL。在许多情况下，GraphQL 被部署为标准 API，它翻译请求（使用 GraphQL 语法发送）到底层 API，以及响应。如果对底层 API 的请求未正确检查授权，则可能导致可能的权限提升。

例如，一个包含参数 `id=1/delete` 的请求可能被解释为 `/api/users/1/delete`。这可能扩展到对属于 `user=1` 的其他资源的操纵。请求也可能被解释为具有授予 GraphQL 节点的授权，而不是真正的请求者。

测试人员应尝试访问底层 API 方法，因为这可能是提升权限的途径。

## 修复

- 限制对内省查询的访问。
- 在生产中禁用字段建议：大多数 GraphQL 服务器允许关闭此功能（例如，在 Apollo Server 中，设置 [hideSchemaDetailsFromClientErrors](https://www.apollographql.com/docs/apollo-server/api/apollo-server#hideschemadetailsfromclienterrors)）。这可以防止通过“您的意思是...？”错误响应进行模式枚举。
- 实施输入验证。
    - GraphQL 没有原生验证输入的方式，然而，有一个名为 ["graphql-constraint-directive"](https://github.com/confuser/graphql-constraint-directive) 的开源项目，允许在模式定义中作为输入验证的一部分。
    - 仅输入验证是有帮助的，但它不是完整的解决方案，应采取措施减轻注入攻击。
- 实施安全措施以防止滥用查询。
    - 超时：限制查询允许运行的时间。
    - 最大查询深度：限制允许的查询深度，这可以防止过深的查询滥用资源。
    - 设置最大查询复杂度：限制查询的复杂度，以减轻 GraphQL 资源的滥用。
    - 使用基于服务器时间的节流：限制用户可以消耗的服务器时间。
    - 使用基于查询复杂度的节流：限制用户可以消耗的查询总复杂度。
- 发送通用错误消息：使用不揭示部署细节的通用错误消息。
- 减轻批处理攻击：
    - 在代码中添加对象请求速率限制。
    - 防止对敏感对象进行批处理。
    - 限制一次可以运行的查询数量。
- 防止查询黑名单绕过：在解析器级别实施访问控制，而不是仅依赖操作名黑名单，因为 GraphQL 别名允许调用者在任意名称下调用任何字段。

有关修复 GraphQL 弱点的更多信息，请参阅 [GraphQL 速查表](https://cheatsheetseries.owasp.org/cheatsheets/GraphQL_Cheat_Sheet.html)。
