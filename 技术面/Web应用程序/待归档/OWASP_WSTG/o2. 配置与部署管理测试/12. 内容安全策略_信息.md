# 测试内容安全策略

## 概述

内容安全策略（CSP）是一种声明式的白名单策略，通过 `Content-Security-Policy` 响应头部或等效的 `<meta>` 元素实施。它允许开发者限制加载资源（如 JavaScript、CSS、图片、文件等）的来源。CSP 是一种有效的纵深防御技术，可降低跨站脚本（XSS）和点击劫持等漏洞的风险。

内容安全策略支持多种指令，可对策略流进行细粒度控制。（详见[参考资料](#参考资料)）

## 测试目标

- 审查 Content-Security-Policy 头部或 meta 元素，以识别配置错误。

## 如何测试

测试内容安全策略（CSP）的弱点不仅仅是验证头部的存在。测试人员应评估该策略是否有意义地减少了攻击面，并得到了正确实施。

### 识别并确认 CSP 的实施

- 检查 HTTP 响应中是否存在 `Content-Security-Policy` 头部。
- 检查是否存在 `Content-Security-Policy-Report-Only`。如果仅存在 Report-Only 版本，则策略未强制执行。
- 验证 CSP 是通过 HTTP 头部还是 `<meta>` 标签传递的（HTTP 头部为首选）。
- 注意，通过 `<meta>` 标签传递的 CSP 不支持某些指令，如 `frame-ancestors`、`report-uri`、`report-to` 或 `sandbox`。
- 确认策略在敏感端点上是否一致应用。

### 审查高风险指令

检查策略中是否存在不安全或过于宽松的指令：

- `unsafe-inline` 允许内联脚本或样式，会显著削弱 XSS 防护。
- `unsafe-eval` 允许动态代码执行（`eval()`），增加了绕过风险。
- `unsafe-hashes` 可能允许内联执行（如果哈希值可预测或范围不当）。
- 通配符（`*`）源可能允许从任何源加载资源。
    - 应仔细评估部分通配符，如 `https://*` 或 `*.cdn.com`。
    - 确定白名单域名是否托管 JSONP 端点或用户可控内容。
- 缺少或误用 `frame-ancestors` 可能使应用程序面临点击劫持风险。
- 缺少 `object-src`、`base-uri` 或限制性的 `default-src` 指令可能会削弱策略有效性。
- 审查 `require-trusted-types-for` 和 `trusted-types` 的使用。在高风险应用程序中，缺少 Trusted Types 可能使基于 DOM 的注入点暴露。如果定义了 `trusted-types` 策略，请确保它们不会过于宽松。
- 检查重复指令或冲突的策略定义，可能导致非预期的执行行为。

### 验证 Nonce 和 strict-dynamic 的使用

如果策略使用 nonce：

- 确认 nonce 是加密随机生成的。
- 验证 nonce 每次响应都会重新生成，且不会被重用。
- 确保旧版内联脚本模式不会被无意信任。

如果使用了 `strict-dynamic`：

- 了解信任从基于 nonce 或哈希的脚本传播。
- 确认不存在不安全的信任链，导致攻击者控制的脚本被加载。

### 评估 CSP 报告机制

如果配置了 `report-uri` 或 `report-to`：

- 验证报告端点是否可达且功能正常。
- 确定报告中是否暴露了敏感信息。
- 确认报告不会产生新的注入或拒绝服务向量。

### 尝试受控绕过技术

尝试通过测试受控负载来验证实施效果：

- 内联脚本注入尝试。
- 基于 Data URL 的负载。
- 从白名单域名的 JSONP 回调操纵。
- 使用受信任脚本源的基于 DOM 的 gadget 链。

注入的 JavaScript 成功执行表明 CSP 配置错误或实施无效。

### 评估策略强度

业务关键型应用程序应致力于实施严格策略。一个健壮的 CSP 通常：

- 避免 `unsafe-inline` 和 `unsafe-eval`。
- 使用基于 nonce 或哈希的脚本控制。
- 限制对象嵌入（`object-src 'none'`）。
- 限制 base 标签操纵（`base-uri 'none'`）。
- 使用 `frame-ancestors` 限制框架嵌入。

### 常见的 CSP 绕过模式

即使存在 CSP，配置错误或设计弱点也可能允许绕过。测试人员应考虑以下常见模式：

#### JSONP 和受信任的第三方端点

如果 CSP 白名单列出了第三方域名（例如 CDN），请确定这些域名是否暴露 JSONP 端点或用户可控内容。攻击者可能利用回调注入来执行任意 JavaScript，同时仍符合策略。

#### 通配符和宽松源策略

依赖通配符（`*`、`https://*`、`*.example.com`）的策略会显著扩大信任边界。子域名接管或受感染的第三方服务可能允许在允许的范围内绕过。

#### Nonce 重用或可预测的 Nonce

如果 nonce 跨请求重用或可预测生成，攻击者可能重用或猜测它们以执行注入脚本。每个响应都应生成一个全新的、加密强度高的 nonce。

#### 过度依赖 strict-dynamic

虽然 `strict-dynamic` 改进了基于 nonce 的策略，但信任从受信任脚本传播。如果受信任脚本加载了攻击者控制的资源，则防护可能被削弱。

#### 缺少纵深防御指令

缺少诸如：

- `object-src 'none'`
- `base-uri 'none'`
- `frame-ancestors`

等指令可能使应用程序暴露于对象注入、base 标签操纵或点击劫持，即使脚本执行受到部分限制。

#### 仅报告模式的 CSP

应用程序有时会无限期地以 `Report-Only` 模式部署 CSP。在这种情况下，违规会被记录但不会被阻止，无法提供真正的保护。

## 修复

组织应实施强大且范围适当的内容安全策略，以有效减少攻击面，而不是仅仅满足合规要求。

### 通用建议

- 避免使用 `unsafe-inline` 和 `unsafe-eval`。
- 优先使用基于 nonce 或哈希的脚本控制。
- 使用限制性的 `default-src` 指令。
- 明确定义：
    - `object-src 'none'`
    - `base-uri 'none'`
    - `frame-ancestors`
- 尽量减少白名单中的第三方域名数量。
- 在引入新的前端依赖项时定期审查 CSP 策略。

### 部署最佳实践

- 暂时部署 `Content-Security-Policy-Report-Only` 进行测试，验证后启用 `Content-Security-Policy` 强制执行。
- 监控违规报告，识别合法的破坏和潜在的攻击尝试。
- 在需要向后兼容的情况下，优先使用 `report-to`（CSP Level 3），同时保留 `report-uri`。
- 确保 nonce 是加密随机生成的，且每次响应重新生成。
- 在前端重大重构或框架更改时审查策略。

### 严格策略指南

严格策略可防范存储型、反射型和某些基于 DOM 的 XSS 攻击，应是高风险或业务关键型应用程序的目标。

基于 nonce 的方法，示例策略包括：

中等严格策略：

```HTTP
script-src 'nonce-r4nd0m' 'strict-dynamic';
object-src 'none';
base-uri 'none';
```

锁定严格策略：

```HTTP
script-src 'nonce-r4nd0m';
object-src 'none';
base-uri 'none';
```

注意：`'nonce-r4nd0m'` 仅作为示例占位符。在实践中，nonce 必须是加密强度高、Base64 编码的值，且每个 HTTP 响应唯一生成。

团队应谨慎调整严格策略，确保与应用程序架构兼容，同时保持安全目标。
