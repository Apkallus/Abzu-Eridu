# 测试 OAuth 客户端弱点

## 概述

OAuth 将资源的访问权限授予客户端。这使得客户端能够代表资源所有者行事。客户端在令牌交换中接收授权码和刷新令牌并存储它们。

如果未能保护令牌交换和凭据，可能导致未授权资源访问和权限提升。

## 测试目标

- 识别 OAuth 客户端中的弱点。

## 如何测试

为了测试客户端弱点，您应旨在：

1. 检索用于授权的凭据。
2. 通过强制浏览授予自己对任意资源的访问权限。
3. 绕过授权。

### 测试暴露的客户端密钥

客户端密钥用于向授权服务器（AS）验证客户端身份，以证明客户端是受信任的来源。

公共客户端通常无法安全地存储客户端密钥。

要识别客户端代码中的客户端密钥，请对客户端代码进行侦察。

1. 浏览到应用程序。
2. 打开浏览器的开发者工具。
3. 导航到“调试器”选项卡。
4. 按 Ctrl+Shift+F 打开搜索。
5. 搜索类似于 `client-secret` 的术语，并确定是否找到任何内容。

如果此方法不成功，您还可以：

1. 使用 HTTP 拦截代理（如 ZAP）逐步执行授权过程。
2. 从 URI 中的参数 `client-secret` 检索客户端密钥。
3. 将上述搜索中的搜索词替换为客户端密钥的值，并确定其是否暴露。

### 测试令牌存储不当

客户端接收访问令牌，理想情况下应将其存储在可防止攻击者获取的位置。

机密客户端应将令牌存储在易失性存储器中，以防止通过其他攻击（如本地文件包含、能够访问环境的攻击者或 SQL 注入攻击）进行访问。

公共客户端（如单页应用程序）则无法安全地存储令牌。例如，跨站脚本攻击允许攻击者访问浏览器中存储的凭据。

公共客户端可以将令牌存储在浏览器的会话存储或 cookie 中，但不应存储在本地存储中。要确定令牌是否存储不当：

1. 浏览到应用程序。
2. 检索一个访问令牌。
3. 打开浏览器的开发者工具。
4. 导航到“应用程序”选项卡。
5. 找到本地存储并查看存储的数据。
6. 找到会话存储并查看存储的数据。
7. 找到 Cookie 存储并查看存储的数据。

### 测试访问令牌注入

此攻击仅在客户端使用直接向客户端颁发访问令牌的响应类型时才有可能。这发生在隐式流、资源所有者密码凭据和机器对机器流的授权类型中。更多描述请参见[测试 OAuth 弱点](05-Testing_for_OAuth_Weaknesses.md)。

当访问令牌泄露给攻击者，然后攻击者使用该令牌通过合法客户端进行身份验证时，访问令牌注入即告成功。

要测试访问令牌注入，请按照以下步骤操作。在此示例中，授权令牌（`ZXhhbXBsZQo=`）已被泄露。

1. 拦截客户端与授权服务器之间的流量。
2. 使用采用隐式流授权类型的客户端启动 OAuth 流程。
3. 注入窃取的访问令牌：
    - 向客户端发送一个带有窃取访问令牌（`ZXhhbXBsZQo=`）的伪造授权响应。
    - 拦截一个有效的授权响应，并将访问令牌（`dGVzdGluZwo=`）替换为泄露的令牌（`ZXhhbXBsZQo=`）。

## 修复

- 仅在客户端能够安全存储客户端密钥的情况下使用客户端密钥。
- 遵循最佳实践安全地存储令牌。像对待其他凭据一样考虑其安全性。
- 避免使用已弃用的 OAuth 授权类型。更多描述请参见[测试 OAuth 弱点](05-Testing_for_OAuth_Weaknesses.md)。
