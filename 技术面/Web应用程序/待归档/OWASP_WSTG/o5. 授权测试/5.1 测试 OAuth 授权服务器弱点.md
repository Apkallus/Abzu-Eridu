# 测试 OAuth 授权服务器弱点

## 概述

OAuth 将用户身份及其相应的访问权限存储在授权服务器（AS）中。AS 在 OAuth 流程中扮演着关键角色，因为它授予客户端对资源的访问权限。为了安全地做到这一点，它必须正确验证 OAuth 流程中涉及的参数。

如果未能正确验证这些参数，可能导致账户接管、未授权资源访问和权限提升。

## 测试目标

- 识别授权服务器中的弱点。

## 如何测试

为了测试 AS 的弱点，您应旨在：

1. 检索用于授权的凭据。
2. 通过强制浏览授予自己对任意资源的访问权限。
3. 绕过授权。

### 测试重定向 URI 验证不足

如果 `redirect_uri` 未正确验证，则可以构造一个包含指向攻击者所控制服务器 URL 的链接。这可用于诱骗 AS 将授权码发送给攻击者。在以下示例中，`client.evil.com` 被用作伪造的 `redirect_uri`。

```text
https://as.example.com/authorize?client_id=example-client&redirect_uri=http%3A%2F%client.evil.com%2F&state=example&response_mode=fragment&response_type=code&scope=openid&nonce=example
```

如果用户在此用户代理中打开此链接，AS 会将用户代理重定向到恶意 URL。

攻击者可以捕获伪造 URL 中传递的 `code` 值，然后将其提交给 AS 的令牌端点。

以下请求演示了一个授权请求，该请求将 `redirect_uri` 发送给授权服务器。客户端 `client.example.com` 向 AS `as.example.com` 发送一个授权请求，其中包含 URL 编码的重定向 URI `http%3A%2F%2Fclient.example.com%2F`。

```http
GET /authorize
    ?redirect_uri=http%3A%2F%2Fclient.example.com%2F
    &client_id=example-client
    &errorPath=%2Ferror
    &scope=openid%20profile%20email
    &response_type=code
    &response_mode=query
    &state=example
    &nonce=example
    &code_challenge=example
    &code_challenge_method=S256 HTTP/1.1
Host: as.example.com
```

AS 响应一个包含授权码的重定向。该授权码可以在令牌请求中交换为访问令牌。如下所示，`Location` 头中的 URL 是先前 `redirect_uri` 参数中给定的 URI。

```http
HTTP/1.1 302 Found
Date: Mon, 18 Oct 2021 20:46:44 GMT
Content-Type: text/html; charset=utf-8
Content-Length: 340
Location: https://client.example.com/?code=example&state=example
```

要测试 AS 是否容易受到重定向 URI 验证不足的影响，请使用 HTTP 拦截代理（如 ZAP）捕获流量。

1. 启动 OAuth 流程并在授权请求处暂停。
2. 更改 `redirect_uri` 的值并观察响应。
3. 调查响应并识别 AS 是否接受了任意的 `redirect_uri` 参数。

如果 AS 将用户代理重定向到您指定的 `redirect_uri`，则 AS 未正确验证 `redirect_uri`。

此外，请参阅[测试服务器端请求伪造](../07-Input_Validation_Testing/19-Testing_for_Server-Side_Request_Forgery.md)中的“常见过滤器绕过”部分，以识别重定向 URI 验证的常见绕过方式。

### 测试授权码注入

在授权码流程代码交换期间，AS 向客户端颁发一个授权码，随后该授权码在令牌端点处交换，以检索授权令牌和刷新令牌。

对 AS 执行以下测试：

1. 为另一个 `client_id` 发送有效的授权码。
2. 为另一个资源所有者发送有效的授权码。
3. 为另一个 `redirect_uri` 发送有效的授权码。
4. 多次重新发送该授权码（代码重放）。

#### 测试公共客户端

发送到令牌端点的请求包含授权码。它被交换为令牌。使用 HTTP 拦截代理（如 ZAP）捕获此请求，并使用修改后的值重新发送。

```http
POST /oauth/token HTTP/1.1
Host: as.example.com
[...]

{
    "errorPath":"/error",
    "client_id":"example-client",
    "code":"INJECT_CODE_HERE",
    "grant_type":"authorization_code",
    "redirect_uri":"https://client.example.com"
}
```

如果 AS 响应一个 `access_token`，则说明代码注入成功。

#### 测试机密客户端

由于机密客户端的 OAuth 流程额外受客户端密钥保护，因此无法直接将授权码提交给令牌端点。相反，将授权码注入客户端。然后，该注入的代码将由机密客户端连同客户端密钥一起在令牌请求中发送。

首先，从 AS 捕获一个授权码：

1. 使用用户 Alice 启动授权码流程。在收到 AS 的授权码时暂停。
2. 不要将代码提交给客户端，并记下该代码和对应的状态值。

然后，注入该代码：

1. 使用用户 Mallory 启动授权码流程，并将先前捕获的 Alice 用户的代码和状态值注入到该流程中。
2. 如果攻击成功，客户端现在应拥有一个 `authorization_token`，该令牌授予对 Alice 用户所拥有资源的访问权限。

```http
GET /callback?code=INJECT_CODE_HERE&state=example HTTP/1.1
Host: client.example.com
[...]

```

### 测试 PKCE 降级攻击

在某些情况下，PKCE 扩展可以从授权码流程中移除。这可能导致公共客户端容易受到 PKCE 扩展本应缓解的攻击。

这种情况可能发生在：

- AS 不支持 PKCE。
- AS 未正确验证 PKCE。

两者都可以使用 HTTP 拦截代理（如 ZAP）进行测试。执行以下测试：

1. 发送不带 `code_challenge=sha256(xyz)` 和 `code_challenge_method` 参数的授权请求。
2. 发送带有空 `code_challenge=sha256(xyz)` 参数值的授权请求。
3. 发送带有伪造 `code_challenge=sha256(xyz)` 参数值的授权请求。

以下示例突出显示了要修改的值：

```http
GET /authorize
    ?redirect_uri=http%3A%2F%client.example.com
    &client_id=example-client
    &errorPath=%2Ferror
    &scope=openid%20profile%20email
    &response_type=code
    &response_mode=web_message
    &state=example-state
    &nonce=example-nonce
    &code_challenge=MODIFY_OR_OMIT_THIS
    &code_challenge_method=MODIFY_OR_OMIT_THIS
    &prompt=none HTTP/1.1
Host: as.example.com
[...]

```

AS 应验证令牌交换中的 `code_verifier` 值。进行以下测试：

1. 发送不带 `code_verifier` 的令牌请求。
2. 发送带有空 `code_verifier` 的令牌请求。
3. 发送带有针对不同授权码的有效 `code_verifier` 的令牌请求。

```http
POST /oauth/token HTTP/1.1
Host: as.example.com
[...]

{
"client_id":"example-client",
"code_verifier":"MODIFY_OR_OMIT_THIS",
"code":"example",
"grant_type":"authorization_code",
"redirect_uri":"https://client.example.com"
}
```

### 测试同意页面的跨站请求伪造

CSRF 攻击在 [CSRF](../06-Session_Management_Testing/05-Testing_for_Cross_Site_Request_Forgery.md) 中有所描述。OAuth 可能遭受 CSRF 攻击。

为防止 CSRF 攻击，OAuth 利用 `state` 参数作为反 CSRF 令牌。

其他措施也可以防止 CSRF 攻击。PKCE 流程可缓解 CSRF。`nonce` 值也可作为反 CSRF 令牌。

根据 [CSRF](../06-Session_Management_Testing/05-Testing_for_Cross_Site_Request_Forgery.md) 测试用例中描述的测试，对 OAuth 使用的每个包含反 CSRF 参数的请求进行测试。

同意页面显示给用户，以验证该用户同意客户端代表用户访问资源。使用 CSRF 攻击同意页面可能授予任意客户端代表用户访问资源的权限。此流程的步骤如下：

1. 客户端生成一个 state 参数并将其与同意请求一起发送。
2. 用户代理显示同意页面。
3. 资源所有者授予客户端访问权限。
4. 同意与已确认的作用域一起发送给 AS。

使用 HTTP 拦截代理（如 ZAP）来测试 state 参数是否被正确验证。

```http
POST /u/consent?state=Tampered_State HTTP/1.1
Host: as.example.com
[...]

state=MODIFY_OR_OMIT_THIS
&audience=https%3A%2F%2Fas.example.com%2Fuserinfo
&scope%5B%5D=profile
&scope%5B%5D=email
&action=accept
```

### 测试点击劫持

点击劫持在[测试点击劫持](../11-Client-side_Testing/09-Testing_for_Clickjacking.md)中有所描述。当同意页面容易受到点击劫持攻击，且攻击者拥有 `client_id`（对于公共客户端）或客户端密钥（对于机密客户端）时，攻击者可以伪造用户的同意，并通过恶意客户端获得对所请求资源的访问权限。

#### 如何测试

要使此攻击成功，攻击者需要将授权页面加载到 iframe 中。

以下 HTML 页面可用于将授权页面加载到 iframe 中：

```html
<html>
    <head>
        <title>Clickjack test page</title>
    </head>
    <body>
        <iframe src="https://as.example.com/auth/realms/example/login-actions/required-action?execution=OAUTH_GRANT&client_id=example-client" width="500" height="500"></iframe>
    </body>
</html>
```

如果成功加载，则该站点容易受到点击劫持攻击。

有关如何实施此类攻击的详细说明，请参阅[测试点击劫持](../11-Client-side_Testing/09-Testing_for_Clickjacking.md)。

### 测试令牌生存期

OAuth 有两种类型的令牌：访问令牌和刷新令牌。访问令牌的有效期应受限。也就是说，它应该是短期的：合适的持续时间取决于应用程序，可能是 5 到 15 分钟。

刷新令牌的有效期应更长。它应该是一次性令牌，每次使用后都会被替换。

#### 测试访问令牌生存期验证

当使用 JSON Web 令牌（JWT）作为访问令牌时，可以从解码后的 JWT 中检索访问令牌的有效期。这在[测试 JSON Web 令牌](../06-Session_Management_Testing/10-Testing_JSON_Web_Tokens.md)中有所描述。AS 可能未正确验证 JWT 的生存期。

要测试访问令牌的生存期，请使用 HTTP 拦截代理（如 ZAP）。拦截一个包含访问令牌的端点请求。将此请求放入重放器中，并等待目标时间过去。访问令牌的有效期应在 5 到 15 分钟之间，具体取决于资源的敏感性。

此类请求可能如下例所示。令牌也可以通过其他方式传输，例如在 cookie 中。

```http
GET /userinfo HTTP/1.1
Host: as.example.com
[...]
Authorization: Bearer eyJhbGciOiJkaXIiL[...]

```

通过在不同时间长度后发送请求来测试生存期验证，例如 5 分钟、10 分钟和 30 分钟后。

可以通过自动化步骤和记录服务器响应来优化此过程。当收到 HTTP 状态 403（而不是 HTTP 状态 200）的响应时，可能表明访问令牌不再有效。

#### 测试刷新令牌生存期验证

刷新令牌的有效期比访问令牌长。由于其有效期长，它们应在用于交换访问令牌后失效。

刷新令牌在与访问令牌一起发放给客户端的同一令牌请求中颁发。

使用 HTTP 拦截代理（如 ZAP）。通过以下步骤进行设置：

1. 检索有效的刷新令牌。
2. 捕获用于将刷新令牌交换为新访问令牌的请求。
3. 将捕获的请求发送到请求重放器。

在以下示例中，刷新令牌作为 POST 正文的一部分发送。

```http
POST /token HTTP/1.1
Host: as.example.com
Cookie: [...]
[...]

grant_type=refresh_token
&refresh_token=eyJhbGciOiJIUz[...]
&client_id=example-client

```

执行以下测试：

1. 发送刷新令牌并确定 AS 是否发放访问令牌。
2. 使用相同的刷新令牌重复步骤，以评估单个刷新令牌被接受的次数。

当使用 JWT 作为刷新令牌时，可以从解码后的 JWT 中检索刷新令牌的有效期。这在[测试 JSON Web 令牌](../06-Session_Management_Testing/10-Testing_JSON_Web_Tokens.md)中有所描述。刷新令牌的有效期可能更长，但应有一个到期日期。

通过盗窃检测机制可以获得额外的安全性。如果刷新令牌在其有效期（或生存期）之外用于令牌交换，AS 将使所有刷新令牌失效。要测试此机制：

1. 发送刷新令牌并确定 AS 是否发放访问令牌。
2. 使用相同的刷新令牌重复步骤，直到其失效。
3. 使用最后一次令牌响应中的刷新令牌。

如果为该资源所有者颁发给客户端的所有刷新令牌都被失效，则 AS 具有令牌盗窃检测功能。

## 修复

对 OAuth AS 的大多数攻击都可以通过在授权码和令牌交换期间验证参数的存在性和内容来缓解。

限制授权码和刷新令牌等凭据的时间范围和允许使用方式。这可以缓解某些类型的攻击，并且如果攻击者获取了此类凭据，也会限制其使用。

正确配置安全缓解措施，如 CORS、反 CSRF 令牌和反点击劫持头部，可以缓解或限制攻击的影响。

- 始终验证所有参数是否存在，并验证其值。
- 使用 PKCE 扩展来正确保护授权码和令牌交换。
- 不允许安全功能（如 PKCE 扩展）的回退。
- 限制凭据的生存期。
- 尽可能仅使用凭据一次，例如授权码。
- 配置可用的安全缓解措施，如 CORS、反 CSRF 令牌和反点击劫持头部。
