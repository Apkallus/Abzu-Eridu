# 测试替代渠道中的弱身份验证

|ID          |
|------------|
|WSTG-ATHN-10|

## 概述

即使主要身份验证机制不包含任何漏洞，同一用户账户的替代合法身份验证用户渠道中也可能存在漏洞。应进行测试以识别替代渠道，并根据测试范围识别漏洞。

替代用户交互渠道可能被用于绕过主要渠道，或暴露可用于辅助攻击主要渠道的信息。其中一些渠道本身可能是使用不同主机名或路径的独立 Web 应用程序。例如：

- 标准网站
- 移动端或特定设备优化网站
- 无障碍优化网站
- 替代国家和语言网站
- 使用相同用户账户的并行网站（例如同一组织提供不同功能的另一个网站，或共享用户账户的合作伙伴网站）
- 标准网站的开发、测试、UAT 和预发布版本

但它们也可能是其他类型的应用程序或业务流程：

- 移动设备应用
- 桌面应用程序
- 呼叫中心客服
- 交互式语音应答或电话树系统

请注意，本测试的重点是替代渠道；某些身份验证替代方案可能表现为通过同一网站传递的不同内容，几乎肯定属于测试范围。此处不再进一步讨论，这些应在信息收集和主要身份验证测试期间已识别。例如：

- 渐进增强和优雅降级导致的功能变化
- 不使用 cookie 的站点使用
- 不使用 JavaScript 的站点使用
- 不使用 Flash 和 Java 等插件的站点使用

即使测试范围不允许测试替代渠道，也应记录其存在。这些可能会削弱身份验证机制的置信度，并可能成为额外测试的前兆。

## 示例

主要网站为 `https://www.example.com`，身份验证功能始终使用 TLS 在 `https://www.example.com/myaccount/` 页面上进行。

然而，存在一个独立的移动优化网站，完全不使用 TLS，且具有较弱的密码恢复机制：`https://m.example.com/myaccount/`。

## 测试目标

- 识别替代身份验证渠道。
- 评估所使用的安全措施，以及替代渠道上是否存在任何绕过。

## 如何测试

### 了解主要机制

全面测试网站的主要身份验证功能。这应识别账户如何发放、创建或更改，以及密码如何恢复、重置或更改。此外，还应了解任何提升权限的身份验证和身份验证保护措施。这些前提对于与任何替代渠道进行比较是必要的。

### 识别其他渠道

可通过以下方法找到其他渠道：

- 阅读站点内容，尤其是首页、联系我们、帮助页面、支持文章和常见问题、条款和条件、隐私声明、robots.txt 文件和任何 sitemap.xml 文件。
- 在先前信息收集和测试期间记录的 HTTP 代理日志中搜索字符串，如 URL 路径和正文内容中的“mobile”、“android”、“blackberry”、“ipad”、“iphone”、“mobile app”、“e-reader”、“wireless”、“auth”、“sso”、“single sign on”。
- 使用搜索引擎查找同一组织或使用相同域名的不同网站，这些网站具有相似的首页内容或也具有身份验证机制。

对于每个可能的渠道，确认用户账户是否跨这些渠道共享，或是否提供对相同或类似功能的访问。

### 枚举身份验证功能

对于每个共享用户账户或功能的替代渠道，识别主要渠道的所有身份验证功能是否可用，以及是否存在额外功能。创建一个类似下面的表格可能很有用：

  | 主要渠道 | 移动端 | 呼叫中心 | 合作伙伴网站 |
  |---------|---------|--------------|-----------------|
  | 注册    | 是     |     -        |       -         |
  | 登录    | 是     |    是        |    是（SSO）    |
  | 注销    |   -     |     -        |       -         |
  | 密码重置 |  是  |   是   |       -         |
  | -       | 更改密码 |   -  |       -         |

在此示例中，移动端具有额外的“更改密码”功能，但不提供“注销”。通过致电呼叫中心也可以执行有限数量的任务。呼叫中心可能很有趣，因为它们的身份确认检查可能比网站更弱，允许使用此渠道辅助攻击用户账户。

在枚举这些时，值得注意会话管理是如何进行的，以防任何渠道之间存在重叠（例如 cookie 作用域限定在同一父域名下，允许跨渠道并发会话，但不允许在同一渠道上）。

### 审查和测试

即使标记为“仅供参考”或“超出范围”，替代渠道也应在测试报告中提及。在某些情况下，测试范围可能包括替代渠道（例如，因为它只是目标主机名上的另一个路径），或者在与所有渠道的负责人讨论后可能会被添加到范围中。如果测试被允许和授权，则应执行本指南中所有其他身份验证测试，并与主要渠道进行比较。

## 相关测试用例

应使用所有其他身份验证测试的测试用例。

## 修复

确保在所有渠道中应用一致的身份验证策略，使它们同样安全。
