# 测试多因素认证（MFA）

|ID          |
|------------|
|WSTG-ATHN-11|

## 概述

许多应用程序将多因素认证（MFA）作为额外的安全层来保护登录过程。这也被称为双因素认证（2FA）或两步验证（2SV）——尽管它们严格来说并不完全相同。MFA 意味着要求用户登录时提供*至少*两种不同的[认证因素](#mfa的类型)。

MFA 为认证功能以及其他安全相关领域（如凭据管理和密码恢复）增加了额外的复杂性，这意味着必须正确且健壮地实现它。

## 测试目标

- 识别应用程序使用的 MFA 类型。
- 确定 MFA 实现是否健壮且安全。
- 尝试绕过 MFA。

## 如何测试

### MFA 的类型

MFA 意味着认证需要*至少*以下因素中的两种：

| 因素 | 示例 |
|--------|----------|
| 你知道的内容 | 密码、PIN 码和安全问题。 |
| 你拥有的内容 | 硬件或软件令牌、证书、电子邮件*、短信和电话呼叫。 |
| 你的特征 | 指纹、面部识别、虹膜扫描、掌纹扫描和行为因素。 |
| 位置 | 源 IP 范围、地理位置。 |

\* 电子邮件只有在电子邮件账户本身受到 MFA 保护时，才真正构成“你拥有的内容”。因此，它应被视为比证书或 TOTP 等其他替代方案更弱，在某些定义下可能不被接受为 MFA。

请注意，要求单一因素的多个示例（例如同时需要密码和 PIN 码）**并不构成 MFA**，尽管它可能比单纯密码提供一些安全优势，并可能被视为两步验证（2SV）。

由于在基于浏览器的环境中实施生物识别的复杂性，“你的特征”很少用于 Web 应用程序，尽管它正开始通过 WebAuthn 等标准被采用。最常见的第二因素是“你拥有的内容”。

### 检查 MFA 绕过

测试 MFA 的第一步是识别应用程序中的所有认证功能，这可能包括：

- 主登录页面。
- 安全关键功能（例如禁用 MFA 或更改密码）。
- 联合登录提供商。
- API 端点（来自主 Web 界面和移动应用）。
- 替代（非 HTTP）协议。
- 测试或调试功能。

应审查所有不同的登录方法，以确保 MFA 得到一致执行。如果某些方法不需要 MFA，则这些方法可能提供简单的绕过方式。

如果认证分多步进行，则可能通过完成认证过程的第一步（输入用户名和密码），然后强制浏览到应用程序或直接发出 API 请求，而不完成第二阶段（输入 MFA 代码）来绕过。

如果认证使用的是允许自定义认证流程（或策略）的 OpenID Connect（OIDC）提供商（如 Azure B2C），则可能定义了多个流程，其中一些可能不需要 MFA。例如，如果应用程序使用名为 `B2C_1_SignInWithMFA` 的流程进行认证，则尝试将其篡改为 `B2C_1_SignIn`、`B2C_1_SignInWithoutMFA` 或其他类似值。

在某些情况下，也可能存在有意的 MFA 绕过实现，例如在以下情况下不要求 MFA：

- 来自特定 IP 地址（可能使用 `X-Forwarded-For` HTTP 头进行欺骗）。
- 设置了特定的 HTTP 头（如非标准头 `X-Debug`）。
- 针对特定硬编码账户（如“root”或“breakglass”账户）。

如果应用程序同时支持本地登录和联合登录，并且这两种账户类型之间没有强隔离，则可能绕过 MFA。例如，如果用户注册了本地账户并为其配置了 MFA，但未在联合登录提供商上的账户配置 MFA，则攻击者可能通过破坏用户在联合登录提供商上的账户，使用相同的电子邮件地址在目标应用程序上重新注册（或关联）一个联合账户。

最后，如果 MFA 在不同于主应用程序的系统上实现（例如在反向代理上，以保护原本不支持 MFA 的旧应用程序），则可能通过直接连接到后端应用服务器来绕过它，如[映射应用架构](../01-Information_Gathering/10-Map_Application_Architecture.md#内容分发网络-cdn)指南中所述。

### 检查 MFA 管理

用于在用户账户内部管理 MFA 的功能应测试漏洞，包括：

- 用户是否需要在删除或更改 MFA 设置时重新认证？
- MFA 管理功能是否容易受到[跨站请求伪造](../06-Session_Management_Testing/05-Testing_for_Cross_Site_Request_Forgery.md)攻击？
- 是否可以通过 [IDOR 漏洞](../05-Authorization_Testing/04-Testing_for_Insecure_Direct_Object_References.md)修改其他用户的 MFA 设置？

### 检查 MFA 恢复选项

许多应用程序会为用户提供一种方法，以便在他们无法使用第二因素认证时重新获得账户访问权限（例如，如果他们丢失了手机）。这些机制通常可能代表应用程序中的重大弱点，因为它们实际上允许绕过第二认证因素。

#### 恢复码

某些应用程序会在用户启用 MFA 时提供一系列恢复码或备用码，可用于登录。应检查这些代码以确保：

- 它们足够长且复杂，以防止暴力破解攻击。
- 它们是安全生成的。
- 它们只能使用一次。
- 实施了暴力破解保护（如账户锁定）。
- 当代码被使用时，用户会收到通知（通过电子邮件、短信等）。

更多详细信息请参见[忘记密码速查表中的“备用码”部分](https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html#backup-codes)。

#### MFA 重置流程

如果应用程序实现了 MFA 重置流程，应像测试[密码重置流程](09-Testing_for_Weak_Password_Change_or_Reset_Functionalities.md)一样对其进行测试。该流程必须*至少*与应用程序的 MFA 实现一样强。

#### 替代认证

某些应用程序允许用户通过其他方式证明其身份，例如使用[安全问题](08-Testing_for_Weak_Security_Question_Answer.md)。这通常代表一个重大弱点，因为安全问题提供的安全级别远低于 MFA。

### 一次性密码

MFA 最常见的形式是一次性密码（OTP），通常是六位数字代码（虽然可以更长或更短）。这些密码可以由服务器和用户同时生成（例如使用身份验证器应用），也可以由服务器生成并发送给用户。OTP 可以通过多种方式提供给用户，包括：

| 类型 | 描述 |
|------|-------------|
| HMAC 一次性密码（HOTP） | 基于秘密和共享计数器的 HMAC 生成代码。 |
| 基于时间的一次性密码（TOTP） | 基于秘密和当前时间的 HMAC 生成代码。 |
| 电子邮件 | 通过电子邮件发送代码。 |
| 短信 | 通过短信发送代码。 |
| 电话 | 通过语音通话发送代码到电话号码。 |

OTP 通常在用户提供用户名和密码后输入。应执行各种检查，包括：

- 多次 MFA 尝试失败后账户是否被锁定？
- 跨不同账户多次 MFA 尝试失败后用户 IP 地址是否被阻止？
- MFA 失败尝试是否被记录？
- 表单是否容易受到注入攻击，包括 [SQL 通配符注入](../07-Input_Validation_Testing/05-Testing_for_SQL_Injection.md#sql-wildcard-injection)？

根据使用的 OTP 类型，还应执行一些其他特定检查：

- OTP 如何发送给用户（电子邮件、短信、电话等）
    - 是否有速率限制以防止短信/电话垃圾信息造成费用？
- OTP 的强度如何（长度和密钥空间）？
- OTP 的有效期多长？
- 是否同时有多个 OTP 有效？
- OTP 能否多次使用？
- OTP 是否绑定到正确的用户账户，还是可以在其他账户上使用它们进行认证？

#### HOTP 和 TOTP

HOTP 和 TOTP 代码都基于服务器和用户之间共享的秘密。对于 TOTP 代码，这通常以二维码的形式提供给用户，用户使用身份验证器应用扫描（也可以以文本秘密的形式提供，供用户手动输入）。

如果秘密在服务器端生成，应检查以确保其足够长且复杂（[RFC 4226](https://www.rfc-editor.org/rfc/rfc4226#section-4) 建议至少 160 位），并且使用[安全随机函数](https://cheatsheetseries.owasp.org/cheatsheets/Cryptographic_Storage_Cheat_Sheet.html#secure-random-number-generation)生成。

如果秘密可由用户提供，则应强制执行适当的最小长度，并检查输入是否存在常见的注入攻击。

TOTP 代码通常有效期为 30 秒，但某些应用程序选择接受多个代码（例如前一个、当前和下一个代码），以处理服务器和用户设备上系统时间的差异。某些应用程序可能允许当前代码两侧的多个代码，这可能使攻击者更容易猜测或暴力破解代码。下表显示了攻击者每秒能发出 10 次请求时，成功暴力破解 OTP 代码的概率，分别针对只接受当前代码的应用程序和接受多个代码的应用程序（有关表格背后的计算，请参见[本文](https://www.codasecurity.co.uk/articles/mfa-testing#case-study---brute-forcing-totp)）。

| 有效代码数 | 1 小时后的成功率 | 4 小时后的成功率 | 12 小时后的成功率 | 24 小时后的成功率 |
|-------------|---------------------------|----------------------------|-----------------------------|-----------------------------|
| 1 | 4%  | 13% | 35% | 58% |
| 3 | 10% | 35% | 72% | 92% |
| 5 | 16% | 51% | 88% | 99% |
| 7 | 22% | 63% | 95% | 99% |

#### 电子邮件、短信和电话

如果代码由服务器生成并发送给客户端，应考虑以下方面：

- 传输机制（电子邮件、短信或语音）对于应用程序是否足够安全？
- 代码是否足够长且复杂？
- 代码是否使用[安全随机函数](https://cheatsheetseries.owasp.org/cheatsheets/Cryptographic_Storage_Cheat_Sheet.html#secure-random-number-generation)生成？
- 代码有效期多长？
- 是否同时有多个代码有效，还是生成新代码会使前一个代码失效？
    - 这能否被用于通过反复请求代码来阻止账户访问？
- 是否有足够的速率限制以防止攻击者请求大量代码？
    - 大量电子邮件代码可能导致服务器因发送垃圾邮件而被阻止。
    - 大量短信或语音呼叫可能花费金钱，或被用于骚扰用户。

### 移动应用和推送通知

OTP 代码的另一种替代方法是将推送通知发送到用户的手机，用户可以批准或拒绝。这种方法不太常见，因为它要求用户安装特定于应用程序的身份验证器。

正确评估其安全性需要将测试范围扩大到覆盖移动应用及其使用的任何支持 API 或服务；这意味着它通常超出传统 Web 应用程序测试的范围。但是，可以在不测试移动应用的情况下执行一些简单的检查，包括：

- 通知是否提供足够的上下文（IP 地址、位置等），以便用户做出批准或拒绝的明智决定？
- 是否有某种质询-响应机制（例如在站点上提供代码，用户需要在应用中输入——通常称为“数字匹配”或“数字挑战”）？
- 是否有速率限制或机制来防止用户被大量通知轰炸，以免用户盲目接受？

### IP 地址和位置过滤

MFA 有时使用的一个因素是位置（“你在哪里”），尽管这是否构成真正的认证因素尚有争议。在 Web 应用程序的上下文中，这通常意味着限制对特定 IP 地址的访问，或者只要用户从特定的受信任 IP 地址连接，就不提示用户输入第二因素。一个常见的场景是，当用户从办公室 IP 范围连接时，仅使用密码进行身份验证，但从其他地方连接时需要 OTP 代码。

根据实现方式，用户可能通过设置 `X-Forwarded-For` 头来欺骗受信任的 IP 地址，从而可能绕过此检查。请注意，如果应用程序未正确清理此头的内容，则还可能执行 SQL 注入等攻击。如果应用程序支持 IPv6，则还应检查以确保对这些连接应用了适当的限制。

此外，应审查受信任的 IP 地址，以确保它们不存在任何弱点，例如是否包括：

- 可能被不受信任的用户访问的 IP 地址（如办公室的访客无线网络）。
- 可能变化的动态分配 IP 地址。
- 攻击者可能托管自己系统的公共网络范围（如 Azure 或 AWS）。

### 证书和智能卡

传输层安全（TLS）通常用于加密客户端和服务器之间的流量，并为客户端提供确认服务器身份的机制（通过将证书上的通用名称（CN）或主题备用名称（SAN）与请求的域进行比较）。然而，它也可以为服务器提供确认客户端身份的机制，即客户端证书认证或相互 TLS（mTLS）。对客户端证书认证的完整讨论超出了本指南的范围，但其关键原则是用户出示数字证书（存储在其机器上或智能卡上），由服务器验证。

测试的第一步是确定目标应用程序是否限制受信任的证书颁发机构（CA）签发证书。此信息可使用各种工具获得，或通过手动检查 TLS 握手获得。最简单的方法是使用 OpenSSL 的 `s_client`：

```bash
$ openssl s_client -connect example:443
[...]
Acceptable client certificate CA names
C = US, ST = Example, L = Example, O = Example Org, CN = Example Org Root Certificate Authority
Client Certificate Types: RSA sign, DSA sign, ECDSA sign
```

如果没有限制，则可能使用来自不同 CA 的证书进行认证。如果有限制但实现不当，则可能使用正确的名称（上例中的“Example Org Root Certificate Authority”）创建本地 CA，并使用此新 CA 签署客户端证书。

如果可以获得有效证书，还应验证该证书只能用于为其颁发的用户（即，你不能使用颁发给 Alice 的证书在 Bob 的账户上进行认证）。此外，应检查证书以确保它们既未过期也未吊销。

## 修复

确保：

- 对应用程序上的所有相关账户和功能实施 MFA。
- 支持的 MFA 方法适合应用程序。
- 用于实施 MFA 的机制得到适当保护并防止暴力破解攻击。
- 对所有 MFA 相关活动进行适当的审计和日志记录。

有关更多建议，请参见 [OWASP 多因素认证速查表](https://cheatsheetseries.owasp.org/cheatsheets/Multifactor_Authentication_Cheat_Sheet.html)。
