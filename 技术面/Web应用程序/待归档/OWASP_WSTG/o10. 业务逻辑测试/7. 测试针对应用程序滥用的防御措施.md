# 测试针对应用程序滥用的防御措施

|ID          |
|------------|
|WSTG-BUSL-07|

## 概述

对有效功能的误用和无效使用可以识别试图枚举 Web 应用程序、发现弱点并利用漏洞的攻击。应进行测试以确定是否存在应用层防御机制来保护应用程序。

缺乏主动防御措施使得攻击者可以毫无顾忌地寻找漏洞。应用程序的所有者因此不会知道其应用程序正受到攻击。

### 示例

一个已认证用户执行了以下（不太可能的）操作序列：

1. 尝试访问其角色不允许下载的文件 ID
2. 用单引号 `'` 替换文件 ID 编号
3. 将 GET 请求改为 POST
4. 添加一个额外参数
5. 复制参数名/值对

应用程序正在监控误用，并在第 5 个事件后以极高的置信度响应用户是攻击者。例如，应用程序：

- 禁用关键功能
- 对剩余功能启用额外的身份验证步骤
- 在每个请求-响应周期中添加时间延迟
- 开始记录有关用户交互的额外数据（例如经过清理的 HTTP 请求头、正文和响应正文）

如果应用程序没有任何响应，并且攻击者可以继续滥用功能并向应用程序提交明显恶意内容，则该应用程序未通过此测试用例。实际上，上述示例中的离散操作不太可能以那种方式发生。更可能的是使用模糊测试工具依次识别每个参数中的弱点。这也是安全测试人员会做的。

## 测试目标

- 从对系统进行的所有测试中生成笔记。
- 审查哪些测试基于攻击性输入表现出不同功能。
- 了解现有的防御措施，并验证它们是否足以保护系统免受绕过技术的攻击。

## 如何测试

此测试的特殊之处在于其结果可以从对 Web 应用程序执行的所有其他测试中得出。在执行所有其他测试时，注意可能表明应用程序具有内置自我防御的措施：

- 更改的响应
- 被阻止的请求
- 将用户注销或锁定其账户的操作

这些可能只是局部的。常见的局部（按功能）防御措施包括：

- 拒绝包含某些字符的输入
- 在多次身份验证失败后临时锁定账户

局部安全控制是不够的。通常没有针对一般误用的防御措施，例如：

- 强制浏览
- 绕过表示层输入验证
- 多次访问控制错误
- 额外、重复或缺失的参数名称
- 多次输入验证或业务逻辑验证失败，其值不可能是用户错误或打字错误的结果
- 接收到无效格式的结构化数据（例如 JSON、XML）
- 接收到明显的跨站脚本或 SQL 注入负载
- 以超出不使用自动化工具可能达到的速度使用应用程序
- 用户的地理位置发生大陆级别变化
- 用户代理更改
- 以错误顺序访问多阶段业务流程
- 大量或高频率使用特定于应用程序的功能（例如优惠券代码提交、信用卡支付失败、文件上传、文件下载、注销等）。

### UI 误导 / 内容欺骗

当用户控制的输入以可能误导用户或管理员的渲染方式显示在受信任的 UI 元素中，且无需脚本执行时，就会发生 UI 误导。与跨站脚本不同，这些问题依赖于视觉或上下文欺骗，并且可以在应用程序中启用工作流滥用或社会工程。

这些问题代表应用程序滥用，因为它们滥用合法的 UI 工作流和信任边界，而不是利用技术漏洞。

常见示例包括：

- 用户控制的文件名、标题或标签显示为系统生成的消息
- 重命名的文件上传显示为受信任的文档或系统工件
- 用户提供的文本显示为批准状态、发件人名称或工作流指示器

例如：一个应用程序允许用户命名上传的文件。攻击者上传一个名为“付款已批准 – 财务系统”的文件。当此文件名在管理审查工作流中显示，且没有明确指示它是用户提供的内容时，审查者可能被误导而批准欺诈流程。

在测试期间，评估：

- 用户控制的数据是否在特权或权威 UI 上下文中反射
- 注入的文本是否可以模仿系统消息、工作流状态或受信任标签
- UI 呈现是否可能影响用户决策或业务流程，即使没有发生技术利用

这些防御措施在应用程序的已认证部分效果最好，尽管在新账户创建速率或访问内容（例如抓取信息）速率方面，在公共区域也可能有用。

并非上述所有内容都需要由应用程序监控，但如果没有任何一项被监控，则存在问题。通过测试 Web 应用程序并执行上述类型的操作，是否对测试人员采取了任何响应？如果没有，测试人员应报告应用程序似乎没有针对误用的全应用程序范围的主动防御措施。请注意，有时对攻击检测的所有响应可能对用户是静默的（例如日志更改、增加监控、向管理员发送警报和请求代理），因此对此发现的置信度无法保证。在实践中，很少有应用程序（或相关基础设施，如 Web 应用防火墙）检测这些类型的误用。
