# 测试上传意外文件类型

|ID          |
|------------|
|WSTG-BUSL-08|

## 概述

许多应用程序的业务流程允许上传和操作通过文件提交的数据。但业务流程必须检查文件，并且只允许某些“已批准”的文件类型。决定哪些文件是“已批准”的由业务逻辑决定，并且是特定于应用程序/系统的。风险在于，通过允许用户上传文件，攻击者可能提交意外文件类型，这些文件可能被执行并对应用程序或系统产生不利影响，攻击方式包括破坏网站、执行远程命令、浏览系统文件、浏览本地资源、攻击其他服务器或利用本地漏洞等，仅举几例。

与上传意外文件类型相关的漏洞是独特的，因为如果文件不具有特定扩展名，上传过程应迅速拒绝该文件。此外，这与上传恶意文件不同，因为在大多数情况下，不正确的文件格式本身可能并不固有“恶意”，但可能对保存的数据有害。例如，如果应用程序接受 Windows Excel 文件，如果上传了类似的数据文件，它可能会被读取，但提取的数据可能会被移动到不正确的位。

应用程序可能只期望处理某些特定文件类型，如 `.csv` 或 `.txt` 文件。应用程序可能不会按扩展名（用于低保证文件验证）或内容（用于高保证文件验证）验证上传的文件。这可能导致应用程序/系统中的意外系统或数据库结果，或为攻击者提供额外方法来利用应用程序/系统。

### 示例

假设一个图片分享应用程序允许用户上传 `.gif` 或 `.jpg` 图形文件到站点。如果攻击者能够上传包含 `<script>` 标签的 HTML 文件或 PHP 文件，会发生什么？系统可能会将文件从临时位置移动到最终位置，现在可以在该位置针对应用程序或系统执行 PHP 代码。

## 测试目标

- 审查项目文档，了解系统拒绝的文件类型。
- 验证不受欢迎的文件类型是否被拒绝并安全处理。
- 验证文件批量上传是否安全，并且不允许绕过已设置的安全措施。

## 如何测试

### 具体测试方法

- 研究应用程序的逻辑要求。
- 准备一个包含“未批准”上传文件的库，这些文件可能包括：jsp、exe 或包含脚本的 HTML 文件。
- 在应用程序中导航到文件提交或上传机制。
- 提交“未批准”的文件进行上传，并验证它们是否被正确阻止上传。
- 检查站点是否仅在客户端 JavaScript 中进行文件类型检查。
- 检查站点是否仅通过 HTTP 请求中的“Content-Type”检查文件类型。
- 检查站点是否仅通过文件扩展名检查文件类型。
- 检查其他上传的文件是否可以通过指定 URL 直接访问。
- 检查上传的文件是否可能包含代码或脚本注入。
- 检查是否存在针对上传文件的文件路径检查。特别是，黑客可能会将具有指定路径的文件压缩到 ZIP 中，以便在上传和解压缩后，可以将解压后的文件上传到预期路径。

## 修复

应用程序应开发为仅接受和操作“可接受”文件的机制，这些文件是应用程序其余功能准备好处理并预期的。一些具体示例包括：文件扩展名的黑名单或白名单、使用头部的“Content-Type”，或使用文件类型识别器，以仅允许指定文件类型进入系统。
