# 测试完整性检查

|ID          |
|------------|
|WSTG-BUSL-03|

## 概述

许多应用程序被设计为根据用户或情况显示不同的字段，使某些输入保持隐藏。然而，在许多情况下，可以通过代理将隐藏字段的值提交给服务器。在这些情况下，服务器端控制必须足够智能，以执行关系或服务器端编辑，确保基于用户和应用程序特定业务逻辑将适当的数据允许发送到服务器。

此外，应用程序不得依赖不可编辑控件、下拉菜单或隐藏字段来进行业务逻辑处理，因为这些字段仅在浏览器的上下文中保持不可编辑。用户可以使用代理编辑器工具编辑其值，并尝试操纵业务逻辑。如果应用程序将数量等与业务规则相关的值暴露为不可编辑字段，则必须在服务器端维护副本，并将其用于业务逻辑处理。最后，除了应用程序/系统数据外，日志系统也必须被保护，以防止读取、写入和更新。

业务逻辑完整性检查漏洞是独特的，因为这些误用案例是特定于应用程序的，如果用户能够进行更改，则根据业务流程逻辑，用户应仅在特定时间能够写入或更新/编辑特定工件。

应用程序必须足够智能以检查关系编辑，并且不允许用户直接向服务器提交无效、不受信任（因为来自不可编辑控件或用户未经授权通过前端提交）的信息。此外，系统工件（如日志）必须“受到保护”，防止未经授权的读取、写入和删除。

### 示例 1

想象一个 ASP.NET GUI 应用程序，只允许管理员用户更改系统中其他用户的密码。管理员用户将看到用户名和密码字段以输入用户名和密码，而其他用户将看不到这两个字段。然而，如果非管理员用户通过代理在用户名和密码字段中提交信息，他们可能能够“欺骗”服务器，使其认为请求来自管理员用户，并更改其他用户的密码。

### 示例 2

大多数 Web 应用程序都有下拉列表，方便用户快速选择其州、出生月份等。假设一个项目管理应用程序允许用户登录，并根据其权限向他们提供他们有权访问的项目下拉列表。如果攻击者找到了他们不应访问的另一个项目的名称，并通过代理提交信息，会发生什么？应用程序会给予对该项目的访问权限吗？即使他们跳过了授权业务逻辑检查，他们也不应拥有访问权限。

### 示例 3

假设机动车管理局要求雇员在签发身份证或驾驶执照时初步核实每位公民的文件和信息。此时，业务流程已创建具有高完整性的数据，因为提交数据的完整性由雇员检查。现在假设应用程序被迁移到互联网，以便雇员可以登录进行全服务操作，或公民可以登录进行有限的自助服务应用程序以更新某些信息。此时，攻击者可能能够使用拦截代理添加或更新他们不应访问的数据，并且他们可以通过声明该公民未婚但提供配偶姓名数据来破坏数据的完整性。这种插入或更新未经验证的数据会破坏数据完整性，如果遵循业务流程逻辑，则可能可以防止。

### 示例 4

许多系统包括用于审计和故障排除目的的日志记录。但是，这些日志中的信息有多好/有效？它们是否可能被攻击者有意或无意地操纵，从而破坏其完整性？

## 测试目标

- 审查项目文档，查找系统中移动、存储或处理数据的组件。
- 确定组件在逻辑上可接受的数据类型，以及系统应防范的类型。
- 确定在每个组件中应被允许修改或读取该数据的人。
- 尝试插入、更新或删除每个组件使用的、根据业务逻辑工作流不应被允许的数据值。

## 如何测试

### 具体测试方法 1

- 使用代理捕获 HTTP 流量，查找隐藏字段。
- 如果找到隐藏字段，查看这些字段与 GUI 应用程序的比较，并通过代理提交不同的数据值开始对这些值进行测试，尝试绕过业务流程并操纵您本不应访问的值。

### 具体测试方法 2

- 使用代理捕获 HTTP 流量，寻找将信息插入应用程序中不可编辑区域的位置。
- 如果找到，查看这些字段与 GUI 应用程序的比较，并通过代理提交不同的数据值开始对这些值进行测试，尝试绕过业务流程并操纵您本不应访问的值。

### 具体测试方法 3

- 列出可能受影响的应用程序或系统组件，例如日志或数据库。
- 对于每个已识别的组件，尝试读取、编辑或删除其信息。例如，应识别日志文件，测试人员应尝试操纵正在收集的数据/信息。

## 修复

应用程序应对如何修改和读取数据及工件以及通过确保数据完整性的可信渠道遵循严格的访问控制。应设置适当的日志记录以审查并确保没有发生未经授权的访问或修改。
