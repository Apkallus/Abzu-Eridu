# 测试 Web 消息传递

|ID          |
|------------|
|WSTG-CLNT-11|

## 概述

Web 消息传递（也称为[跨文档消息传递](https://html.spec.whatwg.org/multipage/web-messaging.html#web-messaging)）允许运行在不同域上的应用程序以安全方式进行通信。在引入 Web 消息传递之前，不同源（iframe、标签页和窗口之间）的通信受到同源策略的限制并由浏览器强制执行。开发人员使用多种技巧来完成这些任务，其中大多数主要是不安全的。

浏览器内的此限制是为了防止恶意网站从其他 iframe、标签页等读取机密数据；然而，在某些合法情况下，两个受信任的网站需要相互交换数据。为满足此需求，跨文档消息传递被引入到 [WHATWG HTML5](https://html.spec.whatwg.org/multipage/) 草案规范中，并已在所有主流浏览器中实现。它支持跨 iframe、标签页和窗口的多个源之间的安全通信。

消息传递 API 引入了 [`postMessage()` 方法](https://developer.mozilla.org/en-US/docs/Web/API/Window/postMessage)，通过该方法可以跨源发送明文消息。它由两个参数组成：消息和域。

当使用 `*` 作为域时，存在一些安全问题，我们将在下文中讨论。为了接收消息，接收网站需要添加一个新的事件处理程序，该处理程序具有以下属性：

- Data，传入消息的内容；
- Origin，发送方文档的来源；以及
- Source，源窗口。

以下是消息传递 API 的使用示例。发送消息：

```js
iframe1.contentWindow.postMessage("Hello world","https://www.example.com");
```

接收消息：

```js
window.addEventListener("message", handler, true);
function handler(event) {
    if(event.origin === 'chat.example.com') {
        /* 处理消息 (event.data) */
    } else {
        /* 忽略来自不受信任域的消息 */
    }
}
```

### 来源安全性

来源由协议、主机名和端口组成。它唯一标识发送或接收消息的域，并且不包括 URL 的路径或片段部分。例如，`https://example.com` 将被视为与 `http://example.com` 不同，因为前者的协议是 `https`，而后者是 `http`。这也适用于在同一域但不同端口上运行的 Web 服务器。

## 测试目标

- 评估消息来源的安全性。
- 验证其是否使用了安全方法并验证了输入。

## 如何测试

### 检查来源安全性

测试人员应检查应用程序代码是否过滤并处理来自受信任域的消息。在发送域内，还要确保明确指定了接收域，并且未将 `*` 用作 `postMessage()` 的第二个参数。这种做法可能引入安全问题，并可能导致在重定向或来源因其他方式发生变化时，网站将数据发送到未知主机，从而将机密数据泄露给恶意服务器。

如果网站未能添加安全控制来限制允许向网站发送消息的域或来源，则可能引入安全风险。测试人员应检查代码中的消息事件侦听器，并从 `addEventListener` 方法获取回调函数以进行进一步分析。必须在数据操作之前始终验证域。

### 检查输入验证

尽管网站理论上仅接受来自受信任域的消息，但数据仍必须被视为外部来源的、不可信的数据，并使用适当的安全控制进行处理。测试人员应分析代码并查找不安全的方法，特别是在通过 `eval()` 评估数据或通过 `innerHTML` 属性插入 DOM 的位置，这可能创建基于 DOM 的 XSS 漏洞。

### 静态代码分析

应分析 JavaScript 代码以确定 Web 消息传递的实现方式。特别是，测试人员应关注网站如何限制来自不受信任域的消息，以及即使对于受信任域，数据如何处理。

在此示例中，需要访问 owasp.org 域内的每个子域（www、chat、forums、...）。该代码试图接受任何包含 `.owasp.org` 的域：

```js
window.addEventListener("message", callback, true);

function callback(e) {
    if(e.origin.indexOf(".owasp.org")!=-1) {
        /* 处理消息 (e.data) */
    }
}
```

意图是允许如下子域：

- `www.owasp.org`
- `chat.owasp.org`
- `forums.owasp.org`

不幸的是，这引入了漏洞。攻击者可以轻松绕过过滤器，因为像 `www.owasp.org.attacker.com` 这样的域也会匹配。

以下是一个缺少来源检查的代码示例。这是非常不安全的，因为它将接受来自任何域的输入：

```js
window.addEventListener("message", callback, true);

function callback(e) {
        /* 处理消息 (e.data) */
}
```

以下是一个存在输入验证漏洞且可能导致 XSS 攻击的示例：

```js
window.addEventListener("message", callback, true);

function callback(e) {
        if(e.origin === "trusted.domain.com") {
            element.innerHTML= e.data;
        }
}
```

更安全的方法是使用属性 `innerText` 代替 `innerHTML`。

有关 Web 消息传递的更多 OWASP 资源，请参见 [OWASP HTML5 安全速查表](https://cheatsheetseries.owasp.org/cheatsheets/HTML5_Security_Cheat_Sheet.html)。
