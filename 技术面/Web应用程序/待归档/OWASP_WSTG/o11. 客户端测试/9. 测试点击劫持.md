# 测试点击劫持

|ID          |
|------------|
|WSTG-CLNT-09|

## 概述

点击劫持是UI伪装的一个子集，是一种恶意技术，通过该技术，Web用户被欺骗与其认为正在交互的对象之外的其他对象进行交互（在大多数情况下是点击）。这种攻击无论是单独使用还是与其他攻击结合使用，都可能在受害者与看似无害的网页交互时，潜在地发送未授权命令或泄露机密信息。“点击劫持”一词由Jeremiah Grossman和Robert Hansen于2008年创造。

点击劫持攻击利用HTML和JavaScript看似无害的功能，强制受害者执行非预期的操作，例如点击一个执行非预期操作的不可见按钮。这是一个影响多种浏览器和平台的客户端安全问题。

为了实施此攻击，攻击者创建一个看似无害的网页，该网页通过使用内联框架（用CSS代码隐藏）加载目标应用程序。完成后，攻击者可以通过其他方式（例如通过社会工程）诱导受害者与网页进行交互。与其他攻击一样，一个常见的前提是受害者已通过目标应用程序的身份验证。

![点击劫持示意图](images/Clickjacking_description.png)\
*图4.11.9-1：点击劫持内联框架示意图*

受害者浏览攻击者的网页，意图与可见的用户界面进行交互，但无意中在隐藏的网页上执行了操作。通过使用隐藏页面，攻击者可以通过在网页中定位隐藏元素来欺骗用户执行他们从未打算执行的操作。

![掩蔽的内联框架示意图](images/Masked_iframe.png)\
*图4.11.9-2：掩蔽的内联框架示意图*

此方法的力量在于，受害者执行的操作源自隐藏但真实的目标网页。因此，开发人员为保护网页免受CSRF攻击而部署的一些反CSRF防护措施可能被绕过。

## 测试目标

- 评估应用程序对点击劫持攻击的脆弱性。

## 如何测试

如上所述，此类攻击通常旨在允许攻击者诱导用户在目标站点上执行操作，即使使用了反CSRF令牌也是如此。

### 使用HTML iframe标签在HTML解释器中加载目标网页

未受框架破坏保护的站点容易受到点击劫持攻击。如果`https://www.target.site`网页成功加载到框架中，则该站点容易受到点击劫持攻击。用于创建此测试网页的HTML代码示例显示在以下代码段中：

```html
    <html>
        <head>
            <title>Clickjack测试网页</title>
        </head>
        <body>
            <iframe src="https://www.target.site" width="400" height="400"></iframe>
        </body>
    </html>
```

### 在禁用JavaScript的情况下测试应用程序

由于这类客户端保护依赖于JavaScript框架破坏代码，如果受害者禁用了JavaScript或攻击者能够禁用JavaScript代码，则该网页将没有任何针对点击劫持的保护机制。

有一些可以与框架一起使用的禁用技术。更深入的技术可以在[点击劫持防御速查表](https://cheatsheetseries.owasp.org/cheatsheets/Clickjacking_Defense_Cheat_Sheet.html)中找到。

### Sandbox属性

HTML5引入了一个名为“sandbox”的新属性。它对加载到iframe中的内容启用一组限制。

示例：

```html
<iframe src="https://example.org" sandbox></iframe>
```

### 在兼容性和无障碍模式下测试应用程序

网页的移动版本通常比桌面版本更小、更快，并且必须比主应用程序更简单。移动变体通常保护较少。然而，攻击者可以伪造Web浏览器给出的真实来源，非移动受害者可能能够访问为移动用户制作的应用程序。这种情况可能允许攻击者利用网页的移动版本。

在无障碍模式下运行的应用程序也应针对点击劫持进行测试，因为站点框架可能会受到影响。

### 服务器端保护：使用内容安全策略的Frame-Ancestors指令

HTTP Content-Security-Policy（CSP）响应头允许网页管理员控制用户代理允许为给定网页加载的资源。HTTP CSP中的`frame-ancestors`指令指定了可使用`<frame>`、`<iframe>`、`<object>`、`<embed>`或`<applet>`标签嵌入网页的可接受父级。

#### 测试Content-Security-Policy响应头

- 使用浏览器，打开开发者工具并访问目标网页。导航到“网络”选项卡。
- 查找加载该网页的请求。它应具有与网页相同的域——通常是“网络”选项卡上的第一项。
- 点击该文件后，将显示更多信息。查找200 OK响应码。
- 向下滚动到响应头部分。Content-Security-Policy部分指示所采用的保护级别。

或者，查看网页源代码以在meta标签中找到Content-Security-Policy。WSTG在[测试Content-Security-Policy](../02-Configuration_and_Deployment_Management_Testing/12-Test_for_Content_Security_Policy.md)中有详细信息。

##### 代理

众所周知，Web代理会添加和剥离头部。如果Web代理剥离了`X-FRAME-OPTIONS`头，则站点将失去其框架保护。

##### 应用程序的移动版本

在这种情况下，由于`X-FRAME-OPTIONS` HTTP头必须在应用程序的每个页面中实现，开发人员可能没有保护移动版本上的每个页面。
