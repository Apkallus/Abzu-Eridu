# 测试客户端模板注入

|ID          |
|------------|
|WSTG-CLNT-15|

## 概述

客户端模板注入（CSTI），也称为基于 DOM 的模板注入，当使用客户端框架（如 Angular、Vue.js 或 Alpine.js）的应用程序将用户输入动态嵌入到网页的 DOM 中时，就会出现此问题。如果该输入被嵌入到模板表达式中或被框架的模板引擎解释，攻击者就可以注入恶意指令。

与模板在服务器端渲染的[服务器端模板注入（SSTI）](../07-Input_Validation_Testing/18-Testing_for_Server-side_Template_Injection.md)不同，CSTI 完全发生在用户浏览器中。当框架扫描 DOM 以查找动态内容时，它可能会执行注入的模板表达式。这通常会导致跨站脚本（XSS），但注入和利用的方法与标准 XSS 不同，因为载荷必须遵循模板引擎的特定语法（例如 {% raw %}`{{ 7*7 }}`{% endraw %}）。

此漏洞在单页应用程序（SPA）中尤为常见，开发人员可能依赖客户端渲染而没有严格的上下文分隔。

## 测试目标

- 识别应用程序使用的客户端框架及其版本。
- 检测用户输入被反射到 DOM 并由模板引擎处理的注入点。
- 评估是否允许通过模板语法执行任意 JavaScript（XSS）。

## 如何测试

### 黑盒测试

#### 框架识别

第一步是确定是否使用了客户端框架。测试人员应在 HTML 源代码或特定的 HTTP 响应头中查找特定属性。

- **Angular：** 查找诸如 `ng-app`、`ng-model` 或 `ng-bind` 等属性。
- **Vue.js：** 查找以 `v-` 开头的属性，如 `v-if`、`v-for`，或在控制台中查找 Vue 全局对象。
- **Alpine.js：** 查找 `x-data`、`x-html`。

#### 注入检测

为了检测 CSTI，测试人员应注入对模板引擎具有语法意义的字符。许多框架中最常见的插值语法是双花括号 {% raw %}`{{ }}`{% endraw %}。

简单的算术运算是标准的探测手段。如果应用程序计算了数学表达式，则存在漏洞。

**通用探测：**

{% raw %}

```txt
注入字符串：{{ 7*7 }}
```

{% endraw %}

- 如果应用程序渲染为 `49`，则 CSTI 存在。
- 如果应用程序渲染为 {% raw %}`{{ 7*7 }}`{% endraw %}，则很可能不存在漏洞，或实施了严格的上下文转义。

#### Angular

Angular 作用于 DOM。如果攻击者能够在 Angular 启动或编译之前将包含 Angular 表达式的字符串注入 DOM，则该表达式将被执行。

**检测载荷：**

{% raw %}

- `{{ 7*7 }}`
- `{{ constructor.constructor('alert(1)')() }}`

{% endraw %}

在旧版本的 Angular（1.x）中，模板引擎在沙盒中工作。利用需要突破此沙盒。载荷的复杂性在很大程度上取决于具体的版本。

**示例载荷（Angular 1.5.x 沙盒绕过）：**

{% raw %}

```javascript
{{x={'y':''.constructor.prototype};x['y'].charAt=[].join;$eval('x=alert(1)');}}
```

{% endraw %}

#### Vue.js

如果开发人员将 `v-html` 指令与用户输入一起使用，或者将 Vue 实例挂载到已包含用户控制 HTML 的 DOM 元素上，则 Vue.js 也容易受到攻击。

**检测载荷：**

{% raw %}

- `{{ 7*7 }}`

{% endraw %}

**示例载荷（Vue.js 2.x）：**

{% raw %}

```javascript
{{_v.constructor('alert(1)')()}}
```

{% endraw %}

#### Alpine.js

Alpine.js 严重依赖 DOM 属性。如果攻击者可以控制属性名称或注入到指令中，他们就可以执行代码。

**示例载荷：**

```html
<div x-data="" x-html="'<img src=x onerror=alert(1)>'"></div>
```

### 灰盒测试

#### 代码审查

在灰盒场景中，测试人员验证前端代码中如何处理用户输入。关键在于识别在何处将解释 HTML 或模板语法的“接收器”与“受污染”源（用户输入）一起使用。

**Angular 接收器：**
查找 `$compile` 或 `ng-bind-html` 的使用。
如果使用了 `ng-bind-html`，请检查 `$sce`（严格上下文转义）是否配置正确，或者是否对不可信数据使用了 `$sce.trustAsHtml()`。

```javascript
// Angular 中的易受攻击示例
$scope.htmlSnippet = $sce.trustAsHtml(userInput);
```

**Vue.js 接收器：**
查找 `v-html` 指令。对用户提供的内容使用 `v-html` 是 Vue 中 CSTI/XSS 的主要原因。

```html
<div v-html="userProvidedComment"></div>
```

**React 接收器：**
虽然 React 在模板注入方面通常更安全，因为它不会扫描 DOM 中的模板（JSX 是编译的），但不当使用 `dangerouslySetInnerHTML` 会导致基于 DOM 的 XSS，这与本文讨论的风险配置类似。

{% raw %}

```javascript
// React 中的易受攻击示例
<div dangerouslySetInnerHTML={{__html: userContent}} />
```

{% endraw %}

#### 配置审查

特别检查内容安全策略（CSP）头部。强大的 CSP 可以通过限制脚本的加载源或阻止内联脚本（包括模板引擎生成的脚本）的执行来减轻 CSTI 的影响。

在 CSP 中查找 `unsafe-eval`。许多模板引擎（尤其是较旧的引擎）需要 `unsafe-eval` 来动态编译模板。如果存在 `unsafe-eval`，则 CSTI 攻击将更容易被利用。

## 修复

- **避免将用户输入渲染为 HTML：** 尽可能使用将输入视为纯文本的安全指令。
    - Angular：使用 `ng-bind` 代替 `ng-bind-html`。
    - Vue：使用 `v-text` 或花括号 {% raw %}`{{ }}`{% endraw %}（自动转义 HTML）代替 `v-html`。
    - React：避免使用 `dangerouslySetInnerHTML`。
- **清理：** 如果需要 HTML 渲染，请在将数据传递给框架之前，使用专用的清理库（如 DOMPurify）来剥离危险的标签和属性。
- **内容安全策略（CSP）：** 实施严格的 CSP，禁用 `unsafe-eval` 并限制脚本源。
- **使用离线编译：** 对于 Vue 或 React 等框架，优先使用构建步骤（webpack、vite）提前编译模板，而不是使用解析 DOM 内容的运行时编译器。

### 工具

- [DOMPurify](https://github.com/cure53/DOMPurify)：一个仅适用于 DOM、超快速、高容错性的 XSS 清理器，用于 HTML、MathML 和 SVG。
