# 测试敏感信息是否通过未加密通道发送

|ID          |
|------------|
|WSTG-CRYP-03|

## 概述

敏感数据在网络中传输时必须受到保护。如果数据通过 HTTPS 传输或以其他方式加密，则保护机制不得存在限制或漏洞，正如更广泛的文章[测试弱传输层安全](01-Testing_for_Weak_Transport_Layer_Security.md)和其他 OWASP 文档中所述：

- [OWASP Top 10 2017 A3-敏感数据暴露](https://owasp.org/www-project-top-ten/2017/A3_2017-Sensitive_Data_Exposure)。
- [OWASP ASVS - 验证 V9](https://github.com/OWASP/ASVS/blob/master/4.0/en/0x17-V9-Communications.md)。
- [传输层保护速查表](https://cheatsheetseries.owasp.org/cheatsheets/Transport_Layer_Protection_Cheat_Sheet.html)。

作为经验法则，如果数据在存储时需要保护，则其在传输期间也必须受到保护。敏感数据的一些示例如下：

- 身份验证中使用的信息（例如凭据、PIN、会话标识符、令牌、Cookie……）
- 受法律、法规或特定组织政策保护的信息（例如信用卡、客户数据）

如果应用程序通过未加密通道（例如 HTTP）传输敏感信息，则被视为安全风险。攻击者可以通过[嗅探网络流量](https://owasp.org/www-community/attacks/Manipulator-in-the-middle_attack)接管账户。一些示例包括通过 HTTP 以明文发送身份验证凭据的基本认证、通过 HTTP 发送的基于表单的认证凭据，或任何其他因法规、法律、组织政策或应用程序业务逻辑而被视为敏感的信息的明文传输。

个人身份信息（PII）的示例包括：

- 社会安全号码
- 银行账号
- 护照信息
- 健康相关信息
- 医疗保险信息
- 学生信息
- 信用卡和借记卡号码
- 驾照和州身份证信息

## 测试目标

- 识别通过各种通道传输的敏感信息。
- 评估所使用通道的隐私和安全性。

## 如何测试

需要受保护的各种类型信息可能由应用程序以明文形式传输。为了检查此信息是否通过 HTTP 而非 HTTPS 传输，请在需要凭据的客户端和 Web 应用服务器之间捕获流量。对于任何包含敏感数据的消息，验证交换是否使用了 HTTPS。有关不安全传输凭据的更多信息，请参见 [OWASP Top 10 2017 A3-敏感数据暴露](https://owasp.org/www-project-top-ten/2017/A3_2017-Sensitive_Data_Exposure) 或 [传输层保护速查表](https://cheatsheetseries.owasp.org/cheatsheets/Transport_Layer_Protection_Cheat_Sheet.html)。

### 示例 1：通过 HTTP 的基本认证

一个典型示例是通过 HTTP 使用基本认证。使用基本认证时，用户凭据被编码而非加密，并作为 HTTP 头部发送。在以下示例中，测试人员使用 [curl](https://curl.haxx.se/) 来测试此问题。注意应用程序如何使用基本认证以及 HTTP 而非 HTTPS。

```bash
$ curl -kis http://example.com/restricted/
HTTP/1.1 401 Authorization Required
Date: Fri, 01 Aug 2013 00:00:00 GMT
WWW-Authenticate: Basic realm="Restricted Area"
Accept-Ranges: bytes Vary:
Accept-Encoding Content-Length: 162
Content-Type: text/html

<html><head><title>401 Authorization Required</title></head>
<body bgcolor=white> <h1>401 Authorization Required</h1>  Invalid login credentials!  </body></html>
```

### 示例 2：通过 HTTP 执行的基于表单的身份验证

另一个典型示例是通过 HTTP 传输用户身份验证凭据的身份验证表单。在以下示例中，可以看到在表单的 `action` 属性中使用了 HTTP。通过使用拦截代理检查 HTTP 流量也可以发现此问题。

```html
<form action="http://example.com/login">
    <label for="username">用户：</label> <input type="text" id="username" name="username" value=""/><br />
    <label for="password">密码：</label> <input type="password" id="password" name="password" value=""/>
    <input type="submit" value="登录"/>
</form>
```

### 示例 3：通过 HTTP 发送包含会话 ID 的 Cookie

会话 ID Cookie 必须通过受保护的通道传输。如果 Cookie 未设置 [secure 标志](../06-Session_Management_Testing/02-Testing_for_Cookies_Attributes.md)，则允许应用程序以未加密方式传输它。注意以下 Cookie 的设置未使用 Secure 标志，并且整个登录过程通过 HTTP 而非 HTTPS 进行。

```http
https://secure.example.com/login

POST /login HTTP/1.1
Host: secure.example.com
[...]
Referer: https://secure.example.com/
Content-Type: application/x-www-form-urlencoded
Content-Length: 188

HTTP/1.1 302 Found
Date: Tue, 03 Dec 2013 21:18:55 GMT
Server: Apache
Set-Cookie: JSESSIONID=BD99F321233AF69593EDF52B123B5BDA; expires=Fri, 01-Jan-2014 00:00:00 GMT; path=/; domain=example.com; httponly
Location: private/
Content-Length: 0
Content-Type: text/html
```

```http
http://example.com/private

GET /private HTTP/1.1
Host: example.com
[...]
Referer: https://secure.example.com/login
Cookie: JSESSIONID=BD99F321233AF69593EDF52B123B5BDA;

HTTP/1.1 200 OK
Content-Type: text/html;charset=UTF-8
Content-Length: 730
Date: Tue, 25 Dec 2013 00:00:00 GMT
```

### 示例 4：密码重置、更改密码或其他账户操作通过 HTTP 进行

如果 Web 应用程序具有允许用户更改账户或使用凭据调用其他服务的功能，请验证所有这些交互是否都使用 HTTPS。要测试的交互包括：

- 允许用户处理忘记密码或其他凭据的表单
- 允许用户编辑凭据的表单
- 要求用户使用其他提供商（例如支付处理）进行身份验证的表单

### 示例 5：测试源代码或日志中的密码敏感信息

使用以下技术之一搜索敏感信息。

检查密码或加密密钥是否硬编码在源代码或配置文件中。

`grep -r –E "Pass | password | pwd |user | guest| admin | encry | key | decrypt | sharekey " ./PathToSearch/`

检查日志或源代码是否可能包含电话号码、电子邮件地址、ID 或任何其他 PII。根据 PII 的格式更改正则表达式。

`grep -r " {2\}[0-9]\{6\} "  ./PathToSearch/`

## 修复

为整个网站使用 HTTPS，并将任何 HTTP 请求重定向到 HTTPS。
