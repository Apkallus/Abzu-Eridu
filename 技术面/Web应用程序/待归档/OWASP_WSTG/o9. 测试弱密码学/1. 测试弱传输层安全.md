# 测试弱传输层安全

|ID          |
|------------|
|WSTG-CRYP-01|

## 概述

当信息在客户端和服务器之间传输时，必须对其进行加密和保护，以防止攻击者读取或修改这些信息。这通常通过 HTTPS 实现，HTTPS 使用[传输层安全（TLS）](https://en.wikipedia.org/wiki/Transport_Layer_Security)协议，该协议是旧版安全套接字层（SSL）协议的替代品。TLS 还提供了一种方式，使服务器能够通过出示可信数字证书向客户端证明它们连接到了正确的服务器。

多年来，在 SSL 和 TLS 协议及其使用的密码套件中发现了大量加密弱点。此外，这些协议的许多实现也存在严重漏洞。因此，测试站点不仅是否实现了 TLS，而且是否以安全方式实现是非常重要的。

## 测试目标

- 验证服务配置。
- 审查数字证书的加密强度和有效性。
- 确保 TLS 安全性不可绕过，并且在应用程序中得到了正确实施。

## 如何测试

传输层安全相关问题可大致分为以下领域：

### 服务器配置

TLS 支持大量协议版本、密码套件和扩展。其中许多被认为是遗留的，并且存在加密弱点，如下所列。请注意，随着时间的推移，可能会发现新的弱点，因此此列表可能不完整。

- [SSLv2（DROWN）](https://drownattack.com/)
- [SSLv3（POODLE）](https://en.wikipedia.org/wiki/POODLE)
- [TLSv1.0（BEAST）](https://www.acunetix.com/blog/web-security-zone/what-is-beast-attack/)
- [TLSv1.1（已被 RFC 8996 弃用）](https://tools.ietf.org/html/rfc8996)
- [EXPORT 密码套件（FREAK）](https://en.wikipedia.org/wiki/FREAK)
- NULL 密码套件（[它们仅提供身份验证](https://tools.ietf.org/html/rfc4785)）。
- 匿名密码套件（SMTP 服务器可能支持这些，如 [RFC 7672](https://tools.ietf.org/html/rfc7672#section-8.2) 中所述）
- [RC4 密码套件（NOMORE）](https://www.rc4nomore.com/)
- CBC 模式密码套件（BEAST、[Lucky 13](https://en.wikipedia.org/wiki/Lucky_Thirteen_attack)）
- [TLS 压缩（CRIME）](https://en.wikipedia.org/wiki/CRIME)
- [弱 DHE 密钥（LOGJAM）](https://weakdh.org/)

[Mozilla 服务器端 TLS 指南](https://wiki.mozilla.org/Security/Server_Side_TLS)详细说明了当前推荐的协议和密码套件。由于 TLS 1.3 与后量子密码套件的兼容性，应优先于 TLS 1.2 使用。

#### 可利用性

需要强调的是，尽管其中许多攻击已在实验室环境中得到验证，但通常不被认为在实际世界中易于利用，因为它们通常需要（主动）中间人攻击和大量资源。因此，除了国家级行为者外，其他人不太可能利用这些漏洞。

### 数字证书

#### 加密弱点

从加密角度来看，数字证书需要审查两个主要领域：证书本身的签名和 TLS 握手期间使用的密钥交换算法。使用经典加密时，应考虑以下几点：

- 使用的密码强度应至少为 128 位安全强度或更高。
- 对于经典签名，应至少使用 RSA-3072 或 ECDSA（P-256）配合 SHA-256 或更强的组合。
- 对于经典密钥交换，应至少使用 ECDHE（P-256）或 X25519 椭圆曲线。

此外，签名和密钥交换算法都应支持后量子加密。应考虑以下几点：

- 对于后量子签名，应至少使用 ML-DSA-44（根据 [FIPS-204](https://csrc.nist.gov/pubs/fips/204/final)）或更高版本。
- 对于后量子密钥交换，应至少使用 ML-KEM-512（根据 [FIPS-203](https://csrc.nist.gov/pubs/fips/203/final)）或更高版本。
- 在后量子算法过渡期间，可以使用混合密码套件，如 X25519+ML-KEM-768。

#### 有效性

除了加密安全之外，证书还必须被认为是有效（或可信）的。这意味着它必须：

- 在定义的有效期内。
    - 2020 年 9 月 1 日之后颁发的任何证书的最长有效期不得超过 [398 天](https://blog.mozilla.org/security/2020/07/09/reducing-tls-certificate-lifespans-to-398-days/)。
    - 证书的有效期将逐渐缩短，到 2029 年 3 月最大寿命为 [47 天](https://github.com/cabforum/servercert/blob/main/docs/BR.md#11-overview)。
- 由受信任的证书颁发机构（CA）签名。
    - 对于面向外部的应用程序，这应是受信任的公共 CA；对于内部应用程序，应是内部 CA。
    - 不要仅仅因为*您的*系统不信任 CA 而将内部应用程序标记为具有不受信任的证书。
- 具有与系统主机名匹配的主题备用名称（SAN）。
    - 现代浏览器忽略通用名称（CN）字段，仅查看 SAN。
    - 确保您使用正确的名称访问系统（例如，如果您通过 IP 访问主机，则任何证书都将显示为不受信任）。

某些证书可能颁发给通配符域（例如 `*.example.org`），这意味着它们对多个子域有效。尽管方便，但围绕此存在许多安全问题需要考虑。这些在 [OWASP 传输层安全速查表](https://cheatsheetseries.owasp.org/cheatsheets/Transport_Layer_Protection_Cheat_Sheet.html#carefully-consider-the-use-of-wildcard-certificates)中进行了讨论。

证书还可能在 Issuer 和 SAN 字段中泄露有关内部系统或域名的信息，这在尝试构建内部网络图或进行社会工程活动时可能很有用。

### 实现漏洞

多年来，各种 TLS 实现中存在漏洞。此处无法一一列举，但一些关键示例如下：

- [Debian OpenSSL 可预测随机数生成器](https://www.debian.org/security/2008/dsa-1571)（CVE-2008-0166）
- [OpenSSL 不安全重新协商](https://www.openssl.org/news/secadv/20091111.txt)（CVE-2009-3555）
- [OpenSSL Heartbleed](https://heartbleed.com)（CVE-2014-0160）
- [F5 TLS POODLE](https://support.f5.com/csp/article/K15882)（CVE-2014-8730）
- [Microsoft Schannel 拒绝服务](https://docs.microsoft.com/en-us/security-updates/securitybulletins/2014/ms14-066)（MS14-066 / CVE-2014-6321）

### 应用程序漏洞

除了底层 TLS 配置安全外，应用程序也需要以安全方式使用它。其中一些要点在本指南的其他部分中有所涉及：

- [设置 HTTP Strict-Transport-Security 头部（WSTG-CONF-07）](../02-Configuration_and_Deployment_Management_Testing/07-Test_HTTP_Strict_Transport_Security.md)

#### 混合活动内容

混合活动内容是指活动资源（如脚本到 CSS）通过未加密的 HTTP 加载，并包含在安全（HTTPS）页面中。这是危险的，因为攻击者可以修改这些文件（因为它们未加密发送），从而允许他们在页面中执行任意代码（JavaScript 或 CSS）。通过不安全连接加载的被动内容（如图像）也可能泄露信息或允许攻击者篡改页面，但不太可能导致完全沦陷。

> 注意：现代浏览器会阻止从非安全源向安全页面加载活动内容。

#### 从 HTTP 重定向到 HTTPS

许多站点会接受未加密 HTTP 的连接，然后立即通过 `301 Moved Permanently` 重定向将用户重定向到站点的安全（HTTPS）版本。站点的 HTTPS 版本随后设置 `Strict-Transport-Security` 头部，指示浏览器将来始终使用 HTTPS。

然而，如果攻击者能够拦截此初始请求，他们可以将用户重定向到恶意站点，或使用 [sslstrip](https://github.com/moxie0/sslstrip) 等工具拦截后续请求。

为了防御此类攻击，站点必须添加到[预加载列表](https://hstspreload.org)中。

## 自动化测试

有大量扫描工具可用于识别服务 SSL/TLS 配置中的弱点，包括专用工具和通用漏洞扫描器。一些较流行的工具包括：

- [Nmap](https://nmap.org)（各种脚本）
- [OWASP O-Saft](https://owasp.org/www-project-o-saft/)
- [sslscan](https://github.com/rbsec/sslscan)
- [sslyze](https://github.com/nabla-c0d3/sslyze)
- [SSL Labs](https://www.ssllabs.com/ssltest/)
- [testssl.sh](https://github.com/drwetter/testssl.sh)

### 手动测试

也可以使用命令行工具（如 `openssl s_client` 或 `gnutls-cli`）手动执行大多数检查，以使用特定协议、密码套件或选项进行连接。

进行此类测试时，请注意大多数现代系统附带的 OpenSSL 或 GnuTLS 版本可能不支持某些过时且不安全的协议，如 SSLv2 或 EXPORT 密码套件。在用于测试之前，请确保您的版本支持这些过时版本，否则会出现漏报。

也可以使用 Web 浏览器进行有限的测试，因为现代浏览器会在其开发者工具中提供正在使用的协议和密码套件的详细信息。它们还提供了一种简单的方法来测试证书是否被视为可信，即通过浏览服务并查看是否出现证书警告。
