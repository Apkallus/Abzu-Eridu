# 测试填充预言机

|ID          |
|------------|
|WSTG-CRYP-02|

## 概述

填充预言机是应用程序的一个功能，它解密客户端提供的加密数据（例如存储在客户端的内部会话状态），并在解密后泄露填充有效性的状态。填充预言机的存在允许攻击者在不知道用于这些加密操作的密钥的情况下解密加密数据并加密任意数据。如果应用程序假设了加密数据的完整性，这可能导致敏感数据泄露或权限提升漏洞。

分组密码仅以特定大小的块加密数据。常用密码的块大小为 8 字节和 16 字节。当数据大小不是所用密码块大小的倍数时，必须以特定方式进行填充，以便解密器能够去除填充。一种常用的填充方案是 PKCS#7。它用填充长度的值填充剩余字节。

### 示例 1

如果填充长度为 5 字节，则在明文之后重复五次字节值 `0x05`。

如果填充不符合所用填充方案的语法，则存在错误条件。如果应用程序针对客户端提供的加密数据泄露了此特定填充错误条件，则存在填充预言机。这可能通过直接暴露异常（例如 Java 中的 `BadPaddingException`）、发送给客户端的响应的细微差异或定时行为等其他侧信道发生。

某些加密操作模式允许位翻转攻击，其中密文中翻转一个位会导致明文中的该位也被翻转。在 CBC 加密数据的第 n 个块中翻转一个位会导致解密数据中第 (n+1) 个块中的相同位被翻转。解密后的密文的第 n 个块会因这种操作而被破坏。

填充预言机攻击使攻击者能够在不知道加密密钥和所用密码的情况下，通过向填充预言机发送精心构造的密文并观察返回的结果来解密加密数据。这会导致加密数据的机密性丧失。例如，在客户端存储会话数据的情况下，攻击者可以获取有关应用程序内部状态和结构的信息。

填充预言机攻击还使攻击者能够在不知道所用密钥和密码的情况下加密任意明文。如果应用程序假设解密数据的完整性和真实性是给定的，则攻击者可能能够操纵内部会话状态并可能获得更高权限。

## 测试目标

- 识别依赖填充的加密消息。
- 尝试破坏加密消息的填充，并分析返回的错误消息以进行进一步分析。

## 如何测试

### 黑盒测试

首先，必须识别填充预言机的可能输入点。通常必须满足以下条件：

1. 数据已加密。好的候选者是看起来随机的值。
2. 使用了分组密码。解码后的密文（通常使用 Base64）长度是常见密码块大小（如 8 或 16 字节）的倍数。不同的密文（例如通过不同会话或操纵会话状态收集的）在长度上具有共同的除数。

#### 示例 2

`Dg6W8OiWMIdVokIDH15T/A==` 经过 Base64 解码后为 `0e 0e 96 f0 e8 96 30 87 55 a2 42 03 1f 5e 53 fc`。这看起来是随机的，长度为 16 字节。

如果识别出这样的输入值候选者，则应验证应用程序对加密值进行逐位篡改时的行为。通常，这个 Base64 编码的值将包括前置在密文前面的初始化向量（IV）。给定明文 *`p`* 和块大小为 *`n`* 的密码，块数为 *`b = ceil( length(p) / n )`*。由于初始化向量的存在，加密字符串的长度为 *`y = (b+1) * n`*。要验证预言机的存在，请解码该字符串，翻转倒数第二个块 *`b-1`* 的最后一位（位于 *`y-n-1`* 处字节的最低有效位），重新编码并发送。接下来，解码原始字符串，翻转块 *`b-2`* 的最后一位（位于 *`y-2*n-1`* 处字节的最低有效位），重新编码并发送。

如果已知加密字符串是单个块（IV 存储在服务器端，或应用程序使用了不良实践的硬编码 IV），则必须依次执行多次位翻转。另一种方法可以是预先添加一个随机块，并翻转位以使添加的块的最后一个字节取所有可能的值（0 到 255）。

测试和基础值应至少导致解密期间和解密后的三种不同状态：

- 密文被解密，生成的数据正确。
- 密文被解密，生成的数据被破坏并导致应用程序逻辑中的某些异常或错误处理。
- 由于填充错误，密文解密失败。

仔细比较响应。特别搜索指出填充有问题的异常和消息。如果出现此类消息，则应用程序包含填充预言机。如果可以隐式观察到上述三种不同状态（不同的错误消息、定时侧信道），则此处很可能存在填充预言机。尝试执行填充预言机攻击以确认这一点。

##### 示例 3

- 如果解密密文的填充被破坏，ASP.NET 会抛出 `System.Security.Cryptography.CryptographicException: Padding is invalid and cannot be removed.`
- 在 Java 中，这种情况下会抛出 `javax.crypto.BadPaddingException`。
- 解密错误或类似错误可能是填充预言机。

> 安全的实现将检查完整性，并且仅产生两种响应：`ok` 和 `failed`。没有可用于确定内部错误状态的侧信道。

### 灰盒测试

验证所有解密的、来自客户端且应仅由服务器知道的加密数据的位置。此类代码应满足以下条件：

1. 密文的完整性应通过安全机制进行验证，如 HMAC 或经过身份验证的密码操作模式（如 GCM 或 CCM）。
2. 解密和进一步处理期间的所有错误状态都应统一处理。
