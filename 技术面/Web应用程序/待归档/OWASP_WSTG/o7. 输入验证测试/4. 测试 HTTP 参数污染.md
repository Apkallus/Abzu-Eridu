# 测试 HTTP 参数污染

|ID          |
|------------|
|WSTG-INPV-04|

## 概述

HTTP 参数污染测试应用程序对接收多个同名 HTTP 参数时的响应；例如，如果参数 `username` 在 GET 或 POST 参数中出现两次。

提供多个同名的 HTTP 参数可能导致应用程序以非预期方式解释值。通过利用这些影响，攻击者可能能够绕过输入验证、触发应用程序错误或修改内部变量值。由于 HTTP 参数污染（简称 *HPP*）影响所有 Web 技术的基础构建块，因此存在服务器端和客户端攻击。

当前的 HTTP 标准未包含关于如何解释多个同名输入参数的指导。例如，[RFC 3986](https://www.ietf.org/rfc/rfc3986.txt) 仅将 *查询字符串* 定义为一组字段-值对，而 [RFC 2396](https://www.ietf.org/rfc/rfc2396.txt) 定义了保留和未保留的查询字符串字符类别。由于缺乏标准，Web 应用程序组件以多种方式处理此边缘情况（详见下表）。

这本身不一定表示存在漏洞。然而，如果开发人员未意识到此问题，重复参数的存在可能会在应用程序中产生异常行为，攻击者可能利用该行为。与安全中常见的情况一样，意外行为通常是弱点的常见来源，在此情况下可能导致 HTTP 参数污染攻击。为了更好地介绍此类漏洞和 HPP 攻击的结果，分析过去发现的一些实际示例是很有趣的。

### 输入验证和过滤器绕过

2009 年，在首次关于 HTTP 参数污染的研究发表后不久，该技术作为可能绕过 Web 应用防火墙的方式受到了安全社区的关注。

其中一个影响 *ModSecurity SQL Injection Core Rules* 的缺陷，是应用程序和过滤器之间阻抗不匹配的完美示例。ModSecurity 过滤器会正确地对以下字符串应用黑名单：`select 1,2,3 from table`，从而阻止该示例 URL 被 Web 服务器处理：`/index.aspx?page=select 1,2,3 from table`。然而，通过利用多个 HTTP 参数的拼接，攻击者可以在 ModSecurity 过滤器已接受输入之后，使应用服务器拼接字符串。例如，URL `/index.aspx?page=select 1&page=2,3 from table` 不会触发 ModSecurity 过滤器，但应用层会将输入拼接回完整的恶意字符串。

另一个 HPP 漏洞影响了 *Apple Cups*，这是许多 Unix 系统使用的著名打印系统。利用 HPP，攻击者可以轻松地使用以下 URL 触发跨站脚本漏洞：`https://127.0.0.1:631/admin/?kerberos=onmouseover=alert(1)&kerberos`。通过添加一个具有有效字符串（例如空字符串）的额外 `kerberos` 参数，可以绕过应用程序的验证检查点。由于验证检查点仅考虑第二次出现，第一个 `kerberos` 参数在用于生成动态 HTML 内容之前未经过适当清理。成功利用将导致在托管站点的上下文中执行 JavaScript 代码。

### 认证绕过

在流行的博客平台 *Blogger* 中发现了一个更严重的 HPP 漏洞。该漏洞允许恶意用户使用以下 HTTP 请求（`https://www.blogger.com/add-authors.do`）获取受害者博客的所有权：

```html
POST /add-authors.do HTTP/1.1
[...]

security_token=attackertoken&blogID=attackerblogidvalue&blogID=victimblogidvalue&authorsList=goldshlager19test%40gmail.com(attacker email)&ok=Invite
```

该缺陷存在于 Web 应用程序使用的身份验证机制中，因为安全检查是在第一个 `blogID` 参数上执行的，而实际操作使用了第二次出现。

## 测试目标

- 识别后端及其使用的解析方法。
- 评估注入点并尝试使用 HPP 绕过输入过滤器。

## 如何测试

幸运的是，HTTP 参数的分配通常由 Web 应用服务器处理，而不是由应用程序代码本身处理，因此对参数污染响应的测试在所有页面和操作中应是标准的。然而，由于需要深入的业务逻辑知识，测试 HPP 需要手动测试。自动化工具只能部分协助审计人员，因为它们倾向于产生过多误报。此外，HPP 可以出现在客户端和服务器端组件中。

### 服务器端 HPP

要测试 HPP 漏洞，请识别任何允许用户提供输入的表单或操作。HTTP GET 请求中的查询字符串参数可以轻松地在浏览器的导航栏中修改。如果表单操作通过 POST 提交数据，测试人员将需要使用拦截代理在数据发送到服务器时篡改 POST 数据。在识别出要测试的特定输入参数后，可以通过拦截请求来编辑 GET 或 POST 数据，或者在响应页面加载后更改查询字符串。要测试 HPP 漏洞，只需将相同的参数附加到 GET 或 POST 数据中，但赋予不同的值。

例如：如果测试查询字符串中的 `search_string` 参数，请求 URL 将包含该参数名称和值：

```text
https://example.com/?search_string=kittens
```

特定参数可能隐藏在其他几个参数中，但方法相同；保留其他参数，并附加重复的参数：

```text
https://example.com/?mode=guest&search_string=kittens&num_results=100
```

使用不同的值附加相同的参数：

```text
https://example.com/?mode=guest&search_string=kittens&num_results=100&search_string=puppies
```

对于基于 JSON 的端点，载荷将如下所示：

```http
POST /search HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "search_string": "kittens",
  "search_string": "puppies"
}
```

然后提交新请求。

分析响应页面以确定解析了哪个（些）值。在上述示例中，搜索结果可能显示 `kittens`、`puppies`、两者的某种组合（`kittens,puppies` 或 `kittens~puppies` 或 `['kittens','puppies']`），可能返回空结果或错误页面。

此行为，无论使用第一个、最后一个还是同名输入参数的组合，很可能在整个应用程序中是一致的。此默认行为是否揭示潜在漏洞取决于特定应用程序的特定输入验证和过滤。作为一般规则：如果现有输入验证和其他安全机制对单个输入足够，并且服务器仅分配第一个或最后一个被污染的参数，则参数污染不会揭示漏洞。如果重复参数被拼接、不同的 Web 应用程序组件使用不同的出现项，或者测试产生错误，则利用参数污染触发安全漏洞的可能性更高。

更深入的分析需要对每个 HTTP 参数执行三个 HTTP 请求：

1. 提交包含标准参数名称和值的 HTTP 请求，并记录 HTTP 响应。例如 `page?par1=val1`
2. 将参数值替换为篡改的值，提交并记录 HTTP 响应。例如 `page?par1=HPP_TEST1`
3. 发送结合步骤（1）和（2）的新请求。再次保存 HTTP 响应。例如 `page?par1=val1&par1=HPP_TEST1`
4. 比较之前所有步骤中获得的响应。如果（3）的响应与（1）不同，并且（3）的响应也与（2）不同，则存在阻抗不匹配，可能最终被滥用以触发 HPP 漏洞。

从参数污染弱点构建完整利用超出了本文的范围。有关示例和详细信息，请参见参考资料。

### 客户端 HPP

与服务器端 HPP 类似，手动测试是审计 Web 应用程序以检测影响客户端组件的参数污染漏洞的唯一可靠技术。在服务器端变体中，攻击者利用易受攻击的 Web 应用程序来访问受保护的数据或执行不允许或不应执行的操作，而客户端攻击旨在破坏客户端组件和技术。

要测试客户端 HPP 漏洞，请识别任何允许用户输入并向用户显示输入结果的表单或操作。搜索页面是理想的，但登录框可能不起作用（因为它可能不会向用户显示无效用户名）。

与服务器端 HPP 类似，使用 `%26HPP_TEST` 污染每个 HTTP 参数，并查找用户提供载荷的 *URL 解码* 出现项：

- `&HPP_TEST`
- `&amp;HPP_TEST`
- 等等。

特别注意响应中在 `data`、`src`、`href` 属性或表单操作中包含 HPP 向量的情况。同样，此默认行为是否揭示潜在漏洞取决于特定的输入验证、过滤和应用程序业务逻辑。此外，重要的是要注意此漏洞也可能影响 XMLHttpRequest（XHR）、运行时属性创建和其他插件技术（例如 Adobe Flash 的 flashvars 变量）中使用的查询字符串参数。
