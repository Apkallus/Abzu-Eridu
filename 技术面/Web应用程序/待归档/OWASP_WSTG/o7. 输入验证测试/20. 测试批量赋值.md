# 测试批量赋值

|ID          |
|------------|
|WSTG-INPV-20|

## 概述

现代 Web 应用程序通常基于框架构建。许多这些 Web 应用程序框架允许将用户输入（以 HTTP 请求参数的形式）自动绑定到内部对象。这通常称为自动绑定。
此功能有时可能被利用来访问本不应从外部修改的字段，从而导致权限提升、数据篡改、安全机制绕过等问题。
在这种情况下，就存在批量赋值漏洞。

敏感属性示例：

- **与权限相关的属性**：应仅由特权用户设置（例如 `is_admin`、`role`、`approved`）。
- **与流程相关的属性**：应仅在流程完成后由内部设置（例如 `balance`、`status`、`email_verified`）。
- **内部属性**：应仅由应用程序内部设置（例如 `created_at`、`updated_at`）。

## 测试目标

- 识别修改对象的请求。
- 评估是否可能修改本不应从外部修改的字段。

## 如何测试

以下是一个经典的示例，有助于说明此问题。

假设一个 Java Web 应用程序具有类似于以下的 `User` 对象：

```java
public class User {
   private String username;
   private String password;
   private String email;
   private boolean isAdmin;

   //Getters & Setters
}
```

为了创建一个新的 `User`，Web 应用程序实现了以下视图：

```html
<form action="/createUser" method="POST">
     <input name="username" type="text">
     <input name="password" type="text">
     <input name="email" text="text">
     <input type="submit" value="Create">
</form>
```

处理创建请求的控制器（Spring 提供与 `User` 模型的自动绑定）：

```java
@RequestMapping(value = "/createUser", method = RequestMethod.POST)
public String createUser(User user) {
   userService.add(user);
   return "successPage";
}
```

当表单提交时，浏览器生成以下请求：

```http
POST /createUser
[...]
username=bob&password=supersecretpassword&email=bob@domain.test
```

然而，由于自动绑定，攻击者可以向请求添加 `isAdmin` 参数，控制器将自动将其绑定到模型。

```http
POST /createUser
[...]
username=bob&password=supersecretpassword&email=bob@domain.test&isAdmin=true
```

然后，用户以 `isAdmin` 属性设置为 `true` 的方式被创建，从而在应用程序中授予其管理权限。

### 黑盒测试

#### 检测处理器

为了确定应用程序的哪个部分容易受到批量赋值攻击，请枚举应用程序中所有接受用户内容并可能映射到模型的部分。这包括所有看似允许在后端执行创建或更新操作的 HTTP 请求（很可能是 GET、POST 和 PUT）。
潜在批量赋值的最简单指标之一是输入参数名称中存在括号语法，例如：

```html
<input name="user[name]" type="text">
```

当遇到此类模式时，尝试添加与不存在的属性相关的输入（例如 `user[nonexistingattribute]`），并分析响应/行为。
如果应用程序未实现任何控制（例如允许字段列表），则很可能由于应用程序找不到与对象关联的属性而返回错误（例如 500）。更有趣的是，这些错误有时有助于发现属性名称和值数据类型，而无需访问源代码。

#### 识别敏感字段

由于在黑盒测试中测试人员无法看到源代码，因此有必要通过其他方式收集与对象关联的属性信息。
分析后端收到的响应，尤其注意：

- HTML 页面源代码
- 自定义 JavaScript 代码
- API 响应

例如，通常可以利用返回对象详细信息的处理器来收集有关关联字段的线索。
假设有一个返回用户个人资料的处理器（例如 `GET /profile`），它可能包含与用户相关的其他属性（在此示例中，`isAdmin` 属性看起来特别有趣）。

```json
{"_id":12345,"username":"bob","age":38,"email":"bob@domain.test","isAdmin":false}
```

然后尝试利用允许修改或创建用户的处理器，添加设置为 `true` 的 `isAdmin` 属性。

另一种方法是使用单词列表来尝试枚举所有潜在属性。然后可以自动化枚举（例如通过 wfuzz、Burp Intruder、ZAP fuzzer 等）。sqlmap 工具包含一个 [common-columns.txt](https://github.com/sqlmapproject/sqlmap/blob/master/data/txt/common-columns.txt) 单词列表，可用于识别潜在的敏感属性。
常见有趣属性名称的小示例包括：

- `is_admin`
- `is_administrator`
- `isAdmin`
- `isAdministrator`
- `admin`
- `administrator`
- `role`

当存在多个角色时，尝试比较不同用户级别发出的请求（特别注意特权角色）。例如，如果管理用户发出的请求中包含额外参数，请尝试以低权限/匿名用户身份使用这些参数。

#### 检查影响

批量赋值的影响可能因上下文而异，因此对于上一阶段尝试的每个测试输入，分析结果并确定它是否代表对 Web 应用程序安全性具有实际影响的漏洞。
例如，修改对象的 `id` 可能导致应用程序拒绝服务或权限提升。另一个示例涉及修改用户角色/状态（例如 `role` 或 `isAdmin`）的可能性，从而导致垂直权限提升。

### 灰盒测试

当使用灰盒测试方法进行分析时，可以遵循相同的方法来验证问题。然而，对应用程序更深入的了解使得更容易识别易受批量赋值漏洞影响的框架和处理器。
特别是，当源代码可用时，可以更轻松、更准确地搜索输入向量。在源代码审查期间，使用简单工具（如 grep 命令）在应用程序代码中搜索一个或多个常见模式。
访问数据库模式或源代码也有助于轻松识别敏感字段。

#### Java

Spring MVC 允许将用户输入自动绑定到对象中。识别处理状态更改请求的控制器（例如查找 `@RequestMapping` 的出现），然后验证是否实施了控制（在控制器上或涉及模型上）。批量赋值利用的限制可以是，例如：

- 通过 `DataBinder` 类的 `setAllowedFields` 方法设置的可绑定字段列表（例如 `binder.setAllowedFields(["username","password","email"])`）
- 通过 `DataBinder` 类的 `setDisallowedFields` 方法设置的非可绑定字段列表（例如 `binder.setDisallowedFields(["isAdmin"])`）

还应注意使用 `@ModelAttribute` 注解，该注解允许指定不同的名称/键。

#### PHP

Laravel Eloquent ORM 提供了 `create` 方法，允许自动赋值属性。然而，最新版本的 Eloquent ORM 提供了针对批量赋值漏洞的默认保护，要求通过 `$fillable` 数组明确指定可以自动赋值的允许属性，或通过 `$guarded` 数组指定需要保护的属性（非可绑定）。因此，通过分析模型（扩展 `Model` 类的类），可以识别哪些属性被允许或拒绝，从而指出潜在的漏洞。

#### .NET

ASP.NET 中的模型绑定自动将用户输入绑定到对象属性。这也适用于复杂类型，并且如果属性名称与输入匹配，它将自动将输入数据转换为属性。
识别控制器，然后验证是否实施了控制（在控制器内部或涉及模型上）。批量赋值利用的限制可以是，例如：

- 声明为 `ReadOnly` 的字段
- 通过 `Bind` 属性设置的可绑定字段列表（例如 `[Bind(Include = "FirstName, LastName")] Student std`），通过 `includeProperties`（例如 `includeProperties: new[] { "FirstName, LastName" }`）或通过 `TryUpdateModel`
- 通过 `Bind` 属性设置的非可绑定字段列表（例如 `[Bind(Exclude = "Status")] Student std`）或通过 `excludeProperties`（例如 `excludeProperties: new[] { "Status" }`）

## 修复

使用框架提供的内置功能来定义可绑定和非可绑定字段。最好采用基于允许字段（可绑定）的方法，其中明确定义了用户应更新的属性。
防止此问题的一种架构方法是使用*数据传输对象*（DTO）模式以避免直接绑定。DTO 应仅包含旨在供用户编辑的字段。
