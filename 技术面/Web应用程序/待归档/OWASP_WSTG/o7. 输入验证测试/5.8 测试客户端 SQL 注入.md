# 测试客户端 SQL 注入

## 概述

当应用程序实现了 [Web SQL Database](https://www.w3.org/TR/webdatabase/) 技术，且未正确验证输入或未对查询变量进行参数化时，就会发生客户端 SQL 注入。此数据库通过使用 JavaScript（JS）API 调用来操作，例如 `openDatabase()`，该函数用于创建或打开现有数据库。

## 测试目标

以下测试场景将验证是否进行了适当的输入验证。如果实现存在漏洞，攻击者可以读取、修改或删除存储在数据库中的信息。

## 如何测试

### 识别 Web SQL DB 的使用

如果被测应用程序实现了 Web SQL DB，则客户端核心代码中将使用以下三个调用：

- `openDatabase()`
- `transaction()`
- `executeSQL()`

以下代码展示了这些 API 实现的示例：

```javascript
var db = openDatabase(shortName, version, displayName, maxSize);

db.transaction(function(transaction) {
    transaction.executeSql('INSERT INTO LOGS (time, id, log) VALUES (?, ?, ?)', [dateTime, id, log]);
});
```

### Web SQL DB 注入

确认使用了 `executeSQL()` 之后，攻击者即可测试和验证其实现的安全性。

Web SQL DB 的实现基于 [SQLite 的语法](https://www.sqlite.org/lang.html)。

#### 绕过条件

以下示例展示了如何在客户端利用此漏洞：

```javascript
// URL 示例：https://example.com/user#15
var userId = document.location.hash.substring(1,); // 获取不带哈希的 ID -> 15

db.transaction(function(transaction){
    transaction.executeSQL('SELECT * FROM users WHERE user = ' + userId);
});
```

为了返回所有用户的信息，而不是仅返回攻击者对应的用户，可以在 URL 片段中使用 `15 OR 1=1`。

> **_注意：_** 在 SQL 查询中注入条件 OR 1=1 时要小心。虽然这在您注入的初始上下文中可能无害，但应用程序通常会从单个请求中获取数据用于多个不同的查询。如果您的条件到达 UPDATE 或 DELETE 语句，例如，可能导致意外的数据丢失。

有关其他 SQL 注入负载，请参阅[测试 SQL 注入](05-Testing_for_SQL_Injection.md)场景。
