# 测试 ORM 注入

## 概述

[对象关系映射（ORM）注入](https://capec.mitre.org/data/definitions/109.html)是一种针对 ORM 生成的数据访问对象模型使用 SQL 注入的攻击。从测试人员的角度来看，这种攻击与 SQL 注入攻击几乎相同。然而，注入漏洞存在于 ORM 层生成的代码中。

使用 ORM 工具的好处包括：快速生成与关系数据库通信的对象层、标准化这些对象的代码模板，并且它们通常提供一组安全函数来防止 SQL 注入攻击。ORM 生成的对象可以使用 SQL，在某些情况下使用 SQL 的变体，对数据库执行 CRUD（创建、读取、更新、删除）操作。然而，如果 Web 应用程序使用的 ORM 生成对象的方法可以接受未经清理的输入参数，则仍然可能存在 SQL 注入漏洞。

## 如何测试

ORM 层可能容易受到攻击，因为它们扩展了攻击面。您不是直接使用 SQL 查询攻击应用程序，而是专注于滥用 ORM 层来发送恶意 SQL 查询。

### 识别 ORM 层

为了有效地测试并理解您的请求与后端查询之间发生的事情，以及与进行适当测试相关的一切，识别所使用的技术至关重要。通过遵循[信息收集](../01-Information_Gathering/README.md)章节，您应该了解当前应用程序所使用的技术。查看此[将语言映射到其相应 ORM 的列表](https://en.wikipedia.org/wiki/List_of_object-relational_mapping_software)。

### 滥用 ORM 层

在识别出可能使用的 ORM 之后，了解其解析器的工作原理、研究滥用方法，甚至如果应用程序使用的是旧版本，识别与所用库相关的 CVE 变得至关重要。有时，ORM 层未正确实现，因此允许测试人员进行常规的 [SQL 注入](05-Testing_for_SQL_Injection.md)，而无需担心 ORM 层。

#### ORM 实现薄弱

一个来自 [SANS](https://software-security.sans.org/developer-how-to/fix-sql-injection-in-java-hibernate) 的脆弱场景，其中 ORM 层未正确实现：

```java
List results = session.createQuery("from Orders as orders where orders.id = " + currentOrder.getId()).list();
List results = session.createSQLQuery("Select * from Books where author = " + book.getAuthor()).list();
```

上述代码未实现位置参数，而位置参数允许开发人员将输入替换为 `?`。示例如下：

```java
Query hqlQuery = session.createQuery("from Orders as orders where orders.id = ?");
List results = hqlQuery.setString(0, "123-ADB-567-QTWYTFDL").list(); // 0 是第一个位置，它被动态替换为设置的字符串
```

这种实现将验证和清理交由 ORM 层完成，绕过它的唯一方法是识别 ORM 层本身的问题。

#### 脆弱的 ORM 层

ORM 层是代码，大多数情况下是第三方库。它们可能像任何其他代码一样存在漏洞。一个例子是 [sequelize ORM npm 库](https://snyk.io/blog/sequelize-orm-npm-library-found-vulnerable-to-sql-injection-attacks/)，该库在 2019 年被发现存在漏洞。在 [RIPS Tech](https://www.ripstech.com/) 进行的另一项研究中，[Java 使用的 Hibernate ORM](https://hibernate.org/orm/) 中发现了绕过方法。

根据他们的[博客文章](https://blog.ripstech.com/2020/exploiting-hibernate-injections/)，可以概述一个允许测试人员识别问题的速查表如下：

| DBMS       | SQL 注入                                                         |
|------------|-----------------------------------------------------------------------|
| MySQL      | `abc\' INTO OUTFILE --`                                               |
<!-- markdownlint-disable-next-line MD055 MD056 -->
| PostgreSQL | `$$='$$=chr(61) || chr(0x27) and 1=pg_sleep(2) || version()'`         |
| Oracle     | `NVL(TO_CHAR(DBMS_XMLGEN.getxml('select 1 where 1337>1')),'1')!='1'`  |
| MS SQL     | `1<LEN(%C2%A0(select%C2%A0top%C2%A01%C2%A0name%C2%A0from%C2%A0users)` |

另一个示例包括 [Laravel Query-Builder](https://laravel.com/docs/7.x/queries)，它在 2019 年被发现存在[漏洞](https://freek.dev/1317-an-important-security-release-for-laravel-query-builder)。
