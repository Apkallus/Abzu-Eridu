# 测试 MS Access

## 概述

如通用 [SQL 注入](https://owasp.org/www-community/attacks/SQL_Injection) 部分所述，当用户提供的输入在构造 SQL 查询时未经过充分约束或清理时，就会出现 SQL 注入漏洞。此类漏洞允许攻击者在用于连接数据库的用户的权限下执行 SQL 代码。本节将讨论利用 [Microsoft Access](https://en.wikipedia.org/wiki/Microsoft_Access) 特定功能的相关 SQL 注入技术。

## 如何测试

### 指纹识别

在测试基于 SQL 的应用程序时，指纹识别具体的数据库技术是正确评估潜在漏洞的第一步。常见方法包括注入标准的 SQL 注入攻击模式（例如单引号、双引号等）以触发数据库异常。假设应用程序未使用自定义页面处理异常，则可以通过观察错误消息来指纹识别底层 DBMS。

根据所使用的具体 Web 技术，MS Access 驱动的应用程序将响应以下错误之一：

`Fatal error: Uncaught exception 'com_exception' with message Source: Microsoft JET Database Engine`

或

`Microsoft JET Database Engine error '80040e14'`

或

`Microsoft Office Access Database Engine`

在所有情况下，我们都可以确认正在测试使用 MS Access 数据库的应用程序。

### 基础测试

不幸的是，MS Access 不支持传统 SQL 注入测试中常用的典型运算符，包括：

- 无注释字符
- 无堆叠查询
- 无 LIMIT 运算符
- 无 SLEEP 或 BENCHMARK 类运算符
- 以及其他许多运算符

尽管如此，可以通过组合多个运算符或使用替代技术来模拟这些功能。如前所述，无法使用插入字符 `/*`、`--` 或 `#` 的技巧来截断查询。然而，幸运的是，我们可以通过注入“空”字符来绕过此限制。在 SQL 查询中使用空字节 `%00` 会导致 MS Access 忽略所有剩余字符。这可以解释为数据库使用的内部表示中所有字符串都以 NULL 结尾。值得一提的是，`null` 字符有时也可能带来麻烦，因为它可能会在 Web 服务器级别截断字符串。在这种情况下，我们可以使用另一个字符：`0x16`（URL 编码格式为 `%16`）。

考虑以下查询：

`SELECT [username],[password] FROM users WHERE [username]='$myUsername' AND [password]='$myPassword'`

我们可以通过以下两个 URL 截断查询：

- `https://www.example.com/page.asp?user=admin'%00&pass=foo`
- `https://www.example.com/page.app?user=admin'%16&pass=foo`

MS Access 未实现 `LIMIT` 运算符，但可以使用 `TOP` 或 `LAST` 运算符来限制结果数量。

`https://www.example.com/page.app?id=2'+UNION+SELECT+TOP+3+name+FROM+appsTable%00`

通过组合这两个运算符，可以选择特定的结果。字符串连接可以使用 `& (%26)` 和 `+ (%2b)` 字符。

在测试 SQL 注入时，还可以使用许多其他函数，包括但不限于：

- ASC：获取传入字符的 ASCII 值
- CHR：获取传入 ASCII 值对应的字符
- LEN：返回传入字符串的长度
- IIF：IF 结构，例如以下语句 `IIF(1=1, 'a', 'b')` 返回 `a`
- MID：此函数允许提取子字符串，例如 `mid('abc',1,1)` 返回 `a`
- TOP：此函数允许指定查询应从顶部返回的最大结果数。例如 `TOP 1` 将仅返回 1 行。
- LAST：此函数用于仅选择一组行中的最后一行。例如以下查询 `SELECT last(*) FROM users` 将仅返回结果的最后一行。

其中一些运算符对于利用盲 SQL 注入至关重要。对于其他高级运算符，请参阅参考资料中的文档。

#### 属性枚举

为了枚举数据库表的列，可以使用常见的基于错误的技术。简而言之，我们可以通过分析错误消息并使用不同的选择器重复查询来获取属性名称。例如，假设我们知道某列的存在，我们可以使用以下查询获取剩余属性的名称：

`' GROUP BY Id%00`

在收到的错误消息中，可以观察到下一列的名称。此时，我们可以迭代该方法，直到获得所有属性的名称。如果我们不知道第一个属性的名称，仍然可以插入一个虚构的列名，并在错误消息中获取第一个属性的名称。

#### 获取数据库模式

MS Access 默认存在多个系统表，可用于获取表名和列名。不幸的是，在较新的 MS Access 数据库版本的默认配置中，这些表不可访问。尽管如此，仍值得尝试：

- MSysObjects
- MSysACEs
- MSysAccessXML

例如，如果存在 union SQL 注入漏洞，可以使用以下查询：

`' UNION SELECT Name FROM MSysObjects WHERE Type = 1%00`

或者，可以使用标准单词列表（例如 [FuzzDb](https://github.com/fuzzdb-project/fuzzdb)）暴力破解数据库模式。

在某些情况下，开发人员或系统管理员未意识到将实际的 `.mdb` 文件包含在应用程序 Web 根目录中可能允许下载整个数据库。数据库文件名可以通过以下查询推断：

`https://www.example.com/page.app?id=1'+UNION+SELECT+1+FROM+name.table%00`

其中 `name` 是 `.mdb` 文件名，`table` 是有效的数据库表。如果数据库受密码保护，可以使用多种软件工具破解密码。请参阅参考资料。

### 盲 SQL 注入测试

[盲 SQL 注入](https://owasp.org/www-community/attacks/Blind_SQL_Injection) 漏洞在测试实际应用时并非最容易利用的 SQL 注入。在较新版本的 MS Access 中，也无法执行 shell 命令或读/写任意文件。

在盲 SQL 注入的情况下，攻击者只能通过评估时间差异或应用程序响应来推断查询结果。假设读者已经了解盲 SQL 注入攻击背后的理论，本节的其余部分将重点介绍 MS Access 的具体细节。

以下示例被使用：

`https://www.example.com/index.php?myId=[sql]`

其中 ID 参数用于以下查询：

`SELECT * FROM orders WHERE [id]=$myId`

假设 `myId` 参数存在盲 SQL 注入漏洞。作为攻击者，我们想提取表 `users` 中 `username` 列的内容，假设我们已经公开了数据库模式。

可用于推断第 10 行用户名的第一个字符的典型查询是：

`https://www.example.com/index.php?id=IIF((select%20MID(LAST(username),1,1)%20from%20(select%20TOP%2010%20username%20from%20users)='a',0,'no')`

如果第一个字符是 `a`，查询将返回 `0`，否则返回字符串 `no`。

通过组合 `IIF、MID、LAST` 和 `TOP` 函数，可以提取特定选定行上用户名的第一个字符。由于内部查询返回一组记录，而不仅仅是一条，因此无法直接使用它。幸运的是，我们可以组合多个函数来提取特定字符串。

假设我们要检索第 10 行的用户名。首先，我们可以使用 TOP 函数选择前 10 行，使用以下查询：

`SELECT TOP 10 username FROM users`

然后，使用此子集，我们可以通过 LAST 函数提取最后一行。一旦我们只有一行，并且正是包含我们字符串的行，我们就可以使用 IIF、MID 和 LAST 函数来推断用户名的实际值。在我们的示例中，我们使用 IIF 返回一个数字或一个字符串。使用此技巧，我们可以通过观察应用程序错误响应来区分是否为真响应。由于 `id` 是数字，与字符串比较会导致 SQL 错误，可能通过 `500 Internal Server Error` 页面泄露。否则，可能会返回标准的 `200 OK` 页面。

例如，我们可以有以下查询：

`https://www.example.com/index.php?id='%20AND%201=0%20OR%20'a'=IIF((select%20MID(LAST(username),1,1)%20from%20(select%20TOP%2010%20username%20from%20users))='a','a','b')%00`

如果第一个字符是 'a'，则为真，否则为假。

如前所述，此方法允许推断数据库中任意字符串的值：

1. 通过尝试所有可打印值，直到找到匹配项
2. 通过使用 `LEN` 函数推断字符串长度，或者在找到所有字符后停止

基于时间的盲 SQL 注入也可以通过滥用[重量级查询](https://docs.microsoft.com/en-us/previous-versions/tn-archive/cc512676(v=technet.10))来实现。
