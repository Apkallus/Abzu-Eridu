# 测试 CSV 注入

|ID          |
|------------|
|WSTG-INPV-21|

## 概述

CSV 注入（也称为公式注入）发生在应用程序将不受信任的、用户控制的输入嵌入到 CSV（或其他电子表格兼容）导出文件中，并且生成的文件在电子表格程序（例如 Microsoft Excel、LibreOffice Calc）中打开时。电子表格应用程序可能将某些单元格值解释为公式，这可能导致安全问题，例如用户欺骗（钓鱼式工作流）、操纵电子表格输出或数据泄露。在某些环境中，公式注入可以通过电子表格“小工具”和遗留功能（例如 DDE / 动态数据交换行为）升级为更高影响，可能达到打开文件的工作站上的命令执行——通常取决于客户端配置和/或用户交互。

此问题的一个关键特征是，漏洞通常仅在导出文件被用户（例如管理员、财务或支持人员）在电子表格应用程序中打开时才会显现。

## 测试目标

- 识别包含不受信任输入的 CSV/电子表格导出功能。
- 验证在常见电子表格应用程序中打开导出文件时，攻击者控制的值是否被解释为公式。
- 检查分隔符/引号注入是否可以将危险前缀移动到单元格的开头。
- 验证缓解措施在 Microsoft Excel 中保存并重新打开 CSV 后是否仍然有效。
- 根据谁打开导出文件以及如何使用来评估实际影响。

## 如何测试

### 公式触发前缀

以以下字符开头的单元格可能被电子表格软件解释为公式：

- 等号（`=`）
- 加号（`+`）
- 减号（`-`）
- At 符号（`@`）
- 制表符（`0x09`）
- 回车符（`0x0D`）
- 换行符（`0x0A`）
- 全角（双字节）变体，例如 `＝`、`＋`、`－`、`＠`（取决于区域设置/应用程序行为）

> 重要（Excel 行为）：当 CSV 文件被保存并重新打开时，Microsoft Excel 可能会从 CSV 单元格中移除引号或转义字符。因此，一些常见建议的缓解措施在保存/重新打开后可能失效，之前被转义的公式可能再次变为活动状态。

还要注意，确保 *整个* 不受信任输入不以危险字符开头是不够的。攻击者可能会注入分隔符和引号来开始一个新单元格，将危险字符放置在单元格的开头。

### 识别 CSV 导出功能和数据源

定位生成 CSV/TSV 或“导出到电子表格”内容的功能：

- 报告（用户、交易、审计日志、工单）
- 导出列表的管理仪表板
- 由应用程序生成的电子邮件附件
- 计划导出 / 集成

识别可能最终出现在导出中的不受信任数据源：

- 用户资料（姓名、电子邮件、公司）
- 自由文本字段（评论、工单主题、备注）
- 导入/集成的外部数据（webhook、CRM 同步、合作伙伴源）

记录哪些角色可以触发导出，以及哪些角色可能打开它。

### 将良性的、可检测的类似公式的值放入候选字段

使用无害的负载来检测公式评估（避免执行命令或执行不受控网络访问的负载）。
测试以每个公式触发字符开头的值：

- `=1+1`
- `+1+1`
- `-1+1`
- `@SUM(1,1)`
- `=HYPERLINK("http://example.invalid/leak?test=1", "Click Me")`

注意：

- `HYPERLINK()` 案例可用于演示实际影响（欺骗/钓鱼式工作流，或点击/打开链接时可能的元数据泄露）。在测试期间使用受控端点（例如本地监听器或内部测试主机）。
- 在验证中不要使用外部“真实攻击者”基础设施。

同时测试控制字符和 Unicode 变体（如果输入处理允许）：

- 以制表符后跟 `=1+1` 开头的值（TAB + `=1+1`）
- 全角前缀变体（例如 `＝1+1`）

### 测试分隔符和引号“单元格突破”场景

由于 CSV 是基于单元格的，测试您是否可以注入内容以开始一个新单元格，然后以危险字符开头。这取决于：

- 字段分隔符（通常为 `,` 或 `;`）
- 引号规则和转义
- 应用程序端 CSV 生成和编码

*良性* 测试模式示例（根据实际导出格式调整分隔符）：

- 包含引号和分隔符的值，旨在创建新单元格，然后 `=1+1`
- 直接包含分隔符的值（如果导出程序未加引号），然后 `=1+1`

您的目标是查看生成的 CSV 是否包含任何以公式触发前缀之一开头的单元格。通过在文本编辑器中检查原始 CSV 输出来验证这一点。

### 在电子表格应用程序中导出并验证

- 导出/下载 CSV。
- 在目标环境中使用的至少一个电子表格应用程序中打开它（Excel 和/或 LibreOffice 等）。
- 确认单元格是否被解释为公式：
    - 电子表格显示计算结果（例如 `2`）而不是文字字符串（例如 `=1+1`），和/或
    - 公式栏显示公式而不是纯文本。

记录：

- 应用程序名称/版本（例如 Microsoft Excel）
- 文件打开方式（双击 vs 导入向导）
- 区域设置是否影响解析

### Excel 保存/重新打开回归测试（缓解措施可靠性）

如果应用程序声称已转义/引用值：

- 在 Microsoft Excel 中打开导出的 CSV。
- 保存文件（例如保存为 CSV）。
- 关闭并重新打开它。
- 重新检查先前“中和”的值是否变为活动公式。

此步骤至关重要，因为 Excel 可能在保存/重新打开后规范化/移除某些转义/引号行为。

### 评估影响

根据以下因素评估实际风险：

- 谁打开文件（特权角色 vs 最终用户）
- 导出是否在外部共享
- 电子表格值是否被依赖用于决策（例如总计、标志、状态）
- 公式是否可能误导用户或在您的环境中触发电子表格行为

提供清晰的复现路径：

1. 在哪里输入输入
2. 如何生成导出
3. 哪个程序将其解释为公式
4. 证据（屏幕截图或描述）表明评估发生

### 更高影响的验证（可选，严格受控）

一些现实世界的案例表明，公式注入可以升级到超出简单计算/欺骗的更高影响（例如通过遗留电子表格功能如 DDE）。此行为高度依赖于环境。

## 修复

没有适用于所有电子表格应用程序和下游消费者的通用 CSV 清理策略。对于旨在在电子表格软件中供人查看的 CSV，请应用纵深防御方法：

- 确保没有单元格以公式触发字符（`=`、`+`、`-`、`@`）及相关控制和 Unicode 变体开头。
- 考虑按字段清理（通常建议）：
    - 将每个单元格用双引号括起来
    - 在每个单元格前添加单引号
    - 通过双写来转义双引号
  注意：在 Microsoft Excel 中保存并重新打开后，这可能无法保持可靠。

- Excel 抗性缓解措施（观察到的行为）：
    - 如果单元格以 `=`、`+`、`-` 或 `@` 开头，在带引号的字段内为内容添加制表符（`0x09`）前缀（例如 `"\t=1+1"`）。
  权衡：制表符成为底层数据的一部分，可能影响下游程序化导入。

始终根据范围内的电子表格应用程序和工作流验证所选的缓解措施。
