# 测试会话超时

|ID          |
|------------|
|WSTG-SESS-07|

## 概述

在此阶段，测试人员检查应用程序是否在用户空闲一定时间后自动将其注销，确保无法“重用”同一会话，并且没有敏感数据残留在浏览器缓存中。

所有应用程序都应为会话实施空闲或非活动超时。该超时定义了在用户无活动的情况下会话将保持活动的时间长度，并在自 Web 应用程序收到特定会话 ID 的最后一个 HTTP 请求以来的定义空闲时间过后关闭并使会话失效。最合适的超时时间应在安全性（较短的超时）和可用性（较长的超时）之间取得平衡，并在很大程度上取决于应用程序所处理数据的敏感级别。例如，公共论坛的 60 分钟注销时间是可以接受的，但在家庭银行应用程序中，这么长的时间就太过了（建议最大超时为 15 分钟）。无论如何，任何不强制执行基于超时的注销的应用程序都应被视为不安全，除非特定功能要求需要此类行为。

空闲超时限制了攻击者猜测和使用来自其他用户的有效会话 ID 的机会，并且在某些情况下可以保护公共计算机免遭会话重用。然而，如果攻击者能够劫持给定的会话，空闲超时并不会限制攻击者的操作，因为他们可以定期在会话上生成活动，以使会话保持活动状态更长时间。

会话超时管理和过期必须在服务器端强制执行。如果使用客户端控制的某些数据来强制执行会话超时，例如使用 cookie 值或其他客户端参数来跟踪时间引用（例如自登录以来的分钟数），攻击者可以操纵这些数据来延长会话持续时间。因此，应用程序必须在服务器端跟踪非活动时间，并在超时过期后自动使当前用户的会话失效，并删除客户端上存储的所有数据。

这两个操作都必须仔细实施，以避免引入弱点，如果用户忘记从应用程序注销，攻击者可能利用这些弱点获得未授权访问。更具体地说，对于注销功能，确保所有会话令牌（例如 cookie）都被正确销毁或使其不可用，并在服务器端实施适当的控制以防止会话令牌的重用，这一点至关重要。如果这些操作未正确执行，攻击者可以重放这些会话令牌以“复活”合法用户的会话并冒充他们（这种攻击通常称为“cookie 重放”）。当然，一个缓解因素是攻击者需要能够访问这些令牌（存储在受害者的 PC 上），但在多种情况下，这可能并非不可能或特别困难。

此类攻击的最常见场景是使用公共计算机访问某些私人信息（例如网络邮件、在线银行账户）。如果用户离开计算机而没有明确注销，并且应用程序未实施会话超时，则攻击者只需按下浏览器的“后退”按钮即可访问同一账户。

## 测试目标

- 验证是否存在严格的会话超时。

## 如何测试

### 黑盒测试

在测量超时注销时，可以应用[测试注销功能](06-Testing_for_Logout_Functionality.md)部分中看到的相同方法。
测试方法非常相似。首先，测试人员必须检查是否存在超时，例如，通过登录并等待超时注销被触发。与注销功能一样，超时过后，所有会话令牌都应被销毁或变得不可用。

然后，如果配置了超时，测试人员需要了解超时是由客户端还是服务器（或两者）强制执行的。如果会话 cookie 是非持久性的（或者更一般地说，会话 cookie 不存储任何关于时间的数据），测试人员可以假设超时由服务器强制执行。如果会话 cookie 包含一些与时间相关的数据（例如登录时间、上次访问时间或持久性 cookie 的过期日期），则客户端可能参与了超时的强制执行。在这种情况下，测试人员可以尝试修改 cookie（如果未加密保护），并观察会话会发生什么。例如，测试人员可以将 cookie 过期日期设置为遥远的未来，看看会话是否可以延长。

作为一般规则，所有内容都应在服务器端检查，并且不应可能通过将会话 cookie 重置为先前值来再次访问应用程序。

### 灰盒测试

测试人员需要检查：

- 注销功能是否有效销毁所有会话令牌，或至少使其不可用，
- 服务器是否对会话状态执行适当的检查，不允许攻击者重放先前已销毁的会话标识符，
- 是否强制执行超时，并且由服务器正确强制执行。如果服务器使用从客户端发送的会话令牌中读取的过期时间（但不建议这样做），则该令牌必须加密保护以防止篡改。

请注意，最重要的是应用程序在服务器端使会话失效。通常，这意味着代码必须调用适当的方法，例如 Java 中的 `HttpSession.invalidate()` 和 .NET 中的 `Session.abandon()`。从浏览器中清除 cookie 是可取的，但并非严格必要，因为如果会话在服务器上被正确失效，浏览器中的 cookie 对攻击者将没有帮助。
