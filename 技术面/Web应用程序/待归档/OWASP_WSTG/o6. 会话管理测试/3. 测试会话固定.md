# 测试会话固定

|ID          |
|------------|
|WSTG-SESS-03|

## 概述

会话固定（Session Fixation）是由于在认证前后保持相同的会话cookie值这一不安全实践而产生的。这种情况通常发生在会话cookie在登录前就被用于存储状态信息时，例如在认证支付之前向购物车添加商品。

在会话固定漏洞的典型利用中，攻击者可以在未认证的情况下从目标站点获取一组会话cookie。然后，攻击者可以使用不同技术将这些cookie强制注入受害者的浏览器。如果受害者随后在目标站点进行认证，且登录时cookie未被刷新，则受害者将被攻击者选择的会话cookie所标识。攻击者随后能够使用这些已知的cookie冒充受害者。

此问题可以通过在认证过程后刷新会话cookie来修复。或者，也可以通过确保会话cookie的完整性来防止攻击。当考虑网络攻击者（即控制受害者所用网络的攻击者）时，应全面采用[HSTS](https://en.wikipedia.org/wiki/HTTP_Strict_Transport_Security)，或在cookie名称中添加`__Host-` / `__Secure-`前缀。

全面采用HSTS是指主机为自身及其所有子域激活HSTS。这在Stefano Calzavara、Alvise Rabitti、Alessio Ragazzo和Michele Bugliesi合著的论文《Testing for Integrity Flaws in Web Sessions》中有所描述。

## 测试目标

- 分析认证机制及其流程。
- 强制注入cookie并评估其影响。

## 如何测试

在本节中，我们将对下一节所示的测试策略进行解释。

第一步是向被测站点（例如 `www.example.com`）发送请求。如果测试人员发送以下请求：

```http
GET / HTTP/1.1
Host: www.example.com
```

他们将获得以下响应：

```html
HTTP/1.1 200 OK
Date: Wed, 14 Aug 2008 08:45:11 GMT
Server: IBM_HTTP_Server
Set-Cookie: JSESSIONID=0000d8eyYq3L0z2fgq10m4v-rt4:-1; Path=/; secure
Cache-Control: no-cache="set-cookie,set-cookie2"
Expires: Thu, 01 Dec 1994 16:00:00 GMT
Keep-Alive: timeout=5, max=100
Connection: Keep-Alive
Content-Type: text/html;charset=Cp1254
Content-Language: en-US
```

应用程序为客户端设置了一个新的会话标识符 `JSESSIONID=0000d8eyYq3L0z2fgq10m4v-rt4:-1`。

接下来，如果测试人员通过以下POST请求成功认证到 `https://www.example.com/authentication.php`：

```http
POST /authentication.php HTTP/1.1
Host: www.example.com
[...]
Referer: https://www.example.com
Cookie: JSESSIONID=0000d8eyYq3L0z2fgq10m4v-rt4:-1
Content-Type: application/x-www-form-urlencoded
Content-length: 57

Name=Meucci&wpPassword=secret!&wpLoginattempt=Log+in
```

测试人员观察到服务器的以下响应：

```http
HTTP/1.1 200 OK
Date: Thu, 14 Aug 2008 14:52:58 GMT
Server: Apache/2.2.2 (Fedora)
X-Powered-By: PHP/5.1.6
Content-language: en
Cache-Control: private, must-revalidate, max-age=0
X-Content-Encoding: gzip
Content-length: 4090
Connection: close
Content-Type: text/html; charset=UTF-8
...
HTML 数据
...
```

由于成功认证后未颁发新的cookie，测试人员知道，除非确保会话cookie的完整性，否则可能进行会话劫持。

测试人员可以向用户发送一个有效的会话标识符（可能使用社会工程技巧），等待他们认证，随后验证该cookie是否已被授予权限。

### 使用强制Cookie进行测试

此测试策略针对的是网络攻击者，因此只需应用于未全面采用HSTS的站点（全面采用HSTS的站点是安全的，因为它们的所有cookie都具有完整性）。我们假设在被测站点上有两个测试账户，一个充当受害者，一个充当攻击者。我们模拟一个场景，攻击者将登录后未重新颁发且不具有完整性的所有cookie强制注入受害者的浏览器。在受害者登录后，攻击者向站点提交强制注入的cookie以访问受害者的账户：如果这些cookie足以代表受害者行事，则会话固定是可能的。

执行此测试的步骤如下：

1. 访问站点的登录页面。
2. 在登录前保存cookie jar的快照，排除名称中包含 `__Host-` 或 `__Secure-` 前缀的cookie。
3. 以受害者身份登录站点，并访问任何需要认证的安全功能页面。
4. 将cookie jar设置为步骤2中保存的快照。
5. 触发步骤3中识别的安全功能。
6. 观察步骤5中的操作是否成功执行。如果是，则攻击成功。
7. 清除cookie jar，以攻击者身份登录并访问步骤3中的页面。
8. 将步骤2中保存的cookie逐一写入cookie jar。
9. 再次触发步骤3中识别的安全功能。
10. 清除cookie jar，再次以受害者身份登录。
11. 观察步骤9中的操作是否在受害者的账户中成功执行。如果是，则攻击成功；否则，站点对会话固定是安全的。

我们建议为受害者和攻击者使用两台不同的机器或浏览器。如果Web应用程序执行指纹识别以验证从给定cookie启用的访问权限，这可以帮助减少误报数量。测试策略的一个更短但精确度稍低的变体只需要一个测试账户。它遵循相同的步骤，但在步骤6处停止。

## 修复

在用户成功认证后实施会话令牌更新。

应用程序应在认证用户之前始终首先使现有会话ID失效，如果认证成功，则提供另一个会话ID。
