# 测试 Cookie 属性

|ID          |
|------------|
|WSTG-SESS-02|

## 概述

Web Cookie（以下简称 Cookie）通常是恶意用户（主要针对其他用户）的关键攻击向量，应用程序应始终尽职尽责地保护 Cookie。

HTTP 是一种无状态协议，意味着它不持有对同一用户发送的请求的任何引用。为了解决此问题，会话被创建并附加到 HTTP 请求中。浏览器（如[测试浏览器存储](../11-Client-side_Testing/12-Testing_Browser_Storage.md)中所述）包含多种存储机制。在本指南的那一部分中，每种机制都进行了详细讨论。

浏览器中最常用的会话存储机制是 Cookie 存储。Cookie 可以通过在 HTTP 响应中包含 [`Set-Cookie`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie) 头由服务器设置，也可以通过 JavaScript 设置。Cookie 可用于多种用途，例如：

- 会话管理
- 个性化
- 跟踪

为了确保 Cookie 数据的安全，业界已经开发了多种方法来帮助锁定这些 Cookie 并限制其攻击面。随着时间的推移，Cookie 已成为 Web 应用程序首选的存储机制，因为它们在用途和保护方面提供了极大的灵活性。

保护 Cookie 的手段包括：

- [Cookie 属性](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#Creating_cookies)
- [Cookie 前缀](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#Cookie_prefixes)

## 测试目标

- 确保为 Cookie 设置了正确的安全配置。

## 如何测试

下面将讨论每个属性和前缀。测试人员应验证应用程序是否正确使用了它们。可以使用[拦截代理](#intercepting-proxy)或通过检查浏览器的 Cookie 存储来审查 Cookie。

### Cookie 属性

#### Secure 属性

[`Secure`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie#Secure) 属性指示浏览器仅在通过安全通道（如 `HTTPS`）发送请求时才发送 Cookie。这有助于保护 Cookie 不被传递到未加密的请求中。如果应用程序可以通过 `HTTP` 和 `HTTPS` 同时访问，攻击者可能能够重定向用户，使其 Cookie 作为非受保护请求的一部分发送。

#### HttpOnly 属性

[`HttpOnly`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie#HttpOnly) 属性用于帮助防止会话泄露等攻击，因为它不允许通过客户端脚本（如 JavaScript）访问 Cookie。

> 这并不限制 XSS 攻击的整个攻击面，因为攻击者仍然可以代表用户发送请求，但极大地限制了 XSS 攻击向量的影响范围。

#### Domain 属性

[`Domain`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#Scope_of_cookies) 属性用于将 Cookie 的域与发起 HTTP 请求的服务器域进行比较。如果域匹配或是子域，则接下来将检查 [`path`](#path-attribute) 属性。

请注意，只有属于指定域的主机才能为该域设置 Cookie。此外，`domain` 属性不能是顶级域（如 `.gov` 或 `.com`），以防止服务器为另一个域设置任意 Cookie（例如为 `owasp.org` 设置 Cookie）。如果未设置 domain 属性，则生成 Cookie 的服务器的主机名将用作 `domain` 的默认值。

例如，如果 `app.mydomain.com` 上的应用程序设置了一个未设置 domain 属性的 Cookie，则该 Cookie 将随之后对 `app.mydomain.com` 的所有请求重新提交，但不会提交给其子域（如 `hacker.app.mydomain.com`）或 `otherapp.mydomain.com`。（然而，旧版本的 Edge/IE 行为不同，*确实*会将这些 Cookie 发送给子域。）如果开发人员想要放宽此限制，则可以将 `domain` 属性设置为 `mydomain.com`。在这种情况下，Cookie 将随对 `app.mydomain.com` 和 `mydomain.com` 子域（如 `hacker.app.mydomain.com`，甚至 `bank.mydomain.com`）的所有请求发送。如果子域上存在易受攻击的服务器（例如 `otherapp.mydomain.com`），并且 `domain` 属性设置得过于宽松（例如 `mydomain.com`），则该易受攻击的服务器可能被用于在 `mydomain.com` 的整个范围内收集 Cookie（如会话令牌）。

#### Path 属性

[`Path`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#Scope_of_cookies) 属性与 [`domain`](#domain-attribute) 一起在设置 Cookie 范围方面起着重要作用。除了域之外，还可以指定 Cookie 有效的 URL 路径。如果域和路径匹配，则 Cookie 将被包含在请求中。与 domain 属性一样，如果 path 属性设置得过于宽松，则可能使应用程序面临同一服务器上其他应用程序的攻击风险。例如，如果 path 属性设置为 Web 服务器根目录 `/`，则应用程序 Cookie 将发送给同一域内的所有应用程序（如果同一服务器下存在多个应用程序）。同一服务器下多个应用程序的几个示例：

- `path=/bank`
- `path=/private`
- `path=/docs`
- `path=/docs/admin`

#### Expires 属性

[`Expires`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#Permanent_cookies) 属性用于：

- 设置持久性 Cookie
- 如果会话持续时间过长，限制其生命周期
- 通过将其设置为过去的日期来强制删除 Cookie

与[会话 Cookie](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#Session_cookies) 不同，持久性 Cookie 将在浏览器中使用，直到 Cookie 过期。一旦过期时间超过设置的时间，浏览器将删除该 Cookie。

#### SameSite 属性

[`SameSite`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#SameSite_cookies) 属性可用于断言是否应随跨站请求一起发送 Cookie。此功能允许服务器降低跨源信息泄露的风险。在某些情况下，它也被用作风险降低（或纵深防御）策略，以防止[跨站请求伪造](05-Testing_for_Cross_Site_Request_Forgery.md)攻击。此属性可以配置为三种不同的模式：

- `Strict`
- `Lax`
- `None`

##### Strict 值

`Strict` 值是 `SameSite` 最严格的用法，允许浏览器仅在第一方上下文中发送 Cookie，且不包含顶级导航。换句话说，与 Cookie 关联的数据将仅在与浏览器 URL 栏中显示的当前站点匹配的请求上发送。Cookie 不会在由第三方站点生成的请求上发送。此值特别推荐用于在同一域上执行的操作。然而，它可能对某些会话管理系统产生一些限制，对用户的导航体验产生负面影响。由于浏览器不会在来自第三方域或电子邮件的任何请求上发送 Cookie，即使用户已经拥有已认证的会话，也将被要求重新登录。

##### Lax 值

`Lax` 值不如 `Strict` 严格。如果 URL 等于 Cookie 的域（第一方），即使链接来自第三方域，Cookie 也会被发送。此值被大多数浏览器视为默认行为，因为它比 `Strict` 值提供了更好的用户体验。它不会针对资源（如图像）触发，因为访问它们可能不需要 Cookie。

##### None 值

`None` 值指定浏览器将在所有上下文中发送 Cookie，包括跨站请求（这是 `SameSite` 实现之前的正常行为）。如果设置了 `SameSite=None`，则必须同时设置 Secure 属性，否则现代浏览器将忽略 SameSite 属性，例如 `SameSite=None; Secure`。

### Cookie 前缀

从设计上讲，Cookie 没有能力保证其中存储信息的完整性和机密性。这些限制使得服务器无法确信给定 Cookie 的属性在创建时是如何设置的。为了以向后兼容的方式为服务器提供此类功能，业界引入了[`Cookie 名称前缀`](https://tools.ietf.org/html/draft-ietf-httpbis-cookie-prefixes-00)的概念，以便将此类细节作为 Cookie 名称的一部分嵌入传递。

#### Host 前缀

[`__Host-`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#Cookie_prefixes) 前缀要求 Cookie 满足以下条件：

  1. Cookie 必须使用 [`Secure` 属性](#secure-attribute)设置。
  2. Cookie 必须从用户代理认为安全的 URI 设置。
  3. 仅发送给设置 Cookie 的主机，且不得包含任何 [`Domain` 属性](#domain-attribute)。
  4. Cookie 必须使用 [`Path` 属性](#path-attribute)设置，且值为 `/`，以便随对主机的每个请求发送。

因此，Cookie `Set-Cookie: __Host-SID=12345; Secure; Path=/` 将被接受，而以下任何一个都将被拒绝：
`Set-Cookie: __Host-SID=12345`
`Set-Cookie: __Host-SID=12345; Secure`
`Set-Cookie: __Host-SID=12345; Domain=site.example`
`Set-Cookie: __Host-SID=12345; Domain=site.example; Path=/`
`Set-Cookie: __Host-SID=12345; Secure; Domain=site.example; Path=/`

#### Secure 前缀

[`__Secure-`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies#Cookie_prefixes) 前缀限制较少，可以通过在 Cookie 名称前添加区分大小写的字符串 `__Secure-` 来引入。任何匹配前缀 `__Secure-` 的 Cookie 都应满足以下条件：

  1. Cookie 必须使用 `Secure` 属性设置。
  2. Cookie 必须从用户代理认为安全的 URI 设置。

### 强化实践

根据应用程序需求以及 Cookie 应有的功能，必须应用相应的属性和前缀。Cookie 锁定得越严格越好。

综合以上所有内容，我们可以定义最安全的 Cookie 属性配置为：`Set-Cookie: __Host-SID=<session token>; path=/; Secure; HttpOnly; SameSite=Strict`。
