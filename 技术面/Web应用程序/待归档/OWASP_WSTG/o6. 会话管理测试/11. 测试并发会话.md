# 测试并发会话

|ID          |
|------------|
|WSTG-SESS-11|

## 概述

并发会话是 Web 应用程序的一个常见方面，它支持多个同时进行的用户交互。本测试用例旨在评估应用程序处理单个用户多个活动会话的能力。此功能对于有效管理并发用户会话至关重要，尤其是在包含个人身份信息（PII）的管理面板、个人用户账户或依赖第三方服务来丰富用户提供数据的 API 等敏感区域。主要目标是确保并发会话符合应用程序的安全要求。

了解应用程序中的安全需求是评估启用并发会话是否符合预期功能的关键。允许并发会话本身并非有害，并且在许多应用程序中是有意允许的。然而，至关重要的是确保应用程序的功能与其关于并发会话的安全措施有效对齐。如果并发会话是有意的，则必须确保额外的安全控制措施，例如管理活动会话、终止会话以及潜在的新会话通知。相反，如果应用程序中未预期或计划并发会话，则验证现有的会话管理漏洞检查至关重要。

要认识到并发会话是必要的，您应考虑以下因素：

- 了解应用程序的性质，特别是用户可能需要从不同位置或设备同时访问的情况。
- 识别关键操作，例如需要安全访问的金融交易。
- 处理敏感数据，如个人身份信息（PII），表明需要安全交互。
- 区分管理面板和普通用户仪表板（用于普通用户访问）。

## 测试目标

- 通过评估单个用户账户多个活动会话的处理来评估应用程序的会话管理。

## 如何测试

1. **生成有效会话：**
   - 提交有效凭据（用户名和密码）以创建会话。
   - 示例 HTTP 请求：

     ```http
     POST /login HTTP/1.1
     Host: www.example.com
     Content-Length: 32

     username=admin&password=admin123
     ```

   - 示例响应：

     ```http
     HTTP/1.1 200 OK
     Set-Cookie: SESSIONID=0add0d8eyYq3HIUy09hhus; Path=/; Secure
     ```

   - 存储生成的认证 cookie。在某些情况下，生成的认证 cookie 会替换为诸如 JSON Web 令牌（JWT）之类的令牌。

2. **测试生成活动会话：**
   - 尝试通过提交登录请求（例如一百次）来创建多个认证 cookie。

   注意：利用隐私浏览模式或多账户容器可能有助于进行这些测试，因为它们可以提供独立的环境来测试会话管理，而不受浏览器中存储的现有会话或 cookie 的干扰。

3. **测试验证活动会话：**
   - 尝试使用初始会话令牌（例如 `SESSIONID=0add0d8eyYq3HIUy09hhus`）访问应用程序。
   - 如果使用第一个生成的令牌成功进行认证，则将其视为一个潜在问题，表明会话管理不充分。

此外，还有一些扩展测试用例，将测试方法扩展到包括来自不同 IP 和位置的多个会话场景。这些测试用例有助于识别与地理或网络因素相关的会话处理中的潜在漏洞或异常：

- 测试来自同一 IP 的多个会话。
- 测试来自不同 IP 的多个会话。
- 测试来自同一用户在短时间内不太可能或不可能访问的位置的多个会话（例如，在特定国家创建了一个会话，五分钟后从另一个国家生成了另一个会话）。

## 修复

应用程序应监控并限制每个用户账户的活动会话数量。如果超过允许的最大会话数，系统必须使先前的会话失效以维护安全性。实施其他解决方案可以进一步缓解此漏洞：

   1. **用户通知：** 每次成功登录后通知用户，以提高对活动会话的认知。
   2. **会话管理页面：** 创建专用页面来显示并允许终止活动会话，以增强用户控制。
   3. **IP 地址跟踪：** 跟踪登录到账户的用户的 IP 地址，并标记任何可疑活动，例如来自不同位置的多次登录。
   4. **IP 地址限制：** 允许用户指定受信任的 IP 地址或范围，以便从其访问账户，通过将会话限制在已知和批准的位置来增强安全性。
