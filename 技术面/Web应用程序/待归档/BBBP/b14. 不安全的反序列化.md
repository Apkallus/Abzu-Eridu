# 第14章：不安全的反序列化（Insecure Deserialization）

当应用程序在没有适当预防措施的情况下反序列化程序对象时，就会发生不安全的反序列化漏洞。攻击者随后可以操纵序列化对象来改变程序的行为。

不安全的反序列化漏洞一直让我着迷。它们很难发现和利用，因为根据构建应用程序所用的编程语言和库的不同，它们往往呈现出不同的形式。这些漏洞还需要深厚的技术理解和独创性才能利用。尽管发现它们可能具有挑战性，但值得付出努力。无数漏洞报告描述了研究人员如何利用这些漏洞在Google和Facebook等公司的关键资产上实现RCE。

在本章中，我将介绍什么是不安全的反序列化，PHP和Java应用程序中不安全的反序列化漏洞如何发生，以及如何利用它们。

## 机制

**序列化**是指将编程语言中的某些数据转换为一种格式的过程，这种格式允许将其保存在数据库中或通过网络传输。**反序列化**是指相反的过程，即程序从文件或网络读取序列化对象并将其转换回对象。

这很有用，因为编程语言中的某些对象难以通过网络传输或在数据库中存储而不损坏。序列化和反序列化允许编程语言在不同的计算环境中重建相同的程序对象。许多编程语言都支持对象的序列化和反序列化，包括Java、PHP、Python和Ruby。

开发人员通常信任用户提供的序列化数据，因为它难以阅读或对用户来说不可读。这种信任假设正是攻击者可以利用的。不安全的反序列化是一种漏洞类型，当攻击者可以操纵序列化对象以在程序中引起意外后果时，就会产生这种漏洞。这可能导致身份验证绕过甚至RCE。例如，如果应用程序从用户获取序列化对象，并使用其中包含的数据来确定谁已登录，恶意用户可能能够篡改该对象并以他人身份进行身份验证。如果应用程序使用不安全的反序列化操作，恶意用户甚至可能能够在对象中嵌入代码片段，并在反序列化期间使其执行。

理解不安全反序列化的最佳方法是学习不同编程语言如何实现序列化和反序列化。由于这些过程在每种语言中看起来都不同，我们将探讨此漏洞在PHP和Java中如何呈现。在继续之前，如果你想要测试本章中的示例代码，需要安装PHP和Java。

你可以按照PHP手册页面（https://www.php.net/manual/en/install.php）上针对你的系统的说明来安装PHP。然后你可以通过命令行运行`php YOUR_PHP_SCRIPT.php`来运行PHP脚本。或者，你可以使用在线PHP测试器（如ExtendsClass，https://extendsclass.com/php.html）来测试示例脚本。在线搜索“PHP tester”可获取更多选项。请注意，并非所有在线PHP测试器都支持序列化和反序列化，因此请确保选择一个支持的。

大多数计算机应该已经安装了Java。如果你在命令行运行`java -version`并看到返回了Java版本号，则无需再次安装Java。否则，你可以在https://java.com/en/download/help/download_options.html找到安装Java的说明。你也可以使用在线Java编译器来测试代码；Tutorials Point有一个在线编译器：https://www.tutorialspoint.com/compile_java_online.php。

### PHP

虽然野外的大多数反序列化漏洞是由Java中的不安全反序列化引起的，但我也发现PHP反序列化漏洞极其常见。在我研究HackerOne上公开披露的反序列化漏洞的项目中，我发现所有披露的反序列化漏洞中有一半是由PHP中的不安全反序列化引起的。我还发现大多数反序列化漏洞被列为高影响或关键影响漏洞；令人难以置信的是，大多数可用于在目标服务器上执行任意代码。

当PHP中发生不安全的反序列化漏洞时，我们有时称之为**PHP对象注入**漏洞。要理解PHP对象注入，你首先需要理解PHP如何序列化和反序列化对象。

当应用程序需要存储PHP对象或通过网络传输它时，它会调用PHP函数`serialize()`来打包它。当应用程序需要使用该数据时，它会调用`unserialize()`来解包并获取底层对象。

例如，以下代码片段将序列化名为`user`的对象：

```php
<?php
① class User{
    public $username;
    public $status;
}
② $user = new User;
③ $user->username = 'vickie';
④ $user->status = 'not admin';
⑤ echo serialize($user);
?>
```

这段PHP代码声明了一个名为`User`的类。每个`User`对象将包含一个`$username`和一个`$status`属性①。然后它创建一个名为`$user`的新`User`对象②。它将`$user`的`$username`属性设置为`'vickie'`③，将其`$status`属性设置为`'not admin'`④。然后，它序列化`$user`对象并打印出表示序列化对象的字符串⑤。

将此代码片段保存为名为`serialize_test.php`的文件，并使用命令`php serialize_test.php`运行它。你应该得到表示用户对象的序列化字符串：

```
O:4:"User":2:{s:8:"username";s:6:"vickie";s:6:"status";s:9:"not admin";}
```

让我们分解这个序列化字符串。PHP序列化字符串的基本结构是`数据类型:数据`。数据类型方面，`b`表示布尔值，`i`表示整数，`d`表示浮点数，`s`表示字符串，`a`表示数组，`O`表示特定类的对象实例。其中一些类型后面会跟有关于数据的附加信息，如下所述：

```
b:THE_BOOLEAN;
i:THE_INTEGER;
d:THE_FLOAT;
s:LENGTH_OF_STRING:"ACTUAL_STRING";
a:NUMBER_OF_ELEMENTS:{ELEMENTS}
O:LENGTH_OF_NAME:"CLASS_NAME":NUMBER_OF_PROPERTIES:{PROPERTIES}
```

以此参考为指导，我们可以看到我们的序列化字符串表示`User`类的对象。它有两个属性。第一个属性名为`username`，值为`vickie`。第二个属性名为`status`，值为`not admin`。名称和值都是字符串。

当你准备再次操作该对象时，你可以使用`unserialize()`反序列化该字符串：

```php
<?php
① class User{
    public $username;
    public $status;
}
$user = new User;
$user->username = 'vickie';
$user->status = 'not admin';
$serialized_string = serialize($user);
② $unserialized_data = unserialize($serialized_string);
③ var_dump($unserialized_data);
var_dump($unserialized_data["status"]);
?>
```

此代码片段的前几行创建一个用户对象，序列化它，并将序列化字符串存储到名为`$serialized_string`的变量中①。然后，它反序列化该字符串并将恢复的对象存储到变量`$unserialized_data`中②。`var_dump()` PHP函数显示变量的值。最后两行显示反序列化对象`$unserialized_data`及其`status`属性③的值。

大多数面向对象编程语言都有类似的序列化和反序列化程序对象的接口，但其序列化对象的格式不同。一些编程语言还允许开发人员序列化为其他标准化格式，如JSON和YAML。

#### 控制变量值

你可能已经注意到这里有些可疑。如果序列化对象未加密或签名，任何人都可以创建`User`对象吗？答案是肯定的！这是不安全的反序列化危害应用程序的常见方式。

利用PHP对象注入漏洞的一种可能方法是操纵对象中的变量。一些应用程序只是将序列化对象作为身份验证方法传递，而不加密或签名，认为序列化本身就能阻止用户篡改值。如果是这种情况，你可以篡改序列化字符串中编码的值：

```php
<?php
class User{
    public $username;
    public $status;
}
$user = new User;
$user->username = 'vickie';
① $user->status = 'admin';
echo serialize($user);
?>
```

在这个我们之前创建的`User`对象示例中，你通过修改PHP脚本将状态更改为`admin`①。然后你可以在代理中拦截传出请求，将新对象插入替换旧对象，查看应用程序是否授予你管理员权限。

你也可以直接更改序列化字符串：

```
O:4:"User":2:{s:8:"username";s:6:"vickie";s:6:"status";s:9:"admin";}
```

如果你直接篡改序列化字符串，记得也要更改字符串的长度标记，因为你的状态字符串长度已更改：

```
O:4:"User":2:{s:8:"username";s:6:"vickie";s:6:"status";s:5:"admin";}
```

#### `unserialize()` 内部原理

要理解`unserialize()`如何导致RCE，让我们看看PHP如何创建和销毁对象。

**PHP魔术方法**是PHP中具有特殊属性的方法名称。如果序列化对象的类实现了任何具有魔术名称的方法，这些方法将具有魔术属性，例如在执行期间的某些时点或满足某些条件时自动运行。其中两个魔术方法是`__wakeup()`和`__destruct()`。

`__wakeup()`方法在实例化期间使用，当程序在内存中创建类的实例时——这正是`unserialize()`所做的；它获取序列化字符串（指定了该类及其对象的属性），并使用该数据创建原始序列化对象的副本。然后它搜索`__wakeup()`方法并执行其中的代码。`__wakeup()`方法通常用于重建对象可能拥有的任何资源，重新建立序列化期间丢失的任何数据库连接，并执行其他重新初始化任务。它在PHP对象注入攻击中通常很有用，因为它提供了访问服务器数据库或程序中其他函数的便捷入口点。

然后程序操作该对象并使用它执行其他操作。当不再存在对反序列化对象的引用时，程序调用`__destruct()`函数来清理该对象。此方法在利用方面通常包含有用的代码。例如，如果`__destruct()`方法包含删除和清理与对象关联的文件的代码，攻击者可能能够通过控制传递给这些函数的输入来破坏文件系统的完整性。

#### 实现RCE

当你控制传递给`unserialize()`的序列化对象时，你就控制了所创建对象的属性。你也可能能够控制传递给自动执行方法（如`__wakeup()`或`__destruct()`）的值。如果你能做到这一点，就有可能实现RCE。

例如，考虑这个易受攻击的代码示例，取自https://www.owasp.org/index.php/PHP_Object_Injection：

```php
① class Example2
{
    private $hook;
    function __construct(){
        // some PHP code...
    }
    function __wakeup(){
        ② if (isset($this->hook)) eval($this->hook);
    }
}
// some PHP code...
③ $user_data = unserialize($_COOKIE['data']);
```

该代码声明了一个名为`Example2`的类。它有一个`$hook`属性和两个方法：`__construct()`和`__wakeup()`①。如果`$hook`不为空，`__wakeup()`函数会将`$hook`中存储的字符串作为PHP代码执行②。PHP `eval()`函数接收字符串并将字符串内容作为PHP代码运行。然后，程序对用户提供的名为`data`的Cookie执行`unserialize()`③。

在这里，你可以实现RCE，因为代码将用户提供的对象传递给`unserialize()`，并且存在一个对象类`Example2`，其魔术方法在对象实例化时自动对用户提供的输入运行`eval()`。

要利用此RCE，你需要将`data` Cookie设置为序列化的`Example2`对象，并将`hook`属性设置为任何你想要执行的PHP代码。你可以使用以下代码片段生成序列化对象：

```php
class Example2
{
    private $hook = "phpinfo();";
}
print ① urlencode(serialize(new Example2));
```

在打印对象之前，我们需要对其进行URL编码①，因为我们将通过Cookie注入对象。将此代码生成的字符串传递到`data` Cookie中将导致服务器执行PHP代码`phpinfo();`，该代码输出服务器上PHP配置的信息。`phpinfo()`函数通常用作漏洞报告中的概念验证函数，以证明成功的PHP命令注入。以下是此攻击期间目标服务器上发生的详细情况：

1. 序列化的`Example2`对象作为`data` Cookie传入程序。
2. 程序对`data` Cookie调用`unserialize()`。
3. 因为`data` Cookie是一个序列化的`Example2`对象，`unserialize()`实例化一个新的`Example2`对象。
4. `unserialize()`函数看到`Example2`类实现了`__wakeup()`，因此调用`__wakeup()`。
5. `__wakeup()`函数查找对象的`$hook`属性，如果它不是NULL，则运行`eval($hook)`。
6. `$hook`属性不是NULL，因为它被设置为`phpinfo();`，因此运行`eval("phpinfo();")`。
7. 你通过执行放在`data` Cookie中的任意PHP代码实现了RCE。

#### 使用其他魔术方法

到目前为止，我们已经提到了魔术方法`__wakeup()`和`__destruct()`。实际上，在尝试利用`unserialize()`漏洞时，你会发现四个特别有用的魔术方法：`__wakeup()`、`__destruct()`、`__toString()`和`__call()`。

与`__wakeup()`和`__destruct()`（在对象创建时始终执行）不同，`__toString()`方法仅在对象被视为字符串时调用。它允许一个类决定当其对象之一被视为字符串时如何反应。例如，它可以决定如果对象被传递给`echo()`或`print()`函数时显示什么。你将在“使用POP链”部分（第238页）中看到在反序列化攻击中使用此方法的示例。

当调用未定义的方法时，程序会调用`__call()`方法。例如，调用`$object->undefined($args)`将变成`$object->__call('undefined', $args)`。同样，此魔术方法的可利用性在很大程度上取决于其实现方式。有时，当应用程序代码包含错误或允许用户定义要调用的方法名称时，攻击者可以利用此魔术方法。

你通常会发现在利用这四个魔术方法时最有用，但还有许多其他方法存在。如果此处提到的那些不可利用，可能值得检查类对其他魔术方法的实现，看看你是否可以从那里开始攻击。在https://www.php.net/manual/en/language.oop5.magic.php阅读更多关于PHP魔术方法的信息。

#### 使用POP链

到目前为止，你知道当攻击者控制传递给`unserialize()`的序列化对象时，他们可以控制所创建对象的属性。这使他们有机会通过选择传递给`__wakeup()`等魔术方法的值来劫持应用程序的流程。

这种利用有时...是有效的。但这种方法有一个问题：如果类的已声明魔术方法不包含任何在利用方面有用的代码怎么办？例如，有时可用于对象注入的类只包含少数方法，其中没有一个包含代码注入机会。那么不安全的反序列化就没用了，利用就失败了，对吗？

即使在这种情况下，我们也有另一种实现RCE的方法：**POP链**。面向属性编程（POP）链是一种利用类型，其名称源于攻击者控制所有反序列化对象属性的事实。POP链通过将代码片段（称为小工具）串联在一起，以实现攻击者的最终目标。这些小工具是从代码库中借用的代码片段。POP链使用魔术方法作为其初始小工具。攻击者然后可以使用这些方法来调用其他小工具。

如果这听起来很抽象，请考虑以下示例应用程序代码，取自https://owasp.org/www-community/vulnerabilities/PHP_Object_Injection：

```php
class Example
{
    ① private $obj;
    function __construct()
    {
        // some PHP code...
    }
    function __wakeup()
    {
        ② if (isset($this->obj)) return $this->obj->evaluate();
    }
}
class CodeSnippet
{
    ③ private $code;
    ④ function evaluate()
    {
        eval($this->code);
    }
}
// some PHP code...
⑤ $user_data = unserialize($_POST['data']);
// some PHP code...
```

在此应用程序中，代码定义了两个类：`Example`和`CodeSnippet`。`Example`有一个名为`obj`的属性①，当反序列化`Example`对象时，会调用其`__wakeup()`函数，该函数调用`obj`的`evaluate()`方法②。`CodeSnippet`类有一个名为`code`的属性，包含要执行的代码字符串③，以及一个`evaluate()`方法④，它对`code`字符串调用`eval()`。

在代码的另一部分，程序接受来自用户的POST参数`data`并对其调用`unserialize()`⑤。

由于最后一行包含不安全的反序列化漏洞，攻击者可以使用以下代码生成序列化对象：

```php
class CodeSnippet
{
    private $code = "phpinfo();";
}
class Example
{
    private $obj;
    function __construct()
    {
        $this->obj = new CodeSnippet;
    }
}
print urlencode(serialize(new Example));
```

此代码片段定义了一个名为`CodeSnippet`的类，并将其`code`属性设置为`phpinfo();`。然后它定义了一个名为`Example`的类，并在实例化时将其`obj`属性设置为一个新的`CodeSnippet`实例。最后，它创建一个`Example`实例，序列化它，并对序列化字符串进行URL编码。攻击者然后可以将生成的字符串输入POST参数`data`。

请注意，攻击者的序列化对象使用了应用程序源代码中其他位置找到的类和属性名称。因此，当程序收到精心构造的`data`字符串时，将执行以下操作：

首先，它将反序列化对象并创建一个`Example`实例。然后，由于`Example`实现了`__wakeup()`，程序将调用`__wakeup()`并看到`obj`属性被设置为一个`CodeSnippet`实例。最后，它将调用`obj`的`evaluate()`方法，由于攻击者将`code`属性设置为`phpinfo();`，该方法运行`eval("phpinfo();")`。攻击者能够执行他们选择的任何PHP代码。

POP链通过链式重用应用程序代码库中的代码来实现RCE。让我们看另一个如何使用POP链实现SQL注入的示例。此示例也取自https://owasp.org/www-community/vulnerabilities/PHP_Object_Injection。

假设应用程序在代码中某处定义了一个名为`Example3`的类，并反序列化来自POST参数`data`的未清理用户输入：

```php
class Example3
{
    protected $obj;
    function __construct()
    {
        // some PHP code...
    }
    ① function __toString()
    {
        if (isset($this->obj)) return $this->obj->getValue();
    }
}
// some PHP code...
$user_data = unserialize($_POST['data']);
// some PHP code...
```

注意`Example3`实现了`__toString()`魔术方法①。在这种情况下，当`Example3`实例被视为字符串时，它将返回对其`$obj`属性运行`getValue()`方法的结果。

同时假设，在应用程序的某处，代码定义了类`SQL_Row_Value`。它有一个名为`getValue()`的方法，执行SQL查询。该SQL查询从`SQL_Row_Value`实例的`$_table`属性获取输入：

```php
class SQL_Row_Value
{
    private $_table;
    // some PHP code...
    function getValue($id)
    {
        $sql = "SELECT * FROM {$this->_table} WHERE id = " . (int)$id;
        $result = mysql_query($sql, $DBFactory::getConnection());
        $row = mysql_fetch_assoc($result);
        return $row['value'];
    }
}
```

攻击者可以通过控制`Example3`中的`$obj`来实现SQL注入。以下代码将创建一个`Example3`实例，其`$obj`设置为`SQL_Row_Value`实例，`$_table`设置为字符串`"SQL Injection"`：

```php
class SQL_Row_Value
{
    private $_table = "SQL Injection";
}
class Example3
{
    protected $obj;
    function __construct()
    {
        $this->obj = new SQL_Row_Value;
    }
}
print urlencode(serialize(new Example3));
```

结果，每当攻击者的`Example3`实例被视为字符串时，其`$obj`的`get_Value()`方法将被执行。这意味着`SQL_Row_Value`的`get_Value()`方法将以`$_table`字符串设置为`"SQL Injection"`执行。

攻击者实现了有限的SQL注入，因为他们可以控制传递给SQL查询`SELECT * FROM {$this->_table} WHERE id = " . (int)$id;`的字符串。

POP链类似于返回导向编程（ROP）攻击，这是一种在二进制利用中使用的有趣技术。你可以在维基百科上阅读更多信息：https://en.wikipedia.org/wiki/Return-oriented_programming。

### Java

现在你已理解PHP中不安全反序列化的工作原理，让我们探索另一种容易出现这些漏洞的编程语言：Java。Java应用程序容易出现不安全的反序列化漏洞，因为其中许多应用程序处理序列化对象。要理解如何利用Java中的反序列化漏洞，让我们看看Java中序列化和反序列化是如何工作的。

对于Java对象要可序列化，它们的类必须实现`java.io.Serializable`接口。这些类还实现了特殊方法`writeObject()`和`readObject()`，分别处理该类对象的序列化和反序列化。让我们看一个示例。将此代码存储在名为`SerializeTest.java`的文件中：

```java
import java.io.ObjectInputStream;
import java.io.ObjectOutputStream;
import java.io.FileInputStream;
import java.io.FileOutputStream;
import java.io.Serializable;
import java.io.IOException;

① class User implements Serializable{
    ② public String username;
}

public class SerializeTest{
    public static void main(String args[]) throws Exception{
        ③ User newUser = new User();
        ④ newUser.username = "vickie";
        FileOutputStream fos = new FileOutputStream("object.ser");
        ObjectOutputStream os = new ObjectOutputStream(fos);
        ⑤ os.writeObject(newUser);
        os.close();

        FileInputStream is = new FileInputStream("object.ser");

        ⑥ User storedUser = (User)ois.readObject();

```

然后，在你存储文件的目录中，运行以下命令。它们将编译程序并运行代码：

```bash
$ javac SerializeTest.java
$ java SerializeTest
```

你应该看到输出字符串`vickie`。让我们稍微分解一下程序。首先，我们定义一个名为`User`的类，它实现了`Serializable`①。只有实现`Serializable`的类才能被序列化和反序列化。`User`类有一个`username`属性，用于存储用户的用户名②。

然后，我们创建一个新的`User`对象③并将其用户名设置为字符串`"vickie"`④。我们将`newUser`的序列化版本写入并存储到文件`object.ser`中⑤。最后，我们从文件中读取对象，反序列化它，并打印出用户的用户名⑥。

要通过不安全的反序列化漏洞利用Java应用程序，我们首先必须找到一个入口点，通过该入口点插入恶意序列化对象。在Java应用程序中，可序列化对象通常用于在HTTP头、参数或Cookie中传输数据。

Java序列化对象不像PHP序列化字符串那样是人类可读的。它们通常也包含不可打印字符。但它们确实有一些特征可以帮助你识别它们并找到潜在的利用入口点：

- 十六进制以`AC ED 00 05`开头，或base64以`rO0`开头。（你可能会在HTTP请求中看到它们作为Cookie或参数。）
- HTTP消息的`Content-Type`头设置为`application/x-java-serialized-object`。

由于Java序列化对象包含许多特殊字符，在传输前对它们进行编码是很常见的，因此也要注意这些特征的不同编码版本。

发现用户提供的序列化对象后，你可以尝试的第一件事是通过篡改对象中存储的信息来操纵程序逻辑。例如，如果Java对象被用作访问控制的Cookie，你可以尝试更改对象中存在的用户名、角色名称和其他身份标记，重新序列化它，并将其发送回应用程序。你也可以尝试篡改对象中任何文件路径、文件指定符或控制流值的值，看看是否可以改变程序流程。

有时，当代码不限制应用程序允许反序列化的类时，它可以反序列化任何它可以访问的可序列化类。这意味着攻击者可以创建任何类的自己的对象。潜在攻击者可以通过构造正确类的对象来实现RCE，这些对象可以导致任意命令的执行。

#### 实现RCE

从Java反序列化漏洞到RCE的路径可能很复杂。要获得代码执行，你通常需要使用一系列小工具来达到所需的代码执行方法。这与在PHP中使用POP链利用反序列化漏洞的方式类似，因此我们在此不再重复整个过程。在Java应用程序中，你会在应用程序加载的库中找到小工具。使用应用程序范围内的小工具，创建一个最终导致RCE的方法调用链。

寻找和链式小工具以制定利用可能很耗时。你还受限于应用程序可用的类，这可能限制你的利用能做什么。为了节省时间，尝试使用流行库中的小工具创建利用链，如Apache Commons-Collections、Spring Framework、Apache Groovy和Apache Commons FileUpload。你会在网上找到许多已发布的此类库。

#### 使用Ysoserial自动化利用

Ysoserial（https://github.com/frohoff/ysoserial/）是一个你可以用来生成利用Java不安全反序列化漏洞的载荷的工具，通过避免你自己开发小工具链来节省大量时间。Ysoserial使用在常见Java库中发现的小工具链集合来制定利用对象。使用Ysoserial，你可以使用单个命令从指定库创建使用小工具链的恶意Java序列化对象：

```bash
$ java -jar ysoserial.jar gadget_chain command_to_execute
```

例如，要创建一个使用Commons-Collections库中的小工具链在目标主机上打开计算器的载荷，执行此命令：

```bash
$ java -jar ysoserial.jar CommonsCollections1 calc.exe
```

Ysoserial生成的小工具链都赋予你在系统上执行命令的能力。该程序获取你指定的命令并生成执行该命令的序列化对象。

有时用于小工具链的库似乎很明显，但通常这是一个试错过程，因为你需要发现目标应用程序实现了哪些易受攻击的库。这就是良好的侦察可以帮助你的地方。

你可以在GitHub上找到更多关于利用Java反序列化的资源：https://github.com/GrrrDog/Java-Deserialization-Cheat-Sheet/。

## 防御措施

防御反序列化漏洞很困难。保护应用程序免受这些漏洞攻击的最佳方法因所使用的编程语言、库和序列化格式而异。没有放之四海而皆准的解决方案。

你应该确保在未经适当检查的情况下不会反序列化任何被用户输入污染的数据。如果反序列化是必要的，使用白名单将反序列化限制在少数允许的类中。

你也可以使用简单数据类型（如字符串和数组），而不是在传输时需要序列化的对象。并且，为了防止序列化Cookie被篡改，你可以将会话状态保存在服务器端，而不是依赖用户输入获取会话信息。最后，你应该关注补丁并确保依赖项是最新的，以避免通过第三方代码引入反序列化漏洞。

一些开发人员尝试通过识别常见易受攻击的类并将其从应用程序中移除来缓解反序列化漏洞。这有效地限制了攻击者在小工具链中可用的小工具。然而，这不是一种可靠的保护形式。限制小工具可以是一层很好的防御，但黑客是有创造力的，总能在其他库中找到更多小工具，想出创造性的方法来实现相同的结果。重要的是解决此漏洞的根本原因：应用程序不安全地反序列化用户数据。

OWASP反序列化速查表是学习如何为你的特定技术防止反序列化缺陷的优秀资源：https://cheatsheetseries.owasp.org/cheatsheets/Deserialization_Cheat_Sheet.html。

## 寻找不安全的反序列化

进行源代码审查是检测反序列化漏洞最可靠的方法。从本章的示例中可以看出，发现不安全的反序列化漏洞的最快方法是在源代码中搜索反序列化函数，并检查用户输入是否被鲁莽地传递给它。例如，在PHP应用程序中，查找`unserialize()`；在Java应用程序中，查找`readObject()`。在Python和Ruby应用程序中，分别查找`pickle.loads()`和`Marshall.load()`函数。

但许多漏洞赏金猎人能够在无需检查任何代码的情况下找到反序列化漏洞。以下是一些你可以用来在无法访问源代码的情况下发现不安全反序列化的策略。

首先，密切关注传递到应用程序中的大型数据块。例如，base64字符串`Tzo0OiJVc2VyIjoyOntzOjg6InVzZXJuYW1lIjtzOjY6InZpY2tpZSI7czo2OiJzdGF0dXMiO3M6OToibm90IGFkbWluIjt9`是PHP序列化字符串`O:4:"User":2:{s:8:"username";s:6:"vickie";s:6:"status";s:9:"not admin";}`的base64编码版本。

这是`Person`类（具有`name`属性`vickie`）的序列化Python对象的base64表示：`gASVLgAAAAAAAACMCF9fbWFpbl9flIwGUGVyc29ulJOUKYGUfZSMBG5hbWWUjAZWaWNraWWUc2Iu`。

这些大型数据块可能表示对象注入机会的序列化对象。如果数据被编码，尝试解码它。传递给Web应用程序的大多数编码数据都是用base64编码的。例如，如前所述，Java序列化对象通常以十六进制字符`AC ED 00 05`或base64字符`rO0`开头。还要注意HTTP请求或响应的`Content-Type`头。例如，`Content-Type`设置为`application/x-java-serialized-object`表示应用程序正在通过Java序列化对象传递信息。

另外，你可以从寻找容易出现反序列化缺陷的功能开始。寻找可能需要反序列化用户提供的对象的功能，如数据库输入、身份验证令牌和HTML表单参数。

一旦找到用户提供的序列化对象，你需要确定它是什么类型的序列化对象。它是PHP对象、Python对象、Ruby对象还是Java对象？阅读每种编程语言的文档以熟悉其序列化对象的结构。

最后，尝试使用我提到的一种技术篡改该对象。如果应用程序将序列化对象用作身份验证机制，尝试篡改字段以查看你是否可以以其他人身份登录。你也可以尝试通过小工具链实现RCE或SQL注入。

## 提升攻击影响

本章已经描述不安全的反序列化漏洞通常会导致远程代码执行，赋予攻击者广泛的能力来影响应用程序。因此，反序列化漏洞是有价值且影响巨大的漏洞。即使RCE不可能，你也可能能够实现身份验证绕过或以其他方式干预应用程序的逻辑流程。

然而，当漏洞依赖于一个不明显的入口点，或需要一定级别的应用程序权限才能利用，或易受攻击的功能对未经身份验证的用户不可用时，不安全的反序列化的影响可能会受到限制。

在提升反序列化缺陷时，要考虑漏洞赏金计划的范围和规则。反序列化漏洞可能很危险，因此请确保在尝试操纵程序逻辑或执行任意代码时不要对目标应用程序造成损害。阅读第18章关于如何为RCE创建安全PoC的技巧。
