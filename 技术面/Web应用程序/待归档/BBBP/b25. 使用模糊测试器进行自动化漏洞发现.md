# 第25章：使用模糊测试器进行自动化漏洞发现（Automatic Vulnerability Discovery Using Fuzzers）

每当我接触一个新目标时，我更喜欢手动搜索漏洞。手动测试非常适合发现新的和意想不到的攻击向量，它还可以帮助你深入学习新的安全概念。但手动测试也耗费大量时间和精力，因此，与自动化侦察一样，你应该努力至少自动化部分漏洞发现过程。自动化测试可以帮助你在短时间内梳理出大量漏洞。

事实上，表现最好的漏洞赏金猎人将大部分黑客过程自动化。他们自动化侦察工作，并编写程序不断在他们选择的目标上寻找漏洞。每当他们的工具通知他们存在潜在漏洞时，他们会立即验证并报告。

通过一种称为**模糊测试（fuzzing）**的自动化技术发现的漏洞，现在占据了大多数新CVE条目。虽然模糊测试通常与二进制漏洞利用开发相关联，但它也可以用于发现Web应用程序中的漏洞。在本章中，我们将讨论使用两个工具（Burp Intruder和Wfuzz）对Web应用程序进行模糊测试，以及它可以帮助你实现什么。

## 什么是模糊测试？

**模糊测试**是向应用程序发送大量无效和意外数据，并监控应用程序是否出现异常的过程。有时黑客为特定目的构造这些无效数据；其他时候，他们随机或使用算法生成数据。在这两种情况下，目标都是诱导意外行为（如崩溃），然后检查错误是否导致可利用的漏洞。模糊测试对于暴露内存泄漏、控制流问题和竞争条件等漏洞特别有用。例如，你可以使用American Fuzzy Lop（AFL，https://github.com/google/AFL/）等工具对编译后的二进制文件进行漏洞模糊测试。

有许多种模糊测试，每种都针对测试应用程序中特定类型的问题进行了优化。Web应用程序模糊测试是一种试图暴露常见Web漏洞的技术，如注入问题、XSS和身份验证绕过。

### Web模糊测试器的工作原理

Web模糊测试器通过将常见漏洞的载荷插入Web应用程序注入点，自动生成恶意请求。然后它们发出这些请求并跟踪服务器的响应。

为了更好地理解这个过程，让我们看看开源Web应用程序模糊测试器Wfuzz（https://github.com/xmendez/wfuzz/）的工作原理。当提供一个单词列表和一个端点时，Wfuzz会将所有标记为`FUZZ`的位置替换为单词列表中的字符串。例如，以下Wfuzz命令会将URL中的`FUZZ`实例替换为`common_paths.txt`单词列表中的每个字符串：

```bash
$ wfuzz -w common_paths.txt http://example.com/FUZZ
```

你应该为每种要扫描的漏洞类型提供不同的单词列表。例如，你可以通过向模糊测试器提供常见文件路径的单词列表，使其表现得像目录枚举器。结果，Wfuzz将生成枚举example.com上路径的请求：

```
http://example.com/admin
http://example.com/admin.php
http://example.com/cgi-bin
http://example.com/secure
http://example.com/authorize.php
http://example.com/cron.php
http://example.com/administrator
```

你也可以通过提供潜在的ID值，使模糊测试器表现得像IDOR扫描器：

```bash
$ wfuzz -w ids.txt http://example.com/view_inbox?user_id=FUZZ
```

假设`ids.txt`是一个数字ID列表。如果`example.com/view_inbox`是用于访问不同用户邮件收件箱的端点，此命令将导致Wfuzz生成一系列尝试访问其他用户收件箱的请求，如下所示：

```
http://example.com/view_inbox?user_id=1
http://example.com/view_inbox?user_id=2
http://example.com/view_inbox?user_id=3
```

一旦收到服务器的响应，你可以分析它们以查看特定路径中是否真的存在文件，或者你是否可以访问其他用户的邮件收件箱。如你所见，与漏洞扫描器不同，模糊测试器在测试漏洞方面非常灵活。你可以通过指定不同的载荷和注入点来最大限度地定制它们。

## 模糊测试过程

现在让我们来看看你可以采取哪些步骤将模糊测试集成到你的黑客过程中！当你面对一个目标时，如何开始对其进行模糊测试？应用程序的模糊测试过程可以分为四个步骤。你可以从确定应用程序中可以模糊测试的端点开始。然后，决定载荷列表并开始模糊测试。最后，监控模糊测试器的结果并寻找异常。

### 第1步：确定数据注入点

对Web应用程序进行模糊测试时，首先要做的是识别用户可以向应用程序提供输入的方式。哪些端点接受用户输入？使用了哪些参数？应用程序使用哪些头？你可以将这些参数和头视为数据注入点或数据入口点，因为这些是攻击者可以向应用程序注入数据的位置。

到目前为止，你应该已经对各种用户输入机会上应寻找哪些漏洞有了直觉。例如，当你看到数字ID时，应测试IDOR；当你看到搜索栏时，应测试反射型XSS。根据你发现的目标上的数据注入点容易受到的漏洞类型对它们进行分类：

**用于测试IDOR的数据入口点**

```
GET /email_inbox?user_id=FUZZ
Host: example.com

POST /delete_user
Host: example.com
(POST request parameter)
user_id=FUZZ
```

**用于测试XSS的数据入口点**

```
GET /search?q=FUZZ
Host: example.com

POST /send_email
Host: example.com
(POST request parameter)
user_id=abc&title=FUZZ&body=FUZZ
```

### 第2步：决定载荷列表

识别数据注入点以及你可能能够利用每个注入点利用的漏洞后，确定向每个注入点提供什么数据。你应该使用最可能漏洞的常见载荷对每个注入点进行模糊测试。向大多数数据入口点提供XSS载荷和SQL注入载荷也是值得的。

使用好的载荷列表对于通过模糊测试发现漏洞至关重要。我推荐下载Daniel Miessler的SecLists（https://github.com/danielmiessler/SecLists/）和Max Woolf的Big List of Naughty Strings（https://github.com/minimaxir/big-list-of-naughty-strings/），以获得相当全面的载荷列表，可用于Web应用程序模糊测试。这些列表包括最常见Web漏洞的载荷，如XSS、SQL注入和XXE。另一个用于枚举和漏洞模糊测试的优秀单词列表数据库是FuzzDB（https://github.com/fuzzdb-project/fuzzdb/）。

除了使用已知载荷，你也可以尝试随机生成载荷。特别是，创建极长的载荷、包含各种编码的奇怪字符的载荷，以及包含某些特殊字符（如换行符、换页符等）的载荷。通过向应用程序提供此类垃圾数据，你可能能够检测到意外行为并发现新的漏洞类别！你可以使用第5章中学到的bash脚本来自动生成随机载荷。如何生成包含特定特殊字符的随机长度字符串？提示：你可以使用`for`循环或Unix系统上的`/dev/random`文件。

### 第3步：进行模糊测试

接下来，系统地将你的载荷列表提供给应用程序的数据入口点。有几种方法可以做到这一点，具体取决于你的需求和编程技能。自动化模糊测试的最简单方法是使用Burp Intruder（图25-1）。Intruder提供了一个带有图形用户界面（GUI）的模糊测试器，可与你的Burp代理无缝集成。每当你遇到想要模糊测试的请求时，你可以右键点击它并选择`Send to Intruder`。

在Intruder选项卡中，你可以配置模糊测试器设置，选择数据注入点和载荷列表，然后开始模糊测试。要将请求的一部分添加为数据注入点，请高亮请求的该部分，然后点击窗口右侧的`Add`。

*（图25-1：Burp Intruder载荷位置选择）*

然后在Payloads选项卡中选择预定义的载荷列表或生成载荷列表（图25-2）。例如，你可以生成数字列表或随机生成的字母数字字符串。

*（图25-2：在Burp Intruder中选择载荷列表）*

Burp Intruder易于使用，但它有一个缺点：Burp的免费版本限制了模糊测试器的功能，并对攻击进行时间限制，这意味着它会减慢你的模糊测试速度并限制你在特定时间段内可以发送的请求数量。你每分钟只能发送一定数量的请求，这使得Intruder的效率远低于没有时间限制的模糊测试器。除非你需要GUI或拥有Burp的专业版，否则最好使用开源模糊测试器，如OWASP ZAP的模糊测试器或Wfuzz。你将在本章后面的“使用Wfuzz进行模糊测试”中学习如何使用Wfuzz对目标进行模糊测试。

请注意，有时为了阻止对应用程序操作的干扰，对模糊测试器进行时间限制是必要的。这对大公司来说不应该是个问题，但如果你在没有时间限制的情况下对没有扩展架构的小公司进行模糊测试，可能会意外发起DoS攻击。在进行模糊测试时，务必谨慎行事并从公司获得许可！

### 第4步：监控结果

分析模糊测试器返回的结果，寻找服务器响应中的模式和异常。寻找什么取决于你使用的载荷集和你希望发现的漏洞。例如，当你使用模糊测试器查找文件路径时，状态码是文件是否存在的良好指示。如果路径名返回的状态码在200范围内，你可能发现了一个有效路径。另一方面，如果状态码是404，文件路径可能无效。

在进行SQL注入模糊测试时，你可能想寻找响应内容长度或时间的变化。如果某个载荷返回的内容比其他载荷长，可能表明你的载荷能够影响数据库的操作并改变其返回的内容。另一方面，如果你使用在应用程序中诱导时间延迟的载荷列表，检查是否有任何载荷使服务器响应时间比平均值慢。使用你在本书中学到的知识来识别漏洞存在的关键指标。

## 使用Wfuzz进行模糊测试

现在你了解了总体方法，让我们通过一个使用Wfuzz的实际示例来演示，你可以使用以下命令安装Wfuzz：

```bash
$ pip install wfuzz
```

模糊测试在侦察阶段和漏洞发现阶段都很有用：你可以使用模糊测试来枚举文件路径、暴力破解身份验证、测试常见Web漏洞等。

### 路径枚举

在侦察阶段，尝试使用Wfuzz枚举服务器上的文件路径。以下是一个可用于枚举example.com上文件路径的命令：

```bash
$ wfuzz -w wordlist.txt -f output.txt --hc 404 --follow http://example.com/FUZZ
```

`-w`标志选项指定用于枚举的单词列表。在这种情况下，你应该选择一个为你的目标所使用的技术设计的好的路径枚举单词列表。`-f`标志指定输出文件位置。这里，我们将结果存储到当前目录中名为`output.txt`的文件中。`--hc 404`选项告诉Wfuzz排除任何具有404状态码的响应。记住，此代码代表“文件未找到”。使用此过滤器，我们可以轻松地从结果列表中丢弃不指向有效文件或目录的URL。`--follow`标志告诉Wfuzz遵循所有HTTP重定向，以便我们的结果显示URL的实际目的地。

让我们使用一个简单的单词列表运行该命令，看看我们能在facebook.com上找到什么。出于演示目的，我们使用一个仅包含四个单词的单词列表`wordlist.txt`：

```
authorize.php
cron.php
administrator
secure
```

运行此命令以枚举Facebook上的路径：

```bash
$ wfuzz -w wordlist.txt -f output.txt --hc 404 --follow http://facebook.com/FUZZ
```

让我们看看结果。从左到右，Wfuzz报告的每一行都有以下列：请求ID、HTTP响应码、响应行数长度、响应词数长度、响应字符数长度和使用的载荷：

```
********************************************************
* Wfuzz 2.4.6 - The Web Fuzzer
*
********************************************************
Target: http://facebook.com/FUZZ
Total requests: 4
===================================================================
ID     Response   Lines    Word     Chars     Payload
===================================================================
000000004: 200      20 L     2904 W   227381 Ch "secure"
Total time: 1.080132
Processed Requests: 4
Filtered Requests: 3
Requests/sec.: 3.703250
```

你可以看到这些结果只包含一个响应。这是因为我们过滤掉了不相关的结果。由于我们丢弃了所有404响应，我们现在可以专注于指向实际路径的URL。看起来`/secure`返回了200 OK状态码，是facebook.com上的一个有效路径。

### 暴力破解身份验证

在目标上收集到有效文件路径后，你可能会发现服务器上的某些页面受到保护。大多数情况下，这些页面会有403 Forbidden响应码。那你该怎么办？

好吧，你可以尝试暴力破解页面上的身份验证。例如，有时页面使用HTTP的基本身份验证方案作为访问控制。在这种情况下，你可以使用Wfuzz模糊测试身份验证头，使用`-H`标志指定自定义头：

```bash
$ wfuzz -w wordlist.txt -H "Authorization: Basic FUZZ" http://example.com/admin
```

基本身份验证方案使用名为`Authorization`的头来传输凭据，这些凭据是用户名和密码对的base64编码字符串。例如，如果你的用户名和密码是`admin`和`password`，你的身份验证字符串将是`base64("admin:password")`，即`YWRtaW46cGFzc3dvcmQ=`。你可以使用脚本从常见的用户名和密码对生成身份验证字符串，然后使用Wfuzz将它们提供给目标受保护页面。

另一种暴力破解基本身份验证的方法是使用Wfuzz的`--basic`选项。给定输入的用户名和密码列表后，此选项自动构造身份验证字符串以暴力破解基本身份验证。在Wfuzz中，你可以用`FUZZ`、`FUZ2Z`、`FUZ3Z`等标记不同的注入点。这些注入点将分别用传入的第一个、第二个和第三个单词列表进行模糊测试。以下是一个可以同时模糊测试用户名字段和密码字段的命令：

```bash
$ wfuzz -w usernames.txt -w passwords.txt --basic FUZZ:FUZ2Z http://example.com/admin
```

`usernames.txt`文件包含两个用户名：`admin`和`administrator`。`passwords.txt`文件包含三个密码：`secret`、`pass`和`password`。如你所见，Wfuzz为你的列表中的每个用户名和密码组合发送一个请求：

```
********************************************************
* Wfuzz 2.4.6 - The Web Fuzzer
*
********************************************************
Target: http://example.com/admin
Total requests: 6
===================================================================
ID     Response   Lines    Word     Chars     Payload
===================================================================
000000002: 404      46 L     120 W    1256 Ch  "admin - pass"
000000001: 404      46 L     120 W    1256 Ch  "admin - secret"
000000003: 404      46 L     120 W    1256 Ch  "admin - password"
000000006: 404      46 L     120 W    1256 Ch  "administrator - password"
000000004: 404      46 L     120 W    1256 Ch  "administrator - secret"
000000005: 404      46 L     120 W    1256 Ch  "administrator - pass"
Total time: 0.153867
Processed Requests: 6
Filtered Requests: 0
Requests/sec.: 38.99447
```

使用暴力破解绕过身份验证的其他方法包括切换`User-Agent`头或伪造用于身份验证的自定义头。你可以通过使用Wfuzz暴力破解HTTP请求头来完成所有这些操作。

### 测试常见Web漏洞

最后，Wfuzz可以帮助你自动测试常见Web漏洞。首先，你可以使用Wfuzz模糊测试URL参数并测试IDOR和开放重定向等漏洞。通过在URL中放置`FUZZ`关键字来模糊测试URL参数。例如，如果站点对聊天消息使用数字ID，使用以下命令测试各种ID：

```bash
$ wfuzz -w wordlist.txt http://example.com/view_message?message_id=FUZZ
```

然后通过检查响应码或响应内容长度来查找有效ID，并查看你是否可以访问他人的消息。指向有效页面的ID通常返回200响应码或更长的网页。

你也可以将载荷插入重定向参数以测试开放重定向：

```bash
$ wfuzz -w wordlist.txt http://example.com?redirect=FUZZ
```

要检查载荷是否引起重定向，请打开Wfuzz的`follow`（`--follow`）和`verbose`（`-v`）选项。`follow`选项指示Wfuzz遵循重定向。`verbose`选项显示更详细的结果，包括请求期间是否发生了重定向。看看你是否可以构造一个将用户重定向到你网站的载荷：

```bash
$ wfuzz -w wordlist.txt -v --follow http://example.com?redirect=FUZZ
```

最后，通过使用常见载荷列表模糊测试URL参数、POST参数或其他用户输入位置，来测试XSS和SQL注入等漏洞。

在使用Wfuzz测试XSS时，尝试创建一个将用户重定向到你页面的脚本列表，然后打开`verbose`选项以监控任何重定向。或者，你可以使用Wfuzz的内容过滤器来检查反射的XSS载荷。`--filter`标志允许你设置结果过滤器。一个特别有用的过滤器是`content~STRING`，它返回包含指定`STRING`的响应：

```bash
$ wfuzz -w xss.txt --filter "content~FUZZ" http://example.com/get_user?user_id=FUZZ
```

对于SQL注入漏洞，尝试使用预制的SQL注入单词列表，并监控每个载荷的响应时间、响应码或响应长度的异常。如果你使用包含时间延迟的SQL注入载荷，查找长响应时间。如果大多数载荷返回某个响应码但有一个不是，进一步调查该响应以查看是否存在SQL注入。更长的响应长度也可能表明你能够从数据库中提取数据。

以下命令使用单词列表`sqli.txt`测试SQL注入。你可以使用`-d`标志指定POST请求体数据：

```bash
$ wfuzz -w sqli.txt -d "user_id=FUZZ" http://example.com/get_user
```

## 模糊测试与静态分析

在第22章中，我讨论了源代码审查对发现Web漏洞的有效性。你现在可能会想：为什么不直接进行代码的静态分析呢？为什么要进行模糊测试？

静态代码分析是识别攻击者可以利用的漏洞和不正确编程实践的宝贵工具。然而，静态分析有其局限性。

首先，它在非运行状态下评估应用程序。对应用程序进行代码审查无法让你模拟应用程序在运行且客户端与之交互时的反应，而且很难预测攻击者可能提供的所有可能恶意输入。静态代码分析还需要访问应用程序的源代码。当你进行黑盒测试时（如在漏洞赏金场景中），除非你能泄露应用程序的源代码或识别应用程序使用的开源组件，否则你可能无法获得源代码。这使得模糊测试成为测试方法论的一个很好的补充，因为你不需要源代码来对应用程序进行模糊测试。

## 模糊测试的缺陷

当然，模糊测试并不是解决所有漏洞检测的万能药。这种技术有其局限性，其中之一是服务器的速率限制。在远程黑盒测试中，你可能无法在不被服务器检测到你的活动或触发某种速率限制的情况下向应用程序发送大量载荷。这可能导致你的测试变慢，或者服务器可能禁止你访问服务。

在黑盒测试中，准确评估通过模糊测试发现的漏洞的影响也可能很困难，因为你无法访问代码，因此获得的应用程序行为样本有限。你通常需要进一步进行手动测试来分类漏洞的有效性和重要性。将模糊测试视为金属探测器：它只是将你指向可疑位置。最终，你需要更仔细地检查，看看是否发现了有价值的东西。

另一个限制涉及模糊测试可以发现的漏洞类别。虽然模糊测试擅长发现某些基本漏洞（如XSS和SQL注入），有时可以帮助发现新的漏洞类型，但它对检测业务逻辑错误或需要多个步骤才能利用的漏洞帮助不大。这些复杂漏洞是潜在攻击的重要来源，仍需要手动梳理。虽然模糊测试应该是你测试过程中的重要组成部分，但它绝不应是唯一的部分。

## 添加到你的自动化测试工具包

像模糊测试器或扫描器这样的自动化测试工具可以帮助你发现一些漏洞，但如果你不花时间了解测试工具包中每个工具的工作原理，它们往往会阻碍你的学习进度。因此，在将工具添加到你的工作流程之前，请务必花时间阅读工具的文档并理解其工作原理。你应该对所有使用的侦察和测试工具都这样做。

除了阅读工具文档外，如果工具是开源的，我还建议阅读其源代码。这可以让你了解其他黑客的方法论，并提供有关该领域最好的黑客如何测试的见解。最后，通过学习其他人如何自动化黑客工作，你也会开始学习如何编写自己的工具。

这里有一个挑战给你：阅读Sublist3r（https://github.com/aboul3la/Sublist3r/）和Wfuzz（https://github.com/xmendez/wfuzz/）的源代码。这两个都是用Python编写的易于理解的工具。Sublist3r是一个子域名枚举工具，而Wfuzz是一个Web应用程序模糊测试器。Sublist3r如何进行子域名枚举？Wfuzz如何对Web应用程序进行模糊测试？你能从它们接收输入目标开始到输出结果结束，写出它们的应用逻辑吗？你能用不同的方法重写它们实现的功能吗？

一旦你对自己的工具如何工作有了扎实的理解，尝试修改它们以添加新功能！如果你认为其他人会觉得你的功能有用，你可以为开源项目做出贡献：建议将你的功能添加到工具的官方版本中。

理解你的工具和漏洞利用如何工作是成为黑客大师的关键。祝你好运，黑客愉快！
