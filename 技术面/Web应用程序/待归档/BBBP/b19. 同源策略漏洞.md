# 第19章：同源策略漏洞（Same-Origin Policy Vulnerabilities）

第3章介绍了同源策略（SOP），这是现代Web应用程序中部署的基本防御措施之一。SOP限制了一个来源的脚本如何与不同来源站点的资源进行交互，对于防止许多常见的Web漏洞至关重要。

但网站为了获得更大的灵活性，经常放宽SOP。这些有控制、有意图的SOP绕过可能会产生不利影响，因为攻击者有时可以利用这些技术中的错误配置来绕过SOP。这些利用可能导致私人信息泄露，并常常导致更多漏洞，如身份验证绕过、账户接管和大规模数据泄露。在本章中，我们将讨论应用程序如何放宽或绕过SOP，以及攻击者如何利用这些功能危及应用程序。

## 机制

让我们快速回顾一下SOP的工作原理。由于SOP的存在，只有页面A和页面B同源时，页面A的脚本才能访问页面B的数据。如果两个URL具有相同的协议、主机名和端口号，则它们被认为具有相同的来源。现代Web应用程序通常基于HTTP Cookie进行身份验证，服务器根据浏览器自动包含的Cookie采取行动。这使得SOP尤为重要。当SOP实施时，恶意网页将无法利用你浏览器中存储的Cookie来访问你的私人信息。你可以在第3章阅读更多关于SOP的详细信息。

实际上，SOP对现代Web应用程序来说往往过于严格。例如，如果遵循该策略，同一组织的多个子域或多个域将无法共享信息。由于SOP不灵活，大多数网站会找到方法来放宽它。这常常是问题所在。

例如，假设你是一个试图从银行站点 a.example.com 窃取信息并找到用户账号的攻击者。你知道用户的银行详情位于 a.example.com/user_info。你的受害者已登录银行站点 a.example.com，同时在同一浏览器中访问你的站点 attacker.com。

你的站点向 a.example.com/user_info 发出GET请求以检索受害者的个人信息。由于你的受害者已登录银行，他们的浏览器会自动将他们的Cookie包含在发送给 a.example.com 的每个请求中，即使该请求是由你恶意站点上的脚本生成的。不幸的是，由于SOP，受害者的浏览器不允许你的站点读取从 a.example.com 返回的数据。

但现在，假设你意识到 a.example.com 通过SOP绕过技术将信息传递给 b.example.com。如果你能找到所使用的技术并利用它，你可能能够窃取银行站点上受害者的私人信息。

网站绕过SOP的最简单方法是通过JavaScript更改页面的来源。在页面的JavaScript中使用`document.domain`将两个页面的来源设置为相同的域，将使页面能够共享资源。例如，你可以将 a.example.com 和 b.example.com 的域都设置为 example.com，以便它们可以交互：

```javascript
document.domain = "example.com";
```

然而，这种方法有其局限性。首先，你只能将页面的`document.domain`设置为父域；例如，你可以将 a.example.com 的来源设置为 example.com，但不能设置为 example2.com。因此，只有当你希望与父域或兄弟子域共享资源时，此方法才有效。

## 利用跨域资源共享

由于这些限制，大多数站点使用**跨域资源共享（CORS）**来放宽SOP。CORS是一种保护服务器数据的机制。它允许服务器通过HTTP响应头`Access-Control-Allow-Origin`显式指定允许访问其资源的来源列表。

例如，假设我们试图将位于 a.example.com/user_info 的以下JSON数据发送到 b.example.com：

```json
{"username": "vickieli", "account_number": "12345"}
```

在SOP下，b.example.com 无法访问该JSON文件，因为 a.example.com 和 b.example.com 来源不同。但使用CORS，用户的浏览器会代表 b.example.com 发送一个`Origin`头：

```
Origin: https://b.example.com
```

如果 b.example.com 是有权限访问 a.example.com 资源的URL白名单的一部分，a.example.com 将向浏览器发送请求的资源以及一个`Access-Control-Allow-Origin`头。此头将向浏览器指示允许特定来源访问该资源：

```
Access-Control-Allow-Origin: b.example.com
```

应用程序也可以返回带有通配符（`*`）的`Access-Control-Allow-Origin`头，以指示该页面上的资源可以被任何域访问：

```
Access-Control-Allow-Origin: *
```

另一方面，如果请求页面的来源不被允许访问该资源，用户的浏览器将阻止请求页面读取数据。

CORS是实现跨域通信的好方法。然而，只有当允许来源列表正确定义时，CORS才是安全的。如果CORS配置错误，攻击者可以利用该错误配置访问受保护的资源。

CORS最基本的错误配置是允许null来源。如果服务器将`Access-Control-Allow-Origin`设置为`null`，浏览器将允许任何具有null来源头的站点访问该资源。这不安全，因为任何来源都可以创建带有null来源的请求。例如，使用`data:` URL方案从文档生成的跨站请求将具有null来源。

另一种错误配置是将`Access-Control-Allow-Origin`头设置为请求页面的来源，而不验证请求者的来源。如果服务器不验证来源，并为任何来源返回`Access-Control-Allow-Origin`，该头将完全绕过SOP，消除对跨域通信的所有限制。

总之，如果服务器将`Access-Control-Allow-Origin`头设置为`null`或请求页面的任意来源，它允许攻击者将信息带出站外：

```
Access-Control-Allow-Origin: null
Access-Control-Allow-Origin: https://attacker.com
```

另一种可利用的错误配置发生在站点使用弱正则表达式来验证来源时。例如，如果策略只检查来源URL是否以`www.example.com`开头，则可以使用如`www.example.com.attacker.com`这样的来源来绕过策略：

```
Access-Control-Allow-Origin: https://www.example.com.attacker.com
```

一个有趣且不可利用的配置是将允许来源设置为通配符（`*`）。这不可利用，因为CORS不允许将凭据（包括Cookie、身份验证头或客户端证书）与对这些页面的请求一起发送。由于无法向这些页面发送带凭据的请求，因此无法访问私人信息：

```
Access-Control-Allow-Origin: *
```

开发人员可以通过创建具有严格白名单和强大URL验证的明确定义的CORS策略来防止CORS错误配置。对于包含敏感信息的页面，只有当请求页面的来源在白名单中时，服务器才应在`Access-Control-Allow-Origin`头中返回该来源。对于公共信息，服务器可以简单地使用通配符`*`作为`Access-Control-Allow-Origin`。

## 利用 postMessage()

一些站点通过使用`postMessage()`绕过SOP。此方法是一个使用JavaScript语法的Web API。你可以使用它向另一个窗口发送基于文本的消息：

```javascript
RECIPIENT_WINDOW.postMessage(MESSAGE_TO_SEND, TARGET_ORIGIN);
```

接收窗口将通过一个事件处理器来处理消息，该处理器在接收窗口收到消息时触发：

```javascript
window.addEventListener("message", EVENT_HANDLER_FUNCTION);
```

由于使用`postMessage()`要求发送方获得接收方窗口的引用，消息只能在窗口与其iframe或弹出窗口之间发送。这是因为只有相互打开的窗口才有办法引用对方。例如，一个窗口可以使用`window.opener`来引用它打开的新窗口。或者，它可以使用`window.openers`来引用生成当前窗口的窗口。它可以使用`window.frames`来引用嵌入的iframe，使用`window.parent`来引用当前iframe的父窗口。

例如，假设我们试图将位于 a.example.com/user_info 的以下JSON数据传递给 b.example.com：

```javascript
{'username': 'vickieli', 'account_number': '12345'}
```

a.example.com 可以打开 b.example.com 并向其窗口发送消息。`window.open()`函数打开特定URL的窗口并返回对其的引用：

```javascript
var recipient_window = window.open("https://b.example.com", "b_domain");
recipient_window.postMessage("{'username': 'vickieli', 'account_number': '12345'}", "*");
```

同时，b.example.com 将设置一个事件监听器来处理它接收到的数据：

```javascript
function parse_data(event) {
    // Parse the data
}
window.addEventListener("message", parse_data);
```

如你所见，`postMessage()`并不直接绕过SOP，而是为不同来源的页面提供了一种相互发送数据的方式。

`postMessage()`方法可以是实现跨域通信的可靠方式。然而，在使用它时，消息的发送方和接收方都应验证对方的来源。当页面执行弱来源检查或完全缺乏来源检查时，就会发生漏洞。

首先，`postMessage()`方法允许发送方指定接收方的来源作为参数。如果发送方页面没有指定目标来源，而是使用通配符目标来源，就可能将信息泄露给其他站点：

```javascript
RECIPIENT_WINDOW.postMessage(MESSAGE_TO_SEND, *);
```

在这种情况下，攻击者可以创建一个恶意HTML页面，监听来自发送方页面的事件。然后，他们可以通过使用恶意链接或虚假图像诱骗用户触发`postMessage()`，使受害者页面将数据发送到攻击者的页面。

为防止此问题，开发人员应始终将`TARGET_ORIGIN`参数设置为目标站点的URL，而不是使用通配符来源：

```javascript
recipient_window.postMessage(
    "{'username': 'vickieli', 'account_number': '12345'}", "https://b.example.com");
```

另一方面，如果消息接收者不验证`postMessage()`的来源页面，攻击者就可能向网站发送任意数据并代表受害者触发非预期操作。例如，假设 b.example.com 允许 a.example.com 基于`postMessage()`触发密码更改，如下所示：

```javascript
recipient_window.postMessage(
    "{'action': 'password_change', 'username': 'vickieli', 'new_password': 'password'}",
    "https://b.example.com");
```

然后 b.example.com 页面将接收消息并处理请求：

```javascript
function parse_data(event) {
    // If "action" is "password_change", change the user's password
}
window.addEventListener("message", parse_data);
```

注意，任何窗口都可以向 b.example.com 发送消息，因此任何页面都可以在 b.example.com 上发起密码更改！为了利用此行为，攻击者可以嵌入或打开受害者页面以获得其窗口引用。然后他们可以自由地向该窗口发送任意消息。

为防止此问题，页面在处理消息之前应验证发送者的来源：

```javascript
function parse_data(event) {
    ① if (event.origin == "https://a.example.com") {
        // If "action" is "password_change", change the user's password
    }
}
window.addEventListener("message", parse_data);
```

这行①通过检查可接受的来源来验证发送者的来源。

## 利用 JSON with Padding

**带填充的JSON（JSONP）**是另一种绕过SOP的技术。它允许发送方将JSON数据作为JavaScript代码发送。不同来源的页面可以通过处理JavaScript来读取JSON数据。

为了了解其工作原理，让我们继续之前的示例，我们试图将位于 a.example.com/user_info 的以下JSON数据传递到 b.example.com：

```json
{"username": "vickieli", "account_number": "12345"}
```

SOP允许HTML `<script>`标签跨来源加载脚本，因此 b.example.com 跨来源检索数据的简单方法是在`<script>`标签中将数据作为脚本加载：

```html
<script src="https://a.example.com/user_info"></script>
```

这样，b.example.com 实际上就是在脚本标签中包含了JSON数据块。但这会导致语法错误，因为JSON数据不是有效的JavaScript：

```html
<script>
{"username": "vickieli", "account_number": "12345"}
</script>
```

JSONP通过将数据包装在JavaScript函数中，并将数据作为JavaScript代码而不是JSON文件发送来解决此问题。请求页面将资源作为脚本包含，并指定一个回调函数，通常在名为`callback`或`jsonp`的URL参数中。这个回调函数是接收页面上预定义的、准备处理数据的函数：

```html
<script src="https://a.example.com/user_info?callback=parseinfo"></script>
```

a.example.com 的页面将返回包装在指定回调函数中的数据：

```javascript
parseinfo({"username": "vickieli", "account_number": "12345"})
```

接收页面实质上将包含这个脚本，它是有效的JavaScript代码：

```html
<script>
parseinfo({"username": "vickieli", "account_number": "12345"})
</script>
```

接收页面然后可以通过运行JavaScript代码并处理`parseinfo()`函数来提取数据。通过将数据作为脚本而不是JSON数据发送，JSONP允许跨来源读取资源。

以下是JSONP工作流程的摘要：

1. 数据请求者在脚本标签中包含数据的URL，以及回调函数的名称。
2. 数据提供者返回包装在指定回调函数中的JSON数据。
3. 数据请求者接收该函数，并通过运行返回的JavaScript代码来处理数据。

你通常可以通过查找包含`jsonp`或`callback`等术语的脚本标签来确定站点是否使用JSONP。

但JSONP带有风险。当端点上启用JSONP时，攻击者可以简单地在他们的站点上嵌入相同的脚本标签，并请求包装在JSONP载荷中的数据，如下所示：

```html
<script src="https://a.example.com/user_info?callback=parseinfo"></script>
```

如果用户在登录 a.example.com 的同时浏览攻击者的站点，用户的浏览器将在此请求中包含他们的凭据，并允许攻击者提取属于受害者的机密数据。

这就是为什么JSONP只适合传输公共数据。虽然可以通过使用CSRF令牌或为JSONP请求维护Referer头白名单来加固JSONP，但这些保护措施通常可以被绕过。

JSONP的另一个问题是 b.example.com 必须完全信任 a.example.com，因为它正在运行来自 a.example.com 的任意JavaScript。如果 a.example.com 被入侵，攻击者可以在 b.example.com 上运行他们想要的任何JavaScript，因为 b.example.com 在`<script>`标签中包含了来自 a.example.com 的文件。这相当于XSS攻击。

现在CORS是跨域通信的可靠选择，站点不再像以前那样频繁使用JSONP。

## 利用XSS绕过SOP

最后，XSS本质上是一种完全的SOP绕过，因为在页面上运行的任何JavaScript都在该页面的安全上下文中操作。如果攻击者能让恶意脚本在受害者页面上执行，该脚本就可以访问受害者页面的资源和数据。因此，记住，如果你能找到XSS，你就基本上绕过了保护该页面的SOP。

## 寻找SOP绕过

让我们开始利用你学到的知识来寻找SOP绕过漏洞吧！SOP绕过漏洞是由SOP放宽技术的不当实施引起的。因此，你需要做的第一件事是确定目标应用程序是否以任何方式放宽了SOP。

### 第1步：确定是否使用了SOP放宽技术

你可以通过查找每种SOP放宽技术的特征来确定目标是否使用了SOP放宽技术。在浏览Web应用程序时，打开代理并查找跨域通信的任何迹象。例如，使用CORS的站点通常会在HTTP响应中返回包含`Access-Control-Allow-Origin`头的内容。如果你检查页面（例如，在Chrome中右键点击并选择“检查”，然后导航到Event Listeners）并找到`message`事件监听器（图19-1），则站点可能正在使用`postMessage()`。

如果看到带有回调函数的`<script>`标签中加载了URL，则站点可能正在使用JSONP：

```html
<script src="https://a.example.com/user_info?callback=parseinfo"></script>
<script src="https://a.example.com/user_info?jsonp=parseinfo"></script>
```

如果你看到跨域通信的线索，尝试本章提到的技术，看看你是否可以绕过SOP并从站点窃取敏感信息！

*（图19-1：在Chrome浏览器中查找页面的事件监听器）*

### 第2步：发现CORS错误配置

如果站点使用CORS，检查`Access-Control-Allow-Origin`响应头是否设置为`null`：

```
Origin: null
```

如果不是，向站点发送带有`origin`头`attacker.com`的请求，查看响应中的`Access-Control-Allow-Origin`是否设置为`attacker.com`。（你可以通过拦截请求并在代理中编辑来添加`Origin`头。）

```
Origin: attacker.com
```

最后，通过提交包含允许站点的`Origin`头（如`www.example.com.attacker.com`）来测试站点是否正确验证来源URL。查看`Access-Control-Allow-Origin`头是否返回攻击者域的来源。

```
Origin: www.example.com.attacker.com
```

如果返回了这些`Access-Control-Allow-Origin`头值之一，你就发现了CORS错误配置。攻击者将能够绕过SOP并将信息带出站外（图19-2）。

*（图19-2：站点是否容易受到CORS错误配置漏洞的攻击？）*

### 第3步：发现 postMessage 漏洞

如果站点使用`postMessage`，查看你是否可以作为不受信任的站点发送或接收消息。创建一个包含iframe的HTML页面，将接受消息的目标页面嵌入其中。尝试向该页面发送触发状态更改行为的消息。如果目标无法被嵌入，则将其作为新窗口打开：

```javascript
var recipient_window = window.open("https://TARGET_URL", "target_domain");
recipient_window.postMessage("RANDOM MESSAGE", "*");
```

你也可以创建一个HTML页面，监听来自目标页面的事件，并从目标站点触发`postMessage`。查看你是否可以从目标页面接收敏感数据：

```javascript
var sender_window = window.open("https://TARGET_URL", "target_domain");
function parse_data(event) {
    // Run some code if we receive data from the target
}
window.addEventListener("message", parse_data);
```

### 第4步：发现JSONP问题

最后，如果站点使用JSONP，查看你是否可以在你的站点上嵌入脚本标签，并请求包装在JSONP载荷中的敏感数据：

```html
<script src="https://TARGET_URL?callback=parseinfo"></script>
```

### 第5步：考虑缓解因素

当目标站点不依赖Cookie进行身份验证时，这些SOP绕过错误配置可能不可利用。例如，当站点使用自定义头或秘密请求参数来验证请求时，你可能需要找到一种方法来伪造它们以窃取敏感数据。

## 提升攻击影响

SOP绕过漏洞通常意味着攻击者可以读取私人信息或以其他用户身份执行操作。这意味着这些漏洞在尝试提升之前通常已经具有高严重性。但你仍然可以通过自动化或利用你找到的信息进行攻击转向来提升SOP绕过问题。你能通过自动化SOP绕过的利用来收集大量用户数据吗？你能利用你找到的信息造成更多损害吗？例如，如果你能提取受害者的安全问题，你能利用该信息完全接管用户的账户吗？

许多研究人员只是报告CORS错误配置，而不展示漏洞的影响。在发送报告之前要考虑问题的影响。例如，如果公开可读的页面带有`null`的`Access-Control-Allow-Origin`头，它不会对应用程序造成损害，因为该页面不包含任何敏感信息。一份好的SOP绕过报告将包括潜在的攻击场景，并指示攻击者如何利用该漏洞。例如，攻击者可以窃取哪些数据？操作有多容易？
