# 第23章：黑客攻防：Android应用（Hacking Android Apps）

到目前为止，你已经学习了整本书的Web应用程序黑客技术。大多数漏洞赏金计划都对其Web应用提供赏金，因此掌握Web黑客技术是开始漏洞赏金狩猎的最简单方式，因为它能解锁最广泛的目标。

另一方面，移动端黑客需要更多的先决技能，并且需要更多时间才能入门。但由于门槛较高，从事移动端计划的黑客往往较少。此外，随着公司越来越多地推出复杂的移动产品，移动端计划的数量也在增加。移动端程序有时会列在公司主要漏洞赏金计划的“移动端”或“物联网”部分下。这意味着，如果你学习攻击移动应用程序，你可能会提交更少的重复报告，并发现更多有趣的漏洞。

尽管设置更为复杂，但攻击移动应用程序与攻击Web应用程序非常相似。本章介绍你在开始分析Android应用之前需要学习的额外技能。

拥有移动应用程序的公司通常同时拥有Android和iOS版本的应用。我们不会介绍iOS应用，本章也绝不是攻击Android应用的全面指南。但结合前面的章节，它应为你开始独立探索该领域提供所需的基础。

> **注意**：移动端黑客攻击的最佳参考资源之一是OWASP移动安全测试指南（https://github.com/OWASP/owasp-mstg/）。

## 设置移动代理

与配置Web浏览器与代理协同工作类似，你需要设置测试移动设备与代理协同工作。这通常涉及在设备上安装代理的证书并调整代理设置。

如果经济条件允许，可以获取另一台移动设备，或使用你的旧设备进行测试。移动端测试是有风险的：你可能意外损坏设备，并且本章提到的许多技术会使设备的保修失效。你也可以使用移动模拟器（模拟移动设备的程序）进行测试。

首先，你需要配置Burp的代理以接受来自移动设备的连接，因为默认情况下，Burp的代理只接受来自运行Burp的机器的连接。导航到Burp的Proxy → Options选项卡。在Proxy Listeners部分，点击Add。在弹出的窗口（图23-1）中，输入一个当前未使用的端口号，并选择`All interfaces`作为Bind to address选项。点击OK。

*（图23-1：设置Burp以接受Wi-Fi网络上所有设备的连接）*

你的代理现在应接受连接到同一Wi-Fi网络的任何设备的连接。因此，我不建议在公共Wi-Fi网络上执行此操作。

接下来，你将配置Android设备以使用代理。这些步骤会根据你使用的系统略有不同，但过程应该是选择Settings → Network → Wi-Fi，选择（通常通过长按）你当前连接的Wi-Fi网络，然后选择Modify Network。然后你应该能够选择代理主机名和端口。在这里，你应该输入你计算机的IP地址和你之前选择的端口号。如果你使用的是Linux计算机，可以通过运行以下命令找到计算机的IP地址：

```bash
hostname -i
```

如果你使用的是Mac，可以通过以下命令找到IP：

```bash
ipconfig getifaddr en0
```

你的Burp代理现在应该准备好开始拦截来自你移动设备的流量了。设置移动模拟器与代理协同工作的过程与此类似，只是某些模拟器要求你在模拟器设置菜单中添加代理详情，而不是在模拟设备本身的网络设置中。

如果你还想拦截和解码来自移动设备的HTTPS流量，你需要在设备上安装Burp的证书。你可以通过在计算机浏览器中访问`http://burp/cert`（该浏览器使用Burp作为代理）来完成此操作。保存下载的证书，通过电子邮件发送给自己，然后下载到你的移动设备上。接下来，在设备上安装证书。此过程也取决于设备上运行的系统，但应该是选择Settings → Security → Install Certificates from Storage。点击你刚下载的证书，并选择`VPN and apps`作为Certificate use选项。你现在就可以使用Burp审计HTTPS流量了。

## 绕过证书绑定

证书绑定是一种将应用程序限制为仅信任预定义证书的机制。也称为SSL绑定或证书绑定，它提供了针对中间人攻击的额外安全层——在中间人攻击中，攻击者秘密拦截、读取和篡改两方之间的通信。如果你想拦截和解码使用证书绑定的应用程序的流量，你必须首先绕过证书绑定，否则应用程序不会信任你的代理SSL证书，你将无法拦截HTTPS流量。

有时必须绕过证书绑定才能拦截受更好保护的应用的流量。如果你已成功设置移动设备与代理协同工作，但仍然无法看到目标应用程序的流量，则该应用可能已实施证书绑定。

绕过证书绑定的过程取决于每个应用程序如何实现证书绑定。对于Android应用程序，你有几种绕过绑定的选项。你可以使用Frida，这是一个允许你向应用程序注入脚本的工具。你可以从https://frida.re/docs/installation/下载Frida。然后使用通用Android SSL绑定绕过Frida脚本（https://codeshare.frida.re/@pcipolloni/universal-android-ssl-pinning-bypass-with-frida/）。另一个可用于自动化此过程的工具是Objection（https://github.com/sensepost/objection/），它使用Frida绕过Android或iOS的绑定。运行Objection命令`android sslpinning disable`以绕过绑定。

对于大多数应用程序，你可以使用这些自动化工具绕过证书绑定。但如果应用程序使用自定义代码实现绑定，你可能需要手动绕过它。你可以用自定义证书覆盖打包的证书。或者，你可以更改或禁用应用程序的证书验证代码。执行这些技术的过程很复杂，且高度依赖于你正在攻击的应用程序，因此我不再详细说明。有关这些方法的更多信息，你需要自行研究。

## APK的构成

在攻击Android应用程序之前，你必须首先了解它们是由什么组成的。Android应用程序以称为Android Package（APK）的文件格式分发和安装。APK就像ZIP文件，包含Android应用程序运行所需的一切：应用程序代码、应用程序清单文件和应用程序资源。本节描述Android APK的主要组件。

首先，`AndroidManifest.xml`文件包含应用程序的包名、版本、组件、访问权限和引用的库，以及其他元数据。它是探索应用程序的良好起点。从此文件中，你可以深入了解应用程序的组件和权限。

理解目标应用程序的组件将为你提供其工作原理的总体概述。有四种类型的应用组件：Activities（在`<activity>`标签中声明）、Services（在`<service>`标签中声明）、BroadcastReceivers（在`<receiver>`标签中声明）和ContentProviders（在`<provider>`标签中声明）。

**Activities**是与用户交互的应用程序组件。你看到的Android应用程序窗口由Activities组成。**Services**是不直接与用户交互的长期运行操作，如在后台检索或发送数据。**BroadcastReceivers**允许应用程序响应来自Android系统和其他应用程序的广播消息。例如，某些应用程序仅在设备连接到Wi-Fi时下载大文件，因此它们需要一种在设备连接到Wi-Fi网络时得到通知的方式。**ContentProviders**提供了一种与其他应用程序共享数据的方式。

应用程序使用的权限（如发送短信的能力）以及其他应用程序与其交互所需的权限也在`AndroidManifest.xml`文件中声明。这将让你很好地了解应用程序可以做什么以及它如何与同一设备上的其他应用程序交互。有关在`AndroidManifest.xml`中可以找到什么的更多信息，请访问https://developer.android.com/guide/topics/manifest/manifest-intro/。

`classes.dex`文件包含以DEX文件格式编译的应用程序源代码。你可以使用本章后面介绍的各种Android黑客工具来提取和解编译此源代码进行分析。有关如何进行源代码审查以发现漏洞的更多信息，请参阅第22章。

`resources.arsc`文件包含应用程序的预编译资源，如字符串、颜色和样式。`res`文件夹包含未编译到`resources.arsc`中的应用程序资源。在`res`文件夹中，`res/values/strings.xml`文件包含应用程序的文字字符串。

`lib`文件夹包含平台相关的编译代码。`lib`中的每个子目录包含用于特定移动架构的特定源代码。编译的内核模块位于此处，通常是漏洞的来源。

`assets`文件夹包含应用程序的资产，如视频、音频和文档模板。最后，`META-INF`文件夹包含`MANIFEST.MF`文件，该文件存储有关应用程序的元数据。此文件夹还包含APK的证书和签名。

## 使用的工具

现在你了解了Android应用程序的主要组件，你需要知道如何处理APK文件并提取Android源代码。除了使用Web代理检查进出测试设备的流量外，你还需要一些对分析Android应用程序至关重要的工具。本节不详细介绍如何使用这些工具，而是说明何时以及为什么使用它们。其余内容你可以通过使用每个工具的文档页面轻松了解。

### Android调试桥（ADB）

Android调试桥（ADB）是一个命令行工具，让你的计算机与连接的Android设备通信。这意味着如果你想在计算机上读取或修改应用程序源代码和资源文件，就不必通过电子邮件在计算机和手机之间来回发送。例如，你可以使用ADB将文件复制到设备或从设备复制文件，或快速安装你正在研究的应用程序的修改版本。ADB的文档在https://developer.android.com/studio/command-line/adb/。

要开始使用ADB，请使用USB电缆将设备连接到笔记本电脑。然后在设备上打开调试模式。每当你想要在通过USB连接到笔记本电脑的设备上使用ADB时，你必须启用USB调试。此过程因移动设备而异，但应类似于选择Settings → System → Developer Options → Debugging。这将使你能够通过ADB从笔记本电脑与设备交互。在Android 4.1及更低版本上，开发者选项屏幕默认可用。在Android 4.2及更高版本中，需要通过选择Settings → About Phone，然后点击Build number七次来启用开发者选项。

在你的移动设备上，你应该看到一个窗口提示你允许来自笔记本电脑的连接。通过在笔记本电脑终端中运行以下命令，确保你的笔记本电脑已连接到设备：

```bash
adb devices -l
```

现在你可以使用此命令安装APK：

```bash
adb install PATH_TO_APK
```

你也可以通过运行以下命令将文件从设备下载到笔记本电脑：

```bash
adb pull REMOTE_PATH LOCAL_PATH
```

或将笔记本电脑上的文件复制到移动设备：

```bash
adb push LOCAL_PATH REMOTE_PATH
```

### Android Studio

Android Studio是用于开发Android应用程序的软件，你可以使用它来修改现有应用程序的源代码。如果你没有实体Android设备，它还包含一个模拟器，允许你在虚拟环境中运行应用程序。你可以从https://developer.android.com/studio/下载并了解Android Studio。

### Apktool

Apktool是一个用于逆向工程APK文件的工具，对于Android黑客攻击至关重要，并且可能是你在分析期间最常用的工具。它将APK转换为可读的源代码文件，并从这些文件重建APK。Apktool的文档在https://ibotpeaches.github.io/Apktool/。

你可以使用Apktool从APK中获取单个文件以进行源代码分析。例如，此命令从名为`example.apk`的APK中提取文件：

```bash
$ apktool d example.apk
```

有时你可能想要修改APK的源代码，看看这是否会改变应用程序的行为。你可以在进行修改后使用Apktool重新打包单个源代码文件。此命令将`example`文件夹的内容打包到文件`example.apk`中：

```bash
$ apktool b example -o example.apk
```

### Frida

Frida（https://frida.re/）是一个出色的instrumentation工具包，允许你将脚本注入应用程序的正在运行的进程中。你可以使用它来检查被调用的函数、分析应用程序的网络连接，以及绕过证书绑定。

Frida使用JavaScript作为其语言，因此你需要了解JavaScript才能充分利用它。但是，你可以访问网上共享的大量预制脚本。

### 移动安全框架（MobSF）

我还强烈推荐移动安全框架（https://github.com/MobSF/Mobile-Security-Framework-MobSF/），即MobSF，用于所有移动应用测试。这个用于Android、iOS和Windows的自动化移动应用测试框架可以进行静态和动态测试。它自动化了本章中提到的许多技术，一旦你了解了Android黑客的基础知识，它是一个很好的工具。

## 寻找漏洞

现在你的移动黑客环境已设置好，是时候开始在移动应用中寻找漏洞了。幸运的是，攻击移动应用程序与攻击Web应用程序并没有太大不同。

首先，提取应用程序的包内容并审查代码中的漏洞。比较同一组织的移动应用和Web应用的身份验证和授权机制。开发人员可能信任来自移动应用的数据，如果你使用移动端点，这可能导致IDOR或身份验证损坏。移动应用也往往存在会话管理问题，如重用会话令牌、使用更长的会话或使用不过期的会话Cookie。这些问题可以与XSS链接，以获取会话Cookie，即使用户注销或更改密码，攻击者也可以接管账户。

有些应用程序对加密或哈希使用自定义实现。查找不安全的算法、已知算法的弱实现和硬编码的加密密钥。在审查应用程序源代码中的潜在漏洞后，你可以通过在模拟器或真实设备上进行动态测试来验证你的发现。

移动应用程序是寻找其Web应用中没有的额外Web漏洞的绝佳场所。你可以使用与发现Web漏洞相同的方法来寻找这些漏洞：在敏感操作期间使用Burp Suite拦截来自移动应用的流量。移动应用通常使用独特的端点，这些端点可能不像Web端点那样经过充分测试，因为很少有黑客在移动应用上进行挖掘。你可以通过寻找在组织的Web应用程序中未见的端点来发现它们。

我建议在深入移动应用之前先测试组织的Web应用程序，因为移动应用通常是其Web版本的简化版。使用你已经学到的技能寻找IDOR、SQL注入、XSS和其他常见Web漏洞。你也可以通过分析移动应用程序的源代码来寻找常见Web漏洞。

除了在Web应用中寻找的漏洞外，还要寻找一些移动特有的漏洞。`AndroidManifest.xml`包含有关应用程序及其功能的基本信息。此文件是你分析的良好起点。解包APK文件后，阅读它以获得对应用程序的基本了解，包括其组件和它使用的权限。然后你可以深入了解其他文件，寻找移动特有的漏洞。

移动应用的源代码通常包含应用访问Web服务所需的硬编码秘密或API密钥。`res/values/strings.xml`文件存储应用程序中的字符串。这是寻找硬编码秘密、密钥、端点和其他类型信息泄露的好地方。你也可以通过使用`grep`搜索第22章中提到的关键字来在其他文件中搜索秘密。

如果你发现带有`.db`或`.sqlite`扩展名的文件，这些是数据库文件。查看这些文件内部，了解哪些信息随应用程序一起分发。这些也是潜在秘密和敏感信息泄露的简单来源。查找会话数据、财务信息以及属于用户或组织的敏感信息。

最终，寻找移动漏洞与攻击Web应用程序并没有太大不同。仔细检查客户端和服务器之间的交互，并深入研究源代码。记住移动应用特有的漏洞类别，如硬编码秘密和数据库文件中敏感数据的存储，这些在移动应用中比在Web应用中更为常见。
