# 测试暴露的会话变量

|ID          |
|------------|
|WSTG-SESS-04|

## 概述

会话令牌（Cookie、SessionID、隐藏字段）如果被暴露，通常会使攻击者能够冒充受害者并非法访问应用程序。因此，必须始终保护它们免受窃听，尤其是在客户端浏览器和应用服务器之间传输期间。

此处信息涉及传输安全性如何应用于敏感会话ID数据的传输，而非一般数据，并且可能比站点所提供数据的缓存和传输策略更为严格。

使用个人代理，可以确定每个请求和响应的以下内容：

- 使用的协议（例如 HTTP vs HTTPS）
- HTTP 头部
- 消息正文（例如 POST 或页面内容）

每次会话ID数据在客户端和服务器之间传递时，都应检查协议、缓存、隐私指令和正文。此处的传输安全性是指通过GET或POST请求、消息正文或其他方式在有效HTTP请求中传递的会话ID。

## 测试目标

- 确保实施了适当的加密。
- 审查缓存配置。
- 评估通道和方法的安全性。

## 如何测试

### 测试会话令牌的加密和重用漏洞

防止窃听通常由TLS加密提供，但也可能包含其他隧道或加密。应注意，会话ID的加密或加密哈希应与传输加密分开考虑，因为受保护的是会话ID本身，而不是它可能代表的数据。

如果攻击者可以向应用程序提供会话ID以获得访问权限，则必须在传输过程中保护它以降低该风险。因此，应确保对于传递会话ID的任何请求或响应，无论使用何种机制（例如隐藏表单字段），加密都是默认且强制执行的。应执行简单检查，例如在与应用程序交互期间将 `https://` 替换为 `http://`，以及修改表单提交，以确定是否在安全和非安全站点之间实施了充分的隔离。

请注意，如果站点中还存在使用会话ID跟踪用户但未启用安全性的部分（例如记录注册用户下载了哪些公共文档），则必须使用不同的会话ID。因此，当客户端从安全切换到非安全元素时，应监控会话ID以确保使用不同的ID。

> 每次认证成功时，用户应期望收到：
>
> - 不同的会话令牌
> - 每次发出HTTP请求时通过加密通道发送的令牌

### 测试代理和缓存漏洞

在审查应用程序安全性时，也必须考虑代理。在许多情况下，客户端将通过企业、ISP或其他代理或协议感知网关（例如防火墙）访问应用程序。HTTP协议提供了控制下游代理行为的指令，应评估这些指令的正确实现。

通常，会话ID绝不应通过未加密的传输发送，也绝不应被缓存。应检查应用程序，以确保加密通信是默认的，并且对于任何会话ID的传输都是强制执行的。此外，每当传递会话ID时，应设置指令以防止中间缓存甚至本地缓存对其进行缓存。

应用程序还应配置为在HTTP/1.0和HTTP/1.1上保护缓存中的数据——RFC 2616讨论了与HTTP相关的适当控制。HTTP/1.1提供了许多缓存控制机制。`Cache-Control: no-cache` 表示代理不得重用任何数据。虽然 `Cache-Control: Private` 看似一个合适的指令，但它仍然允许非共享代理缓存数据。在网吧或其他共享系统中，这明显存在风险。即使在单用户工作站上，缓存的会话ID也可能因文件系统受损或使用网络存储而暴露。HTTP/1.0缓存不识别 `Cache-Control: no-cache` 指令。

> 应使用 `Expires: 0` 和 `Cache-Control: max-age=0` 指令，以进一步确保缓存不会暴露数据。应检查传递会话ID数据的每个请求/响应，以确保使用了适当的缓存指令。

### 测试GET和POST漏洞

通常，不应使用GET请求，因为会话ID可能暴露在代理或防火墙日志中。它们也远比其他传输类型更容易被操纵，但应注意，几乎任何机制都可以被具有适当工具的客户端操纵。此外，[跨站脚本（XSS）](https://owasp.org/www-community/attacks/xss/)攻击最容易通过向受害者发送精心构造的链接来利用。如果数据以POST方式从客户端发送，这种情况可能性要小得多。

应测试所有接收POST数据的服务器端代码，以确保它们不接受以GET方式发送的数据。例如，考虑登录页面生成的以下POST请求（`https://owaspapp.com/login.asp`）。

```http
POST /login.asp HTTP/1.1
Host: owaspapp.com
[...]
Cookie: ASPSESSIONIDABCDEFG=ASKLJDLKJRELKHJG
Content-Length: 51

Login=Username&password=Password&SessionID=12345678
```

如果login.asp实现不佳，则可能通过以下URL登录：`https://owaspapp.com/login.asp?Login=Username&password=Password&SessionID=12345678`

可以通过以这种方式检查每个POST来识别潜在不安全的服务器端脚本。

### 测试传输漏洞

客户端与应用程序之间的所有交互应至少根据以下标准进行测试。

- 会话ID如何传输？例如GET、POST、表单字段（包括隐藏字段）
- 会话ID是否默认始终通过加密传输发送？
- 是否有可能操纵应用程序以未加密方式发送会话ID？例如通过将HTTPS更改为HTTP？
- 对传递会话ID的请求/响应应用了哪些缓存控制指令？
- 这些指令是否始终存在？如果不存在，异常在哪里？
- 是否使用了包含会话ID的GET请求？
- 如果使用POST，是否可以与GET互换？
