# 测试跨站请求伪造

|ID          |
|------------|
|WSTG-SESS-05|

## 概述

跨站请求伪造（[CSRF](https://owasp.org/www-community/attacks/csrf)）是一种攻击，迫使最终用户在他们当前已认证的 Web 应用程序上执行非预期的操作。借助一些社会工程手段（如通过电子邮件或聊天发送链接），攻击者可以强制 Web 应用程序的用户执行攻击者所选择的操作。当 CSRF 攻击针对普通用户时，成功的利用可能会危及最终用户的数据和操作。如果目标最终用户是管理员账户，则 CSRF 攻击可能危及整个 Web 应用程序。

CSRF 依赖于：

1. 浏览器处理会话相关信息（如 cookie 和 HTTP 认证信息）的行为。
2. 攻击者对有效 Web 应用程序 URL、请求或功能的了解。
3. 应用程序会话管理仅依赖浏览器所知的信息。
4. 存在某些 HTML 标签，其存在会导致立即访问 HTTP[S] 资源；例如图像标签 `img`。

要点 1、2 和 3 是漏洞存在的必要条件，而要点 4 有助于实际利用，但并非严格必需。

1. 浏览器自动发送用于标识用户会话的信息。假设 *site* 是一个托管 Web 应用程序的站点，用户 *victim* 刚刚完成对 *site* 的认证。作为响应，*site* 向 *victim* 发送一个 cookie，用于标识 *victim* 发出的请求属于 *victim* 的已认证会话。一旦浏览器收到 *site* 设置的 cookie，它将自动随之后所有指向 *site* 的请求一起发送该 cookie。
2. 如果应用程序不在 URL 中使用会话相关信息，则可以识别应用程序的 URL、参数和合法值。这可以通过代码分析或通过访问应用程序并注意 HTML 或 JavaScript 中嵌入的表单和 URL 来实现。
3. “浏览器所知的信息”是指由浏览器存储并在每次向需要该认证的应用程序区域发出请求时呈现的信息，如 cookie 或基于 HTTP 的认证信息（例如基本认证，而非基于表单的认证）。下面讨论的漏洞适用于完全依赖此类信息来识别用户会话的应用程序。

为简单起见，请考虑可通过 GET 访问的 URL（尽管讨论同样适用于 POST 请求）。如果 *victim* 已通过认证，则提交另一个请求会导致 cookie 自动随其发送。下图说明了用户访问 `www.example.com` 上的应用程序的情况。

![会话骑行](images/Session_riding.GIF)\
*图 4.6.5-1：会话骑行*

GET 请求可以通过多种不同方式由用户发送：

- 使用 Web 应用程序
- 直接在浏览器中键入 URL
- 跟随指向该 URL 的外部链接

应用程序无法区分这些调用。特别是第三种方式可能非常危险。有许多技术和漏洞可以隐藏链接的真实属性。该链接可以嵌入电子邮件中、出现在引诱用户访问的恶意站点上，或出现在第三方托管的内容（如另一个站点或 HTML 邮件）中，并指向应用程序的资源。如果用户点击该链接，由于他们已经在 *site* 的 Web 应用程序上通过认证，浏览器将向 Web 应用程序发出 GET 请求，并附上认证信息（会话 ID cookie）。这会导致在 Web 应用程序上执行用户未预期的有效操作；例如，在 Web 银行应用程序上进行资金转账。

通过使用如上述第 4 点中指定的 `img` 标签，甚至不需要用户点击特定链接。假设攻击者向用户发送一封电子邮件，诱导他们访问一个包含以下（过度简化的）HTML 的页面。

```html
<html>
    <body>
...
<img src="https://www.company.example/action" width="0" height="0">
...
    </body>
</html>
```

当浏览器显示此页面时，它将尝试从 `https://www.company.example` 显示指定尺寸为零（因此不可见）的图像。这会导致自动向托管在 *site* 上的 Web 应用程序发送请求。图像 URL 是否指向真正的图像并不重要，因为它的存在会触发 `src` 字段中指定的 `action` 请求。只要浏览器未禁用图像下载，就会发生这种情况。大多数浏览器不会禁用图像下载，因为那会使大多数 Web 应用程序无法使用。

此处的问题源于：

- 页面上的 HTML 标签导致自动执行 HTTP 请求（`img` 是其中之一）。
- 浏览器无法判断 `img` 引用的资源是否不是合法图像。
- 图像加载不受所谓图像源位置的限制，即表单和图像本身不必位于同一主机甚至同一域。

与 Web 应用程序无关的 HTML 内容可以引用应用程序中的组件，并且浏览器会自动向应用程序构造有效请求，这一事实允许此类攻击。除非使攻击者无法与应用程序功能交互，否则无法禁止此行为。

在集成的邮件/浏览器环境中，仅显示包含图像引用的电子邮件消息将导致执行对 Web 应用程序的请求并附带相关的浏览器 cookie。电子邮件消息可能引用看似有效的图像 URL，例如：

```html
<img src="https://[attacker]/picture.gif" width="0" height="0">
```

在此示例中，`[attacker]` 是攻击者控制的站点。通过利用重定向机制，恶意站点可以使用 `https://[attacker]/picture.gif` 将受害者引导至 `https://[thirdparty]/action` 并触发 `action`。

Cookie 不是此类漏洞中涉及的唯一示例。会话信息完全由浏览器提供的 Web 应用程序也容易受到攻击。这包括仅依赖 HTTP 认证机制的应用程序，因为认证信息由浏览器所知，并在每个请求时自动发送。这不包括基于表单的认证，因为这种认证仅发生一次，并生成某种形式的会话相关信息（通常是 cookie）。

假设受害者登录到防火墙 Web 管理控制台。要登录，用户必须进行认证，会话信息存储在 cookie 中。

假设防火墙 Web 管理控制台有一个功能，允许已认证用户按其数字 ID 删除规则，或者如果用户指定 `*`，则删除配置中的所有规则（这在现实中是一个危险的功能，但能使示例更有趣）。删除页面如下所示。为简单起见，假设表单发出 GET 请求。删除规则编号一：

```text
https://[target]/fwmgt/delete?rule=1
```

删除所有规则：

```text
https://[target]/fwmgt/delete?rule=*
```

此示例特意简化，但以简化方式展示了 CSRF 的危险性。

![会话骑行防火墙管理](images/Session_Riding_Firewall_Management.gif)\
*图 4.6.5-2：会话骑行防火墙管理*

使用上图所示表单，输入值 `*` 并单击“删除”按钮将提交以下 GET 请求：

```text
https://www.company.example/fwmgt/delete?rule=*
```

这将删除所有防火墙规则。

![会话骑行防火墙管理 2](images/Session_Riding_Firewall_Management_2.gif)\
*图 4.6.5-3：会话骑行防火墙管理 2*

用户也可以通过手动提交 URL 来实现相同的结果：

```text
https://[target]/fwmgt/delete?rule=*
```

或者通过跟随一个直接或通过重定向指向上述 URL 的链接。或者，再次通过访问包含指向相同 URL 的嵌入式 `img` 标签的 HTML 页面。

在所有这些情况中，如果用户当前已登录到防火墙管理应用程序，请求将成功并修改防火墙的配置。可以想象针对敏感应用程序的攻击，进行自动拍卖出价、资金转账、订单、更改关键软件组件的配置等。

有趣的是，这些漏洞可以在防火墙后利用；即，被攻击的链接只需受害者可访问，而不必直接由攻击者可访问。特别是，它可以是任何内部网 Web 服务器；例如，在前面提到的防火墙管理场景中，该场景不太可能暴露在互联网上。

自我脆弱应用程序，即同时用作攻击向量和目标的应用程序（如 Web 邮件应用程序），会使情况更糟。由于用户在阅读电子邮件时已登录，此类易受攻击的应用程序可能允许攻击者执行诸如删除邮件或发送看似来自受害者的邮件等操作。

## 测试目标

- 确定是否有可能代表用户发起非用户发起的请求。

## 如何测试

审计应用程序以确定其会话管理是否存在漏洞。如果会话管理仅依赖于客户端值（浏览器可获得的信息），则应用程序存在漏洞。“客户端值”指的是 cookie 和 HTTP 认证凭据（基本认证和其他形式的 HTTP 认证；不包括基于表单的认证，后者是应用程序级认证）。

通过 HTTP GET 请求可访问的资源容易受到攻击，但 POST 请求也可以通过 JavaScript 自动化，因此也存在漏洞；因此，仅使用 POST 不足以纠正 CSRF 漏洞的发生。

对于 POST，可以使用以下示例。

1. 创建类似于下面所示的 HTML 页面
2. 将 HTML 托管在恶意或第三方站点上
3. 将页面链接发送给受害者并诱导他们点击。

```html
<html>
<body onload='document.CSRF.submit()'>

<form action='https://targetWebsite/Authenticate.jsp' method='POST' name='CSRF'>
    <input type='hidden' name='name' value='Hacked'>
    <input type='hidden' name='password' value='Hacked'>
</form>

</body>
</html>
```

在开发人员使用 JSON 进行浏览器与服务器通信的 Web 应用程序中，可能会出现一个问题：JSON 格式没有查询参数，而自提交表单必须使用查询参数。为了绕过这种情况，我们可以使用包含 JSON 负载（包括隐藏输入）的自提交表单来利用 CSRF。我们需要将编码类型（`enctype`）更改为 `text/plain`，以确保负载按原样传递。漏洞利用代码如下所示：

```html
<html>
 <body>
  <script>history.pushState('', '', '/')</script>
   <form action='https://victimsite.com' method='POST' enctype='text/plain'>
     <input type='hidden' name='{"name":"hacked","password":"hacked","padding":"'value='something"}' />
     <input type='submit' value='Submit request' />
   </form>
 </body>
</html>
```

POST 请求将如下：

```http
POST / HTTP/1.1
Host: victimsite.com
Content-Type: text/plain

{"name":"hacked","password":"hacked","padding":"=something"}
```

当此数据作为 POST 请求发送时，服务器将欣然接受 name 和 password 字段，而忽略名为 padding 的字段，因为它不需要。
