# 测试会话劫持

|ID          |
|------------|
|WSTG-SESS-09|

## 概述

能够获取用户会话 cookie 的攻击者可以通过出示这些 cookie 来冒充用户。这种攻击称为会话劫持。当考虑网络攻击者（即控制受害者所用网络的攻击者）时，会话 cookie 可能通过 HTTP 不当地暴露给攻击者。为防止这种情况，会话 cookie 应标记 `Secure` 属性，以便仅通过 HTTPS 进行通信。

请注意，即使 Web 应用程序完全部署在 HTTPS 上，也应使用 `Secure` 属性，否则可能发生以下 cookie 窃取攻击。假设 `example.com` 完全部署在 HTTPS 上，但未将其会话 cookie 标记为 `Secure`。则可能发生以下攻击步骤：

1. 受害者向 `https://another-site.com` 发送请求。
2. 攻击者破坏相应的响应，使其触发对 `https://example.com` 的请求。
3. 浏览器现在尝试访问 `https://example.com`。
4. 尽管请求失败，但会话 cookie 以明文形式通过 HTTP 泄露。

或者，可以使用 [HSTS](https://en.wikipedia.org/wiki/HTTP_Strict_Transport_Security) 禁止使用 HTTP 来防止会话劫持。请注意，这里有一个与 cookie 作用域相关的细微差别。特别是，当会话 cookie 设置了 `Domain` 属性时，需要全面采用 HSTS。

全面采用 HSTS 在 Stefano Calzavara、Alvise Rabitti、Alessio Ragazzo 和 Michele Bugliesi 合著的论文《Testing for Integrity Flaws in Web Sessions》中有所描述。当主机为自身及其所有子域激活 HSTS 时，即全面采用 HSTS。部分采用 HSTS 是指主机仅为自身激活 HSTS。

设置 `Domain` 属性后，会话 cookie 可以跨子域共享。应避免使用 HTTP 与子域通信，以防止未加密的 cookie 通过 HTTP 发送时被泄露。为了说明此安全缺陷，假设站点 `example.com` 激活了 HSTS 但不带 `includeSubDomains` 选项。该站点将会话 cookie 的 `Domain` 属性设置为 `example.com`。则可能发生以下攻击：

1. 受害者向 `https://another-site.com` 发送请求。
2. 攻击者破坏相应的响应，使其触发对 `https://fake.example.com` 的请求。
3. 浏览器现在尝试访问 `https://fake.example.com`，这被 HSTS 配置所允许。
4. 由于请求发送到设置了 `Domain` 属性的 `example.com` 子域，因此请求包含会话 cookie，这些 cookie 以明文形式通过 HTTP 泄露。

应在顶点域上激活全面 HSTS 以防止此攻击。

## 测试目标

- 识别易受攻击的会话 cookie。
- 劫持易受攻击的 cookie 并评估风险级别。

## 如何测试

该测试策略针对网络攻击者，因此仅需应用于未全面采用 HSTS 的站点（全面采用 HSTS 的站点是安全的，因为它们的 cookie 不会通过 HTTP 通信）。我们假设在被测站点上有两个测试账户，一个充当受害者，一个充当攻击者。我们模拟一个场景，攻击者窃取所有未受保护免于 HTTP 泄露的 cookie，并将其提交给站点以访问受害者的账户。如果这些 cookie 足以代表受害者行事，则会话劫持是可能的。

执行此测试的步骤如下：

1. 以受害者身份登录站点，并访问任何需要认证的安全功能页面。
2. 从 cookie jar 中删除所有满足以下任一条件的 cookie：
    - 在未采用 HSTS 的情况下：设置了 `Secure` 属性。
    - 在部分采用 HSTS 的情况下：设置了 `Secure` 属性或未设置 `Domain` 属性。
3. 保存 cookie jar 的快照。
4. 触发步骤 1 中识别的安全功能。
5. 观察步骤 4 中的操作是否成功执行。如果是，则攻击成功。
6. 清除 cookie jar，以攻击者身份登录并访问步骤 1 中的页面。
7. 将步骤 3 中保存的 cookie 逐一写入 cookie jar。
8. 再次触发步骤 1 中识别的安全功能。
9. 清除 cookie jar，再次以受害者身份登录。
10. 观察步骤 8 中的操作是否在受害者的账户中成功执行。如果是，则攻击成功；否则，站点对会话劫持是安全的。

我们建议为受害者和攻击者使用两台不同的机器或浏览器。如果 Web 应用程序执行指纹识别以验证从给定 cookie 启用的访问权限，这可以帮助减少误报数量。测试策略的一个更短但精确度稍低的变体只需要一个测试账户。它遵循相同的模式，但在步骤 5 处停止（注意这使得步骤 3 无用）。
