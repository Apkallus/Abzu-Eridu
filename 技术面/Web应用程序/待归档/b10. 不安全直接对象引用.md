# 第10章：不安全直接对象引用（Insecure Direct Object References）

与 XSS 和开放重定向一样，不安全直接对象引用（IDOR）是几乎每个 Web 应用程序中都存在的一类漏洞。当应用程序基于用户请求直接授予对资源的访问权限而不进行验证时，就会发生 IDOR。

在本章中，我们将探讨这些漏洞的工作原理。然后深入探讨应用程序如何防御 IDOR，以及如何绕过那些常见的防护机制。

## 机制

尽管名字又长又吓人，但 IDOR 其实很容易理解；它本质上就是**访问控制缺失**。当用户能够通过直接引用对象 ID、对象编号或文件名来访问不属于他们的资源时，就会发生 IDOR。

例如，假设 example.com 是一个允许你与他人聊天的社交媒体站点。注册后，你发现你的用户 ID 是 1234。该网站允许你通过点击首页上的 “查看你的消息” 按钮来查看与好友的所有消息。点击该按钮后，你会被重定向到以下位置，显示你的所有私信：`https://example.com/messages?user_id=1234`。

现在，如果你将 URL 栏中的地址改为 `https://example.com/messages?user_id=1233` 呢？

你会发现现在可以看到另一名用户（用户 1233）与其好友之间的所有私信。此时，你就发现了一个 IDOR 漏洞。应用程序未基于用户身份限制对消息的访问，而是允许用户请求他们想要的任何消息。应用程序天真地信任用户输入，直接根据用户提供的 `user_id` 值加载资源，如下示例代码所示：

```python
messages = load_messages(request.user_id)
display_messages(messages)
```

IDOR 不仅限于读取其他用户的信息。你还可以利用它们以其他用户的名义编辑数据。例如，假设用户可以提交 POST 请求来更改密码。POST 请求必须包含该用户的 ID 和新密码，并定向到 `/change_password` 端点：

```
POST /change_password
(POST request body)
user_id=1234&new_password=12345
```

在这种情况下，如果应用程序不验证提交的用户 ID 是否对应当前登录用户，攻击者可能通过发送不属于自己的用户 ID 来更改其他人的密码，如下所示：

```
POST /change_password
(POST request body)
user_id=1233&new_password=12345
```

最后，IDOR 还可以影响数据库对象以外的资源。另一种 IDOR 发生在应用程序直接引用系统文件时。例如，此请求允许用户访问他们已上传的文件：`https://example.com/uploads?file=user1234-01.jpeg`。

由于 `file` 参数的值是 `user1234-01.jpeg`，我们可以轻易推断出用户上传文件的命名规范为 `USER_ID-FILE_NUMBER.FILE_EXTENSION`。因此，其他用户上传的文件可能命名为 `user1233-01.jpeg`。如果应用程序不限制用户访问属于他人的文件，攻击者可以通过猜测文件名来访问任何人的上传文件，例如：`https://example.com/uploads?file=user1233-01.jpeg`。

恶意用户甚至可能通过此端点读取敏感的系统文件！例如，`/etc/shadow` 是 Unix 系统上用于记录用户密码的文件。由于它很敏感，不应暴露给普通用户。如果你能通过类似 `https://example.com/uploads?file=/PATH/TO/etc/shadow` 的 URL 读取该文件，那么你就发现了一个漏洞！攻击者能够读取 Web 根目录之外的文件也称为**路径遍历攻击**或**目录遍历攻击**。我们将在第 17 章详细讨论目录遍历攻击。

## 防御措施

IDOR 的发生源于应用程序在两个方面失败。首先，它未能基于用户身份实施访问控制。其次，它未能随机化对象 ID，而是使对数据对象（如文件或数据库条目）的引用保持可预测性。

在本章的第一个示例中，你能够看到属于用户 1233 的消息，因为服务器在发送私人信息之前未检查登录用户的身份。服务器未验证你实际上就是用户 1233，而是直接返回了你请求的信息。

在这种情况下，由于用户 ID 只是数字，很容易推断你还可以检索用户 1232 和用户 1231 的消息，如下所示：

```
https://example.com/messages?user_id=1232
https://example.com/messages?user_id=1231
```

这就是为什么该漏洞被称为不安全直接对象引用。用户的 ID 被用于直接引用该站点上用户的私信。如果未通过适当的访问控制加以保护，这些可预测的直接对象引用会暴露其背后隐藏的数据，允许任何人获取与该引用关联的信息。

应用程序可以通过两种方式防止 IDOR。首先，应用程序在授予对资源的访问权限之前可以检查用户的身份和权限。例如，应用程序可以检查用户的会话 Cookie 是否与用户请求消息的 `user_id` 相对应。

其次，网站可以使用唯一的、不可预测的密钥或哈希标识符来引用每个用户的资源。**哈希**是指将值转换为另一个字符串的单向过程。使用安全算法和密钥对 ID 进行哈希处理，使攻击者难以猜测哈希 ID 字符串。如果 example.com 将其请求结构改为如下形式，攻击者将无法再访问其他用户的消息，因为攻击者无法猜测如此长且随机的 `user_key` 值：

```
https://example.com/messages?user_key=6MT9EalV9F7r9pns0mK1eDAEW
```

但这种方法并不能完全防御 IDOR。如果攻击者能找到窃取这些 URL 或 `user_key` 的方法，他们仍然可以泄露用户信息。防御 IDOR 的最佳方法是细粒度的访问控制，或者将访问控制与 ID 随机化或哈希相结合。

## 寻找 IDOR

让我们来寻找一些 IDOR！发现 IDOR 的最佳方法是通过源代码审查，检查所有直接对象引用是否都受到访问控制保护。我们将在第 22 章讨论如何进行源代码审查。但如果你无法访问应用程序的源代码，这里有一种简单有效的 IDOR 测试方法。

### 第 1 步：创建两个账户

首先，在目标网站上创建两个不同的账户。如果用户在站点上可以拥有不同的权限，请为每个权限级别创建两个账户。例如，创建两个管理员账户、两个普通用户账户、两个群组成员账户和两个非群组成员账户。这将帮助你在相似用户账户之间以及具有不同权限的用户之间测试访问控制问题。

继续前面的示例，你可以在 example.com 上创建两个账户：用户 1235 和用户 1236。其中一个账户作为你的攻击者账户，用于执行 IDOR 攻击。另一个作为受害者账户，用于观察攻击效果。两个用户的消息页面具有以下 URL：

- `https://example.com/messages?user_id=1235`（攻击者）
- `https://example.com/messages?user_id=1236`（受害者）

如果应用程序不允许你创建这么多账户，你可以联系公司请求更多账户。如果你解释自己正在参与他们的漏洞赏金计划，公司通常会授予你额外账户。此外，如果应用程序有付费会员，请向公司申请高级账户或自己付费购买。通常，付费购买这些会员是值得的，因为你可以获得测试新功能的机会。

除了使用两个账户进行测试外，你还应在未登录的情况下重复测试过程。查看是否可以在未经身份验证的情况下访问合法用户可用的信息或功能。

### 第 2 步：发现功能

接下来，尝试尽可能多地发现应用程序功能。使用你拥有的最高权限账户，浏览应用程序，寻找可供测试的功能。

特别关注那些返回用户信息或修改用户数据的功能。记录它们以备后用。以下是 example.com 上可能存在 IDOR 的一些功能：

- 此端点允许你读取用户消息：
  - `https://example.com/messages?user_id=1236`
- 此端点允许你读取用户文件：
  - `https://example.com/uploads?file=user1236-01.jpeg`
- 此端点删除用户消息：
  - `POST /delete_message`
  - `(POST request body) message_id=user1236-0111`
- 此端点用于访问群组文件：
  - `https://example.com/group_files?group=group3`
- 此端点删除群组：
  - `POST /delete_group`
  - `(POST request body) group=group3`

### 第 3 步：捕获请求

浏览你在上一步中映射的每个应用程序功能，捕获从 Web 客户端发往服务器的所有请求。仔细检查每个请求，找到包含数字、用户名或 ID 的参数。请记住，你可以在请求中的不同位置触发 IDOR，如 URL 参数、表单字段、文件路径、头和 Cookie。

为了提高测试效率，使用两个浏览器，每个浏览器登录不同的账户。然后操纵来自一个浏览器的请求，观察更改是否立即反映在另一个账户上。

例如，假设你创建了两个账户 1235 和 1236。在 Firefox 中登录 1235，在 Chrome 中登录 1236。使用 Burp 修改来自 Firefox 的流量。在 Proxy 选项卡中开启 Intercept，在代理文本窗口中编辑请求（图 10-1）。通过观察在 Chrome 中受害者账户上的变化来检查攻击是否成功。

### 第 4 步：更改 ID

最后，在敏感请求中切换 ID，检查返回的信息是否也随之变化。查看是否可以使用攻击者账户访问受害者账户的信息。并检查是否可以从第一个账户修改第二个账户的信息。

例如，在此设置中，你可以尝试通过 Firefox 浏览器访问用户 1236 有权访问的功能：

- 此端点允许你读取用户消息：
  - `https://example.com/messages?user_id=1236`
- 此端点允许你读取用户文件：
  - `https://example.com/uploads?file=user1236-01.jpeg`
- 此端点删除用户消息：
  - `POST /delete_message`
  - `(POST request body) message_id=user1236-0111`
- 此端点用于访问群组文件：
  - `https://example.com/group_files?group=group3`
- 此端点删除群组：
  - `POST /delete_group`
  - `(POST request body) group=group3`

如果其中任何请求成功访问或修改了用户 1236 的信息，你就发现了一个 IDOR 漏洞。

## 绕过 IDOR 保护

IDOR 并不总是像切换数字 ID 那样简单。随着应用程序功能越来越复杂，它们引用资源的方式也常常变得更加复杂。现代 Web 应用程序也开始实施更多针对 IDOR 的保护，许多现在使用更复杂的 ID 格式。这意味着简单的数字 IDOR 越来越少见。

我们如何绕过这些障碍并仍然找到 IDOR 呢？

IDOR 在应用程序中可以以不同方式表现。除了普通的数字 ID 之外，以下是几个需要关注的地方。

### 编码 ID 和哈希 ID

首先，不要忽略编码和哈希 ID。当面对看似随机的字符串时，始终怀疑它被编码了并尝试解码。你还应学会识别最常见的编码方案，如 base64、URL 编码和 base64url。例如，查看此端点的 ID：

```
https://example.com/messages?user_id=MTIzNQ
https://example.com/messages?user_id=MTIzNg
```

这些 `user_id` 只是用户 ID 的 base64url 编码版本。`MTIzNQ` 是 1235 的 base64url 编码字符串，`MTIzNg` 是 1236 的编码版本。某些应用程序使用的编码方案可以轻松逆向。在这种情况下，你只需使用在线 base64url 编码器编码伪造的 ID 即可执行 IDOR。

你可能一开始无法判断站点使用的是哪种编码方案。此时，可以使用 Burp Decoder 中的 Smart Decode 工具（图 10-2），或者尝试用不同方案（URL 编码、HTML 编码、十六进制编码、八进制编码、base64、base64url 等）解码该字符串，以找出所使用的编码方案。一旦你积累了更多阅读编码数据的经验，就会培养出识别编码方案的直觉。

*（图 10-2：你可以尝试使用不同方法在 Burp Decoder 中解码字符串。或者使用 Smart Decode 工具查看 Burp 能否检测到编码方案。）*

如果应用程序使用哈希或随机 ID，查看 ID 是否可预测。有时应用程序使用的算法产生的熵不足。**熵**是指 ID 的随机程度。字符串的熵越高，越难猜测。有些 ID 没有足够的熵，经过仔细分析后可以预测。在这种情况下，尝试创建几个账户来分析这些 ID 的生成方式。你可能能够找到某种模式，从而预测属于其他用户的 ID。

### 泄露的 ID

应用程序也有可能通过另一个 API 端点或应用程序的其他公共页面（如用户个人资料页面）泄露 ID。

我曾经发现一个 API 端点，允许用户通过哈希的 `conversation_id` 值检索详细的私信。请求如下：

```
GET /messages?conversation_id=O1SUR7GJ43HS93VAR8xxxx
```

乍一看这似乎是安全的，因为 `conversation_id` 是一个长且随机的字母数字序列。但我后来发现，任何人都可以通过使用他们的公开用户 ID 来请求每个用户的 `conversation_id` 列表！以下请求会返回属于该用户的 `conversation_id` 列表：

```
GET /messages?user_id=1236
```

由于 `user_id` 在每个用户的个人资料页面上都是公开的，我可以通过首先在其个人资料页面上获取他们的 `user_id`，检索属于该用户的 `conversation_id` 列表，最后通过 `conversation_id` 加载消息，从而读取任何用户的消息。

### 即使应用程序没有要求，也给它提供一个 ID

在现代 Web 应用程序中，你经常会遇到应用程序使用 Cookie 而非 ID 来识别用户可以访问的资源的情况。

例如，当你向端点发送以下 GET 请求时，应用程序将根据会话 Cookie 推断你的身份，然后向你发送与该用户关联的消息：

```
GET /api_v1/messages
Host: example.com
Cookies: session=YOUR_SESSION_COOKIE
```

由于你不知道其他用户的会话 Cookie，你无法使用这些会话 Cookie 读取他们的消息。这似乎表明应用程序是安全的，免受 IDOR 影响。但某些应用程序会实现另一种使用对象 ID 检索资源的方式。他们有时这样做是为了开发人员方便、向后兼容，或者仅仅因为开发人员忘记删除测试功能。

如果应用程序生成的请求中不存在 ID，请尝试向请求添加一个。将 `id`、`user_id`、`message_id` 或其他对象引用附加到 URL 查询或 POST 主体参数中，查看是否对应用程序行为产生影响。例如，假设此请求显示你的消息：

```
GET /api_v1/messages
```

那么此请求可能显示其他用户的消息：

```
GET /api_v1/messages?user_id=ANOTHER_USERS_ID
```

### 留意盲 IDOR

有时，易受 IDOR 影响的端点不会直接响应泄露的信息。它们可能导致应用程序在其他地方泄露信息：在导出文件、电子邮件甚至文本提醒中。例如，假设 example.com 上的此端点允许用户通过电子邮件向自己发送收据副本：

```
POST /get_receipt
(POST request body)
receipt_id=3001
```

此请求会将收据 3001 的副本发送到当前用户的注册邮箱。现在，如果你请求属于其他用户的收据 2983 呢？

```
POST /get_receipt
(POST request body)
receipt_id=2983
```

虽然 HTTP 响应没有变化，但你的邮箱中可能会收到收据 2983 的副本！通常，恶意请求可能导致未来某个时间的信息泄露。我曾经发现一个 IDOR 导致一个月后的月度报告中发生信息泄露。

### 更改请求方法

如果一种 HTTP 请求方法不起作用，你可以尝试其他方法：GET、POST、PUT、DELETE、PATCH 等。应用程序通常在同一端点上启用多种请求方法，但未能为每种方法实施相同的访问控制。例如，如果此 GET 请求不易受 IDOR 影响，不会返回其他用户的资源：

```
GET example.com/uploads/user1236-01.jpeg
```

你可以尝试使用 DELETE 方法来删除该资源。DELETE 方法从目标 URL 中移除资源：

```
DELETE example.com/uploads/user1236-01.jpeg
```

如果 POST 请求不起作用，你也可以尝试使用 PUT 方法更新其他用户的资源。PUT 方法更新或创建目标 URL 处的资源：

```
PUT example.com/uploads/user1236-01.jpeg
(PUT request body)
NEW_FILE
```

另一个经常有效的技巧是在 POST 和 GET 请求之间切换。如果有一个这样的 POST 请求：

```
POST /get_receipt
(POST request body)
receipt_id=2983
```

你可以尝试将其重写为 GET 请求，如下所示：

```
GET /get_receipt?receipt_id=2983
```

### 更改请求的文件类型

切换请求文件的文件类型有时会导致服务器以不同方式处理授权。应用程序可能对用户标识信息的方式比较灵活：它们可能允许用户使用 ID 引用文件，或直接使用文件名。但应用程序通常未能为每种引用方式实施相同的访问控制。

例如，应用程序通常以 JSON 文件类型存储信息。尝试在请求 URL 末尾添加 `.json` 扩展名，看看会发生什么。如果此请求被服务器阻止：

```
GET /get_receipt?receipt_id=2983
```

那么尝试这个：

```
GET /get_receipt?receipt_id=2983.json
```

## 提升攻击影响

IDOR 的影响取决于受影响的函数，因此为了最大化漏洞的严重性，你应该始终首先在关键功能中寻找 IDOR。基于读取的 IDOR（泄露信息但不更改数据库）和基于写入的 IDOR（可以未经授权地更改数据库）都可能产生高影响。

在状态更改的写入型 IDOR 方面，在密码重置、密码更改和账户恢复功能中寻找 IDOR，因为这些通常具有最高的业务影响。优先于那些更改电子邮件订阅设置的功能。

对于非状态更改（基于读取）的 IDOR，寻找处理应用程序中敏感信息的功能。例如，寻找处理私信、个人信息和私密内容的功能。考虑哪些应用程序功能使用这些信息，并相应寻找 IDOR。

你还可以将 IDOR 与其他漏洞结合以提高影响。例如，写入型 IDOR 可以与自 XSS 结合形成存储型 XSS。密码重置端点上的 IDOR 结合用户名枚举可能导致大规模账户接管。管理员账户上的写入 IDOR 甚至可能导致 RCE！我们将在第 18 章讨论 RCE。

## 自动化攻击

掌握 IDOR 寻找技巧后，你可以尝试使用 Burp 或自己编写的脚本来自动化 IDOR 寻找。例如，你可以使用 Burp Intruder 遍历 ID 以找到有效的 ID。Burp 扩展 Autorize（https://github.com/Quitten/Autorize/）通过使用低权限账户访问高权限账户来扫描授权问题，而 Burp 扩展 Auto Repeater（https://github.com/nccgroup/AutoRepeater/）和 AuthMatrix（https://github.com/SecurityInnovation/AuthMatrix/）允许你自动化切换 Cookie、头和参数的过程。有关如何使用这些工具的更多信息，请前往 Burp 窗口的 Extender 选项卡，然后到 BAppStore 选项卡找到你想使用的扩展。

## 找到你的第一个 IDOR！

现在你知道了什么是 IDOR、如何绕过 IDOR 保护以及如何提升 IDOR 的影响，你准备好寻找第一个了！选择一个漏洞赏金计划，按照本章讨论的步骤操作：

1. **为每个应用角色创建两个账户**，指定一个作为攻击者账户，另一个作为受害者账户。
2. **发现应用程序中可能导致 IDOR 的功能**。特别关注返回敏感信息或修改用户数据的功能。
3. **重新访问你在第 2 步中发现的功能**。使用代理，在浏览敏感功能时拦截浏览器流量。
4. **使用代理拦截每个敏感请求**，并切换请求中看到的 ID。如果切换 ID 后你能访问其他用户的信息或更改其数据，你可能发现了 IDOR。
5. **如果应用程序似乎不受 IDOR 影响，不要灰心**。利用这个机会尝试保护绕过技术！如果应用程序使用编码、哈希或随机 ID，你可以尝试解码或预测 ID。你也可以尝试在应用程序未要求时向它提供 ID。最后，有时更改请求方法类型或文件类型会带来不同。
6. **监控导出文件、电子邮件和文本提醒中的信息泄露**。现在的 IDOR 可能导致未来的信息泄露。
7. **起草你的第一份 IDOR 报告！**
