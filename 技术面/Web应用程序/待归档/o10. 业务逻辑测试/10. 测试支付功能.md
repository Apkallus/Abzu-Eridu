# 测试支付功能

|ID          |
|------------|
|WSTG-BUSL-10|

## 概述

许多应用程序都实现了支付功能，包括电子商务网站、订阅服务、慈善机构、捐赠网站和货币兑换。此功能的安全性至关重要，因为漏洞可能允许攻击者从组织窃取资金、进行欺诈性购买，甚至窃取其他用户的支付卡详细信息。这些问题不仅可能导致组织的声誉受损，还可能导致重大的财务损失，包括直接损失和行业监管机构的罚款。

## 测试目标

- 确定电子商务功能的业务逻辑是否健壮。
- 了解支付功能的工作原理。
- 确定支付功能是否安全。

## 如何测试

### 支付网关集成方法

应用程序可以通过几种不同的方式集成支付功能，测试方法将根据使用的方法而有所不同。最常见的方法有：

- 将用户重定向到第三方支付网关。
- 在应用程序的 IFRAME 中加载第三方支付网关。
- 使用 HTML 表单向第三方支付网关发出跨域 POST 请求。
- 直接接受卡详细信息，然后从应用程序后端向支付网关的 API 发出 POST 请求。

### PCI DSS

支付卡行业数据安全标准（PCI DSS）是组织在处理借记卡和信用卡支付时必须遵循的标准（但需要注意的是，它不是法律）。对本标准的全面讨论超出了本指南（以及大多数渗透测试）的范围——但测试人员了解几个关键点是有用的。

关于 PCI DSS 最常见的误解是它仅适用于存储持卡人数据（即借记卡或信用卡详细信息）的系统。这是不正确的：它适用于任何“存储、处理或传输”这些信息的系统。具体需要遵循哪些要求取决于使用了哪种支付网关集成方法。[Visa 电子商务支付处理指南](https://web.archive.org/web/2023/https://www.visa.co.uk/dam/VCOM/regional/ve/unitedkingdom/PDF/risk/processing-e-commerce-payments-guide-73-17337.pdf)提供了更多细节，但简要总结如下：

| 集成方法 | 自我评估问卷（SAQ） |
|--------------------|-------------------------------------|
| 重定向 | [SAQ A](https://www.pcisecuritystandards.org/documents/PCI-DSS-v3_2_1-SAQ-A.pdf) |
| IFRAME | [SAQ A](https://www.pcisecuritystandards.org/documents/PCI-DSS-v3_2_1-SAQ-A.pdf) |
| 跨域 POST | [SAQ A-EP](https://www.pcisecuritystandards.org/documents/PCI-DSS-v3_2-SAQ-A_EP-rev1_1.pdf) |
| 后端 API | [SAQ D](https://www.pcisecuritystandards.org/documents/PCI-DSS-v3_2_1-SAQ-D_Merchant.pdf) |

除了每种方法的攻击面和风险状况不同之外，组织需要满足的 SAQ A（22 项要求）和 SAQ D（329 项要求）之间的要求数量也存在显著差异。因此，值得强调未使用重定向或 IFRAME 的应用程序，因为它们代表更高的技术和合规风险。

### 数量篡改

大多数电子商务网站允许用户在开始结账流程之前将商品添加到购物篮中。此购物篮应跟踪已添加的商品及每种商品的数量。数量通常应为正整数，但如果网站未正确验证此值，则可能指定商品的小数数量（例如 `0.1`）或负数数量（例如 `-1`）。根据后端处理方式，添加负数数量的商品可能导致负值，从而降低购物篮的总成本。

通常有几种方式可以修改购物篮的内容，应进行测试，例如：

- 添加负数数量的商品。
- 重复移除商品直到数量变为负数。
- 将数量更新为负值。

某些站点可能还会提供有效数量的下拉菜单（例如必须按 10 个一组购买的商品），并且可能通过篡改这些请求来添加其他数量的商品。

如果完整的购物篮详细信息被传递给支付网关（而不是仅传递总价值），则也可能在该阶段篡改值。

最后，如果应用程序存在 [HTTP 参数污染](../07-Input_Validation_Testing/04-Testing_for_HTTP_Parameter_Pollution.md)漏洞，则可能通过多次传递参数而导致意外行为，例如：

```http
POST /api/basket/add
Host: example.org

item_id=1&quantity=5&quantity=4
```

### 价格篡改

#### 在应用程序上

将商品添加到购物篮时，应用程序应仅包含商品和数量，如下例所示：

```http
POST /api/basket/add HTTP/1.1
Host: example.org

item_id=1&quantity=5
```

然而，在某些情况下，应用程序可能还包含价格，这意味着可能可以篡改它：

```http
POST /api/basket/add HTTP/1.1
Host: example.org

item_id=1&quantity=5&price=2.00
```

不同类型的商品可能具有不同的验证规则，因此每种类型都需要单独测试。某些应用程序还允许用户在购买时添加可选的慈善捐款，并且此捐款通常可以是任意金额。如果此金额未经验证，则可能添加负数捐款金额，从而降低购物篮的总价值。

#### 在支付网关上

如果结账流程在第三方支付网关上进行，则可能篡改应用程序和网关之间的价格。

向网关的传输可以使用跨域 POST 到网关来执行，如下面的 HTML 示例所示。

> 注意：此请求中不包含卡详细信息——用户将在支付网关上被提示输入它们：

```html
<form action="https://example.org/process_payment" method="POST">
    <input type="hidden" id="merchant_id" value="123" />
    <input type="hidden" id="basket_id" value="456" />
    <input type="hidden" id="item_id" value="1" />
    <input type="hidden" id="item_quantity" value="5" />
    <input type="hidden" id="item_total" value="20.00" />
    <input type="hidden" id="shipping_total" value="2.00" />
    <input type="hidden" id="basket_total" value="22.00" />
    <input type="hidden" id="currency" value="GBP" />
    <input type="submit" id="submit" value="submit" />
</form>
```

通过修改 HTML 表单或拦截 POST 请求，可能修改商品价格，并有效地以更低价格购买它们。请注意，许多支付网关会拒绝交易金额为 0 的交易，因此总金额为 0.01 更有可能成功。然而，某些支付网关可能接受负值（用于处理退款）。如果存在多个值（如商品价格、运费和购物篮总价），所有这些都应进行测试。

如果支付网关使用 IFRAME，则可能通过修改 IFRAME URL 来执行类似类型的攻击：

```html
<iframe src="https://example.org/payment_iframe?merchant_id=123&basket_total=22.00" />
```

> 注意：支付网关通常由第三方运行，因此可能不在测试范围内。这意味着虽然价格篡改可能是可以接受的，但其他类型的攻击（如 SQL 注入）在未经明确书面批准的情况下不应执行。

#### 加密交易详情

为了防止交易被篡改，某些支付网关会加密向其发出的请求的详细信息。例如，[PayPal](https://developer.paypal.com/api/nvp-soap/paypal-payments-standard/integration-guide/encryptedwebpayments/#link-usingewptoprotectmanuallycreatedpaymentbuttons) 使用公钥加密技术来实现这一点。

首先要尝试的是发出未加密的请求，因为某些支付网关允许不安全的交易，除非它们被特别配置为拒绝。

如果这不起作用，那么您需要找到用于加密交易详情的公钥，该密钥可能暴露在应用程序的备份中，或者如果您能找到目录遍历漏洞的话。

另外，应用程序可能对支付网关及其数字证书重用相同的公钥/私钥对。您可以使用以下命令从服务器获取公钥：

```bash
echo -e '\0' | openssl s_client -connect example.org:443 2>/dev/null | openssl x509 -pubkey -noout
```

获得此密钥后，您可以尝试根据支付网关的文档创建加密请求，并将其提交给网关以查看是否被接受。

#### 安全哈希

其他支付网关使用交易详情的安全哈希（或 HMAC）来防止篡改。具体方式因提供商而异（例如，[Adyen](https://docs.adyen.com/online-payments/classic-integrations/hosted-payment-pages/hmac-signature-calculation) 使用 HMAC-SHA256），但通常包括交易详情和一个秘密值。例如，哈希可以计算为：

```php
$secure_hash = md5($merchant_id . $transaction_id . $items . $total_value . $secret)
```

此值随后被添加到发送到支付网关的 POST 请求中，并进行验证以确保交易未被篡改。

首先要尝试的是移除安全哈希，因为某些支付网关允许不安全的交易，除非设置了特定的配置选项。

POST 请求应包含计算此哈希所需的所有值（密钥除外）。这意味着如果您知道如何计算哈希（应包含在支付网关的文档中），则可以尝试暴力破解该秘密。或者，如果站点运行的是现成应用程序，配置文件中可能有默认秘密。最后，如果您能找到站点的备份或获得对配置文件的访问权限，则可能在那里找到该秘密。

如果可以获取此秘密，则可以篡改交易详情，然后生成自己的安全哈希，该哈希将被支付网关接受。

#### 货币篡改

如果无法篡改实际价格，则可能更改所使用的货币，尤其是在支持多种货币的应用程序中。例如，应用程序可能验证价格为 10，但如果您可以更改货币，使您支付 10 美元而不是 10 英镑，这将允许您更便宜地购买商品。

#### 时间延迟请求

如果站点上商品的价值随时间变化（例如在货币兑换中），则可能通过使用本地代理拦截请求并延迟它们来以旧价格买入或卖出。要使其可利用，价格需要包含在请求中，或与请求中的某些内容（如会话或交易 ID）关联。以下示例显示了在允许用户买卖黄金的应用程序上如何可能利用此漏洞：

- 查看站点上黄金的当前价格。
- 发起购买 1 盎司黄金的请求。
- 拦截并冻结该请求。
- 等待一分钟以再次检查黄金价格：
    - 如果价格上涨，允许交易完成，并以低于当前价值的价格购买黄金。
    - 如果价格下跌，放弃该请求。

如果站点允许用户使用加密货币支付（通常波动性更大），则可能通过以该加密货币获得固定价格，然后等待观察该价值与站点使用的主要货币相比是上涨还是下跌来利用此漏洞。

### 折扣码

如果应用程序支持折扣码，则应执行各种检查：

- 代码是否容易猜测（TEST、TEST10、SORRY、SORRY10、公司名称等）？
    - 如果代码中包含数字，是否可以通过增加数字找到更多代码？
- 是否有暴力破解防护？
- 是否可以同时应用多个折扣码？
- 折扣码是否可以多次应用？
- 是否可以[注入通配符](../07-Input_Validation_Testing/05-Testing_for_SQL_Injection.md#sql-wildcard-injection)，如 `%` 或 `*`？
- 折扣码是否暴露在应用程序任何地方的 HTML 源代码或隐藏的 `<input>` 字段中？

除了这些，还应测试常见的漏洞，如 SQL 注入。

### 破坏支付流程

如果应用程序上的结账或支付流程涉及多个阶段（例如将商品添加到购物篮、输入折扣码、输入送货详情和输入账单信息），则可能通过以非预期顺序执行这些步骤而导致意外行为。例如，您可以尝试：

- 在输入账单详情后修改送货地址以降低运费。
- 在输入送货详情后移除商品，以避免最低购物篮价值。
- 在应用折扣码后修改购物篮内容。
- 在完成结账流程后修改购物篮内容。

也可能跳过交易的整个支付流程。例如，如果应用程序重定向到第三方支付网关，支付流程可能是：

- 用户在应用程序上输入详情。
- 用户被重定向到第三方支付网关。
- 用户输入其卡详细信息。
    - 如果支付成功，他们被重定向到应用程序上的 `success.php`。
    - 如果支付失败，他们被重定向到应用程序上的 `failure.php`。
- 应用程序更新其订单数据库，并在成功时处理订单。

根据应用程序是否实际验证网关上的支付是否成功，可能可以强制浏览到 `success.php` 页面（如果可能需要，可能包含交易 ID），这将导致站点处理订单，就好像支付成功一样。此外，可能可以重复请求 `success.php` 页面，以导致订单被多次处理。

### 利用交易处理费

商户通常需要为每笔交易支付费用，这些费用通常由少量固定费用和总价值的一定百分比组成。这意味着接受非常小的支付（如 $0.01）可能导致商户实际上亏损，因为交易处理费用大于交易的总价值。

此问题在电子商务网站上很少可利用，因为最便宜商品的价格通常足够高以防止这种情况。但是，如果站点允许客户进行任意金额的支付（如捐款），请检查其是否强制执行合理的最小值。

### 测试支付卡

大多数支付网关都有一组定义的测试卡详细信息，开发人员在测试和调试期间可以使用。这些卡详细信息通常只能在网关的开发或沙盒版本上使用，但如果配置错误，可能会在正式站点上被接受。

以下列出了各种支付网关的这些测试详细信息示例：

- [Adyen - 测试卡号](https://docs.adyen.com/development-resources/test-cards-and-credentials/test-card-numbers)
- [Globalpay - 测试卡](https://developer.globalpay.com/resources/test-card-numbers)
- [Stripe - 基本测试卡号](https://stripe.com/docs/testing#cards)

### 测试物流

在应用程序上测试支付功能可能会引入额外的复杂性，尤其是在测试正式站点时。需要考虑的领域包括：

- 为应用程序获取测试卡支付详细信息。
    - 如果这些不可用，则可能可以获得预付卡或替代方案。
- 记录所做的任何订单，以便取消和退款。
- 不要下无法取消的订单，或会导致其他操作（如商品立即从仓库发货）的订单。

#### 源等于目标

如果转账的来源等于目标，则可能导致仅向账户增加价值而没有任何实际转账发生。应测试此场景以确保应用程序防止此类操作。

#### 两步支付或转账

对于需要两步（发起和确认）的支付或转账，确保在两个阶段都执行检查。例如：

- 发起两笔独立的支付。
- 单独确认它们。

验证必要的检查（如每日限额或余额验证）是否在确认阶段执行。如果未能做到这一点，可能导致负余额或绕过限额。

#### 支付发起后添加商品

测试以下场景：支付已发起，随后将商品添加到购物车。确认支付可能导致已添加的商品被标记为已支付，这可能导致支付过程中的不一致。

#### 竞争条件

竞争条件发生在应用程序的行为取决于未正确控制的事件时间或顺序时。在支付环境中，这通常表现为多个并发请求与共享状态（如账户余额、库存计数或一次性优惠码）交互，而无需适当的同步，允许攻击者利用检查和后续操作之间的时间差。这通常被称为检查时间到使用时间（TOCTOU）缺陷。

竞争条件在以下情况下特别有影响：

- 金融交易（转账、支付、提款）
- 一次性令牌或代码（优惠券、礼品卡、推荐奖金）
- 有限可用资源分配（库存、座位、注册）
- 应仅发生一次的状态转换（账户激活、批准工作流）

要测试，请关注检查和修改值的位置，或操作应仅成功一次的位置。同时发送多个相同请求，并观察应用程序是否处理了多个成功请求（而本应只有一个成功）。

使用 curl，向状态更改端点发送 20 个快速请求：

```bash
# 发送 20 个接近并发的请求以兑换一次性优惠码
for i in $(seq 1 20); do
  curl -s -X POST https://example.com/api/redeem-coupon \
    -H "Cookie: session=USER_SESSION" \
    -H "Content-Type: application/json" \
    -d '{"code":"SINGLE-USE-CODE"}' &
done
wait
```

要获得更精确的时间控制，请使用 [Burp Suite Turbo Intruder](https://github.com/PortSwigger/turbo-intruder) 及其 gate 机制，以保持所有请求并同时释放它们：

```python
def queueRequests(target, wordlists):
    engine = RequestEngine(endpoint=target.endpoint,
                           concurrentConnections=20,
                           requestsPerConnection=1,
                           pipeline=False)
    for i in range(20):
        engine.queue(target.req, gate='race')
    engine.openGate('race')

def handleResponse(req, interesting):
    table.add(req)
```

常见的测试场景包括：

- 并发支付确认：
  同时发起多个确认请求（例如 `POST /confirm-payment`）针对同一订单。这可能导致同一订单被多次处理。

- 回调重放或洪水攻击：
  拦截网关的回调请求（例如到 `success.php` 或 `/payment/callback`），并快速并行重放。如果后端缺乏适当的幂等性检查，这可能导致：
    - 触发多个订单履行事件（例如发货、信用）。
    - 将同一订单标记为“已支付”多次。
    - 导致余额膨胀或库存错误。

- 余额操作：
  如果账户有 100 个积分，同时发送多个消耗全部余额的并发请求。如果总计扣除超过原始余额（例如余额变为 -100），则应用程序存在漏洞。

- 一次性令牌：
  对于优惠码、邀请链接或密码重置令牌，同时发送多个尝试使用该令牌的并发请求。如果超过一个成功，则应用程序未能原子地使令牌失效。

漏洞指标包括：对单次操作返回多个成功响应、并发扣除后出现负余额、资源分配超出声明容量，或重复的副作用（多封电子邮件、重复收费）。

为了缓解竞争条件，请将检查-修改操作包装在具有行级锁定的数据库事务中（`SELECT ... FOR UPDATE`），使用版本列实现乐观锁定，为状态更改请求要求幂等性密钥，并使用 `DELETE FROM tokens WHERE token = ? RETURNING *` 等操作原子地使令牌失效。

另请参见 [CWE-362：使用共享资源的并发执行，同步不当](https://cwe.mitre.org/data/definitions/362.html)。

#### 多输入系统（批量支付）

在支持批量支付的系统中，测试总金额保持为正但单个输入包含负值的场景。例如：

```plaintext
account_id_1 = $5
account_id_2 = -$4
Total = 支付 $1，但入账 $5
```

确保应用程序正确处理此类情况并防止利用。

## 修复

- 尽可能避免存储、传输或处理卡详细信息。
    - 对支付网关使用重定向或 IFRAME。
- 审查支付网关文档并使用所有可用的安全功能（如加密和安全哈希）。
- 在服务器端处理所有与定价相关的信息：
    - 客户端请求中应仅包含商品 ID 和数量。
- 实施适当的输入验证和业务逻辑约束（如检查负商品编号或值）。
- 确保应用程序支付流程健壮，且步骤不能按顺序外执行。
