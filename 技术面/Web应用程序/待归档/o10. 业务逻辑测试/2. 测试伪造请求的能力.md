# 测试伪造请求的能力

|ID          |
|------------|
|WSTG-BUSL-02|

## 概述

伪造请求是攻击者用来绕过前端 GUI 应用程序，直接向后端处理提交信息的一种方法。攻击者的目标是通过拦截代理发送带有数据值的 HTTP POST/GET 请求，这些值不受应用程序业务逻辑支持、防范或预期。伪造请求的一些示例包括利用可猜测或可预测的参数，或暴露“隐藏”的特性和功能，例如启用调试或显示特殊屏幕或窗口，这些在开发期间非常有用，但可能泄露信息或绕过业务逻辑。

与伪造请求能力相关的漏洞是每个应用程序特有的，与业务逻辑数据验证不同，其重点在于破坏业务逻辑工作流。

应用程序应具备逻辑检查，以防止系统接受伪造请求，从而避免攻击者有机会利用业务逻辑、流程或应用程序流。请求伪造并非新鲜事；攻击者使用拦截代理向应用程序发送 HTTP POST/GET 请求。通过请求伪造，攻击者可能通过查找、预测和操纵参数来绕过业务逻辑或流程，使应用程序认为某个流程或任务已经发生或尚未发生。

此外，伪造请求可能通过调用“隐藏”的特性或功能（如开发人员和测试人员最初使用的调试功能，有时被称为[“彩蛋”](https://en.wikipedia.org/wiki/Easter_egg_(media))）来颠覆程序化或业务逻辑流。“彩蛋”是作品（如计算机程序、电影、书籍或填字游戏）中的故意内部玩笑、隐藏信息或功能。根据游戏设计师 Warren Robinett 的说法，这个词是由 Atari 的员工创造的，他们被提醒注意 Robinett 在其已广泛发行的游戏《冒险》中隐藏的秘密信息。据说这个名字唤起了传统寻找复活节彩蛋活动的想法。

### 示例 1

假设一个电子商务剧院网站允许用户选择门票，对整笔交易应用一次性的 10% 老年人折扣，查看小计并进行销售结算。如果攻击者通过代理发现应用程序有一个隐藏字段（值为 1 或 0），业务逻辑使用该字段来确定是否已享受折扣。攻击者随后可以多次提交“1”或“未享受折扣”的值，以多次利用相同的折扣。

### 示例 2

假设一个在线视频游戏根据找到海盗宝藏、海盗以及完成每个关卡所获得的积分来支付代币。这些代币稍后可兑换奖品。此外，每个关卡的积分都有一个等于关卡等级的乘数值。如果攻击者通过代理发现应用程序在开发和测试期间使用了一个隐藏字段，可以快速到达游戏最高级别，他们可以迅速达到最高级别并快速积累不应得的积分。

此外，如果攻击者通过代理发现应用程序在开发和测试期间使用了一个隐藏字段来启用日志，该日志指示其他在线玩家或隐藏宝藏相对于攻击者的位置，他们就能快速到达这些位置并获得积分。

## 测试目标

- 审查项目文档，寻找字段的可猜测、可预测或隐藏功能。
- 插入逻辑上有效的数据以绕过正常业务逻辑工作流。

## 如何测试

### 通过识别可猜测值

- 使用拦截代理观察 HTTP POST/GET，寻找值是否以固定间隔递增或易于猜测的迹象。
- 如果发现某个值是可猜测的，则可以更改该值，并可能获得意外的可见性。

### 通过识别隐藏选项

- 使用拦截代理观察 HTTP POST/GET，寻找隐藏功能（如可开启或激活的调试）的迹象。
- 如果找到任何此类功能，尝试猜测并更改这些值，以获得不同的应用程序响应或行为。

## 修复

应用程序必须足够智能，并设计有业务逻辑，以防止攻击者预测和操纵参数以颠覆程序化或业务逻辑流，或利用隐藏/未记录的功能（如调试）。
