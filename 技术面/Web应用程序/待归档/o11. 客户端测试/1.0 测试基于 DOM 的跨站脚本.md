# 测试基于 DOM 的跨站脚本

|ID          |
|------------|
|WSTG-CLNT-01|

## 概述

[基于 DOM 的跨站脚本](https://owasp.org/www-community/attacks/DOM_Based_XSS) 是对 XSS 漏洞的事实上的称呼，这些漏洞是由于页面上的活动浏览器端内容（通常是 JavaScript）通过[源](https://github.com/wisec/domxsswiki/wiki/sources)获取用户输入，并在[接收器](https://github.com/wisec/domxsswiki/wiki/Sinks)中使用，从而导致注入代码的执行。本文档仅讨论导致 XSS 的 JavaScript 漏洞。

DOM，即[文档对象模型](https://en.wikipedia.org/wiki/Document_Object_Model)，是用于在浏览器中表示文档的结构化格式。DOM 使动态脚本（如 JavaScript）能够引用文档的组件，如表单字段或会话 Cookie。DOM 也被浏览器用于安全目的——例如限制不同域上的脚本获取其他域的会话 Cookie。当活动内容（如 JavaScript 函数）被特制请求修改，使得攻击者可以控制 DOM 元素时，就可能发生基于 DOM 的 XSS 漏洞。

并非所有 XSS 漏洞都需要攻击者控制从服务器返回的内容，攻击者反而可能滥用不良的 JavaScript 编码实践来达到相同的结果。其后果与典型的 XSS 缺陷相同，只是传递方式不同。

与其他类型的跨站脚本漏洞（[反射型和存储型](https://owasp.org/www-community/attacks/xss/)）相比，在反射型和存储型 XSS 中，未经过滤的参数由服务器传递然后返回给用户，并在用户浏览器的上下文中执行，而基于 DOM 的 XSS 漏洞则通过使用文档对象模型（DOM）的元素以及攻击者精心构造的代码来控制代码流程，从而改变执行流程。

由于它们的性质，基于 DOM 的 XSS 漏洞在许多情况下可以在服务器无法确定实际执行内容的情况下被执行。这可能使许多常规的 XSS 过滤和检测技术对此类攻击无效。

这个假设示例使用了以下客户端代码：

```html
<script>
document.write("站点位于：" + document.location.href + "。");
</script>
```

攻击者可以将 `#<script>alert('xss')</script>` 附加到受影响页面的 URL 上，当执行时，将显示警告框。在此实例中，附加的代码不会发送到服务器，因为 `#` 字符之后的所有内容不会被浏览器视为查询的一部分，而是视为片段。在此示例中，代码被立即执行，页面显示“xss”的警告。与更常见的跨站脚本类型（[反射型和存储型](https://owasp.org/www-community/attacks/xss/)）不同，在反射型和存储型 XSS 中，代码被发送到服务器然后再返回到浏览器，而此代码直接在用户浏览器中执行，无需与服务器联系。

基于 DOM 的 XSS 缺陷的[后果](https://owasp.org/www-community/attacks/xss/)与更广为人知的 XSS 形式一样广泛，包括 Cookie 检索、进一步的恶意脚本注入等，因此应同样严肃对待。

## 测试目标

- 识别 DOM 接收器。
- 构建针对每种接收器类型的载荷。

## 如何测试

JavaScript 应用程序与其他类型的应用程序有显著不同，因为它们通常由服务器动态生成。为了理解正在执行的代码，需要爬取被测网站，以确定所有正在执行的 JavaScript 实例以及接受用户输入的位置。许多网站依赖大型函数库，这些函数库通常包含数十万行代码，且并非内部开发。在这些情况下，自上而下的测试通常成为唯一可行的选择，因为许多底层函数从未被使用，而分析它们以确定哪些是接收器将消耗比通常可用时间更多的时间。如果一开始未识别出输入或其缺失，则对于自上而下的测试也是如此。

用户输入主要有两种形式：

- 由服务器写入页面的输入，其方式不允许直接 XSS，以及
- 从客户端 JavaScript 对象获取的输入。

以下是服务器可能将数据插入 JavaScript 的两个示例：

```js
var data = "<服务器转义的数据>";
var result = someFunction("<服务器转义的数据>");
```

以下是从客户端 JavaScript 对象获取输入的两个示例：

```js
var data = window.location;
var result = someFunction(window.referrer);
```

虽然在如何检索它们方面，JavaScript 代码几乎没有区别，但重要的是要注意，当输入通过服务器接收时，服务器可以对其应用任何所需的排列。另一方面，JavaScript 对象执行的排列是相当众所周知且有文档记录的。如果上述示例中的 `someFunction` 是一个接收器，则前者的可利用性将取决于服务器所做的过滤，而后者的可利用性将取决于浏览器对 `window.referrer` 对象所做的编码。Stefano Di Paulo 撰写了一篇关于当要求获取 URL 的各种元素时浏览器返回什么的优秀文章，该文章使用了 [document 和 location 属性](https://github.com/wisec/domxsswiki/wiki/location,-documentURI-and-URL-sources)。

此外，JavaScript 经常在 `<script>` 块之外执行，正如过去导致 XSS 过滤器绕过的许多向量所证明的那样。在爬取应用程序时，重要的是要注意在事件处理程序和具有表达式属性的 CSS 块等位置使用脚本。另外，请注意，任何外部 CSS 或脚本对象都需要进行评估，以确定正在执行什么代码。

自动化测试在识别和验证基于 DOM 的 XSS 方面只有非常有限的效果，因为它通常通过发送特定载荷并尝试在服务器响应中观察它来识别 XSS。这对于下面提供的简单示例可能效果很好，其中 message 参数被反射回用户：

```html
<script>
var pos=document.URL.indexOf("message=")+5;
document.write(document.URL.substring(pos,document.URL.length));
</script>
```

然而，在以下人为构造的案例中可能无法检测到：

```html
<script>
var navAgt = navigator.userAgent;

if (navAgt.indexOf("MSIE")!=-1) {
        document.write("您正在使用 IE 作为浏览器访问站点：" + document.location.href + "。");
}
else
{
    document.write("您正在使用未知浏览器。");
}
</script>
```

因此，除非测试工具能够对客户端代码执行额外的分析，否则自动化测试将无法检测到可能容易受到基于 DOM 的 XSS 攻击的区域。

因此，应进行手动测试，可以通过检查代码中引用可能对攻击者有用的参数的区域来完成。此类区域的示例包括动态写入页面的代码、修改 DOM 的位置，甚至直接执行脚本的位置。
