# 测试客户端 URL 重定向

|ID          |
|------------|
|WSTG-CLNT-04|

## 概述

本节介绍如何检查客户端 URL 重定向，也称为开放重定向。这是一种输入验证缺陷，当应用程序接受用户控制的输入（指定指向可能为恶意外部 URL 的链接）时存在。此类漏洞可用于完成钓鱼攻击或将受害者重定向到感染页面。

当应用程序接受包含 URL 值且未经清理的不可信输入时，就会发生此漏洞。此 URL 值可能导致 Web 应用程序将用户重定向到另一个页面，例如攻击者控制的恶意页面。

此漏洞可能使攻击者成功发起钓鱼骗局并窃取用户凭据。由于重定向源自真实应用程序，钓鱼尝试可能具有更可信的外观。

以下是一个钓鱼攻击 URL 的示例。

```text
https://www.target.site?#redirect=www.fake-target.site
```

访问此 URL 的受害者将被自动重定向到 `fake-target.site`，攻击者可以在那里放置一个模仿目标站点的虚假页面，以窃取受害者的凭据。

开放重定向也可用于构造一个 URL，绕过应用程序的访问控制检查，并将攻击者转发到他们通常无法访问的特权功能。

## 测试目标

- 识别处理 URL 或路径的注入点。
- 评估系统可能重定向到的位置。

## 如何测试

当测试人员手动检查此类漏洞时，他们首先识别客户端代码中是否实现了客户端重定向。这些重定向可以通过（以 JavaScript 为例）使用 `window.location` 对象来实现。这可以通过简单地将字符串赋值给它来将浏览器定向到另一个页面。以下代码片段演示了这一点：

```js
var redir = location.hash.substring(1);
if (redir) {
    window.location='https://'+decodeURIComponent(redir);
}
```

在此示例中，脚本未对变量 `redir` 进行任何验证，该变量通过查询字符串包含用户提供的输入。由于未应用任何形式的编码，此未经验证的输入被传递给 `windows.location` 对象，从而创建了 URL 重定向漏洞。

这意味着攻击者只需提交以下查询字符串，即可将受害者重定向到恶意站点：

```text
https://www.victim.site/?#www.malicious.site
```

通过略微修改，上述示例片段可能容易受到 JavaScript 注入攻击。

```js
var redir = location.hash.substring(1);
if (redir) {
    window.location=decodeURIComponent(redir);
}
```

这可以通过提交以下查询字符串来利用：

```text
https://www.victim.site/?#javascript:alert(document.cookie)
```

在测试此漏洞时，请注意某些字符在不同浏览器中的处理方式不同。有关参考，请参见[基于 DOM 的 XSS](https://owasp.org/www-community/attacks/DOM_Based_XSS)。
