# 测试 WebSockets

|ID          |
|------------|
|WSTG-CLNT-10|

## 概述

传统上，HTTP 协议每个 TCP 连接只允许一次请求/响应。异步 JavaScript 和 XML（AJAX）允许客户端异步（在后台无需刷新页面）向服务器发送和接收数据，然而，AJAX 要求客户端发起请求并等待服务器响应（半双工）。

[WebSockets](https://html.spec.whatwg.org/multipage/web-sockets.html#network) 允许客户端或服务器创建“全双工”（双向）通信通道，使客户端和服务器能够真正异步通信。WebSockets 通过 HTTP 进行初始的*升级*握手，此后所有通信都通过 TCP 通道使用帧进行传输。更多信息请参见 [WebSocket 协议](https://tools.ietf.org/html/rfc6455)。

### Origin

服务器有责任验证初始 HTTP WebSocket 握手过程中的 [`Origin` 头](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Origin)。如果服务器未验证初始 WebSocket 握手过程中的 origin 头，WebSocket 服务器可能接受来自任何来源的连接。这可能允许攻击者跨域与 WebSocket 服务器通信，从而导致类似 CSRF 的问题。另请参见 [Top 10-2017 A5-访问控制失效](https://owasp.org/www-project-top-ten/2017/A5_2017-Broken_Access_Control)。此弱点的利用称为跨站 WebSocket 劫持（CSWH 或 CSWSH）。

### 机密性和完整性

WebSockets 可以通过未加密的 TCP 或加密的 TLS 使用。使用未加密的 WebSockets 时，使用 `ws://` URI 方案（默认端口 80）；使用加密（TLS）WebSockets 时，使用 `wss://` URI 方案（默认端口 443）。另请参见 [Top 10-2017 A3-敏感数据暴露](https://owasp.org/www-project-top-ten/2017/A3_2017-Sensitive_Data_Exposure)。

### 输入清理

与任何来自不可信源的数据一样，应对数据进行适当的清理和编码。另请参见 [Top 10-2017 A1-注入](https://owasp.org/www-project-top-ten/2017/A1_2017-Injection) 和 [Top 10-2017 A7-跨站脚本（XSS）](https://owasp.org/www-project-top-ten/2017/A7_2017-Cross-Site_Scripting_(XSS))。

## 测试目标

- 识别 WebSockets 的使用。
- 通过对正常 HTTP 通道使用相同的测试来评估其实现。

## 如何测试

### 黑盒测试

1. 识别应用程序正在使用 WebSockets。
   - 检查客户端源代码中是否存在 `ws://` 或 `wss://` URI 方案。
   - 使用 Google Chrome 的开发者工具查看网络 WebSocket 通信。
   - 使用 [ZAP](https://www.zaproxy.org) 的 WebSocket 选项卡。
2. Origin。
   - 使用 WebSocket 客户端（可在下面的工具部分找到）尝试连接到远程 WebSocket 服务器。如果建立了连接，则服务器可能未检查 WebSocket 握手的 origin 头。
3. 机密性和完整性。
   - 检查 WebSocket 连接是否使用 TLS 传输敏感信息 `wss://`。
   - 检查 HTTPS 实现是否存在安全问题（有效证书、BEAST、CRIME、RC4 等）。请参阅本指南中的[测试弱传输层安全](../09-Testing_for_Weak_Cryptography/01-Testing_for_Weak_Transport_Layer_Security.md)部分。
4. 身份验证。
   - WebSockets 不处理身份验证，应执行常规的黑盒身份验证测试。请参阅本指南中的[身份验证测试](../04-Authentication_Testing/README.md)部分。
5. 授权。
   - WebSockets 不处理授权，应执行常规的黑盒授权测试。请参阅本指南中的[授权测试](../05-Authorization_Testing/README.md)部分。
6. 输入清理。
   - 使用 [ZAP](https://www.zaproxy.org) 的 WebSocket 选项卡重放和模糊测试 WebSocket 请求和响应。请参阅本指南中的[数据验证测试](../07-Input_Validation_Testing/README.md)部分。

### 灰盒测试

灰盒测试类似于黑盒测试。在灰盒测试中，渗透测试人员对应用程序有部分了解。唯一的区别是，您可能拥有被测应用程序的 API 文档，其中包含预期的 WebSocket 请求和响应。
