# API 侦察

|ID          |
|------------|
|WSTG-APIT-01|

## 概述

侦察是任何渗透测试活动中的重要步骤，API 渗透测试也不例外。通过收集有关 API 的信息并加深对目标的理解，侦察能显著提高测试过程的有效性。此阶段不仅增加了发现关键安全问题的可能性，还确保了对 API 安全态势的全面评估。

本指南中有关于[信息收集](../01-Information_Gathering/README.md)的章节，这些内容在审计 API 时同样适用。但两者存在一些差异。作为安全研究人员，我们通常关注特定领域，在本指南中查找适用章节可能比较耗时。为确保研究人员有一个集中关注 API 的单一位置，本节将重点放在那些适用于 API 的内容上，并提供指向指南中其他支持内容的引用。

### API 类型

API 可以是公共的或私有的。

#### 公共 API

公共 API 通常会在 Swagger/OpenAPI 文档中发布其详细信息。获取此文档对于了解攻击面非常重要。同样重要的是找到此文档的旧版本，这些旧版本可能显示已弃用但仍可运行的代码，这些代码可能存在安全漏洞。

请注意，此文档无论多么用心编写，都可能不准确，也可能未公开完整的 API。

公共 API 也可能在共享库或 API 目录中有文档记录。

#### 私有 API

私有 API 的可见性取决于预期的使用者。API 可以是私有的，但仅对订阅的客户端（也称为`合作伙伴`）可访问，或仅对内部客户端（如同公司的其他部门）可访问。使用侦察技术查找私有 API 也很重要。这些 API 可以通过多种技术发现，我们将在下面讨论。

## 测试目标

- 查找后端服务器代码支持的所有 API 端点，无论是有文档记录的还是未记录的。
- 查找每个端点支持的所有参数，无论是有文档记录的还是未记录的。
- 发现发送给客户端的 HTML 和 JavaScript 中与 API 相关的有趣数据。

## 如何测试

### 查找文档

无论是公共还是私有情况，API 文档的实用程度取决于其质量和准确性。公共 API 文档通常与所有人共享，而私有 API 文档仅与预期客户端共享。然而，在两种情况下，查找意外泄露或其他方式获得的文档都将有助于您的调查。

无论 API 的可见性如何，搜索 API 文档都可能找到较旧的、尚未发布或意外泄露的 API 文档。这些文档将非常有助于理解 API 暴露的攻击面。

### API 目录

API 文档的其他来源可能包括 API 目录，例如：

- GitHub 通用搜索
- [GitHub 公共 API 仓库](https://github.com/public-apis/public-apis)
- [APIs.guru](https://apis.guru)
- [RapidAPI](https://rapidapi.com/)
- [PublicAPIs](https://publicapis.dev/) 和 [PublicAPIs](https://publicapis.io/)
- [Postman API Network](https://www.postman.com/explore)

### 查找众所周知的位置

如果文档不明显，您可以根据一些显而易见的名称或路径主动搜索目标以查找文档。这些包括：

- /api-docs
- /doc
- /swagger
- /swagger.json
- /openapi.json
- /.well-known/schema-discovery

### Robots.txt

`robots.txt` 是一个文本文件，站点所有者创建它来指示网络爬虫（如搜索引擎机器人）如何抓取和索引其站点。它是 Robots 排除协议（REP）的一部分，该协议规定了机器人与站点交互的方式。

此文件可能提供有关路径结构或 API 端点的额外线索。

[信息收集](../01-Information_Gathering/README.md)部分在多个用例中提到了 robots.txt，包括 WSTG-INFO-01、WSTG-INFO-03、WSTG-INFO-05 和 WSTG-INFO-08。

### GitDorking

如果应用程序使用 GitHub、GitLab 或其他面向公共的基于 Git 的仓库，我们还可以搜索任何线索或敏感内容（也称为`GitDorking`）。这些信息可能包括密码、API 密钥、配置文件以及开发人员可能意外或无意中提交到其仓库的其他机密数据。组织可能意外共享敏感代码、示例或测试代码，这些代码可能提供实现细节的线索。目标员工的个人 GitHub 账户也可能意外泄露可能提供线索的信息。

### 浏览和爬取应用程序

即使您拥有 API 文档，浏览应用程序也是一个好主意。文档可能已过时、不准确或不完整。

使用诸如 ZAP 或 Burp Suite 等拦截代理浏览应用程序会记录端点以供后续检查。此外，使用其内置的爬取功能，拦截代理可以帮助生成全面的端点列表。从爬取的 URL 中，查找具有明显 API URL 命名方案的链接。这些包括：

- `https://example.com/api/v1`（或 v2 等）
- `https://example.com/graphql`

或者应用程序可能使用或依赖的子域：

- `https://api.example.com/api/v1`

渗透测试人员必须尽可能尝试执行应用程序中的尽可能多的功能。这不仅是为了生成全面的端点列表，也是为了避免延迟加载和代码拆分带来的问题。此外，您的渗透测试应包含不同权限级别的示例账户，以便您的浏览器和爬取可以访问并暴露尽可能多的功能的端点。

完成后，通过浏览和爬取应用程序获得的端点信息可以帮助渗透测试人员使用其他工具（如 Postman）编写目标的 API 文档。

### Google Dorking

使用诸如带有 `site` 和 `inurl` 等指令的 Google Dorking 等被动侦察技术，我们可以定制搜索 Google 索引器可能找到的常见 API 关键字。有关更多信息，请参阅[进行搜索引擎发现侦察以查找信息泄露](../01-Information_Gathering/01-Conduct_Search_Engine_Discovery_Reconnaissance_for_Information_Leakage.md)。

以下是一些 API 特定的示例：

`site:"mytargetsite.com" inurl:"/api"`

`inurl:apikey filetype:env`

其他关键字可能包括 `"v1"`、`"api"`、`"graphql"`。

我们可以将 Google Dorking 扩展到包括目标的子域。

单词列表对于获取 API 中使用的常见单词的全面列表非常有帮助。

### 回顾过去，回溯历史

一般来说，API 会随着时间而变化。但已弃用或较旧的版本可能仍然在运行，无论是出于有意还是配置错误。这些也应进行测试，因为它们很可能包含新版本已修复的漏洞。此外，API 的变化显示了新功能，这些功能可能不太健壮，因此是测试的良好候选对象。

为了发现旧版本，我们可以使用`Wayback Machine`来帮助查找旧端点。一个有用的工具称为 TomNomNom 的 [WayBackUrls](https://github.com/tomnomnom/waybackurls)，它获取 Wayback Machine 所知的域的所有 URL。

- [WayBackUrls](https://github.com/tomnomnom/waybackurls)。获取 Wayback Machine 所知的域的所有 URL。
- [waymore](https://github.com/xnl-h4ck3r/waymore)。从 Wayback Machine、Common Crawl、Alien Vault OTX、URLScan 和 VirusTotal 中查找更多信息。
- [gau](https://github.com/lc/gau)。从 AlienVault 的开放威胁交换、Wayback Machine 和 Common Crawl 获取已知 URL。

### 客户端应用程序

API 和其他信息的绝佳来源是服务器发送给客户端的 HTML 和 JavaScript。有时，客户端应用程序会泄露敏感信息，包括 API 和密钥。[审查网页内容以查找信息泄露](../01-Information_Gathering/05-Review_Web_Page_Content_for_Information_Leakage.md)部分提供了一些审查 Web 内容泄露的通用信息。在这里，我们将扩展重点放在审查 JavaScript 内容以查找 API 相关密钥上。

我们可以使用多种工具来帮助我们提取发送到浏览器的 JavaScript 中的敏感信息。这些工具通常基于两种方法之一：正则表达式或抽象语法树（AST）。然后是帮助我们组织或管理 JS 文件以便由 AST 和正则表达式工具进行调查的通用工具。

正则表达式更直接，通过搜索 JS 或 HTML 内容以查找已知模式。然而，这种方法可能会遗漏正则表达式中未明确标识的内容。鉴于某些 JS 的结构，这种方法可能会遗漏很多内容。另一方面，AST 是树状结构，表示源代码的语法。树中的每个节点对应于代码的一部分。对于 JavaScript，AST 将代码分解为基本组件，使工具和编译器能够轻松理解和修改代码。

#### 通用工具

1. [Uproot](https://github.com/0xDexter0us/uproot-JS)。一个 BurpSuite 插件，可将遇到的任何 JS 文件保存到磁盘。这有助于提取文件以供命令行工具进行分析。
2. [OpenAPI 支持](https://www.zaproxy.org/docs/desktop/addons/openapi-support/)。此 ZAP 附加组件允许您爬取和导入 OpenAPI（Swagger）定义，版本 1.2、2.0 和 3.0。
3. [OpenAPI Parser](https://github.com/aress31/openapi-parser)。一个 BurpSuite 插件，将 OpenAPI 文档解析到 Burp Suite 中，以自动化基于 OpenAPI 的 API 安全评估。

#### 正则表达式工具

1. [JSParser](https://github.com/nahamsec/JSParser)。一个使用 Tornado 和 JSBeautifier 从 JavaScript 文件中解析相对 URL 的 Python 2.7 脚本。
2. [JSMiner](https://github.com/PortSwigger/js-miner)。一个 BurpSuite 插件，尝试在静态文件中查找有趣的内容；主要是 JavaScript 和 JSON 文件。此工具在爬取应用程序时“被动”扫描。
3. [JSpector](https://github.com/hisxo/JSpector)。一个 BurpSuite 插件，被动爬取 JavaScript 文件，并自动创建包含 JS 文件中找到的 URL、端点和危险方法的问题。
4. [Link Finder](https://github.com/GerbenJavado/LinkFinder)。一个在 JavaScript 文件中查找端点的 Python 脚本。

#### AST 工具

1. [JSLuice](https://github.com/BishopFox/jsluice)。一个命令行工具，从 JavaScript 源代码中提取 URL、路径、密钥和其他有趣数据。

### 其他侦察工具

1. [Attack Surface Detector](https://github.com/secdec/attack-surface-detector-burp)。一个 BurpSuite 插件，通过解析路由和识别参数，使用静态代码分析来识别 Web 应用程序端点。
2. [Param Miner](https://github.com/portswigger/param-miner)。一个 BurpSuite 插件，识别隐藏的、未链接的参数。
3. [xnLinkFinder](https://github.com/xnl-h4ck3r/xnLinkFinder)。一个 Python 工具，用于发现端点、潜在参数以及针对给定目标的目标特定单词列表。
4. [GAP](https://github.com/xnl-h4ck3r/GAP-Burp-Extension)。Burp 扩展，用于查找潜在端点、参数并生成自定义目标单词列表。

### 主动模糊测试

主动模糊测试涉及使用带有单词列表和过滤请求结果的工具，以暴力破解端点发现。

#### Kiterunner

[KiteRunner](https://github.com/assetnote/kiterunner) 是一个工具，用于在现代应用程序和 API 中执行传统的内容发现和暴力破解路由/端点。

```console
kr [scan|brute] <input> [flags]
```

要使用单词列表扫描目标以查找 API，我们可以：

```console
kr scan https://example.com/api -w /usr/share/wordlists/apis/routes-large.kite --fail-status-codes 404,403
```

#### FFUF/DirBuster/GoBuster

FFUF、DirBuster 和 GoBuster 三者都旨在通过暴力破解技术发现 Web 服务器上的隐藏路径和文件。三者都使用可定制的单词列表向目标 Web 服务器生成请求，尝试识别有效的目录和文件。三者都支持多线程或高效处理以加速暴力破解过程。

API 的一些常见单词列表文件包括：[SecLists](https://github.com/danielmiessler/SecLists) 中的 Discovery/Web-Content/api 部分、[GraphQL Wordlist](https://github.com/Escape-Technologies/graphql-wordlist) 和 [Assetnote](https://wordlists.assetnote.io/)。

GoBuster 示例：

`gobuster dir -u <target url> -w <wordlist file>`
