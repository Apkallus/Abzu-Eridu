# 测试绕过身份验证模式

|ID          |
|------------|
|WSTG-ATHN-04|

## 概述

在计算机安全中，身份验证是尝试验证通信发送方数字身份的过程。此类过程的一个常见示例是登录过程。测试身份验证模式意味着理解身份验证过程的工作原理，并利用该信息绕过身份验证机制。

虽然大多数应用程序需要身份验证才能访问私有信息或执行任务，但并非每种身份验证方法都能提供足够的安全性。疏忽、无知或对安全威胁的轻视往往导致身份验证方案可通过跳过登录页面并直接调用本应在身份验证后访问的内部页面而被绕过。

此外，通常可以通过篡改请求并诱使应用程序认为用户已经通过身份验证来绕过身份验证措施。这可以通过修改给定的 URL 参数、操纵表单或伪造会话来达成。

与身份验证模式相关的问题可能出现在软件开发生命周期（SDLC）的不同阶段，如设计、开发和部署阶段：

- 在设计阶段，错误可能包括对要保护的应用程序部分的错误定义、选择不使用强加密协议来保护凭据传输等。
- 在开发阶段，错误可能包括输入验证功能的错误实现，或未遵循特定语言的安全最佳实践。
- 在应用程序部署阶段，由于缺乏所需技术技能或缺乏良好文档，应用程序设置（安装和配置活动）期间可能出现问题。

## 测试目标

- 确保身份验证应用于所有需要它的服务。

## 如何测试

有几种方法可以绕过 Web 应用程序使用的身份验证模式：

- 参数修改
- 会话 ID 预测
- SQL 注入

### 参数修改

与身份验证设计相关的另一个问题是当应用程序基于固定值参数验证成功登录时。用户可以修改这些参数以访问受保护区域，而无需提供有效凭据。在下面的示例中，“authenticated”参数被更改为“yes”，允许用户获得访问权限。在此示例中，参数位于 URL 中，但也可以使用代理来修改参数，尤其是当参数作为 POST 请求中的表单元素发送或参数存储在 cookie 中时。

```html
https://www.site.com/page.asp?authenticated=no

raven@blackbox /home $nc www.site.com 80
GET /page.asp?authenticated=yes HTTP/1.0

HTTP/1.1 200 OK
Date: Sat, 11 Nov 2006 10:22:44 GMT
Server: Apache
Connection: close
Content-Type: text/html; charset=iso-8859-1

<!DOCTYPE HTML PUBLIC "-//IETF//DTD HTML 2.0//EN">
<HTML><HEAD>
</HEAD><BODY>
<H1>您已通过身份验证</H1>
</BODY></HTML>
```

### PHP 松散比较

如果攻击者能够通过利用先前发现的漏洞（例如目录遍历）或从 Web 仓库（开源应用程序）检索到应用程序源代码，则可能对身份验证过程的实现进行精细攻击。

在以下示例（PHPBB 2.0.12 - 身份验证绕过漏洞）中，第 2 行的 `unserialize()` 函数解析用户提供的 cookie，并在 `$sessiondata` 数组中设置值。在第 7 行，后端数据库（`$auto_login_key`）中存储的用户 MD5 密码哈希与用户提供的（`$sessiondata['autologinid']`）进行比较。

```php
1. if (isset($HTTP_COOKIE_VARS[$cookiename . '_sid'])) {
2.     $sessiondata = isset($HTTP_COOKIE_VARS[$cookiename . '_data']) ? unserialize(stripslashes($HTTP_COOKIE_VARS[$cookiename . '_data'])) : array();
3.     $sessionmethod = SESSION_METHOD_COOKIE;
4. }
5. $auto_login_key = $userdata['user_password'];
6. // 我们必须自动登录
7. if( $sessiondata['autologinid'] == $auto_login_key )
8. {
9.     // autologinid 匹配密码
10.     $login = 1;
11.     $enable_autologin = 1;
12. }

```

在 PHP 中，字符串值与 `true` 布尔值进行比较的结果始终为 `true`（因为字符串包含值），因此通过向 `unserialize()` 函数提供以下字符串，可以绕过身份验证控制并以管理员身份登录，其 `userid` 为 2：

```php
a:2:{s:11:"autologinid";b:1;s:6:"userid";s:1:"2";}  // 原始值：a:2:{s:11:"autologinid";s:32:"8b8e9715d12e4ca12c4c3eb4865aaf6a";s:6:"userid";s:4:"1337";}
```

让我们分解此字符串中的操作：

1. `autologinid` 现在是一个布尔值，设置为 `true`：这一点可以通过将密码哈希的 MD5 值（`s:32:"8b8e9715d12e4ca12c4c3eb4865aaf6a"`）替换为 `b:1` 来体现。
2. `userid` 现在设置为管理员 ID：这一点可以在字符串的最后部分看到，我们将常规用户 ID（`s:4:"1337"`）替换为 `s:1:"2"`。

### PHP 魔术哈希

当哈希值进行松散比较并最终被评估为数字时，PHP 中也会出现类似的问题。

PHP 支持对数字使用[科学记数法](https://en.wikipedia.org/wiki/Scientific_notation)，因此数字 `2e4` 被视为“2 乘以 10 的 4 次方”，等于 20,000。类似地，数字 `0e4` 则为“0 乘以 10 的 4 次方”，等于零。因为零乘以任何数都为零，这意味着像 `0e2 == 0e3` 这样的比较结果为 true，因为两边都等于零。

当比较两个密码哈希，且两个哈希值都采用一串零、字母“e”和一串数字的形式时，这就可以被利用。考虑以下代码：

```php
<?php
{
    if (md5('240610708') == md5('QLTHNDT')) {
        print("密码匹配");
    } else {
        print("密码不同");
    }
}
```

这两个密码显然不同，但由于它们都哈希为科学记数法中等于零的字符串（分别为 "0e462097431906509019562988736854" 和 "0e405967825401955372549139051580"），比较将返回 true，用户将成功登录。

为了测试这一点，首先需要确定应用程序使用的哈希类型。然后可以在用户账户上设置一个能生成此类哈希的密码，然后尝试使用另一个能生成不同哈希但同样评估为零的密码登录。各种密码哈希的字符串可以在 [spaze/hashes](https://github.com/spaze/hashes) GitHub 仓库中找到。

如果成功，这不仅意味着应用程序正在不安全地比较密码哈希，而且还表明它们未对密码加盐，并且很可能使用了遗留的哈希算法。

### 相同长度的密码有效

偶尔，应用程序会允许用户通过输入与真实密码长度相同的任意密码来登录。这是一个相当不常见的问题，暗示密码存储和处理方式存在重大缺陷。

此问题的一个可能原因是应用程序不同部分之间的字符编码问题，因为未知或无效字符通常用 `?` 字符表示，而某些语言（如 SQL）将其视为单字符通配符。因此，像 `password` 和 `????????` 这样的字符串的比较可能会返回 true，如果通配符被解释，例如在 SQL `LIKE` 比较中。

> 注意：这也意味着密码要么以明文存储，要么使用可逆加密。

测试很简单：尝试使用一个不正确但长度与有效密码相同的密码登录账户。
