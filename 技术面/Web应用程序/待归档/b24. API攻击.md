# 第24章：API黑客攻击（API Hacking）

应用程序编程接口（API）是程序之间相互通信的一种方式，它们为各种各样的应用程序提供支持。随着应用程序变得越来越复杂，开发人员越来越多地使用API来组合应用程序的组件或属于同一组织的多个应用程序。而且，API越来越多地具有执行重要操作或传输敏感信息的能力。

在本章中，我们将讨论什么是API、它们如何工作，以及如何发现和利用API漏洞。

## 什么是API？

简单来说，API是一组允许一个应用程序与另一个应用程序通信的规则。它们使应用程序能够以受控方式共享数据。通过使用API，互联网上的应用程序可以利用其他应用程序的资源来构建更复杂的功能。

例如，考虑Twitter的API（https://developer.twitter.com/en/docs/twitter-api/）。这个公共API允许外部开发人员访问Twitter的数据和操作。例如，如果开发人员希望他们的代码从Twitter的数据库中检索推文内容，他们可以使用一个返回推文信息的Twitter API端点，向位于 api.twitter.com 的Twitter API服务器发送GET请求：

```
GET /1.1/statuses/show.json?id=210462857140252672
Host: api.twitter.com
```

此URL表示开发人员正在使用Twitter的API版本1.1，并请求ID为210462857140252672的名为statuses的资源（这是Twitter对其推文的称呼）。URL中的`id`字段是API端点所需的请求参数。API端点通常需要某些参数来确定返回哪个资源。

然后Twitter的API服务器将以JSON格式将数据返回给请求应用程序（此示例取自Twitter的公共API文档）：

```json
① {
   ② "created_at": "Wed Oct 10 20:19:24 +0000 2018",
    "id": 1050118621198921728,
    "id_str": "1050118621198921728",
    "text": "To make room for more expression, we will now count all emojis as equal—including those with gender... and skin t... https://t.co/MkGjXf9aXm",
    "truncated": true,
    "entities": {
       ③ "hashtags": [],
        "symbols": [],
        "user_mentions": [],
        "urls": [
            {
                "url": "https://t.co/MkGjXf9aXm",
                "expanded_url": "https://twitter.com/i/web/status/1050118621198921728",
                "display_url": "twitter.com/i/web/status/1...",
                "indices": [
                    117,
                    140
                ]
            }
        ]
    },
    ④ "user": {
        "id": 6253282,
        "id_str": "6253282",
        "name": "Twitter API",
        "screen_name": "TwitterAPI",
        "location": "San Francisco, CA",
        "description": "The Real Twitter API. Tweets about API changes, service issues and our Developer Platform. Don't get an answer? It's on my website.",
        [...]
    }
}
```

API通常以JSON或XML格式返回数据。JSON是一种以纯文本表示数据的方式，常用于在Web消息中传输数据。在测试应用程序时，你经常会看到JSON消息，因此学习如何阅读它们会很有帮助。

JSON对象以花括号开始和结束①。在这些花括号内，所表示对象的属性以键值对的形式存储。例如，在代表推文的前述数据块中，`created_at`属性的值为`Wed Oct 10 20:19:24 +0000 2018`。这表示推文创建于2018年10月10日星期三晚上8:19②。

JSON对象还可以包含列表或其他对象。花括号表示对象。前述推文包含一个`user`对象，表示创建推文的用户④。列表用方括号表示。Twitter在前述JSON块中返回了一个空的hashtags列表，这意味着推文中没有使用hashtag③。

你可能会想知道API服务器如何决定谁可以访问数据或执行操作。API通常要求用户在使用其服务之前进行身份验证。通常，用户在其API请求中包含访问令牌以证明其身份。其他时候，用户需要使用特殊的身份验证头或Cookie。然后服务器将使用请求中提供的凭据来确定用户应访问哪些资源和操作。

## REST API

有多种类型的API。这里讨论的Twitter API被称为**表述性状态转移（REST）API**。REST是最常用的API结构之一。大多数情况下，REST API以JSON或纯文本格式返回数据。REST API用户向特定资源端点发送请求以访问该资源。在Twitter的案例中，你向 `https://api.twitter.com/1.1/statuses/show/` 发送GET请求以检索推文信息，向 `https://api.twitter.com/1.1/users/show/` 发送GET请求以检索用户信息。

REST API通常具有定义的查询结构，使用户易于预测他们应向其发送请求的特定端点。例如，要通过Twitter API删除推文，用户可以向 `https://api.twitter.com/1.1/statuses/destroy/` 发送POST请求；要转推推文，用户可以向 `https://api.twitter.com/1.1/statuses/retweet/` 发送POST请求。你可以看到，Twitter的所有API端点都以相同的方式结构化（`https://api.twitter.com/1.1/RESOURCE/ACTION`）：

```
https://api.twitter.com/1.1/users/show
https://api.twitter.com/1.1/statuses/show
https://api.twitter.com/1.1/statuses/destroy
https://api.twitter.com/1.1/statuses/retweet
```

REST API还可以使用各种HTTP方法。例如，GET通常用于检索资源，POST用于更新或创建资源，PUT用于更新资源，DELETE用于删除资源。

## SOAP API

SOAP是一种在现代应用程序中较少使用的API架构。但许多旧应用和物联网应用仍在使用SOAP API。SOAP API使用XML传输数据，其消息具有头和体。一个简单的SOAP请求如下所示：

```
DELETE / HTTPS/1.1
Host: example.s3.amazonaws.com
<DeleteBucket xmlns="http://doc.s3.amazonaws.com/2006-03-01">
    <Bucket>quotes</Bucket>
    <AWSAccessKeyId>AKIAIOSFODNN7EXAMPLE</AWSAccessKeyId>
    <Timestamp>2006-03-01T12:00:00.183Z</Timestamp>
    <Signature>Iuyz3d3P0aTou39dzbqaEXAMPLE=</Signature>
</DeleteBucket>
```

此示例请求取自Amazon S3的SOAP API文档。它删除一个名为`quotes`的S3存储桶。如你所见，API请求参数作为XML文档中的标签传递给服务器。

SOAP响应如下所示：

```xml
<DeleteBucketResponse xmlns="http://s3.amazonaws.com/doc/2006-03-01">
    <DeleteBucketResponse>
        <Code>204</Code>
        <Description>No Content</Description>
    </DeleteBucketResponse>
</DeleteBucketResponse>
```

此响应表示存储桶已成功删除，不再存在。

SOAP API有一个称为**Web服务描述语言（WSDL）**的服务，用于描述API的结构以及如何访问它。如果你能找到SOAP API的WSDL，你可以在攻击之前使用它来理解API。你通常可以通过在API端点末尾添加`.wsdl`或`?wsdl`，或搜索包含术语`wsdl`的URL端点来找到WSDL文件。在WSDL中，你将能够找到一系列可测试的API端点。

## GraphQL API

GraphQL是一种较新的API技术，允许开发人员请求他们需要的精确资源字段，并仅通过单个API调用获取多个资源。由于这些优点，GraphQL正变得越来越普遍。

GraphQL API使用自定义查询语言，并为API的所有功能使用单个端点。这些端点通常位于`/graphql`、`/gql`或`/g`。GraphQL有两种主要操作类型：**查询（queries）**和**变更（mutations）**。查询获取数据，就像REST API中的GET请求一样。变更创建、更新和删除数据，就像REST API中的POST、PUT和DELETE请求一样。

例如，查看以下对Shopify GraphQL API的API请求。Shopify是一个电子商务平台，允许用户通过GraphQL API与其在线商店交互。要访问Shopify的GraphQL API，开发人员需要向端点 `https://SHOPNAME.myshopify.com/admin/api/API_VERSION/graphql.json` 发送POST请求，并在POST请求体中包含GraphQL查询。要检索有关你商店的信息，你可以发送此请求：

```graphql
query {
    shop {
        name
        primaryDomain {
            url
            host
        }
    }
}
```

此GraphQL查询表示我们希望检索商店的`name`和`primaryDomain`，并且我们只需要`primaryDomain`的`url`和`host`属性。

Shopify的服务器将以JSON格式返回请求的信息：

```json
{
    "data": {
        "shop": {
            "name": "example",
            "primaryDomain": {
                "url": "https://example.myshopify.com",
                "host": "example.myshopify.com"
            }
        }
    }
}
```

注意，响应不包含对象的所有字段，而是用户请求的精确字段。根据你的需求，你可以请求同一数据对象的更多或更少字段。以下是一个请求较少字段的示例：

```graphql
query {
    shop {
        name
    }
}
```

你也可以请求资源属性的精确子字段和其他嵌套属性。例如，这里你只请求商店的`primaryDomain`的`url`：

```graphql
query {
    shop {
        primaryDomain {
            url
        }
    }
}
```

这些查询都用于检索数据。

用于编辑数据的**变更**可以有参数和返回值。让我们看一个来自graphql.org的变更示例。此变更创建一个新的客户记录，并接受三个输入参数：`firstName`、`lastName`和`email`。然后它返回新创建客户的ID：

```graphql
mutation {
    customerCreate(
        input: {
            firstName: "John",
            lastName: "Tate",
            email: "john@johns-apparel.com"
        }
    ) {
        customer {
            id
        }
    }
}
```

GraphQL独特的语法起初可能使测试变得困难，但一旦你理解了它，你就可以像测试其他类型的API一样测试这些API。要了解更多关于GraphQL语法的信息，请访问 https://graphql.org/。

GraphQL API还为漏洞猎人包含了一个很好的侦察工具：一个称为**自省（introspection）**的功能，允许API用户向GraphQL系统询问关于自身的信息。换句话说，它们是返回关于如何使用API的信息的查询。例如，`__schema`是一个特殊字段，将返回API中所有可用的类型；以下查询将返回系统中所有类型名称。你可以使用它来查找你可以查询的数据类型：

```graphql
{
    __schema {
        types {
            name
        }
    }
}
```

你也可以使用`__type`查询来查找特定类型的关联字段：

```graphql
{
    __type(name: "customer") {
        name
        fields {
            name
        }
    }
}
```

你将得到如下返回的类型字段。然后你可以使用这些信息来查询API：

```json
{
    "data": {
        "__type": {
            "name": "customer",
            "fields": [
                {
                    "name": "id"
                },
                {
                    "name": "firstName"
                },
                {
                    "name": "lastName"
                },
                {
                    "name": "email"
                }
            ]
        }
    }
}
```

自省功能使API黑客的侦察工作变得轻而易举。为了防止恶意攻击者枚举其API，许多组织在其GraphQL API中禁用了自省功能。

## API中心应用

API越来越多地不仅被用作与外部开发人员共享数据的机制。你还会遇到**API中心应用**，即使用API构建的应用程序。API中心应用不是从服务器检索完整的HTML文档，而是由客户端组件组成，该组件通过API调用从服务器请求和渲染数据。

例如，当用户查看Facebook帖子时，Facebook的移动应用程序使用API调用从服务器检索有关帖子的数据，而不是检索包含嵌入数据的完整HTML文档。然后应用程序在客户端渲染数据以形成网页。

许多移动应用程序都是以这种方式构建的。当公司已经拥有Web应用时，使用API中心方法构建移动应用可以节省时间。API允许开发人员分离应用程序的渲染和数据传输任务：开发人员可以使用API调用来传输数据，然后为移动端构建单独的渲染机制，而不是重新实现相同的功能。

然而，API中心应用的兴起意味着公司和应用程序通过API暴露越来越多的数据和功能。API经常泄露敏感数据和托管应用程序的应用程序逻辑。正如你将看到的，这使得API漏洞成为安全漏洞的广泛来源，也是漏洞猎人的丰硕目标。

## 寻找API漏洞

让我们探讨一些影响API的漏洞以及你可以采取的发现它们的步骤。API漏洞类似于影响非API Web应用程序的漏洞，因此请确保你对我们迄今为止讨论的漏洞有很好的理解。话虽如此，在测试API时，你应该将测试重点放在本节列出的漏洞上，因为它们在API实现中很普遍。

在我们深入之前，有许多开源API开发和测试工具可以帮助你提高API测试过程的效率。Postman（https://www.postman.com/）是一个方便的工具，可以帮助你测试API。你可以使用Postman从头开始构建复杂的API请求，并管理你将发送的大量测试请求。GraphQL Playground（https://github.com/graphql/graphql-playground/）是一个用于构建GraphQL查询的IDE，具有自动补全和错误高亮功能。ZAP有一个GraphQL插件（https://www.zaproxy.org/blog/2020-08-28-introducing-the-graphql-add-on-for-zap/），可以自动化GraphQL自省和测试查询生成。Clairvoyance（https://github.com/nikitastupin/clairvoyance/）帮助你在自省被禁用时深入了解GraphQL API的结构。

### 执行侦察

首先，寻找API漏洞与在常规Web应用程序中寻找漏洞非常相似，它需要侦察。API测试最困难的方面是了解应用程序的预期，然后定制载荷以操纵其功能。

如果你正在攻击GraphQL API，你可以从发送自省查询开始，以了解API的结构。如果你正在测试SOAP API，从查找WSDL文件开始。如果你正在攻击REST或SOAP API，或者如果你正在攻击的GraphQL API上禁用了自省，从枚举API开始。API枚举是指识别尽可能多的API端点的过程，以便你可以测试尽可能多的端点。

要枚举API，首先阅读API的公共文档（如果有的话）。拥有公共API的公司通常会发布关于API端点及其参数的详细文档。你应该能够通过在网上搜索`company_name API`或`company_name developer docs`来找到公共API文档。此文档为枚举API端点提供了一个良好的起点，但不要被愚弄，以为官方文档包含所有你可以测试的端点！API通常有公共端点和私有端点，只有公共端点会在这些开发者指南中找到。

尝试使用Swagger（https://swagger.io/），这是一个开发人员用于开发API的工具包。Swagger包含一个用于生成和维护API文档的工具，开发人员通常使用它在内部记录API。有时公司不会公开发布其API文档，但忘记锁定托管在Swagger上的内部文档。在这种情况下，你可以通过在网上搜索`company_name inurl:swagger`来找到文档。此文档通常包括所有API端点、其输入参数和示例响应。

接下来你可以做的是浏览所有应用程序工作流以捕获API调用。你可以通过使用拦截代理在后台记录HTTP流量来浏览公司的应用程序来做到这一点。你可能会发现应用程序工作流中使用的API调用不在公共文档中。

使用你找到的端点，你可以尝试推断其他端点。例如，REST API通常具有可预测的结构，因此你可以通过研究现有端点来推断新端点。如果`/posts/POST_ID/read`和`/posts/POST_ID/delete`都存在，是否存在一个名为`/posts/POST_ID/edit`的端点？类似地，如果你发现博客文章位于`/posts/1234`和`/posts/1236`，`/posts/1235`也存在吗？

接下来，使用第5章中的侦察技术搜索其他API端点，如研究JavaScript源代码或公司的公共GitHub仓库。你也可以尝试生成错误消息，希望API泄露关于自身的信息。例如，尝试向API端点提供意外的数据类型或格式错误的JSON代码。

模糊测试技术也可以通过使用单词列表帮助你找到额外的API端点。许多在线单词列表是为模糊测试API端点定制的；一个示例单词列表在https://gist.github.com/yassineaboukir/8e12adefbd505ef704674ad6ad48743d/。我们将在第25章详细讨论如何模糊测试端点。

还要注意，API经常更新。虽然应用程序可能不主动使用较旧版本的API，但这些版本可能仍然会从服务器获得响应。对于你在较新版本API中找到的每个端点，你应该测试较旧版本的端点是否有效。例如，如果`/api/v2/user_emails/52603991338963203244`端点存在，那么`/api/v1/user_emails/52603991338963203244`是否存在？API的旧版本通常包含已在新版本中修复的漏洞，因此请确保将查找旧API端点包含在你的侦察策略中。

最后，花时间理解每个API端点的功能、参数和查询结构。你对API的工作原理了解得越多，你就越能理解如何攻击它。识别所有可能的用户数据输入位置以供将来测试。注意任何身份验证机制，包括：

- 需要什么访问令牌？
- 哪些端点需要令牌，哪些不需要？
- 访问令牌如何生成？
- 用户是否可以在不登录的情况下使用API生成有效令牌？
- 更新或重置密码时，访问令牌会过期吗？

在整个侦察过程中，确保做大量笔记。记录你找到的端点及其参数。

### 测试访问控制失效和信息泄露

侦察之后，我喜欢从测试访问控制问题和信息泄露开始。大多数API使用访问令牌来确定客户端的权限；它们向每个API客户端颁发访问令牌，客户端使用这些令牌执行操作或检索数据。如果这些API令牌没有正确颁发和验证，攻击者可能会绕过身份验证并非法访问数据。

例如，有时API令牌在服务器接收后未被验证。其他时候，API令牌不是随机生成的，可以被预测。最后，一些API令牌不会定期失效，因此窃取令牌的攻击者可以无限期地保持对系统的访问。

另一个问题是资源或功能级别的访问控制失效。有时API端点没有与主应用程序相同的访问控制机制。例如，假设具有有效API密钥的用户可以检索关于自己的数据。他们是否也可以读取关于其他用户的数据？或者他们是否可以通过API代表他人执行操作？最后，没有管理员权限的普通用户是否可以从仅限于管理员的端点读取数据？与REST或SOAP API分开，应用程序的GraphQL API可能有其自己的授权机制和配置。这意味着即使应用程序的Web或REST API是安全的，你也可以在GraphQL端点上测试访问控制问题。这些问题类似于第10章讨论的IDOR漏洞。

其他时候，API提供多种方式来执行相同的操作，但并非所有方式都实施了访问控制。例如，假设一个REST API有两种删除博客文章的方式：向`/posts/POST_ID/delete`发送POST请求和向`/posts/POST_ID`发送DELETE请求。你应该问自己：这两个端点是否受到相同的访问控制？

另一个常见的API漏洞是信息泄露。API端点经常返回比应该返回的更多信息，或比渲染网页所需的更多信息。例如，我曾经发现一个填充用户个人资料页面的API端点。当我访问其他人的个人资料页面时，一个API调用被用来返回个人资料所有者的信息。乍一看，个人资料页面没有泄露任何敏感信息，但用于获取用户数据的API响应实际上也返回了个人资料所有者的私有API令牌！攻击者通过访问受害者的个人资料页面窃取其API令牌后，可以使用此访问令牌冒充受害者。

列出应该受到某种形式访问控制限制的端点。对于每个端点，创建两个具有不同权限级别的用户账户：一个应有权访问该功能，一个不应。测试你是否可以使用较低权限的账户访问受限功能。

如果你的较低权限用户无法访问受限功能，尝试移除访问令牌，或向API调用添加额外参数，如Cookie `admin=1`。你也可以切换HTTP请求方法，包括GET、POST、PUT、PATCH和DELETE，以查看是否在所有方法中正确实施了访问控制。例如，如果你无法通过向API端点发送POST请求来编辑其他用户的博客文章，你能通过改用PUT请求来绕过保护吗？

尝试通过切换API调用中的用户ID或其他用户标识参数来查看、修改和删除其他用户的信息。如果用于识别用户和资源的ID不可预测，尝试通过其他端点的信息泄露来泄露ID。例如，我曾经发现一个返回用户信息的API端点；它揭示了用户的ID以及用户所有朋友的ID。有了用户和他们朋友的ID，我能够访问两个用户之间发送的消息。通过结合两个信息泄露并仅使用用户ID，我能够读取用户的私信！

在GraphQL中，一个常见错误配置是允许较低权限的用户通过变更请求修改他们不应修改的数据。尝试捕获从一个用户账户允许的GraphQL查询，看看你是否可以从另一个不应有权限的账户发送相同的查询并获得相同的结果。

在寻找访问控制问题时，仔细研究服务器返回的数据。不要只看生成的HTML页面；深入查看原始API响应，因为API经常返回不在网页上显示的数据。你可能能够在响应体中发现敏感信息泄露。API端点是否返回任何私人用户信息或关于组织的敏感信息？返回的信息是否应对当前用户可用？返回的信息是否对公司构成安全风险？

### 测试速率限制问题

API经常缺乏速率限制；换句话说，API服务器不限制客户端或用户账户在短时间内可以发送的请求数量。缺乏速率限制本身是一个低严重性漏洞，除非被证明可以被攻击者利用。但在关键端点上，缺乏速率限制意味着恶意用户可以向服务器发送大量请求以收集数据库信息或暴力破解凭据。

在没有速率限制时可能危险的端点包括身份验证端点、不受访问控制保护的端点以及返回大量敏感数据的端点。例如，我曾经遇到一个允许用户通过邮件ID检索其邮件的API端点，如下所示：

```
GET /api/v2/user_emails/52603991338963203244
```

此端点不受任何访问控制保护。由于此端点也没有速率限制，攻击者基本上可以通过发送大量请求来猜测邮件ID字段。一旦他们猜到了一个有效ID，他们就可以访问其他用户的私有邮件。

要测试速率限制问题，向端点发送大量请求。你可以使用Burp Intruder或curl在短时间内发送100到200个请求。确保在不同的身份验证阶段重复测试，因为具有不同权限级别的用户可能受到不同的速率限制。

在测试速率限制问题时务必非常小心，因为非常可能意外地通过使请求淹没应用程序来对其发起DoS攻击。在进行速率限制测试之前，你应该获得书面许可，并根据公司的政策对请求进行时间限制。

还要记住，应用程序的速率限制可能高于你的测试工具的能力。例如，应用程序可能设置每秒400个请求的速率限制，而你的工具可能无法达到该限制。

### 测试技术性漏洞

到目前为止，我们在本书中讨论的许多漏洞——如SQL注入、反序列化问题、XXE、模板注入、SSRF和RCE——都是由不正确的输入验证引起的。有时开发人员忘记为API实现适当的输入验证机制。

因此，API也容易受到影响常规Web应用程序的许多其他漏洞的攻击。由于API是应用程序接受用户输入的另一种方式，它们成为攻击者将恶意输入潜入应用程序工作流的另一种方式。

如果API端点可以访问外部URL，它可能容易受到SSRF攻击，因此你应该检查其对内部URL的访问是否不受限制。

竞争条件也可能发生在API中。如果你可以使用API端点访问受竞争条件影响的应用程序功能，这些端点可能成为触发竞争条件的替代方式。

其他漏洞，如路径遍历、文件包含、不安全的反序列化问题、XXE和XSS也可能发生。如果API端点通过文件路径返回内部资源，攻击者可能使用该端点读取存储在服务器上的敏感文件。如果用于文件上传的API端点不限制用户可以上传的数据类型，攻击者可能将恶意文件（如Web shell或其他恶意软件）上传到服务器。API也经常接受序列化格式（如XML）的用户输入。在这种情况下，可能发生不安全的反序列化或XXE。通过文件上传或XXE的RCE在API端点中很常见。最后，如果API的URL参数在响应中被反射，攻击者可以使用该API端点在受害者浏览器上触发反射型XSS。

测试这些问题的过程将类似于在常规Web应用中测试它们。你只需以API形式向应用程序提供载荷。

例如，对于路径遍历和文件包含攻击等漏洞，注意API端点中的绝对和相对文件路径，并尝试干扰路径参数。如果API端点接受XML输入，尝试向请求中插入XXE载荷。如果端点的URL参数在响应中被反射，看看你是否可以通过在URL中放置载荷来触发反射型XSS。

你也可以利用模糊测试技术（我们将在第25章讨论）来发现这些漏洞。

应用程序越来越依赖API，即使API并不总是像其Web应用对应物一样受到良好保护。注意你目标使用的API，你可能会发现主应用程序中不存在的问题。如果你有兴趣了解更多关于攻击API和Web应用程序的内容，OWASP Web安全测试指南（https://github.com/OWASP/wstg/）是一个很好的学习资源。
