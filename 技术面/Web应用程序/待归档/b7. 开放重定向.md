# 第7章：开放重定向（Open Redirects）

网站经常使用 HTTP 或 URL 参数在无需用户操作的情况下将用户重定向到指定 URL。虽然这种行为可能很有用，但也可能导致开放重定向——即攻击者能够操纵该参数的值，将用户重定向到站外。让我们讨论这个常见漏洞，为什么它是个问题，以及如何利用它来升级你发现的其他漏洞。

## 机制

网站经常需要自动重定向用户。例如，当未经身份验证的用户尝试访问需要登录的页面时，这种情况很常见。网站通常会将那些用户重定向到登录页面，然后在认证完成后将其返回原始位置。

例如，当用户访问其账户仪表板 `https://example.com/dashboard` 时，应用程序可能会将其重定向到登录页面 `https://example.com/login`。

为了之后将用户重定向回之前的位置，站点需要记住他们在被重定向到登录页面之前打算访问哪个页面。因此，站点会使用某种附加在 URL 上的重定向参数来跟踪用户的原始位置。该参数决定登录后将用户重定向到哪里。例如，URL `https://example.com/login?redirect=https://example.com/dashboard` 会在登录后重定向到用户仪表板（位于 `https://example.com/dashboard`）。或者，如果用户原本试图浏览其账户设置页面，站点会在登录后将其重定向到设置页面，URL 可能看起来像这样：`https://example.com/login?redirect=https://example.com/settings`。

自动重定向用户可以节省时间并改善体验，因此你会发现许多应用程序都实现了此功能。

在开放重定向攻击中，攻击者通过向用户提供来自合法站点的 URL（该 URL 会重定向到其他地方）来诱骗用户访问外部站点，例如：`https://example.com/login?redirect=https://attacker.com`。这样的 URL 可能诱使受害者点击链接，因为他们认为这指向合法站点 example.com 的页面。但实际上，该页面会自动重定向到恶意页面。然后攻击者可以发起社会工程学攻击，诱骗用户在攻击者网站上输入其在 example.com 的凭据。在网络安全领域，社会工程学是指欺骗受害者的攻击。利用社会工程学窃取凭据和私人信息的攻击称为**网络钓鱼**。

另一种常见的开放重定向技术是基于 Referer 的开放重定向。Referer 是浏览器自动包含的一个 HTTP 请求头，告诉服务器请求来源于何处。Referer 头是确定用户原始位置的常用方式，因为它包含链接到当前页面的 URL。因此，某些站点会在特定用户操作（如登录或退出）后自动重定向到页面的 Referer URL。在这种情况下，攻击者可以托管一个链接到受害者站点的页面，以设置请求的 Referer 头，使用如下 HTML：

```html
<html>
  <a href="https://example.com/login">Click here to log in to example.com</a>
</html>
```

该 HTML 页面包含一个 `<a>` 标签，将标签中的文本链接到另一个位置。此页面包含一个文本为 “Click here to log in to example.com” 的链接。当用户点击该链接时，会被重定向到 `<a>` 标签的 `href` 属性指定的位置，本例中为 `https://example.com/login`。

## 防御措施

为了防止开放重定向，服务器需要确保不会将用户重定向到恶意位置。站点通常实现 URL 验证器，以确保用户提供的重定向 URL 指向合法位置。这些验证器使用黑名单或白名单。

当验证器使用黑名单时，它会检查重定向 URL 是否包含恶意重定向的某些指示符，然后相应阻止这些请求。例如，站点可能会将已知的恶意主机名或开放重定向攻击中常用的特殊 URL 字符列入黑名单。当验证器使用白名单时，它会检查 URL 的主机名部分，确保其与预定义的允许主机列表匹配。如果 URL 的主机名部分匹配某个允许的主机名，则重定向通过；否则，服务器阻止重定向。

这些防御机制听起来很简单，但现实是正确解析和解码 URL 很难做到。验证器通常难以正确识别 URL 的主机名部分。这使得开放重定向成为现代 Web 应用程序中最常见的漏洞之一。我们将在本章后面讨论攻击者如何利用 URL 验证问题来绕过开放重定向保护。

## 寻找开放重定向

让我们从寻找一个简单的开放重定向开始。你可以通过一些侦察技巧发现易受攻击的端点，并手动确认开放重定向。

### 第 1 步：寻找重定向参数

首先搜索用于重定向的参数。这些通常以 URL 参数的形式出现，如下面加粗的部分：

- `https://example.com/login?**redirect**=https://example.com/dashboard`
- `https://example.com/login?**redir**=https://example.com/dashboard`
- `https://example.com/login?**next**=https://example.com/dashboard`
- `https://example.com/login?**next**=/dashboard`

在浏览网站时打开代理工具。然后，在 HTTP 历史记录中查找包含绝对 URL 或相对 URL 的任何参数。**绝对 URL** 是完整的，包含定位其指向资源所需的所有组件，如 `https://example.com/login`。绝对 URL 至少包含 URL 方案、主机名和资源路径。**相对 URL** 必须由服务器与另一个 URL 拼接后才能使用，通常只包含 URL 的路径部分，如 `/login`。有些重定向 URL 甚至会省略相对 URL 的第一个斜杠（`/`）字符，如 `https://example.com/login?next=dashboard`。

请注意，并非所有重定向参数都有像 `redirect` 或 `redir` 这样直白的名称。例如，我曾见过名为 `RelayState`、`next`、`u`、`n` 和 `forward` 的重定向参数。无论参数名称是什么，你都应记录所有似乎用于重定向的参数。

此外，注意那些 URL 中不包含重定向参数但仍然自动重定向用户的页面。这些页面可能是基于 Referer 的开放重定向的候选。要找到这些页面，你可以留意 3XX 响应码（如 301 和 302），这些响应码表示重定向。

### 第 2 步：使用 Google Dork 查找更多重定向参数

Google dork 技术是查找重定向参数的有效方法。要使用 Google dork 在目标站点上查找重定向参数，首先设置站点搜索词为目标站点：

```
site:example.com
```

然后查找 URL 参数中包含 URL 的页面，利用 `%3D`（等号 `=` 的 URL 编码版本）。通过在搜索词中添加 `%3D`，你可以搜索像 `=http` 和 `=https` 这样的词，这些是参数中包含 URL 的指示。以下搜索查找包含绝对 URL 的 URL 参数：

```
inurl:%3Dhttp site:example.com
```

该搜索词可能找到以下页面：

- `https://example.com/login?next=https://example.com/dashboard`
- `https://example.com/login?u=http://example.com/settings`

也可以尝试使用 `%2F`（斜杠 `/` 的 URL 编码版本）。以下搜索词搜索包含 `=/` 的 URL，因此返回包含相对 URL 的 URL 参数：

```
inurl:%3D%2F site:example.com
```

此搜索词会找到类似这样的 URL：

- `https://example.com/login?n=/dashboard`

或者，你可以搜索常见 URL 重定向参数的名称。以下是一些搜索词，很可能会显示用于重定向的参数：

- `inurl:redir site:example.com`
- `inurl:redirect site:example.com`
- `inurl:redirecturi site:example.com`
- `inurl:redirect_uri site:example.com`
- `inurl:redirecturl site:example.com`
- `inurl:return site:example.com`
- `inurl:returnurl site:example.com`
- `inurl:relaystate site:example.com`
- `inurl:forward site:example.com`
- `inurl:forwardurl site:example.com`
- `inurl:forward_url site:example.com`
- `inurl:url site:example.com`
- `inurl:uri site:example.com`
- `inurl:dest site:example.com`
- `inurl:destination site:example.com`
- `inurl:next site:example.com`

这些搜索词会找到如下 URL：

- `https://example.com/logout?dest=/`
- `https://example.com/login?RelayState=https://example.com/home`
- `https://example.com/logout?forward=home`
- `https://example.com/login?return=home/settings`

记录你发现的新参数，以及第 1 步中找到的参数。

### 第 3 步：测试基于参数的开放重定向

接下来，注意你发现的每个重定向参数的功能，并测试每个参数是否存在开放重定向。将一个随机主机名或你自己拥有的主机名插入重定向参数，然后查看站点是否自动重定向到你指定的站点：

- `https://example.com/login?n=http://google.com`
- `https://example.com/login?n=http://attacker.com`

某些站点会在你访问 URL 后立即重定向到目标站点，无需任何用户交互。但对于许多页面，重定向要等到用户操作（如注册、登录或退出）之后才会发生。在这种情况下，请在检查重定向之前确保完成所需的用户交互。

### 第 4 步：测试基于 Referer 的开放重定向

最后，对你在第 1 步中找到的那些虽不包含重定向 URL 参数但仍重定向用户的页面进行基于 Referer 的开放重定向测试。为此，在你拥有的域名上设置一个页面，并托管以下 HTML 页面：

```html
<html>
  <a href="https://example.com/login">Click on this link!</a>
</html>
```

将链接的 URL 替换为目标页面。然后重新加载并访问你的 HTML 页面。点击链接，查看你是否被自动重定向到你的站点，或者在所需的用户交互后重定向。

## 绕过开放重定向保护

作为一名漏洞赏金猎人，我几乎在所有攻击的 Web 目标中都发现了开放重定向。为什么开放重定向在当今 Web 应用中仍然如此普遍？站点通过验证用于重定向用户的 URL 来防止开放重定向，因此开放重定向的根本原因是 URL 验证失败。而遗憾的是，URL 验证极难做到正确。

下面你可以看到 URL 的组成部分。浏览器重定向用户的方式取决于浏览器如何区分这些组件：

```
scheme://userinfo@hostname:port/path?query#fragment
```

URL 验证器需要预测浏览器将如何重定向用户，并拒绝会导致站外重定向的 URL。浏览器将用户重定向到 URL 的主机名部分所指示的位置。然而，URL 并不总是遵循此示例中所示的严格格式。它们可能是格式错误的，组件顺序错乱，包含浏览器不知道如何解码的字符，或者有多余或缺失的组件。例如，浏览器将如何重定向这个 URL？

```
https://user:password:8080/example.com@attacker.com
```

当你在不同浏览器中访问此链接时，你会发现不同浏览器处理该 URL 的方式不同。有时验证器没有考虑到所有可能导致浏览器意外行为的边缘情况。在这种情况下，你可以尝试使用几种策略绕过保护，本节将介绍这些策略。

### 利用浏览器自动更正

首先，你可以利用浏览器的自动更正功能构造替代 URL 来重定向到站外。现代浏览器通常会自动更正缺少正确组件的 URL，以修正用户输入错误导致的混乱 URL。例如，Chrome 会将所有这些 URL 解释为指向 `https://attacker.com`：

- `https:attacker.com`
- `https;attacker.com`
- `https:\\/\\/attacker.com`
- `https:/\\/\\attacker.com`

这些特性可以帮助你绕过基于黑名单的 URL 验证。例如，如果验证器拒绝任何包含字符串 `https://` 或 `http://` 的重定向 URL，你可以使用替代字符串（如 `https;`）达到相同效果。

大多数现代浏览器还会自动将反斜杠（`\`）更正为正斜杠（`/`），这意味着它们会将以下 URL 视为相同：

- `https:\\example.com`
- `https://example.com`

如果验证器不识别此行为，这种不一致可能导致漏洞。例如，以下 URL 可能存在问题：

```
https://attacker.com\\@example.com
```

除非验证器将反斜杠视为路径分隔符，否则它会将主机名解释为 `example.com`，并将 `attacker.com\\` 视为 URL 的用户名部分。但如果浏览器将反斜杠自动更正为正斜杠，它会将用户重定向到 `attacker.com`，并将 `@example.com` 视为 URL 的路径部分，形成以下有效 URL：

```
https://attacker.com/@example.com
```

### 利用有缺陷的验证器逻辑

绕过开放重定向验证器的另一种方法是利用验证器逻辑中的漏洞。例如，作为防御开放重定向的常见措施，URL 验证器通常检查重定向 URL 是否以站点的域名为开头、包含或结尾。你可以通过创建包含目标域名的子域或目录来绕过此类保护：

- `https://example.com/login?redir=http://example.com.attacker.com`
- `https://example.com/login?redir=http://attacker.com/example.com`

为了防止此类攻击成功，验证器可能只接受既以白名单中域名开头又以它结尾的 URL。但是，可以构造一个同时满足这两个规则的 URL。看看这个：

```
https://example.com/login?redir=https://example.com.attacker.com/example.com
```

该 URL 虽然以目标域名开头和结尾，但实际重定向到 `attacker.com`。浏览器会将第一个 `example.com` 解释为子域名，第二个 `example.com` 解释为文件路径。

或者你可以使用 at 符号（`@`）使第一个 `example.com` 成为 URL 的用户名部分：

```
https://example.com/login?redir=https://example.com@attacker.com/example.com
```

自定义构建的 URL 验证器容易受到此类攻击，因为开发人员通常不会考虑所有边缘情况。

### 使用 Data URL

你还可以操纵 URL 的方案部分来欺骗验证器。如第 6 章所述，数据 URL 使用 `data:` 方案将小文件嵌入 URL。其格式如下：

```
data:MEDIA_TYPE[;base64],DATA
```

例如，你可以使用 data 方案发送纯文本消息，如下所示：

```
data:text/plain,hello!
```

可选的 `base64` 规范允许你发送 base64 编码的消息。例如，以下是上述消息的 base64 编码版本：

```
data:text/plain;base64,aGVsbG8h
```

你可以使用 `data:` 方案构造一个 base64 编码的重定向 URL 来绕过验证器。例如，此 URL 将重定向到 `example.com`：

```
data:text/html;base64,PHNjcmlwdD5sb2NhdGlvbj0iaHR0cHM6Ly9leGFtcGxlLmNvbSI8L3NjcmlwdD4=
```

此 URL 中编码的数据 `PHNjcmlwdD5sb2NhdGlvbj0iaHR0cHM6Ly9leGFtcGxlLmNvbSI8L3NjcmlwdD4=` 是以下脚本的 base64 编码版本：

```html
<script>location="https://example.com"</script>
```

这是一段包裹在 HTML `<script>` 标签中的 JavaScript 代码，它将浏览器位置设置为 `https://example.com`，强制浏览器重定向到那里。你可以将此数据 URL 插入重定向参数以绕过黑名单：

```
https://example.com/login?redir=data:text/html;base64,PHNjcmlwdD5sb2NhdGlvbj0iaHR0cHM6Ly9leGFtcGxlLmNvbSI8L3NjcmlwdD4=
```

### 利用 URL 解码

通过互联网发送的 URL 只能包含 ASCII 字符，其中包括英语中常用的一组字符和几个特殊字符。但由于 URL 经常需要包含特殊字符或其他语言的字符，人们使用 URL 编码对字符进行编码。URL 编码将字符转换为百分号后跟两个十六进制数字，例如 `%2f`，这是斜杠字符（`/`）的 URL 编码版本。

当验证器验证 URL 或浏览器重定向用户时，它们必须首先通过解码任何 URL 编码的字符来确定 URL 中包含的内容。如果验证器和浏览器解码 URL 的方式存在任何不一致，你就可以利用这一点。

#### 双重编码

首先，尝试对载荷中的某些特殊字符进行双重或三重 URL 编码。例如，你可以对 `https://example.com/@attacker.com` 中的斜杠字符进行 URL 编码。以下是带有 URL 编码斜杠的 URL：

```
https://example.com%2f@attacker.com
```

以下是带有双重 URL 编码斜杠的 URL：

```
https://example.com%252f@attacker.com
```

最后，这是带有三重 URL 编码斜杠的 URL：

```
https://example.com%25252f@attacker.com
```

每当验证器和浏览器解码这些特殊字符的方式存在不一致时，你就可以利用这种不一致来诱导开放重定向。例如，某些验证器可能完全解码这些 URL，然后假定 URL 重定向到 `example.com`，因为 `@attacker.com` 位于 URL 的路径部分。然而，浏览器可能不完全解码 URL，而是将 `example.com%25252f` 视为 URL 的用户名部分。

另一方面，如果验证器不对 URL 进行双重解码，但浏览器会，那么你可以使用如下载荷：

```
https://attacker.com%252f@example.com
```

验证器会将 `example.com` 视为主机名。但浏览器会重定向到 `attacker.com`，因为 `@example.com` 成为 URL 的路径部分，如下所示：

```
https://attacker.com/@example.com
```

#### 非 ASCII 字符

有时你可以利用验证器和浏览器解码非 ASCII 字符方式的不一致。例如，假设此 URL 已通过 URL 验证：

```
https://attacker.com%ff.example.com
```

`%ff` 是字符 ÿ，这是一个非 ASCII 字符。验证器已确定 `example.com` 是域名，`attacker.comÿ` 是子域名。可能发生几种情况。有时浏览器会将非 ASCII 字符解码为问号。在这种情况下，`example.com` 将成为 URL 查询的一部分，而非主机名，浏览器将导航到 `attacker.com`：

```
https://attacker.com?.example.com
```

另一种常见情况是浏览器会尝试查找 “最相似” 的字符。例如，如果字符 ╱（`%E2%95%B1`）出现在如下 URL 中，验证器可能确定主机名是 `example.com`：

```
https://attacker.com╱.example.com
```

但浏览器会将这个类似斜杠的字符转换为实际斜杠，使 `attacker.com` 成为主机名：

```
https://attacker.com/.example.com
```

浏览器经常以这种方式规范化 URL，以追求用户友好。除了相似符号外，你还可以使用其他语言的字符集来绕过过滤器。Unicode 标准是一套为在计算机上表示世界上所有语言而开发的代码。你可以在 http://www.unicode.org/charts/ 找到 Unicode 字符列表。使用 Unicode 图表查找相似字符并将其插入 URL 以绕过过滤器。西里尔字母字符集尤其有用，因为它包含许多与 ASCII 字符相似的字符。

### 组合利用技术

为了击败更复杂的 URL 验证器，可以组合多种策略来绕过分层防御。我发现以下载荷很有用：

```
https://example.com%252f@attacker.com/example.com
```

该 URL 通过使 URL 既以 `example.com` 开头又以它结尾，绕过了仅检查 URL 是否包含、以白名单主机名开头或结尾的保护。大多数浏览器会将 `example.com%252f` 解释为 URL 的用户名部分。但如果验证器过度解码 URL，它会将 `example.com` 混淆为主机名部分：

```
https://example.com/@attacker.com/example.com
```

你还可以使用更多方法来击败 URL 验证器。在本节中，我概述了最常见的方法。尝试每一种方法以检查你正在测试的验证器中的弱点。如果有时间，可以实验各种 URL，发明新的绕过 URL 验证器的方法。例如，尝试在 URL 中插入随机非 ASCII 字符，或故意弄乱其不同组件，观察浏览器如何解释它。

## 提升攻击影响

攻击者可以单独使用开放重定向使其网络钓鱼攻击更具可信度。例如，他们可以在电子邮件中向用户发送此 URL：`https://example.com/login?next=https://attacker.com/fake_login.html`。虽然此 URL 首先会将用户引向合法网站，但在登录后会将其重定向到攻击者站点。攻击者可以在恶意站点上托管一个镜像合法站点登录页面的伪造登录页面，并通过如下消息提示用户再次登录：

> 抱歉！您提供的密码不正确。请重新输入用户名和密码。

用户相信自己输错了密码，便会向攻击者站点提供其凭据。此时，攻击者站点甚至可以将用户重定向回合法站点，使受害者无法察觉其凭据已被窃取。

由于组织无法完全防止网络钓鱼（因为这些攻击依赖于人为判断），安全团队通常会在单独报告时将开放重定向视为微不足道的漏洞。但开放重定向通常可以作为漏洞链的一部分，以实现更大的影响。例如，开放重定向可以帮助你绕过 URL 黑名单和白名单。以这个 URL 为例：

```
https://example.com/?next=https://attacker.com/
```

即使面对实现良好的 URL 验证器，该 URL 也会通过，因为该 URL 在技术上仍然位于合法网站上。因此，开放重定向可以帮助你最大化如服务器端请求伪造（SSRF）等漏洞的影响，我将在第 13 章中讨论 SSRF。如果站点利用白名单防止 SSRF，并且只允许向预定义 URL 列表发出请求，攻击者可以利用这些白名单页面中的开放重定向将请求重定向到任何位置。

你还可以使用开放重定向窃取凭据和 OAuth 令牌。通常，当页面重定向到另一个站点时，浏览器会将来源 URL 作为 Referer HTTP 请求头包含在内。当来源 URL 包含敏感信息（如身份验证令牌）时，攻击者可以诱导开放重定向，通过 Referer 头窃取令牌。（即使在敏感端点上没有开放重定向，也有办法通过开放重定向链将令牌带出站外。我将在第 20 章详细介绍这些攻击的工作原理。）

2. **搜索执行基于 Referer 重定向的页面**。这些是基于 Referer 的开放重定向的候选。
3. **对你发现的页面和参数进行开放重定向测试**。
4. **如果服务器阻止了开放重定向，尝试本章提到的绕过技术**。
5. **头脑风暴如何将开放重定向用于你的其他漏洞链中！**
