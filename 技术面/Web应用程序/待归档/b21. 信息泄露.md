# 第21章：信息泄露（Information Disclosure）

第10章介绍的IDOR漏洞是应用程序泄露用户私人信息的常见方式。但攻击者还可以通过其他方式从目标应用程序中发现敏感信息。我把这些漏洞称为**信息泄露漏洞**。这些漏洞很常见；事实上，即使我在寻找其他类型的漏洞时，它们也是我在漏洞赏金狩猎中最常发现的漏洞类型。

这些漏洞可能以多种方式发生，具体取决于应用程序。在本章中，我们将讨论几种你可能从应用程序中泄露数据的方式，以及如何最大化自己发现信息泄露的机会。本章深入探讨了第5章提到的一些技术，但侧重于使用这些技术提取敏感和私人信息。

## 机制

当应用程序未能正确保护敏感信息，使用户能够访问他们本不应获得的信息时，就会发生信息泄露。这些敏感信息可能包括有助于攻击的技术细节，如软件版本号、内部IP地址、敏感文件名和文件路径。它还可能包括允许攻击者对应用程序进行源代码审查的源代码。其他时候，应用程序会向未经授权的第三方泄露用户的私人信息，如用户的年龄、银行账号、邮箱地址和邮寄地址。

大多数系统旨在对开发信息（包括软件版本号和配置文件）对外界保密，因为它允许攻击者收集有关应用程序的信息并制定最有效的攻击策略。例如，了解应用程序使用的确切软件版本将允许攻击者查找影响该应用程序的公开漏洞。配置文件通常包含访问令牌和内部IP地址等信息，攻击者可以利用这些信息进一步入侵组织。

通常，应用程序在HTTP响应头、HTTP响应体或其他服务器响应中泄露版本号。例如，许多应用程序使用的`X-Powered-By`头显示应用程序运行的框架：

```
X-Powered-By: PHP/5.2.17
```

另一方面，应用程序通过未对文件应用适当的访问控制，或意外将敏感文件上传到外部用户可以访问的公共存储库，来泄露敏感配置文件。

应用程序应保护的另一个信息是其源代码。当应用程序的后端代码泄露给公众时，泄露的代码可以帮助攻击者理解应用程序的逻辑，并寻找逻辑缺陷漏洞、硬编码凭据或公司基础设施信息（如内部IP）。

应用程序可能通过意外发布私有代码仓库、在公共GitHub或GitLab仓库上共享代码片段，或将其上传到Pastebin等第三方站点来泄露源代码。

最后，应用程序常常通过在公共代码中包含敏感信息来泄露信息。开发人员可能意外地将凭据、内部IP地址、信息性代码注释和用户私人信息等放置在提供给用户的HTML和JavaScript文件等公共源代码中。

## 防御措施

完全防止敏感信息泄露很困难。但你可以在开发过程中通过保护数据来可靠地降低信息泄露的可能性。

你应该采取的最重要措施是避免将凭据和其他敏感信息硬编码到可执行代码中。相反，你可以将敏感信息放在单独的配置文件或像Vault（https://github.com/hashicorp/vault/）这样的秘密存储系统中。此外，定期审计你的公共代码仓库，确保敏感文件没有被意外上传。工具可以帮助你监控代码中的秘密，如secret-bridge（https://github.com/duo-labs/secret-bridge/）。如果必须将敏感文件上传到生产服务器，应用细粒度的访问控制以限制用户对文件的访问。

接下来，从服务和服务器响应中移除会泄露后端服务器设置和软件版本技术细节的数据。通过向用户返回通用错误页面来处理所有异常，而不是显示错误详细信息的页面。

## 寻找信息泄露

根据你攻击的应用程序和你要寻找的内容，你可以使用多种策略来发现信息泄露漏洞。一个好的起点是使用第5章介绍的侦察技术查找软件版本号和配置信息。然后你可以开始查找暴露的配置文件、数据库文件以及上传到生产服务器但未受保护的其他敏感文件。以下步骤讨论了一些你可以尝试的技术。

### 第1步：尝试路径遍历攻击

首先尝试路径遍历攻击以读取服务器的敏感文件。路径遍历攻击用于访问Web应用程序根目录以外的文件。此过程涉及操纵应用程序用于引用文件的文件路径变量，向其添加`../`字符。在Unix系统中，此序列指向当前目录的父目录，因此通过将其添加到文件路径，你通常可以到达Web根目录之外的文件。

例如，假设一个网站允许你使用相对URL加载应用程序图像文件夹中的图像。绝对URL包含完整地址，从URL协议到域名和资源的路径名。另一方面，相对URL仅包含完整URL的一部分。大多数只包含资源的路径或文件名。相对URL用于链接到同一域上的另一个位置。

例如，此URL将把用户重定向到 `https://example.com/images/1.png`：

```
https://example.com/image?url=/images/1.png
```

在这种情况下，`url`参数包含一个相对URL（`/images/1.png`），它引用Web应用程序根目录内的文件。你可以插入`../`序列尝试导航出images文件夹并离开Web根目录。例如，以下URL指向Web应用程序根文件夹（以及images文件夹之外）的`index.html`文件：

```
https://example.com/image?url=/images/../index.html
```

类似地，以下将访问服务器根目录的`/etc/shadow`文件，该文件存储系统用户账户列表及其加密密码：

```
https://example.com/image?url=/images/../../../../../../../etc/shadow
```

可能需要一些试错来确定需要多少`../`序列才能到达系统根目录。此外，如果应用程序实施了某种输入验证，不允许文件路径中出现`../`，你可以使用`../`的编码变体，如`%2e%2e%2f`（URL编码）、`%252e%252e%255f`（双重URL编码）和`..%2f`（部分URL编码）。

### 第2步：搜索Wayback Machine

另一种查找暴露文件的方法是使用Wayback Machine。如第5章所述，Wayback Machine是一个在线存档，记录了网站在不同时间点的样子。你可以使用它来查找隐藏和废弃的端点，以及大量当前端点，而无需主动爬行站点，使其成为了解应用程序可能暴露内容的好方法。

在Wayback Machine网站上，只需搜索一个域即可查看其过去版本。要搜索域的存档文件，请访问 `https://web.archive.org/web/*/DOMAIN`。

在此URL中添加`/*`可获取与域相关的存档URL列表。例如，`https://web.archive.org/web/*/example.com/*`将返回与example.com相关的URL列表。你应该会在Wayback Machine网页上看到显示URL（图21-1）。

*（图21-1：你可以在Wayback Machine上列出域的存档URL。）*

然后你可以使用搜索功能查看是否有任何敏感页面已被存档。例如，要查找管理页面，在找到的URL中搜索`/admin`一词（图21-2）。

*（图21-2：在URL中搜索关键字以查找潜在敏感页面。）*

你也可以通过使用常见文件扩展名（如`.conf`（图21-3）和`.env`）搜索备份文件和配置文件，或通过使用文件扩展名`.js`和`.php`查找JavaScript或PHP文件等源代码。

下载有趣的存档页面并查找任何敏感信息。例如，是否有任何仍在使用的硬编码凭据，或者页面是否泄露了普通用户不应知道的任何隐藏端点？

### 第3步：搜索粘贴转储站点

接下来，查找Pastebin和GitHub gists等粘贴转储站点。这些站点允许用户通过直接链接共享文本文档，而不是通过电子邮件或Google Docs等服务，因此开发人员经常使用它们向同事发送源代码、配置文件和日志文件。但在Pastebin等站点上，共享的文本文件默认是公开的。如果开发人员上传敏感文件，所有人都可以读取它。因此，这些代码共享站点因泄露API密钥和密码等凭据而臭名昭著。

Pastebin有一个API，允许用户使用关键字、邮箱或域名搜索公共粘贴文件。你可以使用此API查找属于特定组织的敏感文件。PasteHunter或pastebin-scraper等工具也可以自动化该过程。Pastebin-scraper（https://github.com/streaak/pastebin-scraper/）使用Pastebin API帮助你搜索粘贴文件。此工具是一个shell脚本，因此将其下载到本地目录并运行以下命令以搜索与特定关键字关联的公共粘贴文件。`-g`选项表示通用关键字搜索：

```bash
./scrape.sh -g KEYWORD
```

此命令将返回与指定`KEYWORD`关联的Pastebin文件ID列表。你可以通过访问`pastebin.com/ID`来访问返回的粘贴文件。

### 第4步：从暴露的 .git 目录重建源代码

另一种查找敏感文件的方法是从暴露的`.git`目录重建源代码。在攻击应用程序时，获取其源代码对于构建漏洞利用可能非常有帮助。这是因为一些漏洞（如SQL注入）通过静态代码分析比黑盒测试更容易发现。第22章涵盖如何审查代码中的漏洞。

当开发人员使用Git对项目的源代码进行版本控制时，Git会将所有项目的版本控制信息（包括项目文件的提交历史）存储在Git目录中。通常，这个`.git`文件夹不应公开，但有时会被意外公开。这就是信息泄露发生的时候。当`.git`目录被暴露时，攻击者可以获取应用程序的源代码，从而通过truffleHog（https://github.com/dxa4481/truffleHog/）或Gitleaks（https://github.com/zricethezav/gitleaks/）等秘密扫描工具获得开发者注释、硬编码API密钥和其他敏感数据。

#### 检查 .git 文件夹是否公开

要检查应用程序的`.git`文件夹是否公开，只需转到应用程序的根目录（例如 example.com）并在URL中添加`/.git`：

```
https://example.com/.git
```

当你浏览`/.git`目录时可能发生三种情况。如果收到404错误，这意味着应用程序的`.git`目录未公开，你将无法通过这种方式泄露信息。如果收到403错误，`.git`目录在服务器上可用，但你无法直接访问文件夹根目录，因此无法列出目录中包含的所有文件。如果你没有收到错误且服务器响应`.git`目录的目录列表，你可以直接浏览文件夹的内容并检索其中包含的任何信息。

#### 下载文件

如果启用了目录列表，你可以浏览文件并检索泄露的信息。`wget`命令从Web服务器检索内容。你可以使用递归模式（`-r`）的`wget`批量下载指定目录及其子目录中存储的所有文件：

```bash
$ wget -r example.com/.git
```

但如果未启用目录列表且目录的文件未显示，你仍然可以重建整个`.git`目录。首先，你需要确认文件夹的内容确实公开。你可以通过尝试访问目录的`config`文件来做到这一点：

```bash
$ curl https://example.com/.git/config
```

如果此文件可访问，只要你知道`.git`目录的一般结构，你可能能够下载Git目录的全部内容。`.git`目录以特定方式布局。当你在Git仓库中执行以下命令时，你应该看到类似于以下的内容：

```bash
$ ls .git
COMMIT_EDITMSG HEAD branches config description hooks index info logs objects refs
```

这里显示的输出列出了几个标准和文件，这些文件和文件夹对于重建项目源代码很重要。特别是`/objects`目录用于存储Git对象。此目录包含其他文件夹；每个都有两个字符的名称，对应于其中存储的Git对象的SHA1哈希的前两个字符。在这些子目录中，你会找到以其中存储的Git对象的SHA1哈希的其余部分命名的文件。换句话说，哈希为`0a082f2656a655c8b0a87956c7bcdc93dfda23f8`的Git对象将存储在目录`.git/objects/0a`中，文件名为`082f2656a655c8b0a87956c7bcdc93dfda23f8`。例如，以下命令将返回一个文件夹列表：

```bash
$ ls .git/objects
00 0a 14 5a 64 6e 82 8c 96 a0 aa b4 be c8 d2 dc e6 f0 fa info pack
```

此命令将显示存储在特定文件夹中的Git对象：

```bash
$ ls .git/objects/0a
082f2656a655c8b0a87956c7bcdc93dfda23f8 4a1ee2f3a3d406411a72e1bea63507560092bd 66452433322af3d3
19a377415a890c70bbd263 8c20ea4482c6d2b0c9cdaf73d4b05c2c8c44e9 ee44c60c73c5a622bb1733338d3fa964
b333f0
0ec99d617a7b78c5466daa1e6317cbd8ee07cc 52113e4f248648117bc4511da04dd4634e6753
72e6850ef963c6aeee4121d38cf9de773865d8
```

Git在`.git/objects`中存储不同类型的对象：提交、树、blob和带注释的标签。你可以使用以下命令确定对象类型：

```bash
$ git cat-file -t OBJECT-HASH
```

提交对象存储提交的树对象哈希、父提交、作者、提交者、日期和提交消息等信息。树对象包含提交的目录列表。Blob对象包含已提交文件的副本（即：实际的源代码！）。最后，标签对象包含有关带标签对象及其相关标签名称的信息。你可以使用以下命令显示与Git对象关联的文件：

```bash
$ git cat-file -p OBJECT-HASH
```

`/config`文件是项目的Git配置文件，`/HEAD`文件包含对当前分支的引用：

```bash
$ cat .git/HEAD
ref: refs/heads/master
```

如果你无法访问`/.git`文件夹的目录列表，你必须逐个下载每个文件，而不是从目录根目录递归下载。但当对象文件具有复杂路径（如`.git/objects/0a/72e6850ef963c6aeee4121d38cf9de773865d8`）时，你如何知道服务器上哪些文件可用？

你从已知存在的文件路径开始，如`.git/HEAD`！读取此文件将给你一个对当前分支的引用（例如`.git/refs/heads/master`），你可以使用它来查找系统上的更多文件：

```bash
$ cat .git/HEAD
ref: refs/heads/master
$ cat .git/refs/heads/master
0a66452433322af3d319a377415a890c70bbd263
$ git cat-file -t 0a66452433322af3d319a377415a890c70bbd263
commit
$ git cat-file -p 0a66452433322af3d319a377415a890c70bbd263
tree 0a72e6850ef963c6aeee4121d38cf9de773865d8
```

`.git/refs/heads/master`文件将指向存储提交目录树的特定对象哈希。从那里，你可以看到该对象是一个提交，并与树对象`0a72e6850ef963c6aeee4121d38cf9de773865d8`关联。现在检查该树对象：

```bash
$ git cat-file -p 0a72e6850ef963c6aeee4121d38cf9de773865d8
100644 blob 6ad5fb6b9a351a77c396b5f1163cc3b0abcde895 .gitignore
040000 blob 4b66088945aab8b967da07ddd8d3cf8c47a3f53c source.py
040000 blob 9a3227dca45b3977423bb1296bbc312316c2aa0d README
040000 tree 3b1127d12ee43977423bb1296b8900a316c2ee32 resources
```

找到了！你发现了一些源代码文件和额外的对象树可供探索。

在远程服务器上，你发现不同文件的请求会略有不同。例如，你可以使用此URL确定HEAD：

```
https://example.com/.git/HEAD
```

使用此URL查找该HEAD中存储的对象：

```
https://example.com/.git/refs/heads/master
```

使用此URL访问与提交关联的树：

```
https://example.com/.git/objects/0a/72e6850ef963c6aeee4121d38cf9de773865d8
```

最后，使用此URL下载存储在`source.py`文件中的源代码：

```
https://example.com/.git/objects/4b/66088945aab8b967da07ddd8d3cf8c47a3f53c
```

如果你从远程服务器下载文件，你还需要在读取之前解压缩下载的对象文件。这可以使用一些代码完成。你可以使用Ruby、Python或你首选语言的zlib库解压缩对象文件：

```bash
ruby -rzlib -e 'print Zlib::Inflate.new.inflate(STDIN.read)' < OBJECT_FILE
python -c 'import zlib, sys; print repr(zlib.decompress(sys.stdin.read()))' < OBJECT_FILE
```

恢复项目源代码后，你可以使用`grep`搜索敏感数据，如硬编码凭据、加密密钥和开发者注释。如果有时间，你可以浏览整个恢复的代码库进行源代码审查并发现潜在漏洞。

### 第5步：在公共文件中查找信息

你也可以尝试在应用程序的公共文件中查找信息泄露，如HTML和JavaScript源代码。根据我的经验，JavaScript文件是信息泄露的丰富来源！

以普通用户身份浏览你攻击的Web应用程序，并注意应用程序在何处显示或使用你的个人信息。然后右键点击这些页面并点击“查看页面源代码”。你应该会看到当前页面的HTML源代码。跟踪此页面上的链接以找到应用程序正在使用的其他HTML文件和JavaScript文件。然后，在找到的HTML文件和JavaScript文件中，使用`password`和`api_key`等关键字`grep`每个页面以查找硬编码凭据、API密钥和个人信息。

你也可以使用LinkFinder（https://github.com/GerbenJavado/LinkFinder/）等工具在站点上定位JavaScript文件。

## 提升攻击影响

找到敏感文件或敏感信息后，你必须在报告之前确定其影响。例如，如果你发现了密码或API密钥等凭据，你需要通过使用它们访问目标系统来验证它们是否正在使用。我经常发现无法用于访问任何内容的过期凭据。在这种情况下，信息泄露不是漏洞。

如果你发现的敏感文件或凭据是有效且当前的，请考虑如何使用它们来危害应用程序的安全。例如，如果你找到了GitHub访问令牌，你可能可以干预组织的项目并访问其私有仓库。如果你找到了其管理门户的密码，你可能可以泄露其客户的私人信息。如果你可以访问目标服务器上的`/etc/shadow`文件，你可能可以破解系统用户的密码并接管系统！报告信息泄露通常是通过强调泄露信息的关键性来向公司传达泄露的影响。

如果你发现的信息影响不是特别关键，你可以探索通过将其与其他安全问题链接来提升漏洞的方法。例如，如果你可以泄露目标网络内的内部IP地址，你可以在SSRF利用期间使用它们来转向网络。或者，如果你能确定应用程序运行的确切软件版本号，查看是否有与该软件版本相关的CVE可以帮助你实现RCE。

## 找到你的第一个信息泄露！

现在你了解了常见的信息泄露类型以及如何发现它们，按照本章讨论的步骤找到你的第一个信息泄露：

1. **使用第5章介绍的侦察技术查找软件版本号和配置信息**。
2. **开始搜索暴露的配置文件、数据库文件以及上传到生产服务器但未受适当保护的其他敏感文件**。你可以使用的技术包括路径遍历、爬取Wayback Machine或粘贴转储站点，以及在暴露的`.git`目录中查找文件。
3. **在应用程序的公共文件（如HTML和JavaScript源代码）中查找信息**，使用关键字`grep`这些文件。
4. **在报告之前考虑你找到的信息的影响**，并探索提升其影响的方法。
5. **起草你的第一份信息泄露报告并发送给漏洞赏金计划！**
