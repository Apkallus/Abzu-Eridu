# 测试 IMAP/SMTP 注入

|ID          |
|------------|
|WSTG-INPV-10|

## 概述

此威胁影响所有与邮件服务器（IMAP/SMTP）通信的应用程序，通常是 Webmail 应用程序。本测试的目的是验证由于输入数据未正确清理，是否能够将任意 IMAP/SMTP 命令注入到邮件服务器中。

如果邮件服务器无法直接从 Internet 访问，IMAP/SMTP 注入技术会更有效。如果可与后端邮件服务器进行完全通信，则建议直接进行测试。

IMAP/SMTP 注入使得能够访问本来无法直接从 Internet 访问的邮件服务器。在某些情况下，这些内部系统并不具备应用于前端 Web 服务器的相同级别的安全加固和基础设施安全。因此，邮件服务器可能更容易受到最终用户的攻击（参见图 1 所示的方案）。

![IMAP SMTP 注入](images/Imap-smtp-injection.png)\
*图 4.7.10-1：使用 IMAP/SMTP 注入技术与邮件服务器通信*

图 1 描绘了使用 Webmail 技术时通常看到的流量流程。步骤 1 和 2 是用户与 Webmail 客户端的交互，而步骤 2 是测试人员绕过 Webmail 客户端直接与后端邮件服务器交互。

该技术允许进行各种各样的操作和攻击。可能性取决于注入的类型和范围，以及被测的邮件服务器技术。

使用 IMAP/SMTP 注入技术的一些攻击示例包括：

- 利用 IMAP/SMTP 协议中的漏洞
- 规避应用程序限制
- 规避自动化流程
- 信息泄露
- 中继/垃圾邮件

## 测试目标

- 识别 IMAP/SMTP 注入点。
- 了解系统的数据流和部署结构。
- 评估注入的影响。

## 如何测试

### 识别易受攻击的参数

为了检测易受攻击的参数，测试人员必须分析应用程序处理输入的能力。输入验证测试要求测试人员向服务器发送虚假或恶意请求，并分析响应。在安全的应用程序中，响应应为错误，并附带一些相应操作，告知客户端出现问题。在易受攻击的应用程序中，恶意请求可能会被后端应用程序处理，并响应 `HTTP 200 OK` 消息。

需要注意的是，发送的请求应与被测技术相匹配。当使用 MySQL 服务器时，发送针对 Microsoft SQL Server 的 SQL 注入字符串将导致误报响应。在这种情况下，发送恶意 IMAP 命令是合理的操作方式，因为 IMAP 是被测的底层协议。

应使用的 IMAP 特殊参数包括：

| 在 IMAP 服务器上     | 在 SMTP 服务器上 |
|------------------------|--------------------|
| 认证         | 发件人邮件地址     |
| 邮箱操作（列表、读取、创建、删除、重命名） | 目标邮件地址 |
| 邮件操作（读取、复制、移动、删除） | 主题   |
| 断开连接          | 邮件正文       |
|                        | 附件     |

在此示例中，通过对所有包含该参数的请求进行操纵，测试 "mailbox" 参数：

`https://<webmail>/src/read_body.php?mailbox=INBOX&passed_id=46106&startMessage=1`

可以使用以下示例：

- 为参数赋予空值：

`https://<webmail>/src/read_body.php?mailbox=&passed_id=46106&startMessage=1`

- 将值替换为随机值：

`https://<webmail>/src/read_body.php?mailbox=NOTEXIST&passed_id=46106&startMessage=1`

- 为参数添加其他值：

`https://<webmail>/src/read_body.php?mailbox=INBOX PARAMETER2&passed_id=46106&startMessage=1`

- 添加非标准特殊字符（例如：`\`、`'`、`"`、`@`、`#`、`!`、`|`）：

`https://<webmail>/src/read_body.php?mailbox=INBOX"&passed_id=46106&startMessage=1`

- 删除该参数：

`https://<webmail>/src/read_body.php?passed_id=46106&startMessage=1`

上述测试的最终结果为测试人员提供了三种可能的情况：
S1 - 应用程序返回错误代码/消息
S2 - 应用程序不返回错误代码/消息，但未执行所请求的操作
S3 - 应用程序不返回错误代码/消息，并正常执行所请求的操作

情况 S1 和 S2 表示成功的 IMAP/SMTP 注入。

攻击者的目标是获得 S1 响应，因为它表明应用程序易受注入和进一步操纵的影响。

假设用户使用以下 HTTP 请求检索邮件头：

`https://<webmail>/src/view_header.php?mailbox=INBOX&passed_id=46105&passed_ent_id=0`

攻击者可能通过注入字符 `"`（使用 URL 编码为 `%22`）来修改参数 INBOX 的值：

`https://<webmail>/src/view_header.php?mailbox=INBOX%22&passed_id=46105&passed_ent_id=0`

在这种情况下，应用程序的响应可能为：

```txt
ERROR: Bad or malformed request.
Query: SELECT "INBOX""
Server responded: Unexpected extra arguments to Select
```

情况 S2 更难成功测试。测试人员需要使用盲命令注入来确定服务器是否存在漏洞。

另一方面，最后一种情况（S3）在本段中不相关。

### 了解客户端的数据流和部署结构

在识别出所有易受攻击的参数（例如 `passed_id`）后，测试人员需要确定可能的注入级别，然后设计测试计划以进一步利用应用程序。

在此测试案例中，我们检测到应用程序的 `passed_id` 参数存在漏洞，并用于以下请求：

`https://<webmail>/src/read_body.php?mailbox=INBOX&passed_id=46225&startMessage=1`

使用以下测试案例（在需要数值时提供字母值）：

`https://<webmail>/src/read_body.php?mailbox=INBOX&passed_id=test&startMessage=1`

将生成以下错误消息：

```txt
ERROR : Bad or malformed request.
Query: FETCH test:test BODY[HEADER]
Server responded: Error in IMAP command received by server.
```

在此示例中，错误消息返回了执行命令的名称及其相应参数。

在其他情况下，错误消息（应用程序`未控制`）包含执行命令的名称，但阅读相应的 [RFC](#references) 可让测试人员了解还可执行哪些其他命令。

如果应用程序不返回描述性错误消息，测试人员需要分析受影响的功能，以推断与上述功能相关的所有可能命令（和参数）。例如，如果在创建邮箱功能中检测到易受攻击的参数，则合乎逻辑的假设是受影响的 IMAP 命令为 `CREATE`。根据 RFC，`CREATE` 命令接受一个参数，该参数指定要创建的邮箱名称。

### IMAP/SMTP 命令注入

一旦测试人员识别出易受攻击的参数并分析了其执行上下文，下一阶段就是利用该功能。

此阶段有两种可能的结果：

1. 注入在未认证状态下是可能的：受影响的功能不需要用户认证。可注入的（IMAP）命令限于：`CAPABILITY`、`NOOP`、`AUTHENTICATE`、`LOGIN` 和 `LOGOUT`。
2. 注入仅在已认证状态下是可能的：成功利用要求用户在继续测试之前完全通过认证。

在任何情况下，IMAP/SMTP 注入的典型结构如下：

- 头部：预期命令的结束部分；
- 主体：新命令的注入；
- 尾部：预期命令的开始部分。

重要的是要记住，为了执行 IMAP/SMTP 命令，前一个命令必须以 CRLF（`%0d%0a`）序列终止。

假设在[识别易受攻击的参数](#identifying-vulnerable-parameters)阶段，攻击者检测到以下请求中的参数 `message_id` 存在漏洞：

`https://<webmail>/read_email.php?message_id=4791`

同时假设在阶段 2（“了解客户端的数据流和部署结构”）中执行的分析结果已识别出与此参数关联的命令和参数为：

`FETCH 4791 BODY[HEADER]`

在此场景中，IMAP 注入结构为：

`https://<webmail>/read_email.php?message_id=4791 BODY[HEADER]%0d%0aV100 CAPABILITY%0d%0aV101 FETCH 4791`

这将生成以下命令：

```sql
???? FETCH 4791 BODY[HEADER]
V100 CAPABILITY
V101 FETCH 4791 BODY[HEADER]
```

其中：

```sql
头部 = 4791 BODY[HEADER]
主体 = %0d%0aV100 CAPABILITY%0d%0a
尾部 = V101 FETCH 4791
```

>
> - 任意 IMAP/SMTP 命令注入
