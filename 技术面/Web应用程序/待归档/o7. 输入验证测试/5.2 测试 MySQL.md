# 测试 MySQL

## 概述

当输入在构造 SQL 查询时未经过充分约束或清理时，就会出现 [SQL 注入](https://owasp.org/www-community/attacks/SQL_Injection) 漏洞。使用动态 SQL（通过字符串拼接构造 SQL 查询）为这些漏洞打开了大门。SQL 注入允许攻击者访问 SQL 服务器。它允许在用于连接数据库的用户权限下执行 SQL 代码。

*MySQL 服务器*有一些特殊性，因此某些漏洞利用需要为此应用程序特别定制。这就是本节的主题。

## 如何测试

当在由 MySQL 数据库支持的应用中发现 SQL 注入漏洞时，根据 MySQL 版本和 DBMS 上的用户权限，可以执行多种攻击。

MySQL 在全球生产环境中至少有四个版本，`3.23.x`、`4.0.x`、`4.1.x` 和 `5.0.x`。每个版本都有一组与版本号成正比的功能。

- 从 4.0 版本开始：支持 UNION
- 从 4.1 版本开始：支持子查询
- 从 5.0 版本开始：支持存储过程、存储函数和名为 `INFORMATION_SCHEMA` 的视图
- 从 5.0.2 版本开始：支持触发器

应该注意，对于 4.0.x 之前的 MySQL 版本，由于子查询功能或 `UNION` 语句尚未实现，只能使用基于布尔或时间的盲注攻击。

从现在开始，我们假设存在经典的 SQL 注入漏洞，可以通过类似于 [测试 SQL 注入](05-Testing_for_SQL_Injection.md) 部分中描述的请求来触发。

`https://www.example.com/page.php?id=2`

### 单引号问题

在利用 MySQL 功能之前，必须考虑字符串在语句中的表示方式，因为 Web 应用经常转义单引号。

MySQL 引号转义如下：

`'A string with \'quotes\''`

也就是说，MySQL 将转义后的撇号 `\'` 解释为字符而非元字符。

因此，如果应用程序为了正常工作需要使用常量字符串，则需区分两种情况：

1. Web 应用转义单引号 `'` => `\'`
2. Web 应用不转义单引号 `'` => `'`

在 MySQL 下，有一种标准方法可以绕过对单引号的需求，即无需单引号即可声明常量字符串。

假设我们想要知道记录中名为 `password` 的字段的值，条件如下：

1. password like `'A%'`
2. 使用连接后的十六进制 ASCII 值：
    `password LIKE 0x4125`
3. 使用 char() 函数：
    `password LIKE CHAR(65,37)`

### 多混合查询

MySQL 库连接器不支持用 `;` 分隔的多查询，因此无法像在 Microsoft SQL Server 中那样在单个 SQL 注入漏洞中注入多个非齐次的 SQL 命令。

例如，以下注入将导致错误：

`1 ; update tablename set code='javascript code' where 1 --`

### 信息收集

#### 指纹识别 MySQL

当然，首先要了解后端数据库是否为 MySQL DBMS。MySQL 服务器有一个功能，用于让其他 DBMS 忽略 MySQL 方言中的子句。当注释块 `'/**/'` 包含感叹号 `'/*! sql here*/'` 时，MySQL 会解释它，而其他 DBMS 则将其视为普通注释块，如 [MySQL 手册](https://dev.mysql.com/doc/refman/8.0/en/comments.html)中所述。

示例：

`1 /*! and 1=0 */`

> 如果存在 MySQL，注释块内的子句将被解释。

#### 版本

有三种方法可以获取此信息：

1. 使用全局变量 `@@version`
2. 使用函数 [VERSION()](https://dev.mysql.com/doc/refman/8.0/en/information-functions.html#function_version)
3. 使用带有版本号的注释指纹识别 `/*!40110 and 1=0*/`

这意味着：

```sql
if(version >= 4.1.10)
   add 'and 1=0' to the query.
```

这些方法是等效的，因为结果是相同的。

带内注入：

`1 AND 1=0 UNION SELECT @@version /*`

推断注入：

`1 AND @@version like '4.0%'`

响应将包含类似以下内容：

`5.0.22-log`

#### 登录用户

MySQL 服务器依赖两种用户：

1. [USER()](https://dev.mysql.com/doc/refman/8.0/en/information-functions.html#function_user)：连接到 MySQL 服务器的用户。
2. [CURRENT_USER()](https://dev.mysql.com/doc/refman/8.0/en/information-functions.html#function_current-user)：执行查询的内部用户。

1 和 2 之间存在一些差异。主要区别在于，匿名用户（如果允许）可以以任何名称连接，但 MySQL 内部用户是一个空名称（''）。另一个区别是，存储过程或存储函数（除非另有声明）将作为创建者用户执行。这可以通过使用 `CURRENT_USER` 来了解。

带内注入：

`1 AND 1=0 UNION SELECT USER()`

推断注入：

`1 AND USER() like 'root%'`

响应将包含类似以下内容：

`user@hostname`

#### 正在使用的数据库名称

有原生函数 `DATABASE()`

带内注入：

`1 AND 1=0 UNION SELECT DATABASE()`

推断注入：

`1 AND DATABASE() like 'db%'`

> 预期结果，类似这样的字符串：
>
> `dbname`

#### INFORMATION_SCHEMA

从 MySQL 5.0 开始，创建了一个名为 [INFORMATION_SCHEMA](https://dev.mysql.com/doc/refman/8.0/en/information-schema.html) 的视图。它允许我们获取关于数据库、表、列以及过程和函数的所有信息。

| INFORMATION_SCHEMA 中的表 | 描述 |
|------------------------------|-------------|
| SCHEMATA  | 用户（至少）拥有 SELECT_priv 权限的所有数据库 |
| SCHEMA_PRIVILEGES  | 用户对每个数据库拥有的权限 |
| TABLES  | 用户（至少）拥有 SELECT_priv 权限的所有表 |
| TABLE_PRIVILEGES | 用户对每个表拥有的权限 |
| COLUMNS | 用户（至少）拥有 SELECT_priv 权限的所有列 |
| COLUMN_PRIVILEGES | 用户对每个列拥有的权限 |
| VIEWS | 用户（至少）拥有 SELECT_priv 权限的所有列 |
| ROUTINES | 过程和函数（需要 EXECUTE_priv） |
| TRIGGERS | 触发器（需要 INSERT_priv） |
| USER_PRIVILEGES | 连接用户拥有的权限 |

所有这些信息都可以使用 SQL 注入部分中描述的已知技术提取。

### 攻击向量

#### 写入文件

如果连接的用户具有 `FILE` 权限且单引号未被转义，则可以使用 `into outfile` 子句将查询结果导出到文件中。

`Select * from table into outfile '/tmp/file'`

注意：无法绕过文件名周围的单引号。因此，如果对单引号进行了某种清理（如转义 `\'`），则无法使用 `into outfile` 子句。

这种攻击可用作带外技术，以获取查询结果的信息，或写入可在 Web 服务器目录内执行的文件。

示例：

`1 limit 1 into outfile '/var/www/root/test.jsp' FIELDS ENCLOSED BY '//'  LINES TERMINATED BY '\n<%jsp code here%>';`

> 结果以 `rw-rw-rw` 权限存储，由 MySQL 用户和组拥有。
>
> 其中 `/var/www/root/test.jsp` 将包含：
>
> `//field values//`
> `<%jsp code here%>`

#### 从文件读取

`load_file` 是一个原生函数，在文件系统权限允许的情况下可以读取文件。如果连接的用户具有 `FILE` 权限，则可以使用它来获取文件内容。可以使用先前描述的技术绕过单引号转义清理。

`load_file('filename')`

> 整个文件将可通过标准技术导出。

### 标准 SQL 注入攻击

在标准 SQL 注入中，结果可以直接在页面中显示为正常输出或 MySQL 错误。通过使用前面提到的 SQL 注入攻击和已描述的 MySQL 功能，直接 SQL 注入可以很容易地实现，其深度主要取决于渗透测试人员所面对的 MySQL 版本。

一种有效的攻击是通过强制函数/过程或服务器本身抛出错误来获取结果。MySQL 抛出的错误列表，特别是原生函数抛出的错误，可以在 [MySQL 手册](https://dev.mysql.com/doc/mysql-errors/8.0/en/server-error-reference.html) 中找到。

### 带外 SQL 注入

带外注入可以通过使用 [`into outfile`](#写入文件) 子句来实现。

### 盲 SQL 注入

对于盲 SQL 注入，MySQL 服务器原生提供了一组有用的函数。

- 字符串长度：
    - `LENGTH(str)`
- 从给定字符串中提取子字符串：
    - `SUBSTRING(string, offset, #chars_returned)`
- 基于时间的盲注入：
    - BENCHMARK 和 SLEEP `BENCHMARK(#ofcycles,action_to_be_performed)`
    当基于布尔值的盲注入无法产生任何结果时，benchmark 函数可用于执行定时攻击。
    参见 `SLEEP()`（MySQL > 5.0.x）作为 benchmark 的替代方案。

有关完整列表，请参考 [MySQL 手册](https://dev.mysql.com/doc/refman/8.0/en/functions.html)。
