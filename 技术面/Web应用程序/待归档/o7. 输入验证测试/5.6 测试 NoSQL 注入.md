# 测试 NoSQL 注入

## 概述

NoSQL 数据库提供比传统 SQL 数据库更宽松的一致性约束。通过减少关系约束和一致性检查，NoSQL 数据库通常提供性能和扩展优势。然而，即使不使用传统 SQL 语法，这些数据库仍然可能存在注入漏洞。由于这些 NoSQL 注入攻击可能在[过程语言](https://en.wikipedia.org/wiki/Procedural_programming)中执行，而不是在[声明式 SQL 语言](https://en.wikipedia.org/wiki/Declarative_programming)中执行，其潜在影响比传统 SQL 注入更大。

NoSQL 数据库调用以应用程序的编程语言、自定义 API 调用或按通用约定（如 `XML`、`JSON`、`LINQ` 等）格式化。针对这些规范的恶意输入可能不会触发应用程序的主要清理检查。例如，过滤掉常见的 HTML 特殊字符（如 `< > & ;`）并不能防止针对 JSON API 的攻击，因为 JSON 中的特殊字符包括 `/ { } :`。

有数百种 NoSQL 数据库可供应用程序使用，提供各种语言和关系模型的 API。每种都提供不同的功能和限制。由于它们之间没有通用语言，示例注入代码并不适用于所有 NoSQL 数据库。因此，任何测试 NoSQL 注入攻击的人员都需要熟悉其语法、数据模型和底层编程语言，以便定制特定的测试。

NoSQL 注入攻击可能在应用程序的不同区域执行，这与传统 SQL 注入不同。SQL 注入会在数据库引擎内执行，而 NoSQL 变体可能在应用层或数据库层执行，具体取决于所使用的 NoSQL API 和数据模型。通常，NoSQL 注入攻击会在攻击字符串被解析、评估或拼接到 NoSQL API 调用时执行。

额外的时间攻击可能与 NoSQL 数据库缺少并发检查有关。这些不在注入测试的范围内。在撰写本文时，MongoDB 是最广泛使用的 NoSQL 数据库，因此所有示例将使用 MongoDB API。

## 如何测试

### 测试 MongoDB 中的 NoSQL 注入漏洞

MongoDB API 期望 BSON（二进制 JSON）调用，并包含一个安全的 BSON 查询组装工具。然而，根据 MongoDB 文档，未序列化的 JSON 和 [JavaScript 表达式](https://docs.mongodb.org/manual/faq/developers/#javascript) 在多个替代查询参数中是允许的。最常用的允许任意 JavaScript 输入的 API 调用是 `$where` 运算符。

MongoDB `$where` 运算符通常用作简单的过滤器或检查，类似于 SQL 中的用法。

`db.myCollection.find( { $where: "this.credits == this.debits" } );`

也可以选择评估 JavaScript 以允许更高级的条件。

`db.myCollection.find( { $where: function() { return obj.credits - obj.debits < 0; } } );`

### 示例 1

如果攻击者能够操纵传递给 `$where` 运算符的数据，该攻击者可以包含任意 JavaScript 作为 MongoDB 查询的一部分进行评估。如果用户输入未经清理直接传递给 MongoDB 查询，以下代码会暴露一个示例漏洞。

`db.myCollection.find( { active: true, $where: function() { return obj.credits - obj.debits < $userInput; } } );`

与测试其他类型的注入一样，无需完全利用漏洞来证明问题。通过注入与目标 API 语言相关的特殊字符并观察结果，测试人员可以确定应用程序是否正确清理了输入。例如，在 MongoDB 中，如果包含以下任何特殊字符的字符串未经过清理传递，将触发数据库错误。

`' " \ ; { }`

对于正常的 SQL 注入，类似的漏洞将允许攻击者执行任意 SQL 命令——随意暴露或操纵数据。然而，由于 JavaScript 是一种功能齐全的语言，这不仅允许攻击者操纵数据，还可以运行任意代码。例如，在测试时，除了引发错误之外，完整的利用将使用特殊字符构造有效的 JavaScript。

将输入 `0;var date=new Date(); do{curDate = new Date();}while(curDate-date<10000)` 插入到上述示例代码的 `$userInput` 中，将导致执行以下 JavaScript 函数。这个特定的攻击字符串将导致整个 MongoDB 实例在 10 秒内以 100% CPU 使用率运行。

`function() { return obj.credits - obj.debits < 0;var date=new Date(); do{curDate = new Date();}while(curDate-date<10000); }`

### 示例 2

即使查询中使用的输入被完全清理或参数化，还有一种替代途径可能触发 NoSQL 注入。许多 NoSQL 实例有自己的保留变量名，独立于应用程序编程语言。

例如，在 MongoDB 中，`$where` 语法本身就是一个保留的查询运算符。它需要按原样传递到查询中；任何更改都会导致数据库错误。然而，由于 `$where` 也是有效的 PHP 变量名，攻击者可能通过创建一个名为 `$where` 的 PHP 变量将代码插入查询。PHP MongoDB 文档明确警告开发人员：

请确保对于所有特殊查询运算符（以 `$` 开头），您使用单引号，以便 PHP 不会尝试用变量 `$exists` 的值替换 `$exists`。

即使查询不依赖于任何用户输入，如以下示例，攻击者也可以通过将运算符替换为恶意数据来利用 MongoDB。

`db.myCollection.find( { $where: function() { return obj.credits - obj.debits < 0; } } );`

一种潜在向 PHP 变量分配数据的方法是通过 HTTP 参数污染（参见：[测试 HTTP 参数污染](04-Testing_for_HTTP_Parameter_Pollution.md)）。通过参数污染创建名为 `$where` 的变量，可以触发 MongoDB 错误，指示查询不再有效。任何不是字符串 `$where` 本身的 `$where` 值都足以证明漏洞。攻击者将通过插入以下内容开发完整的利用：

`$where: function() { //arbitrary JavaScript here }`
