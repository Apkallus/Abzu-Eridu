
# 客户端模板注入

客户端模板引擎允许开发人员将 HTML 的结构与用 JavaScript 编写的逻辑分离。开发人员可以为常见的网页元素（如标题、页脚等）创建模板；这些模板可以在应用程序的不同部分重用，从而减少冗余。这些模板引擎允许动态渲染内容。这样，就可以根据用户交互或事件更改内容，而无需完全重新加载页面。

当用户提供的输入被不适当地混入 Web 模板时，就会产生客户端模板注入（CSTI）漏洞。此问题经常出现在使用客户端模板框架（如 Angular、React 和 Vue.js）的 Web 应用程序中，这些框架将用户输入动态纳入渲染过程。如果这些输入未经正确清理，它们可能会被模板引擎解释为模板代码的一部分，从而可能导致 XSS。

## 什么是客户端模板注入？

当使用客户端模板框架的应用程序将用户输入动态嵌入到网页中时，就会出现客户端模板注入漏洞。在渲染页面时，框架会扫描页面中的模板表达式并执行它遇到的任何表达式。攻击者可以通过提供一个恶意模板表达式来利用这一点，从而发起跨站脚本（XSS）攻击。

## AngularJS 中的 XSS - 历史

AngularJS 的初始版本实现了沙盒来防御 XSS 漏洞。沙盒通过阻止访问全局对象和属性（如 `window` 和 `document`）以及其他潜在危险属性来工作。在这种情况下，如果用户尝试执行 `alert` 函数，它将被限定在本地范围，不会影响全局环境。由于沙盒通过限制对全局属性（如 `window`）的访问来工作，沙盒绕过主要围绕寻找执行 JavaScript 的替代方法。

然而，随着时间的推移，这些沙盒被反复绕过，并最终因为维护安全沙盒的挑战而被放弃，因此 Angular 的后续版本放弃了沙盒方法，并遭遇了与其他 XSS 保护机制（如 IE XSS Filter 和 Chrome XSS Auditor）相同的命运。

相反，重点转向了替代安全机制，如自动转义输入和严格的上下文转义，以安全地处理用户输入和表达式评估 [https://angular.io/guide/security]。

如果使用了像 AngularJS 这样的框架，则可能在没有尖括号或事件的情况下执行 JavaScript。当站点在 HTML 元素上使用 `ng-app` 属性时，AngularJS 的数据绑定的功能将搜索并执行作为模板的双花括号 `{{}}` 内的 JavaScript，这些花括号可以直接出现在 HTML 或属性中。

因此，通过使用诸如 `{{7*7}}` 的表达式结果是否为运算后的 `49`，以评估应用程序是否解析 Angular 表达式。

以下是 AngularJS 版本 1.7.7 的向上原型链遍历最终构造匿名函数的载荷：

```javascript
// 作用域对象 -> 初始原型对象
{{constructor.constructor('alert()')()}}

// 作用域对象 -> 初始原型对象
{{$on.constructor('alert()')()}}

// 字符串实例 -> 字符串原型对象 -> 初始原型对象
{{''.constructor.constructor('alert(1)')()}}
```

这些载荷为相同核心的变体：多次链式调用构造对象 `constructor` 来向上遍历原型链，最终得到初始构造函数。于是，字符串 `alert()` 作为参数传递给函数构造函数，从而创建了一个匿名函数

## 什么是 AngularJS 沙箱？

AngularJS 沙箱是一种机制，用于防止在 AngularJS 模板表达式中访问潜在危险的对象，如 `window` 或 `document`。它还会阻止访问潜在危险的属性，如 `__proto__`。尽管 AngularJS 团队并不将其视为安全边界，但更广泛的开发者社区通常持相反意见。虽然一开始绕过沙箱颇具挑战性，但安全研究人员已经发现了多种方法。因此，它最终在 AngularJS 1.6 版本中被移除。然而，许多遗留应用程序仍在使用旧版本的 AngularJS，因此可能仍然存在漏洞。

### AngularJS 沙箱如何工作？

沙箱的工作原理是解析表达式、重写 JavaScript，然后使用各种函数测试重写后的代码是否包含任何危险对象。例如，`ensureSafeObject()` 函数检查给定对象是否引用自身。这是检测 `window` 对象的一种方式。`Function` 构造器以大致相同的方式被检测，即检查 `constructor` 属性是否引用自身。

`ensureSafeMemberName()` 函数检查对象的每个属性访问，如果包含危险属性（如 `__proto__` 或 `__lookupGetter__`），该对象将被阻止。`ensureSafeFunction()` 函数阻止 `call()`、`apply()`、`bind()` 或 `constructor()` 被调用。

您可以通过访问 [此 fiddle](http://jsfiddle.net/2zs2yv7o/1/) 并在 `angular.js` 文件的第 13275 行设置断点来亲自查看沙箱的运行情况。变量 `fnString` 包含您重写后的代码，因此您可以查看 AngularJS 如何转换它。

### AngularJS 沙箱逃逸如何工作？

沙箱逃逸涉及欺骗沙箱，使其认为恶意表达式是无害的。最著名的逃逸方式是在表达式中全局使用修改后的 `charAt()` 函数：

```javascript
'a'.constructor.prototype.charAt = [].join
```

当它最初被发现时，AngularJS 并未阻止此修改。该攻击通过使用 `[].join` 方法覆盖函数来工作，这导致 `charAt()` 函数返回发送给它的所有字符，而不是特定的单个字符。由于 AngularJS 中 `isIdent()` 函数的逻辑，它会将所认为的单个字符与多个字符进行比较。由于较长字符串的第一个字符在被比较字符的范围内，`isIdent()` 函数总是返回 `true`，如下例所示：

```javascript
isIdent = function(ch) {
    return ('a' <= ch && ch <= 'z' || 'A' <= ch && ch <= 'Z' || '_' === ch || ch === '$');
}
isIdent('x9=9a9l9e9r9t9(919)')
```
- 每次 `charAt()` 尝试对字符串中的特定索引字符进行检测时，由于被覆盖为 `[].join` 导致返回整个字符串
- 单字符与多字符字符串的比较按照字典顺序，当多字符字符串的第一个字符满足时立即结束比较（此时单字符无后续字符，完成比较）
- 设置载荷的第一个字符为合法字符后，其余内容为任意载荷

一旦 `isIdent()` 函数被欺骗，您就可以注入恶意 JavaScript。例如，诸如 `$eval('x=alert(1)')` 这样的表达式将被允许，因为 AngularJS 将每个字符都视为标识符。请注意，我们需要使用 AngularJS 的 `$eval()` 函数，因为覆盖 `charAt()` 函数只会在沙箱代码执行后生效。此技术将绕过沙箱并允许任意 JavaScript 执行。

### 构造高级 AngularJS 沙箱逃逸

AngularJS 的 `$parse` 函数将参数字符串解析为取值函数
- `$parse(key)(dict)` 相当于 `dict[key]`
    - 若可控制字符串对象 `key` 的值，则载荷将被 `$parse` 解析执行
    - 需确保 `dict` 非空

若站点对字符进行更严格的限制，如阻止使用双引号或单引号。此时，需使用诸如 `String.fromCharCode()` 之类的函数来生成字符。虽然 AngularJS 阻止在表达式中访问 `String` 构造器，但您可以通过使用字符串实例的构造 `constructor` 属性（`''.constructor` 即访问 `String()` 对象）来绕过这一点。这显然需要一个字符串，因此要构造此类攻击，您需要找到一种不使用单引号或双引号来创建字符串的方法。

创建字符串实例的方式：
- `toString()` 返回固定字符串 `"[object Undefined]"`
- 列表合并元素为字符串 `[].join()` 返回 `""` 空字符对象
- 加法运算似乎在其中一个操作数拥有 `toString()` 方法是进行字符串运算
    - `[]+[]` => `""`
    - `1+[]` => `"1"`
    - `window+[]` => `"[object Window]"`
    - `[]+{}` => `"[object Object]" `
    - `window+1` => `"[object Window]1"` 
- `[].toLocaleString()` => `""`
- 可能报错的受限方式
    - `(/正则表达式/).source`
    - ``` `` ``` 模板字符串

在标准的沙箱逃逸中，您会使用 `$eval()` 来执行您的 JavaScript 载荷。若 `$eval()` 函数被阻止，使用 `orderBy` 过滤器作为替代。`orderBy` 过滤器的典型语法如下：

```js
[123]|orderBy:'字符串表达式'

[123]|orderBy:String().fromCharCode(字符串对应的 ASCII 码)
```

请注意，`|` 运算符的含义与 JavaScript 中不同。通常，这是按位 `OR` 操作，但在 AngularJS 中，它表示过滤器操作。在上面的代码中，我们将左侧的数组 `[123]` 发送给右侧的 `orderBy` 过滤器。冒号表示要发送给过滤器的参数，在此例中是一个字符串。`orderBy` 过滤器通常用于对对象进行排序，但它也接受表达式，这意味着我们可以使用它来传递载荷。

### AngularJS CSP 绕过如何工作？

内容安全策略（CSP）绕过的工作方式与标准沙箱逃逸类似，但通常涉及一些 HTML 注入。当在 AngularJS 中激活 CSP 模式时，它会以不同方式解析模板表达式，并避免使用 `Function` 构造器。这意味着上述标准沙箱逃逸将不再有效。

根据具体策略，CSP 会阻止 JavaScript 事件。
- AngularJS 定义了自己可用的事件，当在其事件内部时，其特殊的 `$event` 对象简单地引用浏览器事件对象。可使用此对象执行 CSP 绕过：
    - 在 Chrome 上，`$event/event` 对象上的特殊属性 `path` 是一个包含事件冒泡路径上所有 DOM 元素的对象数组，而最后一个属性始终是 `window` 对象。
- 将此数组传递给 `orderBy` 过滤器，以枚举该数组并使用最后一个元素来执行全局函数
    - 数组元素遍历时，`orderBy` 过滤器将执行上下文 `this` 设置为当前元素，到最后的元素时 `this` 即为 `window`

```html
<input autofocus ng-focus="$event.path|orderBy:'[].constructor.from([1],alert)'">
```

注意，这里使用了 `Array().from()` 函数，它允许您将对象转换为数组，并对该数组的每个元素调用给定的函数（在第二个参数中指定）。在此例中，我们调用了 `alert()` 函数。我们不能直接调用该函数，因为 AngularJS 沙箱会解析代码并检测到正在使用 `window` 对象来调用函数。使用 `from()` 函数可以有效地将 `window` 对象隐藏起来，使其不被沙箱发现，从而允许我们注入恶意代码（作为回调函数）。

PortSwigger Research [使用此技术创建了一个仅 56 个字符的 AngularJS CSP 绕过](https://portswigger.net/research/angularjs-csp-bypass-in-56-characters)：

```html
<input id=x ng-focus=$event.path|orderBy:'(y=alert)(1)'>

<!-- 自 Chrome 109 起，`path` 属性已被移除。替代方案是使用 `composedPath()`-->
<input id=x ng-focus=$event.composedPath()|orderBy:'(y=alert)(1)'>
```
- AngularJS 的聚焦事件 `ng-focus` 使用 URL hash 标签聚焦触发

另一种方法是使用 `array.map()` 函数，如下所示：

```javascript
[1].map(alert)
```

`map()` 接受一个函数作为参数，并会对数组中的每个项调用该函数。这将绕过沙箱，因为对 `alert()` 函数的引用是在没有显式引用 `window` 的情况下使用的。

尝试各种方式来执行 `alert()` 而不触发 AngularJS 的 `window` 检测。

## ReactJS 中的 XSS

ReactJS 与 AngularJS 不同，它不使用沙盒来保证安全。相反，它依赖于严格的上下文转义和编码来防止 XSS 攻击。React 自动转义插入 DOM 的所有字符串，将潜在危险字符转换为其安全的编码等效项。这确保了任何输入都被视为纯文本而不是可执行代码。

然而，ReactJS 确实提供了一种通过特定函数直接向 DOM 插入 HTML 内容的机制。这类函数的一个示例是 `dangerouslySetInnerHTML`，它设置 DOM 元素的 `innerHTML` 属性。"dangerously" 这个名称本身就是一个警告，表明应谨慎使用。因此，通过 `dangerouslySetInnerHTML` 分配用户提供的输入可能存在风险：

**示例**
```jsx
<div dangerouslySetInnerHTML={{__html: '<script>alert(1);</script>'}} />
```

在此示例中，`dangerouslySetInnerHTML` 用于插入原始 HTML，如果未经适当清理，可能包含恶意脚本。

有人可能问为什么 ReactJS 不消除此函数而仅依赖安全函数。原因在于，存在开发人员需要插入 HTML 内容的合法情况。这包括与动态生成 HTML 的第三方库集成，或使用富格式化的 WYSIWYG 编辑器等场景。有时，也出于性能原因选择此方法。该方法本身并不危险；当用户提供的输入未经适当清理或验证而处理时，才会变得危险。

类似地，虽然 React 在许多上下文中确实会自动转义值以防止 XSS，但其自动转义并非在所有上下文中都有效。例如，考虑以下场景，用户提供的输入在锚点标签的上下文中使用：

**示例**
```jsx
<a href={data} className="tmgm">Click Here</a>
```

在此情况下，如果用户提供 XSS 有效载荷如 `javascript:alert(1)`，React 的内置转义将无法阻止 JavaScript 的执行。

## 如何防止客户端模板注入漏洞

为了防止客户端模板注入漏洞，请避免使用不受信任的用户输入来生成模板或表达式。如果这不可行，请考虑在将用户输入嵌入客户端模板之前，过滤掉其中的模板表达式语法。

请注意，HTML 编码不足以防止客户端模板注入攻击，因为框架会在定位和执行模板表达式之前对相关内容执行 HTML 解码。
