## 基于变异的 XSS - 浏览器解析顺序的花哨名称

mXSS 特别因浏览器处理畸形 HTML 的独特方式而发生。当浏览器解析畸形 HTML 时，它们遵循特定的解析规则，并尝试解释和尝试纠正畸形输入的结构。此过程在某些浏览器中称为 "tag soup" 解析。解析器进行有根据的猜测来关闭标签、正确嵌套元素，并创建连贯的 DOM 结构。

mXSS 的核心在于将完全安全和无害的输入转换为危险的内容，当浏览器渲染时导致 XSS 漏洞。

关于 mXSS 最早发表的论文之一题为 "mXSS Attacks: Attacking Well-Secured Web-Applications by Using InnerHTML Mutations" [https://cure53.de/fp170.pdf]。该论文重点关注 `innerHTML` 属性的滥用如何导致 mXSS。

该论文描述 mXSS 的流程：攻击者提供无害输入，该输入通过服务器端 XSS 过滤器或 WAF 传递，然后通过客户端 XSS 过滤器传递，最终到达 `innerHTML` 属性，在那里被突变后发送到渲染引擎——这就是 XSS 发生的地方。

注意：`innerHTML` 属性是一种强大的 JavaScript 方法，用于将原始 HTML 字符串插入元素。当设置此属性时，浏览器解析 HTML 字符串并形成 DOM 树，包括对畸形 HTML 的错误修正。读取 `innerHTML` 然后将 DOM 树序列化回 HTML 字符串，也包括对字符串进行的任何突变。

例如，论文中展示的影响旧版 IE 的 mXSS 攻击向量之一如下：

**有效载荷**
```html
<img style="font-fa\22onerror\3d\61lert\28\31\29\20mily:'arial'"src="x:x" />
```

此向量包含一个 `onerror` 事件处理程序，它位于 style 属性内，因此将被视为字符串。然而，当通过易受攻击的 JavaScript 版本中的 `innerHTML` 属性传递时，浏览器会突变该字符串，导致 `onerror` 作为属性跳出 style 上下文并执行 JavaScript。

**输出**
```html
<IMG style="font-fa"onerror=alert(1)mily:'arial'"src="x:x">
```

在当今世界，现代浏览器已弃用客户端 XSS 过滤器；然而，其余概念基本相同。此外，mXSS 的早期变种主要针对 Internet Explorer；今天的变种则针对其他浏览器。在深入探讨示例之前，让我们先探索浏览器中的突变过程：考虑以下畸形 HTML，其中 `<div>` 标签未正确关闭。

**示例 1：**
```html
<div><p>Example</div</p>
```

浏览器修正畸形 HTML；一旦它通过 `innerHTML`，得到的值就不同了 `<div><p></div></p>`。

让我们再举一个开始和结束标签不匹配的示例。

**示例 2：**
```html
<b>Bold text<i>Italic text</b></i>
```

保存标记并打开 DOM 后，标签已被修正 `<b><i></i></b>`。

诸如 Webmail、CMS、论坛等应用程序通常为用户提供插入 HTML 的功能。这些可能导致畸形输入，尤其是允许插入 HTML 的 WYSIWYG 编辑器。这些可能允许用户输入恶意的 HTML/JavaScript 代码。为了防止这种情况，使用了清理器和净化器。它们通过编码剥离恶意输入或将不安全输入转换为安全等效项来工作。然而，这些清理器/净化器如果未正确实现，可能会促成 mXSS。

在 mXSS 攻击中，攻击者构造一个畸形输入，清理器将其视为安全输入，应用修正并将其处理给 HTML 解析器。然而，一旦此输入被浏览器处理，浏览器的解析逻辑会将其转换为可执行的 JavaScript。让我们看一个现实世界的示例，展示清理器和 HTML 解析器之间的不匹配如何导致 mXSS。

### mXSS Mozilla Bleach Clean 函数 CVE-2020-6802

https://bugzilla.mozilla.org/show_bug.cgi?id=1615315

让我们考虑 Mozilla bleach.clean 函数中一个有趣的 mXSS 漏洞案例。该漏洞是一个案例研究，展示了当清理库未能完全考虑 HTML 解析的复杂性以及不同元素和属性使用的上下文时，如何导致 mXSS 漏洞。让我们看一个通过 bleach.clean 函数传递的有效载荷示例：

**有效载荷**
```html
<noscript><style></noscript><img src=x onmouseover=alert(1)>
```

由 bleach.clean 函数渲染的输出如下：

**清理器渲染的输出**
```html
<noscript><style></noscript><img src=x onmouseover=alert(1)></style></noscript>
```

1. noscript 在 bleach 中作为 HTML 处理（尝试在末尾添加闭合标签），而在浏览器中作为纯文本处理（将其中的 style 开标签作为纯文本）。
2. style 属于原始文本标签（标签内容解析为纯文本），从而在 bleach 中不对其内容进行检测
3. 最终导致 img 作为 HTML 元素渲染，并加载事件监听器

当浏览器的 HTML 解析器处理清理后的输出时，它会删除关闭的 style 和 noscript 标签(`</style></noscript>`)。如前所述，这是 HTML 解析器的常见行为，它将修正畸形 HTML，因此由于解析器的修正，之前不活跃的 `<img>` 标签变为活跃，触发 JavaScript 代码，导致 mXSS。

**HTML 解析器输出**
```html
<noscript><style></noscript><img src=x onmouseover=alert(1)>
```

### mXSS: Google 库 Closure

2019 年初，安全研究员 Masato Kinugawa 发现了一个影响 Google 库 Closure（用于 Google 搜索）的 mXSS。

Masato 使用了一个名为 DOMPurify 的清理库（Closure 使用它来过滤潜在的 XSS 字符串）。DOMPurify 在客户端（浏览器）运行，通过读取字符串，然后才允许其作为 innerHTML 插入来执行过滤。这实际上是通过 innerHTML 注入 DOM 的字符串最有效的清理方式，因为浏览器实现各异，浏览器版本也不同（因此服务端过滤不会那么有效）。通过将 DOMPurify 库交付给客户端并执行评估，Google 期望他们能拥有一个跨新旧浏览器工作的强大 XSS 过滤解决方案。

Masato 使用的载荷包含以下内容：

```html
<noscript><p title="</noscript><img src=x onerror=alert(1)">"
```

从技术上讲，这个载荷应该是 DOM 安全的，因为由于标签和引号的设置方式，直接附加此内容不会导致脚本执行。因此，DOMPurify 让它通过了“不是 XSS 风险”的检查，但当它被加载到浏览器 DOM 中时，DOM 执行了一些优化，使其看起来像这样：

```html
<noscript><p title="</noscript>
<img src="x" onerror="alert(1)">
">"
```

发生这种情况的原因是 DOMPurify 在其清理过程中使用了根元素 `<template>`。`<template>` 标签会被解析但不会渲染，因此非常适合用于清理。

在 `<template>` 标签内部，元素脚本被禁用。当脚本被禁用时，`<noscript>` 标签会表示其子元素，但当脚本启用时它什么也不做。换句话说，`img onerror` 在清理器内部无法执行脚本，但当它通过清理并进入真实的浏览器环境后，`<p title="` 被忽略，而 `img onerror` 变得有效。

总结：浏览器 DOM 元素通常根据其父级、子级和兄弟级条件性地行动。在某些情况下，黑客可以利用这一点，构造 XSS 载荷，通过不作为有效脚本来绕过过滤器——但在实际浏览器中运行时却变成有效脚本。
