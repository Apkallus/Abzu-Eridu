# 基于 DOM 的 XSS

文档对象模型（DOM）是浏览器用于渲染网页的模型，定义每个 HTML 元素的基本属性和行为，提供了 JavaScript 用于动态访问和修改网页内容和结构的接口。

DOM-based XSS 漏洞通常出现在 JavaScript 从攻击者可控制的源（如 URL）获取数据，并将其传递给支持动态代码执行的汇（如 `eval()` 或 `innerHTML`）时（当用户提供的输入未经清理而用于动态更新 DOM）。这使得攻击者能够执行恶意 JavaScript，通常可用来劫持其他用户的账户。

DOM XSS 与其他形式的 XSS 之间的一个关键区别是，DOM 型 XSS 直接针对网页的 DOM，它攻击客户端的本地页面副本，这意味着恶意请求永远不会发送到服务器；因此，服务器端安全措施（如 WAF 和服务器端过滤器）对 DOM XSS 攻击无效。

攻击者的JavaScript通过以下过程得以执行。
- 用户请求一个经过专门设计的URL，它由攻击者提交，且其中包含嵌入式JavaScript。
- 服务器的响应中并不以任何形式包含攻击者的脚本。
- 当用户的浏览器处理这个响应时，上述脚本得以处理。

假设示例使用了以下客户端代码：

```html
<script>
document.write("站点位于：" + document.location.href + "。");
</script>
```

攻击者可以将 `#<script>alert('xss')</script>` 附加到受影响页面的 URL 上，当执行时，将显示警告框。在此实例中，附加的代码不会发送到服务器，因为 `#` 字符之后的所有内容不会被浏览器视为查询的一部分，而是视为片段。

像 jQuery 这样的 JavaScript 库容易受到 DOM 型 XSS 的影响，因为它们会动态更改 DOM 元素。

DOM XSS 可以是反射型或存储型，但利用浏览器 DOM 的汇（sink）和源（source）进行执行。由于不同浏览器的 DOM 实现存在差异，某些浏览器可能存在漏洞，而其他浏览器则没有。通常，源是能够存储文本的 DOM 对象，而汇是能够执行存储为文本的脚本的 DOM API。由于 DOM XSS 从不接触服务器，几乎不可能用静态分析工具或任何其他类型的流行扫描器检测到。

DOM XSS 也难以处理，因为当今使用的浏览器种类繁多。很可能一个浏览器所包含的 DOM 实现中的错误，在另一个浏览器的 DOM 实现中并不存在。

浏览器版本也是如此。2015 年的浏览器版本可能存在漏洞，而现代浏览器可能没有。如果未提供足够的浏览器/操作系统详细信息，试图支持多种浏览器的公司可能难以复现 DOM XSS 攻击。JavaScript 和 DOM 都基于开放规范（TC39 和 WHATWG），但每个浏览器的实现差异很大，且通常因设备而异。

DOM XSS 的概念最初由 Amit Klein 在 2005 年发表的题为 "DOM Based Cross Site Scripting or XSS of the Third Kind" [www.webappsec.org/projects/articles/071105.shtml] 的论文中提出。在发表时，由于网站不严重依赖客户端脚本，该论文并未立即受到欢迎。然而，随着 Web 的发展，变得更加动态并依赖客户端 JavaScript，DOM XSS 的相关性显著增加。

让我们检查一个 mega-bank.com 的 DOM XSS 漏洞。

MegaBank 为其 401(k) 管理服务提供了一个投资门户，位于 investors.mega-bank.com。在 investors.mega-bank.com/listing 内，有一个可通过 401(k) 缴款进行投资的基金列表。左侧导航菜单提供了对这些基金的搜索和筛选功能。

由于基金数量有限，搜索和排序在客户端进行。搜索 “oil” 会将页面 URL 修改为 `investors.mega-bank.com/listing?search=oil`。类似地，筛选 “usa” 以仅查看美国基金，会生成 URL `investors.mega-bank.com/listing#usa`，并自动将页面滚动到美国基金集合。

现在要注意，仅仅因为 URL 改变，并不总是意味着正在向服务器发出请求。在现代 Web 应用中，这种情况更常见，它们使用自己的基于 JavaScript 的路由器，因为这可以带来更好的用户体验。

当我们输入一个恶意搜索查询时，我们不会在这个特定网站上遇到任何异常交互。但重要的是要注意，像 `search` 这样的查询参数可能是 DOM XSS 的源，并且可以通过 `window.location.search` 在所有主流浏览器中找到。

同样，哈希值也可以通过 `window.location.hash` 在 DOM 中找到。这意味着载荷可以注入到搜索查询参数或哈希值中。这些源中的许多危险载荷不会引起任何问题，除非另一段代码实际以可能导致脚本执行的方式使用它——因此需要同时具备“源”和“汇”。

让我们想象 MegaBank 在同一页面中有以下代码：

```javascript
/*
 * 从 URL 中获取哈希对象 #<x>。
 * 使用 findNumberOfMatches() 函数查找所有匹配项，
 * 将哈希值作为输入提供。
 */
const hash = document.location.hash;
const funds = [];
const nMatches = findNumberOfMatches(funds, hash);

/*
 * 写入找到的匹配数，并将哈希值附加到 DOM 以改善用户体验。
 */
document.write(nMatches + ' matches found for ' + hash);
```

这里我们利用源（`window.location.hash`）的值来生成一些要显示给用户的文本。这是通过汇（`document.write`）完成的，但也可以通过许多其他汇完成，有些比其他需要更多或更少的努力。

假设我们生成了一个如下链接：

`investors.mega-bank.com/listing#<script>alert(document.cookie);</script>`

一旦注入 DOM 并被解释为 `<script>` 标签，`document.write()` 调用将导致此哈希值作为脚本执行。这将显示当前会话 cookie，但正如我们在之前的 XSS 示例中看到的，它可能做许多有害的事情。

## XSS 汇与源

可利用的 XSS 漏洞由两个关键组件组成。
1. **汇（sink）**：可能不安全的函数，可用于动态生成 HTML 或 JavaScript 而未经清理。
2. **源（source）**：用户输入

除了最常见的汇和源之外，还有很少使用的汇和源，它们更可能被利用，并且更不容易受到过滤或清理的影响。

## 如何测试基于 DOM 的跨站脚本

JavaScript 应用程序与其他类型的应用程序有显著不同，因为它们通常由服务器动态生成。为了理解正在执行的代码，需要爬取被测网站，以确定所有正在执行的 JavaScript 实例以及接受用户输入的位置。许多网站依赖大型函数库，这些函数库通常包含数十万行代码，且并非内部开发。在这些情况下，自上而下（从输入的源追踪数据流走向）的测试通常成为唯一可行的选择，因为许多底层函数从未被使用，而分析它们以确定哪些是汇将消耗比通常可用时间更多的时间。

用户输入主要有两种形式：

- 由服务器处理后写入页面的输入，处理方式未知

    ```js
    var data = "<服务器转义的数据>";
    var result = someFunction("<服务器转义的数据>");
    ```

- 从客户端 JavaScript 对象获取的输入，处理方式可见

    ```js
    var data = window.location;
    var result = someFunction(window.referrer);
    ```

Stefano Di Paulo 撰写了一篇 URL 各部分与 DOM 元素映射的文章 (https://github.com/wisec/domxsswiki/wiki/location,-documentURI-and-URL-sources)


### DOM XSS 与反射和存储数据的结合

某些纯基于 DOM 的漏洞是自包含在单个页面中的。如果脚本从 URL 读取某些数据并将其写入危险汇，则该漏洞完全是客户端的。

然而，源不仅限于浏览器直接暴露的数据——它们也可以来自网站本身。例如，网站通常会在来自服务器的 HTML 响应中反射 URL 参数。这通常与普通 XSS 相关，但也可能导致反射型 DOM XSS 漏洞。

在反射型 DOM XSS 漏洞中，服务器处理请求中的数据，并将数据回显到响应中。反射的数据可能被放入 JavaScript 字符串字面量中，或 DOM 中的数据项（如表单字段）中。随后页面上的脚本以不安全的方式处理反射的数据，最终将其写入危险的汇。

```javascript
eval('var data = "reflected string"');
```

网站也可能在服务器上存储数据并在其他地方反射它。在存储型 DOM XSS 漏洞中，服务器从一个请求接收数据，存储它，然后将数据包含在后续响应中。后续响应中的脚本包含一个汇，它以不安全的方式处理数据。

```javascript
element.innerHTML = comment.author
```

若服务器端实施旨在阻止基于DOM的XSS攻击的过滤，除基础绕过方式外
- 尝试将载荷插入虚构参数中，若服务器与客户端的过滤-使用不匹配（服务器针对特定参数，客户端取用整段 URL），则可能成功
- 尝试将载荷插入到 HTML 片断字符`#`的右边（浏览器并不将 URL 中的片断字符串提交到服务器）

### 测试 HTML 汇

要测试 HTML 汇中的 DOM XSS，请将一个普通的随机字母数字字符串放入源（如 `location.search`），然后使用开发者工具检查 HTML，找到字符串出现的位置。请注意，浏览器的“查看源代码”选项不适用于 JS 的 DOM XSS 测试，因为它不会考虑 JavaScript 对 HTML 所做的更改。在 Chrome 的开发者工具中，您可以使用 `Control+F`（MacOS 上为 `Command+F`）在 DOM 中搜索您的字符串。

对于字符串在 DOM 中出现的每个位置，您需要识别上下文。基于此上下文，您需要调整输入以查看其处理方式。例如，如果字符串出现在双引号属性内，请尝试在字符串中注入双引号，看是否能够突破该属性。

请注意，浏览器在 URL 编码方面的行为有所不同。Chrome、Firefox 和 Safari 会对 `location.search` 和 `location.hash` 进行 URL 编码。如果您的数据在处理前被 URL 编码，那么 XSS 攻击不太可能成功。

### 测试 JavaScript 执行汇

测试 JavaScript 执行汇中的 DOM XSS 稍微困难一些。对于这些汇，您的输入不一定出现在 DOM 中的任何位置，因此您无法搜索它。相反，您需要使用 JavaScript 调试器来确定您的输入是否以及如何被发送到汇。

对于每个潜在的源（如 `location`），您首先需要在页面的 JavaScript 代码中找到引用该源的情况。在 Chrome 的开发者工具中，您可以使用 `Control+Shift+F`（MacOS 上为 `Command+Alt+F`）在所有页面 JavaScript 代码中搜索该源。

一旦找到读取源的位置，您可以使用 JavaScript 调试器添加断点，并跟踪源值的使用方式。您可能会发现该源被赋值给其他变量。如果是这种情况，您需要再次使用搜索功能来跟踪这些变量，并检查它们是否被传递给汇。当您发现某个汇被赋了来自源的数据时，您可以使用调试器悬停变量以查看其在发送到汇之前的值的。然后，与 HTML 汇类似，您需要调整输入，看能否成功实施 XSS 攻击。

在实际环境中识别和利用 DOM XSS 可能是一个繁琐的过程，通常需要手动遍历复杂且压缩的 JavaScript。但是，如果您使用 Burp 的浏览器，则可以利用其内置的 DOM Invader 扩展，它会为您完成大量繁重的工作。

### 识别 JavaScript 的执行位置

JavaScript 不仅在 `<script>` 块中执行，还可能出现在：

- **事件处理器**（如 `onload`、`onclick`）
- **CSS 表达式属性**（如 `expression`）
- **外部 CSS 或脚本对象**（需要进一步评估其中执行的代码）

爬取应用时，务必覆盖这些位置。

### 自动化测试的局限性

自动化测试在识别和验证基于 DOM 的 XSS 方面只有非常有限的效果，因为它通常通过发送特定载荷并尝试在服务器响应中观察它来识别 XSS。这对于下面提供的简单示例可能效果很好，其中 message 参数被反射回用户：

```html
<script>
var pos=document.URL.indexOf("message=")+5;
document.write(document.URL.substring(pos,document.URL.length));
</script>
```

然而，在以下人为构造的案例中可能无法检测到：

```html
<script>
var navAgt = navigator.userAgent;

if (navAgt.indexOf("MSIE")!=-1) {
        document.write("您正在使用 IE 作为浏览器访问站点：" + document.location.href + "。");
}
else
{
    document.write("您正在使用未知浏览器。");
}
</script>
```

因此，除非测试工具能够对客户端代码执行额外的分析，否则自动化测试将无法检测到可能容易受到基于 DOM 的 XSS 攻击的区域。

因此，应进行手动测试，可以通过检查代码中引用可能对攻击者有用的参数的区域来完成。此类区域的示例包括
- 动态写入页面的代码
- 修改 DOM 的位置
- 直接执行脚本的位置。

### 利用不同源和汇的 DOM XSS

原则上，如果存在一条从源到汇的可执行路径，则网站容易受到基于 DOM 的跨站脚本攻击。实际上，不同的源和汇具有不同的属性和行为，这些会影响可利用性，并决定需要使用哪些技术。此外，网站的脚本可能对数据执行验证或其他处理，在尝试利用漏洞时必须考虑这些因素。有多种与基于 DOM 的漏洞相关的汇。

`document.write` 汇可与 `script` 元素配合使用，因此您可以使用简单的载荷，如下所示：

```javascript
document.write('... <script>alert(document.domain)</script> ...');
```

但请注意，在某些情况下，写入 `document.write` 的内容包含一些周围的上下文，您需要在漏洞利用中加以考虑。例如，您可能需要在 JavaScript 载荷之前关闭一些现有元素。
- PortSwigger 使用 select option 元素作为上下文，然而实际上无需逃逸即执行所有 JS 载荷(script 标签、svg onload、img onerror)

`innerHTML` 汇在任何现代浏览器上都不接受 `script` 元素，也不会触发 `svg onload` 事件。这意味着您需要使用其他元素，如 `img` 或 `iframe`。事件处理程序（如 `onload` 和 `onerror`）可与这些元素结合使用。例如：

```javascript
element.innerHTML='... <img src=1 onerror=alert(document.domain)> ...'
```
- PortSwigger 使用 innerHTML 作为汇，然而实际上仅 script 标签 无法执行，svg onload、img onerror 均可执行

## 客户端 XSS 练习 

来自 "DOMGoat" [https://domgo.at/] 的 DOM XSS 示例，该平台包含多个由各种来源/汇点组成的易受 DOM XSS 攻击的场景。

**客户端 XSS 练习 1**

DOMGoat 的第一个练习 [https://domgo.at/cxss/example/1?payload=abcd&sp=x#123452]

包含一个场景，涉及来自 `location.hash` 属性的用户输入，该属性表示浏览器 URL 中井号（`#`）之后的文本。此输入使用 `unescape` 函数解码，然后通过 `innerHTML` 属性（危险汇点）直接注入网页的 HTML 结构中。

**易受攻击的代码：**
```javascript
let hash = location.hash;
if (hash.length > 1) {
    let hashValueToUse = unescape(hash.substr(1));
    let msg = "Welcome " + hashValueToUse + "!!";
    document.getElementById("msgboard").innerHTML = msg;
}
```

POC 非常简单：

**POC**
```
https://domgo.at/cxss/example/1?payload=abcd&sp=x#<img/src=x onerror=prompt(1)>
```

### 根本原因分析

执行有效载荷会在 Chrome 控制台中引发 404 错误，因为 `innerHTML` 将有效载荷解释为图像源，加载失败，从而触发 `onerror` 事件并执行 `prompt(1)`。

检查错误会将我们带到代码的第 178 行，这是易受攻击的代码行。让我们设置一个断点。

刷新页面后，执行在易受攻击行设置的断点处暂停，允许我们检查 DOM 的当前状态以及用户输入的处理方式。这也使我们能够验证输入在插入 DOM 之前是否被清理。

"作用域"面板显示局部作用域变量（`hashValueToUse` 和 `msg`）的当前值。`hash` 变量包含我们的 XSS 有效载荷，该有效载荷已被浏览器编码。然而，通过使用 `unescape` 函数的 `hashValueToUse` 变量处理后，输入被解码，然后分配给 `msg` 变量，该变量将输入传递给 `innerHTML` 属性。

确定根本原因的另一种方法是使用有效载荷中的 `debugger` 语句，这将导致浏览器调试器在触发代码时暂停执行，这也可能有助于精确定位代码的确切位置。

**有效载荷**
```
https://domgo.at/cxss/example/1?payload=abcd&sp=x#<img src=x onerror=debugger>
```

堆栈跟踪指示发生了错误事件。控制台中可见的 URL 参数 `?payload=abcd&sp=x:` 表明此错误可能与这些参数的处理有关，可能指向代码的第 178 行存在漏洞。

DOM XSS 可以进一步分为两种类型：反射型 DOM XSS 和存储型 DOM XSS。虽然我们已经简要探讨了反射型 DOM XSS，但我们将在 HTML5 章节（第 12 章）中进一步探讨此变体以及第二种变体。同样，DOM XSS 也在 WAF 绕过的上下文中进行了讨论。

## 第三方依赖中的源和汇

现代 Web 应用程序通常使用许多第三方库和框架构建，这些通常为开发人员提供额外的功能。请务必记住，其中一些也可能是 DOM XSS 的潜在源和汇，如果这些库用于使用用户提供的输入动态更新 DOM。所有这些第三方库都有其独特的汇点集。

### jQuery 库 DOM XSS

如果使用了诸如 jQuery 之类的 JavaScript 库，请留意能够更改页面上 DOM 元素的汇。

jQuery 提供了诸如 `.after()`、`.before()`、`.prepend()`、`.replaceWith()` 等方法，可用于将内容插入 DOM。如果这些方法在未清理不受信任数据的情况下使用，它们可能导致 DOM XSS。让我们检查几个针对 jQuery 的示例：

#### jQuery 示例 #1

jQuery 中有几种方法可用于设置元素的各种属性或特性。其中一种方法是 `attr` 函数。如果 `.attr()` 使用不受信任的数据设置这些属性（如 `href`、`src`、事件处理程序属性），则可能导致 XSS。以下 JavaScript 使用 URL 中的数据更改锚点元素的 `href` 属性：

**易受攻击的代码**
```javascript
$(function() {
    $('#back').attr("href", (new URLSearchParams(window.location.search)).get('return'));
});
```

该代码从 URL 的查询字符串中提取 `return` 参数的值，然后 `attr` 选择元素的 `href` 属性后设置为 `return` 参数的值。在 JS 代码应用后，单击返回链接将执行它

**POC：**
```
?return=javascript:alert(document.domain)
```

#### jQuery 示例 #2

另一个潜在汇是 `$()`，它可用于将恶意对象注入 DOM。
[`$()` 不仅作为元素选择器，还作为 HTML 解释器来创建元素](https://bugs.jquery.com/ticket/11290/) 

jQuery 曾经非常流行，一个经典的 DOM XSS 漏洞是由网站将此选择器与 `location.hash` 源结合用于动画或自动滚动到页面上特定元素引起的。这种行为通常使用易受攻击的 `hashchange` 事件处理程序实现，类似于以下内容：

```javascript
$(window).on('hashchange', function() { // 触发事件
    // 脆弱代码，源 location.hash，汇 $()
	var element = $(location.hash); 
	element[0].scrollIntoView();
});
```

由于 hash 是用户可控的，攻击者可以利用它在 `$()` 选择/创建器汇中注入 XSS 向量。较新版本的 jQuery 已通过阻止在输入以哈希字符（`#`）**开头**时向选择/创建器注入 HTML 来修补此特定漏洞。但是，您可能仍然会在实际中遇到易受攻击的代码。

要实际利用此经典漏洞，您需要找到一种无需用户交互即可触发 `hashchange` 事件的方法。最简单的方法之一是通过 `iframe` 传递您的漏洞利用：

```html
<iframe src="https://vulnerable-website.com#" onload="this.src+='<img src=1 onerror=alert(1)>'"></iframe>
```

在此示例中
1. 子框架的 `src` 属性指向带有空哈希值的易受攻击页面
2. 当顶级页面的 `iframe` 加载时，顶级页面的 `onload` 事件触发：
    - XSS 向量被附加到子框架 `src` 属性的 URL 的哈希片段中
3. 子框架重载更新后的 URL，子框架内：
    1. 触发 `hashchange` 事件。
    2. 载荷由源 `location.hash` 进入汇 `$()`，导致 JS 执行

此处攻击者控制的顶级页面修改子框架元素的 src 属性，而无关同源策略中访问子框架 DOM 的限制

> **注意：** 即使更新版本的 jQuery，只要您能完全控制来自不需要 `#` 前缀的源的输入，仍可能通过 `$()` 选择器汇受到攻击。
