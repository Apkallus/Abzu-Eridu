## 通过文件上传实现 XSS

在某些场景中，应用程序可能允许用户上传文件，如 SVG、DOCX 和 PDF 文件，以实现合法功能，并阻止危险扩展名（如 PHP、JSP、ASPx 等）。在这些场景中，如果在渲染文件之前未对输入进行清理，则可能导致 XSS。值得一提的是，一些 Web 应用程序专门为渲染用户上传的文件而实现沙盒域。这种做法显著降低了通过此类文件进行的潜在 XSS 攻击的有效性。

## 通过 SVG 文件实现 XSS

由于 SVG 可以包含 JavaScript 代码，如果应用程序接受 SVG 文件作为输入并在未清理的情况下渲染它，则会导致 XSS。一个场景，一个门户具有文件上传功能，该功能被设计为仅接受图像文件，包括 SVG 文件。由于 SVG 文件可以包含 JavaScript，使用以下有效载荷：

**有效载荷：**
```xml
<?xml version="1.0" standalone="no"?>
<!DOCTYPE svg PUBLIC "-//W3C//DTD SVG 1.1//EN" "www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg version="1.1" baseProfile="full" xmlns="www.w3.org/2000/svg">
    <polygon id="triangle" points="0,0 0,50 50,0" fill="#009900" stroke="#004400"/>
    <script type="text/javascript">
        alert(document.domain);
    </script>
</svg>
```

一些 Web 应用使用沙盒域来渲染上传的文件；在这种情况下，其有效性受到严重限制。

## 通过文件元数据实现 XSS

即使应用程序将文件上传限制为安全格式（如 JPG、PNG 和 GIF 文件），仍然存在一些攻击向量，具体取决于应用程序的逻辑和上下文。其中一个例子是当应用程序处理和显示图像元数据（如 EXIF 头）时。如果应用程序未清理此元数据，则可能导致 XSS。

场景，应用程序中的一个端点允许用户上传个人资料图片。该端点接受安全格式（如 JPG 和 PNG），并在未清理的情况下反射 EXIF 数据。然而，有效载荷并未执行，因为响应被当作图像处理。

进一步检查发现，应用程序依赖于请求中设置的 `Content-Type` 来确定响应。因此，通过操纵内容类型，我们能够将页面渲染为 HTML，从而执行嵌入在 EXIF 头中的 XSS 向量。

为了演示这一发现，我们创建了一个包含 XSS 有效载荷的 JPG 图像，并使用 exiftool 将其上传到服务器。

**命令：**
```bash
exiftool download.jpeg -Comment='<script>alert(document.domain)</script>'
```
