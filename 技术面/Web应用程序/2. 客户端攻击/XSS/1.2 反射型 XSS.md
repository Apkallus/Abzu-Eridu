## 反射型 XSS

反射型 XSS，也称为非持久性 XSS，当用户提供的输入被服务器直接回显在响应中时发生。此漏洞源于服务器端代码在返回输入给用户之前未能进行清理。由于利用这种漏洞需要设计一个包含嵌入式 JavaScript 代码的请求，随后这些代码又被反射到任何提出请求的用户，因而它被称作反射型 XSS。

我们再次成为虚构银行 mega-bank.com 的客户。这次，我们想查找有关如何开设新的储蓄账户以补充现有支票账户的支持文档。幸运的是，mega-bank.com 的支持门户 support.mega-bank.com 有一个搜索栏，我们可以用来查找常见的支持请求及其解决方案。

我们首先尝试搜索 
  `open savings account`。
- 此搜索将我们重定向到新的 URL：
  `support.mega-bank.com/search?query=open+savings+account`。
- 在此搜索结果页面上，我们看到标题：
  `3 results for “open savings account”`。

此时初步猜测应用程序的搜索功能在 URL 参数中接收用户提供的搜索词，并将该搜索词原样回显到响应的页面中

接下来，我们尝试将 URL 调整为 
  `support.mega-bank.com/search?query=open+checking+account`。
- 结果页面上的标题现在变为：
  `4 results for “open checking account”`。
  
由此推断，URL 查询参数与结果页面上显示的标题之间存在关联。

让我们尝试向搜索查询中添加一个加粗标签：
  `support.mega-bank.com/search?query=open+<strong>checking</strong>+account`。
  令我们惊讶的是，我们生成的新 URL 确实加粗了结果页面中的标题。

利用这个新发现，让我们在查询参数中包含一个 `<script>` 标签：
  `support.mega-bank.com/search?query=open+<script>alert(test);</script>checking+account`。
  打开此 URL 会加载搜索结果，但首先会弹出一个包含单词 “test” 的警告模态框。

我们在这里发现的是一个 XSS 漏洞——只是这次它不会存储在服务器中。相反，服务器会读取它并将其发送回客户端。这类漏洞被称为“反射型 XSS”。

如果应用程序的另一个用户请求了攻击者构造的 URL，那么攻击者提供的脚本将在受害用户的浏览器中，以该用户与应用程序的会话上下文执行。

在我们的例子中，我们可以构造一个恶意链接载荷，直接发送给我们想要攻击的用户。这可以通过电子邮件、基于 Web 的广告或许多其他方式完成。

此外，我们前面讨论的反射型 XSS 可以很容易地伪装成有效的链接。让我们以这段 HTML 代码片段为例：

```html
Welcome to MegaBank Fans!
Your #1 source for legit MegaBank support info and links.

<a href="https://mega-bank.com/signup">Become a New Customer</a>
<a href="https://mega-bank.com/promos">See Promotional Offers</a>
<a href="https://support.mega-bank.com/search?query=open+<script>alert('test');</script>checking+account">
  Create a New Checking Account</a>
```

这里有三个链接，都有自定义文本。其中两个是合法的。点击最后一个文本为 “Create a New Checking Account” 的链接会进入支持页面。`alert()` 会提示发生了异常，但就像之前的存储型 XSS 示例一样，我们可以很容易地在后台执行一些代码。也许我们可以找到足够的客户信息来冒充用户，或者如果支持门户 UI 中存在支票/路由号码，我们也可以获取它。

这种反射型 XSS 依赖于 URL，这使得攻击者很容易分发。大多数反射型 XSS 不会这么容易分发，可能需要最终用户执行额外操作，比如将 JavaScript 粘贴到 Web 表单中。

### 易受攻击的 PHP 代码示例

**易受攻击的代码：**
```php
<?php
if ($_SERVER["REQUEST_METHOD"] == "GET" && isset($_GET['x'])) {
    echo $_GET['x'];
}
?>
```

该代码通过 GET 参数 `x` 接收输入，并使用 `echo` 直接返回。

**示例**

由于此过程中没有进行清理或编码，任何用户提供的输入（包括 HTML 标签和 JavaScript）都将被视为应用程序的一部分。因此，提供像 `<script>alert(document.domain)</script>` 这样的有效载荷将在应用程序上下文中执行 JavaScript。

**有效载荷：**
```
http://xss-labs.com/?x=<script>alert(document.domain)</script>
```


## 反射型 XSS 攻击的影响

如果攻击者可以控制在受害者浏览器中执行的脚本，那么他们通常可以完全控制该用户。攻击者可以：

- 在应用程序内执行用户能够执行的任何操作。
- 查看用户能够查看的任何信息。
- 修改用户能够修改的任何信息。
- 与其他应用程序用户发起交互（包括恶意攻击），这些交互将看似来自初始受害用户。

攻击者可以通过多种方式诱导受害用户访问其控制的请求，以实施反射型 XSS 攻击。这些方式包括：在攻击者控制的网站上放置链接，或在允许生成内容的其他网站上放置链接，或通过电子邮件、推文或其他消息发送链接。攻击可以直接针对已知用户，也可以是对应用程序所有用户的无差别攻击。

由于攻击需要外部传递机制，反射型 XSS 的影响通常低于存储型 XSS，因为存储型 XSS 可以在易受攻击的应用程序自身内独立完成攻击。


## 不同上下文中的反射型 XSS

反射型跨站脚本有许多不同类型。反射数据在应用程序响应中的位置决定了需要哪种类型的载荷来利用它，并且还可能影响漏洞的影响。

此外，如果应用程序在反射数据之前对提交的数据执行了任何验证或其他处理，这通常会影响所需的 XSS 载荷类型。

## 如何发现和测试反射型 XSS 漏洞

要探查反射型XSS漏洞，最可靠的方法是系统性地检查在解析应用程序过程中确定的所有用户输入进入点，并遵循以下步骤。

- 在每个进入点提交一个良性字母字符串。
- 确定此字符串“反射”在应用程序响应中的所有位置。
- 对于每个反射，确定显示反射型数据时的语法上下文。
- 提交针对反射的语法上下文而修改的数据，尝试在响应中引入任意脚本。
- 如果反射型数据被阻止或净化，导致脚本无法执行，则尝试了解并避开应用程序的防御性过滤。

手动测试反射型 XSS 漏洞包括以下步骤：

- **测试每个入口点。** 分别测试应用程序 HTTP 请求中的每个数据入口点，包括 URL 查询字符串、消息正文中的参数或其他数据，以及 URL 文件路径。也包括 HTTP 标头，但仅能通过某些 HTTP 标头触发的类似 XSS 的行为可能在实际中无法利用。

- **提交随机字母数字值。** 对于每个入口点，提交一个唯一的随机值，并确定该值是否在响应中被反射。该值应设计为能够绕过大多数输入验证，因此需要相当短且仅包含字母数字字符。但它需要足够长，使得在响应中意外匹配的可能性极低。大约 8 个字符的随机字母数字值通常是理想的。您可以使用 Burp Intruder 的[数字载荷](https://portswigger.net/burp/documentation/desktop/tools/intruder/payloads/types#numbers)（随机生成的十六进制值）来生成合适的随机值。您还可以使用 Burp Intruder 的[载荷匹配设置](/burp/documentation/desktop/tools/intruder/configure-attack/settings#grep-payloads)自动标记包含提交值的响应。

- **确定反射上下文。** 对于响应中反射随机值的每个位置，确定其上下文。这可能是在 HTML 标签之间的文本中、在带引号的标签属性内、在 JavaScript 字符串中等等。

- **测试候选载荷。** 根据反射的上下文，测试一个初始的候选 XSS 载荷，如果它未被修改地反射在响应中，该载荷将触发 JavaScript 执行。测试载荷的最简单方法是将请求发送到 [Burp Repeater](/burp/documentation/desktop/tools/repeater)，修改请求插入候选载荷，发出请求，然后检查响应以查看载荷是否生效。一种高效的工作方式是保留原始随机值在请求中，并将候选 XSS 载荷放在其前后。然后将随机值设置为 Burp Repeater 响应视图中的搜索词，Burp 将高亮显示搜索词出现的每个位置，让您快速定位反射点。

- **测试替代载荷。** 如果候选 XSS 载荷被应用程序修改或完全阻止，您将需要测试替代载荷和技术，这些载荷和技术可能根据反射的上下文和所执行输入验证的类型，实现有效的 XSS 攻击。更多详情，请参阅[跨站脚本上下文](/web-security/cross-site-scripting/contexts)。

- **在浏览器中测试攻击。** 最后，如果您成功找到了一个在 Burp Repeater 中似乎有效的载荷，请将攻击转移到真实浏览器（通过将 URL 粘贴到地址栏，或在 [Burp Proxy 的拦截视图](/burp/documentation/desktop/tools/proxy/intercept-messages)中修改请求），然后查看注入的 JavaScript 是否确实执行。通常，最好执行一些简单的 JavaScript，如 `alert(document.domain)`，如果攻击成功，它将在浏览器中触发可见的弹窗。

> 注意，必须测试所有GET与POST请求，检查URL查询字符串与消息主体中的每一个参数。虽然有少数XSS漏洞传送机制只能通过一个POST请求触发，但仍有可能对漏洞加以利用。
>
> 任何时候，一旦在POST请求中发现XSS，应使用Burp中的“更改请求方法”（change request method）选项确定是否可以通过GET请求实施相同的攻击。

## 工具

- [PHP Charset Encoder(PCE)](https://cybersecurity.wtf/encoder/) 帮助您将任意文本编码到 65 种字符集之间，您可以在自定义载荷中使用这些字符集。
- [Hackvertor](https://hackvertor.co.uk/public) 是一个在线工具，允许对 JavaScript（或任何字符串输入）进行多种类型的编码和混淆。
- [XSS-Proxy](https://xss-proxy.sourceforge.net/) 是一个高级跨站脚本（XSS）攻击工具。
- [ratproxy](https://code.google.com/archive/p/ratproxy/) 是一个半自动化、主要是被动式的 Web 应用程序安全审计工具，针对准确和敏感检测以及自动注释潜在问题和安全相关设计模式进行了优化，基于对复杂 Web 2.0 环境中现有用户发起流量的观察。
- [Burp Proxy](https://portswigger.net/burp/) 是一个用于攻击和测试 Web 应用程序的交互式 HTTP/S 代理服务器。
- [Zed Attack Proxy (ZAP)](https://www.zaproxy.org) 是一个带有内置扫描器的交互式 HTTP/S 代理服务器，用于攻击和测试 Web 应用程序。
