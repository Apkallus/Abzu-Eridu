## 如何传递 CSRF 攻击

跨站请求伪造攻击的传递机制与反射型 XSS 基本相同。通常，攻击者将恶意 HTML 放置在他们控制的网站上，然后诱导受害者访问该网站。这可以通过电子邮件或社交媒体消息向用户发送网站链接来实现。或者，如果攻击被放置在热门网站上（例如，在用户评论中），他们可能只是等待用户访问该网站。

请注意，一些简单的 CSRF 攻击使用 GET 方法，并且可以完全自包含在易受攻击网站的一个 URL 中。在这种情况下，攻击者可能无需使用外部站点，可以直接向受害者提供易受攻击域上的恶意 URL。

## 提升攻击影响

发现 CSRF 漏洞后，尝试将 CSRF 影响最大化。
- 查看 CSRF 对应端点的功能
- 与其他设计缺陷或已知漏洞结合使用

### 查看 CSRF 对应端点的功能

#### 通过 CSRF 泄露用户信息

CSRF 对应端点功能导致信息泄露：应用程序通常根据用户偏好发送或披露信息。如果你能通过 CSRF 更改这些设置，就可以为敏感信息披露铺路。

例如，假设 example.com Web 应用程序每月向用户指定的邮箱发送账单邮件。这些邮件包含用户的账单信息，包括街道地址、电话号码和信用卡信息。发送账单邮件的邮箱地址可以通过以下请求更改：

```http
POST /change_billing_email
Host: example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
email=NEW_EMAIL&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

不幸的是，此端点上的 CSRF 验证存在缺陷，服务器接受空白令牌。即使 `csrf_token` 字段留空，请求也会成功

攻击者可以让受害者用户通过 CSRF 发送此请求，更改其账单邮件的接收地址

此后所有账单邮件都将发送到攻击者的邮箱，直到受害者注意到未经授权的更改。一旦账单邮件发送到攻击者邮箱，攻击者即可收集敏感信息，如街道地址、电话号码和与该账户关联的信用卡信息。

#### 通过 CSRF 接管用户账户

CSRF 对应端点功能导致账户接管：当 CSRF 漏洞存在于关键功能（如创建密码、更改密码、更改邮箱地址或重置密码的代码）中时，就会发生账户接管。

例如，假设除了使用邮箱和密码注册外，example.com 还允许用户通过社交媒体账户注册。如果用户选择此选项，则无需创建密码，因为他们只需通过关联账户登录即可。但为了给用户另一种选择，通过社交媒体注册的用户可以通过以下请求设置新密码：

```http
POST /set_password
Host: example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE;

(POST request body)
password=XXXXX&csrf_token=871caef0757a4ac9691aceb9aad8b65b
```

由于用户是通过社交媒体账户注册的，他们无需提供旧密码即可设置新密码，因此如果此端点上的 CSRF 保护失败，攻击者将能够为任何通过社交媒体注册且尚未设置密码的用户设置密码。

虽然大多数 CSRF 都是低严重性问题，但有时关键端点上的 CSRF 可能导致严重后果。

### 与其他设计缺陷或已知漏洞结合使用

由于引入了其他攻击向量，CSRF攻击的可能性改变了许多其他类型的漏洞的影响范围。例如，如果某项管理功能接受参数中的用户标识符，然后显示与指定用户有关的信息。该功能受到严格的访问控制，但它的uid参数中包含SQL注入漏洞。由于应用程序管理员为可信用户，并且在任何情况下都能够完全控制数据库，因此，这种SQL注入漏洞被认为风险较低。但是，由于该功能并不执行任何管理操作（根据最初的设计），因此其并未采取防范CSRF的措施。从攻击者的角度看，该功能与专门供管理员执行任意SQL查询的功能一样重要。如果可以注入一个执行某种敏感操作，或通过带外通道检索数据的查询，那么即使是非管理用户也可以通过CSRF实施这种攻击。

#### 身份验证与 CSRF

由于实施CSRF攻击需要在受害用户的会话中执行某种特权操作，因此，在实施攻击时，用户需要登录到应用程序。

一个存在大量危险的CSRF漏洞的位置，是家庭DSL路由器使用的Web界面。这些设备大多包含敏感功能，如打开面向互联网的防火墙上的所有端口。由于这些功能通常并未采取防范CSRF的措施，并且多数用户也没有修改设备的默认内部IP地址，因此，它们易于受到由恶意外部站点传送的CSRF攻击。但是，相关设备通常需要进行验证才能执行敏感操作，而且许多用户并未登录他们的设备。

如果设备的Web界面使用基于表单的验证，则可以通过首先使用户登录设备，然后执行经过验证的操作，从而实施两步攻击。由于大多数用户并未修改这类设备的默认证书（可能认为该Web界面只能通过内部家庭网络访问），因此，攻击者的网页可以首先提出包含默认证书的登录请求。然后，设备会在用户的浏览器中设置一个会话令牌，随后的任何请求，包括由攻击者生成的请求，将自动传送该令牌。

在其他情况下，攻击者可能需要受害用户以攻击者自身的用户账户登录应用程序才能实施特定的攻击。以一个允许用户上传并存储文件的应用程序为例。这些文件随后可以进行下载，但只能由上传它们的用户下载。假设由于没有对文件内容进行过滤，该功能可用于实施保存型XSS攻击。该漏洞似乎并不会造成任何伤害，因为攻击者只能用它来攻击自己。但实际上，通过使用CSRF技巧，攻击者可以利用保存型XSS漏洞来攻破其他用户。如上文所述，攻击者的网页可以提出一个CSRF请求，强制受害用户使用攻击者的证书登录。然后，攻击者的网页可以提出另一个CSRF请求，以下载某个恶意文件。用户的浏览器处理该文件时，攻击者的XSS有效载荷将会执行，用户在易受攻击的应用程序中的会话将被攻破。虽然受害者当前是使用攻击者的账户登录的，但是，攻击并未就此结束。XSS有效载荷可以在用户的浏览器中持续存在，并执行任意操作，因而可以让用户注销其在易受攻击的应用程序中的会话，并诱使其使用自己的证书登录。

反 CSRF 令牌的本地枚举条件：
- 在URL查询字符串中传输CSRF令牌：受保护的功能通过目标URL中包含令牌的简单超链接即可访问。
- 在整个用户会话中使用相同的反CSRF令牌，或者允许多次使用同一个令牌

#### 通过 CSRF 创建存储型自 XSS

自 XSS 是一种需要受害者输入 XSS 载荷的 XSS 攻击。这些漏洞几乎总被视为非问题，因为它们太难利用；需要受害者大量操作，因此你不太可能成功。然而，当你将 CSRF 与自 XSS 结合时，通常可以将自 XSS 转变为存储型 XSS。

例如，假设 example.com 的金融子域 `finance.example.com` 允许用户为其每个关联银行账户创建昵称。账户昵称字段存在自 XSS 漏洞：该字段对用户输入没有清理、验证或转义。但只有用户自己能编辑和查看该字段，因此攻击者无法直接触发 XSS。

然而，用于更改账户昵称的端点同时存在 CSRF 漏洞。应用程序未正确验证 CSRF 令牌的存在，因此只需在请求中完全省略令牌参数即可绕过 CSRF 保护。

结合两个漏洞构建载荷，绕过端点的 CSRF 保护，更改用户的账户昵称为 XSS 载荷。下次用户登录并查看仪表板时，将触发 XSS。

核心：CSRF （请求伪造）解除了自 XSS （主动特权操作）的限制，任意访问攻击者站点的受害者均自动设置 XSS 载荷



## 找到你的第一个 CSRF

掌握了 CSRF 漏洞、绕过 CSRF 保护和提升 CSRF 影响的知识后，你现在可以开始寻找第一个 CSRF 漏洞了：

1. **识别应用程序上的状态更改操作**，并记录其位置和功能。
2. **检查这些功能是否存在 CSRF 保护**。如果找不到任何保护，你可能发现了漏洞！
3. **如果存在 CSRF 保护机制**，尝试使用本章提到的绕过技术绕过保护。
4. **通过构造恶意 HTML 页面并访问该页面**，确认漏洞是否已执行操作。
5. **思考将载荷交付给最终用户的策略**。
