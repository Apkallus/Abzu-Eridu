
### 头部验证

由于要利用 CSRF 漏洞，攻击者需要诱使受害者导航到托管 CSRF 有效载荷的网页（例如 `evil.com`）。因此，对易受 CSRF 攻击的目标应用程序的请求将源自攻击者域（`evil.com`）。开发人员实施 HTTP `Referer`/`Origin` 头检查来尝试防御 CSRF 攻击，通常通过验证请求是否源自应用程序自身的域。这种方法通常效果较差，并且经常可以被绕过。
- Referer 标头提供关于发起请求的先前页面的 URL 的信息
- Origin 标头提供关于用于发起请求的来源（协议、主机、端口）的信息。

> **Referer 头**  
> HTTP Referer 头（HTTP 规范中无意拼写错误）是一个可选的请求头，包含链接到所请求资源的网页 URL。当用户触发 HTTP 请求时（包括点击链接或提交表单），浏览器通常会自动添加该头。有多种方法允许链接页面省略或修改 `Referer` 头的值，这通常出于隐私原因。

依赖验证 `referrer` 或 `origin` 等头部的 CSRF 防御可能通过设置或处于特定场景不发送这些头部来绕过，导致服务器在其验证步骤中读取 `null`。
- `referrer` 头部，[MDN 文档](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Referrer-Policy)
    - 客户端 HTML 丢弃 referrer 标头
        - 设置特定组件，如 `<a>` 等标签
            ```html
            <a rel="noreferrer">
            <a referrerpolicy="no-referrer">
            ```
        - meta 标签全局设置
            ```html
            <meta name="referrer" content="no-referrer">
            <meta name="referrer" content="never">
            ```

    - 服务器头部丢弃 referrer 标头
        - 设置 `Referrer-Policy: no-referrer`
            - 浏览器不发送 `Referrer` 头部

- `origin` 头部[MDN 文档](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Origin)
    - 服务器头部
        - 设置 `Referrer-Policy: no-referrer`
            - 在 `non-cors` 请求中浏览器发送 `Origin: null`（例如基本的 POST 表单）

> 服务器头部 `Referrer-Policy`，可同时控制两个浏览器头部 `referrer`、`origin`

在现代应用程序中，反向代理和 WAF 可能会移除或修改标头，这可能给 CSRF 检测带来挑战。这可能导致假阴性和假阳性，因为预期的标头可能在请求过程中不存在或已被更改。

此外，在用户从安全站点（HTTPS）导航到非安全站点（HTTP）的场景中，浏览器不会发送 referrer 标头，从而导致不一致。

尽管如此，在某些场景下，根据具体实现，referrer 保护仍然可以被绕过。

#### Referer 的验证依赖于头是否存在

某些应用程序在请求中存在 `Referer` 头时进行验证，但如果省略该头则跳过验证。

在这种情况下，使用上述多种方式构造 CSRF 攻击，使受害用户的浏览器在生成的请求中丢弃 `Referer` 头

#### Referer 的验证可以被绕过

某些应用程序以简陋的可以被绕过的方式验证 `Referer` 头，例如：

- 若应用程序验证 `Referer` 中的域以预期值开头，则攻击者可以将此值作为自己域的子域放置：

    ```http
    http://vulnerable-website.com.attacker-website.com/csrf-attack
    ```

- 若应用程序仅验证 `Referer` 包含其自身的域名，则攻击者可以将所需值放在 URL 的其他位置：

    ```http
    http://attacker-website.com/csrf-attack?vulnerable-website.com
    ```
    - 设置方式
        - 仅在攻击页面使用 JS 修改浏览器 URL，而无需修改载荷 URL 或攻击者服务器端点
            ```js
            history.pushState('', '', '/域名');
            ```
        - 将域名包含在查询参数或路径 `/?域名` `/域名`，修改载荷 URL 或攻击者服务器端点
            - 实验中出现受害者浏览器在 URL 的查询参数后添加路径符号 `/` 的怪癖，导致查询参数型载荷失败（`/?域名/` 端点未设置）

> 许多浏览器现在默认从 `Referer` 头中剥离查询字符串。

攻击者通过设置 referrer 头部 `unsafe-url`，以发送包含详细路径与查询参数的完整 URL
- 客户端 HTML
    ```html
    <meta name="referrer" content="unsafe-url">
    ```
- 服务器头部
    ```http
    Referrer-Policy: unsafe-url
    ```

#### 基于子域的 Referer 验证绕过

应用程序根据 referrer 标头执行域验证。然而，如果允许包含子域，则可以利用子域接管或脆弱子域来绕过验证。

通过利用被入侵的子域，攻击者可以构造在 referrer 标头中包含该子域的请求，从而欺骗应用程序将请求视为合法。
