## XSS vs CSRF

在本节中，我们将解释 XSS 和 CSRF 之间的区别，并讨论 CSRF 令牌是否有助于防止 XSS 攻击。

### XSS 和 CSRF 之间有什么区别？

[跨站脚本]（或 XSS）允许攻击者在受害用户的浏览器中执行任意 JavaScript。

[跨站请求伪造]（或 CSRF）允许攻击者诱导受害用户执行他们并不打算执行的操作。

XSS 漏洞的后果通常比 CSRF 漏洞更严重：

- CSRF 通常仅适用于用户能够执行的部分操作。许多应用程序在整体上实施了 CSRF 防御，但忽略了一两个暴露的操作。相反，成功的 XSS 攻击通常可以诱导用户执行其能够执行的任何操作，无论漏洞出现在哪个功能中。

- CSRF 可以被描述为一种“单向”漏洞，因为虽然攻击者可以诱导受害者发出 HTTP 请求，但他们无法检索该请求的响应。相反，XSS 是“双向”的，攻击者注入的脚本可以发出任意请求、读取响应，并将数据外泄到攻击者选择的外部域。

### CSRF 令牌能阻止 XSS 攻击吗？

某些 XSS 攻击确实可以通过有效使用 CSRF 令牌来阻止。考虑一个简单的反射型 XSS 漏洞，可以像这样轻松利用：

```http
https://insecure-website.com/status?message=<script>/*+Bad+stuff+here...+*/</script>
```

现在，假设易受攻击的功能包含一个 CSRF 令牌：

```http
https://insecure-website.com/status?csrf-token=CIwNZNlR4XbisJF39I8yWnWX9wX4WFoz&message=<script>/*+Bad+stuff+here...+*/</script>
```

假设服务器正确验证了 CSRF 令牌，并拒绝没有有效令牌的请求，那么该令牌确实可以防止 XSS 漏洞的利用。关键点在于其名称：“跨站脚本”——至少在其[反射型]形式中——涉及一个跨站请求。通过防止攻击者伪造跨站请求，应用程序阻止了对 XSS 漏洞的简单利用。

这里需要注意一些重要的限制：

- 如果站内其他地方的某个功能中存在反射型 XSS 漏洞且未受 CSRF 令牌保护，那么该 XSS 仍然可以以正常方式被利用。

- 如果站点上的任何地方存在可利用的 XSS 漏洞，那么该漏洞可以被利用来使受害用户执行某些操作，即使这些操作本身受到 CSRF 令牌的保护。在这种情况下，攻击者的脚本可以请求相关页面以获取有效的 CSRF 令牌，然后使用该令牌执行受保护的操作。

- CSRF 令牌不能防御存储型 XSS 漏洞。如果受 CSRF 令牌保护的页面同时也是存储型 XSS 漏洞的输出点，那么该 XSS 漏洞可以像往常一样被利用，XSS 载荷将在用户访问该页面时执行。
