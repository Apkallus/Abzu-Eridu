# 如何预防 CSRF 漏洞

## 使用 CSRF 令牌

防御 CSRF 攻击最可靠的方法是在相关请求中包含 CSRF 令牌。令牌必须满足以下标准：

- 具有高熵的不可预测性，与通常的会话令牌一样。
- 与用户的会话绑定。
- 在执行相关操作之前，在每种情况下都进行严格验证。

### CSRF 令牌应如何生成？

CSRF 令牌应包含足够的熵并且高度不可预测，具有与一般会话令牌相同的属性。

您应使用加密安全伪随机数生成器（CSPRNG），并以其创建时的时间戳加上一个静态秘密作为种子。

如果您需要超出 CSPRNG 强度之外的进一步保证，您可以通过将其输出与一些用户特定的熵拼接，并对整个结构进行强哈希来生成单个令牌。这为试图基于颁发给他们的样本来分析令牌的攻击者提供了额外的障碍。

### CSRF 令牌应如何传输？

CSRF 令牌应被视为秘密，并在其整个生命周期中以安全的方式处理。通常有效的方法是将令牌作为 HTML 表单中的隐藏字段传输给客户端，该表单使用 POST 方法提交。当表单提交时，令牌将作为请求参数包含在内：

```html
<input type="hidden" name="csrf-token" value="CIwNZNlR4XbisJF39I8yWnWX9wX4WFoz" />
```

为了额外的安全性，包含 CSRF 令牌的字段应尽可能放在 HTML 文档的早期，最好在任何非隐藏输入字段之前，以及在 HTML 中嵌入用户可控数据的任何位置之前。这可以缓解攻击者使用精心构造的数据操纵 HTML 文档并捕获其部分内容的多种技术。

另一种替代方法是将令牌放入 URL 查询字符串中，这不太安全，因为查询字符串：

- 在客户端和服务端的多个位置被记录；
- 容易在 HTTP Referer 头中传输给第三方；
- 可能在用户浏览器中显示在屏幕上。

某些应用程序在自定义请求头中传输 CSRF 令牌。这为攻击者即使预测或捕获了其他用户的令牌也提供了额外的防御，因为浏览器通常不允许跨域发送自定义头。然而，这种方法限制了应用程序使用 XHR（而非 HTML 表单）发出受 CSRF 保护的请求，对于许多情况可能被认为过于复杂。

CSRF 令牌不应在 Cookie 中传输。

### CSRF 令牌应如何验证？

当生成 CSRF 令牌时，它应存储在服务器端的用户会话数据中。当收到需要进行验证的后续请求时，服务器端应用程序应验证该请求是否包含与用户会话中存储的值匹配的令牌。无论请求的 HTTP 方法或内容类型如何，都必须执行此验证。如果请求根本不包含任何令牌，则应以与存在无效令牌时相同的方式拒绝它。

## 使用严格的 SameSite Cookie 限制

除了实施健壮的 CSRF 令牌验证之外，我们建议为您颁发的每个 Cookie 明确设置自己的 SameSite 限制。通过这样做，无论浏览器如何，您都可以精确控制 Cookie 将在哪些上下文中使用。

即使所有浏览器最终都采用“默认 Lax”策略，这也不适用于每个 Cookie，并且比 `Strict` 限制更容易被绕过。与此同时，不同浏览器之间的不一致也意味着只有部分用户能享受到任何 SameSite 保护。

理想情况下，您应默认使用 `Strict` 策略，只有在有充分理由时才降低到 `Lax`。除非您完全了解安全影响，否则绝不要使用 `SameSite=None` 禁用 SameSite 限制。

## 警惕跨源、同站攻击

尽管正确配置的 SameSite 限制可以很好地防御跨站攻击，但至关重要的是要理解它们对于跨源、同站攻击完全没有防御能力。

如果可能，我们建议将不安全的内容（如用户上传的文件）与任何敏感功能或数据隔离开来，放在单独的站点上。在测试站点时，请务必彻底审计属于同一站点的所有可用攻击面，包括任何兄弟域。
