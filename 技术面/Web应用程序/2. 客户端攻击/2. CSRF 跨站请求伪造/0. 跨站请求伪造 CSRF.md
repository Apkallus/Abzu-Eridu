# 跨站请求伪造（CSRF）

跨站请求伪造（CSRF）允许攻击者伪造请求并以用户身份（管理员或特权账户）执行操作。它允许攻击者部分绕过同源策略，该策略旨在防止不同网站之间相互干扰。

在成功的 CSRF 攻击中，攻击者导致受害用户无意中执行了某个操作。例如，这可能是更改其账户上的电子邮件地址、更改密码、进行资金转账、上传文件、删除用户等。这些特权操作通常需要已认证的会话，即用户已成功登录系统。根据操作的性质，攻击者可能能够完全控制用户的账户。如果被入侵的用户在应用程序中拥有特权角色，则攻击者可能能够完全控制应用程序的所有数据和功能。

CSRF 攻击专门针对状态更改请求（如发送推文和修改用户设置），而不是泄露敏感用户信息的请求。这是因为攻击者无法读取 CSRF 攻击期间发送的伪造请求的响应。

## CSRF 如何工作？

大多数现代 Web 应用程序通过会话 Cookie 来认证用户并管理用户会话。首次登录网站时，Web 服务器会建立一个新会话：它向浏览器发送一个与会话关联的会话 Cookie，这个 Cookie 向服务器证明身份。浏览器存储与该网站关联的会话 Cookie，并在对该站点发送的每个后续请求中一并发送。这一切都是自动发生的

要成功实施 CSRF 攻击，必须满足以下条件：

1. **关键操作。** 应用程序中的特权操作（如修改其他用户的权限）或任何针对用户特定数据的操作（如更改用户自己的密码）。
2. **基于 Cookie 的会话处理。** 执行该操作涉及发出一个或多个 HTTP 请求，并且应用程序仅依赖会话 Cookie 来识别发出请求的用户。没有其他用于跟踪会话或验证用户请求的机制。
  - 用户已认证并登录 Web 应用程序，且拥有活动会话。
3. **无不可预测的请求参数。** 执行该操作的请求不包含攻击者无法确定或猜测其值的任何参数。例如，当导致用户更改密码时，如果攻击者需要知道现有密码的值，则该功能不易受攻击。

我们来看一个银行网站请求的示例。考虑以下从一个账户向另一个账户转账的请求：

**请求**
```http
POST /transfer.php HTTP/1.1
Host: vulnerablebank.com
Cookie: PHPSESSID=3e5a8b24b7467fd7e4791ab33412aff1
Content-Type: application/x-www-form-urlencoded

to_account=098855455&amount=1000&currency=usd
```

从这个请求可以清楚地看出，CSRF 攻击的所有三个条件都已满足：

1. 关键操作：向账户转账。  
2. 应用程序使用会话 Cookie 来识别发出请求的用户。没有其他令牌或机制来跟踪用户会话。
  - 用户已认证并拥有有效的会话 ID。  
3. 该请求的所有参数都是可预测的，并且对攻击者已知。

现在，让我们考虑以下不满足条件的请求：

**请求**
```http
POST /transfer.php HTTP/1.1
Host: securebank.com
Cookie: session=abc123
Content-Type: application/x-www-form-urlencoded

to_account=987654321&amount=1000&password=password@12345
```

在此请求中，用户已登录银行应用程序，创建了活动会话。用户拥有执行转账的权限，这是一个特权操作。然而，这次银行网站要求每次交易请求中都包含用户的密码作为额外的安全层，这对攻击者来说是不可预测的，因此第三个条件不满足。

> 虽然 CSRF 通常被描述为与基于 Cookie 的会话处理相关，但它也出现在其他上下文中，即应用程序自动将某些用户凭据添加到请求中，例如 HTTP 基本认证和基于证书的认证。
