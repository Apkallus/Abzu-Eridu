# SameSite Cookie

SameSite Cookie 是一种浏览器安全机制，用于确定网站在何种情况下会在源自其他网站的请求中包含其 Cookie。SameSite Cookie 限制为多种跨站攻击（包括 CSRF、跨站信息泄露以及某些 CORS 攻击）提供了部分防护。

自 2021 年起，如果颁发 Cookie 的网站未显式设置其自身的限制级别，Chrome 会默认应用 `Lax` SameSite 限制。这是一项拟议标准，我们预计其他主流浏览器未来也会采用此行为。因此，为了全面测试跨站攻击向量，务必牢固掌握这些限制的工作原理以及可能被绕过的方式。


## 在 SameSite Cookie 的上下文中，什么是“站点”？

在 SameSite Cookie 限制的上下文中，站点被定义为顶级域（TLD，通常是 `.com` 或 `.net` 等）加上域名的下一级，这通常被称为 TLD+1。

在判断一个请求是否同站时，URL 协议也被考虑在内。这意味着从 `http://app.example.com` 到 `https://app.example.com` 的链接在大多数浏览器中被视为跨站。

> **注意：** 您可能会遇到“有效顶级域”（eTLD）这个术语。它只是为了说明那些实际被当作顶级域处理的保留多级后缀，如 `.co.uk`。

## 站点（site）和源（origin）有什么区别？

站点和源的区别在于它们的范围：站点涵盖多个域名，而源只包含一个。

两个 URL 如果具有完全相同的协议、域名和端口，则被认为是同源。注意，端口通常由协议隐含。

从这个例子可以看出，“站点”一词的粒度要粗得多，因为它只考虑协议和域名的最后部分。关键点在于，**跨源请求仍然可以是同站请求，但反过来则不成立**。

| 请求来源 | 请求目标 | 同站？ | 同源？ |
|---|---|---|---|
| `https://example.com` | `https://example.com` | 是 | 是 |
| `https://app.example.com` | `https://intranet.example.com` | 是 | 否：域名不匹配 |
| `https://example.com` | `https://example.com:8080` | 是 | 否：端口不匹配 |
| `https://example.com` | `https://example.co.uk` | 否：eTLD 不匹配 | 否：域名不匹配 |
| `https://example.com` | `http://example.com` | 否：协议不匹配 | 否：协议不匹配 |

这是一个重要的区别，因为这意味着任何允许执行任意 JavaScript 的漏洞都可以被滥用来绕过同一站点下其他域上基于站点的防御措施。我们将在后面的实验中看到这样的例子。

## SameSite 如何工作？

在 SameSite 机制引入之前，浏览器会在每个指向颁发 Cookie 的域的请求中发送 Cookie，即使该请求是由不相关的第三方网站触发的。SameSite 通过允许浏览器和网站所有者限制哪些跨站请求（如果有的话）应该包含特定 Cookie 来工作。这有助于减少用户遭受 CSRF 攻击的风险——CSRF 攻击会诱导受害者的浏览器发出请求，从而在易受攻击的网站上触发有害操作。由于这些请求通常需要与受害者认证会话关联的 Cookie，如果浏览器不包含该 Cookie，攻击将失败。

目前所有主流浏览器都支持以下 SameSite 限制级别：

- [`Strict`](#strict)
- [`Lax`](#lax)
- [`None`](#none)

开发者可以手动为每个设置的 Cookie 配置限制级别，从而更好地控制这些 Cookie 的使用时机。为此，他们只需在 `Set-Cookie` 响应头中包含 `SameSite` 属性及其偏好的值：

```http
Set-Cookie: session=0F8tgdOhi9ynR1M9wa3ODa; SameSite=Strict
```

尽管这提供了一些针对 CSRF 攻击的保护，但这些限制都不能保证绝对免疫

> **注意：** 如果颁发 Cookie 的网站没有显式设置 `SameSite` 属性，Chrome 默认自动应用 `Lax` 限制。这意味着 Cookie 只会在满足特定条件的跨站请求中发送，即使开发者从未配置过此行为。由于这是一个拟议的新标准，我们预计其他主流浏览器未来也会采用此行为。

Set-Cookie 文档 <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie>

### SameSite Strict

SameSite 属性 `strict`，表示浏览器不会在跨站请求时发送 Cookie。

#### SameSite Strict 绕过

绕过 SameSite Cookie 的方法之一是利用应用程序中现有的功能，通常称为“gadgets”。这些 gadgets 可以有多种形式，如客户端 URL 重定向、JSONP 端点或错误配置的 CORS。

为了说明，考虑我们传统的 `vulnerablebank` CSRF 场景。以下请求发起转账：

**POC**
```
http://vulnerablebank.com/transfer.php?to_account=098855455&amount=1000&currency=usd
```

同一个域 `vulnerablebank.com` 包含一段处理基于 URL 中 `redirect` 参数值的客户端重定向的 JavaScript 代码。由于缺乏验证，该代码容易受到“开放重定向”漏洞的影响。

**易受攻击的代码**
```javascript
var params = new URLSearchParams(window.location.search);
var redirectURL = params.get('redirect');
if (redirectURL) {
    window.location = decodeURIComponent(redirectURL);
}
```

在此场景中，被利用的“gadget”是客户端重定向功能。攻击者可以滥用此功能来创建恶意 URL。我们来看以下 POC：

**POC**
```
http://vulnerablebank.com/csrf/index.html?redirect=/transfer.php?to_account=098855455&amount=1000&currency=usd
```

访问此 URL 后，重定向函数被执行，从而发起转账并成功绕过 SameSite Cookie 保护。由于此操作发生在同一域内，Cookie 被发送。

就浏览器而言，这些客户端重定向实际上并不是真正的重定向；生成的请求只是被视为普通的独立请求。最重要的是，这是一个同站请求，因此将包含与该站点相关的所有 Cookie，无论设置了什么跨站限制。

如果你能操纵这个小工具来引发恶意的二次请求，这可以使你完全绕过任何 SameSite Cookie 限制。

请注意，等效攻击在服务器端重定向中是不可能的。在这种情况下，浏览器识别出遵循重定向的请求最初是由跨站请求产生的，因此仍然应用相应的 Cookie 限制。
- 浏览器将对重定向的 URL 全链进行检查，保存 `重定向污染` 状态
    - -> 攻击者服务器访问应用程序重定向脆弱页面，并注入客户端 DOM 重定向载荷
    - <- 应用程序返回被输入载荷的页面
    - -> 受害者浏览器使用同站的 DOM 客户端重定向访问目标功能页面，并由于同站而携带 cookie


> 客户端重定向可结合客户端路径遍历使用：浏览器对路径遍历序列 `../../` 进行规范化处理，用来突破已存在的路径前缀字符串的限制

#### 通过子/兄弟域绕过 SameSite Strict

即使请求是跨源发出的，它仍然可以是同站的。

如果 Cookie 的作用域没有精确定义，而是设置为 `Domain=vulnerablebank.com`，则该 Cookie 可以被 `vulnerablebank.com` 的所有子域访问。

请确保彻底审计所有可用的攻击面，包括任何兄弟域。特别是，能够让你引发任意二次请求的漏洞（如 XSS）可以完全破坏基于站点的防御，使站点的所有域都暴露于跨站攻击。

在这种情况下，如果从 `sub1.vulnerablebank.com` 向另一个子域（例如 `sub2.vulnerablebank.com`）发出请求，浏览器将在请求中包含 Cookie。这是因为 `sub2.vulnerablebank.com` 是 `vulnerablebank.com` 的子域，因此该请求不被视为跨站。

除了经典的 CSRF，不要忘记如果目标网站支持 WebSocket，该功能可能容易受到跨站 WebSocket 劫持（CSWSH）的攻击，这本质上只是一个针对 WebSocket 握手的 CSRF 攻击。

### SameSite Lax

`Lax` SameSite 限制意味着浏览器将在跨站请求中发送 Cookie，但仅当同时满足以下两个条件时：

- 请求使用安全方法（如 `GET`）
- 请求由用户的顶级导航（如点击链接）产生。

这意味着 Cookie 不会在任何情况下包含在跨站 `POST` 请求中，例如。由于 `POST` 请求通常用于执行修改数据或状态的操作（至少根据最佳实践），它们更有可能成为 CSRF 攻击的目标。

同样，Cookie 不会包含在后台请求中，例如由脚本、iframe 或图片和其他资源的引用发起的请求。

#### 使用 GET 请求绕过 SameSite Lax 限制

在实践中，服务器并不总是严格区分对给定端点收到的是 `GET` 还是 `POST` 请求(由于向后兼容性或开发人员的疏忽)，即使是那些期望表单提交的端点。

在 `Lax` 限制下（显式或由于浏览器默认），仍能够通过诱导受害者浏览器发出 `GET` 请求来实施 CSRF 攻击。

只要请求涉及顶级导航，浏览器仍会包含受害者的会话 Cookie。以下是发起此类攻击的最简单方法之一：

```html
<script>
    document.location = 'https://vulnerable-website.com/account/transfer-payment?recipient=hacker&amount=1000000';
</script>
```

即使普通的 `GET` 请求不被允许，某些框架也提供了覆盖请求行中指定方法的方式。例如，Symfony 支持表单中的 `_method` 参数，该参数在路由方面优先于常规方法：

```html
<form action="https://vulnerable-website.com/account/transfer-payment" method="GET">
    <input type="hidden" name="_method" value="POST">
    <input type="hidden" name="recipient" value="hacker">
    <input type="hidden" name="amount" value="1000000">
</form>
```

对应
```http
GET /account/transfer-payment?_method=POST... HTTP/2
```
- 此时浏览器对 GET 方法携带 cookie，而服务器使用框架参数覆盖为 POST

其他框架支持各种类似的参数。


#### 使用新颁发的 Cookie 绕过 SameSite Lax 限制

具有 `Lax` SameSite 限制的 Cookie 通常不会在任何跨站 `POST` 请求中发送，但有一些例外。

如前所述，如果网站在设置 Cookie 时未包含 `SameSite` 属性，Chrome 默认自动应用 `Lax` 限制。然而，为了避免破坏单点登录（SSO）机制，在顶级 `POST` 请求的前 120 秒内，它实际上并不强制执行这些限制。因此，存在一个两分钟的窗口期，在此期间用户可能容易受到跨站攻击。

> **注意：** 这两分钟的窗口不适用于显式设置了 `SameSite=Lax` 属性的 Cookie。

试图将攻击时机安排在这个短暂的窗口内有些不切实际。另一方面，如果你能在网站上找到一个小工具，迫使受害者获得一个新的会话 Cookie，你可以在发起主要攻击之前先刷新他们的 Cookie。例如，完成基于 OAuth 的登录流程可能会在每次产生新的会话，因为 OAuth 服务不一定知道用户是否仍登录到目标网站。

为了触发应用程序的 Cookie 刷新而无需受害者手动重新登录，在 oauth 验证流程中应当使用顶级页面导航，以在浏览器对 oauth 服务器的 GET 请求中携带 oauth 服务器的已验证 cookie。这带来了额外的挑战，因为你随后需要将用户重定向回你的网站，以便发起 CSRF 攻击。

或者，你可以从新标签页触发 Cookie 刷新，这样浏览器在你能发出最终攻击之前不会离开该页面。这种方法的一个小障碍是，浏览器会阻止弹出标签页，除非它们是通过手动交互打开的。例如，默认情况下，以下弹出窗口将被浏览器阻止：

```javascript
window.open('https://vulnerable-website.com/login/sso');
```

为了解决这个问题，你可以将语句包装在 `onclick` 事件处理程序中，如下所示：

```javascript
window.onclick = () => {
    window.open('https://vulnerable-website.com/login/sso');
}
```

这样，`window.open()` 方法仅在用户点击页面某处时被调用。

#### SameSite None

当 SameSite 属性设置为 `none` 时，将完全禁用 SameSite 限制，表示 Cookie 将在同站和跨站请求以及跨域通信中发送。换句话说，它不提供任何针对 CSRF 攻击的保护。

禁用 SameSite 场景：

- 当 Cookie 旨在用于第三方上下文，且不会授予持有者访问任何敏感数据或功能的权限时。跟踪 Cookie 是典型示例。

- 用于需要跨源访问的场景，例如单点登录（SSO）实现。

如果你遇到设置了 `SameSite=None` 或没有显式限制的 Cookie，值得调查它是否有用。当 Chrome 首次采用“默认 Lax”行为时，其副作用是破坏了许多现有的 Web 功能。作为一种快速解决方法，一些网站选择在所有 Cookie 上禁用 SameSite 限制，包括潜在敏感的 Cookie。

当设置带有 `SameSite=None` 的 Cookie 时，网站还必须包含 `Secure` 属性，这确保 Cookie 仅通过 HTTPS 在加密消息中发送。否则，浏览器将拒绝该 Cookie，不会设置它。

```http
Set-Cookie: trackingId=0F8tgdOhi9ynR1M9wa3ODa; SameSite=None; Secure
```
