## 寻找 CSRF

发现未受 CSRF 防护保护的状态更改请求

### 第 1 步：识别状态更改操作

更改用户数据的操作称为状态更改操作。例如，发送推文和修改用户设置都属于状态更改。发现 CSRF 的第一步是登录目标站点并浏览，寻找任何更改数据的活动。

例如，假设你正在测试 `email.example.com`，这是 example.com 处理电子邮件的子域。遍历应用程序的所有功能，点击所有链接。使用 Burp 等代理拦截生成的请求，并记录其 URL 端点。

逐一记录这些端点，形成类似下面的列表，以便稍后重新访问和测试：

**email.example.com 上的状态更改请求**

- **更改密码**：`email.example.com/password_change`
  - POST 请求
  - 请求参数：`new_password`
- **发送邮件**：`email.example.com/send_email`
  - POST 请求
  - 请求参数：`draft_id`, `recipient_id`
- **删除邮件**：`email.example.com/delete_email`
  - POST 请求
  - 请求参数：`email_id`

### 第 2 步：寻找 CSRF 防护缺失

现在访问这些端点以测试 CSRF。首先，打开 Burp Suite，在 Proxy 选项卡中开始拦截所有发往目标站点的请求。切换 Intercept 按钮，直到显示 `Intercept is on`。设置为 Intercept is on 以捕获浏览器流量。点击 Forward 按钮将当前请求转发到服务器。

在主动寻找 CSRF 时，让 Burp 在后台运行以记录与目标站点相关的其他流量。持续点击 Forward 按钮，直到遇到与状态更改操作关联的请求。例如，假设你正在测试发现的密码更改功能是否易受 CSRF 攻击。你已在 Burp 代理中拦截到该请求：

```http
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE

(POST request body)
new_password=abc123
```

在拦截的请求中，寻找 CSRF 保护机制的迹象。使用窗口底部的搜索栏搜索字符串 `"csrf"` 或 `"state"`。CSRF 令牌不仅可能以 POST 主体参数的形式出现，有时也会出现在请求头、Cookie 和 URL 参数中。例如，它们可能像 Cookie 一样出现：

```http
POST /password_change
Host: email.example.com
Cookie: session_cookie=YOUR_SESSION_COOKIE; csrf_token=871caef0757a4ac9691aceb9aad8b65b

(POST request body)
new_password=abc123
```

若在端点上发现了 CSRF 保护，尝试各种保护绕过技术。
