# 第 11 章：跨站请求伪造（CSRF）

有时我们已经知道存在一个允许我们执行所需操作的 API 端点，但由于它需要特权访问（例如管理员账户），我们无法访问该端点。在本章中，我们将构建跨站请求伪造（CSRF）利用，使得管理员或特权账户代表我们执行操作，而不是使用 JavaScript 代码片段。

CSRF 攻击利用了浏览器的运作方式以及网站与浏览器之间的信任关系。通过寻找那些依赖这种关系来确保安全，但对浏览器赋予过多信任的 API 调用，我们可以精心构造链接和表单，只需稍加努力，就能让用户在不知情的情况下代表自己发起请求。

通常，CSRF 攻击对被攻击的用户来说是不可见的，因为浏览器中的请求在后台发生。这意味着这种攻击可以用来利用特权用户，并在用户完全不知情的情况下对服务器执行操作。它是最隐蔽的攻击之一，自 21 世纪初出现以来，在整个 Web 上造成了严重破坏。

## 查询参数篡改

让我们考虑 CSRF 攻击最基本的形式——通过超链接进行参数篡改（见图 11-1）。Web 上的大多数超链接对应于 HTTP GET 请求。最常见的超链接就是嵌入在 HTML 片段中的 `<a href="https://my-site.com"></a>`。

HTTP GET 请求的结构简单且一致，无论它从何处发送、在何处读取，或如何在网络上传输。要使 HTTP GET 有效，它必须遵循支持的 HTTP 规范版本——因此我们可以放心，GET 请求的结构在不同应用中是相同的。

HTTP GET 请求的结构如下：

```
GET /resource-url?key=value HTTP/1.1
Host: www.mega-bank.com
```

每个 HTTP GET 请求都包括 HTTP 方法（GET），后跟资源 URL，然后是一组可选的查询参数。查询参数的起始由 `?` 表示，一直持续到遇到空格。接着是 HTTP 规范版本，下一行是资源 URL 所在的主机。

当 Web 服务器收到此请求时，它会被路由到适当的处理程序类，该类将接收查询参数以及一些附加信息，以识别发起请求的用户、他们请求时使用的浏览器类型，以及他们期望返回的数据格式。

> **图 11-1. CSRF GET——传播一个恶意链接，点击后，将以已认证用户的名义执行改变状态的 HTTP GET 请求**

让我们看一个例子，使这个概念更具体。第一个例子是一个服务端路由类，它基于 ExpressJS（最流行的基于 Node.js 的 Web 服务器软件）编写：

```javascript
/*
 * 一个示例路由。
 *
 * 将 HTTP 请求提供的查询返回给请求者。
 * 如果未提供查询，则返回错误。
 */
app.get('/account', function(req, res) {
  if (!req.query) { return res.sendStatus(400); }
  return res.json(req.query);
});
```

这是一个极其简单的路由，只做几件事：

- 仅接受对 `/account` 的 HTTP GET 请求
- 如果未提供查询参数，返回 HTTP 400 错误
- 如果提供了查询参数，以 JSON 格式将查询参数反射回发送方

让我们从 Web 浏览器向此端点发起请求：

```javascript
/*
 * 生成一个不带查询的新 HTTP GET 请求。
 *
 * 这将失败并返回错误。
 */
const xhr = new XMLHttpRequest();
xhr.onreadystatechange = function() {
  console.log(xhr.responseText);
}
xhr.open('GET', 'https://www.mega-bank.com/account', true);
xhr.send();
```

这里，我们从浏览器向服务器发起一个 HTTP GET 请求，由于我们没有提供任何查询参数，它将返回 400 错误。我们可以添加查询参数以获得更有趣的结果：

```javascript
/*
 * 生成一个带查询的新 HTTP GET 请求。
 *
 * 这将成功，并且查询将在响应中被反射回来。
 */
const xhr = new XMLHttpRequest();
const params = 'id=12345';
xhr.onreadystatechange = function() {
  console.log(xhr.responseText);
}
xhr.open('GET', `https://www.mega-bank.com/account?${params}`, true);
xhr.send();
```

发起此请求后不久，将返回一个包含以下内容的响应：

```json
{
  id: 12345
}
```

如果您在浏览器中检查网络请求，它还会包含 HTTP 200 状态码。

理解这些请求的流程对于发现和利用 CSRF 漏洞至关重要。让我们回溯一下，再次讨论 CSRF。

CSRF 攻击的两个主要标识是：

- 权限提升
- 发起请求的用户账户通常不知道它发生了（这是一种隐蔽攻击）

大多数遵循 HTTP 规范的 CRUD（创建、读取、更新、删除）Web 应用会使用多种 HTTP 动词，而 GET 只是其中之一。不幸的是，GET 请求是任何请求中最不安全的，也是构造 CSRF 攻击最简单的方式之一。

我们分析的最后一个 GET 端点反射了数据，但重要的是它确实读取了我们发送的查询参数。浏览器中的 URL 地址栏会发起 HTTP GET 请求，浏览器或手机中的 `<a></a>` 链接也是如此。

此外，当我们在互联网上点击链接时，我们很少评估来源以查看链接将我们带到何处。这个链接：

```html
<a href="https://www.my-website.com?id=123">My Website</a>
```

在浏览器中会显示为 “My Website”。大多数用户不会知道有一个参数作为标识符附加到链接上。任何点击该链接的用户都会从他们的浏览器发起一个请求，向关联服务器发送一个查询参数。

让我们想象我们虚构的银行网站 MegaBank 使用了带参数的 GET 请求。看看这个服务端路由：

```javascript
import session from '../authentication/session';
import transferFunds from '../banking/transfers';

/*
 * 将资金从已认证用户的银行账户转移到已认证用户选择的银行账户。
 *
 * 已认证用户可以选择要转账的金额。
 */
app.get('/transfer', function(req, res) {
  if (!session.isAuthenticated) { return res.sendStatus(401); }
  if (!req.query.to_user) { return res.sendStatus(400); }
  if (!req.query.amount) { return res.sendStatus(400); }

  transferFunds(session.currentUser, req.query.to_user, req.query.amount,
    (error) => {
      if (error) { return res.sendStatus(400); }
      return res.json({
        operation: 'transfer',
        amount: req.query.amount,
        from: session.currentUser,
        to: req.query.to_user,
        status: 'complete'
      });
    }
  );
});
```

对于未经训练的眼睛来说，这个路由看起来很简单。它检查用户是否拥有正确的权限，并检查是否指定了转账目标用户。由于用户拥有正确的权限，考虑到用户必须经过身份验证才能发起此请求（它假定请求是代表请求用户发起的），指定的金额应该是准确的。同样，我们假设转账是转给正确的人。不幸的是，因为这是使用 HTTP GET 请求完成的，一个指向此特定路由的超链接可以很容易地被构造并发送给已认证用户。

涉及 HTTP GET 参数篡改的 CSRF 攻击通常按以下步骤进行：

1. 黑客发现 Web 服务器使用 HTTP GET 参数来修改其逻辑流程（在本例中，确定银行转账的金额和目标）。
2. 黑客使用这些参数构造一个 URL 字符串：`<a href="https://www.mega-bank.com/transfer?to_user=<黑客的账户>&amount=10000">click me</a>`。
3. 黑客制定分发策略：通常要么是定向攻击（谁最有可能已登录并拥有正确数量的资金？），要么是批量攻击（如何在被检测到之前的短时间内尽可能多地攻击用户？）。

这些攻击通常通过电子邮件或社交媒体分发。由于分发容易，其影响对公司可能是毁灭性的。黑客甚至曾发起网络广告活动，以将他们的链接尽可能多地传播到用户手中。

## 替代 GET 载荷

由于浏览器中的默认 HTTP 请求是 GET 请求，许多接受 URL 参数的 HTML 标签在被交互或加载到 DOM 时会自动发起 GET 请求。因此，GET 请求最容易通过 CSRF 攻击。

在之前的示例中，我们使用了超链接 `<a></a>` 标签来诱骗用户在其自己的浏览器中执行 GET 请求。或者，我们也可以构造一个图像来做同样的事情：

```html
<!-- 与链接不同，图像在加载到 DOM 时立即执行 HTTP GET 请求。这意味着它不需要加载网页的用户进行任何交互。 -->
<img src="https://www.mega-bank.com/transfer?to_user=<黑客的账户>&amount=10000" width="0" height="0" border="0">
```

当浏览器检测到图像标签时，浏览器会向 `<img>` 标签中包含的 `src` 端点发起 GET 请求（见图 11-2）。这就是图像对象加载到浏览器的方式。因此，一个图像标签（在这种情况下，是一个不可见的 0×0 像素图像）可以用来在无需任何用户交互的情况下发起 CSRF。

> **图 11-2. CSRF IMG——在目标应用内部，发布一个 `<img>` 标签，加载时强制发起 HTTP GET**

同样，大多数其他允许 URL 参数的 HTML 标签也可以用来发起恶意 GET 请求。考虑 HTML5 的 `<video></video>` 标签：

```html
<!-- 视频通常立即加载到 DOM 中，具体取决于浏览器的配置。某些移动浏览器直到元素被交互时才会加载。 -->
<video width="1280" height="720" controls>
  <source src="https://www.mega-bank.com/transfer?to_user=<黑客的账户>&amount=10000" type="video/mp4">
</video>
```

上述视频的功能与使用的图像标签相同。因此，注意任何通过 `src` 属性从服务器请求数据的标签类型非常重要。大多数此类标签都可以用来对不知情的最终用户发起 CSRF 攻击。

## 针对 POST 端点的 CSRF

CSRF 攻击通常针对 GET 端点，因为通过超链接、图像或其他自动发起 HTTP GET 请求的 HTML 标签来分发 CSRF 要容易得多。然而，针对 POST、PUT 或 DELETE 端点发送 CSRF 载荷仍然是可能的。传递 POST 载荷需要更多工作，并且需要一些强制性用户交互（见图 11-3）。

通常，通过 POST 请求传递的 CSRF 攻击是通过浏览器表单创建的，因为 `<form></form>` 对象是少数可以在不需要任何脚本的情况下发起 POST 请求的 HTML 对象之一：

```html
<form action="https://www.mega-bank.com/transfer" method="POST">
  <input type="hidden" name="to_user" value="hacker">
  <input type="hidden" name="amount" value="10000">
  <input type="submit" value="Submit">
</form>
```

在通过 POST 表单进行 CSRF 的情况下，我们可以利用表单输入上的 “hidden” 类型属性来植入不会在浏览器中渲染的数据。

> **图 11-3. CSRF POST——提交一个针对另一台服务器的表单，该服务器对表单创建者不可访问，但对表单提交者可访问**

除了设计 CSRF 载荷所需的隐藏字段外，我们还可以通过提供合法的表单字段来进一步操纵用户：

```html
<form action="https://www.mega-bank.com/transfer" method="POST">
  <input type="hidden" name="to_user" value="hacker">
  <input type="hidden" name="amount" value="10000">
  <input type="text" name="username" value="username">
  <input type="password" name="password" value="password">
  <input type="submit" value="Submit">
</form>
```

在这个例子中，用户将看到一个登录表单——也许是一个合法网站的登录表单。但是当表单被填写并提交时，实际上会向 MegaBank 发起请求——不会发起任何登录尝试。

这是一个示例，说明看起来合法的 HTML 组件如何被用来发送请求，利用用户在浏览器中的当前应用状态。在这种情况下，用户已登录 MegaBank，尽管他们正在与一个完全不同的网站交互，但我们能够利用他们在 MegaBank 中的当前会话，代表他们执行提权操作。

这种技术也可以用来代表有权访问内部网络的用户发起请求。表单的创建者不能向内部网络上的服务器发起请求，但如果一个位于内部网络的用户填写并提交了表单，由于目标用户拥有更高的网络访问权限，该请求将针对内部服务器发出。

自然，这种类型的 CSRF（POST）比通过 `<a></a>` 标签植入 CSRF GET 请求更复杂——但有时您必须对 POST 端点发起提权请求，这时表单是成功发起攻击的最简单方式。

## 绕过 CSRF 防御

根据您所针对的应用，攻击者可能只需很少的努力就能绕过抗 CSRF 防御。

> 请记住，来自浏览器的任何请求都可以被伪造。像 Burp、Zap 甚至 `curl` 这样的工具都可以用来发送带有伪造头部、内容类型或请求体的请求。

并非所有验证器都可以使用相同的技术绕过。因此，尝试多种选项并记录测试进展非常重要，以便深入了解应用的验证工作流程。

### 头部验证

依赖验证 `referrer` 或 `origin` 等头部的 CSRF 防御可能通过移除这些头部来绕过，导致服务器在其验证步骤中读取 `null` 或 `undefined`。在某些情况下，可以通过向 `<a>` 和 `<form>` 标签添加 `rel=noreferrer` 来实现。如果 `rel=noreferrer` 可以添加到这些标签之一，浏览器将自动省略 `referrer` 头部。

示例如下：

```html
<a href="https://example.com/update_password?password=123"
   rel="noreferrer">click me to update password</a>
```

随着您阅读本书的深入，您会注意到我不主张使用头部作为第一道安全防线。HTTP 头部很容易被伪造、绕过或篡改，考虑到它们在实现安全机制中的频繁使用，这令人惊讶。

### 令牌池

一些传统的 CSRF 防御工具会在“池”中评估令牌，而不是逐个评估。在这些系统中，可以从您自己的用户账户内生成一个合法的 CSRF 令牌，并将其附加为 CSRF 攻击的一部分，从而成功验证请求。

合法令牌通常可以通过分析您自己的测试账户中从 Web 应用发起的特定 HTTP 请求，直接从浏览器开发者工具中获取。令牌之后可以通过多种方式附加到请求中，例如通过 Linux 或 macOS 终端的 `curl` 请求：

```bash
curl https://website.com/auth \
  -H "anti-csrf-token": "12345abc"
```

某些服务器会尝试拒绝结构上与合法请求不够相似的请求。因此，可能需要一些努力来模拟有效请求的结构和数据。

### 弱令牌

以可预测方式生成的 CSRF 令牌，例如使用日期、时间、用户名或迭代整数值，可以被模拟和伪造。在评估使用抗 CSRF 令牌的应用的弱点时，请仔细查看生成的 CSRF 令牌的结构。

例如，考虑以下弱 CSRF 令牌：`1691434927`、`1691434928`。经过进一步评估，您可以确定这些 CSRF 令牌实际上是 Unix 纪元时间戳（它们表示自 1970 年 1 月 1 日 00:00:00 UTC 以来经过的毫秒数）。通过理解这些令牌的生成机制，您可以开始伪造类似的令牌，这些令牌将被生成它们的服务端验证器接受。在这个特定示例中，于 2023 年 8 月 7 日晚上 8:04:36 通过网络发送的 CSRF 载荷，使用包含内容 `1691438676` 的伪造 CSRF 令牌将成功验证。

### 内容类型

如果服务器期望特定的内容类型，例如 `application/json`，如果您发送带有 `application/x-url-encoded` 或 `form-multipart` 的载荷，它可能会表现不同。这些替代内容类型可能会跳过验证步骤，并允许您的 CSRF 攻击成功。

在极少数情况下，以下内容类型可能绕过验证：

- `application/x-7z-compressed`
- `application/zip`
- `application/pdf`
- `application/ld+json`
- `application/gzip`
- `text/csv`
- `application/xml`
- `application/xhtml+xml`
- `application/rtf`
- `text/css`

内容类型在所有请求中作为 HTTP 头部声明：

```
Content-Type: text/html; charset=utf-8
Content-Type: multipart/form-data;
```

HTTP 表单将内容类型作为属性 `enctype` 包含：

```html
<form action="/" method="post" enctype="multipart/form-data">
  <button type="submit">Submit</button>
</form>
```

在 HTTP 表单的情况下，只需打开浏览器开发者工具并编辑 `enctype` 字段，就会强制浏览器在表单提交时修改其随 HTTP 请求发送的内容类型。

### 正则表达式过滤器绕过

如果服务器验证使用正则表达式来检测 CSRF 载荷，编写 URL 的替代方法可能会绕过所述正则表达式。示例如下：

- 某些服务器在定义从哪里开始读取查询参数时，除了问号（`?`）外，还接受分号（`;`）。例如，查询 `https://example.com?test=123` 可以写成 `https://example.com;test=123`。
- 某些服务器在定义文件位置时接受正斜杠或反斜杠，例如 `https://example.com/test` 或 `https://example.com\test`。
- 配置不当的服务器会接受相对位置符号（`..`）。例如，`https://example.com/../test`。

确定服务器用于验证的特定正则表达式可能很困难，因此当不确定时，首先尝试针对在搜索引擎上找到的最常见验证器正则表达式进行绕过。许多开发者不会自己设计正则表达式。

### Iframe 载荷

除了使用图像、脚本或链接，还可以通过 `iframe` 的 `src` 属性产生 CSRF，它会自动加载且不需要用户交互。

方法很简单，如下所示：

```html
<iframe src="https://example.com/change_password?password=123"></iframe>
```

当加载到 DOM 中时，`iframe` 将立即向 `src` 属性中的值发起 HTTP 请求。请注意，`iframe` CSRF 仅适用于 HTTP GET 端点。

### AJAX 载荷

如果脚本执行是可能的，能够作为 CSRF 载荷运行的异步 JavaScript 和 XML（AJAX）请求可以从 JavaScript 执行上下文中发出。如下所示：

```javascript
const url = "https://example.com/change_password?password=123";
const xhr = new XMLHTTPRequest();
xhr.open("GET", url);
xhr.setRequestHeader("Content-Type", "text/plain");
xhr.send();
```

这通常出现在允许用户自定义的网站上。否则，在不存在 XSS 漏洞的网站上很少见。

### 零交互表单

如果脚本执行是可能的，CSRF 可以在无需用户交互的表单中执行。相反，一个 JavaScript 载荷（通常通过 XSS 获得）将使用 DOM API 模拟用户与表单的交互：

```html
<form id="pw_form" method="GET" action="https://example.com/change_password">
  <input id="pw" type="hidden" name="password" value="" />
  <input type="submit" value="submit"/>
</form>

<script>
  // 获取表单引用
  const el = document.querySelector("#el")
  const pw = document.querySelector("#pw")
  // 更改密码字段
  pw.val = "new_password_123"
  // 提交表单
  el.submit()
</script>
```

在大多数情况下，所有 DOM 交互都可以通过 JavaScript API 调用来调用，只要 JavaScript 脚本执行是可能的。

## 小结

CSRF 攻击利用了 Web 浏览器、用户和 Web 服务器/API 之间存在的信任关系。默认情况下，浏览器信任从用户设备执行的操作是代表该用户的。

在 CSRF 的情况下，这部分是正确的，因为用户发起了操作，但并不了解操作在后台做了什么。当用户点击一个链接时，浏览器代表他们发起一个 HTTP GET 请求——无论该链接来自何处。因为该链接被信任，有价值的认证数据可以随 GET 请求一起发送。

CSRF 攻击的核心是 WHATWG 等浏览器标准委员会开发的信任模型的结果。这些标准将来可能会改变，使 CSRF 风格攻击更难实施。但就目前而言，这些攻击仍将存在。它们在 Web 上很常见，且易于利用。
