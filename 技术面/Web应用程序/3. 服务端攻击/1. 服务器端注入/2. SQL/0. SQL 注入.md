## SQL 注入

SQL 注入是最经典的注入形式。SQL 字符串在 HTTP 载荷中被逃逸，导致以最终用户的名义执行自定义 SQL 查询。

传统上，许多开源软件包是使用 PHP 和 SQL（通常是 MySQL）的组合构建的。历史上许多最常被引用的 SQL 注入漏洞都是由于 PHP 对视图、逻辑和数据代码之间的插值持宽松态度而发生的。老派的 PHP 开发人员会将 SQL、HTML 和 PHP 交织到他们的 PHP 文件中——这种组织模式虽然受 PHP 支持，但被滥用，导致了大量易受攻击的 PHP 代码。

让我们看一个老派论坛软件的 PHP 代码块示例，它允许用户登录：

```php
<?php if ($_SERVER['REQUEST_METHOD'] != 'POST') {
  echo'
  <div class="row">
    <div class="small-12 columns">
      <form method="post" action="">
        <fieldset class="panel">
          <center>
            <h1>Sign In</h1><br>
          </center>
          <label>
            <input type="text" id="username" name="username"
                   placeholder="Username">
          </label>
          <label>
            <input type="password" id="password" name="password"
                   placeholder="Password">
          </label>
          <center>
            <input type="submit" class="button" value="Sign In">
          </center>
        </fieldset>
      </form>
    </div>
  </div>';
} else {
  // 用户已填写登录表单。
  // 从 config.php 拉取数据库信息
  $servername = getenv('IP');
  $username   = $mysqlUsername;
  $password   = $mysqlPassword;
  $database   = $mysqlDB;
  $dbport     = $mysqlPort;

  $database = new mysqli($servername, $username, $password, $database, $dbport);

  if ($database->connect_error) {
    echo "ERROR: Failed to connect to MySQL";
    die;
  }

  $sql = "SELECT userId, username, admin, moderator FROM users WHERE username = '"
         .$_POST['username']."' AND password = '".sha1($_POST['password'])."';";
  $result = mysqli_query($database, $sql);
}
```

如您在此登录代码中所见，PHP、SQL 和 HTML 全部混杂在一起。此外，SQL 查询是通过连接查询参数生成的，在生成查询字符串之前没有进行任何清理。

HTML、PHP 和 SQL 代码的交织确实使基于 PHP 的 Web 应用更容易受到 SQL 注入攻击。甚至一些最大的开源 PHP 应用，如 WordPress，过去也曾因此受害。

近年来，PHP 编码标准变得更加严格，该语言也实现了降低 SQL 注入发生几率的工具。
事实上，根据国家漏洞数据库的数据，注入漏洞已从 2010 年占所有漏洞的近 5% 下降到当今发现的所有漏洞的不到 1%。

从 PHP 中学到的安全教训在其他语言中得以延续，在当今的 Web 应用中更难找到 SQL 注入漏洞。然而，这仍然是可能的，并且在不使用安全编码最佳实践的应用中仍然很常见。

让我们考虑另一个简单的 Node.js/Express.js 服务器——这次是与 SQL 数据库通信的：

```javascript
const sql = require('mssql');

/*
 * 接收对 /users 的 POST 请求，请求体中带有 user_id 参数。
 *
 * 将执行 SQL 查找，尝试在数据库中找到与 `user_id` 参数中提供的 `id` 匹配的用户。
 *
 * 数据库查询的结果将在响应中返回。
 */
app.post('/users', function(req, res) {
  const user_id = req.body.user_id;

  /*
   * 连接到 SQL 数据库（服务端）。
   */
  await sql.connect('mssql://username:password@localhost/database');

  /*
   * 查询数据库，提供 HTTP 请求体中的 `user_id` 参数。
   */
  const result = await sql.query('SELECT * FROM users WHERE USER = ' + user_id);

  /*
   * 在 HTTP 响应中将 SQL 查询结果返回给请求者。
   */
  return res.json(result);
});
```

在此示例中，开发者使用直接字符串拼接将查询参数附加到 SQL 查询。这假设通过网络发送的查询参数未被篡改，但我们知道这不能作为合法性的可靠指标。

在合法的 `user_id` 情况下，此查询将向请求者返回一个用户对象。在更恶意的 `user_id` 字符串情况下，可能从数据库返回更多对象。让我们看一个例子：

```javascript
const user_id = '1=1'
```

啊，老式的真值判断。现在查询变成 `SELECT * FROM users WHERE USER = true`，这相当于“将所有用户对象返回给请求者”。

如果我们直接在 `user_id` 对象中开始一个新语句呢？我们将在以下代码中这样做：

```javascript
user_id = '123abc; DROP TABLE users;';
```

现在我们的查询如下：`SELECT * FROM users WHERE USER = 123abd; DROP TABLE users;`。换句话说，我们在原始查询之上附加了另一个查询。糟糕，现在我们需要重建用户表了。

一个更隐蔽的例子可能是这样的：

```javascript
const user_id = '123abc; UPDATE users SET credits = 10000 WHERE user = 123abd;'
```

现在，我们不是请求所有用户列表，也不是删除用户表，而是使用第二个查询来更新数据库中我们自己的用户账户——在这种情况下，给自己比应得的更多的应用内积分。

有许多好方法可以防止这些攻击发生，因为 SQL 注入防御已经发展了二十多年。我们将在第三部分详细讨论如何防御这些攻击。
