
## DNS 分析 - 域名信息

DNS 资源记录

    | 记录 | 描述 | 揭示内容 |
    |------|------|----------|
    | SOA | 授权起始 | 创建 DNS 区域的源主机 |
    | NS | 名称服务器 | 给定域的权威 DNS 服务器名称 |
    | A | 地址（IPv4） | 主机名的 IPv4 地址 |
    | AAAA | 地址（IPv6） | 主机名的 IPv6 地址 |
    | PTR | 指针 | 给定 IPv4 或 IPv6 地址的主机名 |
    | CNAME | 规范名称 | CNAME 是其别名的主机名 |
    | MX | 邮件交换 | 给定域的邮件服务器 |
    | HINFO | 主机信息 | 主机的操作系统或其他信息 |
    | SRV | 服务定位器 | 域内的应用服务端点，包括 Kerberos、LDAP、SIP 和 XMPP |
    | TXT | 文本字符串 | 包括 SPF 和 DKIM 字段的材料，根据配置用于提供安全性 |

RFC 1035 详细说明了除 AAAA IPv6 地址和 SRV 服务定位器记录之外的所有记录的底层机制和用法。

RFC 8482 更新了对 any 查询的最小化响应

### 被动 DNS 枚举

被动技术不直接与目标基础设施交互，而是依赖公开可用的数据源。示例包括：

- 公共 DNS 记录（A、AAAA、MX、TXT、NS）
- 反向 DNS 查找（PTR 记录）
- 搜索引擎
- 被动 DNS 数据库
- 证书透明度日志

在早期侦察阶段首选被动技术以避免检测。

### 主动 DNS 枚举

主动技术直接查询目标的 DNS 基础设施，并可能在目标系统上生成日志。这些包括：

- 子域名暴力破解
- DNS 区域传输尝试
- 使用工具进行 DNS 记录枚举

用于 DNS 枚举的常用工具包括：

- `amass` DNS 区域传输、证书解析、搜索引擎和子域数据库
- `subfinder`
- `dnsrecon` 普通查询+区域传输
- `fierce` 区域传输+DNS猜测
- `dig` dig
- `nslookup` 手动
- `Sublist3r` 查询搜索引擎和在线子域数据库
- `SubBrute` 暴破子域
- `Gobuster`
- `Altdns` 生成大量“改变”或“变异”的潜在子域


### 被动收集技术




#### web 子域名获取

如今，使用单个域名来完整承载一个 Web 应用的情况已很少见。通常，Web 应用至少会拆分为客户端域和服务端域，再加上常见的 `https://www` 与纯 `https://` 之别。能够迭代地发现并记录支撑 Web 应用的子域名，是对该应用进行初始侦察的一项实用技术。

存储在 Web 上的公开可用信息量如此之大，以至于偶然的数据泄露可能会在无人注意的情况下溜走数年。包括：

- 曾意外公开后又转为私有的 GitHub 仓库的缓存副本
- SSH 密钥
- 各种服务的密钥，如 Amazon AWS 或 Stripe，这些密钥曾定期暴露，然后从面向公众的 Web 应用中移除
- 不打算面向公众的 DNS 列表和 URL
- 详细描述未发布产品但不应在线的页面
- 托管在 Web 上但不打算被搜索引擎抓取的财务记录
- 电子邮件地址、电话号码和用户名

这些信息可以在许多地方找到，例如：

- 搜索引擎
    - 高级搜索
- 社交媒体帖子
    - 社交媒体企业出售用户数据到其查询 API 端点
- 存档应用，如 archive.org
- 图片搜索和反向图片搜索

在尝试查找子域名时，公共记录也是一个很好的信息来源，因为子域名可能不易通过字典找到，但可能已被前述服务之一索引。

##### 浏览器的内置网络分析工具

首先，我们可以通过浏览应用程序的可见功能并观察后台发出了哪些 API 请求，来收集一些有用的数据。这通常会给我们一些唾手可得的端点。要查看这些正在发出的请求，我们可以使用自己的 Web 浏览器网络工具，或更强大的工具如 Burp 或 ZAP。

使用浏览器中的“网络”标签查看浏览器正在处理的所有网络流量。对于大型网站，过滤起来可能会变得相当繁琐。

通常最有价值的结果来自“网络”标签下的“XHR”子标签，它将显示任何对服务器的 HTTP POST、GET、PUT、DELETE 以及其他请求，并过滤掉字体、图像、视频和依赖文件。

##### 从内容安全策略（CSP）枚举子域名

内容安全策略（CSP）标头允许管理员指定哪些域和子域被允许在其网站上加载内容，如脚本、框架来源、图像来源等。位于响应标头选项卡中的 CSP 标头包含子域名，从 CSP 标头提取域。


##### 使用图标 favicon 哈希枚举子域名

Favicon 是与网站关联的小图标，通常命名为 “favicon.ico”。当网站的子域使用相同的 favicon 时，每个 favicon 将具有相同的哈希值。通过分析和比较这些哈希值，可以枚举子域名。该技术有效，因为相同 favicon 的哈希值在不同子域中是相同的。

**步骤 1：下载 Favicon**

第一步是下载 favicon。为此，我们可以使用 “curl”。

**命令**
```bash
curl -s www.paypalobjects.com/webstatic/icon/favicon.ico -o favicon.ico
```

**步骤 2：生成 MurmurHash**

下一步是为 favicon 生成 MurmurHash。MurmurHash 是从 “favicon.ico” 文件内容计算出的 32 位哈希值。这可以通过使用 Python 的 mmh3 模块的 MurmurHash 来实现。

**代码**
```bash
cat favicon.ico | base64 | python3 -c "import mmh3, sys; print(mmh3.hash(sys.stdin.buffer.read()))"
```

此命令获取 favicon.ico 文件，将其转换为 base64 格式，然后使用 Python 中的 mmh3 模块计算 MurmurHash。

**步骤 3：使用 Shodan 搜索**

下一步是在 Shodan 上搜索该哈希，以检索使用相同 favicon 的更多域。我们将使用 ShodanHQ 中的 “http.favicon.hash” 标志来实现。

**命令**
```
http.favicon.hash:309020573
```

##### TLS 证书透明度日志

证书透明度（CT）日志是已颁发 TLS 证书的公开可访问记录。可以搜索这些日志以识别与目标组织关联的主机名和子域名，包括预发布系统、管理界面、遗留系统或其他外部可访问的服务。

审查 CT 日志可能会揭示通过 DNS 区域传输、反向查找或搜索引擎查询无法直接发现的主机名。测试人员应提取发现的主机名，并通过 DNS 解析进行验证，以确定它们是否活跃且在评估的定义范围内。

在审查 CT 日志数据时，请考虑：

- 指示开发、预发布或测试环境的主机名。
- 管理或管理界面。
- 可能仍然可访问的已弃用或遗留系统。
- 可能暗示额外未发现子域名的通配符证书。

从 CT 日志收集的信息应在进一步测试之前进行验证，以确认所有权和相关性。

查询 CT 日志的一种常见方法是使用聚合证书数据的公开搜索门户。例如，测试人员可以搜索颁发给 `example.com` 的证书，并查看列出的子域名。

例如：`https://crt.sh/?q=%25.example.com`

结果可能列出子域名，如 `dev.example.com`、`staging.example.com` 或其他未直接从主站点引用的主机名。发现的主机名应在进一步测试之前通过 DNS 解析进行验证。

工具：
- crt.sh
- Censys
- Cert Spotter

##### 使用 SSL 证书主题备用名称（SAN）枚举子域名

当网站接受 HTTPS 连接，则 SSL/TLS 证书上的通用名称（CN）和主题备用名称（SAN）可能包含一个或多个主机名。因此，通过检查此字段，可以发现可能无法通过传统 DNS 查找看到的子域名。

可以通过手动检查证书或使用其他工具（如 OpenSSL）获取 CN 和 SAN：

```sh
openssl s_client -connect 93.184.216.34:443 </dev/null 2>/dev/null | openssl x509 -noout -text | grep -E 'DNS:|Subject:'

Subject: C = US, ST = California, L = Los Angeles, O = Internet Corporation for Assigned Names and Numbers, CN = www.example.org
DNS:www.example.org, DNS:example.com, DNS:example.edu, DNS:example.net, DNS:example.org, DNS:www.example.com, DNS:www.example.edu, DNS:www.example.net
```

该命令启动与 `api.paypal.com` 的 SSL/TLS 连接以检索证书，然后从 SAN 字段提取并格式化子域名以便阅读。

但是，如果 Web 服务器没有受信任的证书，或者使用了通配符，则这可能不会返回任何有效信息。

##### 使用 Web 存档进行子域名枚举

Web 存档捕获并存储网站的历史版本，这意味着它们可能包含曾经活跃但现已从 Web 上消失的子域名信息。有几种工具可用于此目的，其中最流行的是 “gau” （[https://github.com/lc/gau](https://github.com/lc/gau)）。该工具特别适用于从各种来源（包括搜索引擎、AlienVault 的 Open Threat Exchange、Wayback Machine、Common Crawl 和 URLScan）获取给定域名的已知 URL。

以下命令利用 “gau” 查询 `example.com`，并使用 “--subs” 标志返回子域名：

**命令**
```bash
echo example.com | gau --subs
```

此命令检索与 `example.com` 子域相关的 URL 列表。要从这些 URL 中提取域，我们可以选择使用带有正则表达式的 “grep”。或者，你可以使用命令行工具 “unfurl” （[https://github.com/tomnomnom/unfurl](https://github.com/tomnomnom/unfurl)） 实现相同结果。

**命令**
```bash
echo example.com | gau --subs | unfurl -u domains | sort -u
```

在此命令中，“gau --subs” 从 Web 存档获取数据，“unfurl -u domains” 提取域，“sort -u” 部分对数据进行排序。

### 主动收集技术

#### DNS 区域传输技术

区域传输通过 TCP 端口 53 执行，以将当前 DNS 区域材料传播到支持该操作的其他名称服务器。

区域文件包含与特定域和 IP 块相关的 DNS 记录。配置错误的服务器会接受公共互联网的传输请求，利用这一点来绘制给定网络的地图。

这种技术如今使用有限，因为 DNS 服务器很大程度上不接受区域传输。然而，仍然值得尝试。

首先，测试人员必须确定目标（或与其相关）的名称服务器。通过 `nslookup`、`host` 或 `dig` 等工具请求 DNS NS 记录来确定其名称服务器。

获取域的权威服务器详细信息后，尝试针对权威名称服务器执行正向区域传输。并在完成端口扫描后，针对范围内任何暴露 TCP 端口 53 的名称服务器进行尝试。

-   示例1：使用 dig

    ```bash
    # 获取域的名称服务器
    dig nmap.org ns

    # 执行区域传送
    dig @ns1.linode.com nmap.org axfr
    ```
-   示例2：使用 host

    ```sh
    # 获取域的名称服务器，-t 参数的查询类型为 ns
    host -t ns 域名

    # -l 执行区域传送
    host -l 域名 名称服务器
    ```

若成功则得到主机内部 IP 地址的详细信息，之后使用 IP 块进行反向查询网段（PTR 记录）。示例使用 `dig` 执行 `198.171.79.0/24` 子网的反向区域传输。

    ```bash
    $ dig @glb-ns4.it.verio.net 79.171.198.in-addr.arpa axfr
    ```

#### 正向 DNS 猜测 - DNS 枚举

通常名称服务器不允许区域传输，于是进行主动猜测策略来识别有效的 DNS 地址记录，包括：
- NSEC 和 NSEC3 记录枚举
- 使用 A 记录请求进行字典攻击

暴力破解尝试容易被记录检测，并且由于速率限制、正则表达式和其他旨在防止此类窥探的简单安全机制，通常极其耗时。并可能导致您的 IP 地址被服务器或其管理员记录或封禁。

##### NSEC 和 NSEC3 枚举

查询支持 DNSSEC 的名称服务器以揭示有效的主机名。对应 Nmap 脚本 `dns-nsec-enum` 和 `dns-nsec3-enum`

步骤
- 使用 Nmap 脚本得到一批主机名，并输出到文件
- 使用 dig 得到对应 ip 地址

DNS 暴力破解是发现子域名最有效的方式。这涉及使用常见子域名字典，因此暴力破解的结果直接取决于字典的质量。

使用 dig 半自动针对特定服务器执行正向猜测
- 步骤
    - 识别 `academi.com` 域的权威 DNS 服务器
    - 准备一个主机名字典文件（`academi.txt`）
    - 并使用 `dig` 查询特定服务器（`ns2.dnsbycomodo.com`）。

    ```bash
    cat /usr/share/fierce/hosts.txt | awk '{printf("%s.academi.com\n",$1);}' > out.txt
    dig @ns2.dnsbycomodo.net -f out.txt +noall +answer
    ```

下级名称服务器通常未加固，测试每个可用的 DNS 服务。

##### DNSValidator - 使用高效 DNS 服务器

在枚举子域名时，工具通常默认使用你的互联网服务提供商（ISP）提供的 DNS 解析器，或常用的如 Google 的 8.8.8.8。然而，这些可能不是最可靠的，因为可能存在来自单个 IP 请求数的速率限制，并且也可能受到地理限制。

因此，在开始子域名枚举之前，验证 DNS 解析器是否受支持且有效非常重要。为此，可以使用基于 Python 的工具 DNSValidator （[https://github.com/vortexau/dnsvalidator](https://github.com/vortexau/dnsvalidator)）。它采用多种方法测试和验证 DNS 解析器。该工具检查解析器列表的公共可访问性和有效性，排除不符合标准的解析器。

使用 DNSValidator，我们将向其提供从 public-dns.info （[https://public-dns.info/nameservers.txt](https://public-dns.info/nameservers.txt)） 获取的外部解析器列表。我们将使用其最佳输出作为子域名枚举工具的 DNS 输入。

注：DNSValidator 与 public-dns.info 均停止更新

### 使用 Amass 进行主动 + 被动子域名枚举

OWASP Amass （[https://github.com/owasp-amass/amass](https://github.com/owasp-amass/amass)） 是最先进的子域名枚举工具之一。它结合了主动和被动子域名枚举的优点。其独特功能包括通过公司名称查询，而不是提供 IP 地址和子域名。

#### Amass Intel 模块

“Intel” 模块可用于对组织执行 OSINT（开源情报）。该命令包含多个标志，可用于查询与组织相关的各种数据，包括 ASN、关联的 IP 段以及许多其他有趣细节。

```bash
# “org” 标志，将返回关联的 “ASN” 和 IP 段。
amass intel -org "google"

# “-asn” 标志针对特定 asn 返回域和子域。
amass intel -asn 44384

# 类似地，“whois” 标志可用于从 whois 记录返回域/子域。
amass intel -whois -d paypal.com
```

#### Amass Enum 模块

如前所述，“amass” 可用于进行主动和被动子域名枚举。这可以通过 “enum” 模块实现。

**主动模式**
主动模式利用正常模式的所有功能，并将其功能扩展到与发现的资产交互。这包括尝试获取 TLS 证书、执行 DNS 区域传输、使用 NSEC 遍历，甚至进行 Web 爬取。

```bash
# 对 `paypal.com` 执行主动枚举
amass enum -active -d paypal.com -src
```

**被动模式**
对于子域名的被动扫描，我们可以使用 enum 模块中的 “passive” 标志。

```bash
# 对 `paypal.com` 执行被动枚举
amass enum -passive -d paypal.com -src
```

#### Amass db 模块

“amass” 的另一个独特功能是它具有内置数据库，可用于存储和访问先前扫描的输出。数据库存储发现域名、IP 地址、子域名以及来自网络侦察活动的相关数据等信息。

```bash
# `-list` 列出数据库中保存的扫描
amass db -list
```

**命令**
```bash
# `-show` 标志查询特定记录的结果，示例显示 `owasp.org` 的结果
amass db -show -d owasp.org
```

#### Amass viz 模块

Amass 采用 “viz” 模块，可用于可视化存储在 Amass 图形数据库中的信息。它生成域之间发现的链接的可视化。这些结果可以导入到可视化 OSINT 工具中

**命令**
```bash
amass viz -d3 -dir paypal
```

#### Amass track 模块

Amass track 是一个模块，允许用户比较针对同一域执行的枚举结果。这对于理解域随时间演变的历时洞察特别有用。。

```bash
# 比较针对 `paypal.com` 的最后两次扫描
amass track -d paypal.com
```

### 使用 Httpx 排除死子域

即使重复域已被删除，收集到的子域文件可能仍包含非活动或已死的子域。

使用 httpx （[https://github.com/projectdiscovery/httpx](https://github.com/projectdiscovery/httpx)）。以下命令以 paypal-subdomain.txt 为输入，通过 “httpx” 遍历每个域，并将输出返回到 “paypal-alive-subdomain.txt”。

**命令**
```bash
cat paypal-subdomain.txt | httpx -sc -cl --title -o paypal-alive-subdomain.txt
```

### IPv6 主机枚举

通过 AAAA 的 DNS 猜测识别 IPv6 服务器。

使用 `thc-ipv6` 项目中的 `dnsdict6` 程序识别 `ripe.net` 域内可预测主机名的 IPv6 地址。

```bash
dnsdict6 -s ripe.net
```

使用 `thc-ipv6` 项目中的 `dnsrevenum6` 程序查询 IPv6 地址的主机名

```sh
./dnsrevenum6 dns.test.com 2001:db8:42a8::/48
```
