## IP 网络扫描

在识别出感兴趣的 IP 地址块后，将进行主动扫描以绘制网络地图、编录可访问的主机并识别暴露的服务
- 使用 Nmap 执行初始网络扫描
- 进行底层评估以了解网络配置
- 使用 Nmap 和 Metasploit 执行轻度漏洞扫描
- 批量漏洞扫描（使用 Nessus、Qualys 等工具）
- 规避入侵检测和防御机制

然后进行手动测试以调查漏洞、利用已知缺陷并发起暴力密码猜测攻击。

端口扫描
    寻找目标上的开放端口以识别处于活动状态的端口，发现目标上运行的服务。这些服务是在目标上执行并运行的应用程序，它们开放了监听端口，准备接受连接。

    通过评估这些开放端口、它们返回的信息或当特定数据发送给它们时的行为，以发现该端口上运行的应用程序名称、版本、配置信息。


## 使用 Nmap 进行初始网络扫描

### ICMP

有用的 ICMPv4 消息类型：
- **类型 8（回显请求）**：由 ping 和其他实用程序用于识别可访问的主机。
- **类型 13（时间戳请求）**：以十进制格式提供来自目标的系统时间信息。

IANA 维护着全面的 ICMPv4 和 ICMPv6 消息类型列表[^1]。近年来许多有用的 ICMPv4 类型已被弃用，包括 17（地址掩码请求）和 37（域名请求）。在遗留网络中，这些类型仍可能产生有用信息。

[^1]: 参见 IANA.org 的“Internet Control Message Protocol version 6 (ICMPv6) Parameters”和“Internet Control Message Protocol (ICMP) Parameters”。

#### 使用 Nmap 进行 ICMPv4 扫描

在 Nmap 中，使用 `-PE -PP -PM` 标志通过 ICMPv4 回显、时间戳和子网掩码请求执行主机发现。

```sh
nmap -PE -PP -PM -sn -vvv -n 10.12.5.0/24
```

#### 使用广播地址

在 IP 网络中，最后一个地址八位组被保留为广播地址（例如 `10.10.5.0/24` 中的 `10.10.5.255`），用于向子网内的所有主机发送流量。

使用 ping 识别子网内的主机

    ```bash
    ping -b 10.10.5.255
    ```

通过 ICMP 回显广播识别其他子网

    ```bash
    ping -b 255.255.255.255
    ```

### TCP

Nmap 支持许多 TCP 扫描模式，在执行隐蔽扫描和理解底层网络配置时特别有用。

使用 Nmap 执行基本的 IPv4 TCP SYN 隐蔽扫描以识别可访问的服务，这种扫描通常可以在不触发防御的情况下检测端口

    ```bash
    nmap -sS 10.10.5.10
    ```

默认情况下，Nmap 将执行主机发现并识别可访问的主机进行扫描。
在测试加固环境时，由于 ICMP 包可能已被防火墙阻止，应使用 `-Pn` 标志强制扫描每个地址。
较慢的时间策略（例如 `-T2`）也很有用，因为激进的策略可能触发 SYN 洪水保护并导致数据包被丢弃。

Nmap 为每个端口返回一个状态（`open`、`closed` 或 `filtered`）。
SYN 探测引发的四种响应变体：
- SYN/ACK 数据包（表示开放端口）
- RST/ACK（表示关闭）
- 无响应
- ICMP 类型 3 消息（表示过滤）

开放端口行为
- SYN →
- ← SYN/ACK
- RST →

关闭端口行为
- SYN →
- ← SYN/ACK

过滤端口行为（无响应）
- SYN →

过滤端口行为（ICMP 响应）
- SYN →
- ← ICMP type3

应该扫描每个 TCP 端口，从 0 到 65,535。Nmap 默认使用有限的内部通用端口列表，这会引入盲点。

IPv4 防火墙和路由器通常会产生 ICMP 类型 3（目的地不可达）响应，这提供了对网络配置的洞察。

常用 ICMP 类型 3 消息代码[^3]

| 代码 | 描述 |
|------|------|
| 0 | 网络不可达 |
| 1 | 主机不可达 |
| 2 | 协议不可达 |
| 3 | 端口不可达 |
| 6 | 目标网络未知 |
| 7 | 目标主机未知 |
| 9 | 通信被管理性禁止（网络） |
| 10 | 通信被管理性禁止（主机） |
| 13 | 通信被管理性禁止（通用） |

[^3]: RFC 792 和 RFC 1812 提供了详尽列表。

### UDP

对 UDP 的无连接特性的扫描方式：
- **负面扫描**
    基于 ICMP 不可达响应推断开放端口
- **载荷扫描**
    使用正确格式的数据报以引发服务响应来识别（例如 DNS、DHCP、TFTP 等）

ICMP 是一个不可靠的指标，因为注重安全的组织倾向于过滤这些消息，并且大多数操作系统默认对 ICMP 响应进行速率限制。

Nmap 通过 `-sU` 标志结合使用负面扫描和载荷扫描（而不仅仅是单一模式）。这常常会使输出变得模糊（同时返回 `open` 和 `open|filtered` 状态）。

```sh
nmap -Pn -sU -open -vvv -n 10.3.0.1
```

使用 `-sU -sV` 标志，可以主动探测每个 UDP 端口并查看哪些端口响应。然而，对于模糊的 `open|filtered` 端口，以这种方式运行 Nmap 非常缓慢，在测试大型网络时不切实际。


其他可用于执行 UDP 载荷扫描的工具
- [naabu](https://github.com/projectdiscovery/naabu)
- [zmap](https://github.com/zmap/zmap)
- [RustScan](https://github.com/bee-san/RustScan)
- [masscan](https://github.com/robertdavidgraham/masscan)
- [nrev](https://github.com/shellrow/nrev)

### SCTP

SCTP 与 TCP 和 UDP 并列，如图 6-1 所示。该协议旨在通过 IP 提供电话数据传输，复制了七号信令系统（SS7）的许多可靠性特性，并支撑了一个称为 SIGTRAN 的更大协议族。SCTP 受到包括 IBM AIX、Oracle Solaris、HP-UX、Linux、Cisco IOS 和 VxWorks 在内的操作系统的支持。

#### 数据包格式

每个 SCTP 数据包包含一个头部和关联的块。源端口和目标端口值为 16 位（范围 0 到 65,535）， 以及 8 位块类型值。根据类型，块值字段会有所不同。

#### Nmap 支持

通过两种数据包类型扫描来发现可访问的 SCTP 服务：
- **INIT**：Nmap（`-sY`）。
    类似于 TCP SYN 扫描，向每个端口发送 SCTP INIT 块。INIT ACK 响应表示端口开放，而 ABORT 块表示端口关闭。

- **COOKIE ECHO**：Nmap （`-sZ`）。
    对于包含 COOKIE ECHO 块的数据包，若目标端口开放则丢弃，若目标端口关闭则回复一个 ABORT 块。这种扫描类型隐蔽，但不能区分开放端口和过滤端口（仅显示关闭的端口）。

其中许多协议不对应于 IETF RFC 文档，而是由国际电信联盟（ITU）和第三代合作伙伴计划（3GPP）标准描述。例如，ITU-T Q.1902.1[^7] 定义了 SCTP 端口 3097 使用的承载独立呼叫控制（BICC）协议。

[^7]: 参见 ITU.int 上的“Q.1902.1: Bearer Independent Call Control protocol (Capability Set 2): Functional description”。

## 底层 IP 评估 - 指纹识别原理

通过精心构造探测数据包并检查响应，执行以下操作：

- 对目标主机或网络设备的操作系统进行指纹识别
- 识别具有 IP 栈实现缺陷的主机
- 枚举过滤设备并推测其策略
- 揭示配置错误的系统的内部 IP 地址

操纵 IP 和 TCP 头值的工具：
- **Hping3**
- **Scapy**
- **Nmap**
- **Firewalk**

通过操纵这些字段并对特定探测的响应进行采样，以推测底层网络配置。

### 构造任意数据包

https://scapy.readthedocs.io/en/latest/usage.html

使用 Scapy，区分
- 端口开放 SYN/ACK
    ```py
    sr1(IP(dst="scanme.nmap.org")/TCP(dport=80,flags="S"))
    ```
- 端口关闭 RST/ACK
- 过滤 ICMP type3
- 丢弃无响应

### TCP/IP 栈指纹识别

操作系统使用不同的 IP TTL 和 WINDOW 值准备数据包。数据包的 TTL 每经过一跳减少一次，因此该值在测试期间会根据与目标主机的距离而减小。

操作系统使用的默认 TCP/IP 值

| 操作系统 | IP TTL（初始） | TCP WINDOW |
|----------|---------------|------------|
| Linux | 64 | 5840 |
| FreeBSD | 64 | 65535 |
| Windows XP | 128 | 65535 |
| Windows 7 及更高版本 | 128 | 8192 |
| Cisco IOS | 255 | 4128 |

### IP ID 分析

许多 TCP/IP 实现为出站数据包设置递增的 IP ID 值。可利用此行为来识别防火墙后的各个主机并测量网络活动。

如果从主机接收到的每个数据包的 IP ID 值都是递增的，则该主机还可以作为一个不知情的第三方（称为 **僵尸**）来促进使用 Nmap 进行隐蔽端口扫描（见 nmap 空闲扫描文档）。

#### 使用 Scapy 进行 IP ID 采样

Scapy 可以对 IP ID 值进行采样并绘图。通过结果图可识别负载均衡器后面的各个主机。Philippe Biondi 和 Arnaud Ebalard 的演讲“Scapy and IPv6 networking”[^13] 描述了其他有用的采样策略。

[^13]: Philippe Biondi and Arnaud Ebalard，“Scapy and IPv6 Networking”，在 Hack in the Box Security Conference 上展示，马来西亚吉隆坡，2006 年 9 月 18-21 日。


```bash
apt install python3-gnuplot
scapy
>>> ans,unans=sr(IP(dst="scanme.nmap.org")/TCP(sport=[RandShort() for _ in range(100)]))
>>> ans.plot(lambda s,r:r.id)
```

#### 使用 Nmap 进行 IP ID 采样

https://nmap.org/book/man-os-detection.html

Nmap 的 `-O` 操作系统检测选项的多项测试中包含 TCP 序列号和 IP ID 生成。在此模式下，TCP 时间戳选项 (RFC 1323) 用于计算主机正常运行时间。

```bash
nmap -F -sS -O -open -v scanme.nmap.org
```

#### 使用 Nmap 进行隐蔽 IP ID 扫描

在识别出返回递增 IP ID 值的合适僵尸候选主机后，可通过 Nmap 进行 IP ID 头空闲扫描。该过程使用僵尸主机作为指示器。这种扫描模式的一个好处是，可以从某个特定角度映射 ACL（例如，识别分支办公室中产生顺序 IP ID 值的路由器，并将其用作僵尸）。

- `-sI 空闲僵尸主机 目标ip`

默认情况下，Nmap 会向僵尸主机的 TCP 端口 80 发送心跳包。使用 `-Pn` 标志抑制探测尤其重要，这样所有数据包都将从僵尸主机的地址发送到目标。


### 操纵 TTL 以分析 ACL

Nmap 基于网络响应或无响应来识别过滤器。更深入的评估使用 **Firewalk** 进行，它操纵数据包的 TTL 字段，从一定距离（即比给定网关多一跳）扫描目标。

Firewalk 通过特定网关（`gw.test.org`）测试六个 TCP 端口到目标 `www.test.org` 。该功能也已通过 `firewalk` NSE 脚本移植到 Nmap。

    ```bash
    firewalk -n -S21,22,23,25,53,80 -pTCP gw.test.org www.test.org
    ```

Firewalk 计算到网关的距离，并向目标发送 TTL 多一跳的数据包。基于响应，该工具能够映射网关的过滤策略。例如：

- 如果收到 ICMP 类型 11，代码 0（传输中 TTL 超时）消息，则数据包已通过并随后生成了响应。
- 如果数据包被静默丢弃，则可能是在网关上完成的。
- 如果收到 ICMP 类型 3，代码 13（通信被管理性禁止）消息，则表示正在使用诸如路由器 ACL 之类的过滤器。

如果数据包被静默丢弃，这并不一定意味着到目标主机和端口的流量被过滤。一些防火墙知道数据包即将过期，并且无论策略是否允许，都会发送过期消息。

**Firewalk 适用于真正的 IP 路由环境，而不是使用网络地址转换（NAT）的环境**。Dave Goldsmith 和 Mike Schiffman 的论文详细描述了这种测试方法[^16]。

⚠️ Firewalk 似乎不适用于普通的家用 NAT 路由器。traceroute 的 UDP/TCP/ICMP 节点本跳过期探测均可成功追踪，而 Firewalk 的网关下跳过期探测无响应 ⚠️

[^16]: Dave Goldsmith and Mike Schiffman，“Firewalking”，Cambridge Technology Partners 白皮书，1998 年 10 月。

### 揭示内部 IP 地址

配置错误的路由器、防火墙和网络设备有时会使用非公共源地址响应网络探测。

使用 `tcpdump` 识别测试期间从私有地址接收到的数据包

只要有可能，避免从使用 NAT 的网络进行扫描，因为网关的行为可能会扭曲结果。为了获得最佳在线结果，请从能够直接路由 IP 流量进出 Internet 的主机进行扫描（即不在执行地址转换的防火墙或路由器之后）。
