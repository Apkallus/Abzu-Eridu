# 第 3 章：现代 Web 应用程序的结构

在您能够有效地对 Web 应用进行侦察之前，最好先了解许多 Web 应用所共有的常见技术。这些依赖关系涵盖了从 JavaScript 辅助库和预定义的 CSS 模块，一直到 Web 服务器甚至操作系统。通过理解这些依赖关系在应用栈中的角色及其常见实现方式，就能更容易地快速识别它们并查找配置错误。

## 现代与传统 Web 应用对比

当今的 Web 应用通常构建在 10 年前并不存在的技术之上。用于构建 Web 应用的工具在这段时间里发展如此之快，以至于如今它有时看起来像是一个完全不同的专业领域。

十年前，大多数 Web 应用使用服务端框架构建，这些框架渲染一个 HTML/JS/CSS 页面，然后发送给客户端。当需要更新时，客户端只需从服务器请求另一个要渲染并通过 HTTP 传输的页面。不久之后，随着 AJAX（异步 JavaScript 和 XML）的兴起，Web 应用开始更频繁地使用 HTTP，允许在页面会话中通过 JavaScript 发起网络请求。

如今，许多应用实际上更适合表示为两个或多个通过网络协议通信的应用，而非单个单体应用。这是当今 Web 应用与十年前 Web 应用之间的一大架构差异。

通常，当今的 Web 应用由多个通过表述性状态传递（REST）API 连接的应用组成。这些 API 是无状态的，仅用于满足从一个应用到另一个应用的请求。这意味着它们实际上不存储任何关于请求者的信息。

如今许多客户端（UI）应用在浏览器中以更接近传统桌面应用的方式运行。这些客户端应用管理自己的生命周期循环，请求自己的数据，并且在初始引导完成后不需要重新加载页面。

部署到 Web 浏览器的独立应用与众多服务器通信并不少见。考虑一个允许用户登录的图像托管应用——它很可能有一个专门的主机/分发服务器位于一个 URL，以及一个单独的 URL 用于管理数据库和登录。

可以肯定地说，当今的应用通常实际上是许多独立但共生应用的组合，协调一致地工作。这可以归因于更清晰定义的网络协议和 API 架构模式的发展。

普通的现代 Web 应用可能会使用以下多项技术：

- REST API
- JSON 或 XML
- JavaScript
- SPA 框架（ReactJS、VueJS、EmberJS、AngularJS）
- 认证和授权系统
- 一个或多个 Web 服务器（通常运行在 Linux 服务器上）
- 一个或多个 Web 服务器软件包（ExpressJS、Apache、NGINX）
- 一个或多个数据库（MySQL、MongoDB 等）
- 客户端本地数据存储（cookie、Web 存储、IndexedDB）

> 这并非一份详尽列表，考虑到互联网上现有数十亿个独立网站，本书不可能涵盖所有 Web 应用技术。如果您需要快速了解本章未列出的特定技术，应使用其他书籍和编程网站（如 Stack Overflow）。

其中一些技术十年前就存在，但不能说它们在那段时间没有改变。数据库已存在数十年，但 NoSQL 数据库和客户端数据库无疑是较新的发展。

全栈 JavaScript 应用的发展也直到 Node.js 和 npm 开始被快速采用才成为可能。在过去大约十年里，Web 应用的格局变化如此之快，以至于这些技术中的许多已经从无人知晓变得无处不在。

还有更多技术即将出现：例如用于在本地存储请求的 Cache API，以及作为客户端到服务器（甚至客户端到客户端）通信替代网络协议的 WebSockets。最终，浏览器打算完全支持一种称为 WebAssembly 的汇编代码变体，这将允许使用非 JavaScript 语言在浏览器中编写客户端代码。

这些新兴技术中的每一种都带来了新的安全漏洞，可以被用于善意或恶意目的。无论是利用还是保护 Web 应用，现在都是一个激动人心的时代。

不幸的是，我无法解释当今 Web 上使用的每一项技术——那需要单独一本书！但本章剩余部分将对前面列出的技术进行介绍。您可以集中精力学习那些您还不够熟悉的技术。

## REST API

REST 代表表述性状态传递（Representational State Transfer），这是一种花哨的说法，用来定义具有几个独特特性的 API：

**必须与客户端分离**  
REST API 旨在构建高度可扩展但简单的 Web 应用。将客户端与 API 分离，同时遵循严格的 API 结构，使得客户端应用可以轻松地从 API 请求资源，而无法自行调用数据库或执行服务端逻辑。

**必须无状态**  
按照设计，REST API 仅接受输入并提供输出。API 不得存储关于客户端连接的任何状态。然而，这并不意味着 REST API 不能执行认证和授权——相反，授权应该被令牌化并在每个请求中发送。

**必须易于缓存**  
为了正确扩展通过互联网交付的 Web 应用，REST API 必须能够轻松地将响应标记为可缓存或不可缓存。由于 REST 还对从哪个端点提供什么数据有非常严格的定义，因此在设计良好的 REST API 上配置这一点实际上非常容易。理想情况下，缓存应以编程方式管理，以避免意外将特权信息泄露给另一个用户。

**每个端点应定义一个特定的对象或方法**  
通常这些是层次化定义的；例如，`/moderators/joe/logs/12_21_2018`。这样，REST API 可以轻松使用 HTTP 动词，如 GET、POST、PUT 和 DELETE。因此，具有多个 HTTP 动词的一个端点就变成了自文档化的。

想要修改版主账户“joe”？使用 `PUT /moderators/joe`。想要删除日志 `12_21_2018`？只需简单推导：`DELETE /moderators/joe/logs/12_21_2018`。

由于 REST API 遵循定义良好的架构模式，Swagger 等工具可以轻松集成到应用中并记录端点，使其他开发者更容易理解端点的意图（见图 3-1）。

图 3-1. Swagger，一个旨在与 REST API 轻松集成的自动 API 文档生成器

在过去，大多数 Web 应用使用简单对象访问协议（SOAP）结构的 API。REST 相对于 SOAP 有几个优势：

- 请求针对数据，而非函数
- 请求易于缓存
- 高度可扩展

此外，虽然 SOAP API 必须使用 XML 作为其传输数据格式，但 REST API 可以接受任何数据格式，但通常使用 JSON。JSON 比 XML 更轻量（更少冗长）且更易于人类阅读，这也使 REST 在竞争中占优。

以下是用 XML 编写的示例载荷：

```xml
<user>
  <username>joe</username>
  <password>correcthorsebatterystaple</password>
  <email>joe@website.com</email>
  <joined>12/21/2005</joined>
  <client-data>
    <timezone>UTF</timezone>
    <operating-system>Windows 10</operating-system>
    <licenses>
      <videoEditor>abc123-2005</videoEditor>
      <imageEditor>123-456-789</imageEditor>
    </licenses>
  </client-data>
</user>
```

同样地，用 JSON 编写的相同载荷：

```json
{
  "username": "joe",
  "password": "correcthorsebatterystaple",
  "email": "joe@website.com",
  "joined": "12/21/2005",
  "client_data": {
    "timezone": "UTF",
    "operating_system": "Windows 10",
    "licenses": {
      "videoEditor": "abc123-2005",
      "imageEditor": "123-456-789"
    }
  }
}
```

您遇到的大多数现代 Web 应用要么使用 RESTful API，要么使用提供 JSON 的类 REST API。在特定的企业应用之外（出于遗留兼容性而保持如此严格的设计），遇到 SOAP API 和 XML 的情况越来越罕见。

当您尝试逆向工程 Web 应用的 API 层时，理解 REST API 的结构很重要。掌握 REST API 的基本原理将给您带来优势，因为您会发现许多您希望调查的 API 遵循 REST 架构——而且，您可能希望使用或集成到工作流中的许多工具也将通过 REST API 暴露。

## JavaScript 对象表示法（JSON）

REST 是一种架构规范，定义了 HTTP 动词应如何映射到服务器上的资源（API 端点和功能）。如今大多数 REST API 使用 JSON 作为其传输数据格式。

考虑一下：应用的 API 服务器必须与其客户端（通常是浏览器或移动应用中的一些代码）通信。没有客户端/服务器关系，我们就无法跨设备存储状态并在账户之间持久化该状态。所有状态都必须在本地存储。

由于现代 Web 应用需要大量客户端/服务器通信（用于数据下游交换和以 HTTP 动词形式的上游请求），以临时格式发送数据是不可行的。数据的传输格式必须标准化。

JSON 是这个问题的一个潜在解决方案。JSON 是一个开放标准（非专有）文件格式，满足许多有趣的要求：

- 非常轻量（减少网络带宽）。
- 需要很少的解析（降低服务器/客户端硬件负载）。
- 易于人类阅读。
- 层次化（可以表示数据间的复杂关系）。
- JSON 对象的表示方式与 JavaScript 对象非常相似，使得在浏览器中消费 JSON 和构建新 JSON 对象变得非常容易。

如今所有主流浏览器都原生支持 JSON 解析（且速度快！），除前述优点外，这使得 JSON 成为在无状态服务器和 Web 浏览器之间传输数据的极佳格式。

以下 JSON：

```json
{
  "first": "Sam",
  "last": "Adams",
  "email": "sam.adams@company.com",
  "role": "Engineering Manager",
  "company": "TechCo.",
  "location": {
    "country": "USA",
    "state": "california",
    "address": "123 main st.",
    "zip": 98404
  }
}
```

可以在浏览器中轻松解析为 JavaScript 对象：

```javascript
const jsonString = `{
  "first": "Sam",
  "last": "Adams",
  "email": "sam.adams@company.com",
  "role": "Engineering Manager",
  "company": "TechCo.",
  "location": {
    "country": "USA",
    "state": "california",
    "address": "123 main st.",
    "zip": 98404
  }
}`;

// 将服务器发送的字符串转换为对象
const jsonObject = JSON.parse(jsonString);
```

JSON 灵活、轻量且易于使用。它并非没有缺点，因为任何轻量级格式与重量级替代方案相比都有权衡。这些将在本书后面评估 JSON 与其竞争对手之间的具体安全差异时讨论，但现在重要的是要明白，如今浏览器和服务器之间的大量网络请求是以 JSON 格式发送的。

熟悉阅读 JSON 字符串，并考虑在浏览器或代码编辑器中安装格式化 JSON 字符串的插件。能够在短时间内快速解析这些字符串并找到特定键，对于渗透测试各种 API 将非常有价值。

## JavaScript

在本书中，我们将不断讨论客户端和服务端应用。服务器是一台计算机（通常是功能强大的计算机），位于数据中心（有时称为云）中，负责处理对网站的请求。有时这些服务器实际上可能是一组多台服务器；其他时候可能只是一台用于开发或日志记录的轻量级服务器。

另一方面，客户端是用户可以访问并用于使用 Web 应用的任何设备。客户端可以是手机、商场售货亭或电动汽车中的触摸屏——但就我们的目的而言，通常只是 Web 浏览器。

服务器可以配置为运行几乎任何您能想象的软件，使用任何您能想象的语言。当今的 Web 服务器运行在 Python、Java、JavaScript（JS）、C++ 等语言上。客户端（特别是浏览器）则没有这种奢侈。JavaScript 是一种动态编程语言，最初设计用于互联网浏览器。JavaScript 不仅是一种编程语言，也是 Web 浏览器中客户端脚本的唯一编程语言。JavaScript 现在已用于许多应用，从移动端到物联网（IoT）。

本书中的许多代码示例都是用 JavaScript 编写的（见图 3-2）。在可能的情况下，后端代码示例也使用 JavaScript 语法编写，以便不在上下文切换上浪费时间。我会尽量保持 JavaScript 代码简洁明了，但我可能会使用一些 JavaScript 支持但在其他语言中不那么流行（或广为人知）的构造。

JavaScript 是一门独特的语言，因为其开发与浏览器及其伙伴文档对象模型（DOM）的发展紧密相连。因此，在继续之前，您可能需要了解一些 quirks。

### 变量与作用域

在 ES6 JavaScript（一个较新版本）中，有四种定义变量的方式：

```javascript
// 全局定义
age = 25;

// 函数作用域
var age = 25;

// 块级作用域
let age = 25;

// 块级作用域，不可重新赋值
const age = 25;
```

这些看起来可能相似，但在功能上非常不同。

如果不包含 `var`、`let` 或 `const` 这样的关键字，您定义的任何变量都会被提升到全局作用域。这意味着作为全局作用域子对象的任何其他对象都能访问该变量。一般来说，这被认为是不良实践，我们应该避免这样做。（这也可能是严重安全漏洞或功能错误的根源。）

注意，所有缺少 `var`、`let` 或 `const` 的变量也会在浏览器中的 `window` 对象上添加一个指针：

```javascript
// 定义全局整数
age = 25;

// 直接调用（返回 25）
console.log(age);

// 通过 window 上的指针调用（返回 25）
console.log(window.age);
```

当然，这可能在 `window`（浏览器 DOM 依赖以维护窗口状态的对象）上引起命名空间冲突，这是另一个避免使用它的好理由：

```javascript
var age = 25;
```

任何使用标识符 `var` 定义的变量，其作用域为最近的函数，如果没有外部函数块定义，则为全局（在全局情况下，它出现在 `window` 上，与无标识符变量类似，如前所示）。

这种变量有点令人困惑，这可能是最终引入 `let` 的部分原因。

```javascript
const func = function() {
  if (true) {
    // 在 if 块内定义 age
    var age = 25;
  }
  /*
   * 记录 age 将返回 25
   *
   * 这是因为 var 标识符绑定到最近的
   * 函数，而非最近的块。
   */
  console.log(age);
};
```

在前面的示例中，使用 `var` 标识符定义了一个值为 25 的变量。在大多数其他现代编程语言中，尝试记录 `age` 时它将是未定义的。不幸的是，`var` 不遵循这些一般规则，而是将自身作用域限定为函数而非块。这可能导致新 JavaScript 开发者陷入调试混乱。

```javascript
let age = 25;
```

ECMAScript 6（JavaScript 规范）引入了 `let` 和 `const`——两种实例化对象的方式，其行为更类似于其他现代语言中的方式。

如您所料，`let` 是块级作用域的。这意味着：

```javascript
const func = function() {
  if (true) {
    // 在 if 块内定义 age
    let age = 25;
  }
  /*
   * 这次，console.log(age) 将返回 `undefined`。
   *
   * 这是因为 `let` 与 `var` 不同，它绑定到最近的块。
   * 将作用域绑定到最近的块而非最近的函数，
   * 通常被认为更利于可读性，
   * 并减少作用域相关错误。
   */
  console.log(age);
};
```

```javascript
const age = 25;
```

`const` 与 `let` 非常相似，也是块级作用域的，但也不能重新赋值。这使得它类似于 Java 等语言中的 `final` 变量：

```javascript
const func = function() {
  const age = 25;
  /*
   * 这将导致：TypeError: invalid assignment to const `age`
   *
   * 与 `let` 非常相似，`const` 是块级作用域的。
   * 主要区别在于 `const` 变量在实例化后不支持重新赋值。
   *
   * 如果对象被声明为 const，其属性仍然可以更改。
   * 因此，`const` 确保内存中指向 `age` 的指针不被改变，
   * 但不关心 `age` 的值或 `age` 上的属性是否改变。
   */
  age = 25;
};
```

总的来说，您应始终努力在代码中使用 `let` 和 `const`，以避免错误并提高可读性。

### 函数

在 JavaScript 中，函数是对象。这意味着它们可以使用上一节中的变量和标识符进行赋值和重新赋值。

这些都是函数：

```javascript
// 匿名函数
function() {};

// 全局声明的命名函数
a = function() {};

// 函数作用域命名函数
var a = function() {};

// 块级作用域命名函数
let a = function() {};

// 块级作用域命名函数，不可重新赋值
const a = function() {};

// 继承父级上下文的匿名函数
() => {};

// 立即调用函数表达式（IIFE）
(function() {})();
```

第一个函数是匿名函数——这意味着它在创建后无法被引用。接下来的四个只是根据提供的标识符指定作用域的函数。这与我们之前创建 `age` 变量的方式非常相似。

第六个函数是箭头函数（简写函数）——它与其父级共享上下文（稍后会更多）。

最后一个函数是一种您可能只在 JavaScript 中找到的特殊类型函数，称为 IIFE——立即调用函数表达式。这是一个在加载时立即触发并在自己的命名空间内运行的函数。更高级的 JavaScript 开发者使用它们来封装代码块，使其无法从其他地方访问。

### 上下文

如果您能用任何其他非 JavaScript 语言编写代码，那么要成为一名优秀的 JavaScript 开发者，您需要学习五件事：作用域、上下文、原型继承、异步和浏览器 DOM。

JavaScript 中的每个函数都有自己附加的一组属性和数据。我们称之为函数的上下文。上下文不是固定不变的，可以在运行时修改。存储在函数上下文中的对象可以使用关键字 `this` 引用：

```javascript
const func = function() {
  this.age = 25;
  // 将返回 25
  console.log(this.age);
};

// 将返回 undefined
console.log(this.age);
```

如您所想象，许多烦人的编程错误都是因为上下文难以调试——尤其是当某个对象的上下文必须传递给另一个函数时。JavaScript 引入了一些解决方案来帮助开发者在函数之间共享上下文：

```javascript
// 创建一个带有 ageData 上下文的 getAge() 函数克隆
// 然后使用参数 'joe' 调用它
const getBoundAge = getAge.bind(ageData)('joe');

// 使用 ageData 上下文和参数 joe 调用 getAge()
const boundAge = getAge.call(ageData, 'joe');

// 使用 ageData 上下文和参数 joe 调用 getAge()
const boundAge = getAge.apply(ageData, ['joe']);
```

这三个函数 `bind`、`call` 和 `apply` 允许开发者将上下文从一个函数移到另一个函数。`call` 和 `apply` 的唯一区别是 `call` 接受参数列表，而 `apply` 接受参数数组。

两者可以轻松互换：

```javascript
// 将数组解构为列表
const boundAge = getAge.call(ageData, ...['joe']);
```

另一个帮助程序员管理上下文的新增功能是箭头函数，也称为简写函数。该函数继承其父级的上下文，允许上下文从父函数共享到子函数，而无需显式 `call`/`apply` 或 `bind`：

```javascript
// 全局上下文
this.garlic = false;

// 汤食谱
const soup = { garlic: true };

// 附加到 soup 对象的标准函数
soup.hasGarlic1 = function() { console.log(this.garlic); } // true

// 附加到全局上下文的箭头函数
soup.hasGarlic2 = () => { console.log(this.garlic); } // false
```

掌握这些管理上下文的方法将使您对基于 JavaScript 的服务器或客户端的侦察工作更轻松、更快捷。您甚至可能发现一些源于这些复杂性的语言特定漏洞。

### 原型继承

与许多建议使用基于类的继承模型的传统服务端语言不同，JavaScript 被设计为具有高度灵活的原型继承系统。不幸的是，由于很少有语言使用这种类型的继承系统，它经常被开发者忽视，其中许多人试图将其转换为基于类的系统。

在基于类的系统中，类像蓝图一样定义对象。在这样的系统中，类可以继承自其他类，并以这种方式创建层次关系。在像 Java 这样的语言中，子类使用 `extends` 关键字生成，或使用 `new` 关键字实例化。

JavaScript 并不真正支持这些类型的类，但由于原型继承的灵活性，可以在 JavaScript 原型系统之上通过一些抽象来模拟类的确切功能。在原型继承系统中，如 JavaScript，创建的任何对象都有一个名为 `prototype` 的属性附加到它上面。`prototype` 属性附带一个 `constructor` 属性，指向拥有该原型的函数。这意味着任何对象都可以用来实例化新对象，因为构造函数指向包含原型的对象，而原型包含构造函数。

这可能令人困惑，但这里有一个例子：

```javascript
/*
 * 用 JavaScript 编写的车辆伪类。
 *
 * 为了更清楚地演示原型继承基础，故意保持简单。
 */
const Vehicle = function(make, model) {
  this.make = make;
  this.model = model;
  this.print = function() {
    return `${this.make}: ${this.model}`;
  };
};

const prius = new Vehicle('Toyota', 'Prius');
console.log(prius.print());
```

当在 JavaScript 中创建新对象时，还会创建一个单独的对象 `__proto__`。该对象指向在创建该对象时调用其构造函数的原型。这允许对象之间进行比较，例如：

```javascript
const prius = new Vehicle('Toyota', 'Prius');
const charger = new Vehicle('Dodge', 'Charger');

/*
 * 如我们所见，“Prius”和“Charger”对象都是
 * 基于“Vehicle”创建的。
 */
prius.__proto__ === charger.__proto__;
```

通常，对象上的原型会被开发者修改，导致 Web 应用功能发生令人困惑的变化。最值得注意的是，由于 JavaScript 中的所有对象默认都是可变的，对 `prototype` 属性的更改可以在运行时的任何时刻发生。

有趣的是，这意味着与更严格设计的继承模型不同，JS 继承树可以在运行时改变。对象因此可以在运行时变形：

```javascript
const prius = new Vehicle('Toyota', 'Prius');
const charger = new Vehicle('Dodge', 'Charger');

/*
 * 这将失败，因为 Vehicle 对象
 * 没有 “getMaxSpeed” 函数。
 *
 * 因此，从 Vehicle 继承的对象也没有这样的函数。
 */
console.log(prius.getMaxSpeed()); // Error: getMaxSpeed is not a function

/*
 * 现在我们将一个 getMaxSpeed() 函数赋值给 Vehicle 的原型，
 * 所有从 Vehicle 继承的对象都将实时更新，因为
 * 原型从 Vehicle 对象传播到其子对象。
 */
Vehicle.prototype.getMaxSpeed = function() {
  return 100; // mph
};

/*
 * 因为 Vehicle 的原型已更新，
 * getMaxSpeed 函数现在将对所有子对象起作用。
 */
prius.getMaxSpeed(); // 100
charger.getMaxSpeed(); // 100
```

原型需要一段时间才能适应，但最终它们的强大和灵活性超过了学习曲线中的任何困难。在深入研究 JavaScript 安全时，理解原型尤其重要，因为很少有开发者完全理解它们。

此外，由于原型在被修改时会传播给子对象，在基于 JavaScript 的系统中发现了一种特殊类型的攻击，称为原型污染。这种攻击涉及修改父 JavaScript 对象，从而无意中改变子对象的功能。

### 异步

异步是那些“难搞懂，易记住”的概念之一，在网络编程中似乎经常出现。由于浏览器必须定期与服务器通信，并且请求和响应之间的时间是不标准的（考虑到载荷大小、延迟和服务器处理时间），因此在 Web 上经常使用异步来处理这种变化。

在同步编程模型中，操作按发生的顺序执行。例如：

```javascript
console.log('a');
console.log('b');
console.log('c');
// a
// b
// c
```

在前述情况下，操作按顺序发生，每次以相同顺序调用这三个函数时都可靠地拼出“abc”。

在异步编程模型中，这三个函数每次可能以相同顺序被解释器读取，但可能不按相同顺序解析。考虑这个依赖于异步日志函数的例子：

```javascript
// --- 尝试 #1 ---
async.log('a');
async.log('b');
async.log('c');
// a
// b
// c

// --- 尝试 #2 ---
async.log('a');
async.log('b');
async.log('c');
// a
// c
// b

// --- 尝试 #3 ---
async.log('a');
async.log('b');
async.log('c');
// a
// b
// c
```

第二次调用日志函数时，它们没有按顺序解析。为什么？在网络编程中，请求经常花费可变时间、超时，并且不可预测地运行。在基于 JavaScript 的 Web 应用中，这通常通过异步编程模型来处理，而不是简单地等待一个请求完成后再发起另一个请求。其好处是巨大的性能提升，可以比同步替代方案快数十倍。我们不强制请求一个接一个地完成，而是同时启动所有请求，然后在解析之前编程它们应在解析时做什么。

在旧版本的 JavaScript 中，这通常通过称为回调的系统来完成：

```javascript
const config = {
  privacy: public,
  acceptRequests: true
};

/*
 * 首先从服务器请求用户对象。
 * 一旦完成，从服务器请求用户资料。
 * 一旦完成，设置用户资料配置。
 * 一旦完成，console.log "success!"
 */
getUser(function(user) {
  getUserProfile(user, function(profile) {
    setUserProfileConfig(profile, config, function(result) {
      console.log('success!');
    });
  });
});
```

虽然回调与同步模型相比非常快速高效，但它们非常难以阅读和调试。后来的一种编程理念建议创建一个可重用的对象，该对象在给定函数完成后调用下一个函数。这些被称为 Promise，如今在许多编程语言中使用：

```javascript
const config = {
  privacy: public,
  acceptRequests: true
};

/*
 * 首先从服务器请求用户对象。
 * 一旦完成，从服务器请求用户资料。
 * 一旦完成，设置用户资料配置。
 * 一旦完成，console.log "success!"
 */
const promise = new Promise((resolve, reject) => {
  getUser(function(user) {
    if (user) { return resolve(user); }
    return reject();
  });
}).then((user) => {
  getUserProfile(user, function(profile) {
    if (profile) { return resolve(profile); }
    return reject();
  });
}).then((profile) => {
  setUserProfile(profile, config, function(result) {
    if (result) { return resolve(result); }
    return reject();
  });
}).catch((err) => {
  console.log('an error occurred!');
});
```

前面两个示例都完成完全相同的应用逻辑。区别在于可读性和组织性。基于 Promise 的方法可以进一步分解，垂直增长而非水平增长，并使错误处理更加容易。Promise 和回调是可互操作的，可以根据程序员的偏好一起使用。

处理异步的最新方法是 `async` 函数。与普通函数对象不同，这些函数旨在使处理异步变得轻而易举。

考虑以下 `async` 函数：

```javascript
const config = {
  privacy: public,
  acceptRequests: true
};

/*
 * 首先从服务器请求用户对象。
 * 一旦完成，从服务器请求用户资料。
 * 一旦完成，设置用户资料配置。
 * 一旦完成，console.log "success!"
 */
const setUserProfile = async function() {
  let user = await getUser();
  let userProfile = await getUserProfile(user);
  let setProfile = await setUserProfile(userProfile, config);
};

setUserProfile();
```

您可能会注意到这更容易阅读。很好，这就是重点！

`async` 函数将函数转换为 Promise。在带有 `await` 的 Promise 内部任何方法调用，都会在该函数内暂停进一步执行，直到该方法调用解析。`async` 函数外部的代码仍可正常运行。

本质上，`async` 函数将普通函数转换为 Promise。随着时间的推移，您将在客户端代码和基于 JavaScript 的服务端代码中越来越多地看到它们。

### 浏览器 DOM

您现在应该对异步编程有足够的理解——这是 Web 和客户端/服务器应用中占主导地位的模型。在您记住这些信息之后，您应该了解的最后一个与 JavaScript 相关的概念是浏览器 DOM。

DOM 是用于在现代 Web 浏览器中管理状态的层次化表示数据。图 3-3 显示了 `window` 对象，这是 DOM 规范定义的最顶层标准对象之一。

图 3-3. DOM 的 window 对象

JavaScript 是一种编程语言，像任何优秀的编程语言一样，它依赖于一个强大的标准库。这个库与其他语言中的标准库不同，被称为 DOM。DOM 提供经过良好测试且性能卓越的常规功能，并在所有主流浏览器中实现，因此您的代码无论在哪个浏览器上运行，功能都应相同或几乎相同。

与其他标准库不同，DOM 的存在不仅是为了填补语言中的功能漏洞或提供常用功能（这是 DOM 的次要功能），主要是为了提供一个公共接口，用于定义表示网页的层次化节点树。您可能不小心调用过 DOM 函数并以为它是 JS 函数。例如 `document.querySelector()` 和 `document.implementation`。

构成 DOM 的主要对象是 `window` 和 `document`。每个都在由名为 WHATWG 的组织维护的规范中仔细定义。

无论您是 JavaScript 开发者、Web 应用渗透测试员还是安全工程师，深入理解浏览器 DOM 及其在 Web 应用中的作用对于发现应用表示层中显而易见的漏洞至关重要。将 DOM 视为基于 JavaScript 的应用向最终用户部署的框架。请记住，并非所有与脚本相关的安全漏洞都是不当 JavaScript 的结果；它们有时也可能源于不当的浏览器 DOM 实现。

## SPA 框架

较旧的网站通常建立在用于操作 DOM 的临时脚本和大量重复使用的 HTML 模板代码的组合之上。这不是一个可扩展的模型，虽然它适用于向最终用户交付静态内容，但不适用于交付复杂的、逻辑丰富的应用。

当时的桌面应用软件功能强大，允许用户存储和维护应用状态。过去的网站不提供这种类型的功能，尽管许多公司更愿意通过 Web 交付其复杂应用，因为这在易用性和防盗版等方面带来了许多好处。

单页应用（SPA）框架旨在弥合网站和桌面应用之间的功能差距。SPA 框架允许开发复杂的基于 JavaScript 的应用，这些应用存储自己的内部状态，并由可重用的 UI 组件组成，每个组件都有自己的自我维护生命周期，从渲染到逻辑执行。

如今 SPA 框架在 Web 上十分普遍，支撑着最大和最复杂的应用（如 Facebook 和 YouTube），这些应用功能至关重要，并提供接近桌面的应用体验。当今一些最大的开源 SPA 框架是 ReactJS、EmberJS、VueJS 和 AngularJS（图 3-4）。它们都构建在 JavaScript 和 DOM 之上，但从安全和功能角度来看都带来了额外的复杂性。

## 认证与授权系统

在大多数应用由客户端（浏览器/手机）和服务器组成，并且服务器持久化最初来自客户端的数据的世界中，必须有系统来确保未来对持久化数据的访问来自正确的用户。我们使用术语“认证”来描述允许系统识别用户的流程。换句话说，认证系统告诉我们“joe123”确实是“joe123”，而不是“susan1988”。

术语“授权”用于描述系统内确定“joe123”有权访问哪些资源，而非“susan1988”有权访问哪些资源的流程。例如，“joe123”应该能够访问他自己上传的私人照片，而“susan1988”应该能够访问她的，但他们不应该能够访问彼此的照片。

这两个过程对于 Web 应用的功能都至关重要，并且都是 Web 应用中需要适当安全控制的关键功能。

### 认证

早期的认证系统本质上很简单。例如，HTTP 基本认证通过在每次请求上附加一个 `Authorization` 头部来执行认证。该头部由一个包含 `Basic: <base64 编码的用户名:密码>` 的字符串组成。服务器接收用户名:密码组合，并在每次请求时将其与数据库进行核对。显然，这种认证方案有若干缺陷——例如，凭据很容易以多种方式泄露，从通过 HTTP 的受损 WiFi 到简单的 XSS 攻击。

后来的认证发展包括摘要认证，它使用加密哈希而非 base64 编码。在摘要认证之后，涌现出了大量新技术和架构用于认证，包括不涉及密码的技术和需要外部设备的技术。

如今，大多数 Web 应用根据业务性质从一套认证架构中选择。例如，OAuth 协议非常适合希望与大型网站集成的网站。OAuth 允许主要网站（如 Facebook、Google 等）向合作伙伴网站提供验证用户身份的令牌。OAuth 对用户可能有用，因为用户的数据只需在一个网站上更新，而非多个网站——但 OAuth 也可能危险，因为一个被入侵的网站可能导致多个配置文件被入侵。

HTTP 基本认证和摘要认证至今仍被广泛使用，其中摘要更受欢迎，因为它对拦截和重放攻击有更多防御。通常这些与多因素认证（MFA）等工具结合，以确保认证令牌不被入侵，并且登录用户的身份没有改变。

### 授权

授权是认证之后的下一步。授权系统更难分类，因为授权在很大程度上取决于 Web 应用内部的业务逻辑。一般来说，设计良好的应用有一个集中的授权类，负责确定用户是否有权访问某些资源或功能。

如果 API 编写得不好，它们会在每个 API 基础上实现检查，手动复制授权功能。通常，如果您能看出一个应用在每个 API 中重新实现授权检查，那么该应用很可能有若干 API 的检查不够充分，仅仅是由于人为错误。

一些应始终有授权检查的常见资源包括：设置/资料更新、密码重置、私人消息读/写、任何付费功能，以及任何高权限用户功能（如版主功能）。

## Web 服务器

现代客户端/服务器 Web 应用依赖于一系列相互叠加的技术，以使服务端和客户端组件按预期运行。在服务器方面，应用逻辑运行在基于软件的 Web 服务器包之上，这样应用开发者就不必担心处理请求和管理进程。Web 服务器软件当然运行在操作系统之上（通常是某种 Linux 发行版，如 Ubuntu、CentOS 或 Red Hat），而操作系统又运行在某个数据中心的物理硬件之上。

就 Web 服务器软件而言，现代 Web 应用世界中有几个主要参与者。Apache 仍然服务于全球近一半的网站，因此我们可以假设 Apache 也服务于大多数 Web 应用。Apache 是开源的，已开发近 30 年，几乎可以在所有 Linux 发行版以及一些 Windows 服务器上运行（见图 3-5）。

图 3-5. Apache，最大且最常实现的 Web 服务器软件包之一，自 1995 年以来一直在开发

Apache 不仅因其庞大的贡献者社区和开源特性而伟大，还因为它变得多么易于配置和可插拔。它是一个灵活的 Web 服务器，您可能会在很长一段时间内看到它。

Apache 最大的竞争对手是 NGINX（发音为“Engine X”）。NGINX 运行约 30% 的 Web 服务器，并且正在快速增长。虽然 NGINX 可以免费使用，但其母公司（目前为 F5 Networks）使用付费+模式，支持和额外功能需要付费。

NGINX 用于具有大量独特连接的高流量应用，而非那些连接少但需要大量数据的应用。当从 Apache 切换到 NGINX 时，同时服务许多用户的 Web 应用可能会看到显著的性能提升，因为 NGINX 架构每个连接的开销要小得多。

在 NGINX 之后是 Microsoft IIS，尽管由于昂贵的许可证和与基于 Unix 的开源软件（OSS）包缺乏兼容性，基于 Windows 的服务器的受欢迎程度已经下降。当处理许多 Microsoft 特定技术时，IIS 是 Web 服务器的正确选择，但对于试图在开源基础上构建的公司来说，它可能是一个负担。

还有许多较小的 Web 服务器，每个都有自己的安全优势和劣势。熟悉这三大服务器将在您继续本书并学习如何发现源于配置不当（而非仅应用逻辑中的漏洞）的漏洞时非常有用。

## 服务端数据库

一旦客户端将待处理的数据发送到服务器，服务器通常必须持久化这些数据，以便在未来的会话中检索。长期在内存中存储数据并不可靠，因为重启和崩溃可能导致数据丢失。此外，与磁盘相比，随机存取内存相当昂贵。

在磁盘上存储数据时，需要采取适当的预防措施，以确保数据能够可靠且快速地检索、存储和查询。如今几乎所有 Web 应用都将其用户提交的数据存储在某类数据库中——通常根据特定的业务逻辑和用例来变更所使用的数据库。

SQL 数据库仍然是市场上最通用的通用数据库。SQL 查询语言严格但可靠快速且易于学习。SQL 可用于从存储用户凭据到管理 JSON 对象或小图像 blob 的任何用途。其中最大的是 PostgreSQL、Microsoft SQL Server、MySQL 和 SQLite。

当需要更灵活的存储时，可以使用无模式的 NoSQL 数据库。像 MongoDB、DocumentDB 和 CouchDB 这样的数据库将信息存储为松散结构的“文档”，这些文档灵活且可以随时修改，但在查询或聚合方面不如 SQL 容易或高效。

在当今的 Web 应用格局中，也存在更高级和特定的数据库。搜索引擎通常使用自己高度专业化的数据库，必须定期与主数据库同步。一个例子是广泛流行的 Elasticsearch。

每种类型的数据库都带来独特的挑战和风险。SQL 注入是一种众所周知的漏洞原型，在查询未正确形成时对主要 SQL 数据库有效。然而，如果黑客愿意学习数据库的查询模型，注入式攻击几乎可以对任何数据库发起。

明智的做法是考虑许多现代 Web 应用可以同时使用多个数据库，而且通常确实如此。具有足够安全 SQL 查询生成的应用可能没有足够安全的 MongoDB 或 Elasticsearch 查询和权限。

## 客户端数据存储

传统上，由于技术限制和跨浏览器兼容性问题，客户端存储的数据很少。这种情况正在迅速改变。许多应用现在在客户端存储大量应用状态，通常以配置数据或大型脚本的形式，如果每次访问都下载这些数据会导致网络拥塞。

在大多数情况下，使用名为本地存储（localStorage）的浏览器管理存储容器来从客户端存储和访问键/值数据。本地存储遵循浏览器强制执行的 SOP，防止其他域（网站）访问彼此的本地存储数据。即使浏览器或标签页关闭，Web 应用也能保持状态（见图 3-6）。

图 3-6. 本地存储是一个强大的持久化键/值存储，所有现代浏览器都支持

本地存储的一个子集称为会话存储（sessionStorage），其操作方式相同，但数据仅在标签页关闭前持久化。当数据更关键且不应在另一用户使用同一台机器时持久化时，可以使用此类存储。

> 在架构不当的 Web 应用中，客户端数据存储也可能暴露敏感信息，如认证令牌或其他秘密。

最后，对于更复杂的应用，所有主流 Web 浏览器都支持 IndexedDB。IndexedDB 是一个基于 JavaScript 的 NoSQL 数据库，能够在 Web 应用后台异步存储和查询。由于 IndexedDB 是可查询的，它提供了比本地存储更强大的开发者接口。IndexedDB 在基于 Web 的游戏和基于 Web 的交互式应用（如图像编辑器）中得到应用。您可以在浏览器开发者控制台中输入以下内容来检查浏览器是否支持 IndexedDB：`if (window.indexedDB) { console.log('true'); }`。

## GraphQL

GraphQL 是一种强大的 API 端点查询新方法，自 2015 年 9 月首次发布以来迅速流行起来。通常，GraphQL 通过包装服务器上现有的 API 端点来实现，并允许通过 GraphQL 查询语言查询端点——这是一种脚本语言，允许捆绑请求，从而在网络层面节省显著的性能开销（见图 3-7）。

GraphQL 的查询语言允许比传统 REST 更高级的服务器请求。由 GraphQL 包装的 REST API 提供的一些额外功能如下：

- 请求特定字段
- 复杂请求参数
- 字段别名
- 操作
- 变量
- 指令
- 变更（Mutations）
- 字段片段（Fragments）

图 3-7. GraphQL 的 JavaScript 实现

通过使用 GraphQL 查询语言提供的结构，您可以组合出如下查询：

```graphql
query GetLeadActorFromMovie($movie: Movie)
  actor(movie: $movie) {
    name, age, gender,
    otherMovies {
      name
    }
  }
```

为查询提供正确的变量：

```json
{
  "movie": "Raiders of the Lost Ark"
}
```

服务器将返回以下内容：

```json
{
  "data": {
    "actor": {
      "name": "Harrison Ford",
      "age": 81,
      "gender": "male",
      "otherMovies": [
        "Blade Runner",
        "Patriot Games",
        "Clear and Present Danger"
      ]
    }
  }
}
```

如您所见，GraphQL 允许比传统 REST API 本身复杂得多的查询。仅使用 REST API，我们首先需要查询 `getLeadActorByMovie/:movieName`，然后查询 `getMoviesByLeadActor/:actorName`。但使用 GraphQL，这些查询已合并为一个——允许一次网络跳转而非两次。

使用 GraphQL 的安全影响将在本书后面全面评估。现在，请注意 GraphQL 允许客户端提供复杂脚本，针对服务器上的现有 REST API 运行，以减少网络延迟。

## 版本控制系统

几乎所有现代 Web 应用都使用某种类型的版本控制系统（VCS）构建。这些系统允许现代 Web 应用的开发者回溯其更改以识别和恢复更改，创建“分支”或功能的替代实现，并处理许多常见部署任务。

最常用的 VCS 是 Git。Git 最初由 Linus Torvalds 和 Linux 基金会开发，但此后被广泛采用，成为最流行的 VCS。Git 提供了分支（branches）、暂存（stashes）和复刻（forks）等工具，允许开发者通过几个命令分支并存储代码库的多个变体。

之后，Git 允许使用多种算法将这些替代副本合并在一起，每种算法都会产生历史记录，如果以后需要追踪问题，可以重新查看。托管 Git 版本控制的最常见实现是 GitHub（现归 Microsoft 所有，见图 3-8）和 GitLab。两者都在 Git 版本控制协议之上提供了简单易用的 API 和集成。

一个常见的 Git 工作流程可能如下：

```bash
git init
git remote add origin <github-repository-url>
touch .gitignore
echo "node_modules" > .gitignore
git add -A
git commit -m "initial commit"
git push --set-upstream origin main
```

此工作流程使用 Git 命令行工具在当前工作目录初始化 Git 仓库。然后，将远程云托管 GitHub 仓库链接到本地 Git 仓库。接下来，创建一个 `.gitignore` 文件，告诉 Git 哪些文件不应在版本控制中跟踪。然后命令 `git add -A` 告诉 Git 将所有更改的文件添加到暂存区以准备提交。`git commit` 命令让 Git 知道暂存文件已准备好记录，`-m` 标志提供一条消息详细说明更改集中的内容。最后，`git push` 告诉 Git 将暂存文件推送到版本控制历史记录，并远程推送到 GitHub 仓库。

图 3-8. GitHub，一个流行的代码共享和协作网站，构建在 Git 版本控制系统之上

然而，这只是 Git 的一个简单工作流程，大多数现代 Web 应用使用更高级的功能。一个典型例子是持续集成和持续交付，即 CI/CD。CI/CD 工具虽然不是 Git 协议的原生部分，但通常与 Git 集成在一起以便使用。

通常，现代 Web 应用配置为在新代码推送到云托管 Git 仓库时自动运行测试并部署构建。这个过程称为持续部署，是最常见的 CI/CD 集成之一。它通常由 Git 钩子或在特定 Git 命令上运行的代码片段触发。

由于这些原因，Web 应用的安全不再仅与应用本身的网络栈和应用代码相关，还与其 VCS 和 CI/CD 管道相关。这些层中的任何一层被入侵都可能是灾难性的，因此应考虑整个应用及其 VCS 和交付机制的全局图景。

## CDN/缓存

传统 Web 应用通常直接将内容提供给客户端（浏览器），并使用智能估算来计算峰值请求时段的硬件需求。许多现代 Web 应用采取不同方法，将静态或很少更新的内容卸载到内容分发网络（CDN），并在应用栈的各个部分（包括客户端缓存）进行缓存。

CDN 可以通过在全球范围内提供服务器来扩展，这些服务器通过 API 复制您的 Web 服务器提供的静态内容。CDN 处理可扩展性，而 Web 服务器直接与 CDN 通信，作为单一事实来源（见图 3-9）。

图 3-9. CDN 将内容从您的主 Web 服务器卸载，并在地理上分发这些内容，以降低服务器负载、提供更低的延迟和更高的正常运行时间

这种方法具有积极效果，允许开发者非常快速地将应用扩展到全球范围，但也引入了一些新的安全风险。主要风险之一是缓存过期，这可能没有可靠或准确的信息，并可能导致权限提升或信息泄露。

虽然 CDN 作为全球缓存运行，但本地缓存和网络缓存更常用。在传统 Web 应用中未使用的一种常见缓存形式是客户端（浏览器）缓存。浏览器提供默认的缓存机制，可以通过 `ETag` 或 `Cache-Control` 等头部进行配置，但许多应用也使用本地存储、会话存储和 IndexedDB 来进一步定制客户端缓存。

由于客户端和第三方 CDN/缓存采用的扩展，工程师需要在许多十年前通常不被考虑的地方评估数据安全性。这些新系统带来了额外的复杂性，因为这些缓存可能损坏或不同步，从而导致新的安全挑战。其中许多挑战将在后续章节中讨论。

## 小结

现代 Web 应用构建在许多旧应用中找不到的新技术之上。由于功能扩展导致的攻击面增加，与过去的网站相比，当今的应用面临更多形式的攻击。

要在当今的应用生态系统中成为安全专家，您不仅需要安全专业知识，还需要一定水平的软件开发技能。这十年中最优秀的黑客和安全专家除了他们的安全技能外，还带来了深厚的工程知识。他们理解应用客户端和服务器之间的关系和架构。他们能够从服务器、客户端或两者之间的网络角度分析应用的行为。

最优秀的人还理解支撑现代 Web 应用这三个层次的技术。因此，他们理解不同数据库、客户端技术和网络协议中固有的弱点。

虽然您不需要成为专家级软件工程师才能成为熟练的黑客或安全工程师，但这些技能将帮助您，并且您会发现它们非常有价值。它们将加快您的研究，使您能够看到深层而困难、否则无法发现的漏洞。
