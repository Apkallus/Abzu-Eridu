# 第 4 章：查找子域名

为了全面了解和测试 API 端点，您首先应熟悉 Web 应用所使用的域名结构。如今，使用单个域名来完整承载一个 Web 应用的情况已很少见。通常，Web 应用至少会拆分为客户端域和服务端域，再加上常见的 `https://www` 与纯 `https://` 之别。能够迭代地发现并记录支撑 Web 应用的子域名，是对该应用进行初始侦察的一项实用技术。

## 每个域名的多个应用

假设我们正试图绘制 MegaBank 的 Web 应用地图，以便更好地执行该银行委托的黑盒渗透测试。我们知道 MegaBank 有一个应用，用户可以登录并访问其银行账户。该应用位于 `https://www.mega-bank.com`。

我们特别好奇 MegaBank 是否还有其他与 mega-bank.com 域名关联的、可通过互联网访问的服务器。我们知道 MegaBank 有漏洞赏金计划，且该计划的范围相当全面地覆盖了主要的 mega-bank.com 域名。因此，mega-bank.com 中任何容易发现的漏洞都已被修复或报告。如果出现新的漏洞，我们必须争分夺秒地在漏洞赏金猎人之前找到它们。

因此，我们希望寻找一些更容易的目标，这些目标仍能对 MegaBank 造成有效打击。这是一次纯粹的公司赞助的道德测试，但这并不意味着我们不能享受其中的乐趣。

我们应该做的第一件事是执行一些侦察，并用附加到 mega-bank.com 的子域名列表来填充我们的 Web 应用地图（见图 4-1）。因为 `www` 指向面向公众的 Web 应用本身，我们可能对此不感兴趣。但大多数大型消费类公司实际上会托管附加到其主域名的各种子域名。这些子域名用于托管各种服务，从电子邮件、管理应用、文件服务器等等。

图 4-1. Mega-bank.com 简单的子域名网络——通常这些网络要复杂得多，可能包含无法从外部网络访问的服务器

有很多方法可以找到这些数据，通常您必须尝试几种才能获得您想要的结果。我们将从最简单的方法开始，逐步深入。

## 浏览器的内置网络分析工具

首先，我们可以通过浏览 MegaBank 的可见功能并观察后台发出了哪些 API 请求，来收集一些有用的数据。这通常会给我们一些唾手可得的端点。要查看这些正在发出的请求，我们可以使用自己的 Web 浏览器网络工具，或更强大的工具如 Burp 或 ZAP。

图 4-2 显示了 Wikipedia 浏览器开发者工具的一个示例，可用于查看、修改、重发和记录网络请求。像这样免费提供的网络分析工具比 10 年前的许多付费网络工具强大得多。由于本书不依赖专用工具，我们现在仅依赖浏览器。

只要您使用的是三大主流浏览器（Chrome、Firefox 或 Edge）之一，您就会发现它们自带的开发者工具非常强大。事实上，浏览器开发者工具已经发展到如此程度，您无需购买任何第三方工具就能成为一名熟练的黑客。现代浏览器提供网络分析、代码分析、带断点和文件引用的 JavaScript 运行时分析、精确的性能测量（也可用作旁路攻击的黑客工具），以及执行小型安全和兼容性审计的工具。

图 4-2. Wikipedia.org 浏览器开发者工具网络标签页，显示了对 Wikipedia API 发起的异步 HTTP 请求

要分析通过浏览器传输的网络流量，请执行以下操作（在 Chrome 中）：

1. 点击导航栏右上角的三点菜单以打开设置菜单。
2. 在“更多工具”下点击“开发者工具”。
3. 在此菜单顶部，点击“网络”标签。如果不可见，请水平扩展开发者工具直到其出现。

现在，在“网络”标签打开的情况下，尝试浏览任何网站中的页面。注意，新的 HTTP 请求将弹出，同时还有其他许多请求（见图 4-3）。

图 4-3. 网络标签，用于分析流向和来自 Web 浏览器的网络流量

您可以使用浏览器中的“网络”标签查看浏览器正在处理的所有网络流量。对于大型网站，过滤起来可能会变得相当繁琐。

通常最有价值的结果来自“网络”标签下的“XHR”子标签，它将显示任何对服务器的 HTTP POST、GET、PUT、DELETE 以及其他请求，并过滤掉字体、图像、视频和依赖文件。您可以在左侧面板中点击任何单个请求以查看更多详细信息。

点击其中一个请求将显示请求的原始和格式化版本，包括任何请求头部和请求体。在选中请求时出现的“预览”标签中，您将能够看到任何 API 请求结果的格式化版本。

“XHR”下的“响应”标签将显示原始的响应载荷，而“时序”标签将显示与请求相关的排队、下载和等待时间的非常具体的指标。这些性能指标实际上非常重要，因为它们可用于发现旁路攻击（一种依赖于响应之外的次要指标来推测服务器上正在运行什么代码的攻击；例如，通过同一端点调用的服务器上两个脚本之间的加载时间）。

现在您应该对浏览器“网络”标签有了足够的熟悉，可以开始四处探索并将其用于侦察。这些工具看起来吓人，但实际上并不难学。

当您浏览任何网站时，您可以检查请求 → 头部 → 常规 → 请求 URL，以查看请求发送到哪个域或响应来自哪个域。通常，这就是查找主网站关联服务器所需的全部信息。

## 利用公共记录

如今，存储在 Web 上的公开可用信息量如此之大，以至于偶然的数据泄露可能会在无人注意的情况下溜走数年。优秀的黑客可以利用这一事实，找到许多有趣的信息碎片，这些信息可能在未来导致一次轻松的攻击。

过去我在执行渗透测试时，曾在网上发现的一些数据包括：

- 曾意外公开后又转为私有的 GitHub 仓库的缓存副本
- SSH 密钥
- 各种服务的密钥，如 Amazon AWS 或 Stripe，这些密钥曾定期暴露，然后从面向公众的 Web 应用中移除
- 不打算面向公众的 DNS 列表和 URL
- 详细描述未发布产品但不应在线的页面
- 托管在 Web 上但不打算被搜索引擎抓取的财务记录
- 电子邮件地址、电话号码和用户名

这些信息可以在许多地方找到，例如：

- 搜索引擎
- 社交媒体帖子
- 存档应用，如 archive.org
- 图片搜索和反向图片搜索

在尝试查找子域名时，公共记录也是一个很好的信息来源，因为子域名可能不易通过字典找到，但可能已被前述服务之一索引。

### 搜索引擎缓存

Google 是世界上使用最广泛的搜索引擎，通常被认为索引的数据比其他任何搜索引擎都多。仅仅依靠 Google 搜索对手动侦察来说并不实用，因为您需要筛选大量数据才能找到有价值的东西。加之 Google 已严厉打击自动化请求，并拒绝那些不能紧密模仿真实 Web 浏览器的请求。

幸运的是，Google 为高级搜索用户提供了特殊的搜索运算符，允许您提高搜索查询的精确度。我们可以使用 `site:<my-site>` 运算符，要求 Google 仅针对特定域名进行查询：

```
site:mega-bank.com log in
```

对热门网站执行此操作通常会返回来自主域名的成页内容，而来自有趣子域名的内容却很少。您需要进一步改进搜索焦点，才能开始发现任何有趣的内容。

使用减号运算符向任何查询字符串添加特定的否定条件。例如，`-inurl:<pattern>` 将拒绝任何与提供模式匹配的 URL。图 4-4 显示了一个组合了 Google 搜索运算符 `site:` 和 `-inurl:<pattern>` 的搜索示例。通过结合这两个运算符，我们可以要求 Google 仅返回 Wikipedia.org 中关于小狗的网页，同时排除任何 URL 中包含“dog”一词的网页。这种技术可用于减少返回的搜索结果数量，并在忽略特定关键字的同时搜索特定子域名。精通 Google 的搜索运算符以及其他搜索引擎的运算符，将使您能够找到原本不易发现的信息。

我们可以使用 `-inurl:<pattern>` 运算符来移除我们已经熟悉的子域名（如 `www`）的结果。注意，它也会过滤掉 URL 其他部分中的 `www` 实例，因为它不指定子域名，而是整个 URL 字符串。这意味着 `https://admin.mega-bank.com/www` 也会被过滤，因此可能存在误删：

```
site:mega-bank.com -inurl:www
```

您可以对许多网站尝试此操作，您会发现一些您甚至没想过会存在的子域名。例如，让我们对热门新闻网站 Reddit 试试：

```
site:reddit.com -inurl:www
```

此查询的第一个结果将是 `code.reddit.com`——Reddit 早期版本中使用的代码存档，工作人员决定将其公开。像 Reddit 这样的网站故意将这些域名暴露给公众。

图 4-4. 一个 Google.com 搜索，结合了 Google 搜索运算符 `site:` 和 `-inurl:<pattern>`

对于我们对 MegaBank 的渗透测试，如果我们发现其他故意暴露且我们不感兴趣的域名，我们也会将其过滤掉。如果 MegaBank 有一个托管在子域名 `mobile.mega-bank.com` 下的移动版本，我们同样可以轻松过滤掉它：

```
site:mega-bank.com -inurl:www -inurl:mobile
```

在尝试查找给定网站的子域名时，您可以重复此过程，直到找不到更多相关结果。尝试在其他搜索引擎（如 Bing）上使用这些技术也可能有益——大型搜索引擎都支持类似运算符。

记录您通过此技术发现的任何有趣内容，然后继续其他子域名侦察方法。

### 意外存档

像 Archive.org 这样的公共存档实用程序很有用，因为它们会定期构建网站的快照，并允许您访问过去的网站副本。

Archive.org 致力于保存互联网的历史，因为许多网站会消亡，而新网站会取代其域名。由于 Archive.org 存储网站的历史快照，有时可以追溯到 20 年前，因此该网站是寻找曾经（有意或无意）披露但后来被删除的信息的金矿。图 4-5 中的特定截图是 Wikipedia.org 在 2003 年（二十多年前！）被索引的主页。

图 4-5. Archive.org，一家位于旧金山的非营利组织，自 1996 年以来一直存在

一般来说，搜索引擎会索引有关网站的数据，但会定期抓取该网站以保持其缓存更新。这意味着您应该在搜索引擎中查找相关的当前数据，但对于相关的历史数据，您可能最好查看网站存档。

《纽约时报》是流量方面最受欢迎的基于网络的媒体公司之一。如果我们在 Archive.org 上查找其主网站（`https://www.nytimes.com`），我们会发现 Archive.org 在 1996 年至今之间保存了超过 50 万张首页快照。

如果我们知道或能猜测出某个 Web 应用发布重大版本或披露严重安全漏洞的时间点，历史快照就特别有价值。在查找子域名时，历史存档通常会通过曾经通过 HTML 或 JS 暴露但不再出现在实时应用中的超链接来披露这些信息。

如果我们右键单击浏览器中的 Archive.org 快照并选择“查看源代码”，我们可以快速搜索常见的 URL 模式。搜索 `file://` 可能会显示以前在线的下载链接，而搜索 `https://` 或 `http://` 应该会调出所有 HTTP 超链接。

我们可以通过以下简单步骤从存档中自动化子域名发现：

1. 打开 10 个来自 10 个不同日期的存档，日期之间要有显著的时间间隔。
2. 右键点击“查看源代码”，然后按 Ctrl-A 高亮所有 HTML。
3. 按 Ctrl-C 将 HTML 复制到剪贴板。
4. 在桌面上创建一个名为 `legacy-source.html` 的文件。
5. 按 Ctrl-V 将存档中的源代码粘贴到该文件中。
6. 对您打开的其他 9 个存档重复此操作。
7. 在您喜欢的文本编辑器（VIM、Atom、VSCode 等）中打开此文件。
8. 搜索最常见的 URL 方案：
   - `http://`
   - `https://`
   - `file://`
   - `ftp://`
   - `ftps://`

您可以在[规范文档](https://oreil.ly/zhTcF)中找到浏览器支持的 URL 方案的完整列表，该文档在所有主流浏览器中用于定义应支持的方案。

### 社交快照

如今，每个主要的社交媒体网站都通过销售用户数据来盈利。根据平台的不同，这可能包括公开帖子、私人帖子，在某些情况下甚至包括直接消息。

不幸的是，如今主要的社交媒体公司都花费大量努力让用户相信他们最私密的数据是安全的。这通常通过营销信息来完成，描述为保护客户数据不被触及所付出的巨大努力。然而，这通常只是为了帮助吸引和维持活跃用户而说的。很少有国家拥有足够现代化的法律和立法者来强制执行这些声明的合法性。这些网站的许多用户很可能并不完全理解哪些数据被共享、通过什么方法共享，以及这些数据被消费用于什么目的。

通过社交媒体数据为公司赞助的渗透测试查找子域名，在大多数人看来并不违反道德。然而，我恳请您在将来使用这些 API 进行更有针对性的侦察时，考虑最终用户的权益。

为简单起见，我们将以 Twitter API 作为侦察示例。但请记住，每个主要的社交媒体公司都提供类似的一套 API，通常遵循相似的 API 结构。从 Twitter API 查询和搜索推文数据所需的概念可以应用于任何其他主要社交媒体网络。

#### Twitter API

X（原 Twitter）公司提供多种用于搜索和过滤其数据的产品（见图 4-6）。

这些产品在范围、功能集和数据集上有所不同。这意味着您想要访问的数据越多，请求和过滤这些数据的方式越多，您需要支付的费用就越高。在某些情况下，甚至可以在 X 的服务器上执行搜索，而不是在本地。请注意，出于恶意目的这样做可能违反 X 的服务条款，因此这种使用应仅限于白帽行为。

在最底层，X 提供试用版“搜索 API”，允许您筛选 30 天的推文，前提是每个查询请求不超过 100 条推文，且每分钟查询不超过 30 次。使用免费层 API，您的每月总查询次数也被限制在 250 次。大约需要 10 分钟的查询时间才能获得此层提供的最大月度数据集。这意味着您只能分析 25,000 条推文，而无需购买更高级的会员层级。

图 4-6. X 的 API 开发者文档将帮助您快速上手搜索和过滤用户数据

这些限制可能使得编码工具来分析 API 变得有点困难。如果您在工作赞助的项目中需要 X 进行侦察，您可能需要考虑升级或寻找其他数据源。

我们可以使用此 API 构建一个包含指向 `*.mega-bank.com` 链接的 JSON，以进一步推进我们的子域名侦察。要开始查询 X 搜索 API，您需要以下内容：

- 一个注册的开发者账户
- 一个注册的应用
- 一个在请求中包含以验证您身份的 Bearer Token

查询此 API 非常简单，尽管文档分散且有时因缺乏示例而难以理解：

```bash
curl --request POST \
  --url https://api.twitter.com/1.1/tweets/search/30day/Prod.json \
  --header 'authorization: Bearer <MY_TOKEN>' \
  --header 'content-type: application/json' \
  --data '{
    "maxResults": "100",
    "keyword": "mega-bank.com"
  }'
```

默认情况下，此 API 对关键字执行模糊搜索。要精确匹配，您必须确保传输的字符串本身包含在双引号中。双引号可以通过以下形式在有效 JSON 中发送：`"keyword": "\"mega-bank.com\""`。

记录此 API 的结果并搜索链接可能会导致发现以前未知的子域名。这些通常来自营销活动、广告跟踪器，甚至与主应用不同服务器关联的招聘活动。

举一个现实生活中的例子，尝试构建一个查询，请求有关 Microsoft 的推文。在筛选足够多的推文后，您会注意到 Microsoft 有多个它在 X 上积极推广的子域名，包括：

- `careers.microsoft.com`（招聘网站）
- `office.microsoft.com`（Microsoft Office 主页）
- `powerbi.microsoft.com`（PowerBI 产品主页）
- `support.microsoft.com`（Microsoft 客户支持）

请注意，如果一条推文变得足够流行，主要搜索引擎将开始索引它。因此，如果您在寻找不太流行的推文，分析 Twitter API 将更有用。高流行的病毒式推文由于入站链接数量多，会被搜索引擎索引。这意味着有时更有效的方法是直接使用正确的运算符查询搜索引擎，如本章前面所述。

如果此 API 的结果不足以满足您的侦察项目，X 还提供另外两个 API：流式 API 和消防水带 API。

X 的流式 API 提供当前推文的实时流，供实时分析；然而，此 API 仅提供实际实时推文的一个非常小的百分比，因为数量太大，无法实时处理和发送给开发者。这意味着在任何给定时间，您可能错过超过 99% 的推文。如果您正在研究的应用正在流行或非常受欢迎，此 API 可能会有益。如果您为一个初创公司做侦察，此 API 对您来说不会有太大用处。

X 的消防水带 API 与流式 API 类似，但保证交付 100% 符合您提供条件的推文。这通常比流式 API 更有价值，因为我们在大多数情况下更看重相关性而非数量。

总之，当使用 X 作为侦察工具时，请遵循以下规则：

- 对于大多数 Web 应用，查询搜索 API 将为您提供最相关的侦察数据。
- 大规模应用或正在流行的应用，可能在消防水带或流式 API 中找到有用的信息。
- 如果历史信息对您的情况可接受，考虑下载大量历史推文数据并在本地针对其查询。

请记住，几乎所有主要社交媒体网站都提供数据 API，可用于侦察或其他形式的分析。如果一个不给您想要的结果，另一个可能会。

## 区域传送攻击

浏览面向公众的 Web 应用并分析网络请求只能让您走到这一步。我们还希望找到附加到 MegaBank 但未以任何方式链接到公共 Web 应用的子域名。

区域传送攻击是一种对配置不当的域名系统（DNS）服务器有效的侦察技巧。它并不是真正的“黑客攻击”，尽管其名称会暗示如此。相反，它只是一种信息收集技术，使用起来不费吹灰之力，如果成功可以给我们一些有价值的信息。其核心是，DNS 区域传送攻击是一个代表个人发出的格式化请求，旨在看起来像是来自有效 DNS 服务器的合法 DNS 区域传送请求。

DNS 服务器负责将人类可读的域名（例如 `https://mega-bank.com`）转换为机器可读的 IP 地址（例如 `195.250.100.195`），这些地址是层次化的，并按照通用模式存储，以便于请求和遍历。DNS 服务器很有价值，因为它们允许服务器的 IP 地址变更而无需更新该服务器的应用用户。换句话说，用户可以持续访问 `https://www.mega-bank.com`，而无需担心请求将解析到哪台服务器。

DNS 系统非常依赖于其将 DNS 记录更新与其他 DNS 服务器同步的能力。DNS 区域传送是 DNS 服务器可以共享 DNS 记录的标准化方式。记录以基于文本的格式（称为区域文件）共享。

区域文件通常包含不打算轻易访问的 DNS 配置数据。因此，正确配置的 DNS 主服务器应仅能解析由其他授权的 DNS 辅助服务器请求的区域传送请求。如果 DNS 服务器未正确配置为仅解析针对其他特定定义 DNS 服务器的请求，它将容易受到恶意行为者的攻击。

总之，如果我们希望对 MegaBank 尝试区域传送攻击，我们需要假装自己是 DNS 服务器，并请求一个 DNS 区域文件，就好像我们需要它来更新自己的记录一样。我们首先需要找到与 `https://www.mega-bank.com` 关联的 DNS 服务器。我们可以在任何基于 Unix 系统的终端中非常轻松地做到这一点：

```bash
host -t ns mega-bank.com
```

命令 `host` 是一个 DNS 查找工具，您可以在大多数 Linux 发行版以及最新版本的 macOS 中找到它。`-t ns` 标志指定我们要请求负责解析 mega-bank.com 的名称服务器。

此命令的输出类似于：

```
mega-bank.com name server ns1.bankhost.com
mega-bank.com name server ns2.bankhost.com
```

我们对此结果感兴趣的字符串是 `ns1.bankhost.com` 和 `ns2.bankhost.com`。它们指的是为 mega-bank.com 解析的两个名称服务器。

使用 `host` 进行区域传送请求非常简单，只需一行：

```bash
host -l mega-bank.com ns1.bankhost.com
```

这里 `-l` 标志表示我们希望从 `ns1.bankhost.com` 获取 mega-bank.com 的区域传送文件以更新我们的记录。如果请求成功，表明 DNS 服务器未正确配置，您将看到类似这样的结果：

```
Using domain server:
Name: ns1.bankhost.com
Address: 195.11.100.25
Aliases:

mega-bank.com has address 195.250.100.195
mega-bank.com name server ns1.bankhost.com
mega-bank.com name server ns2.bankhost.com
mail.mega-bank.com has address 82.31.105.140
admin.mega-bank.com has address 32.45.105.144
internal.mega-bank.com has address 25.44.105.144
```

从这些结果中，您现在有了在 mega-bank.com 域名下托管的其他 Web 应用的列表，以及它们的公共 IP 地址！

您甚至可以尝试导航到这些子域名或 IP 地址，看看什么能解析。如果幸运的话，您已大大扩展了您的攻击面！

不幸的是，DNS 区域传送攻击并不总是像前面的例子那样顺利。正确配置的服务器在您请求区域传送时会给出不同的输出：

```
Using domain server:
Name: ns1.secure-bank.com
Address: 141.122.34.45
Aliases:

: Transfer Failed.
```

区域传送攻击很容易阻止，您会发现许多应用都正确配置以拒绝这些尝试。然而，由于尝试区域传送攻击只需几行 Bash 命令，几乎总是值得一试。如果成功，您将获得许多有趣的子域名，否则可能找不到。

## 暴力破解子域名

作为发现子域名的最后手段，可以使用暴力破解策略。这些策略可能对安全机制较少的 Web 应用有效；然而，对于更成熟和安全的 Web 应用，我们会发现我们的暴力破解必须非常智能地构造。暴力破解子域名应作为最后的选择，因为暴力破解尝试容易被记录，并且由于速率限制、正则表达式和其他旨在防止此类窥探的简单安全机制，通常极其耗时。

> 暴力破解攻击非常容易被检测，并可能导致您的 IP 地址被服务器或其管理员记录或封禁。

暴力破解意味着测试每种可能的子域名组合，直到找到匹配项。对于子域名，可能存在许多可能的匹配项，因此找到第一个匹配项可能不够。

首先，让我们考虑一下，与本地暴力破解不同，针对目标域名的子域名暴力破解需要网络连接。因为我们必须远程执行此暴力破解，网络延迟将进一步减慢我们的尝试。一般来说，每个网络请求预计有 50 到 250 毫秒的延迟。

这意味着我们应该异步发起请求，并尽可能快地全部发出，而不是等待之前的响应。这样做将显著减少暴力破解完成所需的时间。

检测存活子域名的反馈循环非常简单。我们的暴力破解算法生成一个子域名，然后我们向 `<子域名猜测>.mega-bank.com` 发起请求。如果我们收到响应，我们将其标记为存活子域名。否则，我们将其标记为未使用的子域名。

由于您正在阅读的这本书名为《Web 应用程序安全》，在这种背景下，我们最需要熟悉的语言是 JavaScript。JavaScript 不仅是目前可用于 Web 浏览器中客户端脚本的唯一编程语言，而且由于 Node.js 和开源社区，它也是一种极其强大的后端服务器端语言。

让我们用 JavaScript 分两步构建一个暴力破解算法。我们的脚本应执行以下操作：

1. 生成潜在子域名列表。
2. 遍历该子域名列表，每次 ping 以检测子域名是否存活。
3. 记录存活的子域名，对未使用的子域名不采取任何操作。

我们可以使用以下方法生成子域名：

```javascript
/*
 * 给定每个子域名的最大长度，生成子域名列表的简单函数。
 */
const generateSubdomains = function(length) {
  /*
   * 用于生成子域名的字符列表。
   *
   * 可以修改此列表以包含不太常见的字符，如 '-'.
   *
   * 某些浏览器也支持中文、阿拉伯文和拉丁字符。
   */
  const charset = 'abcdefghijklmnopqrstuvwxyz'.split('');

  let subdomains = charset;
  let subdomain;
  let letter;
  let temp;

  /*
   * 时间复杂度：o(n*m)
   * n = 字符串长度
   * m = 有效字符数
   */
  for (let i = 1; i < length; i++) {
    temp = [];
    for (let k = 0; k < subdomains.length; k++) {
      subdomain = subdomains[k];
      for (let m = 0; m < charset.length; m++) {
        letter = charset[m];
        temp.push(subdomain + letter);
      }
    }
    subdomains = temp
  }
  return subdomains;
}

const subdomains = generateSubdomains(4);
```

此脚本生成长度为 n 的所有可能的字符组合，其中用于组合子域名的字符列表是 `charset`。该算法的工作原理是将 `charset` 字符串拆分为字符数组，然后将初始字符集分配给该字符数组。

接下来，我们迭代 `length` 次，在每次迭代中创建一个临时存储数组。然后遍历每个子域名和 `charset` 数组中的每个字符（指定可用字符集）。最后，我们使用现有子域名和字母的组合构建 `temp` 数组。

现在，使用此子域名列表，我们可以开始针对顶级域名（.com、.org、.net 等）如 mega-bank.com 进行查询。为此，我们将编写一个简短脚本，利用 Node.js（一个流行的 JavaScript 运行时）中提供的 DNS 库。

要运行此脚本，您只需在您的环境（假设是基于 Unix 的环境，如 Linux 或 Ubuntu）中安装最新版本的 Node.js：

```javascript
const dns = require('dns');
const promises = [];

/*
 * 此列表可以使用前面的暴力破解脚本填充，
 * 或使用常用子域名字典。
 */
const subdomains = [];

/*
 * 遍历每个子域名，并对每个子域名执行异步 DNS 查询。
 *
 * 这比更常见的 `dns.lookup()` 性能更高，
 * 因为 `dns.lookup()` 从 JavaScript 看起来是异步的，
 * 但依赖于操作系统的 getaddrinfo(3)，它是同步实现的。
 */
subdomains.forEach((subdomain) => {
  promises.push(new Promise((resolve, reject) => {
    dns.resolve(`${subdomain}.mega-bank.com`, function (err, ip) {
      return resolve({ subdomain: subdomain, ip: ip });
    });
  }));
});

// 在所有 DNS 查询完成后，记录结果
Promise.all(promises).then(function(results) {
  results.forEach((result) => {
    if (!!result.ip) {
      console.log(result);
    }
  });
});
```

在此脚本中，我们做了几件事来提高暴力破解代码的清晰度和性能。首先导入 Node DNS 库。然后创建一个数组 `promises`，它将存储一个 Promise 对象列表。Promise 是处理 JavaScript 中异步请求的一种更简单的方式，在所有主流 Web 浏览器和 Node.js 中都有原生支持。

之后，我们创建另一个名为 `subdomains` 的数组，该数组应填充我们从第一个脚本生成的子域名（我们将在本节末尾将两个脚本组合在一起）。接下来，我们使用 `forEach()` 运算符轻松遍历 `subdomains` 数组中的每个子域名。这等效于 `for` 迭代，但在语法上更优雅。

在子域名迭代的每一层，我们向 `promises` 数组推送一个新的 Promise 对象。在此 Promise 对象中，我们调用 `dns.resolve`，这是 Node.js DNS 库中的一个函数，尝试将域名解析为 IP 地址。我们推送到 `promises` 数组的这些 Promise 仅在 DNS 库完成其网络请求后才解析。

最后，`Promise.all` 块接收一个 Promise 对象数组，并且仅当数组中的每个 Promise 都已解析（完成其网络请求）时才调用 `results`（即 `.then()`）。结果中的双 `!!` 运算符表示我们只希望返回已定义的结果，因此我们应忽略返回无 IP 地址的尝试。

如果我们包含一个调用 `reject()` 的条件，我们还需要在末尾有一个 `catch()` 块来处理错误。DNS 库会抛出许多错误，其中一些可能不值得中断我们的暴力破解。为了简洁起见，示例中省略了这一点，但如果您打算进一步扩展此示例，这将是一个很好的练习。

此外，我们使用 `dns.resolve` 而非 `dns.lookup`，因为尽管两者的 JavaScript 实现都是异步解析（无论它们发起的顺序如何），但 `dns.lookup` 所依赖的原生实现基于 libuv，后者同步执行操作。

我们可以很容易地将两个脚本合并为一个程序。首先，我们生成潜在子域名列表，然后执行异步暴力破解以解析子域名：


## 字典攻击

我们可以通过使用字典攻击而不是暴力破解攻击来进一步加速此过程，而不是尝试每种可能的子域名。与暴力破解攻击非常相似，字典攻击遍历大量潜在子域名，但不是随机生成，而是从最常见子域名列表中提取。

字典攻击快得多，通常能为您找到一些有趣的东西。只有最不常见和非标准的子域名才会躲避字典攻击。

一个名为 dnscan 的流行开源 DNS 扫描器附带了一份互联网上最流行子域名的列表，基于来自超过 86,000 个 DNS 区域记录的数百万个子域名。根据 dnscan 的子域名扫描数据，前 25 个最常见的子域名如下：

- www
- ns1
- m
- www2
- admin
- forum
- news
- mail
- webdisk
- ns2
- imap
- test
- ns
- ftp
- localhost
- webmail
- smtp
- cpanel
- whm
- blog
- pop3
- dev
- autodiscover
- autoconfig
- pop

GitHub 上的 dnscan 仓库托管了包含前 10,000 个子域名的文件，由于其非常开放的 GNU v3 许可证，可以集成到您的侦察过程中。您可以在 [GitHub](https://oreil.ly/xOlQg) 上找到 dnscan 的子域名列表和源代码。

我们可以轻松地将像 dnscan 这样的字典插入我们的脚本中。对于较小的列表，您可以直接将字符串复制/粘贴/硬编码到脚本中。对于大型列表，如 dnscan 的 10,000 子域名列表，我们应将数据与脚本分开，并在运行时拉取。这将更容易修改子域名列表，或利用其他子域名列表。这些列表大多为 .csv 格式，因此集成到您的子域名侦察脚本中非常简单：

```javascript
const dns = require('dns');
const csv = require('csv-parser');
const fs = require('fs');

const promises = [];

/*
 * 开始从磁盘流式传输子域名数据（而非一次性将其全部加载到内存中，以防文件很大）。
 *
 * 在每一行，调用 `dns.resolve` 查询子域名并检查其是否存在。
 * 将这些 Promise 存储在 `promises` 数组中。
 *
 * 当所有行被读取，且所有 Promise 都被解析后，
 * 将找到的子域名记录到控制台。
 *
 * 性能提升：如果子域名列表特别大，则应打开第二个文件，
 * 并在 Promise 解析时将结果流式传输到该文件。
 */
fs.createReadStream('subdomains-10000.txt')
  .pipe(csv())
  .on('data', (subdomain) => {
    promises.push(new Promise((resolve, reject) => {
      dns.resolve(`${subdomain}.mega-bank.com`, function (err, ip) {
        return resolve({ subdomain: subdomain, ip: ip });
      });
    }));
  })
  .on('end', () => {
    // 在所有 DNS 查询完成后，记录结果
    Promise.all(promises).then(function(results) {
      results.forEach((result) => {
        if (!!result.ip) {
          console.log(result);
        }
      });
    });
  });
```

是的，就这么简单！如果您能找到一个可靠的子域名字典（只需搜索一下），您可以将其粘贴到暴力破解脚本中，现在您就有了一个字典攻击脚本可用。因为字典方法比暴力破解方法高效得多，明智的做法是首先使用字典，仅当字典未返回您要查找的结果时，再使用暴力破解子域名生成。

## 小结

当对 Web 应用执行侦察时，主要目标应该是构建应用的地图，以便在以后优先排序和部署攻击载荷时使用。此搜索的初始组成部分是了解哪些服务器负责维持应用运行——因此我们要搜索附加到应用主域名的子域名。

面向消费者的域名，如银行网站的客户端，通常受到最严格的审查。错误将很快被修复，因为访问者每天都暴露于其中。

在后台运行的服务器，如邮件服务器或管理员后门，通常充满错误，因为它们使用和暴露的频率要低得多。通常，找到这些“幕后”API 之一可以成为在应用中搜索可利用漏洞的有益开端。

在尝试查找子域名时，应使用多种技术，因为一种技术可能无法提供全面的结果。一旦您认为已执行足够的侦察并为您正在测试的域名收集了一些子域名，您可以继续进行其他侦察技术——但如果您在更明显的攻击向量上没有运气，您随时可以回来寻找更多。
