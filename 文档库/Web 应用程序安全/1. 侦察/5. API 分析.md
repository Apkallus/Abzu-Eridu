# 第 5 章：API 分析

API 端点分析是侦察工具包中继子域名发现之后的下一项逻辑技能。该应用使用了哪些域？如果此应用有三个域（例如 `x.domain`、`y.domain` 和 `z.domain`），我应该注意它们各自可能有自己独特的 API 端点。

一般来说，我们可以使用与尝试查找子域名时非常类似的技术。暴力破解攻击和字典攻击在这里效果不错，但手动努力和逻辑分析也常常会得到回报。

在发现子域名之后，了解 Web 应用结构的第二步是查找 API。这一步将为我们提供开始理解某个暴露 API 用途所需的信息。当我们理解 API 为何通过网络暴露时，我们就能开始看到它如何融入应用以及它的业务目的是什么。

## 端点发现

之前我们讨论了当今大多数企业应用在定义其 API 结构时如何遵循特定方案。通常，API 要么遵循 REST 格式，要么遵循 SOAP 格式。REST 正变得越来越流行，并被认为是当今现代 Web 应用 API 的理想结构。

当我们在浏览应用并分析网络请求时，可以使用浏览器中的开发者工具。如果我们看到许多看起来像这样的 HTTP 请求，那么很有把握地假设这是一个 REST API：

```
GET api.mega-bank.com/users/1234
GET api.mega-bank.com/users/1234/payments
POST api.mega-bank.com/users/1234/payments
```

请注意，每个端点都指定了特定资源而非函数。此外，我们可以假设嵌套资源 `payments` 属于用户 `1234`，这告诉我们这个 API 是层次化的。这是 RESTful 设计的另一个明显标志。

如果我们查看每个请求发送的 cookie，并查看每个请求的头部，我们可能也会发现 RESTful 架构的迹象：

```
POST /users/1234/payments HTTP/1.1
Host: api.mega-bank.com
Authorization: Bearer abc21323
Content-Type: application/x-www-form-urlencoded
User-Agent: Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/1.0 (KHTML, like Gecko)
```

每个请求都发送令牌是 RESTful API 设计的另一个标志。REST API 应该是无状态的，这意味着服务器不应跟踪其请求者。

一旦我们知道这确实是 REST API，我们就可以开始对可用端点进行逻辑假设。表 5-1 列出了 REST 架构支持的 HTTP 动词。使用表 5-1，我们可以查看在浏览器控制台中发现的针对特定资源的请求，并尝试使用不同的 HTTP 动词对这些资源发起请求，看看 API 是否返回任何有趣的内容。

**表 5-1. REST 架构支持的 HTTP 动词**

| REST HTTP 动词 | 用途 |
| :--- | :--- |
| POST | 创建 |
| GET | 读取 |
| PUT | 更新/替换 |
| PATCH | 更新/修改 |
| DELETE | 删除 |

HTTP 规范定义了一种特殊方法，仅用于提供有关特定 API 动词的信息。此方法称为 `OPTIONS`，在对 API 执行侦察时应该是我们的首选。我们可以在终端中使用 `curl` 轻松发起请求：

```bash
curl -i -X OPTIONS https://api.mega-bank.com/users/1234
```

如果 `OPTIONS` 请求成功，我们应该会看到以下响应：

```
200 OK
Allow: HEAD, GET, PUT, DELETE, OPTIONS
```

一般来说，`OPTIONS` 仅适用于专门指定用于公共使用的 API。因此，虽然这是一个简单的第一次尝试，但对于我们尝试测试的大多数应用，我们需要更健壮的发现解决方案。很少有企业应用会暴露 `OPTIONS`。

让我们继续讨论一种更可能用于确定接受的 HTTP 动词的方法。我们在浏览器中看到的第一个 API 调用如下：

```
GET api.mega-bank.com/users/1234
```

我们现在可以将其扩展为：

```
GET api.mega-bank.com/users/1234
POST api.mega-bank.com/users/1234
PUT api.mega-bank.com/users/1234
PATCH api.mega-bank.com/users/1234
DELETE api.mega-bank.com/users/1234
```

有了前面的 HTTP 动词列表，我们可以生成一个脚本来测试我们理论的正确性。

> 暴力破解 API 端点 HTTP 动词可能产生删除或修改应用数据的副作用。在对应用 API 执行任何类型的暴力破解尝试之前，请确保您已获得应用所有者的明确许可。

我们的脚本目的很简单：使用给定的端点（我们知道此端点已接受至少一个 HTTP 动词），尝试每个额外的 HTTP 动词。在对端点尝试每个额外 HTTP 动词后，记录并打印结果：

```javascript
/*
 * 给定一个 URL（对应于 API 端点），
 * 尝试使用各种 HTTP 动词的请求，以确定哪些 HTTP 动词映射到给定端点。
 */
const discoverHTTPVerbs = function(url) {
  const verbs = ['POST', 'GET', 'PUT', 'PATCH', 'DELETE'];
  const promises = [];

  verbs.forEach((verb) => {
    const promise = new Promise((resolve, reject) => {
      const http = new XMLHttpRequest();
      http.open(verb, url, true)
      http.setRequestHeader('Content-type', 'application/x-www-form-urlencoded');

      /*
       * 如果请求成功，解析 Promise 并将状态码包含在结果中。
       */
      http.onreadystatechange = function() {
        if (http.readyState === 4) {
          return resolve({ verb: verb, status: http.status });
        }
      }

      /*
       * 如果请求不成功，或未及时完成，将请求标记为不成功。
       * 超时时间应根据平均响应时间进行调整。
       */
      setTimeout(() => {
        return resolve({ verb: verb, status: -1 });
      }, 1000);

      // 发起 HTTP 请求
      http.send({});
    });

    // 将 Promise 对象添加到 promises 数组
    promises.push(promise);
  });

  /*
   * 当所有动词都尝试完毕后，将其各自 Promise 的结果记录到控制台。
   */
  Promise.all(promises).then(function(values) {
    console.log(values);
  });
}
```

此脚本在技术层面上的功能同样简单。HTTP 端点在返回给浏览器的任何消息旁边都会返回一个状态码。我们实际上不关心这个状态码是什么。我们只是想看到一个状态码。

我们对 API 发起多个 HTTP 请求，每个 HTTP 动词一个。大多数服务器不会响应未映射到有效端点的请求，因此我们有一个额外的情况，如果一个请求在 1 秒内未收到响应，则返回 –1。一般来说，1 秒（在此情况下为 1,000 毫秒）对于 API 响应来说是足够的时间。您可以根据自己的用例向上或向下调整此值。

在所有 Promise 都解析后，您可以查看日志输出以确定哪些 HTTP 动词具有关联的端点。

## 认证机制

猜测 API 端点所需的载荷形状比仅仅断言 API 端点存在要困难得多。最简单的方法是分析通过浏览器发送的已知请求的结构。除此之外，我们必须对 API 端点所需的形状进行有根据的猜测，并手动测试它们。自动化发现 API 端点结构是可能的，但任何不涉及分析现有请求的自动化尝试都非常容易被检测和记录。

通常最好从几乎每个应用都能找到的常见端点开始：登录、注册、密码重置等。这些端点通常采用与其他应用类似的载荷形状，因为认证通常基于标准化方案设计。

每个具有公共 Web 用户界面的应用都应该有登录页面。然而，它们认证您会话的方式可能有所不同。了解您正在处理的认证方案类型很重要，因为许多现代应用在每个请求中都发送认证令牌。这意味着如果我们能逆向工程所使用的认证类型，并理解令牌是如何附加到请求中的，那么分析依赖已认证用户令牌的其他 API 端点将更容易。当今使用的主要认证方案有几种，表 5-2 列出了最常见的几种。

**表 5-2. 主要认证方案**

| 认证方案 | 实现细节 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| HTTP 基本认证 | 每个请求发送用户名和密码 | 所有主流浏览器原生支持 | 会话不过期；易于拦截 |
| HTTP 摘要认证 | 每个请求发送用户名:域:密码的哈希值 | 更难拦截；服务器可拒绝过期令牌 | 加密强度取决于所使用的哈希算法 |
| OAuth | 基于“Bearer”令牌的认证；允许使用其他网站登录，如 Amazon → Twitch | 令牌化权限可在应用间共享以实现集成 | 网络钓鱼风险；中心站点被入侵会连累所有连接的应用 |

如果我们登录 `https://www.mega-bank.com` 并分析网络响应，在登录成功后我们可能会看到类似这样的内容：

```
GET /homepage
HOST mega-bank.com
Authorization: Basic am9lOjEyMzQ=
Content Type: application/json
```

我们一眼就能看出这是 HTTP 基本认证，因为发送了 `Basic` 授权头部。此外，字符串 `am9lOjEyMzQ=` 只是一个 base64 编码的 `username:password` 字符串。这是通过 HTTP 传输用户名和密码组合的最常见格式。

在浏览器控制台中，我们可以使用内置函数 `btoa(str)` 和 `atob(base64)` 在字符串和 base64 之间相互转换。如果我们通过 `atob` 函数运行 base64 编码的字符串，我们将看到通过网络发送的用户名和密码：

```javascript
/*
 * 解码先前用 base64 编码的字符串。
 * 结果 = joe:1234
 */
atob('am9lOjEyMzQ=');
```

由于这种机制非常不安全，基本认证通常仅在使用 SSL/TLS 流量加密的 Web 应用上使用。这样，凭据就不会在空中被拦截——例如，在可疑的商场 WiFi 热点处。

从这次登录/重定向到主页的分析中需要注意的重要一点是，我们的请求确实经过了认证，并且它们使用 `Authorization: Basic am9lOjEyMzQ=` 进行认证。这意味着如果我们遇到另一个使用空载荷未返回任何有趣内容的端点，我们应该尝试的第一件事是附加一个授权头部，看看当我们以认证用户身份请求时，它是否有不同反应。

## 端点形状

在定位了若干子域名以及这些子域名中包含的 HTTP API 之后，开始确定每个资源使用的 HTTP 动词，并将调查结果添加到您的 Web 应用地图中。一旦您有了子域名、API 和形状的综合列表，您可能会开始思考如何实际了解任何给定 API 期望什么类型的载荷。

### 常见形状

有时这个过程很简单——许多 API 期望的载荷形状是行业中常见的。例如，作为 OAuth 2.0 流程一部分设置的授权端点可能期望以下数据：

```json
{
  "response_type": "code",
  "client_id": "id",
  "scope": ["scopes"],
  "state": "state",
  "redirect_uri": "uri"
}
```

由于 [OAuth 2.0](https://oauth.net/2) 是一个广泛实施的公共规范，确定 OAuth 2.0 授权端点要包含的数据通常可以通过结合有根据的猜测和可用的公共文档来完成。OAuth 2.0 授权端点中作用域的命名约定和列表可能因实现略有不同，但整体载荷形状不应改变。

OAuth 2.0 授权端点的一个示例可以在 Discord（即时通讯）的公共文档中找到。Discord 建议对 OAuth 2.0 端点的调用应结构如下：

```
https://discordapp.com/api/oauth2/authorize?response_type=code&client_\
id=157730590492196864&scope=identify%20guilds.\
join&state=15773059ghq9183habn&redirect_uri=https%3A%2F%2Fnicememe.\
website&prompt=consent
```

这里 `response_type`、`client_id`、`scope`、`state` 和 `redirect_uri` 都是官方规范的一部分。

Facebook 关于 OAuth 2.0 的公共文档非常相似，建议对相同功能使用以下请求：

```
GET https://graph.facebook.com/v4.0/oauth/access_token?
  client_id={app-id}
  &redirect_uri={redirect-uri}
  &client_secret={app-secret}
  &code={code-parameter}
```

因此，在处理常见端点原型时，找到 HTTP API 的形状并不复杂。然而，明智的做法是考虑到，虽然许多 API 实现像 OAuth 这样的公共规范，但它们通常不会为其负责启动应用逻辑的内部 API 使用公共规范。

### 应用特定形状

应用特定的形状比基于公共规范的形状更难确定。要确定 API 端点期望的载荷形状，您可能需要依赖多种侦察技术，并通过反复试验慢慢了解端点。

不安全的应用可能会以 HTTP 错误消息的形式给您提示。例如，假设您使用以下请求体调用 `POST https://www.mega-bank.com/users/config`：

```json
{
  "user_id": 12345,
  "privacy": {
    "publicProfile": true
  }
}
```

您很可能会收到类似 `401 not authorized` 或 `400 internal error` 这样的 HTTP 状态码。如果状态码附带了类似 `auth_token not supplied` 的消息，您可能偶然发现了一个缺失的参数。

在另一个带有正确 `auth_token` 的请求中，您可能会收到另一个错误消息：`publicProfile only accepts "auth" and "noAuth" as params`。

找到了。

但更安全的应用可能只会抛出通用错误，您将不得不转向其他技术。

如果您拥有特权账户，您可以在针对另一个账户尝试之前，先通过 UI 对您自己的账户尝试相同的请求，以确定发出的形状是什么样的。这可以在浏览器开发者工具 → 网络标签或使用像 Burp 这样的网络监控工具中找到。

最后，如果您知道载荷中期望的变量名，但不知道值，那么您可以通过重复请求并变化值直到其中一个成功，来暴力破解请求。显然，手动暴力破解值很慢，所以您需要脚本来加速这个过程。您能了解到的关于期望变量的规则越多越好。如果您知道 `auth_token` 总是 12 个字符，那就很好。如果您知道它总是十六进制，那就更好了。您能学习和应用的规则越多，您就越有可能暴力破解出一个成功的组合。

字段的可能组合列表称为解空间。您希望将解空间减少到最小的可行搜索空间。

除了搜索有效解，您可能还想尝试搜索无效解。这些可能帮助您减少解空间，甚至可能发现应用代码中的错误。

## 小结

在建立了关于支撑应用的子域名的思维模型（理想情况下也以某种形式记录下来）之后，下一步是找到托管在这些子域名上的 API 端点，以便您稍后尝试确定其用途。虽然这听起来是一个简单的步骤，但作为一项侦察技术，它至关重要，因为没有它，您可能会浪费时间在保护良好的端点上寻找漏洞，而存在类似功能或数据的保护较弱的端点却被忽视。此外，如果您还不知道 API 的预期用途，查找端点上的端点是理解 API 目的和功能的一步。

一旦您发现并记录了若干 API 端点，确定该端点接受的载荷形状就是下一个逻辑步骤。结合使用有根据的猜测、自动化以及像我们在本章中所做的那样分析常见端点原型，最终将引导您发现这些端点期望的数据以及响应的数据。有了这些知识，您现在理解了应用的功能，这是攻破或保护应用的第一步。
