# 第 2 章：Web 应用程序侦察入门

Web 应用程序侦察通常指在入侵 Web 应用之前进行的探索性数据收集阶段。侦察通常由黑客、渗透测试员或漏洞赏金猎人执行，但它也是安全工程师发现 Web 应用中薄弱机制并在恶意行为者发现之前进行修补的有效方式。侦察（recon）技能本身并没有显著价值，但当与进攻性黑客知识和防御性安全工程经验相结合时，它们会变得越来越有价值。

## 信息搜集

我们已经知道，侦察就是在试图入侵应用之前，建立对它的深入理解。我们也知道侦察是优秀黑客工具包中必不可少的部分。但到目前为止，我们对侦察的了解大致止步于此。所以，让我们从更技术性的角度来头脑风暴一下，为什么侦察很重要。

> 后续章节中介绍的许多侦察技术对于绘制应用地图很有用，但也可能导致您的 IP 被标记，可能造成应用封禁甚至法律诉讼。大多数侦察技术只应针对您拥有或获得书面许可测试的应用执行。

侦察可以通过多种方式完成。有时，只需浏览 Web 应用并记录网络请求，就足以让您深入了解该应用的内部工作原理。但是，需要注意的是，并非所有 Web 应用都有用户界面供我们可视化地探索并记录其功能。

大多数面向公众的应用（通常是像社交媒体这样的企业对消费者应用）都会有面向公众的用户界面。然而，我们不应假设即使在这种情况下我们也能访问完整的用户界面。相反，在进一步调查之前，我们应假设我们只能访问用户界面的一个子集。

让我们逻辑性地思考几分钟。当您去当地的 MegaBank 并开设一个新银行账户（例如一个支票账户）时，您通常也会收到登录凭据，允许您通过网络查看账户信息。通常您的账户信息是由银行员工手动录入的，通常是带您完成文书的银行柜员。这意味着在某时某刻，其他人曾有权访问一个 Web 或联网应用，可以在银行数据库中创建新账户。

此外，如果您打电话给银行经理要求为您开设一个新的储蓄账户，他们也会照做。通常，只要您能提供正确的凭据以正确识别您的身份，他们会远程完成此操作。对于大多数主要银行，这个新的储蓄账户将可以通过您支票账户已有的相同登录信息访问。

由此我们可以推断，也有人有权访问一个允许他们编辑与您（现有）账户相关信息以将其与新创建的储蓄账户关联的应用。这可能是用于创建您支票账户的同一个应用，也可能是一个完全不同的应用。

接下来，您无法在线手动关闭银行账户，但您可以轻松走进当地分行要求关闭账户。在您的请求获批准后，您的账户将很快被关闭，通常在几小时内。

您可以通过 Web 应用访问银行账户查看余额——但通常您只能使用此界面读取余额。这意味着您拥有只读访问权限。

一些银行可能允许我们在线支付账单或转账——但没有银行允许客户在线创建、修改或删除自己的账户。即使使用最先进的数字银行系统，银行的客户也只有有限的写入权限。然而，银行管理员和受信任的员工拥有修改、创建和删除账户所需的权限。请考虑下表，该表描述了银行应用的三种不同用户：

| 用户 | 类型 | 权限 |
| :--- | :--- | :--- |
| 客户 | 外部 | 登录网站。通过 Web UI 读取账户余额。 |
| 柜员 | 内部 | 在客户提供文件后创建新账户。代表客户修改现有账户。 |
| 银行经理 | 内部 | 删除账户。修改账户权限。 |

对于大型银行来说，为每个修改账户的操作手动编写数据库查询是不现实的，因此逻辑上我们可以预期他们编写了自动化程序来完成这些操作，即使我们（外部用户）无法访问它。

大多数自动化数据访问的应用都使用分层用户权限结构。我们将权限结构类似这样的应用称为基于角色的访问控制（RBAC）应用。如今很少有应用对所有用户仅使用一个权限级别。

您可能在自己使用过的软件中也见过这些控制措施；例如，在操作系统上调用危险命令可能会提示您输入管理员凭据。或者，许多社交媒体网站有版主，他们拥有比普通用户更高的权限级别，但通常低于管理员。

如果我们仅通过 Web 应用的用户界面来浏览，可能永远不会了解到为这些高权限用户（如管理员、版主等）设计的 API 端点。但掌握了 Web 应用侦察技术后，我们通常能找到这些 API。我们甚至可以构建一张复杂的图谱，详细描述管理员或版主的全部权限，以便与普通用户的权限集进行比较。偶尔，我们可能会发现一些故障，允许非特权用户利用仅为更高特权用户设计的功能。

侦察技能也可用于收集我们实际上无法访问的应用的信息。这可能是一个学校的内部网络，或一个公司可通过网络访问的文件服务器。如果我们具备逆向工程应用 API 结构及这些 API 所接受载荷的正确技能，我们就不需要用户界面来了解应用的运行方式。

有时在进行侦察时，您可能会遇到完全没有保护的服务器或 API。许多公司依赖多个服务器，包括内部和外部。仅仅忘记一行网络或防火墙配置，就可能导致 HTTP 服务器暴露在公共网络中，而不是仅限于内部网络。

当您构建出一个 Web 应用的技术和架构样貌时，您也将能更好地优先安排您的攻击。您将了解应用的哪些部分受到最严密的保护，哪些部分还需要加强。

## Web 应用映射

随着我们逐步完成第一部分，您将学习如何构建一个代表 Web 应用结构、组织和功能的图谱。这通常应该是您尝试入侵 Web 应用之前的第一步。随着您对 Web 应用侦察越来越熟练，您将发展出自己的技术，以及自己记录和组织所发现信息的方法。

许多人将一组有序的地形点称为地图。“地形”一词意为研究地貌、形状和表面。Web 应用也有特征、形状和表面。这些与自然界中的非常不同，但许多相同概念仍然适用。我们将在此使用“地图”一词来定义关于应用代码、网络结构和功能集所收集的数据点。在接下来的几章中，您将学习如何获取填充地图所需的数据。

根据您所测试应用的复杂性以及您计划测试的时长，将地图存储在简单的草稿笔记中可能就足够了。对于更复杂或您打算长期频繁测试的应用，您可能需要更健壮的解决方案。您如何构建自己的地图最终取决于您——只要易于遍历且能存储相关信息和关系，任何格式都应足够。

就个人而言，我喜欢为大多数笔记使用类似 JavaScript 对象表示法（JSON）的格式。我发现层次化数据结构在 Web 应用中非常常见，它们也让我更容易排序和搜索笔记。

以下是一段类似 JSON 的侦察笔记示例，描述了在 Web 应用的 API 服务器中找到的一组 API 端点：

```json
{
  "api_endpoints": {
    "sign_up": {
      "url": "mywebsite.com/auth/sign_up",
      "method": "POST",
      "shape": {
        "username": { "type": "String", "required": true, "min": 6, "max": 18 },
        "password": { "type": "String", "required": true, "min": 6, "max": 32 },
        "referralCode": { "type": "String", "required": false, "min": 64, "max": 64 }
      }
    },
    "sign_in": {
      "url": "mywebsite.com/auth/sign_in",
      "method": "POST",
      "shape": {
        "username": { "type": "String", "required": true, "min": 6, "max": 18 },
        "password": { "type": "String", "required": true, "min": 6, "max": 32 }
      }
    },
    "reset_password": {
      "url": "mywebsite.com/auth/reset",
      "method": "POST",
      "shape": {
        "username": { "type": "String", "required": true, "min": 6, "max": 18 },
        "password": { "type": "String", "required": true, "min": 6, "max": 32 },
        "newPassword": { "type": "String", "required": true, "min": 6, "max": 32 }
      }
    }
  },
  "features": {
    "comments": {},
    "uploads": {
      "file_sharing": {}
    }
  },
  "integrations": {
    "oauth": {
      "twitter": {},
      "facebook": {},
      "youtube": {}
    }
  }
}
```

像 Notion 这样的层次化笔记软件，或像 XMind 这样的思维导图应用，也是用于记录和组织您通过侦察尝试所学内容的绝佳工具。最终，您需要找到一种适合您的方法，既能保持组织性，又足够健壮，能在需要时扩展到复杂应用之上。

## 小结

侦察技术对于深入理解 Web 应用的技术和结构，以及支持该 Web 应用的服务非常有价值。除了能够对 Web 应用执行侦察外，我们还必须密切关注我们的发现，并以一种组织良好、便于日后轻松遍历的方式记录它们。

本章中展示的类似 JSON 的笔记描述了我个人在记录对 Web 应用的侦察工作时偏好的笔记风格。然而，侦察笔记记录最重要的方面是保留关系和层次结构，同时保持笔记易于阅读和手动遍历。

您必须找到一种适合您、并能从小型应用到大型应用都可扩展的记录风格。如果您找到了更适合您的替代风格或格式，那就使用它；笔记的内容和结构远比存储它们的应用或格式重要。
