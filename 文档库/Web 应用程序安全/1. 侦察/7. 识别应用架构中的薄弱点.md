# 第 7 章：识别应用架构中的薄弱点

到目前为止，我们已经讨论了许多用于识别 Web 应用组件、确定 Web 应用中 API 的形状，以及了解 Web 应用如何与用户浏览器交互的技术。每种技术本身都有价值，但当以有组织的方式将收集到的信息组合起来时，可以获得更多价值。

理想情况下，正如之前建议的，您在整个侦察过程中应保持某种形式的笔记。正确记录您的研究是必不可少的，因为某些 Web 应用非常庞大，探索其所有功能可能需要数月时间。侦察期间产生的文档数量最终取决于您自己（测试员、黑客、爱好者、工程师等），且如果优先级不恰当，更多并不总是更有价值，尽管有数据总比没数据好。

对于您测试的每个应用，理想情况下您最终会得到一组组织良好的笔记。这些笔记应涵盖：

- Web 应用所使用的技术
- 按 HTTP 动词分类的 API 端点列表
- API 端点形状列表（如果有）
- Web 应用包含的功能（例如评论、认证、通知等）
- Web 应用使用的域名
- 找到的配置（例如内容安全策略 [CSP]）
- 认证/会话管理系统

一旦您完成了这份列表的编译，就可以用它来优先排序任何尝试入侵应用或查找漏洞的工作。

与普遍看法相反，Web 应用中的大多数漏洞源于不恰当设计的应用架构，而非编写不良的方法。当然，一个直接将用户提供的 HTML 写入 DOM 的方法确实存在风险，并可能允许用户上传脚本（如果没有适当的清理）并在另一用户的机器上执行该脚本（XSS）。

但有些应用存在数十个 XSS 漏洞，而同一行业中其他规模相似的应用几乎为零。最终，应用的架构以及该应用内模块/依赖的架构是可能产生漏洞的薄弱点的极佳标记。

## 安全与不安全的架构信号

如前所述，单个 XSS 漏洞可能是方法编写不当的结果。但多个漏洞很可能是应用架构薄弱的标志。

让我们设想两个允许用户向其他用户发送直接消息（文本）的简单应用。其中一个应用容易受到 XSS 攻击，而另一个则没有。

不安全的应用在向 API 端点发出存储评论的请求时，可能不会拒绝脚本；其数据库可能不会拒绝脚本，并且可能不会对代表消息的字符串进行适当的过滤和清理。最终，它被加载到 DOM 中并作为 `DOM message<script>alert('hacked');</script>` 进行解析，从而导致脚本执行。

另一方面，安全的应用很可能具备上述一种或多种防护措施。然而，在每种情况下实施多重防护，就开发者时间而言成本很高，且容易被忽视。即使是由精通应用安全的工程师编写的应用，如果其应用架构本质上不安全，最终也可能出现安全漏洞。这是因为安全应用在功能开发之前和开发期间就实现了安全措施，而安全性平庸的应用在功能开发时实现安全措施，不安全的应用可能根本不实现任何安全措施。

如果开发者在 5 年时间内必须为前述示例中的即时通讯（IM）系统编写 10 种变体，那么每种实现很可能在结构上有所不同。然而，每种实现之间的安全风险大致相同。

每个 IM 系统都包含以下功能：

- 编写消息的 UI
- 接收刚刚编写和提交的消息的 API 端点
- 存储消息的数据库表
- 检索一条或多条消息的 API 端点
- 显示一条或多条消息的 UI 代码

最低限度，应用代码如下所示：

**client/write.html**
```html
<!-- 消息输入的基本 UI -->
<h2>向 <span id="target">TestUser</span> 发送消息</h2>
<input type="text" class="input" id="message"></input>
<button class="button" id="send" onclick="send()">发送消息</button>
```

**client/send.js**
```javascript
const session = require('./session');
const messageUtils = require('./messageUtils');

/*
 * 遍历 DOM 并收集两个值：要发送的消息内容和目标消息接收者的用户名或其他唯一标识符（id）。
 * 调用 messageUtils 生成一个经过认证的 HTTP 请求，将提供的数据（message, user）发送到服务器上的 API。
 */
const send = function() {
  const message = document.querySelector('#send').value;
  const target = document.querySelector('#target').value;
  messageUtils.sendMessageToServer(session.token, target, message);
};
```

**server/postMessage.js**
```javascript
const saveMessage = require('./saveMessage');

/*
 * 接收来自客户端 send.js 的数据，验证用户权限，如果所有验证检查通过，将提供的消息保存到数据库中。
 * 如果成功，返回 HTTP 状态码 200。
 */
const postMessage = function(req, res) {
  if (!req.body.token || !req.body.target || !req.body.message) {
    return res.sendStatus(400);
  }

  saveMessage(req.body.token, req.body.target, req.body.message)
    .then(() => {
      return res.sendStatus(200);
    })
    .catch((err) => {
      return res.sendStatus(400);
    });
};
```

**server/messageModel.js**
```javascript
const session = require('./session');

/*
 * 表示一个消息对象。作为模式（schema）使用，以便所有消息对象包含相同的字段。
 */
const Message = function(params) {
  user_from: session.getUser(params.token),
  user_to: params.target,
  message: params.message
};

module.exports = Message;
```

**server/getMessage.js**
```javascript
const session = require('./session');

/*
 * 从服务器请求消息，验证权限，如果成功则从数据库中提取消息，然后将消息返回给通过客户端请求它的用户。
 */
const getMessage = function(req, res) {
  if (!req.body.token) { return res.sendStatus(401); }
  if (!req.body.messageId) { return res.sendStatus(400); }

  session.requestMessage(req.body.token, req.body.messageId)
    .then((msg) => {
      return res.send(msg);
    })
    .catch((err) => {
      return res.sendStatus(400);
    });
};
```

**client/displayMessage.html**
```html
<!-- 显示从服务器请求的单条消息 -->
<h2>显示来自 <span id="message-author"></span> 的消息</h2>
<p class="message" id="message"></p>
```

**client/displayMessage.js**
```javascript
const session = require('./session');
const messageUtils = require('./messageUtils');

/*
 * 使用工具函数通过 HTTP GET 请求单条消息，然后将其附加到 #message 元素，并将作者附加到 #message-author 元素。
 * 如果 HTTP 请求获取消息失败，则将错误记录到控制台。
 */
const displayMessage = function(msgId) {
  messageUtils.getMessageById(session.token, msgId)
    .then((msg) => {
      messageUtils.appendToDOM('#message', msg);
      messageUtils.appendToDOM('#message-author', msg.author);
    })
    .catch(() => console.log('an error occured'));
};
```

保护这个简单应用所需的许多安全机制可以而且应该被抽象到应用架构中，而不是逐个案例地实现。

以 DOM 注入为例。在 UI 中构建一个像下面这样的简单方法将消除大多数 XSS 风险：

```javascript
import { DOMPurify } from '../utils/DOMPurify';
// 使用：https://github.com/cure53/DOMPurify

const appendToDOM = function(data, selector, unsafe = false) {
  const element = document.querySelector(selector);
  // 对于需要 DOM 注入的情况（非默认）
  if (unsafe) {
    element.innerHTML = DOMPurify.sanitize(data);
  } else { // 标准情况（默认）
    element.innerText = data;
  }
};
```

仅围绕这样一个函数构建您的应用，就能显著降低代码库中出现 XSS 漏洞的风险。

然而，这些方法的实现很重要——请注意，前面代码示例中的 DOM 注入标志被专门标记为 `unsafe`。它不仅默认关闭，而且是函数签名中的最后一个参数，这意味着不太可能被意外触发。

像前面的 `appendToDOM` 方法这样的机制是安全应用架构的标志。缺乏这些安全机制的应用更有可能包含漏洞。这就是为什么识别不安全的应用架构对于发现漏洞和优先改进代码库都很重要。

## 多层安全

在我们之前考虑消息服务架构的示例中，我们识别并隔离了 XSS 风险可能发生的多个层。这些层是：

- API POST
- API GET
- 数据库写入
- 数据库读取
- 客户端读取

其他类型的漏洞也是如此，例如 XXE 或跨站请求伪造（CSRF）——每个漏洞都可能因多个层的安全机制不足而发生。例如，假设一个假想的应用（如消息应用）在 API POST 层添加了机制，通过对用户发送的载荷（消息）进行清理来消除 XSS 风险。现在，通过 API POST 层部署 XSS 可能已不可能。

然而，在稍后的时间点，可能会开发并部署另一种发送消息的方法。例如，一个新的 API POST 端点接受消息列表以支持批量消息发送。如果新的 API 端点提供的清理功能不如原始端点强大，它可能被用来将包含脚本的载荷上传到数据库，从而绕过开发者在单消息 API 中的原始意图。

我提出这个简单的例子是为了指出，一个应用的安全性仅取决于其架构中最薄弱的环节。如果该服务的开发者在多个位置（如 API POST 和数据库写入阶段）实现了机制，那么新的攻击就可以得到缓解。

有时，不同的安全层可以支持针对特定攻击类型的不同防御机制。例如，API POST 可以调用无头浏览器并尝试模拟消息在页面上的渲染，如果检测到任何脚本执行，则拒绝消息载荷。在数据库层或客户端层不可能实现涉及无头浏览器的缓解措施。

不同的机制也可以检测不同的攻击载荷。无头浏览器可能检测到脚本执行，但如果浏览器特定的 API 存在错误，脚本可能绕过此机制。这可能发生，因为载荷在无头浏览器中不会执行，但只在具有易受攻击浏览器版本（与服务器上测试的浏览器或版本不同）的用户浏览器中执行。

所有这些例子都表明，最安全的 Web 应用在多个层引入安全机制，而不安全的 Web 应用仅在一两个层引入安全机制。在测试 Web 应用时，您应该寻找应用中那些使用少数安全机制或需要大量层（因此安全机制与层的比率可能较低）的功能。如果您能隔离并确定哪些功能符合此标准，那么在寻找漏洞时应优先于其他功能；它更可能被利用。

## 采用与改造

另一个需要关注的风险因素是开发人员改造现有技术的倾向。通常，这并非始于架构问题。相反，它通常是一个组织问题，这反映并体现在应用架构中。

这在许多软件公司中很常见，因为从开发角度看，改造工具或功能具有许多好处，包括：

- 避免复杂的许可
- 为功能增加额外功能
- 通过营销新工具或功能来制造宣传

除此之外，从头创建功能通常比重新利用现有开源或付费工具更有趣、更具挑战性。但改造并非总是不好的，因此每种情况都必须单独评估。

在某些情况下，改造现有软件可能给公司带来比弊端更多的好处。例如，如果最好的工具有一个许可协议，要求高额佣金导致负利润，或禁止修改以致应用不得不放弃必要功能。

另一方面，从安全角度看，改造是有风险的。风险根据被改造的具体功能而增减，但范围可能从中等安全风险到极端安全风险不等。

特别是，资深安全工程师建议永远不要自己编写加密算法。有才华的软件工程师和数学家可能能够开发自己的哈希算法以避免使用开放算法——但代价是什么？

考虑哈希算法 SHA-3。SHA-3 是 SHA 算法家族的一部分，这是一个开源哈希算法，已开发近 20 年，并接受了美国国家标准与技术研究院（NIST）的严格测试，以及美国最大安全公司的贡献。

从哈希算法生成的哈希值经常受到来自多种攻击向量（例如组合攻击、马尔可夫攻击等）的攻击。开发人员编写的哈希算法必须能承受与最佳开放算法相同的鲁棒性。

推出一个具有与 NIST 和其他组织为 SHA-3 开发提供的同等广泛测试水平的算法，将花费组织数千万美元。但零成本，组织可以采用来自像 OpenJDK 这样的来源的 SHA-3 实现，并仍能获得来自 NIST 和社区测试的所有好处。

决定推出自己哈希算法的孤立软件开发者很可能无法达到相同的标准并进行稳健测试。因此，组织的关键数据将成为黑客的容易目标。

那么我们如何确定哪些功能或工具应该采用，哪些应该改造呢？一般来说，安全架构的应用只会改造纯粹功能性的功能，例如改造用于存储评论或通知系统的模式。

需要数学、操作系统或硬件深层专业知识的功能可能应该由 Web 应用开发者保留。这包括数据库、进程隔离和大多数内存管理。

不可能在所有方面都是专家。优秀的 Web 应用开发者理解这一点，并将精力集中在他们的专业领域，在主要领域之外操作时请求协助。另一方面，糟糕的开发者经常尝试改造关键任务功能——这并不罕见！

充满自定义数据库、自定义加密和特殊硬件级优化的应用通常最容易入侵。这条规则可能有一些罕见的例外，但它们是异常值而非常态。

## 小结

当谈论 Web 应用中的漏洞时，我们通常谈论的是代码级别的问题，或作为编写不当代码的结果。然而，代码级别的问题可以在应用架构中更早地被轻易发现。通常，应用的架构设计要么导致大量安全错误，要么导致相对较少数量的安全错误，这取决于应用的防御措施是如何设计和分布在代码库中的。

因此，识别应用架构中薄弱点的能力是一项有用的侦察技术。在寻找漏洞时，应首先关注架构不良的功能，因为通常具有良好安全架构的功能在从一个端点跳到另一个端点或尝试绕过过滤系统时会保持更一致。

应用架构通常在非常高的层次上讨论，而不是大多数安全工作发生的低层次。如果您不习惯从设计角度考虑应用，这可能使它成为一个令人困惑的主题。

在作为侦察工作的一部分调查 Web 应用时，请确保在绘制地图时考虑应用的整体安全架构。掌握架构分析不仅有助于您在寻找漏洞时集中精力，还可能通过识别导致先前功能中出现错误的模式，帮助您识别未来功能中的薄弱架构。
