# 第 6 章：识别第三方依赖

如今，大多数 Web 应用都是通过内部代码与通过多种集成技术整合的外部代码相结合构建的。外部依赖可以是来自其他公司的专有软件（在特定许可模式下允许集成），也可以是来自开源社区的免费软件。在应用代码中使用此类第三方依赖并非没有风险，而且第三方依赖通常不会像内部代码那样接受同样严格的安全审查。

在侦察过程中，您很可能会遇到许多第三方集成，您需要密切关注依赖本身及其集成方式。这些依赖常常可能变成攻击向量；有时这类依赖中的漏洞是众所周知的，您甚至可能不需要自己准备攻击，而是可以直接从通用漏洞披露（CVE）数据库中复制攻击方法。

## 检测客户端框架

通常，开发者不会自行构建复杂的 UI 基础设施，而是利用经过良好维护和测试的 UI 框架。这些框架通常以 SPA 库（用于处理复杂状态）、仅限 JS 的库（用于修补 JavaScript 语言在跨浏览器中的功能缺陷，如 Lodash、JQuery）或 CSS 框架（用于改善网站外观，如 Bootstrap、Bulma）的形式出现。

通常，这三类框架都很容易检测。如果您能确定版本号，通常可以在网上找到适用的 ReDoS、原型污染和 XSS 漏洞组合（特别是未更新的旧版本）。

### 检测 SPA 框架

截至 2023 年，Web 上最大的 SPA 框架有（排名不分先后）：

- EmberJS（LinkedIn、Netflix）
- Angular（Google）
- React（Facebook）
- VueJS（Adobe、GitLab）

这些框架中的每一个在管理 DOM 元素的方式以及开发者与框架交互的方式上都引入了非常特定的语法和顺序。并非所有框架都这么容易检测。有些需要指纹识别或高级技术。当版本信息直接给出时，一定要记下来。

#### EmberJS

EmberJS 很容易检测，因为当 EmberJS 启动时，它会设置一个全局变量 `Ember`，可以在浏览器控制台中轻松找到（见图 6-1）。

图 6-1. 检测 EmberJS 版本

EmberJS 还会为所有 DOM 元素标记 `ember-id` 供其内部使用。这意味着如果您通过开发者工具 → 元素标签查看任何使用 EmberJS 的网页的 DOM 树，您应该会看到许多包含 `id=ember1`、`id=ember2`、`id=ember3` 等的 `div`。每个这样的 `div` 都应该被包裹在一个 `class="ember-application"` 的父元素中，通常是 `body` 元素。

EmberJS 使得检测正在运行的版本变得容易。只需引用附加到全局 `Ember` 对象的常量：

```javascript
// 3.1.0
console.log(Ember.VERSION);
```

#### Angular

旧版本的 Angular 提供了一个类似于 EmberJS 的全局对象。该全局对象名为 `angular`，版本可以从其属性 `angular.version` 获取。Angular 4.0+ 移除了这个全局对象，这使得确定 Angular 应用的版本变得稍微困难一些。您可以通过检查控制台中是否存在 `ng` 全局来判断应用是否运行 Angular 4.0+。

要检测版本，您需要多做一点工作。首先，获取 Angular 应用中的所有根元素。然后检查第一个根元素上的属性。第一个根元素应该有一个 `ng-version` 属性，它将提供您所调查应用的 Angular 版本：

```javascript
// 返回根元素数组
const elements = getAllAngularRootElements();
const version = elements[0].attributes['ng-version'];
// ng-version="6.1.2"
console.log(version);
```

#### React

React 可以通过全局对象 `React` 来识别，与 EmberJS 类似，它的版本也可以通过一个常量轻松检测：

```javascript
const version = React.version;
// 0.13.3
console.log(version);
```

您还可能注意到带有 `type="text/jsx"` 的脚本标签，引用了 React 的特殊文件格式，该格式将 JavaScript、CSS 和 HTML 都放在同一个文件中。这明确表明您正在处理 React 应用，并且知道组件的每个部分都源自单个 `.jsx` 文件，可以使调查单个组件变得更容易。

#### VueJS

与 React 和 EmberJS 类似，VueJS 暴露了一个全局对象 `Vue`，其中包含版本常量：

```javascript
const version = Vue.version;
// 2.6.10
console.log(version);
```

如果您无法检查 VueJS 应用的元素，很可能是因为该应用被配置为忽略开发者工具。这是附加到全局对象 `Vue` 的一个可切换属性。您可以将此属性设置为 `true`，以便再次在浏览器控制台中检查 VueJS 组件：

```javascript
// Vue 组件现在可以被检查
Vue.config.devtools = true;
```

### 检测 JavaScript 库

JavaScript 辅助库的数量多到无法计数，有些暴露全局对象，而另一些则隐蔽运行。许多 JavaScript 库使用顶级全局对象来命名空间其函数。这些库非常容易检测和遍历（见图 6-2）。

图 6-2. JavaScript 库全局对象

Underscore 和 Lodash 使用下划线符号（`_`）暴露全局对象，而 JQuery 使用 `$` 命名空间。然而，除了主要库之外，您最好运行一个查询来查看页面中加载的所有外部脚本。

我们可以利用 DOM 的 `querySelectorAll` 函数快速找到页面中导入的所有第三方脚本列表：

```javascript
/*
 * 利用内置的 DOM 遍历函数
 * 快速生成当前页面中导入的每个 <script> 标签列表。
 */
const getScripts = function() {
  /*
   * 查询选择器可以以 "." 开头（引用 CSS 类），
   * 以 "#" 开头（引用 `id` 属性），
   * 或无前缀（引用 HTML 元素）。
   *
   * 在本例中，'script' 将查找所有 <script> 实例。
   */
  const scripts = document.querySelectorAll('script');

  /*
   * 遍历每个 `<script>` 元素，并检查该元素是否包含非空的源（src）属性。
   */
  scripts.forEach((script) => {
    if (script.src) {
      console.log(`i: ${script.src}`);
    }
  });
};
```

调用此函数将给出类似这样的输出：

```
getScripts();
[VM183:5 i: https://www.google-analytics.com/analytics.js
[VM183:5 i: https://www.googletagmanager.com/gtag/js?id=UA-1234
[VM183:5 i: https://js.stripe.com/v3/
[VM183:5 i: https://code.jquery.com/jquery-3.4.1.min.js
[VM183:5 i: https://cdnjs.cloudflare.com/ajax/libs/d3/5.9.7/d3.min.js
VM183:5 i: /assets/main.js
```

从这里我们需要直接访问各个脚本以确定顺序、配置等。

### 检测 CSS 库

对检测脚本的算法稍作修改，我们也可以检测 CSS：

```javascript
/*
 * 利用浏览器内置的 DOM 遍历，
 * 快速聚合每个包含值为 `stylesheet` 的 `rel` 属性的 `<link>` 元素。
 */
const getStyles = function() {
  const scripts = document.querySelectorAll('link');

  /*
   * 遍历每个链接，并确认 `link` 元素包含值为 `stylesheet` 的 `rel` 属性。
   *
   * Link 是一个多功能元素，最常用于加载 CSS 样式表，
   * 但也用于预加载、图标或搜索。
   */
  scripts.forEach((link) => {
    if (link.rel === 'stylesheet') {
      console.log(`i: ${link.getAttribute('href')}`);
    }
  });
};
```

同样，此函数将输出导入的 CSS 文件列表：

```
getStyles();
VM213:5 i: /assets/jquery-ui.css
VM213:5 i: /assets/boostrap.css
VM213:5 i: /assets/main.css
VM213:5 i: /assets/components.css
VM213:5 i: /assets/reset.css
```

## 检测服务端框架

检测客户端（浏览器）上运行的软件比检测服务器上运行的软件要容易得多。大多数时候，客户端所需的所有代码都被下载并通过 DOM 引用存储在内存中。有些脚本可能按条件加载或在页面加载后异步加载，但只要您触发正确的条件，它们仍然可以被访问。

检测服务器的依赖要困难得多，但通常并非不可能。有时服务端依赖会在 HTTP 流量（头部、可选字段）中留下明显痕迹，或暴露自己的端点。检测服务端框架需要更多关于各个框架本身的知识，但幸运的是，就像在客户端一样，有几个包被广泛使用。如果您能记住检测顶级包的方法，您就能在许多您调查的 Web 应用上识别它们。

### 头部检测

一些配置不安全的 Web 服务器包会在默认头部中暴露过多数据。一个典型的例子是 `X-Powered-By` 头部，它会直接泄露 Web 服务器的名称和版本。在旧版本的 Microsoft IIS 上，这通常是默认启用的。向这些易受攻击的 Web 服务器发起任何调用，您都会在响应中看到如下返回值：

```
X-Powered-By: ASP.NET
```

如果非常幸运，Web 服务器甚至可能提供更多信息：

```
Server: Microsoft-IIS/4.5
X-AspNet-Version: 4.0.25
```

聪明的服务器管理员会禁用这些头部，聪明的开发团队也会从默认配置中移除它们。但仍有数百万网站暴露这些头部，供任何人读取。

### 默认错误消息和 404 页面

一些流行的框架没有提供非常容易的方法来确定所使用的版本号。如果这些框架是开源的，比如 Ruby on Rails，那么您可能能够通过指纹识别确定所使用的版本。Ruby on Rails 是最大的开源 Web 应用框架之一，其源代码托管在 GitHub 上以便协作。不仅最新版本可用，所有使用 Git 版本控制的历史版本都可以找到。因此，从一次提交到另一次提交的具体更改可以用来指纹识别所使用的 Ruby on Rails 版本（见图 6-3）。

图 6-3. 指纹识别所使用的 Ruby on Rails 版本

您是否曾经访问过某个 Web 应用，却看到标准的 404 页面或弹出开箱即用的错误消息？大多数 Web 服务器都提供自己的默认错误消息和 404 页面，这些页面在 Web 应用所有者替换为自定义替代页面之前，会持续呈现给用户。

这些 404 页面和错误消息可以暴露相当多的关于您服务器设置的信息。它们不仅可能暴露您的服务器软件，而且通常还能暴露版本或版本范围。

例如，全栈 Web 应用框架 Ruby on Rails 有自己的默认 404 页面，这是一个包含“您正在查找的页面不存在”框的 HTML 页面（见图 6-4）。

图 6-4. Ruby on Rails 默认 404 页面

这个页面的 HTML 可以在 Ruby on Rails 的公共 GitHub 仓库中找到，文件路径为 `rails/railties/lib/rails/generators/rails/app/templates/public/404.html`。如果您在本地机器上克隆 Ruby on Rails 仓库（使用 `git clone https://github.com/rails/rails`），并开始筛选对该页面的更改（使用 `git log | grep 404`），您可能会发现一些有趣的信息片段，例如：

- 2017 年 4 月 20 日——向 404 页面添加了命名空间 CSS 选择器
- 2013 年 11 月 21 日——将 U+00A0 替换为空格
- 2012 年 4 月 5 日——移除了 HTML5 type 属性

现在，如果您正在测试一个应用，并且偶然发现了它的 404 页面，您可以搜索在 2012 年被移除的 HTML5 type 属性 `type="text/css"`。如果存在，说明您使用的 Ruby on Rails 版本是 2012 年 4 月 5 日或更早发布的。

接下来，您可以查找 U+00A0 字符。如果存在，那么应用的 Ruby on Rails 版本来自 2013 年 11 月 21 日或更早。

最后，您可以搜索命名空间 CSS 选择器 `.rails-default-error-page`。如果这些不存在，那么您知道 Ruby on Rails 版本来自 2017 年 4 月 20 日或更早。

假设您很幸运，HTML5 type 属性已被移除，U+00A0 已被替换为空格，但命名空间 CSS 选择器尚未出现在您测试的 404 页面中。现在，我们可以将这些时间框架与 Ruby Gems 包管理器网站上列出的官方发布时间表进行交叉参考。通过这种交叉参考，我们可以确定一个版本范围。

通过这个交叉参考练习，我们可以确定正在测试的 Ruby on Rails 版本介于 3.2.16 和 4.2.8 之间。碰巧 Ruby on Rails 版本 3.2.x 到 4.2.7 存在一个 XSS 漏洞，该漏洞在互联网和漏洞数据库中都有详细记录（CVE-2016-6316）。

该攻击允许黑客将带有引号的 HTML 代码注入到 Ruby on Rails 客户端的 Action View Tag 辅助程序读取的任何数据库字段中。此 HTML 中包含 JavaScript 代码的脚本标签，会在任何访问基于 Ruby on Rails 的 Web 应用并以触发 Action View 辅助程序运行的方式与之交互的设备上执行。

这只是调查 Web 应用的依赖和版本如何导致轻松利用的一个例子。我们将在本书的下一部分讨论此类利用，但请记住，这些技术不仅适用于 Ruby on Rails。它们适用于任何第三方依赖，只要您（黑客或测试员）能够确定该软件及其版本。

### 数据库检测

大多数 Web 应用使用服务端数据库（如 MySQL 或 MongoDB）来存储关于用户、对象和其他持久化数据的状态。很少有 Web 应用开发者会自行构建数据库，因为高效可靠地存储和检索大量数据并非易事。

如果数据库错误消息直接发送给客户端，可以使用与检测服务器包类似的技术来确定数据库类型。但通常情况并非如此，因此您必须找到其他发现途径。

一种可用的技术是主键扫描。大多数数据库支持“主键”的概念，它指的是表（SQL）或文档（NoSQL）中在对象创建时自动生成的键，用于在数据库中快速执行查找。这些键的生成方法因数据库而异，有时如果开发人员有特殊需求（如 URL 中使用的较短键）可以配置。如果您能确定几个主流数据库默认主键的生成方式，除非默认方法被覆盖，否则在筛选足够多的网络请求后，您很可能能够确定数据库类型。

以 MongoDB（一个流行的 NoSQL 数据库）为例。默认情况下，MongoDB 为每个创建的文档生成一个名为 `_id` 的字段。`_id` 键使用低碰撞哈希算法生成，结果始终是一个长度为 12 的十六进制兼容字符串。此外，MongoDB 使用的算法在其开源文档中可见。

文档告诉我们以下信息：

- 用于生成这些 ID 的类称为 `ObjectId`。
- 每个 ID 恰好是 12 字节。
- 前 4 个字节表示自 Unix 纪元以来的秒数（Unix 时间戳）。
- 接下来的 5 个字节是随机的。
- 最后 3 个字节是一个以随机值开头的计数器。

一个 `ObjectId` 示例看起来像这样：`507f1f77bcf86cd799439011`。

`ObjectId` 规范还列出了辅助方法，如 `getTimestamp()`，但由于我们将在客户端而非服务器端分析流量和数据，这些辅助方法很可能不会暴露给我们。相反，了解了 MongoDB 主键的结构，我们想查看 HTTP 流量并分析载荷，寻找外观相似的 12 字节字符串。这通常很简单，您会发现一个主键以如下请求形式出现：

```
GET users/:id
其中 :id 是主键

PUT users, body = { id: id }
其中 id 也是主键

GET users?id=id
其中 id 是主键，但在查询参数中
```

有时这些 ID 会出现在您最意想不到的地方，例如元数据或关于用户对象的响应中：

```json
{
  "_id": "507f1f77bcf86cd799439011",
  "username": "joe123",
  "email": "joe123@my-email.com",
  "role": "moderator",
  "biography": "..."
}
```

无论您如何找到主键，如果您能确定该值确实是数据库的主键，那么您就可以开始研究数据库并尝试找到与其键生成算法匹配的数据库。通常这足以确定 Web 应用使用的数据库，但偶尔如果遇到多个数据库使用相同主键生成算法的情况（如顺序整数或其他简单模式），您可能需要将此技术与另一种技术（如强制错误消息）结合使用。

## 小结

多年来，就源代码而言，第一方应用代码是最常见的攻击向量。但如今，由于现代 Web 应用对第三方和开源集成的依赖，这一情况正在改变。

深入了解目标的第三方集成可能会引导您找到应用中易于利用的安全漏洞。通常，这些漏洞对于应用所有者来说也很难检测。除此之外，了解第三方依赖在您自己的代码库中的使用方式，可以让您减轻因粗糙的集成技术或与安全性较低的库集成（当有更安全选项时）而带来的风险。

总之，由于当今大多数应用底层运行着大量代码，第三方集成几乎是强制性的。从头开始构建一个完整的全栈 Web 应用将是一项极其艰巨的任务。因此，理解用于查找和评估应用中依赖关系的技术正成为任何从事安全行业人士必备的技能。
