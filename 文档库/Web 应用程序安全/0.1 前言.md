欢迎阅读《Web 应用程序安全：现代 Web 应用程序的利用与防御对策》。在本前言中，我们将讨论本书的内容、适合的读者群体，以及阅读后续技术章节所需具备的技能。阅读前言有助于您判断本书是否适合您。

此外，本书进行了大量更新，以纳入近年来的新技术。我认为必须增加针对 Web 应用中新兴但常见的技术形式（例如 GraphQL 和 NoSQL 数据库）的攻击与防御示例代码。

第二版包含了大量新的安全内容，覆盖了最新、最流行的 Web 应用技术。同时，每章都加入了更深入的内容，并吸收了数十乃至数百条读者和编辑的建议与请求。

希望您会觉得本书结构合理、阅读愉快，并且在读完本书后，能够获得新的知识和视角，从而提升您的信息安全技能。

## 先修知识与学习目标

本书不仅能帮助您学习如何防御 Web 应用免受黑客攻击，还将带您逐步了解黑客调查并入侵 Web 应用所采取的方法。

在本书中，我们将讨论当今黑客用于入侵企业、政府乃至个人爱好者所托管的 Web 应用的许多技术。在充分研究上述技术之后，我们将开始讨论如何保护 Web 应用免受这些黑客的攻击。

在此过程中，您将发现关于应用架构的全新思考方式。您还将学习如何将安全最佳实践整合到工程组织中。最后，我们将评估多种防御当今 Web 应用最常见、最危险攻击类型的技术。

完成本书学习后，您将具备对无法获取代码级访问权限的应用执行信息收集（recon）技术的必要知识。您还能够识别 Web 应用中的威胁向量和漏洞，并构造用于破坏应用数据、中断执行流程或干扰 Web 应用预期功能的载荷。

借助这些技能，以及从最后一节“防御 Web 应用”中获得的知识，您将能够识别 Web 应用代码库中的风险区域，并理解如何编写代码来防御那些可能使您的应用及其用户面临风险的攻击。

本书内容难度循序渐进，因此如果您选择跳过某些章节并发现自己缺少必要的先修信息，只需回退几章补上即可。本前言中未定义为先修知识的任何主题，在书中出现前都会事先说明。

## 为什么示例使用 JavaScript？

本书中的大多数代码示例使用 JavaScript 编写，这在与软件安全相关的书籍中并不常见。您可能会好奇我为何在策划本书时做出这一选择。

所有优秀的安全工程师都必须理解 JavaScript，因为目前它是唯一可以在 Web 浏览器客户端内运行的编程语言。因此，鉴于客户端示例必须用 JavaScript，我决定将大多数服务端示例也用 JavaScript 呈现。

幸运的是，编程语言大多是相通的，因此如果您能从概念上理解内容，就应该能够将任何服务端攻击或缓解措施应用于其他服务端编程语言。

## 为什么教授概念而非工具？

本书以相当底层的方式解释进攻性和防御性网络安全技术，以揭示流行工具的内部工作原理。例如，在本书中，您将自行开发跨站脚本（XSS）载荷并查找 XSS 接收器（sink），而不是使用像 XSS-Sniper 这样的自动化 XSS 发现工具。

如今，许多安全工程师和渗透测试人员并不了解他们日常使用的工具底层发生了什么。其弊端在于，对工具的依赖会导致这些工程师和测试人员无法利用高级攻击或缓解措施。此外，应用安全行业中的工具尤其短暂易逝——它们经常被替换或过时。

通过教授底层概念而非依赖付费工具，任何阅读本书的人都应该能够从一个工具切换到另一个工具，或部署自定义工具、攻击或缓解措施。如果本书专注于特定工具，就无法做到这一点。

## 建议背景

本书的受众相当广泛。它面向具有中级软件工程背景的任何人编写和构建。什么是“中级软件工程背景”？答案因人而异。

对于技术能力很强的人来说，本书可能实际上只需要初级软件工程背景。换句话说，具有先前 Web 开发和/或脚本编写经验（如果足够）的系统管理员，也可以合理阅读本书并理解所有示例。尽管如此，本书包含需要客户端和服务端编码知识的示例。只了解其中一方不足以深入理解这些示例。

本书还讨论了基于 HTTP 的基本客户端/服务端网络通信。此外，在后几章中，当探讨如何在降低安全风险的情况下将内部软件与第三方软件集成时，会涉及软件架构的讨论。

由于本书涵盖的主题众多，我将成功完成本书所需的技能水平定义为“中级”而非“初级”。本书不适合那些没有编写生产级软件应用程序任何经验或知识的人。

### 最低所需技能

在本书中，“中级软件工程背景”意味着以下能力：

- 您至少能用一种编程语言编写基本的 CRUD（创建、读取、更新、删除）程序。
- 您能编写在服务器上运行的代码（例如后端代码）。
- 您至少能编写一些在浏览器中运行的代码（前端代码，通常是 JavaScript）。
- 您了解 HTTP，并能用某种语言或框架发起或至少读懂通过 HTTP 的 GET/POST 调用。
- 您能编写或至少阅读并理解同时使用服务端和客户端代码，并通过 HTTP 在两者之间通信的应用程序。
- 您熟悉至少一种主流数据库（MySQL、MongoDB 等）。

这些技能是成功跟随本书示例的最低标准。超出这些要点的任何经验都是加分项，将使您更容易吸收本书内容并从中获得教育价值。

尽管为简洁起见，本书中的大多数代码示例用 JavaScript 编写（以便客户端和服务端代码使用同一种语言），但大多数示例可以轻松地应用于其他语言。

我已尽力组织本书主题，使其难度以可维持的节奏递增。同时，我也尽量在解释中做到详尽。这意味着每当我介绍新技术时，都会先简要介绍其背景和基本原理。

### 软件工程师与 Web 应用开发者

本书的主要读者是早期至中期职业的软件工程师或 Web 应用开发者。理想情况下，您有兴趣深入了解黑客使用的进攻技术或安全工程师用于防御黑客的防御技术。

通常，“软件工程师”和“Web 应用开发者”这两个头衔可以互换，这可能会导致一些混淆。在本书中我会同时使用这两个术语。让我们先从一些澄清开始。

**软件工程师**  
当我使用“软件工程师”一词时，我指的是能够编写在多种平台上运行的软件的通才。

软件工程师将从本书中受益良多。首先，本书中的许多知识很容易转移到不在 Web 上运行的软件，也适用于其他类型的网络应用，首先想到的是原生移动应用。

本书中讨论的几种利用方式利用了服务端集成，涉及 Web 应用与其他软件组件之间的通信。因此，可以认为任何与 Web 应用交互的软件（数据库、CRM、会计、日志工具等）都是潜在威胁向量。

**Web 应用开发者**  
在本书中，Web 应用开发者是指高度专注于编写在 Web 上运行的软件的人。他们通常进一步分为前端、后端和全栈开发者。

历史上，许多针对 Web 应用的攻击都针对服务端漏洞。因此，我认为本书对后端或全栈开发者的用处是明确的。我也认为本书对其他类型的 Web 应用开发者有价值，包括那些不编写服务器上运行的代码，而是编写在浏览器中运行代码的人（前端/JavaScript 开发者）。

正如我将解释的，当今黑客利用 Web 应用的许多方式都源于在浏览器中运行的恶意代码。有些黑客甚至利用浏览器 DOM 或 CSS 样式表来攻击应用的用户。对于不编写服务端代码的前端开发者来说，了解其代码可能暴露的安全风险以及如何缓解这些风险非常重要。

## 本书如何组织？

您很快会发现本书的结构与大多数其他技术书籍有很大不同。这是有意为之。本书特意按照攻击（进攻）和安全（防御）章节接近 1:1 的比例构建。

我们的旅程从一点历史课和对过去技术、工具和利用方式的探索开始。然后我们进入主要主题：现代 Web 应用程序的利用与防御对策（即本书副标题）。

主要内容分为三大部分，每部分包含许多独立章节，涵盖广泛的主题。理想情况下，您应该线性地阅读本书，从第一页一直到最后一页。如前所述，按顺序阅读本书将提供最佳学习效果，但也可以将其用作黑客攻击参考或安全工程参考，分别关注前半部分或后半部分。

现在您应该已经了解如何导航本书，让我们概述三个主要部分，以理解各部分的重要性。

### 信息收集（Recon）

本书的第一部分是“信息收集（Recon）”，我们在此评估在不必然尝试入侵 Web 应用的情况下获取其信息的方法。在“信息收集”中，我们讨论了许多重要的技术和概念，如果您想成为黑客，掌握这些内容是必不可少的。这些主题对于任何希望锁定现有应用的人来说也很重要，因为许多这些技术所暴露的信息可以通过适当的规划来缓解。

我有幸与我认为是世界上一些最好的渗透测试人员和漏洞赏金猎人共事过。通过与他们的交流以及对他们工作方式的分析，我意识到这个主题比许多其他书籍所描述的要重要得多。对于世界上许多顶级漏洞赏金猎人来说，专家级的信息收集能力是区分“伟大”黑客和“优秀”黑客的关键。换句话说，拥有快车（在此例中，也许知道如何构建利用）是一回事，但若不知道通往终点的最有效路线，您可能赢不了比赛。一辆较慢的车如果选择了更高效的路径，也能比快车更早到达终点。

如果基于幻想的类比更贴近您的感受，您可以把信息收集技能想象成角色扮演游戏中的盗贼。在我们的例子中，盗贼的工作不是造成大量伤害，而是先行侦察并带回情报。这个角色帮助确定射击目标，并找出哪些战斗能带来最大收益。

最后一点尤其有价值，因为很可能许多类型的攻击都会针对防御良好的目标被记录。这意味着您可能只有一次机会在某个软件漏洞被发现和修补之前加以利用。我们可以安全地得出结论，信息收集的第二个用途是弄清楚如何优先处理您的利用方式。

如果您对渗透测试或漏洞赏金猎人的职业感兴趣，第一部分对您至关重要。这主要是因为漏洞赏金世界中的测试以“黑盒”方式进行，渗透测试也在一定程度上如此。“黑盒”测试是一种测试风格，测试者对被测试应用的结构和代码一无所知，因此必须通过仔细分析和调查来构建自己对应用的理解。

### 攻击（Offense）

第二部分是“攻击（Offense）”。在此，本书的重点从信息收集和数据收集转向分析代码和网络请求。然后，借助这些知识，我们将尝试利用编写不安全或配置不当的 Web 应用。

本部分中的若干章节解释了现实世界中恶意黑帽黑客使用的实际黑客技术。¹ **务必注意，如果您要测试本书中的技术，只能针对您自己拥有或已获得明确书面许可进行利用测试的应用进行测试。** 不当使用本书中介绍的黑客技术可能会导致罚款、监禁等后果，具体取决于您所在国家关于黑客活动的法律。

在第二部分中，我们学习如何构建和部署利用方式。这些利用旨在窃取数据或强制改变应用的行为。本部分建立在第一部分“信息收集”的知识之上。利用先前获得的信息收集技能与新获得的黑客技能，我们将开始接管并攻击演示 Web 应用。

第二部分按利用类型逐章组织。每章详细解释一种不同类型的利用。这些章节从解释利用本身开始，以便您了解其工作机制。然后我们讨论如何搜索可应用此利用的漏洞。最后，我们针对所利用的演示应用构造特定的载荷。然后部署载荷并观察结果。

XSS 是我们深入研究的首个利用之一，它是一种适用于众多 Web 应用的攻击类型，但也可应用于其他应用（例如移动应用、Flash/ActionScript 游戏等）。这种特定攻击涉及在您自己的机器上编写一些恶意代码，然后利用应用中薄弱过滤机制，使您的脚本能在另一用户的机器上执行。

当我们讨论像 XSS 攻击这样的利用时，我们将从一个易受攻击的应用开始。这个演示应用将直接明了，理想情况下只有几段代码。在此基础上，我们将编写一段要作为载荷注入演示应用的代码，然后该载荷将利用另一端的假设用户。

听起来很简单，不是吗？实际上就应该如此。没有任何防御措施时，大多数软件系统都容易被攻破。因此，对于像 XSS 这样有许多防御措施的利用，我们将逐步深入探讨编写和部署攻击的细节。

我们将首先尝试突破常规防御，最终转向绕过更高级的防御机制。记住，仅仅因为有人为他们的代码库建了一堵墙，并不意味着您不能翻越它或从底下穿过。这正是我们将发挥创造力并找到一些独特而有趣解决方案的地方。

第二部分非常重要，因为理解黑客的思维方式对于构建安全代码库往往至关重要。对于任何对黑客、渗透测试或漏洞赏金感兴趣的人来说，这尤其重要。

### 防御（Defense）

本书的第三部分也是最后一部分“防御（Defense）”是关于保护您自己的代码免受黑客攻击。在第三部分中，我们将回顾第二部分中介绍的每种利用类型，并以完全相反的视角重新审视它们。在此，我们不集中于攻破软件系统，而是尝试预防或降低黑客攻破我们系统的可能性。

在第三部分中，您将学习如何防御第二部分中的特定利用，此外还会学习能够保护您的代码库免受多种攻击的通用防护措施。这些通用防护措施范围从“默认安全”的工程方法到可由工程团队使用测试和其他简单自动化工具（如 linter）轻松执行的安全编码最佳实践。

除了学习如何编写更安全的代码外，您还将学习一些越来越有价值的技巧，用于当场抓获黑客并改善您所在组织对软件安全的态度。

第三部分中的大多数章节在结构上有些类似于第二部分中的黑客章节。我们从概述开始，介绍开始为特定类型攻击准备防御所需的技术和技能。

最初，我们将准备基础级别的防御，这应有助于缓解攻击，但可能无法始终抵御最顽固的黑客。最后，我们将改进防御，使其能够阻止大多数（即使不是全部）黑客攻击。

在此，第三部分的结构开始与第二部分不同，因为我们将讨论改进应用安全所带来的权衡。总的来说，所有提升安全的措施都会在安全之外产生某种权衡。也许不是您的职责去建议在何种风险水平下以产品为代价进行接受，但您应该意识到所做出的权衡。

通常，这些权衡以应用性能为代价。您为读取和清理数据付出的努力越多，在应用标准功能之外执行的操作就越多。因此，安全功能通常比不安全功能需要更多的计算资源。

随着更多操作也带来更多代码，这意味着更多的维护、测试和工程时间。这种开发开销也常常体现为日志记录或监控开销。最后，一些安全预防措施会以降低可用性为代价。

一个非常简单的比较安全收益与其在可用性和性能方面成本的过程示例是登录表单。如果在尝试登录时向用户显示无效用户名的错误消息，黑客就更容易暴力破解用户名：密码组合。这是因为黑客不再需要查找活跃登录用户名列表，因为应用会确认用户账户是否存在。黑客只需成功暴力破解几个用户名，这些可以被确认并记录下来供以后入侵尝试使用。

接下来，黑客只需暴力破解密码，而不是用户名：密码组合，这意味着数学复杂度显著降低，所需时间和资源大大减少。

此外，如果应用使用电子邮件和密码方案登录，而不是用户名和密码方案，那么我们还有另一个问题。黑客可以利用此登录表单找到有效的电子邮件地址，这些地址可以出售用于营销或垃圾邮件目的。即使采取了预防暴力破解的措施，精心构造的输入（例如 first.last@company.com, firstlast@company.com, firstl@company.com）也能让黑客逆向工程公司电子邮件账户的命名模式，并精确定位高管的有效账户用于销售或具有重要访问权限的个人进行钓鱼攻击。

因此，通常认为向用户提供更通用的错误消息是最佳实践。当然，这种改变与用户体验相冲突，因为更具体的错误消息无疑对应用的可用性更理想。这是一个很好的例子，展示了为改进应用安全而可以做出的权衡，但代价是降低可用性。这应该让您对本书第三部分中讨论的权衡类型有所了解。

本书的这一部分对于任何希望提升技能的安全工程师，或任何考虑转向安全工程角色的软件工程师来说都极为重要。这里提供的信息将有助于设计和编写更安全的应用程序。

与第二部分一样，了解如何改进应用的安全性对于任何类型的黑客来说都是一项宝贵的资产。这是因为虽然常规防御通常可以轻易绕过，但更复杂的防御需要更深入的理解和知识才能绕过。这进一步证明了我建议从头到尾阅读本书的理由。

尽管根据您的目标，本书的某些部分可能比其他部分提供更有价值的教育，但我认为不会有任何部分被浪费。这种交叉训练尤其有价值，因为本书的每一部分都只是同一谜题的不同视角。

## 语言与术语

您现在应该已经清楚，本书旨在教您一些非常有价值但也非常罕见和特殊的技能。虽然这些技能越来越有价值，并能极大提升您在就业市场上的竞争力，但它们也相当难学，需要专注、天赋以及掌握一种全新的思维模式来审视 Web 应用。

为了成功传达这些新技能，我们需要建立一些共同语言。这对于避免混淆，并帮助您以一种在安全和工程组织中一致的方式表达您的新想法很重要。

每当我引入新术语或短语时，我都会尽力解释它。特别是处理缩写词时，我会先拼写全称再单独使用缩写。您之前看到我拼写了 XSS。除此之外，我已尽力确定哪些术语和短语可能需要解释。我将它们收集并组织到下表（表 P-1 至 P-3）中。如果您遇到不完全理解的术语或短语，请返回本前言（为其添加书签！）查看是否在此列出。如果没有，请随时给我的编辑发邮件，也许我们可以在下一版中收录它。

**表 P-1. 职业角色**

| 职业角色 | 描述 |
| :--- | :--- |
| 黑客（Hacker） | 入侵系统的人，通常为了窃取数据或使系统按开发者原意之外的方式运行。 |
| 白帽（White hat） | 有时称为“道德黑客”——利用黑客技术帮助组织改进安全的人。 |
| 黑帽（Black hat） | 典型黑客——利用黑客技术入侵系统以谋利、制造混乱或满足自身目标和兴趣的人。 |
| 灰帽（Grey hat） | 介于白帽和黑帽之间的黑客；偶尔会违反法律，例如未经许可尝试入侵应用，但通常出于发现或认可而非谋利或制造混乱。 |
| 渗透测试员（Penetration tester） | 受雇以类似黑客方式入侵系统的人。与黑客不同，渗透测试员受雇报告应用软件中的错误和疏忽，以便拥有软件的公司可以在被恶意黑客入侵前修复它。 |
| 漏洞赏金猎人（Bug bounty hunter） | 自由职业的渗透测试员。通常大公司会创建“负责任披露计划”，为报告安全漏洞提供现金奖励。一些漏洞赏金猎人全职从事此工作，但通常他们是全职专业人士，在业余时间参与以赚取额外收入。 |
| 应用安全工程师（Application security engineer） | 有时称为“产品安全工程师”——软件工程师，其职责是评估和改进组织代码库和应用架构的安全性。 |
| 软件安全工程师（Software security engineer） | 软件工程师，其职责是开发与安全相关的产品，但不一定负责评估整个组织的安全性。 |
| 管理员（Admin） | 有时称为“系统管理员”或“sys admin”。负责维护 Web 服务器或 Web 应用配置和正常运行时间的技术人员。 |
| Scrum Master | 工程组织中的领导职位，负责帮助工程团队规划和执行开发工作。 |
| 安全捍卫者（Security champion） | 软件工程师，不属于安全组织，也不负责安全工作，但有兴趣改进组织代码的安全性。 |

**表 P-2. 术语**

| 术语 | 描述 |
| :--- | :--- |
| 漏洞（Vulnerability） | 软件系统中的错误，通常源于工程疏忽或连接多个模块时的意外功能。此类错误允许黑客对软件系统执行非预期操作。 |
| 威胁向量或攻击向量（Threat vector / Attack vector） | 应用功能的一个子集，黑客认为其编写不安全，因此可能包含漏洞，是良好的攻击目标。 |
| 攻击面（Attack surface） | 应用中漏洞的列表，黑客在确定如何最佳攻击软件系统时会构建它。 |
| 利用（Exploit） | 通常是一段代码或一组命令，可用于利用漏洞。 |
| 载荷（Payload） | 已格式化为可通过网络发送到服务器以利用漏洞的利用方式。通常这只是将利用打包成适当的格式以通过网络发送。 |
| 红队（Red team） | 通常由渗透测试员、网络安全工程师和软件安全工程师组成的团队。该团队试图入侵公司软件，以评估公司抵御实际黑客的能力。 |
| 蓝队（Blue team） | 通常由软件安全工程师和网络安全工程师组成的团队。该团队尝试改进公司软件安全，通常利用红队的反馈来驱动优先级。 |
| 紫队（Purple team） | 同时承担红队和蓝队职责的组合团队。通用安全团队而非专业团队；由于技能要求广泛，通常更难正确配备人员。 |
| 网站（Website） | 可通过互联网访问的一系列信息文档，通常通过 HTTP 协议。 |
| Web 应用（Web application） | 通过互联网交付并在浏览器内运行、类似于桌面应用的应用。它们与传统网站不同，具有多层权限，在数据库中存储用户输入，并通常允许用户互相分享内容。 |
| 混合应用（Hybrid application） | 基于 Web 技术构建的移动应用。通常使用另一个库（如 Apache 的 Cordova）以将原生功能与上层的 Web 应用共享。 |

**表 P-3. 缩写词**

| 缩写 | 描述 |
| :--- | :--- |
| API | 应用程序编程接口——由一个代码模块暴露给其他代码消费和使用的一组函数。本书中通常指浏览器可在服务器上调用的、通过 HTTP 暴露的函数。也可用于指本地通信的模块，包括同一软件包中的不同模块。 |
| CSRF | 跨站请求伪造——一种攻击，黑客能够利用特权用户的权限对服务器发起请求。 |
| CSS | 层叠样式表——通常与 HTML 结合使用以创建视觉上吸引人且正确对齐的 UI 的样式语言。 |
| DDoS | 分布式拒绝服务——由多台计算机同时大规模执行的 DoS 攻击，以数量压倒服务器；单台计算机通常无法造成如此大的破坏。 |
| DOM | 文档对象模型——随每个 Web 浏览器提供的 API。包含组织和管理页面中 HTML 所需的所有功能，以及管理历史记录、Cookie、URL 和其他常见浏览器功能的 API。 |
| DoS | 拒绝服务——一种攻击，重点不是窃取数据，而是请求大量服务器或客户端资源，以致应用用户体验恶化或应用无法运行。 |
| HTML | 超文本标记语言——一种在 Web 上随 CSS 和 JavaScript 一起使用的模板语言。 |
| HTTP | 超文本传输协议——在 Web 应用或网站中客户端与服务器通信最常用的网络协议。 |
| HTTPS | 超文本传输协议安全——使用 HTTP over TLS 或 HTTP over SSL 加密的 HTTP 流量。 |
| JSON | JavaScript 对象表示法——一种以轻量、易于人阅读且易于机器读取的方式存储层次化数据的规范。在现代 Web 应用中，常用于浏览器和 Web 服务器之间的通信。 |
| OOP | 面向对象编程——一种围绕对象和数据结构而非功能或逻辑组织代码的编程模型。 |
| OSS | 开源软件——可自由使用和修改的软件。通常按 MIT、Apache、GNU 或 BSD 等许可证发布。 |
| REST | 表述性状态传递——一种构建无状态 API 的特定架构，将 API 端点定义为资源而非功能单元。REST 允许多种数据格式，但通常使用 JSON。 |
| RTC | 实时通信——一种较新的网络协议，允许浏览器之间以及浏览器与 Web 服务器之间通信。 |
| SOAP | 简单对象访问协议——一种用于函数驱动 API 的协议，需要严格编写的模式。仅支持 XML 作为数据格式。 |
| SOP | 同源策略——浏览器强制实施的策略，防止一个源的内容在另一个源中加载。 |
| SPA | 单页应用——也称为“单页 Web 应用”（SPWA）。指互联网上功能类似桌面应用的网站，管理自己的 UI 和状态，而非使用浏览器提供的默认功能。 |
| SSDL | 安全软件开发生命周期——也称为 SDLC/SDL。一种常见框架，使软件工程师和安全工程师能协作编写更安全的代码。 |
| SSL | 安全套接层——一种旨在保护传输中（网络上的）信息安全的加密协议，特别用于 HTTP。 |
| TLS | 传输层安全——一种旨在保护传输中（网络上的）信息安全的加密协议，通常用于 HTTP。该协议取代了现已弃用的 SSL。 |
| VCS | 版本控制系统——一种用于管理代码库历史增删记录的特殊软件。有时还包括依赖管理和协作功能。 |
| XML | 可扩展标记语言——一种存储层次化数据的规范，遵循严格规则集。比 JSON 更重但更可配置。 |
| XSS | 跨站脚本——一种攻击类型，涉及迫使另一客户端（通常是浏览器）运行黑客编写的代码。 |
| XXE | XML 外部实体——一种依赖未正确配置的 XML 解析器来窃取 Web 服务器本地文件或包含来自另一 Web 服务器恶意文件的攻击。 |

## 小结

这是一本多方面的书，旨在惠及具有进攻性和防御性安全兴趣的人。任何具有足够 Web 编程背景（客户端和服务端）的开发人员或管理员都能轻松访问和使用。

《Web 应用程序安全》带您了解优秀黑客和漏洞赏金猎人用于入侵应用的许多技术，然后教您可在自己软件中实施以抵御此类黑客的技术和流程。

本书可以通读，也可以作为特定信息收集技术、攻击和防御攻击的按需参考。最终，它的目标是帮助您以实用、动手且逻辑递进的方式提高 Web 应用程序安全能力，无需重要的安全先验经验。

我真诚希望编写本书所投入的数百小时能对您有所裨益，并且您能从中获得一些有趣的学习内容。欢迎您随时就未来版本提出反馈或建议。
