# 第 23 章：安全用户体验

安全 Web 应用中一个经常被遗忘的组成部分是用户界面（UI）。UI 是最终用户与任何 Web 应用进行交互的标准方式。高度专业化的 Web 应用可能允许用户通过 CLI、REST API，甚至通过 XML 或 JSON 文件进行交互。但大多数应用和最终用户更喜欢通过用户界面进行交互。

在 Web 上，由于浏览器计算模型的限制，UI 被限制在一小套技术中。通常这些界面是 HTML、CSS 和 JavaScript，但尽管存在安全和性能限制，也可能使用插件或小程序（例如 Java）。

在本章中，我们不会关注驱动 UI 的技术（例如 JavaScript、CSS、HTML）。相反，我们将关注这些技术输出的应用的设计、体验和可用性，最终用户通过它进行交互。话不多说，让我们考虑一些开发者在生成 UI 时常见的常见安全错误，然后评估这些安全漏洞的潜在解决方案。

## 信息泄露与枚举

信息泄露漏洞和枚举漏洞共享一组共同的识别特征，并且（通常）源于需要使用有用数据来驱动 UI 的需求。然而，通过智能的用户体验设计，这两种漏洞都可以避免或至少得到缓解。

### 信息泄露

在设计 Web 应用的 UI 时，首先要考虑也是最需要考虑的事情是您打算向最终用户提供多少信息。乍一看，这似乎是一个容易解决的问题，但进一步考虑后，您可能会发现 UI 的主要目的是向最终用户提供信息。事实上，对于最终用户来说，如果界面提供更多信息而不是更少信息，通常更好。

从这个角度来看，我们面临的挑战是确定应该向用户提供多少信息——以及应该对最终用户隐藏哪些信息。这意味着可能需要做出权衡，即最终用户的体验可能会为了更好的信息安全而有所牺牲。

有几个经典且简单的例子，说明额外的信息披露如何给您的 Web 应用带来安全风险——尽管这样做是为了改善用户体验。让我们考虑一个场景，您是一名安全工程师，与几位才华横溢的 UX 设计师一起在 Web 应用上工作。

您团队中的设计师（理所应当地）希望通过向最终用户提供详细的错误消息来改善用户体验，以便最终用户能够了解 Web 应用中的表单提交失败时发生了哪种类型的问题。

实现这样的错误消息有很多方式。让我们考虑以下可能的实现：

- 将服务器的错误消息和错误代码直接回显给用户
- 如果消息在预定义的允许列表中，则将服务器的错误消息和错误代码回显给用户
- 在客户端代码中确定一个通用错误消息，同时提供服务器的错误代码
- 仅提供服务器错误代码
- 既不提供错误消息也不提供服务器的错误代码

有趣的是，事实证明（从技术上讲）既不提供错误消息也不提供服务器错误代码是最安全的选择——尽管由于可用性的权衡，这可能不是正确的选择。

这是一个需要权衡评估的案例。开发者绝不应泄露有关应用源代码的信息，以免被指纹识别，但有关服务器状态的信息可能对用户有价值。

当向最终用户披露有关服务器状态的信息时，重要的是确保这些信息以最小化潜在风险的方式进行过滤和格式化。例如，当查询失败时，不要直接从数据库或中间件回显错误消息，而是创建一份通用语气下的常见失败用例允许列表，并返回最接近的匹配项。

考虑以下错误消息：“无法查询用户 Jonathan Smith，因为提供的地址 905 N. Main St. 与当前文件不匹配。”

这条错误消息可以用一条不会泄露地址的通用消息替换，从而降低意外信息泄露的几率。改进后的通用错误消息可能是：“由于地址不匹配，无法查询该用户。”

虽然比简单返回“发生错误”向客户端提供了更多信息，但采用允许列表通用错误方法以较低的信息泄露风险提供了有关原因的信息。

请记住，随着数据库和中间件的升级，服务端错误消息会随着时间变化。因此，如果没有一个强大的预定义错误消息允许列表，现在可能不会发生信息泄露，但将来可能会发生。

在将 HTTP 错误码连同错误消息一起返回给客户端时，最佳实践通常是使用通用状态码，这样就不容易对您的 Web 服务器进行侦察。通用错误码“400 - Bad Request”通常被选中，因为它比更详细和专用的错误码提供的数据更少。

然而，如果您有点幽默感，您可以选择返回“418 - I'm a teapot”，这是一个未使用的错误码，在 1998 年作为愚人节玩笑被纳入 HTTP 规范（但大多数 Web 服务器仍然支持）。使用像“418 - I'm a teapot”这样的错误码的好处是，因为它没有实际的官方用途，所以不太可能干扰您的开发者使用的任何第三方工具。

在某些情况下，非通用错误码可能更受欢迎，例如，如果您的 API 被多个期望获取网络调用状态详细信息的客户端使用。在这种情况下，提供更专用的错误码是可以接受的，但应逐个端点地评估所提供的错误码是否可能泄露任何可能被利用的应用状态信息。

### 枚举

当多个结构相似的查询导致信息泄露，或者这些查询的总和（或组合）可用于推断客户端本不应有权访问的信息时，就会发生枚举漏洞。有许多枚举漏洞导致信息以非预期方式泄露的例子。

例如，如果 API `GET /user/:id` 允许查询单个用户，并且数据库中用户的 `:id` 列是一个可迭代的整数（例如 1、2、3 等），那么通过连续递增的数字重复请求 `GET /user/:id` 端点，攻击者就可以推断出系统内的用户总数。这可能不是高风险信息，但它仍然是本不应被披露的信息。

一个更严重的枚举漏洞示例如下：

1. 黑客发现了一个本不应暴露在互联网上的管理员界面。这个管理员界面用于 MegaBank，允许 MegaBank 的特定用户子集远程管理用户账户和余额。
2. 黑客无法攻破认证机制，因此选择尝试找到有效的用户名:密码组合以便以员工身份登录。
3. 黑客注意到某些用户名:密码认证尝试导致错误“密码错误”，而其他则导致错误“用户不存在”。
4. 黑客收集了其他网站的数据泄露列表，假设 MegaBank 的一些员工在这些泄露的网站上有账户。泄露数据包含用户名和电子邮件地址。泄露的网站示例包括爱好论坛和小型电子商务网站。
5. 黑客找到 `@megabank.com` 的员工电子邮件地址，并将相关用户名输入到管理员 API 中，生成一个电子表格，列出那些在认证尝试中返回“密码错误”的用户（这是枚举步骤）。
6. 既然黑客已经确定了哪些 MegaBank 用户名在管理员账户上有效，黑客就可以利用其他技术（数据泄露、社会工程学、暴力破解、字典）来找到有效的密码并登录。

您可能会问为什么这种枚举形式很重要。攻击者不能从一开始就暴力破解组合吗？

事实是，暴力破解、字典攻击和其他入侵用户账户的方法缺乏准确性，容易被检测，有时在数学上也不可行。通过发现并利用枚举漏洞，暴力破解的复杂性显著降低，因为目标不再是用户名:密码组合，而只是已知用户名的有效密码。

此外，如果检测存在风险或速率限制是个问题，攻击者可以利用高度针对性的密码收集方法（例如鱼叉式网络钓鱼），而这些方法不会被检测到。

在应用中避免枚举的最常见机制是设计时满足以下三个标准：

- 当向最终用户提供错误时，以通用方式提供（类似于缓解信息泄露），这样多次调用就不能积累可被恶意使用的数据。例如，OWASP 建议所有认证失败都使用通用错误消息“认证失败”，而不是具体说明是用户名还是密码（或组合）导致了失败。
- 在设计任何可以通过用户交互的数据结构或代码模块时，应避免任何易于猜测的模式或字符序列。例如，将 API 端点命名为 1/2/3/4/5 会被认为是不良实践。同样，基于注册日期创建用户 ID 也会被认为是不良实践。
- 任何可能存在枚举风险的 API 端点都应强制执行速率限制，以降低程序化重复访问向攻击者泄露有价值信息的风险。速率限制应通过查看合法用户平均需要访问的请求数来确定，并在合法使用模型的上限设置限制。

总之，枚举漏洞的缓解方式与信息泄露漏洞的缓解方式有很多相同之处。如果您将枚举视为一种信息泄露形式，即相对无害的小块数据被反复泄露，那么缓解此类攻击就变得非常简单。

## 安全用户体验最佳实践

除了在用户体验层出现的安全风险和漏洞之外，应用中的用户体验层也是教育和引导最终用户以安全方式使用应用的理想场所之一。

维基百科将**暗模式**定义为一种 UI 设计模式，它欺骗用户调用其并非意图调用的功能。这通常会导致用户的安全或隐私降级。

暗模式的鲜为人知的反面是我称之为**光模式**的模式，它温和地引导用户改善其安全和隐私（图 23-1）。光模式可能很有用，因为用户通常对 Web 应用的工作原理了解不够，无法为自己做出最佳的安全选择。

> **图 23-1. 流行的 JavaScript 库 React 在其命名约定中使用光模式，温和地引导开发者避免意外使用包含安全风险的函数**

在 Web UI 中安全配置设置旁边的文档链接并不是光模式。那是文档。对于任何具有相当复杂性的 Web 应用的大样本用户，我们可以假设只有一部分用户会完整阅读文档。这就是光模式变得有用的地方。光模式不是通过将文档添加到启用或禁用安全机制的页面来提示用户理解可选的安全机制，而是可以在应用中任何包含安全或隐私风险的功能被调用时散布。

以基于 Web 的加密货币钱包为例，它支持用户之间的转账。默认情况下，大多数钱包允许任何金额的出站转账。但许多基于 Web 的钱包提供了选择加入的限制配置，可以限制被入侵账户的破坏半径。

安全机制的一个示例是，在包含总价值 100,000 美元的加密货币的钱包中，每天仅允许 1,000 美元价值的货币流出。通过每日上限限制每日流出，即使钱包被入侵，攻击者也需要一百天才能清算用户的所有储蓄。这为用户提供了时间访问其钱包并消除攻击者的访问权限。

这些机制对用户的安全至关重要，但很少被启用。通过在应用的 UI 中添加光模式，启用上述安全功能的用户转化率可能会显著提高。但应如何以及在何处实施具有此目标的光模式呢？

添加此类光模式的最佳方式实际上不是在实施安全控制的设置页面上，而是在调用安全风险的所有功能中。在每笔出站交易中，用户都应被警告安全设置未启用，并且输错金额可能导致不可逆转的财务损失。这样，用户不必考虑攻击者，而是被提供了一个边缘情况，即他们自己的意外误用可能导致货币损失。

在每笔交易中向用户提供此场景的转化率会更高，因为用户实际上正在发送货币，并且直接熟悉此类事故发生的可能性。这比使用攻击者获得其账户访问权限的案例更有效，因为用户可能没有账户接管可能性的衡量标准。

每当调用有风险的操作时，温和地向用户提供对应用风险和安全机制的理解，引导用户在操作和后果之间建立联系。这种模式通常比仅仅向用户提供缺乏关联性的文档（在大多数情况下甚至不会被完整阅读）能导致用户启用安全机制的比率高得多。

请注意，默认安全仍然是一种优于光模式的设计模式，但在默认安全（例如，强制用户默认启用安全机制）不可行的情况下，应使用光模式。

## 小结

总而言之，虽然您会发现的大多数漏洞源于应用代码，但您也应始终关注您设计用户体验的方式。由于 UI 的主要目标是向最终用户呈现数据，因此必须了解数据呈现的方式，以免错误数据被访问。

最后，良好的用户体验可用于引导最终用户以更安全的方式使用应用。对于存在滥用可能性的功能丰富的应用，请确保在用户面临风险时识别并提醒他们——引导他们走向最安全的路径，以便他们能够避免因糟糕的用户体验而导致的常见陷阱。
