# 第 26 章：漏洞发现

在安全架构的代码经过设计、编写和审查之后，应建立一套流程，以确保没有漏洞被遗漏。通常，架构最佳的应用经历的漏洞最少，且风险最低。之后，拥有足够安全代码审查流程的应用比没有此类流程的应用经历的漏洞更少（但仍多于那些采用默认安全架构的应用）。

即使经过安全架构设计和充分审查的应用，偶尔仍会成为漏洞的牺牲品。漏洞可能在审查中被遗漏，或者当应用在不同环境中运行或预期环境升级时，由于意外行为而产生。因此，您需要针对生产代码而非预生产代码的漏洞发现流程。

## 安全自动化

在架构和审查阶段之后，发现漏洞的第一步是自动化阶段。自动化漏洞发现至关重要，但这并非因为它能捕获所有漏洞。相反，自动化（通常）成本低廉、有效且持久。

自动化发现技术在发现代码中可能被架构师和代码审查员遗漏的常规安全缺陷方面非常出色。自动化发现技术不擅长发现特定于应用功能的逻辑漏洞，或需要“链式”组合才能发挥作用的漏洞（多个弱漏洞组合在一起产生强大漏洞）。

安全自动化有几种形式；最常见的是：

- 静态分析
- 动态分析
- 漏洞回归测试

这些自动化形式中的每一种在应用开发生命周期中都有不同的目的和位置，但每一种都是必不可少的，因为它们能捕捉到其他形式无法发现的漏洞类型。

### 静态分析

您应该编写的第一种自动化，也是最常见的一种，是静态分析。静态分析器是查看源代码并评估代码是否存在语法错误和常见错误的脚本。静态分析可以在开发期间在本地进行（linter），也可以按需对源代码仓库进行，或在每次提交/推送到主分支时进行。

存在许多健壮且强大的静态分析工具，例如：

- Checkmarx（大多数主要语言——付费）
- PMD（Java——免费）
- Bandit（Python——免费）
- Brakeman（Ruby——免费）

这些工具中的每一个都可以配置为分析包含文本并代表代码文件的文档的语法。这些工具实际上都不执行代码，因为那会将它们归入下一类，称为动态分析或有时称为运行时分析。静态分析工具应配置为查找[常见的 OWASP 十大漏洞](https://oreil.ly/JRMl1)。

这些工具中的许多都以免费和付费形式存在，适用于主要语言。静态分析工具也可以从头开始编写——但内部构建的工具通常在大规模代码库上表现不佳。

例如，以下漏洞通常可通过静态分析检测：

- **通用 XSS**：查找使用 `innerHTML` 的 DOM 操作。
- **反射型 XSS**：查找从 URL 参数中提取的变量。
- **DOM XSS**：查找特定的 DOM 接收器，如 `setInterval()`。
- **SQL 注入**：查找在查询中使用的用户提供的字符串。
- **CSRF**：查找状态更改的 GET 请求。
- **DoS**：查找编写不当的正则表达式。

静态分析工具的进一步配置还可以帮助您强制执行最佳安全编码实践。例如，您的静态分析工具应拒绝未导入适当授权函数的 API 端点，或消费用户输入但未从单一事实来源验证库中提取的函数。

静态分析对于通用漏洞发现非常强大，但它也可能令人沮丧，因为它会报告许多误报。此外，静态分析在处理动态语言（如 JavaScript）时会遇到困难。静态类型语言（如 Java 或 C#）进行静态分析要容易得多，因为工具理解预期的数据类型，并且数据在穿过函数和类时不能改变类型。

另一方面，动态类型语言进行准确的静态分析要困难得多。JavaScript 就是一个很好的例子，因为 JavaScript 变量（包括函数、类等）是可变对象——它们可以在任何时间点改变。此外，由于没有类型转换，如果不运行时评估，很难在任何时候理解 JavaScript 应用的状态。

总之，静态分析工具非常适合发现常见漏洞和错误配置，尤其是在静态类型编程语言方面。静态分析工具在发现涉及深层应用知识、漏洞链或动态类型语言中的漏洞方面效果不佳。

### 动态分析

静态分析通常在代码执行之前查看代码。而动态分析则在代码执行后查看代码。由于动态分析需要代码执行，因此成本更高且速度显著更慢。在大型应用中，动态分析在具有任何实用性之前需要一个类似生产的环境（服务器、许可证等）。

动态分析在实际漏洞的捕捉方面非常出色，而静态分析会捕捉许多潜在漏洞，但确认它们的方式有限。动态分析在执行代码后分析输出，并将其与描述漏洞和错误配置的模型进行比较。这使得它非常适合测试动态语言，因为它可以看到代码的输出而不是（模糊的）输入和流程。它也非常适合发现由于正常应用操作而产生的副作用漏洞——例如，敏感数据在内存中存储不当或旁道攻击。

动态分析工具适用于许多语言和框架。其中一些示例如下：

- IBM AppScan（付费）
- Veracode（付费）
- Iroh（免费）

由于在类生产环境中运行的复杂性增加，更好的工具通常是付费的或需要大量前期配置。简单的应用可以构建自己的动态分析工具，但对于 CI/CD 级别的完全自动化，它们将需要大量工作和一定的前期成本。

与静态分析工具不同，配置得当的动态分析工具应该具有更少的误报，并能对您的应用提供更深入的洞察。与静态分析工具相比，代价在于维护、成本和性能。

### 漏洞回归测试

对于安全的 Web 应用来说，最后一种必不可少的自动化形式是漏洞回归测试网络。静态分析和动态分析工具很酷，但与回归测试相比，它们难以设置、配置和维护。

漏洞回归测试套件很简单。它的工作方式类似于功能或性能测试套件，但漏洞回归测试会测试已知漏洞是否仍然存在。这可以防止缺陷被重新引入代码库。

您不需要为安全漏洞测试使用特殊的框架。任何能够重现漏洞的测试框架都应适用。图 26-1 显示了 Jest，这是一个适用于 JavaScript 应用的快速、简洁且强大的测试库。Jest 可以很容易地修改为测试安全回归。

> **图 26-1. Jest 测试库**

想象以下漏洞。软件工程师 Steve 在应用中引入了一个新的 API 端点，允许用户从其仪表板 UI 组件中按需升级或降级其会员资格：

```javascript
const currentUser = require('../currentUser');
const modifySubscription = require('../../modifySubscription');
const tiers = ['individual', 'business', 'corporation'];

/*
 * 代表当前已认证用户接收 HTTP GET 请求。
 *
 * 接受参数 `newTier` 并尝试将已认证用户的订阅更新为该层级。
 */
app.get('/changeSubscriptionTier', function(req, res) {
  if (!currentUser.isAuthenticated) { return res.sendStatus(401); }
  if (!req.params.newTier) { return res.sendStatus(400); }
  if (!tiers.includes(req.params.newTier)) { return res.sendStatus(400); }

  modifySubscription(currentUser, req.params.newTier)
    .then(() => {
      return res.sendStatus(200);
    })
    .catch(() => {
      return res.sendStatus(400);
    });
});
```

Steve 的老朋友 Jed 经常批评 Steve 的代码，他意识到他可以使用任何层级作为 `newTier` 参数发起类似 `GET /api/changeSubscriptionTier` 的请求，并通过超链接发送给 Steve。当 Steve 点击此链接时，会以其账户名义发起请求，从而改变其公司应用门户中的订阅状态。

Jed 发现了一个 CSRF 漏洞。幸运的是，虽然 Steve 对 Jed 不断的批评感到恼火，但他意识到此漏洞的危险性，并将其报告给组织进行分诊。经过分诊，解决方案是将请求从 HTTP GET 改为 HTTP POST。

不想在朋友 Jed 面前再次丢脸，Steve 编写了一个漏洞回归测试：

```javascript
const tester = require('tester');
const requester = require('requester');

/*
 * 检查 `changeSubscriptionTier` 端点的 HTTP Options。
 *
 * 如果接受超过一个动词，或动词不等于 'POST'，则失败。
 * 超时或 Options 请求失败则失败。
 */
const testTierChange = function() {
  requester.options('http://app.com/api/changeSubscriptionTier')
    .on('response', function(res) {
      if (!res.headers) {
        return tester.fail();
      } else {
        const verbs = res.headers['Allow'].split(',');
        if (verbs.length > 1) { return tester.fail(); }
        if (verbs[0] !== 'POST') { return tester.fail(); }
      }
    })
    .on('error', function(err) {
      console.error(err);
      return tester.fail();
    })
};
```

这个回归测试看起来像功能测试，事实上它正是！

功能测试和漏洞测试之间的区别不在于框架，而在于编写测试的目的。在本例中，CSRF 错误的解决方案是端点应只接受 HTTP POST 请求。回归测试确保端点 `changeSubscriptionTier` 只接受一个 HTTP 动词，且该动词等于 POST。如果将来有更改引入了该端点的非 POST 版本，或修复被覆盖，则此测试将失败，表明漏洞已回归。

漏洞回归测试很简单。有时它们如此简单，甚至可以在漏洞引入之前编写。这对于那些看似无关紧要的小改动可能产生重大影响的代码很有用。最终，漏洞回归测试是防止已关闭的漏洞重新进入代码库的简单而有效的方法。

测试本身应在提交或推送钩子时尽可能运行（如果测试失败则拒绝提交或推送）。对于更复杂的版本控制环境，定期运行（每日）是次优选择。

## 负责任披露计划

除了具备适当的自动化来捕获漏洞外，您的组织还应有一个定义明确且公开的漏洞披露方式。您的内部测试可能无法覆盖客户的所有潜在用例。因此，您的客户很可能会发现否则不会被报告的漏洞。

不幸的是，一些大型组织曾将用户的漏洞报告转化为针对报告者的诉讼和封口令。由于法律并未明确界定道德研究与恶意利用之间的区别，您的应用中最懂技术的用户很可能不会报告意外发现的漏洞，除非您明确界定了负责任披露的路径。

一个好的负责任披露计划应包括一份清单，列出您的用户可以测试应用安全性而不会产生任何法律风险的方式。除此之外，您的披露计划应定义明确的提交方法和优秀提交的模板。

为了降低在漏洞被修复之前公开暴露的风险，您可以在负责任披露计划中包含一个条款，防止研究人员在最近发现漏洞后公开讨论。通常，负责任披露计划会列出一段时期（数周或数月），在此期间，在漏洞被修复之前，报告者不能对外讨论该漏洞。一个正确实施的漏洞披露计划将进一步降低可利用漏洞被遗留开放的风险，并改善公众对您开发团队致力于安全工作的认可。

## 漏洞赏金计划

虽然负责任披露允许研究人员和最终用户报告在您的 Web 应用中发现的漏洞，但它不提供实际测试您的应用和发现漏洞的激励。过去十年来，软件公司一直在实施漏洞赏金计划，以现金奖励换取最终用户、白帽黑客和安全研究人员正确提交并记录的漏洞报告。

启动漏洞赏金计划曾经是一个困难的过程，需要广泛的法律文件、一个分诊团队，以及专门配置的 Sprint 或 Kanban 流程来检测重复和解决漏洞。如今，存在中介公司来促进漏洞赏金计划的开发和增长。

像 HackerOne 和 Bugcrowd 这样的公司提供易于定制的法律模板和用于提交和分诊的 Web 界面。HackerOne 是 Web 上最流行的漏洞赏金平台之一，帮助小公司建立漏洞赏金计划，并与安全研究人员和道德黑客建立联系（见图 26-2）。

> **图 26-2. HackerOne，一个漏洞赏金平台**

除了发布正式的负责任披露政策外，利用漏洞赏金计划将允许自由职业渗透测试员（漏洞赏金猎人）和最终用户不仅发现漏洞，而且有动力报告它们。

## 第三方渗透测试

除了创建负责任披露系统和通过漏洞赏金计划激励披露之外，第三方渗透测试可以让您深入了解代码库的安全性，这是您通过自己的开发团队无法获得的。第三方渗透测试员类似于漏洞赏金猎人，因为他们与您的组织没有直接关联，但能提供对您 Web 应用安全性的洞察。漏洞赏金猎人（大多数，除了前 1%）是自由职业的渗透测试员。他们想工作时就工作，没有特定的议程要遵守。

另一方面，渗透测试公司可以被指派测试应用的特定部分——并且通常通过法律协议可以安全地获得公司源代码（以获得更准确的测试结果）。理想情况下，签约测试应在发布到生产环境之前针对应用代码库的高风险和新编写区域进行。发布后测试对于代码库的高风险区域以及确保安全机制跨平台保持一致也很有价值。

## 小结

有很多方法可以发现 Web 应用代码库中的漏洞，每种方法都有其优缺点，并在应用生命周期中占据不同的位置。理想情况下，应结合使用其中几种技术，以确保您的组织有最大的机会在严重安全漏洞被组织外部的黑客发现或利用之前捕获并解决它们。

通过将本章所述的漏洞发现技术与适当的自动化相结合，并将其反馈到您的 SSDL 中，您将能够自信地发布生产 Web 应用，而不必过于担心在生产中发现严重安全漏洞。
