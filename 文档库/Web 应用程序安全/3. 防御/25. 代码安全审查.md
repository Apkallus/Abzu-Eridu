# 第 25 章：代码安全审查

在注重安全的组织中，代码审查阶段必须始终在架构阶段之后进行——绝不能在之前。

如今，一些科技公司信奉“快速行动，打破常规”的口号，但这种哲学往往被滥用，并用作忽视适当安全流程的借口。即使在一个快速发展的公司中，在交付代码之前审查应用架构也是必不可少的。尽管从安全角度来看，预先审查整个功能架构是理想的，但在不确定的条件下可能不可行。至少，主要和知名功能应在开发前进行架构设计和审查。当新功能出现时，也应在开发前进行架构设计和安全审查。

审查代码是否存在安全漏洞的适当时机是，在代码提交背后的架构经过适当审查之后。这意味着，在遵循安全开发最佳实践的组织中，代码审查应该是第二步。

这样做有两个好处。第一个也是最明显的好处是安全性，但拥有一个通常从开发团队外部审视代码的额外审查者本身也有其价值。这为开发者提供了一双不带偏见的眼睛，可能会发现其他情况下未知的错误和架构缺陷。

因此，代码安全审查阶段对于应用功能和应用安全都至关重要。在仅进行功能审查的组织中，代码安全审查应作为额外的步骤实施。这样做将大幅减少可能被发布到生产环境中的高影响安全错误的数量。

一般来说，代码安全审查在合并请求（传统上也称为“拉取请求”，但在大多数情况下这个术语不太准确）时进行最有意义。在合并时进行代码安全审查是合理的，因为完整的功能集已经开发完成，所有需要连接的系统都应已集成。这是一个可以在一次会话中审查代码全部范围的时间点。

也可以将代码安全审查与开发过程以更细粒度的方法交织在一起，例如每次提交时，甚至采用结对编程的方式。这两种方法都需要持续、不间断的工作，因为二者都会从某个时间点查看代码，无法覆盖代码的全部范围。然而，对于关键任务的安全功能，这可能是一种明智的方法。

当一个人专注于功能，另一个人专注于安全时，可能能够编写出极具安全意识的特性，而这在合并请求时进行审查是不可能实现的。

您的组织选择何时审查代码中的安全漏洞取决于组织自身，并且必须适应其现有流程。然而，前面的方法可能是将安全代码审查集成到您的开发流程中最实用和最有效的方式。

## 如何开始代码审查

代码安全审查的操作方式应与代码功能审查非常相似。功能审查在几乎所有的开发组织中都是标准流程，这使得代码安全审查的学习曲线大大缩短。

审查代码安全的第一步是将相关分支拉到本地开发机器上。一些组织允许在基于 Web 的编辑器中进行审查（由 GitHub 或 GitLab 提供；参见图 25-1），但这些在线工具不如您可以在本地使用的工具全面。

以下是一个可以从终端执行的常见本地审查流程：

1. 使用 `git checkout main` 切换到 main 分支。
2. 使用 `git pull origin main` 获取并合并最新的 master。
3. 使用 `git checkout <username>/feature` 切换到功能分支。
4. 使用 `git diff origin/main...` 运行与 main 的差异比较。

`git diff` 命令应返回两样东西：

- 在 main 分支和当前分支上不同的文件列表
- 这些文件在 main 分支和当前分支之间的更改列表

这是任何代码功能审查和任何代码安全审查的起点。两者之间的差异在此之后开始。

## 典型漏洞与业务逻辑漏洞

代码功能审查检查代码以确保其满足功能规范并且不包含可用性错误。代码安全审查检查常见漏洞，如 XSS、CSRF、注入等，但更重要的是检查需要深入了解代码目的的逻辑级漏洞，这些漏洞不易被自动化工具或扫描器发现。

为了发现由逻辑错误引起的漏洞，我们首先需要了解功能目标的上下文。这意味着我们需要了解功能的用户、功能的功能以及功能的业务影响。

在这里，我们在本书中讨论漏洞时遇到了一些差异。我们研究的大多数漏洞都是众所周知漏洞的常见原型。但同样可能的是，具有非常特定用例的应用存在无法在旨在进行软件安全通识教育的书中列出的漏洞。

考虑以下关于要集成到 MegaBank 的新社交媒体功能——MegaChat 的上下文：

- 我们正在构建一个社交媒体门户，允许注册用户申请会员资格。
- 会员资格由版主根据对用户在会员资格之前的活动的审查来批准。
- 用户具有有限的功能，但当升级为会员时，他们拥有更多的功能。
- 版主自动获得会员功能以及额外的版主能力。
- 与只能发布文本媒体的用户不同，会员可以上传游戏、视频和艺术作品。
- 我们对会员资格进行限制，因为托管这类媒体成本高昂，并且我们希望减少低质量内容，同时保护自己免受机器账户和只想托管内容的闲散用户的侵害。

由此我们可以得出以下结论：

**用户和角色**

- 用户是 MegaBank 的客户。
- 用户分为三个角色：用户（默认）、会员和版主。
- 每个用户角色具有不同的权限和功能。

**功能功能**

- 用户、会员和版主可以发布文本。
- 会员和版主可以发布视频、游戏和图像。
- 版主可以使用版主功能，包括将用户升级为会员。

**业务影响**

- 托管视频、游戏和图像的成本很高。
- 会员资格伴随着闲散用户（存储/带宽成本）和机器人（存储/带宽成本）的风险。

典型漏洞是用户帖子中的 XSS。业务逻辑漏洞则是一个编码不当的特定 API 端点，允许用户发送带有 `isMember: true` 的载荷以便发布视频，即使该用户尚未被版主授予会员功能。

代码审查是我们寻找典型漏洞并试图找到需要深入应用上下文的业务逻辑漏洞的地方。

## 安全审查从何处入手

理想情况下，您应从应用风险最高的组件开始代码审查。然而，如果您被要求对您在设计上没有发言权的应用进行安全审查，您可能并不总是清楚这些组件是什么。这在咨询或处理现有产品时很常见。

因此，我提出一个框架来简化安全代码审查流程，并帮助您开始进行安全审查。在您对给定应用足够熟悉以开始基于风险评估应用功能之前，可以使用此框架。

想象一个具有两个组件的基本 Web 应用：浏览器中的客户端和与该客户端通信的服务器。当然，我们可以从审查服务端代码开始。事实上，这并没有错。但服务器上可能存在未暴露给客户端的功能。这意味着，如果没有很好地理解面向用户的功能（相对于内部方法等），您的努力可能会意外地集中在低风险代码上，而高风险代码本应优先处理。

这是一个难以理解的概念，但就像在第 21 章关于安全应用架构中一样，我们需要意识到，在理想世界中，每段应用代码都会得到同等的审查。不幸的是，那反映的是理想世界。在现实世界中，往往有截止日期、时间表和其他需要关注的项目。

因此，在实际源代码中，一个好的起点是客户端（浏览器）向服务器发起请求的任何地方。从客户端开始很棒，因为这将开始让您很好地了解您正在处理的攻击面。从那里，您可以了解客户端和服务器之间交换了哪些类型的数据，以及是否使用了多个服务器而不是一个。此外，您可以了解交换的载荷以及这些载荷在服务器上如何被解释。

在评估客户端本身之后，跟随客户端的 API 调用回到服务器。开始评估在 Web 应用中连接客户端和服务器的调用。

完成此操作后，您可能应考虑追踪这些 API 所依赖的辅助方法、依赖项和功能。这意味着评估数据库、日志、上传的文件、转换库以及 API 端点直接或通过辅助库调用的任何其他内容。

接下来，通过检查可能暴露给客户端但未被直接调用的每一块功能来覆盖基础。这可能是为支持即将推出的功能而构建的 API，或者只是意外暴露但本应是内部的功能。

最后，在代码库中的这些主要部分被覆盖之后，将您的时间投入到代码库的其余部分。通过分析业务逻辑并根据您预想此类应用会面临的风险确定优先级，来确定审查路径。

总而言之，在 Web 应用的安全审查中确定要审查哪些代码的有效方法如下：

1. 评估客户端代码以了解业务逻辑，并理解用户将能够使用哪些功能。
2. 利用从客户端审查中获得的知识，开始评估 API 层，特别是您通过客户端审查找到的 API。这样做，您应该能够很好地了解 API 层依赖哪些依赖项来运行。
3. 追踪 API 层中的依赖项，仔细审查数据库、辅助库、日志函数等。这样做，您将接近覆盖大多数面向用户的功能。
4. 利用对客户端链接 API 结构的了解，尝试查找任何可能意外暴露或为未来功能发布而准备的面向公众的 API。在发现它们时进行审查。
5. 继续审查代码库的其余部分。这实际上应该相当容易，因为您已经以有机的方式通读了代码库，而不是试图强行理解应用架构。

这不是进行安全审查的唯一方法，某些具有小众安全要求的特定应用可能需要不同的审查路径。然而，我建议这条路径，因为它将使您以有机的节奏熟悉应用，并允许您优先处理面向用户的功能，同时将潜在的低风险功能留到最后。

随着您对安全代码审查流程以及您发现自己审查的特定应用越来越熟悉，您应该能够修改这组指南，以更好地适应您的应用和您的应用面临的风险。

## 安全编码反模式

代码级别的安全审查与编写代码之前进行的架构审查有一些相似之处。代码审查与架构审查的不同之处在于，它们是实际发现漏洞的理想时机，而如果在架构阶段提出，这些漏洞只是假设性的。

在进行任何安全审查时，有许多反模式需要注意。很多时候，反模式只是仓促实现的解决方案，或者是在没有适当的先验知识的情况下实现的解决方案。无论原因如何，了解如何发现反模式将真正有助于加快您的审查过程。

以下反模式都相当常见，但如果它们进入生产构建，每一个都可能对系统造成严重破坏。

### 黑名单

在安全领域，临时缓解措施通常应被忽略，而应找到永久性解决方案，即使需要更长时间。只有当有一个预先计划的时间表，将设计和实施真正的完整解决方案时，才应实施临时或不完整的解决方案。黑名单是临时或不完整安全解决方案的一个例子。

假设您正在为一个可接受域列表构建服务端过滤机制，您的应用可以与之集成：

```javascript
const blocklist = ['http://www.evil.com', 'http://www.badguys.net'];

/*
 * 确定该域是否允许集成。
 */
const isDomainAccepted = function(domain) {
  return !blocklist.includes(domain);
};
```

这是一个常见错误，因为它看起来像一个解决方案。但即使它当前充当解决方案，它也可以被认为是不完整的（除非考虑到所有域的完美知识，这不太可能）和临时的（即使对所有当前域有完美了解，将来也可能引入更多恶意域）。

换句话说，黑名单仅在您对所有可能的当前和未来输入有完美了解时才能保护您的应用。如果这两者中任何一个无法获得，黑名单将无法提供足够的保护，并且通常可以通过一点努力绕过（在本例中，黑客只需购买另一个域）。

在安全领域，白名单总是更可取的。只需翻转允许集成的方式，这个过程就可以安全得多：

```javascript
const allowlist = ['https://happy-site.com', 'https://www.my-friends.com'];

/*
 * 确定该域是否允许集成。
 */
const isDomainAccepted = function(domain) {
  return allowlist.includes(domain);
};
```

有时，工程师会争辩说白名单会创造困难的产品开发环境，因为白名单随着列表的增长需要持续的手动或自动维护。通过手动努力，这确实可能是一个负担，但手动和自动相结合可以使维护更容易，同时保留大部分安全收益。

在这个例子中，要求集成合作伙伴提交其网站、营业执照等以供审查，然后才被加入白名单，这将使恶意集成极难蒙混过关。即使他们成功了，一旦从白名单中移除，他们也很难再次通过（他们需要一个新的域名和营业执照）。

### 样板代码

另一个需要注意的安全反模式是使用样板或默认框架代码。这是一个大问题，而且容易被忽略，因为框架和库通常需要努力加强安全性，而它们实际上应该开箱即用就具有增强的安全性，并需要放松。

一个典型的例子是 MongoDB 中的配置错误，导致旧版本的 MongoDB 数据库在安装到 Web 服务器上时默认可通过互联网访问。再加上数据库没有强制身份验证要求，导致网络上数万个 MongoDB 数据库被脚本劫持，索要比特币以换取归还。配置文件中的几行代码本可以通过阻止 MongoDB 可被互联网访问（仅本地访问）来解决此问题。

在世界各地使用的大多数主要框架中都发现了类似的问题。以 Ruby on Rails 为例，使用样板 404 页面代码可以轻松暴露您正在使用的 Ruby on Rails 版本。EmberJS 也是如此，它有一个默认的登陆页面，在生产应用中应被移除。

框架为开发者抽象了令人厌烦的困难和例行工作，但如果开发者不理解框架中发生的抽象，则很可能抽象执行不正确且没有适当的安全机制。因此，除非样板代码已经过适当评估和配置，否则避免将任何样板代码发布到生产环境中。

### 默认信任

在构建具有多个功能级别的应用时，所有这些功能都从主机操作系统请求资源，为您的代码实施适当的权限模型至关重要。

想象一个能够生成服务端日志、将文件写入磁盘以及对 SQL 数据库执行更新的应用。在许多实现中，将在服务器上生成一个用户账户，具有日志记录、数据库访问和磁盘访问的权限。应用将在该用户账户下运行所有功能。

然而，这意味着如果发现允许代码执行或改变脚本预期执行方式的漏洞，所有这三个有价值的服务端资源都可能被入侵。

相反，一个安全应用会为日志记录、写入磁盘和执行数据库操作生成彼此独立的权限。安全应用中的每个模块都将在其自己的用户下运行，具有专门配置的权限，只允许特定功能运行所需的内容。通过这样做，一个模块的严重故障不会泄漏到其他模块，并且 SQL 模块中的漏洞不应使黑客能够访问服务器上的文件或日志。

### 客户端/服务端分离

最后一个需要注意的反模式是客户端/服务端耦合反模式。当客户端和服务端应用代码紧密绑定，以至于一方无法在没有另一方的情况下运行时，就会出现这种反模式。这种反模式主要存在于较旧的 Web 应用中，但在今天的单体应用中仍然可以找到。一个由客户端和服务器组成的安全应用应使客户端和服务器独立开发，并且两者应使用预定义的数据格式和网络协议通过网络进行通信。

客户端和服务器代码深度耦合的应用（例如，带有身份验证逻辑的 PHP 模板代码）由于缺乏分离而变得更容易利用。模块不是读取网络请求的结果，而是发回其 HTML 代码，包括任何表单数据（例如，在处理身份验证时）。然后，服务器必须负责解析该 HTML 代码，并确保在 HTML 代码和身份验证逻辑中都不会发生脚本执行或参数篡改。

在完全分离的客户端/服务器应用中，服务器不负责 HTML 数据的结构和内容。相反，服务器拒绝任何发送的 HTML，并且只接受使用预定义数据格式的身份验证载荷。

在分布式应用中，每个模块负责较少独特的安全机制。另一方面，耦合客户端和服务器代码的单体应用必须考虑针对多种语言的安全机制，并考虑到接收的数据可能以多种方式格式化，而不是单一的、预定义的方式。

总之，关注点分离从工程角度和安全角度始终都很重要。适当分离的模块产生更容易管理的安全机制，这些机制不需要重叠，也不需要考虑由于多种数据/脚本类型之间的复杂交互而产生的罕见边缘情况。

## 小结

在审查代码安全时，我们不仅需要考虑寻找常见漏洞（我们将在后续章节中讨论）。我们还需要考虑应用中的反模式，这些模式看起来像解决方案，但在后期会成为问题。代码安全审查也应该是全面的——涵盖所有潜在可能发现漏洞的领域。

在代码审查期间，我们需要考虑应用的具体使用要求，以便我们能够理解可能引入哪些不适合常见预定义漏洞原型的逻辑漏洞。在开始代码审查时，我们应采取一条逻辑路径，使我们能够了解应用的用例，从而开始评估应用中的风险。在更成熟的应用中，高风险区域众所周知，大部分审查工作应集中在这些区域，其余区域按风险降序审查。

最终，将安全审查集成到您的代码审查管道中，如果操作正确，将帮助您降低将漏洞引入代码库的可能性。代码安全审查流程应成为任何现代软件开发管道的一部分，并且在可能的情况下，应由具有安全知识的工程师与产品或功能开发者一起执行。
