# 第 21 章：安全应用架构

保障任何 Web 应用安全的第一步是架构阶段。在构建产品时，由软件工程师和产品经理组成的跨职能团队通常会协作，寻找一种能够高效服务于特定业务目标的技术模型。在软件工程中，架构师的角色是高层级地设计模块，并评估模块间相互通信的最佳方式。这可以扩展到确定数据存储的最佳方式、依赖哪些第三方依赖项、整个代码库应采用何种编程范式等。

与建筑架构师类似，软件架构是一个微妙的过程，承担着大量风险，因为一旦应用构建完成，重新架构和重构的成本很高。安全架构具有与软件或建筑架构类似的风险特征。通常，通过仔细规划和评估，漏洞可以在架构阶段轻松预防。然而，如果规划不足，应用代码必须重新架构和重构——这通常会给企业带来巨大成本。

NIST 根据对流行 Web 应用的研究声称：“在设计阶段移除应用安全漏洞的成本，比在生产阶段移除要低 30 到 60 倍。”因此，这巩固了我们对于架构阶段重要性的任何疑虑。

## 分析功能需求

确保产品或功能以安全方式架构的第一步，是收集该产品或功能预期实现的所有业务需求。业务需求可以在被考虑集成到 Web 应用之前进行风险评估。

> 任何设有独立安全和研发团队的组织，都应确保在开发过程中建立两者之间的沟通渠道。功能不能孤立地进行适当分析，此类分析应包括工程和产品开发方面的利益相关者。

考虑以下业务案例：在清理了代码库中的多个安全漏洞后，MegaBank 决定通过推出自己的商品品牌来利用其新获得的人气。MegaBank 的新商品品牌 MegaMerch 将提供一系列高品质棉质 T 恤、舒适的棉/氨纶运动裤，以及带有 MegaMerch（MM）标志的男女泳装。

为了分销 MegaMerch 新品牌下的商品，MegaBank 希望建立一个电子商务应用，满足以下要求：

- 用户可以创建账户并登录。
- 用户账户包含用户的全名、地址和出生日期。
- 用户可以访问显示商品的商店首页。
- 用户可以搜索特定商品。
- 用户可以保存信用卡和银行账户以备将来使用。

对这些需求的高层分析告诉我们一些重要的信息片段：

- 我们存储凭证。
- 我们存储个人身份信息。
- 与访客相比，用户拥有更高的权限。
- 用户可以搜索现有商品。
- 我们存储财务数据。

这些要点虽然并不异常，但使我们能够初步分析，如果架构不当，此应用可能遇到哪些潜在风险。从该分析中得出的一些风险领域如下：

**认证与授权**  
我们如何处理会话、登录和 Cookie？

**个人数据**  
它的处理方式是否与其他数据不同？法律是否影响我们应如何处理这些数据？

**搜索引擎**  
搜索引擎如何实现？它从主数据库中提取数据作为唯一事实来源，还是使用单独的缓存数据库？

这些风险中的每一个都引发了许多关于实现细节的问题，这为安全工程师协助将应用向更安全的方向发展提供了空间。

## 认证与授权

由于我们存储凭证并向访客和注册用户提供不同的用户体验，我们知道我们同时拥有认证系统和授权系统。这意味着我们必须允许用户登录，并能够在确定允许这些用户执行什么操作时区分不同级别的用户。

此外，由于我们存储凭证并支持登录流程，我们知道将通过网络发送凭证。这些凭证也必须存储在数据库中，否则认证流程将中断。

这意味着我们必须考虑以下风险：

- 我们如何处理传输中的数据？
- 我们如何处理凭证的存储？
- 我们如何处理不同级别的用户授权？

### 安全套接层（SSL）与传输层安全（TLS）

由于我们已识别的风险，需要解决的最重要的架构决策之一是如何处理传输中的数据。传输中的数据是架构审查期间重要的第一步评估，因为它将影响整个 Web 应用中所有数据的流动。

初始的传输中数据要求应是所有通过网络发送的数据在传输过程中都应加密。这降低了中间人攻击的风险，中间人攻击可能窃取我们用户的凭证并代表他们进行购买（因为我们存储了他们的财务数据）。

安全套接层（SSL）和传输层安全（TLS）是当今用于保护传输中数据免受任何网络中恶意窥探的两种主要加密协议。SSL 由 Netscape 在 20 世纪 90 年代中期设计，此后发布了该协议的多个版本。

TLS 于 1999 年由 RFC 2246 定义，并针对 SSL 中的几个架构问题提供了升级的安全性。由于两者之间存在大量架构差异，TLS 无法与旧版本的 SSL 进行互操作。TLS 提供了最严格的安全性，而 SSL 虽然采用率更高，但存在多个降低其作为加密协议完整性的漏洞。图 21-1 展示了一种实现 TLS 的方法示例。

> **图 21-1. Let's Encrypt 是为 TLS 加密提供证书的少数非营利安全机构（SA）之一**

如今，当网站通信通过 SSL 或 TLS 得到适当保护时，所有主流浏览器都会在 URL 地址栏中显示一个锁图标。HTTP 规范提供了“HTTPS”或“HTTP Secure”，这是一个统一资源标识符（URI）方案，要求在允许通过网络发送任何数据之前必须存在 TLS/SSL。支持 HTTPS 的浏览器在发出 HTTPS 请求时，如果 TLS/SSL 连接受到损害，会向最终用户显示警告。

对于 MegaMerch，我们希望确保所有数据在通过网络发送之前都经过加密并兼容 TLS。TLS 的实现方式通常因服务器而异，但每个主要的 Web 服务器软件包都提供了简单的集成，以开始加密 Web 流量。

### 安全凭据

密码安全要求的存在有多种原因，但不幸的是，大多数开发者并不理解什么使密码对黑客安全。创建安全密码与长度和特殊字符的数量关系不大，而是与可以在密码中找到的模式有关。在密码学中，这被称为熵——随机性和不确定性的数量。您需要熵高的密码。

信不信由你，Web 上使用的大多数密码都不是唯一的。当黑客试图暴力破解 Web 应用的登录时，最简单的方法是找到一份最常用密码列表，并使用该列表进行字典攻击。高级字典攻击还将包括常用密码的组合、常用密码结构和常用密码组合。除此之外，经典暴力破解涉及遍历所有可能的组合。

如您所见，保护您的不是密码的长度，而是缺乏可观察的模式以及避免使用常用单词和短语。不幸的是，这很难传达给用户。相反，我们应该通过制定某些要求，使用户难以开发包含许多众所周知模式的密码。

例如，我们可以拒绝前一千个密码列表中的任何密码，并告知用户该密码太常见。我们还应该防止用户使用出生日期、名字、姓氏或地址的任何部分。在 MegaMerch，我们可以要求注册时提供名字、姓氏和出生日期，并防止这些内容被允许出现在用户密码中。

### 凭据哈希

在存储敏感凭证时，我们绝不应以明文存储。相反，我们应该在首次看到密码时对其哈希，然后再存储。哈希密码不是一个困难的过程，而且安全收益巨大。

哈希算法与大多数加密算法不同的原因有很多。首先，哈希算法是不可逆的。在处理密码时，这是一个关键点。我们不希望即使是自己的员工也能够窃取用户密码，因为他们可能在其他地方使用这些密码（这是不良实践，但很常见），并且在出现恶意员工的情况下，我们不希望承担此类责任。

其次，现代哈希算法极其高效。当今的哈希算法可以将多兆字节的字符串表示为仅 128 到 264 位的数据。这意味着当我们进行密码检查时，我们将在登录时重新哈希用户密码，并将其与数据库中的哈希密码进行比较。即使密码很长，我们也能够高速执行查找。

使用哈希的另一个关键优势是，现代哈希算法在实际应用中没有碰撞（0 碰撞或在统计上接近 0——1/10 亿以上）。这意味着您可以在数学上确定两个密码具有相同哈希值的概率极低。因此，您不必担心黑客“猜测”密码，除非他们猜测出另一个用户的精确密码。

如果数据库被入侵且数据被盗，正确哈希的密码可以保护您的用户。黑客将只能访问哈希值，并且数据库中即使单个密码被逆向工程的可能性也非常小。

让我们考虑三种情况，黑客获得了 MegaMerch 数据库的访问权限：

| 情况 | 存储方式 | 结果 |
| :--- | :--- | :--- |
| 情况 #1 | 密码以明文存储 | 所有密码都被泄露 |
| 情况 #2 | 密码使用 MD5 算法哈希 | 黑客可以使用彩虹表（预先计算的哈希→密码表；较弱的哈希算法容易受到这种攻击）破解部分密码 |
| 情况 #3 | 密码使用 BCrypt 哈希 | 任何密码被破解的可能性很小 |

如您所见，所有密码都应该被哈希。此外，用于哈希的算法应根据其数学完整性和在现代硬件上的可扩展性进行评估。算法在哈希时应**慢**，从而减少黑客每秒可以做出的猜测次数。

在破解密码时，慢速哈希算法至关重要，因为黑客将自动化密码到哈希的过程。一旦黑客找到与密码相同的哈希（忽略潜在碰撞），密码就被有效破解了。像 BCrypt 这样哈希极其缓慢的算法，在现代硬件上破解一个密码可能需要数年或更长时间。

现代 Web 应用应考虑以下哈希算法来保护其用户凭证的完整性。

**BCrypt**  
BCrypt 是一种哈希函数，其名称来源于两个发展：“B”来自 Blowfish Cipher，这是 Bruce Schneier 于 1993 年开发的一种对称密钥分组密码，设计为通用开源加密算法。“Crypt”是 Unix 操作系统自带的默认哈希函数的名称。

Crypt 哈希函数是为早期 Unix 硬件设计的，这意味着当时硬件每秒无法哈希足够的密码来使用 Crypt 函数逆向工程哈希密码。在开发时，Crypt 每秒可以哈希少于 10 个密码。在现代硬件上，Crypt 函数可以用于每秒哈希数万个密码。这使得破解 Crypt 哈希密码对任何当代黑客来说都是轻而易举的操作。

BCrypt 在 Blowfish 和 Crypt 的基础上进行了迭代，提供了一种在更快硬件上反而变得更慢的哈希算法。BCrypt 哈希密码可以扩展到未来，因为试图使用 BCrypt 进行哈希的硬件越强大，所需操作就越多。因此，当今黑客几乎不可能编写一个能够执行足够哈希以通过暴力破解匹配复杂密码的脚本。

**PBKDF2**  
作为 BCrypt 的替代方案，PBKDF2 哈希算法也可用于保护密码。PBKDF2 基于一种称为密钥拉伸的概念。密钥拉伸算法第一次尝试时会快速生成哈希，但每次额外尝试会变得越来越慢。因此，PBKDF2 使暴力破解成为一个计算成本高昂的过程。PBKDF2 最初并非为哈希密码而设计，但在无法使用类似 BCrypt 的算法时，足以用于哈希密码。

PBKDF2 接受一个配置选项，代表生成哈希的最小迭代次数。此最小值应始终设置为您的硬件可以处理的最大迭代次数。您永远不知道黑客可能拥有什么类型的硬件，因此通过将哈希的最小迭代次数设置为硬件的最大值，您消除了在更快硬件上的潜在迭代，并消除了在较慢硬件上的任何尝试。

在我们对 MegaMerch 的评估中，我们决定使用 BCrypt 哈希密码，并且只比较密码哈希值。

### 多因素认证（MFA）

除了要求传输中加密的安全哈希密码外，我们还应考虑向希望确保其账户完整性不受损害的用户提供多因素认证（MFA）。图 21-2 显示了 Google Authenticator，这是 Android 和 iOS 上最常见的 MFA 应用之一。它兼容许多网站，并具有用于集成到您的应用中的开放 API。MFA 是一项出色的安全功能，基于非常简单的原理有效运行。

> **图 21-2. Google Authenticator——Android 和 iOS 上最常用的 MFA 应用之一**

大多数 MFA 系统要求用户在浏览器中输入密码，以及从移动应用或短信生成的密码。更高级的 MFA 协议实际上使用物理硬件令牌，通常是一个 USB 驱动器，当插入用户计算机时生成唯一的单次使用令牌。一般来说，物理令牌更适用于企业员工而非其用户。为电子商务平台分发和管理物理令牌对每个参与者来说都是痛苦的经历。

基于手机应用/短信的 MFA 可能不如专用 MFA USB 令牌安全，但相比不使用 MFA 的应用，其安全收益仍然大一个数量级。

在 MFA 应用或消息协议没有任何漏洞的情况下，MFA 消除了对您 Web 应用的远程登录——这些登录不是由账户所有者发起的。破坏 MFA 账户的唯一方法是同时获取账户密码和包含 MFA 代码的物理设备（通常是手机）。

在对 MegaMerch 的架构审查期间，我们强烈建议向希望提高其 MegaMerch 账户安全性的用户提供 MFA。

### 个人身份信息（PII）与金融数据

当我们存储用户的个人身份信息时，我们需要确保此类存储在我们运营所在国家是合法的，并且我们遵守这些国家关于 PII 存储的任何适用法律。除此之外，我们希望确保在数据库泄露或服务器被入侵的情况下，PII 不会以易于滥用的格式暴露。与 PII 类似的规则适用于财务数据，如信用卡号（在某些国家也包含在 PII 法律下）。

较小的公司可能会发现，与其自行存储 PII 和财务细节，更有效的策略可能是将此数据的存储外包给专门从事此类数据存储的合规企业。

## 搜索引擎

任何实现自定义搜索引擎的 Web 应用都应考虑此类任务的影响。搜索引擎通常要求数据以某种方式存储，以使特定查询非常高效。搜索引擎中数据的理想存储方式与通用数据库中数据的理想存储方式有很大不同。

因此，大多数实现搜索引擎的 Web 应用将需要一个单独的数据库，搜索引擎从中提取数据。如您所见，这可能导致许多复杂情况，需要从一开始就有适当的安全架构，而不是后来。

同步任何两个数据库都是一项艰巨的任务。如果主数据库中的权限模型被更新，搜索引擎的数据库也必须更新以反映主数据库中的更改。

此外，引入代码库的错误可能导致某些模型在主数据库中被删除，但在搜索数据库中未被删除。或者，搜索数据库中关于特定对象的元数据在对象从主数据库中移除后可能仍然可搜索。

所有这些都是实现搜索时的关注点示例，在实现任何搜索引擎（无论是 Elasticsearch 还是内部解决方案）之前，绝对应考虑这些关注点。Elasticsearch 是最大、最广泛使用的开源分布式搜索（图 21-3）。它易于配置，文档完善，并且可以在任何应用中免费使用。它建立在 Apache 的 Solr 搜索引擎项目之上。

## 零信任架构

零信任架构是将一种称为零信任的哲学应用于软件架构。这两个术语在此可以互换使用。一般来说，零信任架构有许多不同的名称：

- 零信任
- 零信任模式
- 零信任网络访问（ZTNA）
- 零信任设计
- 零信任设计模式

所有这些术语都指同一组设计原则，我们将在本节中讨论。

### 零信任的历史

“零信任”一词首次出现在计算机科学家 Stephen Paul Marsh 于 1994 年在斯特灵大学（苏格兰）的博士论文中。Stephen 认为信任的概念可以分解为计算机可以计算的有限标准集。在 Stephen 的博士论文发表后不久，“零信任”一词从常用术语中消失，直到 2018 年。

2020 年，NIST 发布了一份名为《[零信任架构（SP-800-207）](https://oreil.ly/Spd7b)》的白皮书。该出版物重新激发了“零信任”一词的使用，在其发布后的几年里，它成为信息安全领域搜索最广泛的术语之一。今天，在安全背景下使用“零信任”一词时，大多数时候指的是 2020 年 NIST 同名白皮书中编写的指南。

### 隐式信任与显式信任

零信任架构的核心是一种用于开发安全应用的设计模式。零信任主要通过在两种主要类型的“信任”上区分于其他设计模式：隐式信任和显式信任。在上述 NIST 白皮书中，隐式信任被定义为基于接近度或角色授予的信任类型。

考虑一座带有护城河的城堡。由于护城河的目的是阻止不速之客进入城堡，可以假设任何已经通过护城河边界的人在城堡内都应被信任。这是一种隐式信任模型，因为除了护城河之外没有验证。在此示例中，护城河是决定一个人应被信任还是不信任的唯一因素——没有考虑不速之客游过护城河的边缘情况。

一个类似且更相关的隐式信任示例可能是一个受防火墙保护但没有任何请求通过防火墙后进行验证的网络。NIST 认为这种类型的信任模型已经过时，并导致易受攻击且容易被利用的 Web 应用。

根据 NIST 的指导，更好的模型是“信任但验证”或显式信任模型。在此类模型中，应用被设计为在调用特权功能时进行验证。这意味着仅仅存在于隐式信任区域中（例如 AWS 云账户、通过服务器防火墙、之前成功的身份验证检查等）是不够的。

根据零信任架构，每一个可能导致入侵的操作都必须有一个前置验证步骤，以确保请求者或调用者是他们声称的身份并且仍然拥有适当的权限。

### 认证与授权

到目前为止，您可能已经注意到零信任架构与最小权限原则有很多重叠之处。这是因为零信任架构的核心是花费大量时间确保行为者（脚本、用户等）只能访问为其预期的资源。

但还有其他情况需要考虑，零信任同样有益。考虑企业中的员工担任高级职位，并且该职位在公司内部软件中拥有重要权限。我们称这位员工为 Joe Admin，因为该员工拥有管理员权限，可以高度访问内部软件。

在他的日常工作中，Joe 使用众多软件服务查看公司财务、评估业务交易等功能。他甚至有权访问公司银行账户。

有一天，Joe 因洗钱（或任何其他正当理由）被公司解雇。该公司的内部软件架构设计使得用于认证和授权的令牌具有 48 小时的过期窗口。

这并不罕见，在此情况下，Joe 一直使用的软件只是寻找有效的授权令牌，并没有以任何方式验证 Joe 作为高级员工的状态是否未发生变化。因此，Joe 即使在被解雇后也能使用该软件，并通过修改公司账簿以报复，造成额外的混乱。

零信任授权，或应用零信任原则的授权架构，本可以防止这种边缘情况发生。攻击之所以发生，是因为信任被隐式授予任何拥有未过期访问令牌的用户——而没有考虑用户就业状态可能发生的变化。在每个授权步骤中实施 NIST 定义的零信任将产生持续授权，这将防止此类针对内部系统的攻击。

## 小结

在构建应用时有许多需要考虑的问题。每当产品组织开发新应用时，该应用的设计和架构也应由经验丰富的安全工程师或架构师仔细分析。深层安全缺陷——如不正确的认证方案或不成熟的搜索引擎集成——可能使您的应用面临不易解决的风险。一旦付费客户开始在其工作流程中依赖您的应用，尤其是在合同签订之后，解决架构级安全错误将变得艰巨。

在本章开头，我引用了 NIST 的估计，即在架构阶段发现的安全漏洞的修复成本比在生产阶段发现要低 30 到 60 倍。这可能是多种因素共同作用的结果，包括以下因素：

- 客户可能依赖不安全的功能，因此您需要构建等效的安全功能并为他们提供迁移计划，以避免停机。
- 深层架构级安全缺陷可能要求除了不安全的模块外，还要重写大量模块。例如，一个具有有缺陷多人模块的复杂 3D 视频游戏，可能不仅需要重写网络模块，还需要重写建立在多人网络模块之上的游戏模块。如果底层技术必须更换以提高安全性（例如从 UDP 或 TCP 网络迁移），尤其如此。
- 安全漏洞可能已被利用，除了工程时间外，还给企业造成实际金钱损失。
- 安全漏洞可能被公开，给受影响的 Web 应用带来负面公关，导致企业失去签约机会和选择离开的客户。

最终，发现和解决安全问题的最佳阶段始终是架构阶段。在此阶段消除安全问题将为您节省长期成本，并消除日后外部发现或公开所导致的潜在麻烦。
