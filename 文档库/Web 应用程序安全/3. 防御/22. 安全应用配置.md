# 第 22 章：安全应用配置

成功向客户交付安全 Web 应用的一个组成部分，是确保您所交付的 Web 应用以能够利用浏览器内置安全机制的方式进行了配置。

当今的 Web 应用构建在众多语言、框架和技术之上。然而，由于 Web 应用的唯一交付方式仍然是浏览器，学习如何利用浏览器内置的安全机制对于良好的安全态势至关重要。

在本章中，我们将评估并讨论浏览器实现的几种安全技术。您还将学习如何正确配置它们，以最大限度地提高 Web 应用的安全性。

## 内容安全策略（CSP）

内容安全策略（CSP）是浏览器针对涉及浏览器客户端的最常见网络攻击形式的主要安全机制之一。如果实施正确，它能够防止 XSS、数据注入、网络钓鱼、框架嵌入和重定向攻击。

为了提供干净的开发者体验而不破坏互联网，CSP 被设计为具有大量配置选项。因此，强大的 CSP 策略与薄弱的 CSP 策略有很大不同。事实上，完全可以在没有任何 CSP 策略的情况下运行一个功能完整的网站，导致浏览器不实施针对常见攻击的缓解措施。

让我们从实现的角度更深入地了解 CSP 策略，以便您学习如何在 Web 应用上正确配置 CSP 策略，从而允许浏览器代表您的用户实施安全机制。

### 实施 CSP

CSP 可以通过两种方法之一在 Web 应用上实施。最常见的方法是让您的服务器在每个请求中返回 `Content-Security-Policy` 头部。请注意，`X-Content-Security-Policy` 和 `X-Webkit-CSP` 头部已被弃用，不应再用于实施 CSP 策略。

或者，您可以通过在每个 HTML 页面的 `<HEAD></HEAD>` 中包含一个 `<meta>` 标签来实施 CSP 策略。这样的 `<meta>` 标签如下所示：

```html
<meta
  http-equiv="Content-Security-Policy"
  content="default-src 'self'; img-src data:" />
```

### CSP 结构

无论您选择哪种实现方式，配置 CSP 策略的方法大致相同。CSP 策略由指令组成，指令之间用分号（`;`）分隔。在每个指令之后，您可以包含与该指令对应的配置选项，然后由浏览器实施。

一个示例指令是 `script-src scripts.mega-bank.com`，它允许网站仅执行来自所提供源 `scripts.mega-bank.com` 的 JavaScript 脚本。所有其他脚本将在浏览器控制台中引发 CSP 错误。

### 重要指令

CSP 支持的指令列表因浏览器而异，并随着浏览器更新和遵循最新版本的 CSP 规范而定期扩展。目前，CSP 规范由万维网联盟（W3C）非营利组织维护，该组织维护各种 Web 标准。

从安全角度来看，以下是一些最重要的指令：

**default-src**  
这是其他指令的回退，并白名单列出可以加载图像、脚本、CSS 样式表和其他资源的源。定义此指令可防止您的网站执行非预期脚本，从而显著降低 XSS 风险。

**sandbox**  
配置后，此指令在页面加载时创建一个沙箱，防止资源能够创建弹出窗口、执行脚本或与浏览器插件交互。

**frame-ancestors**  
此指令定义哪些网页可以嵌入当前网页。将其设置为 `'none'` 通常是最安全的选择，因为它可以防止其他网站通过在当前网页前面放置用户界面元素并诱骗用户点击来进行点击劫持。

**eval 和内联脚本**  
仅通过实施任何 CSP 策略，这些功能就会被默认阻止。这是一个巨大的胜利，因为这两种脚本执行方法都是 XSS 攻击的流行攻击向量。但是，可以通过 `'unsafe-inline'` 和 `'unsafe-eval'` 禁用这些阻止。

**report-uri**  
此指令允许您定义一个端点，用于报告 CSP 错误以进行日志记录。

### CSP 源与源列表

CSP 策略中以 `-src` 结尾的指令接受一个称为源列表的输入——一个由空格分隔的源和 CSP 特定配置值的列表。源列表用于告诉指令在加载到浏览器后如何操作。有几个指令是 CSP 特有的，理解它们的功能很重要：

**`*`**  
此通配符运算符允许除 `blob` 和 `file` 之外的任何 URL。例如，`image-src *` 允许来自任何基于 Web 的源的图像，但不允许来自本地文件系统。

**`'none'`**  
阻止任何源加载。请注意，在指令后包含 `none` 的 CSP 策略不得有其他源，否则 CSP 策略将加载失败。例如，`image-src 'none'` 是有效的，但 `image-src 'none' images.mega-bank.com` 无效。

**`data:`**  
允许加载 base64 编码的图像。例如，`img-src data:`。

**`https:`**  
允许加载，前提是源列表中的所有资源都实现 HTTPS。例如，`img-src https: mega-bank.com`。

**`'self'`**  
指加载页面的当前源。如果网页加载到 `test.mega-bank.com/123`，那么来自 `test.mega-bank.com/*` 的任何图像都可以加载，前提是 CSP 策略设置为 `img-src 'self'`。

### 严格 CSP

有时您在构建更复杂的 Web 应用，需要能够加载内联脚本。但是，您希望避免为所有脚本启用内联脚本的陷阱，因为这是最常见的 XSS 攻击向量之一。

使用 CSP，可以通过一点努力安全地启用内联脚本。这种 CSP 配置形式通常称为**严格 CSP**，因为它为在浏览器中运行的脚本提供了额外的安全规则以防止 XSS，但不会限制功能。

实施严格 CSP 策略有两种方法。第一种是基于哈希的严格 CSP，第二种是基于随机数的严格 CSP。这两种方法都要求常见的 XSS 接收器实现在每次脚本执行之前进行验证的随机数或 SHA-256 哈希。

一个简单的基于随机数的严格 CSP 实现如下所示：

```
Content-Security-Policy:
script-src 'nonce-{RANDOM}' 'strict-dynamic';
```

当 CSP 策略设置为带有 `'nonce-{RANDOM}'` 的 `'strict-dynamic'` 时，浏览器将强制内联脚本提供一个新的属性 `nonce`。为遵守此规定，内联脚本将如下所示：

```html
<script src="..." nonce="123">alert()</script>
```

随机数在运行时创建，您的每个脚本都以正确的 `nonce` 值加载。在脚本执行之前，浏览器检查以确保脚本属性中的 `nonce` 与预定义值匹配。如果为 true，脚本继续执行（从具有正确随机数的顶级脚本加载的动态脚本也可以加载）。如果为 false，脚本执行失败，并在控制台中抛出 CSP 错误。

基于哈希的方法操作类似，但不是使用随机数，而是使用 SHA-256 哈希。在哈希方法中，每个内联脚本的哈希都被添加到 CSP 指令 `script-src` 的源列表中。当内联脚本尝试执行时，它被哈希并与源列表进行比较。如果未能通过检查，则会在控制台中抛出错误，脚本执行失败。

基于随机数的严格 CSP 适用于每个网页都在服务器上渲染的场景，允许在每个页面加载时创建新的随机数。

基于哈希的 CSP 更适合需要缓存的应用程序（例如，使用 CDN），因为 SHA-256 哈希的碰撞率非常低，两个脚本发生碰撞（恶意脚本生成与非恶意脚本相同的哈希）的概率大约在 1/43,000,000,000 左右。由于 CDN 和缓存通常持续时间较长，黑客可以构造与当前随机数匹配的内联脚本载荷。然而，他们极不可能构造出与同一页面中的非恶意脚本哈希完全相同的内联脚本载荷。

### 安全 CSP 策略示例

以下是一个默认安全的 CSP 策略，可以在进一步定制之前作为入门策略使用。它提供了一个使用非ce策略实现严格 CSP 的示例，针对脚本源，阻止 `frame-ancestors` 以防止点击劫持攻击，在允许 base64 图像加载的同时对图像强制执行 HTTPS，提供 CSP 错误的报告 URI，并提供一个默认的 `'self'` 作为源列表的回退。实施此 CSP 的代码如下：

```
Content-Security-Policy
default-src: 'self';
script-src: 'self' 'nonce-jgoj23j2o3j2oij26jk2nkn26kjh23' 'strict-dynamic';
frame-ancestors: 'none';
img-src: data: https:;
report-uri: https://reporting.megabank.com
```

## 跨域资源共享（CORS）

跨域资源共享（CORS）是一种浏览器实施的安全机制，经常与 CSP 混淆。虽然 CSP 允许开发者选择哪些脚本被允许在浏览器中执行，但 CORS 能够在脚本到达浏览器中的 JavaScript 执行上下文之前的更早阶段阻止脚本。

CORS 之所以重要，部分原因是在 JavaScript（唯一受浏览器支持的编程语言）中执行网络请求的两种主要方法是 `fetch` 和 `XMLHTTPRequest`。这两个 API 都遵循一个称为同源策略（SOP）的概念，该策略规定，除非在 CORS 策略中定义，Web 应用只能在其自身的（相同）源内进行网络调用。

需要注意的是，如果没有 SOP，来自浏览器内 JavaScript 代码的 `fetch` 或 `XMLHTTPRequest` 可以对另一个网站（而非执行代码所在的网站）执行特权操作。这是因为浏览器会自动将 cookie 附加到网络请求，这意味着如果用户在标签页 #2 中已通过电子商务商店的认证，而标签页 #1 可以在没有 SOP 的情况下对该电子商务商店进行购买（这称为跨站请求伪造或 CSRF 攻击）。

CORS 为我们提供了一种配置解决方案，使开发者能够允许特定的跨域请求完成，而不会显著降低安全态势（如同禁用 SOP 那样）。

### CORS 请求类型

CORS 策略可以通过两种方法进行检查。这些通常称为**简单 CORS 方法**和**预检 CORS 方法**。

WHATWG 组织维护的规范指出，虽然简单版本的检查对于最常见的网络请求已经足够，但预检应用于任何类型的不寻常或状态更改的请求。

### 简单 CORS 请求

简单请求可以是 POST、HEAD 或 GET 请求，但必须遵循规范中概述的几个条件。首先，简单请求只能包含几个选定的头部。这些在文档中被注为“CORS 安全列头”：

- Accept
- Accept-Language
- Content-Language
- Content-Type

其次，`Content-Type` 头部必须仅包含以下头部之一：`application/x-www-form-urlencoded`、`multipart/form-data` 或 `text/plain`。

最后，源自浏览器的网络请求不得使用高级网络请求功能，如附加到 `XMLHTTPRequest` 对象的事件监听器。

如果满足所有这些条件，浏览器将向网络请求附加一个头部：`origin: <source-origin>`。如果请求的服务器配置正确，它将返回 HTTP 状态码 200，并附加另一个头部：`access-control-allow-origin: <source-origin>`。

如果浏览器提供的头部中包含的源与服务器提供的头部不匹配，则来自服务器的响应将不会传回脚本执行，并且浏览器控制台将抛出 CORS Policy 错误。

### 预检 CORS 请求

任何不符合作为简单请求执行标准的请求都将作为预检请求执行。

在预检请求中，浏览器将发送一个预检检查，这只是一个格式化为包含以下头部的 HTTP Options 请求：

- `origin: <source-origin>`
- `access-control-request-method: <permitted HTTP method>`
- `access-control-request-headers: <permitted HTTP headers>`

与简单请求类似，如果预检请求中的任何检查失败，浏览器网络栈将不会将响应传递给脚本执行上下文，而是抛出 CORS Policy 错误。

预检请求背后的设计理念是，它们通过拒绝在确定服务器和浏览器是否被允许通过匹配 CORS 头部进行通信之前，将任何相关数据或操作传递给服务器，从而提供更高的安全性。

### 实施 CORS

虽然浏览器决定将哪些 `origin`、`access-control-request-method` 和 `access-control-request-headers` 附加到出站网络请求，但应用开发者仍需要正确配置服务器，以用适当的响应匹配入站 CORS 网络请求。

大多数现代服务器软件包提供简单的 CORS 中间件，通常抽象了大部分 CORS 配置。在服务端 JavaScript 生态系统中，这通常通过 `cors` npm 模块完成：

```javascript
const express = require('express');
const app = express();
const cors = require('cors');

const corsOptions = {
  origin: 'https://mega-bank.com'
};

app.get('/users/:id', cors(corsOptions), function (req, res, next) {
  res.sendStatus(200);
});

app.listen(443, function () {
  console.log('listening on port 443')
});
```

与浏览器中的许多其他基于配置的工具一样，CORS 接受多种定义允许源的方式——包括通配符（`*`）。始终记住**最小权限原则**，并授予服务端 CORS 策略最小范围的源，而不是最大范围。因此，只将您的应用运行所需的特定源加入白名单。

### 头部

现代浏览器允许在头部级别进行各种类型的安全配置，正如我们在 CSP 和 CORS 中所见。除了这些通过 HTTP 头配置的标志性安全机制之外，还有大量额外的基于头的安全机制，它们提供了一点安全改进，且实现难度很低。以下是每个现代 Web 应用都应考虑使用的几个重要但常被忽视的安全机制。

#### 严格传输安全（HSTS）

HTTP 严格传输安全（HSTS）头部告诉浏览器，网页应仅通过 HTTPS 加载，并且将来的 HTTP 请求应“升级”为 HTTPS。HSTS 是强制请求使用 SSL/TLS（HTTPS）的首选机制，因为 301 重定向需要初始 HTTP 请求，这可能容易受到中间人攻击。

HSTS 很容易作为头部实现，使用以下语法：

```
Strict-Transport-Security: max-age=<expire-time>; includeSubDomains; preload
```

参数如下：

- **max-age**：浏览器记住 HSTS 配置的时间（秒）。
- **includeSubDomains**：将 HSTS 策略应用于当前源的所有子域。
- **preload**：规范中未包含的可选参数。Google 目前运行一个预加载列表，您的应用可以提交到该列表。Firefox 和 Chrome 浏览器在加载网页之前会检查预加载列表，如果存在则加载最新的 HSTS 策略。

#### 跨域打开者策略（COOP）

现代浏览器维护一个“浏览上下文”。此上下文包括有关当前页面、iframe 和标签页数据的信息。此浏览上下文的共享可以通过 `Cross-Origin-Opener-Policy` 安全头部进行修改。

此头部可以通过语法实现：`Cross-Origin-Opener-Policy: <config>`，其中 `config` 参数可以是 `unsafe-none`、`same-origin-allow-popups` 或 `same-origin`。参数如下：

- **unsafe-none**：允许与打开当前页面的页面或文档共享浏览上下文（如果未定义头部，则为默认值）
- **same-origin-allow-popups**：允许与源自当前页面的弹出窗口共享浏览上下文
- **same-origin**：将浏览上下文的共享限制为仅同源页面

尽管这是一个相对较新且鲜为人知的安全机制，将 `Cross-Origin-Opener-Policy` 头部设置为 `same-origin` 可防止新标签页、窗口或其他浏览上下文能够导航回打开上下文。这是一个巨大的安全优势，因为它防止了非预期的信息泄露。

#### 跨域资源策略（CORP）

2018 年，安全研究人员发现，通过对现代 CPU 架构使用旁路时序攻击，可以通过 JavaScript 从另一个标签页读取内存和状态。此漏洞被称为 Spectre，至今仍影响正在使用的 Intel CPU。

针对此类攻击的少有使用但非常有效的缓解措施是 `Cross-Origin-Resource-Policy` 头部。此头部受 Firefox、Edge 和 Chrome 浏览器支持，允许开发者告诉浏览器进一步限制资源（例如文档、窗口）在 Web 浏览器中的访问位置。

此头部的配置值为：

- **same-origin**：限制读取为同源（协议/方案、域、端口）。这是最安全的选择。
- **same-site**：仅限制读取为同站点（顶级域）。这不如 `same-origin` 安全，因为它允许子域访问服务器发送的资源。
- **cross-origin**：允许从任何源读取。这是最不安全的配置选项。

虽然这是一项选择加入的技术，但 CORP 头部指示浏览器在比通过 JavaScript 更低的级别强制执行内存共享——使其成为任何 Web 应用安全头部的一个良好补充。

### 具有安全含义的头部

虽然不如前述头部复杂，但以下头部为您的 Web 应用提供了一些额外的安全性：

**X-Content-Type-Options**  
当设置为 `nosniff` 时，可防止浏览器猜测 MIME 类型（即 MIME 嗅探），这可以防止浏览器尝试将不可执行文件类型转换为可执行文件类型。

**Content-Type**  
最常见的头部之一，可用于指示 HTTP 请求中发送的内容类型，例如 `Content-Type: text/html; charset=UTF-8`。主要的安全益处是它允许客户端正确解释内容。

### 遗留安全头部

在过去，单个头部通常用于在浏览器上启用安全机制。然而，在现代，许多这些头部已被合并到 CSP 的功能中或由浏览器默认启用。

以下“遗留”头部与其现代等效物一起列出，适用于需要保留所述功能但希望以前瞻性方式实现的应用：

- **X-Frame-Options** → CSP `frame-ancestors`
- **X-XSS-Protection** → 已移除。CSP 默认阻止最常见的 XSS 接收器。浏览器不再扫描反射型 XSS。
- **Expect-CT** → 已移除。2018 年 5 月之后生成的所有 TLS 证书默认支持证书透明度（CT）要求。所有先前的 TLS 证书由于最长 39 个月的年龄限制现已过期。
- **Referrer-Policy** → 现代浏览器默认不再发送所有 referrer 信息，但旧浏览器会向同站发送源、路径和查询字符串，向其他网站发送源。可以通过设置头部在旧浏览器中禁用它：`Referrer-Policy: strict-origin-when-cross-origin`。
- **X-Powered-By** → 某些 Web 服务器默认附加此头部，以便客户端知道哪个服务器发送了响应。这允许轻松指纹识别服务器和版本，这是一种信息泄露，使黑客更容易发起攻击。在现代服务器上默认禁用，但如果您使用服务端软件的旧版本，请确保禁用此头部。
- **X-Download-Options** → 仅由 Internet Explorer 支持，此头部允许开发者防止下载在 Internet Explorer 上下文中在内存中运行。由于 Internet Explorer 已弃用，此头部通常不再需要。

## Cookie

尽管一些现代 Web 应用使用替代方法来存储和传输会话信息和令牌，但大多数 Web 应用仍然使用 cookie 作为在每个请求基础上认证用户的主要方式。因此，了解如何锁定和加固浏览器处理令牌的方式以防止泄露或修改至关重要。

### 创建与安全设置 Cookie

Cookie 通常通过 HTTP 头部 `Set-Cookie` 设置。此头部接受一组配置，包括 cookie 数据的内容，它是一个由等号（`=`）分隔的键：值对。在 cookie 数据之后，可以包含一组属性，允许开发者配置浏览器如何处理 cookie。

创建 cookie 的常见示例如下：

```
Set-Cookie: auth_token=abc123; Secure;
```

在前面的示例中，通过 HTTP 头部设置了一个 cookie，此后将在每个 HTTP 请求中在浏览器和服务器之间来回传递。此 cookie 的键为 `auth_token`，值为 `abc123`。它还包含一个属性标志 `Secure`，告诉浏览器仅在网站通过 HTTPS 加载时才发送 cookie。这是为了防止中间人攻击。

保护 cookie 的主要方法是通过属性标志，其中大量标志允许开发者限制浏览器的不安全机制，从而提高 cookie 的安全性和完整性。在这些属性中，最强大的安全标志如下：

- **Path**：默认情况下，浏览器会在当前域（例如 `mega-bank.com`）的每个网络请求中发送所有 cookie。包含 `path=/website` 属性会将浏览器限制为仅在路径中包含 `/website` 的对当前域的网络请求中发送 cookie。
- **Secure**：任何包含敏感数据的 cookie 都应附加 `secure` 属性。它防止 cookie 通过未加密的网络调用发送，以防止中间人攻击窃取 cookie 中的数据。可以通过 `Secure;` 设置。
- **Expires**：强制浏览器在 HTTP 日期时间戳指定的日期之后丢弃 cookie，例如 `Expires: Wed, 20 Dec 2025 01:01:00 GMT`。
- **HttpOnly**：此属性阻止 JavaScript 代码能够读取 cookie。这意味着即使您的网站通过允许黑客对用户浏览器运行 JavaScript 的 XSS 攻击被入侵，JavaScript 也不应能够访问会话 cookie，因此不应能够代表被黑客入侵的用户进行特权网络调用。可以通过 `HttpOnly;` 设置。
- **SameSite**：接受三个值：`Lax`、`Strict` 或 `None`。用于通过防止 cookie 伴随跨站请求发送来阻止 CSRF 攻击。通常应设置为 `Strict`，这仅允许从生成 cookie 的站点发送 cookie。

除非绝对必要，否则不应使用 `domain` 属性，因为它会放宽对 cookie 的限制，使其仅发送到当前域。在 cookie 上使用 `domain` 属性的一个奇怪副作用是，它仅仅因为存在就会将默认 cookie 安全模型从 `my-domain.com` 更改为 `*.my-domain.com`。这意味着如果您使用 `domain` 属性来指定 cookie 可以发送的其他域，您也将在所有子域中发送 cookie。如果您的子域包含用户开发的脚本，这可能产生许多安全隐患，甚至导致账户接管。

### 测试 Cookie

通常，cookie 通过拦截工具进行测试，这些工具允许开发者或安全工程师对开发或暂存环境实施自己的中间人攻击，以拦截和读取其应用程序的 cookie。有多种工具（通常称为 HTTP 代理）可用于拦截。其中一些工具专门用于安全测试，包括非常流行的 Burp Suite 和开源竞争者 ZAP。

当使用这些工具中的任何一个与仅通过 HTTP 通信的应用一起使用时，您需要使用用于测试的 TLS 证书配置本地应用，并向您的 HTTP 拦截工具提供解密密钥。

Cookie 也可以在 Firefox、Safari、Chrome 或 Edge 浏览器提供的浏览器开发者工具中进行测试。这样做非常简单，可以通过控制台或 UI 完成。

要在 Chrome 浏览器上查看 cookie，请执行以下步骤：

1. 按 F12 打开浏览器开发者工具。
2. 转到控制台中的 Application 标签。
3. 展开 Storage 部分下的 Cookies 下拉菜单。
4. 选择您想要查看 cookie 的网站。

在开发者工具 JavaScript 控制台中，您也可以输入 `document.cookie` 来查看当前加载的网页主文档的 cookie。

## 框架与沙箱

有时在开发 Web 应用时，您可能会遇到需要在与第一方代码相同的页面中运行其他开发者的代码的边缘情况。这在允许深度用户定制的网站中很常见，典型的例子是 MySpace，它允许深度定制用户档案。另一种情况可能是您的公司与另一家生产相关 Web 软件的公司之间的合作。

在这两种情况下，默认允许第三方代码在您的网站中运行都是一个巨大的风险。但是，如果在第三方代码在您的 Web 应用中执行之前对其进行沙箱隔离，这些风险可以在一定程度上得到缓解。在本节中，我们将讨论几种沙箱隔离潜在恶意第三方代码方法的优缺点。

### 传统 Iframe

如果正确实施，浏览器的 `iframe` 元素可以允许开发者安全地在与第一方代码相同的页面中使用其他开发者的代码。Iframe 创建一个单独的 DOM 和 JavaScript 上下文，这些上下文被沙箱隔离以防止泄漏到主框架。Iframe 逃逸被视为 P0（严重风险）漏洞，并由浏览器供应商快速解决，这意味着 Google（Chrome）、Microsoft（Edge）和 Mozilla（Firefox）正在资助此机制的安全性，而不是您的公司。

使用 iframe 的缺点很明显，因此尽管它是最简单的方法之一，但并不适用于所有情况。Iframe 与大多数 UI 不能很好地融合，因为它们拥有自己的 DOM，并且必须与主窗口配置相同。因此，iframe 需要代码重复，这意味着如果主框架的 UI 更改了颜色，框架 CSS 中也必须更改。这导致开发过程中的不便并需要额外时间。同样在 UI 方面，iframe 难以像主 DOM 元素那样缩放。`iframe` 对象需要独特的 CSS 格式，增加了复杂性。

除了这些限制之外，iframe 与主窗口的通信仅限于 `postMessage` API。这意味着任何子框架到主框架的通信都需要额外的开发时间。

最后，由于 iframe 启动自己的 DOM 和 JavaScript 上下文，并且需要额外的代码进行配置，当同一页面中使用多个 iframe 时，可能会导致显著的性能下降。

尽管有这些限制，iframe 易于实现。一个简单的 iframe 沙箱如下所示：

```html
<iframe src="https://other-website.com"></iframe>
```

由于上述 iframe 使用了与主窗口 `https://mega-bank.com` 不同的源，它获得了 iframe 隔离和 SOP 隔离机制的好处。

一个更安全的实现如下：

```html
<iframe src="https://other-website.com" sandbox></iframe>
```

请注意附加到 `iframe` 元素的 `sandbox` 属性。`sandbox` 属性强制 iframe 使用几个额外的隔离层，包括：

- 将内容视为独立源（请参阅“跨域资源共享”一节中关于 SOP 的讨论）
- 阻止表单提交
- 阻止脚本执行
- 禁用 API
- 防止链接访问其他浏览上下文
- 防止使用插件
- 防止 iframe 查看父浏览上下文
- 禁用自动播放

在考虑沙箱时，iframe 应是您的第一道防线。尽管如此，始终重要的是权衡其局限性并了解您的其他选择。

### Web Worker

Web Worker 不提供与 iframe 相同级别的隔离，但它们确实比在与第一方代码相同的执行上下文中执行第三方代码提供了更多的隔离。Web Worker 是在浏览器内运行的独立隔离线程，能够执行同源脚本，但无法与 DOM 交互。

在支持的浏览器上创建 Web Worker 很简单：

```javascript
if (window.Worker) {
  // 创建 Web Worker 并让其运行 code.js 中的代码
  const myWorker = new Worker('code.js');
  // 销毁 Web Worker
  myWorker.terminate()
}
```

由于 Web Worker 有自己的 JavaScript 执行上下文，它们无法访问页面的主 JavaScript 执行上下文中的变量和数据，除非通过 `postMessage` 或 `MessageChannel` API 显式传递。然而，Web Worker 能够通过 `XMLHTTPRequest` API 进行同域和跨域的 HTTP 请求，这使得它们不如独立源的 iframe 安全。

### 子资源完整性（SRI）

子资源完整性本身并不是一种隔离技术，而是一种验证来自第三方的代码未被修改的方法。

如果 iframe 不能满足您与第三方 JavaScript 供应商集成的需求，您的团队可以手动审查第三方 JavaScript，然后生成一个 base64 编码的 SHA-256、SHA-384 或 SHA-512 哈希，表示该 JavaScript 代码的内容。

在生成代表第三方 JavaScript 代码内容的哈希后，您可以使用包含先前生成的哈希的 `integrity` 属性将该代码加载到浏览器中：

```html
<script src="other-website.com/stuff.js"
        integrity="sha384-NmJhYmViMzNiMmU1NzllMDMyODdl"
        crossorigin="anonymous"></script>
```

在加载时，浏览器将对 `other-website.com/stuff.js` 执行 HTTP GET 以获取脚本。然而，在将其加载到 JavaScript 执行上下文之前，它会将资源的哈希与 `integrity` 属性中提供的哈希进行比较。如果 `integrity` 哈希与从第三方网站加载的文件的哈希不匹配，浏览器将认为该文件自上次审查以来已被修改，抛出控制台错误，并拒绝加载 JavaScript。

### Shadow Realms

在撰写本文时，Shadow Realms 是一个即将推出的 JavaScript 语言特性，旨在消除 iframe 隔离的一些缺陷，同时保持不同开发者之间的 JavaScript 沙箱隔离。Shadow Realms 引入了一种在隔离环境中执行 JavaScript 的新方法，通过创建具有自己全局对象、内在对象和内建对象（标准对象和函数）的新执行上下文。

如今，可以通过实现一个公开可用的 shim（基于 JavaScript 的代码块，模拟计划在 2025 年前向所有主流浏览器发布的 API）来测试 Shadow Realms。

创建 Shadow Realm 很容易，可以完全用 JavaScript 完成：

```javascript
const shadowRealm = new ShadowRealm();

// 导入在其自身环境中执行的代码。
const doSomething = await shadowRealm.importValue('./file.js', 'redDoSomething');

// 此调用链接到 shadowRealm 的 redDoSomething
doSomething();
```

Shadow Realms 设计为易于实现，并且不会干扰您的 UI。它占用更少的内存，并且能够同步执行代码。这意味着您可以在主执行上下文中编写代码，该代码在 Shadow Realm 的代码之后直接执行。这在 iframe 中是不可能的，因为 iframe 到主窗口的通信默认是异步的。

准备并支持 Shadow Realms 的发布可能是下一代浏览器中沙箱隔离 JavaScript 的最佳方法。Shadow Realms 目前是一个第 3/4 阶段的 TC39 提案，这意味着一旦它能够通过所有主流浏览器上的 Test262（兼容性）测试，它将被添加到 JavaScript 编程语言中。

## 小结

Web 应用安全在很大程度上是关于编写安全代码并以安全方式部署 Web 应用。但随着开发和部署 Web 应用的生态系统不断发展，越来越多的第三方代码、工具和技术成为标准。

作为一名适应性强的安全工程师，重要的是要注意，您的 Web 应用的安全性仅取决于其最薄弱的环节。通常，最薄弱的环节是浏览器、网络栈或第三方集成。正确地配置您的 Web 应用，以最大程度地减轻这些额外风险，为您的 Web 应用提供尽可能安全地部署、使用和维护的机会。
