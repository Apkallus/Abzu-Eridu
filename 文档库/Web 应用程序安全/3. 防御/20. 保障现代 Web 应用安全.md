# 第 20 章：保障现代 Web 应用安全

到目前为止，我们已经花了大量时间分析用于研究、分析和入侵 Web 应用的技术。这些初步技术本身很重要，并且在我们进入本书第三部分也是最后一部分——防御时，为我们提供了重要的洞察。

当今的 Web 应用比其前身更加复杂和分布式。与较老的、单体式的 Web 应用（尤其是那些服务端渲染且几乎没有用户交互的应用）相比，这增加了攻击的表面积。这就是我将本书结构设计为从侦察开始，然后是攻击，最后是防御的原因。

我相信，了解 Web 应用的攻击面，并理解潜在黑客如何映射和分析这种攻击面是很重要的。除此之外，我认为了解黑客用于入侵 Web 应用的技术，对于任何希望保护 Web 应用的人来说也是至关重要的知识。通过理解黑客会用什么方法入侵您的 Web 应用，您应该能够得出优先防御的最佳方法，并对恶意窥探者隐藏您的应用架构和逻辑。

到目前为止，我们所涵盖的所有技能和技术都是协同作用的。提高您在侦察、攻击或防御方面的掌握程度，将使您的时间利用极其高效。

保护 Web 应用在某种程度上类似于保卫一座中世纪城堡。城堡由许多建筑和城墙组成，它们代表了核心应用代码。城堡外有许多建筑，它们以描述应用依赖和集成的方式与城堡主人（通常是领主）集成并支持他。由于城堡及周围王国的庞大表面积，在战争期间必须优先安排防御；在每个潜在入口最大化防御工事是不可行的。

在 Web 应用安全的世界中，这种优先级排序和漏洞管理通常是大公司的安全工程师或小公司中更通用的软件工程师的工作。这些专业人员承担着防御大师的角色，结合软件工程技能与侦察和黑客技能，以降低成功攻击的概率，减轻潜在损害，然后管理正在进行或已发生的损害。

## 防御性软件架构

编写一个防御良好的 Web 应用的第一步，始于实际编写任何软件之前。这是架构阶段。在任何新产品或功能的架构阶段，应深入关注流经整个应用的数据。

可以说，软件工程的大部分工作是高效地将数据从 A 点移动到 B 点。同样，安全工程的大部分工作是高效地保护从 A 点到 B 点传输中的数据，以及数据在传输之前、之后或期间可能驻留的任何位置。

在实际编写和部署软件之前，发现并解决深层架构安全缺陷要容易得多。在应用被用户采用之后，可以为了修复安全漏洞而进行重新架构的深度通常受到限制。

对于消费者在其上构建的任何类型的 Web 应用尤其如此。允许用户开设自己的商店、运行自己的代码等的 Web 应用，重新架构的成本可能极其高昂，因为深度重新架构可能需要客户重做许多耗时的手动流程。

在接下来的章节中，我们将学习许多正确评估应用架构安全性的技术。这些技术从数据分析到新功能的威胁建模，范围广泛。

## 全面代码审查

在已经过安全架构评估的 Web 应用实际编写过程中，下一步是在每次提交发布到代码库之前仔细评估它。大多数公司已经采用了强制性代码审查流程，以提高质量保证、减少技术债务并消除易于发现的编程错误。

代码审查也是确保已发布代码符合安全标准的关键步骤。为减少利益冲突，源代码版本控制的提交不仅应由提交者团队的成员审查，还应由不相关的团队（尤其是在安全方面）审查。

在代码审查级别按提交逐个发现安全漏洞实际上比人们想象的要容易。需要关注的主要方面是：

- 数据如何从 A 点传输到 B 点（通常通过网络，以特定格式）？
- 数据如何存储？
- 当数据到达客户端时，如何呈现给用户？
- 当数据到达服务器时，会对其执行哪些操作，以及如何持久化？

在接下来的章节中，我们将评估执行安全代码审查的更具体措施。但此列表提供了可以构建的基础，证明任何人都可以开始进行安全审查。

## 漏洞发现

假设您的组织和/或代码库已经在编写代码之前（架构）和开发过程中（代码审查）采取了评估安全的措施，下一步是在代码审查过程中不易识别（或被遗漏）的错误所导致的漏洞。

漏洞有多种发现方式，其中一些方式会损害您的业务/声誉，而另一些则不会。

发现漏洞的老式方式要么是客户通知，要么是（最坏情况下）广泛的公开披露。不幸的是，一些公司仍然依赖此作为他们发现 Web 应用漏洞以进行修复的唯一手段。

存在更现代的发现漏洞的方法，它们可以保护您的产品免受负面公关、诉讼和客户流失的浪潮。当今最注重安全的公司使用以下组合：

- 漏洞赏金计划
- 内部红/蓝队
- 第三方渗透测试员
- 公司激励工程师记录已知漏洞

通过使用这些技术中的一种或多种，在客户或公众之前发现漏洞，大公司可以以少量前期支出节省大量资金。

我们将在接下来的章节中评估这些发现漏洞的方法中的每一种。我们还将分析几个未能适当投资于此类主动安全措施的公司的知名案例，以及由此类疏忽导致的巨大财务损失。

## 漏洞分析

在您的 Web 应用中发现漏洞后，应采取几个步骤来正确分类、优先级排序和管理该漏洞。

首先，并非所有漏洞都具有相同的风险。在安全工程中众所周知，有些漏洞可以推迟到开发者有空闲时间时再处理，而另一些则值得放下所有当前开发流程以便进行修补。

漏洞管理的第一步是评估漏洞给公司带来的风险。漏洞的风险级别决定了在确定何时以及以何种顺序修复漏洞时的优先级。

风险和优先级可以从以下因素推导出来：

- 对公司的财务风险
- 利用的难易程度
- 被泄露的数据类型
- 现有的合同协议
- 已到位的缓解措施

在确定漏洞的风险和优先级后，下一步涉及开发跟踪方法，以确保解决方案按时推进并符合您的合同义务。最后一步是编写自动化测试，以确保漏洞在修复部署后不会回归和重新打开。

## 漏洞管理

在评估漏洞的风险并根据列出的因素对其进行优先级排序后，必须跟踪修复直至完成。此类修复应及时完成，截止日期根据风险评估确定。此外，应根据评估的漏洞分析客户合同，以确定是否有任何协议被违反。

此外，如果在此期间可以记录漏洞，则应增加额外的日志记录，以确保在开发修复程序时没有黑客试图利用该漏洞。对已知漏洞缺乏日志记录已导致数家公司倒闭，因为它们在等待工程团队解决期间并未意识到漏洞正在被滥用。

管理漏洞是一个持续的过程。您的漏洞管理流程应仔细规划并记录在案，以便记录您的进展。随着时间的推移，这应该会产生更准确的时间表，并且可以平均化修复时间的消耗率。

## 回归测试

在部署修复漏洞的补丁后，下一步是编写一个回归测试，该测试将断言修复有效且漏洞不再存在。这是一个最佳实践，但使用它的公司数量并不如应有的多。很大比例的漏洞是回归——要么是直接重新打开的错误，要么是原始错误的变体。一家大型软件公司（10,000 多名员工）的安全工程师曾告诉我，他们大约 25% 的安全漏洞是先前关闭的错误回归到打开状态。

构建和实施漏洞回归管理框架很简单。向该框架添加测试用例所需的时间应该只占实际修复所花费时间的一小部分。漏洞回归测试前期成本很低，但从长远来看可以节省大量时间和金钱。我们将在接下来的章节中讨论如何有效构建、部署和维护回归测试框架。

## 缓解策略

最后，对任何注重安全的公司来说，一个总体最佳实践是积极努力减轻应用代码库中发生漏洞的风险。这是一个从架构阶段到回归测试阶段都在进行的实践。

缓解策略应该是广泛的，就像一张网试图捕获尽可能多的鱼。在应用的关键区域，缓解也应该是深入的。

缓解的形式包括安全编码最佳实践、安全应用架构、回归测试框架、安全软件开发生命周期（SSDL）以及默认安全的开发人员思维和开发框架。

在接下来的章节中，我们将学习多种方法来缓解，有时甚至消除特定漏洞可能引入我们代码库的风险。

实践以上所有步骤将大大增强您所工作的任何代码库的安全性。它将消除组织中的大量风险，节省大量资金，同时保护您免受否则会及时出现的巨大品牌损害。

## 应用侦察与攻击技术

我们在第一部分和第二部分学到的技术并非进入第三部分之前所必需的。然而，对侦察和攻击技术的深入了解将使您能够构建更强大的防御，否则无法获得这些洞察。

在我们进行保护 Web 应用的过程中，请记住从第一部分学到的侦察技术。这些技术将使您深入了解如何隐藏您的应用以避开不必要的窥探。它们还将使您了解如何优先修复，因为您会注意到某些漏洞比其他漏洞更容易发现。

第二部分的内容在本节也将很有价值。通过理解黑客为入侵 Web 应用而寻找的常见漏洞，您将更好地理解可以部署哪些类型的防御来减轻此类攻击。对特定漏洞类别的了解也应有助于您优先进行修复，因为您将理解如果这些漏洞之一在您的 Web 应用中被发现，什么类型的数据将面临风险。

本书并非全面的全知参考，但它应提供足够的基础知识，供您寻求更多关于侦察、攻击和防御的信息。它应为您提供所需的基础，以理解如何就侦察技术、漏洞和缓解方法进行沟通。有了这些知识，您应该能够轻松加速您在软件安全领域的学习，并开始将您的自学导向您希望掌握的任何安全领域。

## 小结

保护现代 Web 应用需要广泛的技能。首先，Web 应用必须以架构上安全的方式进行设计。其次，在编写代码时，在发布到生产构建之前，必须经过安全代码审查等流程。接下来，需要引入发现、分析和管理漏洞的系统。最后，需要更新回归测试框架和工作流程，以确保相同的安全漏洞永远不会发生超过一次。
