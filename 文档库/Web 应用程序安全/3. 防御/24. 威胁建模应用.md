# 第 24 章：威胁建模应用

威胁建模是一种提高应用安全态势的宝贵方法——前提是实施正确。如果实施不当或仓促，威胁建模练习只会流于形式，浪费宝贵时间。

如果实施得当，威胁建模能够让软件工程师和安全工程师以凝聚的方式协同工作。一个良好实施的威胁模型所产生共享工程与安全工作流，使公司能够识别威胁（例如潜在漏洞）、威胁行为者（例如潜在黑客）、现有缓解措施（例如安全控制）以及威胁与缓解措施之间的差距。

一个良好实施的威胁建模过程，除了前述优点外，还作为一份动态文档，用于快速有效地将安全知识从一位工程师传递给另一位工程师。它以大多数其他方法无法实现的方式，增强了组织安全知识的稳健性。

虽然您不希望使用仓促实施的威胁建模过程，但学习如何设计和实施一个稳健的威胁建模工作流，从长远来看将显著改善您组织的安全态势。

## 设计有效的威胁模型

一个适当的威胁模型应能够实现多个目标。如果所有这些目标未能实现，威胁模型的影响力将受到限制：

- 记录知识
- 识别威胁行为者
- 识别缓解措施
- 识别风险（攻击向量）
- 识别差距（风险与缓解措施之间的差距）

这些主要目标中的每一个都建立在其他目标之上。例如，威胁模型作为一个动态的前向知识库，当其不仅包含有关应用架构的数据，还包含其威胁行为者、潜在威胁等信息时，价值最大。

同样，一个仅包含风险识别而不记录缓解措施的威胁模型也不尽如人意，因为风险无法付诸行动。尽管风险在某种程度上具有价值，但在了解现有缓解措施之前，风险是不可操作的。为已有缓解措施的风险实施新的缓解措施，是引入更多错误、技术债务，甚至可能向代码库添加更多漏洞的根源。

## 通过示例进行威胁建模

假设一家虚构的公司 MegaBank 正在实施一项新功能，用户可以通过该功能评价 MegaBank 上的现有功能。这将帮助 MegaBank 确定哪些功能受到欢迎，并向其他用户推广这些功能。

鉴于该功能将面向公众，且如果实施不当可能被滥用，MegaBank 要求其应用安全团队与开发者合作开发威胁模型。Holly Hacker 是 MegaBank 安全团队的高级成员，她看到了这一举措的价值，并且了解威胁模型的目标，于是开始与开发团队合作收集数据以生成该威胁模型。

### 逻辑设计

Holly 收集的第一份数据很简单：她请求从逻辑设计角度对新功能进行描述。对于软件产品而言，逻辑设计是指从功能角度对功能或产品的描述。这个视角比工程架构或设计文档更高层次，但比营销或销售描述更低层次。通常，此描述的详细程度与 UX 设计师使用的相同。UX 设计师必须在能够开发有效的 UI 之前理解应用逻辑。

Holly 得出的描述如下：

MegaBank“用户评价”功能是一个面向公众的功能，已认证用户将能够填写 Web 表单，并就任何现有的 MegaBank 功能发布评价。评价表单将包含一个 0 到 5 之间的分数，以及一个基于文本的评价正文。当提交评价时，MegaBank 服务器将把评价内容和分数存储在数据库中，之后用户可以在任何“激活功能”页面上点击“显示评价”来查询。这些页面的示例包括“激活加密货币钱包”或“激活银行贷款”功能。MegaBank 当前用户体验的设计是可选功能默认“关闭”，因此如果功能成功，“用户评价”功能将有助于促进新功能的启用。

——逻辑设计，MegaBank 用户评价威胁模型

逻辑设计概述对威胁模型很重要，但它是经常被忽视的组件之一。它之所以重要，是因为它之后可用于发现逻辑漏洞，即源于应用未遵循其预期功能逻辑的漏洞。这些类型的漏洞不同于传统的应用级漏洞，因为它们对每个应用及其特定的业务需求而言是独一无二的。

在本例中，逻辑漏洞的一个例子是用户能够通过绕过 Web 表单并发送自定义 HTTP 请求来发布大于 5 的分数。仅仅在数据库中存储整数值本身并不是漏洞。然而，在所给的边缘情况下，由于该功能的独特应用逻辑，它确实是一个漏洞。这是因为任何聚合或平均值都会被超出范围的数字所扭曲，误导用户并导致他们对 MegaBank 用户评价的真实性失去信任。这将损害 MegaBank 的品牌声誉。

### 技术设计

技术设计文档和收集是威胁建模过程中比逻辑设计文档和收集更常见的步骤。不幸的是，虽然技术设计对于发现漏洞和描述产品功能实现细节很有效，但它们通常不解释业务目标和逻辑。这就是为什么需要前面提到的逻辑设计数据收集。

话虽如此，技术设计数据收集和文档编制为了有效，有一些特殊要求，因此并不像简单抓取一些数据流图并放入文档中那么容易。

收集技术设计文档并将其汇总到威胁模型中的目标是让工程师和研究人员以后能够识别作为应用架构一部分产生的威胁。

技术设计应足够全面，以识别应用中的以下关键变量：

- 实现此功能使用了哪些工具和技术？
  - 编程语言
  - 数据库
  - 构建工具
  - 包管理器
  - 框架

- 在此功能的实现中使用了哪些第三方服务？
  - 网内第三方服务（例如 AWS 子资源，如 S3）
  - 网外第三方服务（例如 Datadog、CircleCI）

- 数据以何种方式从一个模块流向另一个模块（数据流图 [DFD] 在此处有用）？
  - 传输中的数据格式
  - 传输中的加密
  - 网络协议

- 此功能的网络配置方式（内部/VPN、外部等）？
  - 防火墙
  - VPN
  - 物理网络

- 此功能中使用了哪些认证和授权控制？
  - 是否允许多种类型的认证令牌？
  - 所有 API 端点是否执行相同类型的授权检查？

- 数据库模式是什么？（是否有任何数据受到监管或控制？）
  - 无模式的 NoSQL 数据库应列出所有可能的数据形状。

在 MegaBank 用户评价功能的案例中，我们虚构的安全工程师得出了以下（简化的）技术设计：

首先，我们现有的 React 用户界面将添加两个新组件。第一个组件是 `getReviews`，第二个是 `createReview`。这两个组件都将有相应的数据库条目和 REST API 端点。

`getReviews` 组件将向托管在 AWS EC2 上的 HTTP REST 服务器发起 HTTP GET 请求，然后该服务器将查询 PostgreSQL 数据库，查找与查询参数 `featureID` 中指定的功能相关的评价。与查询结果关联的 JSON 将被返回，除非未找到评价，此时将返回一个包含文本“未找到所选功能的评价”的错误。

`createReview` 组件将使用用户提交的表单信息发起 HTTP POST 请求，该请求将首先通过对照现有会话管理服务检查用户的会话 cookie 来验证用户是否已登录。如果此检查成功，关联的 AWS EC2 REST 端点将验证分数的类型为整数，评价的类型为文本，然后将其存储在 PostgreSQL 数据库中。

在客户端和 HTTP REST 服务器上使用的编程语言都是 JavaScript。HTTP REST 服务器将运行 ExpressJS，并通过 npm 包管理器使用开源模块 `body-parser` 和 `cookie-parser`。所有语言、框架和依赖项都运行在最新的稳定版本上。

此 HTTP 服务器所在的硬件将是公司 AWS 云中的 AWS EC2 实例。

通过网络发送的数据将使用 Let's Encrypt 生成的证书通过 TLS 1.3 加密。

所有 API 也可以通过 GraphQL 进行可选查询。

——技术设计，MegaBank 用户评价威胁模型

在这种粒度级别下，安全工程师可以开始将技术设计与逻辑设计进行对照评估，从而能够开始发现传统的 Web 应用漏洞和逻辑漏洞。

在本例中，文本的渲染方式未被注明，这意味着这可能成为 XSS 攻击向量。SQL 查询的生成逻辑也未被注明，考虑到用户提交的字符串正在被存储，这是危险的。除此之外，在错误消息中显示“未找到所选功能的评价”可能会导致信息泄露。如果某个功能尚未公开，遍历此 API 可能会泄露其存在。

通常，最好等到威胁模型的所有相关数据收集完毕后再进行潜在漏洞的头脑风暴，但通过适当的数据收集，这类漏洞会很快显现。在过程中注意到它们是有益的。

### 威胁识别（威胁行为者）

在逻辑设计和技术设计完成后，是时候让 Holly Hacker 开始考虑威胁行为者了。威胁行为者是典型的攻击者——每个攻击者都有独特的攻击系统路径（在本例中，是 MegaBank 用户评价功能）。

在考虑威胁行为者时，我们应该查看先前的设计文档，并开始头脑风暴每种类型的用户可能有权访问哪些权限层级。这些不仅仅是程序化的授权层级，还包括在任何组织中自然出现的内部人员权限。例如，考虑此假设应用的潜在攻击者和表 24-1 中显示的权限。

**表 24-1. 潜在威胁行为者**

| 威胁行为者 | 权力和权限，风险 |
| :--- | :--- |
| 用户管理员 | 可以使用管理员 UI 读取/更新数据库。可能利用权限窃取 PII。可以修改现有评级。目前，管理员工具的使用未得到适当记录。没有问责制。 |
| 客户支持用户 | 可以读取数据库数据，但不能更新字段。可能窃取 PII。 |
| 评价聚合器脚本 | 定期运行以计算多个评价的平均分数。作为数据库管理员直接访问数据库，无法访问其他工具或执行 Bash。如果被入侵，可以对数据库运行任何查询。重大损害潜力。 |
| 已认证用户 | 可以发布评价和评分。可以阅读其他用户的公开评价。可能尝试通过绕过 Web 表单并直接创建 HTTP POST 来提交恶意载荷，如 SQL 注入或溢出评价。 |
| 未认证（访客）用户 | 可以阅读评分。没有写入或更新访问权限。只要考虑到 DoS 或循环图查询，威胁较低。 |

如您所见，在编译所有潜在威胁行为者的列表后，我们得出了一些意想不到的结论。首先，威胁行为者列表应是应用用户的全面列表——不仅是人类，还包括机器驱动的用户。在本例中，Linux 用户“评价聚合器脚本”被添加到威胁行为者列表中，因为它自主运行，并且对支持此新功能的 SQL 数据库具有特权访问。

仅考虑直接的人类攻击是一个常见错误，而仅考虑外部人类用户则是一个更严重的错误。在编译威胁行为者列表时，确保包括内部、外部和机器用户。此外，如果可能，每个用户都应被分为认证和未认证权限集。例如，您的应用很可能对未认证的公共用户权限与已认证的公共用户权限处理方式不同。因此，在威胁建模练习中，每个都应单独考虑。

请注意以下明显但重要的细节：这些威胁行为者中的每一个都有不同的攻击面，即他们可以利用的攻击类型和方法。虽然这似乎是一个显而易见的细节，但不仅在威胁建模中，而且在开发过程中都要牢记这一点，因为它可以帮助您和组织适当地对风险进行排名，以便在处理危险性较低的威胁之前，识别并解决来自最危险威胁行为者的最重要攻击。

### 威胁识别（攻击向量）

在记录了潜在威胁行为者的知识后，现在该考虑针对您的应用的威胁了。我们可以将这些威胁归类为攻击向量——它们还不是漏洞，因为系统尚未开发。相反，它们是威胁行为者可能攻击您的系统的潜在（未来）路径。

通常，在威胁建模练习中头脑风暴攻击向量的最佳方式是对前述三个组件进行交叉分析：逻辑设计、技术设计和威胁行为者。

逻辑设计暴露了业务逻辑的组件，这些组件在查看技术设计时相关。例如，权限系统可能在技术设计中以实现方式定义，但没有逻辑设计，很难确定权限系统的实现方式是否可能被利用。这类系统中的边缘情况在生产应用中非常常见。

接下来，在评估攻击向量时，我们要考虑威胁行为者。并非所有攻击向量都适合所有威胁行为者。某些攻击向量需要某种特权访问；例如，内部 VPN 访问或登录仅限管理员服务器应用的凭据。通过不断考虑这些类别之间的关系，我们可以识别谁能够以及谁不能利用某个攻击向量。

因此，我们的攻击向量表结合了威胁模型中先前所有标题的知识，以及一个风险排名列（严重性），这将帮助我们优先安排修复的时间和资源。我们的最终表格类似于表 24-2 中总结的信息。

**表 24-2. 攻击向量**

| 威胁名称 | 严重性 a | 威胁行为者 | 描述 |
| :--- | :--- | :--- | :--- |
| 验证不当——分数 | P1 | 除访客用户外的所有用户 | 由于缺乏定义的验证，创建新评价的 POST 可以包含超出 0 和 5 范围的分数，导致聚合后评级偏差。 |
| 信息泄露——FeatureID | P3 | 所有用户 | 在 getReviews 端点上提供的错误消息允许用户扫描可能未发布的功能或他们不应知道存在的功能（受限）。 |
| SQL 注入 | P0 | 除访客用户外的所有用户 | POST 载荷上的不当实现可能导致针对 postReview 端点的 SQL 注入。 |
| GraphQL 循环和大查询 | P1 | 所有用户 | 循环和大查询成为可能，这在传统 REST 中是不可能的。这些不被速率限制阻止，需要最大查询时间来防止。 |
| GraphQL 内省和错误 | P1 | 所有用户 | 应禁用 GraphQL 的内省引擎，否则我们将泄露服务器配置细节。如果 GraphQL 内部错误未被抑制并替换为自定义错误，也会泄露服务器信息。 |
| 高特权用户攻击 | P0 | 管理员和评价聚合器脚本 | 特权令牌拥有读取/更新数据库的权限，但没有任何监控或问责。一个被入侵的令牌可能危及整个系统。 |

a P 代表优先级。P0–P4 是安全行业中常用的严重性标注，P0 最高，P4 最低。

如您所见，通过交叉引用威胁模型的前面部分，我们能够得出几个攻击向量，并将每个攻击向量与唯一的威胁行为者（或一组威胁行为者）相关联。攻击向量表仅用于示例目的，但它展示了几个关键点。

首先，许多攻击向量源自内部威胁行为者。从管理员用户到服务器上运行的内部脚本，您永远不能假设只有预期使用的功能会被调用。内部用户被黑客入侵、变节，或者在机器用户的情况下，发生故障或遇到错误。这些情况中的每一种都需要在威胁建模期间解决，否则系统将以“最佳情况”模型而非“最坏情况”模型运行。在处理安全问题时，始终应首先考虑最坏情况。

其次，我们可以看到这些攻击向量中的每一个都呈现出不同的严重性级别——即对公司或产品的风险。如何计算严重性将在后面的章节中深入讨论，但现在重要的是注意并非所有攻击向量都具有相同的风险。这是一件好事，因为当团队规模或工作能力有限时，它是一种有价值的优先安排工作的方法。

最后，我们注意到从 SQL 注入到逻辑漏洞的各种攻击。威胁建模帮助我们发现产品或功能面临哪些典型风险，这在购买工具或投入工程时间时可能有所帮助。如果没有此类数据，工时或资金可能会被投资不当，将产品或功能针对不太可能发生的攻击进行防护。

### 识别缓解措施

在威胁模型的第一部分，我们收集并组织了理解此新功能所需的所有数据。我们评估了逻辑设计，然后深入研究了技术设计。从数据收集阶段，我们识别了威胁行为者——可能攻击我们系统的用户。

在识别威胁行为者之后，我们使用收集到的信息来识别以攻击向量形式存在的风险。这些攻击向量代表了威胁行为者攻击我们系统的方法。

构建良好威胁模型的下一步是为每个发现的攻击向量识别缓解措施。这一步相当简单；它只是一个数据收集阶段。

我们示例功能的现有缓解措施如表 24-3 所示。

**表 24-3. 现有缓解措施**

| 缓解名称 | 缓解的攻击向量 | 描述 |
| :--- | :--- | :--- |
| 验证逻辑 | 验证不当——分数 | 我们的验证引擎拒绝任何不是整数或超出 0 和 5 范围的分数。这防止了聚合错误。 |
| SQL 注入缓解措施 | SQL 注入 | 所有调用 SQL 代码的调用都使用我们的领域特定语言（DSL），这强制使用预编译语句以防止 SQL 注入攻击。 |

从表中可以看出，此应用已有针对验证不当和 SQL 注入式攻击的现有缓解措施。我们在攻击向量识别期间注意到的其余攻击没有已知缓解措施。

| 信息泄露——FeatureID | P3 | 所有用户 | 在 getReviews 端点上提供的错误消息允许用户扫描可能未发布的功能或他们不应知道存在的功能（受限）。 |
| GraphQL 循环和大查询 | P1 | 所有用户 | 循环和大查询成为可能，这在传统 REST 中是不可能的。这些不被速率限制阻止，需要最大查询时间来防止。 |
| GraphQL 内省和错误 | P1 | 所有用户 | 应禁用 GraphQL 的内省引擎，否则我们将泄露服务器配置细节。如果 GraphQL 内部错误未被抑制并替换为自定义错误，也会泄露服务器信息。 |
| 高特权用户攻击 | P0 | 管理员和评价聚合器脚本 | 特权令牌拥有读取/更新数据库的权限，但没有任何监控或问责。一个被入侵的令牌可能危及整个系统。 |

对于这些未缓解的攻击向量中的每一个，我们都必须头脑风暴一个充分的缓解措施。如果没有为每个未缓解的攻击向量提供充分的缓解措施，该功能不应被允许上线。表 24-5 列出了为已识别的攻击向量所制定的缓解措施。

**表 24-5. 针对已识别攻击向量的缓解措施**

| 缓解名称 | 缓解的攻击向量 | 描述 |
| :--- | :--- | :--- |
| 更好的错误消息 | 信息泄露——FeatureID | 错误消息应通用，不提供有关我们应用或数据的任何信息。 |
| GraphQL 计算限制 | GraphQL 循环和大查询 | 由于 GraphQL 查询可能变得昂贵且自引用，除了速率限制外，我们必须为每个查询实施最大计算时间。在计算时间结束时，请求将失败。 |
| GraphQL 配置 | GraphQL 内省和错误 | GraphQL 不得显示任何其自身的错误消息，并且必须关闭内省查询。 |
| 特权权限重构 | 高特权用户攻击 | 为防止被入侵用户或身份验证令牌造成的损害，特权用户令牌的作用域将仅限于该用户所需的功能。此外，每个 API 调用将使用特权用户无法撤销或删除日志的机制在平台外进行记录。此外，评价聚合器脚本将仅在数据库的一列（聚合分数）中具有写入权限，在另一列（分数）中具有读取权限。 |

在记录了这些缓解措施并到位之后，下一步是在每个缓解措施实施后将其标记为已完成，并将威胁模型指向实现。一旦所有缓解措施都已实施，威胁模型的即时目的就完成了，功能的开发可以继续推进。

然而，如果未来功能范围扩大，那么就该回到威胁模型了。此时，我们将更新前向记录的知识，更新威胁行为者和威胁向量，并重新评估缓解措施和差异。

## 小结

如您所见，威胁模型是一个非常全面的安全工具。一个正确实施的威胁模型能够在一个集中存储库中记录技术知识和安全知识，使其成为一个长期有效的信息来源。

此外，威胁建模练习本身非常能够在应用开发之前识别威胁行为者（攻击者）和攻击向量（潜在漏洞）。

最后，通过识别攻击向量并将其与现有缓解措施并列列出，可以在开发阶段的早期（此时修复此类问题成本最低）开始完善系统的安全态势。有人说，架构阶段 10 小时的修复工作相当于发布后 100 小时的修复工作。如果这是真的，那么一个好的威胁模型可能是任何安全工程师工具箱中最具成本效益的安全工具之一。
