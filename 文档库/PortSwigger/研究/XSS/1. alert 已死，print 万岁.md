# alert() 已死，print() 万岁

[跨站脚本（XSS）](/web-security/cross-site-scripting) 与 `alert()` 函数几十年来一直形影不离。想证明你能执行任意 JavaScript？弹一个 alert。想懒人式地找 XSS 漏洞？到处注入调用 `alert()` 的载荷，看有没有东西弹出来。

然而，地平线上正酝酿着麻烦。恶意广告一直在滥用我们心爱的 `alert`，从其 iframe 内部干扰和社交工程访问者。Google Chrome 决定通过[禁用跨域 iframe 中的 `alert`](https://chromestatus.com/feature/5148698084376576) 来应对这一问题。跨域 iframe 常常被故意内嵌到网站中，同时也是一些[相对高级的 XSS 攻击](https://portswigger.net/web-security/dom-based/controlling-the-web-message-source) 几乎必不可少的组件。

2021 年 7 月 20 日 Chrome 92 发布后，跨域 iframe 内部的 XSS 漏洞将：

- 不再支持基于 `alert()` 的概念验证（PoC）。
- 对任何使用基于 `alert()` 的检测技术的人来说变得不可见。

接下来怎么办？显而易见的变通方法是使用 `prompt` 或 `confirm`，但不幸的是 Chrome 的拦截措施会阻止所有对话框。触发 [DNS 回显](https://portswigger.net/research/hunting-asynchronous-vulnerabilities) 到监听器（[OAST](https://portswigger.net/burp/application-security-testing/oast) 风格）是另一种可行方法，但由于配置要求，不太适合作为 PoC。我们也排除了 `console.log()`，因为控制台函数经常被代理或由 JavaScript 混淆器禁用。

很有趣的是，这种“保护”（阻止跨域显示对话框）拦截了 `alert` 和 `prompt`，但正如 [Yosuke Hasegawa 指出的](https://twitter.com/hasegawayosuke/status/1410963117236916227)，他们忘了基础认证。这在当前的 Canary 版本中仍然有效，不过将来很可能会被拦截。

我们需要一个 `alert` 的替代品，它要满足：

- 简单、免配置、容易记忆
- 高度可见，即使是在不可见的 iframe 中执行

经过数周的深入研究，我们很高兴为您带来……

## `print()`

![print 示意](/cms/images/5d/6a/cb54-article-print-cropped.png)

我们将在近期更新 Web Security Academy 的实验，以支持基于 `print()` 的解决方案。[XSS 速查表](/web-security/cross-site-scripting/cheat-sheet) 也将同步更新，以反映在跨域 iframe 中使用新的 `print()` 载荷。在没有 iframe 的情况下，我们暂时仍会继续使用 `alert`。

**print 万岁！**
