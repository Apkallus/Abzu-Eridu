# XSS in hidden input fields（隐藏输入字段中的 XSS）

https://portswigger.net/research/xss-in-hidden-input-fields

> 作者：**Gareth Heyes**（研究员）  
> Twitter：[@garethheyes](https://twitter.com/garethheyes)

- **发布时间：** 2015 年 11 月 16 日（星期一）11:25 UTC
- **更新日期：** 2019 年 6 月 14 日（星期五）12:03 UTC

在 PortSwigger，我们会定期对内部测试平台中的流行 Web 应用程序运行 Burp Suite 的预发布版本，以确保其行为正常。最近在这样做时，[Liam](https://twitter.com/MetalF0X) 在 [已删除] 的一个隐藏输入元素中发现了一个[跨站脚本](/web-security/cross-site-scripting)（[XSS](/web-security/cross-site-scripting)）漏洞：

```html
<input type="hidden" name="redacted" value="default" injection="xss" />
```

隐藏输入中的 XSS [通常非常难以利用](https://web.archive.org/web/20130724031616/http://sla.ckers.org/forum/read.php?2,17217)，因为典型的 JavaScript 事件（如 `onmouseover` 和 `onfocus`）由于元素不可见而无法触发。

我决定进一步研究，看看是否有可能在现代浏览器上利用它。我尝试了一堆东西，比如 `autofocus`、CSS 技巧等等。最终我想到了访问键（access key），想知道当通过访问键激活时，是否会在隐藏输入上调用 `onclick` 事件。在 Firefox 上确实如此！这意味着我们可以执行隐藏属性中的 XSS 载荷，前提是你能诱使受害者按下组合键。在 Firefox Windows/Linux 上，组合键是 `ALT+SHIFT+X`，在 OS X 上是 `CTRL+ALT+X`。您可以使用访问键属性中的不同键来指定不同的组合键。这里是向量：

```html
<input type="hidden" accesskey="X" onclick="alert(1)">
```

这个向量并不理想，因为它涉及一些用户交互，但它比 `expression()` 好得多（后者仅适用于 IE<=9）。请注意，如果您的反射被重复，则组合键将失效。一种解决方法是在注入另一个属性来破坏第二次反射，例如：`" accesskey="x" onclick="alert(1)" x='`

### 更新 - 现已在 Chrome 以及 link、meta 和其他任何元素上工作

这项技术现在在 Chrome 中也可以工作！它还可以在 `link` 元素中工作，这意味着以前无法利用的 `link` 元素中的 XSS 漏洞（您只能控制属性）现在可以使用此技术进行利用。例如，您可能有一个带有 `rel="canonical"` 的 `link` 元素，如果您注入 `accesskey` 属性和 `onclick` 事件，那么您就拥有了 XSS。

```html
<link rel="canonical" accesskey="X" onclick="alert(1)" />
```
