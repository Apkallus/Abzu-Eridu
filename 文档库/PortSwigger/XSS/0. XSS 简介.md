# 跨站脚本攻击（Cross‑site scripting）

在本节中，我们将说明什么是跨站脚本攻击，描述跨站脚本漏洞的不同类型，并详细阐述如何发现和预防跨站脚本攻击。

## 什么是跨站脚本攻击（XSS）？

跨站脚本攻击（也称为 XSS）是一种 Web 安全漏洞，允许攻击者破坏用户与易受攻击的应用程序之间的交互。它允许攻击者绕过同源策略，该策略旨在将不同网站彼此隔离。跨站脚本漏洞通常允许攻击者伪装成受害用户，执行该用户有权执行的任何操作，并访问该用户的任何数据。如果受害用户在应用程序中拥有特权访问权限，则攻击者可能能够完全控制应用程序的所有功能和数据。

## XSS 是如何工作的？

跨站脚本攻击通过操纵易受攻击的网站，使其向用户返回恶意 JavaScript。当恶意代码在受害者的浏览器中执行时，攻击者就可以完全破坏其与应用程序的交互。

## XSS 概念验证

您可以通过注入一个能让您自己的浏览器执行任意 JavaScript 的载荷来确认大多数类型的 XSS 漏洞。长期以来，通常使用 `alert()` 函数来实现此目的，因为它简短、无害，并且在成功调用时非常显眼。事实上，您可以通过在模拟受害者的浏览器中调用 `alert()` 来解决我们大多数 XSS 实验室。

但不幸的是，如果您使用 Chrome 浏览器，会有一个小问题。从 92 版本（2021 年 7 月 20 日）开始，跨域 iframe 被阻止调用 `alert()`。由于这些 iframe 用于构建一些更高级的 XSS 攻击，您有时需要使用替代的 PoC 载荷。在这种情况下，我们推荐使用 `print()` 函数。如果您想了解有关此更改的更多信息以及我们为何推荐 `print()`，请查阅我们关于此主题的[博客文章](https://portswigger.net/research/alert-is-dead-long-live-print)。

由于我们实验室中的模拟受害者使用 Chrome，我们已修改了受影响的实验室，以便它们也可以使用 `print()` 来解决。我们已在相关说明中予以注明。

## XSS 攻击有哪些类型？

主要有三种类型的 XSS 攻击，分别是：

- [反射型 XSS](#反射型跨站脚本攻击) – 恶意脚本来自当前的 HTTP 请求。
- [存储型 XSS](#存储型跨站脚本攻击) – 恶意脚本来自网站的数据库。
- [基于 DOM 的 XSS](#基于-dom-的跨站脚本攻击) – 漏洞存在于客户端代码中，而非服务器端代码。

## 反射型跨站脚本攻击

反射型 XSS 是最简单的一种跨站脚本攻击。它发生在应用程序从 HTTP 请求中接收数据，并以不安全的方式将该数据包含在即时响应中时。

下面是一个简单的反射型 XSS 漏洞示例：

```
https://insecure-website.com/status?message=All+is+well.
<p>Status: All is well.</p>
```

应用程序不对数据执行任何其他处理，因此攻击者可以轻松构建如下攻击：

```
https://insecure-website.com/status?message=<script>/* Bad stuff here... */</script>
<p>Status: <script>/* Bad stuff here... */</script></p>
```

如果用户访问了攻击者构建的 URL，那么攻击者的脚本将在用户的浏览器中执行，并在该用户与应用程序的会话上下文中运行。此时，脚本可以执行任何操作，并检索该用户有权访问的任何数据。

## 存储型跨站脚本攻击

存储型 XSS（也称为持久型或二阶 XSS）发生在应用程序从不可信源接收数据，并以不安全的方式将该数据包含在后续的 HTTP 响应中时。

相关数据可能通过 HTTP 请求提交给应用程序，例如博客文章评论、聊天室中的用户昵称或客户订单中的联系方式。在其他情况下，数据可能来自其他不可信源，例如显示通过 SMTP 接收的消息的 Web 邮件应用程序、显示社交媒体帖子的营销应用程序，或显示网络流量数据包数据的网络监控应用程序。

下面是一个简单的存储型 XSS 漏洞示例。一个留言板应用程序允许用户提交消息，这些消息会显示给其他用户：

```
<p>Hello, this is my message!</p>
```

应用程序不对数据执行任何其他处理，因此攻击者可以轻松发送一条攻击其他用户的消息：

```
<p><script>/* Bad stuff here... */</script></p>
```

## 基于 DOM 的跨站脚本攻击

基于 DOM 的 XSS（也称为 DOM XSS）发生在应用程序包含某些客户端 JavaScript，这些 JavaScript 以不安全的方式处理来自不可信源的数据，通常是将数据写回 DOM。

在以下示例中，应用程序使用一些 JavaScript 读取输入字段的值，并将该值写入 HTML 中的某个元素：

```javascript
var search = document.getElementById('search').value;
var results = document.getElementById('results');
results.innerHTML = 'You searched for: ' + search;
```

如果攻击者可以控制输入字段的值，他们可以轻松构造一个恶意值，导致自己的脚本执行：

```
You searched for: <img src=1 onerror='/* Bad stuff here... */'>
```

在典型情况下，输入字段的值来自 HTTP 请求的某个部分（例如 URL 查询字符串参数），从而使攻击者能够像反射型 XSS 一样，使用恶意 URL 发起攻击。

## XSS 可以用于什么？

利用跨站脚本漏洞的攻击者通常能够：

- 冒充或伪装成受害用户。
- 执行该用户能够执行的任何操作。
- 读取该用户能够访问的任何数据。
- 捕获用户的登录凭据。
- 对网站进行虚拟篡改。
- 向网站注入木马功能。

## XSS 漏洞的影响

XSS 攻击的实际影响通常取决于应用程序的性质、其功能和数据，以及受感染用户的状态。例如：

- 在纯信息展示型应用程序中，所有用户均为匿名且所有信息都是公开的，影响通常微乎其微。
- 在包含敏感数据的应用程序中（如银行交易、电子邮件或医疗记录），影响通常是严重的。
- 如果受感染用户在应用程序中拥有较高权限，则影响通常是致命的，攻击者可完全控制易受攻击的应用程序，并危害所有用户及其数据。

## 如何发现和测试 XSS 漏洞

绝大多数 XSS 漏洞都可以使用 Burp Suite 的 Web 漏洞扫描器快速可靠地发现。

手动测试反射型和存储型 XSS 通常涉及向应用程序的每个入口点提交一些简单且唯一的输入（例如一个短字母数字字符串），识别提交的输入在 HTTP 响应中返回的所有位置，然后分别测试每个位置，以确定是否可以使用精心构造的输入执行任意 JavaScript。通过这种方式，您可以确定 XSS 出现的[上下文](/web-security/cross-site-scripting/contexts)，并选择合适的载荷进行利用。

手动测试由 URL 参数引起的基于 DOM 的 XSS 涉及类似的过程：在参数中放置一些简单唯一的输入，使用浏览器的开发者工具在 DOM 中搜索该输入，并测试每个位置以确定其是否可利用。然而，其他类型的 DOM XSS 更难检测。要发现基于非 URL 输入（如 `document.cookie`）或非 HTML 接收器（如 `setTimeout`）的 DOM 漏洞，除了审查 JavaScript 代码外别无他法，而这可能非常耗时。Burp Suite 的 Web 漏洞扫描器结合了 JavaScript 的静态和动态分析，能够可靠地自动化检测基于 DOM 的漏洞。

## 内容安全策略

内容安全策略（CSP）是一种浏览器机制，旨在减轻跨站脚本攻击和其他一些漏洞的影响。如果采用 CSP 的应用程序存在类似 XSS 的行为，则 CSP 可能会阻碍或阻止漏洞的利用。通常，CSP 可以被绕过，从而利用底层漏洞。

## 悬挂标记注入

悬挂标记注入是一种技术，在由于输入过滤器或其他防御措施而无法实现完整的跨站脚本攻击时，可用于跨域捕获数据。它通常可用于捕获其他用户可见的敏感信息，包括可用于代表用户执行未经授权操作的 CSRF 令牌。

## 如何预防 XSS 攻击

在某些情况下，预防跨站脚本攻击是微不足道的，但根据应用程序的复杂性及其处理用户可控数据的方式，可能会困难得多。

总体而言，有效预防 XSS 漏洞可能需要结合以下措施：

- **对输入进行过滤。** 在接收用户输入时，根据预期或有效输入尽可能严格地进行过滤。
- **对输出进行编码。** 在将用户可控数据输出到 HTTP 响应时，对输出进行编码，以防止其被解释为活动内容。根据输出上下文，可能需要结合使用 HTML、URL、JavaScript 和 CSS 编码。
- **使用适当的响应头。** 对于不打算包含任何 HTML 或 JavaScript 的 HTTP 响应，可以使用 `Content-Type` 和 `X-Content-Type-Options` 头，确保浏览器按您预期的方式解释响应。
- **内容安全策略。** 作为最后一道防线，您可以使用内容安全策略（CSP）来降低仍然发生的 XSS 漏洞的严重性。

## 关于跨站脚本攻击的常见问题

**XSS 漏洞有多常见？**  
XSS 漏洞非常常见，XSS 可能是最常出现的 Web 安全漏洞。

**XSS 攻击有多常见？**  
很难获得关于真实世界 XSS 攻击的可靠数据，但它可能比其他漏洞被利用的频率低。

**XSS 和 CSRF 有什么区别？**  
XSS 涉及导致网站返回恶意 JavaScript，而 CSRF 涉及诱导受害用户执行他们本无意执行的操作。

**XSS 和 SQL 注入有什么区别？**  
XSS 是针对其他应用程序用户的客户端漏洞，而 SQL 注入是针对应用程序数据库的服务器端漏洞。

**如何在 PHP 中预防 XSS？**  
使用白名单允许的字符过滤输入，并使用类型提示或类型转换。对于 HTML 上下文，使用 `htmlentities` 和 `ENT_QUOTES` 对输出进行转义；对于 JavaScript 上下文，使用 JavaScript Unicode 转义。

**如何在 Java 中预防 XSS？**  
使用白名单允许的字符过滤输入，并使用 Google Guava 等库对 HTML 上下文的输出进行 HTML 编码，或对 JavaScript 上下文使用 JavaScript Unicode 转义。
