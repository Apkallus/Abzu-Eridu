# DOM-based XSS（基于 DOM 的跨站脚本）

在本节中，我们将介绍基于 DOM 的跨站脚本（DOM XSS），解释如何发现 DOM XSS 漏洞，并讨论如何利用不同的源和接收器来利用 DOM XSS。

## 什么是基于 DOM 的跨站脚本？

DOM-based XSS 漏洞通常出现在 JavaScript 从攻击者可控制的源（如 URL）获取数据，并将其传递给支持动态代码执行的接收器（如 `eval()` 或 `innerHTML`）时。这使得攻击者能够执行恶意 JavaScript，通常可用来劫持其他用户的账户。

要实施 DOM-based XSS 攻击，您需要将数据放入某个源，使其传播到接收器并导致任意 JavaScript 的执行。

DOM XSS 最常见的源是 URL，通常通过 `window.location` 对象访问。攻击者可以构造一个链接，将受害者发送到易受攻击的页面，并在 URL 的查询字符串和片段部分放置载荷。在某些情况下（例如针对 404 页面或运行 PHP 的网站），载荷也可以放在路径中。

关于源和接收器之间污点传播的详细说明，请参阅[基于 DOM 的漏洞](/web-security/dom-based)页面。

## 如何测试基于 DOM 的跨站脚本

绝大多数 DOM XSS 漏洞可以使用 Burp Suite 的 Web 漏洞扫描器快速可靠地发现。要手动测试基于 DOM 的跨站脚本，通常需要使用带有开发者工具的浏览器（如 Chrome）。您需要依次遍历每个可用的源，并逐一测试。

### 测试 HTML 接收器

要测试 HTML 接收器中的 DOM XSS，请将一个随机字母数字字符串放入源（如 `location.search`），然后使用开发者工具检查 HTML，找到字符串出现的位置。请注意，浏览器的“查看源代码”选项不适用于 DOM XSS 测试，因为它不会考虑 JavaScript 对 HTML 所做的更改。在 Chrome 的开发者工具中，您可以使用 `Control+F`（MacOS 上为 `Command+F`）在 DOM 中搜索您的字符串。

对于字符串在 DOM 中出现的每个位置，您需要识别上下文。基于此上下文，您需要调整输入以查看其处理方式。例如，如果字符串出现在双引号属性内，请尝试在字符串中注入双引号，看是否能够突破该属性。

请注意，浏览器在 URL 编码方面的行为有所不同。Chrome、Firefox 和 Safari 会对 `location.search` 和 `location.hash` 进行 URL 编码，而 IE11 和 Microsoft Edge（基于 Chromium 之前）则不会对这些源进行 URL 编码。如果您的数据在处理前被 URL 编码，那么 XSS 攻击不太可能成功。

### 测试 JavaScript 执行接收器

测试 JavaScript 执行接收器中的 DOM XSS 稍微困难一些。对于这些接收器，您的输入不一定出现在 DOM 中的任何位置，因此您无法搜索它。相反，您需要使用 JavaScript 调试器来确定您的输入是否以及如何被发送到接收器。

对于每个潜在的源（如 `location`），您首先需要在页面的 JavaScript 代码中找到引用该源的情况。在 Chrome 的开发者工具中，您可以使用 `Control+Shift+F`（MacOS 上为 `Command+Alt+F`）在所有页面 JavaScript 代码中搜索该源。

一旦找到读取源的位置，您可以使用 JavaScript 调试器添加断点，并跟踪源值的使用方式。您可能会发现该源被赋值给其他变量。如果是这种情况，您需要再次使用搜索功能来跟踪这些变量，并检查它们是否被传递给接收器。当您发现某个接收器被赋了来自源的数据时，您可以使用调试器悬停变量以查看其在发送到接收器之前的值的。然后，与 HTML 接收器类似，您需要调整输入，看能否成功实施 XSS 攻击。

### 使用 DOM Invader 测试 DOM XSS

## 利用不同源和接收器的 DOM XSS

原则上，如果存在一条从源到接收器的可执行路径，则网站容易受到基于 DOM 的跨站脚本攻击。实际上，不同的源和接收器具有不同的属性和行为，这些会影响可利用性，并决定需要使用哪些技术。此外，网站的脚本可能对数据执行验证或其他处理，在尝试利用漏洞时必须考虑这些因素。有多种与基于 DOM 的漏洞相关的接收器。有关详细信息，请参阅下面的[列表](#which-sinks-can-lead-to-dom-xss-vulnerabilities)。

`document.write` 接收器可与 `script` 元素配合使用，因此您可以使用简单的载荷，如下所示：

```javascript
document.write('... <script>alert(document.domain)</script> ...');
```

`innerHTML` 接收器在任何现代浏览器上都不接受 `script` 元素，也不会触发 `svg onload` 事件。这意味着您需要使用其他元素，如 `img` 或 `iframe`。事件处理程序（如 `onload` 和 `onerror`）可与这些元素结合使用。例如：

```javascript
element.innerHTML='... <img src=1 onerror=alert(document.domain)> ...'
```

### 第三方依赖中的源和接收器

现代 Web 应用程序通常使用许多第三方库和框架构建，这些通常为开发人员提供额外的功能。请务必记住，其中一些也可能是 DOM XSS 的潜在源和接收器。

#### jQuery 中的 DOM XSS

如果使用了诸如 jQuery 之类的 JavaScript 库，请留意能够更改页面上 DOM 元素的接收器。例如，jQuery 的 `attr()` 函数可以更改 DOM 元素的属性。如果从用户控制的源（如 URL）读取数据，然后传递给 `attr()` 函数，则可能操纵发送的值以导致 XSS。例如，以下 JavaScript 使用 URL 中的数据更改锚点元素的 `href` 属性：

```javascript
$(function() {
	$('#backLink').attr("href",(new URLSearchParams(window.location.search)).get('returnUrl'));
});
```

您可以通过修改 URL，使 `location.search` 源包含恶意 JavaScript URL 来利用此漏洞。在页面的 JavaScript 将此恶意 URL 应用到返回链接的 `href` 之后，单击返回链接将执行它：

```
?returnUrl=javascript:alert(document.domain)
```

> 状态：未解决

另一个需要注意的潜在接收器是 jQuery 的 `$()` 选择器函数，它可用于将恶意对象注入 DOM。

jQuery 曾经非常流行，一个经典的 DOM XSS 漏洞是由网站将此选择器与 `location.hash` 源结合用于动画或自动滚动到页面上特定元素引起的。这种行为通常使用易受攻击的 `hashchange` 事件处理程序实现，类似于以下内容：

```javascript
$(window).on('hashchange', function() {
	var element = $(location.hash);
	element[0].scrollIntoView();
});
```

由于 hash 是用户可控的，攻击者可以利用它在 `$()` 选择器接收器中注入 XSS 向量。较新版本的 jQuery 已通过阻止在输入以哈希字符（`#`）开头时向选择器注入 HTML 来修补此特定漏洞。但是，您可能仍然会在实际中遇到易受攻击的代码。

要实际利用此经典漏洞，您需要找到一种无需用户交互即可触发 `hashchange` 事件的方法。最简单的方法之一是通过 `iframe` 传递您的漏洞利用：

```html
<iframe src="https://vulnerable-website.com#" onload="this.src+='<img src=1 onerror=alert(1)>'"></iframe>
```

在此示例中，`src` 属性指向带有空哈希值的易受攻击页面。当 `iframe` 加载时，XSS 向量被附加到哈希，导致 `hashchange` 事件触发。

> **注意：** 即使更新版本的 jQuery，只要您能完全控制来自不需要 `#` 前缀的源的输入，仍可能通过 `$()` 选择器接收器受到攻击。

## DOM XSS 与反射和存储数据的结合

某些纯基于 DOM 的漏洞是自包含在单个页面中的。如果脚本从 URL 读取某些数据并将其写入危险接收器，则该漏洞完全是客户端的。

然而，源不仅限于浏览器直接暴露的数据——它们也可以来自网站本身。例如，网站通常会在来自服务器的 HTML 响应中反射 URL 参数。这通常与普通 XSS 相关，但也可能导致反射型 DOM XSS 漏洞。

在反射型 DOM XSS 漏洞中，服务器处理请求中的数据，并将数据回显到响应中。反射的数据可能被放入 JavaScript 字符串字面量中，或 DOM 中的数据项（如表单字段）中。随后页面上的脚本以不安全的方式处理反射的数据，最终将其写入危险的接收器。

```javascript
eval('var data = "reflected string"');
```

网站也可能在服务器上存储数据并在其他地方反射它。在存储型 DOM XSS 漏洞中，服务器从一个请求接收数据，存储它，然后将数据包含在后续响应中。后续响应中的脚本包含一个接收器，它以不安全的方式处理数据。

```javascript
element.innerHTML = comment.author
```

## 哪些接收器可能导致 DOM-XSS 漏洞？

以下是一些可能导致 DOM-XSS 漏洞的主要接收器：

```
document.write()
document.writeln()
document.domain
element.innerHTML
element.outerHTML
element.insertAdjacentHTML
element.onevent
```

以下 jQuery 函数也是可能导致 DOM-XSS 漏洞的接收器：

```
add()
after()
append()
animate()
insertAfter()
insertBefore()
before()
html()
prepend()
replaceAll()
replaceWith()
wrap()
wrapInner()
wrapAll()
has()
constructor()
init()
index()
jQuery.parseHTML()
$.parseHTML()
```

## 如何防止 DOM-XSS 漏洞

除了[基于 DOM 的漏洞](/web-security/dom-based)页面中描述的通用措施外，您还应避免允许来自任何不可信源的数据动态写入 HTML 文档。
