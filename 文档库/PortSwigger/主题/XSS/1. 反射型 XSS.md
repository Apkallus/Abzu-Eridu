# 反射型 XSS

在本节中，我们将解释反射型跨站脚本，描述反射型 XSS 攻击的影响，并详细说明如何发现反射型 XSS 漏洞。

## 什么是反射型跨站脚本？

反射型跨站脚本（或 XSS）当应用程序在 HTTP 请求中接收数据，并以不安全的方式将该数据包含在即时响应中时产生。

假设一个网站具有搜索功能，它在 URL 参数中接收用户提供的搜索词：

```
https://insecure-website.com/search?term=gift
```

应用程序将该搜索词原样回显到响应的页面中：

```html
<p>You searched for: gift</p>
```

假设应用程序不对数据执行任何其他处理，攻击者可以构建如下攻击：

```
https://insecure-website.com/search?term=<script>/*+Bad+stuff+here...+*/</script>
```

该 URL 会产生以下响应：

```html
<p>You searched for: <script>/* Bad stuff here... */</script></p>
```

如果应用程序的另一个用户请求了攻击者构造的 URL，那么攻击者提供的脚本将在受害用户的浏览器中，以该用户与应用程序的会话上下文执行。

## 反射型 XSS 攻击的影响

如果攻击者可以控制在受害者浏览器中执行的脚本，那么他们通常可以完全控制该用户。攻击者可以：

- 在应用程序内执行用户能够执行的任何操作。
- 查看用户能够查看的任何信息。
- 修改用户能够修改的任何信息。
- 与其他应用程序用户发起交互（包括恶意攻击），这些交互将看似来自初始受害用户。

攻击者可以通过多种方式诱导受害用户访问其控制的请求，以实施反射型 XSS 攻击。这些方式包括：在攻击者控制的网站上放置链接，或在允许生成内容的其他网站上放置链接，或通过电子邮件、推文或其他消息发送链接。攻击可以直接针对已知用户，也可以是对应用程序所有用户的无差别攻击。

由于攻击需要外部传递机制，反射型 XSS 的影响通常低于存储型 XSS，因为存储型 XSS 可以在易受攻击的应用程序自身内独立完成攻击。

## 不同上下文中的反射型 XSS

反射型跨站脚本有许多不同类型。反射数据在应用程序响应中的位置决定了需要哪种类型的载荷来利用它，并且还可能影响漏洞的影响。

此外，如果应用程序在反射数据之前对提交的数据执行了任何验证或其他处理，这通常会影响所需的 XSS 载荷类型。

## 如何发现和测试反射型 XSS 漏洞

绝大多数反射型跨站脚本漏洞可以使用 Burp Suite 的 Web 漏洞扫描器快速可靠地发现。

手动测试反射型 XSS 漏洞包括以下步骤：

- **测试每个入口点。** 分别测试应用程序 HTTP 请求中的每个数据入口点，包括 URL 查询字符串、消息正文中的参数或其他数据，以及 URL 文件路径。也包括 HTTP 标头，但仅能通过某些 HTTP 标头触发的类似 XSS 的行为可能在实际中无法利用。

- **提交随机字母数字值。** 对于每个入口点，提交一个唯一的随机值，并确定该值是否在响应中被反射。该值应设计为能够绕过大多数输入验证，因此需要相当短且仅包含字母数字字符。但它需要足够长，使得在响应中意外匹配的可能性极低。大约 8 个字符的随机字母数字值通常是理想的。您可以使用 Burp Intruder 的[数字载荷](https://portswigger.net/burp/documentation/desktop/tools/intruder/payloads/types#numbers)（随机生成的十六进制值）来生成合适的随机值。您还可以使用 Burp Intruder 的[载荷匹配设置](/burp/documentation/desktop/tools/intruder/configure-attack/settings#grep-payloads)自动标记包含提交值的响应。

- **确定反射上下文。** 对于响应中反射随机值的每个位置，确定其上下文。这可能是在 HTML 标签之间的文本中、在带引号的标签属性内、在 JavaScript 字符串中等等。

- **测试候选载荷。** 根据反射的上下文，测试一个初始的候选 XSS 载荷，如果它未被修改地反射在响应中，该载荷将触发 JavaScript 执行。测试载荷的最简单方法是将请求发送到 [Burp Repeater](/burp/documentation/desktop/tools/repeater)，修改请求插入候选载荷，发出请求，然后检查响应以查看载荷是否生效。一种高效的工作方式是保留原始随机值在请求中，并将候选 XSS 载荷放在其前后。然后将随机值设置为 Burp Repeater 响应视图中的搜索词，Burp 将高亮显示搜索词出现的每个位置，让您快速定位反射点。

- **测试替代载荷。** 如果候选 XSS 载荷被应用程序修改或完全阻止，您将需要测试替代载荷和技术，这些载荷和技术可能根据反射的上下文和所执行输入验证的类型，实现有效的 XSS 攻击。更多详情，请参阅[跨站脚本上下文](/web-security/cross-site-scripting/contexts)。

- **在浏览器中测试攻击。** 最后，如果您成功找到了一个在 Burp Repeater 中似乎有效的载荷，请将攻击转移到真实浏览器（通过将 URL 粘贴到地址栏，或在 [Burp Proxy 的拦截视图](/burp/documentation/desktop/tools/proxy/intercept-messages)中修改请求），然后查看注入的 JavaScript 是否确实执行。通常，最好执行一些简单的 JavaScript，如 `alert(document.domain)`，如果攻击成功，它将在浏览器中触发可见的弹窗。

## 关于反射型跨站脚本的常见问题

**反射型 XSS 和存储型 XSS 之间有什么区别？**  
反射型 XSS 当应用程序从 HTTP 请求中获取某些输入，并以不安全的方式将该输入嵌入到即时响应中时产生。而存储型 XSS 则是应用程序存储该输入，并以不安全的方式将其嵌入到后续响应中。

**反射型 XSS 和自我型 XSS 之间有什么区别？**  
自我型 XSS（Self-XSS）涉及与常规反射型 XSS 类似的应用程序行为，但无法通过精心构造的 URL 或跨域请求以常规方式触发。相反，该漏洞仅在受害者自己从浏览器提交 XSS 载荷时才会触发。实施自我型 XSS 攻击通常需要社交工程手段，诱使受害者将攻击者提供的输入粘贴到他们的浏览器中。因此，它通常被认为是一个价值较低、影响不大的问题。
