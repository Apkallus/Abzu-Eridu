# 存储型 XSS

在本节中，我们将解释存储型跨站脚本，描述存储型 XSS 攻击的影响，并详细说明如何发现存储型 XSS 漏洞。

## 什么是存储型跨站脚本？

存储型跨站脚本（也称为二阶或持久型 XSS）当应用程序从不可信源接收数据，并以不安全的方式将该数据包含在其后续 HTTP 响应中时产生。

假设一个网站允许用户对博客文章提交评论，这些评论会显示给其他用户。用户通过如下 HTTP 请求提交评论：

```
POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Length: 100

postId=3&comment=This+post+was+extremely+helpful.&name=Carlos+Montoya&email=carlos%40normal-user.net
```

该评论提交后，任何访问该博客文章的用户将在应用程序的响应中看到以下内容：

```html
<p>This post was extremely helpful.</p>
```

假设应用程序不对数据执行任何其他处理，攻击者可以提交如下恶意评论：

```html
<script>/* Bad stuff here... */</script>
```

在攻击者的请求中，该评论将被 URL 编码为：

```
comment=%3Cscript%3E%2F*%2BBad%2Bstuff%2Bhere...%2B*%2F%3C%2Fscript%3E
```

任何访问该博客文章的用户现在将在应用程序的响应中收到以下内容：

```html
<p><script>/* Bad stuff here... */</script></p>
```

攻击者提供的脚本随后将在受害用户的浏览器中，以该用户与应用程序的会话上下文执行。

## 存储型 XSS 攻击的影响

如果攻击者可以控制在受害者浏览器中执行的脚本，那么他们通常可以完全控制该用户。攻击者可以执行与[反射型 XSS 漏洞](/web-security/cross-site-scripting/reflected)影响相同的任何操作。

在可利用性方面，反射型 XSS 与存储型 XSS 的关键区别在于，存储型 XSS 漏洞使得攻击可以在应用程序自身内部独立完成。攻击者无需寻找外部方式来诱导其他用户发出包含其漏洞利用的特定请求。相反，攻击者将其漏洞利用放入应用程序本身，然后等待用户遇到它。

存储型跨站脚本的自包含特性在 XSS 漏洞仅影响当前已登录应用程序用户的情况下尤为相关。如果 XSS 是反射型，则攻击必须时机恰当：如果用户在未登录时被诱导发出攻击者的请求，则不会被入侵。相反，如果 XSS 是存储型，则用户在遇到漏洞利用时保证已登录。

## 不同上下文中的存储型 XSS

存储型跨站脚本有许多不同类型。存储数据在应用程序响应中的位置决定了需要哪种类型的载荷来利用它，并且还可能影响漏洞的影响。

此外，如果应用程序在存储数据之前或在将存储数据合并到响应时执行了任何验证或其他处理，这通常会影响所需的 XSS 载荷类型。

## 如何发现和测试存储型 XSS 漏洞

许多存储型 XSS 漏洞可以使用 Burp Suite 的 Web 漏洞扫描器发现。

手动测试存储型 XSS 漏洞可能具有挑战性。您需要测试所有相关的“入口点”（攻击者可控数据可通过这些点进入应用程序的处理流程），以及所有“出口点”（数据可能在这些点出现在应用程序的响应中）。

应用程序处理的入口点包括：

- URL 查询字符串和消息正文中的参数或其他数据。
- URL 文件路径。
- HTTP 请求标头（这些在反射型 XSS 中可能无法利用）。
- 任何带外路径，攻击者可通过这些路径将数据传送到应用程序中。存在的路径完全取决于应用程序实现的功能：Web 邮件应用程序将处理电子邮件中接收的数据；显示 Twitter 动态的应用程序可能处理第三方推文中包含的数据；新闻聚合器将包含源自其他网站的数据。

存储型 XSS 的出口点是所有可能在任何情况下返回给任何类型应用程序用户的 HTTP 响应。

测试存储型 XSS 漏洞的第一步是定位入口点和出口点之间的链接，即提交到入口点的数据从出口点发出。这可能具有挑战性的原因在于：

- 提交到任何入口点的数据原则上可能从任何出口点发出。例如，用户提供的显示名称可能出现在一个只有部分应用程序用户可见的隐蔽审计日志中。
- 应用程序当前存储的数据可能因应用程序内执行的其他操作而被覆盖。例如，搜索功能可能显示最近的搜索列表，这些列表会随着用户执行其他搜索而迅速被替换。

为了全面识别入口点和出口点之间的链接，需要分别测试每个排列，将特定值提交到入口点，直接导航到出口点，并确定该值是否出现在那里。然而，对于超过几个页面的应用程序，这种方法不切实际。

相反，更实际的方法是系统地遍历数据入口点，向每个入口点提交一个特定值，并监控应用程序的响应以检测提交值出现的情况。可以特别关注相关的应用程序功能，例如博客文章的评论。当在响应中观察到提交的值时，您需要确定数据是否确实跨不同请求存储，而不是仅在即时响应中被反射。

当您识别出应用程序处理中入口点和出口点之间的链接后，需要专门测试每个链接以检测是否存在存储型 XSS 漏洞。这包括确定响应中存储数据出现的上下文，并测试适用于该上下文的合适候选 XSS 载荷。此时，测试方法与发现[反射型 XSS 漏洞](/web-security/cross-site-scripting/reflected)的方法大致相同。
