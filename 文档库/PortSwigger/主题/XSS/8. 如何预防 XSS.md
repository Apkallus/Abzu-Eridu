# 如何预防 XSS

在本节中，我们将描述预防跨站脚本漏洞的一些通用原则，以及如何使用各种常见技术来防御 XSS 攻击。

跨站脚本的防御通常可以通过两层防线来实现：

- [输出时编码数据](#encode-data-on-output)
- [输入时验证数据](#validate-input-on-arrival)

> 你可以使用 Burp Scanner 扫描网站中的众多安全漏洞，包括 XSS。Burp 先进的扫描逻辑模拟了熟练攻击者的行为，能够实现相应高覆盖率的 XSS 漏洞检测。你可以使用 Burp Scanner 来确保你的 XSS 攻击防御措施有效运作。
>
> [了解更多关于 Burp Scanner 的信息](/burp/vulnerability-scanner)

## 输出时编码数据

编码应在用户可控数据写入页面之前直接应用，因为你写入的上下文决定了需要使用哪种编码方式。例如，JavaScript 字符串中的值与 HTML 上下文中的值需要不同类型的转义。

在 HTML 上下文中，应将非白名单值转换为 HTML 实体：

- `<` 转换为 `&lt;`
- `>` 转换为 `&gt;`

在 JavaScript 字符串上下文中，非字母数字值应进行 Unicode 转义：

- `<` 转换为 `\u003c`
- `>` 转换为 `\u003e`

有时你需要按正确顺序应用多层编码。例如，要安全地将用户输入嵌入事件处理程序，你需要同时处理 JavaScript 上下文和 HTML 上下文。因此，你需要首先对输入进行 Unicode 转义，然后再进行 HTML 编码：

```html
<a href="#" onclick="x='This string needs two layers of escaping'">test</a>
```

## 输入时验证数据

编码可能是最重要的 XSS 防线，但在所有上下文中都不足以预防 XSS 漏洞。你还应在从用户处首次接收输入时尽可能严格地验证输入。

输入验证的示例包括：

- 如果用户提交了一个将在响应中返回的 URL，请验证它是否以安全协议（如 HTTP 和 HTTPS）开头。否则，攻击者可能会利用有害协议（如 `javascript` 或 `data`）来攻击你的站点。
- 如果用户提供的值预期为数字，请验证该值确实包含整数。
- 验证输入是否仅包含预期的字符集。

输入验证理想情况下应通过阻止无效输入来实现。另一种方法——尝试清理无效输入使其有效——更容易出错，应尽可能避免。

### 白名单与黑名单

输入验证通常应采用白名单而非黑名单。例如，与其试图列出所有有害协议（`javascript`、`data` 等），不如简单地列出安全协议（HTTP、HTTPS），并禁止不在列表中的任何内容。这将确保你的防御在新的有害协议出现时不会失效，并且不易受到试图混淆无效值以逃避黑名单的攻击的影响。

## 允许“安全”的 HTML

应尽可能避免允许用户发布 HTML 标记，但有时这是业务需求。例如，博客网站可能允许评论包含一些有限的 HTML 标记。

经典方法是尝试过滤掉可能有害的标签和 JavaScript。你可以尝试使用安全标签和属性的白名单来实现这一点，但由于浏览器解析引擎的差异以及变异 XSS 等 quirks，这种方法极难安全地实施。

最好的办法是使用在用户浏览器中执行过滤和编码的 JavaScript 库，如 DOMPurify。其他库允许用户以 Markdown 格式提供内容，并将 Markdown 转换为 HTML。不幸的是，所有这些库时不时都会出现 XSS 漏洞，因此这不是完美的解决方案。如果你使用其中一种，应密切关注安全更新。

> **注意：** 除了 JavaScript，其他内容如 CSS 甚至普通 HTML 在某些情况下也可能是有害的。
>
> [使用恶意 CSS 的攻击](https://portswigger.net/research/detecting-and-exploiting-path-relative-stylesheet-import-prssi-vulnerabilities#badcss)

## 如何使用模板引擎预防 XSS

许多现代网站使用服务器端模板引擎（如 Twig 和 Freemarker）在 HTML 中嵌入动态内容。这些引擎通常定义了自己的转义系统。例如，在 Twig 中，你可以使用 `e()` 过滤器，并附带一个参数来定义上下文：

```
{{ user.firstname | e('html') }}
```

其他一些模板引擎（如 Jinja 和 React）默认转义动态内容，这有效地防止了大多数 XSS 的发生。

我们建议在评估是否使用特定模板引擎或框架时仔细审查其转义特性。

> **注意：** 如果你直接将用户输入拼接到模板字符串中，你将容易受到[服务器端模板注入](https://portswigger.net/kb/issues/00101080_server-side-template-injection)的影响，这通常比 XSS 更严重。

## 如何在 PHP 中预防 XSS

在 PHP 中有一个内置函数 `htmlentities` 用于编码实体。在 HTML 上下文中，你应该调用此函数来转义输入。该函数应使用三个参数调用：

- 你的输入字符串。
- `ENT_QUOTES`，一个标志，指定所有引号都应被编码。
- 字符集，在大多数情况下应为 UTF-8。

例如：

```php
<?php echo htmlentities($input, ENT_QUOTES, 'UTF-8');?>
```

当处于 JavaScript 字符串上下文时，你需要如前所述对输入进行 Unicode 转义。不幸的是，PHP 没有提供用于对字符串进行 Unicode 转义的 API。以下是在 PHP 中执行此操作的代码：

```php
<?php
function jsEscape($str) {
    $output = '';
    $str = str_split($str);
    for($i=0;$i<count($str);$i++) {
        $chrNum = ord($str[$i]);
        $chr = $str[$i];
        if($chrNum === 226) {
            if(isset($str[$i+1]) && ord($str[$i+1]) === 128) {
                if(isset($str[$i+2]) && ord($str[$i+2]) === 168) {
                    $output .= '\u2028';
                    $i += 2;
                    continue;
                }
                if(isset($str[$i+2]) && ord($str[$i+2]) === 169) {
                    $output .= '\u2029';
                    $i += 2;
                    continue;
                }
            }
        }
        switch($chr) {
            case "'":
            case '"':
            case "\n";
            case "\r";
            case "&";
            case "\\";
            case "<":
            case ">":
                $output .= sprintf("\\u%04x", $chrNum);
            break;
            default:
                $output .= $str[$i];
            break;
    }
    }
    return $output;
}
?>
```

以下是在 PHP 中使用 `jsEscape` 函数的方法：

```php
<script>x = '<?php echo jsEscape($_GET['x'])?>';</script>
```

或者，你可以使用模板引擎。

## 如何在 JavaScript 客户端预防 XSS

要在 JavaScript 中的 HTML 上下文转义用户输入，你需要自己的 HTML 编码器，因为 JavaScript 不提供编码 HTML 的 API。以下是将字符串转换为 HTML 实体的示例 JavaScript 代码：

```javascript
function htmlEncode(str){
    return String(str).replace(/[^\w. ]/gi, function(c){
        return '&#'+c.charCodeAt(0)+';';
    });
}
```

然后你可以按如下方式使用此函数：

```javascript
<script>document.body.innerHTML = htmlEncode(untrustedValue)</script>
```

如果你的输入位于 JavaScript 字符串中，你需要一个执行 Unicode 转义的编码器。以下是一个 Unicode 编码器示例：

```javascript
function jsEscape(str){
    return String(str).replace(/[^\w. ]/gi, function(c){
        return '\\u'+('0000'+c.charCodeAt(0).toString(16)).slice(-4);
    });
}
```

然后你可以按如下方式使用此函数：

```javascript
<script>document.write('<script>x="'+jsEscape(untrustedValue)+'";<\/script>')</script>
```

## 如何在 jQuery 中预防 XSS

jQuery 中最常见的 XSS 形式是将用户输入传递给 jQuery 选择器。Web 开发人员经常使用 `location.hash` 并将其传递给选择器，这会导致 XSS，因为 jQuery 会渲染 HTML。jQuery 认识到这个问题并修补了其选择器逻辑，以检查输入是否以哈希开头。现在 jQuery 只有在第一个字符是 `<` 时才会渲染 HTML。如果你将不可信数据传递给 jQuery 选择器，请确保使用上述 `jsEscape` 函数正确转义该值。

## 使用内容安全策略（CSP）缓解 XSS

内容安全策略（CSP）是防御跨站脚本的最后一道防线。如果你的 XSS 防御失败，你可以使用 CSP 通过限制攻击者的操作来缓解 XSS。

CSP 允许你控制各种事项，例如是否允许加载外部脚本以及是否执行内联脚本。要部署 CSP，你需要包含一个名为 `Content-Security-Policy` 的 HTTP 响应头，其值包含你的策略。

CSP 示例如下：

```
default-src 'self'; script-src 'self'; object-src 'none'; frame-src 'none'; base-uri 'none';
```

此策略指定资源（如图片和脚本）只能从与主页面相同的源加载。因此，即使攻击者成功注入了 XSS 载荷，他们也只能从当前源加载资源。这大大降低了攻击者利用 XSS 漏洞的可能性。

如果需要加载外部资源，请确保只允许那些不会帮助攻击者利用你网站的脚本。例如，如果你将某些域加入白名单，攻击者就可以从这些域加载任何脚本。如果可能，尽量将资源托管在你自己的域上。

如果这不可能，你可以使用基于哈希或随机数的策略来允许来自不同域的脚本。随机数是一个随机字符串，作为脚本或资源的属性添加，只有当随机字符串与服务器生成的匹配时才会执行。攻击者无法猜测随机字符串，因此无法调用具有有效随机数的脚本或资源，从而该资源不会被执行。
