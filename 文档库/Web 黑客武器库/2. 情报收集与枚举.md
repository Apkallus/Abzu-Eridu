# 第 2 章 情报收集与枚举

## 2.1 引言

枚举或许是渗透测试或漏洞赏金活动中最重要的环节。尤其是在漏洞赏金计划中，参与者之间存在竞争，拥有高效枚举方法论的漏洞猎人往往能发现“低垂的果实”和复杂漏洞。

枚举方法主要有两种类型：主动式和被动式。主动式通常涉及直接探测目标并发送大量请求，而被动式主要依赖于公开可用数据或先前爬取网站的信息。

在本章中，我们探讨渗透测试和漏洞挖掘中使用的先进枚举方法。这些技术通常需要手动与自动化分析相结合。我们还将讨论如何利用自动化为你所用。本章主题灵活呈现，你可以根据自身情况选择相关章节。本章除了前沿枚举技术外，还将涉及云枚举技术。

本章包含多个 Bash 和 Python 脚本以展示自动化。已尽力在可能的地方解释这些脚本。但如果你不熟悉这些语言，为更好理解，熟悉它们是至关重要的。此外，本章还介绍了各种带有复杂命令的工具。虽然这些命令、参数和标志可能让人望而生畏，但不需要死记硬背。更重要的是理解它们的目的、优点和使用场景，这些可以通过查阅文档来补充。

值得提及的是，尽管有大量自动化工具可用于各种枚举技术，但真正的技能在于有效解读这些工具的输出、关联不同来源的数据并利用这些信息。需要指出的是，在枚举阶段，没有“不必要的信息”。所有收集到的数据都可能对未来有用。

为了实践演示，本章主要使用 `Paypal.com`，这是最古老的漏洞赏金平台之一，因此测试无需额外许可。

### 2.1.1 枚举 ASN 和 IP 段

自治系统号（ASN）是分配给组织或公司的唯一标识符。它代表属于该实体的一系列 IP 地址。从枚举角度看，ASN 很重要，因为它能提供公司拥有的 IP 范围信息。例如，对于 PayPal，我们可以使用 `bgp.he.net` 来识别与该公司关联的 ASN。输出结果提供了分配给 PayPal 的 ASN 列表：

**图 2.1 通过 bgp.he.net 进行 ASN 枚举**

根据这些 ASN，我们可以进一步确定分配给 PayPal 的 IP 范围。

**图 2.2 ASN 对应的 IP 范围**

或者，我们也可以使用公开的 “bgpview” API 来查询 ASN：

**命令**
```bash
curl -s https://api.bgpview.io/search?query_term=paypal | jq
```

**图 2.3 BGPview API 查询结果**

另外，Nmap 脚本 “target-asn” 也可用于基于 ASN 提取 IP 范围。例如，以下脚本将针对 ASN “26444” 显示结果：

**命令**
```bash
nmap --script targets-asn --script-args targets-asn.asn=26444
```

**图 2.4 Nmap 脚本结果**

基于识别出的 IP 段，可以查询特定文件。例如，安全研究员 Patrik Fehernbach 可能使用 Bash 脚本扫描 Yahoo 的 IP 范围，查找 “phpinfo.php” 文件（该文件可能包含有价值的服务器信息）。该脚本将尝试在指定范围内的每个 IP 上访问此文件。

**代码**
```bash
for ipa in 98.13{6..9}.{0..255}.{0..255}; do
    wget -t 1 -T 5 http://${ipa}/phpinfo.php;
done &
```

该脚本可以通过查询类似文件来增强。我们将在后续章节讨论几种自动化工具。

### 2.1.2 反向 IP 查询

反向 IP 查询涉及查询 IP 地址，以识别托管在同一 IP 上的域名。这在共享托管环境中很常见，多个域名托管在同一 IP 上。

有多种在线工具可执行反向 IP 查询，如 YouGetSignal、ViewDNS.info、rapiddns.io 等。

**图 2.5 反向 IP 查询结果**

或者，我们可以使用 curl 命令从 RapidDNS 提取与 PayPal 特定 IP 段相关的域名信息。

**命令**
```bash
curl -s 'https://rapiddns.io/sameip/64.4.250.0/24?full=1#result' | grep 'target="' -B1 | egrep -v '\(--|\) ' | rev | cut -c 6- | rev | cut -c 5- | sort -u
```

**图 2.6 从 rapiddns.io 获取的 PayPal 域名结果**

该命令检索 HTML 内容，使用 grep 和 egrep 过滤相关行，然后使用 rev 和 cut 处理数据以去除 HTML 标签，最终得到干净的域名列表，如 ‘paypal.be’、‘paypal.ca’ 等。从截图中可以看到其他不属于 PayPal 的域，如 “buyindiaonline.com”。这是因为这些数据库并不完全准确，因此应与其他数据库结合使用，并需要人工审核。

## 2.2 使用多线程进行反向 IP 查询

为加快反向 IP 查询过程，尤其是在同时查询多个 IP 地址时，可以使用多线程。其中一个工具是 “Interlace” （[https://github.com/codingo/Interlace](https://github.com/codingo/Interlace)），这是一个基于 Python 的工具，能将单线程命令转换为多线程操作。

**命令**
```bash
interlace -tL ip.txt -c "curl -s 'https://rapiddns.io/sameip/_target_?full=1#result'|grep 'target=' -B1 | egrep -v '\(--|\) ' | rev | cut -c 6- | rev | cut -c 5- | sort -u >>output.txt" -threads 2 --silent --no-color --no-bar
```

该命令使用包含 IP 地址的输入文件（ip.txt），以并行线程（此处为“2”）处理每个 IP 并执行指定的 curl 命令，最终将结果追加到 output.txt。

### 2.2.1 扫描开放端口/服务

进行反向 IP 查询后，下一步自然就是查询开放端口。开放端口可能暴露在非常规端口上运行的 HTTP 服务器，这可能在标准扫描中被忽略。虽然 Nmap 仍是该领域最流行的工具，但 Masscan 最近已成为许多漏洞猎人的首选。其快速扫描大型网络的能力使其在识别潜在入口点（如开放 HTTP 服务器）方面非常高效。

## 2.3 使用 Masscan 扫描开放端口

以下命令可用于扫描开放端口：

**命令**
```bash
sudo masscan --open-only 10.22.144.0/24 -p1-65535,U:1-65535 --rate=10000 --http-user-agent "Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:67.0) Gecko/20100101 Firefox/67.0" -oL "output.txt"
```

**图 2.7 Masscan 开放端口查询结果**

## 2.4 使用 httpx 检测 HTTP 服务

Httpx 是一个用于收集 Web 服务信息的工具。虽然 Masscan 能有效揭示开放端口，但它不指定哪些端口正在运行 HTTP 服务。为此，可以将 httpx 应用于 Masscan 的输出。以下命令序列用于此目的：

**命令**
```bash
cat output.txt | grep tcp | awk '{print $4,":",$3}' | tr -d ' ' | httpx -title -sc -cl
```

**图 2.8 使用 httpx 发现活动子域名**

该命令读取包含 Masscan 输出的 “output.txt” 文件，过滤出 TCP 端口，将 IP 和端口格式化为特定模式，然后使用 “httpx” 检查这些地址是否活跃。“Httpx” 评估标题、状态码和内容长度等参数，以确定服务的活动性和特征。

### 2.4.1 扫描服务版本

既然我们已经使用 “httpx” 成功识别了每个 IP 地址上的 Web 服务器，下一步就是执行服务版本扫描。Nmap 以其广泛的数据库和扫描技术而闻名，通常能准确确定服务器版本。

**命令**
```bash
nmap -sC -sV 10.22.144.147 -T4
```

**图 2.9 Nmap 脚本输出结果**

该命令使用 “-sV” 标志启用服务版本检测，识别目标服务器的协议和软件版本。“-sC” 标志执行 Nmap 脚本以获取额外信息。

**提示：** 使用 Nmap 进行详细扫描可能耗时，尤其是在漏洞赏金活动中。更有效的方法是先使用 Masscan 识别开放端口，然后对这些端口进行有针对性的 Nmap 服务版本扫描。这种方法避免扫描所有端口，从而节省时间。为此，脚本 Masscan_to_nmap.py 能高效地将 Masscan 的输出连接到 Nmap，简化流程 [https://gist.github.com/mosesrenegade/1f09c90376d81630e233c37d2e7d3b3d](https://gist.github.com/mosesrenegade/1f09c90376d81630e233c37d2e7d3b3d)。

## 2.5 子域名枚举

子域名枚举是任何枚举活动的主要组成部分，也是大多数成功的漏洞猎人花费最多时间的地方。虽然有大量工具为此开发，但我们将重点介绍我们认为最有效的工具。子域名枚举可以是主动式或被动式。在主动枚举中，我们直接探测目标，而在被动枚举中，我们依赖各种网站在其查询中已获得的结果。

### 2.5.1 主动子域名枚举

在本节中，我们将深入探讨主动枚举的子域名枚举技术。但需要注意的是，这些方法可能会在服务器端产生大量噪音，可能导致 WAF 或任何其他安全解决方案的速率限制。因此，在需要隐蔽性的测试中，应谨慎使用这些技术。

#### 2.5.1.1 子域名枚举 DNS 暴力破解

DNS 暴力破解是发现子域名最有效的方式。这涉及使用常见子域名字典，因此暴力破解的结果直接取决于字典的质量。一些知名的 DNS 字典来自 “SECLIST” （[https://github.com/danielmiessler/SecLists/tree/master/Discovery/DNS](https://github.com/danielmiessler/SecLists/tree/master/Discovery/DNS)）、 “ASSETNOTE” （[https://wordlists-cdn.assetnote.io/data/manual/best-dns-wordlist.txt](https://wordlists-cdn.assetnote.io/data/manual/best-dns-wordlist.txt)） 和 “Rapid Forward DNS” （[https://opendata.rapid7.com/sonar.fdns_v2/](https://opendata.rapid7.com/sonar.fdns_v2/)） 等，可用于此目的。

**提示：** 在渗透测试或漏洞赏金活动中，你可能通过其他方式（如范围文档或利用其他漏洞）发现子域名。建议手动将这些子域名添加到你的自定义字典中。随着时间的推移，这种方法将为你提供优于其他渗透测试者或漏洞猎人的优势。

## 2.6 DNSValidator

在枚举子域名时，工具通常默认使用你的互联网服务提供商（ISP）提供的 DNS 解析器，或常用的如 Google 的 8.8.8.8。然而，这些可能不是最可靠的，因为可能存在来自单个 IP 请求数的速率限制，并且也可能受到地理限制。

因此，在开始子域名枚举之前，验证 DNS 解析器是否受支持且有效非常重要。为此，可以使用基于 Python 的工具 DNSValidator （[https://github.com/vortexau/dnsvalidator](https://github.com/vortexau/dnsvalidator)）。它采用多种方法测试和验证 DNS 解析器。该工具检查解析器列表的公共可访问性和有效性，排除不符合标准的解析器。

使用 DNSValidator，我们将向其提供从 public-dns.info （[https://public-dns.info/nameservers.txt](https://public-dns.info/nameservers.txt)） 获取的外部解析器列表。我们将使用 “public-dns.info” 上的列表提供已知外部解析器，输出将保存为 “resolvers.txt”，作为子域名枚举工具的输入。

**命令**
```bash
dnsvalidator -tL https://public-dns.info/nameservers.txt -threads 100 -o resolvers.txt
```

**图 2.10 DNSValidator 结果**

## 2.7 ShuffleDNS

ShuffleDNS （[https://github.com/projectdiscovery/shuffledns](https://github.com/projectdiscovery/shuffledns)） 是一个用 “Go” 语言编写的子域名枚举工具，该语言以速度和并发性著称，适合处理大规模 DNS 查询任务。本质上，它是 “massdns” 的封装，后者是为批量 DNS 查找而构建的高性能 DNS 解析器。

其独特功能之一是能够处理基于 DNS 的通配符。它通过跟踪解析到单个 IP 的子域名数量（达到一定阈值）来实现，从而显著减少误报数量。

DNS 通配符是 DNS 配置中的一项功能，允许将所有不存在的子域名自动解析到特定 IP 地址。例如，如果子域名如 `tmgm.paypal.com` 不存在，配置了通配符的 DNS 服务器会将其解析到指定 IP 地址，而不是返回标准的 NXDOMAIN（不存在域）响应。

为了演示该工具，我们将目标设为 `paypal.com`，使用名为 top1million-5000.txt 的子域名列表作为输入。先前通过 DNSValidator 获得的 resolvers.txt 文件包含 DNS 解析器列表。

**命令**
```bash
shuffledns -d paypal.com -w subdomains-top1million-5000.txt -r resolvers.txt
```

**图 2.11 子域名 DNS 发现**

同样，还有其他工具可用于实现相同目的。你可以尝试它们，甚至超越它们，最终找到适合你自己的工具集。

## 2.8 Subbrute

Subbrute 是一个基于 Python 的子域名枚举工具。它利用开放解析器来规避速率限制。以下命令启动 Subbrute，使用 “sub-wordlist.txt” 对 `paypal.com` 进行子域名枚举，然后将结果通过管道传递给 “massdns”，并指定解析器文件、输出格式和文件。

**命令**
```bash
python3 subbrute.py sub-wordlist.txt paypal.com | massdns -r resolvers.txt -o S -w output.txt
```

## 2.9 Gobuster

Gobuster 是另一个枚举工具，主要用于子域名枚举。但它不仅限于子域名，还可以查找开放的 Amazon S3 存储桶、Google Cloud 存储桶等。以下命令以 “sub-wordlist” 为输入，对 `paypal.com` 执行枚举。

**命令**
```bash
gobuster dns -w sub-wordlist.txt -d paypal.com -t 50
```

### 2.9.2 使用 Favicon 哈希枚举子域名

Favicon 是与网站关联的小图标，通常命名为 “favicon.ico”。当网站的子域使用相同的 favicon 时，每个 favicon 将具有相同的哈希值。通过分析和比较这些哈希值，可以枚举子域名。该技术有效，因为相同 favicon 的哈希值在不同子域中是相同的。

例如，如果我们对 PayPal 应用此技术，首先收集各种 PayPal 子域的 favicon 哈希。如果多个子域使用相同的 favicon，它们将具有相同的哈希值。让我们看看实际效果：

**图 2.14 PayPal 的 Favicon**

**步骤 1：下载 Favicon**

第一步是下载 favicon。为此，我们可以使用 “curl”。

**命令**
```bash
curl -s www.paypalobjects.com/webstatic/icon/favicon.ico -o favicon.ico
```

**图 2.15 使用 curl 下载 favicon**

**步骤 2：生成 MurmurHash**

下一步是为 favicon 生成 MurmurHash。MurmurHash 是从 “favicon.ico” 文件内容计算出的 32 位哈希值。这可以通过使用 Python 的 mmh3 模块的 MurmurHash 来实现。

**代码**
```bash
cat favicon.ico | base64 | python3 -c "import mmh3, sys; print(mmh3.hash(sys.stdin.buffer.read()))"
```

此命令获取 favicon.ico 文件，将其转换为 base64 格式，然后使用 Python 中的 mmh3 模块计算 MurmurHash。

**图 2.16 生成 MurmurHash**

**步骤 3：使用 Shodan 搜索**

下一步是在 Shodan 上搜索该哈希，以检索使用相同 favicon 的更多域。我们将使用 ShodanHQ 中的 “http.favicon.hash” 标志来实现。

**命令**
```
http.favicon.hash:309020573
```

**图 2.17 在 Shodan 上搜索 favicon**

## 2.10 整合在一起

从获取 favicon 到通过 Shodan 搜索和检索子域的整个过程，可以使用 curl 命令与 Python 代码的组合来自动化。

**命令**
```bash
curl -s www.paypalobjects.com/webstatic/icon/favicon.ico | base64 | python3 -c 'import mmh3, sys;print(mmh3.hash(sys.stdin.buffer.read()))' | xargs -I{} shodan search http.favicon.hash:{} --fields hostnames | tr ";" "\n"
```

该命令检索 favicon，然后使用 base64 命令将其转换为 base64 格式。接下来，使用 Python mmh3 模块计算转换后的 base64 字符串的 MurmurHash3 哈希。最后，使用 Shodan CLI 工具查找具有相同 favicon 哈希的主机。

### 2.10.1 子域名的被动枚举

被动枚举通常是识别子域名最快的方法，因为它利用各种来源的数据，而不直接与目标域的服务器交互。这种方法确保不会向域的服务器发送实际请求。被动枚举使用的工具从多个来源收集信息，包括 DNS 记录存档、搜索引擎缓存、证书透明度日志和第三方数据聚合器。

#### 2.10.1.1 使用 RapidDNS 枚举子域名

RapidDNS 是用于对目标域进行被动子域名枚举的流行工具。它拥有庞大的五十亿记录数据库，能够跨广泛域全面发现子域名。以下命令可用于查询 rapiddns.io 获取 PayPal 子域名。

**命令**
```bash
curl -s https://rapiddns.io/subdomain/paypal.com?full=1 | grep -Eo '[a-zA-Z0-9.-]+\.([a-zA-Z]{2,})' | sort -u
```

该命令使用 curl 从 RapidDNS 检索 `paypal.com` 子域的数据，然后 grep 过滤数据以提取域名模式，“sort -u” 将排序并删除重复条目，提供唯一的子域名列表。

**图 2.19 使用 rapiddns 输出 paypal.com 子域**

#### 2.10.1.4 使用 GitHub 枚举子域名

GitHub 是一个托管各种公共仓库的平台。这些可能包括源代码中硬编码的 URL、项目中的配置文件（包含有关不同环境的详细信息）、用于代码共享的 GitHub Gists 以及项目文档，所有这些都可能成为发现子域信息的来源。

幸运的是，有工具可以查询这些领域。其中一个工具是 github-subdomains.py （[https://github.com/gwen001/github-search/blob/master/github-subdomains.py](https://github.com/gwen001/github-search/blob/master/github-subdomains.py)）。要使用此工具，可以执行以下命令：

**命令**

#### 2.10.1.6 使用 Web 存档进行子域名枚举

Web 存档捕获并存储网站的历史版本，这意味着它们可能包含曾经活跃但现已从 Web 上消失的子域名信息。有几种工具可用于此目的，其中最流行的是 “gau” （[https://github.com/lc/gau](https://github.com/lc/gau)）。该工具特别适用于从各种来源（包括搜索引擎、AlienVault 的 Open Threat Exchange、Wayback Machine、Common Crawl 和 URLScan）获取给定域名的已知 URL。

## 2.12 Web 应用程序指纹识别

一旦我们识别出子域/域、去除了误报、整合了数据并确定了起点，下一步就是对 Web 应用程序进行指纹识别。这涉及识别隐藏目录、文件结构、端点和输入参数。

### 2.12.1 目录模糊测试

模糊测试目录涉及向目标发送大量请求，以发现可访问的目录、文件或端点。此方法的有效性在很大程度上取决于所使用的字典质量。其中值得注意的是 AssetNote 字典 （[https://wordlists.assetnote.io/](https://wordlists.assetnote.io/)） 和 SecLists （[https://github.com/danielmiessler/SecLists/tree/master/Discovery/Web-Content](https://github.com/danielmiessler/SecLists/tree/master/Discovery/Web-Content)）。

有几种工具可用于目录模糊测试，每个都有其独特的功能集和能力。

#### 2.12.1.1 使用 FFUF 进行目录模糊测试

FFUF（Fuzz Faster U Fool） （[https://github.com/ffuf/ffuf](https://github.com/ffuf/ffuf)） 是一个主要用于模糊测试 Web 应用程序的强大工具。虽然它以识别隐藏目录而闻名，但其用途远不止于此。它可以帮助识别子域、参数模糊测试、标头模糊测试和测试速率限制功能。

以下命令对 `demo-site.com` 执行目录模糊测试。

**命令**
```bash
ffuf -w wordlist.txt -u http://demo-site.com/FUZZ -mc 200 -e .php
```

该命令运行 ffuf，检查 `http://demo-site.com/` 上以 .php 扩展名结尾的有效页面，使用 wordlist.txt 中的条目，仅显示使用 HTTP 状态码 200 成功加载的页面。

注意：你可以根据应用程序调整扩展名；如果应用程序使用 JSP 或 ASP，可以相应调整。

**图 2.34 FFUF 目录模糊测试结果**

#### 2.12.1.2 使用 Dirbuster 进行目录模糊测试

如果你喜欢 GUI 版本，OWASP 的 Dirbuster 是一个用 Java 编码的流行工具，用于执行基于目录的暴力破解。Dirbuster 包含多种模糊测试模式，如字典模式、文件名暴力破解模式和基于自定义扩展名的模糊测试。

**图 2.35 Dirbuster 输出**

对隐藏目录进行模糊测试的最终目标是映射攻击面，并找到可能被用作 Web 应用程序入口点的潜在敏感文件。然而，有时它也可能揭示隐藏的端点，这些端点可能未经测试，很可能存在潜在漏洞。

#### 2.12.2.8 自动分析所有 JavaScript 文件

现代应用程序是动态的，可能包含数十个 JavaScript 文件。手动下载这些 JavaScript 文件并从中提取相关细节可能耗时。因此，自动化此过程可以节省宝贵时间。让我们探索如何自动化整个工作流程，从收集 JavaScript 文件到提取机密。

**步骤 1：收集 JavaScript 文件**

为了收集 JavaScript 文件，我们可以使用先前从 Web 存档和爬虫收集的数据。我们将使用以下命令：

**命令**
```bash
grep "\.js" paypal.txt | sort -u | httpx -silent -mc 200 -o paypal-js.txt
```

该命令在 “paypal.txt” 文件中搜索 JS 文件（该文件是使用 GAU 或任何其他工具检索的）。然后对引用进行排序，删除重复项，并使用 “httpx” 将具有成功响应码（200）的 URL 过滤到 “paypal-js.txt” 文件中。

**图 2.42 输出揭示 PayPal JS 文件**

**步骤 2：从 JavaScript 文件中提取端点**

该命令使用一个循环，读取 “paypal-js.txt” 文件中的每一行（包含在先前步骤中提取的 URL），然后为每个 URL 执行 “Linkfinder.py”，并将结果输出到 “paypal-endpoints.txt”。

## 2.14 新攻击面的自动映射

在漏洞赏金计划中，能够映射新的攻击面或发现目标的新域/子域，可以为你提供相对于其他漏洞猎人的优势。这种优势源于应用程序经常添加新的子域和页面。首先测试这些新元素可以在发现易于利用的漏洞（通常称为“低垂果实”）方面带来显著好处。

有多种方法可以实现这一点，但一种首选方法包括使用 Discord 服务器。Discord 在这方面非常高效，因为其实时通信和通知功能。Discord 允许使用 webhooks，即从应用程序发送到 Discord 的自动消息。该脚本持续运行，每 24 小时执行一次主函数，但频率可以根据需要调整。

这可用于设置 webhooks，在相关事件发生时（如发现新子域或网页更改）在特定频道发布消息。

让我们看看实际效果。在 Discord 服务器上，我们将导航到“集成”页面以创建新的 webhook。这将生成一个唯一的 webhook 链接。

**图 2.50 创建新 webhook 的导航页面**

以下 Python 代码演示了整个过程的实际运行。“target” 参数用于指定需要爬取的 URL。同时，“webhook” 参数用于提供在先前步骤中生成的 Discord webhook 链接。

**代码**
```python
import subprocess
import requests
import sqlite3
import time

target = "http://demo-site.com"

def run_hakrawler(url):
    command = f"echo {url} | hakrawler -u | egrep -v '(\.js|\.css|\.png|\.jpg|\.gif)'"
    process = subprocess.Popen(command, shell=True, stdout=subprocess.PIPE, stderr=subprocess.PIPE)
    output, error = process.communicate()
    line = output.decode('utf-8')
    return line

def discord_notification(url):
    webhook = "[YOUR_DISCORD_WEBHOOK_LINK]"
    message = {"content": "New Endpoint Found: " + url}
    requests.post(webhook, json=message)

def main():
    conn = sqlite3.connect('hakrawler-out.db')
    cursor = conn.cursor()
    cursor.execute("' CREATE TABLE IF NOT EXISTS urls ( column_name TEXT ) '")
    sql = "INSERT INTO urls (column_name) VALUES (?)"
    urls = run_hakrawler(target).splitlines()
    for url in urls:
        if target in url:
            cursor.execute("SELECT * FROM urls WHERE column_name = ?", (url,))
            existing_data = cursor.fetchone()
            if not existing_data:
                cursor.execute(sql, (url,))
                if counter != 1:
                    discord_notification(url)
    conn.commit()
    conn.close()

counter = 0
while True:
    counter = counter + 1
    main()
    time.sleep(24 * 60 * 60)
```

注意：`discord_notification` 函数向指定的 Discord webhook 发送消息。因此，当你收到通知时，需要将 `[YOUR_DISCORD_WEBHOOK_LINK]` 替换为实际的 Discord webhook URL。

运行此脚本后，每当发现新端点时，通知将实时发送到指定的 Discord 频道。发现的端点也将保存到 SQLite 数据库中。

## 2.15 Web 应用程序指纹识别

指纹识别 Web 应用程序的过程涉及识别它们使用的底层技术。这包括检测服务器类型（如 Apache、Nginx 和 Tomcat），以及编程语言的版本（如 PHP、JSP 和 ASP.Net）。它还涵盖依赖项，包括外部包、客户端库以及与外部组件（如 Web 服务器）的集成。此外，此过程还涉及指纹识别 Web 应用防火墙（WAF）、反向代理和负载均衡器，这些将在后续章节中介绍。

虽然前面的部分已经演示了使用 Nmap 等工具进行主动侦察以识别服务器版本的方法，但本节将重点介绍避免在服务器上产生噪音的方法。

每个组织都有其独特的风险偏好，因此如何处理此类风险在很大程度上取决于其安全策略。然而，在我的渗透测试活动中，无论其感知严重性如何，我总是倾向于报告此类问题。这是因为即使是低风险和中等风险的问题，有时也可以串联起来形成更大的安全威胁。

## 2.18 云枚举

现代 Web 导致云平台（如 AWS、GCP 等）的采用日益增加，这归功于其可扩展性和可靠性。这些平台的关键特性之一包括全面的对象存储解决方案，非常适合存储大量非结构化数据，如图像、视频、备份等。然而，这些存储解决方案可能托管因安全配置错误而暴露的敏感数据。在本节中，我们将讨论 AWS 枚举技术以及攻击者如何利用配置错误的 S3 存储桶。

### 2.18.1 AWS S3 存储桶枚举

S3 存储桶用于 Amazon 云存储服务中的存储容器。S3 存储桶已成为托管静态内容（如 HTML、CSS 和 JavaScript 文件）的热门选择。S3 存储桶枚举的目标是识别可公开访问的错误配置存储桶。让我们了解命名约定：

#### 2.18.1.1 命名约定与发现

S3 存储桶 URL 遵循标准命名约定。如果你知道存储桶名称，你可以使用以下格式直接访问它：

**标准约定**
格式：`[bucket-name].s3.amazonaws.com`
示例：`examplebucket.s3.amazonaws.com`

**替代约定**
然而，存在另一种命名约定，如下：
格式：`http://s3.amazonaws.com/[bucket_name]/`
示例：`http://s3.amazonaws.com/examplebucket/`

此格式通常用于需要从不同区域访问 S3 存储桶，或处理某些 DNS 和路径样式访问考虑的场景。

注意：在 Amazon S3 中，每个存储桶名称必须在 S3 全局所有现有存储桶名称中唯一。

#### 2.18.1.2 识别 S3 存储桶

确定网站是否托管在 Amazon S3 存储桶上可以通过一系列调查步骤实现，主要利用 DNS 查询工具。其特定 AWS 区域可以帮助理解站点的托管环境。

例如，考虑对 `flaws.cloud` 等域进行 DNS 查询。

**命令**
```bash
host flaws.cloud
```

该命令显示解析为 “52.218.201.195” 的几个 IP 地址。访问此地址时，它将重定向到 “aws.amazon.com/s3”，表明 flaws.cloud 托管在 S3 存储桶上。

**图 2.58 显示与 flaws.cloud 关联的 IP 地址**

接下来，你可以获取存储桶的区域，我们可以通过 DNS 请求发现 IP 来执行 dig 和 nslookup：

**命令**
```bash
nslookup 52.218.201.195
```

**图 2.59 nslookup 显示区域**

DNS 查询解析为 “s3-website-us-west-2.amazonaws.com”。它确认托管在 AWS 区域 “us-west-2”。这是 AWS 用于分布其服务的地理区域之一。因此，该站点也可使用以下域访问：

**示例**
```
flaws.cloud.s3-website-us-west-2.amazonaws.com
```

或者，你可以直接在 `flaws.cloud.s3-us-west-2.amazonaws.com` 访问存储桶内容。在此替代格式中，省略了 s3-website 部分，可用于访问存储桶，而不是托管的网站。

#### 2.18.1.3 使用 Google Dorks 识别 S3 存储桶

我们还可以使用被动枚举技术，如 Google Dorks，来识别与特定域关联的配置错误的 S3 存储桶。这种方法可能有助于发现敏感文件。例如，以下查询可用于搜索与 `Paypal.com` 关联的 AWS 存储桶。

**命令：**
```
site:s3.amazonaws.com paypal.com
```

要搜索特定文件扩展名，我们使用 filetype 命令：

**命令：**
```
site:s3.amazonaws.com filetype:xls password
```

注意：如果 S3 存储桶可公开访问，其内容可能已被搜索引擎索引或被 Web 爬虫存档，或保留在第三方数据聚合器中。这意味着敏感数据可能缓存在这些平台上，即使在更改存储桶权限后仍可访问。

以下是一些可用于此目的的常见 Google Dorks 示例（表 2.1）：

**表 2.1 常见 Google Dorks**

| 命令 | 描述 |
|------|------|
| `site:s3.amazonaws.com filetype:txt password` | 搜索包含单词 “password” 的文本文件的 S3 存储桶 |
| `site:s3.amazonaws.com filetype:sql` | 搜索包含 SQL 文件的 S3 存储桶 |
| `site:s3.amazonaws.com inurl:backup` | 搜索 URL 中包含 “backup” 的 S3 存储桶 |

### 2.18.2 利用配置错误的 AWS S3 存储桶

### 2.18.3 利用身份验证用户组配置错误

**图 2.63 配置为仅身份验证用户的存储桶**

访问控制列表（ACL）和存储桶策略是管理对 S3 存储桶资源访问的机制。这些设置中的配置错误可能会无意中授予未经授权用户读取或写入权限。例如，考虑以下配置，其中访问控制列表错误地配置为 “Authenticated users group”，意味着任何拥有 AWS 账户的人都可以读取和写入 AWS 存储桶。

当尝试匿名访问存储桶时，返回 “Access denied” 错误，这通常表示存储桶是私有的：

**命令**
```bash
aws s3 sync s3://demo-bucket.redseclabs.com/. --region us-east-1 --no-sign-request
```

**图 2.64 私有存储桶导致 “access denied” 错误**

然而，由于允许任何拥有 AWS 账户的人访问存储桶内容的错误配置，让我们看看如何使用 AWS CLI 实现这一点。

要访问存储桶，我们需要通过配置 AWS 访问密钥 ID 和秘密访问密钥，通过 AWS CLI 进行身份验证。这样做后，你将它们与你机器上的 AWS CLI 关联。CLI 将使用这些凭据来认证你对 AWS 服务的 API 请求。这些细节可以从 AWS 管理控制台检索 [https://us-east-1.console.aws.amazon.com/iamv2/home?region=us-east-1#/security_credentials/access-key-wizard](https://us-east-1.console.aws.amazon.com/iamv2/home?region=us-east-1#/security_credentials/access-key-wizard)。

要配置两个密钥，我们将使用 configure 命令。此命令将提示你输入 AWS 访问密钥、秘密密钥和用于身份验证的默认区域。

**命令**
```bash
aws configure
```

**图 2.65 配置秘密访问密钥和访问密钥 ID**

完成这些配置（包括设置 AWS 凭据和配置 AWS CLI）后，用户可以成功访问存储桶。此访问将根据 ACL 或存储桶策略中设置的权限进行。完成后，可以使用以下命令通过身份验证访问 S3 存储桶。

**命令**
```bash
aws s3 ls s3://demo-bucket.redseclabs.com/ --region us-east-1
```

## 2.19 额外进阶

- **自动化计划扫描脚本**：编写脚本以自动化监控计划变更。应检测任何 Web 应用程序中的重大更新，如新功能、子域等，并通过 Discord 发送警报。
- **探索 RustScan** （[https://github.com/RustScan/RustScan](https://github.com/RustScan/RustScan)）：探索基于 Rust 的 “Masscan” 替代品 “RustScan”，并开发其自动化，类似 Masscan 中演示的那样。
- **漏洞扫描器开发**：利用 Python、Go 或其他语言构建漏洞扫描器，整合我们在本章中探讨的所有技术。
- **Azure、GCP 枚举**：虽然 AWS 目前拥有最大的市场份额，但你也可能遇到 Azure 和 GCP，它们由于其不同的架构和服务产品而具有不同的枚举方法。研究 Azure 和 GCP 枚举技术将有助于你更进一步学习。
