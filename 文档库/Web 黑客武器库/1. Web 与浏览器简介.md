# 第 1 章 Web 与浏览器简介

## 1.1 引言

Web 应用程序已成为现代数字环境中不可或缺的一部分。在过去十年中，它们在技术、特性和功能方面经历了显著的发展，旨在创造丰富的用户体验。然而，功能的每一次进步都带来了一系列新的复杂性。此外，争夺市场份额的浏览器不断向 Web 应用程序引入独特的功能，其中许多以不同方式实现安全策略和机制。由于缺乏一致的标准参考实现，安全策略的实现一直是浏览器特有的，且高度多样化。这种差异不仅扩大了威胁面，也为攻击者利用这些不一致性创造了机会。

Web 应用程序与浏览器技术的交汇是一个关键关注领域。本章将重点说明浏览器特有功能和安全实现如何在更广泛的 Web 安全背景下体现其重要性。我们还将深入浏览器安全的世界，探讨浏览器为保护 Web 应用程序而引入的核心安全策略和机制。理解这些基础知识对于 Web 安全至关重要。

## 1.2 HTTP 简介

超文本传输协议（HTTP）是驱动万维网的协议。从根本上说，它基于客户端-服务器架构，客户端（通常通过浏览器）请求内容，服务器/应用程序交付响应。客户端到服务器的请求可以通过中间设备（如反向代理、负载均衡器和 Web 应用防火墙（WAF））进行路由。HTTP 的默认传输端口是 TCP（传输控制协议）端口 80，但它也可以在其他端口上运行，并可以封装在其他协议中。

### 1.2.1 HTTP 的属性

以下是 HTTP 的一些核心属性：

- **无状态性**：HTTP 是无状态协议，这意味着两个请求之间没有任何关联。但是，为了管理诸如登录等状态，会使用 Cookie 等机制。
- **缺乏固有加密**：HTTP 是未加密的协议，这意味着网络上的任何中间设备（如路由器或代理）都能够读取和修改流量。为了解决这个问题，HTTPS 将 HTTP 封装在 TLS/SSL（传输层安全/安全套接层）加密层中。
- **可扩展性**：HTTP 被设计为可扩展的，这意味着它在请求和响应中都使用标头来传递元数据和其他相关信息。这允许在不修改核心的情况下实现新功能。
- **可靠性**：HTTP 通过 TCP 传输，这意味着可靠性得到保证。TCP 是面向连接的协议，确保数据包按顺序且无错误地交付。

### 1.2.2 HTTP 通信

HTTP 通信基于 HTTP 请求和 HTTP 响应。客户端向服务器发送 HTTP 请求，请求某个资源，服务器以 HTTP 响应进行回应。我们来分析一个示例 HTTP 请求。

以下 HTTP 请求尝试访问托管在 `redseclabs.com/index.html` 的 `index.html` 文件。

**HTTP 请求**

```
GET /index.html HTTP/1.1
Host: www.redseclabs.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/90.0.4430.212 Safari/537.36
Referer: https://google.com
```

- `GET /index.html HTTP/1.1`：GET 表示此请求使用的 HTTP 方法，后跟请求的资源（这里是 `index.html`），然后是 HTTP 协议版本（HTTP/1.0）。

- `Host: www.redseclabs.com`：主机字段指提交请求的目标主机。主机字段是必需的，因为一个 IP 地址可以托管多个网站或虚拟主机。

- `User-Agent: Mozilla/5.0`：用户代理字段指示用于访问网站的浏览器和操作系统。它通常用于提供自定义页面，特别是为确保与不同浏览器的跨兼容性。用户代理字符串中的 `Mozilla/5.0` 是历史遗留问题；大多数浏览器出于兼容性原因以 `Mozilla/5.0` 开头。字符串末尾的 `Chrome/90.0.4430.212` 表示浏览器版本。

- `Referer: www.google.com`：“Referer”标头用于向服务器指示用户来自哪个网页的 URL。例如，如果 Referer 标头显示 `www.google.com`，则表示用户通过点击 Google 搜索上的链接到达当前网站。

此 HTTP 请求将产生以下 HTTP 响应：

**示例：HTTP 响应**

```
HTTP/1.1 200 OK
Date: Sun, 26 Nov 2023 12:00:00 GMT
Server: Apache/2.4.41 (Unix)
Content-Length: 450
Content-Type: text/html; charset=UTF-8
Connection: close

<html><body> <h1>Welcome to RedSecLabs!</h1></body></html>
```

逐行分析 HTTP 响应：

- `HTTP/1.1 200 OK`：该字段表示使用 HTTP 1.1 协议，并且请求已成功处理，HTTP 状态码 200 表明了这一点。

- `Date: Sun, 26 Nov 2023 12:00:00 GMT`：该字段提供响应发送时的时间戳。

- `Server: Apache/2.4.41 (Unix)`：该字段表示服务器正在运行 Apache 2.4.41，并托管在 Unix 操作系统上。披露此字段可能有助于攻击者，因此它不是强制性的，可以删除甚至替换为虚构值。

- `Content-Length: 450`：该字段指定响应内容的大小（以字节为单位），此处为 450 字节。

- `Content-Type: text/html; charset=UTF-8`：该字段指示发送的内容类型（HTML）和所使用的字符编码，即“UTF-8”。

- `Connection: close`：该字段表示一旦响应交付完毕，TCP/IP 套接字将关闭，要求用户在进一步通信之前打开新的套接字。或者，它可以设置为“Keep-Alive”，这将保持连接打开以用于后续请求。

### 1.2.3 HTTP 响应码

HTTP 响应码由三位数字表示，指示响应的状态。每个状态码代表不同的响应类别：

**表 1.1 常见 HTTP 响应码**

| 代码范围 | 类别 | 描述 |
|--------|------|------|
| 1xx    | 信息性 | 用于指示服务器正在更改协议，如 HTTP 切换到 WebSocket。 |
| 2xx    | 成功 | 表示请求已被正确理解。 |
| 3xx    | 重定向 | 表示需要进一步操作才能完成请求，例如重定向到新 URL。 |
| 4xx    | 客户端错误 | 表示请求包含不正确或不准确的信息。 |
| 5xx    | 服务器错误 | 表示请求在处理过程中遇到了问题。 |

### 1.2.4 HTTP 请求方法

HTTP 包含多种方法，但最基础的是 GET 和 POST。虽然这些方法在 Web 交互中常用，但其他方法是可选的，可能用于特定目的。GET 传统上用于检索内容，POST 用于向服务器提交内容。但是，GET 也可以用于向服务器发送内容。让我们以登录表单为例，使用 GET 请求处理用户名和密码来验证用户。

**请求**

```
GET /login.php?username=myusername&password=mypassword HTTP/1.1
Host: www.redseclabs.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/90.0.4430.212 Safari/537.36
Content-Type: application/x-www-form-urlencoded
Content-Length: 34
Connection: close
```

从安全角度来看，这种方法存在几个问题：

- GET 请求会记录在服务器日志中，因此任何可以访问这些日志的人（例如利用安全漏洞的未授权用户）都可能看到完整的 URL。当端点无意中泄漏日志或日志被未经授权的访问（超出预期的管理监督）时，这便成为一个问题。
- 浏览器和中间代理可能会缓存 GET 请求。
- 用户可能会将包含敏感信息的此类 URL 加入书签，并且在分享时可能无意中暴露敏感数据。

现在，让我们看看相同的请求和参数通过 POST 请求处理时的样子：

**请求**

```
POST /login.php HTTP/1.1
Host: www.redseclabs.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/90.0.4430.212 Safari/537.36
Content-Type: application/x-www-form-urlencoded
Content-Length: 34
Connection: close

username=myusername&password=mypassword
```

用户名和密码不是 URL 的一部分，而是 HTTP 请求正文的一部分，因此不易受到上述安全问题的影响。

## 1.3 HTTP 标头中的常见漏洞

让我们讨论一些可能因配置错误而产生的常见漏洞。我们将在本书相应章节中详细讨论这些漏洞。

### 1.3.1 基于 User-Agent 的欺骗

User-Agent 值可以被篡改，因此服务器不能信任它。然而，许多管理员仍然倾向于基于用户代理实施速率限制和其他安全机制。

### 1.3.2 Host 标头注入

如果服务器不验证 Host 标头，攻击者可能注入恶意 Host 值。这可能导致 Web 缓存投毒、密码重置以及将用户重定向到恶意网站等攻击。

### 1.3.3 跨域 Referer 泄漏

跨域 Referer 泄漏发生在 Referer URL 包含敏感信息（如会话 ID、令牌和密码）且用户导航到不同源时。例如，在以下请求中，Referer 标头包含会话令牌，该令牌泄漏到免费图片上传域名。控制该网站的攻击者可能利用此令牌劫持受害者会话并接管帐户。

网站可以通过元标签或 HTTP 标头包含“Referer-Policy”，以建议在导航到不同网站时何时包含 Referer 标头。以下是一些配置：

- `no-referrer`：从不发送 Referer 标头。
- `same-origin`：仅对同源请求发送 Referer 标头。
- `strict-origin-when-cross-origin`：对于跨源请求，仅发送源（origin）。

## 1.4 HTTP/2

HTTP 1.1 的最新升级是 HTTP/2。它在速度和性能方面提供了显著改进。多项关键增强使 HTTP/2 更高效地工作，例如多路复用（Multiplexing），它允许通过单个连接同时交付多个资源。由于这些改进，网站不再需要将内容拆分到多个域。单个连接可以处理多个请求，从而有效减少延迟。

例如，在 HTTP 1.1 中，当用户想在 YouTube 或其他视频流平台上观看视频时，过程涉及加载各种页面元素，如视频文件、脚本、CSS 文件和 JavaScript。这种加载通过使用多个 TCP 连接来管理。然而，浏览器通常限制到单个域名的并发连接数，这可能在包含许多资源的页面上造成瓶颈。

HTTP/2 还具有另一个功能“服务器推送”（server push），允许服务器通过预测客户端需要什么来向客户端发送内容。相反，服务器推送可能被服务器滥用，例如向客户端推送恶意资源或欺骗现有对象。类似地，在 HTTP/2 中，攻击者可以发送包含过多标头字段大小的超大标头帧。服务器可能根据标头大小分配内存，从而导致拒绝服务。

## 1.5 现代 Web 应用程序的演进

在过去十年左右的时间里，Web 在技术栈、架构和基础设施方面发生了重大转变。Web 服务和 RESTful API（应用程序编程接口）的使用已变得普遍，促进了异构服务之间的集成。这种演进反映了行业向可扩展性、效率和可靠性转变的大趋势。以下是这些发展的总结。

### 1.5.1 架构的转变

过去，许多应用程序被构建为单体结构，这意味着所有组件都紧密耦合且相互依赖。这意味着一个组件的故障可能导致整个应用程序崩溃，从而形成单点故障。最近，已经出现向微服务架构的重大转变。这是通过将应用程序分解为更小、独立运行的功能单元来实现的。我们将在第 11 章详细讨论微服务和安全问题。

### 1.5.2 技术栈的演进

技术栈的演进反映了 Web 应用程序开发和部署方式的转变。新技术和架构（如微服务和无服务器计算）以及较新的数据库技术（如 NoSQL）已经出现，为传统的 LAMP 堆栈提供了替代方案。让我们了解不同的堆栈。

### 1.5.3 LAMP 堆栈

在撰写本书时，PHP 被 76.6% 的网站使用。大量使用 PHP 作为服务器端编程语言的开发人员也更倾向于使用 Linux 作为操作系统，Apache 作为 HTTP 服务器，MySQL 作为数据库服务器。这些技术共同构成了所谓的 LAMP 堆栈，即 Linux、Apache、MySQL 和 PHP/Perl/Python 的缩写。多年来，这些组件中的每一个都在不断演进。

Linux 持续更新，仍然是服务器环境的首选。虽然 Apache 被广泛使用，但它面临着 Nginx 等 Web 服务器的竞争。MySQL 在很大程度上仍然流行，但 PostgreSQL 等替代数据库系统已经出现。PHP 随着时间的推移有了显著改进。Python 的受欢迎程度不断增长，尤其是在新兴领域，而 Perl 已变得不那么突出。

### 1.5.4 MEAN/MERN 堆栈

虽然构成 LAMP 的所有单个组件都在不断更新，但开发人员已逐渐远离这些组件（除了 Linux），并且已转向更先进的堆栈，如 MEAN/MERN 堆栈。

MEAN（MongoDB, Express.js, Angular, Node.js）和 MERN（MongoDB, Express.js, React, Node.js）是流行的开发堆栈。这些堆栈在客户端和服务器端提供统一的语言，即 JavaScript，使开发更高效和流畅。NoSQL 数据库（如 MongoDB、Cassandra 和 Redis）的采用为传统关系数据库提供了替代方案。虽然 MERN 已成为流行选择，但它也带来了自身的安全问题，例如引入了新型漏洞，如 Node.js 注入、NoSQL 注入、依赖注入等。

### 1.5.5 单页应用程序（SPA）

单页应用程序最近流行起来，旨在改善用户体验。与传统网站不同，传统网站在每次导航或提交表单时都会从服务器获取新页面，而 SPA 最初只加载整个网页及其所有组件一次。初始加载后，SPA 动态更新同一页面上的内容，无需重新加载网页。

SPA 通常依赖 RESTful API 或 Graph API 来获取数据，使其适合与微服务架构集成。它们使用 JavaScript 框架（如 AngularJS、React 等）构建。

由于严重依赖 JavaScript 来渲染和动态更新内容，SPA 通常容易受到基于 DOM 的跨站脚本攻击。我们将在第 4 章探讨这一点。

### 1.5.6 云组件的使用

云计算通过增强可扩展性极大地促进了微服务的开发。Docker 和 Kubernetes 等技术对于管理微服务至关重要，并提供容器化以实现有效的隔离和扩展。

同样，在 Web 开发实践中，DevOps 文化已变得突出，侧重于开发和运维团队之间的协作和自动化。与此相辅相成的是，CI/CD 流水线自动化了软件交付过程，在容器化环境中实现更快、更高效的发布。

### 1.5.7 无服务器架构

在 SPA 和微服务部署的背景下，无服务器架构已变得越来越流行。尽管名称如此，无服务器架构确实涉及服务器。在这种模型中，云提供商负责管理服务器；开发人员编写代码，并且只为代码执行时间付费。无服务器架构有其自身的安全挑战。我们将在 Web 服务章节（第 11 章）中深入探讨。

## 1.6 理解数据编码

Web 应用程序中的编码用于确保通信遵循指定的一组规则和标准。URL 只能包含有限的字符集，主要是字母数字（字母和数字）和某些特殊字符。当输入插入到允许集之外时，这些字符需要被编码以防止歧义。

编码和解码的过程通常在后台进行：用户可以输入各种字符，浏览器和应用程序负责编码和解码。当用户向浏览器提供不允许的字符时，它会在处理请求之前自动编码。

图 1.1（[https://perishablepress.com/stop-using-unsafe-characters-in-urls/](https://perishablepress.com/stop-using-unsafe-characters-in-urls/)）展示了一个图表，解释了哪些字符可以被视为“安全”以及哪些应该被编码。

值得注意的是，保留字符仅在其用途超出定义范围时才需要编码。让我们讨论 Web 应用程序中使用的主要数据编码类型：

- URL 编码
- HTML 编码
- Base64 编码
- Unicode 编码

### 1.6.1 URL 编码

URL 编码，也称为百分号编码，是一种用于对 URL 中的保留字符进行编码的过程。在 URL 编码中，不属于 URL 允许集的字符被替换为百分号（%）后跟其十六进制值。例如，与符号（“&”）通常用作查询字符串中的分隔符，必须对其进行编码以防止歧义。例如，考虑以下 URL：

**示例**

```
https://example.com/login.php?username=tmgm&password=t&mgm
```

在此示例中，密码包含与符号“&”，如果不编码，它将被视为参数分隔符，将密码设置为“t”和“mgm”。为防止这种情况，与符号“&”被编码为“%26”，最终 URL 如下：

```
https://example.com/login.php?username=tmgm&password=t%26mgm
```

以下是表 1.2，列出了常见编码字符及其编码版本：

**表 1.2 常见字符及其编码版本**

| 字符 | 编码版本 |
|------|----------|
| 空格 | %20 |
| 双引号（“） | %22 |
| 小于号（<） | %3C |
| 大于号（>） | %3E |
| 井号（#） | %23 |
| 与号（&） | %26 |
| 斜杠（/） | %2F |
| 加号（+） | %2B |
| 等号（=） | %3D |

### 1.6.2 双重编码

在双重编码中，字符被编码两次：第一级编码将字符转换为百分号编码形式。第二级编码应用于百分号编码后的字符。换句话说，这意味着它将应用于百分号（%）本身，连同编码后的字符十六进制值再次被编码。

例如，如果你想双重编码字符“<”，你首先将其编码为 %3C。然后，将“%”编码为 %25，得到 %253C 作为双重编码形式。

这是一种常见技术，可用于绕过对 URL 仅解码一次的 WAF 和应用层过滤器。如果 WAF 只解码一次，它会将 %253C 视为无害，而不会意识到第二次解码会将其转换为“<”字符。我们将在后续章节中深入探讨这些技术。

### 1.6.3 HTML 编码

在 HTML 中，某些字符具有特殊含义，如果处理不当可能导致歧义。例如，字符“<”和“>”可以表示 HTML 标签的开头和结尾。为确保这些字符以文本格式显示，而不是被解释为 HTML 语法，它们必须进行 HTML 编码。

HTML 编码涉及将这些特殊字符替换为字符实体。例如，小于号“<”在 HTML 中可以替换为“`&lt;`”，大于号“>”则使用“`&gt;`”。此外，HTML 编码不仅限于在 HTML 语法中具有特殊含义的字符。它还可以表示标准键盘上不易获得的字符。例如，版权符号“©”在 HTML 中可以表示为“`&copy;`”。

根据 HTML 规范，所有字符引用必须以与号“&”开头，后面可以跟多种变体，如十进制和十六进制编码。以下是表示这些字符的多种方式：

**表 1.3 表示字符的多种方式**

| 字符 | 命名实体 | 十进制编码 | 十六进制编码 |
|------|----------|------------|--------------|
| <    | &lt;     | &#60;      | &#X3C;       |
|      |          | &#000060   | &#0x3C       |
|      |          |            | &#0000x3C    |
| >    | &gt;     | &#62;      | &#X3e;       |
|      |          | &#000062   | &#0x3e       |
|      |          |            | &#0000x3e    |
| '    | &apos;   | &#39;      | &#X27;       |
|      |          | &#000039   | &#0x27       |
|      |          |            | &#000027     |
| "    | &quot;   | &#34;      | &#X22;       |
|      |          | &#000034   | &#0x22       |
|      |          |            | &#000022     |

注意：从表中可以看出，我们可以在十进制和十六进制形式中使用前导零。

### 1.6.4 Base64 编码

Base64 编码可用于将二进制数据表示为 ASCII 字符串。

Base64 最常见的用途之一是安全地传输电子邮件附件，因为电子邮件服务器经常会更改或误解某些字符（如换行符），导致数据歧义或损坏。

例如，考虑字符串“Hello\nWorld”，其中“\n”表示换行符。该字符串将被视为：

**示例**

```
Hello
World
```

该字符串在字母“o”之后包含一个换行符。在 ASCII 中，此字符串由十进制值表示：

**示例**

```
72 101 108 108 111 10 119 111 114 108 100
```

在此示例中，字节序列“10”表示“换行符”字符。电子邮件系统可能无法正确解释此字符。因此，通过使用 base64 编码对这些字符进行编码，我们可以将它们表示为 ASCII 字符，从而消除在电子邮件系统中存在问题的换行符等字符。

所有 Web 编程语言都广泛支持 base64 编码和解码。JavaScript 提供了 `btoa()` 和 `atob()` 函数来处理 base64。

值得一提的是，开发人员经常将 base64 误认为是加密方案而非编码方案。即使在真实世界的渗透测试中，你也可能发现敏感信息以 base64 编码，从而导致敏感数据暴露。图 1.2 演示了使用 `btoa` 和 `atob` 函数进行 base64 编码/解码：

## 1.7 浏览器简介

浏览器作为访问 Web 应用程序的界面，负责解释并向最终用户显示内容。它们主要负责通过处理 HTML、CSS 和 JavaScript 来渲染页面。

在不断发展的 Web 安全背景下，Web 浏览器的角色已不仅仅是渲染页面，还不断引入安全控制以保护用户的隐私和安全。例如，为保护用户隐私，许多浏览器提供内置措施，如增强的 Cookie 控制、隐私浏览模式、阻止第三方跟踪器等。在安全方面，浏览器已实施了某些内置安全策略，如同源策略（SOP），该策略限制不同源的内容如何在浏览器内交互，而一些可选的安全机制则以标头形式实现，可由 Web 管理员用于增强安全性。

浏览器支持扩展和插件，提供额外功能，如广告拦截、密码管理等。虽然这些额外功能可以改善用户的安全性，但这些功能也可能被攻击者武器化并成为弱点。详细讨论浏览器是一个复杂的话题，超出了本章的范围。图 1.3 给出了浏览器核心组件的高层概述：

**图 1.3 浏览器内部结构高层概述**

让我们简要介绍每个组件。

### 1.7.1 用户界面

这代表 HTML 和 CSS，并在屏幕上显示解析后的内容。浏览器的实用工具包括除呈现网页的窗口之外你所看到的一切。例如，地址栏、后退/前进按钮和书签菜单都属于用户界面的一部分。

### 1.7.2 浏览器引擎

它充当用户界面和渲染引擎之间的接口。例如，当用户与浏览器界面交互时，如输入 URL、点击链接或与表单交互，浏览器引擎负责处理命令。

### 1.7.3 渲染引擎

渲染引擎是 Web 浏览器不可或缺的一部分，它基本上将 HTML、CSS 和 JavaScript 转换为可视化和交互式页面。每个浏览器都使用特定的引擎：Chrome 和 Opera 使用 Blink，Firefox 使用 Gecko，Safari 使用 WebKit。这些引擎处理标记和脚本以创建网页的文档对象模型（DOM），应用 CSS 样式，执行 JavaScript 以实现动态内容，然后在屏幕上渲染页面的布局和视觉表现。

鉴于每个 Web 浏览器都使用特定的渲染引擎，Blink、Gecko 或 WebKit 等引擎中的漏洞可能使依赖该特定引擎的所有浏览器面临潜在安全风险。

### 1.7.4 网络

该组件负责进行底层网络调用，如 HTTP 和 DNS。

### 1.7.5 UI 后端

用于访问底层操作系统方法，如组合框、用户框等。

### 1.7.6 JavaScript 解释器

该组件负责解析和执行网页中的 JavaScript 代码。

### 1.7.7 数据存储

该组件负责在客户端存储数据。这包括 Cookie、Web Storage、IndexedDB、WebSQL 和 FileSystem 等机制。

## 1.8 核心浏览器安全策略与机制

浏览器厂商已引入多种安全策略和机制来保护其用户。这些策略默认实施，或者可能由每个浏览器以不同方式实施。为了保护用户，浏览器厂商已经实施了广泛的安全策略和机制。这些策略的范围从精细控制可加载的资源类型，到在不同网站之间强制执行严格隔离。

### 1.8.1 同源策略

同源策略（SOP）是浏览器中最基本、最核心的策略之一。该策略本质上阻止一个源中的 Web 页面能够访问不同源中的属性。源通常指方案（协议）、域名和端口的组合。简单来说，如果两个网页的方案、域名和端口号匹配，则认为它们属于同一源。

值得一提的是，SOP 在性质上并不一致且异构，因此其在不同浏览器中的实现可能有所不同。过去 Internet Explorer 的一个示例是仅包含方案和主机，但不考虑端口。

**图 1.4 同源策略中的源**

为了更好地理解 SOP，让我们看一个托管在 `output.jsbin.com` 的代码示例，该代码使用 Ajax 请求获取 `gmail.com` 的响应，并使用 `document.write` 函数将其写入网页。

**代码**

```html
<script>
xhr = new XMLHttpRequest();
xhr.open('GET', 'www.gmail.com', true);
xhr.onreadystatechange = function () {
    if (xhr.readyState === XMLHttpRequest.DONE && xhr.status === 200) {
        document.write(xhr.responseText);
    }
};
xhr.send();
</script>
```

图 1.5 中的截图说明，由于主机名不匹配，从 `https://output.jsbin.com` 访问 `www.gmail.com` 已被阻止。

**图 1.5 SOP 违规**

下表（表 1.5）概述了不同源之间的交互规则，并指定了源被视为相同的条件。

**表 1.5 不同源之间的交互规则**

| 源 1 | 源 2 | 同源？ |
|------|------|--------|
| `http://store.example.com/page.html` | `http://store.example.com/newpage.html` | **是** |
| `http://store.example.com/page.html` | `http://news.example.com/page.html` | **否** |
| `http://store.example.com:80/page.html` | `http://store.example.com:8080/page.html` | **否** |
| `https://store.example.com:8443/page.html` | `about:blank` | **是** |
| `https://store.example.com/dir/page.html` | `https://store.example.com/dir/subdir/page.html` | **是** |

**注意**：在表中，`store.example.com` 和 `about:blank` 被显示为同源。这可能让部分读者感到困惑。“about:blank”没有自己的源，它会继承创建它的文档的源。例如，如果 `https://store.example.com` 上的页面打开了一个指向 `about:blank` 的新窗口，那么这个 `about:blank` 页面会继承 `https://store.example.com` 的源。

### 1.8.2 内容安全策略

内容安全策略（CSP）是一种被所有现代浏览器广泛支持的安全策略，旨在缓解注入攻击，如 XSS、点击劫持以及其他由未经授权的脚本执行导致的代码注入攻击。该策略是可选的，可以通过“Content-Security-Policy”标头实现。让我们看一个示例，展示如何设置 CSP，仅允许浏览器从指定资源 `http://code.jquery.com/jquery-1.11.0.min.js` 包含 JavaScript。

**示例：**

```
Content-Security-Policy: script-src http://code.jquery.com/jquery-1.11.0.min.js;
```

此策略将确保仅执行来自指定 jQuery 库的 JavaScript，阻止任何其他非此来源的脚本。

CSP 最初作为非常严格的策略引入，因此从实际角度来看，破坏大量网站并不现实；然而，随着引入更多级别，策略变得不那么严格且更实用。

CSP 级别从具有基本指令的 Level 1 发展到具有高级控制（允许精细管理资源）的 Level 3。CSP 定义了控制页面可以加载哪些资源的指令，允许开发人员指定要列入白名单的来源。这包括脚本、样式、图像等。

- **CSP Level 1** — 此级别引入了基本指令，允许开发人员指定要列入白名单的来源，包括脚本、样式、图像等。
- **CSP Level 2** — 此级别为开发人员提供了更多控制权，可以控制哪些源可以将你的站点嵌入到框架中。
- **CSP Level 3** — 此级别目前正在开发中，只有部分已在现代浏览器中实现，并提供更精细的资源交互控制。

我们将在后续章节中进一步探讨 CSP 及其因配置错误而可能的绕过。

### 1.8.3 HTTP Cookie

如前所述，HTTP 是无状态协议，因此不了解来自同一客户端的先前交互。为克服此限制，使用了 HTTP Cookie。当客户端首次向 Web 服务器提交请求时，服务器会响应一个“Set-Cookie”标头，其中包含 Cookie 值。然后 Cookie 值存储在客户端的浏览器中。在之后每次向 Web 服务器发出请求时，浏览器都会提交该 Cookie。这使得服务器能够识别客户端并维护会话状态。

Cookie 基于 Domain 和 Path 属性定义作用域，而同源策略（SOP）依赖于方案、主机和端口。例如，考虑以下 Cookie：

**示例**

```
Set-Cookie: key=anyvalue; domain=example.com; Path=/search/
```

在此示例中，为 `example.com` 域设置了 Cookie，并且特别限定在 `/search/` 路径下。这意味着仅当请求指向 `example.com` 域内且处于 `/search/` 路径下的 URL 时，浏览器才会发送包含“key=value”的 Cookie。

与 SOP 不同，Cookie 并非严格绑定到单一源，只要 Cookie 的 Domain 和 Path 属性允许，一个源的网站可以设置一个 Cookie，该 Cookie 可以被另一个源读取。

#### 1.8.3.1 域级 Cookie 作用域

如上所述，在 Cookie 上下文中，Cookie 的作用域通过 domain 和 path 参数设置。然而，如果作用域设置得过于宽松，可能导致不一致。考虑以下示例：

**示例**

```
Set-Cookie: key=anyvalue; domain=.example.com
```

在此示例中，Cookie 的 domain 属性设置为 `.example.com`，使用了前导点，这是用于设置 Cookie 的旧约定。当 Cookie 的域设置为 `.example.com` 时，它将对其所有子域（如 `sub.example.com` 或 `tmgm.sub.example.com`）可访问。

现代浏览器已标准化此行为，当你设置 `domain=example.com`（无前导点）或 `domain=.example.com`（有前导点）时，Cookie 将在指定域及其所有子域中可访问。

注意：限制和限定作用域的正确方法是根本不包含 domain 属性。

#### 1.8.3.2 Cookie 投掷漏洞

如前所述，当 Cookie 未紧密限定在当前域时，它可以被主域的子域访问。如果攻击者设法控制任何子域，此漏洞的影响可能很广泛，包括攻击者固定会话令牌，从而导致帐户接管。以下是攻击的工作原理：

**步骤 1**：控制 `vulnerable.example.com` 的攻击者将名为“sessionID”的 Cookie 设置为已知值。

**示例**

```
Set-Cookie: sessionID=valueknowntoattacker; domain=example.com
```

**步骤 2**：当受害者访问攻击者控制的子域（`vulnerable.example.com`）时，浏览器存储该值。

**步骤 3**：受害者访问 `example.com` 并使用固定的会话 ID 登录应用程序。

**步骤 4**：当受害者登录应用程序时，应用程序不会生成新的 SessionID。

**步骤 5**：攻击者使用相同的会话 ID 接管受害者的帐户。

即使已经设置了具有特定名称的 Cookie，Cookie 投掷也是可能的。当浏览器收到两个同名的 Cookie 时，浏览器会将请求视为有效并同时发送两者。

**示例**

```
Set-Cookie: SessionID=setbytheserver; SessionID=attackerknow; domain=.example.com
```

大多数应用程序在重复的情况下将处理第一个参数。在这种情况下，我们可以通过添加更长的路径来强制我们的 Cookie。这是因为根据 RFC 6265 的文档，路径较长的 Cookie 具有优先权（[https://datatracker.ietf.org/doc/html/rfc6265#section-5.4](https://datatracker.ietf.org/doc/html/rfc6265#section-5.4)）。

**图 1.6 RFC 6265 关于 Cookie 顺序的摘录**

**较短路径的示例**

```
Set-Cookie: SessionID=setbytheserver; domain=.example.com; path=/
```

**攻击者设置较长路径的示例**

```
Set-Cookie: SessionID=attackerknow; domain=.example.com; path=/admin
```

当浏览器处理这些标头时，它将存储两个 Cookie。但是，当访问同时匹配两个 Cookie 的路径（如 `/admin`）时，由攻击者设置的具有较长路径的 Cookie 将优先。

#### 1.8.3.3 Cookie 炸弹漏洞

Cookie 炸弹发生在网站设置过多 Cookie 或几个非常大的 Cookie 时。浏览器对可以存储的 Cookie 数量有限制，限制是针对每个域的 Cookie 数量或 Cookie 大小。在服务器端，过多的 Cookie 可能导致 HTTP 标头中发送过多数据，这会增加加载时间并可能使服务器过载。

让我们看以下 JavaScript 代码，它将设置多个 Cookie，从“testcookie1”开始，递增到 99，每个值的长度为 4000 个“A”：

**示例**

```javascript
// 设置域
let baseDomain = 'hackerone.com';
// 为 Cookie 值创建 4000 个 'A' 的字符串
let cookieValue = 'A'.repeat(4000);
// 循环设置多个 Cookie
for (let cookieNum = 1; cookieNum < 99; cookieNum++) {
    // 设置具有递增名称和长字符串值的 Cookie
    document.cookie = 'testCookie${cookieNum}=${cookieValue};Domain=${baseDomain}';
}
```

在 `hackerone.com` 上下文中执行此 JavaScript 代码后，我们可以看到设置了 Cookie 值。结果，访问 `hackerone.com` 及其子域将导致错误。

**图 1.7 在 hackerone.com 上设置的 Cookie**

**图 1.8 设置大量 Cookie 后 hackerone.com 无法访问**

由于同源策略（SOP）的限制，你不能跨域设置此 Cookie；但是，应用程序中的漏洞代码可能允许用户通过用户可控参数跨域设置 Cookie。

#### 1.8.3.4 会话过期与验证

HTTP Cookie 包含“Max-age”和“Expires”等属性，允许用户设置 Cookie。“Max-age”定义 Cookie 的有效最大时间。例如，如果 max-age 设置为“3600”，意味着 Cookie 有效期为 3600 秒，即一小时。过期后，Cookie 将自动从浏览器中删除。

**示例**

```
Set-Cookie: key=value; Max-Age=3600; Path=/path; Domain=example.com
```

“Expires”属性定义 Cookie 过期的具体时间。

**示例**

```
Set-Cookie: key=value; Expires=Sun, 31 Dec 2024 23:59:59 GMT; Path=/path; domain=example.com
```

明显的安全风险是，如果 Cookie 过期时间设置得很长，获得 Cookie 访问权限的攻击者将能够在较长时间内保持访问权限。一些 Web 应用程序选择接受此风险，并采取补偿控制，例如在访问敏感功能或检测到差异时重新验证用户身份。

#### 1.8.3.5 Cookie 保护

对于 HTTP Cookie 的安全而言，有两个重要的标志会产生重大影响。其中一个标志是“secure”，它向浏览器指示此 Cookie 仅应在安全连接（如 TLS 连接）中发送。另一个与安全相关的标志是“httponly”。此标志指示浏览器不允许从 JavaScript 访问。以下是实现示例：

**示例**

```
Set-Cookie: key=value; Max-Age=3600; Path=/; domain=example.com; HttpOnly; Secure
```

HTTP Cookie 也可以设置为 Same Site Origin，这将告诉浏览器何时发送 Cookie。这用作针对跨站请求伪造（CSRF）的保护机制，将在第 5 章相关部分讨论。

### 1.8.4 Iframe Sandbox

“Iframe”标签是一个强大的 HTML 元素，允许网站将网页嵌入当前文档中。当页面在 Iframe 中加载时，它将从目标地址加载所有内容，包括 HTML、CSS 和 JavaScript。这存在安全风险，因为从外部网页加载的内容可能是恶意的，可能导致父网站的安全受到损害。

为解决此问题，HTML5 规范为 Iframe 引入了“sandbox”属性。这提供了对应加载内容类型的精细控制。以下是使用 sandbox 属性加载 `example.com` 的示例：

**示例**

```html
<iframe sandbox src="http://example.com/"></iframe>
```

sandbox 属性的默认设置非常严格。这些限制包括阻止 JavaScript 执行、禁用表单提交等。但是，它也允许开发人员通过各种属性（如 allow-forms、allow-popups、allow-same-origin 和 allow-scripts）灵活地微调允许的内容。此外，CSP 包含一个 sandbox 指令，当实现时，会对整个文档应用类似的限制。

### 1.8.5 子资源完整性检查

Web 应用程序经常加载外部资源，如 JavaScript 库和 CSS 文件。这些资源有时托管在第三方服务器上，如 `code.jquery.com` 上的 jQuery。然而，如果此类域名被入侵，其内容被替换为恶意版本，则嵌入这些资源的任何网站也可能受到威胁。

这是因为 JavaScript 能够操作网页，可能导致敏感数据被盗，并成为传播恶意软件的载体。虽然 CSP 允许将域名列入白名单以加载资源，但它无法保护来自白名单域名的已入侵外部资源。

为解决此问题，浏览器引入了一种称为子资源完整性（SRI）检查的安全功能，该功能确保脚本自首次加载以来未被修改。这是通过 integrity 属性实现的，该属性接受包含外部文件校验和或哈希值（如 SHA-256、SHA-384 或 SHA-512）的输入，用于确保所加载文件的完整性。

**示例**

```html
<script src="https://code.jquery.com/jquery-3.6.0.min.js"
        integrity="sha256-tmgm3212 . . ."
        crossorigin="anonymous">
</script>
```

在此示例中，integrity 属性包含 jquery-3.6.0.min.js 预期内容的哈希值。如果此文件中的内容更改，哈希值将改变，从而阻止更改后的脚本加载。

### 1.8.6 HTTP 严格传输安全（HSTS）

网站可以使用 HTTP 到 HTTPS 的强制重定向来确保网站仅能通过 HTTPS 访问。但是，仅此并不能防止协议降级攻击。这是一种加密攻击形式，用于将加密连接降级为弱操作模式，从而使攻击者轻松拦截和解密数据。

为解决此问题，浏览器引入了 HTTP 严格传输安全（HSTS），它指示浏览器将所有 HTTP 请求转换为 HTTPS，从而防止攻击者利用不安全的 HTTP 连接。

HSTS 策略可以通过 HTTP 响应中的“Strict-Transport-Security”标头设置。示例如下：

**示例**

```
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

该标头使用 max-age 指令，指定浏览器应记住该站点只能使用 HTTPS 访问的持续时间（以秒为单位）。HSTS 还包含“includeSubdomains”属性，表示该策略应适用于所有子域，而不仅仅是主域。

HSTS 可以包含“preload”指令，这是一个硬编码到浏览器中的列表，用于始终使用 HTTPS，甚至在首次与网站交互之前。HSTS 预加载列表是主要 Web 浏览器共同努力的结果。网站所有者必须满足特定标准才能被列入 HSTS 预加载列表。

**示例**

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

## 1.9 策略例外与策略绕过

从本章前述文献中可以明显看出，安全策略在本质上并非严格，而是为开发人员和网站所有者提供了灵活性。这种灵活性通常通过使用策略例外来实现。在这里，区分策略绕过和策略例外非常重要。

策略绕过通常被视为浏览器内的漏洞，涉及利用漏洞来规避有效实施的策略，而策略例外则被定义为浏览器允许受控规避策略的合法场景。

### 1.9.1 SOP 绕过类型

SOP 绕过已变得普遍，主要由于文档对象模型（DOM）和 JavaScript 的日益复杂。再加上服务器端功能（如重定向）的复杂性，使得问题更加严重。大多数 SOP 绕过是逻辑错误，即浏览器内不同层和组件之间的逻辑混淆或不匹配导致的。

例如，一个组件可能检测到空字节并停止代码执行，而另一个组件选择完全忽略它并执行代码。

若干 SOP 绕过不仅限于单个浏览器，因为它们存在于共享组件中，可能影响多个浏览器。例如，WebKit 渲染引擎中的绕过会影响所有基于同一引擎构建的浏览器。类似地，在 Java 等插件中发现的 SOP 绕过可能影响所有支持这些插件的浏览器。

SOP 绕过有几种分类方式；不过，与 Schwenk 等人（2017）的工作类似，我们将其分为四种不同类型：部分读取、完全读取、部分写入、完全读取和写入。对于每个类别，也提到了属于各自类别的潜在攻击。

**表 1.6 SOP 绕过的分类**

| SOP 绕过类型 | 描述 | 攻击类别 |
|-------------|------|----------|
| **部分读取** | 在 SOP 背景下，部分读取访问意味着在跨源通信期间读取某些属性/子属性，而这些操作在正常情况下是不被 SOP 允许的。此外，也存在某些例外情况，策略允许合法的部分读取，例如，设计上允许读取跨源图片的 “**width**” 和 “**height**” 参数。 | **侧信道攻击，如跨源 CSS 攻击、跨源数据泄漏** |
| **完全读取** | 在 SOP 背景下，完全读取访问意味着一个源上的脚本能够读取另一个源上网页的所有属性。此类别还包括规避本地文件访问限制的利用手段。 | **跨模式数据暴露攻击、跨文件攻击** |
| **部分写入** | 在 SOP 背景下，部分写入是指在跨源通信期间修改某些属性的能力。某些属性如 **location.href** 和 **location.hash**，无论其窗口对象是否属于不同源，都可以被修改。还有其他 DOM 属性（如 `window.name`）会跨源持久存在。 | **浏览器欺骗漏洞** |
| **完全读写** | 在 SOP 背景下，完全读写访问意味着一个源上的脚本能够读取和写入另一个源的所有属性。完全读写是 JavaScript 在另一个源上下文中执行的结果。我们没有将“完全写入”单独列为一类，因为在几乎所有情况下，完全写入都会导致 UXSS（通用跨站脚本）。 | **UXSS、跨区脚本** |

### 1.9.2 SOP 绕过——CVE-2007-0981

让我们来看一个由于层不匹配导致的 SOP 绕过的经典案例，该漏洞在 Firefox 旧版本中被发现，记录为 CVE-2007-0981。我们来看一下 POC：

```javascript
location.hostname = "evil.com\x00www.bing.com"
```

在此 POC 中，`location.hostname` 属性被设置为 `evil.com`，后跟一个空字节 `\x00`，然后是 `bing.com`。空字节在许多编程语言中被普遍视为字符串终止符。然而，在此情况下，DOM（渲染引擎的一部分）不将 `\x00` 视为字符串终止符。因此，它将 `evil.com\x00www.bing.com` 解释为 `bing.com` 的子域。



相反，DNS 解析器（网络层的一部分）确实将空字节识别为字符串终止符，并停止执行，将源视为 `evil.com`，忽略字符串的其余部分。因此，这将允许攻击者设置、更改或删除 `bing.com` 及其子域的 Cookie。

### 1.9.3 SOP 绕过——CVE-2011–3246

CVE-2011–3246 是一个类似示例，影响 Safari 浏览器的旧版本。SOP 绕过又是逻辑混淆和层不匹配的非常类似的情况。然而，这次不匹配发生在持久层中的 Cookie 存储部分和网络层中的 DNS 解析器之间。

**POC**

## 1.10 站点隔离

大多数 SOP 绕过的发生是由于不同源共享同一个渲染进程。这在跨站框架和弹出窗口中尤为常见。渲染进程负责处理网页的渲染，包括解析 HTML 和 CSS、执行 JavaScript 等功能。

以前，浏览器不会将不同源隔离到单独的渲染进程中。因此，当发现 SOP 绕过时，它允许来自一个源的脚本可能访问同一进程内另一个源的数据。

在站点隔离中，每个进程在其独立的渲染进程中运行，并在操作系统级别隔离。使用站点隔离，即使在渲染进程中发生 SOP 绕过，访问跨源数据也变得更加困难。

## 1.11 地址栏欺骗漏洞

根据 Google 的规则（[https://bughunters.google.com/about/rules/6625378258649088/google-and-alphabet-vulnerability-reward-program-vrp-rules](https://bughunters.google.com/about/rules/6625378258649088/google-and-alphabet-vulnerability-reward-program-vrp-rules)），地址栏是现代浏览器中唯一可以信赖的安全指示器。这似乎合乎逻辑，因为地址栏是用户确认自己是否在正确网站上的主要方式。

浏览器厂商意识到这一点，因此实施了内置控制，以确保浏览器中显示的内容与地址栏中的 URL 对应。

地址栏欺骗漏洞发生在地址栏指向用户试图访问的正确域，但该域的内容由攻击者控制时。在底层，基于浏览器的欺骗漏洞是通过利用浏览器中的缺陷来执行的，以制造用户正在合法域上的错觉。让我们看几个示例：

### 1.11.1 地址栏欺骗——示例 1

在 JavaScript 中，有几个函数允许用户延迟执行事件或以固定间隔执行事件。这些方法功能强大，可用于寻找地址栏欺骗漏洞。让我们看这种技术的一个示例：

**代码**

```html
<script>
w = window.open("www.facebook.com", "_blank");
setTimeout(function(){ w.document.write("<html>This is not Facebook</html>") }, 5000);
</script>
```

此脚本使用 `window.open` 函数在新窗口中打开 Facebook 网站，然后尝试在延迟五秒后使用 `document.write` 将页面内容覆盖为文本“This is not Facebook”。执行通过 `setTimeout` 函数延迟。在此过程中，浏览器可能无法更新地址栏，因此当地址栏仍然指向 `facebook.com` 时，实际显示的内容却由攻击者控制。以下是 Yandex 浏览器（CVE-2020–7369）使用相同技术保留地址栏的示例。

### 1.11.2 地址栏欺骗——示例 2

触发地址栏欺骗的另一种技术是使用不存在的端口来保留地址栏，同时使用 JavaScript 定时函数诱导适当延迟。让我们看这种技术的一个示例：

**代码**

```html
<script>
function spoof() {
    document.write("<h1>This is not Bing</h1>");
    document.location = "https://bing.com:1234";
    setInterval(function(){ document.location = "https://bing.com:1234"; }, 9800);
};
</script>
```

该脚本使用 `document.location` 函数将用户重定向到非标准端口“1234”上的 `https://bing.com`。根据浏览器的不同，由于此端口上没有服务运行，最终返回连接超时错误之前会花费一些时间。在此时间段内，当浏览器尝试加载 `https://bing.com:1234` 时，`setInterval` 函数被重复调用，导致再次重定向到该 URL，从而在定义的时间间隔内重复该过程。

此过程在地址栏中保留欺骗的 URL，同时允许攻击者修改网页内容。这将在每次间隔后触发，从而在过程中保留 URL，同时允许攻击者向文档写入内容，导致地址栏欺骗。理想情况下，浏览器的正确行为是在 `setInterval` 函数被调用进行重定向时，重定向到“about:blank”。

相同的 POC 已被多次用于在 Safari 浏览器以及其他流行浏览器（如 Microsoft Edge 和 DuckDuckGo）中触发地址栏欺骗漏洞。有趣的是，Safari 浏览器会隐藏端口，因此用户唯一可见的 URL 是 `bing.com`。

### 1.11.3 利用欺骗绕过反钓鱼过滤器

仅依赖地址栏中的 URL 来识别钓鱼页面的反钓鱼过滤器，在地址栏欺骗漏洞存在的情况下可能无法检测到此类页面。由于这些过滤器依赖来自地址栏的源信息，它们可能被欺骗的 URL 所欺骗。因此，如果浏览器在地址栏中保留欺骗的 URL，这些反钓鱼过滤器会将其视为合法。

即使在不易受地址栏欺骗漏洞影响的现代浏览器中，也有可能欺骗反钓鱼过滤器。这是因为一些现代浏览器会在用户尝试向新窗口写入内容时将地址栏更新为“about:blank”，而不是保留 URL，以此作为防止地址栏欺骗漏洞的措施。

然而，这种对策可以通过打开新窗口并向“about:blank”页面写入内容来欺骗反钓鱼过滤器。因此，当反钓鱼过滤器遇到“about:blank”时，它无法执行站点信誉检查，从而导致绕过。为了说明，考虑以下代码：

**代码**

```html
<script>
function spoof() {
    var x = open('about:blank');
    base64 = 'VE1HTQo=' // phishing page encoded in base64
    x.document.body.innerHTML = atob(base64);
}
</script>
```
