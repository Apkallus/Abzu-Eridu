## 脚本格式

NSE 脚本包含几个描述性字段、一个定义脚本何时执行的规则，以及一个包含实际脚本指令的 `action` 函数。描述性字段的赋值方式与任何其他 Lua 变量相同。它们的名称必须像本节所示那样使用小写字母。

### `description` 字段

`description` 字段描述了脚本测试的内容以及用户应注意的任何重要事项。根据脚本的复杂程度，描述的长度可能从几句话到几段不等。第一段应简要概述脚本功能，适合单独呈现给用户。后续段落可以提供更多的脚本细节。

### `categories` 字段

`categories` 字段定义脚本所属的一个或多个类别（参见[“脚本类别”](nse-usage.html#nse-categories)一节）。类别不区分大小写，可以按任意顺序指定。它们以数组风格的 Lua 表列出，如本例所示：

```lua
categories = {"default", "discovery", "safe"}
```

### `author` 字段

`author` 字段包含脚本作者的姓名，也可以包含联系信息（例如主页 URL）。我们不再建议包含电子邮件地址，因为垃圾邮件发送者可能会从 NSEDoc 网站抓取它们。此可选字段不被 NSE 使用，但可以让脚本作者获得应有的赞誉或责备。

### `license` 字段

Nmap 是一个社区项目，我们欢迎各种代码贡献，包括 NSE 脚本。因此，如果您编写了有价值的脚本，请不要独享！可选的 `license` 字段有助于确保我们拥有分发 Nmap 附带的所有脚本的合法权限。所有这些脚本目前都使用标准的 Nmap 许可证（在[“Nmap 版权与许可”](man-legal.html#nmap-copyright)一节中描述）。它们包含以下行：

```lua
license = "Same as Nmap--See https://nmap.org/book/man-legal.html"
```

Nmap 许可证类似于 GNU GPL。如果脚本作者愿意，也可以使用 BSD 风格的许可证（无广告条款）。对于 BSD 风格的许可证，请包含以下行：

```lua
license = "Simplified (2-clause) BSD license--See https://nmap.org/svn/docs/licenses/BSD-simplified"
```

### `dependencies` 字段

`dependencies` 字段是一个数组，包含应在此脚本之前运行的脚本名称（如果它们也被选中）。当一个脚本可以利用另一个脚本的结果时，就会使用此字段。例如，大多数 `smb-*` 脚本依赖于 `smb-brute`，因为 `smb-brute` 找到的帐户可能允许其他脚本获取更多信息。在 `dependencies` 中列出脚本并不会导致该脚本运行；它仍然需要通过 `--script` 选项或其他方式选中。`dependencies` 仅仅强制在 **已被选中** 的脚本之间进行排序。这是一个来自 `smb-os-discovery` 的 `dependencies` 表示例：

```lua
dependencies = {"smb-brute"}
```

`dependencies` 表是可选的。如果省略该字段，NSE 将假定脚本没有依赖项。

依赖项建立了脚本的内部顺序，为每个脚本分配一个称为“运行级别”的编号[^12]。在运行脚本时，您将在 NSE 的输出中看到每个脚本组的运行级别（以及总运行级别数）：

```screen
NSE: Script scanning 127.0.0.1.
NSE: Starting runlevel 1 (of 3) scan.
Initiating NSE at 17:38
Completed NSE at 17:38, 0.00s elapsed
NSE: Starting runlevel 2 (of 3) scan.
Initiating NSE at 17:38
Completed NSE at 17:38, 0.00s elapsed
NSE: Starting runlevel 3 (of 3) scan.
Initiating NSE at 17:38
Completed NSE at 17:38, 0.00s elapsed
NSE: Script Scanning completed.
```

### 规则

Nmap 使用脚本规则来确定是否应对目标运行脚本。规则是一个 Lua 函数，返回 `true` 或 `false`。仅当规则计算结果为 `true` 时，才会执行脚本的 `action` 函数。

脚本必须包含以下一个或多个函数来确定脚本何时运行：

- `prerule()`
- `hostrule(host)`
- `portrule(host, port)`
- `postrule()`

`prerule` 脚本在扫描任何主机之前的脚本预扫描阶段运行一次。`hostrule` 和 `portrule` 脚本在每批主机扫描之后运行。`postrule` 脚本在所有主机扫描完成后的脚本后扫描阶段运行一次。如果脚本具有多个规则，则可以在多个阶段运行。

`prerule` 和 `postrule` 不接受参数。`hostrule` 接受一个主机表，并可以测试例如目标的 IP 地址或主机名。`portrule` 接受一个主机表和一个端口表，针对任何处于 `open`、`open|filtered` 或 `unfiltered` 状态的端口。端口规则通常测试端口号、端口状态或监听服务名称等因素，以决定是否针对端口运行。示例规则参见[“规则”](nse-tutorial.html#nse-tutorial-rule)一节。

高级用户可以通过在 `--script` 参数中的脚本名称（或类别或其他表达式）前加上 `+` 来强制脚本运行，无论这些规则函数的结果如何。

当前选择 `prerule` 或 `postrule` 的标准是：如果脚本正在执行主机发现或任何其他网络操作，则应使用 `prerule`。`postrule` 保留用于报告扫描期间收集的数据和统计信息。

### Action

Action 是 NSE 脚本的核心。它包含当脚本的 prerule、portrule、hostrule 或 postrule 触发时所有要执行的指令。它是一个 Lua 函数，接受与规则相同的参数。action 的返回值可以是一个名称-值对表、一个字符串或 `nil`。关于 NSE action 的示例，请参见[“Action”](nse-tutorial.html#nse-tutorial-action)一节。

如果 action 的输出是一个表，它会自动以结构化方式格式化，以包含在正常（`-oN`）和 XML（`-oX`）输出格式中。如果是一个字符串，文本将直接显示在正常输出中，并在 XML 输出中写入为 XML 属性。如果脚本返回 `nil`，则不产生任何输出。有关如何处理不同返回值的详细信息，请参见[“结构化与非结构化输出”](nse-api.html#nse-structured-output)一节。

### 环境变量

每个脚本都有自己的环境变量集：

- **`SCRIPT_PATH`**：脚本路径。
- **`SCRIPT_NAME`**：脚本名称。此变量可用于调试输出。
- **`SCRIPT_TYPE`**：由于脚本可以有多个规则函数，此环境变量将显示哪个规则激活了脚本，如果脚本希望在不同脚本扫描阶段之间共享代码，这将非常有用。它将采用以下四个字符串值之一：`"prerule"`、`"hostrule"`、`"portrule"` 或 `"postrule"`。此变量仅在规则函数评估期间及之后可用。

以下是一个使用上述环境变量的调试代码示例，以及来自 `dns-zone-transfer` 的输出消息：

```lua
stdnse.print_debug(3, "Skipping '%s' %s, 'dnszonetransfer.server' argument is missing.", SCRIPT_NAME, SCRIPT_TYPE)
```

```screen
Initiating NSE at 15:31
NSE: Skipping 'dns-zone-transfer' prerule, 'dnszonetransfer.server' argument is missing.
```
