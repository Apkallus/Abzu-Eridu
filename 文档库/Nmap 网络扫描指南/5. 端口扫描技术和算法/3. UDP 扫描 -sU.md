## UDP 扫描（`-sU`）

虽然互联网上大多数流行的服务运行在 TCP 协议之上，但 [UDP](http://www.rfc-editor.org/rfc/rfc768.txt) 服务也被广泛部署。DNS、SNMP 和 DHCP（注册端口 53、161/162 和 67/68）是最常见的三种。由于 UDP 扫描通常比 TCP 更慢、更困难，一些安全审计人员会忽略这些端口。这是一个错误，因为可利用的 UDP 服务相当普遍，攻击者当然不会忽略整个协议。幸运的是，Nmap 可以帮助盘点 UDP 端口。

UDP 扫描通过 `-sU` 选项激活。它可以与 TCP 扫描类型（如 SYN 扫描 `-sS`）结合，在同一次运行中检查两种协议。

UDP 扫描的工作原理是向每个目标端口发送一个 UDP 数据包。对于大多数端口，这个数据包是空的（没有有效载荷），但对于几个更常见的端口，会发送协议特定的有效载荷。根据响应或缺乏响应，端口被分配到四种状态之一，如表 5.3 所示。

**表 5.3 Nmap 如何解释对 UDP 探测的响应**

| 探测响应 | 分配的状态 |
| :--- | :--- |
| 来自目标端口的任何 UDP 响应（不常见） | `open` |
| 未收到响应（即使重传后） | `open|filtered` |
| ICMP 端口不可达错误（类型 3，代码 3） | `closed` |
| 其他 ICMP 不可达错误（类型 3，代码 1、2、9、10 或 13） | `filtered` |

此表中最引人注目的元素可能是 `open|filtered` 状态。它是 UDP 扫描最大挑战的症状：开放端口很少对空探测做出响应。那些 Nmap 有协议特定有效载荷的端口更有可能得到响应并被标记为 `open`，但对于其余端口，目标 TCP/IP 协议栈只是将空数据包传递给监听应用程序，而该应用程序通常立即将其作为无效数据丢弃。如果所有其他状态的端口都会响应，那么开放端口就可以通过排除法推导出来。不幸的是，防火墙和过滤设备也以无响应地丢弃数据包而闻名。因此，当 Nmap 在几次尝试后没有收到响应时，它无法确定端口是 `open` 还是 `filtered`。在 Nmap 发布之初，过滤设备还很少见，Nmap 可以（也确实）简单地假设端口是 `open`。如今互联网受到更好的保护，因此 Nmap 在 2004 年（版本 3.70）更改了行为，将无响应的 UDP 端口报告为 `open|filtered`。我们可以在示例 5.4 中看到这一点，该示例显示了 Ereet 扫描一台名为 Felix 的 Linux 机器。

**示例 5.4 UDP 扫描示例**

```log
krad# nmap -sU -v felix

Starting Nmap ( https://nmap.org )
Nmap scan report for felix.nmap.org (192.168.0.42)
(The 997 ports scanned but not shown below are in state: closed)
PORT    STATE         SERVICE
53/udp  open|filtered domain
67/udp  open|filtered dhcpserver
111/udp open|filtered rpcbind
MAC Address: 00:02:E3:14:11:02 (Lite-on Communications)

Nmap done: 1 IP address (1 host up) scanned in 999.25 seconds
```

这次对 Felix 的扫描展示了 `open|filtered` 歧义问题以及另一个问题：UDP 扫描可能 **很慢**。由于 Felix 和大多数其他 Linux 系统执行的 ICMP 响应速率限制，扫描一千个端口在此例中花费了近 17 分钟。Nmap 提供了解决这两个问题的方法，如下两节所述。

### 区分开放 UDP 端口与被过滤 UDP 端口

在 Felix 扫描的例子中，除了三个 `open|filtered` 端口外，所有端口都是 `closed`。因此，扫描仍然成功将潜在开放端口缩小到了少数几个。情况并非总是如此。示例 5.5 显示了对严重过滤的站点 Scanme 的 UDP 扫描。

**示例 5.5 UDP 扫描示例**

```log
krad# nmap -sU -T4 scanme.nmap.org

Starting Nmap ( https://nmap.org )
All 1000 scanned ports on scanme.nmap.org (64.13.134.52) are open|filtered

Nmap done: 1 IP address (1 host up) scanned in 5.50 seconds
```

在这种情况下，扫描根本没有缩小开放端口的范围。全部 1000 个端口都是 `open|filtered`。需要一种新的策略。

表 5.3“Nmap 如何解释对 UDP 探测的响应”显示，当 Nmap 未能从其 UDP 探测中收到任何响应时，就会出现 `open|filtered` 状态。但它也表明，在极少数情况下，监听端口的 UDP 服务会以同样的方式响应，证明该端口是开放的。这些服务不常响应的原因是 Nmap 发送的空数据包被认为是无效的。不幸的是，UDP 服务通常定义自己的数据包结构，而不是遵循 Nmap 始终可以发送的某种通用格式。SNMP 数据包看起来与 SunRPC、DHCP 或 DNS 请求数据包完全不同。

为了为每个流行的 UDP 服务发送正确的数据包，Nmap 需要一个定义其探测格式的大型数据库。幸运的是，Nmap 以 `nmap-service-probes` 文件的形式拥有这样的数据库，它是第 7 章《服务和应用程序版本检测》中描述的服务和版本检测子系统的一部分。

当使用 `-sV`（或 `-A`）启用版本扫描时，它将向每个 `open|filtered` 端口（以及已知的 `open` 端口）发送 UDP 探测。如果任何探测从 `open|filtered` 端口引发响应，状态将更改为 `open`。将 `-sV` 添加到 Felix 扫描的结果如示例 5.6 所示。

**示例 5.6 使用版本检测改进 Felix 的 UDP 扫描结果**

```log
krad# nmap -sUV -F felix.nmap.org

Starting Nmap ( https://nmap.org )
Nmap scan report for felix.nmap.org (192.168.0.42)
Not shown: 997 closed ports
PORT    STATE         SERVICE    VERSION
53/udp  open          domain     ISC BIND 9.2.1
67/udp  open|filtered dhcpserver
111/udp open          rpcbind    2 (rpc #100000)
MAC Address: 00:02:E3:14:11:02 (Lite-on Communications)

Nmap done: 1 IP address (1 host up) scanned in 1037.57 seconds
```

这次新扫描显示端口 111 和 53 绝对是开放的。系统并不完美——端口 67 仍然是 `open|filtered`。在这个特定情况下，端口是开放的，但 Nmap 没有针对 DHCP 的有效版本探测。另一个困难的服务是 SNMP，它通常仅在给出正确的团体字符串时才响应。许多设备配置有团体字符串 `public`，但并非全部。虽然这些结果并不完美，但了解三个测试端口中的两个的真实状态仍然有帮助。

在成功消除 Felix 结果的歧义后，Ereet 将注意力转回 Scanme，上次 Scanme 将所有端口列为 `open|filtered`。他再次尝试使用版本检测，如示例 5.7 所示。

**示例 5.7 使用版本检测改进 Scanme 的 UDP 扫描结果**

```log
krad# nmap -sUV -T4 scanme.nmap.org

Starting Nmap ( https://nmap.org )
Nmap scan report for scanme.nmap.org (64.13.134.52)
```

> 虽然 Ereet 最终找到了开放端口，但他犯了一个错误，没有首先更新他的 Nmap 版本。Nmap 5.10BETA1 及更新版本有一个有效载荷系统，如果它们被选中进行端口扫描或主机发现，会向三十多个众所周知的 UDP 端口发送适当的服务协议请求。虽然它不如版本检测全面，但它可以快速识别出示例 5.5 中的开放端口 53。

这个结果花了一个小时，而之前的 Scanme 扫描只花了五秒钟，但这些结果实际上是有用的。Ereet 的笑容扩大，眼睛闪闪发光，因为他在想要入侵的机器上发现了一个开放的 ISC BIND 域名服务器。该软件有很长的安全漏洞历史，也许他能在这个较新的版本中找到漏洞。

### 加速 UDP 扫描

UDP 扫描的另一个大挑战是快速完成。开放和被过滤的端口很少发送任何响应，导致 Nmap 超时，然后进行重传，以防探测或响应丢失。关闭端口通常是一个更大的问题。它们通常会发回 ICMP 端口不可达错误。但与关闭的 TCP 端口响应 SYN 或连接扫描发送的 RST 数据包不同，许多主机默认对 ICMP 端口不可达消息进行速率限制。Linux 和 Solaris 在这方面特别严格。例如，Felix 上的 Linux 2.4.20 内核将目标不可达消息限制为每秒一次（在 `net/ipv4/icmp.c` 中）。这解释了为什么示例 5.4“UDP 扫描示例”中的扫描如此缓慢。

Nmap 会检测速率限制并相应减慢速度，以避免用目标机器将丢弃的无用数据包淹没网络。不幸的是，Linux 风格的限制（每秒一个数据包）使得 65,535 端口的扫描需要超过 18 小时。以下是一些提高 UDP 扫描性能的建议。另请阅读第 6 章《优化 Nmap 性能》以获取更详细的讨论和一般性建议。

- **增加主机并行度**  
  如果 Nmap 每秒只从单个目标主机收到一个端口不可达错误，那么通过同时扫描 100 个这样的主机，它每秒可以收到 100 个错误。通过向 `--min-hostgroup` 传递一个较大的值（例如 100）来实现。

- **先扫描流行端口**  
  很少使用 UDP 端口号。对最常见的 100 个 UDP 端口进行扫描（使用 `-F` 选项）将快速完成。然后您可以研究这些结果，同时在后台启动为期数天的 65K 端口网络扫描。

- **在版本检测扫描中添加 `--version-intensity 0`**  
  如前所述，通常需要版本检测（`-sV`）来区分开放的 UDP 端口和被过滤的 UDP 端口。版本检测相对较慢，因为它涉及向目标机器上发现的每个 `open` 或 `open|filtered` 端口发送大量特定于应用程序协议的探测。指定 `--version-intensity 0` 会指示 Nmap 仅尝试最有可能对给定端口号有效的探测。它通过使用 `nmap-service-probes` 文件中的数据来实现。此选项对性能的影响是显著的，本节后面将演示。

- **从防火墙后面进行扫描**  
  与 TCP 一样，数据包过滤器会显著降低扫描速度。许多现代防火墙可以轻松设置数据包速率限制。如果您可以通过从防火墙后面（而不是穿越防火墙）发起扫描来绕过该问题，请这样做。

- **使用 `--host-timeout` 跳过慢速主机**  
  受 ICMP 速率限制的主机可能比那些对每个探测都快速响应目标不可达数据包的主机花费多个数量级的时间。指定最大扫描时间（例如 `15m` 表示 15 分钟）会使 Nmap 放弃在该时间内未完成扫描的单个主机。这允许您快速扫描所有有响应的主机。然后您可以在后台处理慢速主机。

- **使用 `-v` 并放松心情**  
  启用详细输出（`-v`）后，Nmap 会提供每个主机的预计扫描完成时间。无需密切关注。在 Nmap 不知疲倦地为您扫描时，睡一觉，去您最喜欢的酒吧，读一本书，完成其他工作，或以其他方式自娱自乐。

完美展示优化 UDP 扫描需求的是示例 5.7“使用版本检测改进 Scanme 的 UDP 扫描结果”。该扫描获得了所需数据，但扫描这一台主机就花了一个多小时！在示例 5.9 中，Ereet 再次运行该扫描。这次他添加了 `-F --version-intensity 0` 选项，长达一小时的扫描减少到了 13 秒！然而，相同的关键信息（在端口 53 上运行的 ISC Bind 守护进程）被检测到了。
