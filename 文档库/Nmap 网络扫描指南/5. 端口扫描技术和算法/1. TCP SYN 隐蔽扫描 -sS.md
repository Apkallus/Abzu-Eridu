## TCP SYN（隐蔽）扫描（`-sS`）

SYN 扫描成为默认且最流行的扫描选项是有充分理由的。在不受干扰防火墙阻碍的快速网络上，它可以快速执行，每秒扫描数千个端口。SYN 扫描相对低调且隐蔽，因为它从不完成 TCP 连接。它也能在任何合规的 TCP 协议栈上工作，而不像 Nmap 的 FIN/NULL/Xmas、Maimon 和空闲扫描那样依赖于特定平台的特性。它还能清晰、可靠地区分 `open`、`closed` 和 `filtered` 状态。

可以通过向 Nmap 传递 `-sS` 选项来请求 SYN 扫描。它需要原始数据包权限，并且当该权限可用时是默认的 TCP 扫描。因此，当以 root 或 Administrator 身份运行 Nmap 时，通常会省略 `-sS`。这种默认的 SYN 扫描行为如示例 5.1 所示，它找到了三个主要状态中各一个端口。

**示例 5.1 一个显示三种端口状态的 SYN 扫描**

```screen
krad# nmap -p22,113,139 scanme.nmap.org

Starting Nmap ( https://nmap.org )
Nmap scan report for scanme.nmap.org (64.13.134.52)
PORT    STATE    SERVICE
22/tcp  open     ssh
113/tcp closed   auth
139/tcp filtered netbios-ssn

Nmap done: 1 IP address (1 host up) scanned in 1.35 seconds
```

虽然无需底层 [TCP](http://www.rfc-editor.org/rfc/rfc793.txt) 知识也能轻松使用 SYN 扫描，但理解该技术有助于解释异常结果。对我们来说幸运的是，令人生畏的黑帽破解者 Ereet Hagiwara 从 [恐吓日本 Windows 用户](http://www.microsoft.com/japan/security/bulletins/MS04-003e.mspx) 中抽出时间，在数据包层面为我们说明了示例 5.1 中的 SYN 扫描。首先，开放端口 22 的行为如图 5.2 所示。

**图 5.2 对开放端口 22 的 SYN 扫描**

![SYN scan of open port 22](images/ereet/Ereet_Packet_Trace_Syn_Open.png)

如本例所示，Nmap 首先向端口 22 发送一个设置了 SYN 标志的 TCP 数据包（如果你忘记了数据包报头的样子，请参见图 2“TCP 报头”）。这是任何合法连接尝试都会进行的 TCP 三次握手的第一步。由于目标端口是开放的，Scanme 通过返回一个设置了 SYN 和 ACK 标志的响应来执行第二步。在正常连接中，Ereet 的机器（名为 krad）会通过发送一个 ACK 数据包确认 SYN/ACK 来完成三次握手。Nmap 不需要这样做，因为 SYN/ACK 响应已经告诉它端口是开放的。如果 Nmap 完成了连接，它还需要担心关闭连接。这通常涉及另一次握手，使用 FIN 数据包而不是 SYN。因此发送 ACK 是个坏主意，但总得做点什么。如果完全忽略 SYN/ACK，Scanme 会认为它丢失了并不断重发。由于我们不想建立完整连接，正确的响应是如示意图所示的 RST 数据包。这告诉 Scanme 忘记（重置）尝试的连接。Nmap 可以轻松发送这个 RST 数据包，但实际上它不需要。运行在 krad 上的操作系统也会收到 SYN/ACK，因为 Nmap 自己构造了 SYN 探测，所以操作系统并不期待这个响应。因此操作系统会用一个 RST 数据包响应这个意外的 SYN/ACK。本章描述的所有 RST 数据包也设置了 ACK 位，因为它们总是作为对收到的数据包的响应（并确认）而发送。因此 RST 数据包没有显式显示该位。由于三次握手从未完成，SYN 扫描有时被称为半开扫描。

图 5.3 显示了 Nmap 如何确定端口 113 是关闭的。这比开放的情况更简单。第一步总是相同的——Nmap 向 Scanme 发送 SYN 探测。但收到的不是 SYN/ACK，而是一个 RST。这样就确定了——端口是关闭的。不需要关于该端口的更多通信。

**图 5.3 对关闭端口 113 的 SYN 扫描**

![SYN scan of closed port 113](images/ereet/Ereet_Packet_Trace_Syn_Closed.png)

最后，Ereet 向我们展示了在 Nmap 看来过滤端口是什么样的，如图 5.4 所示。和往常一样，首先发送初始 SYN，但 Nmap 没有收到回复。响应可能只是慢。根据之前的响应（或时序默认值），Nmap 知道要等待多长时间，最终放弃接收。无响应的端口通常是被过滤的（被防火墙设备阻止，或者主机已关闭），但这一项测试并不确定。也许端口是开放的，但探测或响应只是丢失了。网络可能不稳定。因此 Nmap 通过重新发送 SYN 探测再次尝试。在又一个超时周期之后，Nmap 放弃并将端口标记为 `filtered`。在这种情况下，只尝试了一次重传。如“扫描代码与算法”一节所述，Nmap 会仔细记录数据包丢失统计信息，并在扫描不太可靠的网络时尝试更多次重传。

**图 5.4 对过滤端口 139 的 SYN 扫描**

![SYN scan of filtered port 139](images/ereet/Ereet_Packet_Trace_Syn_Filtered.png)

如果 Nmap 收到某些 ICMP 错误消息返回，它也会认为端口是 `filtered`。表 5.2 显示了 Nmap 如何根据对 SYN 探测的响应来分配端口状态。

**表 5.2 Nmap 如何解释对 SYN 探测的响应**

| 探测响应 | 分配的状态 |
| :--- | :--- |
| TCP SYN/ACK 响应 | `open` |
| TCP RST 响应 | `closed` |
| 未收到响应（即使重传后） | `filtered` |
| ICMP 不可达错误（类型 3，代码 1、2、3、9、10 或 13） | `filtered` |

虽然本节中的精美插图在你有它们时很有用，但当你除了其他所需命令行标志之外还指定 `--packet-trace` 选项时，Nmap 会在数据包级别准确报告它正在做什么。这是新手在 Ereet 不在身边帮助时理解 Nmap 行为的好方法。即使是高级用户在 Nmap 产生他们意想不到的结果时也会发现它很方便。你可能还想使用 `-d`（甚至 `-d5`）提高调试级别。然后扫描为你的目的所需的最少数量的端口和主机，否则你最终可能会得到数百万行输出。示例 5.2 使用启用了数据包跟踪的方式重复了 Ereet 的三端口 SYN 扫描（输出经过编辑以保持简洁）。阅读命令行，然后在继续阅读之前测试自己，看看将发送哪些数据包。然后，当你读到“SYN Stealth Scan took 1.25s”时，你应该从 RCVD 行知道端口状态表在继续阅读之前会是什么样子。

SYN 扫描长期以来一直被称为隐蔽扫描，因为它比 TCP 连接扫描（下一个讨论的）更隐蔽，后者是 Nmap 发布前最常见的扫描类型。尽管有这个名字，但不要指望默认的 SYN 扫描能在敏感网络中不被检测地滑过。广泛部署的入侵检测系统甚至个人防火墙都非常有能力检测默认的 SYN 扫描。有关隐蔽扫描的更有效技术，请参见第 10 章《检测和破坏防火墙与入侵检测系统》。
