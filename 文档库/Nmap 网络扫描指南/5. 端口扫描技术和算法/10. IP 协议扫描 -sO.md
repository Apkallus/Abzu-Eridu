## IP 协议扫描（`-sO`）

IP 协议扫描允许您确定目标机器支持哪些 IP 协议（TCP、ICMP、IGMP 等）。从技术上讲，这不是端口扫描，因为它循环遍历 IP 协议号而不是 TCP 或 UDP 端口号。但它仍然使用 `-p` 选项来选择扫描的协议号，以正常的端口表格式报告结果，甚至使用与真正端口扫描方法相同的底层扫描引擎。因此它与端口扫描足够接近，属于这里。

除了本身有用之外，协议扫描还展示了开源软件的力量。虽然基本思想很简单，但我之前没有想到添加它，也没有收到任何此类功能请求。然后在 2000 年夏天，Gerhard Rieger 构思了这个想法，编写了一个实现它的优秀补丁，并将其发送到 `nmap-hackers` 邮件列表。我在第二天将该补丁合并到 Nmap 树中并发布了一个新版本。很少有商业软件的用户会热情到设计并贡献自己的改进！

协议扫描的工作方式与 UDP 扫描类似。它不是遍历 UDP 数据包的端口号字段，而是发送 IP 数据包头并遍历 8 位的 IP 协议字段。这些头通常是空的，不包含数据，甚至不包含声称的协议的正确头。某些流行协议（包括 TCP、UDP 和 ICMP）例外。这些协议的适当协议头被包含在内，因为某些系统否则不会发送它们，并且因为 Nmap 已经有创建它们的函数。协议扫描不是关注 ICMP 端口不可达消息，而是寻找 ICMP **协议** 不可达消息。表 5.8 显示了对 IP 探测的响应如何映射到端口状态。

**表 5.8 Nmap 如何解释对 IP 协议探测的响应**

| 探测响应 | 分配的状态 |
| :--- | :--- |
| 来自目标主机的任何协议的任何响应 | `open`（对于响应使用的协议，不一定是探测协议） |
| ICMP 协议不可达错误（类型 3，代码 2） | `closed` |
| 其他 ICMP 不可达错误（类型 3，代码 1、3、9、10 或 13） | `filtered`（但如果从目标机器发送，则证明 ICMP 是开放的） |
| 未收到响应（即使重传后） | `open|filtered` |

与 TCP 或 UDP 协议中的开放端口一样，每个开放协议都是一个潜在的攻击向量。此外，协议扫描结果有助于确定机器的用途以及存在什么样的数据包过滤。终端主机通常只开放 TCP、UDP、ICMP 和（有时）IGMP，而路由器通常提供更多，包括与路由相关的协议如 GRE 和 EGP。防火墙和 VPN 网关可能显示与加密相关的协议如 IPsec 和 SWIPE。

与 UDP 扫描期间收到的 ICMP 端口不可达消息一样，ICMP 协议不可达消息通常也受到速率限制。例如，默认的 Linux 2.4.20 系统每秒发送不超过一个 ICMP 目标不可达响应。由于只有 256 个可能的协议号，这比 65,536 端口的 UDP 扫描问题小。[“加速 UDP 扫描”一节](scan-methods-udp-scan.html#scan-methods-udp-optimizing) 中的建议也适用于加速 IP 协议扫描。

协议扫描的使用方式与命令行中大多数其他扫描技术相同。只需在您喜欢的任何通用 Nmap 选项之外指定 `-sO`。正常的端口（`-p`）选项用于选择协议号。或者您可以使用 `-F` 来扫描 `nmap-protocols` 数据库中列出的所有协议。默认情况下，Nmap 扫描所有 256 个可能的值。示例 5.20 显示了 Ereet 扫描波兰的一台路由器，然后扫描我本地网络上的一台典型 Linux 2.4 机器。
