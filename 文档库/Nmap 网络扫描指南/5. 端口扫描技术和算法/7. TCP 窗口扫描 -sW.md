## TCP 窗口扫描（`-sW`）

窗口扫描与 ACK 扫描完全相同，区别在于它利用某些系统实现细节来区分开放端口和关闭端口，而不是在返回 RST 时总是打印 `unfiltered`。它通过检查返回的 RST 数据包的 TCP 窗口值来实现这一点。在某些系统上，开放端口使用正窗口大小（即使对于 RST 数据包），而关闭端口的窗口为零。窗口扫描发送与 ACK 扫描相同的裸 ACK 探测，并按照表 5.6 解释结果。

**表 5.6 Nmap 如何解释对窗口扫描 ACK 探测的响应**

| 探测响应 | 分配的状态 |
| :--- | :--- |
| 带有非零窗口字段的 TCP RST 响应 | `open` |
| 带有零窗口字段的 TCP RST 响应 | `closed` |
| 未收到响应（即使重传后） | `filtered` |
| ICMP 不可达错误（类型 3，代码 1、2、3、9、10 或 13） | `filtered` |

这种扫描依赖于互联网上少数系统的实现细节，因此您不能总是信任它。不支持它的系统通常会返回所有端口为 `closed`。当然，也可能该机器确实没有开放端口。如果大多数扫描端口是 `closed`，但少数常见端口号（例如 22、25 和 53）是 `open`，则该系统很可能容易受到攻击。偶尔，系统甚至会表现出完全相反的行为。如果您的扫描显示 997 个开放端口和三个关闭或过滤端口，那么这三个可能才是真正开放的端口。

虽然这种扫描并非适用于所有情况，但有时非常有用。回想一下 [示例 5.12“对 Docsrv 的 FIN 扫描”](scan-methods-null-fin-xmas-scan.html#scan-methods-ex-sco-fin-scan)，该扫描显示了许多在基本 SYN 扫描中未发现的 `open|filtered` 端口。问题在于我们无法通过 FIN 扫描区分开放端口和过滤端口。上一节表明，我们可以通过结合 FIN 和 ACK 扫描结果来区分它们。在本例中，窗口扫描使其更加容易，因为不需要 FIN 扫描结果，如示例 5.17 所示。
