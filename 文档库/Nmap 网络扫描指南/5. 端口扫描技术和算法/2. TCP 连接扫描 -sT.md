## TCP 连接扫描（`-sT`）

TCP 连接扫描是当 SYN 扫描不可用时的默认 TCP 扫描类型。这发生在用户没有原始数据包权限或正在扫描 IPv6 网络时。与大多数其他扫描类型写入原始数据包不同，Nmap 通过调用 `connect` 系统调用来请求底层操作系统与目标机器和端口建立连接。这是 Web 浏览器、P2P 客户端以及大多数其他支持网络的应用程序用来建立连接的同一个高级系统调用。它是称为 Berkeley Sockets API 的编程接口的一部分。Nmap 不是从线路上读取原始数据包响应，而是使用此 API 获取每个连接尝试的状态信息。这是除 FTP 反弹扫描（`-b`）之外，非特权用户唯一可用的扫描类型。

当 SYN 扫描可用时，它通常是更好的选择。Nmap 对高级 `connect` 调用的控制比原始数据包少，因此效率较低。系统调用会完成到开放目标端口的连接，而不是像 SYN 扫描那样执行半开重置。这不仅需要更长的时间和更多的数据包来获取相同的信息，而且目标机器更有可能记录连接。一个像样的 IDS 可以检测到任何一种，但大多数机器没有这样的警报系统。在普通的 Unix 系统上，当 Nmap 连接然后在不发送数据的情况下关闭连接时，许多服务会在 syslog 中添加一条记录，有时还会添加一条神秘的错误消息。当这种情况发生时，真正差劲的服务会崩溃，但这并不常见。管理员如果在其日志中看到来自单个系统的大量连接尝试，应该知道她已被连接扫描。

前两步（SYN 和 SYN/ACK）与 SYN 扫描完全相同。然后，krad 没有用 RST 数据包中止半开连接，而是用自己的 ACK 数据包确认 SYN/ACK，完成连接。在这种情况下，Scanme 甚至有时间通过现在开放的连接发送其 SSH 标识字符串（`SSH-1.99-OpenSSH_3.1p1\n`）。一旦 Nmap 从其主机操作系统得知连接成功，它就会终止连接。TCP 连接通常以另一个涉及 FIN 标志的握手结束，但 Nmap 要求主机操作系统立即用 RST 数据包终止连接。

虽然这个连接扫描示例使用的数据包几乎是 SYN 扫描的两倍，但带宽差异很少如此显著。大型扫描中的绝大多数端口将是 `closed` 或 `filtered`。这些的数据包跟踪与图 5.3“对关闭端口 113 的 SYN 扫描”和图 5.4“对过滤端口 139 的 SYN 扫描”中描述的相同。只有开放端口会产生更多的网络流量。

连接扫描的输出与 SYN 扫描没有显著差异。示例 5.3 显示了对 Scanme 的连接扫描。`-sT` 选项可以省略，因为 Nmap 是从非特权账户运行的，因此连接扫描是默认类型。
