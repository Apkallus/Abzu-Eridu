## 引言

作为新手修车，我可能会花几个小时试图用简陋的工具（锤子、胶带、扳手等）来完成手头的任务。当我惨遭失败，把破车拖到真正的修理工那里时，他总会在巨大的工具箱里翻找，直到拿出一个完美的工具，让工作看起来毫不费力。端口扫描的艺术与此类似。专家了解几十种扫描技术，并为给定任务选择合适的一种（或组合）。另一方面，经验不足的用户和脚本小子则试图用默认的 SYN 扫描解决所有问题。由于 Nmap 是免费的，掌握端口扫描的唯一障碍就是知识。这当然比汽车行业强，在汽车行业，你可能需要高超的技能才能判断你需要一个减震弹簧压缩器，然后你还得为此支付数千美元。

前一章从总体角度描述了使用 Nmap 进行端口扫描，包括在“选择扫描技术”一节中对 Nmap 支持的扫描类型的简要总结。本章深入描述了每一种扫描类型。针对每种扫描类型，给出了典型的使用场景和说明，以及说明它们如何工作的在线数据包跟踪。然后讨论了 `ultra_scan` 算法（大多数扫描方法使用该算法），重点是可调整以提高性能的方面。

大多数扫描类型仅适用于特权用户。这是因为它们发送和接收原始 IP 数据包（甚至以太网帧），这需要在 Unix 系统上具有 root 访问权限。在 Windows 上使用管理员账户不是必需的，因为当 Npcap 已经加载到操作系统时，Nmap 在该平台上对非特权用户也能正常工作。在 1997 年 Nmap 发布时，要求 root 权限是一个严重的限制，因为许多用户只能访问共享 shell 账户。现在，世界不同了。电脑更便宜，更多的人拥有永远在线的直接互联网接入，桌面 Unix 系统（包括 Linux 和 Mac OS X）很普遍。现在有了 Windows 版本的 Nmap，使其可以在更多桌面上运行。出于所有这些原因，用户很少需要从受限的共享 shell 账户运行 Nmap。这是幸运的，因为特权选项使 Nmap 更强大、更灵活。

在讨论 Nmap 如何处理探测响应时，许多章节通过类型和代码号讨论 ICMP 错误消息。类型和代码是 ICMP 报头中描述消息目的的两个 8 位字段。Nmap 端口扫描技术只关心 ICMP 类型 3，即目标不可达消息。图 5.1 显示了此类数据包的 ICMP 报头布局（它封装在 IP 数据包的数据部分，如图 1“IPv4 报头”所示）。

有十六种代码代表不同的目标不可达消息。它们都显示在表 5.1 中，但 Nmap 只关心代码 0–3、9、10 和 13，这些代码用星号标记。

**表 5.1 ICMP 目标不可达（类型 3）代码值**

| 代码 | 描述 |
| :--- | :--- |
| 0* | 网络不可达 |
| 1* | 主机不可达 |
| 2* | 协议不可达 |
| 3* | 端口不可达 |
| 4 | 需要分片但设置了不分片标志 |
| 5 | 源路由失败 |
| 6 | 目标网络未知 |
| 7 | 目标主机未知 |
| 8 | 源主机被隔离（已废弃） |
| 9* | 目标网络被管理禁止 |
| 10* | 目标主机被管理禁止 |
| 11 | 服务类型不可达的网络 |
| 12 | 服务类型不可达的主机 |
| 13* | 通信被过滤管理禁止 |
| 14 | 主机优先级违规 |
| 15 | 优先级截断生效 |
