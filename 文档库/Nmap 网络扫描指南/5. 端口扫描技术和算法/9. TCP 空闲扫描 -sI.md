## TCP 空闲扫描（`-sI`）

> **注意**  
> 志愿者已将本节翻译成 [西班牙语](https://nmap.org/idlescan-es.html) 和 [葡萄牙语（巴西）](https://nmap.org/idlescan-pt-br.html)

1998 年，安全研究员 Antirez（也是本书中使用的 hping2 工具的作者）在 Bugtraq 邮件列表中发布了一种巧妙的新端口扫描技术。空闲扫描（Idle scan）正如其名，允许完全盲目的端口扫描。攻击者实际上可以扫描目标，而无需从其自身的 IP 地址向目标发送一个数据包！取而代之的是，一种巧妙的侧信道攻击允许扫描通过一个“僵尸主机”反弹。入侵检测系统（IDS）报告会将无辜的僵尸主机指认为攻击者。除了极其隐蔽之外，这种扫描类型还可以发现机器之间基于 IP 的信任关系。

虽然空闲扫描比目前讨论的任何技术都更复杂，但您不需要成为 TCP/IP 专家就能理解它。它可以通过以下基本事实组合起来：

- 确定 TCP 端口是否开放的一种方法是向该端口发送 SYN（会话建立）数据包。如果端口开放，目标机器将响应 SYN/ACK（会话请求确认）数据包；如果端口关闭，则响应 RST（重置）。这是前面讨论的 SYN 扫描的基础。
- 收到未经请求的 SYN/ACK 数据包的机器将用 RST 响应。未经请求的 RST 将被忽略。
- 互联网上的每个 IP 数据包都有一个分片标识号（IP ID）。由于许多操作系统只是为它们发送的每个数据包递增此编号，因此探测 IPID 可以告诉攻击者自上次探测以来发送了多少数据包。

通过结合这些特性，可以在伪造身份的同时扫描目标网络，使得看起来像是一个无辜的僵尸机器进行了扫描。

### 空闲扫描分步说明

从根本上说，空闲扫描由三个步骤组成，每个端口重复执行：

1. 探测僵尸的 IP ID 并记录下来。
2. 伪造一个来自僵尸的 SYN 数据包，并将其发送到目标的所需端口。根据端口状态，目标的反应可能导致僵尸的 IP ID 增加，也可能不增加。
3. 再次探测僵尸的 IP ID。然后通过将此新 IP ID 与步骤 1 中记录的 IP ID 进行比较来确定目标端口状态。

完成此过程后，僵尸的 IP ID 应该增加了 1 或 2。增加 1 表示僵尸没有发送任何数据包（除了对攻击者探测的回复）。缺乏发送数据包意味着端口不是开放的（目标必须向僵尸发送了一个 RST 数据包（被忽略）或什么也没发送）。增加 2 表示僵尸在两次探测之间发送了一个数据包。这个额外的数据包通常意味着端口是开放的（目标很可能向僵尸发送了一个 SYN/ACK 数据包以响应伪造的 SYN，从而引发僵尸发送 RST 数据包）。大于 2 的增加通常表示僵尸主机有问题。它可能没有可预测的 IP ID 编号，或者可能参与了与空闲扫描无关的通信。

尽管关闭端口发生的情况与过滤端口略有不同，但攻击者在两种情况下测量到的结果是相同的，即 IP ID 增加 1。因此，空闲扫描无法区分关闭端口和过滤端口。当 Nmap 记录 IP ID 增加 1 时，它将端口标记为 `closed|filtered`。

空闲扫描是终极隐蔽扫描。Nmap 提供诱饵扫描（`-D`）来帮助用户隐藏身份，但这（与空闲扫描不同）仍然需要攻击者从其真实 IP 地址向目标发送一些数据包才能获得扫描结果。空闲扫描的一个后果是，入侵检测系统通常会发出警报，声称僵尸机器对它们发起了扫描。因此，它可以用来诬陷其他方进行扫描。在阅读 IDS 警报时请记住这种可能性。

空闲扫描的一个独特优势是它可以用来绕过某些数据包过滤防火墙和路由器。IP 源地址过滤是一种常见（尽管较弱）的安全机制，用于限制可以连接到敏感主机或网络的机器。例如，公司数据库服务器可能只允许访问它的公共 Web 服务器连接。或者家庭用户可能只允许从他工作的机器进行 SSH（交互式登录）连接。

更令人不安的情况发生在某些公司高管要求网络管理员打开防火墙漏洞，以便他从家庭 IP 地址访问内部网络资源时。当高管不愿意或无法使用安全的 VPN 替代方案时，就会发生这种情况。

空闲扫描有时可用于绘制这些信任关系。关键因素是空闲扫描结果列出了从僵尸主机视角看到的开放端口。对前面提到的数据库服务器进行正常扫描可能显示没有开放端口，但使用 Web 服务器的 IP 作为僵尸进行空闲扫描可能会通过显示数据库相关服务端口为开放来暴露信任关系。

绘制这些信任关系对攻击者确定目标优先级非常有用。上面讨论的 Web 服务器对攻击者来说可能看起来很普通，直到她注意到它特殊的数据访问权限。

空闲扫描的一个缺点是它比其他大多数扫描类型花费的时间长得多。尽管采用了优化的算法（见“空闲扫描实现算法”一节），一个 15 秒的 SYN 扫描作为空闲扫描可能需要 15 分钟或更长时间。另一个问题是，您必须能够伪造数据包，使它们看起来像是来自僵尸，并且让它们到达目标机器。许多 ISP（尤其是拨号和住宅宽带提供商）现在实施出口过滤以防止这种数据包欺骗。高端提供商（如托管和 T1 服务）则不太可能这样做。如果此过滤生效，Nmap 将为每个您尝试的僵尸打印一条快速错误消息。如果更换 ISP 不可行，您可以尝试使用同一 ISP 网络上的另一个 IP。有时过滤只阻止欺骗位于客户使用范围 **之外** 的 IP 地址。空闲扫描的另一个挑战是您必须找到一个有效的僵尸主机，如下一节所述。

### 查找有效的空闲扫描僵尸主机

执行 IP ID 空闲扫描的第一步是找到合适的僵尸。它需要以全局（而不是基于它通信的每个主机）的方式递增分配 IP ID 数据包。它应该是空闲的（因此得名扫描），因为无关的流量会增加其 IP ID 序列，混淆扫描逻辑。攻击者与僵尸之间、僵尸与目标之间的延迟越低，扫描进行得越快。

当尝试空闲扫描时，Nmap 会测试提议的僵尸并报告任何问题。如果一个不起作用，尝试另一个。互联网上有足够多的易受攻击的主机，僵尸候选并不难找到。由于主机需要空闲，选择知名主机如 www.yahoo.com 或 google.com 几乎永远不会奏效。

一种常见的方法是简单地对某个网络执行 Nmap ping 扫描。您可以使用 Nmap 的随机 IP 选择模式（`-iR`），但这可能导致距离较远、延迟较大的僵尸。选择靠近源地址或靠近目标的网络会产生更好的结果。您可以尝试使用 ping 扫描结果中的每个可用主机进行空闲扫描，直到找到一个有效的。像往常一样，最好在将别人的机器用于意外目的（如空闲扫描）之前征得许可。

我们不仅仅是为了有趣而在插图中选择打印机图标来表示僵尸——简单的网络设备通常是很棒的僵尸，因为它们通常既未充分利用（空闲）又使用易受 IP ID 流量检测的简单网络协议栈。

对僵尸候选网络执行端口扫描和操作系统识别（`-O`）而不仅仅是 ping 扫描，有助于选择好的僵尸。只要启用了详细模式（`-v`），操作系统检测通常会确定 IP ID 序列生成方法并打印一行如“IP ID Sequence Generation: Incremental”。如果类型为 `Incremental` 或 `Broken little-endian incremental`，则该机器是很好的僵尸候选。这仍然不能保证它有效，因为 Solaris 和其他一些系统为它们通信的每个主机创建一个新的 IP ID 序列。主机也可能太忙。操作系统检测和开放端口列表也有助于识别可能空闲的系统。

识别僵尸候选的另一种方法是针对主机运行 [ipidseq](https://nmap.org/nsedoc/scripts/ipidseq.html) NSE 脚本。此脚本探测主机以分类其 IP ID 生成方法，然后像操作系统检测一样打印 IP ID 分类。与大多数 NSE 脚本一样，`ipidseq.nse` 可以并行对多个主机运行，使其成为扫描整个网络以寻找合适主机的另一个好选择。

虽然识别合适的僵尸需要一些初始工作，但您可以重复使用好的僵尸。

### 执行空闲扫描

一旦找到合适的僵尸，执行扫描就很容易了。只需将僵尸主机名指定给 `-sI` 选项，Nmap 就会完成剩下的工作。示例 5.19 显示了 Ereet 通过一台名为 Kiosk 的 Adobe 机器反弹空闲扫描来扫描美国唱片工业协会（RIAA）的例子。

从上面的扫描中，我们了解到 RIAA 的安全意识不强（注意开放的 PC Anywhere、portmapper 和 Legato nsrexec 端口）。由于他们显然没有防火墙，他们也不太可能有 IDS。但如果有，它会显示 kiosk.adobe.com 是扫描罪魁祸首。`-Pn` 选项阻止 Nmap 向 RIAA 机器发送初始 ping 数据包。否则会暴露 Ereet 的真实地址。扫描花费了很长时间，因为指定了 `-p-` 来扫描所有 65K 端口。不要试图使用 kiosk 进行扫描，因为它已经被移除。

默认情况下，Nmap 从僵尸的源端口 80 伪造探测到目标。您可以通过在僵尸名称后附加冒号和端口号来选择不同的端口（例如 `-sI kiosk.adobe.com:113`）。所选端口不得被攻击者或目标过滤。对僵尸的 SYN 扫描应显示该端口处于 `open` 或 `closed` 状态。

### 空闲扫描实现算法

虽然“空闲扫描分步说明”一节从基础层面描述了空闲扫描，但 Nmap 的实现要复杂得多。关键区别在于并行性以实现快速执行，以及冗余以减少误报。

由于推断端口状态的间接方法，并行化空闲扫描比其他扫描技术更棘手。如果 Nmap 向目标上的许多端口发送探测，然后检查僵尸的新 IP ID 值，IP ID 增加的数量将暴露有多少目标端口是开放的，但不会暴露哪些端口。这实际上不是一个主要问题，因为大型扫描中的绝大多数端口将是 `closed|filtered`。由于只有开放端口会导致 IP ID 值增加，Nmap 将看不到中间的增加，并可以将整个端口组标记为 `closed|filtered`。Nmap 可以并行扫描多达 100 个端口的组。如果 Nmap 探测一组然后发现僵尸 IP ID 增加了 `<N>` 次，那么该组中必须有 `<N>` 个开放端口。然后 Nmap 通过二分搜索找到开放端口。它将组分成两个，并分别向每个子组发送探测。如果一个子组显示零个开放端口，则该子组的所有端口都被标记为 `closed|filtered`。如果一个子组显示一个或多个开放端口，则再次分割，过程继续直到这些端口被识别。虽然这种技术增加了复杂性，但相比一次扫描一个端口，它可以将扫描时间减少一个数量级。

可靠性是空闲扫描的另一个主要问题。如果僵尸主机在扫描期间向任何无关的机器发送数据包，其 IP ID 会增加。这导致 Nmap 认为它找到了一个开放端口。幸运的是，并行扫描在这里也有帮助。如果 Nmap 扫描一组 100 个端口，IP ID 增加表明有两个开放端口，Nmap 将该组分成两个 50 端口的子组。当 Nmap 对两个子组进行 IP ID 扫描时，总僵尸 IP ID 增加必须再次为 2！否则，Nmap 将检测到不一致并重新扫描这些组。它还会根据检测到的僵尸的可靠性率修改组大小和扫描时序。如果 Nmap 检测到太多不一致的结果，它将退出并要求用户提供更好的僵尸。

有时，数据包跟踪是理解此类复杂算法和技术的最佳方式。再一次，Nmap 的 `--packet-trace` 使得在需要时生成这些变得微不足道。本节的其余部分提供了一个实际七端口空闲扫描的带注释数据包跟踪。IP 地址已更改为 `Attacker`、`Zombie` 和 `Target`，并且为了清晰起见，删除了跟踪行的一些无关方面（如 TCP 窗口大小）。

```screen
Attacker# nmap -sI Zombie -Pn -p20-25,110 -r --packet-trace -v Target
Starting Nmap ( https://nmap.org )
```

`-Pn` 对于隐蔽是必要的，否则 ping 数据包将从攻击者的真实地址发送到目标。版本扫描也会暴露真实地址，因此 **不** 指定 `-sV`。`-r` 选项（关闭端口随机化）仅用于使本示例更容易理解。

Nmap 首先通过向僵尸发送六个 SYN/ACK 数据包并分析响应来测试僵尸的 IP ID 序列生成。这有助于 Nmap 立即剔除不良僵尸。这也是必要的，因为某些系统（通常是 Microsoft Windows 机器，尽管并非所有 Windows 机器都这样做）为每个发送的数据包将 IP ID 增加 256 而不是 1。这发生在小端机器上，当它们不将 IP ID 转换为网络字节序（大端）时。Nmap 使用这些初始探测来检测并解决此问题。

该测试表明僵尸工作正常。每个 IP ID 都比前一个增加 1。因此系统似乎是空闲的，并且易受 IP ID 流量检测。这些有希望的结果仍需接受下一步测试，在该测试中，Nmap 伪造四个数据包到僵尸，就好像它们来自目标一样。然后它探测僵尸以确保 IP ID 增加了。如果没有增加，则很可能是攻击者的 ISP 阻止了伪造的数据包，或者僵尸为其通信的每个主机使用单独的 IP ID 序列计数器。两者都很常见，因此 Nmap 总是执行此测试。如上所示，最后一个已知的僵尸 IP ID 是 15674。


四个伪造的数据包加上来自攻击者的探测导致僵尸将其 IP ID 从 15674 增加到 15679。完美！现在真正的扫描开始了。请记住 15679 是最新的僵尸 IP ID。

Nmap 探测端口 20-23。然后它探测僵尸，发现新的 IP ID 是 15680，仅比之前的值 15679 高 1。在这两个已知数据包之间没有 IP ID 增加，这意味着端口 20-23 可能是 `closed|filtered`。也有可能来自目标端口的 SYN/ACK 尚未到达。在这种情况下，僵尸尚未用 RST 响应，因此其 IP ID 尚未增加。为确保准确性，Nmap 稍后将再次尝试这些端口。

```screen
RCVD (1.9990s) TCP Zombie:80 > Attacker:51986 R **id=15681**
```

Nmap 再次探测，因为自上次发送探测以来已经过去了 0.4 秒。僵尸（如果不是真正空闲）在此期间可能与其他主机通信，如果在此处未检测到，稍后会导致不准确。幸运的是，这没有发生：下一个 IP ID 如预期为 15681。

Nmap 探测端口 24、25 和 110，然后查询僵尸 IP ID。它从 15681 跳到了 15684。它跳过了 15682 和 15683，这意味着这三个端口中有两个可能是开放的。Nmap 无法判断哪两个是开放的，也可能是误报。因此 Nmap 进一步深入，将扫描分成子组。

```screen
RCVD (3.1480s) TCP Zombie:80 > Attacker:51979 R **id=15687**
```

第一个子组是端口 24 和 25。IP ID 从 15685 跳到 15687，意味着这两个端口中的一个很可能是开放的。Nmap 再次尝试分而治之的方法，分别探测每个端口。

```screen
SENT (3.3910s) TCP Attacker:51826 > Zombie:80 SA id=32515
RCVD (3.5390s) TCP Zombie:80 > Attacker:51826 R **id=15688**
SENT (3.5390s) TCP Zombie:80 > Target:24 S id=47868
SENT (3.7710s) TCP Attacker:52012 > Zombie:80 SA id=14042
RCVD (3.9190s) TCP Zombie:80 > Attacker:52012 R **id=15689**
```

端口 24 的探测显示 IP ID 没有跳跃。所以该端口不是开放的。根据目前的结果，Nmap 初步确定：

- 端口 20-23 是 `closed|filtered`
- 端口 24、25 和 110 中有两个是 `open`
- 端口 24 和 25 中有一个是 `open`
- 端口 24 是 `closed|filtered`

仔细研究这个谜题，你会发现只有一个解决方案：端口 25 和 110 是开放的，而其他五个是 `closed|filtered`。使用这个逻辑，Nmap 可以停止扫描并立即打印结果。它过去是这样做的，但当僵尸不是真正空闲时，这会产生太多误报的开放端口。因此 Nmap 继续扫描以验证其结果：

```screen

RCVD (5.0690s) TCP Zombie:80 > Attacker:51836 R **id=15694**

```

对端口 25 和 110 的探测表明它们是 `open`，正如我们之前推断的那样。

