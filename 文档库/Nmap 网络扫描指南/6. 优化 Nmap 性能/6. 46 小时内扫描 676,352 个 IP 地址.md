## 46 小时内扫描 676,352 个 IP 地址

本故事由梅奥诊所（Mayo Clinic）的 Jack L. Mogren 提供。它作为一个教程，演示了他为实施定期 Nmap 扫描制度并把这个大型网络的扫描时间从一周缩短到 46 小时所采取的步骤。

梅奥诊所建立了一个相对较大的私有网络，ARP 表显示有超过 70,000 个 IP 地址在使用中。我们的网络管理过去专注于三个主要园区和全国几十个卫星站点的物理架构的创建和维护。我们的座右铭是“你需要？我们建”。对于网络上实际连接了什么却很少关注。网络管理很方便地在数据插孔处结束，并受到“糖果棒综合征”的困扰。它从外部是硬脆和安全的，但内部却是柔软和黏糊的。我们有良好的边界保护，但内部控制很少。

这种态度在 2003 年 1 月突然改变，当时 Slammer 蠕虫（W32.SQLExp）及其变种闯入了我们的环境。突然之间，了解网络上连接了什么变得非常重要。就 Slammer 而言，我们需要知道所有运行 MS SQL Server 2000 或 MSDE 2000 的设备在哪里，以及管理员是谁。由于缺乏这些信息，根除 Slammer 的努力持续了几个月。

因此，“了解网络上有什么”的努力应运而生。这听起来很简单，但考虑到网络的规模、复杂性和历史，这对我们的网络管理服务来说是一个重大的进步和一个新的方向。

Nmap 已被证明是这项努力中的宝贵工具。你无法击败它的价格，而且我很欣赏开源社区为其开发带来的优势。特别是操作系统指纹识别和最终用户提供的许多贡献。

我开始试验 Nmap。我的目标是通过使用 Nmap 的 `-O` 选项，利用 TCP/IP 指纹识别快速执行远程主机识别，从而创建一个有意义的网络清单。

首先，让我介绍一下我们的 IP 环境和我的扫描平台。我们目前拥有一个 B 类地址段和 44 个 C 类地址段，并且使用了大部分私有地址空间。总共有 676,352 个可能的 IP 地址。我在一台运行 Red Hat Linux 8.0 的 Compaq DL380 上执行扫描。我第一次尝试是使用带有操作系统检测（`-O`）和仅 ICMP 回显请求（`-PE`）进行主机发现的普通 TCP SYN 扫描：

```screen
# nmap -O -PE -v -oX mayo.xml -iL ip_networks.txt
```

不幸的是，扫描进行得非常缓慢，扫描我们整个网络将需要一周时间。鉴于我们网络的所有重要部分至少通过 T1 线路（1.54 Mbps）连接，我添加了预设的“疯狂”时序策略（`-T5`）。我还添加了快速扫描模式（`-F`），这将扫描的端口数从大约 1600 个减少到 1200 个[^11]。我还添加了 `--osscan-limit`，以便 Nmap 不会浪费时间在没有任何开放端口的主机上进行操作系统扫描。这产生了以下命令：

```screen
# nmap -O -T5 -PE -F --osscan-limit -v -oX mayo.xml -iL ip_networks.txt
```

不幸的是，这看起来仍然需要几天时间。因此我编辑了 `nmap-services` 文件，将端口数减少到 270 个。然后扫描在大约 49 小时内完成，发现了 66,558 台设备。调整时序变量、移除详细选项并将输出重定向到 `/dev/null` 将时间减少到 46 小时。这给我留下了最终的命令：

```screen
# nmap -O -T5 -PE -F --osscan-limit --max-rtt-timeout 100ms     \
       --max-parallelism 100 --min-hostgroup 100 -oX mayo.xml \
       -iL ip_networks.txt
```

我计划每周执行一次此扫描，并将 XML 格式的输出提供给 MS SQL 数据库。我们其他的扫描方法已经输入到这个数据库中，我们能够创建有助于实现我们“了解网络上有什么”原始目标的报告。我可能会考虑在几个系统上分发扫描负载，以分散工作负担。

[^11]: 在 Nmap 4.75 或更高版本中，`-F` 更加有效，它将扫描的端口数减少到 100 个。
