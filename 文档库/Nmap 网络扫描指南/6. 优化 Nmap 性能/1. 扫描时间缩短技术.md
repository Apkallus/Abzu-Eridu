## 扫描时间缩短技术

解决扫描时间过长的理想方案就是缩短它。本节提供了许多高级技巧。与生活中的许多情况不同，调整 Nmap 命令行可以带来巨大的改变。给你的本田雅阁加装咖啡罐排气管、三英尺高的尾翼和一张大大的红色“R 型”贴纸，并不能显著缩短 0-60 英里加速时间。然而，[“46 小时内扫描 676,352 个 IP 地址”](mayo-scan.html) 一节描述了 Jack Mogren 如何仅仅通过在 Nmap 命令行中添加几个贴纸（我指的是选项）就省去了数天的运行时间。

### 省略非关键测试

当你从不离开铺装路面或只买杂货时，却买了一辆悍马，这在电子世界中的等价物就是启动一次密集而全面的 Nmap 扫描来获取相对微不足道的信息。在家庭网络上，每台主机浪费几秒没什么影响，但对于大型企业来说，这会使日常广域网扫描变得不可行。下面的列表详细说明了避免常见过度扫描错误的方法，从最严重的问题开始，然后是一些即使是高级用户也经常忘记的更微妙的优化。

- **当你只需要确定哪些主机在线时，跳过端口扫描（`-sn`）**  
  有些人使用命令 `nmap <hostname>` 来确定主机是否在线。虽然这有效，但有些过度。Nmap 会发送四个数据包来确定主机在线，然后至少发送 1000 个数据包进行端口扫描。当以这种方式扫描整个网络以查找所有在线主机或特定主机时，问题会被放大。  
  当你只想知道哪些主机在线或它们的 MAC 地址是什么时，与其浪费时间进行端口扫描，不如指定 `-sn` 进行 ping 扫描。

- **限制扫描的端口数量**  
  默认情况下，Nmap 扫描最常见的 1,000 个端口。在响应迅速的机器组成的快速网络上，这可能每台主机只需不到一秒。但是，当遇到速率限制或丢弃探测数据包而不响应的防火墙时，Nmap 必须急剧减速。由于这些原因，UDP 扫描可能会慢得令人痛苦。然而，绝大多数开放端口只集中在几百个端口号内。如果你只扫描 100 个端口而不是默认的 1,000 个，端口扫描速度大约会快 10 倍。你可以使用 `-F`（快速扫描）选项只扫描最常见的 100 个端口，使用 `--top-ports` 指定最常见开放端口的任意数量，或者向 `-p` 提供自定义的端口列表。

- **跳过高级扫描类型（`-sC`、`-sV`、`-O`、`--traceroute` 和 `-A`）**  
  有些人经常指定 `-A` Nmap 选项，这会提供全套功能。它使 Nmap 进行操作系统检测、版本检测、脚本扫描（NSE）和 traceroute，以及默认的端口扫描。版本检测可能非常有用，但也可能拖慢大规模扫描。NSE 也是如此。当时间紧迫时，你可以先在大规模扫描中跳过 `-sC` 和 `-sV`，然后在必要时针对单个端口执行它们。  
  操作系统检测远没有版本检测那么慢，但仍然很容易在每个在线主机上花费 5-10 秒。即使没有它，你也常常可以根据名称、开放端口和局域网上的 MAC 地址猜测操作系统。而且在许多情况下，你可能并不关心操作系统。所以 `-O` 也是另一个仅在必要时使用的候选。作为一种折衷，你可以指定 `--osscan-limit --max-os-tries 1`，这告诉 Nmap 不要重试匹配失败的操作系统检测尝试，并且跳过对任何没有至少一个开放 TCP 端口和一个关闭 TCP 端口的在线主机的操作系统检测。无论如何，针对此类主机的操作系统检测也不那么准确。

- **记得在不需要时关闭 DNS 解析**  
  默认情况下，Nmap 对每个被发现在线的主机执行反向 DNS 解析。如果你使用 `-Pn` 跳过 ping 步骤或指定 `-R`，则会对所有主机进行解析。当使用主机 DNS 库一次查找一个 IP 时，这是一个主要的瓶颈。  
  虽然 Nmap 现在有一个快速的并行反向 DNS 系统来加速查询，但它们仍然可能花费大量时间。当你不需要这些数据时，使用 `-n` 选项禁用它们。对于针对大量主机的简单扫描（如 ping 扫描），省略 DNS 有时可以减少 20% 或更多的扫描时间。在涉及探测数千个端口或使用版本检测等密集功能的更复杂扫描中，DNS 时间不是主要因素。如果你希望 Nmap 主机（使用 `gethostbyaddr` 函数）处理名称解析，请指定 `--system-dns` 选项。这样做会显著降低扫描速度。

### 优化时序参数

Nmap 提供几十个选项来提供提示和规则以控制扫描活动。这些选项从 `-T` 选项提供的高级时序激进程度（在[“时序模板（`-T`）”](performance-timing-templates.html)一节中描述）到[“底层时序控制”](performance-low-level.html)一节中描述的更细粒度的控制。你甚至可以将两者结合。当扫描高度过滤的网络（Nmap 收到的响应很少，无法自行确定其时序估计）时，这些选项特别有用。扫描时间通常可以安全地减半。在充满响应主机的本地局域网上，大多数这些选项效果甚微，因为 Nmap 在这种情况下可以自己确定最优值。

### 分离并优化 UDP 扫描

扫描 UDP 端口很重要，因为许多易受攻击的服务使用该协议，但 UDP 扫描的时序特性和性能要求与 TCP 扫描大不相同。特别需要关注的是 ICMP 错误速率限制，这非常普遍，并且对 UDP 扫描的影响远大于对 TCP 扫描的影响。

由于这些原因，当性能至关重要时，我不建议将 TCP 和 UDP 扫描合并，尽管 Nmap 支持通过诸如 `-sSU` 等选项这样做。你常常希望为每种协议使用不同的时序标志，这需要单独的命令行。[“加速 UDP 扫描”](scan-methods-udp-scan.html#scan-methods-udp-optimizing)一节提供了改进 UDP 扫描性能的有价值的技巧和真实案例。

### 升级 Nmap

有很多次我调查 Nmap 性能不佳的报告，结果发现报告者使用的是多年前过时的旧版本。最新版本的 Nmap 有重要的算法改进、错误修复、性能增强功能（如本地网络 ARP 扫描）等等。面对性能问题，第一步应该是将你的 Nmap 版本（运行 `nmap -V`）与 [https://nmap.org](https://nmap.org) 上提供的最新版本进行比较。如果需要，请升级。如果仍然不够快，请尝试本章中的其他技术。

### 执行并发的 Nmap 实例

有些人试图通过并行运行多个副本（每个针对一个目标）来加速 Nmap。例如，Nessus 扫描器过去默认这样做。这通常比让 Nmap 针对整个网络运行效率低得多、速度慢得多。Nmap 有自己的并行化系统，该系统是根据其需求量身定制的，并且当 Nmap 扫描一个大组时，它可以在了解网络可靠性的过程中加快速度。此外，要求操作系统 fork 65,536 个单独的 Nmap 实例来扫描一个 B 类网络会有很大的开销。同时运行几十个 Nmap 副本也会消耗内存，因为每个实例都加载自己的数据文件副本，如 `nmap-services` 和 `nmap-os-db`。

虽然并行启动单主机 Nmap 扫描是个坏主意，但通常可以通过将扫描分成几个大组并同时执行这些组来提高整体速度。不过不要过度。五到十个 Nmap 进程没问题，但不建议同时启动 100 个 Nmap 进程。启动太多并发的 Nmap 进程会导致资源争用。另一种并发方式是从不同的主机同时运行 Nmap。你可以让 `cron`（或在 Windows 上是 `At`）安排你每个网络上的本地主机同时开始扫描它们本地的机器，然后将结果通过邮件发送到中央数据服务器。从美国扫描你的澳大利亚网络将比从该网络上的本地机器扫描要慢。如果美国的机器必须穿越额外的防火墙才能到达远端网络，差异将更大。

### 从有利的网络位置扫描

限制性的防火墙可以将五秒钟的扫描变成数小时的任务。某些互联网路由相关的延迟和丢包也没有帮助。如果你可以从目标网络本地的主机运行 Nmap，那就这样做。当然，如果目标是像外部攻击者一样看待网络，或者测试防火墙，则需要进行外部扫描。另一方面，扫描和保护内部网络提供了纵深防御，这对于应对内部威胁和那些绕过防火墙的狡猾攻击者至关重要（参见[第 10 章《检测和破坏防火墙与入侵检测系统》](firewalls.html)）。

在进行反向 DNS 解析时，尤其是如果你有一个负担较重的本地名称服务器，使用一个不太繁忙的名称服务器或直接查询权威名称服务器会有所帮助。这种收益通常很小，只对重复或巨大的扫描才有价值。当然，有时选择名称服务器也有非性能方面的原因。

### 增加可用带宽和 CPU 时间

你可以通过增加可用带宽或 CPU 能力来偶尔改善 Nmap 扫描时间。这可以通过安装新的数据线路或 CPU，或者停止同时运行的争用这些资源的应用程序来实现。例如，如果你同时通过下载盗版《黑客帝国2：重装上阵》的 torrent 使 DSL 线路饱和，Nmap 运行会更慢。

Nmap 更常见的约束是其自身的拥塞控制算法，而不是受 CPU 限制或受可用本地带宽限制。这些控制有助于防止网络泛滥并提高准确性。增加 CPU 能力和本地带宽无助于 Nmap 的这种自我限制——必须调整时序选项。你可以通过使用 Unix 上的 `top` 或 Windows 上的任务管理器等应用程序监控 CPU 负载来测试 Nmap 是否受 CPU 限制。如果你的 CPU 大部分时间处于空闲状态，那么升级不会有太大帮助。要测试 Nmap 的带宽使用情况，请在详细模式（`-v`）下运行它。然后 Nmap 将报告发送和接收的字节数及其执行时间，如示例 6.1 所示。

**示例 6.1 本地 100 Mbps 以太网上的带宽使用**

```screen
# nmap -v -n -p- sec.titan.net

Starting Nmap ( https://nmap.org )
[10 lines cut]
Nmap scan report for 192.168.0.8
Not shown: 65534 closed ports
PORT   STATE SERVICE
22/tcp open  ssh
MAC Address: 00:1A:6B:C1:33:37 (USI)

Nmap done: 1 IP address (1 host up) scanned in 2.20 seconds
           Raw packets sent: 65536 (2.884MB) | Rcvd: 65536 (2.621MB)
```

将字节值乘以 8 再除以执行时间，得到以比特每秒为单位的平均带宽使用量。在示例 6.1 中，Nmap 在 2.20 秒内接收了 2,621,000 字节（Nmap 认为 1,000,000 字节为 1 MB）。因此接收流量约为 9.5 Mbps（发送速率为 10.5 Mbps）。因此 100 Mbps 以太网链路不太可能限制 Nmap，升级到千兆以太网不会有太大帮助。

一些消费级宽带设备和其他设备难以处理 Nmap 发送的数据包速率，尽管数据包尺寸小（Nmap 通常发送空报头）保持了较低的带宽。在示例 6.1“本地 100 Mbps 以太网上的带宽使用”中，Nmap 每秒发送约 30,000 个数据包，并接收类似数量的数据包。如此高的数据包速率可能会导致低质量设备出现问题。在本例中，我们看到发送和接收的数据包计数都是 65,536，即扫描的端口数（65,535）加上初始的 ARP ping 探测。因此 Nmap 没有遇到需要重传的任何丢包。这再次表明网络设备不是限制因素——Nmap 可能受 CPU 限制。
