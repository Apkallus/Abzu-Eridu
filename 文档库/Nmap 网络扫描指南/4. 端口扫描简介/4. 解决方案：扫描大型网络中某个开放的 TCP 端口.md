## 解决方案：扫描大型网络中某个开放的 TCP 端口

### 问题

您希望快速找到网络中所有开放了某个 TCP 端口的主机。例如，当发现新的 Microsoft IIS 漏洞后，您可能想要扫描所有开放 TCP 80 端口的机器，并确保它们没有运行存在漏洞的软件版本。或者，在调查一台被入侵的机器时，发现攻击者留下了监听 31337 端口的后门，扫描整个网络的这个端口可以快速找到其他被入侵的系统。完整的（所有端口）扫描可以稍后进行。

### 解决方案

直接的方法是运行：

```bash
nmap -Pn -p<portnumber> -oG <logfilename.gnmap> <target networks>
```

下面是一个具体示例，搜索 4096 个 IP 中开放 Web 服务（80 端口）的主机：

```bash
nmap -Pn -p80 -oG logs/pb-port80scan-%D.gnmap 216.163.128.0/20
```

文件名中的 `%D` 会被替换为扫描运行的日期数字（例如 2007 年 9 月 1 日的 `090107`）。虽然该扫描命令可以工作，但为所扫描的网络选择合适的时序值可以大幅缩短扫描时间。上面的扫描耗时 1236 秒，而下面的优化版本在 869 秒内给出了相同的结果：

```bash
nmap -T4 -Pn -p80 --max-rtt-timeout 200ms --initial-rtt-timeout 150ms \
  --min-hostgroup 512 -oG logs/pb-port80scan2-%D.gnmap \
  216.163.128.0/20
```

其中大部分时间花在了反向 DNS 解析上。通过在命令行中添加 `-n` 来排除该步骤，可将 4096 个主机的扫描时间减少到 193 秒。耐心等待三分钟远比之前等待 21 分钟容易得多。

上述命令将 grepable 格式的结果存储在指定文件中。然后，一个简单的 egrep 命令就可以找到开放 80 端口的主机：

```bash
egrep '[^0-9]80/open' logs/pb-port80scan2-*.gnmap
```

egrep 模式前加 `[^0-9]` 是为了避免错误匹配像 3180 这样的端口。当然，由于我们只扫描端口 80，这种情况不可能发生，但对于多端口扫描来说，这是一个值得记住的好习惯。如果您只想要 IP 地址，可以将 egrep 输出通过管道传递给 `awk '{print $2}'`。

### 讨论

有时，一个故事是理解决策的最佳方式，例如我如何在解决方案部分确定命令行。我在家感到无聊，开始探索一本名为《Playboy》的流行杂志的网络。他们的主站点包含大量图片库，但大多数都被付费订阅认证系统锁定。我很好奇是否能在他们的网络上找到其他免费提供图片的系统。我推测他们可能有一些依赖于隐蔽性而非密码认证的临时或开发服务器。虽然理论上这些服务器可以监听任何端口，但最可能的是 TCP 80 端口。因此，我决定尽可能快地扫描他们的整个网络，寻找开放的该端口。

第一步是确定要扫描的 IP 地址。我对美国 Internet 号码注册机构（ARIN）进行了 whois 搜索，查找名为 Playboy 的组织。结果如示例 4.5 所示。

接下来，我想估算到这些机器的延迟，以便 Nmap 知道预期情况。这不是必须的，但为 Nmap 提供合适的时序值可以加快速度。对于单端口的 `-Pn` 扫描尤其如此。Nmap 无法从每个主机收到足够多的响应来准确估算延迟和数据包丢失率，所以我将在命令行上帮助它。我首先想到的是 ping 他们的主 Web 服务器，如示例 4.6 所示。

这些是我要找的结果。洛杉矶的主机响应时间从未超过 16 毫秒，而芝加哥的主机响应时间长达 62 毫秒。考虑到我是从加利福尼亚的一台机器上探测的，这并不奇怪。谨慎一点是值得的，并且在大量扫描期间延迟可能会增加，因此我决定让 Nmap 等待响应最多 200 毫秒。我将让它以 150 毫秒的超时开始。因此我传递了选项 `--max-rtt-timeout 200ms --initial-rtt-timeout 150ms`。为了设置一个总体积极的时序模式，我在命令行开头指定了 `-T4`。

由于我更看重最小化整个扫描的完成时间，而不是最小化返回第一批主机结果之前的时间，因此我指定了一个较大的扫描组大小。指定了 `--min-hostgroup 512` 选项，以便至少并行扫描 512 个 IP（如果可能）。使用目标网络大小（4096）的精确因子可以避免出现小的、效率较低的 96 个主机块（如果我指定 `--min-hostgroup 500`，扫描结束时会出现这样的块）。所有这些时序问题在第 6 章《优化 Nmap 性能》中有更深入的讲解。

没有必要浪费时间进行预先的 ping 阶段，因为 ping 本身就会花费与单端口扫描一样长的时间。所以指定了 `-Pn` 来禁用该阶段。通过 `-n` 参数跳过反向 DNS 解析可以节省大量时间。否则，在禁用 ping 扫描的情况下，Nmap 会尝试查找所有 4096 个 IP。我正在搜索 Web 服务器，因此使用 `-p80` 请求端口 80。当然，我会错过任何在非标准端口（如 81 或 8080）上运行的 HTTP 服务器。端口 443 上的 SSL 服务器也不会被找到。可以将它们添加到 `-p` 选项中，但即使多一个端口也会使扫描时间加倍，扫描时间大致与扫描的端口数成正比。

最后一个选项是 `-oG`，后面跟着我想要存储 grepable 结果的文件名。我将目标网络附加到命令末尾，然后按回车执行 Nmap。输出如示例 4.10 所示。

Nmap 在大约三分钟内扫描了全部 4096 个 IP。正常输出显示许多端口处于 `filtered` 状态。这些 IP 中的大多数可能不是活跃主机——端口之所以显示为被过滤，是因为 Nmap 没有收到对其 SYN 探测的响应。我通过对输出文件执行简单的 egrep 来获得 Web 服务器列表，如示例 4.11 所示。

**示例 4.11 使用 egrep 查找开放端口**

```screen
# egrep '[^0-9]80/open' pb-port80scan-*.gnmap
Host: 216.163.140.20 () Ports: 80/open/tcp//http///
Host: 216.163.142.135 ()     Ports: 80/open/tcp//http///
```

经过这么多努力，在 4096 个 IP 中只发现了两个可访问的 Web 服务器！有时会发生这种情况。第一个 IP 216.163.140.20（没有反向 DNS 名称）将我带到了 Microsoft Outlook Web Access（webmail）服务器。如果我试图入侵他们的网络，这可能会让我兴奋，但现在并不令人满意。下一个服务器（反向名称 mirrors.playboy.com）要好得多。它提供了我所希望的免费图片！特别是它提供了 Linux ISO 镜像以及大量的 FreeBSD、CPAN 和 Apache 存档！我以可观的 6 Mbps 下载了最新的 Fedora Core ISO。Playboy 拥有丰富的带宽并不令人惊讶。后来我扫描了 Playboy 的其他网络块，发现了更多的 Web 服务器，尽管其中一些内容不适合本书。

虽然这是一个不寻常的端口扫描理由，但单端口扫描对于前面提到的许多其他目的来说是常见的。这里描述的技术可以轻松应用于任何单端口 TCP 扫描。
