## 后处理器

一旦 Nmap 如上所示推断出服务和版本信息，通常就完成了对端口的处理。然而，对于某些服务，Nmap 还会执行额外的工作。目前可用的后处理器包括 Nmap 脚本引擎集成、RPC 磨削和 SSL 隧道。Windows SMB 探测也在考虑之中。

### Nmap 脚本引擎集成

基于正则表达式的版本检测方法功能强大，但无法识别所有服务。某些服务无法通过简单地发送标准探测并将模式与响应匹配来识别。有些服务需要自定义探测字符串或复杂的多步握手过程。其他服务则需要比正则表达式更高级的处理才能识别响应。例如，Skype v2 服务被设计得难以检测，因为在位运营商（如提供 DSL 线路的电话公司）可能会将其视为竞争对手，从而降级或阻止其用户使用该服务。我们发现检测该服务的唯一方法是分析对两个不同探测的响应。类似地，如果我们通过暴力尝试几百个不同的团体名称，就能识别更多的 SNMP 服务。这些任务都不适合传统的 Nmap 版本检测，但两者都通过 [第 9 章《Nmap 脚本引擎》](nse.html) 完成了。

出于这些原因，版本检测现在默认调用 NSE 来处理一些棘手的服务，如 [“使用 NSE 进行版本检测”](nse-vscan.html) 一节所述。

### RPC 磨削

SunRPC（Sun 远程过程调用）是一种常见的 Unix 协议，用于实现包括 NFS 在内的许多服务。Nmap 附带了一个包含近 600 个 RPC 程序的 `nmap-rpc` 数据库。许多 RPC 服务使用高位端口和/或 UDP 传输协议，这使得它们能够通过许多配置不当的防火墙访问。RPC 程序（以及基础设施库本身）也长期存在严重的远程可利用安全漏洞。因此，网络管理员和安全审计员通常希望更多地了解其网络上的任何 RPC 程序。

如果端口映射器（rpcbind）服务（UDP 或 TCP 端口 111）可用，则可以使用 Unix 的 `rpcinfo` 命令枚举 RPC 服务。[示例 7.5](#ex-version-detection-rpcinfo) 演示了针对默认 Solaris 9 服务器的此操作。

**示例 7.5 使用 `rpcinfo` 枚举 RPC 服务**

```screen
> rpcinfo -p ultra

```

此示例显示主机经常提供许多 RPC 服务，这增加了其中一个容易被利用的可能性。您还应该注意到，大多数服务都在奇怪的高位端口上（可能因多种原因而变化），并且分布在 UDP 和 TCP 传输协议之间。

由于 RPC 信息非常敏感，许多管理员试图通过阻止端口映射器端口（111）来掩盖此信息。不幸的是，这并没有堵住漏洞。Nmap 可以通过以下三步过程直接与开放的 RPC 端口通信来确定所有相同的信息。

1. TCP 和/或 UDP 端口扫描找到所有开放端口。
2. 版本检测确定哪些开放端口使用 SunRPC 协议。
3. RPC 暴力引擎通过针对 `nmap-rpc` 中近 600 个程序号中的每一个尝试一个**空命令**来确定每个 RPC 端口的程序标识。大多数时候 Nmap 猜测错误并收到错误消息，说明请求的程序号未在端口上监听。Nmap 继续尝试列表中的每个号，直到成功返回其中一个。如果 Nmap 耗尽其已知的所有程序号，或者端口发送格式错误的响应（表明它不是真正的 RPC），则 Nmap 放弃。

RPC 程序识别探测是并行完成的，并且为 UDP 端口处理重传。只要版本检测发现任何 RPC 端口，此功能就会自动激活。[示例 7.6](#ex-version-detection-rpcscan) 演示了作为版本检测一部分完成的直接 RPC 扫描。

### SSL 后处理器说明

正如技术部分所讨论的，Nmap 能够检测 SSL 加密协议，然后启动加密会话，通过该会话执行正常的版本检测。与前面讨论的 RPC 磨削一样，只要检测到适当的（SSL）端口，SSL 后处理器就会自动执行。[示例 7.7](#ex-version-detection-ssl) 演示了这一点。

**示例 7.7 通过 SSL 进行版本扫描**

```screen
$ nmap -Pn -sSV -T4 -F www.amazon.com

Starting Nmap ( https://nmap.org )
Nmap scan report for 207-171-184-16.amazon.com (207.171.184.16)
(The 1214 ports scanned but not shown below are in state: filtered)
PORT    STATE SERVICE  VERSION
80/tcp  open  http     Apache Stronghold httpd 2.4.2 (based on Apache 1.3.6)
443/tcp open  ssl/http Apache Stronghold httpd 2.4.2 (based on Apache 1.3.6)

Nmap finished: 1 IP address (1 host up) scanned in 35.038 seconds
```

请注意，两个开放端口的版本信息相同，但端口 80 上的服务是 `http`，而端口 443 上的服务是 `ssl/http`。端口 443 上 HTTPS 的常见情况不是硬编码的——Nmap 应该能够检测任何端口上的 SSL，并确定 Nmap 可以在明文中检测到的任何服务的底层协议。如果 Nmap 没有检测到 SSL 后面监听的服务器，列出的服务将是 `ssl/unknown`。如果 Nmap 没有在启用 SSL 支持的情况下构建，列出的服务将仅仅是 `ssl`。在这两种情况下，版本字段都将为空。

Nmap 的 SSL 支持依赖于免费的 [OpenSSL 库](http://www.openssl.org)。它不包含在 Linux RPM 二进制文件中，以避免破坏缺少这些库的系统。Nmap 源代码分发版会尝试检测系统上的 OpenSSL，并在可用时链接到它。有关自定义构建过程以包含或排除 OpenSSL 的详细信息，请参见 [第 2 章《获取、编译、安装和移除 Nmap》](install.html)。
