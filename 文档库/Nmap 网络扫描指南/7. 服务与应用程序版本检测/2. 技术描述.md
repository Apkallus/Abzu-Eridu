## 技术描述

Nmap 版本扫描实际上相当直接。它被设计得尽可能简单，同时仍然具有可扩展性、快速和准确性。真正最细微的细节最好通过下载和查看源代码来发现，但接下来是对所用技术的概要介绍。

Nmap 首先按照你的指示进行端口扫描，然后将所有 `open` 或 `open|filtered` 的 TCP 和/或 UDP 端口传递给服务扫描模块。这些端口随后被并行探测，但为简单起见，这里只描述单个端口。

1. Nmap 检查该端口是否是要排除的端口之一（由 `nmap-service-probes` 文件中的 `Exclude` 指令指定）。如果是，Nmap 将不会扫描此端口，原因见 [“`nmap-service-probes` 文件格式”](vscan-fileformat.html) 一节。

2. 如果端口是 TCP，Nmap 首先尝试连接。如果连接成功并且该端口之前处于 `open|filtered` 状态，则将其更改为 `open`。这种情况对于 TCP 很少见，因为那些为了隐蔽而使用产生 `open|filtered` 端口（如 FIN 扫描）的 TCP 扫描类型的人通常知道，执行版本检测会破坏他们的隐蔽性。

3. 一旦 TCP 连接建立，Nmap 会监听大约五秒钟。许多常见服务，包括大多数 FTP、SSH、SMTP、Telnet、POP3 和 IMAP 服务器，会在初始欢迎标语中标识自己。Nmap 将此称为“NULL 探测”，因为 Nmap 只是监听响应而不发送任何探测数据。如果收到任何数据，Nmap 会将其与其 `nmap-service-probes` 文件（在 [“`nmap-service-probes` 文件格式”](vscan-fileformat.html) 中描述）中的大约 3,000 个 NULL 探测签名进行比较。如果服务被完全识别，该端口的处理就完成了！签名中的正则表达式可以包含子串匹配，以从响应中提取版本号。在某些情况下，Nmap 会在服务类型上获得“软匹配”，但没有版本信息。此时，Nmap 继续但仅发送已知能识别该软匹配服务类型的探测。

4. 这是 UDP 探测的起点，而 TCP 连接则在上面的 NULL 探测失败或软匹配后继续。由于现实情况是大多数端口被用于它们在 `nmap-services` 中注册的服务，每个探测都有一个被认为最有效的端口号列表。例如，识别 Web 服务器（以及许多其他服务）的名为 GetRequest 的探测，其可能端口列表中包含 80-85、8000-8010 和 8080-8085。Nmap 按文件中出现的顺序依次执行与正在扫描的端口号匹配的探测。

   每个探测包含一个探测字符串（可以是任意 ASCII 文本或 \xHH 转义的二进制数据），被发送到端口。返回的响应与一组签名正则表达式进行比较，其类型与上面 NULL 探测描述中讨论的相同。与 NULL 探测一样，这些测试可以产生完全匹配（终止对远程服务的处理）、软匹配（将未来探测限制为与特定服务匹配的探测）或根本不匹配。Nmap 用于测试匹配的确切正则表达式列表取决于探测的回退配置。例如，从 X11Probe 返回的数据不太可能匹配为 GetRequest 探测设计的任何正则表达式。另一方面，从 RTSPRequest 等探测返回的结果可能匹配为 GetRequest 设计的正则表达式，因为被测试的两个协议密切相关。因此，RTSPRequest 探测具有回退到 GetRequest 匹配的功能。更全面的解释请参见 [“作弊和回退”](#vscan-cheats-and-fallbacks) 一节。

   如果在版本检测期间从处于 `open|filtered` 状态的 UDP 端口收到任何响应，则该状态将更改为 `open`。这使得版本检测成为 UDP 扫描的绝佳补充，因为当某些常见防火墙规则生效时，UDP 扫描被迫将所有扫描的 UDP 端口标记为 `open|filtered`。虽然将 UDP 扫描与版本检测结合可能比普通 UDP 扫描多花费数倍时间，但它是有效且有用的技术。此方法在 [“区分开放 UDP 端口与被过滤 UDP 端口”](scan-methods-udp-scan.html#scan-methods-disambiguate) 一节中进行了描述。

5. 在大多数情况下，NULL 探测或上述的可能端口探测（通常只有一个）会匹配服务。由于 NULL 探测与可能端口探测共享连接，这允许在大多数情况下仅通过一次简短连接完成服务检测。对于 UDP，通常只需要一个数据包。但如果 NULL 探测和可能端口探测失败，Nmap 会依次遍历其他现有探测。对于 TCP，Nmap 必须为每个探测建立新连接，以避免先前探测破坏结果。这种最坏情况可能需要一些时间，尤其是因为由于网络连接缓慢或服务响应缓慢，Nmap 必须为每个探测等待大约五秒钟才能得到结果。幸运的是，Nmap 利用了几种自动技术来加速扫描：

   - Nmap 使大多数探测足够通用以匹配许多服务。例如，GenericLines 探测向服务发送两个空行（`\r\n\r\n`）。这匹配多种服务类型的守护进程，包括 FTP、ident、POP3、UUCP、Postgres 和 whois。GetRequest 探测匹配更多的服务类型。其他示例包括 `help\r\n` 以及通用 RPC 和 MS SMB 探测。
   - 如果服务匹配了 `softmatch` 指令，Nmap 只需要尝试可能匹配该服务的探测。
   - 并非所有探测都是平等的！有些探测比其他的匹配更多服务。因此，Nmap 使用“稀有度”指标来避免尝试极不可能匹配的探测。有经验的 Nmap 用户可以使用 [“探测选择与稀有度”](#vscan-selection-and-rarity) 一节中讨论的 `--version-intensity`、`--version-all` 和 `--version-light` 选项，强制尝试所有探测或比默认更严格地限制探测尝试。

6. 其中一个探测测试目标端口是否运行 SSL。如果是（并且如果 OpenSSL 可用），Nmap 会通过 SSL 重新连接并重新启动服务扫描，以确定加密后面监听的是什么。一个特殊的指令允许普通连接和 SSL 隧道连接使用不同的可能端口。例如，Nmap 应该使用 SSL 探测针对端口 443（HTTPS）开始。但在检测并启用 SSL 后，Nmap 应该针对端口 443 尝试 GetRequest 探测，因为该端口通常有一个 Web 服务器在 SSL 加密后面监听。

7. 另一个通用探测识别基于 RPC 的服务。当发现这些服务时，会启动 Nmap RPC 磨床（稍后讨论）以暴力破解 RPC 程序号/名称和受支持的版本号。类似地，用于 Windows 服务指纹识别的 SMB 后处理器作为 [第 9 章《Nmap 脚本引擎》](nse.html) 的一部分提供。

8. 如果至少有一个探测引发了某种响应，但 Nmap 无法识别该服务，则响应内容将以**指纹**的形式打印给用户。如果用户知道实际监听的是什么服务，鼓励他们向 Nmap 开发者提交指纹以集成到 Nmap 中，如 [“提交服务指纹”](vscan-community.html#vscan-submit-prints) 所述。

### 作弊和回退

尽管 Nmap 为服务回复等待了足够长的时间，但有时应用程序对 NULL 探测的响应较慢。这可能由多种原因引起，包括某些服务执行缓慢的反向 DNS 查找。因此，Nmap 有时可以将后续探测的结果与为 NULL 探测设计的匹配行进行匹配。

例如，假设我们扫描服务器上的端口 25（SMTP）以确定监听的是什么。一旦我们连接，该服务可能会进行一系列 DNS 黑名单查找，以确定我们是否应被视为垃圾邮件发送者并被拒绝服务。在它完成之前，Nmap 放弃了等待 NULL 探测响应，并发送了端口 25 注册的下一个探测，即 `HELP\r\n`。当服务最终完成其反垃圾邮件检查时，它会打印一个问候标语，读取 Help 探测，并如[示例 7.4](#ex-version-null-probe-cheat) 所示响应。

**示例 7.4 NULL 探测作弊示例输出**

```screen
220 hcsw.org ESMTP Sendmail 8.12.3/8.12.3/Debian-7.1; Tue, [cut]
214-2.0.0 This is sendmail version 8.12.3
214-2.0.0 Topics:
214-2.0.0       HELO    EHLO    MAIL    RCPT    DATA
214-2.0.0       RSET    NOOP    QUIT    HELP    VRFY
214-2.0.0       EXPN    VERB    ETRN    DSN     AUTH
214-2.0.0       STARTTLS
214-2.0.0 For more info use "HELP &lt;topic&gt;".
214-2.0.0 To report bugs in the implementation send email to
214-2.0.0       sendmail-bugs@sendmail.org.
214-2.0.0 For local information send email to Postmaster at your site.
214 2.0.0 End of HELP info
```

Nmap 从套接字读取这些数据，发现没有来自 Help 探测的正则表达式与返回的数据匹配。这是因为 Nmap 通常期望在 NULL 探测期间接收 ESMTP 标语并在那里匹配它。

由于这是一个相对常见的场景，Nmap 会“作弊”，如果没有探测特定的行匹配，则尝试将响应与任何 NULL 探测匹配行进行匹配。在这种情况下，存在一个 NULL 匹配行，它报告程序是 Sendmail，版本是 8.12.3/8.12.3/Debian-7.1，主机名是 hcsw.org。

NULL 探测作弊实际上只是 Nmap 更通用功能的一个具体例子：回退。回退指令在 [“`nmap-service-probes` 文件格式”](vscan-fileformat.html) 中有详细描述。本质上，任何可能遇到可以被其他探测中的正则表达式匹配的结果的探测，都有一个指定这些其他探测的回退指令。

例如，在流行的 Apache Web 服务器的某些配置中，Apache 不会响应 GetRequest（`GET / HTTP/1.0\r\n\r\n`）探测，因为没有指定虚拟主机名。Nmap 仍然能够正确识别这些服务器，因为这些服务器通常会对 HTTPOptions 探测做出响应。该探测回退到 GetRequest 正则表达式，这些正则表达式足够通用，可以识别 Apache 对 HTTPOptions 探测的响应。

### 探测选择与稀有度

在确定使用哪些探测时，Nmap 会考虑它们的 `rarity`（稀有度）。这表示探测返回有用数据的可能性。如果探测的稀有度较高，则认为它不太常见，并且不太可能被尝试。Nmap 用户可以通过更改版本扫描的强度级别来指定尝试哪些探测，如下所述。Nmap 在决定使用哪些探测时使用的精确算法如下：

1. 对于 TCP，始终首先尝试 NULL 探测。
2. 所有将正在扫描的端口列为可能端口的探测（参见 [“`nmap-service-probes` 文件格式”](vscan-fileformat.html)）按照它们在 `nmap-service-probes` 中出现的顺序被尝试。
3. 所有其他稀有度值小于或等于扫描当前强度值的探测，也按照它们在 `nmap-service-probes` 中出现的顺序被尝试。

一旦发现探测匹配，算法终止并报告结果。

由于 Nmap 的所有探测（NULL 探测除外）都有与之关联的稀有度值，因此在执行版本扫描时控制尝试多少个探测相对容易。只需选择适合扫描的强度级别即可。强度级别越高，尝试的探测就越多。因此，如果需要非常全面的扫描，高强度级别是合适的——尽管它可能比在较低强度级别下进行的扫描花费更长时间。Nmap 的默认强度级别是 7，但 Nmap 为不同的扫描需求提供了以下开关：

- **`--version-intensity <intensity level between 0 and 9>`**  
  将版本扫描的强度级别设置为指定值。如果指定为 0，则仅尝试 NULL 探测（对于 TCP）以及将端口列为可能端口的探测。示例：`nmap -sV --version-intensity 3 scanme.nmap.org`

- **`--version-light`**  
  将强度级别设置为 2。示例：`nmap -sV --version-light scanme.nmap.org`

- **`--version-all`**  
  将强度级别设置为 9。由于所有探测的稀有度级别都在 1 到 9 之间，这将尝试所有探测。即使发生软匹配，也会尝试所有额外的探测。示例：`nmap -sV --version-all scanme.nmap.org`
