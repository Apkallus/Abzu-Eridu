## `nmap-service-probes` 文件格式

与远程操作系统检测（`-O`）一样，Nmap 使用一个纯文本文件来存储版本检测的探测和匹配字符串。虽然 Nmap 分发的 `nmap-service-probes` 版本对大多数用户来说已经足够，但理解文件格式允许高级 Nmap 黑客将自己的服务添加到检测引擎中。与许多 Unix 文件一样，`nmap-service-probes` 是面向行的。以井号（`#`）开头的行被视为注释，解析器会忽略它们。空行也会被忽略。其他行必须包含下面描述的指令之一。一些读者可能更喜欢在阅读下面的详细剖析之前，先看一下[“综合示例”](#vscan-fileformat-example)中的例子。

### `Exclude` 指令

**语法**：`Exclude` _`<port specification>`_

**示例**：

```plaintext
Exclude 53,T:9100,U:30000-40000
```

该指令从版本扫描中排除指定的端口。它只能使用一次，并且应靠近文件顶部，位于任何 `Probe` 指令之前。`Exclude` 指令使用与 Nmap `-p` 开关相同的格式，因此支持范围和逗号分隔的端口列表。在 Nmap 附带的 `nmap-service-probes` 中，唯一排除的端口是 TCP 端口 9100 到 9107。这些是打印机常见的监听端口，它们通常会打印发送给它们的任何数据。因此，版本扫描可能会导致它们打印许多页的探测（如 SunRPC 请求、help 语句和 X11 探测）。

这种行为通常是不希望的，尤其是当扫描目的是隐蔽时。然而，Nmap 默认避免扫描此端口，这可能会使狡猾的用户更容易隐藏服务：只需在排除的端口（如 9100）上运行服务，它就不太可能被识别出名称。端口扫描仍会显示它是开放的。用户可以使用 `--allports` 选项覆盖 `Exclude` 指令。这将导致版本检测询问所有开放端口。

### `Probe` 指令

**语法**：`Probe` _`<protocol>`_ _`<probename>`_ _`<probestring>`_ [`no-payload`]

**示例**：

```plaintext
Probe TCP GetRequest q|GET / HTTP/1.0\r\n\r\n|
Probe UDP DNSStatusRequest q|\0\0\x10\0\0\0\0\0\0\0\0\0|
Probe TCP NULL q||
Probe UDP Sqlping q|\x02| no-payload
```

`Probe` 指令告诉 Nmap 发送什么字符串来识别各种服务。后面讨论的所有指令都作用于最近的 `Probe` 语句。参数如下：

- **_`<protocol>`_**：必须是 `TCP` 或 `UDP`。Nmap 只使用与其尝试扫描的服务协议相匹配的探测。

- **_`<probename>`_**：探测的纯英文名称。它在服务指纹中用于描述哪些探测引发了响应。

- **_`<probestring>`_**：告诉 Nmap 发送什么。它必须以 `q` 开头，然后是一个分隔符字符，该字符作为字符串的开头和结尾。分隔符之间的字符是实际发送的字符串。它的格式类似于 C 或 Perl 字符串，允许以下标准转义字符：`\\`、`\0`、`\a`、`\b`、`\f`、`\n`、`\r`、`\t`、`\v` 和 `\xHH`（其中 `H` 是任何十六进制数字）。`nmap-service-probes` 中有一个 `Probe` 行具有空探测字符串，如上面的第三个示例所示。这是 TCP NULL 探测，它只是监听许多服务发送的初始标语。如果您的分隔符（这些示例中的 `|`）是探测字符串所需要的，您需要选择一个不同的分隔符。

- **`[no-payload]`**：此关键字用于指示 Nmap 在 UDP 端口扫描期间不要将给定的探测用作特定于协议的有效载荷。例如，`Sqlping` 探测有时会触发 IDS 的“MS-SQL ping attempt”警报。

### `match` 指令

**语法**：`match` _`<service>`_ _`<pattern>`_ [`_`<versioninfo>`_`]

**示例**：

```plaintext
match ftp m/^220.*Welcome to .*Pure-?FTPd (\d\S+\s*)/ p/Pure-FTPd/ v/$1/ cpe:/a:pureftpd:pure-ftpd:$1/
match ssh m/^SSH-([\d.]+)-OpenSSH[_-]([\w.]+)\r?\n/i p/OpenSSH/ v/$2/ i/protocol $1/ cpe:/a:openbsd:openssh:$2/
match mysql m|^\x10\0\0\x01\xff\x13\x04Bad handshake$| p/MySQL/ cpe:/a:mysql:mysql/
match chargen m|@ABCDEFGHIJKLMNOPQRSTUVWXYZ|
match uucp m|^login: login: login: $| p/NetBSD uucpd/ o/NetBSD/ cpe:/o:netbsd:netbsd/a
match printer m|^([\w-_.]+): lpd: Illegal service request\n$| p/lpd/ h/$1/
match afs m|^[\d\D]{28}\s*(OpenAFS)([\d\.]{3}[^\s\0]*)\0| p/$1/ v/$2/
```

`match` 指令告诉 Nmap 如何根据对前一个 `Probe` 指令发送的字符串的响应来识别服务。一个 `Probe` 行后面可以跟几十或几百个 `match` 语句。如果给定的模式匹配，可选的版本说明符将构建应用程序名称、版本号以及供 Nmap 报告的其他信息。该指令的参数如下：

- **_`<service>`_**：仅仅是模式匹配的服务名称。例如 `ssh`、`smtp`、`http` 或 `snmp`。作为一种特殊情况，您可以在服务名称前加上 `ssl/`，如 `ssl/vmware-auth`。在这种情况下，服务将被存储为被 SSL 封装的 `vmware-auth`。这对于无需建立 SSL 连接的开销就能完全识别的服务很有用。

- **_`<pattern>`_**：此模式用于确定接收到的响应是否与上一个参数中给出的服务匹配。格式类似于 Perl，语法为 `m/[regex]/[opts]`。`m` 告诉 Nmap 一个匹配字符串开始了。斜杠（`/`）是一个分隔符，只要第二个斜杠也相应替换，可以用几乎任何可打印字符代替。正则表达式是 [Perl 风格的正则表达式](http://www.perl.com/doc/manual/html/pod/perlre.html)，这得益于出色的 [PCRE 库](http://www.pcre.org)。当前支持的选项只有 `i`（使匹配不区分大小写）和 `s`（在点号说明符中包含换行符）。正如您所期望的，这两个选项的语义与 Perl 中的相同。要捕获的子表达式（如版本号）用括号括起来，如大多数上面的示例所示。

- **_`<versioninfo>`_**（可选）：  
  _`<versioninfo>`_ 部分实际上包含几个可选字段。每个字段以标识字母开头（例如 `h` 表示“主机名”）。然后是签名编写者选择的分隔符。首选分隔符是斜杠（`/`），除非该字符在字段本身中使用。接下来是字段值，然后是分隔符。下表描述了六个字段：

  **表 7.1 `versioninfo` 字段格式和值**

  | 字段格式 | 值描述 |
  | :--- | :--- |
  | `p/vendorproductname/` | 包含供应商名称，通常还有服务名称，形式如 `Sun Solaris rexecd`、`ISC BIND named` 或 `Apache httpd`。 |
  | `v/version/` | 应用程序版本“号”，可能包含非数字字符甚至多个单词。 |
  | `i/info/` | 立即可用且可能有用的杂项信息。示例包括 X 服务器是否对未经身份验证的连接开放，或 SSH 服务器的协议号。 |
  | `h/hostname/` | 服务提供的主机名（如果有）。这在 SMTP 和 POP3 等协议中很常见，并且很有用，因为这些主机名可能用于内部网络，或者与直接的反向 DNS 响应不同。 |
  | `o/operatingsystem/` | 服务运行的操作系统。这可能与基于 Nmap IP 协议栈的操作系统检测报告的操作系统不同。例如，目标 IP 可能是一个 Linux 机器，它使用网络地址转换将请求转发到 DMZ 中的 Microsoft IIS 服务器。在这种情况下，协议栈操作系统检测应报告操作系统为 Linux，而服务检测报告端口 80 为 Windows。 |
  | `d/devicetype/` | 服务运行的设备类型，如 `print server` 或 `webcam`。某些服务会披露此信息，并且在更多情况下可以推断出来。例如，HP-ChaiServer Web 服务器仅在打印机上运行。有关设备类型的完整列表，请参阅[“设备类型”](osdetect-device-types.html)一节。 |
  | `cpe:/cpename/[a]` | 服务某个方面的 CPE 名称。可以多次使用；不仅可以识别服务（`cpe:/a` 名称），还可以识别操作系统（`cpe:/o` 名称）和硬件平台（`cpe:/h` 名称）。末尾的斜杠不是 CPE 语法的一部分，但为了匹配其他字段的格式而包含。关于 CPE 的更多信息，请参阅[“通用平台枚举（CPE）”](output-formats-cpe.html)一节。 |

  任何字段都可以省略。事实上，如果没有关于服务的进一步信息，所有字段都可以省略。任何版本字段都可以包含编号字符串，如 `$1` 或 `$2`，它们将以类似 Perl 的方式替换为 _`<pattern>`_ 中对应的带括号的子串。在 `cpe://` 模板中，这样的替换会进行如下转换：某些字符（如冒号）被转义；空格转换为下划线，并且所有字符变为小写。

  在极少数情况下，可以在插入前对替换文本应用一个**辅助函数**。下表描述了三种可用的辅助函数：

  **表 7.2 `versioninfo` 辅助函数**

  | 辅助函数 | 功能描述 |
  | :--- | :--- |
  | `$P()` | 过滤掉不可打印字符。这对于将 Unicode UTF-16 编码字符串（如 `W\0O\0R\0K\0G\0R\0O\0U\0P\0`）转换为 ASCII 近似值（如 `WORKGROUP`）很有用。可以通过向其传递要转换为可打印的匹配编号在任何 `versioninfo` 字段中使用，例如：`i/$P(3)/`。 |
  | `$SUBST()` | 在打印前对匹配项进行替换。它接受三个参数。第一个是模式中的替换编号，就像您在普通替换变量（如 `$1` 或 `$3`）中使用的一样。第二个和第三个参数分别指定要查找和替换的子字符串。匹配字符串的所有实例都会被替换，而不仅仅是第一个。例如，VanDyke VShell sshd 给出的版本号格式如 `2_2_3_578`。我们使用版本信息字段 `v/$SUBST(1,"_",".")/` 将其转换为更传统的形式 `2.2.3.578`。 |
  | `$I()` | 从捕获的字节中解包一个无符号整数。给定最多 8 个字节的捕获字符串，它将把它们视为无符号整数并转换为十进制格式。它接受两个参数。第一个是模式中的替换编号。第二个是表示大端字节序的 `">"` 或表示小端字节序的 `"<"`。 |

### `softmatch` 指令

**语法**：`softmatch` _`<service>`_ _`<pattern>`_

**示例**：

```plaintext
softmatch ftp m/^220 [-.\w ]+ftp.*\r\n$/i
softmatch smtp m|^220 [-.\w ]+SMTP.*\r\n|
softmatch pop3 m|^\+OK [-\[\]\(\)!,/+:&lt;&gt;@.\w ]+\r\n$|
```

`softmatch` 指令的格式与上面讨论的 `match` 指令相似。主要区别在于，在软匹配之后扫描继续进行，但仅限于已知能匹配给定服务的探测。这允许稍后找到正常的（“硬”）匹配，从而提供有用的版本信息。有关其工作原理的更多详细信息，请参见[“技术描述”](vscan-technique.html)一节。这里不定义参数，因为它们与上面的 `match` 相同，只是没有 _`<versioninfo>`_ 参数。与 `match` 一样，在单个 `Probe` 部分中可以存在多个 `softmatch` 语句。

### `ports` 和 `sslports` 指令

**语法**：`ports` _`<portlist>`_

**示例**：

```plaintext
ports 21,43,110,113,199,505,540,1248,5432,30444
ports 111,4045,32750-32810,38978
```

该行告诉 Nmap，此探测识别的服务通常出现在哪些端口上。它应该在每个 `Probe` 部分中只使用一次。语法是 Nmap `-p` 选项的简化版本。参见上面的示例。有关其工作原理的更多详细信息，请参见[“技术描述”](vscan-technique.html)一节。

**语法**：`sslports` _`<portlist>`_

**示例**：

```plaintext
sslports 443
```

这与上面描述的 `ports` 指令相同，只是这些端口通常用于将服务包装在 SSL 中。例如，HTTP 探测声明了 `sslports 443`，而 SMTP 检测探测有 `sslports 465` 行，因为它们分别是 HTTPS 和 SMTPS 的标准端口。_`<portlist>`_ 格式与 `ports` 相同。此可选指令在每个 `Probe` 中不能出现超过一次。

### `totalwaitms` 指令

**语法**：`totalwaitms` _`<milliseconds>`_

**示例**：

```plaintext
totalwaitms 6000
```

这个很少需要的指令指定了 Nmap 在放弃针对特定服务的最新的 `Probe` 之前应等待的时间。Nmap 的默认值通常就很好。

### `tcpwrappedms` 指令

**语法**：`tcpwrappedms` _`<milliseconds>`_

**示例**：

```plaintext
tcpwrappedms 3000
```

该指令仅用于 Null 探测。如果服务在此计时器结束之前关闭 TCP 连接，则该服务被标记为 `tcpwrapped`。否则，匹配将照常继续。

### `rarity` 指令

**语法**：`rarity` _`<value between 1 and 9>`_

**示例**：

```plaintext
rarity 6
```

`rarity` 指令大致对应于期望此探测返回有用结果的频率。数字越高，探测被认为越稀有，越不可能针对服务进行尝试。更多详细信息请参见[“探测选择与稀有度”](vscan-technique.html#vscan-selection-and-rarity)一节。

### `fallback` 指令

**语法**：`fallback` _`<Comma separated list of probes>`_

**示例**：

```plaintext
fallback GetRequest,GenericLines
```

此可选指令指定当当前 `Probe` 部分中没有匹配项时应作为回退使用的探测。有关回退的更多信息，请参见[“作弊和回退”](vscan-technique.html#vscan-cheats-and-fallbacks)一节。对于没有回退指令的 TCP 探测，Nmap 首先尝试探测本身的匹配行，然后隐式回退到 NULL 探测。如果存在回退指令，Nmap 首先尝试探测本身的匹配行，然后是回退指令中指定的探测（从左到右）的匹配行。最后，Nmap 会尝试 NULL 探测。对于 UDP，行为相同，只是从不尝试 NULL 探测。

### 综合示例

以下是来自 `nmap-service-probes` 的一些示例，它们将所有这些综合在一起（为节省空间跳过了许多行）。在读完本节之后，以下内容应该能够理解。

```plaintext
# Exclude 指令使用逗号分隔的端口列表。
# 格式与 -p 开关完全相同。
Exclude T:9100-9107

# 这是 NULL 探测，仅比较给予我们的任何标语
##############################NEXT PROBE##############################
Probe TCP NULL q||
# 等待至少 5 秒钟的数据。否则使用 Nmap 默认值。
totalwaitms 5000
# Windows 2003
match ftp m/^220[ -]Microsoft FTP Service\r\n/ p/Microsoft ftpd/
match ftp m/^220 ProFTPD (\d\S+) Server/ p/ProFTPD/ v/$1/
softmatch ftp m/^220 [-.\w ]+ftp.*\r\n$/i
match ident m|^flock\(\) on closed filehandle .*midentd| p/midentd/ i/broken/
match imap m|^\* OK Welcome to Binc IMAP v(\d[-.\w]+)| p/Binc IMAPd/ v$1/
softmatch imap m/^\* OK [-.\w ]+imap[-.\w ]+\r\n$/i
match lucent-fwadm m|^0001;2$| p/Lucent Secure Management Server/
match meetingmaker m/^\xc1,$/ p/Meeting Maker calendaring/
# Linux 1.1 上的 lopster 1.2.0.1
match napster m|^1$| p/Lopster Napster P2P client/

Probe UDP Help q|help\r\n\r\n|
rarity 3
ports 7,13,37
match chargen m|@ABCDEFGHIJKLMNOPQRSTUVWXYZ|
match echo m|^help\r\n\r\n$|
