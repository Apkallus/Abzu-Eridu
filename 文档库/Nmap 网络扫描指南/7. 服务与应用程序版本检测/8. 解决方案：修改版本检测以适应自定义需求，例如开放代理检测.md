## 解决方案：修改版本检测以适应自定义需求，例如开放代理检测

### 问题

保护任何网络安全的一个重要部分是识别危险主机。Nmap 的服务检测系统是一种灵活、可靠的方法。它可以帮助识别易受攻击的软件版本、发现配置错误的服务器等等。但有时，以标准版本扫描不敢尝试的方式实际滥用服务，是确定它们是否真正易受攻击的最佳方法。

开放代理是那些会盲目地将来自不受信任主机的请求中继到它们自己选择的服务器上的服务器。在网络内部运行这些代理可能极其危险，原因有很多，因为攻击者可以：

- 发起看似来自您网络的攻击
- 窃取您的带宽或其他网络服务
- 冒充内部客户端，以在您的组织内进一步升级权限

这为修改版本检测以专门尝试利用开放代理提供了很好的动机。我们可能可以通过 Nmap 的正常代理匹配行来映射哪些端口是代理，但证明应用程序易受攻击的最佳且唯一真正的方法是亲自利用它。

> **注意**：此解决方案由 Nmap 开发者 Doug Hoyte 贡献。

### 解决方案

我们首先要做的是复制 `nmap-service-probes` 文件，以便在临时副本上工作：

```screen
mkdir ~/proxydetect
cp /usr/local/share/nmap/nmap-service-probes ~/proxydetect
```

接下来，我们想临时强制 Nmap 使用我们的临时文件：

```screen
export NMAPDIR=$HOME/proxydetect
```

现在我们需要向文件中添加一个探测和匹配行，因此打开您最喜欢的编辑器，并将以下文本放入您复制的 `nmap-service-probes` 中。一个好的放置位置是在 NULL 探测的所有匹配行之后，但在下一个 Probe 行（GenericLines）之前。

```plaintext
Probe TCP ProxyProbe q|GET https://insecure.org/ HTTP/1.1\r\nHost: insecure.org\r\n\r\n|
rarity 1
ports 1-65535
totalwaitms 20000
match proxy m|^HTTP/1.[01] 200 OK\r?\n.*TITLE&gt;Insecure.O|s p/Open HTTP Proxy!!/
```

现在，`Nmap` 实际上会尝试通过将任何扫描到的端口视为代理来从 `insecure.org` 请求一个 HTTP 下载。我们将在包含开放代理的网络扫描中开始看到以下内容：

```screen
PORT   STATE SERVICE VERSION
80/tcp open  proxy   Open HTTP Proxy!!
```

### 讨论

我们的探测位置、较低的 `rarity` 值以及广泛的 `ports` 范围有助于确保我们的自定义探测在服务扫描中很快被尝试，这样其他探测（如 GetRequest）就不会在我们有机会使用主动探测之前简单地将此识别为代理。

我们还使用了 `totalwaitms` 指令使 `Nmap` 等待更长时间以使此探测超时。这可能是必要的，因为我们不仅要处理我们与代理之间连接的延迟和不可靠性，还要处理代理与我们请求的页面所在服务器（`insecure.org`）之间连接的延迟和不可靠性。

请记住，除了 HTTP 之外，许多其他协议也可以被代理。版本检测将识别其中许多协议的代理，包括 FTP、POP3、IMAP 和 SMTP。SOCKS 代理有特殊的匹配行，可以确定代理配置的身份验证选项信息。正如我们在本解决方案中所做的那样，我们通常可以使用版本检测，通过自定义探测文件来判断此类代理是否开放。然而，更复杂的测试最好使用 NSE 脚本完成。
