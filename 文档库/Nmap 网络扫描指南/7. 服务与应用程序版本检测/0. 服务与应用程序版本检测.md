## 引言

虽然 Nmap 能做很多事情，但其最核心的功能是端口扫描。将 Nmap 指向一台远程机器，它可能会告诉你端口 `25/tcp`、`80/tcp` 和 `53/udp` 是开放的。使用其包含 2200 多个众所周知服务的 `nmap-services` 数据库，Nmap 会报告这些端口可能分别对应邮件服务器（SMTP）、Web 服务器（HTTP）和域名服务器（DNS）。这种查找通常是准确的——事实上，绝大多数监听在 TCP 25 端口上的守护进程都是邮件服务器。然而，你不应该把安全押在这上面！人们可以并且确实会在奇怪的端口上运行服务。也许他们的主 Web 服务器已经在 80 端口上，所以他们为临时或测试服务器选择了另一个端口。也许他们认为将易受攻击的服务隐藏在某个 obscure 端口上可以防止“邪恶黑客”找到它。最近更常见的是，人们选择端口不是基于他们想要运行的服务，而是基于什么能穿过防火墙。当 ISP 在微软 IIS 蠕虫 CodeRed 和 Nimda 爆发后封锁了 80 端口时，大量用户将他们的个人 Web 服务器转移到其他端口作为回应。当公司因 Telnet 存在严重安全风险而封锁它时，我见过用户干脆在 Secure Shell（SSH）端口上运行 telnetd。

即使 Nmap 是正确的，上述假设的服务器运行的是 SMTP、HTTP 和 DNS 服务，那也不是很多信息。在对您的公司或客户进行漏洞评估（甚至简单的网络清单）时，您真正想知道的是运行的是哪个邮件服务器、哪个 DNS 服务器以及它们的版本。拥有准确的版本号有助于极大地确定服务器易受哪些漏洞攻击。请记住，安全修复通常会向后移植到软件的早期版本，因此您不能仅依靠版本号来证明服务存在漏洞。误报较少见，但可能发生在愚蠢的管理员伪造易受攻击服务的版本号以使其看起来已打补丁时。

确定服务类型和版本号的另一个好理由是许多服务共享相同的端口号。例如，端口 `258/tcp` 被 Checkpoint Firewall-1 GUI 管理界面和 `yak` Windows 聊天客户端同时使用。这使得基于 `nmap-services` 表的猜测更加不准确。任何做过大量扫描的人都知道，您也经常会在未注册的端口上发现监听服务——如果没有版本检测，这些服务就完全是个谜。最后一个问题是，对于简单的端口扫描器来说，被过滤的 UDP 端口通常看起来与开放端口相同（参见 [“UDP 扫描（`-sU`）”](scan-methods-udp-scan.html)）。但如果它们响应 Nmap 版本检测发送的服务特定探测，那么您就可以确信它们是开放的（并且通常确切知道运行的是什么）。

服务扫描有时会揭示超出服务类型和版本号的目标信息。关于服务的杂项信息收集在“info”字段中。该字段在 `VERSION` 列中显示，位于产品名称和版本号后面的括号内。该字段可以包含 SSH 协议号、Apache 模块等等。

有些服务还会报告它们配置的主机名，这些主机名与机器的反向 DNS 主机名往往惊人地不同。主机名字段在端口表之后的 `Service Info` 行中报告。这听起来像是次要的信息泄露，但可能会产生后果。在 CanSecWest 安全会议的一年，我正蜷缩在房间里的笔记本电脑前。突然，屏幕角落的 `tcpdump` 窗口疯狂滚动起来，我意识到我的机器正受到攻击。我反扫描回去，发现一个不寻常的高位端口处于开放状态。连接后，该端口吐出一堆二进制字符，但输出中的一个 ASCII 字段给出了一个配置的域名。这个域名属于一家足够小的安全公司，我确切知道是谁干的。我让前台打电话到他酒店房间，当我请他停止探测我的机器时，他非常惊讶。

版本检测可以发现的另外两个字段是操作系统和设备类型。它们也在 `Service Info` 行中报告。我们在这里使用两种技术。一种是应用程序独占性。如果我们将服务识别为 Microsoft Exchange，我们就知道操作系统是 Windows，因为 Exchange 不在其他任何系统上运行。另一种技术是说服更多可移植的应用程序泄露平台信息。许多服务器（尤其是 Web 服务器）只需很少的诱导。这种类型的操作系统检测旨在补充 Nmap 的操作系统检测系统（`-O`），并且有时会报告不同的结果。例如，考虑一个隐藏在端口转发 Unix 防火墙后面的 Microsoft Exchange 服务器。

Nmap 版本扫描子系统通过连接到开放端口并使用特定服务理解的探测来询问它们以获取更多信息，从而获得所有这些数据。这使得 Nmap 能够详细评估真正在运行什么，而不仅仅是哪些端口号是开放的。[示例 7.1](#ex-version-detection-scan1) 显示了实际输出。

**示例 7.1 版本检测的简单用法**

```screen
# nmap -sV -T4 -F insecure.org

Host is up (0.016s latency).
rDNS record for 74.207.254.18: web.insecure.org
Not shown: 95 filtered ports
PORT    STATE  SERVICE  VERSION
22/tcp  open   ssh      OpenSSH 4.3 (protocol 2.0)
25/tcp  open   smtp     Postfix smtpd
80/tcp  open   http     Apache httpd 2.2.3 ((CentOS))

Nmap done: 1 IP address (1 host up) scanned in 14.82 seconds
```

Nmap 版本检测提供以下高级功能（后面会完整描述）：

- 通过非阻塞套接字和专为高效而强大的实现设计的探测/匹配定义语法，实现高速并行操作。
- 在可用时确定应用程序名称和版本号——而不仅仅是服务协议。
- 支持 TCP 和 UDP 协议，以及文本 ASCII 和打包的二进制服务。
- 多平台支持，包括 Linux、Windows、Mac OS X、FreeBSD/NetBSD/OpenBSD、Solaris 以及 Nmap 已知能工作的所有其他平台。
- 如果检测到 SSL，Nmap 会使用 OpenSSL（如果可用）进行连接，并尝试确定在该加密层后面监听的是什么服务。这使它能够发现像 HTTPS、POP3S、IMAPS 等服务，并提供版本细节。
- 如果发现 SunRPC 服务，Nmap 会启动其暴力 RPC 磨床来查找程序号、名称和版本号。
- 支持 IPv6，包括 TCP、UDP 和基于 TCP 的 SSL。
- 通用平台枚举（CPE）输出，用于与其他软件互操作（某些信息仅包含在 XML 输出中）。关于 CPE 的更多信息，请参见 [“通用平台枚举（CPE）”](output-formats-cpe.html) 一节。
- 社区贡献：如果 Nmap 从服务收到无法识别的数据，则会打印一个 **服务指纹** 以及一个提交 URL。该系统模仿了极其成功的 Nmap 操作系统检测指纹提交流程。也可以提交新的探测和修正。
- 综合数据库：Nmap 识别超过一千个服务签名，涵盖从 ACAP、AFP、AIM 到 XML-RPC、Zebedee 和 Zebra 的 180 多种独特服务协议。
