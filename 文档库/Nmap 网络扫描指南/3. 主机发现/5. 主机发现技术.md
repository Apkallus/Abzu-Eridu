## 主机发现技术

曾几何时，判断一个 IP 地址是否对应活跃主机非常容易：只需发送一个 ICMP 回显请求（ping）数据包并等待响应即可。防火墙很少会阻止这些请求，绝大多数主机也会乖乖响应。自 1989 年起，RFC 1122 就要求“每个主机都必须实现 ICMP 回显服务器功能，接收回显请求并发送相应的回显应答”。

不幸的是，对于网络探索者来说，许多管理员认为安全问题比 RFC 要求更重要，因此屏蔽了 ICMP ping 消息。示例 3.8 使用仅 ICMP 的 Nmap ping 扫描对六个流行网站进行测试，但只收到两个响应。这表明，不能再因为主机没有响应 ICMP ping 探测就假定它不可用。示例中的 `-sn -PE` 选项指定了仅 ICMP 的 ping 扫描。`-R` 选项告诉 Nmap 对所有主机（即使是关机的）执行反向 DNS 解析。

**示例 3.8 尝试 ping 流行的互联网主机**

```screen
# nmap -sn -PE -R -v microsoft.com ebay.com citibank.com google.com \
                     slashdot.org yahoo.com

Starting Nmap ( https://nmap.org )
Host origin2.microsoft.com (207.46.250.252) appears to be down.
Host pages.ebay.com (66.135.192.87) appears to be down.
Host ld1-www.citicorp.com (192.193.195.132) appears to be down.
Host 216.239.57.99 appears to be up.
Host slashdot.org (66.35.250.150) appears to be down.
Host w3.rc.dcn.yahoo.com (216.109.127.30) appears to be up.
Nmap done: 6 IP addresses (2 hosts up) scanned in 3.76 seconds
```

幸运的是，Nmap 提供了标准 ICMP 回显请求之外的多种主机发现技术。下面将逐一介绍。请注意，如果你使用了本节讨论的任何 `-P` 选项，它们将 **替换** 默认的发现探测，而不是附加到默认探测上。

### TCP SYN Ping（`-PS<port list>`）

`-PS` 选项发送一个设置了 SYN 标志的空 TCP 数据包。默认目标端口是 80（编译时可通过修改 `nmap.h` 中的 `DEFAULT_TCP_PROBE_PORT_SPEC` 来配置），但也可以指定其他端口。可以指定一个端口列表（例如 `-PS22-25,80,113,1050,35000`），此时将并行向每个端口发送探测。

SYN 标志暗示远程系统你正在尝试建立连接。通常目标端口是关闭的，会返回一个 RST（复位）数据包。如果端口恰好开放，目标将执行 TCP 三次握手的第二步，用 SYN/ACK TCP 数据包响应。然后运行 Nmap 的机器不会发送完成三次握手并建立完整连接的 ACK 数据包，而是用 RST 数据包拆除这个刚建立的连接。[10]

Nmap 不关心端口是开放还是关闭。无论是前面讨论的 RST 还是 SYN/ACK 响应，都告诉 Nmap 该主机是存活的并且有响应。

在 Unix 系统上，通常只有特权用户 `root` 才能发送和接收原始 TCP 数据包。对于非特权用户，会自动采用一种变通方法：对每个目标端口发起 `connect` 系统调用。这实际上向目标主机发送了一个 SYN 数据包，试图建立连接。如果 `connect` 快速返回成功或 ECONNREFUSED 失败，说明底层 TCP 栈一定收到了 SYN/ACK 或 RST，因此将该主机标记为可用。如果连接尝试一直挂起直到超时，则主机被标记为关闭。这种变通方法也用于 IPv6 连接，因为 Nmap 目前还不支持构建原始 IPv6 数据包。

示例 3.8 中，由于目标没有响应 ICMP 回显请求，六台机器中有四台未能检测到。使用端口 80 的 SYN 探测重复实验，则全部六台都有响应，如示例 3.9 所示。

**示例 3.9 使用端口 80 SYN 探测重试主机发现**

```screen
# nmap -sn -PS80 -R -v microsoft.com ebay.com citibank.com google.com \
                       slashdot.org yahoo.com

Starting Nmap ( https://nmap.org )
Host origin2.microsoft.com (207.46.249.252) appears to be up.
Host pages.ebay.com (66.135.192.87) appears to be up.
Host ld1-www.citicorp.com (192.193.195.132) appears to be up.
Host 216.239.57.99 appears to be up.
Host slashdot.org (66.35.250.150) appears to be up.
Host w3.rc.dcn.yahoo.com (216.109.127.30) appears to be up.
Nmap done: 6 IP addresses (6 hosts up) scanned in 0.48 seconds
```

第二次运行不仅检测到了所有六台机器，而且速度更快（不到半秒），因为机器被并行扫描，且扫描从未因等待响应而超时。这个测试不完全公平，因为这些机器都是流行的 Web 服务器，可以预期它们监听 TCP 80 端口。但它仍然说明了一个要点：不同类型的主机响应不同类型的探测。Nmap 支持并行使用多种扫描类型，以便有效扫描多样化的网络。

### TCP ACK Ping（`-PA<port list>`）

TCP ACK ping 与 SYN ping 非常相似。区别正如你猜测的那样：设置的是 TCP ACK 标志而非 SYN 标志。这样的 ACK 数据包声称是对已建立 TCP 连接上的数据进行确认，但实际并不存在这样的连接。因此远程主机应该总是以 RST 数据包响应，从而在这个过程中暴露自己的存在。

`-PA` 选项使用与 SYN 探测相同的默认端口（80），也可以采用相同格式指定目标端口列表。如果非特权用户尝试此选项，或指定了 IPv6 目标，则会使用之前讨论的 `connect` 变通方法。这种变通方法并不完美，因为 `connect` 实际发送的是 SYN 数据包而不是 ACK。

提供 SYN 和 ACK 两种 ping 探测的原因是为了最大程度地绕过防火墙。许多管理员配置路由器或其他简单防火墙来阻止入站 SYN 数据包，除非它们去往公司的公共服务（如 Web 服务器或邮件服务器）。这种无状态方法占用防火墙/路由器的资源很少，并且被硬件和软件过滤器广泛支持。作为这种方法普遍性的一个例子，Linux Netfilter/iptables 防火墙软件提供了 `--syn` 便利选项，其手册页描述如下：

> 仅匹配设置了 SYN 位且清除了 ACK 和 RST 位的 TCP 数据包。此类数据包用于请求 TCP 连接初始化；例如，在接口上阻止此类数据包将阻止入站 TCP 连接，但出站 TCP 连接不受影响。它等价于 `--tcp-flags SYN,RST,ACK SYN`。

当存在此类防火墙规则时，发送到关闭目标端口的 SYN ping 探测（`-PS`）很可能会被阻止。在这种情况下，ACK 探测能够直接穿透这些规则。

另一种常见的防火墙类型使用状态规则，丢弃意外的数据包。这种功能最初主要出现在高端防火墙上，但多年来已变得普遍得多。Linux Netfilter/iptables 系统通过 `--state` 选项支持此功能，根据连接状态对数据包进行分类，如下手册摘录所述：

> 可能的状态包括：INVALID（数据包与任何已知连接无关）、ESTABLISHED（数据包与已在两个方向有数据包流过的连接相关）、NEW（数据包已启动新连接，或与尚未在两个方向有数据包流过的连接相关）、RELATED（数据包正在启动新连接，但与现有连接相关，如 FTP 数据传输或 ICMP 错误）。

ACK 探测不太可能绕过采用这种方法的防火墙，因为这种意外数据包将被归类为 INVALID 状态，很可能被丢弃。示例 3.10 显示了尝试对 Microsoft 进行 ACK ping 的结果。它们的状态防火墙丢弃了数据包，导致 Nmap 错误地认为主机已关闭。在这种情况下，SYN 探测成功的机会大得多。

这就引出一个问题：当目标网络的防火墙规则未知或不一致时，应该使用哪种技术？正确的答案通常是“两者都用”。Nmap 可以并行向多个端口发送 SYN 和 ACK 探测，同时执行其他主机发现技术。这将在“综合运用：主机发现策略”一节中进一步讨论。

**示例 3.10 尝试对 Microsoft 进行 ACK ping**

```screen
# nmap -sn -PA www.microsoft.com

Starting Nmap ( https://nmap.org )
Warning: Hostname www.microsoft.com resolves to 5 IPs. Using 207.46.192.254.
Note: Host seems down. If it is really up, but blocking ping probes, try -Pn
Nmap done: 1 IP address (0 hosts up) scanned in 2.22 seconds
```

### UDP Ping（`-PU<port list>`）

另一个主机发现选项是 UDP ping，它向指定端口发送 UDP 数据包。端口列表的格式与前面讨论的 `-PS` 和 `-PA` 选项相同。如果未指定端口，默认值为 40,125。这个默认值可以在编译时通过修改 `nmap.h` 中的 `DEFAULT_UDP_PROBE_PORT_SPEC` 来配置。之所以默认使用一个非常不常见的端口，是因为对于这种特定的扫描类型，发送到开放端口通常是不希望的。

对于大多数端口，数据包是空的；但对于少数常见端口（如 53 和 161），会发送特定协议的有效载荷，这样更有可能得到响应。`--data-length` 选项为所有端口发送固定长度的随机有效载荷。

当命中目标机器上的关闭端口时，UDP 探测应引发一个 ICMP 端口不可达数据包作为返回。这向 Nmap 表明机器处于活动状态且可用。其他许多类型的 ICMP 错误（如主机/网络不可达或 TTL 超时）表示主机已关闭或不可达。没有响应也同样解释为不可达。如果命中开放端口，大多数服务只是忽略空数据包而不返回任何响应。这就是为什么默认探测端口是 40,125——它极不可能被使用。少数服务（如字符生成协议 chargen）会响应空 UDP 数据包，从而向 Nmap 透露机器可用。对于具有自定义有效载荷的端口，探测更有可能获得响应。

此扫描类型的主要优点是它能够绕过仅过滤 TCP 的防火墙和过滤器。例如，我曾经有一个 Linksys BEFW11S4 无线路由器。该设备的外部接口默认过滤所有 TCP 端口，但 UDP 探测仍然会引发端口不可达消息，从而暴露设备的存在。

### ICMP Ping 类型（`-PE`、`-PP` 和 `-PM`）

除了前面讨论的不常见的 TCP 和 UDP 主机发现类型，Nmap 还可以发送无处不在的 `ping` 程序所使用的标准数据包。Nmap 向目标 IP 地址发送 ICMP 类型 8（回显请求）数据包，期望从可用主机返回类型 0（回显应答）。正如本章开头所述，许多主机和防火墙现在会阻止这些数据包，而不是按照 RFC 1122 的要求进行响应。因此，仅靠 ICMP 扫描对互联网上未知目标的可靠性通常不足。但对于监控内部网络的系统管理员来说，这是一种实用且高效的方法。使用 `-PE` 选项启用此回显请求行为。

虽然回显请求是标准的 ICMP ping 查询，但 Nmap 并不止于此。ICMP 标准（RFC 792 和 RFC 950）还规定了时间戳请求（代码 13）、信息请求（代码 15）和地址掩码请求（代码 17）数据包。尽管这些查询的表面目的是获取地址掩码和当前时间等信息，但它们可以很容易地用于主机发现。Nmap 目前未实现信息请求数据包，因为它们未被广泛支持（RFC 1122 声明“主机不应实现这些消息”）。时间戳查询和地址掩码查询可以分别使用 `-PP` 和 `-PM` 选项发送。时间戳应答（ICMP 代码 14）或地址掩码应答（代码 18）表明主机可用。当管理员专门屏蔽了回显请求数据包，却忘记其他 ICMP 查询也可用于相同目的时，这两种查询就非常有价值。

### IP 协议 Ping（`-PO<protocol list>`）

最新的主机发现选项是 IP 协议 ping，它发送 IP 数据包，并在其 IP 报头中设置指定的协议号。协议列表的格式与前面讨论的 TCP 和 UDP 主机发现选项中的端口列表相同。如果未指定协议，默认发送多个 IP 数据包，对应 ICMP（协议 1）、IGMP（协议 2）和 IP-in-IP（协议 4）。默认协议可以在编译时通过修改 `nmap.h` 中的 `DEFAULT_PROTO_PROBE_PORT_SPEC` 来配置。注意，对于 ICMP、IGMP、TCP（协议 6）和 UDP（协议 17），数据包会带有正确的协议报头；而对于其他协议，除了 IP 报头外不发送额外数据（除非指定了 `--data-length` 选项）。

此主机发现方法寻找两种响应：一是与探测使用相同协议的响应，二是 ICMP 协议不可达消息（表示目标主机不支持给定的协议）。任何一种类型的响应都表明目标主机处于活跃状态。

### ARP 扫描（`-PR`）

最常见的 Nmap 使用场景之一是扫描以太网 LAN。在大多数 LAN 上，尤其是那些使用 RFC 1918 私有地址范围的 LAN，绝大多数 IP 地址在任何给定时间都未被使用。当 Nmap 尝试发送原始 IP 数据包（如 ICMP 回显请求）时，操作系统必须确定目标 IP 对应的目的地硬件（ARP）地址，以便正确构造以太网帧。这需要它发出一系列 ARP 请求。示例 3.11 显示了这一点：尝试对本地以太网主机进行 ping 扫描。`--send-ip` 选项告诉 Nmap 即使在本地网络上也发送 IP 层数据包（而不是原始以太网）。Wireshark 输出的三个 ARP 请求及其时序被粘贴到会话中。

**示例 3.11 对离线目标进行原始 IP ping 扫描**

```screen
# nmap -n -sn --send-ip 192.168.33.37

Starting Nmap ( https://nmap.org )
  0.000000 00:01:29:f5:27:f2 -> ff:ff:ff:ff:ff:ff ARP Who has 192.168.33.37? 
  0.999836 00:01:29:f5:27:f2 -> ff:ff:ff:ff:ff:ff ARP Who has 192.168.33.37?
  1.999684 00:01:29:f5:27:f2 -> ff:ff:ff:ff:ff:ff ARP Who has 192.168.33.37?
Note: Host seems down. If it is really up, but blocking ping probes, try -Pn
Nmap done: 1 IP address (0 hosts up) scanned in 2.04 seconds
```

这个例子用了两秒多才完成，因为（Linux）操作系统在放弃该主机之前发送了三个 ARP 请求，间隔一秒。鉴于 ARP 应答通常在几毫秒内到达，等待数秒是过度的。操作系统供应商并不优先考虑缩短这个超时时间，因为绝大多数数据包都是发送给实际存在的主机。另一方面，当给定像 10.0.0.0/8 这样的目标时，Nmap 必须向 1600 万个 IP 发送数据包。即使许多目标被并行 ping，每个目标等待两秒也会造成巨大的延迟。

在 LAN 上进行原始 IP ping 扫描还有另一个问题。当发现目标主机无响应（如前面的例子）时，源主机通常会在其内核 ARP 表中为该目标 IP 添加一个不完整的条目。ARP 表空间是有限的，当它填满时，某些操作系统反应不佳。当 Nmap 以原始 IP 模式（`--send-ip`）使用时，有时必须等待几分钟让 ARP 缓存条目过期，才能继续主机发现。

ARP 扫描通过让 Nmap 掌握控制权来解决了这两个问题。Nmap 自行发出原始 ARP 请求并处理重传和超时时间，绕过了系统 ARP 缓存。示例 3.12 显示了不同之处。这个 ARP 扫描所花时间仅为 IP 版本扫描时间的十分之一多一点。

**示例 3.12 对离线目标进行 ARP ping 扫描**

```screen
# nmap -n -sn -PR --packet-trace --send-eth 192.168.33.37

Starting Nmap ( https://nmap.org )
SENT (0.0060s) ARP who-has 192.168.33.37 tell 192.168.0.100
SENT (0.1180s) ARP who-has 192.168.33.37 tell 192.168.0.100
Note: Host seems down. If it is really up, but blocking ping probes, try -Pn
Nmap done: 1 IP address (0 hosts up) scanned in 0.23 seconds
```

在示例 3.12 中，`-PR` 或 `--send-eth` 选项实际上没有产生任何效果，因为当扫描 Nmap 检测到位于本地以太网网络上的以太网主机时，ARP 是默认的扫描类型。这包括传统的有线以太网以及 802.11 无线网络。ARP 扫描不仅如上所述更高效，而且更准确。主机经常阻塞基于 IP 的 ping 数据包，但它们通常无法在不影响网络通信的情况下阻塞 ARP 请求或响应。即使指定了不同的 ping 类型（例如 `-PE` 或 `-PS`），Nmap 也会对同一 LAN 上的任何目标改用 ARP。如果你绝对不想进行 ARP 扫描，可以像示例 3.11 那样指定 `--send-ip` 选项。

让 Nmap 控制发送原始以太网帧还允许 Nmap 控制源 MAC 地址。如果你在安全会议上成为房间里唯一拥有 PowerBook 的人，并且从注册给 Apple 的 MAC 地址发起大规模 ARP 扫描，所有人的目光可能会转向你。你可以使用 `--spoof-mac` 选项伪造 MAC 地址，这在“MAC 地址欺骗”一节中讨论。

### 默认组合

如果未选择任何主机发现技术，Nmap 会使用一个默认组合。对于 Windows 或特权（root）Unix 用户，该组合等效于 `-PE -PS443 -PA80 -PP`。细心的读者知道，这意味着向每台机器发送一个 ICMP 回显请求、一个 TCP SYN 数据包、一个 TCP ACK 数据包和一个 ICMP 时间戳请求。例外情况是：对于位于本地以太网网络上的任何目标，将使用 ARP 扫描。对于非特权的 Unix shell 用户，默认等效于 `-PS80,443`（对目标主机的 80 和 443 端口发起 TCP `connect` 调用）。对于安全审计，我建议使用更全面的 ping 类型集合，例如“设计理想的探测组合”一节中讨论的那些。
