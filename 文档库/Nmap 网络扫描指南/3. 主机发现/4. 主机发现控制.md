## 主机发现控制

默认情况下，Nmap 会在端口扫描、操作系统检测、Nmap 脚本引擎或版本检测等更具侵入性的探测之前包含一个 ping 扫描阶段。Nmap 通常仅对在 ping 扫描阶段显示为可用的机器执行侵入性扫描。与针对每一个 IP 地址进行全面扫描相比，这节省了大量时间和带宽。然而，这种方法并非适用于所有情况。有时你确实想扫描每一个 IP（`-Pn`），有时你只想执行主机发现而不进行端口扫描（`-sn`）。甚至有时你只想打印出目标主机并在发送任何 ping 探测之前退出（`-sL`）。Nmap 提供了几个高级选项来控制此行为。

### 列表扫描（`-sL`）

列表扫描是主机发现的一种退化形式，它仅列出指定网络上的每个主机，而不向目标主机发送任何数据包。默认情况下，Nmap 仍然对主机执行反向 DNS 解析以获取其名称。Nmap 还会在最后报告 IP 地址的总数。列表扫描是一个很好的合理性检查，可确保你拥有正确的目标 IP 地址。如果主机显示了你无法识别的域名，则值得进一步调查，以防止扫描错误公司的网络。

目标 IP 范围可能出错的原因有很多。即使是网络管理员也可能打错自己的网络块，而渗透测试人员有更多需要担心的地方。在某些情况下，安全顾问被给予了错误的地址。在其他情况下，他们尝试通过 whois 数据库和路由表等资源找到正确的 IP 范围。这些数据库可能已过时，或者公司可能将 IP 空间借给其他组织。是否扫描母公司、兄弟公司、服务提供商和子公司是一个重要问题，应事先与客户协商确定。初步的列表扫描有助于确认正在扫描的确切目标。

提前进行列表扫描的另一个原因是隐蔽性。在某些情况下，你不想以大规模攻击目标网络开始，这可能会触发 IDS 警报并引起不必要的注意。列表扫描不引人注目，并提供可能有助于选择要定位的单个机器的信息。目标注意到所有反向 DNS 请求是可能的，尽管可能性极低。如果担心这一点，你可以使用 `--dns-servers` 选项通过匿名递归 DNS 服务器中继请求。

列表扫描通过命令行选项 `-sL` 指定。由于其目的仅仅是打印目标主机列表，因此不能将更高级别的功能选项（如端口扫描、操作系统检测或 ping 扫描）与 `-sL` 结合使用。如果你希望禁用 ping 扫描但仍执行此类更高级别的功能，请阅读 `-Pn` 选项。示例 3.6 展示了如何使用列表扫描枚举斯坦福大学主 Web 服务器周围的 CIDR /28 网络范围（16 个 IP 地址）。

### 禁用端口扫描（`-sn`）

此选项告诉 Nmap 在主机发现后不运行端口扫描。单独使用时，它使 Nmap 执行主机发现，然后打印出响应该扫描的可用主机。这通常被称为“ping 扫描”。即使没有进行端口扫描，你仍然可以请求 Nmap 脚本引擎（`--script`）主机脚本和 traceroute 探测（`--traceroute`）。仅 ping 扫描比列表扫描更具侵入性，并且通常可用于相同的目的。它可以快速、不引人注目地对目标网络进行轻量级侦察。对于攻击者来说，知道有多少主机在线比列表扫描提供的每一个 IP 和主机名的列表更有价值。

系统管理员也常常发现此选项很有价值。它可以轻松地用于计算网络上的可用机器或监控服务器可用性。这通常称为 ping 扫荡，并且比 ping 广播地址更可靠，因为许多主机不响应广播查询。

示例 3.7 显示了对我最喜欢的网站之一 Linux Weekly News 周围的 CIDR /24（256 个 IP）进行的快速 ping 扫描。

此示例仅用了 13 秒，但提供了有价值的信息。在那个 C 类大小的地址范围内，有 105 个主机在线。从挤在这么小 IP 空间中的不相关域名可以清楚地看出，LWN 使用了托管或专用服务器提供商。如果 LWN 的机器被证明是高度安全的，攻击者可能会攻击那些邻近的机器之一，然后使用诸如 Ettercap 或 Dsniff 之类的工具执行本地以太网攻击。对这些数据的道德使用是：正在考虑将机器迁移到此提供商的网络管理员可能会给列出的几个组织发送电子邮件，询问他们对服务的意见，然后再签署长期合同或进行昂贵且具有破坏性的数据中心迁移。

默认情况下，`-sn` 选项发送一个 ICMP 回显请求、一个到端口 443 的 TCP SYN 数据包、一个到端口 80 的 TCP ACK 数据包以及一个 ICMP 时间戳请求。由于非特权 Unix 用户（或未安装 Npcap 的 Windows 用户）无法发送这些原始数据包，因此在那些情况下只发送 SYN 数据包。SYN 数据包通过 TCP `connect` 系统调用发送到目标主机的端口 80 和 443。当特权用户尝试扫描本地以太网上的目标时，除非指定了 `--send-ip` 选项，否则会使用 ARP 请求（`-PR`）。

`-sn` 选项可以与“主机发现技术”一节中讨论的任何技术结合使用，以获得更大的灵活性。如果使用了任何那些探测类型和端口号选项，则会覆盖默认探测。当运行 Nmap 的源主机与目标网络之间存在严格的防火墙时，建议使用那些高级技术。否则，当防火墙丢弃探测或其响应时，可能会错过主机。

### 禁用 Ping（`-Pn`）

另一个选项是完全跳过 Nmap 的发现阶段。通常，Nmap 使用此阶段确定活跃机器以进行更重量级的扫描。默认情况下，Nmap 仅对发现为在线的主机执行重量级探测，例如端口扫描、版本检测或操作系统检测。使用 `-Pn` 选项禁用主机发现会导致 Nmap 对指定的 **每一个** 目标 IP 地址尝试请求的扫描功能。因此，如果在命令行上指定了 B 类大小的目标地址空间（/16），则会扫描所有 65,536 个 IP 地址。与列表扫描一样，会跳过正常的主机发现，但 Nmap 不会停止并打印目标列表，而是继续执行请求的功能，就好像每个目标 IP 都处于活动状态一样。

禁用 Nmap ping 测试的原因有很多。最常见的原因之一是侵入性漏洞评估。人们可以指定几十种不同的 ping 探测，试图从所有可用主机获得响应，但一个处于活动状态但受到严格防火墙保护的机器仍然可能不响应任何这些探测。因此，为了避免遗漏任何内容，审计员经常对目标网络上的每个 IP 进行高强度扫描，例如扫描所有 65,536 个 TCP 端口。向可能没有主机监听的 IP 地址发送数十万个数据包似乎是浪费，并且可能使扫描时间减慢一个数量级或更多。Nmap 必须向每个端口发送重传，以防原始探测在传输过程中丢失，并且 Nmap 必须花费大量时间等待响应，因为它对这些无响应的 IP 地址没有往返时间估计。但严肃的渗透测试人员愿意付出这个代价，以避免哪怕是一点点错过活跃机器的风险。他们也可以同时进行快速扫描，让大规模的 `-Pn` 扫描在后台运行，同时进行其他工作。第 6 章《优化 Nmap 性能》提供了更多性能调优建议。

使用 `-Pn` 的另一个常见原因是测试人员已经有一个已知在线的机器列表。因此用户认为浪费时间进行主机发现阶段没有意义。用户创建自己的活动主机列表，然后使用 `-iL`（从列表输入）选项将其传递给 Nmap。从节省时间的角度来看，这种策略很少有益。由于上一段讨论的重传和 RTT 估计问题，即使一个大列表中的一个无响应的 IP 地址，通常也会比整个 ping 扫描阶段花费更多的时间来扫描。此外，ping 阶段允许 Nmap 收集 RTT 样本，这可以加速后续的端口扫描，特别是当目标主机有严格的防火墙规则时。虽然指定 `-Pn` 作为节省时间的方法很少有帮助，但如果你的列表中的某些机器阻塞了所有否则会指定的发现技术，那么 `-Pn` 就很重要。用户必须在扫描速度和可能错过高度隐藏的机器之间取得平衡。
