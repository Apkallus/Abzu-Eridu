## 综合运用：主机发现策略

### 相关选项

前面的章节描述了用于控制 Nmap 主机发现阶段和自定义使用技术的主要选项。不过，这里还有许多通用的 Nmap 选项与此相关。本节简要介绍这些选项标志与 ping 扫描的关系。有关每个选项的完整说明，请参见第 15 章《Nmap 参考指南》。

- **`-v`（同 `--verbose`）**  
  默认情况下，Nmap 通常只打印活跃的、有响应的主机。详细模式会使 Nmap 打印出关闭的主机，以及有关活跃主机的额外信息。

- **`--source-port <portnum>`（同 `-g`）**  
  设置固定的源端口对 ping 扫描（TCP 和 UDP）的作用与对 Nmap 其他功能的作用相同。一些天真的防火墙管理员为了让 DNS（端口 53）或 FTP-DATA（端口 20）正常工作而破例设置了规则集。当然，这打开了一个足够让 Nmap ping 扫描穿过的漏洞。“源端口操纵”一节提供了有关此技术的更多细节。

- **`-n`、`-R`**  
  `-n` 选项禁用所有 DNS 解析，而 `-R` 选项对所有主机（即使是关闭的）启用 DNS 查询。默认行为是将 DNS 解析限制在活跃主机上。这些选项对 ping 扫描尤为重要，因为 DNS 解析会极大地影响扫描时间。

- **`--dns-servers <server1>[,<server2>[...]]`（用于反向 DNS 查询的服务器）**  
  默认情况下，Nmap 会尝试从您的 `resolv.conf` 文件（Unix）或注册表（Win32）中确定用于反向 DNS 解析的 DNS 服务器。或者，您可以使用此选项指定备用服务器。如果您正在使用 `--system-dns` 或进行 IPv6 扫描，则此选项无效。使用多个 DNS 服务器通常比只查询一个更快且更隐蔽。指定目标 IP 空间的所有权威服务器通常能获得最佳性能。

- **`--data-length <length>`**  
  此选项向每个数据包添加 `<length>` 个随机字节的数据，适用于 TCP、UDP 和 ICMP ping 扫描类型（对于扫描 IPv4 的特权用户）。这有助于使扫描不那么显眼，更像无处不在的 `ping` 诊断程序生成的数据包。包括 Snort 在内的几个入侵检测系统会对零字节 ping 数据包发出警报。此选项可以规避这些警报。选项值 32 使回显请求看起来更可能来自 Windows，而 56 则模拟默认的 Linux `ping`。

- **`--ttl <value>`**  
  为进行 IPv4 ping 扫描的特权用户设置发送 TTL。这可以作为安全预防措施，确保扫描不会传播到本地网络之外。它也可以用来更逼真地模拟原生 `ping` 程序。一些企业网络存在已知的路由环路，且不容易修复。当遇到环路时，使用 `--ttl` 降低发送 TTL 有助于减少路由器 CPU 负载。

- **预设时序选项（`-T3`、`-T4`、`-T5` 等）**  
  较高的 `-T` 值会加快 ping 扫描的速度，就像它们加快其他 Nmap 功能一样。如果源网络和目标网络之间有适度快速且可靠的连接（即任何比拨号调制解调器更好的连接），建议使用 `-T4` 选项。

- **`--max-parallelism`、`--min-parallelism <value>`**  
  这些选项影响同时发出的探测数量。使用默认 ping 类型（两个探测）时，并行度值大致是并行扫描的机器数量。将每台主机的 ping 技术减少到一个探测（例如 `-PE`）会使给定并行度下同时扫描的主机数量加倍，而增加到每台主机四个探测（例如 `-PE -PS22,113,50000`）则会减半。大多数用户只使用预设时序选项（如 `-T4`）。

- **`--min-rtt-timeout`、`--max-rtt-timeout`、`--initial-rtt-timeout <time>`**  
  这些选项控制 Nmap 等待 ping 响应的时间。

- **输入选项（`-iL <filename>`、`-iR <number>`）**  
  主机输入选项与 Nmap 其他部分一样受支持。用户经常将列表输入（`-iL`）选项与 `-Pn` 结合使用，以避免对已知在线的主机进行 ping 扫描。在试图节省时间而这样做之前，请阅读“禁用 Ping（`-Pn`）”一节。`-iR` 选项从已分配的互联网 IP 空间中随机选择主机。其参数是您希望扫描的随机主机数量。使用 0 表示永不停止的扫描（直到您中止或杀死 Nmap 进程）。

- **输出选项（`-oA`、`-oN`、`-oG`、`-oX` 等）**  
  所有 Nmap 输出类型（正常、grepable 和 XML）都支持 ping 扫描。第 13 章《Nmap 输出格式》进一步描述了它们的工作方式。

- **`--randomize-hosts`**  
  使用此选项打乱主机扫描顺序可能会使扫描不那么显眼，尽管它也会使扫描输出稍微难以跟踪。

- **`--reason`**  
  正常的 Nmap 输出会指示主机是否在线，但不描述主机响应了哪个发现测试。要获取此详细信息，请添加 `--reason` 选项。对于主机发现，结果可能令人困惑，因为 Nmap 并不总是尝试每一个探测。它一旦收到第一个响应就会停止。因此，Nmap 可能在一次运行中报告来自主机的 ICMP 回显响应，但在第二次运行中可能首先收到 RST 响应，导致 Nmap 报告该响应。

- **`--packet-trace`**  
  当您想要比 `--reason` 提供更多细节时，请尝试 `--packet-trace`。此选项显示 Nmap 发送和接收的每个数据包，包括序列号、TTL 值和 TCP 标志等细节。

- **`-D <decoy1,decoy2,...>`**  
  特权 IPv4 ping 扫描完全支持诱饵，可以伪装真正的攻击者。

- **`-6`**  
  基于 TCP `connect` 的 ping 扫描（`-PS`）支持 IPv6 协议，包括多端口模式（例如 `-PS22,80,113`）。

- **`-S <source IP address>`、`-e <sending device name>`**  
  与 Nmap 的其他功能一样，可以使用这些选项指定源地址和发送设备。

- **通用选项**  
  默认情况下，除非指定了 `-sn` 或 `-sL`，否则 Nmap 在主机发现阶段之后会进行更具侵入性的扫描。因此，可以使用几十种通用端口扫描、操作系统检测和版本检测选项。有关更多信息，请参阅参考指南或相关章节。

### 选择和组合 Ping 选项

有效的扫描不仅仅需要了解本章和前面章节中描述的所有选项。用户必须理解如何以及何时使用它们以适应目标网络拓扑和扫描目标。

从表 3.2 中我们可以找出一些有趣的特性。到目前为止，最好的单一探测是 `-PE`（ICMP 回显）。其次是针对常见端口的各种 SYN 和 ACK 探测（`-PS` 和 `-PA`）。ICMP 时间戳探测 `-PP` 表现得相当好。最好的 UDP 探测（`-PU`）是那些针对随机高端口的探测，当与 `--source-port 53 --data-length 24` 结合使用时效果更好。针对端口 25（SMTP）的 SYN 和 ACK 探测明显比其他 TCP 探测差，可能是因为该端口通常被过滤。各种 IP 协议 ping（`-PO`）效果不佳，而 `-PM`（ICMP 地址掩码请求）排在最后。

最好的多探测组合并不是简单地从表 3.2 顶部挑选条目就能得到的。这是因为某些探测找到的主机与另一探测找到的大量重叠，因此同时使用两者比只使用一个好不了多少。我们通过计算机程序检查每个探测找到的主机数量以及它们与其他探测的重叠程度，找出了不同探测数量的最佳组合。最佳组合如表 3.3 所示。Nmap 默认的四探测组合就是从该列表中选出的，作为全面性和速度之间的折衷。

#### TCP 探测与端口选择

TCP ping 选项是 Nmap 中最强大的发现技术之一。管理员可以屏蔽 ICMP 回显请求数据包而不影响大多数用户，但服务器绝对必须响应发送到其提供的公共服务的 SYN 数据包。同时，ACK 数据包通常能穿透非状态防火墙。我建议同时使用 SYN 和 ACK 探测，使用基于您可能了解的目标网络信息以及更通用的流行端口的端口列表。表 3.4 根据经验测试列出了最可能响应（开放或关闭）的端口。在具有默认丢弃过滤器（最难穿透的类型）的主机上，这些是最可能引发响应的端口。

**表 3.4 最有价值的 TCP 探测端口（按可达性降序排列）**

| 端口号 / 服务 | 原因 |
| :--- | :--- |
| `80/http` | Web 服务器在互联网上的普及使得许多新手认为 Web 就是互联网。 |
| `443/https` | SSL 是网站保护机密目录信息的流行方式。 |
| `113/auth` | auth（identd）服务允许服务器（通常是邮件或 IRC）请求连接它们的客户端的用户名。管理员通常不过滤此端口，以避免防火墙规则阻止服务器连接回端口 113 时可能发生的长时间超时。使用此端口进行 ping 扫描有时会导致误报，因为已知一些管理员会配置防火墙伪造 RST 数据包来响应对其网络上任何 IP 的 auth 查询，即使该 IP 上不存在机器。管理员这样做是为了避免服务器超时，同时仍然阻止端口被访问。 |
| `21/ftp` | 这个文件传输协议仍然存在，尽管许多防火墙管理员不会为它的消亡感到惋惜。 |
| `23/telnet` | 许多设备仍然提供这种管理界面，尽管它是安全噩梦。 |
| `25/smtp` | 邮件是另一个公司允许通过其防火墙的“杀手级应用”。 |
| `53/domain` | 域名服务器极其普遍。 |
| `22/ssh` | SSH 似乎最终超越了 Telnet，成为远程终端管理的标准。 |
| `110/pop3` | POP3 是一种远程读取电子邮件的常用方法。 |
| `3389/ms-term-server` | Microsoft 终端服务允许用户（有时也包括黑客）访问远程计算机上的应用程序和数据。 |
| `1723/pptp` | 点对点隧道协议通常用于在 Microsoft Windows 上实现 VPN 解决方案。 |
| `8080/http-proxy` | 在端口 8080 上运行 Web 代理很常见。当端口 80 不可用时，该端口有时也用作备用 HTTP 端口。 |

除了上述列表中的流行端口外，建议至少选择一个高端口号。许多配置不佳的防火墙仅对特权端口（即 1024 以下的端口）进行默认拒绝。我通常会随机选择一个高端口号，例如 40,000 或 10,042，以捕获位于此类防火墙后面的机器。

在选择要探测的端口时，请记住要强调平台多样性。如果您将 ping 扫描限制为两个端口，HTTP（80）和 SSH（22）可能比 HTTP（80）和 HTTPS（443）更好，因为后两者是相关的 Web 服务，并且许多拥有 HTTPS 的机器通常也会提供 HTTP。对于 ping 扫描的目的来说，在同一台机器上找到两个可访问端口并不比找到一个更好。目标是选择端口，使得广泛的主机集至少匹配其中一个。

请注意，有价值的端口表不包括许多面向客户端的端口，例如无处不在的 Windows SMB 端口 135。主要原因是该表仅查看了默认丢弃防火墙后面的主机，其中绝大多数端口被过滤。在这些情况下，Windows 端口（如 135–139 和 445）通常被阻止。当这些机器不在防火墙后面时，开放端口对于 ping 扫描并不重要，因为数千个关闭端口同样有效。

#### UDP 端口选择

在选择 UDP 端口时，请记住开放端口不太可能响应探测。理想的是未过滤的端口。为避免开放端口，您可以考虑排除常见的 UDP 服务，如 DNS（端口 53）和 SNMP（161）。另一方面，防火墙规则通常非常宽松，以至于这些探测（特别是对端口 53）可能穿透并命中关闭端口。因此，我建议至少选择端口 53 和一个任意选择的高端口号，例如 37,452。

#### ICMP 探测选择

对于 ICMP，标准 ping（回显请求）通常值得尝试。许多管理员特意允许它，因为它对调试有用，或者因为 RFC 1122 要求它。我还会使用地址掩码或时间戳请求中的至少一种。这对于管理员故意阻止回显请求数据包但忘记其他 ICMP 查询的网络非常有价值。

#### 设计理想的探测组合

如何将这些 ping 类型组合成 ping 扫描策略取决于目标网络的特征和扫描目标。对于内部网络，默认 ping 类型通常效果良好。默认值也适用于大多数随意扫描，偶尔遗漏一台主机没什么大不了的。增加更多探测可以帮助捕获那些偶尔出现的隐蔽机器，但代价是使 ping 扫描花费更长的时间。所花时间大致与发送到每台机器的探测数量成正比。对于通过互联网对目标网络进行安全扫描，通常建议增加更多探测。尝试包含前面讨论的各种技术。以下是一组 ping 选项，应该能捕获绝大多数主机：`-PE -PP -PS21,22,23,25,80,113,443,31339 -PA80,113,443,10042`。加上 `--source-port 53` 可能也值得。结果能好多少，又需要多花多少时间？这当然取决于目标网络，但 Nmap 的随机目标选择选项（`-iR`）使得进行快速测试变得容易。示例 3.13 显示了 Nmap 生成 50,000 个随机 IP 地址，然后使用默认选项进行 ping 扫描。您应该记住，默认是四个探测：一个 ICMP 回显请求、一个到端口 443 的 TCP SYN、一个到端口 80 的 TCP ACK 和一个 ICMP 时间戳请求。

**示例 3.13 生成 50,000 个 IP 地址，然后使用默认选项进行 ping 扫描**

```sh
# nmap -n -sL -iR 50000 | awk '/^Host / {print $2}' | sort -n > 50K_IPs
# head -5 50K_IPs
3.0.47.195
3.100.112.251
3.100.98.39
3.10.89.120
3.101.183.79
# nmap -n -sn -T4 -iL 50K_IPs -oA 50KHosts_DefaultPing

Starting Nmap ( https://nmap.org )
Host 4.178.9.27 is up (0.27s latency).
Host 12.135.202.138 is up (0.20s latency).
Host 12.151.172.161 is up (0.089s latency).
[数千行输出被截断]
Host 222.99.77.130 is up (0.20s latency).
Nmap done: 50000 IP addresses (3927 hosts up) scanned in 2532.05 seconds
```

扫描 50,000 个地址花了 42 分钟多一点，检测到 3,927 台主机。为了确定使用更广泛 ping 技术的效果，使用每端口 14 个探测（而不是默认的 4 个）重新扫描了同样的 50,000 个主机。如示例 3.14 所示，Nmap 能够多检测到 785 台主机（增加了 20%）。花费了约 147 分钟，几乎延长了 3.5 倍。考虑到检测到的新主机数量，多花的时间是值得的。请注意，并非所有新主机都是合法的。增加 ping 探测的数量会增加 Nmap 命中网络伪像（使不存在的主机看起来活跃）的机会。防火墙对发往端口 113 的 SYN 或 ACK 数据包返回 RST 就是一个例子。

**示例 3.14 使用额外探测重复 ping 扫描**

```sh
# nmap -n -sn -PE -PP -PS21,22,23,25,80,113,443,31339 -PA80,113,443,10042 \
  -T4 --source-port 53 -iL 50K_IPs -oA 50KHosts_ExtendedPing
Starting Nmap ( https://nmap.org )
Host 4.238.177.186 is up (0.44s latency).
Host 12.135.202.138 is up (0.13s latency).
Host 12.151.172.161 is up (0.092s latency).
[数千行输出被截断]
Host 222.94.94.113 is up (0.23s latency).
Nmap done: 50000 IP addresses (4712 hosts up) scanned in 8842.31 seconds
```

在为客户进行安全审计时，我通常先使用最常用的 1,000 个端口（默认）进行端口扫描，并结合像示例 3.14 中所示的全面 ping 扫描选项。这样的扫描不会花费特别长的时间，使我能够快速开始工作。我还会在工作时在后台启动针对所有 65K TCP 端口的 `-Pn`（禁用 ping）扫描。当它们完成时（可能是几天后），我会将它们与我最初的快速扫描进行比较，并调查发现的任何新端口或机器。
