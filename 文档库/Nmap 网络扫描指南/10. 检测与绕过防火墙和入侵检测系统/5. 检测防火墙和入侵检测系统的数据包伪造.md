## 检测防火墙和入侵检测系统的数据包伪造

前面几节提到，一些防火墙和入侵检测系统可以配置为伪造数据包，就好像它们来自设备后面受保护的某个系统一样。TCP RST 数据包就是一个常见的例子。负载均衡器、SSL 加速器、网络地址转换设备以及某些蜜网也可能导致令人困惑或不一致的结果。了解 Nmap 如何解释响应对于拼凑复杂的远程网络拓扑有很大帮助。当 Nmap 报告异常或出乎意料的结果时，您可以添加 `--packet-trace` 选项来查看 Nmap 得出结论所依据的原始数据包。在令人困惑的情况下，您可能还需要更进一步，发起自定义探测并使用 Nping 和 Wireshark 等其他工具分析数据包。目标通常是找到不一致之处，以帮助您理解实际的网络设置。以下各节描述了实现这一目标的几种有用技术。虽然这些测试大多不直接涉及 Nmap，但它们对于解释意外的 Nmap 结果可能很有用。

### 寻找 TTL 一致性

防火墙、负载均衡器、NAT 网关和类似设备通常位于它们所保护的机器之前的一跳或多跳。在这种情况下，可以创建具有特定 TTL 值的数据包，使其能够到达网络设备但无法到达最终主机。如果从这样的探测中收到 RST，则它一定是由该设备发送的。

在一次非正式评估中，我通过互联网扫描了一家大型杂志出版商的网络（你可能还记得他们在[“解决方案：扫描大型网络中的某个开放 TCP 端口”](solution-find-open-port.html)一节中出现过）。几乎所有 IP 地址都显示端口 113 为关闭状态。怀疑是防火墙伪造了 RST，我进一步深入挖掘。因为该主机同时包含开放、关闭和过滤的端口，我决定特别关注这台主机：

```sh
# nmap -sS -Pn -T4 mx.chi.playboy.com
Starting Nmap ( https://nmap.org )
Nmap scan report for mx.chi.playboy.com (216.163.143.4)
Not shown: 998 filtered ports
PORT    STATE  SERVICE
25/tcp  open   smtp
113/tcp closed auth

Nmap done: 1 IP address (1 host up) scanned in 53.20 seconds
```

端口 113 是真的关闭，还是防火墙在伪造 RST 数据包？我使用免费工具 hping2 的自定义 traceroute 模式计算了到端口 25 和端口 113 的距离（以网络跳数计），如[示例 10.22](#defeating-firewalls-ids-customtraceroute) 所示。我本可以使用更快的 Nmap `--traceroute` 选项来做这件事，但当时该选项还不存在。

**示例 10.22 检测关闭和过滤的 TCP 端口**

```sh
# hping2 -t 5 --traceroute -p 25 -S mx.chi.playboy.com
[combined with results from hping2 -i 1 --ttl \* -p 25 -S mx.chi.playboy.com]
```

这个自定义 traceroute 显示到达开放端口 25 需要 13 跳。在 12 跳处有一个位于芝加哥的防火墙，其名称 fw.chi.playboy.com 很有帮助。我们期望同一台机器上的不同端口具有相同的跳数距离。然而，端口 113 在仅 12 跳后就以 RST 响应。该 RST 是由 fw.chi.playboy.com 伪造的。由于已知该防火墙会伪造端口 113 的响应，因此不应将这些数据包作为给定 IP 地址上存在主机的指示。我通过再次使用常见探测类型（如 ICMP 回显请求（`-PE`）和发往端口 22 和 80 的 SYN 数据包（`-PS22,80`））进行 ping 扫描来找到在线主机，但省略了任何涉及 TCP 端口 113 的 ping 探测。

### 寻找 IP ID 和序列号的一致性

每个 IP 数据包都包含一个 16 位的标识字段，用于分片重组。它也可以被利用来获取关于远程主机的惊人信息。这包括使用 Nmap 空闲扫描技术进行端口扫描、流量估算、主机别名检测等等。它还有助于检测许多网络设备，例如负载均衡器。我曾经在扫描 beta.search.microsoft.com 时注意到奇怪的操作系统检测结果。于是我启动了针对 TCP 端口 80 的 hping2 SYN 探测，以了解发生了什么。[示例 10.23](#defeating-firewalls-ids-ipid-ms) 显示了结果。

查看粗体显示的 IP ID 编号序列，可以清楚地看到实际上有两台机器通过某种负载均衡器共享这个 IP 地址。一个的 IP ID 序列在 57K 范围内，而另一个使用 18K。鉴于这些信息，难怪 Nmap 难以确定唯一的操作系统猜测。它们可能运行在非常不同的系统上。

可以对其他数字字段执行类似的测试，例如 TCP 时间戳选项或开放端口返回的初始序列号。在这个特定案例中，您可以看到 TCP 窗口大小和 TCP 标志也暴露了主机的差异。

### 伪造 TCP 校验和技巧

确定 IDS 或防火墙是否在伪造响应数据包的另一个有用技巧是发送带有伪造 TCP 校验和的探测。基本上所有最终主机在进一步处理之前都会检查校验和，并且不会响应这些损坏的数据包。另一方面，防火墙出于性能考虑常常省略此检查。我们可以通过 `--badsum` 选项检测这种行为，如[示例 10.24](#defeating-firewalls-ids-badsum-example) 所示。

**示例 10.24 通过错误的 TCP 校验和查找防火墙**

```sh
# nmap -sS -p 113 -Pn --badsum google.com

Starting Nmap ( https://nmap.org )
Warning: Hostname google.com resolves to 3 IPs. Using 64.233.187.99.
Nmap scan report for jc-in-f99.google.com (64.233.187.99)
PORT    STATE  SERVICE
113/tcp closed auth

Nmap done: 1 IP address (1 host up) scanned in 0.44 seconds
```

从[示例 10.24](#defeating-firewalls-ids-badsum-example) 我们可以推断，存在某种网络设备（可能是防火墙）在不验证 TCP 校验和的情况下处理发往 google.com 端口 113 的数据包。通常情况下，最终主机会静默丢弃带有错误 TCP 校验和的数据包，我们将看到过滤端口而不是关闭端口。`--badsum` 也会对 IP 之上的其他协议（包括 UDP、ICMP 和 IGMP）使用错误的校验和。

该技术以及故意发送带有格式错误校验和的数据包的其他原因，在 Ed3f 的 [《Phrack》60 期第 12 篇文章](https://nmap.org/p60-12.html) 中有进一步描述。虽然这有时是一种有用的技术，但有几个注意事项需要考虑：

1. 许多现代防火墙现在会验证 TCP 校验和（至少在决定是否响应数据包时），以避免泄露此信息。因此，此技术对于证明 `--badsum` 探测响应是由防火墙（或其他具有不完整 TCP 协议栈的设备）发送的，比证明被过滤的 `--badsum` 探测是由最终主机丢弃的更有用。

2. 使用 `--badsum` 并不能保证在所有平台上都会发送带有错误校验和的数据包。在一些系统上，内核或网卡会执行校验和计算并插入正确的值，覆盖期望的错误值。确保这没有发生在您身上的一种方法是使用远程机器嗅探您正在发送的数据包。例如，使用 `tcpdump` 嗅探时，带有错误 TCP 校验和的数据包会显示为 `[bad tcp cksum aa79 (->ab79)!]`。另一种方法是针对您的一台主机（至少有一个开放端口）进行正常的 SYN 扫描。然后使用 `--badsum` 进行相同的扫描。如果仍然显示相同的开放端口，那么 `--badsum` 可能对您不起作用。请按照[“错误报告”](man-bugs.html)一节中的描述报告该问题。

### 往返时间

当防火墙伪造探测响应时，该响应通常会比真正的目标主机返回的响应稍快一些。毕竟，防火墙通常至少近一跳。它也针对快速解析和处理数据包进行了优化，并且几乎不做其他事情。另一方面，目标主机可能忙于运行应用程序，需要多花几毫秒来响应探测。因此，仔细比较往返时间通常可以暴露防火墙的诡计。

这种技术的一个挑战是防火墙响应和真实目标响应之间的时间差异可能只有几分之一毫秒。正常的往返时间变化可能大于此，因此仅发送两个探测（一个已知会引发目标主机响应的探测，和一个可能来自防火墙的可疑响应探测）通常是不够的。发送每种探测类型各一千个可以抵消大部分 RTT 变化，从而可以辨别出根本差异。这不需要花费很长时间——使用 `nping` 和选项 `-c 1000 --rate 20` 可以在不到一分钟内发送一千个探测。从这些结果中，计算中位数而不是使用平均值。这可以防止巨大的时间值（例如，因丢失的响应在两秒后重传）扭曲数据。再进行一两次千次探测以确定结果的一致性。然后对可疑探测进行相同的操作，并比较两者。如果时间精确到最后一个有效数字都相同，则很可能是同一台主机发送了两种响应。如果您始终看到一种探测类型比另一种响应得更快，则可能是数据包伪造造成的。

这种方法并不完美。时间差异可能由防火墙以外的许多其他因素引起。它仍然是一种有价值的技术，因为检测网络异常（如数据包伪造）就像在法庭上证明案件一样。每一点证据都有助于得出结论。这种差异甚至可能带来比防火墙伪造更有趣的发现。也许目标上的某些端口被重定向到蜜网，以便更好地研究攻击。

### 对数据包头和内容的细致分析

即使是一个小的 TCP 报头中，有多少元素可能不同是令人惊讶的。请参阅[第 8 章《远程操作系统检测》](osdetect.html)以了解数十个可以指示不同操作系统的细微细节。例如，不同的系统会以不同的 TCP 选项、RST 数据包文本、服务类型值等做出响应。如果负载均衡器后面有多个系统，或者数据包是由防火墙或入侵检测系统发送的，这些数据包很少会完全匹配。

一个用于剖析数据包头的优秀工具是 Wireshark，因为它可以将报头分解成单独的字段，并提供数据包二进制内容的文本描述。比较数据包的技巧是收集一个您认为可能来自防火墙的数据包，以及另一个来自目标主机或目标操作系统的同类型数据包。您可能能够收集到的两种数据包类型是 TCP 重置数据包和 ICMP 错误数据包。通过使用 Nping 或 Nmap 的 `--scanflags` 选项，应该能够引发具有不同 IP、TCP 或 ICMP 头的响应。

### 异常的网络一致性

当响应数据包由防火墙发送时，它们通常比预期的单台机器集群更加一致。在扫描前面 TTL 检查部分讨论的杂志公司时，我发现数百个连续 IP 的机器都对端口 113 以 RST 响应。在一个真实的机器集群中，您会期望在给定时间至少有几台离线。此外，我无法从这些地址中的大多数引发任何其他类型的响应。这个可疑的结果促使我进行 TTL 测试，结果表明 `fw.chi` 主机实际上在伪造 RST 数据包。

防火墙甚至不必伪造数据包就会暴露自己。另一种常见的防火墙配置是丢弃发往特定端口的数据包。许多 ISP 会过滤 Windows 端口 135、139 和 445，以减少蠕虫的传播。如果大量相邻的在线主机出现相同的一组过滤端口，网络防火墙很可能是罪魁祸首。在确定哪些端口被防火墙过滤后，您通常可以通过扫描许多网络块中这些被过滤的端口来推断出有多少主机受到这些防火墙规则的保护。这可能导致发现任何意外的漏洞或组织的 DMZ（非军事区），后者通常托管公共服务，防火墙规则要宽松得多。
