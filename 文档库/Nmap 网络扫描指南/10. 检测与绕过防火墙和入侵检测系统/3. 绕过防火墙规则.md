## 绕过防火墙规则

虽然映射防火墙规则可能很有价值，但绕过规则通常是主要目标。Nmap 实现了许多技术来实现这一点，尽管大多数仅对配置不当的网络有效。不幸的是，这些网络很常见。每种技术成功的概率都很低，因此应尽可能多尝试不同的方法。攻击者只需要找到一个错误配置就能成功，而网络防御者必须堵住每一个漏洞。

### 奇特的扫描标志

上一节讨论了使用 ACK 扫描来映射哪些目标网络端口被过滤。然而，它无法确定哪些可访问的端口是开放还是关闭。Nmap 提供了几种扫描方法，善于在绕过防火墙的同时仍然提供所需的端口状态信息。FIN 扫描就是其中一种技术。在[“ACK 扫描”一节](determining-firewall-rules.html#defeating-firewalls-ids-ackscan)中，对一台名为 Para 的机器运行了 SYN 和 ACK 扫描。SYN 扫描仅显示两个开放端口，可能是由于防火墙限制。同时，ACK 扫描无法区分开放端口和关闭端口。[示例 10.6](#defeating-firewalls-fin-scan) 显示了针对 Para 的另一次扫描尝试，这次使用的是 FIN 扫描。由于设置了纯粹的 FIN 数据包，此数据包可以飞过阻止 SYN 数据包的规则。虽然 SYN 扫描在 100 以下只发现了一个开放端口，但 FIN 扫描发现了两个。

**示例 10.6 针对无状态防火墙的 FIN 扫描**

```sh
# nmap -sF -p1-100 -T4 para

Starting Nmap ( https://nmap.org )
Nmap scan report for para (192.168.10.191)
Not shown: 98 filtered ports
PORT   STATE         SERVICE
22/tcp open|filtered ssh
53/tcp open|filtered domain
MAC Address: 00:60:1D:38:32:90 (Lucent Technologies)

Nmap done: 1 IP address (1 host up) scanned in 1.61 seconds
```

许多其他扫描类型也值得尝试，因为目标防火墙规则和目标主机类型决定了哪些技术会生效。一些特别有价值的扫描类型包括 FIN、Maimon、Window、SYN/FIN 和 NULL 扫描。所有这些都在[第 5 章《端口扫描技术与算法》](scan-methods.html)中进行了描述。

### 源端口操纵

一种令人惊讶的常见错误配置是仅根据源端口号来信任流量。这很容易理解。管理员会设置一个崭新的防火墙，却被那些应用程序停止工作的用户抱怨淹没。特别是，DNS 可能被破坏，因为来自外部服务器的 UDP DNS 回复无法再进入网络。FTP 是另一个常见例子。在主动 FTP 传输中，远程服务器试图建立回客户端的连接以传输请求的文件。

这些问题有安全的解决方案，通常以应用级代理或协议解析防火墙模块的形式存在。不幸的是，还有更简单但不安全的解决方案。注意到 DNS 回复来自端口 53，主动 FTP 来自端口 20，许多管理员陷入了一个陷阱：简单地允许来自这些端口的入站流量。他们常常认为没有攻击者会注意到并利用这些防火墙漏洞。在其他情况下，管理员认为这是短期补救措施，直到他们能够实施更安全的解决方案。然后他们忘记了安全升级。

工作过度的网络管理员并不是唯一陷入这个陷阱的人。许多产品都带有这些不安全的规则。甚至连微软也曾有过。Windows 2000 和 Windows XP 附带的 IPsec 过滤器包含一个隐式规则，允许来自端口 88（Kerberos）的所有 TCP 或 UDP 流量。苹果粉丝不应该对此太沾沾自喜，因为 Mac OS X Tiger 附带的防火墙同样糟糕。Jay Beale 发现，即使你在防火墙 GUI 中勾选了“阻止 UDP 流量”框，来自端口 67（DHCP）和 5353（Zeroconf）的数据包仍然可以通过。另一个可悲的例子是，Zone Alarm 个人防火墙（最高版本 2.1.25）允许任何源端口为 53（DNS）或 67（DHCP）的入站 UDP 数据包。

Nmap 提供了 `-g` 和 `--source-port` 选项（它们等效）来利用这些弱点。只需提供一个端口号，Nmap 就会在可能的情况下从该端口发送数据包。Nmap 必须为某些操作系统检测测试使用不同的端口号才能正常工作。大多数 TCP 扫描（包括 SYN 扫描）以及 UDP 扫描都完全支持该选项。2004 年 5 月，JJ Gray 在 Bugtraq 上发布了示例 Nmap 扫描，演示了如何利用 Windows IPsec 源端口 88 漏洞对付他的一个客户。正常扫描和 `-g 88` 扫描如[示例 10.7](#defeating-firewalls-sourceport88) 所示。为了简洁和清晰，省略了一些输出。

**示例 10.7 使用源端口 88 绕过 Windows IPsec 过滤器**

```sh
# nmap -sS -v -v -Pn 172.25.0.14

Nmap scan report for 172.25.0.14
Not shown: 1658 filtered ports
PORT   STATE  SERVICE

# nmap -sS -v -v -Pn -g 88 172.25.0.14

### IPv6 攻击

虽然 IPv6 尚未席卷全球，但在日本和其他一些地区相当流行。当组织采用此协议时，他们常常忘记像对 IPv4 那样本能地锁定它。或者他们可能试图这样做，但发现其硬件不支持 IPv6 过滤规则。过滤 IPv6 有时比 IPv4 更关键，因为扩展的地址空间通常允许将全球可寻址的 IPv6 地址分配给通常必须使用 [RFC 1918](http://www.rfc-editor.org/rfc/rfc1918.txt) 指定的私有 IPv4 地址的主机。

执行 IPv6 扫描而不是默认的 IPv4 扫描通常与在命令行中添加 `-6` 一样简单。此协议的某些功能（如操作系统检测和 UDP 扫描）尚不支持，但最流行的功能可以工作。[示例 10.8](#defeating-firewalls-ex-ipv6) 演示了很久以前对一家著名的 IPv6 开发和倡导组织进行的 IPv4 和 IPv6 扫描。

### IP ID 空闲扫描

IP ID 空闲扫描因其是最隐蔽的扫描类型之一而闻名，因为从您的真实地址没有数据包发送到目标。开放端口通过所选僵尸机器的 IP ID 序列推断。空闲扫描的一个不太为人知的特点是，获得的结果实际上就是僵尸直接扫描目标主机所得到的结果。与 `-g` 选项允许利用受信任的源端口类似，空闲扫描有时可以利用受信任的源 IP 地址。这种巧妙的扫描类型最初由安全研究员 Antirez 构想，在[“TCP 空闲扫描（`-sI`）”](idlescan.html)一节中有完整描述。

### 多重 Ping 探测

尝试扫描经过防火墙的网络时，一个常见问题是丢弃的 ping 探测可能导致遗漏主机。为了减少这个问题，Nmap 允许并行发送多种探测。希望至少有一个能通过。[第 3 章《主机发现（“Ping 扫描”）》](host-discovery.html)深入讨论了这些技术，包括关于最佳防火墙突破技术的经验数据。

### 分片

一些数据包过滤器难以处理 IP 数据包分片。它们可以自己重组数据包，但这需要额外资源。也有可能分片会走不同的路径，阻止重组。由于这种复杂性，一些过滤器忽略所有分片，而另一些则自动传递除第一个分片之外的所有分片。如果第一个分片不够长，无法容纳整个 TCP 报头，或者第二个数据包部分覆盖它，可能会发生有趣的事情。受这些问题影响的过滤设备数量正在减少，但尝试一下也无妨。

如果指定了 `-f`，Nmap 扫描将使用微小的 IP 分片。默认情况下，Nmap 会在每个分片中包含最多 8 字节的数据，因此一个典型的 20 或 24 字节（取决于选项）的 TCP 数据包被分成三个微小的分片。每个 `-f` 实例将最大分片数据大小增加 8。因此，`-f -f` 允许每个分片最多 16 字节的数据。或者，您可以指定 `--mtu` 选项，并将最大数据字节数作为参数。`--mtu` 参数必须是 8 的倍数，并且不能与 `-f` 选项结合使用。

某些源系统在内核中对传出数据包进行重组。Linux 与 iptables 连接跟踪模块就是一个例子。在运行嗅探器（如 Wireshark）时进行扫描，以确保发送的数据包被分片。如果您的主机操作系统引起问题，请尝试使用 `--send-eth` 选项绕过 IP 层并发送原始以太网帧。

分片仅支持 Nmap 的原始数据包功能，这包括 TCP 和 UDP 端口扫描（除了连接扫描和 FTP 反弹扫描）以及操作系统检测。版本检测和 Nmap 脚本引擎等功能通常不支持分片，因为它们依赖于您主机的 TCP 堆栈与目标服务通信。

无序和部分重叠的 IP 分片可用于网络研究和漏洞利用，但这需要比 Nmap 更底层的网络工具。Nmap 按顺序发送分片，没有任何重叠。

如果分片端口扫描成功通过，可以使用诸如 [Fragroute](http://www.monkey.org/~dugsong/fragroute/) 之类的工具对用于攻击主机的其他工具和漏洞利用进行分片。

### 代理

应用级代理，尤其是 Web 代理，由于被认为具有安全性和网络效率（通过缓存）优势而变得流行。与防火墙和 IDS 一样，配置不当的代理可能导致的安全问题比它们解决的问题更多。最常见的问题是未能设置适当的访问控制。互联网上存在成千上万的开放代理，允许任何人使用它们作为匿名跳点访问其他互联网站点。许多组织使用自动扫描器来查找这些开放代理并分发 IP 地址。有时代理被用于可以说是有益的事情，比如逃避中国政府对其居民实施的严厉审查。已知这个“中国防火墙”会屏蔽《纽约时报》网站以及政府不同意的其他新闻、政治和宗教网站。不幸的是，开放代理更常被更邪恶的人滥用，他们想匿名入侵网站、进行信用卡欺诈或用垃圾邮件淹没互联网。

虽然托管一个对互联网资源完全开放的代理会导致许多问题，更严重的情况是开放代理允许连接回受保护的网络。决定内部主机必须使用代理访问互联网资源的管理员常常无意中也允许了相反方向的流量。黑客 Adrian Lamo 因利用这种反向代理技术入侵微软、Excite、雅虎、WorldCom、纽约时报和其他大型网络而闻名。

Nmap 目前不提供代理穿透扫描选项，尽管它在优先级列表中很高。[“解决方案：修改版本检测以适应自定义需求，例如开放代理检测”](vscan-hack-it.html)一节讨论了一种使用 Nmap 版本检测查找开放代理的方法。此外，在 [Packet Storm](http://packetstormsecurity.nl/) 等互联网站点上也有大量专用的免费代理扫描器。成千上万的开放代理列表也很普遍。

### MAC 地址欺骗

以太网设备（包括 Wi-Fi）由唯一的六字节介质访问控制（MAC）地址标识。前三个字节构成组织唯一标识符（OUI）。此前缀由 IEEE 分配给供应商。供应商负责在其销售的适配器和设备中唯一分配剩余的三个字节。Nmap 包含一个数据库，将 OUI 映射到它们被分配给的供应商名称。这有助于在扫描网络时识别设备，尽管本节解释了为什么不能完全信任它。OUI 数据库文件 `nmap-mac-prefixes` 在[“MAC 地址供应商前缀：`nmap-mac-prefixes`”](nmap-mac-prefixes.html)一节中描述。

虽然 MAC 地址预分配给以太网设备，但在大多数现代硬件上可以通过驱动程序更改。但由于很少有人更改其 MAC 地址（甚至知道他们有一个），许多网络使用它们进行识别和授权。例如，大多数无线接入点提供配置选项，限制访问某个 MAC 地址集。类似地，一些付费或私人网络会在您使用 Web 表单连接后强制您进行身份验证或付款。然后他们根据您的 MAC 地址允许您访问网络的其余部分。鉴于通常很容易嗅探 MAC 地址（它们必须在发送和接收的每个帧中发送），然后欺骗该 MAC 以获得未经授权的网络访问，这种访问控制形式相当薄弱。它也仅在网络边缘有效，因为终端主机的 MAC 地址在穿越路由器时会被替换。

除了访问控制，MAC 地址有时也用于问责。网络管理员在获得 DHCP 租约或新机器在网络中通信时记录 MAC 地址。如果稍后收到网络滥用或盗版投诉，他们根据 IP 地址和事件时间找出 MAC 地址。然后他们使用 MAC 追踪负责的机器及其所有者。MAC 地址欺骗的易用性在一定程度上削弱了这种方法。即使用户有罪，他们也可能提出 MAC 地址欺骗的幽灵来推卸责任。

Nmap 通过 `--spoof-mac` 选项支持 MAC 地址欺骗。提供的参数可以有多种形式。如果参数只是数字 `0`，Nmap 会为会话选择一个完全随机的 MAC 地址。如果给定的字符串是偶数个十六进制数字（可选地用冒号分隔对），Nmap 将使用这些作为 MAC。如果提供的十六进制数字少于 12 个，Nmap 会用随机值填充剩下的六字节。如果参数不是零或十六进制字符串，Nmap 会搜索 `nmap-mac-prefixes` 以查找包含给定字符串的供应商名称（不区分大小写）。如果找到匹配项，Nmap 使用供应商的 OUI 并随机填充剩余的三个字节。有效的 `--spoof-mac` 参数示例包括 `Apple`、`0`、`01:02:03:04:05:06`、`deadbeefcafe`、`0020F2` 和 `Cisco`。此选项隐含了 `--send-eth`，以确保 Nmap 实际发送以太网级数据包。此选项仅影响原始数据包扫描（如 SYN 扫描或操作系统检测），不影响面向连接的功能（如版本检测或 Nmap 脚本引擎）。

即使不需要 MAC 地址欺骗来访问网络，它也可以用于欺骗。如果我在会议上从我的 Thinkpad 启动扫描，使用 `--spoof-mac Apple`，怀疑的目光可能会转向房间里使用 MacBook 的用户。

### 源路由

这种老派技术在某些情况下仍然有效。如果路径上的某个特定路由器给您带来了麻烦，请尝试找到绕过它的路由。此技术的有效性有限，因为数据包过滤问题通常发生在目标网络或其附近。这些机器要么丢弃所有源路由数据包，要么是进入网络的唯一途径。Nmap 通过 `--ip-options` 选项支持松散和严格源路由。例如，指定 `--ip-options "L 192.168.0.7 192.168.30.9"` 要求数据包通过给定的两个 IP 航点进行松散源路由。将 `L` 替换为 `S` 表示严格源路由。如果您选择严格源路由，请记住您必须指定路径上的每一跳。

有关使用源路由绕过现代网络过滤策略的真实示例，请参见[“防火墙破坏的实际示例”](#fw-subversion-example)一节。虽然 IPv4 源路由非常常见地被阻止，但 IPv6 形式的源路由更为普遍。关于该问题的有趣文章可在 [http://lwn.net/Articles/232781/](http://lwn.net/Articles/232781/) 找到。

如果使用 Nmap 发现到目标机器的源路由路径，利用不仅限于端口扫描。[Ncat](https://nmap.org/ncat) 可以通过源路由路径启用 TCP 和 UDP 通信（使用 `-g` 选项）。

### FTP 反弹扫描

虽然只有一小部分 FTP 服务器仍然易受攻击，但检查您所有客户系统是否存在此问题是值得的。至少，它允许外部攻击者利用易受攻击的系统扫描其他方。更糟糕的配置甚至允许攻击者绕过组织的防火墙。此技术的细节和示例在[“TCP FTP 反弹扫描（`-b`）”](scan-methods-ftp-bounce-scan.html)一节中提供。[示例 10.9](#defeating-firewalls-ftpbounce-working) 显示了一台 HP 打印机被用于中继端口扫描。如果这台打印机位于组织防火墙后面，它也可以用于扫描正常情况下（对攻击者）无法访问的内部地址。

Nmap scan report for scanme.nmap.org (64.13.134.52)
PORT    STATE    SERVICE
22/tcp  open     ssh
25/tcp  open     smtp
135/tcp filtered msrpc

Nmap done: 1 IP address (1 host up) scanned in 21.79 seconds
```

### 采取替代路径

我讨厌过度使用“跳出框框思考”的陈词滥调，但不断敲打安全网络的前门并不总是最佳方法。寻找其他进入方式。拨打他们的电话线路，攻击可能拥有特殊网络访问权限的子公司，或者带着 Wi-Fi 嗅探设备出现在他们的办公室，甚至潜入并插入一个方便的以太网插孔。Nmap 通过所有这些连接都能很好地工作。只需确保您的渗透测试合同涵盖了这些方法，以免您的客户在忍者服中抓住您，试图爬上他们的数据中心屋顶。

### 防火墙破坏的实际示例

现在我们已经介绍了很多绕过防火墙规则的单个技术，是时候将它们放在一个真实的渗透测试场景中来组合使用了。这一切始于安全专业人士 Michael Cain 在 SecurityFocus 的 `pen-test` 邮件列表上发布的一篇[帖子](https://seclists.org/pen-test/2008/Mar/10)。他和同事 Demetris Papapetrou 正在渗透测试一家大公司的内部网络，并刚刚绕过了旨在阻止一个 VLAN 访问另一个 VLAN 的防火墙规则。我很高兴读到他们使用 Nmap 完成了这一壮举，于是我向他们询问了整个故事。它既具有教育意义又鼓舞人心，因为它展示了坚持和尝试每一种你所知道的技术（即使在最常见的漏洞利用失败之后）的价值。不要让防火墙打败你！

故事开始于 Michael 和 Demetris 执行 Nmap 扫描，显示他们被困在一个严重过滤的网络中。他们可以访问一些公司服务器，但无法访问任何（可能易受攻击的）台式客户端机器，而这些机器必然存在于网络的某个地方。也许他们在一个受限的会议室或大厅网络，或者一个为公司访客设置的无线接入点。发现的一些主机和网络如[示例 10.10](#fw-megacorp-layout) 所示。出于保密原因，这个故事中的一些细节（如 IP 地址）已被更改。我将目标公司称为 Megacorp。

**示例 10.10 Megacorp 的一些有趣主机和网络**

```log
10.10.5.1  - 稍后会给我们带来麻烦的路由器/防火墙
10.10.5.42 - 我们的主角从此机器进行扫描
10.10.6.30 - files2.megacorp.com；Nmap 显示这是一台 Windows 机器，端口 445 开放。
10.10.6.60 - mail.megacorp.com；Nmap 操作系统检测显示它是 Solaris 8。端口 25 开放且可访问。
10.10.10.0/24 - 这里没有显示任何内容，但许多 IP 都有反向 DNS 名称，所以 Demetris 怀疑防火墙可能阻止了他的探测。目标是到达此子网上的任何可用主机。
```

鉴于确定 10.10.10.0/24 网络上是否有任何主机隐藏的目标，Demetris 开始使用 ICMP 回显请求查询（`-PE`）进行简单的 ping 扫描。结果如[示例 10.11](#fw-megacorp-pingscan) 所示。

**示例 10.11 针对目标网络的 Ping 扫描**

```sh
# nmap -n -sn -PE -T4 10.10.10.0/24
Starting Nmap ( https://nmap.org )
Nmap done: 256 IP addresses (0 hosts up) scanned in 26.167 seconds
```

Ping 扫描未能找到任何有响应的主机。Demetris 理所当然地感到失望，但至少这让本节更有趣和更具指导性。也许网络真的是空的，但它也可能挤满了易受攻击的机器，而 Demetris 被阻止访问这些机器。他需要更深入地挖掘。在[示例 10.12](#fw-megacorp-pingscan-pt) 中，Demetris 选择了该网络上的一个 IP 并执行了 ping 扫描。他指定了数据包跟踪（`--packet-trace`）和额外详细程度（`-vv`）选项，以确定数据包级别发生了什么。选择只有一个 IP 是为了避免数百个数据包混乱地涌入。

**示例 10.12 针对单个 IP 的数据包跟踪**

```sh
# nmap -vv -n -sn -PE -T4 --packet-trace 10.10.10.7
Starting Nmap ( https://nmap.org )
SENT (0.3130s) ICMP 10.10.5.42 > 10.10.10.7 echo request (type=8/code=0)
               ttl=41 id=7193 iplen=28 
RCVD (0.3130s) ICMP 10.10.5.1 > 10.10.5.42 host 10.10.10.7 unreachable 
               (type=3/code=1) ttl=255 id=25980 iplen=56 
Nmap done: 1 IP address (0 hosts up) scanned in 0.313 seconds
```

看起来 Demetris 在尝试扫描这些 IP（或至少这个 IP）时收到了 ICMP 主机不可达消息。当主机不可用且路由器无法确定 MAC 地址时，路由器通常会这样做。有时它也是由过滤引起的。Demetris 扫描了网络上的其他主机，并验证了它们的行为相同。可能只有 ICMP 数据包被过滤，因此 Demetris 决定尝试 TCP SYN 扫描。他运行命令 `nmap -vv -n -sS -T4 -Pn --reason 10.10.10.0/24`。所有端口都显示为过滤，`--reason` 结果归咎于一些主机不可达消息和一些无响应的端口。无响应的端口可能是由于路由器发送的主机不可达消息的速率限制。许多路由器每几秒只会发送一个这样的消息。Demetris 可以通过再次运行扫描并查看主机不可达消息是否来自完全相同的一组端口来验证速率限制是否是原因。如果端口相同，则可能是基于特定端口的过滤器。如果 Nmap 每次收到不同端口的主机不可达消息，则速率限制很可能是原因。

如果过滤器导致问题，它可能是一个简单的无状态防火墙，常见于路由器和交换机。如前几节所述，这些防火墙有时允许 TCP ACK 数据包不受干扰地通过。Demetris 重复了扫描，但指定了 `-sA` 进行 ACK 扫描，而不是 `-sS`。扫描发现的任何 `unfiltered` 端口将表明 ACK 数据包成功通过并引发了来自目标主机的 TCP RST 响应。不幸的是，在这种情况下，结果与 SYN 扫描一样，都是 `filtered`。

Demetris 决定尝试更高级的方法。他已经知道，从最初的 Nmap 扫描中，10.10.6.30（files2.megacorp.com）上的 Windows 机器上的端口 445 是开放的。虽然 Demetris 无法直接访问 10.10.10.0/24 网络，但也许 files2（作为重要的公司文件服务器）能够访问该 IP 范围。Demetris 决定尝试使用 IPID 空闲扫描通过 files2 反弹他的扫描。首先，他想确保 files2 能够作为僵尸工作，方法是针对已知响应的机器 10.10.6.60（端口 25 开放）进行测试。此测试的结果如[示例 10.13](#fw-megacorp-idlescan) 所示。

**示例 10.13 测试空闲扫描**

```sh
# nmap -vv -n -Pn -sI 10.10.6.30:445 -p 25 10.10.6.60

Starting Nmap ( https://nmap.org )

Initiating idle scan against 10.10.6.60 at 13:10
Idle scan using zombie 10.10.6.30 (10.10.6.30:445); Class: Incremental
Even though your Zombie (10.10.6.30) appears to be vulnerable to IP ID
sequence prediction (class: Incremental), our attempts have failed.  This
generally means that either the Zombie uses a separate IP ID base for each
host (like Solaris), or because you cannot spoof IP packets (perhaps your ISP
has enabled egress filtering to prevent IP spoofing), or maybe the target
network recognizes the packet source as bogus and drops them
QUITTING!
```

使用 10.10.6.30 作为空闲僵尸没有成功。如果问题是由于流量过大造成的，他可以在午夜再次尝试。`--packet-trace` 选项结合仔细阅读[“TCP 空闲扫描（`-sI`）”](idlescan.html)一节可以帮助确定为什么 10.10.6.30 不能作为僵尸工作。Demetris 尝试了他在网络上找到的其他几台主机，但没有一个能作为僵尸工作。

Demetris 开始担心他是否能够破解进入 10.10.10.0/24 网络。幸运的是，他在这方面是老手，并且还有另一项技巧——IP 源路由。在互联网的早期（甚至在今天的 IPv6 中），源路由是一项重要且广泛部署的网络诊断功能。它允许您指定数据包到达目标所需经过的跳数，而不是依赖常规路由规则。使用严格源路由，您必须指定每一跳。松散源路由允许您填充关键 IP 航点，而正常的互联网路由则填补这些航点之间的跳数细节。

很久以前，网络社区达成共识，认为源路由带来的麻烦（尤其是安全方面）超过了其价值。许多（如果不是大多数）路由器被配置为丢弃源路由的 IPv4 数据包，因此一些人认为自 90 年代初以来问题已经解决。然而，源路由，就像 SYN 洪水攻击和 Telnet 密码嗅探一样，仍然是一种罕见但潜在的风险。Demetris 通过使用经过 10.10.6.60 邮件服务器松散源路由的数据包对 files2（10.10.6.30）进行 ping 扫描来测试此攻击。结果如[示例 10.14](#fw-megacorp-source-route) 所示。

**示例 10.14 测试源路由**

```sh
# nmap -n -sn -PE --ip-options "L 10.10.6.60" --reason 10.10.6.30

Starting Nmap ( https://nmap.org )
Host 10.10.6.30 appears to be up, received echo-reply.
Nmap done: 1 IP address (1 host up) scanned in .313 seconds
```

Demetris 对测试成功感到既惊讶又高兴。他立即将注意力转向他真正的目标网络，在初始 ping 扫描中添加了一个额外选项：`--ip-options "L 10.10.6.60"`。这一次，Nmap 报告 10.10.10.7 机器有响应。Demetris 了解到之前无法到达它是因为 10.10.10.0/24 和 10.10.5.0/24 子网位于不同的路由器 VLAN 上，这些 VLAN 被配置为防止它们相互通信。Demetris 的源路由技术在那条策略中打开了一个巨大的漏洞！Demetris 随后对 10.10.10.7 机器进行了 SYN 扫描，如[示例 10.15](#fw-megacorp-source-route-success) 所示。

Demetris 在本次初始扫描中省略了操作系统检测和版本检测，但从开放端口配置文件来看，这看起来像是一台 Windows 机器。Demetris 现在可以连接并访问这些端口，只要他使用提供源路由选项的工具，如 Ncat。我不知道故事接下来会发生什么，但我猜涉及 Demetris 完全渗透网络，然后帮助公司更安全地重新设计它。
