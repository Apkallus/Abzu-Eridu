## 入侵检测系统的规避

防火墙并非现代攻击者面临的唯一障碍。入侵检测和防御系统也可能带来问题。网络管理员并不总是喜欢在凌晨 2 点被 IDS 的入侵警报页面吵醒。体贴的黑客会尽量避免让自己的行为引发所有这些警报。第一步是检测是否存在 IDS——许多小公司并没有使用它们。如果怀疑或检测到 IDS，有许多有效的技术可以规避它们。它们分为三类，侵入性各不相同：像攻击者不存在一样避开 IDS、用误导性数据迷惑 IDS，以及利用 IDS 来获得进一步的网络权限或干脆将其关闭。另一种选择是，不关心隐蔽性的攻击者可以在攻击目标网络时完全忽略 IDS。

### 入侵检测系统的检测

在网络管理员和恶意黑客之间永无休止的战斗早期，管理员通过加固系统甚至安装防火墙作为边界屏障来保卫自己的领地。黑客开发了新的工具来穿透或绕过防火墙并利用易受攻击的主机。随着管理员引入入侵检测系统（持续监视可疑活动），军备竞赛进一步升级。当然，攻击者通过设计检测和欺骗 IDS 的系统来回应。虽然入侵检测系统本应是被动设备，但许多可以通过网络被攻击者检测到。

最不显眼的 IDS 是被动监听网络流量而不发送任何数据包的设备。专用的网络分路硬件设备可以确保 IDS **无法** 发送数据包，即使它被攻击者入侵。尽管这种设置具有安全优势，但由于实际考虑，并未广泛部署。现代 IDS 希望能够向中央管理控制台等发送警报。如果这是 IDS 传输的全部内容，风险将很小。但为了提供更广泛的警报数据，它们通常会发起探测，这些探测可能被攻击者看到。

#### 反向探测

IDS 常发起的一种探测是对攻击者 IP 地址的反向 DNS 查询。毕竟，警报中的域名比单纯的 IP 地址更有价值。不幸的是，控制自己 rDNS 的攻击者（这很常见）可以实时查看日志并得知自己被检测到了。这是攻击者提供错误信息（如虚假名称和缓存条目）给请求 IDS 的好时机。

有些 IDS 走得更远，向明显的攻击者发送更具侵入性的探测。当攻击者看到他的目标回扫他时，毫无疑问他已经触发了警报。一些 IDS 会向攻击者发送 Windows NetBIOS 信息请求。ISS BlackICE Defender 就是一个默认这样做的厂商（至少过去是这样）。我编写了一个名为 icepick 的小工具，它发送一个简单的数据包，从监听的 BlackICE 实例生成警报。然后它监视特征性的 NetBIOS 查询并报告发现的任何 BlackICE 安装。人们可以轻松扫描大型网络寻找此 IDS，然后尝试利用本章后面讨论的漏洞。

不满足于仅仅定位 BlackICE 安装或在渗透测试中检测它们，我编写了一个名为 windentd 的简单 Unix 程序，用错误信息回复探测。[图 10.1](#fig-blackice-your-mother) 显示了一个 BlackICE 控制台，其中入侵者因 windentd 和 icepick 而被列为“Your Mother”。这些简单的工具可从 [https://nmap.org/presentations/CanSecWest01/](https://nmap.org/presentations/CanSecWest01/) 获得，但不受支持。

#### 突然的防火墙更改和可疑数据包

许多入侵检测系统已经演变成市场部门所称的入侵防御系统。有些只能像普通 IDS 一样嗅探网络并发送触发的数据包响应。最好的 IPS 系统是内联在网络上的，以便在检测到可疑活动时限制数据包流。例如，IPS 可能会阻止来自其认为已进行端口扫描或尝试缓冲区溢出攻击的 IP 地址的任何进一步流量。如果攻击者端口扫描一个系统，然后无法连接到报告的开放端口，他们很可能会注意到这一点。攻击者可以通过尝试从另一个 IP 地址连接来确认他们被阻止了。

可疑的响应数据包也可能是攻击者行为已被 IDS 标记的提示。特别是，许多 **非** 内联的 IDS 会伪造 RST 数据包，试图拆除连接。确定这些数据包是伪造的方法在[“检测防火墙和入侵检测系统的数据包伪造”](firewall-ids-packet-forgery.html)一节中介绍。

#### 命名约定

命名约定可能是 IDS 存在的另一个泄密点。如果 Nmap 列表扫描返回的主机名如 realsecure、ids-monitor 或 dragon-ids，你可能已经找到了一个入侵检测系统。管理员可能无意中泄露了该信息，或者他们认为这就像房子和汽车窗户上的警报贴纸一样。也许他们认为脚本小子会被与 IDS 相关的名字吓跑。这也可能是错误信息。你永远不能完全信任 DNS 名称。例如，你可能认为 bugzilla.securityfocus.com 是一台运行流行的 Bugzilla 基于 Web 的 bug 跟踪软件的 Web 服务器。事实并非如此。[示例 10.16](#defeating-ids-bugzilla-secfocus) 中的 Nmap 扫描显示它可能是一台 Symantec Raptor 防火墙。没有 Web 服务器可访问，尽管可能有一台隐藏在 Raptor 后面。

#### 无法解释的 TTL 跳变

检测某些 IDS 的另一种方法是观察 traceroute 中无法解释的间隙（或可疑机器）。虽然大多数操作系统都包含 `traceroute` 命令（在 Windows 上缩写为 `tracert`），但 Nmap 通过 `--traceroute` 选项提供了一种更快、更有效的替代方法。与标准的 `traceroute` 不同，Nmap 并行发送探测，并能够根据扫描结果确定哪种探测最有效。在[示例 10.17](#ex-traceroute-jump)（为简单起见而构造）中，traceroute 在第 5 跳没有发现任何东西。那可能是一个保护目标公司的内联 IDS 或防火墙。当然，这只能检测内联 IDS，而无法检测那些不在路由路径中、被动嗅探网络的 IDS。即使是一些内联设备也可能不会被发现，因为它们不会递减 TTL 或拒绝从受保护网络传回 ICMP ttl-exceeded 消息。

### 规避入侵检测系统

击败入侵检测系统最微妙的方法是完全避开它们警惕的目光。现实情况是，管理 IDS 的规则相当脆弱，通常可以通过稍微修改攻击来击败。攻击者拥有几十种技术，从 URL 编码到多态 shellcode 生成器，用于逃避对其漏洞利用的 IDS 检测。本节重点介绍隐蔽端口扫描，这比隐蔽利用漏洞更容易。

#### 放慢速度

在避免 IDS 警报方面，耐心是一种美德。端口扫描检测通常是基于阈值的。系统会监视在特定时间内的探测数量。这有助于防止无辜用户的误报。同时，节省资源也很重要——永远保存连接探测会消耗内存并使实时列表搜索变得太慢。这种阈值方法的缺点是攻击者可以通过将扫描速率保持在阈值以下来规避它。Nmap 提供了几种预设时序模式，可以通过 `-T` 选项选择来实现这一点。例如，`-T paranoid` 选项使 Nmap 一次只发送一个探测，等待五分钟。大型扫描可能需要数周，但至少可能不会被检测到。`-T sneaky` 选项类似，但探测之间只等待 15 秒。

与其指定像 `sneaky` 这样的预设时序模式，可以使用 `--max-parallelism`、`--min-rtt-timeout` 和 `--scan-delay` 等选项精确自定义时序变量。[第 6 章《优化 Nmap 性能》](performance.html)深入描述了这些选项。

##### 实际示例：绕过默认 Snort 2.2.0 规则

研究方便的开放源代码 Snort IDS 提供了关于在雷达下潜行的教训。Snort 已经有几代端口扫描检测器。Flow-Portscan 模块相当强大。能够绕过它的扫描很可能也能逃脱许多其他 IDS 的检测。

Flow-portscan 由两个检测系统组成，它们可以协同工作（或单独启用）来检测端口扫描器。该系统及其数十个配置变量在 Snort 发行版的 `docs/README.flow-portscan` 中有文档记录，但我会提供一个快速总结。

Flow-portscan 中较简单的检测方法称为**固定时间尺度**。它只是监视在 `scanner-fixed-window` 秒内是否有 `scanner-fixed-threshold` 个探测数据包。这两个变量在 `snort.conf` 中设置，默认值均为 15。请注意，计数器包括从单台机器发送到受保护网络上任何主机的任何探测。因此，快速扫描 15 台受保护机器上的单个端口，就像扫描单台机器上的 15 个端口一样肯定会生成警报。

如果这是唯一的检测方法，解决方案将非常简单。传递 `--scan-delay 1075ms` 选项以确保 Nmap 在发送探测之间等待 1.075 秒。直观的选择可能是每秒一个数据包以避免在 15 秒内发送 15 个数据包，但这还不够。发送第一个数据包和第十五个数据包之间只有 14 次等待，因此等待时间必须至少为 15/14，即 1.07143 秒。选择 `--scan-delay 1000ms` 的某个倒霉蛋会极大地减慢扫描速度，同时仍然会触发警报。如果网络上的多个主机正在被探测，必须分别扫描它们以避免触发警报。`--max-hostgroup 1` 选项将确保一次只扫描一个主机，但并不完全安全，因为它不会在向一个主机发送的最后一个探测和向下一个主机发送的第一个探测之间强制执行 `--scan-delay`。只要每台主机至少扫描 15 个端口，你可以通过将 `--scan-delay` 至少设置为 1155 毫秒来补偿，或者简单地从一个 shell 脚本启动单目标 Nmap 实例，并在它们之间等待 1075 毫秒。[示例 10.19](#defeating-ids-slow-nmap-scan) 显示了对网络上多台机器的这种隐蔽扫描。使用 Bash shell 语法处理多个 Nmap 实例。这里手动指定了 IP。如果需要许多目标，可以使用 `-sL`（列表扫描）选项将其枚举到一个文件中，然后使用普通的 shell 循环对每个目标启动 Nmap。这些扫描每个端口花费超过 1.075 秒的原因是，对于过滤端口需要重传，以确保它们不是因为网络拥塞而丢失。

不幸的是，对于端口扫描爱好者来说，击败 Snort 并非如此简单。它还有另一种检测方法，称为**滑动时间尺度**。此方法类似于刚刚讨论的固定窗口方法，不同之处在于每当检测到来自主机的新的探测时，它会增加窗口。如果在窗口期间检测到 `scanner-sliding-threshold` 个探测，就会发出警报。窗口从 `scanner-sliding-window` 秒开始，并为检测到的每个探测增加窗口内经过的时间乘以 `scanner-sliding-scale-factor`。这三个变量在 `snort.conf` 中的默认值分别为 40 个探测、20 秒和因子 0.5。

滑动尺度随着新数据包的进入而不断增长，这是一种相当隐蔽的方式。最简单的（如果慢的话）解决方案是每 20.1 秒发送一个探测。这将同时规避默认的固定和滑动尺度。可以像[示例 10.19](#defeating-ids-slow-nmap-scan) 那样做，但使用更高的值。你可以通过快速发送 14 个数据包，等待 20 秒让窗口过期，然后用另外 14 个探测重复，将速度提高一个数量级。你可以使用 shell 脚本控制 Nmap 来做到这一点，但为此定制任务编写自己的简单 SYN 扫描程序可能更可取。

#### 在网络间分散探测，而不是连续扫描主机

如前所述，IDS 通常被设置为只有在达到可疑活动阈值时才发出警报。这个阈值通常是全局的，适用于 IDS 保护的整个网络，而不仅仅是单个主机。有时它们专门监视来自给定源地址到连续主机的流量。如果一个主机发送 SYN 数据包到主机 10.0.0.1 的 139 端口，这本身并不太可疑。但是，如果该探测之后是到 10.0.0.2、.3、.4 和 .5 的类似数据包，则明显指示了端口扫描。

避免触发这些警报的一种方法是在大量主机之间分散探测，而不是连续扫描它们。有时你可以避免在同一网络上扫描太多主机。如果你只是为了研究调查，可以考虑使用 `-iR` 在整个互联网上分散探测，而不是扫描一个大型网络。无论如何，结果可能更具代表性。

在大多数情况下，你想扫描特定网络，而互联网范围的采样是不够的。避免连续主机探测警报很容易。Nmap 提供 `--randomize-hosts` 选项，它将目标网络分成 16384 个 IP 的块，然后随机化每个块中的主机。如果你扫描的是一个巨大的网络，比如 B 类或更大，通过随机化更大的块可能会获得更好（更隐蔽）的结果。你可以通过增加 `nmap.h` 中的 `PING_GROUP_SZ` 然后重新编译来实现这一点。`--randomize-hosts` 扫描中使用的块大小是 `PING_GROUP_SZ` 值的四倍。请注意，较高的 `PING_GROUP_SZ` 值会占用更多主机内存。另一种解决方案是使用列表扫描（`-sL -n -oN <filename>`）生成目标 IP 列表，用 Perl 脚本随机化，然后使用 `-iL` 将整个列表提供给 Nmap。如果你正在扫描像 10.0.0.0/8 这样的大网络并希望随机化所有 1600 万个 IP 地址，你可能必须使用这种方法。

#### 数据包分片

IP 分片可能是入侵检测系统的主要问题，特别是因为对重叠分片和分片重组超时等异常的处理方式不明确，并且在不同平台之间存在很大差异。因此，IDS 通常不得不猜测远程系统将如何解释数据包。分片重组也消耗大量资源。由于这些原因，许多入侵检测系统仍然不能很好地支持分片。指定 `-f` 可使端口扫描使用微小的（8 个数据字节或更少）IP 分片。更多重要细节请参见[“分片”](firewall-subversion.html#defeating-firewalls-fragmentation)一节。

#### 规避特定规则

大多数 IDS 厂商吹嘘他们支持多少警报，但许多（如果不是大多数）很容易被绕过。Nmap 用户中最流行的 IDS 是开源的 [Snort](http://www.snort.org)。[示例 10.20](#defeating-ids-snortrules) 显示了 Snort 2.0.0 中所有引用 Nmap 的默认规则。


现在让我们从攻击者的角度审视这些规则。第一条规则查找没有有效载荷（`dsize:0`）的 ICMP ping 数据包。只需指定一个非零的 `--data-length` 选项即可击败该规则。或者用户完全可以指定不同类型的 ping 扫描，例如 TCP SYN ping。

下一条规则查找设置了 FIN、PSH 和 URG 标志的 TCP 数据包（flags:FPU），并发出 Nmap Xmas 扫描警报。在 Xmas 扫描选项中添加 `--scanflags FINPSH` 选项将移除 URG 标志。扫描仍将按预期工作，但规则将无法触发。

列表中的第三条规则查找设置了 ACK 位但确认号为零的 TCP 数据包（flags:A;ack:0）。Nmap 的旧版本有这种行为，但在 1999 年为响应 Snort 规则而修复了。

第四条规则查找设置了 SYN、FIN、PSH 和 URG 标志的 TCP 数据包（flags:SFPU）。然后它声明了 Nmap 操作系统指纹尝试。攻击者可以通过省略 `-O` 标志来避免触发此规则。如果他确实想进行操作系统检测，可以在 `osscan2.cc` 中注释掉该单个测试。操作系统检测仍然相当准确，但不会触发 IDS 警报。

最后一条规则查找向 Web 服务器发送字符串 `nmap ` 的人。他们在寻找通过 Web 服务器执行命令的尝试。攻击者可以通过重命名 Nmap、使用制表符而不是空格，或者如果可用则使用 SSL 加密连接来击败这一点。

当然，还有其他相关的规则，名字中没有 Nmap，但仍然可能被侵入性端口扫描触发。高级攻击者在自己网络上安装他们关心的 IDS，然后提前修改和测试扫描，以确保它们不会触发警报。

在此示例中选择 Snort 仅仅是因为其规则数据库是公开的，并且它是一个同为开源的网络安全工具。商业 IDS 也存在类似问题。

#### 避免容易检测的 Nmap 功能

Nmap 的某些功能比其他功能更显眼。特别是，版本检测会连接到许多不同的服务，这常常会在这些机器上留下日志并在入侵检测系统上触发警报。操作系统检测也很容易被入侵检测系统发现，因为一些测试使用了相当不寻常的数据包和数据包序列。[示例 10.20“引用 Nmap 的默认 Snort 规则”](#defeating-ids-snortrules)中显示的 Snort 规则演示了典型的 Nmap 操作系统检测签名。

希望保持隐蔽的渗透测试人员的一个解决方案是完全跳过这些显眼的探测。服务和操作系统检测很有价值，但对于成功攻击并非必需。它们也可以根据具体情况针对看起来有趣的主机或端口使用，而不是用它们探测整个目标网络。

### 误导入侵检测系统

上一节讨论了使用巧妙手段避开入侵检测系统的警惕目光。另一种方法是使用数据包伪造主动误导或迷惑 IDS。Nmap 提供了许多选项来实现这一点。

#### 诱饵

街头犯罪分子知道，犯罪后避免当局追捕的一种有效方法是混入附近的人群。警察可能无法从所有无辜的路人中分辨出扒手。在网络领域，Nmap 可以构建一个看似来自世界各地数十台主机的扫描。目标将难以确定哪个主机代表攻击者，哪些是无辜的诱饵。虽然可以通过路由追踪、响应丢弃和其他主动机制来击败这一点，但它通常是隐藏扫描源的有效技术。[图 10.2](#fig-blackice-decoys) 显示了一个被数十个诱饵淹没的 BlackICE 报告屏幕。管理员不能向列表中的每个 ISP 提供商投诉。那会花费很长时间，而且除了一个主机外，其他都是无辜的。

**图 10.2 被数十个诱饵掩盖的攻击者**

![被数十个诱饵掩盖的攻击者](images/BlackICE_Decoys.png)

> **警告**  
> 许多零售（拨号、电缆调制解调器、DSL 等）ISP 会过滤掉大多数伪造数据包，尽管来自与您相同网络范围的伪造数据包可能会通过。首先在您控制的通过互联网连接的一些机器上进行测试，或者您甚至可以使用类似于[“IP ID 技巧”](determining-firewall-rules.html#defeating-firewalls-ipid-tricks)一节中讨论的 IP ID 技巧，针对第三方服务器进行测试。

诱饵通过 `-D` 选项添加。参数是一个主机列表，用逗号分隔。字符串 `ME` 可以作为诱饵之一，表示真实源主机应在扫描顺序中出现的位置。否则它将是随机位置。将 `ME` 包含在列表的第 6 个或更靠后的位置可以防止一些常见的端口扫描检测器报告该活动。例如，Solar Designer 优秀的 Scanlogd 只报告前五个扫描源，以避免日志被诱饵淹没。

您还可以使用 `RND` 请求一个随机的非保留 IP 地址，或者使用 `RND:<number>` 生成 `<number>` 个随机地址。

请注意，用作诱饵的主机应该在线且运行。如果网络上只有一台主机实际在线，则很容易确定哪台主机在扫描。使用太多离线的诱饵也可能导致目标端口暂时无响应，这是由于一种称为 SYN 洪泛的情况。建议使用 IP 地址而不是名称，以避免出现在诱饵网络的名称服务器日志中。理想情况下，目标本身也应通过 IP 地址表示。

诱饵既用于初始 ping 扫描（使用 ICMP、SYN、ACK 等），也用于实际端口扫描阶段。诱饵也用于远程操作系统检测。它们不用于 DNS 查询或服务/版本检测，因此如果您使用诸如 `-sV` 或 `-A` 之类的选项，您会暴露自己。使用太多诱饵会极大地减慢扫描速度，有时甚至使其不那么准确。

#### 端口扫描欺骗

虽然一大群诱饵在隐藏端口扫描的真实来源方面非常有效，但 IDS 警报会清楚地表明有人在使用诱饵。一种更微妙但有限的方法是欺骗来自单个地址的端口扫描。指定 `-S` 后跟一个源 IP，Nmap 将从该给定源发起请求的端口扫描。由于目标会响应欺骗的 IP，而 Nmap 将看不到这些响应，因此无法获得有用的 Nmap 结果。目标的 IDS 警报会将扫描归咎于欺骗的源。您可能需要指定 `-e <interfacename>` 来选择适当的接口名称（如 eth0、ppp0 等），以便 Nmap 通过它发送伪造的数据包。这对于陷害无辜方、使管理员对 IDS 的准确性产生怀疑以及[“对响应系统的 DoS 攻击”](#defeating-ids-dos-attack)一节中讨论的拒绝服务攻击非常有用。

#### 空闲扫描

空闲扫描是一种巧妙的技巧，允许在仍然获得准确 TCP 端口扫描结果的同时欺骗源 IP 地址，如上一节所述。这是通过滥用许多系统实现的 IP 标识字段的属性来完成的。在[“TCP 空闲扫描（`-sI`）”](idlescan.html)一节中有更深入的描述。

#### DNS 代理

即使是最精心设计的计划也可能因一个被忽视的小细节而失败。如果计划涉及超隐蔽端口扫描，那个小细节可能就是 DNS。如[“DNS 解析”](host-discovery-dns.html)一节所述，Nmap 默认对每个响应的主机执行反向 DNS 解析。如果目标网络管理员是偏执的记录一切类型，或者他们有一个极其敏感的 IDS，这些 DNS 查找探测可能会被检测到。即使是像列表扫描（`-sL`）这样无侵入性的操作也可能以这种方式被检测到。探测将来自为运行 Nmap 的机器配置的 DNS 服务器。这通常是由您的 ISP 或组织维护的单独机器，尽管有时是您自己的系统。

消除此风险的最有效方法是指定 `-n` 禁用所有反向 DNS 解析。这种方法的问题在于您丢失了 DNS 提供的宝贵信息。幸运的是，Nmap 提供了一种在隐藏源的同时收集此信息的方法。互联网上很大一部分 DNS 服务器对任何人的递归查询都是开放的。向 Nmap 的 `--dns-servers` 选项指定一个或多个此类名称服务器，所有 rDNS 查询将通过它们进行代理。[示例 10.21](#defeating-firewalls-dns-proxy-ex) 演示了此技术，使用公共递归 DNS 服务器 `4.2.2.1` 和 `4.2.2.2` 对 SecurityFocus 的一些 IP 进行列表扫描，以隐藏任何痕迹。请记住，正向 DNS 仍然使用您主机的配置 DNS 服务器，因此指定目标 IP 地址而不是域名，以防止即使是很小的潜在信息泄漏。出于这个原因，[示例 10.21](#defeating-firewalls-dns-proxy-ex) 首先显示了 Linux `host` 命令用于查找 `www.securityfocus.com`，而不是在 Nmap 命令行中指定该主机名。为了避免基于来自单个 DNS 服务器的请求数量的 IDS 阈值，您可以向 `--dns-servers` 指定数十个逗号分隔的 DNS 服务器，Nmap 将在它们之间轮询其请求。

### 对响应系统的 DoS 攻击

许多厂商正在推广他们所谓的入侵**防御**系统。这些基本上是可以主动阻止流量并重置被视为恶意的已建立连接的 IDS。它们通常是内联在网络上或基于主机，以更好地控制网络活动。其他（非内联）系统进行混杂监听并尝试通过伪造 TCP RST 数据包来处理可疑连接。除了试图阻止各种可疑活动的传统 IPS 厂商外，许多流行的小程序如 [Port Sentry](http://sourceforge.net/projects/sentrytools/) 专门设计用于阻止端口扫描器。

虽然阻止端口扫描器乍看似乎是个好主意，但这种方法存在许多问题。最明显的是，如前几节所示，端口扫描通常很容易伪造。攻击者也通常很容易判断这种扫描阻止软件是否到位，因为他们在进行端口扫描后将无法连接到声称开放的端口。他们会从另一个系统再次尝试并成功连接，确认原始 IP 被阻止了。然后攻击者可以使用前面讨论的主机欺骗技术（`-S` 选项）使目标主机阻止攻击者想要的任何系统。这可能包括重要的 DNS 服务器、主要网站、软件更新档案、邮件服务器等。很可能不需要很长时间就能使合法的管理员感到厌烦，从而禁用响应式阻止。虽然大多数此类产品提供白名单选项以防止阻止某些重要主机，但枚举所有这些主机是非常困难的。攻击者通常可以找到一个新的常用主机来阻止，激怒用户，直到管理员确定问题并相应调整白名单。

### 利用入侵检测系统

破坏入侵检测系统最大胆的方法是攻击它们。许多商业和开源厂商在产品可利用性方面的安全记录都很可怜。Internet Security System 的旗舰产品 `RealSecure` 和 `BlackICE` IDS 产品存在一个漏洞，使得 Witty 蠕虫能够感染超过一万个安装，然后通过破坏其文件系统来禁用 IDS。其他 IDS 和防火墙厂商如 Cisco、Checkpoint、Netgear 和 Symantec 也遭受了严重的远程可利用漏洞。开源嗅探器也没有好多少，在 `Snort`、`Wireshark`、`tcpdump`、`FakeBO` 等中发现了可利用的漏洞。以安全高效的方式进行协议解析极其困难，而且大多数应用程序需要解析数百种协议。导致 IDS 崩溃（通常用一个数据包）的拒绝服务攻击比这些权限提升漏洞更为常见。崩溃的 IDS 将无法检测任何 Nmap 扫描。

鉴于所有这些漏洞，利用 IDS 可能是进入目标网络的最可行方式。这种方法的一个好处是，你甚至不必找到 IDS。向网络上任何“受保护”的机器发送一个恶意数据包通常就足以触发这些 IDS 错误。

### 忽略入侵检测系统

虽然高级攻击者通常会采用本章描述的 IDS 规避技术，但更为常见的初级攻击者（脚本小子）很少关心 IDS。许多公司甚至不部署 IDS，而那些部署的公司通常配置不当或很少关注警报。面向互联网的 IDS 会看到来自脚本小子和蠕虫的大量攻击，因此定位易受攻击服务的一些 Nmap 扫描不太可能引起任何注意。

即使这样的攻击者入侵了网络，被监控的 IDS 检测到，然后被踢出系统，这也是一个小小的损失。黑客对他们来说常常是数字游戏，因此在成千上万个网络中失去一个被入侵的网络是微不足道的。这样一个巡逻良好的网络可能很快就会注意到他们的使用（如拒绝服务攻击、大规模扫描或发送垃圾邮件）并关闭他们。黑客想要入侵的是那些管理疏忽、监控不佳的网络，这些网络将为犯罪活动提供持久的节点。

被追踪和起诉很少是忽略 IDS 的攻击者所关心的问题。他们通常从其他被入侵的网络发起攻击，这些网络通常距离他们的真实位置有多个跨洲跳数。或者他们可能使用匿名连接，如某些网吧、学校计算机实验室、图书馆或普遍存在的开放无线接入点提供的连接。一次性拨号帐户也常被使用。即使被踢出，再用另一个（或同一个）提供商注册只需要几分钟。许多攻击者来自罗马尼亚、中国、韩国和其他极不可能被起诉的国家。

互联网蠕虫是另一类很少费心进行 IDS 规避的攻击。蠕虫和脚本小子都喜欢无耻地扫描数百万个 IP 地址，因为这比强调隐蔽的谨慎、有针对性的方法每小时带来更多的入侵。

虽然大多数攻击不努力隐蔽，但大多数入侵检测系统如此容易被规避的事实是一个主要问题。熟练的攻击者占少数，但往往是最严重的威胁。不要被 IDS 喷出的大量警报所麻痹。它们无法检测到所有事情，而且常常错过最重要的东西。

即使是熟练的黑客有时也会在初始侦察中忽略 IDS 的担忧。他们只是从某个无法追踪的 IP 地址扫描，希望与互联网上所有其他攻击者和探测流量混在一起。在分析结果后，他们可能会从其他系统发起更谨慎、更隐蔽的攻击。
