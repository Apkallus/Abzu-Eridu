## 确定防火墙规则

绕过防火墙规则的第一步是理解它们。在可能的情况下，Nmap 会区分可访问但关闭的端口与主动被过滤的端口。一种有效的技术是从正常的 SYN 端口扫描开始，然后转向更特殊的技术，如 ACK 扫描和 IP ID 序列分析，以更好地了解网络。

### 标准 SYN 扫描

TCP 协议的一个有用特性是，根据 RFC 793，系统必须对意外的连接请求以 TCP RST（复位）数据包的形式发送否定响应。RST 数据包使得 Nmap 能够轻松识别关闭的端口。另一方面，防火墙等过滤设备往往会丢弃发往不允许端口的数据包。在某些情况下，它们会发送 ICMP 错误消息（通常是端口不可达）。由于丢弃的数据包和 ICMP 错误与 RST 数据包容易区分，Nmap 可以可靠地将被过滤的 TCP 端口与开放或关闭的端口区分开来，并且它会自动执行此操作。示例 10.1 展示了这一点。

**示例 10.1 检测关闭和过滤的 TCP 端口**

```sh
# nmap -sS -T4 scanme.nmap.org

Starting Nmap ( https://nmap.org )
Nmap scan report for scanme.nmap.org (64.13.134.52)
Not shown: 994 filtered ports
PORT    STATE  SERVICE
22/tcp  open   ssh
25/tcp  closed smtp
53/tcp  open   domain
70/tcp  closed gopher
80/tcp  open   http
113/tcp closed auth

Nmap done: 1 IP address (1 host up) scanned in 5.40 seconds
```

示例 10.1 中最重要的行之一是“Not shown: 994 filtered ports”（未显示：994 个过滤端口）。换句话说，该主机具有恰当的默认拒绝防火墙策略。只有管理员明确允许的那些端口是可访问的，而默认操作是拒绝（过滤）它们。列举的端口中有三个处于开放状态（22、53 和 80），另外三个处于关闭状态（25、70 和 113）。其余 994 个被测试的端口无法通过此标准扫描到达（被过滤）。

#### 返回 RST 的狡猾防火墙

虽然 Nmap 区分关闭 TCP 端口（返回 RST 数据包）和过滤端口（不返回任何内容或返回 ICMP 错误）通常是准确的，但现在许多防火墙设备能够伪造 RST 数据包，就好像它们来自目标主机并声称端口已关闭。Linux iptables 系统就是这种能力的一个例子，它提供了多种拒绝不需要的数据包的方法。iptables 手册页将此功能描述如下：

> --reject-with *type*
>
> 给定的 *type* 可以是 icmp-net-unreachable、icmp-host-unreachable、icmp-port-unreachable、icmp-proto-unreachable、icmp-net-prohibited 或 icmp-host-prohibited，它们返回相应的 ICMP 错误消息（port-unreachable 是默认值）。选项 tcp-reset 可用于仅匹配 TCP 协议的规则：这会导致发送回一个 TCP RST 数据包。这主要用于阻止 ident（`113/tcp`）探测，当向有问题的邮件主机发送邮件时（否则这些主机不会接受您的邮件），此类探测经常发生。

在端口 113 之外，防火墙和 IDS/IPS 伪造 RST 数据包并不特别常见，因为它可能使合法的网络操作员感到困惑，而且它允许扫描仪立即转到下一个端口，而无需等待由丢包引起的超时。尽管如此，这种情况确实会发生。通过将 RST 数据包与机器发送的其他数据包进行仔细比较，通常可以检测到这种伪造。“检测防火墙和入侵检测系统的数据包伪造”一节描述了有效的检测技术。

### ACK 扫描

如“TCP ACK 扫描（`-sA`）”一节中详细描述的那样，ACK 扫描发送仅设置 ACK 位的 TCP 数据包。无论端口是开放还是关闭，根据 RFC 793，目标都需要以 RST 数据包响应。另一方面，阻止探测的防火墙通常不响应或发回 ICMP 目标不可达错误。这种区别使得 Nmap 能够报告 ACK 数据包是否被过滤。与针对同一机器的 SYN 扫描相比，Nmap ACK 扫描报告的过滤端口集通常更小，因为 ACK 扫描更难过滤。许多网络允许几乎不受限制的出站连接，但希望阻止互联网主机向它们发起连接。阻止入站 SYN 数据包（未设置 ACK 位）是一种简单的方法，但它仍然允许任何 ACK 数据包通过。阻止这些 ACK 数据包更加困难，因为它们无法说明哪一方发起了连接。为了阻止未经请求的 ACK 数据包（如 Nmap ACK 扫描发送的），同时允许属于合法连接的 ACK 数据包，防火墙必须有状态地监视每个已建立的连接，以确定给定的 ACK 是否合适。这些状态防火墙通常更安全，因为它们可以更具限制性。阻止 ACK 扫描是一项额外的可用限制。缺点是它们需要更多的资源来运行，并且状态防火墙重启可能导致设备丢失状态并终止所有通过它的已建立连接。

虽然状态防火墙很普遍且日益流行，但无状态方法仍然相当常见。例如，Linux Netfilter/iptables 系统支持 `--syn` 便利选项，使上述无状态方法易于实现。

在上一节中，SYN 扫描显示 scanme.nmap.org 上 1,000 个常见端口中除了 6 个之外的所有端口都处于过滤状态。示例 10.2 演示了针对同一主机的 ACK 扫描，以确定它是否使用状态防火墙。

**示例 10.2 针对 Scanme 的 ACK 扫描**

```sh
# nmap -sA -T4 scanme.nmap.org

Starting Nmap ( https://nmap.org )
Nmap scan report for scanme.nmap.org (64.13.134.52)
Not shown: 994 filtered ports
PORT    STATE      SERVICE
22/tcp  unfiltered ssh
25/tcp  unfiltered smtp
53/tcp  unfiltered domain
70/tcp  unfiltered gopher
80/tcp  unfiltered http
113/tcp unfiltered auth

Nmap done: 1 IP address (1 host up) scanned in 5.96 seconds
```

这里显示了 SYN 扫描中出现的相同六个端口。其他 994 个仍然被过滤。这是因为 Scanme 受此状态 iptables 指令保护：`iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT`。这仅接受属于或与已建立连接相关的数据包。Nmap 发送的未经请求的 ACK 数据包会被丢弃，但向所示的六个特殊端口发送的除外。特殊规则允许所有发往端口 22、25、53、70 和 80 的数据包，以及响应端口 113 探测而发送 RST 数据包。请注意，显示的六个端口处于 `unfiltered` 状态，因为 ACK 扫描无法进一步将它们划分为 `open`（22、53 和 80）或 `closed`（25、70、113）。

现在让我们看另一个例子。我本地网络上名为 Para 的 Linux 主机使用以下（为节省空间而简化）防火墙脚本：

```sh
#!/bin/sh
#
# A simple, stateless, host-based firewall script.

# First of all, flush & delete any existing tables
iptables -F
iptables -X

# Deny by default (input/forward)
iptables --policy INPUT DROP
iptables --policy OUTPUT ACCEPT
iptables --policy FORWARD DROP

# I want to make ssh and www accessible from outside
iptables -A INPUT -m multiport -p tcp --destination-port 22,80 -j ACCEPT

# Allow responses to outgoing TCP requests
iptables -A INPUT --proto tcp ! --syn -j ACCEPT
```

此防火墙是无状态的，因为没有 `--state` 选项或 `-m state` 模块请求的迹象。示例 10.3 显示了针对此主机的 SYN 和 ACK 扫描。

**示例 10.3 针对 Para 的 SYN 和 ACK 扫描对比**

```sh
# nmap -sS -p1-100 -T4 para

Starting Nmap ( https://nmap.org )
Nmap scan report for para (192.168.10.191)
Not shown: 98 filtered ports
PORT   STATE  SERVICE
22/tcp open   ssh
80/tcp closed http
MAC Address: 00:60:1D:38:32:90 (Lucent Technologies)

Nmap done: 1 IP address (1 host up) scanned in 3.81 seconds

# nmap -sA -p1-100 -T4 para

Starting Nmap ( https://nmap.org )
All 100 scanned ports on para (192.168.10.191) are: unfiltered
MAC Address: 00:60:1D:38:32:90 (Lucent Technologies)

Nmap done: 1 IP address (1 host up) scanned in 0.70 seconds
```

在 SYN 扫描中，100 个端口中有 98 个被过滤。然而，ACK 扫描显示每个扫描的端口都是 `unfiltered`。换句话说，所有 ACK 数据包都在不受阻碍地悄悄通过并引发 RST 响应。这些响应也使扫描速度快了五倍以上，因为它不必等待超时。

现在我们知道了如何区分状态防火墙和无状态防火墙，但这有什么用呢？对 Para 的 ACK 扫描表明，某些数据包可能到达了目标主机。我说“可能”是因为防火墙伪造总是可能的。虽然你可能无法建立到这些端口的 TCP 连接，但它们可用于确定哪些 IP 地址正在使用、操作系统检测测试、某些 IP ID 操作，以及作为向这些机器上安装的 rootkit 隧道传输命令的通道。其他扫描类型，如 FIN 扫描，甚至可能确定哪些端口是开放的，从而推断主机的用途。此类主机可能可用作 IP ID 空闲扫描的僵尸。

这一对扫描还表明，我们所谓的端口状态并不仅仅是端口本身的属性。在这里，相同的端口号被一种扫描类型视为 `filtered`，而被另一种视为 `unfiltered`。你从哪个 IP 地址扫描、沿途任何过滤设备的规则以及你访问目标机器的哪个接口，都会影响 Nmap 如何看待这些端口。端口表仅反映 Nmap 在特定时间、使用特定选项集从特定机器运行时看到的情况。

### IP ID 技巧

IP 报头中不起眼的标识字段可以泄露惊人的信息。本章后面将用它进行端口扫描（空闲扫描技术）以及检测防火墙和入侵检测系统何时伪造 RST 数据包，就好像它们来自受保护的主机。另一个巧妙的技巧是辨别哪些源地址能够通过防火墙。如果沿途某个防火墙丢弃了所有来自 192.168.0.1 的欺骗数据包，那么花费数小时进行盲目的欺骗攻击就没有意义。

我通常使用 Nmap 附带的免费网络探测工具 Nping 来测试这种情况。这是一种相当复杂的技术，但有时可能很有价值。以下是我采取的步骤：

1. **找到内部网络上某台机器的至少一个可访问（开放或关闭）端口。** 路由器、打印机和 Windows 机器通常效果不错。最近发布的 Linux、Solaris 和 OpenBSD 在很大程度上解决了 IP ID 序列号可预测的问题，因此不适用。选择的机器应该网络流量较少，以避免结果混乱。

这个受信任源地址问题的一个具体例子是，我曾经发现一家公司的自定义 UDP 服务允许用户跳过身份验证——如果他们来自配置文件中输入的特殊网络块。这些网络块对应不同的公司位置，该功能旨在简化管理和调试。他们面向互联网的防火墙明智地试图阻止这些地址，因为实际员工可以通过私有链接访问生产环境。但是，通过使用本节中描述的技术，我发现防火墙与配置文件不完全同步。有几个地址我可以成功伪造 UDP 控制消息并接管他们的应用程序。

这种映射防火墙规则的技术不使用 Nmap，但其结果对未来的运行很有价值。例如，这个测试可以显示是否使用某些诱饵（`-D`）。最好的诱饵能够一路到达目标系统。此外，伪造的数据包必须能够通过，IP ID 空闲扫描（稍后讨论）才能工作。用这种技术测试潜在的源 IP 通常比在网络中查找和测试每个潜在的空闲代理机器更容易。潜在的闲置代理只需通过上面的第二步即可测试。

### UDP 版本扫描

前面的部分都集中在流行的 TCP 协议上。使用 UDP 通常更困难，因为该协议不像 TCP 那样提供开放端口的确认。许多 UDP 应用程序会简单地忽略意外的数据包，使 Nmap 无法确定端口是开放还是被过滤。因此，Nmap 将这些模糊的端口置于 `open|filtered` 状态，如示例 10.4 所示。

**示例 10.4 针对防火墙主机的 UDP 扫描**

```sh
# nmap -sU -p50-59 scanme.nmap.org

52/udp open|filtered xns-time
53/udp open|filtered domain
54/udp open|filtered xns-ch
55/udp open|filtered isi-gl
56/udp open|filtered xns-auth
57/udp open|filtered priv-term
58/udp open|filtered xns-mail
59/udp open|filtered priv-file

Nmap done: 1 IP address (1 host up) scanned in 1.38 seconds
```

这个 10 端口的扫描帮助不大。没有端口响应探测数据包，因此它们都被列为开放或过滤。更好地理解哪些端口实际上开放的一种方法是发送大量针对几十种不同已知 UDP 服务的 UDP 探测，以期从任何开放端口获得响应。Nmap 版本检测（第 7 章《服务和应用程序版本检测》）正是这样做的。示例 10.5 显示了相同的扫描，但添加了版本检测（`-sV`）。

**示例 10.5 针对防火墙主机的 UDP 版本扫描**

```sh
# nmap -sV -sU -p50-59 scanme.nmap.org

Starting Nmap ( https://nmap.org )
Nmap scan report for scanme.nmap.org (64.13.134.52)
PORT   STATE         SERVICE    VERSION
50/udp open|filtered re-mail-ck
51/udp open|filtered la-maint
52/udp open|filtered xns-time
53/udp open          domain     ISC BIND 9.3.4
54/udp open|filtered xns-ch
55/udp open|filtered isi-gl
56/udp open|filtered xns-auth
57/udp open|filtered priv-term
58/udp open|filtered xns-mail
59/udp open|filtered priv-file

Nmap done: 1 IP address (1 host up) scanned in 56.59 seconds
```

版本检测毫无疑问地表明端口 53（domain）是开放的，甚至显示了它在运行什么。其他端口仍然是 `open|filtered`，因为它们没有响应任何探测。它们很可能是被过滤的，尽管这不能保证。它们可能正在运行像 SNMP 这样的服务，该服务仅响应具有正确团体字符串的数据包。或者它们可能正在运行一个没有 Nmap 版本检测探针的 obscure 或自定义 UDP 服务。另请注意，这次扫描花费的时间是前一次扫描的 40 倍以上。向每个端口发送所有这些探测是一个相对缓慢的过程。添加 `--version-intensity 0` 选项将显著减少扫描时间，因为只发送最有可能从给定端口号的服务获得响应的探测。
