# 第2章 保持你的成功

即使你理解了本书中的技术信息，你可能仍然难以把握漏洞赏金计划的细微差别。或者你可能正在努力寻找真正的漏洞，却不确定自己为何卡住了。

在本章中，我们将探讨成为一名成功的漏洞赏金猎人所涉及的一些因素。我们将涵盖如何撰写一份能向安全团队恰当描述你发现的报告，如何与你合作的组织建立持久的关系，以及如何在寻找漏洞的过程中克服障碍。

## 撰写一份好报告

漏洞赏金猎人的工作不仅仅是发现漏洞；还要向组织的安全团队解释这些漏洞。如果你提供一份写得好的报告，你将帮助你合作的团队重现漏洞，将其分配给合适的内部工程团队，并更快地修复问题。漏洞修复得越快，恶意黑客利用它的可能性就越小。在本节中，我将分解一份好的漏洞报告的组成部分，并介绍我在此过程中学到的一些技巧和窍门。

### 步骤1：撰写描述性标题

一份优秀漏洞报告的第一部分总是一个描述性的标题。目标是写一个能用一句话总结问题的标题。理想情况下，它应该能让安全团队立即了解漏洞是什么、发生在哪里以及其潜在严重性。为此，它应该回答以下问题：你发现的漏洞是什么？它是否是众所周知的漏洞类型（如 IDOR 或 XSS）的实例？你在目标应用程序的哪个位置找到它的？

例如，不要使用像“关键端点上的 IDOR”这样的报告标题，而应使用类似“https://example.com/change_password 上的 IDOR 导致所有用户账户被接管”的标题。你的目标是让阅读你报告的安全工程师对你将在报告其余部分讨论的内容有一个清晰的认识。

### 步骤2：提供清晰的摘要

接下来，提供报告摘要。本节包含你无法在标题中传达的所有相关细节，例如攻击中使用的 HTTP 请求参数、你是如何发现它的，等等。

以下是一个有效的报告摘要示例：

> `https://example.com/change_password` 端点接受两个 POST body 参数：`user_id` 和 `new_password`。对该端点的 POST 请求会将 `user_id` 用户的密码更改为 `new_password`。该端点没有验证 `user_id` 参数，因此任何用户都可以通过操纵 `user_id` 参数来更改任何其他人的密码。

一份好的报告摘要清晰简洁。它包含理解漏洞所需的所有信息，包括漏洞是什么、漏洞在哪里发现，以及攻击者利用它可以做什么。

### 步骤3：包含严重性评估

你的报告还应包含对漏洞严重性的诚实评估。安全团队除了与你合作修复漏洞外，还有其他职责要处理。包含严重性评估将帮助他们优先修复哪些漏洞，并确保他们立即处理严重漏洞。

你可以使用以下等级来传达严重性：

- **低严重性**：该漏洞没有造成很大损害的潜力。例如，一个只能用于网络钓鱼的开放重定向就是一个低严重性漏洞。
- **中严重性**：该漏洞以中等程度影响用户或组织，或者是一个恶意黑客难以利用的高严重性问题。安全团队应首先关注高严重性和严重性漏洞。例如，在更改密码等敏感操作上的跨站请求伪造（CSRF）通常被视为中严重性问题。
- **高严重性**：该漏洞影响大量用户，其后果对这些用户可能是灾难性的。安全团队应尽快修复高安全性漏洞。例如，一个可用于窃取 OAuth 令牌的开放重定向就是一个高严重性漏洞。
- **严重**：该漏洞影响大多数用户群体或危及组织的核心基础设施。安全团队应立即修复严重性漏洞。例如，导致生产服务器上远程代码执行（RCE）的 SQL 注入将被视为严重问题。

请查阅 [https://www.first.org/cvss/](https://www.first.org/cvss/) 上的通用漏洞评分系统（CVSS），以了解每种漏洞类型的严重程度。CVSS 量表考虑了诸如漏洞如何影响组织、漏洞利用难度以及漏洞利用是否需要特殊权限或用户交互等因素。

然后，试着想象你的客户公司关心什么，以及哪些漏洞会带来最大的业务影响。根据客户的业务优先级调整你的评估。例如，一个约会网站可能认为泄露用户出生日期的漏洞无关紧要，因为用户的年龄在网站上已经是公开信息；而一个求职网站可能会认为类似的漏洞很严重，因为在求职过程中申请人的年龄应该是保密的。另一方面，用户银行信息的泄露几乎总是被认为是一个高严重性问题。

如果你不确定你的漏洞属于哪个严重性等级，可以使用漏洞赏金平台的评级量表。例如，Bugcrowd 的评级系统考虑了漏洞类型和受影响的功能（[https://bugcrowd.com/vulnerability-rating-taxonomy/](https://bugcrowd.com/vulnerability-rating-taxonomy/)），而 HackerOne 则基于 CVSS 量表提供了一个严重性计算器（[https://docs.hackerone.com/hackers/severity.html](https://docs.hackerone.com/hackers/severity.html)）。

你可以用一行列出严重性，如下所示：

> **问题严重性：高**

提供准确的严重性评估将使每个人的工作更轻松，并有助于你和安全团队之间建立积极的关系。

### 步骤4：给出清晰的重现步骤

接下来，提供重现漏洞的分步说明。包括你能想到的所有相关设置先决条件和细节。最好假定另一边的工程师对该漏洞一无所知，也不知道应用程序是如何工作的。

例如，一份马马虎虎的报告可能包含以下重现步骤：

1.  登录网站并访问 https://example.com/change_password。
2.  点击“更改密码”按钮。
3.  拦截请求，并将 `user_id` 参数更改为另一个用户的 ID。

请注意，这些步骤不够全面或明确。它们没有说明你需要两个测试账户来测试漏洞。它们还假设你对应用程序及其请求格式有足够的了解，无需更多说明即可执行每个步骤。

现在，这是一份更好报告中的示例：

1.  在 example.com 上创建两个账户：账户 A 和账户 B。
2.  以账户 A 登录 example.com，并访问 https://example.com/change_password。
3.  在页面左上角的“新密码”字段中输入所需的新密码。
4.  点击页面右上角的“更改密码”按钮。
5.  拦截对 https://example.com/change_password 的 POST 请求，并将 POST 参数 `user_id` 更改为账户 B 的用户 ID。
6.  现在，你可以使用你选择的新密码登录账户 B。

尽管安全团队可能仍然能理解第一份报告，但第二份报告要具体得多。通过提供许多相关细节，你可以避免任何误解并加快修复过程。

### 步骤5：提供概念验证

对于简单的漏洞，你提供的步骤可能就是安全团队重现问题所需的一切。但对于更复杂的漏洞，包含一个记录你利用过程的视频、截图或照片会很有帮助，这被称为概念验证（POC）文件。

例如，对于 CSRF 漏洞，你可以包含一个嵌入了 CSRF 载荷的 HTML 文件。这样，安全团队重现问题所需要做的就是在其浏览器中打开该 HTML 文件。对于 XML 外部实体攻击，请包含你用来执行攻击的精心构造的 XML 文件。对于需要多个复杂步骤才能重现的漏洞，你可以录制一个屏幕录像视频，展示你逐步操作的过程。

像这样的 POC 文件可以节省安全团队的时间，因为他们不必自己准备攻击载荷。你还可以包含你用来攻击应用程序的任何精心构造的 URL、脚本或上传的文件。

### 步骤6：描述影响和攻击场景

为了帮助安全团队充分理解漏洞的潜在影响，你还可以说明一个合理的漏洞利用场景。请注意，本节与我之前提到的严重性评估不同。严重性评估描述了攻击者利用漏洞造成后果的严重程度，而攻击场景则解释这些后果实际会是什么样子。

如果黑客利用了这个漏洞，他们能接管用户账户吗？或者他们能窃取用户信息并导致大规模数据泄露吗？站在恶意黑客的角度思考，并尝试尽可能扩大漏洞的影响。让客户公司对最坏情况有一个现实的认识。这将帮助公司内部确定修复的优先级，并判断是否需要采取任何额外步骤或内部调查。

以下是一个影响部分的示例：

> 利用此漏洞，攻击者更改用户密码所需的一切就是他们的 `user_id`。由于每个用户的公开个人资料页面都列出了账户的 `user_id`，任何人都可以访问任何用户的个人资料，找出他们的 `user_id`，然后更改他们的密码。而且，因为 `user_id` 只是简单的序列号，黑客甚至可以枚举所有 `user_id` 并更改所有用户的密码！这个漏洞将使攻击者能够毫不费力地接管任何人的账户。

一个好的影响部分说明了攻击者如何现实地利用漏洞。它既考虑了任何缓解因素，也考虑了可以达到的最大影响。它不应夸大漏洞的影响或包含任何假设情况。

### 步骤7：推荐可能的缓解措施

你还可以建议安全团队可以采取的缓解漏洞的可能步骤。这将节省团队开始研究缓解措施的时间。通常，由于你是发现漏洞的安全研究员，你会熟悉该应用程序功能的特定行为，因此处于提出全面修复方案的有利位置。

但是，除非你很好地理解了问题的根本原因，否则不要提出修复方案。内部团队可能有更多的背景和专业知识来提供适用于其环境的适当缓解策略。如果你不确定是什么导致了漏洞或可能的修复方案是什么，则避免给出任何建议，以免混淆读者。

以下是你可能提出的一个可能的缓解措施：

> 应用程序应在更改密码请求中验证用户的 `user_id` 参数，以确保用户有权进行账户修改。未经授权的请求应被应用程序拒绝并记录。

由于你不了解应用程序的底层代码库，因此不必深入修复的技术细节。但作为了解漏洞类别的人，你可以为缓解措施提供一个方向。

### 步骤8：验证报告

最后，始终验证你的报告。最后再通读一遍你的报告，确保没有技术错误，或任何可能妨碍安全团队理解它的内容。按照你自己的“重现步骤”操作，确保它们包含足够的细节。检查你所有的 POC 文件和代码，确保它们有效。通过验证你的报告，你可以最大程度地减少提交无效报告的可能性。

### 撰写更好报告的额外技巧

以下是一些额外的技巧，可帮助你提供尽可能好的报告。

**不要做任何假设**

首先，不要假设安全团队能够理解你报告中的所有内容。请记住，你可能已经针对这个漏洞工作了一周，但对于收到报告的安全团队来说，这都是新信息。他们还有一大堆其他职责要处理，并且通常不像你那样熟悉该功能。此外，报告并不总是分配给安全团队。较新的项目、开源项目和初创公司可能依赖开发者或技术支持人员来处理漏洞报告，而不是拥有专门的安全团队。帮助他们理解你发现了什么。

尽可能详细，并包含你能想到的所有相关细节。提供指向解释安全团队可能不熟悉的晦涩安全知识的参考文献的链接也是好的。考虑一下“啰嗦”与“遗漏关键细节”的潜在后果。如果你太啰嗦，最坏的情况是你的报告需要多花两分钟阅读。但如果你遗漏了重要细节，漏洞的修复可能会被延迟，而恶意黑客可能会利用该漏洞。

**清晰简洁**

另一方面，不要包含任何不必要的信息，例如冗长的问候语、笑话或表情包。安全报告是一份商业文件，而不是给你朋友的信。它应该直截了当、切中要点。在不遗漏关键细节的情况下，尽量使你的报告尽可能简短。你应该始终努力节省安全团队的时间，以便他们能够立即开始修复漏洞。

**写你想读的内容**

写作时始终将读者放在心上，并努力为他们创造良好的阅读体验。使用对话式的语气，不要使用 leetspeak、俚语或缩写。这些会使文本更难阅读，并增加读者的烦恼。

**保持专业**

最后，始终以尊重和专业的态度与安全团队沟通。耐心并迅速地提供有关报告的解释。

你在撰写报告时可能会犯错，沟通不畅也必然会发生。但请记住，作为安全研究员，你有能力通过投入时间和精力来撰写报告，从而将这种可能性降到最低。通过磨练你的报告技巧以及你的黑客技能，你可以节省每个人的时间，并最大化你作为黑客的价值。

## 与开发团队建立关系

你作为黑客的工作在提交报告的那一刻并没有结束。作为发现漏洞的人，你应该帮助公司修复问题，并确保漏洞被完全修补。

让我们谈谈如何在提交报告后处理与安全团队的互动，以及如何与他们建立牢固的关系。与安全团队建立牢固的关系将有助于你的报告更快、更顺利地得到解决。如果你能持续为组织的安全做出贡献，这甚至可能导致更大的漏洞赏金。一些漏洞赏金猎人甚至因为他们的漏洞赏金发现而获得了顶级科技公司的面试机会或工作机会！我们将讨论报告的不同状态、在修复过程的每个阶段你应该做什么，以及在与安全团队沟通时如何处理冲突。

### 理解报告状态

一旦你提交了报告，安全团队会将其归类到一个报告状态，该状态描述了你报告的当前状态。随着修复过程的推进，报告状态会发生变化。你可以在漏洞赏金平台的界面上或从安全团队收到的消息中找到列出的报告状态。

**需要更多信息**

你会看到的最常见的报告状态之一是“需要更多信息”。这意味着安全团队没有完全理解你的报告，或者无法使用你提供的信息重现问题。安全团队通常会跟进提问或请求提供有关漏洞的更多信息。

在这种情况下，你应该修改你的报告，提供任何缺失的信息，并解决安全团队的额外疑虑。

**信息性**

如果安全团队将你的报告标记为“信息性”，他们将不会修复该漏洞。这意味着他们认为你报告的问题是一个安全问题，但不够严重，不值得修复。不影响其他用户的漏洞，例如能够在线游戏中提高自己的分数，通常属于这一类。另一种经常被标记为信息性的漏洞是缺少安全最佳实践，例如允许用户重复使用密码。

在这种情况下，你对这份报告无能为力了！公司不会向你支付赏金，你也不必跟进，除非你认为安全团队犯了错误。但是，我确实建议你跟踪信息性问题，并尝试将它们串联成更大、更有影响力的漏洞。

**重复**

“重复”报告状态意味着另一个黑客已经找到了这个漏洞，公司正在修复该漏洞的过程中。

不幸的是，由于公司只向第一个发现漏洞的黑客支付赏金，你不会因为重复报告而获得报酬。除了帮助公司解决问题之外，对这份报告没有什么可做的了。你也可以尝试将漏洞升级或串联成更有影响力的漏洞。这样，安全团队可能会将新报告视为一个单独的问题并奖励你。

**不适用（N/A）**

“不适用”（N/A）状态意味着你的报告不包含具有安全影响的有效安全问题。当你的报告包含技术错误，或者该漏洞是应用程序的预期行为时，可能会发生这种情况。

N/A 报告不支付报酬。你除了继续前进并继续黑客活动之外，没有什么可做的了！

**已分类**

当安全团队在其端验证了报告后，他们会将报告“分类”。这对你来说是个好消息，因为这通常意味着安全团队将修复漏洞并奖励你一笔赏金。

一旦报告被分类，你应该帮助安全团队修复问题。及时跟进他们的问题，并提供他们要求的任何额外信息。

**已解决**

当你的报告被标记为“已解决”时，意味着报告的漏洞已被修复。此时，拍拍自己的肩膀，为你让互联网变得更安全一点而欣慰。如果你参与的是有偿漏洞赏金计划，此时你可以期待收到你的报酬！

除了庆祝并继续黑客活动之外，对这份报告没有什么可做的了。

### 处理冲突

并非所有报告都能快速顺利地解决。当黑客和安全团队对漏洞的有效性、漏洞的严重性或适当的奖励金额存在分歧时，冲突不可避免地会发生。即便如此，冲突可能会损害你作为黑客的声誉，因此以专业的方式处理冲突是成功的漏洞狩猎生涯的关键。如果你发现自己与安全团队发生冲突，你应该这样做。

当你与安全团队对漏洞的有效性存在分歧时，首先确保你最初报告中的所有信息都是正确的。通常，安全团队因为技术或写作错误而将报告标记为“信息性”或“N/A”。例如，如果你在 POC 中包含了错误的 URL，安全团队可能无法重现问题。如果这是导致分歧的原因，请尽快发送一份包含正确信息的后续报告。

另一方面，如果你在报告中没有犯错，但仍然认为他们对问题的标记不正确，请发送一份后续报告，解释你为什么认为该漏洞是一个安全问题。如果这仍然不能解决误解，你可以请求漏洞赏金平台或团队中的其他安全工程师进行调解。

大多数时候，如果漏洞不属于众所周知的漏洞类别，其他人很难看到其影响。如果安全团队否定了所报告问题的严重性，你应该解释一些潜在的攻击场景，以充分说明其影响。

最后，如果你对赏金金额不满意，请不带怨气地沟通。询问组织分配该赏金的理由，并解释为什么你认为你应该获得更高的奖励。例如，如果你报告的责任人低估了漏洞的严重性，你可以在要求更高奖励时详细说明问题的影响。无论你做什么，总是避免毫无解释地要求更多钱。

请记住，我们都会犯错。如果你认为处理你报告的人处理不当，请礼貌地要求重新考虑。一旦你陈述了你的理由，请尊重公司关于修复和赏金金额的最终决定。

### 建立合作伙伴关系

漏洞赏金之旅并不会在报告解决后停止。你应该努力与组织建立长期的合作伙伴关系。这可以帮助你的报告更顺利地得到解决，甚至可能为你带来面试或工作机会。你可以通过尊重他们的时间和以专业精神沟通来与公司建立良好的关系。

首先，通过始终提交经过验证的报告来赢得尊重。不要通过发送垃圾信息、纠缠他们要钱或辱骂安全团队来破坏公司的信任。反过来，他们会尊重你，并将你视为优先考虑的研究员。公司经常会封禁不尊重或不讲道理的猎人，所以要不惜一切代价避免落入这些类别。

还要学习与你合作的每个组织的沟通风格。他们期望在他们的报告中包含多少细节？你可以通过阅读他们公开披露的报告，或者通过将他们对你报告的反馈纳入未来的消息中来了解安全团队的沟通风格。他们是否期望大量照片和视频来记录漏洞？定制你的报告，让你的读者工作更轻松。

最后，确保你支持安全团队直到他们解决问题。许多组织会在报告分类时支付赏金，但请不要在收到奖励后就弃安全团队于不顾！如果需要，提供帮助缓解漏洞的建议，并协助安全团队确认问题已被修复。有时，组织会要求你进行重新测试并支付费用。如果可以，一定要抓住这个机会。你不仅能赚钱，还能帮助公司更快地解决问题。

## 理解你失败的原因

你投入了大量时间寻找漏洞，却没有发现一个。或者你不断提交被标记为“信息性”、“N/A”或“重复”的报告。

你已经遵守了所有规则。你已经使用了所有工具。到底哪里出了问题？排行榜上的黑客向你隐瞒了什么秘密？在本节中，我将讨论阻碍你在漏洞赏金中成功的原因，以及你可以如何改进。

### 为什么你找不到漏洞

如果你在漏洞赏金上花了大量时间，但仍然难以找到漏洞，以下是一些可能的原因。

**你参与了错误的计划**

你可能一直都在瞄准错误的计划。漏洞赏金计划并非生来平等，选择正确的计划至关重要。有些计划因缺乏处理报告的资源而延迟修复漏洞。有些计划淡化漏洞的严重性以避免支付黑客报酬。最后，还有一些计划将其范围限制在其资产的一小部分。他们运行漏洞赏金计划是为了获得正面宣传，并不打算实际修复漏洞。避开这些计划，以免自己头疼。

你可以通过阅读公开披露的报告、分析漏洞赏金平台上的计划统计数据或与其他黑客交谈来识别这些计划。漏洞赏金平台上列出的计划统计数据提供了大量关于计划执行情况的信息。避免响应时间长和平均赏金低的计划。仔细选择目标，并优先考虑那些投资于其漏洞赏金计划的公司。

**你不坚持一个计划**

你应该瞄准一个计划多长时间？如果你的答案是几个小时或几天，那就是你什么也没找到的原因。从一个计划跳到另一个计划是初学者常犯的另一个错误。

每个漏洞赏金计划都有无数的漏洞赏金猎人在攻击它。让自己从竞争中脱颖而出，否则就有可能什么也找不到！你可以通过两种方式让自己脱颖而出：深度挖掘或广泛搜索。例如，深入挖掘应用程序的单个功能以寻找复杂的漏洞。或者发现并攻击公司鲜为人知的资产。

做好这些事情需要时间。不要期望在一个全新的计划上马上就能找到漏洞。如果你第一天找不到漏洞，也不要放弃一个计划。

**你不做侦察**

不进行侦察就直接跳进大型公开计划是在漏洞赏金中失败的另一种方式。我们在第 5 章中讨论的有效侦察，可以帮助你发现新的攻击面：新的子域、新的端点和新的功能。

花时间进行侦察会让你比其他黑客拥有不可思议的优势，因为你会是第一个注意到你发现的所有冷门资产上的漏洞的人，这让你有更好的机会找到非重复的漏洞。

**你只找低垂的果实**

初学者常犯的另一个错误是依赖漏洞扫描器。公司会定期扫描和审计他们的应用程序，其他漏洞赏金猎人也经常这样做，所以这种方法不会给你带来好结果。

此外，避免只寻找明显的漏洞类型。大目标上的简单漏洞可能已经被发现了。许多漏洞赏金计划在公司向公众开放之前都是私有的。这意味着一些有经验的黑客可能已经报告了最容易找到的漏洞。例如，许多黑客很可能已经测试过论坛评论字段上的存储型 XSS 漏洞。

这并不是说你完全不应该寻找低垂的果实。只是如果你那样做没有找到任何东西，也不要气馁。相反，努力更深入地理解应用程序的底层架构和逻辑。从那里，你可以开发出独特的测试方法，从而产生更独特、更有价值的漏洞。

**你不进入私有计划**

在你开始攻击私有计划之后，找到漏洞会变得容易得多。许多成功的黑客表示，他们的大部分发现都来自私有计划。私有计划比公开计划人少得多，所以你的竞争会更少，而竞争少通常意味着更容易找到漏洞和更少的重复报告。

### 为什么你的报告被驳回

如前所述，有三种类型的报告不会带来赏金：N/A、信息性和重复报告。在本节中，我将讨论你可以做些什么来减少这些令人失望的情况。

减少无效报告的数量对每个人都有好处。它不仅会节省你的时间和精力，还会节省安全团队用于处理这些报告的工作时间。以下是你报告不断被驳回的一些原因。

**你不阅读赏金政策**

报告被标记为 N/A 的最常见原因之一是它们超出了范围。计划的政策页面通常有一个名为“范围”的部分，告诉你允许攻击公司的哪些资产。大多数时候，政策页面还会列出范围外的漏洞和资产，意味着你不允许报告它们。

防止提交 N/A 报告的最佳方法是仔细并反复阅读赏金政策。哪些漏洞类型是范围外的？组织的哪些资产是范围外的？尊重这些边界，不要提交范围外的漏洞。

如果你确实偶然发现了一个范围外的严重问题，并且你认为组织必须知道这件事，那就报告它！你可能得不到奖励，但你仍然可以为公司的安全做出贡献。

**你不设身处地为组织着想**

信息性报告比 N/A 报告更难防止。大多数时候，你会得到信息性评级，因为公司不在乎你报告的问题。

想象一下你是一名安全工程师。如果你每天都忙于保护数百万用户的数据，你会关心一个只能用于网络钓鱼的开放重定向吗？尽管这是一个有效的安全漏洞，但你可能不会。你还有其他职责要处理，所以修复低严重性漏洞是你待办事项列表中的最后一项。如果安全团队没有额外的人手来处理这些报告，他们有时会忽略它并将其标记为信息性。

我发现减少信息性报告最有帮助的方法是设身处地为组织着想。了解组织，这样你就能识别其产品、它正在保护的数据以及其应用程序中最重要的部分。一旦你了解了业务的优先级，你就可以去追捕安全团队关心的漏洞。

请记住，不同的公司有不同的优先级。对一家组织来说是信息性的报告，对另一家来说可能是严重的。就像本章前面提到的约会网站与求职网站的例子一样，一切都是相对的。有时，很难弄清楚一个漏洞对组织有多重要。我报告为严重的一些问题最终被标记为信息性。而我归类为低影响的一些漏洞却被作为严重问题奖励。

这就是试错可以带来回报的地方。每次安全团队将你的报告归类为信息性时，都要记录下来以备将来参考。下次你发现漏洞时，问自己：这家公司过去关心过这样的问题吗？了解每家公司关心什么，并根据其业务优先级调整你的黑客努力。你最终会培养出一种直觉，知道哪种漏洞能产生最大的影响。

**你不串联漏洞**

你得到信息性报告也可能是因为你总是报告你发现的第一个小漏洞。

但是，如果你学会串联漏洞，被归类为信息性的小问题可能会变成大问题。当你发现一个可能被驳回的低严重性漏洞时，不要立即报告。相反，尝试在未来的漏洞链中使用它。例如，不要报告一个开放重定向，而是在服务端请求伪造（SSRF）攻击中使用它！

**你写的报告很糟糕**

初学者常犯的另一个错误是他们未能在报告中传达漏洞的影响。即使一个漏洞很有影响力，如果你不能向安全团队传达其含义，他们也会驳回报告。

**那重复报告呢？**

不幸的是，有时你无法避免重复报告。但是你可以通过以下方式降低获得重复报告的几率：在范围大的计划上狩猎、攻击私有计划、广泛进行侦察，以及开发你独特的狩猎方法。

## 当你卡住时该怎么办

当我刚开始进行漏洞赏金时，我常常几天或几周都找不到一个漏洞。我的第一个目标是有一个大范围的社交媒体网站。但在报告了我的第一批 CSRF 和 IDOR 之后，我很快就没了想法（和运气）。我开始一遍又一遍地检查同样的漏洞，并尝试不同的自动工具，但都无济于事。

后来我发现我并不孤单；这种漏洞低谷在新黑客中出奇地普遍。让我们谈谈当你卡住时如何从沮丧中恢复过来并改善你的结果。

### 步骤1：休息一下！

首先，休息一下。黑客工作很辛苦。与电影中展示的不同，寻找漏洞是乏味而困难的。它需要耐心、毅力和对细节的关注，所以它非常消耗脑力。

在你继续黑客活动之前，问问自己：我累了吗？缺乏灵感可能是你的大脑在告诉你它已经达到了极限。在这种情况下，你最好的做法是休息一下。到外面去。见见朋友。吃点冰淇淋。或者呆在室内。泡杯茶。读一本好书。

生活中除了 SQL 注入和 XSS 载荷，还有更多的东西。如果你从黑客活动中休息一下，你常常会发现当你回来时，你会更有创造力。

### 步骤2：提升你的技能

利用你的黑客低谷作为提升技能的机会。黑客常常卡住是因为他们对某些熟悉的技术太安逸了，当这些技术不再有效时，他们错误地认为没有什么可尝试的了。学习新技能将让你走出舒适区，并为未来加强你的黑客技能。

首先，如果你还不熟悉基本的黑客技术，请参考测试指南和最佳实践来巩固你的技能。例如，开放 Web 应用安全项目（OWASP）发布了针对各种资产类型的测试指南。你可以在 [https://owasp.org/www-project-web-security-testing-guide/](https://owasp.org/www-project-web-security-testing-guide/) 和 [https://owasp.org/www-project-mobile-security-testing-guide/](https://owasp.org/www-project-mobile-security-testing-guide/) 找到 OWASP 的 Web 和移动测试指南。

学习一种新的黑客技术，无论是新的 Web 利用技术、新的侦察角度，还是不同的平台（如 Android）。专注于你想要建立的一项特定技能，阅读相关内容，并将其应用到你正在攻击的目标上。谁知道呢？你可能会发现一个全新的方法来接近目标应用程序！你也可以利用这个机会，通过阅读许多黑客博客和 write-up 网站来了解其他黑客在做什么。理解其他黑客的方法可以为你与目标互动提供令人耳目一新的新视角。

接下来，玩夺旗赛（CTF）。在这些安全竞赛中，玩家寻找证明他们已入侵系统的“旗帜”。CTF 是了解新漏洞的好方法。它们也很有趣，并且通常以有趣的新漏洞类别为特色。研究人员不断发现新型的利用技术，掌握这些技术将确保你不断发现漏洞。

### 步骤3：获得新视角

当你准备好再次攻击真实目标时，这里有一些技巧可以帮助你保持动力。

首先，攻击单一目标可能会变得无聊，因此多样化你的目标，而不是只关注一个。我发现有几个目标可以轮换总是有帮助的。当你厌倦了一个应用程序时，切换到另一个，稍后再回来。

其次，确保你在目标中寻找特定的东西，而不是漫无目的地寻找任何东西。列出你学到的新技能并尝试它们。寻找一种新的漏洞，或者尝试一个新的侦察角度。然后，反复进行，直到找到合适的新工作流程。

最后，请记住，黑客攻击并不总是关于发现单个漏洞，而是将应用程序的几个弱点组合成严重的问题。在这种情况下，专门寻找奇怪的行为而不是漏洞会很有帮助。然后记下这些奇怪的行为和弱点，看看你是否能将它们串联成值得报告的东西。

## 最后，几点经验之谈

漏洞赏金狩猎是困难的。当我开始寻找漏洞时，我有时会几个月都找不到一个。而当我找到一个时，也是琐碎且低严重性的。

任何事情变得更好的关键在于练习。如果你愿意投入时间和精力，你的黑客技能将会提高，你很快就会看到自己出现在排行榜和私有邀请名单上！如果你在这个过程中感到沮丧，请记住，随着时间的推移，一切都会变得更容易。如果需要帮助，请向黑客社区寻求帮助。祝你好运！
