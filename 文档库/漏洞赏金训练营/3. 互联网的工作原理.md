# 第3章 互联网的工作原理

在你开始寻找漏洞之前，让我们花些时间了解互联网的工作原理。发现 Web 漏洞完全是关于利用这项技术中的弱点，因此所有优秀的黑客都应该对它有扎实的理解。如果你已经熟悉这些过程，可以随意跳到后面关于互联网安全控制的部分。

以下问题是一个很好的起点：当你在浏览器中输入 `www.google.com` 时会发生什么？换句话说，你的浏览器是如何知道从 google.com 这样的域名转到你要找的网页的？让我们一探究竟。

## 客户端-服务器模型

互联网由两种设备组成：客户端和服务器。客户端请求资源或服务，服务器提供这些资源和服务。当你用浏览器访问网站时，浏览器充当客户端，向 Web 服务器请求网页。然后 Web 服务器会将网页发送给你的浏览器（图 3-1）。

**图 3-1：互联网客户端向服务器请求资源。**

网页不过是 Web 服务器发送的资源或文件的集合。例如，至少，服务器会发送一个用超文本标记语言（HTML）编写的文本文件，这种语言告诉浏览器要显示什么。大多数网页还包括层叠样式表（CSS）文件来美化页面。有时网页还包含 JavaScript（JS）文件，使网站能够为网页添加动画效果，并在不经过服务器的情况下响应用户输入。例如，JavaScript 可以在用户滚动页面时调整图像大小，并在将用户输入发送到服务器之前在客户端进行验证。最后，你的浏览器可能会收到嵌入式资源，如图像和视频。你的浏览器会组合这些资源来显示你看到的网页。

服务器也不仅仅向用户返回网页。Web API 使应用程序能够请求其他系统的数据。这使得应用程序能够以受控的方式相互交互并共享数据和资源。例如，Twitter 的 API 允许其他网站向 Twitter 的服务器发送请求，以检索诸如公开推文及其作者的列表等数据。API 支持了除此之外的许多互联网功能，我们将在第 24 章中再次讨论它们及其安全问题。

## 域名系统

你的浏览器和其他 Web 客户端如何知道在哪里找到这些资源？连接到互联网的每台设备都有一个唯一的互联网协议（IP）地址，其他设备可以用它来找到该设备。然而，IP 地址由数字和字母组成，人类很难记住。例如，较旧格式的 IP 地址 IPv4 看起来像这样：123.45.67.89。新版本 IPv6 看起来更复杂：2001:db8::ff00:42:8329。

这就是域名系统（DNS）发挥作用的地方。DNS 服务器充当互联网的电话簿，将域名转换为 IP 地址（图 3-2）。当你在浏览器中输入域名时，DNS 服务器必须首先将域名转换为 IP 地址。你的浏览器问 DNS 服务器：“这个域名位于哪个 IP 地址？”

**图 3-2：DNS 服务器会将域名转换为 IP 地址。**

## 互联网端口

在你的浏览器获取到正确的 IP 地址后，它将尝试通过一个端口连接到该 IP 地址。端口是设备上的逻辑划分，用于标识特定的网络服务。我们通过端口号来标识端口，端口号范围从 0 到 65,535。

端口允许服务器同时向互联网提供多种服务。由于某些端口接收的流量存在约定，端口号还允许服务器快速将到达的互联网消息转发到相应的服务进行处理。例如，如果互联网客户端连接到端口 80，Web 服务器就会明白客户端希望访问其 Web 服务（图 3-3）。

**图 3-3：端口允许服务器提供多种服务。端口号有助于将客户端请求转发到正确的服务。**

默认情况下，我们对 HTTP 消息使用端口 80，对加密版本的 HTTP（HTTPS）使用端口 443。

## HTTP 请求与响应

一旦连接建立，浏览器和服务器通过超文本传输协议（HTTP）进行通信。HTTP 是一组规则，规定了如何构建和解释互联网消息，以及 Web 客户端和 Web 服务器应如何交换信息。

当你的浏览器想要与服务器交互时，它会向服务器发送一个 HTTP 请求。HTTP 请求有不同的类型，最常见的两种是 GET 和 POST。按照惯例，GET 请求从服务器检索数据，而 POST 请求向服务器提交数据。其他常见的 HTTP 方法包括：OPTIONS（用于请求给定 URL 允许的 HTTP 方法）、PUT（用于更新资源）和 DELETE（用于删除资源）。

以下是一个 GET 请求示例，请求服务器提供 `www.google.com` 的主页：

```
GET / HTTP/1.1
Host: www.google.com
User-Agent: Mozilla/5.0
Accept: text/html,application/xhtml+xml,application/xml
Accept-Language: en-US
Accept-Encoding: gzip, deflate
Connection: close
```

让我们逐步了解这个请求的结构，因为你会在本书中看到很多这样的请求。所有 HTTP 请求都由请求行、请求头和可选的请求体组成。前面的示例只包含请求行和请求头。

请求行是 HTTP 请求的第一行。它指定了请求方法、请求的 URL 以及使用的 HTTP 版本。在这里，你可以看到客户端正在使用 HTTP 版本 1.1 向 `www.google.com` 的主页发送 HTTP GET 请求。

其余的行是 HTTP 请求头。它们用于向服务器传递有关请求的附加信息。这允许服务器定制发送给客户端的结果。在前面的示例中，`Host` 头指定了请求的主机名。`User-Agent` 头包含请求软件（例如用户的 Web 浏览器）的操作系统和软件版本。`Accept`、`Accept-Language` 和 `Accept-Encoding` 头告诉服务器响应应该采用哪种格式。`Connection` 头告诉服务器在服务器响应后网络连接是否应保持打开状态。

你可能会在请求中看到其他几个常见的头。`Cookie` 头用于从客户端向服务器发送 cookie。`Referer` 头指定了链接到当前页面的上一个网页的地址。`Authorization` 头包含用于向服务器认证用户的凭证。

服务器收到请求后，将尝试完成它。服务器将使用 HTTP 响应返回用于构建网页的所有资源。HTTP 响应包含多个部分：一个 HTTP 状态码，用于指示请求是否成功；HTTP 头，它们是浏览器和服务器用来相互通信关于认证、内容格式和安全策略的信息片段；以及 HTTP 响应体，即你请求的实际 Web 内容。Web 内容可能包括 HTML 代码、CSS 样式表、JavaScript 代码、图像等。

以下是一个 HTTP 响应的示例：

```
1 HTTP/1.1 200 OK
2 Date: Tue, 31 Aug 2021 17:38:14 GMT
[...]
3 Content-Type: text/html; charset=UTF-8
4 Server: gws
5 Content-Length: 190532

<!doctype html>
[...]
<title>Google</title>
[...]
<html>
```

注意第一行的 `200 OK` 消息 ➊。这是状态码。200 范围内的 HTTP 状态码表示请求成功。300 范围内的状态码表示重定向到另一个页面，而 400 范围内表示客户端出错，例如请求不存在的页面。500 范围内表示服务器本身遇到错误。

作为漏洞赏金猎人，你应该始终关注这些状态码，因为它们可以告诉你很多关于服务器运行方式的信息。例如，状态码 403 表示该资源对你禁止访问。这可能意味着敏感数据隐藏在页面上，如果你能绕过访问控制，就可能访问到。

响应中用冒号（`:`）分隔的几行是 HTTP 响应头。它们允许服务器向客户端传递有关响应的附加信息。在本例中，你可以看到响应时间是 `Tue, 31 Aug 2021 17:38:14 GMT` ➋。`Content-Type` 头指示响应体的文件类型。在这里，此页面的 `Content-Type` 是 `text/html` ➌。服务器版本是 Google Web Server（`gws`）➍，`Content-Length` 是 190,532 字节 ➎。通常，额外的响应头会指定内容的格式、语言和安全策略。

除此之外，你可能会遇到其他几个常见的响应头。`Set-Cookie` 头由服务器发送给客户端以设置 cookie。`Location` 头指示页面要重定向到的 URL。`Access-Control-Allow-Origin` 头指示哪些源可以访问页面内容（我们将在第 19 章中更多地讨论这一点）。`Content-Security-Policy` 控制浏览器允许加载的资源的来源，而 `X-Frame-Options` 头指示页面是否可以加载到 iframe 中（在第 8 章中进一步讨论）。

空行之后的数据是响应体。它包含网页的实际内容，例如 HTML 和 JavaScript 代码。一旦你的浏览器收到了构建网页所需的所有信息，它就会为你渲染所有内容。

## 互联网安全控制

现在你对信息如何通过互联网通信有了高层次的理解，让我们深入了解一些保护互联网免受攻击者侵害的基本安全控制。为了有效地寻找漏洞，你常常需要创造性地绕过这些控制，因此你首先需要理解它们是如何工作的。

### 内容编码

在 HTTP 请求和响应中传输的数据并不总是以纯文本形式传输。网站通常以不同方式对其消息进行编码，以防止数据损坏。

数据编码被用作一种在不同机器之间可靠传输二进制数据的方式，这些机器对不同内容类型的支持有限。用于编码的字符是那些不作为互联网协议中控制字符使用的常见字符。因此，当你使用常见编码方案对内容进行编码时，你可以确信你的数据将完好无损地到达目的地。相反，当你以原始状态传输数据时，当互联网协议误解消息中的特殊字符时，数据可能会被破坏。

Base64 编码是最常见的数据编码方式之一。它通常用于在 Web 消息中传输图像和加密信息。这是字符串 "Content Encoding" 的 base64 编码版本：

```
Q29udGVudCBFbmNvZGluZw==
```

Base64 编码的字符集包括大写字母 A 到 Z、小写字母 a 到 z、数字 0 到 9、字符 `+` 和 `/`，以及最后的填充字符 `=`。Base64url 编码是用于 URL 格式的 base64 修改版本。它与 base64 类似，但使用不同的非字母数字字符并省略了填充。

另一种流行的编码方法是十六进制编码。十六进制编码是一种以 base-16 格式表示字符的方式，其中字符范围从 0 到 F。十六进制编码比 base64 占用更多空间且效率较低，但提供了更可读的编码字符串。这是字符串 "Content Encoding" 的十六进制编码版本；你可以看到它比 base64 版本占用更多字符：

```
436f6e74656e7420456e636f64696e67
```

URL 编码是一种将字符转换为更容易通过互联网传输的格式的方法。URL 编码字符串中的每个字符都可以用其指定的十六进制数（前面加 `%` 符号）来表示。更多关于 URL 编码的信息请参见维基百科：[https://en.wikipedia.org/wiki/Percent-encoding](https://en.wikipedia.org/wiki/Percent-encoding)。

例如，单词 `localhost` 可以用其 URL 编码等效形式 `%6c%6f%63%61%6c%68%6f%73%74` 表示。你可以使用 URL 计算器（如 URL Decode and Encode，[https://www.urlencoder.org/](https://www.urlencoder.org/)）来计算主机名的 URL 编码等效形式。

我们将在第 13 章讨论 SSRF 时介绍另外几种字符编码类型——八进制编码和双字编码。当你在调查网站时看到编码内容，始终尝试解码以发现网站试图传递什么信息。你可以使用 Burp Suite 的解码器来解码编码内容。我们将在下一章介绍如何做到这一点。或者，你也可以使用 CyberChef（[https://gchq.github.io/CyberChef/](https://gchq.github.io/CyberChef/)）来解码 base64 内容和其他类型的编码内容。

服务器有时也会在传输前加密其内容。这可以保持客户端和服务器之间数据的私密性，并防止任何拦截流量的人窃听消息。

### 会话管理与 HTTP Cookie

为什么你每次关闭电子邮件标签页时不必重新登录？这是因为网站会记住你的会话。会话管理是一个过程，它允许服务器处理来自同一用户的多个请求，而无需要求用户再次登录。

网站为每个登录的用户维护一个会话，当你登录网站时会开始一个新会话（图 3-4）。服务器会为你的浏览器分配一个关联的会话 ID，作为你身份的证明。会话 ID 通常是一个长且不可预测的序列，旨在不可猜测。当你注销时，服务器会结束会话并撤销会话 ID。如果你没有手动注销，网站也可能定期结束会话。

**图 3-4：登录后，服务器会为你创建一个会话并颁发一个会话 ID，该 ID 唯一标识一个会话。**

大多数网站使用 cookie 在 HTTP 请求中传递会话信息。HTTP cookie 是 Web 服务器发送给你的浏览器的小块数据。当你登录一个网站时，服务器会为你创建一个会话，并将会话 ID 作为 cookie 发送给你的浏览器。收到 cookie 后，你的浏览器会存储它，并将其包含在发送给同一服务器的每个请求中（图 3-5）。

**图 3-5：你的会话 ID 与服务器上存储的会话信息相关联。**

这就是服务器如何知道是你！会话的 cookie 生成后，服务器将跟踪它并使用它来验证你的身份。最后，当你注销时，服务器会使会话 cookie 失效，使其无法再次使用。下次你登录时，服务器将为你创建一个新的会话和新的关联会话 cookie。

### 基于令牌的认证

在基于会话的认证中，服务器存储你的信息，并使用相应的会话 ID 来验证你的身份，而基于令牌的认证系统则将这些信息直接存储在某种令牌中。令牌允许服务器通过解码令牌本身来推断你的身份，而不是将你的信息存储在服务器端并使用会话 ID 来查询。这样，应用程序就不必在服务器端存储和维护会话信息。

这种系统伴随着一个风险：如果服务器使用令牌中包含的信息来确定用户身份，那么用户是否能够修改令牌中的信息并登录为其他人？为了防止诸如此类的令牌伪造攻击，一些应用程序会加密其令牌，或对令牌进行编码，使得只有应用程序本身或其他授权方才能读取。如果用户无法理解令牌的内容，他们可能也无法有效地篡改它。加密或编码令牌并不能完全防止令牌伪造。攻击者有可能在不理解其内容的情况下篡改加密令牌。但这比篡改明文令牌要困难得多。攻击者通常可以解码编码后的令牌来篡改它们。

应用程序保护令牌完整性的另一种更可靠的方法是对令牌进行签名，并在令牌到达服务器时验证令牌签名。签名用于验证一段数据的完整性。它们是只有知道密钥才能生成的特殊字符串。由于没有密钥就无法生成有效的签名，并且只有服务器知道密钥是什么，因此有效的签名表明令牌可能没有被客户端或任何第三方篡改。尽管应用程序的实现可能不同，但基于令牌的认证的工作方式如下：

1.  用户使用其凭证登录。
2.  服务器验证这些凭证，并向用户提供一个签名的令牌。
3.  用户在每次请求中发送令牌以证明其身份。
4.  收到并验证令牌后，服务器从令牌中读取用户身份信息，并使用机密数据进行响应。

### JSON Web 令牌

JSON Web 令牌（JWT）是最常用的认证令牌类型之一。它包含三个部分：头部、载荷和签名。

头部标识用于生成签名的算法。它是一个包含算法名称的 base64url 编码字符串。以下是 JWT 头部的样子：

```
eyBhbGcgOiBIUzI1NiwgdHlwIDogSldUIH0K
```

该字符串是以下文本的 base64url 编码版本：

```
{ "alg" : "HS256", "typ" : "JWT" }
```

载荷部分包含有关用户身份的信息。这部分在用于令牌之前也经过 base64url 编码。以下是载荷部分的示例，它是 `{ "user_name" : "admin" }` 的 base64url 编码字符串：

```
eyB1c2VyX25hbWUgOiBhZG1pbiB9Cg
```

最后，签名部分用于验证用户没有篡改令牌。它是通过将头部与载荷连接起来，然后使用头部中指定的算法和密钥进行签名来计算的。以下是 JWT 签名的样子：

```
4Hb/6ibbViPOzq9SJflsNGPWSk6B8F6EqVrkNjpXh7M
```

对于这个特定的令牌，签名是通过使用密钥 `key` 和 HS256 算法对字符串 `eyBhbGcgOiBIUzI1NiwgdHlwIDogSldUIH0K.eyB1c2VyX25hbWUgOiBhZG1pbiB9Cg` 进行签名生成的。完整的令牌将每个部分（头部、载荷和签名）连接起来，用句点（`.`）分隔：

```
eyBhbGcgOiBIUzI1NiwgdHlwIDogSldUIH0K.eyB1c2VyX25hbWUgOiBhZG1pbiB9Cg.4Hb/6ibbViPOzq9SJflsNGPWSk6B8F6EqVrkNjpXh7M
```

当正确实现时，JSON Web 令牌提供了一种安全的用户身份识别方式。当令牌到达服务器时，服务器可以通过检查签名是否正确来验证令牌未被篡改。然后服务器可以使用载荷部分包含的信息来推断用户身份。由于用户无法访问用于签名令牌的密钥，他们无法更改载荷并自己签名令牌。

但如果实现不正确，攻击者有可能绕过安全机制并伪造任意令牌。

#### 操纵 `alg` 字段

有时应用程序在令牌到达服务器后未能验证其签名。这允许攻击者只需提供无效或空签名即可绕过安全机制。

攻击者伪造自己令牌的一种方法是篡改令牌头部的 `alg` 字段，该字段列出了用于编码签名的算法。如果应用程序不限制 JWT 中使用的算法类型，攻击者可以指定使用哪种算法，这可能会危及令牌的安全性。

JWT 支持 `none` 作为算法类型的选项。如果 `alg` 字段设置为 `none`，即使是签名部分为空的令牌也会被认为是有效的。例如，考虑以下令牌：

```
eyAiYWxnIiA6ICJOb25lIiwgInR5cCIgOiAiSldUIiB9Cg.eyB1c2VyX25hbWUgOiBhZG1pbiB9Cg.
```

这个令牌只是以下两个数据块的 base64url 编码版本，没有签名存在：

```
{ "alg" : "none", "typ" : "JWT" } { "user" : "admin" }
```

此功能最初用于调试目的，但如果生产环境中未关闭，它将允许攻击者伪造他们想要的任何令牌，并冒充网站上的任何人。

攻击者利用 `alg` 字段的另一种方式是更改所使用的算法类型。JWT 最常用的两种签名算法是 HMAC 和 RSA。HMAC 要求令牌使用密钥签名，然后使用相同的密钥进行验证。使用 RSA 时，令牌首先使用私钥创建，然后使用相应的公钥进行验证，任何人都可以读取公钥。HMAC 令牌的密钥和 RSA 令牌的私钥必须保密，这一点至关重要。

现在假设一个应用程序最初被设计为使用 RSA 令牌。令牌使用私钥 A 签名，该私钥对公众保密。然后令牌使用公钥 B 进行验证，公钥 B 对任何人都是可用的。只要令牌始终被视为 RSA 令牌，这是可以的。现在，如果攻击者将 `alg` 字段更改为 HMAC，他们可能能够通过使用 RSA 公钥 B 对伪造的令牌进行签名来创建有效的令牌。当签名算法切换到 HMAC 时，令牌仍然使用 RSA 公钥 B 进行验证，但这次，令牌也可以使用相同的公钥进行签名。

#### 暴力破解密钥

也有可能猜测或暴力破解用于签名 JWT 的密钥。攻击者有很多信息可以开始：用于签名令牌的算法、被签名的载荷以及生成的签名。如果用于签名令牌的密钥不够复杂，他们可能能够轻松地暴力破解它。如果攻击者无法暴力破解密钥，他们可能会尝试泄露密钥。如果存在其他漏洞，如目录遍历、外部实体攻击（XXE）或 SSRF，允许攻击者读取存储密钥值的文件，攻击者就可以窃取密钥并签名他们选择的任意令牌。我们将在后面的章节中讨论这些漏洞。

#### 读取敏感信息

由于 JSON Web 令牌用于访问控制，它们通常包含有关用户的信息。如果令牌未加密，任何人都可以对令牌进行 base64 解码并读取令牌的载荷。如果令牌包含敏感信息，它可能成为信息泄露的来源。正确实现的 JSON Web 令牌签名部分提供数据完整性，而不是机密性。

这些只是 JWT 安全问题的几个例子。有关 JWT 漏洞的更多示例，请使用搜索词 "JWT security issues"。任何认证机制的安全性不仅取决于其设计，还取决于其实现。JWT 可以是安全的，但前提是正确实现。

## 同源策略

同源策略（SOP）是一条规则，限制来自一个源的脚本如何与不同源的资源进行交互。用一句话概括，SOP 是：来自页面 A 的脚本只有在页面 A 和页面 B 同源的情况下才能访问页面 B 的数据。这条规则保护了现代 Web 应用程序，并防止了许多常见的 Web 漏洞。

如果两个 URL 共享相同的协议、主机名和端口号，则称它们具有相同的源。让我们看一些例子。页面 A 的 URL 是：

`https://medium.com/@vickieli`

它使用 HTTPS，记得吗，HTTPS 默认使用端口 443。现在查看以下页面，根据 SOP 判断哪些页面与页面 A 具有相同的源：

- `https://medium.com/`
- `http://medium.com/`
- `https://twitter.com/@vickieli7`
- `https://medium.com:8080/@vickieli`

URL `https://medium.com/` 与页面 A 同源，因为两个页面共享相同的协议、主机名和端口号。其他三个页面与页面 A 不同源。`http://medium.com/` 与页面 A 不同源，因为它们的协议不同（`https://medium.com/` 使用 HTTPS，而 `http://medium.com/` 使用 HTTP）。`https://twitter.com/@vickieli7` 也不同源，因为它有不同的主机名。最后，`https://medium.com:8080/@vickieli` 不同源，因为它使用端口 8080 而不是端口 443。

现在让我们考虑一个例子，看看 SOP 是如何保护我们的。想象你登录了你的银行网站 `onlinebank.com`。不幸的是，你在同一个浏览器中点击了一个恶意网站 `attacker.com`。

恶意网站向 `onlinebank.com` 发出 GET 请求以检索你的个人信息。由于你已登录银行，你的浏览器会自动将你的 cookie 包含在你发送给 `onlinebank.com` 的每个请求中，即使该请求是由恶意网站上的脚本生成的。由于请求包含有效的会话 ID，`onlinebank.com` 的服务器通过发送包含你信息的 HTML 页面来满足请求。然后恶意脚本读取并检索页面中包含的私人电子邮件地址、家庭地址和银行信息。

幸运的是，SOP 将阻止托管在 `attacker.com` 上的恶意脚本读取从 `onlinebank.com` 返回的 HTML 数据。这可以防止页面 A 上的恶意脚本获取嵌入在页面 B 中的敏感信息。

## 学习编程

你现在应该有了坚实的基础，能够理解我们将要介绍的大部分漏洞。在设置你的黑客工具之前，我建议你学习编程。编程技能很有帮助，因为寻找漏洞涉及许多重复性任务，通过学习 Python 或 shell 脚本等编程语言，你可以自动化这些任务，从而节省大量时间。

你还应该学习阅读 JavaScript，这是大多数网站使用的语言。阅读网站的 JavaScript 可以让你了解它是如何工作的，为你找到漏洞提供了一条快速通道。许多顶级黑客说他们的秘诀就是阅读 JavaScript 并搜索隐藏的端点、不安全的编程逻辑和密钥。我也通过阅读 JavaScript 源代码发现过许多漏洞。

Codecademy 是一个学习编程的好资源。如果你更喜欢看书，Zed Shaw 的《Learn Python the Hard Way》（Addison-Wesley Professional，2013）是学习 Python 的好方法。而阅读 Marijn Haverbeke 的《Eloquent JavaScript, Third Edition》（No Starch Press，2019）是掌握 JavaScript 的最佳方法之一。
