# 第17章：应用程序逻辑错误与访问控制失效

应用程序逻辑错误和访问控制失效漏洞与我们之前讨论的漏洞有很大不同。前面章节介绍的大多数漏洞都是由输入验证缺陷引起的：当被污染的用户输入未经适当清理而被处理时，就会发生这些漏洞。这些恶意输入在语法上与正常用户输入不同，旨在操纵应用程序逻辑并对应用程序或其用户造成损害。

另一方面，应用程序逻辑错误和访问控制失效问题通常是由完全有效的HTTP请求触发的，这些请求不包含非法或畸形的字符序列。尽管如此，这些请求是故意构造的，目的是为了恶意目的滥用应用程序逻辑或绕过应用程序的访问控制。

应用程序逻辑错误是应用程序中的逻辑缺陷。有时攻击者可以利用它们对组织、应用程序或其用户造成损害。当敏感资源或功能未得到适当保护时，就会发生访问控制失效。要发现这些漏洞，你不能仅仅依赖技术知识。相反，你需要运用创造力和直觉来绕过开发人员设置的限制。

本章将解释这些漏洞、它们在应用程序中如何表现，以及如何测试它们。

## 应用程序逻辑错误

应用程序逻辑错误（或业务逻辑漏洞）是指利用应用程序的合法逻辑流程，却给组织带来负面后果的方式。听起来有点抽象？理解它们的最佳方式是看几个例子。

我在目标网站中常见的一个应用程序逻辑错误是站点多因素认证功能中的缺陷。多因素认证（MFA）是一种要求用户以多种方式证明其身份的做法。MFA通过要求用户同时使用密码和另一种身份证明（通常是手机号或邮箱，有时是通过认证应用、物理密钥甚至指纹）进行认证，从而在密码泄露时保护用户。

大多数MFA实现会提示用户使用密码和通过电子邮件或短信发送的授权码进行认证。但MFA实现常常被一个我称之为**可跳过认证步骤**的逻辑错误所破坏，该错误允许用户跳过认证过程中的某个步骤。

例如，假设一个应用程序实现了三步登录流程。首先，应用程序检查用户的密码。然后，它向用户发送MFA码并验证。最后，应用程序在登录用户之前询问安全问题：

```
步骤1（密码检查）→ 步骤2（MFA）→ 步骤3（安全问题）
```

正常的认证流程如下：

1. 用户访问 `https://example.com/login/`。应用程序提示用户输入密码，用户输入。
2. 如果密码输入正确，应用程序向用户的邮箱发送MFA码，并将用户重定向到 `https://example.com/mfa/`。在这里，用户输入MFA码。
3. 应用程序检查MFA码，如果正确，将用户重定向到 `https://example.com/security_questions/`。在那里，应用程序询问用户几个安全问题，如果提供的答案正确则登录用户。

但有时，用户可以在未完成步骤1和2的情况下到达认证流程的步骤3。虽然易受攻击的应用程序在步骤2完成后将用户重定向到步骤3，但它没有验证用户在允许进入步骤3之前是否已完成步骤2。在这种情况下，攻击者所要做的就是操纵站点的URL并直接请求后续阶段的页面。

如果攻击者可以直接访问 `https://example.com/security_questions/`，他们就可以完全绕过多因素认证。他们可能只需凭某人的密码和安全问题答案就能登录，而无需其MFA设备。

应用程序逻辑错误的另一个常见表现是在多步骤结账过程中。假设一个在线商店允许用户通过已保存的支付方式付款。当用户保存新的支付方式时，站点会验证信用卡是否有效和当前可用。这样，当用户通过已保存的支付方式提交订单时，应用程序就不必再次验证。

假设使用已保存支付方式提交订单的POST请求如下所示，其中 `payment_id` 参数指向用户已保存信用卡的ID：

```
POST /new_order
Host: shop.example.com
(POST request body)
item_id=123
&quantity=1
&saved_card=1
&payment_id=1
```

用户也可以每笔订单使用新的信用卡支付。如果用户使用新信用卡支付，该卡将在结账时被验证。假设使用新支付方式提交订单的POST请求如下：

```
POST /new_order
Host: shop.example.com
(POST request body)
item_id=123
&quantity=1
&card_number=1234-1234-1234-1234
```

重申一下，应用程序仅在客户使用新支付方式时验证信用卡号。但应用程序还通过HTTP请求中 `saved_card` 参数的存在来判断支付方式是否为新。因此，恶意用户可以提交带有 `saved_card` 参数和虚假信用卡号的请求。由于支付验证中的这个错误，他们可以使用未经验证的卡无限免费订购商品：

```
POST /new_order
Host: shop.example.com
(POST request body)
item_id=123
&quantity=1
&saved_card=1
&card_number=0000-0000-0000-0000
```

这类应用程序逻辑错误非常普遍，因为这些缺陷无法自动扫描。它们的表现形式太多，而且大多数当前的漏洞扫描器不具备理解应用程序逻辑或业务需求的能力。

## 访问控制失效

我们的信用卡处理示例也可以归类为访问控制失效问题。当应用程序中的访问控制实现不当且可被攻击者绕过时，就会发生访问控制失效。

例如，第10章讨论的IDOR漏洞是应用程序面临的常见访问控制失效问题。但Web应用程序中还有许多其他常见的访问控制失效问题，如果你希望成为一名有效的黑客，就应该了解它们。让我们来看几个。

### 暴露的管理面板

应用程序有时会忽略或忘记锁定敏感功能，如用于监控应用程序的管理面板。开发人员可能错误地认为用户无法访问这些功能，因为它们没有从主应用程序链接，或者它们隐藏在模糊的URL或端口后面。但攻击者如果能够找到它们，通常可以在未经身份验证的情况下访问这些管理面板。例如，即使应用程序 example.com 将其管理面板隐藏在模糊的URL（如 `https://example.com/YWRtaW4/admin.php`）后面，攻击者仍可能通过Google dork或URL暴力破解找到它。

有时应用程序不会为访问其敏感功能的各种方式实施相同的访问控制机制。假设管理面板已得到适当保护，只有具有有效管理员凭据的人才能访问。但如果请求来自机器信任的内部IP地址，管理面板不会要求用户进行身份验证。在这种情况下，如果攻击者能找到允许他们发送内部请求的SSRF漏洞，他们就可以在未经身份验证的情况下访问管理面板。

如果Cookie或请求头是可预测的，攻击者也可能通过篡改它们来绕过访问控制。假设管理面板不要求凭据，只要请求访问的用户在其HTTP请求中出示 `admin=1` Cookie即可。攻击者绕过此控制所要做的就是将Cookie `admin=1` 添加到他们的请求中。

最后，另一种常见的访问控制问题发生在用户可以强制绕过访问控制点进行浏览时。为了理解这意味着什么，假设访问 example.com 管理面板的常规方式是通过URL `https://example.com/YWRtaW4/admin.php`。如果你浏览到该URL，系统会提示你使用凭据登录。之后，你将被重定向到 `https://example.com/YWRtaW4/dashboard.php`，这是管理面板所在的位置。如果应用程序在仪表板页面未实施访问控制，用户可能能够直接浏览到 `https://example.com/YWRtaW4/dashboard.php` 并访问管理面板，而无需提供凭据。

### 目录遍历漏洞

目录遍历漏洞是另一种类型的访问控制失效。当攻击者可以通过操纵用户输入字段中的文件路径来查看、修改或执行他们不应访问的文件时，就会发生这种情况。

假设 example.com 有一个允许用户访问其上传文件的功能。浏览到URL `http://example.com/uploads?file=example.jpeg` 将导致应用程序显示位于 `/var/www/html/uploads/USERNAME/` 的用户上传文件夹中名为 `example.jpeg` 的文件。

如果应用程序未对 `file` 参数实施输入清理，恶意用户可以使用 `../` 序列逃出上传文件夹并读取系统上的任意文件。在Unix系统上，`../` 序列指向当前目录的父目录。例如，攻击者可以使用此请求访问系统上的 `/etc/shadow` 文件：

```
http://example.com/upload?file=../../../../../etc/shadow
```

页面将导航到 `/var/www/html/uploads/USERNAME/../../../../../etc/shadow`，这指向系统根目录下的 `/etc/shadow` 文件！在Linux系统中，`/etc/shadow` 文件包含系统用户的哈希密码。如果运行Web服务器的用户具有查看此文件的权限，攻击者现在也可以查看它。然后他们可以破解此文件中找到的密码，以获得系统上特权用户的访问权限。攻击者还可能获得对配置文件、日志文件和源代码等敏感文件的访问权限。

## 防御措施

你可以通过执行测试来验证应用程序的逻辑是否按预期工作，从而防止应用程序逻辑错误。这最好由既了解组织的业务需求又了解应用程序开发过程的人来完成。你需要详细了解应用程序如何工作、用户如何相互交互、功能如何执行以及复杂流程如何工作。

仔细审查每个流程，查找可能导致安全问题的任何逻辑缺陷。对每个对应用程序安全至关重要的功能进行严格和定期的测试。

接下来，通过多种对策防止访问控制失效问题。首先，在系统上的所有文件和操作上实施细粒度的访问控制策略。实施访问控制策略的代码也应进行潜在绕过的审计。你可以进行渗透测试，尝试在访问策略或其实现中寻找漏洞。确保访问控制策略是准确的。同时，确保访问服务的多种方式具有一致的访问控制机制。例如，无论应用程序是通过移动设备、桌面设备还是API端点访问，都不应有所不同。相同的认证要求（如MFA）应适用于每个单独的访问点。

## 寻找应用程序逻辑错误和访问控制失效

应用程序逻辑错误和访问控制问题是一些初学者最容易发现的漏洞。寻找这些漏洞不涉及篡改代码或构造恶意输入；相反，它需要创造性思维和实验意愿。

### 第1步：了解你的目标

首先了解你的目标应用程序。以普通用户的身份浏览应用程序，以发现功能和有趣的功能。你还可以阅读应用程序的工程博客和文档。你对该应用程序的架构、开发过程和业务需求了解得越多，就越善于发现这些漏洞。

例如，如果你发现应用程序刚刚为其在线商店添加了新的支付选项，你可以首先测试该支付选项，因为新功能通常是其他黑客测试最少的。如果你发现应用程序使用WordPress，你应该尝试访问 `/wp-admin/admin.php`，这是WordPress管理门户的默认路径。

### 第2步：浏览时拦截请求

在浏览站点时拦截请求，并注意敏感功能。跟踪在这些操作期间发送的每个请求。注意敏感功能和访问控制是如何实现的，以及它们如何与客户端请求交互。对于你发现的新支付选项，完成支付需要哪些请求？是否有任何请求参数指示支付类型或将被收取多少费用？在访问 `/wp-admin/admin.php` 的管理门户时，是否发送了任何特殊的HTTP头或参数？

### 第3步：跳出思维定式

最后，运用你的创造力思考绕过访问控制或以其他方式干扰应用程序逻辑的方法。玩弄你拦截的请求，构造不应被授予的请求。如果你修改请求参数中的收费金额，应用程序是否仍会处理交易但收取较低金额？即使你没有礼品卡，你能将支付类型切换为礼品卡吗？你能通过添加特殊Cookie（如 `admin=1`）来访问管理页面吗？

## 提升攻击影响

提升应用程序逻辑错误和访问控制失效的影响完全取决于你发现的缺陷的性质。但一个通用规则是，你可以尝试将应用程序逻辑错误或访问控制失效与其他漏洞结合，以增加它们的影响。

例如，一个允许你访问具有控制台或应用程序部署功能的管理面板的访问控制失效可能导致远程代码执行。如果你能找到Web应用程序的配置文件，你可以搜索与正在使用的软件版本相关的CVE，以进一步入侵应用程序。你还可能在文件中找到可用于访问网络上不同机器的凭据。

虽然SQL注入或存储型XSS等漏洞的影响通常很明确，但攻击者通过应用程序逻辑错误和访问控制失效漏洞能实现什么并不总是显而易见的。思考恶意用户可能如何充分利用这些漏洞，并在报告中详细说明其影响。

## 找到你的第一个应用程序逻辑错误或访问控制失效！

使用你在本章学到的技巧，找到你的第一个应用程序逻辑错误或访问控制失效漏洞：

1. **了解你的目标应用程序**。你对Web应用程序的架构和开发过程了解得越多，就越善于发现这些漏洞。
2. **浏览时拦截请求**，并注意敏感功能。跟踪在这些操作期间发送的每个请求。
3. **运用你的创造力**思考绕过访问控制或以其他方式干扰应用程序逻辑的方法。
4. **思考将你发现的漏洞与其他漏洞结合的方法**，以最大化缺陷的潜在影响。
5. **起草你的报告！** 务必向报告接收者说明恶意用户如何利用此问题。
