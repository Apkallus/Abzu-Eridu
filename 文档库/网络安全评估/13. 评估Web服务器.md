# 第13章 评估Web服务器

Web服务器需要高度的安全保障，因为它们通常暴露在不可信网络中。本章讨论用于测试这些服务器及其启用子系统的策略和工具。对应用框架（如Microsoft ASP.NET和Rails）的评估将在第14章中介绍。

Web服务器、框架和应用的评估与加固可以写满整本书。在此，我提出一个用于指纹识别、调查和确认可用HTTP服务中漏洞的简洁方法论，包括以下步骤：

1.  识别代理机制
2.  枚举虚拟主机和可访问的网站
3.  对每个识别出的站点：
    a.  分析服务器软件和可用子系统
    b.  主动扫描和爬取以识别有用的内容和功能
    c.  攻击暴露的认证机制
    d.  确认服务器软件中的漏洞

Web应用通常通过负载均衡器呈现，因此前两步很重要。考虑图13-1，其中客户端通过TLS连接到负载均衡器，然后负载均衡器根据提供的Host值将请求定向到内部的应用服务器（通过HTTP）。

**图13-1. 通过HTTP 1.1和TLS连接到虚拟主机**

在测试期间，您会遇到以下三种情况之一：

*   直接访问托管单个站点的单台服务器
*   直接访问托管多个站点（虚拟主机）的单台服务器
*   通过代理间接访问多个应用服务器

通过对从每个服务器端点接收到的材料进行主动测试和被动分析，您可以映射并测试可用的Web应用组件。

## 识别代理机制

负载均衡器和反向代理在大型环境中很常见；它们用于在多个后端应用服务器之间分发和定向请求。系统通常支持HTTP 1.1方法（特别是GET、POST和HEAD）。

识别服务器将连接转发到其他地方的一种简单方法是提供一个不带Host字段的HEAD请求，以及另一个带有效Host字段的请求，如示例13-1中使用Akamai基础设施所示。

## 枚举有效主机

大多数Web服务器和反向代理解析HTTP 1.1的Host值并相应地定向请求。获取有效主机名有三种基本方法：

*   客户端提供其环境中使用的名称列表
*   通过Netcraft、Google、DNS等渠道进行开源查询
*   对暴露的Web服务器和应用进行主动测试

主动测试技术包括：

*   网站爬取和HTML解析以识别主机名
*   X.509证书分析，通过TLS获取服务器主机名值
*   分析服务器响应以获取主机名和IP地址详细信息
*   暴力破解有效主机名

图13-3显示了使用Wikto[^1]通过主动爬取识别与barclays.com域关联的主机名。使用Metasploit的`vhost_scanner`模块进行主动暴力破解主机名也很有效，如示例13-2所示。在测试期间，使用更大的字典[^2]可能会产生结果。

## Web服务器分析

掌握了环境中的有效网站列表（即IP地址、协议和主机组合）后，您可以采用手动和自动化的策略来测试每个可用服务器的配置。

包括Apache HTTP Server和Microsoft IIS在内的实现支持许多模块和子系统（例如认证机制、WebDAV和TLS）。有些是服务器软件自带的（例如Apache中的mod_cgi和IIS中的Microsoft ASP.NET支持），尽管许多需要安装和配置。

您可以通过以下方式推断Web服务器的配置：

*   分析对HTTP请求的响应
*   审查请求内容后返回的服务器HTTP头
*   爬取每个站点并分析目录结构、文件名和内容

我将在后续章节中描述这些策略。

### 分析服务器响应

您可以使用HEAD方法为给定的HTTP资源从Web服务器检索状态码和头（本质上执行ping操作）。在返回的头中，您通常会找到服务器软件和底层配置的详细信息。示例13-3演示了向Apache软件基金会的Web服务器发出的HEAD请求。

**示例13-3. 向www.apache.org发出HEAD请求**

```bash
root@kali:~# telnet www.apache.org 80
Trying 140.211.11.131...
Connected to www.apache.org.
Escape character is '^]'.
HEAD / HTTP/1.1
Host: www.apache.org

HTTP/1.1 200 OK
Date: Mon, 11 Aug 2014 21:34:16 GMT
Server: Apache/2.4.10 (Unix) mod_wsgi/3.5 Python/2.7.5 OpenSSL/1.0.1i
Last-Modified: Mon, 11 Aug 2014 21:10:43 GMT
ETag: "9e28-50060fce7b9a5"
Accept-Ranges: bytes
Content-Length: 40488
Vary: Accept-Encoding
Cache-Control: max-age=3600
Expires: Mon, 11 Aug 2014 22:34:16 GMT
Connection: close
Content-Type: text/html; charset=utf-8
```

我们得知服务器在Unix系统上运行Apache 2.4.10，通过mod_wsgi支持Python，通过OpenSSL支持TLS。通过浏览这些子系统的项目页面并搜索NVD，您可以调查已知的安全漏洞。

示例13-4演示了Microsoft IIS 8.5服务器的响应。该站点通过Akamai进行负载均衡，并呈现了有效的Host字段以引发响应。如果服务器不支持HEAD方法，GET请求应随请求内容一起显示头值。

**示例13-4. 向www.microsoft.com发出HEAD请求**

```bash
root@kali:~# telnet www.microsoft.com 80
Trying 134.170.188.84...
Connected to lb1.www.ms.akadns.net.
Escape character is '^]'.
HEAD / HTTP/1.1
Host: www.microsoft.com

HTTP/1.1 200 OK
Cache-Control: no-cache
Content-Length: 1020
Content-Type: text/html
Last-Modified: Mon, 16 Mar 2009 20:35:26 GMT
Accept-Ranges: bytes
ETag: "67991fbd76a6c91:0"
Server: Microsoft-IIS/8.5
X-Powered-By: ASP.NET
Date: Mon, 11 Aug 2014 21:47:49 GMT
```

如果Web服务器返回的`Server`头被修改，您可以通过响应HEAD和OPTIONS请求时字段格式的差异轻松区分Apache、Microsoft IIS和其他Web服务器。

OPTIONS请求应返回给定资源允许的HTTP请求方法。示例13-5显示Apache软件基金会的Web服务器接受对根目录（`/`）的GET、HEAD、POST、OPTIONS和TRACE请求。

**示例13-5. 向www.apache.org发出OPTIONS请求**

```bash
root@kali:~# telnet www.apache.org 80
Trying 192.87.106.229...
Connected to www.apache.org.
Escape character is '^]'.
OPTIONS / HTTP/1.1
Host: www.apache.org

HTTP/1.1 200 OK
Date: Mon, 11 Aug 2014 23:18:15 GMT
Server: Apache/2.4.10 (Unix) OpenSSL/1.0.1i
Allow: GET,HEAD,POST,OPTIONS,TRACE
Cache-Control: max-age=3600
Expires: Tue, 12 Aug 2014 00:18:15 GMT
Content-Length: 0
Content-Type: text/html; charset=utf-8
```

某些方法支持服务器端文件上传和内容修改。例如，对某个资源发出OPTIONS请求时，您可能会发现支持有用的方法，包括PUT和PROPPATCH（分别用于上传内容和修改文件属性）。

### HTTP头审查

对资源的请求通常会返回包含有用HTTP头的响应，如表13-1所列。示例13-6显示了许多年前eBay的Web服务器泄露了内部IP地址信息以及环境中使用的NetApp缓存硬件的详细信息。

**表13-1. 服务器响应中发现的有用头**

| 头                   | 备注                                                     |
|----------------------|----------------------------------------------------------|
| ETag                 | 可用于指纹识别设备固件                                   |
| Content-Location     | 可能泄露主机名或内部IP地址详细信息                       |
| Location             | 重定向时使用，可能指向内部IP或主机名                     |
| Set-Cookie           | 可能泄露负载均衡器和其他系统的详细信息                   |
| Server               | 提供Web服务器软件和子系统的详细信息                      |
| Via                  | 泄露代理或负载均衡器详细信息                             |
| WWW-Authenticate     | 通常通过realm字段提供IP和主机名详细信息                  |
| X-Powered-By         | 详细说明应用框架（例如Microsoft ASP.NET）                |

### 调查支持的HTTP方法

常规的HTTP 1.1方法很少容易受到攻击（例如，代理连接到任意主机，或访问敏感内容）。在测试期间可能遇到的有用方法包括：

*   TRACE
*   PUT和DELETE
*   WebDAV方法

在后续章节中，我将讨论如何在测试期间评估这些方法。

#### TRACE

如果支持TRACE方法，并且Web服务器运行的应用程序容易受到XSS攻击，则可以发起跨站追踪（XST）[^14]攻击以获取用户会话信息。此向量特别有用，因为它可以揭示受`HttpOnly`标志保护的cookie。

相比之下，如果服务器仅运行不处理用户输入的静态网站，则TRACE支持的实际安全影响可以忽略不计。

#### PUT和DELETE

### 已知的Apache Coyote弱点

Apache Coyote是HTTP/1.1连接器（Web服务器），用于代理用户与应用服务器（包括JBoss Application Server、Apache Struts和Catalina）之间的入站连接。Coyote本身是更大Apache Tomcat软件包的一部分（与Catalina servlet容器和其他组件捆绑在一起）；通常通过`Server` HTTP头字段识别，如下所示：

```http
HTTP/1.1 200 OK
Server: Apache-Coyote/1.1
X-Powered-By: Servlet 2.5; JBoss-5.0/JBossWeb-2.1
Accept-Ranges: bytes
ETag: W/"100-1353333077000"
Last-Modified: Mon, 19 Nov 2012 13:51:17 GMT
Content-Type: text/html
Content-Length: 100
Date: Sat, 11 Jul 2015 14:18:13 GMT
```

通过检查其他头（例如`X-Powered-By`和`struts-time`）、设置的cookie、目录结构以及呈现的内容（例如HTML和JavaScript），您可以推断出通过Coyote HTTP连接器暴露的底层应用服务器。

运行过时版本Apache Tomcat软件包的服务器容易受到攻击。NVD列出了影响Coyote的多个拒绝服务条件。影响服务器的可远程利用缺陷列于表13-9。

## Web服务器加固

在加固Web服务器时，您应考虑以下对抗措施：

*   确保服务器软件、库和依赖项都已打上最新补丁。适当的维护可以缓解最严重的缺陷。
*   通过移除不必要的模块和禁用冗余子系统来减少攻击面（例如Apache HTTP Server中的mod_cgi、mod_perl和PHP，以及Microsoft IIS中的WebDAV和ISAPI扩展等组件）。同时考虑禁用认证子系统以消除暴力破解的风险。
*   在您的环境中禁用对不必要HTTP方法的支持（例如PUT、DELETE、TRACE和WebDAV方法）。
*   在Apache HTTP Server中，使用`httpd.conf`中的`Header always unset`[^21]和`ServerSignature off`[^22]指令移除向攻击者泄露有用Web和应用服务器详细信息的HTTP头（例如`Server`、`X-Powered-By`和`X-Runtime`）。
*   如果不存在索引文件（例如`default.asp`、`index.htm`和`index.html`），防止目录索引，以防止Web爬虫和机会主义攻击者识别敏感信息。
*   如果环境中发生应用程序异常（崩溃），不要向公众暴露调试信息。返回不包含任何敏感材料的通用404或500错误页面。

[^21]: Shanison, [“Unset/Remove Apache Response Header – Protect Your Server Information”](http://bit.ly/2aBfNzK), 2012年7月5日。
[^22]: Tarunika Shrivastava, [“13 Apache Web Server Security and Hardening Tips”](http://bit.ly/2aDf2FU), TecMint, 2015年1月7日。
