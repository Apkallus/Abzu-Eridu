# 第 6 章 IP 网络扫描

在识别出感兴趣的 IP 地址块后，将进行主动扫描以绘制网络地图、编录可访问的主机并识别暴露的服务。本章涵盖以下策略：

- 使用 Nmap 执行初始网络扫描
- 进行底层评估以了解网络配置
- 使用 Nmap 和 Metasploit 执行轻度漏洞扫描
- 批量漏洞扫描（使用 Nessus、Qualys 等工具）
- 规避入侵检测和防御机制

然后进行手动测试以调查漏洞、利用已知缺陷并发起暴力密码猜测攻击。作为参考，图 6-1 展示了本章所涵盖的 IP 协议之间的关系。

## 使用 Nmap 进行初始网络扫描

Nmap 可用于 Apple OS X、Microsoft Windows 和 Linux 等平台，支持通过 ICMP、TCP、UDP 和 SCTP 进行 IPv4 和 IPv6 网络扫描。下面介绍如何使用 Nmap 扫描环境。

### ICMP

Nmap 支持通过 IPv4 和 IPv6 进行 ICMP 扫描，以映射更大 IP 块内的子网并从主机获取响应（取决于配置）。以下两种 ICMPv4 消息类型特别有用：

- **类型 8（回显请求）**：由 ping 和其他实用程序用于识别可访问的主机。
- **类型 13（时间戳请求）**：以十进制格式提供来自目标的系统时间信息。

IANA 维护着全面的 ICMPv4 和 ICMPv6 消息类型列表[^1]。近年来许多有用的 ICMPv4 类型已被弃用，包括 17（地址掩码请求）和 37（域名请求）。在遗留网络中，这些类型仍可能产生有用信息。

[^1]: 参见 IANA.org 的“Internet Control Message Protocol version 6 (ICMPv6) Parameters”和“Internet Control Message Protocol (ICMP) Parameters”。

#### 使用 Nmap 进行 ICMPv4 扫描

在 Nmap 中，使用 `-PEPM` 标志通过 ICMPv4 回显、时间戳和子网掩码请求执行主机发现，如示例 6-1 所示。

**示例 6-1. 使用 Nmap 进行 IPv4 网络发现**

```bash
root@kali:~# nmap -PEPM -sP -vvv -n 10.12.5.0/24

Starting Nmap 6.49BETA4 (https://nmap.org) at 2015-09-07 18:22 EDT
Initiating Ping Scan at 18:22
Scanning 256 hosts [3 ports/host]
Completed Ping Scan at 18:23, 7.24s elapsed (256 total hosts)
Nmap scan report for 10.12.5.0 [host down, received no-response]
Nmap scan report for 10.12.5.1
Host is up, received echo-reply ttl 128 (0.012s latency).
Nmap scan report for 10.12.5.16
Host is up, received echo-reply ttl 128 (0.023s latency).
...
```

旧版本的 Nmap 会报告子网和广播地址的多个响应，从而允许映射子网分配和大小。不幸的是，6.49 版本不会这样，因此你必须使用 ping 手动调查网络行为。

#### 使用广播地址

在 IP 网络中，最后一个地址八位组被保留为广播地址（例如 `10.10.5.0/24` 中的 `10.10.5.255`），用于向子网内的所有主机发送流量。示例 6-2 演示了使用 `ping` 命令向 IPv4 广播地址发送 ICMP 回显请求，示例 6-3 显示了如何通过 `255.255.255.255` 广播地址揭示另一个子网（`10.12.0.0/24`）。

**示例 6-2. 使用 ping 识别子网内的主机**

```bash
root@kali:~# ping -b 10.10.5.255
WARNING: pinging broadcast address
PING 10.10.5.255 (10.10.5.255) 56(84) bytes of data.
64 bytes from 10.10.5.79: icmp_seq=1 ttl=64 time=1.01 ms
64 bytes from 10.10.5.10: icmp_seq=1 ttl=64 time=1.79 ms (DUP!)
...
```

**示例 6-3. 通过 ICMP 回显广播识别其他子网**

```bash
root@kali:~# ping -b 255.255.255.255
WARNING: pinging broadcast address
PING 255.255.255.255 (255.255.255.255) 56(84) bytes of data.
64 bytes from 10.10.5.79: icmp_seq=1 ttl=64 time=1.28 ms
64 bytes from 10.12.5.251: icmp_seq=1 ttl=63 time=1.28 ms (DUP!)
...
```

### TCP

Nmap 支持许多 TCP 扫描模式，在执行隐蔽扫描和理解底层网络配置时特别有用。为了识别可访问的服务，应使用基本的 TCP SYN（`-sS`）模式，如示例 6-4 所示。

**示例 6-4. 使用 Nmap 执行 IPv4 TCP SYN 扫描**

```bash
root@kali:~# nmap -sS 10.10.5.10

Starting Nmap 6.49BETA4 (https://nmap.org) at 2015-09-07 18:45 EDT
Nmap scan report for 10.10.5.10
Not shown: 933 filtered ports, 60 closed ports
PORT      STATE SERVICE
80/tcp    open  http
135/tcp   open  msrpc
445/tcp   open  microsoft-ds
3389/tcp  open  ms-wbt-server
49152/tcp open  unknown
49153/tcp open  unknown
49154/tcp open  unknown
```

默认情况下，Nmap 将执行主机发现并识别可访问的主机进行扫描[^2]。在测试加固环境时，应使用 `-Pn` 标志强制扫描每个地址。较慢的时间策略（例如 `-T2`）也很有用，因为激进的策略可能触发 SYN 洪水保护并导致数据包被丢弃。

[^2]: 参见 Gordon “Fyodor” Lyon，“Host Discovery” in *Nmap Network Scanning* (Sunnyvale, CA: Insecure.com LLC, 2009)。

Nmap 为每个端口返回一个状态（`open`、`closed` 或 `filtered`）。图 6-2 至图 6-5 演示了 SYN 探测引发的四种响应变体：SYN/ACK 数据包（表示开放端口）；RST/ACK（表示关闭）；无响应；或 ICMP 类型 3 消息（表示过滤）。

**图 6-3. 关闭端口行为**

**图 6-4. 过滤端口行为（无响应）**

IPv4 防火墙和路由器通常会产生 ICMP 类型 3（目的地不可达）响应，这提供了对网络配置的洞察。表 6-1 列出了常见的 ICMP 类型 3 消息代码[^3]。

### UDP

UDP 的无连接特性意味着服务要么通过**负面扫描**（基于 ICMP 不可达响应推断开放端口），要么通过使用正确格式的数据报以引发服务响应（例如 DNS、DHCP、TFTP 等，列于 `nmap-payloads`[^4]），即**载荷扫描**来识别。

[^4]: 参见 Nmap Payload Database。

ICMP 是一个不可靠的指标，因为注重安全的组织倾向于过滤这些消息，并且大多数操作系统默认对 ICMP 响应进行速率限制。

Nmap 通过 `-sU` 标志结合使用负面扫描和载荷扫描（而不仅仅是单一模式）。这常常会使输出变得模糊，如示例 6-5 所示，其中同时返回了 `open` 和 `open|filtered` 状态。

**示例 6-5. 使用 Nmap 执行 UDP 扫描**

```bash
root@kali:~# nmap -Pn -sU -open -F -vvv -n 10.3.0.1

Starting Nmap 6.46 (http://nmap.org) at 2014-10-27 02:37 UTC
Initiating UDP Scan at 02:37
Scanning 10.3.0.1 [100 ports]
Discovered open port 137/udp on 10.3.0.1
Discovered open port 123/udp on 10.3.0.1
Completed UDP Scan at 02:38, 13.25s elapsed (100 total ports)
Nmap scan report for 10.3.0.1
Scanned at 2014-10-27 02:37:49 UTC for 13s
PORT      STATE         SERVICE
7/udp     open|filtered echo
...
123/udp   open          ntp
137/udp   open          netbios-ns
...
```

详细输出显示，基于对精心构造的载荷的响应，端口 123（NTP）和 137（NetBIOS 名称服务）是开放的。其他端口则基于不可靠的 ICMP 反馈列出。

使用 `-sUV` 标志，你可以主动探测每个 UDP 端口并查看哪些端口响应。然而，对于模糊的 `open|filtered` 端口，以这种方式运行 Nmap 非常缓慢，在测试大型网络时不切实际。

示例 6-6 演示了使用 Nmap 扫描单个主机的五个 UDP 端口，耗时 114 秒。更深入的测试显示端口 53 确实在监听。

**示例 6-6. 进一步探测五个 UDP 端口**

```bash
root@kali:~# nmap -Pn -sUV -open -p53,123,135,137,161 -vvv -n 10.3.0.1

Starting Nmap 6.46 (http://nmap.org) at 2014-10-27 02:53 UTC
...
PORT    STATE         SERVICE    VERSION
53/udp  open          domain     dnsmasq 2.50
123/udp open          ntp        NTP v4
135/udp open|filtered msrpc
137/udp open          netbios-ns Samba nmbd (workgroup: UCOPIA)
161/udp open|filtered snmp
Service Info: Host: CONTROLLER
```

另一种可用于执行 UDP 载荷扫描的工具是 **Unicornscan**[^5]。针对 `10.3.0.1` 目标，结果几乎立即返回：

[^5]: 更多关于 Unicornscan 的信息，请参见 Robert E. Lee 和 Jack C. Louis，“Introducing Unicornscan”，在 Defcon 13 Hacking Conference 上展示，内华达州拉斯维加斯，2005 年 7 月 29-31 日。

```bash
root@kali:~# unicornscan -mU 10.3.0.1
UDP open          domain[   53] from 10.3.0.1  ttl 128
UDP open      netbios-ns[  137] from 10.3.0.1  ttl 128
```

UDP 扫描结果因工具选择和网络条件而异。Nmap 通过 `-sUV` 提供了一个全面的选项，但使用 `-F` 选项（扫描 100 个端口）测试单个主机可能需要 10 多分钟才能完成。

### SCTP

SCTP 与 TCP 和 UDP 并列，如图 6-1 所示。该协议旨在通过 IP 提供电话数据传输，复制了七号信令系统（SS7）的许多可靠性特性，并支撑了一个称为 SIGTRAN 的更大协议族。SCTP 受到包括 IBM AIX、Oracle Solaris、HP-UX、Linux、Cisco IOS 和 VxWorks 在内的操作系统的支持。

#### 数据包格式

每个 SCTP 数据包包含一个头部和关联的块，如图 6-6 所示。源端口和目标端口值为 16 位（范围 0 到 65,535），8 位块类型值列于表 6-2 中。根据类型，块值字段会有所不同。

**图 6-6. SCTP 数据包格式**

**表 6-2. SCTP 块类型**

| ID | 值 | 描述 |
|----|-----|------|
| 0 | DATA | 载荷数据 |
| 1 | INIT | 初始化请求 |
| 2 | INIT ACK | 初始化确认 |
| 3 | SACK | 选择性确认 |
| 4 | HEARTBEAT | 心跳请求 |
| 5 | HEARTBEAT ACK | 心跳确认 |
| 6 | ABORT | 中止请求 |
| 7 | SHUTDOWN | 关闭请求 |
| 8 | SHUTDOWN ACK | 关闭确认 |
| 9 | ERROR | 操作错误 |
| 10 | COOKIE ECHO | 状态 cookie 回显 |
| 11 | COOKIE ACK | Cookie 确认 |
| 12 | ECNE | 显式拥塞通知回显 |
| 13 | CWR | 拥塞窗口减少 |
| 14 | SHUTDOWN COMPLETE | 关闭完成 |

#### Nmap 支持

你可以通过两种数据包类型扫描来发现可访问的 SCTP 服务：

- **INIT**：类似于 TCP SYN 扫描，Nmap（使用 `-sY` 标志）向每个端口发送 SCTP INIT 块。INIT ACK 响应表示端口开放，而 ABORT 块表示端口关闭。

- **COOKIE ECHO**：实现应丢弃发送到开放端口的包含 COOKIE ECHO 块的数据包，并在端口关闭时发送 ABORT 块。这种扫描类型隐蔽，但不能区分开放端口和过滤端口（仅显示关闭的端口）。Nmap 使用 `-sZ` 标志支持此扫描策略。

示例 6-7 演示了通过 IPv6 进行的 Nmap SCTP 扫描。表 6-3 列出了常见的 SCTP 服务。在识别出有效服务后，使用 **sctpscan**[^6] 手动调查（使用 `-t` 标志将 TCP 套接字桥接到特定的 SCTP 服务）。

[^6]: Philippe Langlois，“SCTPscan and SIGTRAN Research Paper”，在 Black Hat Europe 2007 上展示，荷兰阿姆斯特丹，2007 年 3 月 27-30 日。

### 整合所有扫描

详细的评估涉及对范围内每个 IP 地址的所有 65,536 个 TCP 和 SCTP 端口进行扫描，同时测试常见的 UDP 端口（以节省时间）。在测试期间，我尚未在非标准端口上发现 UDP 服务，因此使用 Nmap 的默认服务列表运行 UDP 扫描就足够了。

#### IPv4 扫描

对于 IPv4，我首选的方法是首先运行三次 Nmap 扫描以识别可访问的主机。你可以使用 `-iL` 标志将目标网络列表从文件加载到 Nmap 中。

```bash
nmap -T4 -Pn -n -sS -F -oG tcp.gnmap 192.168.0.0/24
nmap -T4 -Pn -n -sY -F -oG sctp.gnmap 192.168.0.0/24
nmap -T4 -Pn -n -sU -p53,69,111,123,137,161,500,514,520 -oG udp.gnmap 192.168.0.0/24
```

这些扫描生成的输出文件扩展名为 `gnmap`。特别注意 UDP 结果非常重要，因为它们可能包含误报。如果 UDP 数据集看起来噪点很多（即所有主机都报告有开放端口），直接忽略它。当你对这些文件的内容满意后，使用 `grep` 和 `awk` 生成精简的目标列表，如下所示：

```bash
grep open *.gnmap | awk '{print $2}' | sort | uniq > targets.txt
```

然后应将此列表输入到随后的四次扫描中：

1. **常见服务的快速 TCP 扫描**
   ```bash
   nmap -T4 -Pn -open -sS -A -oA tcp_fast -iL targets.txt
   ```

2. **所有端口的 TCP 扫描（加上指纹识别和通过默认 NSE 脚本的测试）**
   ```bash
   nmap -T4 -Pn -open -sSVC -A -p0-65535 -oA tcp_full -iL targets.txt
   ```

3. **所有端口的 SCTP 扫描**
   ```bash
   nmap -T4 -Pn -open -sY -p0-65535 -oA sctp -iL targets.txt
   ```

4. **常见服务的 UDP 扫描**
   ```bash
   nmap -T3 -Pn -open -sU -oA udp -iL targets.txt
   ```

`-oA` 标志将为每种扫描类型生成多个输出文件，包括易于解析的 `gnmap` 文件和人类可读的完整文本文件。

当使用 Nmap 运行大型并发扫描时，建议使用合理的时间策略以避免使你的互联网连接饱和。你还可以使用 **screen** 实用程序（在 Kali Linux、Apple OS X 和其他平台中都有）将各个扫描会话移至后台。这样，即使你与服务器断开连接，它们也会继续运行。

这些扫描模式不会对暴露的网络服务进行深入分析。漏洞扫描和深度测试将在本章后面介绍，同时介绍商业支持工具的使用。

#### IPv6 扫描

第 4 章和第 5 章描述了 IPv6 地址的枚举。在收集到 IPv6 网络前缀列表后，使用与 IPv4 描述相同的分阶段方法识别主机和服务（即扫描运行常见网络服务的主机，然后对该子集进行全面扫描）。在对大型 IPv6 网络进行 TCP 扫描时，我建议减少端口列表以提高速度，从 `-F`（100 个常见端口）减少到 `-p22,25,53,80,111,139,443`。

准备好目标列表（例如 `targets.txt`）后，运行与之前相同的扫描，这次使用 `-6` 标志通过 IPv6 进行扫描：

```bash
nmap -6 -T4 -Pn -open -sS -A -oA ipv6_tcp_fast -iL targets.txt
nmap -6 -T4 -Pn -open -sSVC -A -p0-65535 -oA ipv6_tcp_full -iL targets.txt
nmap -6 -T4 -Pn -open -sY -p0-65535 -oA ipv6_sctp -iL targets.txt
nmap -6 -T3 -Pn -open -sU -oA ipv6_udp -iL targets.txt
```

## 底层 IP 评估

通过精心构造探测数据包并检查响应，你可以执行以下操作：

- 对目标主机或网络设备的操作系统进行指纹识别

### 操纵 TTL 以逆向工程 ACL

Nmap 基于网络响应或无响应来识别过滤器。更深入的评估使用 **Firewalk** 进行，它操纵数据包的 TTL 字段，从一定距离（即比给定网关多一跳）扫描目标。

示例 6-11 演示了 Firewalk 通过特定网关（`gw.test.org`）测试六个 TCP 端口到目标 `www.test.org` 的过程。该功能也已通过 `firewalk` NSE 脚本移植到 Nmap。

**示例 6-11. 运行 Firewalk**

```bash
$ firewalk -n -S21,22,23,25,53,80 -pTCP gw.test.org www.test.org
Firewalk 5.0 [gateway ACL scanner]
Firewalk state initialization completed successfully.
TCP-based scan.
Ramping phase source port: 53, destination port: 33434
Hotfoot through 217.41.132.201 using 217.41.132.161 as a metric.
Ramping Phase:
 1 (TTL  1): expired [192.168.102.254]
 2 (TTL  2): expired [212.38.177.41]
 3 (TTL  3): expired [217.41.132.201]
Binding host reached.
Scan bound at 4 hops.
Scanning Phase:
port  21: A! open (port listen) [217.41.132.161]
port  22: A! open (port not listen) [217.41.132.161]
...
```

Firewalk 计算到网关的距离，并向目标发送 TTL 多一跳的数据包。基于响应，该工具能够映射网关的过滤策略。例如：

- 如果收到 ICMP 类型 11，代码 0（传输中 TTL 超时）消息，则数据包已通过并随后生成了响应。
- 如果数据包被静默丢弃，则可能是在网关上完成的。
- 如果收到 ICMP 类型 3，代码 13（通信被管理性禁止）消息，则表示正在使用诸如路由器 ACL 之类的过滤器。

如果数据包被静默丢弃，这并不一定意味着到目标主机和端口的流量被过滤。一些防火墙知道数据包即将过期，并且无论策略是否允许，都会发送过期消息。

Firewalk 适用于真正的 IP 路由环境，而不是使用网络地址转换（NAT）的环境。Dave Goldsmith 和 Mike Schiffman 的论文详细描述了这种测试方法[^16]。

[^16]: Dave Goldsmith and Mike Schiffman，“Firewalking”，Cambridge Technology Partners 白皮书，1998 年 10 月。

## 批量漏洞扫描

在测试大型异构环境时，NSE 的功能有些有限。安全专业人员通常依赖功能丰富的商业支持工具来对 IP 网络执行深度自动评估。四种流行的扫描实用程序如下：

- **Nessus**
- **OpenVAS**
- **Qualys**
- **Rapid7 Nexpose**

这些工具通过 IPv4 和 IPv6 执行主机发现、端口扫描和暴露服务评估。OpenVAS 可免费使用，并包含在 Kali Linux 中，在线教程详细介绍了扫描仪的设置和使用[^17]。

[^17]: 具体请参见 Alexandre Borges 的教程“Fast Configuration of OpenVAS on Kali Linux 1.5”以及 NetSecNow 的视频“Setting up OpenVAS on Kali Linux + Config and Scanning Howto + Free Startup Script”（YouTube，2013 年 5 月 28 日发布）。

批量扫描实用程序的操作带来了两个挑战：

- 默认的主机发现和扫描策略会引入覆盖范围的空白
- 输出通常包含误报和没有实际意义的数据

为了获得最佳覆盖范围，我建议同时使用 Nmap（用于执行初始网络扫描和通过 NSE 进行批量测试）和以全面模式运行的扫描实用程序（扫描 65,536 个端口、常见 UDP 服务等）。最重要的是，比较两个过程的输出将让你识别出由扫描配置和主机发现设置引起的任何空白。

关于批量扫描输出中的信噪比，我建议将材料导出为 CSV 或 XML 格式并对其进行解析。例如，Qualys 和其他工具为每个报告的问题提供 CVSS[^18] 分数；这使得你可以忽略影响较低的结果。

[^18]: 参见 Common Vulnerability Scoring System (CVSS)。

## IDS 和 IPS 规避

注重安全的组织使用 IDS 和 IPS 技术来被动监控和主动阻止可疑的网络流量。你可以在第 3 层和第 4 层采用三种特定的策略来干扰或绕过检测：

- 插入传感器可见但目标主机忽略的数据
- 分片数据包，使传感器忽略它们（但稍后会被重组）
- 修改数据包数据（例如，附加数据和设置特定标志）

Kali Linux 中的 **SniffJoke** 实用程序[^19] 通过插件支持前两种方法，这些插件定义了如何操纵出口流量。根据网络配置和目标主机的操作系统，你可以采用不同的策略。Nmap 的规避功能[^20] 包括数据包分片和数据修改，用于掩盖端口扫描（一个受欢迎的功能是 `--data-length`，它向每个数据包附加数据，欺骗基于签名的检测系统）。第三个值得考虑的工具是 **Stonesoft Evader**[^21]，它可以通过分片和修改出口数据包来绕过 Palo Alto Networks 设备的检测。

[^19]: 参见 Not In My Name，“SniffJoke 0.4”，SlideShare.net，2011 年 5 月 30 日。
[^20]: Gordon “Fyodor” Lyon，“Firewall/IDS Evasion and Spoofing”，载于 *Nmap Network Scanning* (Sunnyvale, CA: Insecure.com LLC, 2009)。
[^21]: 更多信息，请参见 Dameon D. Welch-Abernathy，“Who’ll Stop the Evaders?” PhoneBoy Blog，2016 年 3 月 5 日。

### 使用 SniffJoke 进行数据插入和加扰

在 SniffJoke 中，数据操纵和插入攻击被称为 **hacks** 和 **scrambles**，它们作为插件实现，如项目文档[^22] 所述。

[^22]: 参见 GitHub 上的 `tcp-hacks-and-scrambling.txt`。

**hacks** 的示例包括插入虚假载荷（导致传感器解析未被目的地处理的会话数据）、提供虚假的序列信息（导致传感器丢失状态并解析错误数据）以及注入虚假的信令信息（通过插入被目的地忽略但导致传感器认为会话已结束或新会话已建立的 FIN、RST 或 SYN 数据包）。

数据包可以通过多种方式进行畸形化和分片——关键在于采用一种方法，使得数据包被传感器处理但被目标主机忽略，或者反之亦然。这些方法中的许多依赖于传感器和目标主机的 IP 栈实现（通过识别不匹配并利用它来获利）。SniffJoke 采用的加扰策略侧重于生成被目标主机忽略的数据包，如下所示：

- **设置数据包的错误校验和**：高吞吐量环境中的传感器通常出于性能原因不计算数据包校验和。如果是这种情况，恶意内容会被传感器解析，但在目标主机收到时被忽略。

- **使用不常见的 IP 和 TCP 选项**：传感器可能会忽略在 IP 和 TCP 头中设置了某些标志和选项的数据包，而目标主机在收到时会接受该数据包。

SniffJoke 文档中未强调的一个重要主题是数据包分片和段重叠。攻击者可以使用这些策略来规避和绕过 IDS 和 IPS 机制，因为分片数据包被传感器重组的方式可能与目标主机不同[^23]。

[^23]: 参见 Wikipedia 上的“IP fragmentation attack”。

### 配置和运行 SniffJoke

如果你能够在受某个 IDS 或 IPS 机制保护的 Web 服务器上放置一个精心制作的 PHP 脚本，你可以使用 **sniffjoke-autotest** 来识别有效的规避技术并创建一系列本地配置文件。

该 PHP 脚本如下，来自 `sniffjoke-autotest` GitHub 页面[^24]：

```php
<?php
  if(isset($_POST['sparedata'])) {
    for($x = 0; $x < strlen($_POST['sparedata']); $x++)
    {
      if( is_numeric($_POST['sparedata'][$x]) )
        continue;
      echo "bad value in $x offset";
      exit;
    }
    echo $_POST['sparedata'];
  }
?>
```

[^24]: 参见 GitHub 上的 `sniffjoke-autotest`。

将脚本放置在 Web 服务器上后，运行 `sniffjoke-autotest`：

```bash
root@kali:~# sniffjoke-autotest -l home -d /usr/var/sniffjoke/ -s \
http://www.example.org/test.php -a 192.168.0.10
```

该实用程序在向 `http://www.example.org/test.php`（IP 地址为 `192.168.0.10`）发送请求后，将创建一个配置（称为 **location**，本例中使用标签 `home`）。执行后，SniffJoke 主目录（Kali Linux 中为 `/usr/var/sniffjoke/home/`）将包含多个配置文件，如表 6-7 所列。

**表 6-7. 给定位置的 SniffJoke 配置文件**

| 文件名 | 描述 |
|--------|------|
| `ipblacklist.conf` | 包含要忽略的目标 IP 地址 |
| `iptcp-options.conf` | 包含有效的 IP 和 TCP 选项组合 |
| `ipwhitelist.conf` | 列出要由 SniffJoke 覆盖的目标 IP 地址 |
| `plugins-enabled.conf` | 列出有效的插件和加扰组合 |
| `port-aggressivity.conf` | 定义数据包注入特定流的频率 |
| `sniffjoke-service.conf` | SniffJoke 服务的配置文件 |

编辑配置文件以定义在规避范围内的 IP 地址和网络服务。当你对配置满意后，使用特定的位置标签（本例中为 `home`）运行 SniffJoke：

```bash
root@kali:~# sniffjoke --location home
```

该程序在后台运行并更新默认网关，以便所有数据包都通过它进行路由以进行操纵。执行后，`sniffjokectl` 实用程序用于与服务接口交互，如下所示：

## 网络扫描回顾

使用自动和手动测试方法来绘制目标环境地图并识别暴露的网络服务。以下是有效的网络测试技术列表：

- **初始网络扫描**：使用 Nmap 识别可访问的主机，然后对所有 TCP 和 SCTP 端口以及常见的 UDP 端口进行全面扫描。使用 `-A` 标志执行操作系统和网络服务指纹识别。

- **底层 IP 评估**：使用带有 `-O` 标志的 Nmap 对 TCP 序列号、IP ID 和 TCP 时间戳值进行采样。你可以通过使用 Firewalk、Hping3 和 Scapy 构造带有特定标志的数据包来调查底层配置。

- **使用 Nmap 进行漏洞扫描**：NSE 支持对特定服务（包括 DNS、HTTP、MongoDB、Microsoft SQL Server、Oracle 和 SNMP）进行测试。尽管这些测试绝非详尽无遗，但它们通常提供有用的信息。

- **批量漏洞扫描**：包括 Nessus、OpenVAS、Qualys 和 Rapid7 Nexpose 在内的工具根据 PCI DSS 和其他要求执行广泛评估。漏洞扫描器通常会产生大量数据和误报。因此，你应该将输出与 Nmap 结果进行交叉检查以验证适当的覆盖范围。

- **IDS 和 IPS 规避**：通过使用 SniffJoke 和 Nmap 进行数据包分片和注入，可以规避包括 IDS 和 IPS 在内的网络安全机制。

## 网络扫描对抗措施

以下是在考虑加固网络设备和服务器以减少未经授权的网络扫描有效性时应使用的对抗措施清单：

- 在网络边缘过滤入站 ICMP 消息以防止 ping 扫描。
- 在边界路由器和防火墙处过滤出站 ICMP 不可达（类型 3）消息，以防止端口扫描和对你的 ACL 进行逆向工程。
- 确保关键网络设备（核心交换机、边缘路由器和防火墙）上运行的软件已更新到最新补丁。这对于缓解拒绝服务和可用性问题非常重要。
- 审查网络设备的日志记录和审计配置，以确保自动漏洞扫描和大量恶意数据不会因日志基础设施过载而导致拒绝服务。
- 在执行扫描和探测练习时，使用 Nmap 和 SniffJoke 评估网络设备处理分片和畸形数据包的方式。大量不良数据可能耗尽安全机制，导致它们“失败开放”。
- 通过对你自己的 IP 地址空间发起网络扫描，了解你自己的网络配置和公开可访问的服务。令人惊讶的是，很少有组织执行简单的端口扫描练习，并被机会主义攻击者打个措手不及。
