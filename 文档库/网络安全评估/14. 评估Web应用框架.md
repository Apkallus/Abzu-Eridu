# 第14章 评估Web应用框架

应用框架解释并执行用Java、PHP、Python和Ruby等语言编写的代码。框架可以捆绑在更大的Web服务器软件包中（例如Microsoft IIS中的ASP.NET），也可以使用不同的应用和Web服务器组件运行（例如运行在Apache Tomcat之上的JBoss Application Server）。图14-1展示了流行的应用和Web服务器配置（注意：这些框架中的许多也可以使用其他Web服务器呈现）。

在许多情况下，开发人员负责应用程序安全，而IT运维人员负责基础设施安全。如果双方忽视加固这些层之间的框架组件，可能存在可利用的缺口。

本章描述应用框架的调查与利用。

## 框架与数据存储分析

第13章描述了通过以下方式进行应用框架指纹识别：

*   审查HTTP头（例如`Server`和`X-Powered-By`）
*   分析连接时设置的cookie（主要是会话变量）
*   分析文件名、扩展名和目录结构
*   审查内容中的元数据（例如HTML和JavaScript）

可用于进一步调查和分析应用框架组件的自动化工具包括BlindElephant[^1]、WAFP[^2]和clusterd[^3]。前两个工具对用PHP编写的内容管理系统（CMS）平台（例如WordPress、Joomla和Drupal）进行指纹识别，而clusterd可以对应用服务器（包括Apache Tomcat、JBoss和Adobe ColdFusion）进行指纹识别。

示例14-1演示了针对drupal.org运行的BlindElephant。该工具需要平台名称来进行指纹识别[^4]（例如`confluence`、`drupal`、`wordpress`、`joomla`）。根据CMS的不同，您应随后使用特定工具，如CMS Explorer[^5]、WPScan[^6]和OWASP Joomla Vulnerability Scanner[^7]来识别可利用的漏洞。

## 理解常见缺陷

在未经加固的应用框架组件中存在可远程利用的漏洞。严重的可远程利用缺陷源于软件缺陷和缺乏安全控制，可能导致以下后果：

*   未授权访问管理接口
*   信息泄露和敏感数据暴露
*   任意文件上传和代码执行

通过减少攻击面和维护软件（即及时打补丁），您可以缓解并解决大多数问题。应用服务器和框架组件中的已知缺陷将在后续章节中详述。

## Apache Tomcat

Tomcat由Coyote（代理入站连接的HTTP连接器）、Catalina（Java servlet容器）和Jasper（JSP引擎，为Catalina编译代码以供处理）组成。这些组件中可远程利用的缺陷将在后续章节中详述。

包括JBoss和Apache Struts在内的应用服务器也使用Coyote作为Web服务器。在测试期间，您会发现此类服务器返回的HTTP头为`Server: Apache-Coyote/1.1`。为避免将应用服务器误标为Apache Tomcat（即Catalina），您应通过审查返回的内容（包括HTTP头和cookie）并扫描特定文件的存在来识别底层容器。

### Manager应用程序

manager应用程序通过`/manager/html`暴露管理界面。需要凭证，常见的默认用户名/密码组合包括`admin/（空）`、`cxuser/cxuser`、`j2deployer/j2deployer`[^11]、`ovwebusr/OvW*busr1`[^12]、`tomcat/tomcat`和`vcx/vcx`。

### 利用DefaultActionMapper

一个特别严重的漏洞涉及Struts中的DefaultActionMapper。[^26]该机制旨在执行操作，如以下JSP片段所示：

```jsp
<s:form action="foo">
  ...
  <s:submit value="Register"/>
  <s:submit name="redirect:http://www.google.com/" value="Cancel"/>
</s:form>
```

如果用户点击取消按钮，将请求以下URI：

*   `/foo.action?redirect:http://www.google.com/`

Struts 2.3.15及更早版本中的DefaultActionMapper未正确清理输入，这意味着在请求操作文件并提供有效的redirect或action参数时，可以注入OGNL表达式（导致服务器端代码执行）（例如`/foo.action?action:%{3*4}`，如果服务器评估该表达式，将返回“12”）。

您可以使用以下两种结构与DefaultActionMapper交互：

*   `/struts2-blank/example/X.action`
*   `/struts2-showcase/employee/save.action`

Google V8 JavaScript引擎的`Math.random()`函数中存在一个细微缺陷，导致碰撞和Node.js应用程序中的其他问题，正如Betable CTO Mike Malone[^43]所述。Node.js Crypto API包含`crypto.randomBytes()`，在生成随机数时您应优先使用它而不是原生的V8 PRNG函数。

## 应用框架安全检查清单

在加固Web应用服务器时，您应考虑以下对抗措施：

*   确保应用框架组件（包括依赖项和间接暴露的元素）已打上最新补丁，以缓解已知缺陷。
*   仔细处理和考虑仓库和版本控制系统中的秘密。例如，您可以使用Rails应用程序的`secret_token`值在服务器端执行代码。
*   不要将管理界面或特权功能（例如phpMyAdmin和Django管理控制台）暴露给不可信网络。
*   如果可能，将公开暴露的Web应用组件与管理功能解耦，以防止通过CSRF、持久性XSS或社会工程进行权限提升。

Apache HTTP Server建议：

*   使用`httpd.conf`中的`Header always unset`指令[^45]移除向攻击者泄露有用Web和应用服务器详细信息的HTTP头（例如`X-Powered-By`和`X-Runtime`）。[^46]
*   考虑使用`mod_security`提供针对常见攻击类别（包括XSS和命令注入）的全面保护。

[^45]: Shanison, [“Unset/Remove Apache Response Header – Protect Your Server Information”](http://bit.ly/2aBfNzK), 2012年7月5日。
[^46]: 您可以对 [Nginx](http://bit.ly/2axsluF) 应用类似指令。
