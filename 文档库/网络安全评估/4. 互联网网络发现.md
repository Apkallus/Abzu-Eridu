# 第 4 章 互联网网络发现

本章描述了扮演基于互联网的攻击者角色时所采取的最初步骤。一个有能力的对手会使用开放资源来绘制组织的网络地图并识别其用户。以下是其中三种资源：

- 网络搜索引擎和网站（例如 Google、Netcraft 和 LinkedIn）
- WHOIS 注册机构
- 可访问的 DNS 服务器

这些探测中的大部分是间接的，向包括 Google、公共 WHOIS 和 DNS 服务器在内的网站发送流量。然而，有两种策略会向目标网络发送流量，具体如下：

- 探测目标自己的 DNS 服务器
- 通过 SMTP 进行网络探测

通过初步侦察，你可以识别潜在的弱点。与知名的 Web、应用服务器和邮件端点相比，外围系统通常是不安全的。

侦察过程是迭代的，在发现新信息（例如域名或办公地点）时会重复执行枚举任务。评估工作的约定范围定义了边界，有时可能包括第三方。Target 公司在 2013 年遭受了严重入侵，据称一名供应商的 VPN 凭据被用于访问内部网络，导致 7000 万客户记录暴露[^1]。

[^1]: Brian Krebs，“Target Hackers Broke in via HVAC Company”，Krebs on Security，2014 年 2 月 5 日。[链接](https://krebsonsecurity.com/2014/02/target-hackers-broke-in-via-hvac-company/)

## 查询搜索引擎和网站

搜索引擎编录了有用的信息。Google 和其他网站提供了高级搜索功能，使攻击者能够清晰地了解他们计划稍后攻击的网络。

特别是，以下几类数据常常被暴露：

- 办公地点的物理地址
- 联系方式，包括电子邮件地址和电话号码
- 内部电子邮件系统和路由的详细信息
- DNS 布局和命名规范
- 驻留在公共可访问服务器上的文件

关于此列表中的最后一项，Offensive Security 维护着 **Google Hacking Database**[^2]，其中包含可揭示易受攻击 Web 服务器上敏感材料的搜索细节。也有整本书专门讨论该主题，包括《Google Hacking for Penetration Testers》（Syngress，2015）。以下各节详细介绍了攻击者可以用来映射网络和识别有用内容的流行策略。

[^2]: 参见 Offensive Security 的 Exploit Database 存档中的“Google Hacking Database (GHDB)”。[链接](https://www.exploit-db.com/google-hacking-database)

### Google 搜索

攻击者使用 Google 通过其高级搜索面板收集有用信息。搜索可以被精炼以包含或排除某些关键词，或针对特定文件格式、特定互联网域或网页的特定部分（例如页面标题）中的关键词进行命中。表 4-1 列出了有用的 Google 搜索指令。

通过 Google 找到的公开可用材料（例如 Microsoft Office 格式和 PDF 文件）中的元数据也可以被解析以揭示用户名和客户端软件详细信息，如 **Metagoofil** 工具所演示的那样。

**表 4-1. Google 搜索指令**

| 指令 | 示例 | 描述 |
|------|------|------|
| `intext` | `intext:password filetype:xlsx` | 显示页面文本中的命中结果 |
| `intitle` | `intitle:"index of /backup"` | 显示页面标题中的命中结果 |
| `inurl` | `inurl:dyn_sensors.htm` | 显示页面 URL 中的命中结果 |
| `filetype` | `filetype:log intext:password` | 返回特定文件类型的结果（例如日志文件中的密码） |
| `site` | `site:edu filetype:key intext:private` | 显示特定域内的结果（例如 EDU 顶级域中的 RSA 私钥） |

#### 获取 VPN 配置文件

一些组织公开发布 VPN 系统的配置文件和密钥。**Cisco 配置文件（PCF）** 包含 IPsec VPN 客户端变量，包括以下内容：

- VPN 服务器端点地址
- 明文凭据（组名和密码）
- 加密凭据（混淆的组密码）

表 4-2 列出了可用于发现 Cisco VPN、OpenVPN 和 SSH 配置材料及密钥的 Google 搜索字符串。图 4-4 展示了一个搜索，揭示了美国学术机构的 PCF 文件。在测试期间，更改 `site` 变量以包含范围内的每个域。

**表 4-2. 揭示 VPN 和 SSH 配置文件的搜索字符串**

| 技术 | 示例查询字符串 |
|------|----------------|
| Cisco VPN | `filetype:pcf site:edu grouppwd` |
| OpenVPN | `filetype:ovpn site:tk`<br>`filetype:key site:edu +client` |
| SSH | `filetype:key site:edu +id_dsa`<br>`filetype:id_rsa` |

图 4-4 中显示的许多暴露的 PCF 文件包含一个 3DES 加密的密码（`enc_GroupPwd`）。一个设计缺陷意味着用于解密密码的密钥包含在密文中，因此加密机制在很大程度上只是混淆。

以下网站提供即时解密这些密码的工具：

- Cisco VPN 客户端密码解码器
- Cisco VPN 客户端密码破解器

**图 4-4. 通过 Google 识别 Cisco PCF 文件**

Maurice Massar 发布了 `cisco-decrypt.c`[^3]，可以在基于 Unix 的环境中本地构建和运行（需安装 `libgpg-error` 和 `libgcrypt`[^4]），如示例 4-1 所示。

**示例 4-1. 在 Apple OS X 中构建和执行 cisco-decrypt**

```bash
$ wget http://bit.ly/2aAs1IM
$ gcc -Wall -o cisco-decrypt cisco-decrypt.c $(libgcrypt-config --libs --cflags)
$ ./cisco-decrypt 992C9F91B9AF94528891390F09C783805E33FDBB1C6146556CDADAD4A06D
secret
```

[^3]: `cisco-decrypt.c` 可从 http://bit.ly/2aAs1IM 获取。
[^4]: 两者均可从 https://www.gnupg.org/download/ 获取。

## 域名 WHOIS

在测试期间，你可以查询域名注册机构以获取有关目标组织注册的域名 useful 信息。撰写时有多种顶级域（TLD）和相关注册机构，包括通用 TLD 和国家代码 TLD。ICANN 和 IANA 在以下位置维护注册机构列表：

- gTLD 注册机构
- ccTLD 注册机构

这些注册机构提供以下信息：

- 管理联系人详细信息（姓名、电子邮件地址和电话号码）
- 与目标组织相关的办公地点的邮寄地址
- 每个域名的权威名称服务器详细信息

以下是一些可用于执行域名 WHOIS 查询的工具：

- `whois` 命令行客户端
- 每个域名注册机构维护的 Web 界面
- Hurricane Electric（HE）、DomainTools 和其他第三方服务

## IP WHOIS

区域互联网注册机构（RIR）提供与 IP 网络分配相关的 useful 信息。IP WHOIS 数据库对象定义了互联网空间的哪些区域注册给哪些组织，包括路由信息和联系详细信息。

互联网地址属于不同的地理区域。表 4-3 列出了你可以查询以收集 useful 信息（包括运维人员姓名、IP 网络块详细信息和物理办公地点）的相应 RIR。

**表 4-4. 区域互联网注册机构**

| 名称 | 区域 | 网站 |
|------|------|------|
| ARIN | 北美 | http://www.arin.net |
| RIPE | 欧洲 | http://www.ripe.net |
| APNIC | 亚太 | http://www.apnic.net |
| LACNIC | 拉丁美洲和加勒比 | http://www.lacnic.net |
| AFRNIC | 非洲 | http://www.afrnic.net |

每个区域 WHOIS 数据库都包含与该特定区域相关的信息。例如，RIPE 数据库不包含在美洲发现的对象的详细信息。

## DNS 查询

你可以使用命令行实用程序（例如 `nslookup` 和 `dig`）查询名称服务器。自动化工具也用于对可访问的名称服务器执行反向扫描和正向猜测攻击。

表 4-5 列出了测试期间名称服务器提供的 useful DNS 资源记录。RFC 1035 详细说明了除 AAAA IPv6 地址和 SRV 服务定位器记录之外的所有记录的底层机制和用法。

**表 4-5. 有用的 DNS 资源记录**

| 记录 | 描述 | 揭示内容 |
|------|------|----------|
| SOA | 授权起始 | 创建 DNS 区域的源主机 |
| NS | 名称服务器 | 给定域的权威 DNS 服务器名称 |
| A | 地址（IPv4） | 主机名的 IPv4 地址 |
| AAAA | 地址（IPv6） | 主机名的 IPv6 地址 |
| PTR | 指针 | 给定 IPv4 或 IPv6 地址的主机名 |
| CNAME | 规范名称 | CNAME 是其别名的主机名 |
| MX | 邮件交换 | 给定域的邮件服务器 |
| HINFO | 主机信息 | 主机的操作系统或其他信息 |
| SRV | 服务定位器 | 域内的应用服务端点，包括 Kerberos、LDAP、SIP 和 XMPP |
| TXT | 文本字符串 | 包括 SPF 和 DKIM 字段的材料，根据配置用于提供安全性 |

### 正向 DNS 查询

大多数网络应用程序使用 DNS 记录。两种常见场景是网页浏览（即通过 A 或 CNAME 记录将域解析为 Web 服务器 IP 地址）和发送邮件（即通过呈现有效的 MX 记录将消息发送给域内的用户）。

### DNS 区域传输技术

组织出于负载平衡和容错原因使用多个名称服务器。区域传输通过 TCP 端口 53 执行，以将当前 DNS 区域材料传播到支持该操作的其他名称服务器。

区域文件包含与特定域和 IP 块相关的 DNS 记录。配置错误的服务器会接受来自不受信任源（例如公共互联网）的传输请求，你可以利用这一点来绘制给定网络的地图。

### 正向 DNS 猜测

如果可用的名称服务器不允许区域传输，你应该采用主动猜测策略来识别有效的 DNS 地址记录，包括：

- 使用 A 记录请求进行字典攻击
- NSEC 和 NSEC3 记录枚举

## 枚举技术回顾

以下是对基于互联网的查询技术及其应用的概述：

- **Web 搜索**：使用 Google、Netcraft、Shodan、LinkedIn、PGP 密钥服务器和其他站点，针对已知域名和 IP 块进行搜索，以识别人员、主机名、域名以及驻留在暴露 Web 服务器上的有用数据。

- **WHOIS 查询**：查询域名和 IP 注册机构，以检索与目标网络和域名相关的网络块、路由和联系详细信息。IP WHOIS 查询提供有关保留网络块大小（在后续执行侵入性网络扫描时有用）和 AS 号详细信息的信息。

- **BGP 枚举**：将 AS 号与 BGP 站点交叉引用，以枚举该 AS 下关联的 IP 块，然后将这些详细信息反馈到其他查询路径（如 DNS 或进一步的 WHOIS 查询）。

- **DNS 查询**：查询可访问的名称服务器以枚举域、子域、主机名和非公开 IP 地址详细信息。配置错误的 DNS 服务器还会提供列出子域、主机名、设备操作系统平台和内部网络详细信息的区域文件。

- **SMTP 探测**：向目标域的不存在账户发送邮件，通过分析邮件系统的响应（包括中继服务器和内容过滤设备）来映射内部网络空间。

## 枚举对抗措施

使用以下对抗措施清单，有效配置面向互联网的系统，使其不向对手泄露敏感信息：

- 通过禁用不包含 `index.html` 或类似文件（例如 Microsoft IIS 下的 `default.asp`）的目录的目录索引来加固 Web 服务器，并在外围服务器（即你不想被搜索引擎索引的服务器）上使用 `robots.txt` 指令以防止内容索引。

- 不要依赖 `robots.txt` 指令来保护敏感的 Web 服务器内容。

- 在 WHOIS 数据库和 TLS 证书中使用通用的、集中的网络管理联系详细信息，以防止针对 IT 部门的社会工程和 war dialing 攻击生效。

- 配置名称服务器以不允许向不受信任的主机进行 DNS 区域传输，并从互联网主动测试你的网络（即对 TCP 和 UDP 端口 53 进行端口扫描）以识别 rogue 名称服务器。

- 精简 DNS 区域文件，使得不必要的信息不被披露（主要是非公开 IP 地址和主机名详细信息），并且 DNS 猜测攻击无效。理想情况下，仅在绝对需要时使用 PTR 记录（用于 SMTP 邮件服务器和其他需要双向解析的关键系统）。

- 配置 SMTP 服务器在遇到问题时（例如不存在的邮箱）不发送 NDN，这将防止攻击者枚举内部邮件服务器和配置。

- 考虑并审查你的 IPv6 网络和 DNS 配置（如果有）。
