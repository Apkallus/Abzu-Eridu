## FTP

文件传输协议（FTP）提供远程文件系统访问，通常用于 Web 应用程序的维护。服务器使用两个端口：TCP 端口 21（入站服务器控制端口，处理来自客户端的 FTP 命令）和 TCP 端口 20（出站数据端口，用于向客户端传输数据）。文件传输通过控制端口（21）进行协调，其中使用 PORT 等命令通过出站数据端口发起数据传输。

TLS 通常用于包装 FTP（即 FTPS）或通过 STARTTLS 命令提供传输安全。TLS 实现中的已知漏洞在第 11 章中描述。

FTP 服务容易受到以下类别的攻击：

- 暴力密码猜测
- 匿名浏览和利用软件缺陷
- 认证后的漏洞利用（需要特定权限）

## TFTP

TFTP 使用 UDP 端口 69，不需要身份验证——客户端使用 RFC 1350 中概述的数据报格式从服务器读取和写入。由于协议本身的缺陷（即缺乏身份验证和没有传输安全），在公共互联网上很少见到 TFTP 服务器。然而，在大型内部网络中，TFTP 用于向 VoIP 话机和其他设备提供配置文件和 ROM 映像。

TFTP 服务器可通过以下攻击类别被利用：

- 从服务器获取材料（例如包含秘密的配置文件）
- 绕过控制覆盖服务器上的数据（例如替换 ROM 映像）
- 通过溢出或内存损坏缺陷执行代码

Kali Linux 中的 `tftp` 实用程序用于手动连接 TFTP 服务器并发出读取（`get`）和写入（`put`）请求。该协议不提供列出目录内容的方法，因此必须知道精确的文件名。

Nmap 的 `tftp-enum` 脚本使用常见文件名字典发出读取请求，通常能揭示有用的内容。Metasploit 包含一个类似的暴力破解模块[^7]。示例 7-3 演示了对可用服务器运行 Nmap，以及使用 `tftp` 客户端检索文件（本例中为 `sip.cfg`）。

[^7]: Metasploit `tftpbrute` 模块。

## SSH

SSH 服务为包括嵌入式设备和基于 Unix 的主机在内的系统提供加密访问。通常向用户公开的三个子系统如下：

- **安全 Shell（SSH）**：提供命令行访问
- **安全复制（SCP）**：允许用户发送和检索文件
- **安全 FTP（SFTP）**：提供功能丰富的文件传输

默认使用 TCP 端口 22 公开 SSH 及其子系统。SSH 还支持网络连接的隧道和转发；因此，你可以将其用作 VPN 来安全地访问资源。

SSH 协议的工作方式如下：

- 使用 Diffie-Hellman 密钥交换建立共享秘密
- 客户端和服务器均使用伪随机函数（例如 SHA-1 或 SHA-256）从共享秘密派生三对密钥（每方一对）：
  - 两个初始化向量（IV）值
  - 两个加密密钥
  - 两个签名密钥
- 服务器将其公钥以及一个随机签名值发送给客户端
- 客户端验证随机值的签名（认证服务器）
- 服务器对客户端进行身份验证
- 认证后，建立通道以提供对资源的访问

SSH 服务容易受到以下类别的攻击：

- 暴力密码猜测
- 由于私钥泄露或密钥生成弱点而被授予访问权限
- 远程匿名利用已知软件缺陷（无需凭据）
- 认证后利用已知缺陷，导致权限提升

许多缺陷的实际利用依赖于启用或使用某些功能。因此，调查和了解服务配置非常重要。

第 11 章详细介绍了这些算法和特性，因为它们也在 TLS 中使用。SSH 中可利用的协议弱点源于以下方面：

- **使用不安全群组进行密钥交换**：示例 7-7 列出了 `diffie-hellman-group1-sha1` 作为受支持的密钥交换算法，它使用固定的 1024 位参数（也称为群组）。针对一篇研究论文[^13]，Cisco 建议避免使用该群组[^14]。论文作者描述了国家行为体通过离散对数预计算解密使用 768 位和 1024 位群组协商的 SSH 会话的可能性。Gotham Digital Science 博客上的文章[^15]提供了详细信息和一个测试弱群组支持的实用程序。

[^13]: Adrian David 等人，“Imperfect Forward Secrecy: How Diffie-Hellman Fails in Practice”，第 22 届 ACM SIGSAC 计算机与通信安全会议论文集，科罗拉多州丹佛，2015 年 10 月 12-16 日。
[^14]: 参见 Cisco.com 上的“Next Generation Encryption”，2012 年 4 月。
[^15]: Fabian Foerg，“SSH Weak Diffie-Hellman Group Identification Tool”，Gotham Digital Science Blog，2015 年 8 月 3 日。

### 不安全生成的主机密钥

如果 RSA 或 DSA SSH 主机密钥对生成不安全（例如使用熵不足的 PRNG[^25]），攻击者可以计算私钥，并通过中间人攻击冒充合法的服务器端点。

[^25]: 参见 GitHub 上的 `debian-ssh`。

RSA 公钥由两个整数组成：指数 `e` 和模数 `n`。模数是两个选定素数（`p` 和 `q`）的乘积。私钥是解密指数 `d`，如下所示：

```
d = e⁻¹ mod (p - 1)(q - 1)
```

如果攻击者知道 `n` 的因式分解，他们可以计算任何公钥 `(e, n)` 的私钥。当 `p` 和 `q` 未知时，最有效的已知方法是将 `n` 分解为两个素数以计算私钥 `d`。

当找到两个共享单个素数（无论是 `p` 还是 `q`）的不同 RSA 模数 `(n₁, n₂)` 时存在漏洞——攻击者可以计算最大公约数并发现另一个素数（例如，如果共享 `p`，则发现 `q₁` 和 `q₂`）[^26]。通过扫描互联网，该团队能够攻破在线 SSH 服务器使用的 0.03% 的 RSA 主机密钥和 1% 的 DSA 密钥。

[^26]: 该方法详见 Nadia Heninger 等人，“Missing Your Ps and Qs: Detection of Widespread Weak Keys in Network Devices”，第 21 届 USENIX 安全研讨会论文集，华盛顿州贝尔维尤，2012 年 8 月 8-10 日。

## DNS

第 4 章描述了通过 DNS 进行枚举和网络映射的策略。名称服务器使用两个端口来满足请求：UDP 端口 53 用于处理常规请求（即将名称解析为 IP 地址，反之亦然），TCP 端口 53 用于可靠地发送大量数据，如 DNS 区域文件。

DNS 服务容易受到以下类别的攻击：

- 拒绝服务，限制名称服务可用性
- 通过服务器软件缺陷导致的内存损坏和代码执行
- 缓存投毒和破坏，损害名称服务的完整性

要调查配置，首先对服务进行指纹识别，然后枚举对递归和其他功能的支持，如下文所述。

### 测试递归支持

递归是 DNS 的一项基本功能，名称服务器代表客户端转发请求。内部名称服务器通常支持递归；然而，面向互联网的名称服务器不应响应来自不受信任来源的递归查询。公开可访问的名称服务器支持递归可能导致拒绝服务，因为来自伪造源的 UDP 查询会导致流量被放大并发送到任意位置[^33]。

[^33]: 更多信息，请参见 US-CERT.gov 上的“Alert (TA13-088A) DNS Amplification Attacks”，2013 年 3 月 29 日。

如果 UDP 源端口或 TXID 值可预测，支持递归的服务器也存在缓存投毒风险[^34]。示例 7-17 演示了使用 Nmap 的 `dns-recursion`、`dns-random-srcport` 和 `dns-random-txid` 脚本来评估递归支持和采样随机性。

[^34]: 参见 CVE-2008-1447。

**示例 7-17. 使用 Nmap 测试 DNS 递归配置**

```bash
root@kali:~# nmap -sSUV -p53 --script dns-recursion,dns-random-srcport,dns-random-txid \
192.168.208.2

Starting Nmap 6.46 (http://nmap.org) at 2014-12-16 14:17 UTC
Nmap scan report for 192.168.208.2
PORT   STATE  SERVICE VERSION
53/udp open   domain  Microsoft DNS
|_dns-random-srcport: GREAT: 7 queries in 0.6 seconds from 7 ports with std dev 10785
|_dns-random-txid: GREAT: 11 queries in 24.6 seconds from 11 txids with std dev 17480
|_dns-recursion: Recursion appears to be enabled
```

## SNMP

简单网络管理协议（SNMP）服务通常运行在可管理的交换机、路由器和服务器操作系统（例如 Microsoft Windows Server 和 Linux）上，用于监控目的。SNMP 通过在 UDP 数据报中向端口 161 提供有效的团体字符串进行访问。大多数服务器配置有两个团体字符串：一个提供对 SNMP 管理信息库（MIB）的只读访问，另一个提供读写访问。

MIB 是对象标识符（OID）值的层次结构，如示例 7-20 所示。在本例中，我们使用 SNMP 版本 1 和团体字符串 `public` 连接到 `192.168.0.42`。

**示例 7-20. 通过 SNMP 获取 MIB**

```bash
root@kali:~# snmpwalk -v 1 -c public 192.168.0.42
.1.3.6.1.2.1.1.1.0 = STRING: "Cisco Internetwork Operating System Software IOS (tm) C837 
Software (C837-K9O3Y6-M), Version 12.3(2)XC2, EARLY DEPLOYMENT RELEASE SOFTWARE (fc1)
Synched to technology version 12.3(1.6)T
Technical Support: http://www.cisco.com/techsupport
Copyright (c"
iso.3.6.1.2.1.1.2.0 = OID: .1.3.6.1.4.1.9.1.495
iso.6.1.2.1.1.3.0 = Timeticks: (749383984) 86 days, 17:37:19.84
iso.3.6.1.2.1.1.4.0 = "admin@localhost"
iso.3.6.1.2.1.1.5.0 = STRING: "pipex-gw.trustmatta.com"
iso.3.6.1.2.1.1.6.0 = "4th floor"
```

Kali Linux 中的 SNMP 实用程序不会将 OID 条目解析为人类可读的值。要启用此支持，请使用以下命令下载 MIB 数据并覆盖 `/etc/snmp/snmp.conf` 中的指令：

```bash
apt-get install snmp-mibs-downloader
download-mibs
echo "" > /etc/snmp/snmp.conf
```

然后 `snmpwalk` 实用程序将为每个值提供描述，如下所示：

```bash
SNMPv2-MIB::sysDescr.0 = STRING: Cisco Internetwork Operating System Software IOS (tm) C837 
Software (C837-K9O3Y6-M), Version 12.3(2)XC2, EARLY DEPLOYMENT RELEASE SOFTWARE (fc1)

Copyright (c

DISMAN-EVENT-MIB::sysUpTimeInstance = Timeticks: (749894097) 86 days, 17:39:01.14
SNMPv2-MIB::sysContact.0 = STRING: admin@localhost
SNMPv2-MIB::sysName.0 = STRING: pipex-gw.trustmatta.com
SNMPv2-MIB::sysLocation.0 = STRING: 4th floor
```

服务器可能支持 SNMP 协议版本 1、2 或 3，如表 7-15 所述。使用 `snmpwalk` 时，使用 `-v` 标志指定协议。SNMPv3 服务器也可以运行在 TCP 端口 161 上，并使用 TLS 提供传输安全。

**表 7-15. SNMP 协议版本**

| 版本 | 认证 | 传输安全（可选） |
|------|------|------------------|
| 1 | 团体字符串 | 无 |
| 2 | 团体字符串 | 无 |
| 3 | 用户名和密码，使用 MD5 或 SHA-1 哈希 | 168 位 3DES 或 256 位 AES |

### 利用 SNMP

SNMP 服务容易受到以下类别的远程攻击：

- 通过 SNMPv3 枚举用户
- 暴力猜测团体字符串和用户密码值
- 通过读取 SNMP 数据暴露有用信息（低权限）
- 通过写入 SNMP 数据进行利用（高权限）
- 利用软件实现缺陷，导致意外后果（例如特权远程代码执行）

以下各节讨论具体策略。

#### 通过 SNMPv3 枚举用户名

要查询运行版本 3 的可访问 SNMP 服务并枚举用户名，请安装 SNMP MIBS 包并下载 Rory McCune 的 `snmpv3enum.rb` 脚本，如下所示：

```bash
apt-get install snmp-mibs-downloader
download-mibs
wget http://bit.ly/2ccg7cj
wget http://bit.ly/2cch18I
chmod 755 snmpv3enum.rb
```

脚本就位后，使用默认用户名列表发起攻击：

```bash
root@kali:~# ./snmpv3enum.rb -i 10.0.0.5 -u usernames
valid username : snmpAdmin on host : 10.0.0.5
```

#### 通过写入 SNMP 破坏设备

Metasploit 包含两个模块[^42][^43]，在获得 SNMP 写入访问权限后，可用于读取运行配置并将文件上传到 Cisco 设备。两者都启动一个 TFTP 服务器，并覆盖目标 MIB 中的值以引发文件上传或下载（要求目标主机可访问 TFTP 服务）。

## LDAP

轻量级目录访问协议（LDAP）服务通常运行在 Microsoft Active Directory、Exchange 和 IBM Domino 服务器上。在 Active Directory 中，LDAP 服务称为全局目录。表 7-18 列出了支持 LDAP 服务的各个端口。许多实现使用的当前协议是 LDAP 3.0。

**表 7-18. LDAP 服务端口**

| 端口 | 协议 | TLS | 名称 | 描述 |
|------|------|-----|------|------|
| **TCP** | **UDP** | | | |
| 389 | ● | ● | – | ldap | LDAP |
| 636 | ● | – | ● | ldaps | LDAP over TLS |
| 3268 | ● | – | – | globalcat | Microsoft 全局目录 |
| 3269 | ● | – | ● | globalcats | LDAP over TLS |

LDAP 是一个开放协议，通过 IP 提供目录信息服务。目录服务提供有关网络中用户、系统、网络、服务和应用程序的信息。

暴露的 LDAP 服务器容易受到以下类别的远程攻击：

- 通过匿名绑定泄露信息
- 暴力密码猜测
- 认证后修改 LDAP 目录中的数据
- 利用 LDAP 服务器软件缺陷（有或无凭据）

我将在后续章节中讨论这些策略。在此之前，让我们先了解 LDAP 协议[^46] 及其在认证、操作和目录结构方面的特性。

[^46]: 参见 RFC 4511。

### LDAP 认证

客户端在连接服务器时使用两种认证方法之一[^47]：

[^47]: 参见 RFC 4513。

- **简单（Simple）**：简单认证在 LDAP 绑定请求中发送明文凭据。如果执行匿名绑定操作，则不提供凭据。

- **SASL**：简单认证和安全层（SASL）[^48] 提供对 DIGEST-MD5 和 CRAM-MD5 等认证机制的支持。

[^48]: 参见 RFC 4422。

**表 7-19. SASL 认证机制**

| 机制 | 说明 |
|------|------|
| CRAM-MD5 | MD5 质询-响应认证机制[^49]。此方法容易受到已知明文攻击，包括 Cain & Abel 在内的工具可以在嗅探质询-响应数据后破解密码 |

### LDAP 目录结构

目录由 X.500 属性组成[^54]。在 LDAP 层次结构中，四个属性定义了父域、组织、组织单位（OU）和对象（例如用户或单个系统），如表 7-21 所列，并在图 7-4 中展示。

[^54]: 参见 RFC 4519。

**表 7-21. LDAP 中使用的 X.500 属性**

| 属性 | 描述 | 示例 |
|------|------|------|
| DC | 域组件 | `dc=example,dc=com` |
| O | 组织 | `o=Example LLC` |
| OU | 组织单位名称 | `ou=Marketing` |
| CN | 通用名称 | `cn=John Smith` |

在 LDAP 中，**可分辨名称（DN）** 是到对象的完整路径。以下是图 7-4 中 LDAP DN 的几个例子：

```
cn=John West,ou=Engineering,dc=example,dc=com
cn=Sally Stevens,ou=Sales,dc=example,dc=com
```

**图 7-4. LDAP 目录结构**

其他属性（例如 `telephoneNumber` 和 `roomNumber`）可以定义与各个对象相关的值。你可以以这种方式定义用户密码哈希、命令 shell、UID 值和其他变量。Microsoft Active Directory 中的结构和属性通常包括以下内容：

- 域、用户、组和 OU
- 组策略对象（附加到 OU 以强制执行特定策略）
- 系统（即工作站、服务器和网络设备）
- 服务器应用程序和功能
- 站点和网络（用于 Microsoft Exchange 中的邮件路由）

## Kerberos

Kerberos[^61] 是 Microsoft Windows 网络和基于 Unix 环境中使用的认证协议。该协议的一个优点是用户密码不用于与各个服务进行认证；而是使用由密钥分发中心（KDC）生成的加密票据。

[^61]: 参见 RFC 4120。

KDC 提供认证和票据授予服务，如图 7-5 所示。这些机制向客户端提供两种类型的票据：票据授予票据和各个服务票据。在 Microsoft 网络中，登录域时 KDC 提供票据授予票据（TGT）。在基于 Unix 的环境中，调用 `kinit`。TGT 用于请求服务票据，服务票据又用于访问各个应用程序。图 7-5 中显示的消息在表 7-24 中描述。

**图 7-5. Kerberos KDC 认证和票据发放**

**表 7-24. Kerberos 消息**

| 消息 | 描述 |
|------|------|
| KRB_AS_REQ | 对 TGT 的认证服务请求，包含主体名称和时间戳，使用主体的长期密钥（作为共享秘密）加密 |
| KRB_AS_REP | 使用共享秘密解密时间戳后，认证服务释放一个包含会话密钥的 TGT，使用主体的长期密钥加密；TGT 还包含一个使用 KDC 主密钥加密的票据块 |
| KRB_TGS_REQ | 票据授予服务请求，其中 TGT 与对特定服务的访问请求组合在一起，使用会话密钥加密 |
| KRB_TGS_REP | 验证请求后，票据授予服务生成一个共享会话密钥（将在客户端和服务器之间使用），使用目标服务器的长期密钥加密一份副本，并创建一个服务票据发送给客户端（使用原始会话密钥加密） |
| KRB_AP_REQ | 客户端向目标服务器提供服务票据，服务器使用其长期密钥解密并获得共享会话密钥，解密并验证票据本身，然后授予访问权限 |
| KRB_AP_REP | 用于相互认证场景的可选消息，服务器使用共享会话密钥加密时间戳值 |
| KRB_ERROR | 用于从服务器向客户端发送错误消息，并使用特定加密诱导认证，或通信异常 |
| KRB_SAFE | 用于传输带有校验和的数据（提供完整性） |
| KRB_PRIV | 用于传输带有校验和和加密的数据 |
| KRB_CRED | 用于将票据转发给其他主体 |

该协议是无状态的，票据描述了用户权限。因此，如果 KDC 使用的主密钥被泄露，攻击者可以创建任意票据，称为**黄金票据**[^62]。如果主体密码、密钥或票据被泄露，攻击者也可以使用它们生成票据并访问服务。Alva Duckwall 和 Benjamin Delpy 在 Black Hat USA 2014 的演讲[^63] 详细介绍了这些场景，Fulvio Ricciardi 对 Kerberos 协议的描述是一个极好的资源[^64]。

[^62]: Balazs Bucsay，“Mimikatz — Golden Ticket”，Rycon.hu，2014 年 1 月 24 日。
[^63]: Alva Duckwall and Benjamin Delpy，“Abusing Microsoft Kerberos - Sorry You Guys Don’t Get It”，在 Black Hat USA 2014 上展示，内华达州拉斯维加斯，2014 年 8 月 2-7 日。
[^64]: 参见 ZeroShell Net Services 上的 Kerberos 协议及其实现。

Kerberos 术语使用**主体**来描述 realm 内的实体（即用户、系统、应用程序和服务）。为生成票据，KDC 和主体使用共享秘密，这些是长期密钥，通常从密码派生（例如用户或计算机账户密码）。

### Kerberos 密钥

Kerberos realm 中使用的长期密钥如下：

- **KDC 主密钥（认证服务主体长期密钥）**：在 Windows Active Directory 服务器中，这些值从本地 `krbtgt` 账户的密码派生。使用的哈希函数通常是 RC4-HMAC，并且越来越多地使用 AES256-CTS-HMAC，具体取决于操作系统（参见表 7-28）。

- **主体长期密钥**：客户端和服务器都使用与 KDC 共享的长期密钥，用于加密票据。密钥通常从密码派生，与 KDC 主密钥一样，可以使用不同的函数进行哈希。

密钥强度、生成和安全处理至关重要。例如，KDC 主密钥很少更改，如果攻击者泄露了该值（通过访问 KDC 的内存、文件系统甚至备份文件），他们可以生成黄金票据。在 Windows 环境中，主体长期 RC4 密钥是无盐的 NTLM 哈希，容易受到暴力攻击。

你可以配置 Kerberos 使用 X.509 证书和 PKI。使用证书颁发机构，通过 PKINIT[^65] 进行预认证，这可以缓解与离线破解用户密码相关的问题。

[^65]: 参见 RFC 4556 和 Microsoft Developer Network 上的“[MS-PKCA]: Public Key Cryptography for Initial Authentication (PKINIT) in Kerberos Protocol”。

### 票据格式

Kerberos 票据采用 ASN.1 编码，并包含一个使用认证服务长期密钥（即 KDC 主密钥）或各个服务主体（如网络服务或应用程序）长期密钥加密的票据块。Microsoft 的实现包含一个**特权属性证书（PAC）** 数据结构，该结构经过签名，并包含用户名、域、用户和组详细信息。

#### Microsoft PAC 字段

PAC 位于加密票据中的授权数据内。PAC 中有用的字段位于 `KERB_VALIDATION_INFO` 结构下，包括用户名、域和组成员详细信息。Microsoft 的规范文档描述了这些字段[^66]。

[^66]: 参见 Microsoft Developer Network 上的“[MS-PAC]: Privilege Attribute Certificate Data Structure”。

#### 票据块加密和签名

TGT 使用用户主体长期密钥进行加密以防止窃听。解密 TGT 后，获得会话密钥，并缓存加密的票据块。然后使用会话密钥加密服务票据，服务票据包含一个加密的票据块。在 Microsoft 实现中，票据块和 PAC 数据结构按照表 7-26 所述进行加密和签名。

**表 7-26. Microsoft Kerberos 票据块加密和签名**

| | 票据块加密 | PAC 签名（KDC） | PAC 签名（服务器） |
|--|------------|----------------|-------------------|
| 票据授予票据（TGT） | `krbtgt` | `krbtgt` | `krbtgt` |
| 服务票据 | `target` | `krbtgt` | `target` |

在 TGT 中，Microsoft KDC 服务器对 PAC 数据结构进行签名（使用 RC4-HMAC 或 HMAC-SHA1-96，使用 128 位或 256 位密钥，取决于配置）以防止篡改。由于目标服务器必须验证服务票据中的 PAC，KDC 对 PAC 签名两次，首先使用认证服务的长期密钥，然后使用目标服务器的长期密钥。

一旦泄露 KDC 认证服务（`krbtgt`）的长期密钥，你可以生成任意的 TGT（黄金票据）。掌握了服务主体（目标）的长期密钥后，你可以修改服务票据以生成包含伪造 PAC 结构的**白银票据**[^67]。

[^67]: 详情请参见 Ben Lincoln，“Mimikatz 2.0—Silver Ticket Walkthrough”，Beneath the Waves Blog，2014 年 12 月 18 日。

### 本地攻击

具有本地系统或网络访问权限的攻击者可以采用以下策略：

- 被动网络嗅探 Kerberos 认证服务请求
- 通过 MITM 进行主动降级攻击以削弱加密
- 泄露 KDC 主密钥，导致生成黄金票据
- 泄露用户密钥和票据，这些可以被修改、传递和重用

在 Windows 域中进行横向移动的攻击者采用这些策略，因为大多数环境出于兼容性原因支持弱加密类型。密钥捕获和重用也使攻击者能够维持访问（直到密钥更改）。

#### 被动网络嗅探

你可以使用 Arne Vidstrom 的 `kerbcrack`[^68] 收集认证服务请求并暴力破解账户密码。Cain & Abel 和 John the Ripper 也支持捕获和离线暴力破解通过 Kerberos 获得的密码哈希。

[^68]: 参见 NTSecurity 上的 `kerbcrack`。

#### 主动降级和离线暴力破解

MIT Kerberos 1.7、Windows Server 2008 和 Windows Vista 支持在 Kerberos 认证中使用 56 位 DES 加密。Windows 7 还支持出口级 40 位 RC4。因此，你可以通过 MITM 降级传输安全，并破解账户密码。

表 7-28 详细列出了常见的 Microsoft Windows 加密类型（称为 Etypes），这些是用于生成长期密钥和执行认证的哈希函数。在 Windows Server 2008 R2 中，默认禁用对弱出口级 RC4 和 DES Etypes 的支持；然而，支持此类 Etypes 的客户端可能被诱骗向恶意 KDC 发送 `KRB_AS_REQ` 材料，然后这些材料容易受到暴力攻击。Microsoft 企业支持目录服务博客包含一篇有用的文章，演示了 Kerberos 网络数据包捕获和推荐的加固步骤[^69]。

[^69]: 参见 RFC 6113。

## VNC

Olivetti & Oracle 研究实验室于 1998 年发布了远程帧缓冲（RFB）协议规范。虚拟网络计算（VNC）是使用该协议提供远程访问主机的应用程序。该实验室于 2002 年关闭，促使开发人员成立 RealVNC Ltd. 并发布后续的 RFB 协议规范。

RFB 服务通常监听在 TCP 端口 5900 上，但也可以使用其他端口（例如 4900 和 6000）。该协议可通过任意编码类型进行扩展，这些编码类型支持 UltraVNC 和 TightVNC 等软件包中的文件传输和压缩。连接后，服务器会提供一个协议字符串，如示例 7-34 所示。常见的协议版本包括 000.000、003.003、003.007、003.008、003.889、004.000 和 004.001。

**示例 7-34. 识别支持的 RFB 协议**

```bash
root@kali:~# telnet 121.163.21.135 5900
Trying 121.163.21.135...
Connected to 121.163.21.135.
Escape character is '^]'.
RFB 004.000
```

连接并提供一个版本字符串以协商连接后，服务器返回一个安全类型值。表 7-31 列出了常见的类型。最常见的是 VNC 认证，这是一种仅需要密码的 DES 质询-响应机制。

**表 7-31. RFB 安全类型**

| 0 | 无效的安全类型（连接关闭） |
| 1 | 不需要认证（连接已建立） |

## 常见网络服务评估回顾

执行以下操作以发现常见网络服务中的漏洞：

- **指纹识别**：使用 Nmap 版本扫描（`-sV`）和手动技术审查横幅材料并对可用服务进行指纹识别。同时考虑交叉引用操作系统版本和配置以推断某些实现的版本（例如 OpenSSH 5 与 6）。

- **枚举支持的功能**：使用手动评估技术和 Nmap 脚本列出给定服务支持的功能（例如 DNS 递归或 LDAP 匿名绑定）。某些实现缺陷的成功利用依赖于对特定功能的支持，因此调查非常重要。

- **识别和确认已知弱点**：查阅本章中的表格以及其他来源（例如 NVD），以识别暴露的网络服务中的已知弱点。这些可能包括提供有用数据的信息泄露缺陷。

- **暴力密码猜测**：使用 Hydra 和其他工具对支持认证的暴露服务（包括 FTP、SSH、Telnet、SNMP、LDAP 和 VNC）执行暴力密码猜测。根据你正在测试的系统类型定制字典文件，以减少测试时间和网络流量。

- **调查获取的材料**：FTP、TFTP、SNMP、LDAP 和 Unix RPC 服务通常会生成有用的材料，你可以将这些材料提供给进一步的测试过程（例如可在密码猜测攻击中使用的用户名）。审查和调查可用材料，确保你最大限度地发挥其价值。

## 服务加固和对抗措施

在加固网络服务时，请考虑以下对抗措施：

- **尽可能减少网络攻击面**。例如，不要同时提供 FTP、SFTP 和 SCP 进行文件传输，而选择仅使用 SCP。此外，通过禁用对不必要功能的支持，减少每个网络服务中的暴露逻辑和应用程序攻击面。

- **维护服务器软件包和库（例如 NTP、BIND 和 OpenSSL）**，以消除剩余暴露攻击面中的已知弱点。

- **禁用 Telnet、FTP、SNMP、VNC 和其他缺乏加密传输安全的管理协议**。远程维护操作应通过安全的认证连接（例如 VPN 或 SSH）或通过封闭的管理网络提供。

- **如果使用 SNMP，请确保使用强凭据**。考虑使用 ACL 将 SNMP 访问限制在可信源，并防止未经授权的 TFTP 文件传输到你的设备。

- **了解你服务中暴露的认证机制，并确保配置了审计**，以便暴力密码猜测攻击能够被突出显示。

- **加固 SSH 服务器**，如下所示：
  - 强制使用协议版本 2.0，并禁用向后兼容性，以减轻 SSH 1.0 中的已知弱点。
  - 根据你运行的服务器软件和需要支持的客户端，精简支持的密钥交换机制和密码[^82]。
  - 通过为用户禁用密码认证，并强制使用一次性密码（OTP）、公钥或多因素认证（通过 Google Authenticator、Duo Security 等平台）来缓解暴力密码猜测问题。

[^82]: stribika，“Secure Secure Shell”，2015 年 1 月 4 日。

- **加固 DNS 服务器**：
  - 禁用来自不受信任来源的递归查询支持。
  - 确保区域文件不包含多余或敏感信息。

- **加固 Kerberos 服务器**：
  - 禁用对弱 HMAC 算法的支持（特别是 56 位 DES、40 位出口级 RC4 和 128 位 RC4）。现代操作系统支持 AES128 和 AES256，应强制执行。
  - 在 Microsoft 环境中，考虑强制使用最高的域功能级别。Windows Server 2012 引入了一系列改进，包括 Kerberos armoring，这尤其可以缓解降级攻击。
