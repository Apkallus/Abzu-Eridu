# 第 9 章 评估邮件服务

邮件服务在互联网和专用网络中中继消息。攻击者通常利用邮件协议形成的通道来攻击内部系统。本章详细介绍了可用于识别可用邮件服务缺陷的策略——包括服务识别、启用选项的枚举以及已知弱点的测试。

## SMTP

SMTP 服务器（称为消息传输代理或 MTA）使用 Sendmail 和 Microsoft Exchange 等软件包传输电子邮件。图 9-1 展示了一个典型配置，其中使用内容过滤机制来清洗电子邮件。

**图 9-1. 处理互联网邮件的 SMTP 服务器**

在这种情况下，入站邮件首先发送给托管安全服务提供商（MSSP）以隔离恶意软件、垃圾邮件和其他威胁。MSSP 将处理后的邮件中继到组织的外部 SMTP 接口（通常是执行进一步过滤的防火墙或设备），然后该接口将邮件投递到内部邮件服务器。

整个链路中设备和邮件服务器的配置非常重要。例如，网络过滤不足可能允许攻击者与组织的外部 SMTP 接口建立会话并绕过 MSSP。如果许多服务器无法将电子邮件中继到预期收件人，它们还会发送未送达通知（NDN）消息，从而揭示软件和网络配置细节。

通过 SMTP 发起的攻击可能有不同的目标和目的。例如，攻击者可以直接利用暴露服务中的缺陷（例如利用 Microsoft Exchange 中的已知错误），或者使用 SMTP 作为传递机制，向更大系统中的易受攻击组件（例如运行内部邮件服务器的防病毒引擎）提供恶意内容。

### 服务指纹识别

在准备好邮件服务器和有效域的列表后，你可以对每个 SMTP 端点进行指纹识别，并识别已启用的子系统和功能。邮件服务器软件详细信息可通过横幅抓取、行为分析和被动审查 NDN 获得。

连接时呈现的 SMTP 横幅通常描述实现。如果横幅被混淆或未提供足够详细信息，`HELP` 命令可能会提供有意义的反馈。示例 9-1 演示了对 SMTP 服务的手动指纹识别，随后使用 Nmap 进行扫描。

**示例 9-1. 对 SMTP 端点进行指纹识别**

```bash

10 mxb-00082601.gslb.pphosted.com.
root@kali:~# telnet mxa-00082601.gslb.pphosted.com 25
```

### 映射 SMTP 架构

如果邮件服务器无法将邮件路由到收件人，它们通常会向源发送详细的 NDN 消息。这给了攻击者推断有效邮箱的机会（这些邮箱稍后会在钓鱼活动中使用）。第二个好处是 NDN 消息包含有用的环境详细信息，包括：

- 主机名和 IP 地址
- 邮件服务器软件版本和配置
- 底层操作系统和服务器配置
- 邮件服务器的物理位置（基于时区和格式）
- 服务器之间的 TLS 配置和支持

RFC 5321 规定 SMTP 标头不得被邮件服务器软件更改。查看消息源时，我们发现每个邮件服务器都添加了一个 `Received` 标头，如示例 9-2 所示。在本例中，我提取了针对 `blah@nintendo.com` 的 NDN 消息中的标头。

#### 识别防病毒和内容检查机制

NDN 消息还可能包含由内容过滤机制（要么是独立的硬件设备，要么是邮件服务器上运行的软件）生成的标头。示例 9-4 演示了 Barracuda Networks 内容过滤器在 `barracuda.noa.nintendo.com` 添加的标头。

**示例 9-4. 内容过滤器插入的 SMTP 标头**

```

X-Barracuda-Start-Time: 1411818632
X-Barracuda-Encrypted: AES256-SHA
X-Barracuda-URL: http://barracuda.noa.nintendo.com:80/cgi-mod/mark.cgi
X-Virus-Scanned: by bsmtpd at noa.nintendo.com
X-Barracuda-Spam-Score: 0.00
X-Barracuda-Spam-Status: No, SCORE=0.00 using per-user scores of TAG_LEVE= L=2.0
```

通过发送不同内容的消息，你可以使用响应来逆向工程内容过滤策略和防病毒配置。例如，你可以通过在电子邮件中发送 EICAR 测试文件[^2] 来强制识别和警报恶意内容。

### 可远程利用的缺陷

SMTP 服务器软件可能运行在设备上（例如 Barracuda Spam Firewall 和 Cisco IronPort）、轻量级 MTA（例如 qmail 和 Exim）以及功能丰富的邮件服务器平台上（例如 Microsoft Exchange 和 Sendmail）。

根据撰写时的 NVD，Barracuda Spam Firewall、Cisco IronPort 或 Proofpoint 平台中尚无已知的通过 SMTP 向量可利用的缺陷。然而，这些产品中存在许多 Web 应用程序缺陷（例如 XSS、命令注入和数据暴露），可通过暴露的 HTTP 和 HTTPS 接口利用。

### 用户账户枚举

Sendmail 和其他服务器允许邮箱和本地用户账户枚举。在 Kali Linux 中，你可以使用 **smtp-user-enum** 实用程序通过 `EXPN`、`VRFY` 和 `RCPT TO` 命令识别账户。在以下各节中，我将手动演示每种技术。

#### EXPN

`EXPN` 命令扩展给定邮件地址的详细信息，如示例 9-6 所示。通过分析服务器响应，我们发现 `test` 用户账户不存在，`root` 的邮件被转发到 `chris@example.org`，并且存在一个 `sshd` 账户用于权限分离。

**示例 9-6. 使用 EXPN 枚举本地用户**

```bash
root@kali:~# telnet 10.0.10.11 25
Trying 10.0.10.11...
Connected to 10.0.10.11.
Escape character is '^]'.
220 mail2 ESMTP Sendmail 8.13.8/8.12.8; Thu, 13 Nov 2014 03:20:37
HELO world
250 mail2 Hello onyx [192.168.10.1] (may be forged), pleased to meet you
EXPN test
550 5.1.1 test... User unknown
EXPN root
250 2.1.5 <chris@example.org>
EXPN sshd
250 2.1.5 sshd privsep <sshd@mail2>
```

#### VRFY

`VRFY` 命令验证给定的 SMTP 邮件地址是否有效，如示例 9-7 所示。我们可以滥用此功能来枚举本地账户（本例中为 `chris`）。

**示例 9-7. 使用 VRFY 枚举本地用户**


### 绕过内容检查

组织运行内容检查软件以根据给定策略清洗邮件。服务器和客户端软件包以不同方式解析通过电子邮件发送的材料，并且过滤软件和相关组件（即防病毒引擎）中存在缺陷。

多年前，我通过修改消息中的 MIME 标头绕过了 Clearswift MAILsweeper。示例 9-11 显示了 Microsoft Outlook 生成的一封合法电子邮件，从 `john@example.org` 发送到 `mickey@example.org`，附件为 `report.txt`。

**示例 9-11. Microsoft Outlook 生成的带附件消息**

```
From: John Smith <john@example.org>
To: Mickey Mouse <mickey@example.org>
Subject: That report
Date: Thurs, 22 Feb 2001 13:38:19 -0000
MIME-Version: 1.0
X-Mailer: Internet Mail Service (5.5.23)
Content-Type: multipart/mixed ;
boundary="----_=_NextPart_000_02D35B68.BA121FA3"
Status: RO

This message is in MIME format. Since your mail reader doesn't understand this format, some or  
all of this message may not be legible.

- ------_=_NextPart_000_02D35B68.BA121FA3
Content-Type: text/plain; charset="iso-8859-1"

Mickey,

Here's that report you were after.

- ------_=_NextPart_000_02D35B68.BA121FA3
Content-Type: text/plain;
        name="report.txt"
Content-Disposition: attachment;
        filename="report.txt"

< data for the text document here >

- ------_=_NextPart_000_02D35B68.BA121FA3
```

存在可利用的条件是因为内容检查系统和客户端软件解析附件 MIME 标头的方式——MAILsweeper 使用 `name`，而 Microsoft Outlook 使用 `filename`。可以修改标头，向 MAILsweeper 呈现一个良性的文本文件，并通过 Outlook 向用户呈现恶意的 VBScript：

```
- ------_=_NextPart_000_02D35B68.BA121FA3
Content-Type: text/plain;
        name="report.txt"
Content-Disposition: attachment;
        filename="report.vbs"
```

这种策略类似于通过分片和乱序发送数据包来绕过网络 IPS 时采用的策略——我们修改数据，使其通过安全控制，然后在目的地重新组装。在测试期间，考虑这种策略的变体并识别可利用的条件非常重要。

### 邮件安全特性审查

SMTP 消息很容易被伪造，因此组织使用 SPF、DKIM 和 DMARC 功能来防止各方发送未经授权的电子邮件。这些机制以及你可以采取的审查其配置的步骤将在以下各节中概述。

#### SPF

发件人策略框架（SPF）[^12] 提供了一种机制，MTA 可以检查为给定域发送电子邮件的主机是否被授权。组织在特殊格式的 TXT DNS 记录中定义域的授权邮件服务器列表。

[^12]: 参见 RFC 7208。

我们可以通过 `dig` 评估 Google 的 SPF 配置，如示例 9-12 所示。大型组织通常使用 `include` 和 `redirect` 指令，你必须迭代地逐步检查每个记录以了解配置。在此示例中，返回的 IPv6 和 IPv4 范围是授权源。

#### DKIM

[^13]: 参见 RFC 6376。

#### DMARC

基于域的消息认证、报告和一致性（DMARC）[^14] 是一种扩展 SPF 和 DKIM 的邮件认证方法。策略指示邮件服务器如何处理给定域的电子邮件并报告执行的操作。DMARC 与 SPF 和 DKIM 一起使用的方式如图 9-3 所示。

[^14]: 参见 RFC 7489。

**图 9-3. DMARC、SPF 和 DKIM**

DMARC 策略通过 DNS 分发。表 9-11 列出了字段和示例值，示例 9-14 演示了 Google、Yahoo 和 PayPal 使用的策略。

**表 9-11. DMARC 策略字段**

| 名称 | 目的 | 示例 |
|------|------|------|
| `v` | 协议版本 | `v=DMARCv1` |
| `p` | 对源自域名的电子邮件的请求处理策略（即 `none`、`quarantine` 或 `reject`） | `p=quarantine` |
| `sp` | 对子域的请求策略 | `sp=reject` |
| `pct` | 将策略应用于一定百分比的消息（用于控制 DMARC 采用并避免意外的报告泛滥） | `pct=20` |
| `ruf` | 取证报告的报告 URI | `ruf=mailto:authfail@example.org` |
| `rua` | 聚合报告的报告 URI | `rua=mailto:aggrep@example.org` |
| `rf` | 定义取证报告格式 | `rf=afrf` |
| `ri` | 定义聚合报告间隔 | `ri=86400` |
| `adkim` | DKIM 对齐模式（`r`（宽松）是默认值，`s` 强制执行严格模式） | `adkim=s` |
| `aspf` | SPF 对齐模式，使用与 DKIM 对齐模式相同的值 | `aspf=r` |

**示例 9-14. 通过 dig 检索 DMARC 策略**

```bash
root@kali:~# dig _dmarc.yahoo.com txt | grep DMARC
_dmarc.yahoo.com.  1785 IN TXT "v=DMARC1\; p=reject\; sp=none\; pct=100\; 
rua=mailto:dmarc-yahoo-rua@yahoo-inc.com, mailto:dmarc_y_rua@yahoo.com\;"
root@kali:~# dig _dmarc.google.com txt | grep DMARC
_dmarc.google.com. 600 IN TXT "v=DMARC1\; p=quarantine\; rua=mailto:mailauth-reports@google.com"
root@kali:~# dig _dmarc.paypal.com txt | grep DMARC
_dmarc.paypal.com. 300 IN TXT "v=DMARC1\; p=reject\; rua=mailto:d@rua.agari.com\; 
ruf=mailto:dk@bounce.paypal.com,mailto:d@ruf.agari.com"
```

PayPal 和 Yahoo 指示邮件服务器拒绝包含无效 DKIM 签名或非源自其网络的消息。然后通知被发送到每个组织内的相应电子邮件地址。Google 以类似方式配置，尽管它指示邮件服务器隔离消息而不是直接拒绝它们。

### 通过 SMTP 进行钓鱼

通过发送精心制作的电子邮件，可以诱骗用户点击超链接、提供凭据和执行代码（例如 JavaScript 和 Microsoft Office 宏）。根据组织的邮件安全功能和配置，你还可以通过外部 SMTP 接口伪造内部电子邮件。

在以下各节中，我将描述钓鱼的高级方法。Kali Linux 中的 **社会工程工具包（SET）**[^15] 是一个强大的平台，你可以从中发起钓鱼活动。

[^15]: 参见 TrustedSec.com 上的“The Social-Engineer Toolkit (SET)”。

#### 侦察

有效的钓鱼活动通过利用对目标环境的了解来欺骗安全意识强的用户。需要关注的重要细节如下：

- 地址格式和命名约定（例如 Smith, Stan <stan.smith@intel.com>）
- 组织内使用的客户端软件（例如 Microsoft Outlook）
- 消息详细信息或特点，包括某些用户采用的签名格式
- 识别要克隆的候选 Web 接口

你可以通过获取源自组织的材料（例如包含标头和 HTML 内容的电子邮件）来覆盖上述列表中的前三项。邮件列表存档和 Google 是很好的候选，以及通过向市场或销售部门请求信息来胁迫用户。

如果组织使用多因素认证进行远程访问，你可以通过克隆 SSL VPN Web 端点来捕获凭据并立即将其重放到合法服务来破坏它。

#### 着陆页准备

你可以使用 SET 克隆登录页面并收集凭据。为了获得最佳结果，我建议更进一步：注册一个用于活动的域，获取有效的 TLS 证书，并使用 `stunnel` 在受害者和 SET 之间代理 HTTPS 流量。

例如，如果克隆的 Web 端点是 `vpn.victim.com`，请考虑获取 `victim-corp.com`，设置 DNS 使 `sslvpn.victim-corp.com` 指向你的 SET 实例，并购买关联的证书，以便用户看到看起来合法的加密连接。

SET 是一个功能强大的实用程序，具有丰富的功能。你可以使用它来收集凭据（通过传统的钓鱼攻击）以及利用易受攻击的浏览器插件和组件来获得代码执行[^16]。

[^16]: 网上有无数视频和教程展示其功能，包括 Javi Oliu，“Social Engineering Toolkit”，YouTube 视频，发布于 2014 年 5 月 24 日；以及 Jeremy Martin，“Exploitation with Social Engineering Toolkit SET”，YouTube 视频，发布于 2013 年 4 月 21 日。

#### 发送电子邮件

SET 中的鱼叉式钓鱼模块使用 Kali Linux 中的 Sendmail MTA 发送电子邮件。然而，为了获得最佳效果，请考虑从先前获取的材料中手动制作电子邮件消息（使用与合法电子邮件相同的字体和消息格式），通过 Swaks 将此材料管道传输到本地 Sendmail 服务，然后发送到目标 SMTP 服务器。

在继续之前评估目标的安全状况非常重要。查看 SPF、DKIM 和 DMARC 策略（如果有），并评估暴露的 SMTP 接口的行为，看它们是否会接受来自内部域的材料。如果环境是加固的，你需要设置一个用于发送电子邮件的域（例如 `victim-corp.com`）。

图 9-4 展示了一封从公司 IT 部门伪造的有效 HTML 电子邮件。电子邮件的外观和感觉至关重要，语言也很重要，因为你希望用户产生紧迫感以点击恶意链接。

**图 9-4. HTML 钓鱼电子邮件内容**

影响钓鱼活动成功的因素包括：

- SMTP 接口不接受来自公共互联网的、从组织内部域发送的电子邮件
- 使用内容过滤在源自组织外部的电子邮件的主题行中添加材料（例如外部或不可信指示符）
- 存在评估目标 URL 信誉的 Web 代理设备
- 使用特定的浏览器或桌面软件客户端

Eric Smith 在 2013 年 DerbyCon 上的演讲[^17] 记录了许多成功的专业社会工程活动，详细说明了用于以下目的的策略：

[^17]: Eric Smith，“DerbyCon Cheat Codez: Level Up Your SE Game”，YouTube 视频，由 Adrian Crenshaw 于 2013 年 9 月 30 日发布。

- 确保活动中使用的 URL 被标记为在线信誉良好
- 确保用于发起活动的邮件服务器看起来也信誉良好
- 通过跟踪会话标识符来衡量活动的有效性
- 利用特权访问（通过 OWA 或 VPN）获取情报
- 占据并分散事件响应团队的注意力

通过侦察、适当的准备（提供域、证书以及信誉良好的 Web 和邮件服务器地址）和集中执行，你很可能会通过钓鱼取得成功。通过准备多个冗余的攻击平台并分散目标安全团队的注意力，你还应该能够实现持久性。

## POP3

邮件服务器包（例如 Courier、Dovecot 和 Microsoft Exchange）提供 POP3 服务。如果软件包没有维护，攻击者可以利用缺陷破坏系统。然而，在撰写本文时，NVD 中发现的唯一可远程利用的 POP3 漏洞是 IBM Domino 缺陷[^18]。

### 暴力密码猜测

邮件服务器是暴力密码猜测的目标，原因有几个：

- 实现通常不关注账户锁定策略
- POP3 和 IMAP 服务器在断开连接前允许多次登录尝试
- 许多邮件服务器不记录失败的登录尝试

客户端可以使用明文或 MD5 摘要认证与 POP3 服务进行认证。摘要机制（使用 `APOP` 指令）容易受到网络嗅探和通过 Cain & Abel 等工具的攻击，因为该实现存在已知明文缺陷[^19]。

[^19]: Fanbao Liu et al.,“Fast Password Recovery Attack: Application to APOP”，Journal of Intelligent Manufacturing 25, no. 2 (2014): 251–261。

为了缓解这些风险，许多 POP3 服务器通过 SASL 支持其他机制，包括 `DIGEST-MD5` 和 `NTLM`。Hydra 支持多种 SASL 机制[^20]。示例 9-16 显示了 Hydra 用于对暴露的、具有基本 MD5 认证（通过 `APOP`）的 POP3S 服务执行暴力密码猜测。

[^20]: 参见 THC.org 上的“Comparison of Features and Services Coverage”。

**示例 9-16. 使用 Hydra 执行 POP3S 密码猜测**


## IMAP

IMAP 协议与 POP3 类似。因此，暴露的服务容易受到暴力密码猜测和进程操纵攻击（在识别出可利用的软件缺陷后）。

## 邮件服务测试回顾

以下是本章讨论的策略的简明列表：

- **服务识别**：Nmap 可有效识别邮件服务并枚举支持的功能（例如认证机制）。将自动发现与手动验证相结合，以正确识别每个暴露的服务及其功能。

- **NDN 审查**：使用 Swaks 通过每个暴露的 SMTP 网关中继消息，目标是每个已知域内的不存在用户。在识别出以 NDN 消息响应的服务器后，利用该行为揭示内部 IP 地址和主机名详细信息、软件版本和内容过滤策略。

- **用户枚举**：你可以查询可访问的 SMTP 服务（特别是 Sendmail），通过 Kali Linux 中的 `smtp-user-enum` 实用程序获取本地用户账户详细信息。Hydra 和 Nmap 也包含执行枚举的模块。

- **暴力密码猜测**：在准备好有效用户账户列表后，可以使用 Hydra、Nmap 和 Swaks 攻击暴露邮件服务中的认证机制。通过邮件服务进行暴力破解是有效的，因为审计日志通常不被仔细检查，并且账户锁定策略可能不被遵守。

- **钓鱼**：根据你可能获得的访问级别（未经身份验证或经过身份验证）以及目标环境的配置，你可以尝试发起令人信服的钓鱼活动并欺骗用户。Kali Linux 中的 SET 自动化了钓鱼过程。

- **利用邮件软件中的已知缺陷**：邮件系统是模糊的，具有很大的攻击面（考虑 SMTP、IMAP 和 POP3 服务器软件、防病毒和内容检查引擎，以及用户设备上运行的邮件客户端应用程序）。在识别出特定的防病毒引擎、商业内容检查设备、邮件服务器或客户端后，你可以尝试利用已知漏洞。

## 邮件服务对抗措施

在加固邮件系统时，应考虑以下对抗措施：

- **不要将功能丰富的 SMTP 服务器暴露给公共互联网或不可信网络**。Sendmail 和 Microsoft Exchange 具有大量代码库，并引入不必要的暴露。使用专用的内容过滤器（以物理设备或云服务的形式）来解析电子邮件，或部署轻量级 MTA，如 qmail 或 Exim。

- **使用 SPF、DKIM 和 DMARC** 防止服务器传输和接收未经授权的内容。此外，配置暴露的 SMTP 接口，不接受来自不可信网络（例如公共互联网）的、看似来自组织内部域的电子邮件。

- **配置外部内容过滤机制** 在源自环境外部的电子邮件的主题字段中添加材料（例如通知用户消息来自不可信源的字符串）。

- **使用防病毒和内容过滤软件会增加攻击面**，并且此类软件包中存在可利用的缺陷。确保过滤和防病毒软件已更新到最新补丁。

- **禁用邮件服务中对弱认证机制的支持**（例如 `LOGIN`、`PLAIN` 和 `CRAM-MD5`）。如果 SMTP 不使用认证，请完全禁用该功能。

- **尽可能强制使用 TLS**（在 SMTP 服务器之间，以及支持通过 POP3 和 IMAP 收集电子邮件的服务之间）。考虑使用客户端证书提供额外的认证层，并防止未经授权的方与服务交互。

- **通过强制在邮件服务器上实施强密码策略来最小化暴力密码猜测攻击的影响**（涵盖所有向量）。

- **通过记录针对邮件服务的失败认证尝试来提高可见性**。某些平台（例如 Microsoft Windows）中也可以定义账户锁定策略，这将缓解暴力密码猜测的风险。
