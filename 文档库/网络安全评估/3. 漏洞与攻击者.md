软件工程师构建日益精巧的系统，却连一半的搞砸方式都未曾考虑。由于云计算和不断增加的抽象层，不安全的产物将持续存在，而计算机安全行业在未来多年仍将是一块有利可图的领域。

## 基本黑客概念

黑客是操纵系统以执行有用行动的艺术。

一个基本的例子是普通的搜索引擎：它将用户输入与数据库交叉引用并返回结果。处理发生在服务端，通过理解这些系统的工程设计方式，攻击者可以试图操纵应用程序并获取敏感内容。

几十年前，美国五角大楼、空军和海军的网站就存在这一问题。一个名为 `multigate` 搜索引擎接受两个特定参数：`Surf QueryString` 和 `f`。通过精心构造的 URL，服务器 `/etc/passwd` 文件的内容被泄露，如图 3-1 所示。

**图 3-1. 利用 multigate 搜索引擎**

这些站点在网络层受到防火墙和安全设备的保护。然而，由于存储的信息量巨大，一个搜索引擎被实现，从而在应用层引入了漏洞。

一个有用的思考方式是将利用过程视为攻击者对系统进行“编程”以执行有用行动。所采用的具体编程策略取决于系统组件、功能以及攻击者的目标——例如数据暴露、权限提升、任意代码执行或拒绝服务。

## 软件为何存在漏洞

30 多年来，攻击者一直在利用软件中的漏洞，而供应商也一直致力于提高其产品的安全性。由于质量保证尤其不足，这些日益精巧的系统中很可能始终存在缺陷。

软件供应商发现自己处于这样一种境地：随着产品变得愈发复杂，使其安全所需的成本也更高。静态代码审查和动态软件测试是一项重大的工作，它会影响利润底线并延迟产品上市时间。

## 考量攻击面

在对计算机系统的每一次成功攻击中，攻击者都利用了可用的攻击面来实现其目标。这个攻击面通常涵盖服务器应用程序、客户端端点、用户、通信通道和基础设施。每个暴露的组件都是公平的游戏对象。

环境可能被入侵的实际方式包括以下内容：

- 入侵托管服务提供商或云管理控制台[^1]
- 基础设施中的漏洞（例如，虚拟机监视器侧信道攻击[^2]）
- 操作系统漏洞（例如，网络驱动程序或内核缺陷）
- 服务器软件缺陷（例如，Nginx 或 OpenSSL 错误）
- Web 应用程序框架缺陷（例如，会话管理缺陷）
- Web 应用程序错误（例如，命令注入或业务逻辑缺陷）
- 攻击用户的 HTTPS 或 SSH 会话
- 入侵合法凭据（例如，SSH 密钥或身份验证令牌）
- 桌面软件攻击（例如，PuTTY SSH 客户端或浏览器漏洞利用）
- 通过社会工程学或点击劫持攻击用户

某些攻击可能远程进行，而另一些则需要靠近才能操纵易受攻击的系统组件。例如，在多租户云环境中，获得对内部地址空间的访问通常会暴露非公开访问的管理接口。

## 软件安全错误分类法

图 3-2 展示了构成巨大攻击面的各个软件组件。在每个软件内部，可能存在可被利用以获利的错误。Gary McGraw、Katrina Tsipenyuk 和 Brian Chess 发布了一个分类法[^3]，用于对软件缺陷进行分类，列出了七个“王国”：

[^3]: Katrina Tsipenyuk, Brian Chess, and Gary McGraw，“Seven Pernicious Kingdoms: A Taxonomy of Software Security Errors”，IEEE Security & Privacy（2005 年 11/12 月）。[链接](https://www.researchgate.net/publication/3410197_Seven_Pernicious_Kingdoms_A_Taxonomy_of_Software_Security_Errors)

- **输入验证与表示**：未能正确验证输入或编码数据，导致恶意内容被处理。此王国下的“门”示例包括缓冲区溢出、XSS、命令注入和 XXE 处理。

- **API 滥用**：攻击者利用系统库和可访问 API 暴露的函数来执行任意代码、读取敏感材料或绕过安全控制。

- **安全特性**：软件中安全特性的设计和底层实现应当健全。未能执行安全的随机数生成、实施最小权限或安全存储敏感数据，往往会导致被入侵。

- **时间与状态**：在应用程序中，时间和状态漏洞通过竞争条件、不安全临时文件的存在或会话固定（例如在登录或状态更改时不生成新的会话令牌）被利用。

- **错误处理**：在发现错误条件后，攻击者攻击错误处理代码以影响逻辑程序流或泄露敏感材料。

- **代码质量**：糟糕的代码质量导致应用程序行为不可预测。在底层语言（如 C/C++）中，此类行为可能是灾难性的，并导致任意指令的执行。

- **封装**：应用程序在提供数据和代码执行路径访问方式上的模糊性可能导致封装缺陷。示例包括用户间的数据泄露，以及攻击者可利用的遗留调试代码。

第八个王国是**环境**——用于对软件源代码之外发现的漏洞进行分类（例如，解释器或编译器软件、Web 应用程序框架或底层基础设施中的缺陷）。

使用该分类法来定义导致应用程序意外行为的缺陷来源。注意：这些王国并不定义攻击类别（如侧信道攻击）。

## 威胁建模

攻击者以系统组件和功能中的弱点为目标并进行利用。分类法允许我们描述和分类每个软件包内的底层缺陷，但并未解决环境中的更大问题（例如传输中数据的完整性，或加密密钥的处理方式）。为了理解系统面临的实际风险，你可以通过考虑以下内容来建模威胁：

- 各个系统组件
- 攻击者的目标
- 暴露的系统组件（可用的攻击面）
- 每个攻击向量的经济成本和可行性

### 系统组件

可利用的漏洞可能存在于基础设施（例如，虚拟机监视器、软件交换机、存储节点和负载均衡器）、操作系统、服务器软件、客户端应用程序以及最终用户自身中。图 3-3 显示了典型环境中硬件、软件和“湿件”组件之间的关系。

### 攻击者目标

攻击者追求的目标可能包括以下内容：

- **数据暴露或修改**：在计算机系统中，数据可以有多种形式，从服务端数据库到传输中的数据以及客户端软件处理的材料。攻击者通常试图暴露敏感数据，或修改数据以获利（例如，获取可用于提升权限的身份验证令牌，或修改内存中的标志以禁用安全特性）。

- **权限提升**：软件安全特性强制实施边界和权限级别。攻击者可以通过提供合法凭据（如被盗的身份验证令牌）或利用控制机制本身的缺陷来提升其权限。

- **任意代码执行**：在环境中执行任意指令使攻击者能够直接访问资源，并通常可以修改底层系统配置。

- **拒绝服务**：使系统组件变得无响应（或延迟超过给定阈值）的条件会给受害者带来成本，并使攻击者受益。

使用存在内存安全问题的语言（如 C/C++）编写的软件容易通过缓冲区溢出、过度读取和指针滥用而被操纵，这可能导致上述许多目标的实现。使用内存安全语言（包括 Java 和 Microsoft .NET）编写应用程序可以使整类错误变得无关紧要。

### 系统访问和执行上下文

攻击者可以获得的三个广泛级别的系统访问权限如下：

- **远程**：大多数威胁来自给定系统外部，通过广域网（如互联网）授予访问权限。因此，远程攻击者无法拦截组件之间的网络流量（例如，客户端、服务器以及支持身份验证、内容交付和其他功能的系统服务），也无法通过侧信道观察本地系统操作来发现秘密。

- **近距**：逻辑邻近或网络访问（例如，与合法用户坐在同一家咖啡店，或获得对共享云基础设施的访问）通常提供对系统组件之间流动的数据或共享资源（如服务器内存或持久存储）的访问。如果材料未通过传输层安全得到充分保护，或者敏感数据被泄露，则可能发生入侵。

- **直接物理访问**：对系统组件的直接、无监督访问通常会导致完全入侵。近年来，已知政府机构曾截获基础设施硬件[^4]（例如路由器、交换机和防火墙）的运输，以在目标环境中植入复杂的后门。数据中心也曾遭受物理入侵[^5]。

[^4]: Sean Gallagher，“Photos of an NSA ‘Upgrade’ Factory Show Cisco Router Getting Implant”，Ars Technica，2014 年 5 月 14 日。[链接](https://arstechnica.com/tech-policy/2014/05/photos-of-an-nsa-upgrade-factory-show-cisco-router-getting-implant/)
[^5]: Jesse Robbins，“Failure Happens: Taser-Wielding Thieves Steal Servers, Attack Staff, and Cause Outages at Chicago Colocation Facility”，O’Reilly Radar，2007 年 11 月 3 日。[链接](https://www.oreilly.com/radar/failure-happens-taser-wielding-thieves-steal-servers-attack-staff-and-cause-outages-at-chicago-colocation-facility/)

服务器应用程序正变得越来越解耦，消息队列、内容交付、存储、身份验证和其他功能都在 IP 网络上运行。传输中的数据成为攻击目标，无论是通过执行网络嗅探的本地攻击者，还是通过广泛的监控计划[^6]。系统组件之间的传输安全变得越来越重要。

[^6]: Steven J. Vaughan-Nichols，“Google, the NSA, and the Need for Locking Down Datacenter Traffic”，ZDNet，2013 年 10 月 30 日。[链接](https://www.zdnet.com/article/google-the-nsa-and-the-need-for-locking-down-datacenter-traffic/)

一个常见的目标是代码执行，其中暴露的逻辑被用来代表攻击者执行特定指令。然而，执行通常发生在一个狭窄的上下文中。考虑三种场景：

- **XSS**：攻击者向 Web 应用程序提供恶意 JavaScript，该 JavaScript 又被呈现给不知情的用户，并在其浏览器中执行以执行有用操作（例如泄露 cookie 材料或生成任意 HTTP 请求）。
- **HTML 注入**：浏览器通常通过 MITM 攻击被利用，其中呈现恶意 HTML，导致内存损坏和在沙箱浏览器进程中有限的代码执行。
- **恶意 PHP 文件创建**：许多 PHP 解释器缺陷通过使用恶意内容被利用——通过服务端构造脚本（通过上传或文件创建错误）并解析以利用已知错误来实现任意代码执行。

攻击者执行代码的上下文各不相同。例如，Web 应用程序缺陷通常导致通过解释型语言（如 JavaScript、Python、Ruby 或 PHP）执行，并且很少提供对底层操作系统的特权访问。沙箱逃逸和本地权限提升策略必须被采用以实现持久性。

## 攻击 C/C++ 应用程序

操作系统、服务器软件包和桌面客户端应用程序（包括浏览器）通常用 C/C++ 编写。通过理解运行时内存布局、应用程序如何处理数据（以及如何不安全地处理），以及可以被读取和覆盖的值，你可以寻求理解并缓解底层软件缺陷。

### 运行时内存布局

不同操作系统和硬件平台上的内存布局各不相同。图 3-6 展示了在 Intel 和 AMD x86 硬件上的 Windows、Linux 及类似操作系统中常见的高级布局。从外部来源（用户和其他应用程序）提供的输入使用栈和堆进行处理和存储。如果软件未能安全地处理输入，攻击者可以覆盖敏感值并改变程序流程。

**图 3-6. 运行时内存布局**

#### 文本段
该段包含程序的编译可执行代码及其依赖项（例如共享库）。通常禁用写权限，原因有二：
- 代码不包含变量，因此没有理由覆盖自身
- 只读代码段可以在并发程序副本之间共享

在过去，代码会修改自身以提高运行时速度。大多数处理器针对只读代码进行了优化，因此修改会带来性能损失。你可以安全地假设，如果程序试图修改文本段中的材料，那是不经意的。

像 Java 和 Microsoft .NET 这样的即时（JIT）编译器会准备可执行内存页并用指令填充它们；因此，应用程序在文本段之外写入和修改代码是很常见的。

#### 数据段和 BSS 段
数据段和 BSS（块起始于符号）段包含程序的静态和全局变量。这些内存段通常启用了读写访问，在某些情况下，这些段中的指令可以被执行。

#### 堆
堆通常是程序分配的最大内存段。应用程序使用堆来存储持久性数据，这些数据在函数返回后（其局部变量不再可访问）仍然存在。分配器和释放器函数管理堆上的数据。在 C 语言中，`malloc` 通常用于分配一块内存，`free` 用于释放，尽管也可能使用其他函数来优化分配。

#### 栈
栈是用于存储局部函数变量（例如用于存储用户输入的字符缓冲区）以及用于维护程序流的数据的内存区域。当程序进入一个新函数时，会建立一个栈帧。尽管帧的确切布局和使用的处理器寄存器因实现（称为调用约定）而异，但 Microsoft 和 GCC 在 Intel IA-32 硬件上使用的常见布局如下：

- 函数的参数
- 存储的程序执行变量（保存的指令指针和帧指针）
- 局部函数变量的操作空间

当调整栈的大小以创建此空间时，处理器栈指针 (`esp`) 寄存器被修改为指向栈的末尾。栈帧指针 (`ebp`) 指向帧的开始。当进入一个函数时，父函数的栈指针和帧指针的位置被保存在栈上，并在退出时恢复。

栈是一个后进先出（LIFO）数据结构：处理器压入栈的数据存在于底部，并且在压在其上的所有数据被弹出之前不能弹出。

### 处理器寄存器和内存

易失性内存包含文本段中的代码、数据段和 BSS 段中的全局变量和静态变量、堆上的数据，以及栈上的局部函数变量和参数。

在程序执行期间，处理器通过使用指向内存中结构的寄存器来读取和解释数据。寄存器名称、数量和大小因处理器架构而异。为简单起见，这里我使用 Intel IA-32 寄存器名称（特别是 `eip`、`ebp` 和 `esp`）。图 3-7 显示了在内存中执行的程序的高级表示，包括这些处理器寄存器和各种内存段。

从安全角度来看，三个重要的寄存器是指令指针（`eip`）以及前面提到的 `ebp` 和 `esp`。它们在代码执行期间用于从内存读取数据，具体如下：

- `eip` 指向 CPU 要执行的下一条指令
- `ebp` 应始终指向当前函数栈帧的开始
- `esp` 应始终指向栈的底部

在操作期间，大多数底层处理器操作（例如 `push`、`pop`、`ret` 和 `xchg`）会自动修改 CPU 寄存器，以便栈布局有效，并且处理器获取并执行下一条指令。

## 写入内存

溢出已分配的缓冲区通常会导致程序崩溃，因为处理器使用的关键值（例如栈上保存的帧指针和指令指针，或堆上的控制结构）被破坏。攻击者可以利用此行为来改变逻辑程序流。

### 覆盖内存结构以获利

根据输入最终进入的内存区域（例如栈、堆或数据段）以及溢出的方向，攻击者可以影响逻辑程序流。以下是攻击者可以针对获利的结构列表：

- **保存的指令指针**：当进入一个新函数时，一个保存的 `eip` 值被存储在栈上，该值指向一个包含处理器在函数返回时（即返回到父函数）要使用的代码的地址。攻击者可能能够覆盖保存的 `eip` 以指向一个有用的位置，从而在函数退出时导致代码执行。

- **保存的帧指针**：当进入一个新函数时，父 `ebp` 的位置也存储在栈上。通过利用差一错误或类似条件，可以移动父栈帧的表观位置，用恶意内容填充它（取决于父函数及其期望的变量），并在函数返回时获利。

- **函数指针**：堆、栈和 BSS 内存段通常包含函数指针。例如，Microsoft Windows 结构化异常处理（SEH）指针条目在栈上提供异常处理。通过覆盖 SEH 指针，诱导程序崩溃将导致任意代码执行[^7]。

[^7]: David Litchfield，“Defeating the Stack-Based Buffer Overflow Prevention Mechanism of Microsoft Windows 2003 Server”，NGSSoftware Ltd.，2003 年 9 月 8 日。[链接](https://www.ngssoftware.com/papers/ms03-011.pdf)

- **堆控制结构**：根据实现的不同，可能存在可被攻击的堆控制结构。通过操纵控制结构，可以欺骗堆管理过程读取或写入某些内存位置。

- **堆指针**：C++ 对象在堆虚函数表结构中存储函数指针。将虚函数表指向不同的内存位置会导致在函数指针查找时使用新内容，从而控制执行流。其他链接的数据结构和指针可能存在于堆上，根据应用程序的不同，可用于执行受控的读或写操作。

- **全局变量**：程序通常将敏感材料存储在数据段的全局变量中，例如用户 ID 和身份验证标志值。通过向易受攻击的程序（例如 Oracle Solaris 下的 `/bin/login`[^8]）传递恶意环境变量，可以绕过身份验证并获得系统访问权限。

[^8]: 参见 CVE-2001-0797 和 CVE-2007-0882。[CVE-2001-0797](https://nvd.nist.gov/vuln/detail/CVE-2001-0797) | [CVE-2007-0882](https://nvd.nist.gov/vuln/detail/CVE-2007-0882)

- **全局偏移表（GOT）条目**：Unix ELF 二进制文件支持包含共享函数（如 `printf`、`strcpy`、`fork` 等[^9]）的库的动态链接。GOT 存在于数据段中，用于指定库函数的地址。由于该段通常是可写的，攻击者可以寻求覆盖 GOT 中共享函数的地址并接管逻辑程序流。

[^9]: 如 GNU C 库中所见。

- **过程链接表（PLT）条目**：ELF 二进制文件使用过程链接表，其中包含文本段内共享函数的地址。虽然 PLT 条目无法被覆盖，但它们可以用作底层系统调用（如 `write(2)`）的入口点，可用于读取进程内存并绕过安全控制。

- **构造函数（.ctors）和析构函数（.dtors）**：ELF 二进制文件的构造函数定义在数据段内；它们是在程序进入 `main()` 之前执行的函数。析构函数以类似方式定义，并在进程退出时执行。实际上，析构函数被用于在父函数退出时执行代码。

- **应用程序数据**：应用程序用于跟踪状态（例如身份验证标志）或在内存中定位材料的数据可以被覆盖，并导致应用程序行为异常。

## 从内存读取

### 读取内存结构以获利

可能暴露的存储在易失性内存中的秘密包括以下内容：

- **私钥**：应用程序通过 TLS、SSH 和 IPsec 提供传输安全。通过泄露服务器的私钥，你可以冒充它而不被发现（通过 MITM），并且如果未使用完美前向保密，可以破坏会话内容。

- **其他加密材料**：密码学原语使用的重要值包括密钥、种子和初始化向量。如果攻击者能够泄露这些，他们可以攻击密码系统并产生意想不到的后果。

- **安全特性使用的敏感值**：包括 ASLR 在内的编译器和操作系统安全机制依赖于内存中的秘密。如果获得指针地址等值，这些机制可以被绕过。

- **凭据**：应用程序经常使用微服务、第三方 API 和外部数据源。如果这些凭据已知，攻击者可以使用它们获取敏感数据。

- **会话令牌**：Web 和移动应用程序使用会话令牌来跟踪已认证的用户。通过获取这些令牌，你可以冒充合法用户，包括管理员。

随着系统变得越来越分布式，保护秘密（例如 API 密钥和数据库连接字符串）成为高度优先事项。近年来导致 PII 数据暴露的严重入侵事件源于以这种方式泄露的凭据。

## 编译器和操作系统安全特性

为了提供对导致敏感数据被覆盖或从易受攻击应用程序读取的攻击的弹性，供应商在其操作系统和编译器中实现了安全特性。一篇题为[“SoK: Eternal War in Memory”的论文](https://people.eecs.berkeley.edu/~dawnsong/papers/Oakland13-SoK-CR.pdf)详细介绍了针对具有内存安全问题的语言（如 C/C++）的缓解策略和攻击策略。以下是常见安全特性的列表、其用途以及使用它们的平台的详细信息：

- **数据执行保护（DEP）**：在大多数操作系统（包括 Microsoft Windows、Apple OS X 和 Linux）中，DEP 用于将内存的可写区域（包括栈和堆）标记为不可执行。DEP 与处理器硬件一起使用，以防止从内存的可写区域执行指令。然而，DEP 支持和覆盖范围存在例外：许多 32 位二进制文件与 DEP 不兼容，并且某些库（例如 Microsoft Thunk Emulation[^14]）合法地从堆中执行指令。

[^14]: Alexander Sotirov and Mark Dowd，“Bypassing Browser Memory Protections”，在 BlackHat USA 上展示，内华达州拉斯维加斯，2008 年 8 月 2-7 日。[链接](https://www.blackhat.com/presentations/bh-usa-08/Sotirov_Dowd/bh08-sotirov-dowd.pdf)

- **地址空间布局随机化（ASLR）**：包括 Linux、Oracle Solaris、Microsoft Windows 和 Apple OS X 在内的平台也支持 ASLR，用于随机化加载的库和二进制代码的位置。内存的随机化使攻击者难以接管逻辑程序流；但是，存在实现缺陷，并且某些库在没有 ASLR 支持的情况下编译（例如 Windows 7 下的 Microsoft 的 `mscorie.dll`）。

- **代码签名**：Apple 的 iOS 和其他平台使用代码签名[^15]，其中内存的每个可执行页都被签名，并且仅在签名正确时才执行指令。

[^15]: 更多信息，请参见 Apple 开发者支持页面上的“Code Signing”。[链接](https://developer.apple.com/support/code-signing/)

- **SEHOP**：为了防止 SEH 指针在 Windows 应用程序中被覆盖和滥用，Microsoft 实现了 SEHOP[^16]，它在允许调用线程的异常处理程序列表之前检查其有效性。

[^16]: 参见“Preventing the Exploitation of Structured Exception Handler (SEH) Overwrites with SEHOP”，Microsoft TechNet 博客，2009 年 2 月 2 日。[链接](https://docs.microsoft.com/en-us/previous-versions/windows/desktop/xperf/preventing-the-exploitation-of-structured-exception-handler-overwrites-with-sehop)

- **指针编码**：在 Microsoft Visual Studio 和 GCC 中，函数指针可以被编码（使用 XOR 或其他方式）。如果编码的指针在未首先应用正确的 XOR 操作的情况下被覆盖，它们将导致程序崩溃。

- **栈金丝雀**：包括 Microsoft Visual Studio、GCC 和 LLVM Clang 在内的编译器在重要值（例如栈上保存的 `eip` 和函数指针）之前将金丝雀放置在栈上。在函数返回之前检查金丝雀的值；如果它已被修改，程序将崩溃。

- **堆保护**：Microsoft Windows 和 Linux 系统在堆管理代码中包含健全性检查。这些改进可防止双重释放和导致控制结构被覆盖的堆溢出。

- **重定位只读（RELRO）**：可以通过指示 GCC 链接器在运行时解析所有动态链接的函数并将条目标记为只读来保护数据段中的 GOT、析构函数和构造函数条目。

- **格式化字符串保护**：大多数编译器提供对格式化字符串攻击的保护，在这种攻击中，格式标记（如 `%s` 和 `%x`）和其他内容被传递给 `printf`、`scanf` 和 `syslog` 等函数，导致攻击者写入或读取任意内存位置。

## 绕过常见安全特性

你可以通过覆盖内存中的某些值来修改应用程序的执行路径。根据所使用的安全机制（包括 DEP、ASLR、栈金丝雀和堆保护），你可能需要采用特定策略来实现任意代码执行，如下所述。

### 绕过 DEP

DEP 阻止从内存的可写区域执行任意指令。因此，你必须从可执行的文本段中识别和借用有用的指令来实现你的目标。挑战在于识别可用于形成返回导向编程（ROP）链的有用指令序列；以下小节将详细介绍这些内容。

#### CPU 操作码序列
处理器操作码因架构而异（例如 Intel 和 AMD x86-64、ARMv7、SPARC V8 等）。表 3-2 列出了有用的 Intel IA-32 操作码、相应的指令助记符和说明。通用寄存器（`eax`、`ebx`、`ecx` 和 `edx`）用于存储 32 位（4 字节字）值，这些值在不同操作期间被操作和使用。

**表 3-2. 有用的 IA-32 指令**

| 操作码 | 汇编 | 说明 |
|--------|------|------|
| `\x58` | `pop eax` | 移除栈顶的字并写入 `eax` |
| `\x59` | `pop ecx` | 移除栈顶的字并写入 `ecx` |
| `\x5c` | `pop esp` | 移除栈顶的字并写入 `esp` |
| `\x83\xec\x10` | `sub esp, 10h` | 从 `esp` 存储的值中减去 10（十六进制） |
| `\x89\x01` | `mov (ecx), eax` | 将 `eax` 写入 `ecx` 指向的内存位置 |
| `\x8b\x01` | `mov eax, (ecx)` | 将 `ecx` 指向的内存位置写入 `eax` |
| `\x8b\xc1` | `mov eax, ecx` | 将 `ecx` 的值复制到 `eax` |
| `\x8b\xec` | `mov ebp, esp` | 将 `esp` 的值复制到 `ebp` |
| `\x94` | `xchg eax, esp` | 交换 `eax` 和 `esp` 的值（栈枢转） |
| `\xc3` | `ret` | 返回并将 `eip` 设置为栈顶的当前字 |
| `\xff\xe0` | `jmp eax` | 跳转（设置 `eip`）到 `eax` 的值 |

许多这些操作更新 `esp`（栈指针）寄存器，使其指向栈的底部。例如，当一个字写入栈时，`push` 将指针减 4，而当一个字被移除时，`pop` 将其加 4。

`ret` 指令在 ROP 中很重要——它用于将控制权转移到下一个指令序列，由位于栈上的返回地址定义。因此，每个序列必须以 `\xc3` 值或类似的转移执行的指令结束。

程序二进制文件和加载的库通常包含数百万个有效的 CPU 指令。通过搜索数据中的特定序列（例如，有用的指令后跟 `\xc3`），你可以用借来的代码构建简单的程序。两个扫描库和二进制文件以查找指令序列的工具是 ROPgadget[^17] 和 ROPEME[^18]。

[^17]: Jonathan Salwan，“ROPgadget – Gadgets Finder and Auto-Roper”，Shell-storm.org，2011 年 3 月 12 日。[链接](http://shell-storm.org/project/ROPgadget/)
[^18]: longld，“ROPEME – ROP Exploit Made Easy”，VNSecurity.net，2010 年 8 月 13 日。[链接](https://github.com/longld/ropeme)

#### 将数据写入内存中的任意位置
在扫描文本段以查找指令时，你可能会找到以下两个序列（第一个值是内存中找到每个序列的位置，后跟十六进制操作码和相应的指令助记符）：

```
0x08056c56: "\x59\x58\xc3"  <==> pop ecx ; pop eax ; ret
0x080488b2: "\x89\x01\xc3"  <==> mov (ecx), eax ; ret
```

两个序列都只有三个字节长。在将它们链接在一起执行之前，你首先需要填充栈，如图 3-10 所示。

**图 3-10. 准备好的栈帧和 CPU 寄存器值**

在执行来自 `0x08056c56` 的第一个序列时，会发生以下情况：

1. 栈底的两个字被移除并存储在 `ecx` 和 `eax` 寄存器中。每个 `pop` 指令也会将 `esp` 增加 4，使其指向栈的底部。
2. 第一个序列末尾的返回指令（`ret`）读取栈上的下一个字，该字指向 `0x080488b2` 处的第二个链。
3. 第二个序列将 `eax` 的值写入 `ecx` 指向的内存地址。

#### ROP 小工具
有用的序列被链接起来形成小工具。常见用途包括：

- 从内存中读取和向内存中写入任意值
- 对存储在寄存器中的值执行操作（例如 `add`、`sub`、`and`、`xor`）
- 调用共享库中的函数

在攻击具有 DEP 的平台时，ROP 小工具通常用作初始载荷，以准备一个可写且可执行的内存区域。这是通过为内存管理函数（例如 Linux 下的 `memset` 和 `mmap64`，以及 Microsoft Windows 下的 `VirtualAlloc` 和 `VirtualProtect`）构建虚假栈帧，然后调用它们来实现的。通过使用借用的指令建立可执行区域，你最终注入并运行任意指令（称为 shellcode）。

Dino Dai Zovi 的演讲[^19]详细介绍了在利用 Microsoft Internet Explorer 8 Aurora 漏洞[^20]期间准备 ROP 小工具的过程，使用栈枢转来建立攻击者控制的栈帧。

[^19]: Dino A. Dai Zovi，“Practical Return-Oriented Programming”，在 RSA Conference 2010 上展示，加利福尼亚州旧金山，2010 年 3 月 1-5 日。[链接](https://www.rsaconference.com/events/us10/sessions/468/practical-return-oriented-programming)
[^20]: 参见 CVE-2010-0249。[CVE 详情](https://nvd.nist.gov/vuln/detail/CVE-2010-0249)

### 绕过 ASLR

ASLR 打乱了内存中页面的位置。结果是，在利用缺陷（例如溢出条件）时，你既不知道有用指令的位置，也不知道数据的位置。示例 3-1 显示了 Ubuntu Linux 在程序执行时随机化每个已加载库的基地址。

**示例 3-1. 使用 `ldd` 打印程序的共享库位置**

```bash
$ ldd ./test
        linux-gate.so.1 =>  (0xb78d3000)
        libc.so.6 => /lib/libc.so.6 (0xb7764000)
        /lib/ld-linux.so.2 (0xb78d4000)
$ ldd ./test
        linux-gate.so.1 =>  (0xb78ab000)
        libc.so.6 => /lib/libc.so.6 (0xb773c000)
        /lib/ld-linux.so.2 (0xb78ac000)
$ ldd ./test
        linux-gate.so.1 =>  (0xb7781000)
        libc.so.6 => /lib/libc.so.6 (0xb7612000)
        /lib/ld-linux.so.2 (0xb7782000)
```

尽管这些库被加载到随机位置，但有用的函数和指令序列存在于已知偏移量处（因为页面位置被随机化，而不是内容）。问题就变成了识别每个库被加载到内存中的基地址。

在以下情况下，ASLR 未启用：

- 程序二进制文件未编译为位置无关可执行文件（PIE）
- 程序使用的共享库未编译为支持 ASLR

在这些情况下，你只需引用选择退出 ASLR 的二进制文件中的绝对位置。Microsoft Windows 中的某些 DLL 在没有 ASLR 的情况下编译——如果一个易受攻击的程序加载它们，你可以借用指令并构建 ROP 小工具。

某些平台将数据加载到固定地址，包括函数指针。例如，在 Pwn2Own 2013 期间，VUPEN 使用 `KiFastSystemCall` 和 `LdrHotPathRoutine` 函数指针绕过了 Windows 7 上的 ASLR[^21]。这两个指针在未修补系统上存在于固定地址，你可以使用它们来执行任意指令。

[^21]: 参见 CVE-2013-2556。[CVE 详情](https://nvd.nist.gov/vuln/detail/CVE-2013-2556)

如果程序二进制文件和加载的库使用动态基地址，你可以通过其他方式计算和获取它们，包括：

- 进行暴力攻击以定位有效数据和指令
- 通过信息泄露（例如堆过度读取）揭示内存内容

暴力攻击在某些条件下是可实现的；取决于系统访问级别，以及目标应用程序和底层操作系统的运行方式。32 位进程通常比 64 位进程更容易攻击，因为需要暴力尝试的迭代次数更少。

### 绕过栈金丝雀

根据其实现，攻击者可以通过以下方法克服栈金丝雀机制：

- 利用信息泄露错误来揭示金丝雀
- 通过迭代溢出尝试推断金丝雀（如果金丝雀是静态的）
- 通过实现中的弱点计算金丝雀
- 在金丝雀被检查之前覆盖函数指针并转移逻辑程序流（在利用 Microsoft SEH 指针时可能）
- 覆盖金丝雀所比较的值

根据第二点推断金丝雀是一个有趣的话题——Andrea Bittau 和斯坦福大学的其他人于 2014 年发表了一篇题为“Hacking Blind”的论文[^22]，其中他们通过 CVE-2013-2028 成功远程利用了 Nginx。目标系统运行的是带有栈金丝雀、DEP 和 ASLR 的 64 位 Linux。

[^22]: Andrea Bittau 等人，“Hacking Blind”，2014 年 IEEE 安全与隐私研讨会论文集，加利福尼亚州伯克利，2014 年 5 月 18-21 日。[链接](https://ieeexplore.ieee.org/document/6956573)

Bittau 等人的攻击通过以下方式工作：识别他们的材料覆盖金丝雀的点（导致崩溃），然后逐字节迭代覆盖它，直到服务不再崩溃，这意味着金丝雀有效。后续溢出尝试中的栈布局如图 3-11 所示。

**图 3-11. 通过覆盖推断栈金丝雀**

写入金丝雀之后会修改保存的帧指针和指令指针。通过以相同方式逐字节推断这些值，你可以开始绘制内存布局（识别文本段和父栈帧的位置）并绕过 ASLR。

Nginx、Apache HTTP Server、Samba 和其他服务器包通过未能每次通过 `fork` 执行工作进程时生成新的金丝雀来破坏安全性，因此栈金丝雀在不同会话之间是一致的。

## 逻辑缺陷和其他错误

我已经演示了攻击者如何通过内存操纵来滥用操作系统、服务器软件和桌面应用程序。许多 Web 和移动应用程序是使用内存安全语言（包括 Java 和 Microsoft .NET）开发的，因此攻击者必须利用逻辑缺陷和其他漏洞来实现某些目标。常见缺陷导致以下类型的漏洞被利用：

- 在帐户登录或密码重置逻辑中推断用户名
- 围绕令牌生成和使用的会话管理问题（例如会话固定，用户在身份验证后不重新生成会话令牌）
- 命令注入（例如 SQL、LDAP 或操作系统注入）
- 封装错误，其中请求被无条件接受并处理材料（例如直接对象引用、XXE 缺陷或 XSS 攻击中使用的恶意 JavaScript）
- 信息泄露缺陷，通过精心构造的请求从文件系统或后端数据存储（例如数据库或键值存储）中泄露材料

在本书中，你将发现漏洞范围从微妙的底层内存管理缺陷到易于利用的逻辑缺陷。理解潜在问题的广度至关重要，以便通过深度防御实施有效的安全控制。

## 密码学弱点

随着系统组件变得日益解耦和边界崩溃，对密码学的依赖（通过生成随机数、HMAC 值和提供机密性来强制实施安全边界）正在增加。

以下是计算机系统中常见的一些加密功能：

- 伪随机数生成器（PRNG）
- 提供传输层安全的协议（如 TLS 和 IPsec）
- 静态数据的加密
- 数据签名以提供完整性检查（例如 HMAC 计算）

在这些任何一项中，都可能存在可利用的缺陷，原因是不当的实现或底层协议中的缺陷。一个组件中的失败也可能允许攻击者利用另一个组件（例如，完整性检查不足允许内容被发送到预言机，进而揭示加密密钥）。

如果 PRNG 生成可预测的值，攻击者可以利用此行为。2013 年 8 月，Android PRNG 被发现不安全[^23]，导致移动比特币钱包变得可被攻击者访问[^24]。

[^23]: 参见 CVE-2013-7373。[CVE 详情](https://nvd.nist.gov/vuln/detail/CVE-2013-7373)
[^24]: Bitcoin，“Android Security Vulnerability”，Bitcoin.org，2013 年 8 月 11 日。[链接](https://bitcoin.org/en/alert/2013-08-11-android)

加密组件中许多弱点的利用通常需要特定的访问权限，例如网络访问以拦截流量，或本地操作系统访问以获取 PRNG 使用的值。

密钥泄露可能导致灾难性失败。2014 年，一名攻击者获得了 Primedice 游戏网站使用的服务器种子值，导致约 100 万美元的比特币损失[^25]。

[^25]: Stunna，“Breaking the House”，Medium.com，2015 年 6 月 28 日。[链接](https://medium.com/@stunna/breaking-the-house-9d46ebc2da0a)

针对密码系统的常见攻击类别包括以下内容：

- **碰撞**：消息摘要用于完整性检查，如果发现碰撞（两个不同的消息生成相同的摘要/签名），则可以被绕过。MD5 碰撞可以轻松生成[^26]。碰撞可用于生成恶意材料（例如证书或票据），然后被信任。

[^26]: Nat McHugh，“How I Created Two Images with the Same MD5 Hash”，Nat McHugh Blog，2014 年 10 月 31 日。[链接](https://natmchugh.com/blog/how-i-created-two-images-with-the-same-md5-hash/)

- **密文修改**：如果由流密码（如 RC4）或某些分组密码（ECB、CBC 和 CTR）生成的密文缺乏完整性检查，攻击者可以修改它，以便在接收者处理后生成有用的明文。比特翻转和块重排可以在解密时产生可预测的明文变化。

- **密文重放**：许多协议（包括较旧的 802.11 WiFi 标准）不跟踪状态。例如，加密的 802.11 网络帧可以被重放到接入点或客户端，产生意外结果。在身份验证上下文中，缺乏状态跟踪或使用加密随机数的实现也容易受到重放攻击——攻击者捕获一个令牌，然后稍后呈现它以进行身份验证。

- **侧信道攻击**：敏感材料有时通过预言机泄露。在大多数情况下，侧信道攻击涉及与定时预言机或错误预言机的交互。定时预言机根据事件的时间泄露信息，而错误预言机允许攻击者基于服务器错误消息推断秘密（通常是逐字节迭代进行）。

其他攻击取决于系统实现。在设计密码系统时，考虑安全的密钥生成和处理，以及应用正确的密码学原语（例如使用 HMAC 而不是简单的哈希函数）至关重要。操作的顺序也可能引入漏洞。例如，“先签名后加密”可能导致问题，因为密文未经身份验证。

## 漏洞与攻击者回顾

缺陷可能存在于计算机系统的不同层级：

- **硬件**：负责物理数据处理的基础设施
- **软件**：提供计算和数据处理的应用程序组件
- **湿件**：与软件交互并从数据中得出结论的用户

本书描述了在软件领域中发现的漏洞，并涉及硬件和湿件攻击（通过讨论物理系统入侵和社会工程学）。

通过威胁建模，你可以理解攻击者如何为了获利而影响或观察系统。在绘制暴露路径之后，你可以准备安全控制措施以减轻已知风险并提高安全性。
