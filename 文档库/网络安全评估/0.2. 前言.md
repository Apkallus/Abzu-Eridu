# 前言

攻击者通常以网络为目标获取利益。在我准备《网络安全评估》第三版之际，对事件响应专业知识的需求也在增长。尽管过去十年中，软件供应商已努力提升其产品的安全性，但系统复杂性和攻击面也在扩大，总体而言，互联网的完整性甚至有所下降。

攻击者的手段日益精炼，他们将软件缺陷的复杂利用、社会工程学和物理攻击战术相结合，以高价值资产为目标。更糟糕的是，许多部署用于保护网络的技术已被证明效果不佳。Google Project Zero[^1] 团队成员 Tavis Ormandy 公开披露了许多安全产品中存在的严重可远程利用的漏洞[^2]。

[^1]: 关于 Project Zero 的概述，参见 Andy Greenberg 的文章《Meet Project Zero, Google's Secret Team of Bug-Hunting Hackers》，Wired，2014 年 7 月 15 日。[原文链接](https://www.wired.com/2014/07/google-project-zero/)
[^2]: Ormandy 分别在《FireEye Exploitation: Project Zero's Vulnerability of the Beast》和《Sophail: Applied Attacks Against Sophos Antivirus》中描述了影响 FireEye 和 Sophos 产品的漏洞。此外，他还发推文提及一个卡巴斯基的漏洞，并指出了赛门铁克和趋势科技产品中的问题。[FireEye 报告](https://googleprojectzero.blogspot.com/2015/08/fireeye-exploitation-project-zeros.html) | [Sophail 报告](https://googleprojectzero.blogspot.com/2015/07/sophail-applied-attacks-against-sophos.html)

随着风险增加，研究成果的价值也在提升。安全研究人员出于经济激励，会向第三方和经纪人披露零日漏洞，后者再将发现分享给其客户，并在某些情况下负责任地通知产品供应商。如今存在一个日益扩大的差距：只有特权群体（如政府和有组织犯罪分子）才知道的严重缺陷数量每天都在增加。

一种下意识的反应是起诉黑客并遏制其工具的扩散。然而，我们面对的对手以及他们采用的战术，不过是一个严重问题的症状：我们使用的产品并不适合其用途。对许多科技公司而言，产品安全是事后才考虑的事情，我们今天面临的挑战正是这一问题的体现。

更糟糕的是，各国政府已将互联网军事化，并侵蚀了用于保护数据的密码系统的完整性[^3]。作为安全专业人士，我们必须倡导深度防御，以降低那些可能永远存在的风险，并努力确保我们的网络是进行商业活动、存储数据和相互通信的安全场所。没有互联网及其提供的自由，我们所有人的生活将大不相同。

[^3]: 参见 Daniel J. Bernstein 的《Making Sure Crypto Stays Insecure》以及观看 Matthew Green 的 TEDx 演讲《Why the NSA Is Breaking Our Encryption—And Why We Should Care》。[Bernstein 文章](https://cr.yp.to/talks/2015.09.14/slides.pdf) | [Green 演讲](https://www.youtube.com/watch?v=ZPn_7r-5BXw)

## 概述

本书详细探讨了计算机安全的一个特定领域——以结构化的方式进行基于网络的渗透测试。我所呈现的方法论描述了坚定的攻击者如何搜寻基于互联网的网络以寻找易受攻击的组件，以及你如何执行类似的练习来评估你自己的环境。

评估是任何组织管理其风险应迈出的第一步。通过以坚定对手相同的方式测试你的网络，你可以主动识别其中的弱点。在本书中，我将攻击性内容与项目符号清单的对抗措施相结合，帮助你制定清晰的技术策略并相应地加固你的环境。

## 读者对象

本书假定你熟悉网络协议和基于 Unix 的操作系统管理。如果你是一位经验丰富的网络工程师或安全顾问，你应该能够轻松理解各章内容。为了从本书中获得最大收益，你应该熟悉以下内容：

- OSI 第二层网络操作（主要是 ARP 和 802.1Q VLAN 标记）
- IPv4 协议套件，包括 TCP、UDP 和 ICMP
- 常见网络协议的操作（例如 FTP、SMTP 和 HTTP）
- 基本的运行时内存布局和 Intel x86 处理器寄存器
- 密码学原语（例如 Diffie-Hellman 和 RSA 密钥交换）
- 常见的 Web 应用缺陷（XSS、CSRF、命令注入等）
- 在您的环境中配置和构建基于 Unix 的工具

## 组织结构

本书由 15 章和 3 个附录组成。每章末尾都有一个清单，总结所描述的威胁和技术，以及推荐的应对措施。附录提供参考资料，包括你在测试中可能遇到的 TCP 和 UDP 端口列表。以下是各章和附录的简要描述：

- **第 1 章，网络安全评估简介**，讨论了网络安全评估的基本原理，并介绍了作为过程而非产品的信息保障。
- **第 2 章，评估工作流与工具**，涵盖了构成专业安全顾问攻击平台的工具，以及应采用的评估策略。
- **第 3 章，漏洞与攻击者**，通过分类法对软件漏洞进行分类，并包括漏洞类别和攻击者类型的底层描述。
- **第 4 章，互联网网络发现**，描述了潜在攻击者为绘制你的网络地图而采用的基于互联网的策略——从开放的网络搜索到 DNS 扫描和邮件服务器查询。
- **第 5 章，本地网络发现**，定义了执行局域网发现和嗅探的步骤，以及绕过 802.1Q 和 802.1X 安全功能的方法。
- **第 6 章，IP 网络扫描**，讨论了流行的网络扫描技术及其相关应用。还列出了支持这些扫描类型的工具，并涵盖了 IDS 规避和底层数据包分析技术。
- **第 7 章，评估常见网络服务**，详细介绍了用于测试在许多操作平台上运行的服务的方法。本章涵盖的协议包括 SSH、FTP、Kerberos、SNMP 和 VNC。
- **第 8 章，评估 Microsoft 服务**，涵盖了企业环境中 Microsoft 服务（NetBIOS、SMB Direct、RPC 和 RDP）的测试。
- **第 9 章，评估邮件服务**，详细评估了传输电子邮件的 SMTP、POP3 和 IMAP 服务。这些服务经常容易受到信息泄露和暴力破解攻击，在某些情况下甚至可导致远程代码执行。
- **第 10 章，评估 VPN 服务**，涵盖了基于网络的 IPsec 和 PPTP 服务测试，这些服务提供安全的网络访问和传输中数据的机密性。
- **第 11 章，评估 TLS 服务**，详细评估了为 Web、邮件和其他网络服务提供安全访问的 TLS 协议和功能。
- **第 12 章，Web 应用架构**，定义了 Web 应用服务器组件，并描述了它们之间的交互方式，包括协议和数据格式。
- **第 13 章，评估 Web 服务器**，涵盖了 Web 服务器软件的评估，包括 Microsoft IIS、Apache HTTP Server 和 Nginx。
- **第 14 章，评估 Web 应用框架**，详细介绍了用于发现框架内缺陷的策略，包括 Apache Struts、Rails、Django、Microsoft ASP.NET 和 PHP。
- **第 15 章，评估数据存储**，涵盖了数据库服务器（例如 Oracle Database、Microsoft SQL Server 和 MySQL）、存储协议以及在较大系统中发现的分布式键值存储的远程评估。
- **附录 A，常见端口与消息类型**，包含 TCP、UDP 和 ICMP 列表，并详细说明了相应的评估章节。
- **附录 B，漏洞信息来源**，列出了漏洞和利用信息的公共来源，以便你设计矩阵，在评估暴露的服务时快速识别潜在风险领域。
- **附录 C，不安全的 TLS 密码套件**，详细介绍了 TLS 支持的易受攻击的密码套件，这些套件应被禁用并尽可能避免使用。

## 本书涵盖的漏洞

本书特别描述了未认证用户和已认证用户针对网络服务利用的漏洞。主要不在讨论范围内的策略示例包括：本地权限提升、拒绝服务条件以及通过本地网络访问执行的入侵（包括中间人攻击）。

带有 2008 年及更早 CVE 参考的漏洞不在本书涵盖范围内。本书前几版分别于 2004 年和 2007 年出版，它们详细介绍了包括 Microsoft IIS、Apache 和 OpenSSL 在内的服务器软件包中的较旧漏洞。

为简洁起见，一些不太常见的服务器软件包未予涵盖。在测试期间，你应手动搜索 NIST 国家漏洞数据库（NVD）[^4] 以调查你已识别服务中的已知问题。

## 公认的评估标准

本书是根据公认的渗透测试标准编写的，包括 NIST SP 800-115、NSA IAM、CESG CHECK、CREST、Tiger Scheme、The Cyber Scheme、PCI DSS 和 PTES。你可以使用本书中的材料，为这些认证机构的基础设施和 Web 应用程序测试考试做准备。

### NIST SP 800-115

2008 年，美国国家标准与技术研究院（NIST）发布了特别出版物 800-115[^5]，这是一份安全测试技术指南。PCI DSS 材料将该文档作为行业公认最佳实践的示例进行引用。SP 800-115 从高层次描述了评估过程，并概述了应对系统进行的低层次测试。

[^5]: Karen Scarfone 等人，《Technical Guide to Information Security Testing and Assessment》，美国国家标准与技术研究院，2008 年 9 月。[NIST 下载页面](https://csrc.nist.gov/publications/detail/sp/800-115/final)

### NSA IAM

美国国家安全局（NSA）发布了 INFOSEC 评估方法论（IAM）框架，以帮助 NSA 外部的顾问和安全专业人员为客户提供评估服务。IAM 框架定义了与计算机网络测试相关的三个评估级别：

**评估（第 1 级）**
此级别涉及合作性的高层发现，包括目标组织的策略、流程以及系统内信息流的细节。在此级别不进行动手的网络或系统测试。

**评价（第 2 级）**
评价是一个动手的合作过程，涉及网络扫描、使用渗透测试工具以及应用特定的技术专长。

**红队（第 3 级）**
红队评估是对目标网络的非合作性外部测试，涉及模拟适当攻击者的渗透测试。红队评估包括对漏洞的全面定性。

本书描述了 IAM 框架第 2 级和第 3 级中使用的技术漏洞扫描和渗透测试技术。

### CESG CHECK

英国政府通信总部（GCHQ）设有一个信息保障部门，称为通信电子安全组（CESG）。正如 NSA IAM 框架使政府外部的安全顾问能够提供评估服务一样，CESG 运营着一个名为 CHECK[^6] 的项目，用于评估和认证英国境内的测试团队。

与涵盖信息保障许多方面（包括安全策略审查、防病毒、备份和灾难恢复）的 NSA IAM 不同，CHECK 直接处理网络安全评估。另一个项目是 CESG 所列顾问计划（CLAS），它通过处理 ISO/IEC 27001、安全策略创建和审计，在更广泛的意义上涵盖信息安全。

顾问们需要完成 CESG 批准的突击课程（主要是由 CREST 和 Tiger Scheme 维护的课程）以展示能力并获得认证。CESG CHECK 说明中列出了以下技术能力示例：

- 使用 DNS 信息检索工具获取单条和多条记录，包括理解与目标主机相关的 DNS 记录结构
- 使用 ICMP、TCP 和 UDP 网络映射和探测工具
- 演示 TCP 服务横幅抓取
- 使用 SNMP 检索信息，包括理解与目标系统配置和网络路由相关的 MIB 结构
- 理解路由器和交换机中与 Telnet、HTTP、SNMP 和 TFTP 访问及配置相关的常见弱点

以下是 Unix 特定能力：

- 用户枚举（通过 finger、rusers、rwho 和 SMTP 技术）
- RPC 服务枚举及安全影响演示
- 识别网络文件系统（NFS）弱点
- 测试 r 服务（rsh、rexec 和 rlogin）中的弱点
- 检测不安全的 X Windows 服务器
- 识别 Web、FTP 和 Samba 服务中的弱点

以下是 Windows 特定能力：

- 评估 NetBIOS、SMB 和 RPC 服务，以枚举用户、组、共享、域、密码策略及相关弱点
- 通过 SMB 和 RPC 服务进行用户名和密码猜测
- 演示 Microsoft IIS 和 SQL Server 中已知缺陷的存在

本书记录了这些领域中的评估策略，并提供了支持信息，帮助你深入理解漏洞。尽管 CHECK 项目评估希望从事英国政府安全测试工作的顾问的方法论，但其他地区的安全团队和组织也应了解该框架。

[^6]: 《CHECK Fundamental Principles》，国家网络安全中心，2015 年 10 月 23 日。[NCSC 页面](https://www.ncsc.gov.uk/guidance/check-fundamental-principles)

### CESG 认可资格

在英国，多个机构通过 CESG 批准的突击课程提供培训和考试。这些组织提供的资格被 CESG 认定为等同于 CHECK，具体如下：

**CREST**
CREST 是一个非营利组织，通过其认证的基础设施测试员和认证的 Web 应用程序测试员项目提供认证，从而规范渗透测试行业。通过与 CESG 的合作，CREST 认证的测试员资格授予 CHECK 团队负责人身份，因此许多组织使用该教学大纲来认证测试团队成员。

**Tiger Scheme**
第二个与政府和行业合作的考试机构是 Tiger Scheme。认证级别为准会员、合格会员和高级会员，这些级别被 CESG 认可，可用于获得 CHECK 团队成员和团队负责人身份。

**The Cyber Scheme**
The Cyber Scheme 团队成员（CSTM）认证被 CESG 认可为等同于 CHECK 团队成员。该组织通过其批准的合作伙伴提供培训和认证。

### PCI DSS

支付卡行业安全标准委员会（PCI SSC）维护着 PCI 数据安全标准（PCI DSS），该标准要求支付处理商、商户以及使用支付卡数据的人员遵守特定的控制目标，包括：

- 构建和维护安全的网络
- 保护持卡人数据
- 维护漏洞管理程序
- 实施强大的访问控制措施
- 定期监控和测试网络
- 维护信息安全策略

当前标准为 PCI DSS 3.1 版。在该文档中，针对支付处理商和商户的漏洞扫描和渗透测试有两项要求：

**要求 11.2**
强制要求每季度进行内部和外部漏洞扫描。必须聘请 PCI SSC 批准的扫描供应商（ASV）执行外部测试，但内部测试目的不需要 ASV 认证。

**要求 11.3**
要求由合格资源按照行业公认的最佳实践（即 NIST SP 800-115）每年执行内部和外部渗透测试。

本书是根据 NIST SP 800-115 和其他已发布标准编写的，因此你可以使用该方法执行内部和外部测试，并特别满足 PCI DSS 要求 11.3。

### PTES

PCI SSC 认可渗透测试执行标准（PTES）[^7] 作为测试的参考框架，该标准由七个部分组成。PTES 网站包含各部分的详细材料，如下所示：

- 参与前互动
- 利用
- 情报收集
- 威胁建模
- 漏洞分析
- 后利用
- 报告

[^7]: 关于该标准的详细信息，请访问 [PTES 网站](http://www.pentest-standard.org)
