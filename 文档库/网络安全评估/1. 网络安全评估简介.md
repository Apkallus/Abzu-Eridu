# 第 1 章 网络安全评估简介

本章介绍计算机网络攻击与防御背后的基本经济原理，描述当前形势以及近年来的变化。要实现一个可防御的环境，你必须采取主动的安全方法——从评估开始，了解自身的暴露面。评估有多种类型，从对给定应用程序及其代码的静态分析，到对运行系统的动态测试。我在此对这些测试选项进行分类，并列出本书详细涵盖的领域。

## 现状

大约 20 年前，我开始撰写本书的第一版，那时计算机网络攻击尚未被政府和有组织犯罪分子工业化到我们今天所知的规模。零日漏洞销售业务尚未出现，黑客是网络上的顶级掠食者，通过 IRC 交易破解软件。

当前形势令人深感担忧。现代生活严重依赖计算机网络和应用程序，这些系统复杂且向多个方向加速发展（想想云应用、医疗设备和自动驾驶汽车）。越来越多的消费者使用有缺陷的产品，从而引入了漏洞。

互联网是全球经济体系的主要推动力，几乎被用于一切。国际应用系统分析研究所（IIASA）的一项研究预测，一个国家如果完全失去互联网服务，粮食供应链将在三天内崩溃。[^1]

[^1]: Leena Ilmola-Sheppard 和 John Casti，《案例研究：七次冲击与芬兰》，《创新与供应链管理》第 7 卷，第 3 期（2013 年）：112–124。[链接](https://www.iiasa.ac.at/)

攻击的受害者往往遭受严重的经济负面影响。2009 年，Stuxnet 蠕虫破坏了伊朗纳坦兹的铀浓缩设施，使其运行能力在数月内下降了 30%。[^2] 2012 年，全球市值最高的公司沙特阿美遭遇恶意软件爆发，导致公司瘫痪 10 天。[^3] 破坏性攻击一直持续到 2015 年和 2016 年，索尼影视娱乐公司和好莱坞长老会医疗中心均遭受了服务中断。[^4]

在资源丰富的攻击者与负责保护计算机网络的人员之间存在明显的差距。由于劳动力流动性增加、无线技术的采用以及云服务，即使是最注重安全的组织（包括 Facebook）的系统也可能被控制。[^5] 更糟糕的是，在我们用于通过密码学提供保障的技术和库中，已经发现了严重缺陷，包括：

- RSA BSAFE，默认使用有缺陷的 Dual_EC_DRBG 算法[^6]
- OpenSSL 1.0.1 心跳扩展信息泄露漏洞[^7]

可防御的网络确实存在，但通常是国防工业基地内的组织构建的小型高保障飞地。这些飞地经过严密配置和监控，运行受信任的操作系统和经过评估的软件。许多还使用硬件在特定系统组件之间强制执行单向通信。这些加固后的环境并非无懈可击，但可以通过主动监控进行有效防御。

风险最大的网络是那些拥有大量元素的网络。多个入口点增加了被入侵的可能性，风险管理变得困难。这些因素造成了防御者的困境——防御者必须确保整个系统的完整性，而攻击者只需要利用一个漏洞。

## 威胁与攻击面

主动入侵计算机系统的对手包括国家支持的组织、有组织犯罪分子和业余爱好者。国家支持的攻击者受益于比其他群体总和更深的资源，已成为网络上的顶级掠食者。

理解了攻击面之后，你就可以开始量化风险。大多数企业的暴露攻击面包括客户端系统（如笔记本电脑和移动设备）、基于互联网的服务器、Web 应用和网络基础设施（如 VPN 设备、路由器和防火墙）。

### 攻击客户端软件

桌面应用程序和客户端软件包（例如 Microsoft Office、包括 Google Chrome 在内的 Web 浏览器，以及 PuTTY 等实用程序）可以被具有网络访问权限的攻击者直接攻击，或者通过攻击者发送恶意内容（如精心制作的 Microsoft Excel 文件）进行解析来间接攻击。

要证明客户端软件包的不安全性，你只需看看 Pwn2Own 竞赛的结果。2014 年，法国安全公司 VUPEN[^8] 在 64 位 Windows 8.1 上成功利用 Microsoft Internet Explorer 11、Adobe Reader XI、Google Chrome、Adobe Flash 和 Mozilla Firefox 后赢得了 40 万美元。该公司共使用了 11 个不同的零日漏洞来实现其目标。[^9]

[^9]: Michael Mimoso，《VUPEN 披露已修复的 Firefox Pwn2Own 零日漏洞详情》，Threatpost 博客，2014 年 5 月 21 日。[链接](https://threatpost.com/vupen-discloses-details-of-patched-firefox-pwn2own-zero-day/106241/)

具有网络访问权限的攻击者通常利用浏览器漏洞，通过中间人（MITM）攻击将恶意内容注入明文 HTTP 会话，以高价值资产为目标（例如 OPEC[^10] 等组织的系统管理员），如图 1-1 所示。

MITM 攻击不仅限于明文网络会话。一个资源丰富的攻击者如果通过网络访问，在通过软件缺陷或利用操作安全漏洞获得私钥材料后，也可能破坏合法的加密会话（例如 HTTPS）。

### 攻击服务器软件

由于抽象层级的增加和新兴技术的采用，服务器软件的情况也好不到哪里去。Rails 应用服务器和 Nginx 反向代理在 2013 年均遭受了严重的代码执行缺陷，具体如下：

- Rails 2.3 和 3.x Action Pack YAML 反序列化缺陷[^11]
- Nginx 1.3.9 到 1.4.0 分块编码栈溢出漏洞[^12]

[^13]: 参见 CVE-2002-0392。[CVE 详情](https://nvd.nist.gov/vuln/detail/CVE-2002-0392)

## 暴露的逻辑

理解计算机系统内攻击面的一种方法是考虑存在的暴露逻辑。这些逻辑可以是互联网可访问的网络服务的一部分（例如 Web 服务器中对压缩或分块编码的支持），也可以是客户端使用的解析机制（如 PDF 渲染）。在操作系统内部，特权内核组件和设备驱动程序向运行中的应用程序暴露逻辑。

攻击面是可以被攻击者操纵以获取利益的逻辑（通常是特权逻辑）——无论是泄露秘密、促成代码执行还是引发拒绝服务。暴露的逻辑被基于网络的攻击者以两种方式定位：

- 直接通过暴露的网络服务中的易受攻击函数。
- 间接地，例如通过 MITM 攻击、电子邮件、消息传递或其他方式向客户端系统中的解析器提供恶意材料。

### 暴露逻辑的示例

一个有用的例子是 GNU bash 命令执行缺陷，被称为 Shellshock。[^15] Unix 系统（包括 Linux 和 Apple OS X）中的许多应用程序使用 bash 命令 shell 作为代理来执行底层系统操作。1989 年引入了一个缺陷，意味着可以通过向 bash 传递恶意环境变量来执行任意命令。该问题在 25 年后由 Stéphane Chazelas 发现。[^16]

[^15]: 参见 CVE-2014-6271。[CVE 详情](https://nvd.nist.gov/vuln/detail/CVE-2014-6271)
[^16]: Nicole Perlroth，《安全专家预计 Bash 中的 ‘Shellshock’ 软件漏洞将影响重大》，《纽约时报》，2014 年 9 月 25 日。[链接](https://www.nytimes.com/2014/09/26/technology/security-experts-expect-shellshock-software-bug-in-bash-to-be-significant.html)

要利用该缺陷，攻击者必须找到一条路径，将用户控制的内容传递给易受攻击的命令 shell。两个可远程利用的路径和前置条件示例如下：

**Apache HTTP Server**
通过 `mod_cgi` 向 CGI 脚本提交包含恶意环境变量的 `User-Agent` 头会导致命令执行，正如 Metasploit 所演示的那样。[^17]

[^17]: Metasploit `apache_mod_cgi_bash_env_exec` 模块。[Metasploit 框架](https://www.metasploit.com/)

**DHCP**
许多 DHCP 客户端在配置网络接口时通过 bash 传递和执行命令。当配置一个恶意 DHCP 服务器向客户端发送包含恶意环境变量的 DHCP 响应时，任意命令会在易受攻击的系统上执行，正如 Metasploit 所演示的那样。[^18]

[^18]: Metasploit `dhclient_bash_env` 模块。[Metasploit 框架](https://www.metasploit.com/)

### 利用暴露的逻辑

安全研究行业建立在利用暴露逻辑执行有价值操作的基础上。大多数浏览器利用依赖于串联多个缺陷以绕过安全保护并执行任意代码。

2012 年，著名黑客 Pinkie Pie 开发了一个 Google Chrome 漏洞利用程序，使用了六个不同的缺陷来实现代码执行。[^19] 他在 Chrome 的预渲染、GPU 命令缓冲区、IPC 层和扩展管理器等组件中发现了可利用的条件。

[^19]: Jorge Lucangeli Obes 和 Justin Schuh，《两只小马的故事（第一部分）》，Chromium 博客，2012 年 5 月 22 日。[链接](https://blog.chromium.org/2012/05/tale-of-two-pwnies-part-1.html)

底层软件评估是一门艺术，相对较少数量的熟练研究人员发现软件中的细微缺陷，并将其串联起来以获取利益。在每个迭代测试步骤中，都会评估可用的攻击面，以识别对更大系统的风险。

## 评估类型

为了映射和测试系统的暴露逻辑路径，组织采用了多种方法。在当今市场上，有许多供应商提供静态和动态测试服务，以及用于识别潜在风险和漏洞的分析工具。

### 静态分析

审计应用程序源代码、服务器配置、基础设施配置和架构可能耗时，但这是识别系统中漏洞的最有效方法之一。在大型环境中进行静态分析的一个缺点是成本（主要由于这些工具产生的大量材料以及排除误报的成本），因此重要的是缩小测试范围并正确确定工作优先级。

技术审计和审查方法包括：

- 设计审查
- 配置审查
- 静态代码分析

较少技术性的考虑可能包括数据分类和标记、物理环境审查、人员安全以及教育、培训和意识。

#### 设计审查

系统架构的审查首先涉及理解安全控制在环境中的位置和配置（无论是基于网络的控制（如 ACL）还是底层系统控制（如沙箱）），评估这些控制的效力，并在适用的情况下提出架构变更建议。

通用标准[^20] 是计算机安全认证的国际标准，适用于具有安全声明的操作系统、应用程序和产品。存在七个保证级别，从 EAL1（功能测试）到 EAL4（方法学设计、测试和审查），再到 EAL7（正式验证的设计和测试）。许多商业操作系统通常在 EAL4 级别进行评估，提供多级安全性的操作系统至少评估为 EAL4。还有其他项目，包括英国的 CESG CPA 计划。

[^20]: 参见 ISO/IEC 15408。[CC 门户](https://www.commoncriteriaportal.org/)

正式验证系统设计可能成本高昂。通常，由经验丰富的安全专业人员进行粗略的架构审查就能突显应缓解的潜在陷阱和问题，例如网络分段不佳或传输中数据保护不足。

#### 配置审查

系统组件的底层审计可以包括对基础设施（即防火墙、路由器、交换机、存储和虚拟化基础设施）、服务器和设备操作系统配置（例如 Windows Server、Linux 或 F5 Networks 硬件）以及应用程序配置（例如 Apache 或 OpenSSL 服务器配置）的审查。

NIST、NSA 和 DISA 等组织提供了操作系统的检查清单和配置指南，包括 Apple OS X、Microsoft Windows 和 Linux。这些资源可在线获取：

- NIST 的“国家检查清单计划存储库”[^21]
- NSA 的“IA 指南”[^22]
- DISA 的“安全技术实施指南（STIG）”[^23]

[^21]: [https://nvd.nist.gov/ncp/repository](https://nvd.nist.gov/ncp/repository)
[^22]: [https://www.nsa.gov/resources/everyone/ia-guidance/](https://www.nsa.gov/resources/everyone/ia-guidance/)
[^23]: [https://public.cyber.mil/stigs/](https://public.cyber.mil/stigs/)

通过对这些标准进行差距分析，可以识别操作系统配置中的不足，并努力确保整个环境具有统一的加固配置。批量漏洞扫描器（包括 Rapid7 Nexpose）包含 DISA STIG 扫描策略，因此通过认证测试可以轻松识别差距。

#### 静态代码分析

NIST 和维基百科维护着代码分析工具列表，如下所示：

- NIST 的“源代码安全分析器”[^24]
- 维基百科的“静态代码分析工具列表”[^25]

[^24]: [https://www.nist.gov/itl/ssd/software-quality-group/source-code-security-analyzers](https://www.nist.gov/itl/ssd/software-quality-group/source-code-security-analyzers)
[^25]: [https://en.wikipedia.org/wiki/List_of_tools_for_static_code_analysis](https://en.wikipedia.org/wiki/List_of_tools_for_static_code_analysis)

这些工具可以识别用 C/C++、Java 和 Microsoft .NET 等语言编写的软件中的常见缺陷。HP Fortify 团队发布了一份可通过静态代码分析识别的软件安全错误分类法[^26]，包括输入验证与表示、API 滥用、有缺陷的安全特性、时间和状态漏洞以及错误处理缺陷。第 3 章将详细讨论这些类别。

[^26]: [https://vulncat.hpefod.com](https://vulncat.hpefod.com)

静态代码分析工具需要调优以减少结果中的噪音，并将发现集中在可访问的代码路径上（例如，可被实际利用的缺陷）。这种底层静态分析适用于关键系统组件，因为审查输出的成本可能相当高。

### 动态测试

暴露的逻辑通过对运行系统的动态测试进行评估，包括：

- 网络基础设施测试
- Web 应用测试
- Web 服务测试（例如支持移动应用的 API）
- 基于互联网的社会工程学

由于测试是从攻击者的角度进行的（例如，未经身份验证的基于网络的攻击者、已认证的应用用户或移动客户端），发现与系统的真实威胁相一致。

#### 网络基础设施测试

扫描工具（例如 Nmap、Nessus、Rapid7 Nexpose 和 QualysGuard）会映射和评估环境的网络攻击面，以查找已知漏洞。然后应用手动评估周期进一步调查攻击面，并评估可访问的网络服务。

还可以进行内部网络测试，以识别和利用 OSI 第 2 层和第 3 层的漏洞（例如 ARP 缓存投毒和 802.1Q VLAN 跳跃）。第 5 章讨论了应在本地采用的网络发现和评估技术，以识别弱点。

#### Web 应用测试

Web 应用逻辑通常以未经身份验证或经过身份验证的方式进行评估。大多数组织选择对应用程序进行经过身份验证的测试，模拟持有有效凭证或会话令牌、试图提升权限的攻击者。

开放式 Web 应用程序安全项目（OWASP）Top 10 是常见 Web 应用缺陷的列表。可以可靠测试此类缺陷的工具包括 Burp Suite、IBM Security AppScan、HP WebInspect 和 Acunetix。这些工具提供的测试功能使组织能够广泛扫描其暴露的 Web 应用逻辑，以查找常见问题，包括跨站脚本（XSS）、跨站请求伪造（CSRF）、命令注入会话管理缺陷和信息泄露缺陷。深度 Web 应用测试不在本书范围内；但是，第 12 章至第 14 章详细介绍了 Web 服务器和应用程序框架的评估。

> **提示**：在 2012 年和 2013 年期间，我为遭受 Alexsey Belan 攻击的公司进行了事件响应和取证工作。在每起案件中，他都通过入侵内部 Web 应用来提升权限并横向移动。这突显了对环境中非互联网可访问的 Web 应用进行测试和加固的重要性。

#### 基于互联网的社会工程学

在我的职业生涯中，测试中完成的一些最大入侵是通过基于互联网的社会工程学实现的。两种有效的攻击场景如下：

- 配置一个伪装成合法资源的基于互联网的 Web 服务器，然后通过电子邮件向用户发送包含恶意页面链接的材料。
- 通过电子邮件、消息传递或其他方式，看似来自可信来源（如朋友或同事），直接将恶意材料（例如，用于利用 Microsoft Excel 或 Adobe Acrobat Reader 的精心制作的文档）发送给用户。

我最近对一家金融服务机构进行了一次基于互联网的鱼叉式网络钓鱼演练，涉及一个伪造的 SSL VPN 端点，并通过电子邮件向 200 名用户发送了登录“新公司 VPN 网关”的说明。在两小时内，有 13 名用户输入了他们的 Active Directory 用户名、域密码和双因素认证令牌值。第 9 章详细介绍了网络钓鱼策略和工具。
