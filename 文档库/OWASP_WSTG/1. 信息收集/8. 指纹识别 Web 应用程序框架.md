# 指纹识别 Web 应用程序框架

|ID          |
|------------|
|WSTG-INFO-08|

## 概要

毫不夸张地说，几乎所有能想到的 Web 应用程序想法都已经投入开发。由于全球范围内有大量积极开发和部署的免费开源软件项目，应用程序安全测试很可能遇到完全或部分依赖这些知名应用程序或框架（例如 WordPress、phpBB、Mediawiki 等）的目标。了解被测的 Web 应用程序组件将极大地帮助测试过程，并且还会显著减少测试期间所需的工作量。这些知名的 Web 应用程序具有特定的 HTML 头部、cookie 和目录结构，可以通过枚举来识别应用程序。大多数 Web 框架在这些位置都有几个标记，可以帮助攻击者或测试人员识别它们。这基本上就是所有自动化工具所做的：它们从预定义的位置查找标记，然后与已知签名数据库进行比较。为了提高准确性，通常会使用多个标记。

## 测试目标

- 指纹识别 Web 应用程序所使用的组件。

## 如何测试

### 黑盒测试

有几个常见的位置可以考虑用于识别框架或组件：

- HTTP 头部
- Cookie
- HTML 源代码
- 特定文件和文件夹
- 文件扩展名
- 错误消息

#### HTTP 头部

识别 Web 框架最基本的形式是查看 HTTP 响应头中的 `X-Powered-By` 字段。许多工具可用于对目标进行指纹识别，最简单的是 netcat。

考虑以下 HTTP 请求-响应：

```html
$ nc 127.0.0.1 80
HEAD / HTTP/1.0

HTTP/1.1 200 OK
Server: nginx/1.0.14
[...]
X-Powered-By: Mono
```

从 `X-Powered-By` 字段，我们可以了解到 Web 应用程序框架很可能是 `Mono`。然而，虽然这种方法简单快捷，但并非在所有情况下都有效。通过适当的配置可以轻松禁用 `X-Powered-By` 头部。还有一些技术可以让站点混淆 HTTP 头部（参见[修复措施](#remediation)部分的示例）。在上面的示例中，我们还可以注意到使用了特定版本的 `nginx` 来提供内容。

在同样的示例中，测试人员可能会错过 `X-Powered-By` 头部，或者得到如下答案：

```html
HTTP/1.1 200 OK
Server: nginx/1.0.14
Date: Sat, 07 Sep 2013 08:19:15 GMT
Content-Type: text/html;charset=ISO-8859-1
Connection: close
Vary: Accept-Encoding
X-Powered-By: Blood, sweat and tears
```

有时会有更多的 HTTP 头部指向某个特定框架。在以下示例中，根据 HTTP 请求中的信息，可以看到 `X-Powered-By` 头部包含 PHP 版本。然而，`X-Generator` 头部指出实际使用的框架是 `Swiftlet`，这有助于渗透测试人员扩展其攻击向量。在执行指纹识别时，应仔细检查每个 HTTP 头部是否存在此类泄露。

```html
HTTP/1.1 200 OK
Server: nginx/1.4.1
Date: Sat, 07 Sep 2013 09:22:52 GMT
Content-Type: text/html
Connection: keep-alive
Vary: Accept-Encoding
X-Powered-By: PHP/5.4.16-1~dotdeb.1
Expires: Thu, 19 Nov 1981 08:52:00 GMT
Cache-Control: no-store, no-cache, must-revalidate, post-check=0, pre-check=0
Pragma: no-cache
X-Generator: Swiftlet
```

#### Cookie

另一种类似且更可靠的方法是使用框架特定的 cookie 来确定当前 Web 框架。

考虑以下 HTTP 请求：

![Cakephp HTTP 请求](images/Cakephp_cookie.png)\
*图 4.1.8-7：Cakephp HTTP 请求*

`CAKEPHP` cookie 已被自动设置，这提供了有关正在使用的框架的信息。常见 cookie 名称列表在 [Cookie](#cookies-1) 部分给出。依赖这种识别机制仍然存在局限性——可以更改 cookie 的名称。例如，对于所选的 `CakePHP` 框架，可以通过以下配置完成（摘自 `core.php`）：

```php
/**
* The name of CakePHP's session cookie.
*
* Note the guidelines for Session names states: "The session name references
* the session id in cookies and URLs. It should contain only alphanumeric
* characters."
* @link https://php.net/session_name
*/
Configure::write('Session.cookie', 'CAKEPHP');
```

然而，与更改 `X-Powered-By` 头部相比，这些更改不太可能发生，因此这种识别方法更可靠。

#### HTML 源代码

这种技术基于在 HTML 页面源代码中寻找特定模式。通常可以找到大量有助于测试人员识别特定组件的信息。常见的标记之一是直接泄露框架的 HTML 注释。更常见的是，可以找到特定于框架的路径，即指向特定于框架的 CSS 或 JS 文件夹的链接。最后，特定的脚本变量也可能指向某个框架。

从下面的截图中，可以通过提到的标记轻松确定正在使用的框架及其版本。注释、特定路径和脚本变量都可以帮助攻击者快速识别 ZK 框架的实例。

![ZK 框架示例](images/Zk_html_source.png)\
*图 4.1.8-2：ZK 框架 HTML 源代码示例*

此类信息通常位于 HTTP 响应的 `<head>` 部分、`<meta>` 标签中或页面末尾。尽管如此，应分析整个响应，因为它可用于其他目的，例如检查其他有用的注释和隐藏字段。有时，Web 开发人员可能没有充分隐藏有关所用框架或组件的信息。仍然可能在页面底部偶然发现类似这样的内容：

### 特定文件和文件夹

另一种方法极大地帮助攻击者或测试人员高精度地识别应用程序或组件。每个 Web 应用程序组件在服务器上都有其特定的文件和文件夹结构。已经注意到，可以从 HTML 页面源代码中看到特定路径，但有时它们并未明确显示在那里，但仍可能存在于服务器上。

为了发现它们，使用了一种称为强制浏览或“目录爆破”的技术。目录爆破是使用已知的文件夹和文件名对目标进行暴力破解，并监控 HTTP 响应以枚举服务器内容。此信息既可用于查找默认文件并攻击它们，也可用于对 Web 应用程序进行指纹识别。目录爆破可以通过多种方式完成，下面的示例展示了在 Burp Suite 的列表和 intruder 功能的帮助下，针对 WordPress 驱动的目标进行成功的目录爆破攻击。

![使用 Burp 进行目录爆破](images/Wordpress_dirbusting.png)\
*图 4.1.8-4：使用 Burp 进行目录爆破*

我们可以看到，对于某些 WordPress 特定文件夹（例如 `/wp-includes/`、`/wp-admin/` 和 `/wp-content/`），HTTP 响应分别为 403（禁止）、302（找到，重定向到 `wp-login.php`）和 200（OK）。这表明目标是由 WordPress 驱动的。同样地，可以目录爆破不同的应用程序插件文件夹及其版本。在下面的截图中，可以看到一个典型的 Drupal 插件的 CHANGELOG 文件，它提供了有关正在使用的应用程序的信息并披露了易受攻击的插件版本。

*图 4.1.8-5：Drupal Botcha 信息泄露*

提示：在开始目录爆破之前，请先检查 `robots.txt` 文件。有时也可以在其中找到应用程序特定的文件夹和其他敏感信息。下面的截图中展示了这样一个 `robots.txt` 文件的示例。

![Robots 信息泄露](images/Robots-info-disclosure.png)\

特定文件和文件夹因具体应用程序而异。如果识别出的应用程序或组件是开源的，那么在渗透测试期间进行临时安装可能是有价值的，以便更好地了解呈现了哪些基础架构或功能，以及服务器上可能留下了哪些文件。然而，已有几个有用的文件列表；一个值得注意的例子是 [FuzzDB 的可预测文件/文件夹单词列表](https://github.com/fuzzdb-project/fuzzdb)。

#### 文件扩展名

URL 可能包含文件扩展名，这也有助于识别 Web 平台或技术。

例如，OWASP wiki 使用了 PHP：

```text
https://wiki.owasp.org/index.php?title=Fingerprint_Web_Application_Framework&action=edit&section=4
```

以下是一些常见的 Web 文件扩展名及相关技术：

- `.php` -- PHP
- `.aspx` -- Microsoft ASP.NET
- `.jsp` -- Java Server Pages

## 修复措施

虽然可以努力使用不同的 cookie 名称（通过更改配置）、隐藏或更改文件/目录路径（通过重写或源代码更改）、删除已知头部等，但这些努力归结为“不安全的隐藏”。系统所有者/管理员应认识到，这些努力只会减慢最初级的对手。时间和精力可能更好地用于提高利益相关者的意识以及维护解决方案。

## 工具

下面列出了一些通用且知名的工具。还有许多其他工具，以及基于框架的指纹识别工具。

### WhatWeb

网站：[https://github.com/urbanadventurer/WhatWeb](https://github.com/urbanadventurer/WhatWeb)

WhatWeb 是当前市场上最好的开源指纹识别工具之一，并包含在默认的 [Kali Linux](https://www.kali.org/) 构建中。语言：Ruby 指纹识别匹配通过以下方式进行：

- 文本字符串（区分大小写）
- 正则表达式
- Google Hack 数据库查询（有限的关键字集）
- MD5 哈希
- URL 识别
- HTML 标签模式
- 用于被动和主动操作的自定义 Ruby 代码

示例输出如下截图所示：

![Whatweb 输出示例](images/Whatweb-sample.png)\
*图 4.1.8-8：Whatweb 输出示例*

### Wappalyzer

网站：[https://www.wappalyzer.com/](https://www.wappalyzer.com/)

Wappalyzer 有多种使用模式，其中最流行的可能是 Firefox/Chrome 扩展。它们主要基于正则表达式匹配，除了在浏览器中加载页面外不需要任何其他东西。它完全在浏览器级别工作，并以图标形式给出结果。虽然有时会有误报，但在浏览页面后立即了解构建目标站点所使用的技术是非常方便的。
