# 映射应用程序架构

## 概要

为了有效地测试应用程序，并能够就如何解决发现的任何问题提供有意义的建议，了解实际测试的内容非常重要。此外，确定特定组件是否应被视为测试范围之外也可能有所帮助。

现代 Web 应用程序的复杂性差异很大，从在单个服务器上运行的简单脚本到分布在数十个不同系统、语言和组件中的高度复杂的应用程序。还可能存在额外的网络级组件，如防火墙或入侵防御系统，它们可能对测试产生重大影响。

## 测试目标

- 了解应用程序的架构以及所使用的技术。

## 如何测试

从黑盒角度进行测试时，重要的是尝试构建应用程序如何工作以及使用了哪些技术和组件的清晰图景。在某些情况下，可以测试特定组件（如 Web 应用程序防火墙），而其他组件可以通过检查应用程序的行为来识别。

以下部分提供了常见架构组件的高级概述，以及如何识别它们的详细信息。

### 应用程序组件

#### Web 服务器

简单的应用程序可能在单个服务器上运行，可以使用本指南的[指纹识别 Web 服务器](02-Fingerprint_Web_Server.md)部分中讨论的步骤进行识别。

#### 平台即服务（PaaS）

在平台即服务（PaaS）模型中，Web 服务器和底层基础设施由服务提供商管理，客户仅负责部署在其上的应用程序。从测试的角度来看，有两个主要区别：

- 应用程序所有者无法访问底层基础设施，这意味着他们将无法直接修复任何问题。
- 基础设施测试很可能在任何参与中都超出范围。

在某些情况下，可以识别出 PaaS 的使用，因为应用程序可能使用特定的域名（例如，部署在 Azure App Services 上的应用程序将具有 `*.azurewebsites.net` 域——尽管它们也可能使用自定义域）。在其他情况下，很难确定是否正在使用 PaaS。

#### 无服务器（Serverless）

在无服务器模型中，开发人员提供的代码直接在托管平台上作为独立函数运行，而不是运行部署在 Web 根目录中的传统大型 Web 应用程序。这使得它非常适合基于微服务的架构。与 PaaS 环境一样，基础设施测试很可能超出范围。

在某些情况下，无服务器代码的使用可能通过特定的 HTTP 头部的存在来指示。例如，AWS Lambda 函数通常会返回以下头部：

```http
X-Amz-Invocation-Type
X-Amz-Log-Type
X-Amz-Client-Context
```

Azure Functions 则不那么明显。它们通常会返回 `Server: Kestrel` 头部——但这本身不足以确定它是 Azure App 函数，因为它可能是其他在 Kestrel 上运行的代码。

#### 微服务

在基于微服务的架构中，应用程序 API 由多个离散服务组成，而不是作为一个单体应用程序运行。服务本身通常在容器内运行（通常使用 Kubernetes），并且可以使用各种不同的操作系统和语言。尽管它们通常位于单个 API 网关和域之后，但多种语言的使用（通常在详细的错误消息中表明）可能暗示正在使用微服务。

#### 静态存储

许多应用程序将静态内容存储在专用的存储平台上，而不是直接托管在主 Web 服务器上。两个最常见的平台是 Amazon 的 S3 存储桶和 Azure 的存储帐户，可以通过域名轻松识别：

- Amazon S3 存储桶：`BUCKET.s3.amazonaws.com` 或 `s3.REGION.amazonaws.com/BUCKET`
- Azure 存储帐户：`ACCOUNT.blob.core.windows.net`

这些存储帐户通常会暴露敏感文件，如[测试云存储指南](../02-Configuration_and_Deployment_Management_Testing/11-Test_Cloud_Storage.md)部分所述。

#### 数据库

大多数重要的 Web 应用程序都使用某种数据库来存储动态内容。在某些情况下，可以确定数据库的类型。这通常可以通过以下方式完成：

- 对服务器进行端口扫描，查找与特定数据库关联的任何开放端口。
- 触发 SQL（或 NoSQL）相关的错误消息（或从[搜索引擎](../01-Information_Gathering/01-Conduct_Search_Engine_Discovery_Reconnaissance_for_Information_Leakage.md)中找到现有错误）。

当无法最终确定数据库时，测试人员通常可以根据应用程序的其他方面进行有根据的猜测：

- Windows、IIS 和 ASP.NET 通常使用 Microsoft SQL Server。
- 嵌入式系统通常使用 SQLite。
- PHP 通常使用 MySQL 或 PostgreSQL。
- APEX 通常使用 Oracle。

这些不是硬性规则，但如果没有更好的信息，它们当然可以为您提供一个合理的起点。

#### 身份验证

大多数应用程序都有用户身份验证。可以使用多种身份验证后端，例如：

- Web 服务器配置（包括 `.htaccess` 文件）或在脚本中硬编码密码。
    - 通常显示为 HTTP Basic 身份验证，表现为浏览器中的弹窗和 `WWW-Authenticate: Basic` HTTP 头部。
- 数据库中的本地用户帐户。
    - 通常集成到应用程序的表单或 API 端点中。
- 现有的中央身份验证源，如 Active Directory 或 LDAP 服务器。
    - 可能使用 NTLM 身份验证，由 `WWW-Authenticate: NTLM` HTTP 头部指示。
    - 可能集成到 Web 应用程序的表单中。
    - 可能需要以 `DOMAIN\username` 格式输入用户名，或者可能给出可用域的下拉列表。
- 与内部或外部提供商进行单点登录（SSO）。
    - 通常使用 OAuth、OpenID Connect 或 SAML。

应用程序可能为用户提供多种身份验证选项（例如注册本地帐户，或使用其现有的 Facebook 帐户），并且可能对普通用户和管理员使用不同的机制。

#### 第三方服务和 API

几乎所有的 Web 应用程序都包含加载的或客户端与之交互的第三方资源。这些可以包括：

- [活动内容](https://developer.mozilla.org/en-US/docs/Web/Security/Mixed_content#mixed_active_content)（例如脚本、样式表、字体和 iframe）
- [被动内容](https://developer.mozilla.org/en-US/docs/Web/Security/Mixed_content#mixed_passivedisplay_content)（例如图像和视频）
- 外部 API
- 社交媒体按钮
- 广告网络
- 支付网关

这些资源由用户的浏览器直接请求，因此使用开发者工具或拦截代理更容易识别。虽然识别它们很重要（因为它们可能影响应用程序的安全性），但请记住，*它们通常超出测试范围*，因为它们属于第三方。

### 网络组件

#### 反向代理

反向代理位于一个或多个后端服务器之前，并将请求重定向到适当的目标。它们可用于实现各种功能，例如：

- 充当[负载均衡器](#load-balancer)或[Web 应用程序防火墙](#web-application-firewall-waf)
- 允许在单个 IP 地址或域（子文件夹中）托管多个应用程序
- 实施 IP 过滤或其他限制
- 缓存来自后端的内容以提高性能

并不总是能够检测到反向代理（尤其是当其后面只有一个应用程序时），但有时可以通过以下方式识别：

- 前端服务器和后端应用程序之间的不匹配（例如，带有 `Server: nginx` 头部的 ASP.NET 应用程序）
    - 这有时可能导致[请求走私漏洞](https://portswigger.net/web-security/request-smuggling)
- 重复的头部（尤其是 `Server` 头部）
- 在同一个 IP 地址或域上托管多个应用程序（特别是如果它们使用不同的语言）

#### 负载均衡器

负载均衡器位于多个后端服务器之前，并在它们之间分配请求，以便为应用程序提供更大的冗余和处理能力。

负载均衡器可能难以检测，但有时可以通过发出多个请求并检查响应中的差异来识别，例如：

- 不一致的系统时间
- 详细错误消息中的不同内部 IP 地址或主机名
- [服务器端请求伪造（SSRF）](../07-Input_Validation_Testing/19-Testing_for_Server-Side_Request_Forgery.md)返回的不同地址

它们也可能通过特定 cookie 的存在来指示（例如，F5 BIG-IP 负载均衡器会创建一个名为 `BIGipServer` 的 cookie）。

#### 内容分发网络（CDN）

内容分发网络（CDN）是一组地理上分布的缓存代理服务器，旨在提高站点性能。

通常通过将面向公众的域指向 CDN 的服务器，然后配置 CDN 连接到正确的后端服务器（有时称为“源站”）来完成配置。

检测 CDN 的最简单方法是执行 WHOIS 查找域解析到的 IP 地址。如果它们属于 CDN 公司（例如 Akamai、Cloudflare 或 Fastly - 参见 [Wikipedia](https://en.wikipedia.org/wiki/Content_delivery_network#Notable_content_delivery_service_providers) 获取更完整的列表），则很可能正在使用 CDN。

在测试位于 CDN 后面的站点时，应记住以下几点：

- IP 地址和服务器属于 CDN 提供商，很可能不在基础设施测试范围内。
- 许多 CDN 还包含诸如机器人检测、速率限制和 Web 应用程序防火墙等功能。
- CDN 通常会缓存内容。因此，在后端所做的更改可能不会立即出现在站点上。

如果站点位于 CDN 后面，识别后端服务器可能很有用。如果没有实施适当的访问控制，测试人员可能通过直接访问后端服务器来绕过 CDN（及其提供的任何保护）。有多种不同的方法可以识别后端系统：

- 应用程序发送的电子邮件可能直接来自后端服务器，这可能会暴露其 IP 地址。
- 域的 DNS 爆破、区域传输或证书透明度列表可能通过子域暴露它。
- 扫描已知由公司使用的 IP 范围可能有助于识别后端服务器。
- 利用[服务器端请求伪造（SSRF）](../07-Input_Validation_Testing/19-Testing_for_Server-Side_Request_Forgery.md)可能暴露 IP 地址。
- 应用程序的详细错误消息可能暴露 IP 地址或主机名。

### 安全组件

#### 网络防火墙

大多数 Web 服务器都会受到数据包过滤或状态检测防火墙的保护，该防火墙会阻止所有不必要的网络流量。要检测这一点，请对服务器执行端口扫描并检查结果。

如果大多数端口显示为“关闭”（即，它们响应初始 `SYN` 数据包时返回 `RST` 数据包），则表明服务器可能未受防火墙保护。如果端口显示为“已过滤”（即，向未使用的端口发送 `SYN` 数据包时未收到响应），则很可能是存在防火墙。

此外，如果不适当的服务暴露给外部（例如 SMTP、IMAP、MySQL 等），则表明要么没有防火墙，要么防火墙配置不当。

#### 网络入侵检测和防御系统

网络入侵检测系统（IDS）旨在检测可疑或恶意的网络级活动，例如端口或漏洞扫描，并发出警报。入侵防御系统（IPS）功能类似，但还会采取措施阻止活动，通常是通过阻止源 IP 地址。

通常可以通过对目标运行自动化扫描工具（例如端口扫描器）并观察源 IP 是否被阻止来检测 IPS。然而，许多应用级工具可能不会被 IPS 检测到（特别是如果它不解密 TLS）。

#### Web 应用程序防火墙（WAF）

Web 应用程序防火墙（WAF）检查 HTTP 请求的内容，并阻止那些看似可疑或恶意的请求。它们还可用于动态应用其他控制，如验证码或速率限制。它们通常使用一组已知的恶意签名和正则表达式（例如 [OWASP 核心规则集](https://owasp.org/www-project-modsecurity-core-rule-set/)）来识别恶意流量。WAF 可以有效防御某些类型的攻击，如 SQL 注入或跨站脚本，但对其他类型的攻击（如访问控制或业务逻辑相关问题）效果较差。

WAF 可以部署在多个位置，包括：

- Web 服务器本身
- 单独的虚拟机或硬件设备上
- 云端，位于后端服务器之前

由于 WAF 会阻止恶意请求，因此可以通过向参数添加常见的攻击字符串并观察它们是否被阻止来检测。例如，尝试添加一个名为 `foo` 的参数，其值为 `' UNION SELECT 1` 或 `><script>alert(1)</script>`。如果这些请求被阻止，则很可能存在 WAF。此外，阻止页面的内容可能提供有关正在使用的特定技术的信息。最后，某些 WAF 可能会向响应中添加 cookie 或 HTTP 头部，从而暴露其存在。

如果使用了基于云的 WAF，则可能通过直接访问后端服务器来绕过它，使用与[内容分发网络](#content-delivery-network-cdn)部分中讨论的相同方法。
