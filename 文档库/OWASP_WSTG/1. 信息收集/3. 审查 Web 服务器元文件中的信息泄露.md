# 审查 Web 服务器元文件中的信息泄露

|ID          |
|------------|
|WSTG-INFO-03|

## 概要

本节描述如何测试各种元文件，以发现 Web 应用程序路径或功能的信息泄露。此外，蜘蛛、机器人或爬虫应避免访问的目录列表也可以作为[通过应用程序映射执行路径](07-Map_Execution_Paths_Through_Application.md)的依赖项而创建。还可能收集其他信息，以识别攻击面、技术细节，或用于社会工程学参与。

## 测试目标

- 通过分析元文件识别隐藏或混淆的路径和功能。
- 提取并映射其他有助于更好理解当前系统的信息。

## 如何测试

> 以下使用 `wget` 执行的任何操作也可以使用 `curl` 完成。许多动态应用程序安全测试（DAST）工具（如 ZAP 和 Burp Suite）在其爬虫/蜘蛛功能中包含对这些资源的检查或解析。也可以使用各种 [Google Dorks](https://en.wikipedia.org/wiki/Google_hacking) 或利用高级搜索功能（如 `inurl:`）来识别它们。

### Robots 文件

Web 蜘蛛、机器人或爬虫获取一个网页，然后递归遍历超链接以获取更多 Web 内容。它们的行为受 Web 根目录下 `robots.txt` 文件中的[机器人排除协议](https://www.robotstxt.org)约束。

例如，以下摘录自 2020 年 5 月 5 日采样的 [Google](https://www.google.com/robots.txt) 的 `robots.txt` 文件开头：

```text
User-agent: *
Disallow: /search
Allow: /search/about
Allow: /search/static
Allow: /search/howsearchworks
Disallow: /sdch
...
```

[User-Agent](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/User-Agent) 指令指向特定的 Web 蜘蛛/机器人/爬虫。例如，`User-Agent: Googlebot` 指来自 Google 的蜘蛛，而 `User-Agent: bingbot` 指来自 Microsoft 的爬虫。上面示例中的 `User-Agent: *` 适用于所有 [Web 蜘蛛/机器人/爬虫](https://support.google.com/webmasters/answer/6062608?visit_id=637173940975499736-3548411022&rd=1)。

`Disallow` 指令指定禁止蜘蛛/机器人/爬虫访问的资源。在上面的示例中，以下资源被禁止：

```text
...
Disallow: /search
...
Disallow: /sdch
...
```

Web 蜘蛛/机器人/爬虫可能[故意忽略](https://blog.isc2.org/isc2_blog/2008/07/the-attack-of-t.html) `robots.txt` 文件中指定的 `Disallow` 指令。因此，不应将 `robots.txt` 视为强制限制第三方如何访问、存储或重新发布 Web 内容的机制。

`robots.txt` 文件从 Web 服务器的 Web 根目录获取。例如，使用 `wget` 或 `curl` 从 `www.google.com` 获取 `robots.txt`：

```bash
$ curl -O -Ss https://www.google.com/robots.txt && head -n5 robots.txt
User-agent: *
Disallow: /search
Allow: /search/about
Allow: /search/static
Allow: /search/howsearchworks
...
```

#### 使用 Google Webmaster Tools 分析 robots.txt

网站所有者可以使用 Google 的“分析 robots.txt”功能，作为其 [Google Webmaster Tools](https://www.google.com/webmasters/tools) 的一部分来分析网站。该工具可以帮助测试，步骤如下：

1. 使用 Google 帐户登录 Google Webmaster Tools。
2. 在仪表板上，输入要分析的网站的 URL。
3. 选择可用的方法并按照屏幕上的说明操作。

### META 标签

`<META>` 标签位于每个 HTML 文档的 `HEAD` 部分内，如果机器人/蜘蛛/爬虫的起始点不是从 Web 根目录以外的文档链接（即[深度链接](https://en.wikipedia.org/wiki/Deep_linking)），则整个站点应保持一致。机器人指令也可以使用特定的 [META 标签](https://www.robotstxt.org/meta.html)指定。

#### Robots META 标签

如果没有 `<META NAME="ROBOTS" ... >` 条目，则“机器人排除协议”默认分别为 `INDEX,FOLLOW`。因此，该协议定义的其他两个有效条目以 `NO...` 为前缀，即 `NOINDEX` 和 `NOFOLLOW`。

基于 Web 根目录中 `robots.txt` 文件列出的 Disallow 指令，在每个网页中进行 `<META NAME="ROBOTS"` 的正则表达式搜索。然后将结果与 Web 根目录中的 `robots.txt` 文件进行比较。

#### 其他 META 信息标签

组织通常会在 Web 内容中嵌入信息性 META 标签，以支持屏幕阅读器、社交网络预览、搜索引擎索引等各种技术。这些元信息对于测试人员识别所使用的技术、探索和测试额外的路径/功能具有价值。以下元信息于 2020 年 5 月 5 日通过查看页面源代码从 `www.whitehouse.gov` 获取：

### Sitemaps

站点地图（sitemap）是一个文件，开发人员或组织可以在其中提供有关站点或应用程序提供的页面、视频和其他文件及其之间关系的信息。搜索引擎可以使用此文件更有效地导航您的站点。同样，测试人员可以利用 `sitemap.xml` 文件更深入地了解所调查的站点或应用程序。

以下摘录自 2020 年 5 月 5 日获取的 Google 主站点地图。

```bash
$ wget --no-verbose https://www.google.com/sitemap.xml && head -n8 sitemap.xml
2020-05-05 12:23:30 URL:https://www.google.com/sitemap.xml [2049] -> "sitemap.xml" [1]

<?xml version="1.0" encoding="UTF-8"?>
<sitemapindex xmlns="https://www.google.com/schemas/sitemap/0.84">
  <sitemap>
    <loc>https://www.google.com/gmail/sitemap.xml</loc>
  </sitemap>
  <sitemap>
    <loc>https://www.google.com/forms/sitemaps.xml</loc>
  </sitemap>
...
```

从那里开始探索，测试人员可能希望获取 gmail 站点地图 `https://www.google.com/gmail/sitemap.xml`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="https://www.sitemaps.org/schemas/sitemap/0.9" xmlns:xhtml="https://www.w3.org/1999/xhtml">
  <url>
    <loc>https://www.google.com/intl/am/gmail/about/</loc>
    <xhtml:link href="https://www.google.com/gmail/about/" hreflang="x-default" rel="alternate"/>
    <xhtml:link href="https://www.google.com/intl/el/gmail/about/" hreflang="el" rel="alternate"/>
    <xhtml:link href="https://www.google.com/intl/it/gmail/about/" hreflang="it" rel="alternate"/>
    <xhtml:link href="https://www.google.com/intl/ar/gmail/about/" hreflang="ar" rel="alternate"/>
...
```

### Security TXT

[security.txt](https://securitytxt.org) 已被 IETF 批准为 [RFC 9116 - 帮助安全漏洞披露的文件格式](https://www.rfc-editor.org/rfc/rfc9116.html)，允许站点定义安全策略和联系信息。在测试场景中，这可能引起兴趣的原因有很多，包括但不限于：

- 识别要纳入发现/分析的更多路径或资源。
- 开源情报收集。
- 查找有关漏洞赏金计划的信息等。
- 社会工程学。

该文件可能存在于 Web 服务器的根目录或 `.well-known/` 目录中，例如：

- `https://example.com/security.txt`
- `https://example.com/.well-known/security.txt`

以下是从 LinkedIn 于 2020 年 5 月 5 日获取的真实示例：

```bash
$ wget --no-verbose https://www.linkedin.com/.well-known/security.txt && cat security.txt
2020-05-07 12:56:51 URL:https://www.linkedin.com/.well-known/security.txt [333/333] -> "security.txt" [1]
# Conforms to IETF `draft-foudil-securitytxt-07`
Contact: mailto:security@linkedin.com
Contact: https://www.linkedin.com/help/linkedin/answer/62924
Encryption: https://www.linkedin.com/help/linkedin/answer/79676
Canonical: https://www.linkedin.com/.well-known/security.txt
Policy: https://www.linkedin.com/help/linkedin/answer/62924
```

OpenPGP 公钥包含一些可以提供有关密钥本身信息的元数据。以下是可以从 OpenPGP 公钥中提取的一些常见元数据元素：

- **密钥 ID**：密钥 ID 是从公钥材料派生的短标识符。它有助于识别密钥，通常显示为八个字符的十六进制值。
- **密钥指纹**：密钥指纹是从密钥材料派生的更长且更唯一的标识符。通常显示为 40 个字符的十六进制值。密钥指纹通常用于验证公钥的完整性和真实性。
- **密钥算法**：密钥算法表示公钥使用的加密算法。OpenPGP 支持各种算法，如 RSA、DSA 和 ECC（椭圆曲线密码学）。
- **密钥大小**：密钥大小指加密密钥的位长度或大小。它表示密钥的强度，并决定密钥提供的安全级别。
- **密钥创建日期**：密钥创建日期指示密钥生成或创建的时间。
- **密钥过期日期**：OpenPGP 公钥可以设置过期日期，之后密钥被视为无效。密钥过期日期指定密钥不再有效的时间。
- **用户 ID**：公钥可以有一个或多个关联的用户 ID，用于标识与密钥关联的所有者或实体。用户 ID 通常包括诸如姓名、电子邮件地址和可选注释等信息。

### Humans TXT

`humans.txt` 是一项了解网站背后人员的倡议。它采用文本文件的形式，包含有关为构建网站做出贡献的不同人员的信息。该文件通常（但不总是）包含与职业或招聘网站/路径相关的信息。

以下示例于 2020 年 5 月 5 日从 Google 获取：

```bash
$ wget --no-verbose  https://www.google.com/humans.txt && cat humans.txt
2020-05-07 12:57:52 URL:https://www.google.com/humans.txt [286/286] -> "humans.txt" [1]
Google is built by a large team of engineers, designers, researchers, robots, and others in many different sites across the globe. It is updated continuously, and built with more tools and technologies than we can shake a stick at. If you'd like to help us out, see careers.google.com.
```

### 其他 .well-known 信息源

还有其他 RFC 和互联网草案建议在 `.well-known/` 目录中标准化使用文件。这些列表可以在 [Wikipedia 上](https://en.wikipedia.org/wiki/List_of_/.well-known/_services_offered_by_webservers)或[通过 IANA 此处](https://www.iana.org/assignments/well-known-uris/well-known-uris.xhtml)找到。

测试人员可以很容易地查看 RFC/草案并创建一个列表提供给爬虫或模糊测试工具，以验证此类文件的存在或内容。
