# 识别应用程序入口点

|ID          |
|------------|
|WSTG-INFO-06|

## 概要

在进行任何深入测试之前，枚举应用程序及其攻击面是一个关键的前置步骤，因为它允许测试人员识别可能的薄弱区域。本节旨在帮助识别和映射应用程序中在枚举和映射完成后应进行调查的区域。

## 测试目标

- 通过请求和响应分析识别可能的入口点和注入点。

## 如何测试

在任何测试开始之前，测试人员应始终充分了解应用程序以及用户和浏览器如何与其通信。当测试人员遍历应用程序时，他们应注意所有 HTTP 请求以及传递给应用程序的每个参数和表单字段。他们应特别注意何时使用 GET 请求以及何时使用 POST 请求将参数传递给应用程序。此外，他们还需要注意 RESTful 服务何时使用其他方法。

请注意，为了查看 POST 请求等请求主体中发送的参数，测试人员可能希望使用诸如拦截代理之类的工具（请参阅[工具](#tools)）。在 POST 请求中，测试人员还应特别注意传递给应用程序的任何隐藏表单字段，因为这些字段通常包含敏感信息，如状态信息、商品数量、商品价格等，开发人员从未打算让任何人看到或更改这些信息。

对于此测试阶段，使用拦截代理和笔记应用程序（例如电子表格软件）在测试人员中很流行。代理将记录测试人员在探索过程中与应用程序之间的每个请求和响应。此外，此时测试人员通常会拦截每个请求和响应，以便他们能够准确看到传递给应用程序的每个头部、参数等以及返回的内容。有时这可能相当繁琐，尤其是在大型交互式站点上（例如银行应用程序）。然而，经验会告诉你要寻找什么，这个阶段可以显著优化。

当测试人员遍历应用程序时，他们应注意 URL、自定义头部或请求/响应主体中的任何有趣参数，并将其保存在电子表格中。电子表格应包括请求的页面（最好还添加代理中的请求编号以供将来参考）、有趣的参数、请求类型（GET、POST 等）、访问是否需要身份验证、是否使用 TLS、是否是多步骤流程的一部分、是否使用 WebSocket 以及任何其他相关注释。一旦他们映射了应用程序的每个区域，他们就可以遍历应用程序并测试他们已识别的每个区域，并记录哪些有效、哪些无效。本指南的其余部分将介绍如何测试每个感兴趣的区域，但必须在任何实际测试开始之前进行本节。

以下是所有请求和响应的一些关注点。在请求部分，重点关注 GET 和 POST 方法，因为它们构成了大部分请求。请注意，也可以找到其他方法，如 PUT 和 DELETE。通常，如果允许此类较罕见的请求，它们可能会暴露漏洞。本指南中有一个专门用于测试这些 HTTP 方法的特殊章节。

### 请求

- 识别哪里使用了 GET 以及哪里使用了 POST。
- 识别 POST 请求中使用的所有参数（这些参数在请求主体中）。
- 在 POST 请求中，特别注意任何隐藏参数。当发送 POST 请求时，所有表单字段（包括隐藏参数）都将通过 HTTP 消息主体发送到应用程序。除非使用代理或查看 HTML 源代码，否则通常看不到这些参数。更改这些隐藏参数可能会导致后续加载的页面、它们包含的数据以及授予的访问权限程度发生变化。
- 识别 GET 请求中使用的所有参数（即在 URL 中），特别是查询字符串（通常出现在 ? 标记之后）。
- 识别查询字符串的所有参数。这些通常采用成对格式，例如 `foo=bar`。还要注意，许多参数可以包含在一个查询字符串中，用 `&`、`\~`、`:` 或任何其他特殊字符或编码分隔。
- 请注意，在识别一个字符串中或 POST 请求内的多个参数时，执行攻击可能需要部分或全部参数。测试人员需要识别所有参数（即使它们是编码或加密的），并确定哪些参数被应用程序处理。本指南的后续部分将介绍如何测试这些参数。此时，确保识别出每一个参数非常重要。
- 还要注意任何不常见的附加或自定义类型头部（例如 `debug: false`）。

### 响应

- 识别哪里设置了新 cookie（`Set-Cookie` 头部）、修改或添加了 cookie。
- 识别正常响应（即未修改的请求）中的任何重定向（3xx HTTP 状态码）、400 状态码（特别是 403 Forbidden）和 500 内部服务器错误。
- 还要注意使用了哪些有趣的头部。例如，`Server: BIG-IP` 表示站点已负载均衡。因此，如果一个站点已负载均衡且一台服务器配置错误，则测试人员可能需要根据所使用的负载均衡类型发送多个请求以访问易受攻击的服务器。

### OWASP 攻击面检测器

攻击面检测器（ASD）工具可以研究源代码并发现 Web 应用程序的端点、这些端点接受的参数以及这些参数的数据类型。这包括爬虫无法找到的未链接端点，以及客户端代码中完全未使用的可选参数。它还能够计算应用程序两个版本之间攻击面的变化。

攻击面检测器可作为 ZAP 和 Burp Suite 的插件使用，同时还提供命令行工具。命令行工具将攻击面导出为 JSON 输出，然后可由 ZAP 和 Burp Suite 插件使用。这对于未直接向渗透测试人员提供源代码的情况很有帮助。例如，渗透测试人员可以从不希望提供源代码本身的客户那里获取 JSON 输出文件。

#### 如何使用

CLI jar 文件可从 [https://github.com/secdec/attack-surface-detector-cli/releases](https://github.com/secdec/attack-surface-detector-cli/releases) 下载。

您可以运行以下命令让 ASD 从目标 Web 应用程序的源代码中识别端点。

`java -jar attack-surface-detector-cli-1.3.5.jar <source-code-path> [flags]`

### 测试应用程序入口点

以下是两个检查应用程序入口点的示例。

#### 示例 1

此示例显示了一个从在线购物应用程序购买商品的 GET 请求。

```http
GET /shoppingApp/buyme.asp?CUSTOMERID=100&ITEM=z101a&PRICE=62.50&IP=x.x.x.x HTTP/1.1
Host: x.x.x.x
Cookie: SESSIONID=Z29vZCBqb2IgcGFkYXdhIG15IHVzZXJuYW1lIGlzIGZvbyBhbmQgcGFzc3dvcmQgaXMgYmFy
```

> 请求的所有参数，如 CUSTOMERID、ITEM、PRICE、IP 和 Cookie，它们可能只是编码参数或用于会话状态的参数。

#### 示例 2

此示例显示了一个用于登录应用程序的 POST 请求。

```http
POST /example/authenticate.asp?service=login HTTP/1.1
Host: x.x.x.x
Cookie: SESSIONID=dGhpcyBpcyBhIGJhZCBhcHAgdGhhdCBzZXRzIHByZWRpY3RhYmxlIGNvb2tpZXMgYW5kIG1pbmUgaXMgMTIzNA==;CustomCookie=00my00trusted00ip00is00x.x.x.x00

user=admin&pass=pass123&debug=true&fromtrustIP=true
```

可以注意到参数在多个位置发送：

1. 在查询字符串中：`service`
2. 在 Cookie 头部中：`SESSIONID`、`CustomCookie`
3. 在请求主体中：`user`、`pass`、`debug`、`fromtrustIP`

拥有多种注入位置为攻击者提供了链式攻击的可能性，可以增加在处理代码中发现错误的机会。
