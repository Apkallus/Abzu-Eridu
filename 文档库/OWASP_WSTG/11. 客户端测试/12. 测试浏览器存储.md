# 测试浏览器存储

|ID          |
|------------|
|WSTG-CLNT-12|

## 概述

浏览器为开发者提供以下客户端存储机制，用于存储和检索数据：

- LocalStorage
- SessionStorage
- IndexedDB
- Web SQL（已弃用）
- Cookies

这些存储机制可以使用浏览器的开发者工具进行查看和编辑，例如 [Google Chrome DevTools](https://developers.google.com/web/tools/chrome-devtools/storage/localstorage) 或 [Firefox 的存储检查器](https://developer.mozilla.org/en-US/docs/Tools/Storage_Inspector)。

注意：虽然缓存也是一种存储形式，但在[单独章节](../04-Authentication_Testing/06-Testing_for_Browser_Cache_Weaknesses.md)中涵盖了其自身的特性和注意事项。

## 测试目标

- 确定站点是否在客户端存储中保存了敏感数据。
- 应检查存储对象处理代码是否存在注入攻击的可能性，例如使用未经验证的输入或有漏洞的库。

## 如何测试

### LocalStorage

`window.localStorage` 是一个全局属性，实现了 [Web Storage API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Storage_API)，并在浏览器中提供**持久性**键值存储。

键和值都只能是字符串，因此任何非字符串值在存储之前必须先转换为字符串，通常通过 [JSON.stringify](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/JSON/stringify) 完成。

即使在浏览器窗口关闭后，`localStorage` 中的条目仍然持久存在，但隐私/无痕模式下的窗口除外。

`localStorage` 的最大存储容量因浏览器而异。

#### 列出所有键值对

```javascript
for (let i = 0; i < localStorage.length; i++) {
  const key = localStorage.key(i);
  const value = localStorage.getItem(key);
  console.log(`${key}: ${value}`);
}
```

### SessionStorage

`window.sessionStorage` 是一个全局属性，实现了 [Web Storage API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Storage_API)，并在浏览器中提供**临时性**键值存储。

键和值都只能是字符串，因此任何非字符串值在存储之前必须先转换为字符串，通常通过 [JSON.stringify](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/JSON/stringify) 完成。

`sessionStorage` 中的条目是临时性的，因为当浏览器标签页/窗口关闭时它们会被清除。

`sessionStorage` 的最大存储容量因浏览器而异。

#### 列出所有键值对

```javascript
for (let i = 0; i < sessionStorage.length; i++) {
  const key = sessionStorage.key(i);
  const value = sessionStorage.getItem(key);
  console.log(`${key}: ${value}`);
}
```

### IndexedDB

IndexedDB 是一个事务型、面向对象的数据库，用于存储结构化数据。一个 IndexedDB 数据库可以有多个对象存储，每个对象存储可以有多个对象。

与 localStorage 和 sessionStorage 不同，IndexedDB 可以存储的不仅仅是字符串。[结构化克隆算法](https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API/Structured_clone_algorithm)支持的任何对象都可以存储在 IndexedDB 中。

一个可以存储在 IndexedDB 中但不能存储在 localStorage/sessionStorage 中的复杂 JavaScript 对象示例是 [CryptoKeys](https://developer.mozilla.org/en-US/docs/Web/API/CryptoKey)。

W3C 关于 [Web Crypto API](https://www.w3.org/TR/WebCryptoAPI/) 的[建议](https://www.w3.org/TR/WebCryptoAPI/#concepts-key-storage)指出，需要在浏览器中持久保存的 CryptoKeys 应存储在 IndexedDB 中。在测试网页时，查找 IndexedDB 中的任何 CryptoKeys，并检查当它们应设置为 `extractable: false` 时，是否被设置为 `extractable: true`（即确保底层私钥材料在加密操作期间永远不会暴露）。

#### 打印 IndexedDB 的所有内容

```javascript
const dumpIndexedDB = dbName => {
  const DB_VERSION = 1;
  const req = indexedDB.open(dbName, DB_VERSION);
  req.onsuccess = function() {
    const db = req.result;
    const objectStoreNames = db.objectStoreNames || [];

    console.log(`[*] Database: ${dbName}`);

    Array.from(objectStoreNames).forEach(storeName => {
      const txn = db.transaction(storeName, 'readonly');
      const objectStore = txn.objectStore(storeName);

      console.log(`\t[+] ObjectStore: ${storeName}`);

      // 打印名称为 `storeName` 的对象存储中的所有条目
      objectStore.getAll().onsuccess = event => {
        const items = event.target.result || [];
        items.forEach(item => console.log(`\t\t[-] `, item));
      };
    });
  };
};

indexedDB.databases().then(dbs => dbs.forEach(db => dumpIndexedDB(db.name)));
```

### Web SQL

Web SQL 自 2010 年 11 月 18 日起已被弃用，建议 Web 开发者不要使用它。

### Cookies

Cookies 是一种键值存储机制，主要用于会话管理，但 Web 开发者仍然可以使用它来存储任意字符串数据。

Cookies 在[测试 Cookie 属性](../06-Session_Management_Testing/02-Testing_for_Cookies_Attributes.md)场景中有广泛涵盖。

#### 列出所有 Cookies

```javascript
console.log(window.document.cookie);
```

### 全局 Window 对象

有时 Web 开发者会通过向全局 `window` 对象分配自定义属性来初始化和维护仅在页面运行时生命周期内可用的全局状态。例如：

```javascript
window.MY_STATE = {
  counter: 0,
  flag: false,
};
```

当页面刷新或关闭时，附加在 `window` 对象上的任何数据都会丢失。

#### 列出 Window 对象上的所有条目

```javascript
(() => {
  // 创建一个 iframe 并追加到 body 以加载一个干净的 window 对象
  const iframe = document.createElement('iframe');
  iframe.style.display = 'none';
  document.body.appendChild(iframe);

  // 获取 window 上当前属性的列表
  const currentWindow = Object.getOwnPropertyNames(window);

  // 过滤列表，排除存在于干净 window 中的属性
  const results = currentWindow.filter(
    prop => !iframe.contentWindow.hasOwnProperty(prop)
  );

  // 移除 iframe
  document.body.removeChild(iframe);

  // 记录不同的键值条目
  results.forEach(key => console.log(`${key}: ${window[key]}`));
})();
```

### 安全影响

在审查浏览器存储机制时，测试人员应评估敏感数据是否不必要地暴露在客户端。现代应用程序，尤其是 SPA，经常在浏览器存储中存储身份验证令牌或应用程序状态，这可能引入安全风险。

常见关注点包括：

- 存储在 `localStorage` 或 `sessionStorage` 中的身份验证令牌（例如 JWT），可通过 JavaScript 访问，并可能通过 XSS 暴露。
- 注销后仍然存在的令牌或会话标识符。
- 存储在 IndexedDB 或 localStorage 中的敏感业务数据，但没有明确的需求。
- 本应受保护的加密材料被存储为可提取状态。

不当的客户端存储可能加剧客户端攻击（如基于 DOM 的 XSS）的影响。

### 通用测试指南

除了枚举存储条目外，测试人员还应：

- 使用开发者工具（Application/Storage 选项卡）检查浏览器存储。
- 识别身份验证令牌、会话标识符或敏感业务数据。
- 尝试通过 JavaScript 控制台访问存储的值。
- 验证注销或会话过期后存储条目是否被清除。
- 评估存储的数据是否可在客户端攻击链中被利用。

### 攻击链

在识别出上述任何攻击向量后，可以与不同类型的客户端攻击（如[基于 DOM 的 XSS](01-Testing_for_DOM-based_Cross_Site_Scripting.md)攻击）形成攻击链。

## 修复

应用程序应以安全的方式遵循最佳实践，将敏感数据存储在服务器端，而不是客户端。
