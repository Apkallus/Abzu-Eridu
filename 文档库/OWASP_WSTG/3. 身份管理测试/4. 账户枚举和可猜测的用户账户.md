# 测试账户枚举和可猜测的用户账户

|ID          |
|------------|
|WSTG-IDNT-04|

## 概述

本测试的范围是验证是否可以通过与应用程序的身份验证机制交互来收集一组有效的用户名。此测试将对暴力破解测试有用，在暴力破解测试中，测试人员验证在给定有效用户名的情况下，是否可以找到对应的密码。

通常，Web 应用程序会因配置错误或设计决策而在系统上揭示用户名是否存在。例如，有时当我们提交错误的凭据时，会收到一条消息，指出用户名在系统中存在或提供的密码错误。获取的信息可被攻击者用来获取系统上的用户列表。此信息可用于攻击 Web 应用程序，例如通过暴力破解或默认用户名和密码攻击。

测试人员应与应用程序的身份验证机制交互，以了解发送特定请求是否会导致应用程序以不同方式响应。此问题存在的原因是，当用户提供有效用户名时，Web 应用程序或 Web 服务器释放的信息与提供无效用户名时不同。

在某些情况下，会收到一条消息，揭示提供的凭据错误是因为使用了无效用户名或无效密码。有时，测试人员可以通过发送用户名和空密码来枚举现有用户。

## 测试目标

- 审查与用户标识相关的流程（例如注册、登录等）。
- 在可能的情况下，通过响应分析枚举用户。

## 如何测试

在黑盒测试中，测试人员对特定应用程序、用户名、应用程序逻辑、登录页面上的错误消息或密码恢复设施一无所知。如果应用程序存在漏洞，测试人员会收到一条响应消息，直接或间接地揭示一些有助于枚举用户的信息。

### HTTP 响应消息

#### 测试有效凭据

记录提交有效用户 ID 和有效密码时服务器的应答。

> 使用 Web 代理，注意从此成功身份验证中检索到的信息（HTTP 200 响应、响应长度）。

#### 测试有效用户与错误密码

现在，测试人员应尝试插入有效用户 ID 和错误密码，并记录应用程序生成的错误消息。

> 浏览器应显示类似于以下内容的消息：
>
> ![身份验证失败](images/AuthenticationFailed.png)\
> *图 4.3.4-1：身份验证失败*
>
> 不同于任何揭示用户存在的消息，例如：
>
> `用户 foo 的登录：密码无效`
>
> 使用 Web 代理，注意从此失败身份验证尝试中检索到的信息（HTTP 200 响应、响应长度）。

#### 测试不存在的用户名

现在，测试人员应尝试插入无效的用户 ID 和错误的密码，并记录服务器的应答（测试人员应确信该用户名在应用程序中无效）。记录错误消息和服务器应答。

> 如果测试人员输入不存在的用户 ID，他们可能会收到类似以下的消息：
>
> ![此用户未激活](images/Userisnotactive.png)\
> *图 4.3.4-3：此用户未激活*
>
> 或类似以下的消息：
>
> `用户 foo 登录失败：无效账户`
>
> 通常，应用程序应对不同的错误请求响应相同的错误消息和长度。如果响应不相同，测试人员应调查并找出两个响应之间产生差异的关键点。例如：
>
> 1. 客户端请求：有效用户/错误密码
> 2. 服务器响应：密码不正确
> 3. 客户端请求：错误用户/错误密码
> 4. 服务器响应：用户无法识别
>
> 上述响应让客户端知道对于第一个请求他们拥有有效用户名。因此，他们可以与应用程序交互，请求一组可能的用户 ID 并观察应答。
>
> 查看第二个服务器响应，测试人员以相同方式了解他们不持有有效用户名。因此，他们可以以相同方式交互，并通过查看服务器应答创建有效用户 ID 列表。

### 枚举用户的其他方式

测试人员可以通过多种方式枚举用户，例如：

#### 分析登录页面收到的错误代码

某些 Web 应用程序会释放特定的错误代码或消息，我们可以对其进行分析。

#### 分析 URL 和 URL 重定向

例如：

- `https://www.foo.com/err.jsp?User=baduser&Error=0`
- `https://www.foo.com/err.jsp?User=gooduser&Error=2`

如上所示，当测试人员向 Web 应用程序提供用户 ID 和密码时，他们会在 URL 中看到指示错误已发生的消息。在第一种情况下，他们提供了错误的用户 ID 和错误的密码。在第二种情况下，提供了正确的用户 ID 和错误的密码，因此他们可以识别有效的用户 ID。

#### URI 探测

有时，Web 服务器在收到对现有目录或不存在的目录的请求时响应不同。例如，在某些门户中，每个用户都与一个目录相关联。如果测试人员尝试访问现有目录，他们可能会收到 Web 服务器错误。

Web 服务器返回的一些常见错误包括：

- 403 禁止访问错误代码
- 404 未找到错误代码

示例：

- `https://www.foo.com/account1` - 我们从 Web 服务器收到：403 禁止访问
- `https://www.foo.com/account2` - 我们从 Web 服务器收到：404 文件未找到

在第一种情况下，用户存在，但测试人员无法查看网页；在第二种情况下，用户 "account2" 不存在。通过收集这些信息，测试人员可以枚举用户。

#### 分析网页标题

测试人员可以在网页标题上获得有用信息，他们可以在其中获得特定的错误代码或消息，揭示问题是出在用户名还是密码上。

例如，如果用户无法通过应用程序身份验证并收到标题类似于以下的网页：

- `无效用户`
- `无效身份验证`

#### 分析从恢复设施收到的消息

当我们使用恢复设施（例如忘记密码功能）时，存在漏洞的应用程序可能会返回一条消息，揭示用户名是否存在。

例如，类似于以下的消息：

- `无效用户名：电子邮件地址无效或未找到指定用户。`
- `有效用户名：您的密码已成功发送到您注册的电子邮件地址。`

#### 友好的 404 错误消息

当我们请求目录中不存在的用户时，并不总是收到 404 错误代码。相反，我们可能收到带有图片的 "200 OK"，在这种情况下，我们可以假设当收到特定图片时，该用户不存在。此逻辑可应用于其他 Web 服务器响应；关键在于对 Web 服务器和 Web 应用程序消息进行良好分析。

#### 分析响应时间

除了查看响应内容外，还应考虑响应所花费的时间。特别是当请求涉及与外部服务交互（例如发送忘记密码电子邮件）时，这可能会为响应增加几百毫秒的时间，可用于确定所请求的用户是否有效。

### 猜测用户

#### 可预测的用户名结构

在许多组织中，用户名遵循一致的模式（例如，名字首字母加姓氏，如 jbloggs，或结构化标识符，如 CN000100）。如果可以识别命名约定，通常可以系统地派生有效的用户名。

测试人员应：

- 识别用户名是否遵循可预测的结构。
- 尝试根据观察到的模式生成其他用户名。
- 使用从组织数据（例如员工姓名、电子邮件格式）派生的用户名字典。
- 观察应用程序响应以确定生成的用户名是否有效。

可预测的用户名结构显著减少了枚举有效账户所需的工作量，并可促进进一步的攻击，如暴力破解尝试。

在某些情况下，用户 ID 是根据管理员或公司的特定策略创建的。例如，我们可以看到按顺序创建的用户 ID：

```text
CN000100
CN000101
...
```

有时用户名使用 REALM 别名加序号创建：

- R1001 – REALM1 的用户 001
- R2001 – REALM2 的用户 001

在上述示例中，我们可以创建简单的 shell 脚本来组合用户 ID，并使用 wget 等工具提交请求，以自动化 Web 查询来识别有效的用户 ID。要创建脚本，我们还可以使用 Perl 和 cURL。

其他可能性包括：
- 与信用卡号或一般具有模式的数字相关联的用户 ID。
- 与真实姓名相关联的用户 ID，例如，如果 Freddie Mercury 的用户 ID 为 "fmercury"，那么你可以猜测 Roger Taylor 的用户 ID 为 "rtaylor"。

同样，我们可以从 LDAP 查询或 Google 信息收集（例如来自特定域）中获取的信息来猜测用户名。Google 可以通过特定查询或简单的 shell 脚本或工具帮助查找域用户。

> 通过枚举用户账户，您可能在达到预定义的失败探测次数后锁定账户（基于应用程序策略）。此外，有时您的 IP 地址可能会被应用程序防火墙或入侵防御系统的动态规则禁止。

### 测试员工冒充

确保未注册用户在注册过程中无法选择保留的用户名（例如 admin、administrator、moderator）。此外，验证用户在个人资料编辑页面上无法将其当前用户名更改为这些保留用户名之一。

如果 Web 应用程序具有允许用户访问 Web 应用程序注册和个人资料编辑功能的功能，则要测试的交互包括：

- 注册流程：
    - 以未注册用户身份访问注册页面，填写注册表单，输入一个保留用户名（例如 admin、administrator、moderator），提交注册表单，然后验证响应。
    - 注册过程应拒绝表单提交，并显示错误消息，指示所选用户名不可用于注册。
- 个人资料编辑页面：
    - 使用有效凭据登录 Web 应用程序，导航到个人资料编辑页面。尝试将当前用户名更改为其中一个保留用户名（例如 admin、administrator、moderator），并保存更改以验证行为。
    - 个人资料编辑过程应拒绝用户名更改请求，并显示错误消息，指示所选用户名不可用。
- 测试变体和相似性：
    - 对保留用户名的不同变体（例如 Admin、ADMIN、Administrator）重复上述步骤，并使用不同的大小写组合进行测试，以确保正确处理大小写不敏感。
    - Web 应用程序应将此类变体视为与保留用户名相同，拒绝其选择或修改。

### 灰盒测试

#### 测试身份验证错误消息

验证应用程序对每个导致身份验证失败的客户端请求是否以相同方式响应。对于此问题，黑盒测试和灰盒测试基于对从 Web 应用程序收到的消息或错误代码的分析具有相同的概念。

> 应用程序应对每次失败的身份验证尝试以相同方式响应。
>
> 例如：*提交的凭据无效*

## 修复

确保应用程序在登录过程中输入无效账户名称、密码或其他用户凭据时返回一致的通用错误消息。

确保在将系统发布到生产环境（或暴露给不受信任的网络）之前删除默认系统账户和测试账户。
