# 测试会话管理模式

|ID          |
|------------|
|WSTG-SESS-01|

## 概述

任何基于 Web 的应用程序的核心组件之一，是其控制并维护与用户交互状态的机制。为了避免对站点的每个页面进行持续认证，Web 应用程序实现了各种机制，在预定的时间范围内存储和验证凭据。这些机制称为会话管理。

在本测试中，测试人员希望检查 cookie 和其他会话令牌是否以安全且不可预测的方式创建。能够预测和伪造弱 cookie 的攻击者可以轻易劫持合法用户的会话。

Cookie 用于实现会话管理，并在 RFC 2965 中有详细描述。简而言之，当用户访问一个需要跨多个请求跟踪该用户操作和身份的应用程序时，服务器会生成一个 cookie（或多个 cookie）并发送给客户端。随后，客户端将在所有后续连接中将 cookie 发送回服务器，直到 cookie 过期或被销毁。Cookie 中存储的数据可以向服务器提供关于用户是谁、迄今为止执行了哪些操作、偏好是什么等广泛信息，从而为 HTTP 这样的无状态协议提供状态。

一个典型的例子是在线购物车。在用户的整个会话期间，应用程序必须跟踪其身份、个人资料、选择购买的产品、数量、单价、折扣等。Cookie 是高效存储和来回传递这些信息的方式（其他方式包括 URL 参数和隐藏字段）。

由于 cookie 存储的数据至关重要，它们对应用程序的整体安全性至关重要。篡改 cookie 可能导致劫持合法用户的会话、在活动会话中获得更高权限，并以未经授权的方式影响应用程序的操作。

在本测试中，测试人员需要检查发放给客户端的 cookie 是否能够抵御一系列旨在干扰合法用户会话和应用程序本身的攻击。总体目标是能够伪造一个被应用程序视为有效的 cookie，并提供某种未经授权的访问（会话劫持、权限提升等）。

通常攻击模式的主要步骤如下：

- **cookie 收集**：收集足够数量的 cookie 样本；
- **cookie 逆向工程**：分析 cookie 生成算法；
- **cookie 操纵**：伪造有效 cookie 以执行攻击。最后一步可能需要大量尝试，具体取决于 cookie 的创建方式（cookie 暴力破解攻击）。

另一种攻击模式是溢出 cookie。严格来说，这种攻击具有不同的性质，因为测试人员不是试图重新创建一个完全有效的 cookie。相反，目标是溢出内存区域，从而干扰应用程序的正确行为，并可能注入（并远程执行）恶意代码。

## 测试目标

- 收集会话令牌，尽可能针对同一用户和不同用户。
- 分析并确保存在足够的随机性，以防止会话伪造攻击。
- 修改未签名且包含可被操纵信息的 cookie。

## 如何测试

### 黑盒测试与示例

客户端与应用程序之间的所有交互应至少根据以下标准进行测试：

- 所有 `Set-Cookie` 指令是否都标记为 `Secure`？
- 是否有任何 Cookie 操作通过未加密的传输进行？
- Cookie 是否可以被强制通过未加密的传输？
- 如果是，应用程序如何维护安全性？
- 是否有任何 Cookie 是持久性的？
- 持久性 Cookie 使用了什么 `Expires` 时间，这些时间是否合理？
- 预期为临时性的 Cookie 是否如此配置？
- 使用了哪些 HTTP/1.1 `Cache-Control` 设置来保护 Cookie？
- 使用了哪些 HTTP/1.0 `Cache-Control` 设置来保护 Cookie？

#### Cookie 收集

操作 cookie 的第一步是了解应用程序如何创建和管理 cookie。为此，测试人员应尝试回答以下问题：

- 应用程序使用了多少个 cookie？

  浏览应用程序。注意 cookie 何时被创建。列出接收到的 cookie、设置它们的页面（带有 set-cookie 指令）、它们有效的域、它们的值及其特征。

- 应用程序的哪些部分生成或修改 cookie？

  浏览应用程序，找出哪些 cookie 保持不变，哪些被修改。哪些事件修改了 cookie？

- 应用程序的哪些部分需要此 cookie 才能访问和使用？

  找出应用程序的哪些部分需要 cookie。访问一个页面，然后在不使用 cookie 或使用修改后的值的情况下再次尝试。尝试映射哪些 cookie 在何处使用。

将每个 cookie 映射到相应的应用程序部分及相关信息的电子表格，可以作为此阶段有价值的输出。

#### 会话分析

会话令牌（Cookie、SessionID 或隐藏字段）本身应被检查，以确保其从安全角度具备足够的质量。应针对随机性、唯一性、对统计和密码分析的抵抗力以及信息泄露等标准进行测试。

- 令牌结构与信息泄露

第一阶段是检查应用程序提供的会话 ID 的结构和内容。一个常见错误是在令牌中包含特定数据，而不是颁发通用值并在服务器端引用实际数据。

如果会话 ID 是明文，结构和相关数据可能立即显而易见，例如 `192.168.100.1:owaspuser:password:15:58`。

如果部分或全部令牌看起来被编码或哈希，应将其与各种技术进行比较，以检查是否存在明显的混淆。例如，字符串 `192.168.100.1:owaspuser:password:15:58` 的十六进制、Base64 和 MD5 哈希表示分别为：

- 十六进制：`3139322E3136382E3130302E313A6F77617370757365723A70617373776F72643A31353A3538`
- Base64：`MTkyLjE2OC4xMDAuMTpvd2FzcHVzZXI6cGFzc3dvcmQ6MTU6NTg=`
- MD5：`01c2fc4f0a817afd8366689bd29dd40a`

确定了混淆类型后，可能可以解码回原始数据。然而，在大多数情况下，这不太可能。即便如此，从消息格式中枚举存在的编码也可能是有用的。此外，如果既能推断出格式又能推断出混淆技术，就可以设计自动化暴力破解攻击。

混合令牌可能包括诸如 IP 地址或用户 ID 等信息，以及编码部分，例如 `owaspuser:192.168.100.1:a7656fafe94dae72b1e1487670148412`。

分析单个会话令牌后，应检查代表性样本。对令牌的简单分析应立即揭示任何明显的模式。例如，一个 32 位令牌可能包含 16 位静态数据和 16 位可变数据。这可能表明前 16 位代表用户的固定属性——例如用户名或 IP 地址。如果第二个 16 位块以固定速率递增，则可能表明令牌生成中存在顺序甚至基于时间的元素。参见示例。

如果识别出令牌中的静态元素，应收集更多样本，一次改变一个潜在输入元素。例如，通过不同用户账户或不同 IP 地址进行的登录尝试，可能会在先前静态的部分产生变化。

在单个和多个会话 ID 结构测试期间应解决以下领域：

- 会话 ID 的哪些部分是静态的？
- 会话 ID 中存储了哪些明文机密信息？例如用户名/UID、IP 地址
- 存储了哪些易于解码的机密信息？
- 从会话 ID 的结构中可以推断出哪些信息？
- 在相同的登录条件下，会话 ID 的哪些部分是静态的？
- 整个会话 ID 或各个部分中存在哪些明显的模式？

#### 会话 ID 的可预测性与随机性

应对会话 ID 的可变区域（如果有）进行分析，以确定是否存在任何可识别或可预测的模式。这些分析可以手动执行，也可以使用定制或现成的统计或密码分析工具来推断会话 ID 内容中的任何模式。手动检查应包括对相同登录条件（例如相同的用户名、密码和 IP 地址）下颁发的会话 ID 进行比较。

时间是一个必须控制的重要因素。应在同一时间窗口内进行大量并发连接以收集样本，并保持该变量恒定。即使 50ms 甚至更小的量化粒度可能仍然太粗，以此方式获取的样本可能会揭示本来会被忽略的基于时间的组件。

应随时间分析可变元素，以确定它们是否本质上是递增的。如果是递增的，应调查与绝对时间或经过时间相关的模式。许多系统使用时间作为其伪随机元素的种子。如果模式看似随机，应考虑时间或其它环境变量的单向哈希作为可能性。通常，密码哈希的结果是十进制或十六进制数，因此应是可识别的。

在分析会话 ID 序列、模式或周期、静态元素和客户端依赖性时，所有这些都应被视为对应用程序结构和功能的可能贡献因素。

- 会话 ID 在本质上是否可证明是随机的？结果值是否可以重现？
- 相同的输入条件是否会在后续运行中产生相同的 ID？
- 会话 ID 是否可证明能够抵抗统计或密码分析？
- 会话 ID 的哪些元素与时间相关？
- 会话 ID 的哪些部分是可预测的？
- 在完全了解生成算法和先前 ID 的情况下，是否可以推断出下一个 ID？

#### Cookie 逆向工程

现在测试人员已经枚举了 cookie 并对它们的用途有了大致了解，是时候更深入地研究看起来有趣的 cookie 了。测试人员对哪些 cookie 感兴趣？一个 cookie 为了提供安全的会话管理方法，必须结合多种特性，每个特性都旨在保护 cookie 免受不同类别的攻击。

这些特性总结如下：

1. 不可预测性：cookie 必须包含一定量的难以猜测的数据。伪造有效 cookie 越困难，闯入合法用户会话就越困难。如果攻击者可以猜测合法用户活动会话中使用的 cookie，他们将能够完全冒充该用户（会话劫持）。为了使 cookie 不可预测，可以使用随机值或加密。
2. 防篡改性：cookie 必须抵抗恶意修改尝试。如果测试人员收到一个类似于 `IsAdmin=No` 的 cookie，将其修改以获得管理员权限是微不足道的，除非应用程序执行双重检查（例如，在 cookie 后附加其值的加密哈希）。
3. 过期：关键 cookie 只能在适当的时间段内有效，之后必须从磁盘或内存中删除，以避免被重放的风险。这不适用于存储需要在会话间记住的非关键数据的 cookie（例如站点外观）。
4. `Secure` 标志：其值对会话完整性至关重要的 cookie 应启用此标志，以允许其仅在加密通道中传输，防止窃听。

此处的方法是收集足够数量的 cookie 实例，并开始寻找其值中的模式。“足够”的确切含义可能从少数样本（如果 cookie 生成方法非常容易破解）到数千个样本（如果测试人员需要进行一些数学分析，如卡方检验、吸引子分析等）不等。

特别注意应用程序的工作流程非常重要，因为会话状态会对收集到的 cookie 产生重大影响。在认证前收集的 cookie 与认证后获得的 cookie 可能大不相同。

另一个需要考虑的方面是时间。始终记录获取 cookie 的确切时间，当时间可能在 cookie 值中起作用时（服务器可能使用时间戳作为 cookie 值的一部分）。记录的时间可以是本地时间，也可以是 HTTP 响应中包含的服务器时间戳（或两者）。

在分析收集到的值时，测试人员应尝试找出所有可能影响 cookie 值的变量，并一次改变一个。向服务器传递相同 cookie 的修改版本对于理解应用程序如何读取和处理 cookie 非常有帮助。

此阶段应执行的检查示例包括：

- cookie 中使用了什么字符集？cookie 是数值？字母数字？十六进制？如果在 cookie 中插入不属于预期字符集的字符会发生什么？
- cookie 是否由携带不同信息的不同子部分组成？不同部分如何分隔？使用哪些分隔符？cookie 的某些部分可能具有较高的变化性，其他部分可能保持不变，还有一些可能仅取有限的一组值。将 cookie 分解为其基本组件是首要且基本的步骤。

一个易于发现的结构化 cookie 示例：

```text
ID=5a0acfc7ffeb919:CR=1:TM=1120514521:LM=1120514521:S=j3am5KzC4v01ba3q
```

此示例显示了 5 个不同的字段，携带不同类型的数据：

- ID – 十六进制
- CR – 小整数
- TM 和 LM – 大整数。（有趣的是它们持有相同的值。值得看看修改其中一个会发生什么）
- S – 字母数字

即使未使用分隔符，拥有足够的样本也有助于理解结构。

#### 暴力破解攻击

暴力破解攻击不可避免地源于与可预测性和随机性相关的问题。会话 ID 中的变化性必须与应用程序会话持续时间和超时一起考虑。如果会话 ID 中的变化性相对较小，且会话 ID 有效期较长，则成功的暴力破解攻击的可能性要高得多。

长会话 ID（或更确切地说，具有大量变化性的会话 ID）和较短的有效期将使暴力破解攻击更难成功。

- 对所有可能的会话 ID 进行暴力破解攻击需要多长时间？
- 会话 ID 空间是否足够大以防止暴力破解？例如，与有效生命周期相比，密钥长度是否足够？
- 具有不同会话 ID 的连接尝试之间的延迟是否缓解了此攻击的风险？

### 灰盒测试与示例

如果测试人员可以访问会话管理模式的实现，他们可以检查以下内容：

- 随机会话令牌

  颁发给客户端的会话 ID 或 Cookie 不应容易被预测（不要使用基于可预测变量如客户端 IP 地址的线性算法）。鼓励使用密钥长度为 256 位的加密算法（如 AES）。

- 令牌长度

  会话 ID 长度至少为 50 个字符。

- 会话超时

  会话令牌应具有定义的超时时间（取决于应用程序所管理数据的关键性）。

- Cookie 配置：
    - 非持久性：仅存在于 RAM 内存中
    - secure（仅在 HTTPS 通道上设置）：`Set-Cookie: cookie=data; path=/; domain=.aaa.it; secure`
    - [HTTPOnly](https://owasp.org/www-community/HttpOnly)（不可被脚本读取）：`Set-Cookie: cookie=data; path=/; domain=.aaa.it; HttpOnly`

更多信息请参见：[测试 Cookie 属性](02-Testing_for_Cookies_Attributes.md)
