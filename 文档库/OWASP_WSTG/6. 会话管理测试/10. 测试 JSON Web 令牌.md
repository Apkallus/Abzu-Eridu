# 测试 JSON Web 令牌

|ID          |
|------------|
|WSTG-SESS-10|

## 概述

JSON Web 令牌（JWT）是经过加密签名的 JSON 令牌，旨在系统之间共享声明。它们经常用作认证或会话令牌，尤其在 REST API 中。

JWT 是常见的漏洞来源，无论是在应用程序中的实现方式还是底层库中。由于它们用于认证，一个漏洞很容易导致应用程序被完全攻陷。

## 测试目标

- 确定 JWT 是否暴露敏感信息。
- 确定 JWT 是否可被篡改或修改。

## 如何测试

### 概述

JWT 由三个部分组成：

- 头部
- 载荷（或正文）
- 签名

每个部分都经过 Base64 编码，并用句点（`.`）分隔。请注意，JWT 中使用的 Base64 编码会去除等号（`=`），因此您可能需要将其添加回来才能解码各部分。

### 分析内容

#### 头部

头部定义了令牌类型（通常为 `JWT`）和用于签名的算法。一个解码后的头部示例如下：

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

用于计算签名的主要算法有三种类型：

| 算法 | 描述 |
|-----------|-------------|
| HSxxx | 使用密钥和 SHA-xxx 的 HMAC。 |
| RSxxx 和 PSxxx | 使用 RSA 的公钥签名。 |
| ESxxx | 使用 ECDSA 的公钥签名。 |

还有多种[其他算法](https://www.iana.org/assignments/jose/jose.xhtml#web-signature-encryption-algorithms)可用于加密令牌（JWE），尽管这些不太常见。

#### 载荷

JWT 的载荷包含实际数据。一个示例载荷如下所示：

```json
{
  "username": "administrator",
  "is_admin": true,
  "iat": 1516239022,
  "exp": 1516242622
}
```

载荷通常未加密，因此请检查其中是否包含任何敏感或不适当的数据。

此 JWT 包含用户名和管理员状态，以及两个标准声明（`iat` 和 `exp`）。这些声明在 [RFC 7519](https://tools.ietf.org/html/rfc7519#section-4.1) 中定义，下表简要总结了它们：

| 声明 | 全称 | 描述 |
|-------|-----------|-------------|
| `iss` | 颁发者 | 颁发令牌的方的身份。 |
| `iat` | 颁发时间 | 令牌颁发时的 Unix 时间戳。 |
| `nbf` | 生效时间 | 令牌可被使用的最早日期的 Unix 时间戳。 |
| `exp` | 过期时间 | 令牌过期的 Unix 时间戳。 |

#### 签名

签名使用 JWT 头部中定义的算法计算，然后进行 Base64 编码并附加到令牌。修改 JWT 的任何部分都应导致签名无效，并且令牌被服务器拒绝。

### 审查使用情况

除了自身加密安全外，JWT 还需要以安全的方式存储和发送。这应包括检查：

- 始终通过[加密（HTTPS）连接发送](../09-Testing_for_Weak_Cryptography/03-Testing_for_Sensitive_Information_Sent_via_Unencrypted_Channels.md)。
- 如果存储在 cookie 中，则应[标记适当的属性](../06-Session_Management_Testing/02-Testing_for_Cookies_Attributes.md)。

还应基于 `iat`、`nbf` 和 `exp` 声明审查 JWT 的有效性，以确定：

- JWT 对应用程序具有合理的生命周期。
- 过期的令牌被应用程序拒绝。

### 签名验证

JWT 遇到的最严重漏洞之一是应用程序未能验证签名是否正确。这通常发生在开发人员使用诸如 Node.js `jwt.decode()` 之类的函数时，该函数仅解码 JWT 的正文，而不是 `jwt.verify()`，后者在解码 JWT 之前验证签名。

这一点很容易测试：修改 JWT 的正文而不更改头部或签名中的任何内容，将其提交到请求中，查看应用程序是否接受它。

#### None 算法

除了公钥和基于 HMAC 的算法外，JWT 规范还定义了一种名为 `none` 的签名算法。顾名思义，这意味着 JWT 没有签名，允许对其进行修改。

这可以通过将 JWT 头部中的签名算法（`alg`）修改为 `none` 来测试，如下例所示：

```json
{
        "alg": "none",
        "typ": "JWT"
}
```

然后重新对头部和载荷进行 Base64 编码，并移除签名（保留尾随的句点）。使用上面的头部和[载荷](#载荷)部分列出的载荷，将得到以下 JWT：

```txt
eyJhbGciOiAibm9uZSIsICJ0eXAiOiAiSldUIn0K.eyJ1c2VybmFtZSI6ImFkbWluaW5pc3RyYXRvciIsImlzX2FkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjIsImV4cCI6MTUxNjI0MjYyMn0.
```

某些实现尝试通过显式阻止使用 `none` 算法来避免这种情况。如果以不区分大小写的方式执行此操作，则可以通过指定诸如 `NoNe` 之类的算法来绕过。

#### ECDSA “心灵签名”

在 Java 版本 15 到 18 中发现了一个漏洞，在某些情况下它们无法正确验证 ECDSA 签名（[CVE-2022-21449](https://neilmadden.blog/2022/04/19/psychic-signatures-in-java/)，称为“心灵签名”）。如果使用这些易受攻击的版本之一解析使用 `ES256` 算法的 JWT，则可以通过篡改正文，然后将签名替换为以下值来完全绕过签名验证：

```txt
MAYCAQACAQA
```

结果得到如下所示的 JWT：

```txt
eyJhbGciOiJFUzI1NiIsInR5cCI6IkpXVCJ9.eyJhZG1pbiI6InRydWUifQ.MAYCAQACAQA
```

### 弱 HMAC 密钥

如果 JWT 使用基于 HMAC 的算法（如 HS256）签名，则签名的安全性完全依赖于 HMAC 中使用的密钥的强度。

如果应用程序使用现成或开源软件，第一步应该是调查代码，查看是否存在使用的默认 HMAC 签名密钥。

如果没有默认密钥，则可能猜测或暴力破解该密钥。最简单的方法是使用 [crackjwt.py](https://github.com/Sjord/jwtcrack) 脚本，该脚本只需 JWT 和字典文件。

更强大的选择是使用 [jwt2john.py](https://github.com/Sjord/jwtcrack/blob/master/jwt2john.py) 脚本将 JWT 转换为 [John the Ripper](https://github.com/openwall/john) 可用的格式。然后可以使用 John 对密钥进行更高级的攻击。

如果 JWT 很大，可能超过 John 支持的最大大小。可以通过增加 `/src/hmacSHA256_fmt_plug.c`（或其他 HMAC 格式的等效文件）中的 `SALT_LIMBS` 变量并重新编译 John 来解决，如以下 [GitHub issue](https://github.com/openwall/john/issues/1904) 中所述。

如果可以获取此密钥，则可以创建和签署任意 JWT，这通常会导致应用程序完全被攻陷。

### HMAC 与公钥混淆

如果应用程序使用基于公钥签名的 JWT，但未检查算法是否正确，则可能利用此漏洞进行签名类型混淆攻击。为此需要满足以下条件：

1. 应用程序必须期望 JWT 使用基于公钥的算法（即 `RSxxx` 或 `ESxxx`）签名。
2. 应用程序不得检查 JWT 实际用于签名的算法。
3. 用于验证 JWT 的公钥必须对攻击者可用。

如果所有这些条件都成立，攻击者可以使用公钥使用基于 HMAC 的算法（如 `HS256`）对 JWT 进行签名。例如，[Node.js jsonwebtoken](https://www.npmjs.com/package/jsonwebtoken) 库对公钥和基于 HMAC 的令牌使用相同的函数，如下例所示：

```javascript
// 验证使用 RS256 签名的 JWT
jwt.verify(token, publicKey);

// 验证使用 HS256 签名的 JWT
jwt.verify(token, secretKey);
```

这意味着，如果 JWT 使用 `publicKey` 作为 `HS256` 算法的密钥进行签名，签名将被视为有效。

为了利用此问题，必须获取公钥。最常见的方式是应用程序将同一密钥同时用于签名 JWT 和作为 TLS 证书的一部分。在这种情况下，可以使用以下命令从服务器下载密钥：

```sh
openssl s_client -connect example.org:443 | openssl x509 -pubkey -noout
```

或者，密钥可能位于站点上公共文件中的常见位置，例如 `/.well-known/jwks.json`。

为了测试这一点，修改 JWT 的内容，然后使用先前获取的公钥使用 `HS256` 算法对 JWT 进行签名。在无法访问源代码或实现细节的情况下，这通常很难执行，因为密钥的格式必须与服务器使用的完全相同，因此空格或 CRLF 编码等问题可能导致密钥不匹配。

### 攻击者提供的公钥

[JSON Web 签名（JWS）标准](https://tools.ietf.org/html/rfc7515)（定义了 JWT 使用的头部和签名）允许用于签署令牌的密钥嵌入在头部中。如果用于验证令牌的库支持此功能，并且不将密钥与批准的密钥列表进行核对，则攻击者可以使用他们提供的任意密钥签署 JWT。

有多种脚本可用于执行此操作，例如 [jwk-node-jose.py](https://github.com/zi0Black/POC-CVE-2018-0114) 或 [jwt_tool](https://github.com/ticarpi/jwt_tool)。

### 密钥 ID（kid）操纵

`kid` 头部参数通常用于从文件系统或数据库中检索验证签名所需的密钥。它可能容易受到多种注入攻击。

#### 目录遍历

如果应用程序使用 `kid` 参数从文件系统读取密钥文件，攻击者可以指定指向已知空文件的路径，例如 `../../../../dev/null`（在 Linux 上）或 `nul`（在 Windows 上）。

例如，攻击者可以修改头部以指向空文件：

```json
{
  "alg": "HS256",
  "typ": "JWT",
  "kid": "../../../../../dev/null"
}
```

由于 `/dev/null` 的内容为空，攻击者可以使用**空字符串**作为密钥来签署恶意令牌。如果服务器存在漏洞，它将读取空文件，使用空字符串验证签名，并接受伪造的令牌。

#### 命令/SQL 注入

如果 `kid` 未经清理直接传递给数据库查询或系统命令以检索密钥，则可能容易受到 SQL 注入或命令注入的攻击。

例如，攻击者可以将 SQL 载荷注入 `kid` 参数，以控制数据库返回的密钥：

```json
{
  "alg": "HS256",
  "typ": "JWT",
  "kid": "invalid-key' UNION SELECT 'attacker-controlled-key'--"
}
```

这允许攻击者强制应用程序使用已知密钥（例如“attacker-controlled-key”）进行验证，从而能够伪造有效令牌。

## 修复

- 使用安全且最新的库来处理 JWT。
- 确保签名有效，并且使用预期的算法。
- 使用强 HMAC 密钥或唯一的私钥进行签名。
- 确保载荷中未暴露敏感信息。
- 确保 JWT 安全存储和传输。
- 请参阅 [OWASP JSON Web 令牌速查表](https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html)。
