# 测试 SSI 注入

|ID          |
|------------|
|WSTG-INPV-08|

## 概述

Web 服务器通常允许开发人员在静态 HTML 页面中添加少量动态代码，而无需使用完整的服务器端或客户端语言。此功能由[服务器端包含（SSI）](https://owasp.org/www-community/attacks/Server-Side_Includes_%28SSI%29_Injection)提供。

服务器端包含是 Web 服务器在将页面提供给用户之前解析的指令。当仅需要执行非常简单的任务时，它们是编写 CGI 程序或使用服务器端脚本语言嵌入代码的替代方案。常见的 SSI 实现提供包含外部文件、设置和打印 Web 服务器 CGI 环境变量，或执行外部 CGI 脚本或系统命令的指令（命令）。

SSI 可能导致远程命令执行（RCE），但大多数 Web 服务器默认禁用 `exec` 指令。

这是一种与经典脚本语言注入漏洞非常相似的漏洞。一种缓解措施是 Web 服务器需要配置为允许 SSI。另一方面，SSI 注入漏洞通常更容易利用，因为 SSI 指令易于理解且功能强大，例如，它们可以输出文件内容并执行系统命令。

## 测试目标

- 识别 SSI 注入点。
- 评估注入的严重性。

## 如何测试

为了测试可利用的 SSI，将 SSI 指令作为用户输入注入。如果启用了 SSI 且用户输入验证未正确实施，服务器将执行该指令。这与经典脚本语言注入漏洞非常相似，都发生在用户输入未经正确验证和清理时。

首先确定 Web 服务器是否支持 SSI 指令。通常答案是肯定的，因为 SSI 支持相当普遍。要确定是否支持 SSI 指令，请使用信息收集技术发现目标运行的 Web 服务器类型（参见[指纹识别 Web 服务器](../01-Information_Gathering/02-Fingerprint_Web_Server.md)）。如果您可以访问代码，请通过搜索 Web 服务器配置文件中的特定关键字来确定是否使用了 SSI 指令。

验证是否启用 SSI 指令的另一种方法是检查是否有 `.shtml` 扩展名的页面，该扩展名与 SSI 指令相关联。使用 `.shtml` 扩展名不是强制性的，因此如果没有找到任何 `.shtml` 文件并不一定意味着目标不易受到 SSI 注入攻击。

下一步是确定所有可能的用户输入向量，并测试 SSI 注入是否可利用。

首先找到所有允许用户输入的页面。可能的输入向量也可能包括头部和 Cookie。确定输入如何存储和使用，即输入是作为错误消息还是页面元素返回，以及是否以某种方式被修改。访问源代码可以帮助您更轻松地确定输入向量的位置以及输入的处理方式。

一旦有了潜在注入点的列表，就可以确定输入是否被正确验证。确保可以注入 SSI 指令中使用的字符，例如 `<!#=/."->` 和 `[a-zA-Z0-9]`。

以下示例返回变量的值。[参考资料](#references)部分包含有用的链接，其中包含特定于服务器的文档，可帮助您更好地评估特定系统。

```html
<!--#echo var="VAR" -->
```

使用 `include` 指令时，如果提供的文件是 CGI 脚本，此指令将包含 CGI 脚本的输出。此指令也可用于包含文件内容或列出目录中的文件：

```html
<!--#include virtual="FILENAME" -->
```

要返回系统命令的输出：

```html
<!--#exec cmd="OS_COMMAND" -->
```

如果应用程序存在漏洞，则注入的指令将在下次提供页面时由服务器解释。

SSI 指令也可以注入到 HTTP 头部中，如果 Web 应用程序使用该数据构建动态生成的页面：

```text
GET / HTTP/1.1
Host: www.example.com
Referer: <!--#exec cmd="/bin/ps ax"-->
User-Agent: <!--#include virtual="/proc/version"-->
```
