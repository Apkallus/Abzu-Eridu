# 测试不当错误处理

|ID          |
|------------|
|WSTG-ERRH-01|

## 概述

所有类型的应用程序（Web 应用、Web 服务器、数据库等）都会因各种原因产生错误。开发人员通常忽略处理这些错误，或者认为用户不会故意触发错误（例如，在期望整数的地方发送字符串）。当开发人员只考虑正常路径时，他们忽略了代码可能收到但无法处理的所有其他可能的用户输入。

错误有时表现为：

- 堆栈跟踪，
- 网络超时，
- 输入不匹配，
- 以及内存转储。

不当的错误处理可能允许攻击者：

- 了解内部使用的 API。
- 通过获取内部系统和框架的使用信息，映射相互集成的各种服务，从而为攻击链打开大门。
- 收集正在使用的应用程序的版本和类型。
- 通过强制系统进入死锁或未处理异常，向运行它的引擎发送恐慌信号，从而导致 DoS。
- 绕过控制，即某些异常不受围绕正常路径设置的逻辑的限制。

## 测试目标

- 识别现有的错误输出。
- 分析返回的不同输出。

## 如何测试

错误通常被视为无害，因为它们提供诊断数据和消息，可以帮助用户理解当前问题，或帮助开发人员调试该错误。

通过尝试发送意外数据，或强制系统进入某些边缘情况和场景，系统或应用程序大多时候会透露一些内部正在发生的事情，除非开发人员关闭了所有可能的错误并返回某个自定义消息。

### Web 服务器

所有 Web 应用都运行在 Web 服务器上，无论是集成的还是完整的。Web 应用必须处理和解析 HTTP 请求，因此 Web 服务器始终是技术栈的一部分。一些最著名的 Web 服务器是 Nginx、Apache 和 IIS。

Web 服务器具有已知的错误消息和格式。如果不熟悉它们的样子，可以在线搜索示例。另一种方法是查阅其文档，或者简单地在本地设置一个服务器，并通过浏览 Web 服务器使用的页面来发现错误。

为了触发错误消息，测试人员必须：

- 搜索随机的文件和文件夹，这些将不会被找到（404）。
- 尝试请求存在的文件夹并观察服务器行为（403、空白页面或目录列表）。
- 尝试发送违反 [HTTP RFC](https://tools.ietf.org/html/rfc7231) 的请求。一个示例是发送一个非常长的路径、破坏头部格式或更改 HTTP 版本。
    - 即使错误在应用程序级别被处理，破坏 HTTP RFC 也可能使集成的 Web 服务器暴露自身，因为它必须处理请求，而开发人员忘记了覆盖这些错误。

### 应用程序

应用程序最容易泄露各种各样的错误消息，包括：堆栈跟踪、内存转储、未处理异常和通用错误。这是因为应用程序大多数时候是自定义构建的，开发人员需要观察和处理所有可能的错误情况（或有一个全局错误捕获机制），并且这些错误可能来自与其他服务的集成。

为了使应用程序抛出这些错误，测试人员必须：

1. 识别应用程序期望数据的可能输入点。
2. 分析预期的输入类型（字符串、整数、JSON、XML 等）。
3. 基于前面的步骤对每个输入点进行模糊测试，以获得更聚焦的测试场景。
   - 对所有可能的注入进行模糊测试并不是最佳解决方案，除非您有无限的测试时间并且应用程序可以处理那么多输入。
   - 如果模糊测试不可行，请手动选择最有可能破坏某个解析器的输入（例如，JSON 主体的右括号，在只期望几个字符的地方输入大量文本，使用可能被服务器和输入验证控制解析的参数的 CLRF 注入，不适用于文件名的特殊字符等）。
   - 应对每种类型进行行话数据模糊测试，因为有时解释器会在开发人员的异常处理之外崩溃。
4. 了解响应的服务，并尝试制作更精细的模糊测试列表，以从该服务中引出更多信息或错误详情（可能是数据库、独立服务等）。

错误消息有时是映射系统的主要弱点，尤其是在微服务架构下。如果服务没有正确设置以通用和统一的方式处理错误，错误消息将使测试人员能够识别哪个服务处理哪些请求，并允许针对每个服务进行更聚焦的攻击。

> 测试人员需要警惕响应类型。有时错误以成功状态返回，但带有错误主体，或者将错误隐藏在 302 重定向中，或者只是以自定义方式表示该错误。
