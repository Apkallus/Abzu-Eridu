# 测试权限提升

|ID          |
|------------|
|WSTG-ATHZ-03|

## 概述

本节描述权限从一个级别提升到另一个级别的问题。在此阶段，测试人员应验证用户是否无法以可能导致权限提升攻击的方式修改其在应用程序中的权限或角色。

当用户获得超出其正常允许范围的更多资源或功能访问权限时，即发生权限提升，而这种提升或更改本应被应用程序阻止。这通常由应用程序中的缺陷引起。结果是应用程序以超出开发人员或系统管理员预期的权限执行操作。

提升的程度取决于攻击者被授权拥有的权限，以及成功利用后可获得的权限。例如，允许用户在成功认证后获得额外权限的编程错误会限制提升程度，因为用户已被授权拥有某些权限。同样，无需任何认证即可获得超级用户权限的远程攻击者则呈现更高程度的提升。

通常，当有可能访问授予更高权限账户的资源时（例如获取应用程序的管理权限），人们称之为*垂直提升*；而当有可能访问授予配置相似账户的资源时（例如在在线银行应用程序中，访问与另一个用户相关的信息），则称之为*水平提升*。

## 测试目标

- 识别与权限操纵相关的注入点。
- 模糊测试或以其他方式尝试绕过安全措施。

## 如何测试

### 测试角色/权限操纵

在应用程序中用户可以在数据库中创建信息（例如付款、添加联系人或发送消息）、接收信息（账户对账单、订单详情等）或删除信息（删除用户、消息等）的每个部分，都需要记录该功能。测试人员应尝试以另一个用户的身份访问这些功能，以验证是否可以访问该用户角色/权限不应允许（但可能作为另一个用户被允许）的功能。

#### 操纵用户组

例如：
以下 HTTP POST 允许属于 `grp001` 的用户访问订单 #0001：

```http
POST /user/viewOrder.jsp HTTP/1.1
Host: www.example.com
...

groupID=grp001&orderID=0001
```

验证不属于 `grp001` 的用户是否可以修改 `groupID` 和 `orderID` 参数的值以访问该特权数据。

#### 操作用户配置文件

例如：
以下服务器响应显示在成功认证后返回给用户的 HTML 中的隐藏字段。

```html
HTTP/1.1 200 OK
Server: Netscape-Enterprise/6.0
Date: Wed, 1 Apr 2006 13:51:20 GMT
Set-Cookie: USER=aW78ryrGrTWs4MnOd32Fs51yDqp; path=/; domain=www.example.com
Set-Cookie: SESSION=k+KmKeHXTgDi1J5fT7Zz; path=/; domain= www.example.com
Cache-Control: no-cache
Pragma: No-cache
Content-length: 247
Content-Type: text/html
Expires: Thu, 01 Jan 1970 00:00:00 GMT
Connection: close

<form  name="autoriz" method="POST" action = "visual.jsp">
<input type="hidden" name="profile" value="SysAdmin">\

<body onload="document.forms.autoriz.submit()">
</td>
</tr>
```

如果测试人员将变量 `profile` 的值修改为 `SysAdmin` 会怎样？是否可能成为**管理员**？

#### 操纵条件值

例如：
在服务器将错误消息作为特定参数值包含在一组响应代码中的环境中，如下所示：

```text
@0`1`3`3``0`UC`1`Status`OK`SEC`5`1`0`ResultSet`0`PVValid`-1`0`0` Notifications`0`0`3`Command  Manager`0`0`0` StateToolsBar`0`0`0`
StateExecToolBar`0`0`0`FlagsToolBar`0
```

服务器对用户给予隐式信任。它相信用户将以以上消息结束会话。

在此情况下，验证是否无法通过修改参数值来提升权限。在此特定示例中，通过将 `PVValid` 值从 `-1` 修改为 `0`（无错误条件），可能可以以管理员身份认证到服务器。

#### 操纵 IP 地址

某些站点基于 IP 地址限制访问或计算失败登录尝试次数。

例如：

```text
X-Forwarded-For: 8.1.1.1
```

在这种情况下，如果站点使用 `X-forwarded-For` 的值作为客户端 IP 地址，测试人员可以更改 `X-forwarded-For` HTTP 头的 IP 值以绕过 IP 源识别。

### 测试垂直绕过授权模式

垂直授权绕过特指攻击者获得高于其自身角色的情况。对此绕过的测试侧重于验证每个角色的垂直授权模式如何实现。对于应用程序执行的每个功能、页面、特定角色或请求，需要验证是否可能：

- 访问仅应授予更高角色用户的资源。
- 对仅应由持有更高或特定角色身份的用户操作的资源执行功能。

对于每个角色：

1. 注册一个用户。
2. 基于两个不同角色建立并维护两个不同的会话。
3. 对于每个请求，将会话标识符从原始角色更改为另一个角色的会话标识符，并评估每个的响应。
4. 如果权限较弱的会话包含相同数据，或指示对更高权限功能的操作成功，则认为应用程序存在漏洞。

#### 银行站点角色场景

下表展示了银行站点上的系统角色。每个角色对事件菜单功能具有特定权限：

|     角色     |     权限     | 附加权限 |
|---------------|-------------------|-----------------------|
| 管理员 | 完全控制      | 删除                |
| 经理   | 修改、添加、读取 | 添加                   |
| 员工   | 读取、修改      | 修改                   |
| 客户   | 只读         |                       |

如果以下情况发生，则应用程序将被视为存在漏洞：

1. 客户可以操作管理员、经理或员工功能；
2. 员工用户可以操作经理或管理员功能；
3. 经理可以操作管理员功能。

假设 `deleteEvent` 函数是应用程序管理员账户菜单的一部分，并且可以通过请求以下 URL 访问：`https://www.example.com/account/deleteEvent`。然后，调用 `deleteEvent` 函数时生成以下 HTTP 请求：

```http
POST /account/deleteEvent HTTP/1.1
Host: www.example.com
[其他 HTTP 头]
Cookie: SessionID=ADMINISTRATOR_USER_SESSION

EventID=1000001
```

有效响应：

```http
HTTP/1.1 200 OK
[其他 HTTP 头]

{"message": "事件已删除"}
```

攻击者可能尝试执行相同的请求：

```http
POST /account/deleteEvent HTTP/1.1
Host: www.example.com
[其他 HTTP 头]
Cookie: SessionID=CUSTOMER_USER_SESSION

EventID=1000002
```

如果攻击者请求的响应包含相同数据 `{"message": "事件已删除"}`，则应用程序存在漏洞。

#### 管理员页面访问

假设管理员菜单是管理员账户的一部分。

如果除管理员外的任何角色可以访问管理员菜单，则应用程序将被视为存在漏洞。有时，开发人员仅在 GUI 级别执行授权验证，而将功能本身留无授权验证，因此可能导致漏洞。

### URL 遍历

尝试遍历站点并检查是否存在可能缺少授权检查的页面。

例如：

```text
/../.././userInfo.html
```

### 白盒

如果 URL 授权检查仅通过部分 URL 匹配完成，则测试人员或黑客可能通过 URL 编码技术绕过授权。

例如：

```text
startswith()、endswith()、contains()、indexOf()
```
