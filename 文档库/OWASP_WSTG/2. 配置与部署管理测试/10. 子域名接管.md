# 测试子域名接管

## 概述

成功利用此类漏洞可使攻击者声明并控制受害者的子域名。此攻击依赖于以下条件：

1. 受害者外部 DNS 服务器的子域名记录配置为指向一个不存在或非活跃的资源/外部服务/端点。XaaS（一切皆服务）产品和公共云服务的普及提供了大量潜在目标。
2. 托管该资源/外部服务/端点的服务提供商未正确处理子域名所有权验证。

如果子域名接管成功，则可能发起多种攻击（提供恶意内容、钓鱼、窃取用户会话 cookie、凭据等）。此漏洞可利用多种 DNS 资源记录，包括：`A`、`CNAME`、`MX`、`NS`、`TXT` 等。就攻击严重性而言，`NS` 子域名接管（尽管可能性较低）影响最大，因为成功的攻击可能导致完全控制整个 DNS 区域和受害者的域名。

### GitHub 示例

1. 受害者（victim.com）使用 GitHub 进行开发，并配置了 DNS 记录（`coderepo.victim.com`）以访问它。
2. 受害者决定将代码仓库从 GitHub 迁移到商业平台，但未从 DNS 服务器中删除 `coderepo.victim.com`。
3. 攻击者发现 `coderepo.victim.com` 托管在 GitHub 上，并通过 GitHub Pages 及其自己的 GitHub 账户声明了它。

### 过期域名示例

1. 受害者（victim.com）拥有另一个域名（victimotherdomain.com），并使用 CNAME 记录（www）引用该其他域名（`www.victim.com` --> `victimotherdomain.com`）
2. 在某个时刻，victimotherdomain.com 过期，任何人都可以注册。由于 CNAME 记录未从 victim.com 的 DNS 区域中删除，任何注册 `victimotherdomain.com` 的人都可以完全控制 `www.victim.com`，直到该 DNS 记录被删除或更新。

## 测试目标

- 枚举所有可能的域名（过去的和当前的）。
- 识别任何被遗忘或错误配置的域名。

## 如何测试

### 黑盒测试

子域名接管测试遵循三个阶段：子域名枚举、基于指纹的自动化检测和手动验证。

当 DNS 条目指向一个不再存在或被取消配置的外部资源时，就会产生悬空 DNS 记录。例如，指向所有者已删除的 GitHub Pages 站点的 CNAME 记录仍然解析，但底层资源未被认领。攻击者可以注册该资源并控制子域名。

#### 子域名枚举

使用 [subfinder](https://github.com/projectdiscovery/subfinder) 发现目标域名的子域名：`subfinder -d victim.com -o subdomains.txt`

这将生成一个子域名列表，用于检测阶段。

#### 基于指纹的检测

基于指纹的检测通过将每个子域名的 HTTP 响应与已知存在漏洞的服务响应数据库进行比较来工作。[can-i-take-over-xyz](https://github.com/EdOverflow/can-i-take-over-xyz) 项目维护着这个数据库，其中分类了服务提供商（如 GitHub Pages、AWS S3、Heroku 和 Fastly）在资源未被认领时返回的特定响应字符串。

使用 [subzy](https://github.com/LukaSikic/subzy) 进行快速的初步扫描：`subzy run --targets subdomains.txt`

随后使用 [nuclei](https://github.com/projectdiscovery/nuclei) 并利用专用的接管模板以获得更准确的结果：`nuclei -l subdomains.txt -t takeovers/`

任一工具的阳性结果都表明子域名的响应匹配了已知的漏洞指纹，表明存在指向第三方服务上未认领资源的悬空 DNS 记录。

例如，指向未认领的 GitHub Pages 站点的子域名返回以下响应：

```http
HTTP/1.1 404 Not Found
...
<p>There isn't a GitHub Pages site here.</p>
```

这个特定字符串在 can-i-take-over-xyz 中被列为 GitHub Pages 指纹。当 subzy 或 nuclei 匹配到此响应时，会将子域名标记为可能易受攻击。

#### 手动验证

手动验证发现。

1. 确认 DNS 记录及其指向位置：`dig CNAME subdomain.victim.com`

2. 确认响应与 [can-i-take-over-xyz](https://github.com/EdOverflow/can-i-take-over-xyz) 中列出的该服务提供商的预期指纹匹配：`curl -i http://subdomain.victim.com`

3. 确认该资源在服务提供商的平台上未被认领。

#### 特定云服务的接管

主要云提供商具有值得特别关注的独特接管模式：

- AWS S3：指向不再存在的 S3 存储桶 URL（例如 `bucket.s3.amazonaws.com`）的 CNAME 会返回 `NoSuchBucket` 响应。任何在任何 AWS 账户中创建同名存储桶的人都可以认领该子域名。
- Azure：指向已取消配置的 Azure 资源（如应用服务或流量管理器终结点）的悬空 CNAME 可以通过在不同的 Azure 订阅中注册相同的资源名称来认领。
- GCP：云存储存储桶和 Firebase Hosting 端点也存在类似的模式。

### 灰盒测试

测试人员拥有 DNS 区域文件，因此无需进行 DNS 枚举。测试方法相同。

## 修复

为了降低子域名接管的风险，应从 DNS 区域中删除易受攻击的 DNS 资源记录。建议将持续监控和定期检查作为最佳实践。
