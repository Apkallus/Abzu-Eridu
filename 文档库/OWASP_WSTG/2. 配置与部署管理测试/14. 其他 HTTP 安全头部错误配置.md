# 测试其他 HTTP 安全头部错误配置

| ID         |
|------------|
|WSTG-CONF-14|

## 概述

安全头部在保护 Web 应用程序免受广泛攻击（包括跨站脚本（XSS）、点击劫持和数据注入攻击）方面发挥着至关重要的作用。这些头部指示浏览器如何处理网站通信的安全相关方面，从而减少对已知攻击向量的暴露。然而，错误配置可能导致漏洞，削弱预期的安全防护或使其失效。本节概述了常见的安全头部错误配置、其风险以及如何正确测试它们。

## 测试目标

- 识别配置不当的安全头部。
- 评估错误配置的安全头部的影响。
- 验证所需安全头部的正确实施。

## 常见的安全头部错误配置

- **安全头部值为空：** 头部存在但缺少值，浏览器可能忽略，使其无效。
- **安全头部值或名称无效（拼写错误）：** 错误的头部名称或拼写错误导致头部无法被识别或执行。
- **过度宽松的安全头部：** 配置过于宽泛的头部（例如使用通配符 `*` 或过于宽松的指令）可能泄露信息或允许访问超出预期范围的资源。
- **重复的安全头部：** 同一头部多次出现且值冲突，可能导致浏览器行为不可预测，甚至完全禁用安全措施。
- **遗留或已弃用的头部：** 包含过时的头部（如 HPKP）或指令（如 X-Frame-Options 中的 `ALLOW-FROM`），现代浏览器不再支持，可能带来不必要的风险。
- **安全头部放置位置不当：** 某些头部仅在特定条件下有效。例如，HSTS 等头部必须通过 HTTPS 传递；如果通过 HTTP 发送，则无效。
- **META 标签处理错误：** 如果安全策略（如 CSP）同时通过 HTTP 头部和 META 标签（使用 `http-equiv`）强制执行，则存在 META 标签值可能覆盖或与 HTTP 头部中定义的安全逻辑冲突的风险。这可能导致不安全的策略无意中优先，从而削弱整体安全态势。
- **逐跳头部注入：** 当中间件错误处理 `Connection` 头部时，攻击者可以列出并“剥离”敏感的内部安全头部（如 `X-Forwarded-For`），使其在请求到达后端之前被移除。

## 错误配置安全头部的风险

- **有效性降低：** 配置错误的头部可能无法提供预期的保护，使应用程序面临 XSS、点击劫持或 CORS 相关漏洞等攻击。
- **安全措施失效：** 重复的头部或冲突的指令可能导致浏览器完全忽略 HTTP 安全头部，从而禁用预期的保护。
- **引入新的攻击向量：** 使用遗留或已弃用的头部，如果现代浏览器不再支持预期的安全措施，则可能引入风险而非降低风险。

## 如何测试

### 获取并审查 HTTP 安全头部

要检查应用程序使用的安全头部，可使用以下方法：

- **拦截代理：** 使用 **Burp Suite** 等工具分析服务器响应。
- **命令行工具：** 执行 curl 命令获取 HTTP 响应头部：`curl -I https://example.com`
    - 有时 Web 应用程序会重定向到新页面，为跟随重定向，可使用以下命令：`curl -L -I https://example.com`
    - 某些防火墙可能会阻止 curl 的默认 User-Agent，某些 TLS/SSL 错误也会阻止返回正确信息，此时可尝试使用以下命令：
    `curl -I -L -k --user-agent "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/94.0.4606.81 Safari/537.36" https://example.com`
- **浏览器开发者工具：** 打开开发者工具（F12），导航到 **网络** 选项卡，选择一个请求，查看 **头部** 部分。

### 检查过度宽松的安全头部

- **识别风险头部：** 寻找可能允许过度访问的头部，例如：
- **评估指令：** 验证是否强制执行严格指令。例如，过度宽松的配置可能如下：

    ```http
    Access-Control-Allow-Origin: *
    Access-Control-Allow-Credentials: true
    X-Permitted-Cross-Domain-Policies: all
    Referrer-Policy: unsafe-url
    ```

    安全的配置应类似：

    ```http
    Access-Control-Allow-Origin: {允许的来源URL}
    X-Permitted-Cross-Domain-Policies: none
    Referrer-Policy: no-referrer
    ```

- **交叉参考文档：** 使用 [Mozilla 开发者网络：安全头部](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers) 等资源来审查安全和不安全的指令。

### 检查重复、已弃用/过时的头部

- **重复头部：** 确保同一头部未使用冲突值定义多次。
- **过时头部：** 识别并移除已弃用的头部（如 HPKP）和过时的指令（如 X-Frame-Options 中的 `ALLOW-FROM`）。参考 [Mozilla 开发者网络：X-Frame-Options](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/X-Frame-Options) 等来源以获取当前标准。

### 确认安全头部的正确放置

- **特定协议要求：** 验证为安全上下文设计的头部（如 HSTS）仅在适当条件下传递（即通过 HTTPS）。
- **条件传递：** 某些头部可能仅在特定情况下有效。验证这些条件是否满足，以使头部按预期运行。

### 评估 META 标签处理

- **双重执行检查：** 当安全策略（如 CSP）同时通过 HTTP 头部和使用 `http-equiv` 的 META 标签应用时，确认 HTTP 头部（通常被认为更具权威性）未被 META 标签无意覆盖。
- **审查浏览器行为：** 在不同浏览器中测试应用程序，查看是否存在因冲突指令导致的差异。如有可能，避免使用双重定义以防止意外的安全漏洞。

### 测试头部剥离（逐跳注入）

攻击者可以通过在 `Connection` 头部中列出敏感安全头部来利用此漏洞。代理遵循标准，在将请求转发到后端之前剥离这些头部。这可能导致：

- 绕过基于 IP 的访问控制列表（ACL）。
- 绕过边缘执行的身份/认证检查。
- 禁用由中间件头部强制执行的安全功能。

#### 识别内部/敏感头部（侦察）

要执行此测试，首先需要识别内部基础设施使用了哪些头部。可通过以下方式识别：

- **触发错误页面：** 发送格式错误的请求以触发错误页面（404、500），这些页面可能在响应中泄露内部头部。
- **反射端点：** 搜索调试或“回显”页面（如 `/phpinfo`、`/debug`、`/env`），这些页面会显示后端接收到的所有头部。
- **头部猜测：** 常见目标包括 `X-Forwarded-For`、`X-Real-IP`、`X-Forwarded-Proto` 和 `X-Authenticated-User`。

#### 执行注入

尝试通过将目标头部添加为 `Connection` 头部的值来“剥离”它。

##### 场景 A：绕过基于 IP 的限制

```http
GET /admin HTTP/1.1
Host: example.com
X-Forwarded-For: 203.0.113.10
Connection: close, X-Forwarded-For
```

##### 场景 B：剥离认证上下文

```http
GET /api/user/profile HTTP/1.1
Host: example.com
X-Authenticated-User: victim_user
Connection: close, X-Authenticated-User
```

#### 分析响应

- **存在漏洞：** 应用程序行为发生变化（例如，授予访问权限，或反射的 IP 消失）。
- **安全：** 应用程序行为保持不变，或代理返回 `400 Bad Request`。

## 修复

- **正确配置头部：** 确保头部实现正确，值准确且无拼写错误。
- **强制执行严格指令：** 在满足所需功能的前提下，以最安全的设置配置头部。例如，除非绝对必要，避免在 CORS 策略中使用 `*`。
- **移除已弃用的头部：** 用现代等效项替换遗留安全头部，并移除不再支持的任何头部。
- **避免冲突定义：** 防止重复的头部定义，并确保 META 标签不与安全策略的 HTTP 头部冲突。
- **限制 Connection 头部：** 配置代理忽略客户端在 `Connection` 头部中提供的与敏感内部头部匹配的值。
- **零信任：** 避免仅依赖逐跳头部进行关键安全决策。
