# 测试应用平台配置

|ID          |
|------------|
|WSTG-CONF-02|

## 概述

正确配置构成应用架构的各个组件，对于避免因配置失误而危及整个架构安全至关重要。

审查和测试配置是创建和维护架构的关键任务。这是因为各种系统通常带有通用配置，这些配置可能无法很好地适应其在特定站点上所需执行的任务。

虽然典型的 Web 服务器和应用服务器安装会包含大量功能（例如示例应用、文档、测试页面），但在部署之前，应移除所有非必需的内容，以防止安装后的漏洞利用。

## 测试目标

- 确保默认文件和已知文件已被移除。
- 验证生产环境中未遗留调试代码或扩展。
- 审查应用程序已建立的日志记录机制。

## 如何测试

### 黑盒测试

#### 示例文件和已知文件及目录

在默认安装中，许多 Web 服务器和应用服务器会提供示例应用程序和文件，以方便开发人员在安装后立即测试服务器是否正常运行。然而，许多默认的 Web 服务器应用程序后来被发现存在漏洞。例如，CVE-1999-0449（IIS 中安装了 Exair 示例站点时的拒绝服务）、CAN-2002-1744（Microsoft IIS 5.0 中 CodeBrws.asp 的目录遍历漏洞）、CAN-2002-1630（Oracle 9iAS 中 sendmail.jsp 的使用）或 CAN-2003-1172（Apache Cocoon 中 view-source 示例的目录遍历）等。

CGI 扫描器通常包含详细的已知文件和目录样本列表，这些样本来自不同的 Web 或应用服务器，可快速确定这些文件是否存在。然而，真正可靠的方法是对 Web 服务器或应用服务器的内容进行全面审查，以确定哪些内容与应用程序本身相关，哪些无关。

#### 注释审查

程序员在开发大型 Web 应用程序时添加注释非常常见。然而，HTML 代码中包含的内联注释可能会泄露内部信息，这些信息本不应被攻击者获取。有时，当某项功能不再需要时，部分源代码会被注释掉，但无意中会泄露到返回给用户的 HTML 页面中。

应进行注释审查，以确定是否通过注释泄露了任何信息。这种审查只能通过对 Web 服务器的静态和动态内容进行分析以及文件搜索来彻底完成。可以自动或引导式地浏览站点，并存储所有检索到的内容，然后对检索到的内容进行搜索，以分析代码中可能存在的任何 HTML 注释。

#### 系统配置

可以使用各种工具、文档或检查清单，帮助 IT 和安全专业人员对目标系统是否符合各种配置基线或基准进行详细评估。此类工具包括但不限于：

- [CIS-CAT Lite](https://www.cisecurity.org/blog/introducing-cis-cat-lite/)
- [Microsoft 的攻击面分析器](https://github.com/microsoft/AttackSurfaceAnalyzer)
- [NIST 的国家检查清单计划](https://nvd.nist.gov/ncp/repository)

### 灰盒测试

#### 配置审查

Web 服务器或应用服务器的配置在保护站点内容方面起着重要作用，必须仔细审查以发现常见的配置错误。显然，推荐的配置取决于站点策略以及服务器软件应提供的功能。但在大多数情况下，应遵循配置指南（无论是软件供应商还是外部方提供的），以确定服务器是否已得到妥善保护。

无法泛泛地说明服务器应如何配置，但应考虑以下常见准则：

- 仅启用应用程序所需的服务器模块（对于 IIS，即 ISAPI 扩展）。这可以减少攻击面，因为禁用软件模块可以缩小服务器的规模和复杂性。同时，如果漏洞只存在于已禁用的模块中，则可以防止供应商软件中的漏洞影响站点。
- 使用自定义错误页面处理服务器错误（40x 或 50x），而不是使用默认的 Web 服务器错误页面。尤其要确保任何应用程序错误都不会返回给最终用户，并且不会通过这些错误泄露任何代码，因为这会帮助攻击者。实际上，这一点常常被忽略，因为开发人员在预生产环境中确实需要这些信息。
- 确保服务器软件以操作系统中的最小权限运行。这可以防止服务器软件中的错误直接危害整个系统，尽管攻击者在以 Web 服务器身份运行代码后仍可能提升权限。
- 确保服务器软件正确记录合法访问和错误。
- 确保服务器配置为妥善处理过载并防止拒绝服务攻击。确保服务器已进行适当的性能调优。
- 切勿向非管理员身份（`NT SERVICE\WMSvc` 除外）授予对 applicationHost.config、redirection.config 和 administration.config 的访问权限（无论是读取还是写入）。
- 切勿通过网络共享 applicationHost.config、redirection.config 和 administration.config。
- 不要在可读的配置文件中存储敏感信息。
- 根据需要加密敏感配置数据。
- 限制对共享配置的访问并适当加以保护。

#### 日志记录

日志记录是应用程序安全的重要组成部分。它支持监控、事件响应和调查。应同时审查平台日志和应用程序日志，以确定是否记录了安全相关事件、记录是否有用，以及日志记录过程是否会引入额外风险。

审查期间，请验证以下事项：

- 安全相关事件被记录。
- 日志条目包含足够的上下文以支持调查。
- 日志不存储秘密或不必要的敏感数据。
- 在存储和显示之前，对记录的数据进行安全处理。
- 对日志的访问仅限于授权人员。
- 日志受到防篡改保护并适当保留。
- 日志记录失败不会泄露敏感调试信息。
- 在分布式环境中，日志可以集中或关联。

测试应确认日志是可用的、一致的，并能有效支持检测和响应活动。
