# 审查旧文件、备份文件和未引用文件中的敏感信息

|ID          |
|------------|
|WSTG-CONF-04|

## 概述

虽然 Web 服务器中的大多数文件由服务器自身直接处理，但发现未引用或被遗忘的文件并不罕见，这些文件可能被用于获取有关基础设施或凭据的重要信息。

最常见的场景包括：存在已修改文件的旧版本（重命名后保留）、作为源文件被加载并下载的包含文件，以及以压缩归档形式存在的自动或手动备份。备份文件也可能由托管应用程序的底层文件系统自动生成，这种功能通常称为“快照”。

所有这些文件都可能使测试人员访问内部工作机制、后门、管理界面，甚至用于连接管理界面或数据库服务器的凭据。

漏洞的一个重要来源是与应用程序无关的文件。这些文件可能是在编辑应用程序文件时创建、即时生成备份副本，或在 Web 目录树中遗留旧文件或未引用文件时产生的。在生产 Web 服务器上执行原地编辑或其他管理操作可能会无意中留下备份副本，这些副本可能是编辑器在编辑文件时自动生成的，也可能是管理员为了备份而压缩一组文件时产生的。

这些文件很容易被遗忘，并可能对应用程序构成严重的安全威胁。这是因为备份副本可能使用与原始文件不同的文件扩展名。我们生成（并可能忘记）的 `.tar`、`.zip` 或 `.gz` 归档文件显然具有不同的扩展名，许多编辑器生成的自动副本也是如此（例如，emacs 在编辑 `file` 时会生成名为 `file~` 的备份副本）。手动复制也可能产生类似效果，例如将 `file` 复制为 `file.old`。应用程序所在的底层文件系统可能会在您不知情的情况下，在不同时间点对您的应用程序进行“快照”，这些快照也可能通过 Web 访问，从而对您的应用程序构成类似但形式不同的“备份文件”威胁。

因此，这些活动会生成应用程序不需要的文件，并且 Web 服务器对这些文件的处理方式可能与原始文件不同。例如，如果我们复制 login.asp 并将其命名为 login.asp.old 而没有采取适当的安全措施，则可能允许用户下载 login.asp 的源代码。这是因为 `login.asp.old` 通常会作为文本或纯文本提供，而不是因其扩展名而被执行。换句话说，访问 `login.asp` 会导致 `login.asp` 的服务器端代码执行，而访问 `login.asp.old` 会导致 `login.asp.old` 的内容（同样是服务器端代码）以纯文本形式返回给用户并显示在浏览器中。这可能带来安全风险，因为敏感信息可能被泄露。

通常，暴露服务器端代码是个坏主意。您不仅不必要地暴露了业务逻辑，还可能无意中泄露了有助于攻击者的应用程序相关信息（路径名、数据结构等）。更不用说还有太多脚本以明文形式嵌入了用户名和密码（这是一种粗心且极其危险的做法）。

未引用文件的其他原因还包括设计或配置选择，当这些选择允许将各种类型的应用程序相关文件（如数据文件、配置文件、日志文件）存储在可由 Web 服务器访问的文件系统目录中时。这些文件通常没有理由存在于可通过 Web 访问的文件系统空间中，因为它们只应在应用程序级别由应用程序自身访问（而不是由随意浏览的用户访问）。

### 威胁

旧文件、备份文件和未引用文件对 Web 应用程序的安全性构成多种威胁：

- 未引用文件可能泄露敏感信息，从而有助于针对应用程序进行重点攻击；例如，包含数据库凭据的包含文件、包含对其他隐藏内容引用的配置文件、绝对文件路径等。
- 未引用页面可能包含可用于攻击应用程序的强大功能；例如，一个管理页面未从已发布内容中链接，但任何知道在哪里找到它的用户都可以访问。
- 旧文件和备份文件可能包含在较新版本中已修复的漏洞；例如，`viewdoc.old.jsp` 可能包含一个目录遍历漏洞，该漏洞已在 `viewdoc.jsp` 中修复，但任何找到旧版本的人仍可加以利用。
- 备份文件可能泄露设计为在服务器上执行的页面的源代码；例如，请求 `viewdoc.bak` 可能返回 `viewdoc.jsp` 的源代码，可对其审查以发现难以通过向可执行页面发送盲请求来发现的漏洞。虽然此威胁适用于 Perl、PHP、ASP、shell 脚本、JSP 等脚本语言，但并不限于它们，如下一点中的示例所示。
- 备份归档可能包含 Web 根目录内（甚至外部）所有文件的副本。这使得攻击者能够快速枚举整个应用程序，包括未引用页面、源代码、包含文件等。例如，如果您忘记了一个名为 `myservlets.jar.old` 的文件（其中包含您的 servlet 实现类的备份副本），则您将暴露大量敏感信息，这些信息可被反编译和逆向工程。
- 在某些情况下，复制或编辑文件会修改文件名，但保留文件扩展名不变。这在 Windows 环境中很常见，其中文件复制操作会生成以“Copy of ”或该字符串的本地化版本为前缀的文件名。由于文件扩展名保持不变，因此这不是 Web 服务器将可执行文件以纯文本返回的情况，因此也不是源代码泄露的情况。然而，这些文件同样危险，因为它们可能包含过时和错误的逻辑，当被调用时可能触发应用程序错误，如果启用了诊断消息显示，则可能向攻击者提供有价值的信息。
- 日志文件可能包含有关应用程序用户活动的敏感信息，例如，在 URL 参数中传递的敏感数据、会话 ID、访问的 URL（可能泄露其他未引用内容）等。其他日志文件（例如 ftp 日志）可能包含有关系统管理员维护应用程序的敏感信息。
- 文件系统快照可能包含代码副本，其中包含在较新版本中已修复的漏洞。例如，`/.snapshot/monthly.1/view.php` 可能包含一个目录遍历漏洞，该漏洞已在 `/view.php` 中修复，但任何找到旧版本的人仍可加以利用。

## 测试目标

- 查找并分析可能包含敏感信息的未引用文件。

## 如何测试

### 黑盒测试

对未引用文件的测试使用自动化和手动技术，通常涉及以下组合：

#### 从已发布内容的命名方案推断

枚举应用程序的所有页面和功能。这可以手动使用浏览器完成，或使用应用程序爬虫工具完成。大多数应用程序使用可识别的命名方案，并使用描述其功能的词将资源组织到页面和目录中。通常可以根据已发布内容使用的命名方案推断未引用页面的名称和位置。例如，如果找到名为 viewuser.asp 的页面，还应查找 edituser.asp、adduser.asp 和 deleteuser.asp。类似地，如果发现目录 /app/user，则还应搜索 /app/admin 和 /app/manager。

#### 已发布内容中的其他线索

许多 Web 应用程序在已发布内容中留下线索，可导致发现隐藏页面和功能。这些线索通常可以在 HTML 和 JavaScript 文件的源代码中找到。应手动审查所有已发布内容的源代码，以识别有关其他页面和功能的线索。例如：

程序员注释和源代码中注释掉的部分可能引用隐藏内容：

```html
<!-- <A HREF="uploadfile.jsp">上传文档到服务器</A> -->
<!-- 链接在 uploadfile.jsp 中的错误修复前被移除          -->
```

JavaScript 可能包含仅在用户 GUI 中特定条件下才呈现的页面链接：

```javascript
var adminUser=false;
if (adminUser) menu.add (new menuItem ("维护用户", "/admin/useradmin.jsp"));
```

HTML 页面可能包含通过禁用 SUBMIT 元素而隐藏的 FORM：

```html
<form action="forgotPassword.jsp" method="post">
    <input type="hidden" name="userID" value="123">
    <!-- <input type="submit" value="忘记密码"> -->
</form>
```

关于未引用目录的另一个线索来源是 `/robots.txt` 文件，用于向 Web 机器人提供指令：

```html
User-agent: *
Disallow: /Admin
Disallow: /uploads
Disallow: /backup
Disallow: /~jbloggs
Disallow: /include
```

#### 盲目猜测

其最简单形式涉及通过请求引擎运行常见文件名列表，以尝试猜测服务器上存在的文件和目录。以下 netcat 包装脚本将从标准输入读取单词列表并执行基本猜测攻击：

```bash
#!/bin/bash

server=example.org
port=80

while read url
do
echo -ne "$url\t"
echo -e "GET /$url HTTP/1.0\nHost: $server\n" | netcat $server $port | head -1
done | tee outputfile
```

根据服务器的不同，可以将 GET 替换为 HEAD 以获得更快的结果。可以对指定的输出文件进行 grep 以查找“有趣”的响应代码。响应代码 200（OK）通常表示找到了有效资源（前提是服务器未使用 200 代码提供自定义“未找到”页面）。但也要关注 301（已移动）、302（已找到）、401（未授权）、403（禁止）和 500（内部错误），这些也可能表示值得进一步调查的资源或目录。

基本猜测攻击应针对 Web 根目录以及通过其他枚举技术已识别的所有目录进行。更高级/有效的猜测攻击可按如下方式执行：

- 识别应用程序已知区域中使用的文件扩展名（例如 JSP、ASPX、HTML），并使用附加了每个扩展名的基础单词列表（如果资源允许，也可使用更长的常见扩展名列表）。
- 对于通过其他枚举技术识别的每个文件，根据该文件名创建自定义单词列表。获取常见文件扩展名列表（包括 ~、bak、txt、src、dev、old、inc、orig、copy、tmp、swp 等），并在实际文件名的扩展名之前、之后和替换其扩展名使用每个扩展名。

注意：Windows 文件复制操作会生成以“Copy of ”或该短语的本地化版本为前缀的文件名，因此它们不会更改文件扩展名。虽然“Copy of ”文件在访问时通常不会泄露源代码，但如果调用时导致错误，则可能产生有价值的信息。

#### 通过服务器漏洞和错误配置获得的信息

错误配置的服务器泄露未引用页面的最明显方式是通过目录列表。请求所有已枚举的目录，以识别任何提供目录列表的目录。

在单个 Web 服务器中发现了大量漏洞，允许攻击者枚举未引用内容，例如：

- Apache ?M=D 目录列表漏洞。
- 各种 IIS 脚本源代码泄露漏洞。
- IIS WebDAV 目录列表漏洞。

#### 利用公开可用信息

面向 Internet 的 Web 应用程序中未从应用程序内部引用的页面和功能可能从其他公共领域来源引用。这些引用有各种来源：

- 曾经被引用的页面可能仍出现在互联网搜索引擎的存档中。例如，`1998results.asp` 可能不再从公司站点链接，但可能仍保留在服务器和搜索引擎数据库中。这个旧脚本可能包含可用于攻陷整个站点的漏洞。Google 搜索的 `site:` 运算符可用于仅针对所选域运行查询，例如：`site:www.example.com`。以这种方式使用搜索引擎已经产生了广泛的技术，您可能会觉得有用，这些技术在本文档的“Google 黑客”部分中有所描述。请查看它，以通过 Google 磨练您的测试技能。备份文件不太可能被任何其他文件引用，因此可能未被 Google 索引，但如果它们位于可浏览目录中，搜索引擎可能会知道它们。
- 此外，Google 和 Yahoo 会保留其机器人找到的页面的缓存版本。即使 `1998results.asp` 已从目标服务器中删除，这些搜索引擎仍可能存储其输出的版本。缓存版本可能包含对服务器上仍存在的其他隐藏内容的引用或线索。
- 未从目标应用程序内部引用的内容可能由第三方站点链接。例如，代表第三方交易商处理在线支付的应用程序可能包含各种定制功能，这些功能通常只能通过其客户站点内的链接找到。

#### 文件名过滤器绕过

由于拒绝列表过滤器基于正则表达式，有时可以利用不常见的操作系统文件名扩展功能，这些功能以开发人员未曾预料的方式工作。测试人员有时可以利用应用程序、Web 服务器和底层操作系统解析文件名方式及其文件名约定之间的差异。

示例：Windows 8.3 文件名扩展 `c:\program files` 变为 `C:\PROGRA~1`

- 移除不兼容字符
- 将空格转换为下划线
- 取基名的前六个字符
- 添加 `~<数字>`，用于区分使用相同六个初始字符的文件名
- 此约定在前 3 次名称冲突后发生变化
- 将文件扩展名截断为三个字符
- 将所有字符转换为大写

### 灰盒测试

对旧文件和备份文件执行灰盒测试需要检查属于 Web 应用程序基础架构中 Web 服务器所服务的 Web 目录集合内的文件。理论上，为了彻底，应手动进行检查。然而，由于在大多数情况下，文件副本或备份文件往往使用相同的命名约定创建，因此搜索可以轻松地通过脚本实现。例如，编辑器通过使用可识别的扩展名或结尾来命名备份副本，而人们往往留下带有 `.old` 或类似可预测扩展名的文件。一个有用的策略是定期安排后台作业检查具有可能被识别为副本或备份文件的扩展名的文件，同时进行较长时间间隔的手动检查。

## 修复

为了有效的保护策略，测试应与明确禁止危险实践的安全策略相结合，包括：

- 在 Web 服务器或应用服务器文件系统上原地编辑文件。这是一个特别糟糕的习惯，因为它很可能由编辑器生成备份或临时文件。令人惊讶的是，即使在大型组织中，这种情况也经常发生。如果您绝对需要在生产系统上编辑文件，请确保您没有留下任何非明确意图的内容，并记住您这样做需要自担风险。
- 仔细检查对 Web 服务器暴露的文件系统执行的任何其他活动，例如临时管理活动。例如，如果您偶尔需要拍摄几个目录的快照（您不应该在生产系统上这样做），您可能会先压缩它们。注意不要留下此类归档文件。
- 适当的配置管理策略应有助于防止过时和未引用的文件。
- 应用程序的设计应避免创建（或依赖）存储在 Web 服务器所服务的 Web 目录树下的文件。数据文件、日志文件、配置文件等应存储在 Web 服务器无法访问的目录中，以防止信息泄露的可能性，更不用说如果 Web 目录权限允许写入，则存在数据修改的潜在风险。
- 如果文档根目录位于使用快照技术的文件系统上，则文件系统快照不应通过 Web 访问。配置 Web 服务器拒绝访问此类目录，例如，在 Apache 下，应使用如下 location 指令：

```xml
<Location ~ ".snapshot">
    Order deny,allow
    Deny from all
</Location>
```

## 工具

漏洞评估工具通常包含检查以发现具有标准名称的 Web 目录（例如“admin”、“test”、“backup”等），并报告任何允许索引的 Web 目录。如果测试人员无法找到目录列表，则应尝试检查可能的备份扩展名。一些有助于此目的的工具包括：

- [Nessus](https://www.tenable.com/products/nessus)
- [Nikto2](https://cirt.net/Nikto2)

### Web 爬虫工具

- [wget](https://www.gnu.org/software/wget/)
- Spike Proxy（已弃用——不再维护；已被 Burp Suite 等工具取代）
- Xenu（已弃用——最后更新于 2010 年，无活跃维护）
- [curl](https://curl.haxx.se)

### 现代替代品

- [Burp Suite](https://portswigger.net/burp) – 广泛使用的 Web 安全测试代理
- [ZAP](https://www.zaproxy.org/) – 开源 Web 应用程序安全测试工具
- [Screaming Frog SEO Spider](https://www.screamingfrog.co.uk/seo-spider/) – 网站爬虫和断链检查器

其中一些也包含在标准 Linux 发行版中。Web 开发工具通常包含识别断链和未引用文件的功能。
