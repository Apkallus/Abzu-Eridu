# 测试针对敏感信息的文件扩展名处理

|ID          |
|------------|
|WSTG-CONF-03|

## 概述

Web 服务器通常使用文件扩展名来确定处理 Web 请求时应使用哪些技术、语言和插件。虽然此行为符合 RFC 和 Web 标准，但使用标准文件扩展名会为渗透测试人员提供有关 Web 设备底层技术的有用信息，并极大地简化了确定针对特定技术使用何种攻击场景的任务。此外，Web 服务器的错误配置可能轻易泄露有关访问凭证的机密信息。

文件扩展名检查通常用于在文件上传到服务器之前对其进行验证。不加限制的文件上传可能导致意外结果，因为内容可能并非预期内容，或者由于操作系统文件名处理的意外行为。

理解 Web 服务器如何处理对不同扩展名文件的请求，有助于阐明基于所访问文件类型的服务器行为。例如，它有助于理解哪些文件扩展名会以文本或纯文本形式返回，哪些会导致服务器端执行。后者表明 Web 服务器或应用服务器所使用的技术、语言或插件。这些信息可提供有关 Web 应用程序工程方式的更多见解。例如，虽然“.pl”扩展名通常与服务器端 Perl 支持相关联，但仅凭文件扩展名可能会产生误导，并不能完全表明底层技术。例如，用 Perl 编写的服务器端资源可能会被重命名以隐藏 Perl 的使用。有关识别服务器端技术和组件的更多信息，请参阅下一节“Web 服务器组件”。

## 测试目标

- 枚举可能包含原始数据（如脚本或凭据）的敏感文件扩展名
- 验证不存在针对已设定规则的系统框架绕过

## 如何测试

### 强制浏览

提交带有不同文件扩展名的请求，并验证它们如何处理。验证应针对每个 Web 目录进行。验证允许执行脚本的目录。Web 服务器目录可通过扫描工具识别，这些工具会查找是否存在知名目录。此外，镜像站点结构有助于测试人员重建应用程序提供的目录树。

如果 Web 应用程序架构是负载均衡的，则评估所有 Web 服务器非常重要。此任务的难易程度取决于负载均衡基础设施的配置。在具有冗余组件的基础设施中，各个 Web 或应用服务器的配置可能存在细微差异。如果 Web 架构采用异构技术（例如，在负载均衡配置中同时使用 IIS 和 Apache Web 服务器），则可能会出现这种情况，这可能引入两者之间的轻微不对称行为，并可能导致不同的漏洞。

#### 示例

测试人员发现存在一个名为 `connection.inc` 的文件。尝试直接访问它会返回其内容，内容如下：

```php
<?
    mysql_connect("127.0.0.1", "root", "password")
        or die("Could not connect");
?>
```

测试人员确定存在 MySQL DBMS 后端，以及 Web 应用程序用于访问它的弱凭据。

以下文件扩展名绝不应由 Web 服务器返回，因为它们属于可能包含敏感信息的文件或没有合理理由被提供的文件。

- `.asa`
- `.inc`
- `.config`

以下文件扩展名与文件相关，当访问这些文件时，浏览器会显示或下载它们。因此，必须检查具有这些扩展名的文件，以验证它们确实应该被提供（而不是遗留文件），并且不包含敏感信息。

- `.zip`, `.tar`, `.gz`, `.tgz`, `.rar` 等：（压缩）归档文件
- `.java`：没有理由提供 Java 源代码文件
- `.txt`：文本文件
- `.pdf`：PDF 文档
- `.docx`, `.rtf`, `.xlsx`, `.pptx` 等：Office 文档
- `.bak`, `.old` 及其他表示备份文件的扩展名（例如：Emacs 备份文件的 `~`）

上面列出的列表仅列举了几个示例，因为文件扩展名太多，无法在此全面涵盖。有关更全面的扩展名数据库，请参阅 [FILExt](https://filext.com/)。

为了识别具有给定扩展名的文件，可以使用多种技术组合。这些技术可包括使用漏洞扫描器、爬虫和镜像工具，以及查询搜索引擎（参见[测试：爬虫和谷歌搜索](../01-Information_Gathering/01-Conduct_Search_Engine_Discovery_Reconnaissance_for_Information_Leakage.md)）。手动检查应用程序也可能有益，因为它可以克服自动爬虫的局限性。另请参见[测试旧文件、备份文件和未引用文件](04-Review_Old_Backup_and_Unreferenced_Files_for_Sensitive_Information.md)，该章节涉及与“被遗忘”文件相关的安全问题。

### 文件上传

在基于 Windows 的系统中，Windows 8.3 旧版文件名处理可能会影响 Web 服务器对文件的解析和访问方式。虽然此行为通常在上传限制的背景下讨论，但在评估现有非标准或旧版名称的文件如何被暴露时，它同样具有相关性。

在启用 8.3 文件名生成的环境中，原本无法通过长文件名直接访问的敏感文件，仍可能通过其缩短后的等效名称被访问。如果访问控制未得到一致执行，这可能导致源代码或配置文件被意外泄露。

可能导致意外文件泄露的 8.3 文件名解析行为示例：

1. 诸如 `file.phtml` 之类的文件可能被当作 PHP 代码处理。
2. 根据服务器和处理程序配置，相应的缩短文件名（例如 `FILE~1.PHT`）可能可被访问。
3. 具有误导性或扩展名的文件，一旦被操作系统扩展，仍可能解析为可执行处理程序。

测试应侧重于识别旧版文件名处理是否允许访问本不应提供的敏感文件。对文件上传机制本身的测试在专门的文件上传和业务逻辑测试用例中涵盖。

### 灰盒测试

文件扩展名处理的白盒测试包括检查 Web 应用程序架构中的服务器配置，并验证用于提供不同文件扩展名的规则。

如果 Web 应用程序依赖于负载均衡的异构基础设施，请确定这是否会导致不同的行为。

## 工具

漏洞扫描器（如 Nessus 和 Nikto）会检查是否存在知名的 Web 目录。它们可能允许测试人员下载站点结构，这对于尝试确定 Web 目录的配置以及单个文件扩展名如何被提供很有帮助。可用于此目的的其他工具包括：

- [wget](https://www.gnu.org/software/wget)
- [cURL](https://curl.haxx.se)
- 执行 Google 搜索“web mirroring tools”
