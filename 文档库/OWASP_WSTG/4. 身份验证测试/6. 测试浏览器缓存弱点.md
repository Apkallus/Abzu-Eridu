# 测试浏览器缓存弱点

|ID          |
|------------|
|WSTG-ATHN-06|

## 概述

在此阶段，测试人员检查应用程序是否正确指示浏览器不保留敏感数据。

浏览器可以存储信息用于缓存和历史记录目的。缓存用于提高性能，以便先前显示的信息无需再次下载。历史机制用于方便用户，以便用户可以确切地看到检索资源时所看到的内容。如果向用户显示敏感信息（例如其地址、信用卡详细信息、社会安全号码或用户名），则此信息可能被存储用于缓存或历史记录目的，因此可通过检查浏览器缓存或仅通过按下浏览器的 **后退** 按钮来检索。

## 测试目标

- 审查应用程序是否在客户端存储敏感信息。
- 审查是否可以在未经授权的情况下访问。

## 如何测试

### 浏览器历史记录

从技术上讲，**后退** 按钮是历史记录而非缓存（参见 [HTTP 中的缓存：历史记录列表](https://www.w3.org/Protocols/rfc2616/rfc2616-sec13.html#sec13.13)）。缓存和历史记录是两个不同的实体。然而，它们具有相同的弱点：呈现先前显示的敏感信息。

第一个也是最简单的测试是向应用程序输入敏感信息并注销。然后测试人员点击浏览器的 **后退** 按钮，检查在未认证状态下是否可以访问先前显示的敏感信息。

如果通过按 **后退** 按钮测试人员可以访问之前的页面但无法访问新页面，则这不是身份验证问题，而是浏览器历史记录问题。如果这些页面包含敏感数据，则意味着应用程序未禁止浏览器存储它们。

认证不一定需要参与测试。例如，当用户输入其电子邮件地址以注册新闻通讯时，如果处理不当，此信息也可能被检索到。

可以通过以下方式阻止 **后退** 按钮显示敏感数据：

- 通过 HTTPS 提供页面。
- 设置 `Cache-Control: no-store`

`Cache-Control: no-store` 指示浏览器不要将响应存储在任何缓存或历史记录存储中，这有助于防止用户注销后导航回页面时敏感数据被访问。

### 浏览器缓存

在此，测试人员检查应用程序是否将任何敏感数据泄露到浏览器缓存中。为此，他们可以使用代理（如 ZAP）并搜索属于会话的服务器响应，检查每个包含敏感信息的页面是否服务器指示浏览器不要缓存任何数据。此类指令可以在 HTTP 响应头部中使用以下指令发出：

- `Cache-Control: no-store`

当使用 `Cache-Control: no-store` 时，对于现代浏览器，通常不再需要额外的指令如 `must-revalidate`、`max-age` 或 `Expires` 头部。

```http
HTTP/1.1
Cache-Control: no-store
```

例如，如果测试人员正在测试电子商务应用程序，他们应查找所有包含信用卡号或其他财务信息的页面，并检查所有这些页面是否都强制执行 `no-store` 指令。如果他们发现包含关键信息但未指示浏览器不要缓存其内容的页面，则他们知道敏感信息将存储在磁盘上，他们可以通过在浏览器缓存中查找该页面来轻松验证这一点。

这些信息存储的确切位置取决于客户端操作系统和所使用的浏览器。以下是一些示例：

- Mozilla Firefox：
    - Unix/Linux：`~/.cache/mozilla/firefox/`
    - Windows：`C:\Users\<用户名>\AppData\Local\Mozilla\Firefox\Profiles\<配置文件id>\Cache2\`
- Internet Explorer：
    - `C:\Users\<用户名>\AppData\Local\Microsoft\Windows\INetCache\`
- Chrome：
    - Windows：`C:\Users\<用户名>\AppData\Local\Google\Chrome\User Data\Default\Cache`
    - Unix/Linux：`~/.cache/google-chrome`

#### 审查缓存信息

Firefox 提供了查看缓存信息的功能，这对测试人员可能有利。当然，业界也产生了各种扩展和外部应用程序，您可能在 Chrome、Internet Explorer 或 Edge 中更喜欢或需要使用它们。

在大多数现代浏览器（如 [Firefox](https://developer.mozilla.org/en-US/docs/Tools/Storage_Inspector#Cache_Storage)、[Chrome](https://developers.google.com/web/tools/chrome-devtools/storage/cache) 和 Edge）中，也可以通过开发者工具获取缓存详细信息。使用 Firefox 还可以使用 URL `about:cache` 来检查缓存详细信息。

#### 检查移动浏览器的处理

缓存指令的处理对于移动浏览器可能完全不同。因此，测试人员应以干净的缓存启动新的浏览会话，并利用 Chrome 的[设备模式](https://developers.google.com/web/tools/chrome-devtools/device-mode)或 Firefox 的[响应式设计模式](https://developer.mozilla.org/en-US/docs/Tools/Responsive_Design_Mode)等功能重新测试或单独测试上述概念。

此外，ZAP 和 Burp Suite 等个人代理允许测试人员指定其爬虫/抓取工具应发送的 `User-Agent`。可以将其设置为匹配移动浏览器的 `User-Agent` 字符串，并用于查看被测应用程序发送了哪些缓存指令。

### 灰盒测试

测试方法等同于黑盒情况，因为在这两种场景中，测试人员都可以完全访问服务器响应头部和 HTML 代码。但是，在灰盒测试中，测试人员可能拥有账户凭据，这将允许他们测试仅认证用户可访问的敏感页面。
