# 测试弱密码更改或重置功能

|ID          |
|------------|
|WSTG-ATHN-09|

## 概述

对于任何需要用户使用密码进行身份验证的应用程序，必须有一种机制允许用户在忘记密码时重新获得对其账户的访问权限。尽管这有时可能是一个涉及联系网站所有者或支持团队的手动过程，但通常允许用户进行自助密码重置，并通过提供其他身份证明来重新获得对其账户的访问权限。

由于此功能提供了破坏用户账户的直接途径，因此其安全实现至关重要。

## 测试目标

- 确定密码更改和重置功能是否可能导致账户被破坏。

## 如何测试

### 信息收集

第一步是收集有关应用程序向用户提供哪些密码重置机制的信息。如果同一站点上有多个接口（例如 Web 界面、移动应用和 API），则应全部审查，以防它们提供不同的功能。

确定后，需要了解用户发起密码重置所需的信息。这可以是用户名或电子邮件地址（两者均可从公开信息中获得），但也可能是内部生成的用户 ID。

### 通用关注点

无论密码重置使用的具体方法如何，都需要考虑许多常见领域：

- 密码重置过程是否比身份验证过程更弱？

  密码重置过程提供了访问用户账户的替代机制，因此应至少与通常的身份验证过程一样安全。但是，如果它使用较弱的身验证因子（如安全问题），它可能提供更容易破坏账户的方式。

  此外，密码重置过程可能绕过使用多因素身份验证（MFA）的要求，这可能会大大降低应用程序的安全性。

- 是否有速率限制或其他防止自动化攻击的保护措施？

  与任何身份验证机制一样，密码重置过程应具有防止自动化或暴力破解攻击的保护措施。可以使用多种不同的方法来实现这一点，例如速率限制或使用验证码。这些在触发外部操作（如发送电子邮件或短信）的功能上，或在用户输入密码重置令牌时尤其重要。

  通过在连续一定次数尝试后将账户锁定在密码重置过程之外，也可以防止暴力破解攻击。然而，这也可能阻止合法用户重置密码并重新获得其账户的访问权限。

- 是否容易受到常见攻击？

  除了本指南中讨论的特定领域外，检查其他常见漏洞（如 SQL 注入或跨站脚本）也很重要。

- 重置过程是否允许用户枚举？

  更多信息请参阅[测试账户枚举](../03-Identity_Management_Testing/04-Testing_for_Account_Enumeration_and_Guessable_User_Account.md)指南。

### 电子邮件 - 发送新密码

在此模式中，用户在证明其身份后，通过电子邮件收到一个新密码。这被认为不太安全，主要有两个原因：

- 密码以未加密形式发送给用户。
- 发出请求时账户密码即被更改，实际上使用户在收到电子邮件之前被锁定在其账户之外。通过重复请求，可以阻止用户访问其账户。

在使用此方法的情况下，应审查以下方面：

- 用户首次登录时是否被强制更改密码？

  新密码通过未加密的电子邮件发送，如果用户不删除该电子邮件，它可能会无限期地留在用户的收件箱中。因此，用户应在首次登录时被要求更改密码。

- 密码是否安全生成？

  密码应使用加密安全伪随机数生成器（CSPRNG）生成，并且应足够长以防止密码猜测或暴力破解攻击。为了获得安全且用户友好的体验，应使用安全密码短语风格的方法（即组合多个单词）生成，而不是随机字符串。

- 用户的现有密码是否被发送给他们？

  有些应用程序不会为用户生成新密码，而是将用户现有的密码发送给他们。这是一种非常不安全的方法，因为它通过未加密的电子邮件暴露了他们当前的密码。此外，如果站点能够恢复现有密码，则意味着密码要么使用可逆加密存储，要么（更可能）以未加密的明文存储，这两种情况都代表了严重的安全弱点。

- 邮件是否从具有防欺骗保护的域发送？

  该域应实施 SPF、DKIM 和 DMARC，以防止攻击者伪造来自该域的电子邮件，这可能被用作社会工程攻击的一部分。

- 电子邮件是否被认为足够安全？

  电子邮件通常以未加密方式发送，并且在许多情况下，用户的电子邮件账户不会受到 MFA 的保护。它也可能在多人之间共享，尤其是在企业环境中。

  根据被测应用程序的上下文，考虑基于电子邮件的密码重置功能是否适当。

### 电子邮件 - 发送链接

在此模式中，用户会收到一封包含令牌的电子邮件链接。然后他们可以点击此链接，并被提示在站点上输入新密码。这是密码重置最常用的方法，但比前述方法实现起来更复杂。要测试的关键领域是：

- 链接是否使用 HTTPS？

  如果令牌通过未加密的 HTTP 发送，则攻击者可能拦截它。

- 链接能否多次使用？

  链接应在使用后过期，否则它们会为账户提供持久的后门。

- 未使用的链接是否会过期？

  链接应具有时间限制。具体多长合适取决于站点，但通常不应超过一小时。

- 令牌是否足够长且随机？

  此过程的安全性完全依赖于攻击者无法猜测或暴力破解令牌。令牌应使用加密安全伪随机数生成器（CSPRNG）生成，并且应足够长，使攻击者猜测或暴力破解变得不切实际。至少 128 位（或 32 个十六进制字符）足以使此类在线攻击变得不切实际。

  令牌绝不应基于已知值生成，例如通过使用 `md5($email)` 获取用户电子邮件的 MD5 哈希，或使用可能使用不安全 PRNG 函数甚至可能根本不随机的 GUID（取决于其类型）。

  测试人员还应验证密码重置令牌在服务器上的存储方式。如果重置令牌以明文形式存储在数据库中，则获得数据库访问权限的攻击者可能能够重用它们来重置用户密码。更安全的实现会存储令牌的哈希值，并在验证期间比较哈希值。

  随机令牌的替代方法是使用加密签名令牌，如 JWT。在这种情况下，应执行常规的 JWT 检查（签名是否验证、是否可使用“none”算法、HMAC 密钥是否可被暴力破解等）。更多信息请参阅[测试 JSON Web 令牌](../06-Session_Management_Testing/10-Testing_JSON_Web_Tokens.md)指南。

- 链接是否包含用户 ID？

  有时密码重置链接可能包含用户 ID 以及令牌，例如 `reset.php?userid=1&token=123456`。在这种情况下，可能可以通过修改 `userid` 参数来重置其他用户的密码。

- 是否可以注入不同的主机头？

  如果应用程序信任 `Host` 头的值并使用它生成密码重置链接，则可能通过向请求注入修改后的 `Host` 头来窃取令牌。更多信息请参阅[测试主机头注入](../07-Input_Validation_Testing/17-Testing_for_Host_Header_Injection.md)指南。

- 链接是否会暴露给第三方？

  如果用户被带到的页面包含来自其他方的内容（例如从其他域加载脚本），则 URL 中的重置令牌可能会在这些请求发送的 HTTP `Referer` 头中暴露。HTTP `Referrer-Policy` 头可用于防止此问题，因此请检查页面是否定义了该头。

  此外，如果页面包含任何跟踪、分析或广告脚本，令牌也将暴露给它们。

- 邮件是否从具有防欺骗保护的域发送？

  该域应实施 SPF、DKIM 和 DMARC，以防止攻击者伪造来自该域的电子邮件，这可能被用作社会工程攻击的一部分。

- 电子邮件是否被认为足够安全？

  电子邮件通常以未加密方式发送，并且在许多情况下，用户的电子邮件账户不会受到 MFA 的保护。它也可能在多人之间共享，尤其是在企业环境中。

  根据被测应用程序的上下文，考虑基于电子邮件的密码重置功能是否适当。

### 通过短信或电话发送令牌

另一种方法是通过短信或自动电话呼叫发送令牌，而不是通过电子邮件发送，用户随后在应用程序中输入该令牌。要测试的关键领域是：

- 令牌是否足够长且随机？

  以这种方式发送的令牌通常较短，因为它们旨在由用户手动输入，而不是嵌入链接中。应用程序使用六位数字相当常见，这仅提供约 20 位的安全性（在线暴力破解是可行的），而不是通常较长的电子邮件令牌。

  这使得密码重置功能受到暴力破解攻击的保护变得更加重要。

- 令牌能否多次使用？

  令牌应在使用后失效，否则它们会为账户提供持久的后门。

- 未使用的令牌是否会过期？

  由于较短的令牌更容易受到暴力破解攻击，应实施更短的过期时间，以限制攻击者实施攻击的时间窗口。

- 是否有适当的速率限制和限制？

  向用户发送短信或触发自动电话呼叫比发送电子邮件更具干扰性，可能被用来骚扰用户，甚至对其电话发起拒绝服务攻击。应用程序应实施速率限制以防止此问题。

  此外，短信和电话呼叫通常会为发送方带来财务成本。如果攻击者能够导致发送大量消息，可能会给网站运营商带来巨大费用。如果发送到国际或高费率号码，尤其如此。然而，允许国际号码可能是应用程序的要求。

- 短信或电话呼叫是否被认为足够安全？

  已有[多种攻击](https://www.ncsc.gov.uk/guidance/protecting-sms-messages-used-in-critical-business-processes#section_4)被证明可允许攻击者有效劫持短信，对于短信是否足够安全以用作身份验证因子存在不同观点。

  通常可以通过物理访问设备来接听自动电话呼叫，而无需任何类型的 PIN 或指纹来解锁手机。在某些情况下（如共享办公环境），这可能允许内部攻击者在同事不在办公室时走到其办公桌，轻松重置其密码。

  根据被测应用程序的上下文，考虑短信或自动电话呼叫是否适当。

### 安全问题

安全问题可用作用户身份验证的机制，而不是向他们发送链接或新密码。这被认为是一种弱方法，如果有更好的选择，不应使用。

更多信息请参阅[测试弱安全问题](08-Testing_for_Weak_Security_Question_Answer.md)指南。

### 已认证身份和配置更改

如果应用程序支持修改用于密码更改和重置功能的主要标识符（如电子邮件地址或电话号码），则应强制用户重新进行身份验证。如果在无需重新认证的情况下可以修改密码更改功能中使用的主要标识符，则允许绕过密码更改功能中的重新认证。总体而言，任何影响账户安全的内容（电子邮件、MFA、备份设置等）在修改前都应要求重新认证。

例如：一个应用程序具有密码重置流程，将重置链接发送到账户的电子邮件地址。如果从已认证用户的角度尝试更改密码，该应用程序也需要重新认证。如果攻击者获得了对账户的访问权限（通过窃取的 cookie、物理访问计算机等），并在无需重新认证的情况下更改了账户的电子邮件地址，则可以使用密码重置流程更改密码，从而绕过已认证的密码更改流程。

### 已认证密码更改

一旦用户证明其身份（通过密码重置链接、恢复代码或登录应用程序），他们应能够更改密码。要测试的关键领域是：

- 设置密码时，是否可以指定用户 ID？

  如果用户 ID 包含在密码重置请求中且未验证，则可能修改它并更改其他用户的密码。

- 是否要求用户重新认证？

  如果已登录用户尝试更改密码，应要求他们使用当前密码重新认证，以防止攻击者临时访问无人值守的会话。如果用户启用了 MFA，则通常使用 MFA 重新认证，而不是密码。

- 密码更改表单是否容易受到 CSRF 攻击？

  如果不要求用户重新认证，则可能对密码重置表单发起 CSRF 攻击，导致其账户被破坏。更多信息请参阅[测试跨站请求伪造](../06-Session_Management_Testing/05-Testing_for_Cross_Site_Request_Forgery.md)指南。

- 是否应用了强而有效的密码策略？

  密码策略应在注册、密码更改和密码重置功能之间保持一致。更多信息请参阅[测试弱身份验证方法](07-Testing_for_Weak_Authentication_Methods.md)指南。
