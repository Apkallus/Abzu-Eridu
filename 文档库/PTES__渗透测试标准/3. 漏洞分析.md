## 测试

漏洞测试是发现系统和应用程序中可被攻击者利用的缺陷的过程。这些缺陷范围广泛，包括主机和服务配置错误，或应用程序设计不安全等。虽然查找缺陷的过程因被测组件的不同而有很大差异，但该过程仍适用一些关键原则。

在进行任何类型的漏洞分析时，测试人员应适当确定测试范围，包括深度和广度，以满足预期结果的目标和/或要求。深度值可以包括评估工具的位置、认证要求等。例如：在某些情况下，测试的目标可能是验证缓解措施已到位且有效，并且漏洞不可访问；而在其他情况下，目标可能是在通过认证访问的情况下测试所有适用的变量，以发现所有适用的漏洞。无论您的范围如何，测试都应量身定制，以满足达到目标的深度要求。应始终验证测试深度，以确保评估结果符合预期（例如，所有机器是否都通过了认证？）。除了深度，在进行漏洞测试时还必须考虑广度。广度值可以包括目标网络、网段、主机、应用程序、资产清单等。在最简单的形式下，您的测试可能是查找主机系统上的所有漏洞；而在其他情况下，您可能需要查找给定资产清单或边界内主机上的所有漏洞。此外，应始终验证测试广度，以确保您已满足测试范围（例如，资产清单中的每台机器在扫描时是否都存活？如果没有，原因是什么）。

## 主动测试

主动测试涉及与待测安全漏洞组件进行直接交互。这可以是低层组件，例如网络设备上的 TCP 栈，也可以是栈上更高层的组件，例如用于管理此类设备的基于 Web 的界面。与目标组件交互有两种不同的方式：自动化和手动。

### 自动化

自动化测试利用软件与目标交互、检查响应，并根据这些响应确定是否存在漏洞。自动化过程有助于减少时间和人力需求。例如，虽然连接到系统上的单个 TCP 端口以确定它是否开放接收传入数据很简单，但如果手动执行，对可用的 65,535 个端口中的每一个执行此步骤将需要大量时间。当必须在多个网络地址上重复此类测试时，所需时间可能过长，以至于无法在没有某种自动化的情况下完成测试。使用软件执行这些功能使测试人员能够完成手头的任务，并将注意力集中在处理数据和执行更适合手动测试的任务上。

#### 网络/通用漏洞扫描器

##### 基于端口

基于端口的自动化扫描通常是传统渗透测试中的第一步，因为它有助于基本了解目标网络或主机上可能存在什么。基于端口的扫描器检查远程主机上的端口是否能够接收连接。通常，这涉及使用 IP 的协议（如 TCP、UDP、ICMP 等）。但是，根据环境的不同，也可能存在其他网络协议上的端口（例如，在大型大型机环境中，SNA 的使用非常普遍）。通常，端口可以有两种状态之一：
- **开放**：端口能够接收数据
- **关闭**：端口无法接收数据

如果扫描器无法准确确定给定端口是开放还是关闭，它可能会列出其他状态，例如“已过滤”。

当扫描器确定某个端口开放时，扫描器会假设是否存在漏洞。例如，如果基于端口的扫描器连接到 TCP 端口 23，并且该端口正在监听，扫描器很可能会报告远程主机上存在 telnet 服务，并将其标记为启用了明文认证协议。

##### 基于服务

基于服务的漏洞扫描器是一种利用特定协议与远程主机上的开放端口通信，以确定该端口上运行的服务更多信息的扫描器。这比端口扫描更精确，因为它不仅仅依赖端口来确定运行的服务。例如，端口扫描可以识别主机上的 TCP 端口 8000 是开放的，但仅凭该信息无法知道该端口上运行着什么服务。服务扫描器会尝试使用不同的协议与端口通信。如果端口 8000 上运行的服务能够正确使用 HTTP 通信，那么它将被识别为 Web 服务器。

##### 横幅抓取

横幅抓取是连接到特定端口并检查从远程主机返回的数据以识别绑定到该端口的服务/应用程序的过程。通常在连接过程中，软件会提供一个标识字符串，其中可能包括应用程序名称或正在运行的软件特定版本信息等信息。

#### Web 应用程序扫描器

##### 通用应用程序缺陷扫描器

大多数 Web 应用程序扫描从网站、Web 应用程序或 Web 服务的地址开始。然后扫描器通过跟踪链接和目录结构来爬取站点。在编译了网页、资源、服务和/或其他提供的媒体列表后，扫描器将对爬取结果执行测试或审计。例如，如果在爬取中发现的网页有表单字段，扫描器可能会尝试 SQL 注入或跨站脚本。如果爬取的页面包含错误，扫描器可能会查找错误详细信息中显示的敏感信息，依此类推。

需要注意的是，爬取和测试阶段可以交错进行并同时执行，以减少整体扫描时间。这是许多 Web 应用程序扫描器的默认行为。

##### 目录列举/暴力破解

假设网站上存在爬虫通过跟踪链接无法找到的目录。如果用户没有事先知道这些目录，扫描器至少有两个额外的选择。

扫描器/爬虫可以搜索“常见”目录。这些目录的名称和名称变体很常见，并且包含在根据多年经验和扫描编译的列表中。大多数 Web 应用程序都有一个这种类型的内置列表，而一些渗透测试人员则维护自己的自定义列表。有时目录名称足够独特，可以用来以相当高的准确度识别第三方 Web 应用程序。一个准确的目录列表通常是找到网站“管理”部分的关键——这是大多数渗透测试人员非常感兴趣的发现部分。

暴力破解目录是一种类似的方法，但不是使用静态列表，而是使用工具枚举目录名称可能的所有组合。使用这种方法的不利之处在于，它有可能使 Web 服务器崩溃或因请求过多而淹没服务器，从而导致拒绝服务条件。在执行目录暴力破解时应当小心，并有人密切关注 Web 服务器的状况，尤其是在生产环境中。

作为渗透测试人员，您想要执行目录列举的原因是为了扩展您的攻击面或找到可能包含敏感信息的目录（根据渗透测试的目标，这可能会导致一个主要发现）。

##### Web 服务器版本/漏洞识别

许多 Web 应用程序扫描器会尝试将 Web 服务器的版本与安全公告中已知的存在漏洞的版本进行比较。这种方法有时会导致误报；因为在某些情况下，开源 Web 服务器被分支或复制，并赋予新名称、横幅和不同的版本号。应采取额外步骤来验证 Web 服务器是否确实在运行横幅或 Web 扫描器报告的内容。

##### 方法

有几种 Web 服务器方法被认为是不安全的，可能允许攻击者获得不同级别的 Web 服务器内容访问权限。这些方法是 Web 服务器软件的一部分，而不是网站内容，这一点将其与迄今为止讨论的其他漏洞区分开来。
一些不安全的方法包括：

- **OPTIONS**
  虽然 HTTP OPTIONS 方法本身并不不安全，但它允许攻击者轻松枚举目标服务器接受的 HTTP 方法类型。请注意，OPTIONS 方法并不总是准确的，应单独验证以下每种方法。

- **PUT/DELETE**
  使用 PUT 方法，攻击者可以上传恶意内容，例如可用于传输信息、更改 Web 内容或在 Web 服务器上安装恶意软件的 HTML 页面。使用 DELETE 方法，攻击者可以删除内容或篡改网站，导致服务中断。

  此外，现代 REST 应用程序以不同的方式使用 PUT：
  - 创建 -> POST
  - 读取 -> GET
  - 更新 -> PUT
  - 删除 -> DELETE

- **WebDAV**
  WebDAV 是微软 Internet 信息服务 (IIS) 的一个组件。WebDAV 代表“基于 Web 的分布式创作和版本控制”，用于编辑和文件管理。管理员使用 WebDAV 扩展在 IIS Web 服务器上远程管理和编辑 Web 内容，可以包括 PROPFIND、COPY、MOVE、PROPPATCH、MKCOL、LOCK 和 UNLOCK。WebDAV 与核心操作系统组件交互，这可能使系统面临多种可能的漏洞。这些潜在风险包括：
  - 由于用户请求处理不当导致的缓冲区溢出情况
  - 格式错误请求导致的拒绝服务情况
  - 基于域的脚本攻击
  - 权限提升
  - 任意代码执行

- **TRACE/TRACK**
  现代 Web 服务器支持 TRACE HTTP 方法，该方法包含一个可能导致未经授权的信息泄露的缺陷。TRACE 方法用于调试 Web 服务器连接，并允许客户端查看请求链另一端接收到的内容。在所有主要 Web 服务器中默认启用，远程攻击者可能滥用 HTTP TRACE 功能来泄露敏感信息，导致机密性丧失。

#### 网络漏洞扫描器/特定协议

##### VPN

传统的漏洞评估工具无法与提供互联网密钥交换 (IKE) 服务的 VPN 设备进行正确的协议协商。在使用 IKE 的情况下，需要使用额外的工具包来执行诸如准确指纹识别、退避模式识别以及识别正在使用的认证机制等功能。通过识别 VPN 设备的这些属性，可以识别正在运行的代码版本以及认证类型（如静态预共享密钥）中的弱点。

##### 语音网络扫描器

###### 战争拨号
许多组织仍然使用电话线进行带外访问。使用专为战争拨号设计的漏洞评估工具可以确定认证和网络架构中的弱点。

###### VoIP
IP 语音技术现在在大多数组织中都很普遍。已经开发了许多工具来对 VoIP 基础设施进行漏洞分析。使用这些工具，可以识别 VoIP 网络是否被适当隔离，以及是否存在利用这些网络访问核心基础设施系统或记录目标网络上电话通话的潜在可能性。

#### 手动直接连接

与任何自动化流程或技术一样，总是存在误差的可能性。系统、网络设备和网络连接的不稳定性可能会在测试期间引入不准确的结果。始终建议对目标系统上可用的每个协议或服务执行手动直接连接，以验证自动化测试的结果，并识别所有潜在的攻击向量和先前未识别的弱点。

#### 混淆

##### 多出口节点
安全监控和防御系统基于从特定 IP 地址识别恶意活动的前提运行。在部署了入侵检测系统并监控活动的情况下，从多个 IP 地址发起评估和攻击活动可以提供更准确的结果，并减少目标网络上的监控设备识别和响应的机会。诸如 TOR 代理之类的技术可以提供一种在不从单个 IP 地址发起的情况下进行评估活动的手段。

##### IDS 逃逸
在对部署了 IDS 技术的目标环境进行评估活动时，可能有必要执行逃逸。使用诸如字符串操作、多态性、会话拼接和分片等方法可以在绕过 IDS 设备中实现的签名匹配模式的同时提供更准确的结果。

## 被动测试

### 元数据分析

元数据分析涉及查看描述文件的数据，而不是文件数据本身。例如，Microsoft Office 文档可能列出文档作者、公司、上次保存文档的时间、文档创建时间等。许多文档甚至允许输入自定义元数据。这可能包含内部地址和服务器路径、内部 IP 地址以及其他渗透测试人员可以用来获得额外访问权限或信息的信息。

尽管元数据在公司内部网络的文档上很常见，但公司在向公众或公共互联网提供文档之前应注意清除元数据。因此，攻击者能够被动（无需直接攻击目标）访问的任何元数据都应被视为安全问题。

### 流量监控

流量监控是连接到内部网络并捕获数据以供离线分析的概念。此阶段排除了路由中毒，因为这些会在网络上产生“噪音”并且很容易被检测到。从“交换”网络中能够收集到多少敏感数据通常令人惊讶。这种数据“泄漏”到交换网络可以分类如下：

- **ARP/MAC 缓存溢出**，导致交换数据包被广播——这在 ARP/MAC 缓存时序配置不当的 Cisco 交换机上很常见。
- **Etherleak**——一些较旧的网络驱动程序和某些嵌入式驱动程序将使用系统内存中的数据来填充 ARP 数据包。如果可以收集到足够多的 ARP 数据包，则可以捕获内部内存中的敏感信息。
- **配置错误的集群或负载均衡器**
- **连接到网络的集线器**

请注意，其中一些类别仅导致数据泄漏到单个子网，而其他类别可能导致泄漏到更大的网段。

## 验证

### 工具之间的关联

当使用多个工具时，发现结果之间的关联需求可能变得复杂。关联可以分为两种不同的风格：特定项关联和分类关联，两者都很有用，具体取决于您想在目标上收集的信息类型、指标和统计数据。

- **特定关联** 涉及特定的可定义问题，例如漏洞 ID、CVE、OSVDB、供应商索引号、软件产品的已知问题等，并且可以与微观因素（如主机名、IP、FQDN、MAC 地址等）进行分组。例如，按 CVE 编号对主机 x 的发现结果进行分组，因为它们会在多个工具中索引相同的问题。

- **分类关联** 涉及问题的分类结构，例如在合规框架中（如 NIST SP 800-53、DoD 5300 系列、PCI、HIPPA、OWASP 列表等），允许您按宏观因素（如漏洞类型、配置问题等）对项目进行分组。例如，将具有默认密码的主机的所有发现结果分组到 NIST 800-53 (IA-5) 中的密码复杂性组中。

在大多数情况下，渗透测试人员将关注同一主机上多个工具之间冗余发现的特定漏洞的微观问题。这种冗余可能会扭曲测试输出中的统计结果，导致风险状况被虚假地提高。

相反的情况是过度简化或宏观关联（例如前 10/20 列表）导致结果扭曲，从而虚假地降低风险状况。

### 手动测试/特定协议

#### VPN

##### 指纹识别
指纹识别有助于确定 VPN 设备的类型和已安装代码的正确版本。通过准确识别设备指纹，可以对目标系统进行适当的研究和分析。

##### 认证
VPN 设备可以使用各种形式的认证进行操作。使用不属于传统漏洞评估工具的 VPN 工具包可以正确识别认证机制，并确定可能存在的弱点，例如预共享密钥或默认组 ID。

#### Citrix

##### 枚举
许多默认安装和配置不当的 Citrix 设备提供了一种枚举已发布应用程序和确定配置为认证到设备的有效用户名的方法。这些信息在暴力破解攻击和尝试突破授权用户预定义配置文件的过程中变得至关重要。

#### DNS

域名系统在未得到适当加固时可以为攻击者提供大量信息。版本信息允许进行正确的识别和准确的研究分析。诸如区域传输之类的弱点提供了用于攻击的额外目标列表，以及与目标组织相关的潜在敏感数据的信息泄露。

#### Web

Web 服务为攻击者提供了广阔的领域。与大多数其他协议和服务不同，Web 服务通常发现运行在单个系统的多个端口上。管理员可能会将加固工作集中在 Web 服务的常见端口或已发布的目录上，而忽略了对其他属性的适当加固。应始终以手动方式审查 Web 服务，因为自动化评估工具无法识别其服务中的大多数弱点。

#### 邮件

邮件服务器可以提供有关目标组织的大量信息。利用目标设备的固有功能，可以进行有效帐户的确认以及开发用于对其他系统进行额外攻击的潜在用户名列表。诸如邮件中继之类的漏洞可用于对组织进行额外攻击，例如网络钓鱼。通常，邮件服务器会提供用于远程访问的 Web 界面，这可以作为暴力破解攻击的目标。

### 攻击途径

#### 创建攻击树

在安全评估期间，随着测试在参与过程中的进展，开发攻击树对于最终报告的准确性至关重要。当新的系统、服务和潜在的漏洞被识别出来时，应该开发并定期更新攻击树。这在参与过程的利用阶段尤其重要，因为一个实现的具体切入点可以在攻击树开发过程中映射出的其他向量上重复使用。

#### 隔离的实验室测试

当在隔离的实验室中设置复制环境时，漏洞分析和利用的准确性会大大提高。通常，系统可能使用特定的控制集或额外的保护机制进行加固。通过设计一个模仿目标组织的实验室，顾问可以确保针对目标识别出的漏洞和尝试的利用是可靠的，并减少不准确结果或系统无法运行的可能性。

#### 视觉确认

##### 手动连接与审查
虽然适当的关联可以帮助减少错误发现并提高整体准确性，但没有什么可以替代对目标系统进行视觉检查。评估工具旨在检查协议/服务连接的结果或响应，并将其与已知的漏洞特征进行比较。然而，工具在识别非标准端口上的服务或应用程序中内置的自定义逻辑时并不总是准确的。通过手动评估目标系统、其可用服务以及为这些服务提供功能的应用程序，测试人员可以确保已完成正确的验证和漏洞识别。

## 研究

### 公开研究

一旦在目标系统中报告了漏洞，就需要确定问题识别的准确性，并在渗透测试范围内研究该漏洞的潜在可利用性。在许多情况下，该漏洞将是商业或开源软件包中报告的软件漏洞，而在其他情况下，该漏洞可能是业务流程中的缺陷，或常见的管理错误，如配置错误或使用默认密码。

#### 漏洞数据库
漏洞数据库可用于验证自动化工具报告的问题，或手动审查目标应用程序的漏洞。大多数工具将使用给定漏洞的 CVE 标识符，可用于访问摘要信息和指向 CVE 数据库中其他来源的链接。CVE 也可用于在 OSVDB 和 Bugtraq 等漏洞数据库或利用数据库和框架中搜索该问题。

应使用漏洞数据库来验证报告问题的准确性。例如，Apache Web 服务器缺陷可能存在于 Windows 上，但不存在于 Linux 上，自动化扫描器可能没有考虑到这一点。

#### 供应商公告
供应商发布的安全公告和变更日志可以提供自动化工具可能未报告的任何漏洞信息的指针。许多主要软件供应商报告内部发现的问题以及独立研究人员协调披露的问题的有限细节。如果研究人员选择对漏洞细节保持沉默，供应商公告通常是唯一可用的数据。在这些情况下，其他研究人员可能会独立发现更多细节，并将这些细节添加到漏洞数据库中。搜索供应商公告中使用的 CVE 可能会发现有关潜在可利用问题的更多细节。

变更日志可以为额外研究提供指导，尤其是在开源产品中，版本之间的差异可以揭示一个已修复但不广为人知、因此可能未被优先考虑升级或安装的漏洞。

#### 利用数据库和框架模块

许多利用数据库在互联网上被积极维护并公开访问。安全研究人员和利用编写者并不总是将他们的利用代码提交到多个站点，因此建议熟悉多个站点，并检查每个站点以获取针对潜在易受攻击的应用程序的利用代码。虽然一些漏洞数据库跟踪利用的可用性，但它们的覆盖范围通常不完整，不应被认为是详尽无遗的。

商业和开源的利用框架在研究漏洞时也很有用。在大多数情况下，可用的利用模块会在其公共网站上列出，并且可以作为问题可利用性的宝贵指标。

#### 常见/默认密码

通常，管理员和技术人员会选择弱密码，从不更改默认密码，或者根本不设置密码。大多数软件和硬件的手册都可以在网上轻松找到，并提供默认凭据。互联网论坛和官方供应商邮件列表可以提供有关未记录帐户、常用密码和经常配置错误的帐户的信息。最后，许多网站记录了默认/后门密码，应对每个识别出的系统进行检查。

#### 加固指南/常见配置错误

渗透测试的主要目标之一是模拟真实攻击者的策略和行为。虽然自动化扫描可以缩短测试的时间窗口，但没有扫描器能像人类一样行为。加固指南对渗透测试人员来说是非常宝贵的参考。它们不仅突出了系统最薄弱的部分，而且您可以通过验证管理员实施了多少建议来了解其勤勉程度。在每次渗透测试期间，都应该花时间审查每个主要系统及其建议的加固设置，以发现管理员留下的漏洞。

用户论坛和邮件列表可以提供有关系统以及管理员在配置和保护它们时遇到的各种问题的宝贵信息。测试人员应该像自己安装系统一样研究目标系统，并发现痛点和可能的配置错误所在。

### 私有研究

#### 设置复制环境
虚拟化技术允许安全研究人员运行各种操作系统和应用程序，而无需专用硬件。当识别出目标操作系统或应用程序时，可以快速创建一个虚拟机环境来模仿目标。测试人员可以使用此虚拟机来探索应用程序的配置参数和行为，而无需直接连接到目标。

#### 测试配置
一个测试虚拟机实验室应包含所有常见操作系统的基础镜像，包括 Windows XP、Vista、7、Server 2003 和 Server 2008、Debian、Ubuntu、Red Hat 以及可能的 Mac OS X。为每个服务包级别维护单独的镜像将简化重现目标环境的过程。一个完整的虚拟机库与支持克隆的虚拟机环境相结合，将允许测试人员在几分钟内启动一个新的目标虚拟机。此外，使用快照功能可以更高效地工作并重现错误。

#### 模糊测试
模糊测试或故障注入是一种通过程序化地向应用程序提交有效、随机或意外输入来发现应用程序缺陷的暴力技术。基本过程涉及将一个调试器附加到目标应用程序，然后针对特定的输入区域运行模糊测试例程，然后在任何崩溃后分析程序状态。许多模糊测试应用程序是可用的，尽管一些测试人员会为特定目标编写自己的模糊器。

#### 识别潜在的途径/向量
登录或连接到目标网络应用程序以识别命令和其他输入区域。如果目标是读取文件和/或网页的桌面应用程序，请分析接受的文件格式以寻找数据输入的途径。一些简单的测试涉及提交无效字符或非常长的字符串以导致崩溃。附加一个调试器以在成功崩溃的情况下分析程序状态。

#### 反汇编和代码分析
某些编程语言允许反编译，并且某些特定应用程序在编译时带有调试符号。测试人员可以利用这些功能来分析程序流程并识别潜在的漏洞。应分析开源应用程序的源代码以查找缺陷。用 PHP 编写的 Web 应用程序共享许多相同的漏洞，它们的源代码应作为任何测试的一部分进行检查。
