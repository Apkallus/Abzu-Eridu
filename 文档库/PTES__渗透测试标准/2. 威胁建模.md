# 威胁建模

本节定义了正确执行渗透测试所需的威胁建模方法。本标准不使用特定的模型，而是要求所使用的模型在表示威胁、威胁能力、根据被测组织对其进行的分类，以及在未来的测试中能够重复应用并得到相同结果等方面保持一致。

本标准侧重于传统威胁建模的两个关键要素——资产和攻击者（威胁社区/代理）。每个要素分别细分为业务资产和业务流程，以及威胁社区及其能力。

作为最低要求，在每次渗透测试中，这四个要素都应被清晰地识别并记录在案。

在对攻击者方进行建模时，除了威胁社区（这主要是语义上的，可以追溯到组织的业务SWOT分析）和能力（这主要是技术上的）之外，还应提供动机建模的其他方面。这些额外要点实质上考虑了目标上不同可用资产的价值，并结合了获取这些资产的成本。

作为补充模型，还应对组织进行影响建模，以便为每个已识别资产的丢失事件提供更准确的“假设？”情景视图。这应考虑资产的“净”价值、其内在价值以及与丢失事件相关的其他间接成本。

任何渗透测试项目的威胁建模阶段对于测试人员和组织都至关重要。它清晰地阐明了组织的风险承受能力和优先级（哪些资产比其他资产更重要？哪些威胁社区比其他威胁社区更相关？）。此外，它使测试人员能够专注于交付一个紧密模拟攻击者的工具、技术、能力、可访问性和一般特征的渗透测试，同时牢记组织内部的实际目标是什么，以便测试更相关的控制措施、流程和基础设施，而不是IT元素的清单。

威胁模型应尽可能与被测组织协调构建，即使在完全黑盒的情况下（测试人员没有任何关于组织的事先信息），测试人员也应根据攻击者的视角并结合与目标组织相关的OSINT来创建威胁模型。

该模型应清晰记录，并作为最终报告的一部分交付，因为报告中的发现将引用威胁模型，以便创建特定于组织的更准确的关联性和风险评分（而不是通用的技术评分）。

### 高层级威胁建模过程

1. 收集相关文档
2. 识别并分类主要和次要资产
3. 识别并分类威胁和威胁社区
4. 将威胁社区映射到主要和次要资产

### 示例

在PTES评估中，内部托管的CRM应用程序可能在范围内。存储在后端数据库中的客户信息是一个易于识别的主要资产，因为它直接与范围内的应用程序相关联。然而，通过审查数据库服务器的技术设计，还可以识别出存储在同一个后端数据库服务器上的HR数据库是一个次要资产。攻击者可以利用CRM应用程序作为跳板来获取员工信息。在基本的威胁建模练习中，某些威胁社区在映射到CRM应用程序时可能被认为不相关，但通过识别次要资产，威胁格局突然发生了变化。

### 高层级建模工具

有多种工具可用于识别目标和映射攻击向量。这些工具通常侧重于业务资产（要攻击哪些系统）和业务流程（如何攻击它们）。根据渗透测试项目，渗透测试团队可能在没有任何客户输入的情况下执行这些练习；也可能花费大量时间与客户利益相关者一起识别感兴趣的目标。侧重于业务资产的工具通常需要定量输入来描述每个潜在目标对测试的重要性。输入也可以是定性的，例如客户CIO描述某个系统是任务关键型的。

侧重于业务流程、信息流和技术架构的工具用于识别潜在的攻击向量，并选择哪些最有可能成功，或最可能被某类对手使用。

## 业务资产分析

在威胁建模练习的业务资产分析部分，对所有范围内的资产及其支持的业务流程采用以资产为中心的视角。通过分析收集到的文档并与组织内的相关人员进行访谈，渗透测试人员能够识别出攻击者最可能瞄准的资产、它们的价值以及（部分）丢失这些资产会造成什么影响。

### 组织数据

#### 政策、计划和程序
内部政策、计划和程序定义了组织的业务方式。这些文档特别重要，因为它们可以帮助识别组织内的关键角色以及维持公司运营的关键业务流程。

#### 产品信息（例如商业秘密、研发数据）
产品相关信息包括任何专利、商业秘密、未来计划、源代码、直接影响产品市场价值的支持系统、算法以及组织视为该产品业务成功关键因素的任何其他信息。

#### 营销信息（计划、路线图等）
促销、产品发布、产品变更、定位、合作伙伴关系、第三方供应商、与组织内部或外部活动相关的业务计划等营销计划。
此外，公关相关数据，如合作伙伴、记者、咨询公司的详细信息，以及与此类实体的任何通信，也被视为高度受追捧的目标。

#### 财务信息（例如银行、信贷、股权账户）
财务信息通常是组织拥有的一些最受保护的信息。这些信息可以包括银行账户信息、信用卡账户信息和/或信用卡号以及投资账户等。

#### 技术信息
关于组织及其运营的技术信息对渗透测试人员具有特殊的兴趣。此类信息通常不是渗透测试的预期交付成果，然而，它通过向其他领域提供有价值的信息来促进测试过程；基础设施设计信息可能为情报收集过程提供有价值的数据。

* **基础设施设计信息**
  基础设施设计相关信息涉及用于运行组织的所有核心技术和设施。建筑蓝图、技术布线和连接图、计算设备/网络设计以及应用程序级数据处理都被视为基础设施设计信息。
* **系统配置信息**
  系统配置信息包括配置基线文档、配置清单和加固程序、组策略信息、操作系统镜像、软件清单等。这些信息可能有助于发现漏洞（例如通过了解配置错误或过时的软件安装）。
* **用户账户凭据**
  用户账户凭据有助于在存在身份验证手段（例如VPN、Web门户等）的情况下，以非特权级别促进对信息系统的访问。
* **特权用户账户凭据**
  特权用户账户凭据有助于在存在身份验证手段（例如VPN、Web门户等）的情况下，以提升的访问级别促进对信息系统的访问。获取特权用户账户凭据通常会导致被测信息系统被攻陷。

#### 员工数据
这里分析员工数据是指任何一旦被攻击者获取或攻陷就可能对组织产生直接影响的数据。必须遵守某些合规性要求的组织，如果此类数据丢失或暴露会被处以罚款，显然是这种直接损失后果的候选对象。此外，员工可能被视为关键资产的组织也可能受到此类审查（特定的政府机构、与商业秘密相关的专门员工/部门等）。

以下列表提供了可能被视为威胁建模中业务资产的个人数据领域的示例：
* 国民识别号码（SSN等）
* 个人身份信息（PII）
* 受保护的健康信息（PHI）
* 财务信息（例如银行、信贷账户）

#### 客户数据
与员工数据非常相似，在威胁建模过程中，当此类信息的丢失会给组织带来直接/间接损失时，客户数据被视为业务资产。除了监管/合规需求（基于罚款）之外，当此类数据可用于实施欺诈时，还有一个额外的因素在起作用，在这种情况下，组织可能需要为与欺诈相关的损失承担责任或被起诉（基于导致欺诈发生的客户信息丢失）。

以下列表提供了可能包含相关客户数据并应在威胁建模中视为业务资产的此类信息领域的示例：
* 国民识别号码（SSN等）
* 个人身份信息（PII）
* 受保护的健康信息（PHI）
* 财务账户（例如银行、信贷、股权账户）

#### 供应商数据
与供应商相关的、对组织至关重要的信息（例如关键组件制造商、可能与商业秘密相关的供应商协议、供应组件的成本分析），以及任何可能通过供应商影响组织业务运营的数据，都被视为业务资产。

### 人力资源资产
在识别组织中的人力资产时，我们必须记住，其背景是这些资产是更大规模攻陷组织努力的一部分。因此，被识别为业务资产的人力资产是那些可能被利用来泄露信息、被操纵做出会对组织产生不利影响或使攻击者能够进一步攻陷组织的决策或行动的人员。

人力资源资产不一定是公司层级中最高级别的，而更多是与先前识别的业务资产相关的关键人员，或者处于能够访问此类资产的位置的人员。此列表还可以包括通常与访问受限公司资产无关的员工，但可能处于能够为破坏安全或程序的行为提供物理访问的位置。

以下列表提供了此类资产的一些示例，并应针对被测组织进行调整：
* 执行管理层
* 行政助理
* 中层管理人员
* 行政助理
* 技术/团队负责人
* 工程师
* 技术人员
* 人力资源

## 业务流程分析
如果一个企业不赚钱，那它就不是一个企业。赚钱的方式是将原材料或知识通过各个流程进行加工，创造附加值。这会产生收入。业务流程和支持它们的资产（人员、技术、资金）构成了价值链。通过映射这些流程，识别关键与非关键流程，并最终发现其中的缺陷，我们能够理解业务是如何运作的，是什么为他们赚钱，以及最终特定的威胁社区如何让他们赔钱。

在业务流程分析中，我们区分关键业务流程和非关键业务流程。对于每个类别，分析是相同的，并考虑相同的要素。主要区别在于关键业务流程的威胁权重与非关键业务流程的权重不同。

尽管如此，必须记住，几个非关键业务流程的聚合可以组合成一个场景，本质上在一个元素/流程中形成一个关键缺陷。此类威胁场景也应在该阶段被识别并映射出来，以便在渗透测试中后续使用。

### 支持流程的技术基础设施
由于业务流程通常由IT基础设施支持（例如计算机网络、处理能力、用于输入信息和管理业务流程的PC等），所有这些要素都必须被识别和映射。这种映射应该足够清晰，以便在后续阶段将威胁模型转换为漏洞映射和利用时使用。

### 支持流程的信息资产
与技术基础设施相反，信息资产是组织中现有的知识库，用作参考或支持材料（决策、法律、营销等）。此类资产通常已在业务流程中识别，应与技术基础设施一起映射，以及支持信息资产本身的任何其他技术基础设施。

### 支持流程的人力资产
与流程分析本身（无论是否记录在案）一起，应识别参与业务流程的人力资源，并且每个有任何形式参与的人员（即使与特定信息资产或技术基础设施元素无关）都应被记录并映射到流程中。此类人力资源资产通常是审批子流程、验证子流程甚至参考（如法律建议）的一部分。这些类型的资产（尤其是那些与信息资产或技术基础设施无关的资产）稍后将映射到本质上更具社会性而非技术性的攻击向量。

### 第三方的集成和/或流程的使用
类似于支持流程的人力资产，任何与业务流程有任何关系的第三方也应被映射。此类别可能难以映射，因为它可能同时包含人力资源资产以及信息/技术资产（例如SaaS提供商）。

## 威胁代理/社区分析
在定义相关的威胁社区和代理时，应明确识别威胁的位置（组织内部/外部）、该位置内的特定社区，以及任何有助于建立特定代理/社区的能力/动机资料的其他相关信息。

在可能的情况下，应识别特定的代理。否则，应概述更一般的社区，以及任何支持材料和情报。

以下是一些威胁代理/社区分类的示例：

| 内部 | 外部 |
| :--- | :--- |
| 员工 | 业务合作伙伴 |
| 管理层（高管、中层） | 竞争对手 |
| 管理员（网络、系统、服务器） | 承包商 |
| 开发人员 | 供应商 |
| 工程师 | 国家行为体 |
| 技术人员 | 有组织犯罪 |

### 员工
以兼职或全职合同直接为公司工作的人员。通常，他们不被视为构成严重威胁，因为他们中的大多数依赖公司谋生，并且假设他们受到良好对待，他们倾向于保护公司而不是伤害公司。通常涉及数据丢失事件或意外泄露。在极少数情况下，他们可能被外部人员激励协助入侵，或者可能自行进行恶意行为（例如流氓交易员）。虽然技能水平可能有所不同，但通常为低到中等。

## 威胁能力分析
一旦识别出威胁社区，还必须分析该社区的能力，以便构建一个准确的威胁模型，反映此类社区/代理成功对组织采取行动并攻陷它的实际概率。

该分析既需要技术分析，也需要（在适用的情况下）机会分析。

### 使用中的工具分析
任何已知对威胁社区/代理可用的工具都应包含在此处。此外，应分析可能免费提供的工具，了解充分利用它们所需的技能水平，并将其映射到威胁能力中。

### 相关漏洞利用/载荷的可用性
应从获取或开发针对组织相关环境的漏洞利用的能力方面分析威胁社区/代理。此外，通过第三方、业务合作伙伴或地下社区对此类漏洞利用/载荷的可访问性也应在该分析中考虑。

### 通信机制
应对威胁代理/社区可用的通信机制进行分析，以评估针对组织的攻击的复杂性。这些通信机制范围从简单且公开可用的技术（如加密）到专业工具和服务（如抗投诉托管、投放点的使用，以及使用已知或未知的僵尸网络来执行攻击或掩盖源信息）。

例如，作为测试的一部分，我们测试从外部看组织的整体攻击面是什么。然而，还有另一个经常被遗漏的完整组件。后利用阶段可能存在哪些类型的威胁？这属于检测外泄通道的上下文。巧合的是，渗透测试人员处于独特的位置，可以测试组织检测当今现代恶意软件的命令和控制通道的能力。

当这在范围内时，我们建议测试人员创建一系列恶意软件样本，逐步增加用于隐藏C2的混淆级别。目标是创建容易检测的恶意软件，然后逐渐增加混淆程度，直到不再被检测到。

### 可访问性
威胁行为者能力分析的最后一个要素是他们可以访问组织和/或相关特定资产的程度。完成上述资料并考虑可访问性分析将使渗透测试能够创建与组织风险相关的清晰场景。

## 动机建模
应记录威胁代理/社区的可能动机以供进一步分析。攻击者的动机不断变化，从像Anonymous和Antisec这样的团体发起的黑客行动主义品牌攻击的增加可以看出。基于每个组织和/或垂直市场，独特动机会有细微差别，一些常见的动机包括：
* 利润（直接或间接）
* 黑客行动主义
* 直接怨恨
* 乐趣/声誉
* 进一步访问合作伙伴/连接的系统

## 寻找可比组织被攻陷的相关新闻
为了提供一个完整的威胁模型，应提供与同一垂直行业内的其他组织的比较。此比较应包括与此类组织相关的任何事件或新闻以及它们面临的挑战。这种比较用于验证威胁模型，并为组织提供一个与其自身进行比较的基线（考虑到这些公开可用的信息仅代表所比较组织实际面临的威胁和事件的一部分）。
