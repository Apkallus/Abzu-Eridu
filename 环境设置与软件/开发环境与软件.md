物理机 vscode 客户端连接虚拟机 SSH 服务器 [vscode 文档](https://code.visualstudio.com/docs/remote/ssh)。[教程](https://umatechnology.org/how-to-enable-ssh-in-vmware-workstation/)
- 物理机 vscode 安装 `Remote - SSH` 插件
- 虚拟机
    - 终端命令
        - 安装 ssh 服务器
            `sudo apt-get install openssh-server`

        - 启用 ssh 服务
            `sudo systemctl enable --now ssh`
            `sudo systemctl status ssh`

        - 若 UFW 启用则允许 ssh 服务连接
            `sudo ufw status`
            `sudo ufw allow ssh`

- 在物理机测试虚拟机的连接 
    - 连接虚拟机
        `ssh 虚拟机用户名@主机名` 

    - 显示确认提示后输入 yes
        `Are you sure you want to continue connecting (yes/no/[fingerprint])? yes`

    - 输入密码后连接到虚拟机

- 使用物理机 ssh 密钥
    - 物理机创建密钥
        `ssh-keygen -f key -N '密码'`
        - key 为私钥
        - key.pub 为公钥
    - 查看公钥
        `type key.pub`
    - 复制物理机公钥到虚拟机 ssh 授权配置文件
        `echo '公钥内容' >> ./.ssh/authorized_keys`
    - 使用 ssh 私钥连接虚拟机
        `ssh -i key 虚拟机用户名@主机名`
- 设置虚拟机的服务器 ssh 配置文件 
    `/etc/ssh/sshd_config`

    ```log
    # 开启公钥验证
    PubkeyAuthentication yes
    # 关闭密码登录
    PasswordAuthentication no
    # 关闭 root 登录
    PermitRootLogin no
    ```
    - 重启 ssh 服务以应用修改
        `sudo systemctl restart ssh`

- 设置物理机的客户端 ssh 配置文件
    - vscode 的远程资源管理器中打开 ssh 配置文件并设置参数
        `~/.ssh/config`
        - `HostName 主机名`
        - `User 用户名`
        - `IdentityFile 私钥文件`
- 此时物理机 vscode 的 ssh 客户端成功连接到虚拟机 ssh 服务器
- 之后在 vmware 中手动启动并在后台运行，或使用 vmrun 命令行工具指定虚拟机后台运行

#### WSL (Windows 的 Linux 子系统)

- [WSL Github releases](https://github.com/microsoft/WSL/releases)

- [wsl ipv6](https://learn.microsoft.com/en-us/windows/wsl/networking#ipv6-access)
    - 设置 WSL 为镜像模式
    - 设置防火墙

    ⚠ 镜像模式下 nmap 的原始套接字 sS 包的接收异常：
    
    WSL 在开启镜像模式时（官网文档中启用 ipv6的方式）:
    - 运行 nmap 的 -sS 时
        - 在物理机运行的流量嗅探总能够得到 SA 响应
        - 但 WSL 运行的流量嗅探偶尔才可得到 SA 响应（nmap 正确根据是否得到 SA 响应标记open或过滤）。
    - 运行 nmap 的 -sT 时，一切正常，物理机与 WSL 得到所有 SA 响应

    wsl 在关闭镜像模式时：
    - 一切正常，物理机与 WSL 得到所有 SA 响应

    即，启用 WSL 网络设置中的镜像模式导致使用原始套接字（-sS）的选项大多时候失灵，仅偶尔可得到 SA 响应。

    WSL 的 github 项目在几年前即存在类似的[报告](https://github.com/microsoft/WSL/issues/12185)，项目组无意修复

WSL 子系统使用物理机的转发 VPN

用法：
- 审查用户提供/选定的材料
    - 用户：提供基础材料作为核心/种子
    - LLM：对材料进行扩展介绍/debug/模式匹配
- 显式使用功能提示语(搜索、浏览)，以确保 LLM 使用对应功能

高效：
- LLM 仅为发散的可能性，最佳效果为对人类选定材料的匹配/扩展/校对，列出少量要点后由人类快速识别。而不是在 LLM 生成材料后，由人类花费大量精力对全文纠正
