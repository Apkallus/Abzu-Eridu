### smartdns - 本地 DNS 服务器

<https://github.com/pymumu/smartdns>

1. docker 设置镜像后 pull 容器
2. 设置本地文件夹映射到 `/etc/smartdns`

    ```powershell
    docker run -d `
        --name smartdns `
        --restart=always `
        -p 53:53/udp `
        -p 6080:6080/tcp `
        -v E:\Abzu-Eridu\tool\smartdns\config:/etc/smartdns `
        -v E:\Abzu-Eridu\.cache\smartdns\datalib:/var/lib/smartdns `
        pymumu/smartdns:latest
    ```
    - `-d` 后台运行，`--name 容器名称`
    - `--restart=always` 总是重启
    - 53 dns 查询端口
    - 6080 web 界面端口
    - `-v` 映射文件，主配置与数据库

3. 设置本机的 DNS 服务器地址为本机 
    - ipv4 `127.0.0.1`
    - ipv6 `::1`

### CDN本地反向代理

本地反代：服务于并由最终用户设置的本地运行的反向代理
- 禁用 hosts 模式：导致各种失败的修改 hosts 文件到本地反代的 DNS 劫持
    - 劫持 DNS 到本地即触发 CSP 的错误
- 或需禁用干扰 DNS 查询的 DNS 重定向模式

工具
- [Steamcommunity 302](https://www.dogfight360.com/blog/18682/)
    - [说明手册](https://www.dogfight360.com/blog/knowledge-base/steamcommunity_302_13_manual/)

    - `hosts 模式设置` 
        - `自动修改系统hosts`
            - 修改 hosts 文件以 DNS 劫持到本机反向代理
        - 若开启，则需移除 google 验证码项
            - hosts 规则为 127.0.0.1 的本机地址，DNS 劫持导致连接失败
            - 使用 DNS 重定向模式或在 smartdns 设置 `recaptcha.net` 为中国 DNS 解析

    - `DNS 重定向模式`
        - `启用 DNS 重定向模式`
            - DNS 请求拦截并重定向，干扰 DNS 查询
        - 开启 `启用 DNS 重定向模式` 以及 `DNS 重定向 CDN 优选` 
            - 将 CDN IP 优选应用到**所有**目标为 CDN 地址的查询

    - `系统代理模式`
        - `自动修改系统代理`
            - 修改系统代理设置以使用正向代理
        - `停止监听 & 禁用代理` 
            - 未启用正向代理
        - `开启监听 & PAC 模式`
            - 自动修改功能将在 windows 的 `网络和 Internet - 代理 - 使用设置脚本` 中启用并设置脚本地址 `http://127.0.0.1/pac.txt?S302` 
            - 白名单正向代理，除默认列表外，可手动添加尝试应用 CDN IP 优选的域名

    - DNS 重定向模式异常
        - DNS 重定向模式仍将列表中的 github 重定向到本机而非 CDN IP，有时 firefox 无法访问页面
            - 似乎任何有助于不重定向 DNS 到本机的设置均有益
        - cloudflare 的 IP 双栈信息均被 DNS 重定向到 ipv4 地址，导致 ipv6 地址显式异常
            - 添加 `*ipv6*` 到黑名单后恢复
        - cloudflare speed 测速的丢包项目连接失败
            - 添加 `*speed*` 到黑名单后恢复

    - 系统代理模式
        - 似乎没有实际效果

    - 重定向的 DNS 似乎被 smartdns 缓存
    - 黑名单功能在设置与 smartdns 相同的10万条目的直连列表后启动缓慢
    - 黑名单无法覆盖白名单

- Firefox [下载](https://www.firefox.com/zh-CN/download/all/)
    或 ZAP 内置版 Firefox
    - 若进行 lab 时修改默认安全设置，测试漏洞在 Firefox 浏览器上的实际效果时需重置到默认设置或使用其他版本
    - :warning: 默认安全设置提供“全方位 Cookie 保护”，部分靶场内容无法在此设置下复现（跨站请求伪造）
        - 将“设置 - 隐私与安全 - 浏览器因素”从标准改为自定义并设置
    - :warning: 默认不代理本机地址，在进行 OWASP Juice Shop 时发现本机靶场流量不经过拦截代理服务器，搜索到[CSDN博客](https://blog.csdn.net/HUC0723/article/details/119953560)或[Microsoft云开发文档](https://learn.microsoft.com/en-us/microsoft-cloud/dev/dev-proxy/how-to/intercept-localhost-requests)：
        > 若要将 Mozilla Firefox 配置为向系统代理发送 localhost 的 URL 请求，需要将 network.proxy.allow_hijacking_localhost 首选项设置为 true。 为此，请在 Firefox 中打开 about:config 页面，搜索 network.proxy.allow_hijacking_localhost 首选项并将其设置为 true。
    - :warning: 默认使用更严格的同源策略，部分靶场内容无法在此设置下复现 （ XSS 子框架）
        - 在 Firefox 中打开 about:config 页面，搜索 security.fileuri.strict_origin_policy 首选项，将其设置为 true。

    - 排障
        - 连接被重置，仅当由于 TLS 版本不匹配时，才可通过降低 TLS 版本正常访问：
            `建立安全连接失败由于不能验证所收到的数据是否可信，无法显示您想要查看的页面。错误代码：PR_CONNECT_RESET_ERROR`
            - 打开浏览器高级设置页面 `about:config`
            - 设置 tls 版本控制 security.tls.version.min 为 1
            - 重启浏览器
    - 设置
        - 字体大小 18
        - 权限与数据 
            设置 `设备应用和服务` 与 `本地网络设备` 开启 `禁止新的访问`，从每次弹窗询问修改为默认拒绝
    - 高级设置 `about:config` 
        - 性能
            - `gfx.webrender = true` 启用硬件渲染 
    - 本地安装插件
        - [uBlock](https://github.com/gorhill/uBlock/tree/master/dist#firefox)
            - 当前插件构建使用了签名，于是无需按照旧版指南关闭火狐的签名验证

    - 安卓 firefox 在 `关于` 内点击几次 `logo` 后打开开发者选项
